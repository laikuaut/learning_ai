# 第6章：キャッシュとパフォーマンス 演習問題

## 基本問題

### 問題6-1：キャッシュの種類

以下のキャッシュの種類について、それぞれの特徴と配置場所を答えてください。

1. ブラウザキャッシュ
2. プロキシキャッシュ
3. CDNキャッシュ

<details>
<summary>解答</summary>

| 種類 | 配置場所 | 共有範囲 | 特徴 |
|------|---------|---------|------|
| **ブラウザキャッシュ** | クライアント端末（ブラウザ内） | そのユーザーのみ | ユーザー固有のデータもキャッシュ可能。Cache-Controlの`private`に対応 |
| **プロキシキャッシュ** | 企業や組織の中間サーバー | 組織内のユーザー | 社内で同じリソースへのアクセスを集約 |
| **CDNキャッシュ** | 世界各地のエッジサーバー | 全ユーザー | ユーザーに最も近いサーバーから配信。Cache-Controlの`public`に対応 |

</details>

### 問題6-2：Cache-Controlの理解

以下のCache-Controlヘッダーの意味をそれぞれ説明してください。

1. `Cache-Control: no-store`
2. `Cache-Control: no-cache`
3. `Cache-Control: public, max-age=31536000, immutable`
4. `Cache-Control: private, max-age=300`
5. `Cache-Control: public, max-age=60, s-maxage=3600`

<details>
<summary>解答</summary>

1. **`no-store`**: キャッシュを一切保存しない。毎回サーバーからデータを取得する。機密情報や個人データに使用する。

2. **`no-cache`**: キャッシュは保存するが、使用する前に毎回サーバーに確認する（条件付きリクエストで変更を確認）。常に最新であるべきHTMLページなどに使用する。

3. **`public, max-age=31536000, immutable`**: プロキシやCDNもキャッシュ可能（public）。1年間（31536000秒）キャッシュを新鮮と見なす（max-age）。キャッシュ期間中は再検証も行わない（immutable）。ハッシュ付きファイル名の静的リソースに使用する。

4. **`private, max-age=300`**: ブラウザのみキャッシュ可能（private）で、プロキシやCDNにはキャッシュされない。5分間（300秒）有効。ユーザー固有のデータに使用する。

5. **`public, max-age=60, s-maxage=3600`**: プロキシやCDNもキャッシュ可能（public）。ブラウザキャッシュは1分（60秒）有効。共有キャッシュ（CDN等）は1時間（3600秒）有効。ブラウザには頻繁に更新を確認させつつ、CDNではオリジンへの負荷を抑える設定。

</details>

### 問題6-3：no-cache vs no-store

以下のシナリオで、`no-cache` と `no-store` のどちらを使うべきか選び、理由を説明してください。

1. ユーザーのクレジットカード情報を含むページ
2. ニュースサイトのトップページ（数分ごとに更新される）
3. オンラインバンキングの残高照会ページ
4. 企業ブログの記事一覧ページ

<details>
<summary>解答</summary>

1. **`no-store`** - クレジットカード情報は機密性が非常に高いため、キャッシュに保存してはいけません。ディスクやメモリにデータが残るリスクを排除します。

2. **`no-cache`** - ニュースサイトは最新の情報を表示すべきですが、毎回全データを転送する必要はありません。`no-cache`を使えば、サーバーに変更を確認し、変更がなければ304で高速にレスポンスできます。

3. **`no-store`** - 残高情報は個人の機密情報であり、共有PCや公共端末での情報漏洩を防ぐため、キャッシュに保存してはいけません。

4. **`no-cache`** - ブログ記事の一覧は新しい記事が追加される可能性があるため最新性は重要ですが、機密情報ではないのでキャッシュの利用で効率化できます。

**判断基準:**
- 機密情報・個人情報 → `no-store`（キャッシュに一切残さない）
- 最新であるべきが機密ではないデータ → `no-cache`（キャッシュを利用しつつ毎回確認）

</details>

## 応用問題

### 問題6-4：条件付きリクエストの流れ

以下のHTTP通信の空欄を埋めてください。

**1回目のリクエスト:**

```
GET /api/data HTTP/1.1
Host: api.example.com
```

**1回目のレスポンス:**

```
HTTP/1.1 200 OK
Content-Type: application/json
ETag: "v1-abc123"
Last-Modified: Mon, 01 Apr 2026 10:00:00 GMT
Cache-Control: no-cache

{"data": "hello"}
```

**2回目のリクエスト（データ変更なしの場合）:**

```
GET /api/data HTTP/1.1
Host: api.example.com
(a): _______________
(b): _______________
```

**2回目のレスポンス（データ変更なしの場合）:**

```
HTTP/1.1 (c) _______________
ETag: "v1-abc123"
```

**3回目のリクエスト（データが変更された場合）:**

```
GET /api/data HTTP/1.1
Host: api.example.com
If-None-Match: "v1-abc123"
```

**3回目のレスポンス（データが変更された場合）:**

```
HTTP/1.1 (d) _______________
ETag: (e) _______________
Content-Type: application/json

{"data": "world"}
```

<details>
<summary>解答</summary>

**(a)**: `If-None-Match: "v1-abc123"` - ETagを使った条件付きリクエスト

**(b)**: `If-Modified-Since: Mon, 01 Apr 2026 10:00:00 GMT` - Last-Modifiedを使った条件付きリクエスト

**(c)**: `304 Not Modified` - データに変更がないため、ボディなしで304を返す

**(d)**: `200 OK` - データが変更されたため、新しいデータを200で返す

**(e)**: `"v2-xyz789"` などの新しいETag値 - データが変更されたのでETagも新しくなる

**解説**: 条件付きリクエストでは、ETag（If-None-Match）とLast-Modified（If-Modified-Since）の両方を送信できます。サーバーは両方を受け取った場合、ETagを優先して検証します。

</details>

### 問題6-5：HTTP/2の改善点

HTTP/1.1とHTTP/2の違いについて、以下の項目を比較して説明してください。

1. リクエストの同時送信
2. ヘッダーの扱い
3. サーバープッシュ
4. 接続数

<details>
<summary>解答</summary>

**1. リクエストの同時送信:**
- **HTTP/1.1**: 1つのTCP接続で1つずつ順番にリクエスト・レスポンスを処理（HOLブロッキングが発生）。並列化するにはTCP接続を複数開く必要がある（ブラウザは通常6〜8接続）
- **HTTP/2**: 1つのTCP接続で**複数のリクエスト・レスポンスを同時に並行処理**（多重化 / Multiplexing）。HOLブロッキングが解消される

**2. ヘッダーの扱い:**
- **HTTP/1.1**: ヘッダーをテキスト形式で毎回すべて送信する。CookieやUser-Agentなど、リクエストごとに同じ内容を繰り返し送信
- **HTTP/2**: **HPACK圧縮**によりヘッダーを圧縮。2回目以降は前回との差分のみ送信するため、大幅にデータ量を削減（90%以上の削減も可能）

**3. サーバープッシュ:**
- **HTTP/1.1**: サーバーはクライアントからのリクエストに対してのみレスポンスを返す
- **HTTP/2**: サーバーがクライアントのリクエストを予測して、先にリソースを送信できる（ただし、実用上の効果が限定的なため、Chrome 106でデフォルト無効化された）

**4. 接続数:**
- **HTTP/1.1**: 並列化のために**複数のTCP接続**を開く（ドメインあたり6〜8接続）
- **HTTP/2**: 原則**1つのTCP接続**で全リクエストを処理。TCP接続の確立コストとメモリ使用量を削減

</details>

### 問題6-6：キャッシュ戦略の設計

以下のWebアプリケーションのリソースに対して、適切なCache-Controlヘッダーを設計してください。それぞれの設定理由も説明してください。

1. `index.html`（アプリのエントリーポイント）
2. `app.a1b2c3d4.js`（ビルドでハッシュが付与されたJSファイル）
3. `logo.png`（企業ロゴ、めったに変わらない）
4. `/api/user/profile`（ログインユーザーのプロフィール情報）
5. `/api/products`（商品一覧、頻繁に更新される）

<details>
<summary>解答</summary>

```
# 1. index.html
Cache-Control: no-cache
```
**理由**: HTMLファイルはJS/CSSの参照URLを含むため、常に最新版を取得する必要があります。`no-cache`を使うことで、毎回サーバーに確認しつつ、変更がなければ304で高速レスポンスできます。

```
# 2. app.a1b2c3d4.js
Cache-Control: public, max-age=31536000, immutable
```
**理由**: ファイル名にハッシュ（a1b2c3d4）が含まれているため、内容が変われば別のURLになります。1年間の長期キャッシュを設定し、`immutable`で不要な再検証リクエストも防止できます。CDNでもキャッシュするため`public`を指定します。

```
# 3. logo.png
Cache-Control: public, max-age=604800
```
**理由**: ロゴはめったに変わりませんが、ハッシュ付きファイル名ではないため、永続キャッシュは避けます。1週間（604800秒）のキャッシュで、変更時は1週間以内に反映されます。ETagも併用すると更新の検知が可能です。

```
# 4. /api/user/profile
Cache-Control: private, no-store
```
**理由**: ユーザー固有の個人情報なので、CDNやプロキシにキャッシュさせてはいけません。`no-store`でブラウザにもキャッシュさせず、常に最新データを取得します。

```
# 5. /api/products
Cache-Control: public, max-age=30, s-maxage=300, must-revalidate
```
**理由**: 商品一覧は全ユーザー共通で、頻繁に更新されます。ブラウザキャッシュは30秒と短くし、CDNキャッシュは5分（300秒）で設定。`must-revalidate`で期限切れ後は必ずサーバーに確認させます。

</details>

### 問題6-7：圧縮の効果

以下の問いに答えてください。

1. gzip圧縮が特に効果的なファイル形式を3つ挙げてください
2. gzip圧縮がほとんど効果のないファイル形式を2つ挙げ、その理由を説明してください
3. BrotliとgzipではどちらがHTTPS環境で推奨されますか？理由も答えてください
4. 圧縮に関するHTTPヘッダーを、リクエスト側とレスポンス側それぞれ1つずつ答えてください

<details>
<summary>解答</summary>

**1. gzip圧縮が特に効果的なファイル形式:**
- **HTML** （テキストベースで冗長な構造が多い、約75%削減）
- **CSS** （繰り返しのプロパティ名やセレクタが多い、約80%削減）
- **JSON** （キー名の繰り返しが多い、約85%削減）
- その他: JavaScript、XML、SVG なども効果的

**2. gzip圧縮がほとんど効果のないファイル形式:**
- **JPEG画像** - すでにJPEG形式で圧縮されているため、追加圧縮の余地がほとんどない
- **PNG画像** - すでにDeflate圧縮が適用されているため、gzipで二重圧縮してもサイズが変わらない（逆にわずかに大きくなることもある）
- その他: MP4動画、MP3音声、ZIP/GZファイルなど

**3. BrotliがHTTPS環境で推奨される理由:**
- **Brotliが推奨**されます
- Brotliはgzipより**20〜30%高い圧縮率**を達成できる
- Brotliは**HTTPS通信でのみ利用可能**（HTTP通信ではブラウザが対応しない）
- 圧縮速度はgzipよりやや遅いが、静的ファイルは事前圧縮が可能なため問題にならない
- 主要ブラウザはすべてBrotliに対応済み

**4. 圧縮に関するHTTPヘッダー:**
- **リクエスト側**: `Accept-Encoding: gzip, br`（クライアントが対応する圧縮方式を伝える）
- **レスポンス側**: `Content-Encoding: gzip`（サーバーが使用した圧縮方式を伝える）

</details>

## チャレンジ問題

### 問題6-8：Pythonで条件付きリクエストを実装

Pythonの `http.client` を使って、以下の処理を行うプログラムを作成してください。

1. `https://www.example.com/` にGETリクエストを送信する
2. レスポンスからETagとLast-Modifiedを取得する
3. 取得したETagを使って条件付きリクエスト（If-None-Match）を送信する
4. 1回目と2回目のステータスコードとデータ転送量を比較して表示する

<details>
<summary>ヒント</summary>

- `response.getheader("ETag")` でETagを取得できます
- `If-None-Match` ヘッダーにETagの値をセットして送信します
- 304レスポンスの場合、ボディは空（0バイト）です

</details>

<details>
<summary>解答</summary>

```python
import http.client

def make_request(conn, path, headers=None):
    """リクエストを送信してレスポンス情報を返す"""
    conn.request("GET", path, headers=headers or {})
    response = conn.getresponse()
    body = response.read()
    return {
        "status": response.status,
        "reason": response.reason,
        "etag": response.getheader("ETag"),
        "last_modified": response.getheader("Last-Modified"),
        "content_length": len(body),
        "headers": dict(response.getheaders()),
    }

# --- 1回目: 通常のリクエスト ---
print("=== 1回目: 通常のリクエスト ===")
conn = http.client.HTTPSConnection("www.example.com")
result1 = make_request(conn, "/")
conn.close()

print(f"ステータス: {result1['status']} {result1['reason']}")
print(f"ETag: {result1['etag']}")
print(f"Last-Modified: {result1['last_modified']}")
print(f"受信データサイズ: {result1['content_length']} バイト")

# --- 2回目: 条件付きリクエスト (If-None-Match) ---
print("\n=== 2回目: 条件付きリクエスト (ETag使用) ===")
conn = http.client.HTTPSConnection("www.example.com")
headers = {}
if result1["etag"]:
    headers["If-None-Match"] = result1["etag"]
result2 = make_request(conn, "/", headers)
conn.close()

print(f"ステータス: {result2['status']} {result2['reason']}")
print(f"受信データサイズ: {result2['content_length']} バイト")

# --- 比較結果 ---
print("\n=== 比較結果 ===")
print(f"1回目: {result1['status']} - {result1['content_length']} バイト")
print(f"2回目: {result2['status']} - {result2['content_length']} バイト")

if result2["status"] == 304:
    saved = result1["content_length"] - result2["content_length"]
    print(f"節約量: {saved} バイト ({saved / result1['content_length'] * 100:.1f}% 削減)")
    print("→ サーバーはデータに変更がないことを確認し、ボディを送信しませんでした")
else:
    print("→ データが変更されていたため、新しいデータが送信されました")
```

**実行結果の例:**

```
=== 1回目: 通常のリクエスト ===
ステータス: 200 OK
ETag: "3147526947+gzip"
Last-Modified: Thu, 17 Oct 2019 07:18:26 GMT
受信データサイズ: 1256 バイト

=== 2回目: 条件付きリクエスト (ETag使用) ===
ステータス: 304 Not Modified
受信データサイズ: 0 バイト

=== 比較結果 ===
1回目: 200 - 1256 バイト
2回目: 304 - 0 バイト
節約量: 1256 バイト (100.0% 削減)
→ サーバーはデータに変更がないことを確認し、ボディを送信しませんでした
```

</details>

### 問題6-9：総合問題 - パフォーマンス最適化

以下のWebサイトのパフォーマンスを改善してください。現状の構成を見て、問題点を5つ以上指摘し、それぞれの改善策を具体的に（使用するHTTPヘッダーやツールを含めて）記述してください。

**現状の構成:**

```
サイト: http://myshop.example.com（ECサイト）
- HTTP/1.1 を使用
- HTTPS未対応
- すべてのリソースにキャッシュ設定なし
- 画像は元サイズ（5MB程度のJPEG）をそのまま配信
- CSS/JSファイルは圧縮せずに配信
- 全リソースを1台のオリジンサーバーから配信
- CSSファイル名: style.css（更新のたびにクエリパラメータ ?v=123 で管理）
```

<details>
<summary>ヒント</summary>

以下の観点から考えてみましょう。
- セキュリティ（HTTPS）
- プロトコルバージョン（HTTP/2以上）
- キャッシュ戦略
- 圧縮
- CDNの活用
- 画像の最適化
- ファイルのバージョニング

</details>

<details>
<summary>解答</summary>

**問題点と改善策:**

**1. HTTPS未対応**
- **問題**: 通信が暗号化されておらず、盗聴・改ざん・なりすましのリスクがある。ブラウザが「保護されていない通信」と警告を表示し、ユーザーの信頼を損なう
- **改善策**: Let's EncryptでSSL証明書を取得し、HTTPSを有効化。HTTP→HTTPSの301リダイレクトとHSTSを設定
  ```
  Strict-Transport-Security: max-age=31536000; includeSubDomains
  ```

**2. HTTP/1.1 の使用**
- **問題**: リクエストの多重化ができず、HOLブロッキングが発生。多数の小さなリソースの読み込みが遅い
- **改善策**: HTTP/2（またはHTTP/3）を有効化。1つのTCP接続で多重化が可能になり、ヘッダー圧縮（HPACK）も適用される

**3. キャッシュ設定なし**
- **問題**: 毎回すべてのリソースをサーバーから再取得するため、表示が遅く、帯域幅も浪費
- **改善策**: リソースの種類に応じてCache-Controlを設定
  ```
  # HTML
  Cache-Control: no-cache
  
  # CSS/JS（ハッシュ付きファイル名の場合）
  Cache-Control: public, max-age=31536000, immutable
  
  # 画像
  Cache-Control: public, max-age=604800
  
  # APIレスポンス（個人情報）
  Cache-Control: private, no-store
  ```

**4. CSS/JSの非圧縮配信**
- **問題**: テキストファイルを圧縮せずに配信しており、データ転送量が不必要に多い
- **改善策**: サーバーでgzip/Brotli圧縮を有効化。Accept-EncodingヘッダーでBrotli対応のクライアントにはBrotli、それ以外にはgzipで配信
  ```
  Content-Encoding: br
  ```
  （CSS/JSで70〜85%のサイズ削減が期待できる）

**5. 画像の未最適化**
- **問題**: 5MBの元画像をそのまま配信しており、ページ読み込みが非常に遅い
- **改善策**: 
  - 画像を適切なサイズにリサイズ（表示サイズに合わせる）
  - WebP/AVIF形式に変換（JPEGより30〜50%小さい）
  - レスポンシブイメージ（`<picture>`タグやsrcset）で画面サイズに合った画像を配信
  - 遅延読み込み（`loading="lazy"`）で初期表示を高速化

**6. CDN未使用**
- **問題**: 全リソースを1台のオリジンサーバーから配信しており、地理的に遠いユーザーのレイテンシが高い。サーバーに負荷が集中
- **改善策**: Cloudflare、CloudFront等のCDNを導入し、静的リソースをエッジサーバーから配信。`s-maxage`でCDNのキャッシュ時間を別途制御

**7. クエリパラメータによるバージョン管理**
- **問題**: `?v=123` によるバージョン管理は、一部のプロキシやCDNがクエリパラメータの違いを無視してキャッシュする可能性がある
- **改善策**: ファイル名にハッシュを含める方式に変更
  ```
  ❌ style.css?v=123
  ✅ style.a1b2c3d4.css
  ```
  ビルドツール（webpack, Vite等）で自動的にハッシュを付与する設定にする

</details>

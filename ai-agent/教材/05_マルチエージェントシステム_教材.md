## 5.1 マルチエージェントシステムとは

### 定義

マルチエージェントシステム（Multi-Agent System, MAS）とは、複数のAIエージェントが協調・競争しながらタスクを遂行するシステムです。各エージェントは独自の役割・知識・能力を持ち、相互にやり取りしながら、単一のエージェントでは困難な複雑なタスクを解決します。

```
┌─────────────────────────────────────────────┐
│         マルチエージェントシステム              │
│                                             │
│  ┌─────────┐   通信   ┌─────────┐          │
│  │エージェントA│◄──────►│エージェントB│          │
│  │(リサーチ) │         │(分析)    │          │
│  └────┬────┘         └────┬────┘          │
│       │                   │               │
│       │      通信         │               │
│       └──────┐  ┌────────┘               │
│              ▼  ▼                         │
│         ┌─────────┐                       │
│         │エージェントC│                       │
│         │(レポート) │                       │
│         └─────────┘                       │
│                                             │
│  共有環境（メモリ・ツール・データ）              │
└─────────────────────────────────────────────┘
```

### シングルエージェントとの違い

| 観点 | シングルエージェント | マルチエージェント |
|------|---------------------|-------------------|
| 構造 | 1つのエージェントが全処理を担当 | 複数エージェントが分担 |
| 専門性 | 汎用的（何でも1人で対応） | 各エージェントが専門特化 |
| スケーラビリティ | コンテキスト長の制約を受けやすい | 役割ごとにコンテキストを分離 |
| 信頼性 | 単一障害点になりやすい | 冗長性を持たせやすい |
| コスト | API呼び出し回数が少ない | API呼び出し回数が増加 |
| 複雑さ | 実装がシンプル | 通信・協調の設計が必要 |

### なぜ複数エージェントが必要か

1. **専門性の確保**: 1つのプロンプトに全ての指示を詰め込むと品質が低下します。役割を分けることで、各エージェントの専門性を高められます
2. **コンテキスト長の制約回避**: LLMにはトークン制限があります。タスクを分割することで、各エージェントが必要な情報だけを扱えます
3. **品質向上**: レビュアーやバリデーターを別エージェントにすることで、チェック体制を構築できます
4. **並列処理**: 独立したタスクを複数エージェントが同時に処理できます

> **ポイントまとめ**
> - マルチエージェントシステムは複数の専門エージェントが協調するシステム
> - シングルエージェントでは難しい複雑なタスクに有効
> - コンテキスト分離・専門特化・品質チェックがメリット
> - ただしコストと複雑さが増すトレードオフがある

## 5.2 エージェント間通信パターン

エージェント同士がどのように情報をやり取りするかは、システム設計の根幹です。主要な4つのパターンを紹介します。

### 直接通信（Direct Communication）

エージェントが互いに直接メッセージを送り合うパターンです。最もシンプルですが、エージェント数が増えると通信経路が爆発的に増加します。

```python
# 直接通信の概念コード
class Agent:
    def __init__(self, name: str, role: str):
        self.name = name
        self.role = role

    def send_message(self, recipient: "Agent", message: str) -> str:
        """相手エージェントに直接メッセージを送る"""
        print(f"[{self.name}] → [{recipient.name}]: {message[:50]}...")
        return recipient.receive_message(self, message)

    def receive_message(self, sender: "Agent", message: str) -> str:
        """メッセージを受け取り応答する"""
        response = f"{self.name}が{sender.name}のメッセージを処理しました"
        return response

# 使用例
researcher = Agent("リサーチャー", "調査担当")
writer = Agent("ライター", "執筆担当")
researcher.send_message(writer, "調査結果をまとめました: ...")
```

### メッセージパッシング（Message Passing）

中央のメッセージキュー（Message Queue）を介して非同期にメッセージを交換するパターンです。エージェント同士が直接依存しないため、疎結合（Loose Coupling）な設計が可能です。

```
┌────────┐    publish    ┌──────────────┐    subscribe    ┌────────┐
│Agent A │──────────────►│メッセージキュー│───────────────►│Agent B │
└────────┘               └──────────────┘                └────────┘
                               ▲                              │
                               │          publish             │
                               └──────────────────────────────┘
```

### 共有メモリ（Shared Memory）

全エージェントがアクセスできる共有のデータストアを用いるパターンです。状態の一元管理ができますが、競合制御が課題になります。

```python
from typing import Any

class SharedMemory:
    """エージェント間の共有メモリ"""
    def __init__(self):
        self._store: dict[str, Any] = {}

    def write(self, key: str, value: Any, author: str) -> None:
        self._store[key] = {
            "value": value,
            "author": author,
        }
        print(f"[SharedMemory] {author} が '{key}' を書き込みました")

    def read(self, key: str) -> Any:
        if key in self._store:
            return self._store[key]["value"]
        return None

    def list_keys(self) -> list[str]:
        return list(self._store.keys())

# 使用例
memory = SharedMemory()
memory.write("research_result", {"topic": "AI Agent", "findings": [...]}, "リサーチャー")
result = memory.read("research_result")  # ライターが読み取る
```

### ブラックボード方式（Blackboard Architecture）

共有メモリの発展形です。中央の「ブラックボード」に問題の現在の状態が書かれ、各エージェント（知識源）が自分の専門領域に関する部分を更新していきます。コントローラーが次にどのエージェントを呼ぶかを制御します。

```
┌─────────────────────────────────────────┐
│           コントローラー                  │
│    (次に実行するエージェントを選択)         │
└────────┬──────────┬──────────┬─────────┘
         │          │          │
    ┌────▼───┐ ┌───▼────┐ ┌──▼─────┐
    │知識源A  │ │知識源B  │ │知識源C  │
    └────┬───┘ └───┬────┘ └──┬─────┘
         │          │          │
    ┌────▼──────────▼──────────▼────┐
    │       ブラックボード            │
    │  (問題の現在の状態を保持)        │
    └───────────────────────────────┘
```

### 通信パターンの比較

| パターン | 結合度 | 複雑さ | 適用場面 |
|---------|--------|--------|---------|
| 直接通信 | 高い | 低い | エージェント数が少ない場合 |
| メッセージパッシング | 低い | 中程度 | 非同期処理、スケーラブルな設計 |
| 共有メモリ | 中程度 | 中程度 | 状態を共有しながら並列処理する場合 |
| ブラックボード | 低い | 高い | 複雑な問題を段階的に解く場合 |

> **よくある間違い**
> - 全てのエージェントに全情報を渡してしまう → コンテキストの汚染が起き、各エージェントの判断精度が下がります。必要な情報だけを渡す設計にしましょう
> - 通信のログを残さない → デバッグ時にどのエージェントが何を送ったか追跡不能になります。必ずログを残してください

> **ポイントまとめ**
> - 通信パターンはシステムの拡張性と保守性に直結する
> - 少数エージェントなら直接通信、スケールするならメッセージパッシング
> - 共有メモリは状態管理に便利だが競合に注意
> - ブラックボードは複雑な問題分割に適している

## 5.3 協調パターン

マルチエージェントがどのように連携してタスクを遂行するかの設計パターンを紹介します。

### 階層型（Hierarchical）：マネージャー/ワーカー

1つの上位エージェント（マネージャー）が全体を管理し、下位エージェント（ワーカー）にタスクを委譲するパターンです。最も直感的で実装しやすい構造です。

```
         ┌──────────┐
         │マネージャー│  ← タスクの分解と指示
         └─────┬────┘
        ┌──────┼──────┐
        ▼      ▼      ▼
   ┌─────┐ ┌─────┐ ┌─────┐
   │ワーカー│ │ワーカー│ │ワーカー│  ← 実作業
   │  A   │ │  B   │ │  C   │
   └─────┘ └─────┘ └─────┘
```

**メリット**: 制御が明確、デバッグしやすい、責任の所在が明確
**デメリット**: マネージャーがボトルネックになりうる、柔軟性が低い

### 水平型（Horizontal）：ピアツーピア

全エージェントが対等な立場で、互いに協力してタスクを進めるパターンです。

**メリット**: 柔軟性が高い、単一障害点がない
**デメリット**: 合意形成が難しい、制御が複雑

### ディベート型（Debate）

複数のエージェントが異なる立場から議論し、より良い結論を導き出すパターンです。特に意思決定や分析タスクに効果的です。

```python
def debate_round(topic: str, agents: list[dict], rounds: int = 3) -> str:
    """ディベート型の協調を模擬する概念コード"""
    history = []

    for round_num in range(1, rounds + 1):
        print(f"\n=== ラウンド {round_num} ===")
        for agent in agents:
            # 各エージェントが過去の議論を踏まえて意見を述べる
            prompt = f"""あなたは{agent['role']}です。
トピック: {topic}
これまでの議論:
{chr(10).join(history) if history else '(まだ議論は始まっていません)'}

あなたの立場から意見を述べてください。"""

            # LLM呼び出し（概念的）
            response = call_llm(prompt)  # 実際にはAPI呼び出し
            history.append(f"[{agent['role']}] {response}")
            print(f"[{agent['role']}]: {response[:100]}...")

    # 最終的にまとめ役が結論を出す
    summary_prompt = f"以下の議論を総合し、最終的な結論をまとめてください:\n{chr(10).join(history)}"
    return call_llm(summary_prompt)

# 使用例
agents = [
    {"role": "賛成派アナリスト", "stance": "肯定的"},
    {"role": "反対派アナリスト", "stance": "批判的"},
    {"role": "中立の調停者",     "stance": "中立"},
]
conclusion = debate_round("この技術を本番環境に導入すべきか", agents)
```

### リレー型（Relay / Sequential Pipeline）

エージェントがバトンリレーのように順番に処理を引き継ぐパターンです。各段階で成果物が洗練されていきます。

```
┌──────┐    成果物    ┌──────┐    成果物    ┌──────┐    成果物    ┌──────┐
│調査   │───────────►│設計   │───────────►│実装   │───────────►│検証   │
│エージェント│           │エージェント│           │エージェント│           │エージェント│
└──────┘             └──────┘             └──────┘             └──────┘
```

**メリット**: 流れが明確、各段階の品質管理がしやすい
**デメリット**: 前工程の失敗が後工程に伝播、手戻りが発生しやすい

### 協調パターンの使い分け

| パターン | 適したタスク | 例 |
|---------|------------|-----|
| 階層型 | 明確に分解できるタスク | プロジェクト管理、バッチ処理 |
| 水平型 | 創造的・探索的タスク | ブレインストーミング、研究 |
| ディベート型 | 意思決定、分析 | 技術選定、リスク評価 |
| リレー型 | 段階的に品質を高めるタスク | コンテンツ制作、ソフトウェア開発 |

> **ポイントまとめ**
> - 階層型は制御しやすくデバッグも容易 → まず最初に検討すべきパターン
> - ディベート型は意思決定の質を高めるのに有効
> - リレー型はソフトウェア開発パイプラインとの相性が良い
> - パターンを組み合わせることも可能（例: 階層型 + リレー型）

## 5.4 役割分担の設計

### 専門化のメリット

エージェントを専門化させることで、以下の効果が得られます。

1. **プロンプトの集中**: 1つの役割に絞ったシステムプロンプトは、LLMの出力品質を大幅に向上させます
2. **コンテキストの効率化**: 不要な情報を排除し、必要な情報だけを保持できます
3. **テスタビリティ**: 個々のエージェントを独立してテスト・改善できます
4. **再利用性**: 同じ専門エージェントを複数のシステムで活用できます

### 役割定義の方法

各エージェントには以下の要素を明確に定義します。

```python
agent_definition = {
    "name": "コードレビュアー",
    "role": "提出されたコードの品質を評価する専門家",
    "responsibilities": [
        "コードの可読性を評価する",
        "バグやセキュリティ上の問題を指摘する",
        "改善案を具体的に提示する",
    ],
    "constraints": [
        "コードの書き直しは行わない（指摘のみ）",
        "ビジネスロジックの正否は判断しない",
    ],
    "input_format": "レビュー対象のソースコード",
    "output_format": "問題点リスト（重要度・該当行・指摘内容・改善案）",
}
```

### コンテキスト管理

マルチエージェントシステムで最も重要な設計課題の1つがコンテキスト管理です。

```
  全体コンテキスト（プロジェクトの目標・制約）
  ├── エージェントAのコンテキスト
  │   ├── 共有コンテキスト（他エージェントから受け取った情報）
  │   └── 固有コンテキスト（自分の役割固有の情報）
  ├── エージェントBのコンテキスト
  │   ├── 共有コンテキスト
  │   └── 固有コンテキスト
  └── エージェントCのコンテキスト
      ├── 共有コンテキスト
      └── 固有コンテキスト
```

**設計原則**:
- 各エージェントには「必要最小限」の情報だけを渡す（最小権限の原則）
- 成果物のサマリーを次のエージェントに渡し、生データは共有メモリに保存する
- コンテキストウィンドウの消費量を常に意識する

> **よくある間違い**
> - 役割を曖昧にしてしまう → 「何でもできるエージェント」は何も上手くできません。具体的な責務を定義しましょう
> - 1つのエージェントに多すぎる役割を持たせる → シングルエージェントと同じ問題に陥ります。1エージェント1責務を心がけましょう
> - エージェント間で渡すデータのスキーマを決めない → 出力形式の不一致で後続エージェントがパースに失敗します

> **ポイントまとめ**
> - 役割は具体的に定義し、責務・制約・入出力を明確にする
> - コンテキストは必要最小限に絞り、トークン消費を抑える
> - エージェント間のインターフェース（入出力形式）を事前に設計する

## 5.5 実装例：チーム開発エージェント

ここでは、リレー型の協調パターンを用いた「チーム開発エージェント」の実装例を紹介します。PM（プロジェクトマネージャー）→ 設計者 → 開発者 → レビュアーの4段階で協調するフローです。

```python
"""
チーム開発エージェント - マルチエージェント協調の実装例
動作要件: Python 3.10以上、anthropic パッケージ
実行方法: python team_development_agent.py
"""

import anthropic

client = anthropic.Anthropic()  # ANTHROPIC_API_KEY 環境変数を使用
MODEL = "claude-sonnet-4-20250514"


def call_agent(role: str, system_prompt: str, user_message: str) -> str:
    """個別エージェントを呼び出す共通関数"""
    print(f"\n{'='*50}")
    print(f"[{role}] 処理を開始します...")
    print(f"{'='*50}")

    response = client.messages.create(
        model=MODEL,
        max_tokens=2048,
        system=system_prompt,
        messages=[{"role": "user", "content": user_message}],
    )
    result = response.content[0].text
    print(f"[{role}] 処理完了（{response.usage.output_tokens} tokens）")
    return result


def run_team_development(requirement: str) -> dict[str, str]:
    """チーム開発パイプラインを実行する"""
    results = {}

    # ステップ1: PMがタスクを分析・分解
    pm_system = """あなたはプロジェクトマネージャーです。
ユーザーの要件を分析し、以下の形式でタスクを整理してください：
1. 要件の明確化（何を作るか）
2. 技術要件（使用する技術・制約）
3. 実装タスクのリスト（優先度付き）
4. 設計者への指示"""

    results["pm"] = call_agent(
        "PM（プロジェクトマネージャー）",
        pm_system,
        f"以下の要件を分析してください:\n{requirement}",
    )

    # ステップ2: 設計者がアーキテクチャを設計
    designer_system = """あなたはソフトウェア設計者です。
PMからの指示を基に、以下を含む設計書を作成してください：
1. 全体アーキテクチャ
2. データ構造の定義
3. 関数/クラスのインターフェース設計
4. 開発者への実装指示"""

    results["designer"] = call_agent(
        "設計者",
        designer_system,
        f"PMからの分析結果:\n{results['pm']}\n\n上記を基に設計してください。",
    )

    # ステップ3: 開発者がコードを実装
    developer_system = """あなたはソフトウェア開発者です。
設計書に基づいて、動作するPythonコードを実装してください。
- コードは完全に動作するものにしてください
- 適切なコメントを付けてください
- エラーハンドリングを含めてください"""

    results["developer"] = call_agent(
        "開発者",
        developer_system,
        f"設計書:\n{results['designer']}\n\n上記の設計に基づいて実装してください。",
    )

    # ステップ4: レビュアーがコードをレビュー
    reviewer_system = """あなたはシニアコードレビュアーです。
提出されたコードを以下の観点でレビューしてください：
1. バグの有無
2. セキュリティ上の問題
3. パフォーマンスの懸念
4. コードの可読性
5. 改善提案

問題がある場合は具体的な修正案を提示してください。"""

    results["reviewer"] = call_agent(
        "レビュアー",
        reviewer_system,
        f"以下のコードをレビューしてください:\n{results['developer']}",
    )

    return results


# 実行例
if __name__ == "__main__":
    requirement = "シンプルなTODOリストのCLIアプリケーションを作成してください"
    results = run_team_development(requirement)

    print("\n" + "=" * 60)
    print("チーム開発完了 - 全工程の成果物サマリー")
    print("=" * 60)
    for role, output in results.items():
        print(f"\n--- {role} ---")
        print(output[:200] + "..." if len(output) > 200 else output)
```

### フローの全体像

```
ユーザー要件
    │
    ▼
┌────────────┐  タスク分解   ┌────────────┐  設計書     ┌────────────┐
│     PM     │────────────►│   設計者    │──────────►│   開発者    │
│ 要件分析   │              │アーキテクチャ│            │ コード実装  │
└────────────┘              └────────────┘            └─────┬──────┘
                                                           │コード
                                                           ▼
                                                    ┌────────────┐
                                                    │  レビュアー  │
                                                    │ 品質検証    │
                                                    └─────┬──────┘
                                                           │
                                                           ▼
                                                    最終成果物
```

> **ポイントまとめ**
> - リレー型パイプラインは段階的に品質を高めるのに適している
> - 各エージェントのシステムプロンプトで役割と出力形式を明確に指定する
> - 前段階の出力を次の段階の入力として渡す設計がシンプルで効果的
> - 実運用では各段階の出力を保存し、手戻り時に途中から再開できるようにする

## 5.6 課題と対策

マルチエージェントシステムには多くのメリットがありますが、固有の課題もあります。

### コスト増大

エージェント数が増えるほど、LLMのAPI呼び出し回数とトークン消費量が増加します。

| 構成 | 推定API呼び出し回数 | 推定コスト倍率 |
|------|-------------------|-------------|
| シングルエージェント | 1〜3回 | 1x |
| 3エージェント（リレー型） | 3〜9回 | 3〜5x |
| 5エージェント（階層型） | 5〜15回 | 5〜10x |
| ディベート型（3ラウンド） | 9〜18回 | 8〜15x |

**対策**:
- 小さいモデル（Claude Haiku等）をワーカーに使い、大きいモデル（Claude Sonnet等）をマネージャーに限定する
- キャッシュを活用して同じ問い合わせを減らす
- タスクの複雑さに応じて、シングル/マルチを動的に切り替える

### レイテンシ

直列処理ではエージェント数に比例して応答時間が増加します。

**対策**:
- 独立したタスクは並列実行する（`asyncio.gather` 等）
- ストリーミング出力で体感速度を改善する
- 中間結果をユーザーに逐次表示する

```python
import asyncio

async def run_parallel_agents(tasks: list[dict]) -> list[str]:
    """独立したエージェントタスクを並列実行する"""
    async def call_agent_async(task):
        # 非同期APIクライアントを使用
        response = await async_client.messages.create(
            model=MODEL,
            max_tokens=1024,
            system=task["system"],
            messages=[{"role": "user", "content": task["prompt"]}],
        )
        return response.content[0].text

    results = await asyncio.gather(*[call_agent_async(t) for t in tasks])
    return results
```

### デバッグの難しさ

複数エージェントの対話は、問題の原因特定が困難です。

**対策**:
- 全エージェント間のメッセージを構造化ログに記録する
- 各エージェントの入出力を保存し、再現可能にする
- トレーシングツール（LangSmith、Langfuse等）を活用する

### 循環参照の回避

エージェントAがBを呼び、BがAを呼び返すような循環が発生すると、無限ループに陥ります。

```
  ┌──────┐         ┌──────┐
  │Agent A│────────►│Agent B│
  └──────┘◄────────└──────┘
     循環参照 → 無限ループ！
```

**対策**:
- 最大ステップ数を設定し、超過したら強制終了する
- 呼び出し履歴を管理し、同じパスの再通過を検出する
- DAG（有向非巡回グラフ）として通信フローを設計する

```python
MAX_STEPS = 10

def run_with_loop_guard(agents: list, initial_input: str) -> str:
    """循環参照を防ぐガード付き実行"""
    current_input = initial_input
    visited = []

    for step in range(MAX_STEPS):
        current_agent = select_next_agent(current_input, agents)

        # 同じエージェントの連続呼び出しを検出
        if len(visited) >= 2 and visited[-1] == visited[-2] == current_agent.name:
            print(f"警告: {current_agent.name} が連続で呼ばれています。終了します。")
            break

        visited.append(current_agent.name)
        current_input = current_agent.process(current_input)

        if current_agent.is_task_complete(current_input):
            return current_input

    return f"最大ステップ数({MAX_STEPS})に到達しました。最終結果: {current_input}"
```

> **よくある間違い**
> - 最初からマルチエージェントで設計してしまう → まずシングルエージェントで試し、限界を感じてからマルチに拡張しましょう
> - エージェント数を増やしすぎる → 3〜5体が実用的な上限です。それ以上はコストと複雑さが急増します
> - エラーハンドリングを省略する → 1つのエージェントの失敗がシステム全体を止めます。各段階でリトライやフォールバックを実装しましょう

> **ポイントまとめ**
> - コスト: モデルの使い分けとキャッシュで最適化する
> - レイテンシ: 並列実行とストリーミングで対策する
> - デバッグ: 構造化ログとトレーシングツールを必ず導入する
> - 循環参照: 最大ステップ数とDAG設計で防止する
> - まずシングルエージェントで検証し、必要に応じてマルチに拡張するのがベストプラクティス

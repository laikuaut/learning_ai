## 2.1 プロトコルの基本構造

### JSON-RPCベースの通信

MCPは、JSON-RPC 2.0をベースにしたプロトコルです。JSON-RPC（JSON Remote Procedure Call）とは、JSON形式でリモート手続き呼び出しを行う軽量なプロトコルです。すべてのメッセージがJSON形式のテキストデータであるため、デバッグや理解が容易です。

### 3種類のメッセージ

| 種類 | 説明 | idフィールド | methodフィールド |
|---|---|---|---|
| リクエスト（Request） | 応答を期待するメッセージ | あり | あり |
| レスポンス（Response） | リクエストへの返答 | あり（対応するリクエストと同じ） | なし |
| 通知（Notification） | 応答を期待しない一方通行メッセージ | なし | あり |

### 各メッセージの例

```json
// リクエスト（応答を期待する）
{ "jsonrpc": "2.0", "id": 1, "method": "tools/call",
  "params": { "name": "read_file", "arguments": { "path": "/home/user/example.txt" } } }

// レスポンス（成功時）
{ "jsonrpc": "2.0", "id": 1,
  "result": { "content": [{ "type": "text", "text": "ファイルの内容" }] } }

// レスポンス（エラー時）
{ "jsonrpc": "2.0", "id": 1,
  "error": { "code": -32602, "message": "ファイルが見つかりません" } }

// 通知（応答を期待しない）
{ "jsonrpc": "2.0", "method": "notifications/initialized" }
```

### 主要なエラーコード

| コード | 名前 | 意味 |
|---|---|---|
| -32700 | Parse error | JSONの解析に失敗 |
| -32600 | Invalid Request | リクエストの形式が不正 |
| -32601 | Method not found | 指定されたメソッドが存在しない |
| -32602 | Invalid params | パラメータが不正 |
| -32603 | Internal error | サーバー内部エラー |

### ポイントまとめ
- MCPはJSON-RPC 2.0ベースで、すべてのメッセージがJSON形式
- リクエスト（応答あり）、レスポンス（返答）、通知（応答なし）の3種類
- リクエストとレスポンスは `id` フィールドで対応づけられる

## 2.2 ホスト・クライアント・サーバーの詳細

### 各コンポーネントの構造

```
┌────────────────────────────────────────────┐
│              ホスト（Host）                   │
│  ┌─────────────────────────────────┐       │
│  │         LLM との対話管理         │       │
│  └─────────────────────────────────┘       │
│  ┌──────────────┐  ┌──────────────┐        │
│  │ MCPクライアント│  │ MCPクライアント│        │
│  │     (1)      │  │     (2)      │        │
│  └──────┬───────┘  └──────┬───────┘        │
│  ┌─────────────────────────────────┐       │
│  │    セキュリティポリシーの適用      │       │
│  └─────────────────────────────────┘       │
└─────────┼───────────────────┼───────────────┘
          ▼                   ▼
   ┌──────────────┐    ┌──────────────┐
   │ MCPサーバー(1)│    │ MCPサーバー(2)│
   └──────────────┘    └──────────────┘
```

### ホスト（Host）の責務

| 責務 | 説明 |
|---|---|
| MCPクライアントの生成・管理 | サーバーごとにクライアントインスタンスを作成 |
| LLMとの対話管理 | ユーザー入力をLLMに送り、応答を表示 |
| ユーザー同意の管理 | ツール実行前にユーザーの許可を取得 |
| セキュリティポリシーの適用 | サーバーへのアクセス範囲を制限 |
| 複数サーバー間の調整 | 複数サーバーからの情報を統合してLLMに渡す |

### クライアント（Client）の責務

| 責務 | 説明 |
|---|---|
| プロトコルネゴシエーション | サーバーとのケーパビリティ交換を実行 |
| メッセージのルーティング | リクエスト/レスポンスの送受信を管理 |
| サーバーのライフサイクル管理 | サーバープロセスの起動・監視・停止 |
| エラーハンドリング | 通信エラーやタイムアウトの処理 |

### サーバー（Server）の責務

| 責務 | 説明 |
|---|---|
| ケーパビリティの公開 | 自身が提供するツール・リソース・プロンプトを公開 |
| リクエストの処理 | クライアントからのリクエストを実行して結果を返す |
| 外部システムとの通信 | API呼び出し、DB接続、ファイル操作等を実行 |

### よくある間違い

> **誤解：MCPサーバーは常にリモートのクラウドサーバーに接続する**
>
> MCPサーバー自体はローカルで動作するプログラムです。ファイルシステムサーバーのように純粋にローカルで完結するものも多くあります。

### ポイントまとめ
- ホストはユーザー対話・セキュリティ・複数サーバーの統合を担当
- クライアントは1つのサーバーとの接続管理に特化
- サーバーはツール・リソース・プロンプトの提供に集中

## 2.3 接続とトランスポート

### トランスポート層の役割

トランスポート（transport）は、MCPクライアントとサーバーの間でJSON-RPCメッセージを運ぶ通信手段です。MCPは2種類のトランスポートを定義しています。

### stdio（標準入出力）

stdioトランスポートは、プロセスの標準入出力を使って通信します。

```
┌──────────────┐  stdin   ┌──────────────┐
│ MCPクライアント├────────→│ MCPサーバー   │
│              │  stdout  │ (子プロセス)  │
│              │←────────┤              │
└──────────────┘         └──────────────┘
```

Claude Desktopの設定例：

```json
{
  "mcpServers": {
    "filesystem": {
      "command": "npx",
      "args": ["-y", "@modelcontextprotocol/server-filesystem", "/home/user"],
      "env": { "NODE_ENV": "production" }
    }
  }
}
```

### Streamable HTTP

Streamable HTTPトランスポートは、HTTPベースの通信方法です。以前はSSE（Server-Sent Events）トランスポートと呼ばれていましたが、仕様の進化に伴い名称が変更されました。

```
┌──────────────┐  HTTP POST    ┌──────────────┐
│ MCPクライアント├────────────→│ MCPサーバー   │
│              │  Response/SSE │ (HTTPサーバー) │
│              │←────────────┤              │
└──────────────┘              └──────────────┘
```

### トランスポートの使い分け

| 判断基準 | stdio | Streamable HTTP |
|---|---|---|
| ローカル開発 | 推奨 | - |
| リモートサーバー | - | 推奨 |
| 複数ユーザー共有 | - | 推奨 |
| 設定の簡便さ | 推奨 | - |
| デスクトップアプリ | 推奨 | - |

### よくある間違い

> **誤解：SSEトランスポートを新規実装に使うべき**
>
> SSEトランスポートは旧仕様です。新規実装ではStreamable HTTPを使用してください。既存のSSE実装は動作しますが、将来的に非推奨となる予定です。

### ポイントまとめ
- stdioはローカル開発に最適、設定が簡単で追加インフラ不要
- Streamable HTTPはリモート接続やチーム共有に適している
- 新規実装ではSSEではなくStreamable HTTPを使用すること

## 2.4 プロトコルのライフサイクル

### 4つのフェーズ

MCPの接続は、4つのフェーズ（phase）を経て管理されます。

```
初期化 ──→ ケーパビリティ交換 ──→ 通常操作 ──→ シャットダウン
Initialize    Negotiation        Operation      Shutdown
```

### シーケンスとinitializeリクエストの例

```
クライアント                     サーバー
    │  initialize               │
    ├──────────────────────────→│
    │  initialize (response)    │
    │←──────────────────────────┤
    │  notifications/initialized│
    ├──────────────────────────→│
    │  === 通常操作フェーズ ===   │
    │  tools/list, tools/call等 │
    │←────────────────────────→│
    │  接続終了                  │
    ├──────────────────────────X
```

```json
{
  "jsonrpc": "2.0",
  "id": 1,
  "method": "initialize",
  "params": {
    "protocolVersion": "2025-03-26",
    "capabilities": {
      "roots": { "listChanged": true }
    },
    "clientInfo": {
      "name": "my-mcp-client",
      "version": "1.0.0"
    }
  }
}
```

初期化レスポンスを受信後、クライアントは `notifications/initialized` 通知を送信します。この通知をもってケーパビリティ交換が完了し、通常操作フェーズに移行します。

### ポイントまとめ
- MCPの接続は 初期化 → ケーパビリティ交換 → 通常操作 → シャットダウン の4段階
- 初期化フェーズでプロトコルバージョンとケーパビリティを交換する
- `initialized` 通知の送信後に通常操作が可能になる

## 2.5 ケーパビリティ（Capabilities）

### ケーパビリティネゴシエーション

ケーパビリティ（capabilities）とは、クライアントとサーバーがそれぞれ「何ができるか」を宣言する仕組みです。初期化フェーズで互いのケーパビリティを交換し、利用可能な機能を確認します。

### サーバーケーパビリティ

| ケーパビリティ | 説明 | 主なメソッド |
|---|---|---|
| `tools` | ツールの提供 | `tools/list`, `tools/call` |
| `resources` | リソースの提供 | `resources/list`, `resources/read` |
| `prompts` | プロンプトの提供 | `prompts/list`, `prompts/get` |
| `logging` | ログメッセージの送信 | `logging/setLevel` |

```json
{
  "capabilities": {
    "tools": { "listChanged": true },
    "resources": { "subscribe": true, "listChanged": true },
    "prompts": { "listChanged": true }
  }
}
```

`listChanged: true` は、一覧が動的に変化する可能性があることを示します。

### クライアントケーパビリティ

| ケーパビリティ | 説明 |
|---|---|
| `roots` | ルートディレクトリ情報の提供 |
| `sampling` | サーバーからのLLMサンプリング要求の受け入れ |

### バージョン管理

MCPはプロトコルバージョンを日付形式（例：`2025-03-26`）で管理します。

```
1. クライアントがサポートするバージョンを送信
2. サーバーが対応可能なバージョンで応答
3. バージョン不一致の場合 → 接続を終了
```

### ポイントまとめ
- ケーパビリティは「何ができるか」の宣言で、初期化時に交換される
- サーバーはtools/resources/prompts等のケーパビリティを宣言
- プロトコルバージョンは日付形式で管理され、互換性を保証する

## 2.6 メッセージフロー

### ツール呼び出しの完全なシーケンス

ユーザーがAIアプリに質問し、MCPツールが呼び出される一連の流れです。

```
ユーザー   ホスト      LLM     クライアント  サーバー
  │        │          │         │          │
  │ 質問   │          │         │          │
  ├──────→│ +ツール定義│         │          │
  │        ├────────→│         │          │
  │        │ ツール指示│         │          │
  │        │←────────┤         │          │
  │ 同意？ │          │         │          │
  │←──────┤          │         │          │
  │ 許可   │          │         │          │
  ├──────→│ tools/call│         │          │
  │        ├───────────────────→│ tools/call│
  │        │          │         ├────────→│
  │        │          │         │ 結果     │
  │        │          │         │←────────┤
  │        │ 結果     │         │          │
  │        │←───────────────────┤          │
  │        ├────────→│         │          │
  │        │ 最終応答 │         │          │
  │        │←────────┤         │          │
  │ 応答   │          │         │          │
  │←──────┤          │         │          │
```

### tools/call の具体例

**リクエスト：**
```json
{
  "jsonrpc": "2.0",
  "id": 42,
  "method": "tools/call",
  "params": {
    "name": "read_file",
    "arguments": { "path": "/home/user/notes.txt" }
  }
}
```

**レスポンス：**
```json
{
  "jsonrpc": "2.0",
  "id": 42,
  "result": {
    "content": [{ "type": "text", "text": "ファイルの内容..." }],
    "isError": false
  }
}
```

### リソース取得のフロー

```
MCPクライアント                     MCPサーバー
    │  resources/list                │
    ├───────────────────────────────→│
    │  リソース一覧                   │
    │←───────────────────────────────┤
    │  resources/read (uri指定)      │
    ├───────────────────────────────→│
    │  リソース内容                   │
    │←───────────────────────────────┤
```

### よくある間違い

> **誤解：MCPサーバーがLLMを直接呼び出す**
>
> MCPサーバーはLLMと直接通信しません。サーバーはクライアントからのリクエストを処理して結果を返すだけです。LLMとの対話はホストが管理します。ただし、サンプリング（sampling）機能を使えば、サーバーがホスト経由でLLMにリクエストを送ることは可能です。

### ポイントまとめ
- ツール呼び出しは ユーザー → ホスト → LLM → クライアント → サーバー の流れ
- ユーザー同意のステップがセキュリティの重要なポイント
- サーバーはLLMと直接通信せず、必ずホストを経由する

## 2.7 セキュリティモデル

### MCPの4つのセキュリティ原則

#### 原則1：最小権限の原則（Principle of Least Privilege）

MCPサーバーには、必要最小限のアクセス権限のみを付与します。

```json
{
  "mcpServers": {
    "filesystem": {
      "command": "npx",
      "args": [
        "-y", "@modelcontextprotocol/server-filesystem",
        "/home/user/documents",
        "/home/user/projects"
      ]
    }
  }
}
```

上記の例では、`/home/user/documents` と `/home/user/projects` のみにアクセスを限定しています。`/`（ルート）を指定してシステム全体へのアクセスを許可するのは避けてください。

#### 原則2：ユーザー同意（User Consent）

ホストアプリは、ツール実行前にユーザーの明示的な同意を取得します。

| 同意レベル | 説明 |
|---|---|
| 毎回確認 | ツール実行のたびに許可を求める（デフォルト） |
| セッション単位 | 同一セッション中は再確認しない |
| 永続的許可 | 以降は確認なしで実行（信頼済みサーバー用） |

#### 原則3：プロセス分離（Process Isolation）

MCPサーバーはホストとは別プロセスで動作し、サーバーの不具合がホストに波及することを防ぎます。

```
┌────────────────────┐
│ ホストプロセス       │  ← 独立したメモリ空間
├────────────────────┤  ← プロセス境界
│ サーバープロセス A   │  ← 独立したメモリ空間
├────────────────────┤  ← プロセス境界
│ サーバープロセス B   │  ← 独立したメモリ空間
└────────────────────┘
```

#### 原則4：データプライバシー

| トランスポート | データ経路 | 暗号化 |
|---|---|---|
| stdio | ローカルプロセス間 | 不要（同一マシン内） |
| Streamable HTTP（ローカル） | localhost | オプション |
| Streamable HTTP（リモート） | ネットワーク | HTTPS推奨 |

### よくある間違い

> **誤解：MCPサーバーは自動的に安全である**
>
> MCPはセキュリティの「枠組み」を提供しますが、実際の安全性はサーバーの実装品質と設定に依存します。信頼できないソースのMCPサーバーはファイルの不正読み取りやデータ漏洩のリスクがあります。公式サーバーやコミュニティで評価の高いサーバーを優先しましょう。

### ポイントまとめ
- 最小権限の原則：サーバーには必要最小限のアクセス権だけを付与
- ユーザー同意：ツール実行前に必ずユーザーの明示的な許可を取得
- プロセス分離：サーバーは独立プロセスで動作し、障害の波及を防止
- リモート通信にはHTTPSの使用を推奨

# 第3章：基本設計書（外部設計書）

## この章のゴール

- 基本設計書（Basic Design Document）の目的と構成を理解する
- システムアーキテクチャ図の読み書きができるようになる
- 機能分割とモジュール構成の考え方を理解する
- 外部インタフェース設計の基本を身につける
- 基本設計書のレビューポイントを理解する

---

## 3.1 基本設計書とは

### 基本設計書の位置づけ

基本設計書（外部設計書）は、**要件定義で決めた「何を作るか」を「どのような構造で作るか」に変換する文書**です。システムの外側から見た振る舞い（外部設計）を中心に記述します。

```
【設計書の粒度】

要件定義書          基本設計書              詳細設計書
（What）           （How：全体構造）        （How：内部ロジック）

「商品を検索    →  「Webサーバー +        →  「SQLクエリの
 できること」       APIサーバー +             組み立てロジック、
                   検索エンジンで実現。       Elasticsearchとの
                   REST APIで通信」          インデックス同期処理」
```

### 基本設計書の構成

```
基本設計書の目次（例）
━━━━━━━━━━━━━━━━━━━━━━━━
1. システム概要
   1.1 システム構成図
   1.2 ネットワーク構成図
   1.3 技術スタック

2. アプリケーション設計
   2.1 機能構成図

   （→ 第7章で詳述）

   （→ 第6章で詳述）
```

## 3.2 システム構成図

### システム構成図とは

システム構成図は、システムを構成するサーバー、ネットワーク、ミドルウェアなどの物理的・論理的な構成を示す図です。

```
【システム構成図の例：ECサイト】

                        ┌──────────┐
                        │ ユーザー  │
                        └────┬─────┘
                             │ HTTPS
                        ┌────▼─────┐
                        │   CDN     │  静的コンテンツ配信
                        │(CloudFront)│
                        └────┬─────┘
                             │
                        ┌────▼─────┐
                        │   ALB    │  ロードバランサー
                        │          │
                        └────┬─────┘
                     ┌───────┴───────┐
                ┌────▼─────┐   ┌────▼─────┐
                │ Web/API   │   │ Web/API   │  アプリケーション
                │ サーバー①  │   │ サーバー②  │  サーバー（冗長化）
                │ (ECS)     │   │ (ECS)     │
                └────┬─────┘   └────┬─────┘
                     └───────┬───────┘
               ┌─────────────┼─────────────┐
          ┌────▼─────┐  ┌───▼────┐  ┌────▼─────┐
          │   RDS    │  │ Redis  │  │   S3     │
          │(PostgreSQL)│ │(Cache) │  │(ファイル) │
          │  Primary  │  └────────┘  └──────────┘
          └────┬─────┘
          ┌────▼─────┐
          │   RDS    │
          │  Replica │  読み取り用レプリカ
          └──────────┘
```

### 技術スタック表

システムで使用する技術要素を一覧化します。

```
【技術スタック表（例）】

| カテゴリ           | 技術・ツール          | バージョン | 選定理由                |
|-------------------|---------------------|-----------|------------------------|
| フロントエンド      | React               | 18.x      | コンポーネント設計の柔軟性 |
| UIフレームワーク    | Material UI         | 5.x       | 豊富なコンポーネント      |
| バックエンド        | Node.js + Express   | 20.x LTS  | フロントと言語統一        |
| データベース        | PostgreSQL          | 16.x      | ACID準拠、JSON対応       |
| キャッシュ          | Redis               | 7.x       | セッション・クエリキャッシュ|
| ファイルストレージ   | Amazon S3           | -         | スケーラブルなファイル管理  |
| コンテナ           | Docker + ECS        | -         | デプロイの統一化          |
| CI/CD             | GitHub Actions      | -         | リポジトリとの統合         |
| 監視              | CloudWatch + Datadog | -         | メトリクス・ログ・APM     |
```

> **実務でのポイント**: 技術選定では「なぜその技術を選んだのか」を記録することが重要です。ADR（Architecture Decision Record）として、選定理由・代替案・トレードオフを残しておくと、後から「なぜこの構成にしたのか？」という問いに答えられます。

## 3.4 処理方式設計

### オンライン処理とバッチ処理

```
【処理方式の分類】

┌──────────────────────────────────────────────────────┐
│ オンライン処理（リアルタイム処理）                       │
│                                                      │
│  ・ユーザーの操作に対してリアルタイムに応答する処理       │
│  ・例：打刻、申請、検索、一覧表示                       │
│  ・応答時間：3秒以内                                   │
├──────────────────────────────────────────────────────┤
│ バッチ処理（一括処理）                                  │
│                                                      │
│  ・定期的またはイベント駆動で一括実行する処理             │
│  ・例：月次集計、CSVエクスポート、データバックアップ      │
│  ・実行タイミング：スケジュール実行                      │
└──────────────────────────────────────────────────────┘
```

### バッチ処理一覧

```
【バッチ処理一覧（例）】

| バッチID | バッチ名       | 実行タイミング     | 処理概要              | 想定処理時間 |
|---------|---------------|-------------------|----------------------|------------|
| BAT-001 | 月次勤怠集計   | 毎月1日 2:00      | 前月の勤怠データを集計  | 30分以内    |
| BAT-002 | 未打刻チェック  | 平日 22:00        | 当日の未打刻者を検出   | 5分以内     |
| BAT-003 | 残業超過チェック | 毎週月曜 8:00     | 週40時間超過者を検出   | 10分以内    |
| BAT-004 | データバックアップ| 毎日 3:00         | DBの日次バックアップ   | 15分以内    |
| BAT-005 | CSV連携        | 毎月25日 6:00     | 給与システムへCSV出力  | 10分以内    |
```

### 排他制御方式

複数ユーザーが同時にデータを更新する場合の制御方式を定義します。

```
【排他制御方式】

┌────────────┬───────────────────────────────────────┐
│ 楽観的排他  │ 更新時にバージョン番号をチェックし、       │
│ (Optimistic)│ 競合した場合はエラーとして再操作を促す     │
│            │                                       │
│ 採用場面    │ 同時更新の頻度が低い画面                  │
│            │ 例：プロフィール編集、マスタ更新           │
├────────────┼───────────────────────────────────────┤
│ 悲観的排他  │ データ取得時にロックを取得し、              │
│(Pessimistic)│ 他ユーザーの更新をブロックする            │
│            │                                       │
│ 採用場面    │ 同時更新が許容されない重要データ            │
│            │ 例：在庫数の更新、残高の更新               │
└────────────┴───────────────────────────────────────┘

【楽観的排他制御の動作フロー】

  ユーザーA                     ユーザーB
     │                            │
     │ ① データ取得               │
     │   (version = 1)            │
     │                       ② データ取得
     │                          (version = 1)
     │ ③ 更新送信                 │
     │   (version = 1 → 2)       │
     │   → 成功 ✓                │
     │                       ④ 更新送信
     │                          (version = 1 → ?)
     │                          → 失敗 ✗ (version不一致)
     │                          → エラー「他のユーザーが
     │                             更新済みです。再読込
     │                             してください」
```

```
                   ┌──────▼───────┐

                   └──────────────┘
```

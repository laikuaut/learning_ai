## この章のゴール
- 関係データベースの基本概念（表・行・列・キー）を正確に理解する

## 5.1 データベースの基礎概念

### データベースとは

データベース（Database, DB）とは、複数のアプリケーションやユーザーが共有して利用するために、体系的に整理・蓄積されたデータの集まりです。

### データベース管理システム（DBMS: Database Management System）

データベースを管理するソフトウェアです。代表的な製品にはOracle Database、MySQL、PostgreSQL、SQL Serverなどがあります。

**DBMSの主な機能：**

| 機能 | 説明 |
|------|------|
| データ定義 | テーブルやビューの構造を定義する |
| データ操作 | 検索、追加、更新、削除を行う |
| データ制御 | アクセス権限、トランザクション管理 |
| 排他制御 | 同時アクセスの整合性を保つ |
| 障害回復 | 障害発生時のデータ復旧 |

### データベースの種類

| 種類 | 特徴 | 代表例 |
|------|------|--------|
| 関係（リレーショナル）DB | 表形式でデータを管理 | Oracle, MySQL |
| 階層型DB | 木構造でデータを管理 | IMS |
| ネットワーク型DB | 網構造でデータを管理 | IDMS |
| オブジェクト指向DB | オブジェクトとしてデータを管理 | ObjectStore |

> **試験の重要ポイント：** 基本情報で問われるのは、ほぼすべて「関係データベース」です。

## 5.2 関係データベースの基本用語

### 表（テーブル / リレーション）

関係データベースでは、データを「表」の形で管理します。

```
【社員テーブル】
+--------+--------+--------+--------+
| 社員番号 | 氏名   | 部署ID | 給与    |
+--------+--------+--------+--------+
| E001   | 田中太郎| D01    | 350000 |
| E002   | 鈴木花子| D02    | 420000 |
| E003   | 佐藤次郎| D01    | 380000 |
+--------+--------+--------+--------+
```

### 用語の対応

| 正式名称 | 別名（よく使われる名称） | 意味 |
|---------|----------------------|------|
| リレーション（Relation） | テーブル、表 | データの集合 |
| タプル（Tuple） | 行、レコード | 1件のデータ |
| 属性（Attribute） | 列、フィールド、カラム | データの項目 |
| ドメイン（Domain） | 定義域 | 属性が取りうる値の範囲 |

### キーの種類

| キーの種類 | 説明 | 例 |
|-----------|------|-----|
| 候補キー（Candidate Key） | 行を一意に識別できる属性（の組） | 社員番号、メールアドレス |
| 主キー（Primary Key） | 候補キーの中から選ばれた代表 | 社員番号 |
| 外部キー（Foreign Key） | 他の表の主キーを参照する属性 | 部署ID（→部署テーブル） |
| スーパーキー（Super Key） | 一意に識別できる属性の組（冗長な属性を含んでよい） | {社員番号, 氏名} |

**主キーの条件：**
- **一意性：** 同じ値が2つ以上存在してはならない
- **NOT NULL：** NULL（値なし）は許されない

**外部キーの役割（参照整合性制約）：**
外部キーの値は、参照先テーブルの主キーに存在する値でなければなりません。これを**参照整合性**といいます。

## 5.3 正規化（Normalization）

### 正規化とは

データの冗長性（無駄な重複）を排除し、更新時の矛盾（更新異常）を防ぐために、テーブルの構造を改善する手順です。

### 関数従属性

正規化の基礎となる概念です。属性 X の値が決まると属性 Y の値が一意に決まるとき、「Y は X に関数従属する」といい、**X → Y** と書きます。

```
社員番号 → 氏名        （社員番号が決まれば氏名が決まる）
社員番号 → 部署名      （社員番号が決まれば部署名が決まる）
部署ID → 部署名        （部署IDが決まれば部署名が決まる）
```

### 非正規形（正規化前）

繰り返し項目（反復する属性グループ）を含む状態です。

```
【注文テーブル（非正規形）】
注文番号 | 顧客名 | 商品1 | 数量1 | 商品2 | 数量2 | 商品3 | 数量3
O001    | 田中   | りんご | 3    | みかん | 5    |      |
O002    | 鈴木   | バナナ | 2    |       |      |      |
```

問題点：商品数が増えるとカラムの追加が必要。空欄が多数発生する。

### 第1正規形（1NF: First Normal Form）

**条件：** 繰り返し項目を排除し、すべての属性値がスカラ値（単一値）であること。

```
【注文テーブル（第1正規形）】
注文番号 | 顧客名 | 商品名  | 数量
O001    | 田中   | りんご  | 3
O001    | 田中   | みかん  | 5
O002    | 鈴木   | バナナ  | 2
```

主キー：{注文番号, 商品名}（複合主キー）

### 第2正規形（2NF: Second Normal Form）

**条件：** 第1正規形であり、かつ**部分関数従属**がないこと。

部分関数従属とは、複合主キーの一部に対して従属している状態です。

```
上の例では：
{注文番号, 商品名} → 数量   ... 完全関数従属（OK）
{注文番号} → 顧客名          ... 部分関数従属（NG：主キーの一部に従属）
```

**解決：** 部分関数従属する属性を別テーブルに分離します。

```
【注文テーブル】           【注文明細テーブル】
注文番号 | 顧客名          注文番号 | 商品名 | 数量
O001    | 田中            O001    | りんご | 3
O002    | 鈴木            O001    | みかん | 5
                          O002    | バナナ | 2
```

### 第3正規形（3NF: Third Normal Form）

**条件：** 第2正規形であり、かつ**推移的関数従属**がないこと。

推移的関数従属とは、主キー → A → B のように、主キー以外の属性を介して間接的に従属している状態です。

```
例：
社員番号 → 部署ID → 部署名
（社員番号が決まると部署IDが決まり、部署IDが決まると部署名が決まる）
```

**解決：** 推移的従属する属性を別テーブルに分離します。

```
正規化前：
社員番号 | 氏名 | 部署ID | 部署名

正規化後：
【社員テーブル】            【部署テーブル】
社員番号 | 氏名 | 部署ID    部署ID | 部署名
```

### 正規化のまとめ

| 正規形 | 排除する従属性 | ひとことで |
|--------|---------------|-----------|
| 第1正規形 | 繰り返し項目 | 「1つのセルに1つの値」 |
| 第2正規形 | 部分関数従属 | 「複合キーの一部への従属を排除」 |
| 第3正規形 | 推移的関数従属 | 「キー以外への従属を排除」 |

### よくある間違い
- 第2正規形は主キーが単一属性の場合は自動的に満たされます（部分関数従属が起こりえないため）。
- 正規化しすぎるとテーブル数が増え、結合（JOIN）の回数が増えてパフォーマンスが低下する場合があります。実務では意図的に正規化を崩す（非正規化）こともあります。

## 5.4 ER図（Entity-Relationship Diagram）

### ER図とは

実世界のデータの構造を、エンティティ（実体）、属性、リレーションシップ（関連）で表現する図です。データベース設計の初期段階で使います。

### 基本記号

```
┌─────────┐          ┌─────────┐
│  社員    │──────────│  部署    │
│─────────│          │─────────│
│ 社員番号 │          │ 部署ID   │
│ 氏名     │          │ 部署名   │
│ 給与     │          │ 所在地   │
└─────────┘          └─────────┘
```

### カーディナリティ（多重度）

エンティティ間の数量関係を示します。

| 表記 | 意味 | 例 |
|------|------|-----|
| 1 : 1 | 1対1 | 社員と社員証 |
| 1 : 多（1 : *） | 1対多 | 部署と社員 |
| 多 : 多（* : *） | 多対多 | 学生と科目 |

**多対多の関係は、中間テーブル（連関エンティティ）を使って1対多に分解します。**

```
【学生】 * ──── * 【科目】

↓ 中間テーブルで分解

【学生】 1 ──── * 【履修】 * ──── 1 【科目】
```

## 5.5 SQL（Structured Query Language）

### SQLの分類

| 分類 | 名称 | 主なコマンド |
|------|------|-------------|
| DDL | Data Definition Language（データ定義言語） | CREATE, ALTER, DROP |
| DML | Data Manipulation Language（データ操作言語） | SELECT, INSERT, UPDATE, DELETE |
| DCL | Data Control Language（データ制御言語） | GRANT, REVOKE |

### 5.5.1 テーブルの作成（CREATE TABLE）

```sql
CREATE TABLE 社員 (
    社員番号  CHAR(4)     PRIMARY KEY,
    氏名     VARCHAR(20)  NOT NULL,
    部署ID   CHAR(3),
    給与     INTEGER,
    FOREIGN KEY (部署ID) REFERENCES 部署(部署ID)
);
```

### 5.5.2 データの検索（SELECT）

**基本構文：**
```sql
SELECT 列名1, 列名2, ...
FROM テーブル名
WHERE 条件
ORDER BY 列名 [ASC | DESC];
```

**実行順序（重要）：**
```
FROM → WHERE → GROUP BY → HAVING → SELECT → ORDER BY
```

> **試験のポイント：** SELECT の各句の実行順序は頻出です。WHERE は GROUP BY より先、HAVING は GROUP BY の後に実行されます。

**条件の指定：**

| 演算子 | 意味 | 例 |
|--------|------|-----|
| = | 等しい | 給与 = 350000 |
| <>, != | 等しくない | 部署ID <> 'D01' |
| <, >, <=, >= | 大小比較 | 給与 >= 400000 |
| BETWEEN | 範囲指定 | 給与 BETWEEN 300000 AND 400000 |
| IN | いずれかに一致 | 部署ID IN ('D01', 'D02') |
| LIKE | パターン一致 | 氏名 LIKE '田%' |
| IS NULL | NULLの判定 | 給与 IS NULL |
| IS NOT NULL | NULLでないことの判定 | 給与 IS NOT NULL |

**LIKE のワイルドカード：**
- `%`：任意の0文字以上の文字列
- `_`：任意の1文字

```sql
-- '田' で始まる氏名
WHERE 氏名 LIKE '田%'

-- 2文字目が '藤' の氏名
WHERE 氏名 LIKE '_藤%'
```

### 5.5.3 集約関数（Aggregate Functions）

| 関数 | 意味 | NULLの扱い |
|------|------|-----------|
| COUNT(*) | 行数を数える | NULLを含む |
| COUNT(列名) | 指定列がNULLでない行数 | NULLを除外 |
| SUM(列名) | 合計 | NULLを除外 |
| AVG(列名) | 平均 | NULLを除外 |
| MAX(列名) | 最大値 | NULLを除外 |
| MIN(列名) | 最小値 | NULLを除外 |

> **よくある間違い：** COUNT(*) は NULL を含む全行を数えますが、COUNT(列名) は NULLを除外します。この違いは試験の頻出ポイントです。

### 5.5.4 グループ化（GROUP BY）

```sql
-- 部署ごとの平均給与
SELECT 部署ID, AVG(給与) AS 平均給与
FROM 社員
GROUP BY 部署ID;
```

**HAVING句：** グループ化した結果に対する条件です。

```sql
-- 平均給与が40万円以上の部署
SELECT 部署ID, AVG(給与) AS 平均給与
FROM 社員
GROUP BY 部署ID
HAVING AVG(給与) >= 400000;
```

> **WHEREとHAVINGの違い：** WHERE は個々の行に対する条件、HAVING はグループに対する条件です。

### 5.5.5 結合（JOIN）

複数のテーブルを関連付けてデータを取得します。

**内部結合（INNER JOIN）：** 結合条件に一致する行のみ取得

```sql
SELECT 社員.氏名, 部署.部署名
FROM 社員
INNER JOIN 部署 ON 社員.部署ID = 部署.部署ID;
```

**左外部結合（LEFT OUTER JOIN）：** 左テーブルの全行 + 一致する右テーブルの行

```sql
SELECT 社員.氏名, 部署.部署名
FROM 社員
LEFT OUTER JOIN 部署 ON 社員.部署ID = 部署.部署ID;
```

結合の種類：

| 結合の種類 | 説明 |
|-----------|------|
| 内部結合（INNER JOIN） | 両方に存在する行のみ |
| 左外部結合（LEFT OUTER JOIN） | 左テーブルの全行 + 右の一致分 |
| 右外部結合（RIGHT OUTER JOIN） | 右テーブルの全行 + 左の一致分 |
| 完全外部結合（FULL OUTER JOIN） | 両方の全行 |
| 交差結合（CROSS JOIN） | 全組み合わせ（直積） |

### 5.5.6 副問合せ（サブクエリ）

SELECT文の中にさらにSELECT文を入れる手法です。

```sql
-- 平均給与以上の社員を取得
SELECT 氏名, 給与
FROM 社員
WHERE 給与 >= (SELECT AVG(給与) FROM 社員);
```

**IN を使った副問合せ：**
```sql
-- 東京にある部署に所属する社員
SELECT 氏名
FROM 社員
WHERE 部署ID IN (SELECT 部署ID FROM 部署 WHERE 所在地 = '東京');
```

**EXISTS を使った副問合せ：**
```sql
-- 注文が存在する顧客を取得
SELECT 顧客名
FROM 顧客
WHERE EXISTS (SELECT * FROM 注文 WHERE 注文.顧客ID = 顧客.顧客ID);
```

### 5.5.7 データの追加・更新・削除

**INSERT（追加）：**
```sql
INSERT INTO 社員 (社員番号, 氏名, 部署ID, 給与)
VALUES ('E004', '山田一郎', 'D02', 300000);
```

**UPDATE（更新）：**
```sql
UPDATE 社員
SET 給与 = 給与 * 1.1
WHERE 部署ID = 'D01';
```

**DELETE（削除）：**
```sql
DELETE FROM 社員
WHERE 社員番号 = 'E004';
```

### 5.5.8 ビュー（View）

仮想的なテーブルです。元のテーブルの一部のデータを、あたかも別のテーブルであるかのように見せます。

```sql
CREATE VIEW 高給社員 AS
SELECT 氏名, 給与
FROM 社員
WHERE 給与 >= 400000;
```

ビューの利点：
- セキュリティ（見せたいデータだけを公開）
- 複雑な問い合わせの簡略化
- データの独立性の確保

## 5.6 トランザクション管理

### トランザクションとは

データベースに対する一連の操作をひとまとまりとして扱う処理の単位です。「全て成功するか、全て失敗するか」のどちらかです。

**例：** 銀行の送金処理
```
トランザクション開始
  1. Aさんの口座から10万円を引く
  2. Bさんの口座に10万円を足す
トランザクション終了
```

1だけ成功して2が失敗すると、10万円が消えてしまいます。そのため、2つの操作は「まとめて成功」か「まとめて失敗（取り消し）」のどちらかにしなければなりません。

### ACID特性

| 特性 | 英語 | 意味 |
|------|------|------|
| 原子性 | Atomicity | トランザクションは「全て実行」か「全て取り消し」のどちらか |
| 一貫性 | Consistency | トランザクション前後でデータの整合性が保たれる |
| 独立性（隔離性） | Isolation | 複数のトランザクションが互いに干渉しない |
| 耐久性 | Durability | コミットされたデータは永続的に保持される |

### コミットとロールバック

| 操作 | 意味 |
|------|------|
| COMMIT | トランザクションの処理を確定する |
| ROLLBACK | トランザクションの処理を取り消し、開始前の状態に戻す |

### 排他制御（ロック）

複数のトランザクションが同時にデータを操作するとき、データの整合性を保つ仕組みです。

| ロックの種類 | 別名 | 説明 |
|-------------|------|------|
| 共有ロック（Shared Lock） | 読取りロック | 他のトランザクションも読取りは可能 |
| 占有ロック（Exclusive Lock） | 書込みロック | 他のトランザクションは読取りも書込みも不可 |

**ロックの互換性：**

| | 共有ロック | 占有ロック |
|---|----------|-----------|
| 共有ロック | ○（両立可能） | ×（待ち） |
| 占有ロック | ×（待ち） | ×（待ち） |

### デッドロック

2つのトランザクションが互いに相手のロック解除を待ち合う状態です。

```
トランザクションA: テーブルX をロック → テーブルY のロック待ち
トランザクションB: テーブルY をロック → テーブルX のロック待ち
→ 互いに永遠に待ち続ける（デッドロック）
```

**対策：**
- ロックする順序を統一する
- タイムアウトを設定する
- デッドロック検出機構で一方のトランザクションをロールバックする

### 障害回復

| 手法 | 説明 |
|------|------|
| ログ（ジャーナル） | 更新前後の値を記録する |
| ロールフォワード | ログを使って、コミット済みのデータを再実行し復旧 |
| ロールバック | ログを使って、未コミットの更新を取り消す |
| チェックポイント | 一定時点でメモリの内容をディスクに書き出す |

**障害回復の流れ：**
```
チェックポイント → (T1コミット) → (障害発生) → T2は未コミット

回復処理：
- T1: コミット済み → ロールフォワード（やり直し）
- T2: 未コミット → ロールバック（取り消し）
```

## まとめ

| 項目 | 重要ポイント |
|------|-------------|
| 正規化 | 第1NF＝繰り返し排除、第2NF＝部分従属排除、第3NF＝推移従属排除 |
| 主キー | 一意性 + NOT NULL |
| SQL実行順序 | FROM → WHERE → GROUP BY → HAVING → SELECT → ORDER BY |
| 結合 | INNER=一致のみ、LEFT OUTER=左全行+右一致 |
| ACID特性 | 原子性・一貫性・独立性・耐久性 |
| デッドロック | 互いにロック待ちの状態。ロック順統一で予防 |

## 確認問題

1. 第2正規形を満たすための条件を述べよ。
2. COUNT(*) と COUNT(列名) の違いを説明せよ。
3. WHERE と HAVING の違いを述べよ。
4. ACID特性の4つの性質を日本語で述べよ。
5. デッドロックとは何か、また防止策を1つ述べよ。

<details>
<summary>解答</summary>

1. 第1正規形を満たし、かつ部分関数従属が存在しないこと。すなわち、主キーを構成するすべての属性に完全関数従属していること。
2. COUNT(*) はNULLを含むすべての行を数える。COUNT(列名) は指定された列がNULLでない行のみを数える。
3. WHERE は GROUP BY の前に個々の行に対して適用される条件。HAVING は GROUP BY の後にグループに対して適用される条件。
4. 原子性（Atomicity）：全て実行か全て取消し。一貫性（Consistency）：データの整合性が保たれる。独立性（Isolation）：トランザクション同士が干渉しない。耐久性（Durability）：コミットされた結果は永続的に保持される。
5. デッドロックとは、2つ以上のトランザクションが互いに相手のロック解除を待ち合い、処理が永遠に進まなくなる状態。防止策としては、ロックする資源の順序をすべてのトランザクションで統一することが挙げられる。

## 2.1 エージェントループの基本

### 入力→推論→行動→観察のサイクル

AIエージェントの動作は、**エージェントループ（Agent Loop）** と呼ばれる繰り返し処理で成り立っています。

```
┌──────────────────────────────────────────┐
│            エージェントループ               │
│                                          │
│   ┌────────┐                             │
│   │  入力   │  ユーザーの指示 / 環境情報   │
│   │ Input  │                             │
│   └───┬────┘                             │
│       │                                  │
│       ▼                                  │
│   ┌────────┐                             │
│   │  推論   │  LLMが状況を分析し          │
│   │Reason  │  次の行動を決定              │
│   └───┬────┘                             │
│       │                                  │
│       ├──→ 完了と判断 ──→ 最終回答を返す   │
│       │                                  │
│       ▼                                  │
│   ┌────────┐                             │
│   │  行動   │  ツールを呼び出す           │
│   │  Act   │  (ファイル操作、検索等)      │
│   └───┬────┘                             │
│       │                                  │
│       ▼                                  │
│   ┌────────┐                             │
│   │  観察   │  行動の結果を取得           │
│   │Observe │  結果をコンテキストに追加     │
│   └───┬────┘                             │
│       │                                  │
│       └──────→ 推論に戻る                 │
└──────────────────────────────────────────┘
```

### ループの終了条件

エージェントループは以下の条件で終了します。

1. **タスク完了**：LLMが「目標を達成した」と判断した場合
2. **最大ステップ数到達**：無限ループ防止のための上限に達した場合
3. **エラーによる中断**：回復不能なエラーが発生した場合
4. **ユーザーによる中断**：人間が手動で停止した場合

### Pythonでの基本実装（擬似コード）

```python
def agent_loop(user_input: str, max_steps: int = 20) -> str:
    """エージェントループの基本構造"""
    messages = [{"role": "user", "content": user_input}]

    for step in range(max_steps):
        # 推論：LLMに現在の状況を渡して次の行動を決定
        response = llm.chat(messages)

        # 完了判定：ツール呼び出しがなければタスク完了
        if not response.has_tool_calls():
            return response.content  # 最終回答を返す

        # 行動：ツールを実行
        for tool_call in response.tool_calls:
            result = execute_tool(tool_call.name, tool_call.arguments)

            # 観察：結果をメッセージ履歴に追加
            messages.append({
                "role": "tool",
                "content": result,
                "tool_call_id": tool_call.id
            })

    return "最大ステップ数に達しました。タスクを完了できませんでした。"
```

### よくある間違い

| 間違い | 正しい理解 |
|---|---|
| 「1回のLLM呼び出しで全て完了する」 | 複数回のループで段階的にタスクを進める |
| 「ループ回数を制限しない」 | 必ず最大ステップ数を設定し、無限ループを防ぐ |
| 「行動の結果を保存しない」 | 観察結果をコンテキストに追加しないと、同じ行動を繰り返す |

> **ポイントまとめ**
> - エージェントループは「推論→行動→観察」の繰り返し
> - ツール呼び出しがなくなった時点でループが終了する
> - 最大ステップ数の設定は安全のために必須

## 2.2 ReActパターン

### Reasoning + Acting

ReAct（リアクト）は、**Reasoning（推論）** と **Acting（行動）** を交互に実行するパターンです。2022年にYaoらが提案し、現在のAIエージェントの基本設計として広く採用されています。

ReActの特徴は、行動の前に必ず「思考（Thought）」を明示することです。

```
┌─────────────────────────────────────────────┐
│              ReActパターン                    │
│                                             │
│  Thought: 「まずファイル一覧を確認しよう」    │
│      │                                      │
│      ▼                                      │
│  Action:  list_files("./src")               │
│      │                                      │
│      ▼                                      │
│  Observation: ["main.py", "utils.py", ...]  │
│      │                                      │
│      ▼                                      │
│  Thought: 「main.pyにバグがありそうだ」       │
│      │                                      │
│      ▼                                      │
│  Action:  read_file("./src/main.py")        │
│      │                                      │
│      ▼                                      │
│  Observation: (ファイルの内容)               │
│      │                                      │
│      ▼                                      │
│  Thought: 「15行目の条件式が間違っている」    │
│      │                                      │
│      ▼                                      │
│  Action:  edit_file("./src/main.py", ...)   │
│      │                                      │
│      ▼                                      │
│  Observation: "ファイルを更新しました"        │
│      │                                      │
│      ▼                                      │
│  Thought: 「修正完了。テストを実行しよう」    │
│      │                                      │
│      ▼                                      │
│  ...                                        │
└─────────────────────────────────────────────┘
```

### ReActの実装例（擬似コード）

```python
REACT_SYSTEM_PROMPT = """
あなたはAIエージェントです。以下の形式で思考と行動を繰り返してください。

Thought: (現在の状況の分析と次の行動の理由)
Action: (実行するツール名と引数)

ツールの実行結果がObservationとして返されます。
タスクが完了したら、最終回答を返してください。

利用可能なツール:
- search_web(query): Web検索を実行
- read_file(path): ファイルを読む
- write_file(path, content): ファイルに書き込む
- run_command(cmd): コマンドを実行
"""

def react_loop(user_input: str, max_steps: int = 15) -> str:
    """ReActパターンによるエージェントループ"""
    messages = [
        {"role": "system", "content": REACT_SYSTEM_PROMPT},
        {"role": "user", "content": user_input}
    ]

    for step in range(max_steps):
        # LLMがThought + Actionを生成
        response = llm.chat(messages)
        messages.append({"role": "assistant", "content": response.content})

        # Actionが含まれていなければ完了
        if not contains_action(response.content):
            return extract_final_answer(response.content)

        # Actionを解析して実行
        action = parse_action(response.content)
        observation = execute_tool(action.tool, action.args)

        # Observationをメッセージに追加
        messages.append({
            "role": "user",
            "content": f"Observation: {observation}"
        })

    return "最大ステップ数に到達しました。"
```

### ReActパターンの利点

| 利点 | 説明 |
|---|---|
| 透明性 | Thoughtが明示されるため、判断過程が追跡可能 |
| 精度向上 | 行動前に推論することで、不要なアクションを減らせる |
| デバッグ容易性 | どのステップで判断を誤ったか特定しやすい |
| 柔軟性 | 観察結果に応じて計画を動的に変更できる |

> **ポイントまとめ**
> - ReActは「Thought→Action→Observation」を繰り返すパターン
> - 行動の前に思考を明示することで、透明性と精度が向上する
> - 現在の主要なAIエージェントの多くがReActパターンを採用している

## 2.3 プランニング（計画立案）

複雑なタスクを遂行するには、事前に計画を立てることが重要です。主要なプランニング手法を3つ紹介します。

### タスク分解（Task Decomposition）

大きなタスクを小さなサブタスクに分解する手法です。

```
目標: 「Webアプリケーションにログイン機能を追加する」

分解結果:
├── 1. データベースにユーザーテーブルを作成
├── 2. ユーザー登録APIを実装
├── 3. ログインAPIを実装
├── 4. パスワードのハッシュ化処理を追加
├── 5. セッション管理を実装
├── 6. ログイン画面のUIを作成
├── 7. フロントエンドとAPIを接続
└── 8. テストを作成して実行
```

### Plan-and-Execute（計画と実行の分離）

最初に全体計画を立て、その後で1ステップずつ実行する手法です。

```
┌──────────────┐
│  Planner     │  全体計画を策定
│  (計画者)     │  ステップ1〜Nを列挙
└──────┬───────┘
       │
       ▼
┌──────────────┐     ┌──────────────┐
│  Executor    │────>│  Executor    │────> ...
│  Step 1 実行 │     │  Step 2 実行 │
└──────┬───────┘     └──────┬───────┘
       │                    │
       ▼                    ▼
   結果を確認            結果を確認
  (必要なら再計画)      (必要なら再計画)
```

```python
def plan_and_execute(user_input: str) -> str:
    """Plan-and-Executeパターンの擬似コード"""

    # Step 1: 計画を立てる
    plan = llm.chat([
        {"role": "system", "content": "タスクをステップに分解してください。"},
        {"role": "user", "content": user_input}
    ])
    steps = parse_plan(plan.content)  # ["ステップ1", "ステップ2", ...]

    results = []
    for i, step in enumerate(steps):
        # Step 2: 各ステップを実行
        result = execute_step(step, context=results)
        results.append(result)

        # Step 3: 計画の見直しが必要か判断
        if needs_replan(step, result, steps[i+1:]):
            remaining_steps = replan(user_input, results, steps[i+1:])
            steps = steps[:i+1] + remaining_steps

    return summarize_results(results)
```

### Tree of Thought（思考の木）

複数の思考パスを並列で探索し、最も有望なものを選択する手法です。

```
                    問題
                     │
          ┌──────────┼──────────┐
          ▼          ▼          ▼
       思考A       思考B       思考C
        │           │           │
     評価:70点   評価:90点   評価:40点
                    │
             ┌──────┼──────┐
             ▼      ▼      ▼
           B-1    B-2    B-3
            │      │      │
         85点   95点   60点
                   │
                最良の解
```

### 3つの手法の比較

| 手法 | 計画タイミング | 柔軟性 | 計算コスト | 適したタスク |
|---|---|---|---|---|
| タスク分解 | 実行前に1回 | 低い | 低い | 手順が明確なタスク |
| Plan-and-Execute | 実行前 + 必要に応じて再計画 | 中程度 | 中程度 | 中〜大規模タスク |
| Tree of Thought | 各ステップで複数候補を探索 | 高い | 高い | 最適解が必要なタスク |

> **ポイントまとめ**
> - タスク分解は最もシンプルで実用的なプランニング手法
> - Plan-and-Executeは実行結果に応じた再計画が可能
> - Tree of Thoughtは探索的だが計算コストが高い
> - 実務ではタスク分解 + 必要に応じた再計画の組み合わせが多い

## 2.4 メモリの仕組み

エージェントが効果的に動作するためには、適切なメモリ管理が不可欠です。

### 3種類のメモリ

```
┌─────────────────────────────────────────────────┐
│                  エージェントのメモリ              │
│                                                 │
│  ┌─────────────────────────────────────────┐    │
│  │  短期メモリ（Short-term Memory）         │    │
│  │  = コンテキストウィンドウ                  │    │
│  │                                         │    │
│  │  現在の会話履歴、直近のツール実行結果      │    │
│  │  容量制限あり（例: 200Kトークン）         │    │
│  └─────────────────────────────────────────┘    │
│                                                 │
│  ┌─────────────────────────────────────────┐    │
│  │  長期メモリ（Long-term Memory）          │    │
│  │  = 外部ストレージ                         │    │
│  │                                         │    │
│  │  ベクトルDB、ファイルシステム、            │    │
│  │  ナレッジベース、過去のタスク履歴          │    │
│  │  容量制限なし（実質的に）                  │    │
│  └─────────────────────────────────────────┘    │
│                                                 │
│  ┌─────────────────────────────────────────┐    │
│  │  作業メモリ（Working Memory）            │    │
│  │  = スクラッチパッド                       │    │
│  │                                         │    │
│  │  現在のタスクの計画、中間結果、            │    │
│  │  TODOリスト、メモ                        │    │
│  └─────────────────────────────────────────┘    │
└─────────────────────────────────────────────────┘
```

### 各メモリの比較

| メモリ種別 | 保存先 | 容量 | 速度 | 用途 |
|---|---|---|---|---|
| 短期メモリ | コンテキストウィンドウ | 制限あり | 最速 | 現在の会話の文脈保持 |
| 長期メモリ | 外部DB / ファイル | 実質無制限 | 検索が必要 | 過去の知識・経験の保存 |
| 作業メモリ | ファイル / 変数 | 中程度 | 速い | 現在のタスクの進捗管理 |

### コンテキストウィンドウの管理

短期メモリであるコンテキストウィンドウには容量制限があります。長いタスクでは以下の戦略が重要です。

```python
def manage_context(messages: list, max_tokens: int = 150000) -> list:
    """コンテキストウィンドウの管理戦略"""

    total_tokens = count_tokens(messages)

    if total_tokens <= max_tokens:
        return messages  # 制限内ならそのまま

    # 戦略1: 古いメッセージを要約して圧縮
    summary = llm.chat([{
        "role": "user",
        "content": f"以下の会話を要約してください:\n{format_messages(messages[:10])}"
    }])

    # 戦略2: 要約 + 直近のメッセージを保持
    compressed = [
        {"role": "system", "content": f"これまでの要約: {summary.content}"},
        *messages[-10:]  # 直近10メッセージは保持
    ]

    return compressed
```

### よくある間違い

| 間違い | 正しい理解 |
|---|---|
| 「コンテキストウィンドウは無限」 | 必ず上限がある。超えると古い情報が失われる |
| 「全ての情報をコンテキストに入れる」 | 必要な情報だけを選択的に含める。不要な情報はノイズになる |
| 「メモリの管理は不要」 | 長時間のタスクではメモリ管理が成否を分ける |

> **ポイントまとめ**
> - 短期メモリ（コンテキスト）、長期メモリ（外部DB）、作業メモリの3種類がある
> - コンテキストウィンドウの容量管理がエージェントの性能に直結する
> - 要約や選択的な情報保持で、限られたコンテキストを有効活用する

## 2.5 ツール使用（Tool Use）の基本

### Function Calling の仕組み

Function Calling（関数呼び出し）は、LLMが外部のツールや関数を呼び出すための仕組みです。

```
┌──────┐   (1) ツール定義 + ユーザー入力   ┌──────┐
│      │ ──────────────────────────────>   │      │
│      │                                   │      │
│クライ│   (2) ツール呼び出し要求           │ LLM  │
│アント│ <──────────────────────────────   │      │
│      │                                   │      │
│      │   (3) ツール実行結果               │      │
│      │ ──────────────────────────────>   │      │
│      │                                   │      │
│      │   (4) 最終回答                     │      │
│      │ <──────────────────────────────   │      │
└──────┘                                   └──────┘
```

### ツール定義（JSON Schema）

ツールはJSON Schema形式で定義します。LLMはこの定義を読んで、適切なタイミングでツールを呼び出します。

```python
tools = [
    {
        "name": "read_file",
        "description": "指定されたパスのファイルを読み込んで内容を返します",
        "input_schema": {
            "type": "object",
            "properties": {
                "path": {
                    "type": "string",
                    "description": "読み込むファイルのパス"
                }
            },
            "required": ["path"]
        }
    },
    {
        "name": "write_file",
        "description": "指定されたパスにファイルを書き込みます",
        "input_schema": {
            "type": "object",
            "properties": {
                "path": {
                    "type": "string",
                    "description": "書き込み先のファイルパス"
                },
                "content": {
                    "type": "string",
                    "description": "書き込む内容"
                }
            },
            "required": ["path", "content"]
        }
    },
    {
        "name": "run_command",
        "description": "シェルコマンドを実行して結果を返します",
        "input_schema": {
            "type": "object",
            "properties": {
                "command": {
                    "type": "string",
                    "description": "実行するコマンド"
                }
            },
            "required": ["command"]
        }
    }
]
```

### ツール実行フローの実装例

```python
import json

def tool_use_loop(user_input: str, tools: list) -> str:
    """Function Callingを使ったエージェントループ"""
    messages = [{"role": "user", "content": user_input}]

    while True:
        # LLMにツール定義と会話履歴を渡す
        response = llm.chat(
            messages=messages,
            tools=tools,
            max_tokens=4096
        )

        # ツール呼び出しがなければ完了
        if response.stop_reason == "end_turn":
            return response.content

        # ツール呼び出しを処理
        if response.stop_reason == "tool_use":
            for block in response.content:
                if block.type == "tool_use":
                    # ツールを実行
                    tool_result = execute_tool(
                        name=block.name,
                        arguments=block.input
                    )

                    # 結果をメッセージに追加
                    messages.append({
                        "role": "assistant",
                        "content": response.content
                    })
                    messages.append({
                        "role": "user",
                        "content": [{
                            "type": "tool_result",
                            "tool_use_id": block.id,
                            "content": str(tool_result)
                        }]
                    })

def execute_tool(name: str, arguments: dict) -> str:
    """ツールの実行（実装例）"""
    if name == "read_file":
        with open(arguments["path"], "r") as f:
            return f.read()
    elif name == "write_file":
        with open(arguments["path"], "w") as f:
            f.write(arguments["content"])
        return "ファイルを書き込みました。"
    elif name == "run_command":
        import subprocess
        result = subprocess.run(
            arguments["command"],
            shell=True,
            capture_output=True,
            text=True
        )
        return result.stdout + result.stderr
    else:
        return f"不明なツール: {name}"
```

### よくある間違い

| 間違い | 正しい理解 |
|---|---|
| 「ツールは何でも自由に実行できる」 | セキュリティのため、許可されたツールのみ実行可能にする |
| 「ツールの説明文は適当でよい」 | 説明文がLLMの判断基準になる。明確で具体的に書く |
| 「ツール実行結果を検証しない」 | 実行結果のバリデーションとエラーハンドリングが必要 |

> **ポイントまとめ**
> - Function CallingはLLMが外部ツールを呼び出す標準的な仕組み
> - ツールはJSON Schemaで名前・説明・パラメータを定義する
> - ツールの説明文の品質がエージェントの精度に直結する

## 2.6 エラーハンドリングと自己修正

実際のタスク遂行では、エラーの発生は避けられません。優れたエージェントはエラーから回復する能力を持っています。

### リトライ戦略

```python
def retry_with_backoff(func, max_retries: int = 3) -> str:
    """指数バックオフ付きリトライ"""
    for attempt in range(max_retries):
        try:
            result = func()
            return result
        except Exception as e:
            if attempt == max_retries - 1:
                raise  # 最終試行で失敗したら例外を投げる
            wait_time = 2 ** attempt  # 1秒, 2秒, 4秒...
            print(f"エラー発生: {e}. {wait_time}秒後にリトライ...")
            time.sleep(wait_time)
```

### フォールバック戦略

メインの手段が失敗した場合に、代替手段に切り替える戦略です。

```
メイン手段 ──失敗──> 代替手段1 ──失敗──> 代替手段2 ──失敗──> エラー報告

例:
API呼び出し ──失敗──> キャッシュ参照 ──失敗──> デフォルト値を使用
```

```python
def with_fallback(primary_fn, fallback_fn, final_fallback=None):
    """フォールバック付き実行"""
    try:
        return primary_fn()
    except Exception as e:
        print(f"メイン手段が失敗: {e}")
        try:
            return fallback_fn()
        except Exception as e2:
            print(f"代替手段も失敗: {e2}")
            if final_fallback:
                return final_fallback()
            raise
```

### 自己反省（Reflection）パターン

エージェントが自分の行動を振り返り、改善する手法です。

```
┌──────────────┐
│   行動実行    │
└──────┬───────┘
       │
       ▼
┌──────────────┐
│   結果確認    │  期待通りか？
└──────┬───────┘
       │
       ├── 成功 → 次のステップへ
       │
       ▼ 失敗
┌──────────────┐
│   自己反省    │  何が間違っていたか？
│  Reflection  │  どうすればよかったか？
└──────┬───────┘
       │
       ▼
┌──────────────┐
│   改善行動    │  反省を踏まえて再試行
└──────────────┘
```

```python
def reflect_and_retry(task: str, previous_attempt: str, error: str) -> str:
    """自己反省パターンの擬似コード"""

    # 反省フェーズ: 何が間違っていたか分析
    reflection = llm.chat([
        {"role": "system", "content": "あなたは自分の行動を振り返るAIです。"},
        {"role": "user", "content": f"""
タスク: {task}
前回の試行: {previous_attempt}
エラー: {error}

以下を分析してください:
1. 何が間違っていたか
2. なぜそのエラーが発生したか
3. 次にどうすればよいか
"""}
    ])

    # 改善フェーズ: 反省を踏まえて再試行
    improved_result = llm.chat([
        {"role": "system", "content": f"""
前回の反省: {reflection.content}
この反省を踏まえて、タスクを再実行してください。
"""},
        {"role": "user", "content": task}
    ])

    return improved_result.content
```

### エラーハンドリング戦略の比較

| 戦略 | 概要 | 適用場面 |
|---|---|---|
| リトライ | 同じ操作を再試行 | 一時的なエラー（ネットワーク障害等） |
| フォールバック | 代替手段に切り替え | メイン手段が利用不可の場合 |
| 自己反省 | 原因を分析して改善 | ロジックのミスや方針の誤り |
| エスカレーション | 人間に判断を委ねる | 自動では解決できない問題 |

### よくある間違い

| 間違い | 正しい理解 |
|---|---|
| 「エラーが出たら即座に停止する」 | 回復可能なエラーはリトライや代替手段で対処する |
| 「無限にリトライする」 | リトライ上限を設定し、超えたらエスカレーションする |
| 「エラーの原因を分析しない」 | 自己反省で原因を特定し、同じエラーの再発を防ぐ |

> **ポイントまとめ**
> - リトライ、フォールバック、自己反省、エスカレーションの4つの戦略がある
> - 自己反省（Reflection）パターンは高度なエージェントの重要な特徴
> - エラーの種類に応じて適切な戦略を選択することが重要
> - 最終的に解決できない場合は人間にエスカレーションする設計が安全

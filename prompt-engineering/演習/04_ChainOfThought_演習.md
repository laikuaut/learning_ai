### 問題1：Zero-Shot CoTの適用

以下の問題にZero-Shot CoTを適用するプロンプトを作成し、LLMに投入してみてください。

**問題：** あるWebサービスのサーバーは1秒間に500リクエストを処理できます。ピーク時のトラフィックは1秒間に2,000リクエストです。ロードバランサーで複数台に分散する場合、最低何台のサーバーが必要ですか？また、1台が故障した場合にも対応できるようにするには何台必要ですか？

**期待される出力例：**
```
ステップバイステップで考えます。

ステップ1：必要な最低台数を計算
2,000 ÷ 500 = 4台

ステップ2：1台故障時の冗長性を考慮
4台 + 1台（予備）= 5台
```

<summary>解答例</summary>

```
■ プロンプト：
「あるWebサービスのサーバーは1秒間に500リクエストを処理できます。
ピーク時のトラフィックは1秒間に2,000リクエストです。
ロードバランサーで複数台に分散する場合、最低何台のサーバーが
必要ですか？また、1台が故障した場合にも対応できるようにするには
何台必要ですか？

ステップバイステップで考えてください。」

■ 期待される回答：

ステップ1：必要な処理能力を確認する
・ピーク時トラフィック：2,000 リクエスト/秒
・1台の処理能力：500 リクエスト/秒

ステップ2：最低必要台数を計算する
・2,000 ÷ 500 = 4台
・ピーク時のトラフィックを処理するには最低4台必要

ステップ3：冗長性を考慮する（N+1構成）
・4台中1台が故障すると、残り3台で1,500 リクエスト/秒しか処理できない
・2,000 リクエスト/秒を維持するには、4台 + 予備1台 = 5台必要

ステップ4：追加の考慮事項
・実際には各サーバーを80%程度の負荷で運用するのが望ましい
・80%負荷で計算：2,000 ÷ (500 × 0.8) = 5台 + 予備1台 = 6台

結論：
・最低台数：4台
・1台故障に対応（N+1）：5台
・推奨（80%負荷 + N+1）：6台
```
</details>

  US-5(5SP)の代わりにUS-6(8SP)は入らない（29+8=37SP）
  US-5は次スプリントに回す

ステップ4：結論
  取り込むストーリー：US-1, US-2, US-3, US-4
  合計：29SP（ベロシティ30SPの97%、適切な計画）

### 問題4：逆方向推論の実践

以下のバグ報告について、逆方向推論（症状から原因を追跡）で原因を特定するプロンプトを作成し、LLMに投入してみてください。

**バグ報告：**
- 症状：ECサイトで特定の商品の価格が「0円」と表示される
- 発生条件：割引キャンペーン適用中の商品のみ
- 正常なケース：キャンペーン対象外の商品は正しく表示
- 環境：本番環境のみ（ステージング環境では再現しない）

<details>
<summary>解答例</summary>

```
■ プロンプト：
「以下のバグについて、逆方向推論で原因を特定してください。
症状から出発し、直接的な原因、その原因の原因と遡ってください。

バグ報告：
- 症状：ECサイトで特定の商品の価格が「0円」と表示される
- 発生条件：割引キャンペーン適用中の商品のみ
- 正常なケース：キャンペーン対象外の商品は正しく表示
- 環境：本番環境のみ（ステージングでは再現しない）

ステップ1：最終的な症状から出発する
ステップ2：直接的な原因を推測する（なぜ0円？）
ステップ3：その原因のさらに上流の原因を推測する
ステップ4：本番とステージングの差異から原因を絞り込む
ステップ5：最も可能性の高い根本原因を結論づける」

■ 期待される回答：

ステップ1：症状の確認
  価格が「0円」→ 表示値が0になっている

ステップ2：直接的な原因の推測
  なぜ価格が0になるか？
  a) 商品マスタの価格が0に設定されている
  b) 割引計算で100%引きになっている
  c) APIが価格フィールドにnull/0を返している
  → 発生条件が「キャンペーン適用中のみ」なので b) が有力

ステップ3：上流原因の推測
  なぜ100%引きになるか？
  a) キャンペーンの割引率が100%に設定されている
  b) 割引計算ロジックにバグがある（例：金額と%の混同）
  c) キャンペーンデータの型が間違っている
     （例：「30」を30%でなく30円引きと解釈）

ステップ4：本番 vs ステージングの差異
  ステージングで再現しない理由：
  a) キャンペーンデータが異なる
  b) 環境変数や設定が異なる
  c) データベースのデータが異なる
  → 最有力：本番のキャンペーンデータの設定値が想定と異なる

ステップ5：根本原因の結論
  最も可能性が高い原因：
  本番環境のキャンペーンマスタで、割引率の入力形式が
  不統一（例：「30%」と入力すべきところを「30」と入力し、
  コードが「30」を金額（30円引き）ではなく割引率（3000%）と
  解釈している）。

  確認すべきこと：
  1. 本番のキャンペーンテーブルの割引率カラムの値
  2. 割引計算のコードで単位（%か金額か）の判定ロジック
  3. ステージングと本番のキャンペーンデータの差異
```

## チャレンジ問題

### 問題5：自己整合性の実践

以下のシステム設計の問題を、3つの異なるアプローチ（観点）から推論し、最終的に最も信頼性の高い結論を導くプロンプトを作成してください。

**問題：** 月間100万PVのブログサービスで、記事の全文検索機能を実装したい。検索エンジンとしてElasticsearch、PostgreSQLの全文検索、Algoliaのどれを選ぶべきか？

3つのアプローチ：
1. コストの観点
2. 技術的な適合性の観点
3. 運用・保守の観点

<details>
<summary>解答例</summary>

```
■ プロンプト：
「以下の問題を3つの異なるアプローチで分析し、
最終的に最も信頼性の高い結論を導いてください。

問題：月間100万PVのブログサービスで記事の全文検索を実装。
選択肢：Elasticsearch、PostgreSQL全文検索、Algolia

### アプローチ1：コストの観点
初期導入コスト、運用コスト、スケール時のコストを比較して推論。

### アプローチ2：技術的な適合性の観点
検索精度、レスポンス速度、日本語対応、スケーラビリティを
比較して推論。

### アプローチ3：運用・保守の観点
運用負荷、チームのスキルセット、障害時の対応、
将来の拡張性を比較して推論。

### 総合判断
3つのアプローチの結論を比較し、一致する点と異なる点を整理した上で、
最終的な推奨を述べてください。」

■ 期待される出力の構造：
・各アプローチで独立に推論し結論を出す
・3つの結論を比較して多数決または重み付けで最終判断
・選ばなかった選択肢についても「この条件なら有効」を補足
```
</details>

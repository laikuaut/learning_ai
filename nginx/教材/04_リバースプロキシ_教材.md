# 第4章：リバースプロキシ

> **この章のゴール**: Nginxをリバースプロキシとして使い、バックエンドアプリケーションへリクエストを中継できるようになる。
> proxy_pass の基本から、ヘッダー設定、WebSocket対応、タイムアウト設定まで理解すれば、実務で使われるプロキシ構成が組めるようになります。

---

## なぜリバースプロキシを学ぶのか？

現代のWebアプリケーションでは、Node.js、Python（Django/Flask）、Rubyなどのアプリケーションサーバーが直接インターネットに公開されることは稀です。Nginxをリバースプロキシとして前段に置くのが標準的な構成です。

```
インターネット         Nginx              バックエンド
┌──────────┐      ┌──────────┐      ┌──────────────┐
│クライアント│─────▶│リバース   │─────▶│ Node.js      │
│(ブラウザ) │◀─────│プロキシ   │◀─────│ (port 3000)  │
└──────────┘      │(port 80) │      └──────────────┘
                  └──────────┘      ┌──────────────┐
                       │           │ Python/Flask  │
                       └──────────▶│ (port 5000)  │
                                   └──────────────┘
```

### リバースプロキシのメリット

- **SSL終端**: NginxでSSL/TLSを処理し、バックエンドは平文通信で済む
- **静的ファイル配信**: 静的ファイルはNginxが直接配信（高速）
- **負荷分散**: 複数のバックエンドサーバーに分散可能（第5章で詳しく解説）
- **セキュリティ**: バックエンドの情報（ポート番号、サーバーソフト等）を隠蔽
- **バッファリング**: レスポンスをバッファリングしてバックエンドの負荷を軽減

## 4.1 プロキシとリバースプロキシの違い

### フォワードプロキシ（Forward Proxy）

**フォワードプロキシ（Forward Proxy）** は、クライアント側に設置され、クライアントの代理としてサーバーにアクセスします。

```
クライアント → フォワードプロキシ → インターネット → サーバー
              ├ クライアントのIPを隠す
              └ コンテンツフィルタリング
```

### リバースプロキシ（Reverse Proxy）

**リバースプロキシ（Reverse Proxy）** は、サーバー側に設置され、サーバーの代理としてクライアントのリクエストを受け取ります。

```
クライアント → インターネット → リバースプロキシ → バックエンドサーバー
                                ├ バックエンドを隠す
                                ├ SSL終端
                                └ 負荷分散
```

| 特徴 | フォワードプロキシ | リバースプロキシ |
|---|---|---|
| 設置場所 | クライアント側 | サーバー側 |
| 目的 | クライアントの匿名化、フィルタリング | サーバーの保護、負荷分散 |
| 代表例 | Squid, 企業のプロキシ | Nginx, HAProxy |
| 誰のために動く | クライアント | サーバー |

## 4.2 proxy_pass の基本設定

### 最もシンプルなリバースプロキシ

```nginx
server {
    listen 80;
    server_name example.com;

    location / {
        proxy_pass http://127.0.0.1:3000;
    }
}
```

この設定で、`example.com` へのすべてのリクエストが `127.0.0.1:3000` に転送されます。

### proxy_pass の末尾スラッシュに注意

`proxy_pass` の末尾に `/` があるかないかで、URIの渡し方が変わります。

```nginx
# パターン1：末尾スラッシュなし
location /app/ {
    proxy_pass http://127.0.0.1:3000;
    # /app/hello → http://127.0.0.1:3000/app/hello（パスがそのまま渡る）
}

# パターン2：末尾スラッシュあり
location /app/ {
    proxy_pass http://127.0.0.1:3000/;
    # /app/hello → http://127.0.0.1:3000/hello（/app/ が除去される）
}
```

```
パターン1（スラッシュなし）:
  クライアント: GET /app/hello
  バックエンド: GET /app/hello  ← パスがそのまま

パターン2（スラッシュあり）:
  クライアント: GET /app/hello
  バックエンド: GET /hello      ← /app/ が取り除かれる
```

> **⚠️ よくある間違い**: `proxy_pass` の末尾スラッシュの有無は非常に重要です。これを間違えると、バックエンドに意図しないパスでリクエストが届き、404エラーの原因になります。

### パスの書き換え例

```nginx
# /api/v1/users → http://backend:8080/users
location /api/v1/ {
    proxy_pass http://backend:8080/;
}

# /legacy/ → http://old-server:9000/v2/
location /legacy/ {
    proxy_pass http://old-server:9000/v2/;
}
```

## 4.3 upstream ブロックによるバックエンド定義

### upstream の基本

**upstream ブロック** を使うと、バックエンドサーバーに名前を付けて管理できます。

```nginx
upstream backend_app {
    server 127.0.0.1:3000;
}

server {
    listen 80;
    server_name example.com;

    location / {
        proxy_pass http://backend_app;
    }
}
```

### 複数のバックエンドサーバー

upstream では複数のサーバーを定義でき、ロードバランシング（第5章で詳解）に使えます。

```nginx
upstream backend_app {
    server 127.0.0.1:3000;
    server 127.0.0.1:3001;
    server 127.0.0.1:3002;
}

server {
    listen 80;
    server_name example.com;

    location / {
        proxy_pass http://backend_app;
    }
}
```

## 4.4 プロキシヘッダーの設定

### なぜヘッダー設定が必要か？

リバースプロキシを経由すると、バックエンドアプリケーションから見たクライアント情報が失われます。

```
ヘッダー未設定の場合:
  クライアント(203.0.113.10) → Nginx(192.168.1.1) → バックエンド
  バックエンドが見るクライアントIP: 192.168.1.1（Nginxのアドレス）
  → 本当のクライアントIPが分からない！

ヘッダー設定済みの場合:
  クライアント(203.0.113.10) → Nginx(192.168.1.1) → バックエンド
  X-Real-IP: 203.0.113.10
  X-Forwarded-For: 203.0.113.10
  → バックエンドが本当のクライアントIPを取得できる
```

### 標準的なプロキシヘッダー設定

```nginx
server {
    listen 80;
    server_name example.com;

    location / {
        proxy_pass http://127.0.0.1:3000;

        # クライアントの実IPアドレスを渡す
        proxy_set_header X-Real-IP $remote_addr;

        # プロキシチェーン全体のIPアドレスを渡す
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;

        # 元のホスト名を渡す（バックエンドがドメイン名を参照できるように）
        proxy_set_header Host $host;

        # プロトコル情報を渡す（HTTPSかどうか）
        proxy_set_header X-Forwarded-Proto $scheme;
    }
}
```

### 各ヘッダーの役割

| ヘッダー | 値の例 | 用途 |
|---|---|---|
| `X-Real-IP` | `203.0.113.10` | クライアントの実IPアドレス |
| `X-Forwarded-For` | `203.0.113.10, 10.0.0.1` | 経由したプロキシチェーンのIP一覧 |
| `Host` | `example.com` | 元のリクエストのドメイン名 |
| `X-Forwarded-Proto` | `https` | 元のリクエストのプロトコル |

> **📖 ポイント**: `$proxy_add_x_forwarded_for` は、既存の `X-Forwarded-For` ヘッダーにクライアントIPを追加する変数です。複数段のプロキシがある場合にも対応できます。

### 共通設定のファイル分離

プロキシヘッダーの設定は複数の location で共通することが多いため、別ファイルに分離するのが実務的です。

```nginx
# /etc/nginx/proxy_params（共通ヘッダー設定）
proxy_set_header X-Real-IP $remote_addr;
proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
proxy_set_header Host $host;
proxy_set_header X-Forwarded-Proto $scheme;
```

```nginx
# server ブロックで include
server {
    listen 80;
    server_name example.com;

    location / {
        proxy_pass http://127.0.0.1:3000;
        include proxy_params;
    }

    location /api/ {
        proxy_pass http://127.0.0.1:8080;
        include proxy_params;
    }
}
```

## 4.5 WebSocket のプロキシ

### WebSocket とは

**WebSocket** は、クライアントとサーバー間で双方向のリアルタイム通信を行うためのプロトコルです。チャットアプリやリアルタイム通知などで使われます。

WebSocketは最初にHTTPで接続を開始し、「アップグレード（Upgrade）」によってWebSocketプロトコルに切り替わります。

```
クライアント                    サーバー
    │  GET /ws HTTP/1.1           │
    │  Upgrade: websocket         │
    │  Connection: Upgrade        │
    │ ──────────────────────────▶ │
    │                             │
    │  HTTP/1.1 101 Switching     │
    │ ◀────────────────────────── │
    │                             │
    │ ◀═══ WebSocket通信 ═══════▶ │
    │    （双方向リアルタイム）     │
```

### WebSocket対応のプロキシ設定

```nginx
server {
    listen 80;
    server_name example.com;

    # 通常のHTTPリクエスト
    location / {
        proxy_pass http://127.0.0.1:3000;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
    }

    # WebSocket用のパス
    location /ws/ {
        proxy_pass http://127.0.0.1:3000;
        proxy_http_version 1.1;

        # WebSocketアップグレードに必要なヘッダー
        proxy_set_header Upgrade $http_upgrade;
        proxy_set_header Connection "upgrade";

        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;

        # WebSocketは長時間接続のため、タイムアウトを長めに設定
        proxy_read_timeout 86400s;
        proxy_send_timeout 86400s;
    }
}
```

> **⚠️ よくある間違い**: WebSocketプロキシで `proxy_http_version 1.1;` を忘れると、HTTP/1.0が使われてアップグレードに失敗します。WebSocketにはHTTP/1.1が必須です。

## 4.6 タイムアウト設定

### タイムアウトの種類

バックエンドとの通信では、適切なタイムアウトを設定しないと、応答しないバックエンドに接続が占有されてしまいます。

```nginx
server {
    listen 80;
    server_name example.com;

    location / {
        proxy_pass http://127.0.0.1:3000;

        # バックエンドへの接続タイムアウト（デフォルト: 60s）
        proxy_connect_timeout 5s;

        # バックエンドからのレスポンス読み取りタイムアウト（デフォルト: 60s）
        proxy_read_timeout 60s;

        # バックエンドへのリクエスト送信タイムアウト（デフォルト: 60s）
        proxy_send_timeout 60s;
    }
}
```

### 各タイムアウトの流れ

```
クライアント        Nginx           バックエンド
    │                │                  │
    │── リクエスト ──▶│                  │
    │                │── 接続開始 ─────▶│  ← proxy_connect_timeout
    │                │                  │    （接続確立までの時間）
    │                │── データ送信 ───▶│  ← proxy_send_timeout
    │                │                  │    （リクエスト送信の時間）
    │                │                  │
    │                │◀─ データ受信 ───│  ← proxy_read_timeout
    │                │                  │    （レスポンス受信の時間）
    │◀── レスポンス ──│                  │
```

### 用途別のタイムアウト設定例

```nginx
# 一般的なWebアプリケーション
location / {
    proxy_pass http://backend;
    proxy_connect_timeout 5s;    # 接続は素早く
    proxy_read_timeout 30s;      # レスポンスは30秒以内
    proxy_send_timeout 30s;
}

# ファイルアップロード用（大きなファイルの場合）
location /upload/ {
    proxy_pass http://backend;
    proxy_connect_timeout 5s;
    proxy_read_timeout 300s;     # 処理に時間がかかるため長めに
    proxy_send_timeout 300s;     # アップロードにも時間がかかる
    client_max_body_size 100m;   # アップロードサイズ上限
}

# 長時間処理API（レポート生成など）
location /api/reports/ {
    proxy_pass http://backend;
    proxy_connect_timeout 5s;
    proxy_read_timeout 600s;     # 10分まで待つ
    proxy_send_timeout 60s;
}
```

## 4.7 バッファリング設定

### バッファリングとは

Nginxはバックエンドからのレスポンスをメモリ（またはディスク）に**バッファリング（buffering）**してからクライアントに返すことができます。

```
バッファリングあり（デフォルト）:
  バックエンド ──全データ──▶ Nginx（バッファ） ──全データ──▶ クライアント
  → バックエンドは素早く解放される

バッファリングなし:
  バックエンド ──少しずつ──▶ Nginx ──少しずつ──▶ クライアント
  → ストリーミング向き（Server-Sent Events等）
```

### バッファリングの設定

```nginx
server {
    listen 80;
    server_name example.com;

    location / {
        proxy_pass http://127.0.0.1:3000;

        # バッファリングを有効化（デフォルト: on）
        proxy_buffering on;

        # バッファの数とサイズ
        proxy_buffers 16 16k;

        # バッファリング前に読み取るレスポンスヘッダーのサイズ
        proxy_buffer_size 32k;

        # バッファが溢れた場合に一時ファイルに書き出す最大サイズ
        proxy_max_temp_file_size 1024m;
    }

    # ストリーミング用（Server-Sent Events, イベントストリーム）
    location /events/ {
        proxy_pass http://127.0.0.1:3000;
        proxy_buffering off;        # バッファリング無効
        proxy_cache off;            # キャッシュも無効
        proxy_read_timeout 86400s;  # 長時間接続
    }
}
```

> **📖 ポイント**: バッファリングが有効な場合、Nginxはバックエンドからのレスポンスを全て受け取ってからクライアントに返します。これにより、バックエンドの接続が早く解放され、全体の処理効率が向上します。

## 4.8 実用例：Node.js / Python アプリへのプロキシ

### Node.js（Express）アプリへのプロキシ

```nginx
upstream nodejs_app {
    server 127.0.0.1:3000;
}

server {
    listen 80;
    server_name app.example.com;

    access_log /var/log/nginx/app.access.log;
    error_log  /var/log/nginx/app.error.log;

    # 静的ファイルはNginxが直接配信（高速）
    location /static/ {
        alias /var/www/app/static/;
        expires 30d;
        add_header Cache-Control "public, immutable";
    }

    # favicon.ico
    location = /favicon.ico {
        alias /var/www/app/static/favicon.ico;
        access_log off;
    }

    # それ以外はNode.jsに転送
    location / {
        proxy_pass http://nodejs_app;
        proxy_http_version 1.1;

        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $scheme;

        # WebSocket対応
        proxy_set_header Upgrade $http_upgrade;
        proxy_set_header Connection "upgrade";
    }
}
```

### Python（Gunicorn + Django/Flask）アプリへのプロキシ

```nginx
upstream python_app {
    # Gunicornのソケット接続（パフォーマンスが高い）
    server unix:/run/gunicorn/app.sock;

    # TCP接続も可能
    # server 127.0.0.1:8000;
}

server {
    listen 80;
    server_name api.example.com;

    access_log /var/log/nginx/api.access.log;
    error_log  /var/log/nginx/api.error.log;

    # クライアントからのリクエストボディの最大サイズ
    client_max_body_size 10m;

    # 静的ファイル（Django の collectstatic で生成されたもの）
    location /static/ {
        alias /var/www/api/staticfiles/;
        expires 30d;
    }

    # メディアファイル（ユーザーアップロード）
    location /media/ {
        alias /var/www/api/media/;
        expires 7d;
    }

    # APIリクエストをGunicornに転送
    location / {
        proxy_pass http://python_app;

        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $scheme;

        # タイムアウト設定
        proxy_connect_timeout 5s;
        proxy_read_timeout 60s;
        proxy_send_timeout 60s;
    }
}
```

### UNIXソケット vs TCP接続

```
UNIXソケット（推奨：同一サーバーの場合）:
  server unix:/run/gunicorn/app.sock;
  ├ ネットワークスタックを経由しないため高速
  ├ ファイルパーミッションでアクセス制御可能
  └ 同一マシン上でのみ使用可能

TCP接続:
  server 127.0.0.1:8000;
  ├ 異なるマシンにバックエンドがある場合に必須
  ├ ポート番号で管理しやすい
  └ ネットワークスタックを経由するためやや遅い
```

### 完全な設定例：フロントエンド + API の構成

```nginx
# フロントエンド（React SPAなど）
server {
    listen 80;
    server_name example.com;

    root /var/www/frontend/build;
    index index.html;

    # SPAルーティング：存在しないパスは index.html にフォールバック
    location / {
        try_files $uri $uri/ /index.html;
    }

    # APIリクエストをバックエンドに転送
    location /api/ {
        proxy_pass http://127.0.0.1:8000;

        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $scheme;
    }

    # WebSocket
    location /ws/ {
        proxy_pass http://127.0.0.1:8000;
        proxy_http_version 1.1;
        proxy_set_header Upgrade $http_upgrade;
        proxy_set_header Connection "upgrade";
        proxy_set_header Host $host;
        proxy_read_timeout 86400s;
    }
}
```

## ポイントまとめ

| 項目 | 内容 |
|---|---|
| リバースプロキシ | サーバー側に配置し、クライアントのリクエストをバックエンドに中継する |
| proxy_pass | バックエンドへの転送先を指定（末尾スラッシュの有無に注意） |
| upstream | バックエンドサーバーに名前を付けて管理する |
| プロキシヘッダー | X-Real-IP, X-Forwarded-For, Host で元の接続情報を保持する |
| WebSocket | `Upgrade` / `Connection` ヘッダーと `proxy_http_version 1.1` が必要 |
| タイムアウト | connect / read / send の3種類を用途に合わせて設定する |
| バッファリング | デフォルト有効。ストリーミング用途では無効にする |

> **次の章では**: ロードバランシングについて学びます。upstream ブロックを活用して複数のバックエンドサーバーにリクエストを分散する方法を理解します。

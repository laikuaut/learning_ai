## 課題の要件

1. **パーソナルアシスタントエージェント**を題材に、メモリシステムを設計すること
   - ユーザーの好みや過去の依頼内容を記憶する
   - 長期間（数週間〜数ヶ月）にわたって一貫した対応を提供する
2. **短期メモリ**の設計（以下を含む）
   - 会話履歴の管理方法（コンテキストウィンドウの制約への対処）
   - タスク実行中の中間状態の保持
   - ツール実行結果のキャッシュ
3. **長期メモリ**の設計（以下を含む）
   - ユーザープロファイル（user profile）のデータ構造
   - 過去の対話から抽出する情報の種類と抽出方法
   - メモリの検索・呼び出し方法
4. **メモリの書き込み・読み出しタイミング**を明確に定義すること
5. **メモリの劣化対策**（古い情報の扱い、矛盾する情報の解決）を設計すること
6. 全体を**データ構造（JSON Schema）** と**擬似コード**で表現すること

## ステップガイド

<details>
<summary>ステップ1：メモリの種類と特性を整理する</summary>

人間の記憶システムとのアナロジーで考えると理解しやすいです。

| メモリ種別 | 人間の記憶 | エージェントでの対応 | 保持期間 | 容量 |
|-----------|----------|-------------------|---------|------|
| 短期メモリ | ワーキングメモリ | 現在の会話コンテキスト | セッション中 | 制限あり（コンテキストウィンドウ） |
| 長期メモリ（エピソード） | エピソード記憶 | 過去の対話記録 | 永続 | 大容量 |
| 長期メモリ（意味） | 意味記憶 | ユーザーの好み・事実 | 永続 | 中容量 |
| 長期メモリ（手続き） | 手続き記憶 | 学習済みパターン | 永続 | 小容量 |

</details>

<details>
<summary>ステップ2：短期メモリの設計</summary>

短期メモリで最も重要なのは「コンテキストウィンドウの制約」への対処です。

LLMのコンテキストウィンドウ（context window）には上限があるため、会話が長くなると古いメッセージを削除する必要があります。

主な戦略:
- **スライディングウィンドウ（sliding window）**: 直近N件のメッセージのみ保持
- **要約ベース（summarization）**: 古いメッセージを要約して圧縮
- **重要度ベース**: 重要なメッセージを優先的に保持

</details>

<details>
<summary>ステップ3：長期メモリの設計</summary>

長期メモリの設計では以下を決める必要があります。

1. **何を記憶するか**: すべての会話を保存するのか、重要な情報だけを抽出するのか
2. **どう保存するか**: 構造化データ（JSON）か、非構造化データ（テキスト）か
3. **どう検索するか**: キーワード検索か、ベクトル検索か、構造化クエリか

**よくある間違い**: すべての会話をそのまま長期メモリに保存してしまうこと。ノイズが増えて検索精度が下がります。重要な情報（ユーザーの好み、決定事項、学習したパターン）を抽出して構造化して保存しましょう。

</details>

<details>
<summary>ステップ4：メモリの読み書きタイミングを定義する</summary>

以下のようなイベントでメモリの読み書きが発生します。

| イベント | 読み出し | 書き込み |
|---------|---------|---------|
| セッション開始 | 長期メモリからユーザープロファイルを読み出し | - |
| ユーザー発話 | 関連する過去の対話を検索 | 短期メモリに追加 |
| ツール実行後 | - | 実行結果を短期メモリに追加 |
| エージェント応答後 | - | 応答を短期メモリに追加 |
| セッション終了 | - | 重要情報を長期メモリに書き込み |

</details>

# 擬似コード: スライディングウィンドウ方式

def manage_context_window(memory, max_tokens=100000):
    """コンテキストウィンドウに収まるよう会話履歴を管理する"""

    # 現在のトークン数を計算
    total_tokens = count_tokens(memory.conversation_history)

    if total_tokens <= max_tokens:
        return memory  # 制限内なのでそのまま

    # 制限を超えた場合: 古いメッセージを要約
    old_messages = memory.conversation_history[:-10]  # 直近10件以外
    recent_messages = memory.conversation_history[-10:]  # 直近10件は保持

    summary = llm.summarize(old_messages)

    memory.conversation_history = [
        {"role": "system", "content": f"これまでの会話の要約: {summary}"},
        *recent_messages
    ]

    return memory
```

def save_to_long_term_memory(session, long_term_memory):
    """セッションから重要情報を抽出して長期メモリに保存する"""

    # LLMを使って重要情報を抽出
    extraction_prompt = f"""
    以下の会話から、今後の対話に役立つ情報を抽出してください。

    抽出する情報:
    1. ユーザーの好み（新しく判明したもの）
    2. 重要な事実（部署異動、役職変更など）
    3. 対話の要約（1〜2文）

    会話:
    {session.conversation_history}
    """

    extracted = llm.generate(extraction_prompt)

    # ユーザープロファイルを更新
    for preference in extracted.preferences:
        long_term_memory.user_profile.preferences.update(preference)

    for fact in extracted.facts:
        long_term_memory.user_profile.facts.append(fact)

    # 過去のインタラクション記録を追加
    long_term_memory.past_interactions.append({
        "date": session.started_at,
        "summary": extracted.summary,
        "tags": extracted.tags,
        "importance": extracted.importance
    })

    return long_term_memory
```

### 3. メモリ減衰メカニズム

```python
# 擬似コード: メモリの減衰と整理

import math
from datetime import datetime, timedelta

def calculate_memory_score(entry, current_time):
    """エピソードメモリのエントリのスコアを計算する"""

    # 時間減衰（Ebbinghaus の忘却曲線を模倣）
    days_since_creation = (current_time - entry.session_date).days
    time_decay = math.exp(-0.1 * days_since_creation)

    # アクセス頻度による補正（よく参照される記憶は忘れにくい）
    access_boost = math.log(1 + entry.access_count) * 0.2

    # 重要度による補正
    importance_weight = {
        "critical": 2.0,  # 絶対に忘れてはいけない
        "high": 1.5,
        "medium": 1.0,
        "low": 0.5
    }.get(entry.importance, 1.0)

    score = (time_decay + access_boost) * importance_weight
    return score


def cleanup_episodic_memory(episodic_memory, threshold=0.1):
    """スコアが閾値を下回ったエントリをアーカイブする"""

    current_time = datetime.now()
    active_entries = []
    archived_entries = []

    for entry in episodic_memory.entries:
        score = calculate_memory_score(entry, current_time)
        entry.decay_factor = score

        if score >= threshold:
            active_entries.append(entry)
        else:
            archived_entries.append(entry)

    episodic_memory.entries = active_entries

    # アーカイブ前に、重要なパターンをセマンティックメモリに昇格
    patterns = extract_patterns(archived_entries)
    for pattern in patterns:
        semantic_memory.learned_patterns.append(pattern)

    return episodic_memory
```

### 4. 矛盾解決メカニズム

```python
# 擬似コード: メモリの矛盾解決

def resolve_contradiction(existing_memory, new_information):
    """既存メモリと新情報の矛盾を解決する"""

    # 例: 既存「田中さんは営業部」新規「田中さんは企画部に異動」

    # 1. 矛盾を検出
    conflicts = find_conflicts(existing_memory, new_information)

    for conflict in conflicts:
        old_value = conflict.existing
        new_value = conflict.new

        # 2. 情報源の信頼度を比較
        if new_value.source_reliability > old_value.source_reliability:
            resolution = "replace"
        elif new_value.timestamp > old_value.timestamp:
            # 新しい情報を優先（ただしユーザーに確認）
            resolution = "replace_with_confirmation"
        else:
            resolution = "keep_both"

        # 3. 解決策を適用
        if resolution == "replace":
            existing_memory.update(conflict.key, new_value)
            # 変更履歴を記録
            existing_memory.add_change_log({
                "key": conflict.key,
                "old": old_value.value,
                "new": new_value.value,
                "reason": "より信頼度の高い情報源から更新",
                "timestamp": datetime.now()
            })

        elif resolution == "replace_with_confirmation":
            # 次回の対話時にユーザーに確認する
            existing_memory.add_pending_confirmation({
                "question": f"以前は「{old_value.value}」と伺いましたが、"
                           f"「{new_value.value}」に変わりましたか？",
                "key": conflict.key,
                "old": old_value.value,
                "new": new_value.value
            })

    return existing_memory
```

**プロの知見**: メモリ設計で最も難しいのは「何を記憶し、何を忘れるか」の判断です。すべてを記憶するとノイズが増え検索精度が下がり、記憶しなさすぎるとユーザー体験が損なわれます。実務では、まず最小限のメモリ（ユーザープロファイル + 直近5セッションの要約）から始め、ユーザーフィードバックに基づいて段階的にメモリの範囲を拡大するアプローチが効果的です。

## 基本問題

### 問題1：システム構成図の読解

以下のシステム構成図を読み、質問に答えてください。

```
  ユーザー
     │ HTTPS
  ┌──▼──┐
  │ ALB  │
  └──┬──┘
  ┌──▼──┐  ┌──────┐
  │ ECS  │──│ Redis │
  │(API) │  └──────┘
  └──┬──┘
  ┌──▼──┐
  │ RDS  │
  │(MySQL)│
  └─────┘
```

1. ユーザーとALBの間の通信プロトコルは何ですか？
2. ALBの役割は何ですか？
3. Redisはどのような用途で使われていると考えられますか？
4. この構成でECSサーバーが1台停止した場合、サービスは継続できますか？

**期待される出力例：**

```
1. HTTPS（HTTP over TLS）
2. ロードバランサー。リクエストを複数のサーバーに振り分ける
```

<details>
<summary>ヒント</summary>

- ALB = Application Load Balancer（AWSのロードバランサー）
- ECS = Elastic Container Service（コンテナ実行環境）
- RDS = Relational Database Service（マネージドDB）
- Redis = インメモリデータストア

</details>

<details>
<summary>解答例</summary>

```
1. HTTPS（HTTP over TLS/SSL）
   ユーザーとALB間の通信はTLSで暗号化されている。

2. ALBはロードバランサー（負荷分散装置）である。
   複数のECSコンテナにリクエストを振り分け、
   負荷を分散する役割を持つ。また、ヘルスチェックにより
   障害が発生したコンテナへのルーティングを停止する。

3. Redisの用途として以下が考えられる。
   ・セッション管理：ユーザーのログインセッションを保持
   ・キャッシュ：頻繁に参照するデータのキャッシュ
   ・APIレスポンスのキャッシュ：DB負荷の軽減

4. この図だけでは判断できない。
   ECSが複数タスク（コンテナ）で動いていれば、ALBが
   生存しているコンテナにルーティングするためサービスは
   継続できる。ただし、図にはECSのタスク数が1台か
   複数台かが明示されていないため、構成図に
   「ECS × 2台（最小）」等の記載が必要である。

   → 基本設計書では冗長構成を明示することが重要。
```

</details>

### 問題2：技術スタックの選定

以下のプロジェクトに対して、技術スタックを選定し、選定理由を記述してください。

**プロジェクト：** 社内向けタスク管理Webアプリケーション
- 利用者：50名程度
- 機能：タスクのCRUD、担当者割り当て、進捗管理、通知
- 制約：AWSを使用、開発期間3ヶ月、開発者2名

**期待される出力例：**

```
| カテゴリ      | 技術          | 選定理由                   |
|-------------|--------------|---------------------------|
| フロントエンド | React        | コンポーネント設計の柔軟性   |
| ...         | ...          | ...                       |
```

<details>
<summary>ヒント</summary>

以下の観点で選定してください。
- 開発者2名で3ヶ月の工期 → 学習コストが低い技術
- 利用者50名 → 大規模な負荷分散は不要
- 社内向け → 外部公開のセキュリティほどの要件は不要

</details>

<details>
<summary>解答例</summary>

```
| カテゴリ           | 技術              | 選定理由                            |
|-------------------|-------------------|-----------------------------------|
| フロントエンド      | Next.js (React)   | SSRによる高速表示、                  |
|                   |                   | ファイルベースルーティングで開発効率向上 |
| UIライブラリ       | shadcn/ui         | 軽量でカスタマイズ性が高い             |
| バックエンド       | Next.js API Routes | フロントと一体化で開発工数削減          |
| ORM              | Prisma            | 型安全なDB操作、マイグレーション管理    |
| データベース       | PostgreSQL (RDS)   | ACID準拠、JSON型対応                  |
| 認証              | NextAuth.js        | 社内AD連携が容易                      |
| ホスティング       | AWS App Runner     | コンテナの簡易デプロイ、運用負荷低     |
| 通知              | Amazon SES         | メール通知のマネージドサービス          |
| CI/CD            | GitHub Actions     | リポジトリと一体化、無料枠で十分       |

【選定の方針】
・開発者2名×3ヶ月の制約から、フルスタックフレームワーク
 （Next.js）を採用し、フロント/バックの分離を最小化する。
・利用者50名のため、大規模な負荷分散は不要。
  App Runnerで運用負荷を最小化する。
・Prismaの型安全なORM機能により、DBアクセスの
  バグを減らし、少人数での品質確保を図る。

【代替案と却下理由】
・Vue.js + Express：バックエンドを別に作る必要があり、
  工数が増える。
・Firebase：NoSQLのため、タスク間のリレーション管理が複雑。
・Heroku：AWSの制約に合致しない。
```

</details>

### 問題5：排他制御方式の選定

以下の5つの機能について、楽観的排他制御と悲観的排他制御のどちらが適切か選び、理由を述べてください。

1. ECサイトの在庫数の更新
2. ユーザープロフィールの編集
3. 銀行口座の残高更新
4. ブログ記事の下書き保存
5. 予約システムの空席管理

**期待される出力例：**

```
1. 悲観的排他制御 - 在庫数の整合性が最重要で、
   同時に複数の注文が発生する可能性が高い
```

<details>
<summary>ヒント</summary>

選定のポイントは以下の通りです。
- 同時更新の頻度が高いか低いか
- データの整合性がどの程度重要か
- 競合時にリトライが許容されるか

</details>

<details>
<summary>解答例</summary>

```
1. ECサイトの在庫数の更新 → 悲観的排他制御
   理由：人気商品のセール時に同時注文が集中し、
   在庫が0になった後も注文を受け付ける
   「売り越し」が発生してはならない。
   在庫の整合性は業務上最重要であるため、
   更新時にロックをかけて整合性を保証する。

2. ユーザープロフィールの編集 → 楽観的排他制御
   理由：プロフィールを同時に編集するケースは稀。
   仮に競合が発生しても、エラーメッセージで
   再読込を促せば業務上の問題はない。
   ロックによる待ち時間を発生させる必要がない。

3. 銀行口座の残高更新 → 悲観的排他制御
   理由：金額の整合性は絶対に保証する必要があり、
   同一口座への同時振込・引落が発生する可能性がある。
   データベースのトランザクション + 行ロックで
   整合性を保証する。

4. ブログ記事の下書き保存 → 楽観的排他制御
   理由：下書きを同時に編集するケースは稀で、
   仮に競合しても再保存すれば済む。
   ロックをかけると、保存忘れでロックが
   残り続けるリスクがある。

5. 予約システムの空席管理 → 悲観的排他制御
   理由：同一時間帯の予約が集中した場合、
   ダブルブッキング（二重予約）を防ぐ必要がある。
   残り1席の状態で2人が同時に予約操作した場合、
   一方の予約を確実にブロックする必要がある。
```

</details>

## 基本問題

### 問題1：評価基準の適用

以下のプロンプトとその出力を、5つの評価軸（正確性・関連性・完全性・一貫性・有用性）で評価してください。各軸を5段階でスコアリングしてください。

**プロンプト：** 「Pythonでリストの重複を削除する方法を教えてください」

**出力例：**
```
リストの重複を削除するには、set()を使います。

my_list = [1, 2, 2, 3, 3, 4]
unique_list = list(set(my_list))
print(unique_list)  # [1, 2, 3, 4]
```

<details>
<summary>解答例</summary>

```
■ 評価

| 評価軸    | スコア | 理由                                            |
|----------|--------|------------------------------------------------|
| 正確性    | 5/5    | コードは正しく動作する                           |
| 関連性    | 5/5    | 質問に対して的確に回答している                     |
| 完全性    | 3/5    | set()による方法のみ。順序保持の注意点、             |
|          |        | dict.fromkeys()やfor文での方法が欠けている         |
| 一貫性    | 4/5    | フォーマットは適切だが、説明が1行のみで             |
|          |        | やや簡素すぎる                                    |
| 有用性    | 3/5    | 基本的なケースは解決できるが、set()では             |
|          |        | 元の順序が保持されない点の注意がない               |

■ 総合スコア：4.0/5

■ 改善提案：
1. set()では順序が保持されない注意点を追記
2. 順序を保持する方法（dict.fromkeys()）も紹介
3. それぞれのメリット・デメリットを表形式で比較
```
</details>

### 問題2：プロンプトの改善提案

以下のプロンプトの出力に問題があります。問題点を特定し、プロンプトを改善してください。

**プロンプト：** 「データベースについて説明してください」

**出力の問題点：**
- 回答が5,000文字を超える長文になった
- RDBMSとNoSQLを網羅的に解説し、焦点が定まらない
- 初心者には難しい専門用語が説明なく使われている
- 結論や推奨がない

<details>
<summary>解答例</summary>

```
■ 問題点の分析

1. 出力が長すぎる → 文字数制限を指定していない
2. 焦点が定まらない → 対象範囲を限定していない
3. 専門用語が多い → 対象読者（レベル）を指定していない
4. 結論がない → 出力構成を指定していない

■ 改善後のプロンプト

「あなたはプログラミングスクールの講師です。
Webアプリ開発初心者に向けて、リレーショナルデータベース
（RDB）の基本概念を説明してください。

含める内容：
1. データベースとは何か（身近な例を使って）
2. テーブル、行、列の概念
3. SQLの基本（SELECT、INSERT、UPDATE、DELETE）
4. Webアプリでどう使われるか

制約：
- 500文字以内で回答する
- 専門用語を使う場合は必ず説明を添える
- NoSQLには触れない（別の章で扱うため）
- 最後に「まとめ」を3行以内で記載する

対象読者：HTMLとCSSは書けるが、バックエンドは初めての学習者」

■ 改善のポイント
- 対象読者を明示（初心者、HTML/CSS経験あり）
- 範囲を限定（RDBのみ、NoSQLは除外）
- 文字数制限を設定（500文字以内）
- 出力構成を指定（4セクション + まとめ）
- ペルソナ設定（プログラミングスクール講師）
```
</details>

### 問題3：A/Bテストの設計

コードレビュー用のプロンプトをA/Bテストしたいと考えています。以下を設計してください。

1. テストする仮説
2. プロンプトAとプロンプトB
3. テスト用の入力（コードサンプル）3件
4. 評価基準

<details>
<summary>解答例</summary>

```
■ 1. テストする仮説
「コードレビューのプロンプトにチェックリストを追加すると、
指摘の網羅性が向上するのではないか」

■ 2. プロンプト

プロンプトA（チェックリストなし）：
「以下のPythonコードをレビューしてください。
問題点を重要度（高/中/低）とともに指摘し、
改善案を提示してください。」

プロンプトB（チェックリストあり）：
「以下のPythonコードをレビューしてください。
以下のチェックリストの各項目について確認し、
問題があれば重要度（高/中/低）とともに指摘してください。

チェックリスト：
□ バグや論理エラー
□ セキュリティの脆弱性（SQL インジェクション、XSS等）
□ エラーハンドリングの漏れ
□ 入力バリデーション
□ パフォーマンス問題
□ 変数名・関数名の適切さ
□ コメントの適切さ
□ テストの有無

問題がない項目は「問題なし」と記載してください。」

■ 3. テスト用コードサンプル

サンプル1（セキュリティ問題あり）：
```python
def get_user(user_id):
    query = f"SELECT * FROM users WHERE id = {user_id}"
    return db.execute(query)
```

サンプル2（エラーハンドリング不足）：
```python
def read_config(path):
    with open(path) as f:
        return json.load(f)
```

サンプル3（複合的な問題）：
```python
def process(d):
    x = d['value'] * 1.1
    if x > 100:
        db.execute(f"UPDATE t SET v={x} WHERE id={d['id']}")
    return x
```

■ 4. 評価基準

| 基準 | 評価方法 | 重み |
|------|---------|------|
| 指摘の正確性 | 実際のバグを正しく指摘できたか | 30% |
| 網羅性 | 既知の問題をすべて検出できたか | 30% |
| 改善案の品質 | 修正コードが正しく実用的か | 20% |
| 説明の分かりやすさ | ジュニアエンジニアが理解できるか | 10% |
| フォーマット | 読みやすく整理されているか | 10% |

各コードサンプルの既知の問題リスト（正解データ）：
サンプル1：SQLインジェクション、型チェックなし
サンプル2：FileNotFoundError未処理、JSONDecodeError未処理
サンプル3：変数名不明確、SQLインジェクション、マジックナンバー(1.1)、
           エラーハンドリングなし、KeyError未処理
```
</details>

### 問題4：イテレーション改善の実践

以下のプロンプトを3回のイテレーションで改善してください。各イテレーションで1つの問題を解決してください。

**初版プロンプト：** 「エラーメッセージを説明してください」

想定される問題点：
- 回答が漠然としている
- 解決策が含まれない
- エラーの深刻度が判断できない

<details>
<summary>解答例</summary>

```
■ イテレーション1：出力構成の追加
問題：回答が漠然としている

改善後：
「以下のエラーメッセージについて、以下の構成で説明してください。

1. エラーの意味（1〜2文で簡潔に）
2. 発生原因（箇条書きで考えられる原因を列挙）
3. 解決方法（最も一般的な解決手順）

エラーメッセージ：
"""
（エラーメッセージ）
"""」

テスト結果：解決策が含まれるようになった。
残る問題：深刻度が分からない、コード例がない。

───────────────────────────────────────

■ イテレーション2：深刻度と具体例の追加
問題：深刻度が判断できない、具体例がない

改善後：
「以下のエラーメッセージについて、以下の構成で説明してください。

1. 深刻度：🔴 重大 / 🟡 注意 / 🟢 軽微
2. エラーの意味（1〜2文で簡潔に）
3. 発生原因（箇条書きで考えられる原因を列挙）
4. 解決方法（手順付き）
5. 修正前/修正後のコード例

エラーメッセージ：
"""
（エラーメッセージ）
"""」

テスト結果：深刻度とコード例が含まれるようになった。
残る問題：文脈が不足すると的外れな回答になることがある。

───────────────────────────────────────

■ イテレーション3：文脈の補完を指示
問題：文脈が不足すると的外れな回答になる

改善後：
「あなたはデバッグ支援のエキスパートです。
以下のエラーメッセージを分析してください。

コンテキスト（分かる範囲で）：
- 言語/フレームワーク：
- 発生箇所（ファイル名等）：
- 直前の操作：

エラーメッセージ：
"""
（エラーメッセージ）
"""

以下の構成で回答してください：

1. 深刻度：🔴 重大 / 🟡 注意 / 🟢 軽微
2. エラーの意味（1〜2文）
3. 発生原因（箇条書き、可能性の高い順）
4. 解決手順（番号付き）
5. 修正前/修正後のコード例
6. 再発防止策（1〜2つ）

コンテキストが不足している場合は、
追加で確認すべき情報を質問してください。」

テスト結果：文脈が不足する場合は追加質問するように。
精度が大幅に向上。
```
</details>

## チャレンジ問題

### 問題5：LLM-as-Judgeの設計

あなたのチームでは、FAQチャットボットの回答品質を自動評価したいと考えています。LLM-as-Judge（LLMを評価者として使う）の仕組みを設計してください。

設計に含める内容：
1. 評価用プロンプト
2. 評価基準（スコアリングルーブリック）
3. テストデータセットの設計（5件のQ&Aペア）
4. 自動評価の限界と人間のレビューが必要なケース

<details>
<summary>解答例</summary>

```
■ 1. 評価用プロンプト

「あなたはFAQチャットボットの品質評価担当です。
以下の質問、正解（参考回答）、チャットボットの回答を比較し、
チャットボットの回答を評価してください。

## 質問
{question}

## 正解（参考回答）
{reference_answer}

## チャットボットの回答
{bot_answer}

## 評価基準
以下の4項目をそれぞれ1〜5で評価してください。

1. 正確性（1-5）：
   5=完全に正確 4=概ね正確 3=一部不正確
   2=重要な誤りあり 1=完全に不正確

2. 完全性（1-5）：
   5=必要な情報がすべて含まれている
   3=主要な情報は含まれているが一部欠落
   1=重要な情報が大幅に欠落

3. 簡潔性（1-5）：
   5=適切な長さで無駄がない
   3=やや冗長または簡素すぎる
   1=極端に冗長または情報不足

4. 有用性（1-5）：
   5=ユーザーの問題を完全に解決できる
   3=部分的に解決に役立つ
   1=まったく役に立たない

## 出力形式
- 正確性：X/5 - （理由）
- 完全性：X/5 - （理由）
- 簡潔性：X/5 - （理由）
- 有用性：X/5 - （理由）
- 総合スコア：X/5
- 改善提案：（あれば）」

■ 2. テストデータセット（5件）

| # | 質問 | カテゴリ | 難易度 |
|---|------|---------|-------|
| 1 | 「有給休暇の申請方法は？」 | 典型的 | 易 |
| 2 | 「育児休暇中に副業はできますか？」 | 境界的 | 中 |
| 3 | 「上司のパワハラを相談したい」 | 敏感 | 難 |
| 4 | 「明日の天気は？」 | 範囲外 | 易 |
| 5 | 「休暇の件」 | 曖昧 | 中 |

■ 3. 自動評価の限界

自動評価が有効なケース：
  - 事実ベースの正確性チェック
  - 出力フォーマットの準拠確認
  - 回答の長さや構成の評価

人間のレビューが必要なケース：
  - 敏感なトピック（ハラスメント、メンタルヘルス等）
  - 曖昧な質問への対応の適切さ
  - トーンや共感性の評価
  - 法的・倫理的な判断が必要な回答
  - LLM-as-Judgeのスコアが3/5未満の回答（エスカレーション）
```
</details>

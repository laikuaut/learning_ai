# 第8章 ITサービスマネジメント

> ITIL（Information Technology Infrastructure Library）を中心に、SLA、可用性管理、キャパシティ管理、変更管理、インシデント管理など、ITサービスの運用管理に必要な知識を体系的に学ぶ。

---

## 8.1 ITILの概要

### 8.1.1 ITILとは

**ITIL（IT Infrastructure Library）** は、ITサービスマネジメントのベストプラクティス（成功事例）を体系化したフレームワークです。

### 8.1.2 ITILのサービスライフサイクル（ITIL v3/2011）

```
┌────────────────────────────────────────┐
│            継続的サービス改善             │
│   ┌────────────────────────────────┐   │
│   │                                │   │
│   │  サービス ──→ サービス ──→ サービス │   │
│   │  ストラテジ    デザイン    トランジション│  │
│   │  (戦略)       (設計)      (移行)    │   │
│   │         ╲      │      ╱           │   │
│   │          ╲     ▼     ╱            │   │
│   │       サービスオペレーション          │   │
│   │          (運用)                    │   │
│   └────────────────────────────────┘   │
└────────────────────────────────────────┘
```

| ライフサイクル段階 | 目的 | 主なプロセス |
|------------------|------|-------------|
| サービスストラテジ | 事業目標に合致するIT戦略 | 財務管理、需要管理 |
| サービスデザイン | サービスの設計 | SLA管理、可用性管理、キャパシティ管理 |
| サービストランジション | 本番環境への移行 | 変更管理、リリース管理、構成管理 |
| サービスオペレーション | 日常的な運用 | インシデント管理、問題管理、イベント管理 |
| 継続的サービス改善 | サービスの改善 | 改善の7ステッププロセス |

### 8.1.3 ITIL 4の概要

ITIL 4は2019年に発表された最新版で、アジャイルやDevOpsの概念を取り込んでいます。

**ITIL 4の4つの側面:**
1. **組織と人材** - スキル、文化、役割
2. **情報と技術** - 知識、ツール、技術
3. **パートナとサプライヤ** - 外部関係者との協力
4. **バリューストリームとプロセス** - 価値提供のフロー

**ポイントまとめ:**
- ITIL v3の5つのライフサイクル段階と主なプロセスを覚える
- 午前試験では各プロセスがどの段階に属するかを問われる
- ITIL 4のキーワード（価値共創、サービスバリューシステム）も理解する

## 8.2 SLA（Service Level Agreement）

### 8.2.1 SLAとは

**SLA（サービスレベル合意書 / Service Level Agreement）** は、ITサービスの提供者と利用者の間で合意するサービス品質の目標値と条件を文書化したものです。

### 8.2.2 SLAに含まれる項目

| 項目 | 内容例 |
|------|--------|
| サービス稼働率 | 99.9%以上（月間ダウンタイム43分以内） |
| 応答時間 | Webページの表示3秒以内 |
| インシデント対応時間 | 重大障害は30分以内に一次対応 |
| バックアップ | 日次フルバックアップ、RPO=4時間 |
| 報告 | 月次のサービスレベルレポート提出 |
| ペナルティ | SLA未達時のサービス料金減額 |

### 8.2.3 SLM（Service Level Management）プロセス

```
SLA策定 → SLA監視 → SLAレビュー → SLA改善
   │          │          │          │
   ▼          ▼          ▼          ▼
要件の合意  KPI測定   達成度評価  改善計画

関連文書:
  SLR（Service Level Requirement）: サービスレベル要件
  SLA: 合意したサービスレベル
  OLA（Operational Level Agreement）: 内部運用チーム間の合意
  UC（Underpinning Contract）: 外部サプライヤとの契約
```

**ポイントまとめ:**
- SLAは「合意」であり、一方的な提示ではない
- SLA/OLA/UCの違いを区別する（SLA:顧客間、OLA:内部間、UC:外部間）
- RPO/RTOなどの目標値を具体的に設定できることが重要

## 8.3 可用性管理

### 8.3.1 可用性の指標

```
可用性(%) = MTBF / (MTBF + MTTR) × 100

  MTBF: 平均故障間隔（Mean Time Between Failures）
  MTTR: 平均修復時間（Mean Time To Repair）

  稼働───障害───稼働───障害───稼働
  ←MTBF→←MTTR→←MTBF→←MTTR→
```

### 8.3.2 可用性に関する目標

| 指標 | 英語 | 意味 |
|------|------|------|
| RTO | Recovery Time Objective | 目標復旧時間（いつまでに復旧するか） |
| RPO | Recovery Point Objective | 目標復旧地点（どの時点まで戻せるか） |
| MTPD | Maximum Tolerable Period of Disruption | 最大許容停止時間 |

```
                    RPO          RTO
                     │            │
    ←──データ復旧──→│←─復旧作業─→│
                     │            │
  ──正常運用──障害発生──復旧作業──サービス再開──
              ↑                   ↑
         最後のバックアップ      復旧完了
```

### 8.3.3 可用性を高める技術

| 技術 | 方法 |
|------|------|
| 冗長化 | HA構成、クラスタリング |
| 災害対策（DR） | 遠隔地バックアップサイト |
| 自動フェイルオーバー | 障害時に自動で待機系に切替 |
| 定期的なバックアップ | RPO目標に基づくバックアップ間隔 |

**ポイントまとめ:**
- MTBF/MTTR/可用性の計算は必須
- RTO/RPOの違いを明確に区別する（時間 vs データ地点）
- 午後試験ではRTO/RPO目標に基づくDR設計が出題される

## 8.4 キャパシティ管理

### 8.4.1 キャパシティ管理の3つのサブプロセス

| サブプロセス | 対象 | 内容 |
|------------|------|------|
| ビジネスキャパシティ管理 | 将来の事業ニーズ | ビジネス計画からIT需要を予測 |
| サービスキャパシティ管理 | ITサービス | サービスのリソース使用状況を監視・分析 |
| コンポーネントキャパシティ管理 | ITインフラ | CPU、メモリ、ディスク等の使用率を管理 |

### 8.4.2 キャパシティ計画

```
需要予測 → リソース算出 → 調達計画 → 実装 → 監視
                                           │
                                      フィードバック
```

**需要予測の手法:**
- トレンド分析: 過去のデータから将来を予測
- モデリング: シミュレーションによる予測
- ベースラインの設定: 現在の使用状況を基準値として記録

**ポイントまとめ:**
- キャパシティ管理の3つのサブプロセスを区別する
- ビジネス→サービス→コンポーネントの階層を理解する

## 8.5 変更管理

### 8.5.1 変更管理の目的

ITサービスやインフラへの変更を**計画的かつ安全に実施**するためのプロセスです。未計画の変更による障害を防止します。

### 8.5.2 変更管理のプロセス

```
RFC(変更要求)作成
  │
  ▼
変更の記録と分類
  │
  ▼
変更の評価（影響度、リスク、リソース）
  │
  ▼
CAB(変更諮問委員会)での承認
  │
  ▼
変更の実施
  │
  ▼
変更のレビュー（PIR: 実施後レビュー）
```

### 8.5.3 変更の分類

| 分類 | 説明 | 承認プロセス |
|------|------|------------|
| 標準変更 | リスクが低い定型的な変更 | 事前承認済み（都度承認不要） |
| 通常変更 | 一般的な変更 | CABでの承認が必要 |
| 緊急変更 | 重大インシデントへの対応 | ECAB（緊急CAB）で迅速に承認 |

### 8.5.4 構成管理（CMDB）

**CMDB（Configuration Management Database: 構成管理データベース）** は、ITサービスを構成する全てのCI（構成アイテム）とその関連を管理するデータベースです。

```
CMDB
┌─────────────────────────────────┐
│ CI: Webサーバ01                  │
│  ├── OS: Linux                  │
│  ├── ミドルウェア: Apache 2.4    │
│  ├── アプリ: 受注システム v2.1   │
│  ├── 関連CI: DBサーバ01          │
│  └── SLA: 99.9%稼働率           │
│                                  │
│ CI: DBサーバ01                   │
│  ├── OS: Linux                  │
│  ├── DB: PostgreSQL 15           │
│  └── 関連CI: Webサーバ01         │
└─────────────────────────────────┘
```

**ポイントまとめ:**
- 変更管理のプロセス（RFC→評価→CAB承認→実施→PIR）を覚える
- 標準/通常/緊急変更の違いと承認プロセスを区別する
- CMDBはCI（構成アイテム）とその関連を管理する

## 8.6 インシデント管理と問題管理

### 8.6.1 インシデント管理

**インシデント** = サービスの中断または品質低下を引き起こす事象

```
インシデント検出
  │
  ▼
記録・分類
  │
  ▼
優先度の決定（影響度 × 緊急度）
  │
  ▼
初期診断
  │
  ├── 解決可能 → 解決・復旧 → クローズ
  │
  └── 解決不可 → エスカレーション
                   ├── 機能的エスカレーション（専門チームへ）
                   └── 階層的エスカレーション（管理層へ）
```

**優先度マトリクス:**
```
          緊急度
          高    中    低
  高  │ P1  │ P2  │ P3  │
影 ───├─────┼─────┼─────┤
響 中  │ P2  │ P3  │ P4  │
度 ───├─────┼─────┼─────┤
  低  │ P3  │ P4  │ P5  │
      └─────┴─────┴─────┘

P1: 最優先（即時対応）
P5: 最低優先（次回リリースで対応等）
```

### 8.6.2 問題管理

**問題** = インシデントの根本原因（1つ以上のインシデントの背後にある原因）

```
インシデント管理           問題管理
  │                        │
  │ インシデント対応         │ 根本原因の特定と恒久対策
  │ (迅速な復旧が目的)      │ (再発防止が目的)
  │                        │
  ▼                        ▼
  ワークアラウンド ←───── 既知のエラー(Known Error)
  (暫定対処)               (原因が判明した問題)
```

**インシデント管理と問題管理の違い:**

| 項目 | インシデント管理 | 問題管理 |
|------|--------------|---------|
| 目的 | サービスの迅速な復旧 | 根本原因の特定と再発防止 |
| 対応速度 | 即座に対応 | じっくり分析 |
| 成果物 | ワークアラウンド | 恒久対策、既知のエラーDB |
| 視点 | 今起きている障害の解消 | なぜ障害が起きたかの究明 |

**ポイントまとめ:**
- インシデント管理は「復旧」、問題管理は「根本原因」が目的
- 優先度 = 影響度 × 緊急度 の考え方を理解する
- エスカレーション（機能的/階層的）の違いを区別する

## 8.7 その他の重要プロセス

### 8.7.1 リリース管理

変更をテスト環境から本番環境に安全に移行するプロセスです。

| リリース種別 | 説明 |
|------------|------|
| ビッグバンリリース | 全ユーザに一斉展開 |
| 段階的リリース | ユーザグループごとに順次展開 |
| ブルーグリーンデプロイメント | 新旧環境を切替えて展開 |
| カナリアリリース | 一部ユーザのみに先行展開 |

### 8.7.2 イベント管理

ITインフラやサービスの状態変化（イベント）を監視・検出するプロセスです。

| イベント種別 | 説明 | 対応 |
|------------|------|------|
| 情報 | 正常動作の確認 | ログ記録のみ |
| 警告 | 閾値に近づいている | 予防的対応 |
| 例外 | 異常が発生 | インシデント管理へ連携 |

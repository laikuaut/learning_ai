# 第2章：認証と認可 演習問題

## 基本問題

### 問題2-1：認証と認可の違い
認証（Authentication）と認可（Authorization）の違いを、具体例を交えて説明してください。

<details>
<summary>解答</summary>

- **認証（Authentication）**：「あなたは誰ですか？」 → 本人確認のプロセス
  例：ログイン画面でユーザIDとパスワードを入力し、本人であることを証明する

- **認可（Authorization）**：「あなたは何をしてよいですか？」 → 権限確認のプロセス
  例：一般ユーザは自分のプロフィールのみ閲覧可能、管理者は全ユーザの情報を閲覧・編集可能

認証が先に行われ、その後で認可が行われます。認証が成功しても、必要な権限がなければリソースにアクセスできません。

</details>

### 問題2-2：認証の3要素
認証の3つの要素を挙げ、それぞれの具体例を2つずつ示してください。

<details>
<summary>解答</summary>

| 認証要素 | 説明 | 具体例 |
|---------|------|--------|
| **知識要素（Something you know）** | 本人だけが知っている情報 | パスワード、PIN、秘密の質問 |
| **所持要素（Something you have）** | 本人だけが持っている物 | スマートフォン（SMS認証）、ICカード、ハードウェアトークン |
| **生体要素（Something you are）** | 本人の身体的特徴 | 指紋認証、顔認証、虹彩認証、静脈認証 |

</details>

### 問題2-3：多要素認証（MFA）
以下のうち、多要素認証（MFA）に該当するものを全て選んでください。

(a) パスワード + SMS認証コード
(b) パスワード + 秘密の質問
(c) 指紋認証 + ICカード
(d) パスワード + 顔認証
(e) PIN + パスワード

<details>
<summary>解答</summary>

MFAに該当するのは **(a)、(c)、(d)** です。

- (a) パスワード（知識）+ SMS認証コード（所持）→ **2要素** ○
- (b) パスワード（知識）+ 秘密の質問（知識）→ **同一要素の2段階**（MFAではない）×
- (c) 指紋（生体）+ ICカード（所持）→ **2要素** ○
- (d) パスワード（知識）+ 顔認証（生体）→ **2要素** ○
- (e) PIN（知識）+ パスワード（知識）→ **同一要素の2段階**（MFAではない）×

MFAは「異なる種類の要素」を2つ以上組み合わせることがポイントです。同じ種類の要素を複数使っても多要素にはなりません。

</details>

### 問題2-4：パスワードの安全性
安全なパスワード管理のベストプラクティスを5つ挙げてください。

<details>
<summary>解答</summary>

1. **十分な長さ**：12文字以上（推奨は16文字以上）
2. **複雑性**：大文字・小文字・数字・記号を組み合わせる
3. **パスワードの使い回し禁止**：サービスごとに異なるパスワードを使用
4. **パスワードマネージャの利用**：ランダムで強力なパスワードを自動生成・管理
5. **安全な保存**：サーバ側ではbcrypt等のハッシュ関数（ソルト付き）で保存し、平文やMD5/SHA-1での保存は行わない

</details>

## 応用問題

### 問題2-5：OAuth 2.0
OAuth 2.0 の認可コードフロー（Authorization Code Flow）の流れを、4つの登場人物（リソースオーナー、クライアント、認可サーバ、リソースサーバ）を使って説明してください。

<details>
<summary>解答</summary>

```
① リソースオーナー（ユーザ）がクライアント（アプリ）にアクセス
② クライアントが認可サーバにリダイレクト（認可リクエスト）
③ リソースオーナーが認可サーバで認証し、権限を許可
④ 認可サーバがクライアントに認可コードを発行（リダイレクト）
⑤ クライアントが認可コードを使って認可サーバにアクセストークンを要求
⑥ 認可サーバがアクセストークンを発行
⑦ クライアントがアクセストークンを使ってリソースサーバにアクセス
⑧ リソースサーバがリソースを返却
```

ポイント：OAuth 2.0 は「認可」のプロトコルであり、「認証」のプロトコルではありません。認証が必要な場合はOAuth 2.0 の上にOpenID Connect を組み合わせて使用します。

</details>

### 問題2-6：セッション管理の脅威
以下のセッション管理に関する攻撃を説明し、対策を1つずつ挙げてください。

1. セッションハイジャック
2. セッションフィクセーション

<details>
<summary>解答</summary>

**1. セッションハイジャック**
- **攻撃**：正規ユーザのセッションID を盗み、そのセッションを乗っ取る攻撃。XSSやネットワーク盗聴でセッションIDを取得する
- **対策**：CookieにHttpOnly属性とSecure属性を設定する。通信をHTTPSで暗号化する。セッションIDを十分にランダムで推測困難にする

**2. セッションフィクセーション**
- **攻撃**：攻撃者が事前に用意したセッションIDを被害者に使わせ、被害者がログインした後にそのセッションIDでアクセスする攻撃
- **対策**：ログイン成功時にセッションIDを再生成（新しいIDに変更）する。ログイン前のセッションIDを引き継がない

</details>

## チャレンジ問題

### 問題2-7：JWTの構造
JWT（JSON Web Token）の3つの構成部分を説明し、JWTを使用する際のセキュリティ上の注意点を3つ挙げてください。

<details>
<summary>解答</summary>

**JWTの3つの構成部分：**

1. **ヘッダ（Header）**：トークンの種類（JWT）と使用する署名アルゴリズム（HS256、RS256等）を含む
2. **ペイロード（Payload）**：クレーム（ユーザ情報や有効期限等）を含む。Base64URLエンコードされているが暗号化されていない
3. **署名（Signature）**：ヘッダとペイロードを秘密鍵で署名したもの。改ざん検知に使用

**セキュリティ上の注意点：**

1. **ペイロードに機密情報を含めない**：Base64URLエンコードは暗号化ではなく、誰でもデコードして内容を読める。パスワードや個人情報を含めてはいけない
2. **alg: "none" を許可しない**：algヘッダをnoneにして署名検証をバイパスする攻撃がある。サーバ側で使用するアルゴリズムを厳格に指定する
3. **適切な有効期限（exp）を設定する**：JWTはサーバ側で無効化が難しいため、有効期限を短く設定し、リフレッシュトークンで更新する仕組みにする

</details>

### 問題2-8：アクセス制御モデル
以下のアクセス制御モデルを説明し、それぞれの適用場面を述べてください。

1. DAC（任意アクセス制御）
2. MAC（強制アクセス制御）
3. RBAC（ロールベースアクセス制御）

<details>
<summary>解答</summary>

1. **DAC（Discretionary Access Control / 任意アクセス制御）**
   - リソースの所有者が他のユーザへのアクセス権を設定する方式
   - 適用場面：ファイルシステム（Linux/Windowsのファイル権限）、Googleドライブ等の共有設定

2. **MAC（Mandatory Access Control / 強制アクセス制御）**
   - システム管理者がセキュリティラベル（機密レベル等）に基づいてアクセスを制御する方式。ユーザが権限を変更することはできない
   - 適用場面：軍事・政府機関のシステム、SELinux

3. **RBAC（Role-Based Access Control / ロールベースアクセス制御）**
   - ロール（役割）にアクセス権を割り当て、ユーザにロールを付与する方式。ユーザの追加・異動時にロールを変更するだけで権限を管理できる
   - 適用場面：企業の業務システム（一般ユーザ、管理者、部門長等のロール）、AWS IAM

</details>

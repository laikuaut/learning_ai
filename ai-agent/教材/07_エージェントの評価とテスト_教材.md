## 7.1 エージェント評価の難しさ

### 従来のソフトウェアテストとの違い

従来のソフトウェアは「同じ入力には同じ出力」が期待されます。しかし、AIエージェントには以下の3つの本質的な難しさがあります。

```
┌─────────────────────────────────────────────────────┐
│          AIエージェント評価の3大課題                  │
├─────────────────────────────────────────────────────┤
│                                                     │
│  1. 非決定性（Non-determinism）                      │
│     同じプロンプトでも毎回異なる出力が返る            │
│                                                     │
│  2. 多段階タスク（Multi-step Tasks）                 │
│     中間ステップの正しさをどう測るか                  │
│                                                     │
│  3. 副作用（Side Effects）                           │
│     外部APIの呼び出し、ファイル操作など               │
│     元に戻せない操作をどうテストするか                │
│                                                     │
└─────────────────────────────────────────────────────┘
```

### 非決定性への対処

LLM（Large Language Model）は temperature パラメータにより出力の多様性を制御できますが、temperature=0 でも完全な再現性は保証されません。

```python
# 非決定性を考慮した評価の基本パターン
import statistics

def evaluate_with_retries(agent, test_case, n_runs=5):
    """同じテストケースをN回実行し、統計的に評価します"""
    results = []
    for i in range(n_runs):
        result = agent.run(test_case["input"])
        score = score_result(result, test_case["expected"])
        results.append(score)

    return {
        "mean": statistics.mean(results),
        "median": statistics.median(results),
        "stdev": statistics.stdev(results) if len(results) > 1 else 0,
        "min": min(results),
        "max": max(results),
        "pass_rate": sum(1 for r in results if r >= 0.8) / len(results),
    }
```

### 多段階タスクの評価

エージェントが5ステップのタスクを実行する場合、最終結果だけでなく各ステップの妥当性も評価する必要があります。

```
タスク: 「売上データを分析してレポートを作成せよ」

ステップ1: データ読み込み  → 正しいファイルを選んだか？
ステップ2: 前処理          → 欠損値を適切に処理したか？
ステップ3: 分析            → 適切な手法を選んだか？
ステップ4: 可視化          → グラフは正しいか？
ステップ5: レポート作成    → 結論は妥当か？

→ 最終結果が正しくても、ステップ3で不適切な手法を使っていれば
  別のデータでは失敗する可能性がある
```

### 副作用の問題

エージェントがメール送信やファイル削除などの副作用を伴う操作を行う場合、テスト環境で安全に検証する仕組みが必要です。

```python
# モック（Mock）を使った副作用のテスト
from unittest.mock import MagicMock

class MockEmailTool:
    """メール送信ツールのモック"""
    def __init__(self):
        self.sent_emails = []

    def send(self, to, subject, body):
        self.sent_emails.append({
            "to": to, "subject": subject, "body": body
        })
        return {"status": "sent", "message_id": "mock-123"}

# テストでモックを注入する
mock_email = MockEmailTool()

# 副作用を検証
assert len(mock_email.sent_emails) == 1
assert "田中" in mock_email.sent_emails[0]["to"]
```

> **ポイントまとめ**
> - AIエージェントの評価は非決定性・多段階性・副作用の3つが課題
> - 複数回実行して統計的に評価する手法が基本
> - 副作用はモックを使って安全にテストする

## 7.2 評価指標の設計

### 主要な評価指標

エージェントの品質を多角的に測るために、以下の指標を組み合わせて使います。

| 指標 | 説明 | 計算方法 | 目安 |
|------|------|----------|------|
| タスク成功率（Success Rate） | タスクが正しく完了した割合 | 成功数 / 総試行数 | 80%以上 |
| ステップ効率（Step Efficiency） | 最適ステップ数との比率 | 最適ステップ数 / 実際のステップ数 | 0.7以上 |
| コスト効率（Cost Efficiency） | 1タスクあたりのAPI費用 | 総トークン数 × 単価 | 用途依存 |
| 品質スコア（Quality Score） | 出力の品質を0-1で評価 | LLM-as-Judge等で採点 | 0.8以上 |
| レイテンシ（Latency） | 応答にかかる時間 | 開始〜完了の経過時間 | 用途依存 |
| エラー率（Error Rate） | エラーで終了した割合 | エラー数 / 総試行数 | 5%以下 |

### 評価指標の実装例

```python
import time
from dataclasses import dataclass, field
from typing import Optional

@dataclass
class EvaluationResult:
    """エージェント実行1回分の評価結果"""
    task_id: str
    success: bool
    steps_taken: int
    optimal_steps: int
    total_tokens: int
    cost_usd: float
    latency_seconds: float
    quality_score: float  # 0.0 ~ 1.0
    error_message: Optional[str] = None

    @property
    def step_efficiency(self) -> float:
        if self.steps_taken == 0:
            return 0.0
        return min(self.optimal_steps / self.steps_taken, 1.0)


@dataclass
class EvaluationSuite:
    """評価スイート：複数テストケースの集計"""
    results: list = field(default_factory=list)

    def add(self, result: EvaluationResult):
        self.results.append(result)

    def summary(self) -> dict:
        n = len(self.results)
        if n == 0:
            return {}
        return {
            "total_cases": n,
            "success_rate": sum(r.success for r in self.results) / n,
            "avg_step_efficiency": sum(r.step_efficiency for r in self.results) / n,
            "avg_cost_usd": sum(r.cost_usd for r in self.results) / n,
            "avg_latency_sec": sum(r.latency_seconds for r in self.results) / n,
            "avg_quality": sum(r.quality_score for r in self.results) / n,
            "error_rate": sum(1 for r in self.results if r.error_message) / n,
        }
```

### LLM-as-Judge による品質評価

人手で全出力を評価するのは現実的ではありません。LLM自身を評価者（Judge）として使う手法が広く用いられています。

```python
JUDGE_PROMPT = """あなたはAIエージェントの出力を評価する審査員です。
以下の基準で0.0〜1.0のスコアをつけてください。

【評価基準】
- 正確性（0.3）: 事実に基づいているか
- 完全性（0.3）: 必要な情報が網羅されているか
- 有用性（0.2）: ユーザーの意図に沿っているか
- 簡潔性（0.2）: 不要な情報が含まれていないか

【タスク】
{task_description}

【エージェントの出力】
{agent_output}

【期待される出力の要点】
{expected_points}

JSON形式で回答してください:
{{"accuracy": 0.0-1.0, "completeness": 0.0-1.0, "usefulness": 0.0-1.0, "conciseness": 0.0-1.0, "total": 0.0-1.0, "reasoning": "..."}}
"""
```

> **よくある間違い**
> - 成功率だけで評価してしまう → コスト度外視で100回リトライすれば成功率は上がるが実用的ではありません
> - LLM-as-Judge の結果を無条件に信頼する → 判定のブレを確認するため、複数回評価して一致率を測りましょう

> **ポイントまとめ**
> - 評価指標は複数組み合わせて多角的に測定する
> - ステップ効率とコスト効率は実運用で特に重要
> - LLM-as-Judge は便利だが、信頼性の検証も忘れずに

## 7.3 テスト手法

### テストピラミッドのエージェント版

従来のテストピラミッドをAIエージェント向けに拡張すると、以下のようになります。

```
          ╱╲
         ╱  ╲        E2Eテスト
        ╱    ╲       (少数・高コスト)
       ╱──────╲      エージェント全体を本番に近い環境でテスト
      ╱        ╲
     ╱  統合     ╲    統合テスト
    ╱  テスト     ╲   (中程度)
   ╱──────────────╲   エージェントループ全体の動作確認
  ╱                ╲
 ╱ ユニットテスト   ╲  ユニットテスト
╱──────────────────────╲ (多数・低コスト)
                         個別ツール・パーサーの動作確認
```

### ユニットテスト：個別ツールのテスト

各ツール関数が正しく動作するかを独立してテストします。LLMの呼び出しは不要なため、高速かつ低コストです。

```python
import pytest

# --- テスト対象のツール ---
def search_faq(query: str, faq_db: list[dict]) -> list[dict]:
    """FAQデータベースから関連する回答を検索します"""
    results = []
    for item in faq_db:
        if any(keyword in item["question"] for keyword in query.split()):
            results.append(item)
    return sorted(results, key=lambda x: x.get("priority", 0), reverse=True)

# --- ユニットテスト ---
class TestSearchFAQ:
    @pytest.fixture
    def sample_faq(self):
        return [
            {"question": "パスワードのリセット方法", "answer": "...", "priority": 10},
            {"question": "料金プランの変更方法", "answer": "...", "priority": 5},
            {"question": "パスワードの要件", "answer": "...", "priority": 8},
        ]

    def test_keyword_match(self, sample_faq):
        results = search_faq("パスワード", sample_faq)
        assert len(results) == 2
        # 優先度順にソートされているか
        assert results[0]["priority"] >= results[1]["priority"]

    def test_no_match(self, sample_faq):
        results = search_faq("存在しないキーワード", sample_faq)
        assert len(results) == 0

    def test_empty_query(self, sample_faq):
        results = search_faq("", sample_faq)
        assert len(results) == 0
```

### 統合テスト：エージェントループのテスト

エージェントのループ全体（LLM呼び出し → ツール選択 → 実行 → 結果統合）をテストします。LLMの応答をモックにすることで、安定したテストが可能です。

```python
def test_agent_loop_with_mock_llm():
    """エージェントループの統合テスト（LLMをモック化）"""

    # LLMの応答を事前に定義
    mock_responses = [
        {  # ステップ1: ツール呼び出しを決定
            "tool_call": {"name": "search_faq", "args": {"query": "返品"}},
        },
        {  # ステップ2: 検索結果をもとに回答を生成
            "content": "返品は購入から30日以内であれば可能です。",
        },
    ]

    agent = Agent(
        llm=MockLLM(mock_responses),
        tools={"search_faq": search_faq},
        max_steps=5,
    )

    result = agent.run("返品したいのですが")

    # 検証
    assert result.success is True
    assert result.steps_taken == 2
    assert "返品" in result.output
    assert "30日" in result.output
```

### E2Eテスト：本番環境に近い形でのテスト

実際のLLMを使い、本番に近い環境でエージェント全体を検証します。コストがかかるため、重要なシナリオに絞って実施します。

```python
# E2Eテスト（実際のLLMを使用）
@pytest.mark.e2e
@pytest.mark.slow
def test_customer_support_e2e():
    """カスタマーサポートエージェントのE2Eテスト"""
    agent = CustomerSupportAgent(
        model="gpt-4o",
        tools=[search_faq, create_ticket, send_email],
    )

    test_scenarios = [
        {
            "input": "注文番号12345の商品がまだ届きません",
            "expected_actions": ["search_faq", "create_ticket"],
            "expected_keywords": ["配送", "確認"],
        },
    ]

    for scenario in test_scenarios:
        result = agent.run(scenario["input"])
        # 期待されるアクションが実行されたか
        actions_used = [s.tool_name for s in result.steps if s.tool_name]
        for expected in scenario["expected_actions"]:
            assert expected in actions_used, f"{expected} が実行されませんでした"
```

> **ポイントまとめ**
> - ユニットテストで個別ツールの品質を担保する（最も重要）
> - 統合テストではLLMをモック化して安定性を確保する
> - E2Eテストはコストが高いため、重要シナリオに限定する

## 7.4 ベンチマークとデータセット

### 主要なベンチマーク一覧

| ベンチマーク | 対象タスク | 評価方法 | 特徴 |
|-------------|-----------|---------|------|
| SWE-bench | GitHub Issue の修正 | テストスイートの通過率 | 実際のOSSリポジトリが対象 |
| GAIA | 汎用アシスタントタスク | 正解との一致 | Web検索・計算等の複合タスク |
| WebArena | Webブラウジング | タスク完了率 | 実際のWebサイトを操作 |
| HumanEval | コード生成 | テストケースの通過率 | Python関数の実装 |
| MINT | マルチターン対話 | タスク成功率 | ツール使用を含む対話 |
| AgentBench | 多様なエージェントタスク | タスク別スコア | OS操作・DB操作等8種 |

### SWE-bench の仕組み

SWE-bench は、実際のGitHubリポジトリのIssueとPull Requestから作られたベンチマークです。

```
┌──────────────────────────────────────────────────┐
│  SWE-bench の評価フロー                           │
│                                                  │
│  1. Issue の説明文を入力としてエージェントに渡す    │
│  2. エージェントがリポジトリのコードを修正           │
│  3. リポジトリのテストスイートを実行                │
│  4. 該当テストが通れば「解決」と判定                │
│                                                  │
│  ┌────────┐    ┌──────────┐    ┌────────────┐    │
│  │ Issue  │ →  │ Agent が │ →  │ テスト実行  │    │
│  │ 説明文 │    │ コード修正│    │ Pass/Fail  │    │
│  └────────┘    └──────────┘    └────────────┘    │
│                                                  │
│  SWE-bench Lite: 300問（軽量版）                  │
│  SWE-bench Full: 2,294問（完全版）                │
└──────────────────────────────────────────────────┘
```

### 自作ベンチマークの作成

汎用ベンチマークだけでなく、自分のユースケースに特化したベンチマークを作ることも重要です。

```python

evaluation_dataset = [
    {

        "category": "技術サポート",

        "expected_tools": ["search_faq", "search_known_issues", "create_ticket"],
        "expected_keywords": ["E-5012", "チケット"],
        "difficulty": "hard",
    },
]

def run_benchmark(agent, dataset):
    """自作ベンチマークを実行します"""
    results = []
    for case in dataset:
        result = agent.run(case["input"])
        score = evaluate_case(result, case)
        results.append({"id": case["id"], **score})

    # カテゴリ別集計
    by_category = {}
    for r in results:
        case = next(c for c in dataset if c["id"] == r["id"])
        cat = case["category"]
        by_category.setdefault(cat, []).append(r)

    return results, by_category
```

> **よくある間違い**
> - ベンチマークのスコアだけを追い求める → 自社ユースケースでの性能が重要です
> - ベンチマークのテストデータで学習（汚染）してしまう → 評価用データは訓練に使わないよう厳密に管理しましょう

> **ポイントまとめ**
> - SWE-bench、GAIA等の公開ベンチマークで業界水準と比較できる
> - 自社ユースケースに特化したベンチマークを必ず作成する
> - データ汚染（Data Contamination）に注意する

## 7.5 デバッグ手法

### トレーシング（Tracing）

エージェントの各ステップを詳細に記録し、問題の特定に役立てます。LangSmith や Langfuse などのツールが利用できます。

```python
import time
import uuid
import json
from dataclasses import dataclass, field, asdict

@dataclass
class TraceStep:
    """エージェントの1ステップ分のトレース情報"""
    step_id: str = field(default_factory=lambda: str(uuid.uuid4())[:8])
    step_type: str = ""       # "llm_call", "tool_call", "decision"
    input_data: dict = field(default_factory=dict)
    output_data: dict = field(default_factory=dict)
    duration_ms: float = 0
    tokens_used: int = 0
    error: str = None

@dataclass
class Trace:
    """エージェント実行全体のトレース"""
    trace_id: str = field(default_factory=lambda: str(uuid.uuid4())[:12])
    steps: list = field(default_factory=list)
    start_time: float = field(default_factory=time.time)

    def add_step(self, step: TraceStep):
        self.steps.append(step)

    def to_json(self) -> str:
        return json.dumps(asdict(self), indent=2, ensure_ascii=False)

    def print_summary(self):
        print(f"=== Trace {self.trace_id} ===")
        print(f"総ステップ数: {len(self.steps)}")
        total_time = sum(s.duration_ms for s in self.steps)
        print(f"総実行時間: {total_time:.0f}ms")
        total_tokens = sum(s.tokens_used for s in self.steps)
        print(f"総トークン数: {total_tokens}")
        print()
        for i, step in enumerate(self.steps, 1):
            status = "ERROR" if step.error else "OK"
            print(f"  Step {i}: [{step.step_type}] {status} ({step.duration_ms:.0f}ms)")
            if step.error:
                print(f"          -> {step.error}")
```

### ステップ実行によるデバッグ

エージェントを1ステップずつ実行し、各ステップの判断を確認します。

```python
class DebuggableAgent:
    """ステップ実行が可能なデバッグ用エージェント"""

    def __init__(self, agent, breakpoints=None):
        self.agent = agent
        self.breakpoints = breakpoints or []  # 停止条件のリスト
        self.step_history = []

    def step(self, user_input: str = None):
        """1ステップだけ実行して停止します"""
        result = self.agent.execute_one_step(user_input)
        self.step_history.append(result)

        print(f"--- Step {len(self.step_history)} ---")
        print(f"  判断: {result.decision}")
        print(f"  ツール: {result.tool_name or 'なし'}")
        print(f"  引数: {result.tool_args or 'なし'}")
        print(f"  結果: {result.output[:200]}...")
        return result

    def run_until_breakpoint(self, user_input: str):
        """ブレークポイントに到達するまで実行します"""
        result = self.step(user_input)
        while not result.is_final:
            if self._should_break(result):
                print("** ブレークポイントで停止 **")
                return result
            result = self.step()
        return result

    def _should_break(self, result) -> bool:
        for bp in self.breakpoints:
            if bp(result):
                return True
        return False

# 使用例: 特定のツールが呼ばれたら停止
debug_agent = DebuggableAgent(
    agent=my_agent,
    breakpoints=[
        lambda r: r.tool_name == "send_email",  # メール送信前に停止
        lambda r: r.error is not None,            # エラー発生時に停止
    ]
)
```

### ログ分析のパターン

```python
import logging

# エージェント用ロガーの設定
logger = logging.getLogger("agent")
logger.setLevel(logging.DEBUG)

# ファイルハンドラ（詳細ログ）
file_handler = logging.FileHandler("agent_debug.log", encoding="utf-8")
file_handler.setLevel(logging.DEBUG)

# コンソールハンドラ（重要なもののみ）
console_handler = logging.StreamHandler()
console_handler.setLevel(logging.INFO)

formatter = logging.Formatter(
    "%(asctime)s [%(levelname)s] %(name)s: %(message)s"
)
file_handler.setFormatter(formatter)
console_handler.setFormatter(formatter)

logger.addHandler(file_handler)
logger.addHandler(console_handler)

# エージェント内での使用例
def agent_step(self, messages):
    logger.info(f"Step開始 - メッセージ数: {len(messages)}")
    logger.debug(f"入力メッセージ: {messages[-1]}")

    response = self.llm.call(messages)
    logger.info(f"LLM応答 - トークン: {response.usage.total_tokens}")
    logger.debug(f"LLM出力: {response.content[:500]}")

    if response.tool_calls:
        for tc in response.tool_calls:
            logger.info(f"ツール呼出: {tc.name}({tc.args})")
```

> **ポイントまとめ**
> - トレーシングで各ステップの詳細を記録・可視化する
> - ステップ実行で問題のあるステップを特定する
> - 構造化ログを出力し、後から分析できるようにする

## 7.6 継続的改善

### A/Bテスト

プロンプトやモデルの変更が性能に与える影響を定量的に測定します。

```python
import random
from datetime import datetime

class ABTestRunner:
    """エージェントのA/Bテストを管理します"""

    def __init__(self, agent_a, agent_b, traffic_split=0.5):
        self.agent_a = agent_a  # 既存版（コントロール群）
        self.agent_b = agent_b  # 新版（実験群）
        self.traffic_split = traffic_split
        self.results_a = []
        self.results_b = []

    def route(self, user_input: str) -> tuple:
        """トラフィックを振り分けます"""
        if random.random() < self.traffic_split:
            result = self.agent_a.run(user_input)
            self.results_a.append(result)
            return "A", result
        else:
            result = self.agent_b.run(user_input)
            self.results_b.append(result)
            return "B", result

    def report(self) -> dict:
        """A/Bテストの結果を比較します"""
        return {
            "variant_A": {
                "sample_size": len(self.results_a),
                "success_rate": self._success_rate(self.results_a),
                "avg_cost": self._avg_cost(self.results_a),
                "avg_latency": self._avg_latency(self.results_a),
            },
            "variant_B": {
                "sample_size": len(self.results_b),
                "success_rate": self._success_rate(self.results_b),
                "avg_cost": self._avg_cost(self.results_b),
                "avg_latency": self._avg_latency(self.results_b),
            },
        }

    def _success_rate(self, results):
        if not results:
            return 0
        return sum(1 for r in results if r.success) / len(results)

    def _avg_cost(self, results):
        if not results:
            return 0
        return sum(r.cost_usd for r in results) / len(results)

    def _avg_latency(self, results):
        if not results:
            return 0
        return sum(r.latency_seconds for r in results) / len(results)
```

> - プロンプトはバージョン管理し、変更履歴と指標を記録する
> - 1回の改善では1つの変数だけを変更する

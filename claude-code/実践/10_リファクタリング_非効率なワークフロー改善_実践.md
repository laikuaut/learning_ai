# 実践課題10：リファクタリング ─ 非効率なワークフロー改善 ★4

> **難易度**: ★★★★☆（上級）
> **前提知識**: 第1章〜第8章（全章の知識を活用）
> **課題の種類**: リファクタリング
> **学習目標**: 非効率なClaude Code活用パターンを改善し、フック・CLAUDE.md・MCPを組み合わせた最適なワークフローを設計する

---

## 課題の説明

以下は、ある開発チームがClaude Codeを使って日常的に行っている開発ワークフローです。
しかし、このワークフローには**非効率な点やベストプラクティスに反する点**が複数あります。

問題点を特定し、改善したワークフローを設計してください。

---

### プロジェクト構造

```
my-app/
├── src/
│   ├── index.ts
│   ├── routes/
│   ├── services/
│   └── utils/
├── tests/
├── package.json
└── tsconfig.json
```

**注意: CLAUDE.mdもsettings.jsonも存在しません。**

### チームの日常的な作業フロー

```

> src/routes/にAPIエンドポイント、src/services/にビジネスロジックがあります。

（↑ 毎回セッション開始時に同じ説明をしている）
```

## 解答例

<details>
<summary>解答例（初心者向け）── 問題点と基本的な改善</summary>

### 特定した問題点

| # | 問題点 | 影響 |
|---|---|---|
| 1 | CLAUDE.mdがない | 毎回プロジェクト説明を入力しており、トークンの無駄遣い |
| 2 | リントを手動で依頼 | ファイル編集のたびに手動でESLint実行を依頼している |
| 3 | リントエラーの修正→再実行が手動 | 修正と再チェックのサイクルが非効率 |
| 4 | mainブランチに直接push | ブランチ戦略がなく、誤ったコードが本番に混入するリスク |
| 5 | `git add .` で全ファイルを追加 | 意図しないファイル（.env等）がコミットされる危険性 |
| 6 | コミットメッセージが曖昧 | 「新機能追加」では変更内容が不明で、履歴の追跡が困難 |
| 7 | 危険なコマンドへの防護がない | rm -rfやDROP TABLE等の実行を防止する仕組みがない |
| 8 | PRレビューが毎回手動 | 同じ手順を毎回入力している |

### 基本的な改善

**CLAUDE.md の作成:**

```markdown
# My App

## 概要
TypeScript + ExpressのバックエンドAPI。

## 技術スタック
- TypeScript, Express, Jest

## 開発コマンド
- `npm run dev` : 開発サーバー起動
- `npm test` : テスト実行
- `npm run lint` : ESLint実行

## コーディング規約
- 命名: camelCase
- コミットメッセージ: 日本語で具体的に書く（例：「ユーザー検索APIを追加」）

## ルール
- mainブランチへの直接pushは禁止。必ずfeatureブランチを作成すること
- .envファイルはコミットしないこと
```

**`.claude/settings.json` の作成:**

```json
{
  "hooks": {
    "PostToolUse": [
      {
        "matcher": "Edit|Write",
        "command": "if echo \"$CLAUDE_FILE_PATH\" | grep -qE '\\.(ts|tsx)$'; then cd \"$CLAUDE_PROJECT_DIR\" && npx eslint --fix \"$CLAUDE_FILE_PATH\" 2>&1 || true; fi"
      }
    ]
  }
}
```

### 問題点と改善策の対応表

| # | 問題点 | 改善策 | 実装方法 |
|---|---|---|---|
| 1 | CLAUDE.mdがない | 作成して自動読み込み | CLAUDE.md作成 |
| 2 | リントが手動 | PostToolUseで自動化 | フック設定 |
| 3 | リント修正→再チェックが手動 | `--fix`で自動修正 | フックに`--fix`追加 |
| 4 | mainに直接push | PreToolUseでブロック | フック設定 |
| 5 | `git add .`が危険 | CLAUDE.mdで注意喚起 + gitignore | CLAUDE.md + .gitignore |
| 6 | コミットメッセージが曖昧 | `/commit`コマンド活用 | CLAUDE.mdで規約化 |
| 7 | 危険コマンドの防護なし | PreToolUseでブロック | フック設定 |
| 8 | PRレビューが手動 | スクリプト化 | シェルスクリプト |
| 9 | テスト実行を毎回手動依頼 | CLAUDE.mdに手順記載 | CLAUDE.md |
| 10 | /compact未使用 | 長時間セッションで圧縮 | CLAUDE.mdで推奨 |

### 改善後の設定ファイル

**CLAUDE.md:**

```markdown
# My App - バックエンドAPI

## 概要
ユーザー管理・商品管理を提供するREST API。

## 技術スタック
- 言語: TypeScript 5.x
- フレームワーク: Express 4.x
- テスト: Jest + Supertest
- リンター: ESLint
- DB: PostgreSQL + Prisma

## ディレクトリ構成
- `src/routes/` : APIエンドポイント定義
- `src/services/` : ビジネスロジック（DB操作はここ）
- `src/utils/` : 共通ユーティリティ
- `tests/` : テストファイル

## 開発コマンド
- `npm run dev` : 開発サーバー起動
- `npm test` : 全テスト実行
- `npm run lint` : ESLint実行
- `npm run lint:fix` : ESLint自動修正

## コーディング規約
- 変数名・関数名: camelCase
- クラス名・型名: PascalCase
- ファイル名: camelCase.ts
- コミットメッセージ: 日本語で具体的に（例: 「ユーザー検索APIのページネーション対応」）
- `/commit`コマンドを使ってAIにメッセージを生成させることを推奨

## Gitルール
- **mainブランチへの直接pushは禁止**
- 必ず `feature/xxx` ブランチを作成して開発
- PRを作成してレビュー後にマージ
- `git add .` ではなく、変更ファイルを個別にステージング

## 重要な注意事項
- `.env` ファイルは絶対にコミットしない
- テスト追加なしの機能追加は禁止
- 長時間の作業では `/compact` でコンテキストを圧縮すること
```

**`.claude/settings.json`:**

```json
{
  "hooks": {
    "PreToolUse": [
      {
        "matcher": "Bash",
        "command": "if echo \"$CLAUDE_TOOL_INPUT\" | grep -qE 'rm\\s+-rf|DROP\\s+TABLE|TRUNCATE'; then echo 'BLOCKED: 破壊的操作は禁止です' >&2; exit 2; fi"
      },
      {
        "matcher": "Bash",
        "command": "if echo \"$CLAUDE_TOOL_INPUT\" | grep -qE 'git\\s+push.*main|git\\s+push.*master'; then echo 'BLOCKED: mainブランチへの直接pushは禁止です。featureブランチを使用してください。' >&2; exit 2; fi"
      }
    ],
    "PostToolUse": [
      {
        "matcher": "Edit|Write",
        "command": "if echo \"$CLAUDE_FILE_PATH\" | grep -qE '\\.(ts|tsx)$'; then cd \"$CLAUDE_PROJECT_DIR\" && npx eslint --fix \"$CLAUDE_FILE_PATH\" 2>&1 || true; fi"
      }
    ],
    "Stop": [
      {
        "matcher": "",
        "command": "cd \"$CLAUDE_PROJECT_DIR\" && echo '--- 未コミットの変更 ---' && git diff --stat HEAD 2>/dev/null || true"
      }
    ]
  }
}
```

**`scripts/review-pr.sh`:**

```bash
#!/bin/bash
# PRレビュー自動化スクリプト
PR_NUM=${1:?'使い方: ./scripts/review-pr.sh <PR番号>'}

echo "PR #${PR_NUM} をレビュー中..."
gh pr diff "$PR_NUM" | claude -p "
このPRの差分をレビューしてください。
観点：コード品質、セキュリティ、テストの十分性、命名規約の遵守
" --max-turns 3
```

### 改善後のワークフロー

```
=== 毎朝の作業開始 ===
$ cd ~/my-app
$ claude
# → CLAUDE.mdが自動読み込み。プロジェクト説明は不要！

=== 機能開発 ===
> feature/user-search ブランチを作成して、ユーザー検索APIを追加してください
# → ブランチ作成→実装が一度に行われる
# → PostToolUseフックで編集後にESLint自動実行＆自動修正
# → リントの手動依頼は不要！

=== テストとコミット ===
> テストを実行してください
> /commit
# → AIが変更内容に基づいた適切なコミットメッセージを生成

=== PR作成 ===
> PRを作成してください
# → gh pr create が実行される

=== 別のPRをレビュー ===
$ ./scripts/review-pr.sh 123
# → スクリプト1つでPRレビューが完了
```

**初心者向けとの違い:**
- 初心者向けは主にCLAUDE.mdとPostToolUseフックで最小限の改善を行います
- 改良版はPreToolUseによるmainブランチ保護、Stopフックによる変更サマリー、PRレビュースクリプトまで含む包括的な改善です
- ワークフロー全体をデザインすることで、チーム全体の生産性向上につなげています

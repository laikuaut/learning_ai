# 第8章：実践的なスクリプト開発

## この章のゴール

- `set -euo pipefail` で堅牢なスクリプトを書けるようになること
- trapコマンドでエラーハンドリングとクリーンアップを実装できること
- getoptsでコマンドライン引数を柔軟に処理できること
- デバッグ技法を身につけ、問題を素早く特定できるようになること
- スクリプトの構造化とベストプラクティスを実践できること

---

## 8.1 堅牢なスクリプトの書き方

### set -euo pipefail

実務のスクリプトでは、冒頭に以下を書くのが鉄則です。

```bash
#!/usr/bin/env bash
set -euo pipefail
```

各オプションの意味：

| オプション | 効果 |
|-----------|------|
| `set -e` (errexit) | コマンドが失敗したら即座にスクリプトを終了 |
| `set -u` (nounset) | 未定義の変数を参照するとエラーで終了 |
| `set -o pipefail` | パイプラインの途中でエラーが起きても検出 |

```bash
#!/usr/bin/env bash
set -euo pipefail

# set -e の効果
# false          # この行があるとスクリプトはここで終了する
echo "この行は実行される"

# set -u の効果
# echo "$undefined_var"  # エラー: unbound variable

# set -o pipefail の効果
# false | true   # pipefail なしだと成功（trueの終了ステータス）
                  # pipefail ありだと失敗（falseのエラーを検出）
```

### set -e の例外処理

```bash
#!/usr/bin/env bash
set -euo pipefail

# コマンドの失敗を許容したい場合
if ! grep -q "pattern" file.txt 2>/dev/null; then
    echo "パターンが見つかりませんでした"
fi

# || true で失敗を無視
rm /tmp/maybe_not_exist.txt 2>/dev/null || true

# 終了ステータスを保存
set +e  # 一時的に -e を無効化
risky_command
status=$?
set -e  # 再び有効化
echo "ステータス: $status"
```

### trap コマンド

`trap` はシグナルやイベントが発生したときの処理を登録します。

```bash
#!/usr/bin/env bash
set -euo pipefail

# 一時ファイルの作成
tmpfile=$(mktemp)
tmpdir=$(mktemp -d)

# EXIT シグナルで確実にクリーンアップ
cleanup() {
    echo "クリーンアップ中..."
    rm -f "$tmpfile"
    rm -rf "$tmpdir"
}
trap cleanup EXIT

# エラー発生時の処理
on_error() {
    echo "エラー発生: 行 $1" >&2
}
trap 'on_error $LINENO' ERR

# メイン処理
echo "一時ファイル: $tmpfile"
echo "一時ディレクトリ: $tmpdir"
echo "データ" > "$tmpfile"

# スクリプト終了時に cleanup が自動実行される
```

### 主なシグナル

| シグナル | いつ発生するか |
|---------|---------------|
| `EXIT` | スクリプト終了時（正常・異常問わず） |
| `ERR` | コマンドがエラーで終了したとき |
| `INT` | Ctrl+C が押されたとき |
| `TERM` | kill コマンドで終了要求されたとき |
| `DEBUG` | 各コマンドの実行前 |

### ポイントまとめ

- `set -euo pipefail` は実務スクリプトの必須設定
- `trap cleanup EXIT` で後処理を確実に実行
- `trap 'handler $LINENO' ERR` でエラー行を特定

## 8.2 デバッグ技法

### set -x でトレース出力

```bash
#!/usr/bin/env bash

# スクリプト全体をデバッグ
set -x  # デバッグモードON

name="Tanaka"
echo "Hello, $name"

set +x  # デバッグモードOFF
echo "デバッグ終了"
```

実行結果：
```
+ name=Tanaka
+ echo 'Hello, Tanaka'
Hello, Tanaka
+ set +x
デバッグ終了
```

### 外部からのデバッグ実行

```bash
# スクリプト全体をデバッグモードで実行
$ bash -x script.sh

# 特定の変数が変更されたときにトレース
$ bash -x script.sh 2>&1 | grep "variable_name"
```

### PS4 でデバッグ出力をカスタマイズ

```bash
#!/usr/bin/env bash
# デバッグ出力にファイル名と行番号を表示
export PS4='+[${BASH_SOURCE}:${LINENO}] '
set -x

name="Tanaka"
echo "$name"
```

出力：
```
+[script.sh:5] name=Tanaka
+[script.sh:6] echo Tanaka
Tanaka
```

### 条件付きデバッグ

```bash
#!/usr/bin/env bash
set -euo pipefail

# 環境変数でデバッグを切り替え
DEBUG="${DEBUG:-false}"

debug_log() {
    if [[ "$DEBUG" = "true" ]]; then
        echo "[DEBUG] $*" >&2
    fi
}

debug_log "処理を開始します"
debug_log "変数: name=Tanaka"

# 使い方: DEBUG=true bash script.sh
```

### ポイントまとめ

- `set -x` でコマンドのトレース出力
- `PS4` でデバッグ出力をカスタマイズ
- `DEBUG` 環境変数で切り替え可能なログを実装

## 8.3 コマンドライン引数の処理

### getopts で引数を解析

```bash
#!/usr/bin/env bash
set -euo pipefail

usage() {
    cat <<EOF
使い方: $(basename "$0") [オプション] <ファイル>

オプション:
  -v          詳細モード
  -o FILE     出力先ファイル
  -n NUMBER   処理する行数
  -h          ヘルプを表示
EOF
}

# デフォルト値
verbose=false
output=""
num_lines=10

# getopts で引数を解析
while getopts "vo:n:h" opt; do
    case "$opt" in
        v) verbose=true ;;
        o) output="$OPTARG" ;;
        n) num_lines="$OPTARG" ;;
        h) usage; exit 0 ;;
        ?) usage >&2; exit 1 ;;
    esac
done

# 残りの位置引数を取得
shift $((OPTIND - 1))

# 必須引数のチェック
if [[ $# -eq 0 ]]; then
    echo "エラー: ファイルを指定してください" >&2
    usage >&2
    exit 1
fi

input_file="$1"

echo "詳細モード: $verbose"
echo "出力先: ${output:-（標準出力）}"
echo "行数: $num_lines"
echo "入力ファイル: $input_file"
```

### getopts の構文

```
getopts "オプション文字列" 変数名
```

- `v` : `-v`（引数なしオプション）
- `o:` : `-o value`（引数ありオプション、`:`が付く）
- 先頭の`:` : エラーメッセージを自分で処理（サイレントモード）

### shift でロングオプション風の処理

```bash
#!/usr/bin/env bash
set -euo pipefail

verbose=false
output=""

while [[ $# -gt 0 ]]; do
    case "$1" in
        -v|--verbose)
            verbose=true
            shift
            ;;
        -o|--output)
            output="$2"
            shift 2
            ;;
        -h|--help)
            echo "使い方: $(basename "$0") [-v] [-o file] <input>"
            exit 0
            ;;
        -*)
            echo "不明なオプション: $1" >&2
            exit 1
            ;;
        *)
            break  # オプション以外の引数に到達
            ;;
    esac
done

echo "verbose=$verbose, output=$output, args=$*"
```

### ポイントまとめ

- `getopts` は短いオプション（`-v`、`-o val`）の解析に使う
- `shift` + `case` でロングオプション（`--verbose`）も処理可能
- `OPTARG` でオプションの引数を取得

## 8.4 一時ファイルとクリーンアップ

### mktemp の使い方

```bash
#!/usr/bin/env bash
set -euo pipefail

# 一時ファイルを安全に作成
tmpfile=$(mktemp)                        # /tmp/tmp.XXXXXXXXXX
tmpfile2=$(mktemp /tmp/myapp.XXXXXX)     # /tmp/myapp.XXXXXX
tmpdir=$(mktemp -d)                      # 一時ディレクトリ

# trap で確実にクリーンアップ
trap 'rm -f "$tmpfile" "$tmpfile2"; rm -rf "$tmpdir"' EXIT

echo "一時ファイル1: $tmpfile"
echo "一時ファイル2: $tmpfile2"
echo "一時ディレクトリ: $tmpdir"

# 一時ファイルを使った処理例
echo "データ" > "$tmpfile"
cat "$tmpfile"
```

> **重要**: 一時ファイルは必ず `trap EXIT` でクリーンアップしましょう。スクリプトが途中で異常終了しても確実に削除されます。

### ポイントまとめ

- `mktemp` で安全な一時ファイルを作成（予測不能なファイル名）
- `mktemp -d` で一時ディレクトリ
- 必ず `trap EXIT` と組み合わせる

## 8.5 外部コマンドとの連携

### xargs

```bash
#!/usr/bin/env bash

# find の結果を xargs でコマンドに渡す
find . -name "*.log" -print0 | xargs -0 rm -f

# 並列実行
find . -name "*.jpg" -print0 | xargs -0 -P 4 -I {} convert {} -resize 50% {}

# 1行ずつ処理
echo -e "file1\nfile2\nfile3" | xargs -I {} echo "Processing: {}"
```

### find -exec

```bash
#!/usr/bin/env bash

# 各ファイルに対してコマンドを実行
find /var/log -name "*.log" -mtime +30 -exec rm -f {} \;

# 確認してから削除
find /tmp -name "*.tmp" -mtime +7 -exec rm -i {} \;

# 複数のコマンドを実行
find . -name "*.sh" -exec chmod +x {} \; -exec echo "権限付与: {}" \;
```

### ポイントまとめ

- `xargs -0` は `find -print0` と組み合わせてスペース入りファイル名に対応
- `xargs -P N` で並列実行
- `find -exec` でファイルごとの処理

## 8.6 スクリプトの構造化

### main関数パターン

```bash
#!/usr/bin/env bash
set -euo pipefail

# --- 定数 ---
readonly SCRIPT_DIR="$(cd "$(dirname "$0")" && pwd)"
readonly SCRIPT_NAME="$(basename "$0")"

# --- 関数定義 ---
usage() {
    cat <<EOF
使い方: $SCRIPT_NAME [オプション]

オプション:
  -h, --help    ヘルプを表示
  -v, --verbose 詳細モード
EOF
}

log_info() {
    echo "[INFO] $(date '+%H:%M:%S') $*"
}

log_error() {
    echo "[ERROR] $(date '+%H:%M:%S') $*" >&2
}

cleanup() {
    log_info "クリーンアップ中..."
}

# --- メイン処理 ---
main() {
    trap cleanup EXIT

    # 引数解析
    local verbose=false
    while [[ $# -gt 0 ]]; do
        case "$1" in
            -h|--help) usage; exit 0 ;;
            -v|--verbose) verbose=true; shift ;;
            *) break ;;
        esac
    done

    log_info "スクリプトを開始します"
    log_info "スクリプトディレクトリ: $SCRIPT_DIR"

    # ここにメイン処理を書く
    if [[ "$verbose" = "true" ]]; then
        log_info "詳細モードが有効です"
    fi

    log_info "スクリプトが完了しました"
}

# スクリプト実行
main "$@"
```

> **なぜmain関数を使うのか**: 全体を関数にまとめることで、`local` 変数が使え、コードの構造が明確になります。また、ファイルの読み込み順序に依存しなくなります。

### ポイントまとめ

- main関数パターンで構造化されたスクリプトに
- 定数は `readonly` で保護
- スクリプトの最後で `main "$@"` を呼ぶ

## 8.7 シェルスクリプトのベストプラクティス

### 命名規則

```bash
# 変数名: snake_case（小文字）
user_name="tanaka"
max_retry=3

# 定数: UPPER_SNAKE_CASE（大文字）
readonly MAX_CONNECTIONS=100
readonly DEFAULT_PORT=8080

# 関数名: snake_case（小文字）
get_user_info() { ... }
validate_input() { ... }
```

### ShellCheck の活用

[ShellCheck](https://www.shellcheck.net/) はシェルスクリプトの静的解析ツールです。

```bash
# インストール
# Ubuntu/Debian: apt install shellcheck
# macOS: brew install shellcheck

# 使い方
$ shellcheck script.sh
```

ShellCheckが検出する一般的な問題：

```bash
# SC2086: 変数のクォート忘れ
echo $name         # 警告
echo "$name"       # OK

# SC2034: 使われていない変数
unused_var="hello" # 警告

# SC2155: declare と代入を同時にすると終了ステータスが隠れる
local val=$(command)  # 警告
local val              # OK
val=$(command)         # OK
```

### 安全なスクリプトのチェックリスト

```
✅ set -euo pipefail を冒頭に
✅ 変数は "${var}" でクォート
✅ 一時ファイルは mktemp + trap EXIT
✅ エラーメッセージは >&2 で stderr に
✅ 使い方（usage）関数を用意
✅ ShellCheck でチェック
✅ コメントで意図を説明
✅ 入力値のバリデーション
```

### テストの書き方

```bash
#!/usr/bin/env bash
# test_functions.sh - 簡易テストフレームワーク

source ./lib/utils.sh

# テスト関数
test_is_number() {
    local test_name="is_number"

    is_number "123" || { echo "FAIL: $test_name - 123 は数値"; return 1; }
    is_number "0" || { echo "FAIL: $test_name - 0 は数値"; return 1; }
    ! is_number "abc" || { echo "FAIL: $test_name - abc は数値でない"; return 1; }
    ! is_number "" || { echo "FAIL: $test_name - 空文字は数値でない"; return 1; }

    echo "PASS: $test_name"
}

# テスト実行
test_is_number
```

### ポイントまとめ

- 変数は `snake_case`、定数は `UPPER_SNAKE_CASE`
- ShellCheck を必ず使う（CI/CDに組み込むのが理想）
- `"${var}"` のクォートを徹底

## 問題1：エラーログレベルの理解（基本）

以下のエラーログレベルを、重大度の高い順に並べ替えてください。

`info`, `error`, `emerg`, `warn`, `debug`, `crit`, `alert`, `notice`

また、本番環境で通常設定すべきレベルはどれですか？

<details>
<summary>解答例</summary>

```
【重大度の高い順】
emerg > alert > crit > error > warn > notice > info > debug

【本番環境の推奨設定】
error_log /var/log/nginx/error.log warn;

理由：
- warn 以上を記録することで、問題の兆候を検知できる
- info や debug は大量にログが出力され、
  ディスクを圧迫する可能性がある
- デバッグ時のみ一時的に debug に変更する
```

</details>

## 問題2：ログフォーマットの設計（基本）

以下の情報を含むJSON形式のカスタムログフォーマットを設計してください。

- 日時（ISO 8601形式）
- クライアントIP
- HTTPメソッド
- リクエストURI
- ステータスコード
- レスポンスサイズ
- 処理時間
- User-Agent

<details>
<summary>解答例</summary>

```nginx
log_format json_custom escape=json '{'
    '"timestamp": "$time_iso8601",'
    '"client_ip": "$remote_addr",'
    '"method": "$request_method",'
    '"uri": "$request_uri",'
    '"status": $status,'
    '"body_bytes": $body_bytes_sent,'
    '"request_time": $request_time,'
    '"user_agent": "$http_user_agent"'
'}';

access_log /var/log/nginx/access.json json_custom;

# 出力例:
# {"timestamp":"2026-04-05T10:00:00+09:00","client_ip":"192.168.1.10",
#  "method":"GET","uri":"/api/users","status":200,"body_bytes":1234,
#  "request_time":0.025,"user_agent":"Mozilla/5.0 ..."}
```

</details>

## 問題3：パフォーマンスチューニング（基本）

以下の各設定の目的を説明してください。

```nginx
worker_processes auto;
worker_connections 4096;
sendfile on;
tcp_nopush on;
keepalive_timeout 65;
gzip on;
gzip_comp_level 6;
```

<details>
<summary>解答例</summary>

```
worker_processes auto;
  → ワーカープロセス数をCPUコア数に自動設定。
    コア数に合わせることで最大効率でリクエストを処理する。

worker_connections 4096;
  → 1ワーカーあたりの最大同時接続数。
    4コア×4096接続 = 最大約16,000同時接続。

sendfile on;
  → カーネルレベルのファイル転送を有効化。
    ユーザー空間を経由せず直接ファイルを送信し、高速化。

tcp_nopush on;
  → sendfile使用時にHTTPヘッダーとファイル本体を
    1つのTCPパケットにまとめて送信。ネットワーク効率向上。

keepalive_timeout 65;
  → Keep-Alive接続のアイドルタイムアウト（65秒）。
    同じクライアントからの連続リクエストで
    TCP接続を再利用し、接続コストを削減。

gzip on;
  → レスポンスのgzip圧縮を有効化。
    転送サイズを削減し、ページ読み込み速度を向上。

gzip_comp_level 6;
  → 圧縮レベル（1-9）。6はサイズと
    CPU負荷のバランスが良い推奨値。
```

</details>

## 問題4：エラー原因の特定（応用）

以下の各エラーの原因として考えられるものと、調査・解決方法を述べてください。

1. ブラウザに `502 Bad Gateway` が表示される
2. ブラウザに `403 Forbidden` が表示される
3. `nginx -t` は成功するが、ブラウザに `404 Not Found` が表示される
4. ファイルをアップロードすると `413 Request Entity Too Large` になる

<details>
<summary>解答例</summary>

```
1. 502 Bad Gateway
   原因: proxy_pass で指定したバックエンドサーバーが応答しない
   調査:
     $ sudo tail -f /var/log/nginx/error.log
     → "connect() failed" や "upstream" のエラーを確認
     $ curl http://localhost:3000  # バックエンドに直接アクセス
     $ sudo systemctl status myapp  # アプリの状態確認
   解決:
     ・バックエンドアプリケーションを起動/再起動する
     ・proxy_pass のアドレス・ポートが正しいか確認する

2. 403 Forbidden
   原因: ファイル/ディレクトリの権限不足、またはインデックスファイル未配置
   調査:
     $ ls -la /var/www/html/
     $ namei -l /var/www/html/index.html
     $ sudo tail /var/log/nginx/error.log
   解決:
     ・chmod 755 ディレクトリ, chmod 644 ファイル
     ・所有者: chown -R www-data:www-data /var/www/html/
     ・index.html が存在するか確認
     ・SELinux: sudo setsebool -P httpd_read_user_content on

3. 404 Not Found
   原因: root ディレクティブのパスが間違っている、
         またはファイルが実際に存在しない
   調査:
     $ sudo nginx -T | grep root  # 設定されているrootを確認
     $ ls -la /設定されたパス/
     $ sudo tail /var/log/nginx/error.log
     → "open() failed" でファイルパスを確認
   解決:
     ・root のパスを正しいディレクトリに修正
     ・ファイルが実際に存在するか確認
     ・try_files の設定を見直す

4. 413 Request Entity Too Large
   原因: リクエストボディ（アップロードファイル）が
         client_max_body_size の上限を超えている
   解決:
     client_max_body_size 50m;  # 必要なサイズに変更
     ※ アップロード用のlocationだけ緩めることも可能
```

</details>

## 問題5：gzip設定の最適化（応用）

以下のgzip設定に改善点が3つあります。指摘して修正してください。

```nginx
http {
    gzip on;
    gzip_comp_level 9;
    gzip_types text/html;
    gzip_min_length 0;
}
```

<details>
<summary>ヒント</summary>

- 圧縮レベル9はCPU負荷が非常に高い
- text/htmlはデフォルトで圧縮されるが、他のタイプが含まれていない
- min_length 0は小さなファイルも圧縮し、かえってオーバーヘッドが増える

</details>

<details>
<summary>解答例</summary>

```
問題点:

1. gzip_comp_level 9 → CPU負荷が高すぎる
   修正: gzip_comp_level 6（推奨バランス）

2. gzip_types text/html のみ → CSS/JS/JSON等が圧縮されない
   修正: よく使うテキスト系MIMEタイプを追加
   ※ text/html はデフォルトで含まれるため、明示不要

3. gzip_min_length 0 → 小さなファイルも圧縮される
   修正: gzip_min_length 256（256バイト未満は圧縮しない）
   圧縮のオーバーヘッドの方が大きくなる
```

```nginx
# 改善後
http {
    gzip on;
    gzip_vary on;
    gzip_proxied any;
    gzip_comp_level 6;
    gzip_min_length 256;
    gzip_types
        text/plain
        text/css
        text/javascript
        application/json
        application/javascript
        application/xml
        application/xml+rss
        image/svg+xml;
}
```

</details>

## 問題6：stub_status による監視（応用）

stub_status の以下の出力を分析し、質問に答えてください。

```
Active connections: 250
server accepts handled requests
 50000 50000 120000
Reading: 10 Writing: 30 Waiting: 210
```

1. 現在のアクティブ接続数は？
2. accepts と handled が同じ値ですが、これは何を意味しますか？
3. 平均して1接続あたり何リクエスト処理されていますか？
4. Waiting: 210 は問題ですか？

<details>
<summary>解答例</summary>

```
1. アクティブ接続数: 250
   現在nginxが処理中の接続（Reading + Writing + Waiting）。

2. accepts = handled = 50000
   受け付けた接続をすべて正常に処理できていることを意味する。
   handled < accepts の場合は、リソース不足で
   接続を拒否（ドロップ）していることを示す。
   → 現状は正常。

3. 1接続あたりの平均リクエスト数: 120000 / 50000 = 2.4リクエスト
   Keep-Aliveにより、1つのTCP接続で複数のHTTPリクエストを
   処理していることがわかる。

4. Waiting: 210 は通常は問題なし。
   WaitingはKeep-Alive接続でアイドル状態のクライアント数。
   keepalive_timeout で待機している接続であり、
   次のリクエストを効率的に処理できる状態。
   ただし、Waiting が極端に多い場合は
   keepalive_timeout を短くすることを検討する。
```

</details>

## 問題7：本番環境の構築（チャレンジ）

以下の要件を満たすnginxの本番環境設定を作成してください。

**要件：**
- アプリケーション: `app.example.com`（React SPA + Node.js API）
- フロントエンド: `/var/www/app/build` に静的ファイル
- API: `http://localhost:3000` にプロキシ
- HTTPS対応（Let's Encrypt証明書）
- gzip圧縮を有効化
- 静的ファイルのキャッシュ（CSS/JS: 30日、画像: 1年）
- JSON形式のアクセスログ
- セキュリティヘッダーを適用
- レートリミット（API: 30r/s、ログイン: 1r/s）
- パフォーマンスチューニング済み

<details>
<summary>解答例</summary>

```nginx
# === /etc/nginx/nginx.conf ===

user www-data;
worker_processes auto;
worker_rlimit_nofile 65535;
pid /run/nginx.pid;
error_log /var/log/nginx/error.log warn;

events {
    worker_connections 4096;
    multi_accept on;
}

http {
    include mime.types;
    default_type application/octet-stream;
    server_tokens off;

    # パフォーマンス
    sendfile on;
    tcp_nopush on;
    tcp_nodelay on;
    keepalive_timeout 65;

    # gzip
    gzip on;
    gzip_vary on;
    gzip_proxied any;
    gzip_comp_level 6;
    gzip_min_length 256;
    gzip_types text/plain text/css text/javascript
               application/json application/javascript
               application/xml image/svg+xml;

    # ログ
    log_format json escape=json '{'
        '"time":"$time_iso8601",'
        '"ip":"$remote_addr",'
        '"method":"$request_method",'
        '"uri":"$request_uri",'
        '"status":$status,'
        '"bytes":$body_bytes_sent,'
        '"time_ms":$request_time,'
        '"upstream_ms":"$upstream_response_time",'
        '"ua":"$http_user_agent"'
    '}';

    # レートリミット
    limit_req_zone $binary_remote_addr zone=api:10m rate=30r/s;
    limit_req_zone $binary_remote_addr zone=login:10m rate=1r/s;
    limit_req_status 429;

    include /etc/nginx/sites-enabled/*.conf;
}


# === /etc/nginx/sites-enabled/app.example.com.conf ===

# HTTP → HTTPS リダイレクト
server {
    listen 80;
    server_name app.example.com;
    return 301 https://$server_name$request_uri;
}

# メインサーバー
server {
    listen 443 ssl http2;
    server_name app.example.com;

    root /var/www/app/build;
    index index.html;
    client_max_body_size 10m;

    access_log /var/log/nginx/app.access.json json;
    error_log /var/log/nginx/app.error.log warn;

    # SSL
    ssl_certificate /etc/letsencrypt/live/app.example.com/fullchain.pem;
    ssl_certificate_key /etc/letsencrypt/live/app.example.com/privkey.pem;
    ssl_protocols TLSv1.2 TLSv1.3;
    ssl_prefer_server_ciphers on;
    ssl_session_cache shared:SSL:10m;
    ssl_session_timeout 1d;

    # セキュリティヘッダー
    add_header X-Content-Type-Options "nosniff" always;
    add_header X-Frame-Options "SAMEORIGIN" always;
    add_header X-XSS-Protection "1; mode=block" always;
    add_header Referrer-Policy "strict-origin-when-cross-origin" always;
    add_header Strict-Transport-Security "max-age=63072000; includeSubDomains" always;

    # 隠しファイル拒否
    location ~ /\. { deny all; access_log off; log_not_found off; }

    # favicon / robots
    location = /favicon.ico { log_not_found off; access_log off; }
    location = /robots.txt  { log_not_found off; access_log off; }

    # 静的ファイルキャッシュ
    location ~* \.(css|js)$ {
        expires 30d;
        add_header Cache-Control "public, immutable";
    }
    location ~* \.(jpg|jpeg|png|gif|ico|svg|webp|woff2?)$ {
        expires 365d;
        add_header Cache-Control "public, immutable";
    }

    # API ログイン（厳しいレートリミット）
    location /api/login {
        limit_req zone=login burst=5;
        proxy_pass http://localhost:3000;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $scheme;
    }

    # API
    location /api/ {
        limit_req zone=api burst=50 nodelay;
        proxy_pass http://localhost:3000;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $scheme;
    }

    # SPA フォールバック
    location / {
        try_files $uri $uri/ /index.html;
    }
}
```

</details>

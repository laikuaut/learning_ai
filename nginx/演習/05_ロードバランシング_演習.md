## 問題1：基本的なロードバランシング設定（基本）

3台のバックエンドサーバー（ポート3001, 3002, 3003）にラウンドロビンでリクエストを分散する設定を作成してください。すべて `127.0.0.1` で動作しているものとします。

**期待される動作：**
```
リクエスト 1 → 127.0.0.1:3001
リクエスト 2 → 127.0.0.1:3002
リクエスト 3 → 127.0.0.1:3003
リクエスト 4 → 127.0.0.1:3001（一巡）
```

<details>
<summary>ヒント</summary>

`upstream` ブロックで3つのサーバーを定義し、`proxy_pass` で参照します。ラウンドロビンはデフォルトなので、特別なディレクティブは不要です。

</details>

<details>
<summary>解答例</summary>

```nginx
upstream app_servers {
    server 127.0.0.1:3001;
    server 127.0.0.1:3002;
    server 127.0.0.1:3003;
}

server {
    listen 80;
    server_name example.com;

    location / {
        proxy_pass http://app_servers;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
    }
}
```

**補足**: ラウンドロビンはNginxのデフォルトの分散アルゴリズムです。upstream ブロック内に特別なディレクティブを書く必要はありません。

</details>

## 問題2：分散アルゴリズムの選択（基本）

以下のそれぞれのケースに最適な分散アルゴリズムを答え、その設定を書いてください。

1. APIサーバーで、リクエストの処理時間が0.1秒〜10秒とばらつきが大きい
2. ログイン機能があり、セッション情報がサーバーのメモリに保存されている
3. CDNのキャッシュサーバーで、同じURLのリクエストは同じサーバーに送りたい

<details>
<summary>ヒント</summary>

1. 処理時間にばらつきがある → 接続数ベースが有効
2. セッション維持が必要 → IPアドレスベースが有効
3. URLベースの振り分け → 汎用ハッシュが有効

</details>

<details>
<summary>解答例</summary>

**1. least_conn（最小接続数）:**
```nginx
upstream api_backend {
    least_conn;
    server 192.168.1.101:8000;
    server 192.168.1.102:8000;
    server 192.168.1.103:8000;
}
```
処理時間にばらつきがある場合、接続数が少ないサーバーにリクエストを送ることで負荷が均等化されます。

**2. ip_hash:**
```nginx
upstream web_backend {
    ip_hash;
    server 192.168.1.101:3000;
    server 192.168.1.102:3000;
    server 192.168.1.103:3000;
}
```
同じIPアドレスのクライアントが常に同じサーバーに振り分けられるため、セッション情報が維持されます。

**3. hash $request_uri consistent:**
```nginx
upstream cache_backend {
    hash $request_uri consistent;
    server 192.168.1.101:8080;
    server 192.168.1.102:8080;
    server 192.168.1.103:8080;
}
```
同じURLのリクエストが常に同じサーバーに送られるため、キャッシュのヒット率が向上します。`consistent` を付けることで、サーバー追加・削除時のキャッシュ再配分を最小限に抑えます。

</details>

## 問題3：重み付きロードバランシング（基本）

以下のスペックが異なる3台のサーバーに、性能に見合った比率でリクエストを分散してください。

| サーバー | スペック | 想定処理能力 |
|---|---|---|
| 192.168.1.101 | 8コアCPU、32GB RAM | 高 |
| 192.168.1.102 | 4コアCPU、16GB RAM | 中 |
| 192.168.1.103 | 2コアCPU、8GB RAM | 低 |

<details>
<summary>ヒント</summary>

`weight` パラメータで振り分け比率を指定します。CPUコア数を基準にすると、8:4:2 = 4:2:1 の比率が目安になります。

</details>

<details>
<summary>解答例</summary>

```nginx
upstream backend {
    server 192.168.1.101:3000 weight=4;  # 高性能: 4倍の配分
    server 192.168.1.102:3000 weight=2;  # 中程度: 2倍の配分
    server 192.168.1.103:3000 weight=1;  # 低性能: 1倍の配分
}

server {
    listen 80;
    server_name example.com;

    location / {
        proxy_pass http://backend;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
    }
}
```

**補足**: 7リクエストあたりの配分はおおよそ以下のようになります。
- 192.168.1.101: 4リクエスト（約57%）
- 192.168.1.102: 2リクエスト（約29%）
- 192.168.1.103: 1リクエスト（約14%）

</details>

## 問題4：ヘルスチェックの設定（応用）

以下の要件を満たすヘルスチェック設定を作成してください。

- 3台のバックエンドサーバー
- 5回連続失敗したらサーバーを一時除外
- 60秒間除外した後、再度確認
- 全サーバーがダウンした場合に備えてバックアップサーバーを1台用意

<details>
<summary>ヒント</summary>

`max_fails` で失敗回数、`fail_timeout` で除外期間を設定します。バックアップサーバーは `backup` パラメータで指定します。

</details>

<details>
<summary>解答例</summary>

```nginx
upstream backend {
    server 192.168.1.101:3000 max_fails=5 fail_timeout=60s;
    server 192.168.1.102:3000 max_fails=5 fail_timeout=60s;
    server 192.168.1.103:3000 max_fails=5 fail_timeout=60s;

    # 全メインサーバーがダウンした場合のみ使用
    server 192.168.1.200:3000 backup;
}

server {
    listen 80;
    server_name example.com;

    location / {
        proxy_pass http://backend;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;

        proxy_connect_timeout 5s;
        proxy_read_timeout 30s;
    }
}
```

**動作フロー:**
1. Backend #1 に5回連続失敗 → 60秒間除外
2. 60秒後、Backend #1 にリクエストを送って再確認
3. 成功すれば復帰、失敗すれば再び60秒間除外
4. 全メインサーバーがダウン → バックアップサーバーが稼働

</details>

## 問題5：proxy_next_upstream の設定（応用）

バックエンドのエラー時に自動リトライする設定を作成してください。

**要件：**
- 502, 503, 504 エラーおよびタイムアウト時にリトライ
- 最大2回までリトライ
- リトライ全体のタイムアウトは15秒
- 500エラーではリトライしない（理由も説明してください）

<details>
<summary>ヒント</summary>

`proxy_next_upstream` でリトライ条件を、`proxy_next_upstream_tries` でリトライ回数を、`proxy_next_upstream_timeout` でタイムアウトを設定します。

</details>

<details>
<summary>解答例</summary>

```nginx
upstream backend {
    server 192.168.1.101:3000 max_fails=3 fail_timeout=30s;
    server 192.168.1.102:3000 max_fails=3 fail_timeout=30s;
    server 192.168.1.103:3000 max_fails=3 fail_timeout=30s;
}

server {
    listen 80;
    server_name example.com;

    location / {
        proxy_pass http://backend;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;

        # リトライ条件（500は含めない）
        proxy_next_upstream error timeout http_502 http_503 http_504;

        # 最大2回リトライ（初回 + リトライ2回 = 最大3サーバーに送信）
        proxy_next_upstream_tries 2;

        # リトライ全体のタイムアウト
        proxy_next_upstream_timeout 15s;
    }
}
```

**500エラーをリトライしない理由:**
- 500はアプリケーション内部のバグやエラーであることが多い
- POSTリクエストの場合、リトライによって同じデータが複数サーバーで処理される（例：注文の二重登録）
- 500エラーの原因がデータに起因する場合、どのサーバーに送っても同じ結果になる

</details>

## 問題6：メンテナンス時のサーバー除外（応用）

3台のバックエンドサーバーで運用中に、`192.168.1.102` をメンテナンスのために除外する設定を作成してください。メンテナンス完了後に復帰させる手順も記載してください。

**要件：**
- `ip_hash` アルゴリズムを使用中
- メンテナンス中はハッシュの計算結果を変えない

<details>
<summary>ヒント</summary>

`ip_hash` を使用している場合、サーバーを削除するとハッシュの計算結果が変わります。`down` フラグを使うと、ハッシュテーブルを維持したまま特定のサーバーを除外できます。

</details>

<details>
<summary>解答例</summary>

**メンテナンス前（通常運用）:**
```nginx
upstream backend {
    ip_hash;
    server 192.168.1.101:3000;
    server 192.168.1.102:3000;
    server 192.168.1.103:3000;
}
```

**メンテナンス中（192.168.1.102 を除外）:**
```nginx
upstream backend {
    ip_hash;
    server 192.168.1.101:3000;
    server 192.168.1.102:3000 down;   # down フラグで除外
    server 192.168.1.103:3000;
}
```

**手順:**
```bash
# 1. 設定ファイルを編集して down フラグを追加
sudo nano /etc/nginx/sites-available/example.com

# 2. 設定テストとリロード
sudo nginx -t && sudo systemctl reload nginx

# 3. メンテナンス作業を実施
ssh 192.168.1.102 "sudo systemctl stop app && ... && sudo systemctl start app"

# 4. 設定ファイルを編集して down フラグを削除
sudo nano /etc/nginx/sites-available/example.com

# 5. 設定テストとリロード
sudo nginx -t && sudo systemctl reload nginx
```

**補足**: `down` フラグを使うと `ip_hash` のハッシュテーブル上でサーバーの位置は維持されるため、他のサーバーへのセッション振り分けが変わりません。サーバーを upstream から完全に削除してしまうとハッシュが再計算され、全ユーザーのセッション割り当てが変わる可能性があります。

</details>

## 問題7：本番環境の冗長構成（チャレンジ）

以下の要件を満たすロードバランシング設定を作成してください。

**サーバー構成：**
- メインサーバー2台（192.168.1.101, 192.168.1.102）：高性能（weight=5）
- サブサーバー1台（192.168.1.103）：標準性能（weight=2）
- バックアップサーバー1台（192.168.1.200）

**要件：**
- 最小接続数アルゴリズムと重み付けを組み合わせる
- 3回失敗で30秒間サーバーを除外
- バックエンドエラー（502, 503, 504）時に最大2回リトライ
- 静的ファイル（`/static/`）はNginxが直接配信
- 全バックエンドダウン時にメンテナンスページを表示
- アクセスログにリクエスト処理時間とバックエンドのレスポンス時間を記録

<details>
<summary>ヒント</summary>

- `least_conn` と `weight` は組み合わせ可能です
- メンテナンスページは `error_page` と `location = /maintenance.html` で設定します
- カスタムログフォーマットに `$request_time` と `$upstream_response_time` を追加します

</details>

<details>
<summary>解答例</summary>

```nginx
http {
    # カスタムログフォーマット（処理時間を記録）
    log_format lb_detailed '$remote_addr - [$time_local] '
                           '"$request" $status $body_bytes_sent '
                           'rt=$request_time urt=$upstream_response_time '
                           'upstream=$upstream_addr';

    upstream webapp {
        least_conn;

        # メインサーバー（高性能）
        server 192.168.1.101:3000 weight=5 max_fails=3 fail_timeout=30s;
        server 192.168.1.102:3000 weight=5 max_fails=3 fail_timeout=30s;

        # サブサーバー（標準性能）
        server 192.168.1.103:3000 weight=2 max_fails=3 fail_timeout=30s;

        # バックアップサーバー
        server 192.168.1.200:3000 backup;
    }

    server {
        listen 80;
        server_name example.com;

        # カスタムログフォーマットを使用
        access_log /var/log/nginx/example.com.access.log lb_detailed;
        error_log  /var/log/nginx/example.com.error.log;

        # 静的ファイルはNginxが直接配信
        location /static/ {
            alias /var/www/shared/static/;
            expires 30d;
            add_header Cache-Control "public, immutable";
            access_log off;
        }

        # アプリケーションリクエスト
        location / {
            proxy_pass http://webapp;
            proxy_set_header Host $host;
            proxy_set_header X-Real-IP $remote_addr;
            proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
            proxy_set_header X-Forwarded-Proto $scheme;

            proxy_connect_timeout 5s;
            proxy_read_timeout 60s;
            proxy_send_timeout 60s;

            # エラー時のリトライ設定
            proxy_next_upstream error timeout http_502 http_503 http_504;
            proxy_next_upstream_tries 2;
            proxy_next_upstream_timeout 10s;
        }

        # メンテナンスページ
        error_page 502 503 504 /maintenance.html;
        location = /maintenance.html {
            root /var/www/shared;
            internal;
        }
    }
}
```

**メンテナンスページ（`/var/www/shared/maintenance.html`）:**
```html
<!DOCTYPE html>
<html lang="ja">
<head>
    <meta charset="UTF-8">
    <title>メンテナンス中</title>
    <style>
        body { font-family: sans-serif; text-align: center; padding: 50px; }
        h1 { color: #333; }
    </style>
</head>
<body>
    <h1>ただいまメンテナンス中です</h1>
    <p>しばらくお待ちください。</p>
</body>
</html>
```

</details>

## 問題8：ゼロダウンタイムデプロイ手順（チャレンジ）

3台のバックエンドサーバーで運用中に、ゼロダウンタイムでアプリケーションをアップデートする手順を、Nginxの設定変更を含めて記述してください。

**前提条件：**
- upstream は以下の通り：
  ```nginx
  upstream backend {
      server 192.168.1.101:3000 max_fails=3 fail_timeout=30s;
      server 192.168.1.102:3000 max_fails=3 fail_timeout=30s;
      server 192.168.1.103:3000 max_fails=3 fail_timeout=30s;
  }
  ```
- 各サーバーにSSH接続可能
- アプリケーションは `systemctl restart app` で再起動

<details>
<summary>ヒント</summary>

ローリングアップデートの手順を考えます。1台ずつ `down` フラグで除外 → アップデート → 復帰を繰り返します。設定変更のたびに `nginx -t` で構文チェックし、`systemctl reload nginx` でリロードします。

</details>

<details>
<summary>解答例</summary>

**ローリングアップデート手順:**

```bash
# ===== サーバー1のアップデート =====

# 1. サーバー1を down に設定
# upstream 内の server 192.168.1.101:3000 に down を追加
sudo sed -i 's/server 192.168.1.101:3000 max_fails=3/server 192.168.1.101:3000 down max_fails=3/' /etc/nginx/sites-available/example.com
sudo nginx -t && sudo systemctl reload nginx

# 2. サーバー1の処理中リクエストが完了するのを待つ（10秒程度）
sleep 10

# 3. サーバー1をアップデート
ssh 192.168.1.101 << 'REMOTE'
sudo systemctl stop app
# アプリケーションのアップデート処理
cd /var/www/app && git pull origin main
npm install --production
sudo systemctl start app
# ヘルスチェックエンドポイントで起動確認
curl -f http://localhost:3000/health || exit 1
REMOTE

# 4. サーバー1を復帰（down フラグを削除）
sudo sed -i 's/server 192.168.1.101:3000 down max_fails=3/server 192.168.1.101:3000 max_fails=3/' /etc/nginx/sites-available/example.com
sudo nginx -t && sudo systemctl reload nginx

# ===== サーバー2のアップデート =====
# （サーバー1と同じ手順を 192.168.1.102 に対して実施）

# 5. サーバー2を down に設定
sudo sed -i 's/server 192.168.1.102:3000 max_fails=3/server 192.168.1.102:3000 down max_fails=3/' /etc/nginx/sites-available/example.com
sudo nginx -t && sudo systemctl reload nginx
sleep 10

# 6. サーバー2をアップデート
ssh 192.168.1.102 << 'REMOTE'
sudo systemctl stop app
cd /var/www/app && git pull origin main
npm install --production
sudo systemctl start app
curl -f http://localhost:3000/health || exit 1
REMOTE

# 7. サーバー2を復帰
sudo sed -i 's/server 192.168.1.102:3000 down max_fails=3/server 192.168.1.102:3000 max_fails=3/' /etc/nginx/sites-available/example.com
sudo nginx -t && sudo systemctl reload nginx

# ===== サーバー3のアップデート =====
# （同様の手順を 192.168.1.103 に対して実施）

# 8. サーバー3を down に設定
sudo sed -i 's/server 192.168.1.103:3000 max_fails=3/server 192.168.1.103:3000 down max_fails=3/' /etc/nginx/sites-available/example.com
sudo nginx -t && sudo systemctl reload nginx
sleep 10

# 9. サーバー3をアップデート
ssh 192.168.1.103 << 'REMOTE'
sudo systemctl stop app
cd /var/www/app && git pull origin main
npm install --production
sudo systemctl start app
curl -f http://localhost:3000/health || exit 1
REMOTE

# 10. サーバー3を復帰
sudo sed -i 's/server 192.168.1.103:3000 down max_fails=3/server 192.168.1.103:3000 max_fails=3/' /etc/nginx/sites-available/example.com
sudo nginx -t && sudo systemctl reload nginx

echo "全サーバーのアップデートが完了しました"
```

**ポイント:**
- 常に最低2台が稼働しているため、サービスは中断しません
- 各サーバーの `down` → アップデート → 復帰を順番に行います
- `nginx -t` で構文チェックしてからリロードし、設定ミスを防ぎます
- ヘルスチェックでサーバーの起動を確認してから復帰させます
- 実務ではこの手順をAnsibleやシェルスクリプトで自動化します

</details>

# 第5章：ロードバランシング

> **この章のゴール**: Nginxを使って複数のバックエンドサーバーにリクエストを分散できるようになる。
> 分散アルゴリズムの違い、ヘルスチェック、セッション維持の仕組みを理解し、可用性の高いWeb構成を設計できるようになります。

---

## なぜロードバランシングを学ぶのか？

Webサービスのアクセスが増えると、1台のバックエンドサーバーでは処理しきれなくなります。**ロードバランシング（Load Balancing）** は、複数のサーバーにリクエストを分散し、処理能力の向上と障害耐性の確保を実現する技術です。

```
                                    ┌──────────────┐
                                ┌──▶│ Backend #1   │
                                │   │ (port 3001)  │
┌──────────┐    ┌──────────┐   │   └──────────────┘
│クライアント│───▶│  Nginx    │───┤   ┌──────────────┐
│          │◀───│  LB      │───┼──▶│ Backend #2   │
└──────────┘    └──────────┘   │   │ (port 3002)  │
                                │   └──────────────┘
                                │   ┌──────────────┐
                                └──▶│ Backend #3   │
                                    │ (port 3003)  │
                                    └──────────────┘
```

### ロードバランシングのメリット

| メリット | 説明 |
|---|---|
| **スケーラビリティ（拡張性）** | サーバーを追加するだけで処理能力を増やせる |
| **可用性（Availability）** | 1台が故障しても他のサーバーが処理を続行 |
| **メンテナンス容易性** | 1台ずつ停止してアップデートできる（ローリングアップデート） |
| **パフォーマンス向上** | リクエストが分散されレスポンス時間が短縮 |

## 5.1 upstream ブロックによるバックエンドサーバーグループの定義

### 基本構文

`upstream` ブロックで、ロードバランシング対象のサーバーグループを定義します。

```nginx
upstream backend {
    server 192.168.1.101:3000;
    server 192.168.1.102:3000;
    server 192.168.1.103:3000;
}

server {
    listen 80;
    server_name example.com;

    location / {
        proxy_pass http://backend;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
    }
}
```

### 同一サーバーの異なるポート

同じマシンで複数プロセスを動かす場合にも使えます。

```nginx
upstream app_cluster {
    server 127.0.0.1:3001;
    server 127.0.0.1:3002;
    server 127.0.0.1:3003;
    server 127.0.0.1:3004;
}
```

### UNIXソケット

```nginx
upstream gunicorn_workers {
    server unix:/run/gunicorn/worker1.sock;
    server unix:/run/gunicorn/worker2.sock;
}
```

## 5.2 分散アルゴリズム

Nginxはリクエストをどのバックエンドに送るか、複数のアルゴリズムから選択できます。

### Round Robin（ラウンドロビン）- デフォルト

リクエストを順番に各サーバーに割り振ります。特にディレクティブを指定しなければ、これがデフォルトです。

```nginx
upstream backend {
    # ラウンドロビン（デフォルト・指定不要）
    server 192.168.1.101:3000;
    server 192.168.1.102:3000;
    server 192.168.1.103:3000;
}
```

```
リクエスト 1 → Backend #1
リクエスト 2 → Backend #2
リクエスト 3 → Backend #3
リクエスト 4 → Backend #1  ← 一巡して戻る
リクエスト 5 → Backend #2
...
```

**適している場面**: バックエンドサーバーの性能がほぼ同じ場合

### Least Connections（最小接続数）

現在のアクティブ接続数が最も少ないサーバーにリクエストを送ります。

```nginx
upstream backend {
    least_conn;
    server 192.168.1.101:3000;
    server 192.168.1.102:3000;
    server 192.168.1.103:3000;
}
```

```
Backend #1: 接続数 5
Backend #2: 接続数 2  ← 次のリクエストはここに送られる
Backend #3: 接続数 8
```

**適している場面**: リクエストの処理時間にばらつきがある場合（重い処理と軽い処理が混在）

### IP Hash

クライアントのIPアドレスに基づいてサーバーを決定します。同じIPからのリクエストは常に同じサーバーに送られます。

```nginx
upstream backend {
    ip_hash;
    server 192.168.1.101:3000;
    server 192.168.1.102:3000;
    server 192.168.1.103:3000;
}
```

```
クライアント 203.0.113.10 → 常に Backend #2
クライアント 203.0.113.20 → 常に Backend #1
クライアント 203.0.113.30 → 常に Backend #3
```

**適している場面**: セッション維持が必要な場合（ログイン状態の保持など）

> **⚠️ よくある間違い**: `ip_hash` はクライアントのIPアドレスのクラスC（/24）をハッシュキーにします。同じネットワーク内の複数クライアントが同じサーバーに振り分けられる場合があり、偏りが生じることがあります。

### Hash（汎用ハッシュ）

任意のキーでハッシュを計算し、サーバーを決定します。

```nginx
upstream backend {
    hash $request_uri consistent;
    server 192.168.1.101:3000;
    server 192.168.1.102:3000;
    server 192.168.1.103:3000;
}
```

`consistent` パラメータを付けると、**コンシステントハッシュ（Consistent Hashing）** が使われ、サーバーの追加・削除時にキャッシュの無効化を最小限に抑えます。

**適している場面**: 同じURLへのリクエストを常に同じサーバーに送りたい場合（キャッシュ効率の向上）

### アルゴリズム比較表

| アルゴリズム | 分散方式 | セッション維持 | 推奨用途 |
|---|---|---|---|
| Round Robin | 順番に分散 | なし | 一般的なWebアプリ |
| Least Conn | 接続数が少ない方へ | なし | 処理時間にばらつきがあるAPI |
| IP Hash | IPアドレスベース | あり | ログイン機能のあるアプリ |
| Hash | 任意キーベース | キー依存 | キャッシュサーバーの分散 |

## 5.3 weight（重み付け）の設定

サーバーの性能差がある場合、**重み（weight）** を設定して振り分け比率を調整できます。

```nginx
upstream backend {
    server 192.168.1.101:3000 weight=5;  # 高性能サーバー：5倍の配分
    server 192.168.1.102:3000 weight=3;  # 中程度サーバー：3倍の配分
    server 192.168.1.103:3000 weight=1;  # 低性能サーバー：1倍の配分
}
```

```
9リクエストあたりの配分（おおよそ）:
  Backend #1 (weight=5): ★★★★★       → 約5リクエスト
  Backend #2 (weight=3): ★★★          → 約3リクエスト
  Backend #3 (weight=1): ★             → 約1リクエスト
```

> **📖 ポイント**: `weight` のデフォルト値は `1` です。指定しなければ全サーバーが同じ比率で処理します。

### 重み付けと least_conn の組み合わせ

```nginx
upstream backend {
    least_conn;
    server 192.168.1.101:3000 weight=5;
    server 192.168.1.102:3000 weight=3;
    server 192.168.1.103:3000 weight=1;
}
```

この場合、最小接続数のアルゴリズムに加えて重みも考慮されます。重みが大きいサーバーに対しては、接続数が多少多くてもリクエストが回ってきます。

## 5.4 ヘルスチェック（max_fails, fail_timeout）

### パッシブヘルスチェック

Nginxのオープンソース版は、リクエストの結果に基づく**パッシブヘルスチェック（Passive Health Check）** を提供します。

```nginx
upstream backend {
    server 192.168.1.101:3000 max_fails=3 fail_timeout=30s;
    server 192.168.1.102:3000 max_fails=3 fail_timeout=30s;
    server 192.168.1.103:3000 max_fails=3 fail_timeout=30s;
}
```

| パラメータ | デフォルト | 説明 |
|---|---|---|
| `max_fails` | 1 | この回数失敗したら、サーバーを一時的に利用不可とみなす |
| `fail_timeout` | 10s | 失敗カウントのリセット期間 & 利用不可とする期間 |

### ヘルスチェックの動作フロー

```
時刻 00:00 - Backend #1 にリクエスト → 失敗 (1/3)
時刻 00:05 - Backend #1 にリクエスト → 失敗 (2/3)
時刻 00:10 - Backend #1 にリクエスト → 失敗 (3/3)
           → Backend #1 を「利用不可」にマーク！
           → 以降30秒間は Backend #2, #3 にのみ振り分け

時刻 00:40 - 30秒経過 → Backend #1 を「利用可能」に戻す
           → 次のリクエストを Backend #1 に送って確認
           → 成功すれば完全復帰
           → 失敗すれば再び「利用不可」に
```

> **📖 ポイント**: `fail_timeout` は2つの役割を持ちます。(1) 失敗カウントをリセットする期間、(2) サーバーを利用不可にする期間です。例えば `fail_timeout=30s` なら、30秒以内に `max_fails` 回失敗したらそのサーバーを30秒間外します。

## 5.5 バックアップサーバーと down フラグ

### backup サーバー

`backup` を指定したサーバーは、他の全サーバーが利用不可になった場合にのみ使われます。

```nginx
upstream backend {
    server 192.168.1.101:3000;                 # メインサーバー1
    server 192.168.1.102:3000;                 # メインサーバー2
    server 192.168.1.200:3000 backup;          # バックアップサーバー
}
```

```
通常時:
  リクエスト → Backend #1 or #2（#1, #2 でラウンドロビン）
  Backend #3 はスタンバイ

#1, #2 両方ダウン時:
  リクエスト → Backend #3（バックアップが稼働）
```

### down フラグ

`down` を指定したサーバーは、リクエストの振り分け対象から完全に除外されます。メンテナンス時に使います。

```nginx
upstream backend {
    server 192.168.1.101:3000;
    server 192.168.1.102:3000 down;   # メンテナンス中：リクエストを送らない
    server 192.168.1.103:3000;
}
```

> **📖 ポイント**: `down` フラグは `ip_hash` アルゴリズムを使用している場合に特に便利です。サーバーを upstream から削除すると、ハッシュの計算結果が変わりセッションの振り分けが乱れますが、`down` フラグならハッシュ計算結果を維持したままサーバーを除外できます。

## 5.6 sticky session（セッション維持）

### セッション維持の課題

ロードバランシング環境では、同じユーザーのリクエストが異なるサーバーに振り分けられる可能性があります。サーバーにセッション情報を保存している場合、ログイン状態が失われる問題が発生します。

```
問題のあるケース:
  ユーザーA: ログインリクエスト → Backend #1（セッション作成）
  ユーザーA: 次のリクエスト     → Backend #2（セッションがない！→ ログアウト状態）
```

### 解決方法の比較

| 方法 | 説明 | メリット | デメリット |
|---|---|---|---|
| `ip_hash` | IPアドレスで固定 | 設定が簡単 | NAT環境で偏りが発生 |
| `hash` | Cookieなどで固定 | 柔軟 | 設定がやや複雑 |
| 外部セッションストア | Redis等に保存 | 最も信頼性が高い | インフラが追加で必要 |

### ip_hash によるセッション維持

```nginx
upstream backend {
    ip_hash;
    server 192.168.1.101:3000;
    server 192.168.1.102:3000;
    server 192.168.1.103:3000;
}
```

### hash でCookieを使ったセッション維持

```nginx
upstream backend {
    hash $cookie_sessionid consistent;
    server 192.168.1.101:3000;
    server 192.168.1.102:3000;
    server 192.168.1.103:3000;
}
```

この設定では、リクエストの `sessionid` Cookieの値に基づいてサーバーを決定します。同じセッションIDのリクエストは常に同じサーバーに送られます。

### 推奨: 外部セッションストアの利用

実務では、セッション情報をRedisやMemcachedなどの外部ストアに保存するのが最も堅牢な方法です。

```
┌──────────┐    ┌──────────┐    ┌──────────────┐
│クライアント│───▶│  Nginx    │───▶│ Backend #1   │──┐
└──────────┘    │  LB      │───▶│ Backend #2   │──┤
                └──────────┘───▶│ Backend #3   │──┤
                                └──────────────┘  │
                                                  ▼
                                          ┌──────────────┐
                                          │    Redis      │
                                          │ (セッション)  │
                                          └──────────────┘
                どのバックエンドに振り分けられても
                同じセッション情報を参照できる
```

## 5.7 実用例：Webアプリケーションの冗長構成

### 完全な設定例

以下は、Webアプリケーションの本番環境で使える冗長構成の設定例です。

```nginx
# バックエンドサーバーグループ
upstream webapp {
    # 最小接続数アルゴリズム（重み付き）
    least_conn;

    # メインサーバー（高性能マシン）
    server 192.168.1.101:3000 weight=5 max_fails=3 fail_timeout=30s;
    server 192.168.1.102:3000 weight=5 max_fails=3 fail_timeout=30s;

    # サブサーバー（標準マシン）
    server 192.168.1.103:3000 weight=3 max_fails=3 fail_timeout=30s;

    # バックアップサーバー（メイン全滅時に使用）
    server 192.168.1.200:3000 backup;
}

server {
    listen 80;
    server_name example.com;

    access_log /var/log/nginx/example.com.access.log;
    error_log  /var/log/nginx/example.com.error.log;

    # 静的ファイルはNginxが直接配信
    location /static/ {
        alias /var/www/shared/static/;
        expires 30d;
        add_header Cache-Control "public, immutable";
    }

    # アプリケーションリクエスト
    location / {
        proxy_pass http://webapp;

        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $scheme;

        proxy_connect_timeout 5s;
        proxy_read_timeout 60s;
        proxy_send_timeout 60s;

        # バックエンドエラー時に次のサーバーを試す
        proxy_next_upstream error timeout http_502 http_503 http_504;
        proxy_next_upstream_tries 3;
        proxy_next_upstream_timeout 10s;
    }

    # メンテナンスページ（全バックエンドダウン時）
    error_page 502 503 504 /maintenance.html;
    location = /maintenance.html {
        root /var/www/shared;
        internal;
    }
}
```

### proxy_next_upstream の詳細

`proxy_next_upstream` は、バックエンドとの通信が失敗した場合に、自動的に次のサーバーにリトライする設定です。

```nginx
# リトライする条件を指定
proxy_next_upstream error timeout http_502 http_503 http_504;

# 最大リトライ回数（0 = 制限なし）
proxy_next_upstream_tries 3;

# リトライ全体のタイムアウト
proxy_next_upstream_timeout 10s;
```

| 条件 | 説明 |
|---|---|
| `error` | サーバーとの接続確立エラー |
| `timeout` | タイムアウト |
| `http_502` | 502 Bad Gateway レスポンス |
| `http_503` | 503 Service Unavailable レスポンス |
| `http_504` | 504 Gateway Timeout レスポンス |

> **⚠️ よくある間違い**: `proxy_next_upstream` に `http_500` を含めると、アプリケーションのバグによる500エラーでもリトライが発生し、同じリクエストが複数のサーバーで処理されてしまいます。POSTリクエストでデータが重複する原因になるため、500は含めないのが安全です。

### ローリングアップデートの手順

```bash
# 1. サーバー1を down にマーク（設定変更）
#    server 192.168.1.101:3000 down;
sudo nano /etc/nginx/sites-available/example.com
sudo nginx -t && sudo systemctl reload nginx

# 2. サーバー1をアップデート
ssh 192.168.1.101 "sudo systemctl stop app && sudo apt upgrade -y && sudo systemctl start app"

# 3. サーバー1を復帰（down を削除）
sudo nano /etc/nginx/sites-available/example.com
sudo nginx -t && sudo systemctl reload nginx

# 4. サーバー2, 3も同様に繰り返す
```

## ポイントまとめ

| 項目 | 内容 |
|---|---|
| ロードバランシング | 複数サーバーにリクエストを分散し、処理能力と可用性を向上させる |
| upstream ブロック | バックエンドサーバーのグループを定義する |
| Round Robin | デフォルト。順番に振り分ける |
| least_conn | アクティブ接続数が最も少ないサーバーに送る |
| ip_hash | 同じクライアントIPを常に同じサーバーに送る |
| weight | サーバーの性能差に応じて振り分け比率を調整する |
| max_fails / fail_timeout | パッシブヘルスチェックで異常サーバーを自動除外する |
| backup | 全メインサーバーがダウンした時のみ使用する待機サーバー |
| proxy_next_upstream | 失敗時に自動で次のサーバーにリトライする |

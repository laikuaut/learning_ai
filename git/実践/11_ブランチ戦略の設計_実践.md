# 実践課題11：ブランチ戦略の設計 ★5

> **難易度**: ★★★★★（総合）
> **前提知識**: 第1章〜第8章（全章の知識を総合的に活用）
> **課題の種類**: 設計課題
> **学習目標**: プロジェクトの規模や要件に応じて適切なブランチ戦略を選択・設計し、チームに提案する力を養う

---

## 課題の説明

あなたは新しいプロジェクトのテックリードに任命されました。以下の3つのシナリオについて、それぞれ最適なブランチ戦略を設計し、実際にGitリポジトリ上でデモンストレーションしてください。

---

## シナリオ1：スタートアップのWebアプリ（GitHub Flow）

### 条件

- チーム: 3人（フロントエンド1名、バックエンド1名、フルスタック1名）
- デプロイ頻度: 週に3〜5回（頻繁にリリース）
- 環境: 本番（production）のみ
- CI/CD: GitHub Actions で自動テスト・自動デプロイ

### 要件

1. ブランチ戦略を設計してドキュメント化する
2. 実際にリポジトリを作成して、3人分の開発フローをシミュレーションする
3. 以下の作業を実行する
   - フロントエンド担当: `feature/landing-page` でLP作成
   - バックエンド担当: `feature/api-users` でAPI実装
   - 緊急修正: `hotfix/xss-fix` でセキュリティ修正

<details>
<summary>ステップガイド</summary>

```bash
mkdir /tmp/startup-project
cd /tmp/startup-project
git init

# ===== 初期セットアップ =====
cat << 'EOF' > README.md
# StartupApp

## ブランチ戦略: GitHub Flow

### ルール
1. `main` ブランチは常にデプロイ可能な状態を維持する
2. 新しい作業は `main` から `feature/xxx` ブランチを作成する
3. 定期的にコミットしてプッシュする
4. 準備ができたらプルリクエストを作成する
5. レビュー後に `main` にマージする
6. マージしたら即座にデプロイする

### ブランチ命名規則
- `feature/xxx` — 新機能
- `fix/xxx` — バグ修正
- `hotfix/xxx` — 緊急修正
- `docs/xxx` — ドキュメント更新
EOF

mkdir -p src/{frontend,backend}
echo "<!DOCTYPE html><html><body><h1>App</h1></body></html>" > src/frontend/index.html
echo '{"name": "startup-app", "version": "1.0.0"}' > package.json
git add .
git commit -m "chore: プロジェクト初期化"

# ===== フロントエンド担当: LP作成 =====
git switch -c feature/landing-page

cat << 'EOF' > src/frontend/landing.html
<!DOCTYPE html>
<html>
<head><title>StartupApp - LP</title></head>
<body>
  <header><h1>StartupApp へようこそ</h1></header>
  <main>
    <section class="hero">
      <p>革新的なタスク管理を体験しましょう。</p>
      <button>無料で始める</button>
    </section>
  </main>
</body>
</html>
EOF

git add src/frontend/landing.html
git commit -m "feat(frontend): ランディングページを作成"

cat << 'EOF' > src/frontend/landing.css
.hero { text-align: center; padding: 60px 20px; }
.hero button { background: #4CAF50; color: white; padding: 15px 30px; border: none; cursor: pointer; font-size: 18px; }
EOF

git add src/frontend/landing.css
git commit -m "feat(frontend): LPのスタイリングを追加"

# ===== main にマージ =====
git switch main
git merge --no-ff feature/landing-page -m "Merge: ランディングページを追加 (#1)"
git branch -d feature/landing-page

# ===== バックエンド担当: API実装 =====
git switch -c feature/api-users

cat << 'EOF' > src/backend/users.js
// ユーザーAPI
const users = [];

function createUser(name, email) {
  const user = { id: users.length + 1, name, email, createdAt: new Date() };
  users.push(user);
  return user;
}

function getUsers() {
  return [...users];
}

function getUserById(id) {
  return users.find(u => u.id === id) || null;
}

module.exports = { createUser, getUsers, getUserById };
EOF

git add src/backend/users.js
git commit -m "feat(api): ユーザーCRUD APIを実装"

git switch main
git merge --no-ff feature/api-users -m "Merge: ユーザーAPIを追加 (#2)"
git branch -d feature/api-users

# ===== 緊急修正: XSSバグ =====
git switch -c hotfix/xss-fix

cat << 'EOF' > src/frontend/sanitize.js
// XSS対策: HTMLエスケープ
function escapeHtml(str) {
  return str
    .replace(/&/g, '&amp;')
    .replace(/</g, '&lt;')
    .replace(/>/g, '&gt;')
    .replace(/"/g, '&quot;')
    .replace(/'/g, '&#039;');
}
module.exports = { escapeHtml };
EOF

git add src/frontend/sanitize.js
git commit -m "fix(security): XSS対策のエスケープ処理を追加"

git switch main
git merge --no-ff hotfix/xss-fix -m "Merge: XSS修正 (#3)"
git branch -d hotfix/xss-fix

# ===== 最終確認 =====
git log --oneline --graph --all
```

## シナリオ2：企業向けWebサービス（Git Flow）

### 条件

- チーム: 8人（4人x2チーム）
- デプロイ頻度: 月に1回（計画的なリリース）
- 環境: 開発（dev）、ステージング（staging）、本番（production）
- テスト: QAチームが手動テストを実施

### 要件

1. Git Flow のブランチ構造を構築する
2. リリースフローをシミュレーションする
3. 以下のブランチを使い分ける
   - `main` — 本番環境
   - `develop` — 開発統合
   - `feature/xxx` — 機能開発
   - `release/x.x.x` — リリース準備
   - `hotfix/xxx` — 緊急修正

<details>
<summary>ステップガイド</summary>

```bash
mkdir /tmp/enterprise-project
cd /tmp/enterprise-project
git init

# ===== 初期セットアップ =====
echo "# EnterpriseApp v1.0.0" > README.md
git add README.md
git commit -m "chore: プロジェクト初期化"
git tag v1.0.0

# develop ブランチを作成
git switch -c develop

# ===== 機能開発（feature → develop） =====
# 機能1: 検索機能
git switch -c feature/search
echo "function search(query) { /* ... */ }" > search.js
git add search.js
git commit -m "feat: 検索機能を実装"
git switch develop
git merge --no-ff feature/search -m "Merge feature/search into develop"
git branch -d feature/search

# 機能2: 通知機能
git switch -c feature/notification
echo "function notify(message) { /* ... */ }" > notification.js
git add notification.js
git commit -m "feat: 通知機能を実装"
git switch develop
git merge --no-ff feature/notification -m "Merge feature/notification into develop"
git branch -d feature/notification

# ===== リリース準備（release ブランチ） =====
git switch -c release/1.1.0

# リリースに向けたバージョン更新
echo "# EnterpriseApp v1.1.0" > README.md
git add README.md
git commit -m "chore: バージョンを1.1.0に更新"

# QAテスト中にバグ発見 → release ブランチで修正
echo "function search(query) { if (!query) return []; /* ... */ }" > search.js
git add search.js
git commit -m "fix: 空クエリのバリデーションを追加"

# ===== リリース（main と develop にマージ） =====
# main にマージ
git switch main
git merge --no-ff release/1.1.0 -m "Merge release/1.1.0: v1.1.0 リリース"
git tag v1.1.0

# develop にもマージ（リリースブランチの修正を取り込む）
git switch develop
git merge --no-ff release/1.1.0 -m "Merge release/1.1.0 back into develop"
git branch -d release/1.1.0

# ===== 本番バグの緊急修正（hotfix） =====
git switch main
git switch -c hotfix/crash-fix
echo "// crash fix applied" >> search.js
git add search.js
git commit -m "hotfix: 検索時のクラッシュを修正"

# main にマージ
git switch main
git merge --no-ff hotfix/crash-fix -m "Merge hotfix/crash-fix"
git tag v1.1.1

# develop にもマージ
git switch develop
git merge --no-ff hotfix/crash-fix -m "Merge hotfix/crash-fix into develop"
git branch -d hotfix/crash-fix

# ===== 最終確認 =====
git log --oneline --graph --all
git tag -l
```

## シナリオ3：戦略の比較と提案書の作成

### 要件

以下のフォーマットで、ブランチ戦略の比較提案書を作成してください。

<details>
<summary>提案書テンプレートと解答例</summary>

```markdown
# ブランチ戦略 比較提案書

## 1. GitHub Flow

### 概要
main ブランチから feature ブランチを作成し、PR でマージする。

### 向いているプロジェクト
- デプロイ頻度が高い（週に複数回）
- 少人数チーム（2〜5人）
- CI/CD が整備されている
- 環境が少ない（本番のみ、または本番 + ステージング）

### メリット
- シンプルで覚えやすい
- マージまでの時間が短い
- 素早いフィードバックループ

### デメリット
- リリースの厳密な管理が難しい
- 大人数だと main が不安定になりやすい

### ブランチ構成
main ── feature/xxx
     └── hotfix/xxx

---

## 2. Git Flow

### 概要
main と develop の2本の長寿命ブランチを軸に、
feature / release / hotfix ブランチを使い分ける。

### 向いているプロジェクト
- デプロイ頻度が低い（月1回〜四半期に1回）
- 中〜大人数チーム（5人以上）
- QA（品質保証）プロセスがある
- 複数環境（dev / staging / production）

### メリット
- リリースの品質管理がしやすい
- 並行開発とリリース準備を同時に進められる
- hotfix のルートが明確

### デメリット
- ブランチが多くて複雑
- マージの手順が多い
- develop と main の乖離が起きやすい

### ブランチ構成
main ──────────────────── 本番
  └── hotfix/xxx        緊急修正
develop ─────────────── 開発統合
  ├── feature/xxx       機能開発
  └── release/x.x.x    リリース準備

---

## 3. トランクベース開発（Trunk-Based Development）

### 概要
全員が main（trunk）に直接コミット、または非常に短命な
ブランチ（1日以内にマージ）で開発する。

### 向いているプロジェクト
- CI/CD とテスト自動化が高度に整備されている
- 少数精鋭のシニアチーム
- フィーチャーフラグ（Feature Flag）を活用している

### メリット
- マージの競合が最小限
- 常に全員が最新コードで作業
- デプロイが最速

### デメリット
- 高いテストカバレッジが必須
- フィーチャーフラグの管理が必要
- ジュニア開発者には難易度が高い

---

## 推奨事項

| プロジェクト規模 | 推奨ワークフロー |
|---|---|
| 個人〜3人 | GitHub Flow |
| 4〜10人（アジャイル） | GitHub Flow + 保護ルール |
| 10人以上（計画的リリース） | Git Flow |
| シニアチーム（高自動化） | トランクベース開発 |
```

</details>

<summary>解答例（シナリオ1〜2の一括実行スクリプト）</summary>

```bash
#!/bin/bash
# ブランチ戦略の実践デモ

echo "===== シナリオ1: GitHub Flow ====="
rm -rf /tmp/startup-project
mkdir /tmp/startup-project && cd /tmp/startup-project
git init

echo "# StartupApp" > README.md
mkdir -p src/{frontend,backend}
echo "<h1>App</h1>" > src/frontend/index.html
git add . && git commit -m "chore: プロジェクト初期化"

# Feature 1
git switch -c feature/landing-page
echo "<h1>LP</h1>" > src/frontend/landing.html
git add . && git commit -m "feat(frontend): LPを作成"
git switch main
git merge --no-ff feature/landing-page -m "Merge: LP (#1)"
git branch -d feature/landing-page

# Feature 2
git switch -c feature/api-users
echo "function createUser(){}" > src/backend/users.js
git add . && git commit -m "feat(api): ユーザーAPIを実装"
git switch main
git merge --no-ff feature/api-users -m "Merge: API (#2)"
git branch -d feature/api-users

# Hotfix
git switch -c hotfix/xss-fix
echo "function escapeHtml(){}" > src/frontend/sanitize.js
git add . && git commit -m "fix: XSS対策を追加"
git switch main
git merge --no-ff hotfix/xss-fix -m "Merge: XSS fix (#3)"
git branch -d hotfix/xss-fix

echo "--- GitHub Flow 完了 ---"
git log --oneline --graph

echo ""
echo "===== シナリオ2: Git Flow ====="
rm -rf /tmp/enterprise-project
mkdir /tmp/enterprise-project && cd /tmp/enterprise-project
git init

echo "# EnterpriseApp" > README.md
git add . && git commit -m "chore: 初期化"
git tag v1.0.0

git switch -c develop

# Feature
git switch -c feature/search
echo "function search(){}" > search.js
git add . && git commit -m "feat: 検索機能"
git switch develop
git merge --no-ff feature/search -m "Merge feature/search"
git branch -d feature/search

# Release
git switch -c release/1.1.0
echo "v1.1.0" > VERSION
git add . && git commit -m "chore: v1.1.0準備"

git switch main
git merge --no-ff release/1.1.0 -m "Release v1.1.0"
git tag v1.1.0

git switch develop
git merge --no-ff release/1.1.0 -m "Merge release back"
git branch -d release/1.1.0

# Hotfix
git switch main
git switch -c hotfix/crash-fix
echo "// fix" >> search.js
git add . && git commit -m "hotfix: クラッシュ修正"
git switch main
git merge --no-ff hotfix/crash-fix -m "Merge hotfix"
git tag v1.1.1
git switch develop
git merge --no-ff hotfix/crash-fix -m "Merge hotfix to develop"
git branch -d hotfix/crash-fix

echo "--- Git Flow 完了 ---"
git log --oneline --graph --all
echo "Tags:"
git tag -l

# ===== 後片付け =====
# rm -rf /tmp/startup-project /tmp/enterprise-project
```

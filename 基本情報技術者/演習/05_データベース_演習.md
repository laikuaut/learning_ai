## 基本問題

### 問題1：用語の理解
関係データベースにおいて、以下の用語の正式名称を答えてください。

(1) 行（レコード）
(2) 列（フィールド）
(3) 表（テーブル）

<details>
<summary>解答</summary>

(1) **タプル（Tuple）**
(2) **属性（Attribute）**
(3) **リレーション（Relation）**
</details>

### 問題2：キーの識別
以下の社員テーブルにおいて、主キー、外部キーをそれぞれ答えてください。

```
社員テーブル（社員番号, 氏名, 部署コード, メールアドレス）
部署テーブル（部署コード, 部署名）
```

<details>
<summary>解答</summary>

- 主キー：**社員番号**（各行を一意に識別する）
- 外部キー：**部署コード**（部署テーブルの主キーを参照している）

※ メールアドレスも候補キー（一意に識別可能）ですが、主キーとして選ばれたのは社員番号です。
</details>

### 問題3：SQL（SELECT文の基本）
以下の社員テーブルから、給与が 400000 以上の社員の氏名と給与を取得するSQLを書いてください。

<details>
<summary>解答</summary>

```sql
SELECT 氏名, 給与
FROM 社員
WHERE 給与 >= 400000;
```
</details>

### 問題4：四択問題
SQL の SELECT 文の実行順序として正しいものはどれですか。

ア. SELECT → FROM → WHERE → GROUP BY → HAVING → ORDER BY
イ. FROM → WHERE → GROUP BY → HAVING → SELECT → ORDER BY
ウ. FROM → SELECT → WHERE → GROUP BY → HAVING → ORDER BY
エ. SELECT → FROM → WHERE → ORDER BY → GROUP BY → HAVING

<details>
<summary>解答</summary>

答え：**イ. FROM → WHERE → GROUP BY → HAVING → SELECT → ORDER BY**

SELECT は ORDER BY の直前に実行されます。この実行順序は試験で頻出です。
</details>

### 問題5：正規化の判定
以下のテーブルは第何正規形ですか。

```
注文テーブル（注文番号, 顧客名, 商品名, 数量）
主キー：{注文番号, 商品名}

関数従属性：
{注文番号, 商品名} → 数量
{注文番号} → 顧客名
```

<details>
<summary>ヒント</summary>
主キーの一部（注文番号のみ）で決まる属性があるかどうかを確認してください。
</details>

<details>
<summary>解答</summary>

答え：**第1正規形**

理由：{注文番号} → 顧客名 という**部分関数従属**が存在するため、第2正規形を満たしていません。顧客名は複合主キーの一部（注文番号）だけで決まっています。

第2正規形にするには、顧客名を別テーブルに分離します：
- 注文テーブル（注文番号, 顧客名）
- 注文明細テーブル（注文番号, 商品名, 数量）
</details>

## 応用問題

### 問題6：SQL（集約関数とGROUP BY）
以下の社員テーブルから、部署ごとの社員数と平均給与を求めるSQLを書いてください。ただし、平均給与が 350000 以上の部署のみ表示してください。

<details>
<summary>解答</summary>

```sql
SELECT 部署ID, COUNT(*) AS 社員数, AVG(給与) AS 平均給与
FROM 社員
GROUP BY 部署ID
HAVING AVG(給与) >= 350000;
```

WHERE は個々の行に対する条件、HAVING はグループに対する条件です。
</details>

---

### 問題7：SQL（結合）
社員テーブルと部署テーブルを内部結合して、社員の氏名と所属する部署名を取得するSQLを書いてください。

```
社員（社員番号, 氏名, 部署ID, 給与）
部署（部署ID, 部署名, 所在地）
```

<details>
<summary>解答</summary>

```sql
SELECT 社員.氏名, 部署.部署名
FROM 社員
INNER JOIN 部署 ON 社員.部署ID = 部署.部署ID;
```
</details>

---

### 問題8：SQL（副問合せ）
全社員の平均給与よりも給与が高い社員の氏名と給与を取得するSQLを書いてください。

<details>
<summary>解答</summary>

```sql
SELECT 氏名, 給与
FROM 社員
WHERE 給与 > (SELECT AVG(給与) FROM 社員);
```

副問合せ（サブクエリ）で全社員の平均給与を算出し、それとWHERE句で比較しています。
</details>

### 問題9：四択問題（ACID特性）
トランザクションの ACID 特性のうち、「トランザクションの処理は全て実行されるか、全て取り消されるかのどちらかである」という性質はどれですか。

ア. 原子性（Atomicity）
イ. 一貫性（Consistency）
ウ. 独立性（Isolation）
エ. 耐久性（Durability）

<details>
<summary>解答</summary>

答え：**ア. 原子性（Atomicity）**

- 原子性：全部実行 or 全部取消し
- 一貫性：データの整合性が保たれる
- 独立性：トランザクション同士が干渉しない
- 耐久性：コミット済みデータは永続的に保持
</details>

### 問題10：正規化の実施
以下のテーブルを第3正規形まで正規化してください。

```
受注テーブル（受注番号, 顧客ID, 顧客名, 商品コード, 商品名, 単価, 数量）
主キー：{受注番号, 商品コード}

関数従属性：
{受注番号} → 顧客ID
{顧客ID} → 顧客名
{商品コード} → 商品名, 単価
{受注番号, 商品コード} → 数量
```

<details>
<summary>解答</summary>

**第2正規形（部分関数従属の排除）：**

部分関数従属を排除して3テーブルに分離します。

- 受注（受注番号, 顧客ID, 顧客名）← {受注番号}→顧客ID, 顧客名
- 商品（商品コード, 商品名, 単価）← {商品コード}→商品名, 単価
- 受注明細（受注番号, 商品コード, 数量）

**第3正規形（推移的関数従属の排除）：**

受注テーブルに推移的従属があります：受注番号 → 顧客ID → 顧客名

- **受注（受注番号, 顧客ID）**
- **顧客（顧客ID, 顧客名）**
- **商品（商品コード, 商品名, 単価）**
- **受注明細（受注番号, 商品コード, 数量）**
</details>

## チャレンジ問題

### 問題11：デッドロック
以下の状況でデッドロックが発生するかどうか判定し、理由を説明してください。

```
トランザクションT1: テーブルAにロック → テーブルBにロック要求
トランザクションT2: テーブルBにロック → テーブルAにロック要求
```

<details>
<summary>解答</summary>

**デッドロックが発生します。**

T1はテーブルAをロック済みでテーブルBのロック解除を待っています。
T2はテーブルBをロック済みでテーブルAのロック解除を待っています。
互いに相手のロック解除を待ち合う循環待ちが発生しています。

**対策：** ロック順序を統一する（例：常にA→Bの順でロック）ことで防止できます。
</details>

### 問題12：SQL（複合問題）
以下のテーブルを使って、東京に所在する部署に所属し、かつ給与が全社員の平均以上の社員の氏名、給与、部署名を給与の降順で取得するSQLを書いてください。

```
社員（社員番号, 氏名, 部署ID, 給与）
部署（部署ID, 部署名, 所在地）
```

<details>
<summary>解答</summary>

```sql
SELECT 社員.氏名, 社員.給与, 部署.部署名
FROM 社員
INNER JOIN 部署 ON 社員.部署ID = 部署.部署ID
WHERE 部署.所在地 = '東京'
  AND 社員.給与 >= (SELECT AVG(給与) FROM 社員)
ORDER BY 社員.給与 DESC;
```

結合 + 副問合せ + ORDER BY を組み合わせた複合問題です。
</details>

### 問題13：障害回復
以下の状況で、障害発生後に T1 と T2 それぞれに対してどのような回復処理が必要ですか。

```
時間軸: ──→
チェックポイント → T1開始 → T1コミット → T2開始 → 障害発生
```

<details>
<summary>解答</summary>

- **T1：ロールフォワード（前進復帰）が必要**
  - T1はチェックポイント後にコミット済みなので、ログを使って変更を再実行します。

- **T2：ロールバック（後退復帰）が必要**
  - T2は障害発生時に未コミットなので、ログを使って変更を取り消します。

チェックポイント以降にコミットしたトランザクションはロールフォワード、未コミットのトランザクションはロールバックします。
</details>

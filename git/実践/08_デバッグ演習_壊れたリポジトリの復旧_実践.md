# 実践課題08：デバッグ演習 ─ 壊れたリポジトリの復旧 ★4

> **難易度**: ★★★★☆（上級）
> **前提知識**: 第2章（基本操作）、第3章（ブランチとマージ）、第8章（高度なGit操作とトラブルシューティング）
> **課題の種類**: デバッグ
> **学習目標**: 実務で遭遇しがちなGitのトラブルを自力で解決する力を養う

---

## 課題の説明

以下のスクリプトは、**意図的に7つのトラブルを含んだGitリポジトリ** を構築します。
スクリプトを実行した後、各トラブルを診断して復旧してください。

### 進め方

1. 下のセットアップスクリプトを実行する
2. 各トラブルの症状を確認する
3. 適切な Git コマンドで1つずつ復旧する
4. すべて復旧できれば完了

---

## セットアップスクリプト

以下を `setup_broken_repo.sh` として保存し、実行してください。

```bash
#!/bin/bash
# 壊れたリポジトリのセットアップスクリプト

set -e

REPO_DIR="/tmp/broken-repo"
rm -rf "$REPO_DIR"
mkdir "$REPO_DIR"
cd "$REPO_DIR"
git init

# === 正常なコミットを作成 ===
echo "# プロジェクト" > README.md
git add README.md
git commit -m "初回コミット"

echo "機能A" > feature_a.txt
git add feature_a.txt
git commit -m "機能Aを追加"

echo "機能B" > feature_b.txt
git add feature_b.txt
git commit -m "機能Bを追加"

# === トラブル1: detached HEAD 状態にする ===
git checkout HEAD~1

# === トラブル2: 追跡されていないファイルを大量に散乱させる ===
for i in $(seq 1 20); do
  echo "temp $i" > "temp_file_$i.tmp"
done

# === トラブル3: マージコンフリクトを発生させたまま放置 ===
git checkout -b conflict-branch HEAD~1
echo "conflict-branch の変更" > shared.txt
git add shared.txt
git commit -m "conflict-branch: sharedを変更"

git checkout main
echo "main の変更" > shared.txt
git add shared.txt
git commit -m "main: sharedを変更"

git merge conflict-branch || true  # コンフリクトが発生するが続行

# === トラブル4: .gitignore がないためビルド成果物が追跡候補に ===
mkdir -p build dist node_modules/.cache
echo "compiled" > build/output.js
echo "bundled" > dist/bundle.js
echo "cache" > node_modules/.cache/data.json

# === トラブル5: 巨大ファイルがステージングされている ===
dd if=/dev/zero of=huge_file.bin bs=1M count=5 2>/dev/null
git add huge_file.bin

# === トラブル6: 秘密情報がコミットされている ===
echo "DB_PASSWORD=super_secret_123" > .env
echo "AWS_KEY=AKIAIOSFODNN7EXAMPLE" >> .env
git add .env
# （まだコミットはしていないが、ステージされている状態）

echo ""
echo "========================================="
echo "壊れたリポジトリの構築が完了しました。"
echo "リポジトリの場所: $REPO_DIR"
echo "cd $REPO_DIR で移動して復旧してください。"
echo "========================================="
echo ""
echo "トラブル一覧："
echo "1. detached HEAD 状態になっている"
echo "2. 不要な一時ファイルが20個散乱している"
echo "3. マージコンフリクトが未解決のまま放置されている"
echo "4. ビルド成果物が git status に表示されている"
echo "5. 巨大ファイル(5MB)がステージングされている"
echo "6. 秘密情報(.env)がステージングされている"
echo "7. conflict-branch が不要なまま残っている"
```

---

## トラブルの症状と復旧目標

| # | 症状 | 復旧目標 |
|---|---|---|
| 1 | `git status` で "HEAD detached" と表示される | `main` ブランチに戻る |
| 2 | 20個の `.tmp` ファイルが散乱している | すべて削除する |
| 3 | "Unmerged paths" がある | コンフリクトを解消してマージを完了する |
| 4 | `build/`, `dist/`, `node_modules/` が表示される | `.gitignore` で除外する |
| 5 | 5MBの巨大ファイルがステージされている | ステージングから除外する |
| 6 | `.env`（秘密情報）がステージされている | ステージングから除外して `.gitignore` に追加する |
| 7 | 不要な `conflict-branch` が残っている | マージ後に削除する |

## ヒント

<details>
<summary>ヒント（軽め）：各トラブルに使うコマンドの種類</summary>

| # | 使うコマンド |
|---|---|
| 1 | `git switch` または `git checkout` |
| 2 | `git clean` |
| 3 | ファイル編集 → `git add` → `git commit` |
| 4 | `.gitignore` ファイルの作成 |
| 5 | `git reset` または `git restore --staged` |
| 6 | `git reset` + `.gitignore` |
| 7 | `git branch -d` または `git branch -D` |

</details>

<details>
<summary>ヒント（詳細）：各トラブルの復旧手順の概要</summary>

1. **detached HEAD**: `git switch main` で main ブランチに切り替える。コンフリクト中のため先にマージを処理する必要があるかもしれません。
2. **一時ファイル**: `git clean -f` で追跡されていないファイルを削除。`-n` で事前確認。
3. **コンフリクト**: ファイルを編集してマーカーを除去 → `git add` → `git commit`
4. **ビルド成果物**: `.gitignore` に `build/`, `dist/`, `node_modules/` を追記
5. **巨大ファイル**: `git restore --staged huge_file.bin` でステージング解除 → `.gitignore` に追加
6. **秘密情報**: `git restore --staged .env` → `.gitignore` に `.env` を追加
7. **不要ブランチ**: マージ完了後に `git branch -d conflict-branch`

</details>

## 解答例

<details>
<summary>全7つのトラブル復旧手順</summary>

```bash
cd /tmp/broken-repo

# ===== 現状確認 =====
git status
# HEAD detached, unmerged paths, untracked files が見える

# ===== トラブル3: コンフリクト解消（最初に対処すべき）=====
# コンフリクトマーカーを確認
cat shared.txt
# <<<<<<< HEAD
# main の変更
# =======
# conflict-branch の変更
# >>>>>>> conflict-branch

# 両方の変更を統合
cat << 'EOF' > shared.txt
main の変更
conflict-branch の変更
EOF

git add shared.txt
git commit -m "コンフリクト解消：shared.txt を統合"

# ===== トラブル1: detached HEAD の解消 =====
# main ブランチに切り替え
git switch main
# もし main がないと言われたら:
# git switch -c main

git branch  # main にいることを確認

# ===== トラブル5: 巨大ファイルのステージング解除 =====
git restore --staged huge_file.bin
# または: git reset HEAD huge_file.bin

# ===== トラブル6: 秘密情報のステージング解除 =====
git restore --staged .env
# または: git reset HEAD .env

# ===== トラブル4: .gitignore の作成 =====
cat << 'EOF' > .gitignore
# ビルド成果物
build/
dist/

# 依存パッケージ
node_modules/

# 環境変数（秘密情報）
.env

# 巨大ファイル
*.bin

# 一時ファイル
*.tmp
EOF

git add .gitignore
git commit -m ".gitignoreを追加：ビルド成果物・秘密情報・一時ファイルを除外"

# ===== トラブル2: 一時ファイルの削除 =====
# 事前確認（削除されるファイルの一覧を表示）
git clean -n
# 出力例: Would remove temp_file_1.tmp ... huge_file.bin

# .gitignore で無視されるファイルも含めて削除
git clean -f -X  # .gitignore にマッチするファイルを削除
git clean -f      # 追跡外のファイルを削除

# 確認
ls *.tmp 2>/dev/null || echo "一時ファイルなし"
ls *.bin 2>/dev/null || echo "巨大ファイルなし"

# ===== トラブル7: 不要ブランチの削除 =====
# マージ済みか確認
git branch --merged
# conflict-branch がマージ済みなら:
git branch -d conflict-branch
# マージされていない場合は強制削除:
# git branch -D conflict-branch

# ===== 最終確認 =====
echo "--- git status ---"
git status
# On branch main
# nothing to commit, working tree clean

echo "--- git branch ---"
git branch
# * main

echo "--- git log ---"
git log --oneline --graph --all
```

</details>

<details>
<summary>復旧済みの状態の確認ポイント</summary>

以下のすべてが満たされていれば復旧完了です。

```bash
# 1. main ブランチにいる
git branch --show-current
# 出力: main

# 2. 未追跡ファイルがない
git status --porcelain
# （何も出力されなければOK）

# 3. .tmp ファイルがない
ls *.tmp 2>/dev/null
# （何も出力されなければOK）

# 4. .gitignore が存在する
cat .gitignore
# build/, dist/, node_modules/, .env, *.bin, *.tmp が含まれている

# 5. 巨大ファイルがステージされていない
git diff --staged --name-only
# （何も出力されなければOK）

# 6. .env がステージされていない
git ls-files --cached | grep ".env"
# （何も出力されなければOK）

# 7. conflict-branch が存在しない
git branch | grep conflict
# （何も出力されなければOK）
```

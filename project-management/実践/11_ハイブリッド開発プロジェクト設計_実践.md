# 実践課題11：ハイブリッド開発プロジェクト設計 ★4

> **難易度**: ★★★★☆（上級）
> **前提知識**: 第1章〜第8章（全章の知識を活用）
> **課題の種類**: 設計課題
> **学習目標**: ウォーターフォールとアジャイルの特性を理解し、プロジェクトの特性に合わせたハイブリッド開発プロセスを設計できるようになる

---

## 完成イメージ

以下のような開発プロセス設計書を作成します。

```
=============================================
  開発プロセス設計書
  プロジェクト: ○○システム
=============================================

■ 採用方式: ハイブリッド（WF + アジャイル）
  - フェーズ1（要件定義・基本設計）: ウォーターフォール
    → 契約・承認が必要な上流工程は計画駆動で進める
  - フェーズ2（詳細設計・開発・テスト）: スクラム（2週間スプリント）
    → 変化に対応しやすい開発工程はアジャイルで進める
  - フェーズ3（移行・リリース）: ウォーターフォール
    → リリース計画は厳密な手順管理が必要
```

---

## 課題の要件

以下のシナリオを読み、最適な開発プロセスを設計してください。

### シナリオ

あなたは大手保険会社のIT部門のPMです。以下のプロジェクトの開発プロセスを設計する役割を担っています。

**プロジェクト概要：**
- 内容：保険契約管理システムの大規模刷新
- 規模：開発チーム15名（社内8名 + 協力会社7名）
- 期間：18ヶ月
- 予算：1.5億円
- 現行システム：20年前のCOBOLベースのメインフレーム
- 新システム：Java + クラウド（AWS）

**プロジェクトの特徴：**

| 特性 | 詳細 |
|------|------|
| 規制 | 保険業法・金融庁のガイドラインに準拠が必要 |
| 契約 | 協力会社との準委任契約＋一部請負契約 |
| データ | 契約者100万件のデータ移行が必須 |
| 要件の確定度 | コア業務ロジック: 高（現行踏襲）、新規UI/UX: 低（未確定） |
| ステークホルダー | 経営層、業務部門、コンプライアンス部門、IT部門、協力会社 |
| 品質要件 | 金融システムとしての高い信頼性（99.99%稼働率） |
| リスク | COBOLからの業務ロジック移植の正確性が最大のリスク |

### 作成すべきもの

1. **開発方式の選定と根拠**: WF / アジャイル / ハイブリッドのいずれを採用するか、根拠を示す
2. **フェーズ設計**: 各フェーズの進め方、期間、成果物の定義
3. **チーム体制設計**: 15名の役割分担と組織構造
4. **ガバナンス設計**: 意思決定プロセス、品質ゲート、報告体制
5. **リスク対応計画**: 上位5つのリスクと対応策

## ステップガイド

<details>
<summary>ステップ1：WF/アジャイル/ハイブリッドの選定基準を整理する</summary>

プロジェクトの特性に基づいて、各方式の適合度を評価します。

```
評価マトリクス：

  特性                    WF寄り      アジャイル寄り
  ─────────────────────────────────────────────
  規制準拠の厳しさ        ★★★★★      ★★
  契約形態（請負含む）    ★★★★        ★★
  要件の確定度（コア）    ★★★★★      ★★
  要件の確定度（UI/UX）   ★★          ★★★★★
  チーム規模（15名）      ★★★         ★★★
  期間（18ヶ月）          ★★★★        ★★★
  変更の可能性            ★★          ★★★★
  品質要件の厳しさ        ★★★★★      ★★

  → ハイブリッドが最適（工程によって使い分ける）
```

</details>

<details>
<summary>ステップ2：フェーズを設計する</summary>

ハイブリッドの場合、各フェーズの特性に応じて方式を切り替えます。

```
考え方：
  ・契約上の「承認」が必要な工程 → WF（計画駆動）
  ・不確実性が高く反復が有効な工程 → アジャイル（適応型）
  ・厳密な手順管理が必要な工程 → WF（計画駆動）

典型的なハイブリッドパターン：

パターン1: 上流WF + 中流アジャイル + 下流WF
  要件定義 → 基本設計 → [アジャイル開発] → テスト → 移行

パターン2: WFの中にアジャイルを組み込む
  WFの各フェーズの中で、短いイテレーションを回す

パターン3: アジャイルの中にWFゲートを組み込む
  スクラムで進めつつ、重要な節目でWF的な承認ゲートを設ける
```

</details>

<details>
<summary>ステップ3：チーム体制を設計する</summary>

15名のチームを機能別または機能横断型に編成します。

```
体制案の検討ポイント：
  - 社内メンバーと協力会社メンバーの役割分担
  - コア業務ロジック（高リスク）は社内メンバーが主導
  - UI/UX（変更が多い）はアジャイルチームで進める
  - データ移行は専門チームが必要

大規模アジャイルの場合、SAFe/LeSSなどのスケーリング
フレームワークの適用も検討する
```

</details>

## 解答例

<details>
<summary>解答例（初心者向け）</summary>

### 1. 開発方式の選定

```
採用方式: ハイブリッド（上流WF + 中流アジャイル + 下流WF）

選定根拠：

WFを採用すべき理由:
  - 保険業法・金融庁ガイドラインの準拠に厳密な文書化が必要
  - 協力会社との請負契約では、スコープの事前確定が求められる
  - COBOLからの業務ロジック移植は要件が明確（現行踏襲）
  - 100万件のデータ移行は計画的に進める必要がある

アジャイルを採用すべき理由:
  - UI/UXの要件が未確定であり、反復的に確認したい
  - 業務部門のフィードバックを早期に得たい
  - 18ヶ月のプロジェクトで途中の軌道修正が不可避

→ 工程の特性に応じて使い分けるハイブリッドが最適
```

### 2. フェーズ設計

```
フェーズ          期間      方式         主な成果物
─────────────────────────────────────────────────
Phase 0: 準備     1ヶ月    WF          プロジェクト計画書
Phase 1: 要件定義  3ヶ月    WF          要件定義書、承認書
Phase 2: 基本設計  2ヶ月    WF          基本設計書、承認書
Phase 3: 開発      7ヶ月    スクラム     動くソフトウェア
Phase 4: 総合テスト 2ヶ月    WF          テスト結果報告書
Phase 5: 移行      2ヶ月    WF          移行計画書、移行結果
Phase 6: 並行稼働  1ヶ月    WF          並行稼働結果報告書
─────────────────────────────────────────────────
合計: 18ヶ月

Phase 3（開発）の詳細:
  スプリント期間: 2週間
  スプリント数: 14回
  ベロシティ目標: チームA 30SP、チームB 25SP
  デモ: 毎スプリント末にPO + 業務部門代表にデモ
  品質ゲート: 4スプリントごとにコンプライアンス部門レビュー
```

### 3. チーム体制

```
プロジェクト全体:
  PMO: PM 1名（社内）
  アーキテクト: 1名（社内）

チームA: コア業務ロジック（社内主導）
  リーダー: 社内SE 1名
  開発者: 社内SE 2名 + 協力会社PG 2名
  担当: 契約管理、保険料計算、査定処理

チームB: UI/UX + API（アジャイル運用）
  リーダー: 社内SE 1名
  開発者: 社内SE 1名 + 協力会社PG 2名
  担当: 画面開発、API連携、ユーザー体験

チームC: データ移行 + インフラ
  リーダー: 協力会社SE 1名
  開発者: 社内SE 1名 + 協力会社PG 1名
  担当: データ移行、クラウド環境構築、性能チューニング

QAチーム:
  テスター: 協力会社 1名（Phase 3後半から増員予定）
```

### 4. ガバナンス設計

```
意思決定構造：

  ステアリング委員会（月次）
    参加: 経営層、IT部門長、業務部門長、コンプライアンス部長
    判断: 予算変更、スコープ変更（大）、Go/No-Go判定

  プロジェクト運営会議（週次）
    参加: PM、各チームリーダー、業務部門代表
    判断: スコープ変更（小〜中）、課題対応方針

  デイリースタンドアップ（日次、Phase 3のみ）
    参加: 各チームメンバー
    目的: タスク確認、障害の早期発見

品質ゲート:
  Gate 1: 要件定義完了 → コンプライアンス部門承認
  Gate 2: 基本設計完了 → アーキテクチャレビュー
  Gate 3: 開発完了 → セキュリティ診断
  Gate 4: 総合テスト完了 → リリース判定会議
  Gate 5: 並行稼働完了 → 本番切替判定
```

### 5. リスク対応計画

```
R1: COBOLロジック移植の正確性（スコア9）
  対応: 現行ソースの解析を要件定義と並行して早期開始
  対応: 移植対象のロジックを一覧化し、テストケースを事前作成

R2: データ移行の品質（スコア9）
  対応: 移行リハーサルを3回実施（本番の3ヶ月前、1ヶ月前、1週間前）
  対応: データクレンジングルールを早期に策定

R3: 協力会社メンバーの品質ばらつき（スコア6）
  対応: コーディング規約とレビュープロセスの統一
  対応: 社内メンバーがリーダーとしてレビュー責任を持つ

R4: コンプライアンス要件の追加・変更（スコア6）
  対応: コンプライアンス部門を早期から巻き込む
  対応: 規制変更のモニタリングを月次で実施

R5: 18ヶ月の長期プロジェクトでのモチベーション低下（スコア4）
  対応: マイルストーンごとに成果を可視化
  対応: スプリントレビューで成果を体感する機会を設ける
```

<details>
<summary>解答例（改良版 ─ 移行戦略とスケーリングフレームワーク付き）</summary>

### 開発方式設計（詳細版）

```
============================================================
  開発プロセス設計書 v2.0
  プロジェクト: 保険契約管理システム刷新
============================================================

1. アーキテクチャと開発方式の連動
────────────────────────────────

新システムのアーキテクチャ:
  ┌──────────────┐  ┌──────────────┐
  │  フロントエンド  │  │  管理画面     │
  │  (React SPA)    │  │  (React)     │
  └───────┬────────┘  └──────┬───────┘
          │                    │
  ┌───────┴────────────────────┴───────┐
  │         API Gateway (AWS)          │
  └───────┬────────────────────┬───────┘
          │                    │
  ┌───────┴────────┐  ┌───────┴───────┐
  │ 業務ロジック層  │  │ データ移行    │
  │ (Java/Spring)  │  │ バッチ        │
  └───────┬────────┘  └───────┬───────┘
          │                    │
  ┌───────┴────────────────────┴───────┐
  │         RDS + S3 (AWS)             │
  └────────────────────────────────────┘

  コンポーネントごとの開発方式:
    フロントエンド:   アジャイル（変更頻度高）
    業務ロジック層:   WF寄り（正確性重視）
    データ移行バッチ: WF（計画的に段階実行）
    インフラ:         IaC + CI/CD（自動化）

2. スケーリングフレームワーク: Scrum of Scrums
──────────────────────────────────────────

  3チームの横断調整:
    - 週2回のScrum of Scrums（各チームリーダー + PM）
    - チーム間の依存関係を可視化する依存関係ボード
    - 共通の定義: DoD、コーディング規約、ブランチ戦略

  インテグレーションスプリント:
    - 4スプリントに1回、チーム統合のためのスプリントを設ける
    - 結合テスト、性能テスト、セキュリティテストを実施
    - コンプライアンス部門のレビューをこのタイミングで実施

3. データ移行戦略
──────────────────

  移行方式: 段階的ビッグバン方式

  Step 1（-6ヶ月）: データアセスメント
    - 現行データの品質分析
    - クレンジングルール策定
    - マッピングルール定義

  Step 2（-3ヶ月）: リハーサル1回目
    - 全量データで移行テスト
    - 件数照合、金額照合
    - 移行所要時間の計測

  Step 3（-1ヶ月）: リハーサル2回目
    - 改善版での再テスト
    - 本番想定のタイムスケジュールで実施

  Step 4（-1週間）: リハーサル3回目（本番リハ）
    - 本番環境での最終テスト
    - 切り戻し手順の確認

  Step 5（本番）: データ移行実行
    - 金曜夜間に開始、日曜中に完了目標
    - 異常時は切り戻し判断（日曜12:00がデッドライン）

4. コンプライアンス対応の統合
──────────────────────────────

  金融庁ガイドライン対応チェックポイント:

  要件定義完了時:
    □ 個人情報保護方針との整合性確認
    □ システムリスク管理に関する要件の明文化
    □ 監査証跡（Audit Trail）の要件定義

  設計完了時:
    □ セキュリティ設計書のコンプライアンス部門承認
    □ データ暗号化方式の妥当性確認
    □ アクセス制御設計の審査

  リリース前:
    □ 第三者によるセキュリティ診断
    □ BCP（事業継続計画）テストの実施
    □ 監督官庁への届出事項の確認

5. 契約管理（社内チーム + 協力会社）
──────────────────────────────────

  協力会社との契約形態と方式の整合:

  準委任契約（時間精算）: → アジャイル開発に適用
    チームB（UI/UX）のメンバーに適用
    スプリント単位で成果を確認
    ベロシティ実績に基づく透明な工数管理

  請負契約（成果物納品）: → WF工程に適用
    データ移行バッチの開発・テストに適用
    成果物: 移行プログラム一式、テスト結果報告書
    検収基準を契約時に明確化

  注意点:
    - 請負部分の要件変更は正式な変更管理プロセスで処理
    - 準委任部分は柔軟にスコープ調整可能
    - 両者の境界を明確にし、責任の曖昧さを排除
```

**初心者向けとの違い：**
- **アーキテクチャと開発方式の連動**: コンポーネントの特性に応じて方式を使い分けることで、最適な進め方ができます
- **Scrum of Scrums**: 複数チームの場合、チーム間の調整手法が必要です
- **データ移行戦略の詳細化**: 100万件のデータ移行は最大リスクであり、段階的なリハーサルが不可欠です
- **コンプライアンス対応の統合**: 金融系プロジェクトでは規制対応を開発プロセスに組み込む必要があります
- **契約形態と方式の整合**: アジャイルと請負契約は相性が悪いため、契約形態に応じた使い分けが実務では重要です

</details>

## 8.1 脆弱性診断

### 8.1.1 脆弱性診断とは

脆弱性診断（Vulnerability Assessment）は、システムやネットワークに存在するセキュリティ上の弱点（脆弱性）を発見し、対策の優先度を評価する活動です。

```
【脆弱性診断の種類】

  脆弱性診断
  ├── プラットフォーム診断
  │   ├── ネットワーク機器の設定不備
  │   ├── OSの既知の脆弱性
  │   └── 不要なサービスの公開
  │
  ├── Webアプリケーション診断
  │   ├── SQLインジェクション
  │   ├── XSS（クロスサイトスクリプティング）
  │   ├── CSRF
  │   └── 認証・認可の不備
  │
  └── ソースコード診断（静的解析）
      ├── バッファオーバーフロー
      ├── ハードコードされた認証情報
      └── 安全でない関数の使用
```

### 8.1.2 診断手法

| 手法 | 説明 | 特徴 |
|------|------|------|
| **自動診断（ツール診断）** | 脆弱性スキャナを使って自動的に検査 | 短時間で広範囲をカバー。誤検知あり |
| **手動診断** | 専門家が手動で検査 | 精度が高い。ビジネスロジックの脆弱性も発見可能 |
| **静的解析（SAST）** | ソースコードを解析して脆弱性を発見 | 開発段階で発見可能 |
| **動的解析（DAST）** | 実行中のアプリケーションに対して検査 | 実際の動作環境での脆弱性を発見 |
| **IAST** | アプリ実行時にコード内部からも検査 | SASTとDASTの利点を組み合わせ |

### 8.1.3 CVSS（共通脆弱性評価システム）

脆弱性の深刻度を数値で評価する国際的な基準です。

```
【CVSSスコア】

  スコア範囲    深刻度レベル
  9.0 - 10.0    緊急（Critical）  ← 即時対応が必要
  7.0 - 8.9     重要（High）
  4.0 - 6.9     警告（Medium）
  0.1 - 3.9     注意（Low）
  0.0            なし（None）

  CVSSスコアの構成:
  ┌─────────────────────────────────┐
  │ 基本評価基準（Base Metrics）      │ ← 脆弱性自体の特性
  │  攻撃の難易度、影響の大きさ等     │
  ├─────────────────────────────────┤
  │ 現状評価基準（Temporal Metrics）   │ ← 時間経過による変化
  │  攻撃コードの有無、対策の状況等   │
  ├─────────────────────────────────┤
  │ 環境評価基準（Environmental）      │ ← 利用環境による影響
  │  対象システムの重要度等           │
  └─────────────────────────────────┘
```

### 8.1.4 CVE（共通脆弱性識別子）

CVE（Common Vulnerabilities and Exposures）は、公開された脆弱性に一意のIDを付与する仕組みです。

```
  CVE-2024-12345
  │     │    │
  CVE   年   連番

  例: CVE-2021-44228 = Apache Log4j の脆弱性（Log4Shell）
```

## 8.2 ペネトレーションテスト

### 8.2.1 ペネトレーションテストとは

ペネトレーションテスト（Penetration Test / ペンテスト）は、実際の攻撃者の視点からシステムに侵入を試み、セキュリティ上の問題を発見するテストです。脆弱性診断が「弱点の発見」を目的とするのに対し、ペネトレーションテストは「実際に侵入が可能か」を検証します。

### 8.2.2 脆弱性診断との違い

| 項目 | 脆弱性診断 | ペネトレーションテスト |
|------|-----------|---------------------|
| 目的 | 脆弱性の網羅的な発見 | 実際の侵入可否の検証 |
| 手法 | ツール + 手動 | 攻撃者と同じ手法 |
| 範囲 | 広範囲・網羅的 | 特定の目標に集中 |
| 成果物 | 脆弱性一覧と対策案 | 侵入経路と影響範囲の実証 |
| 頻度 | 定期的（四半期〜年1回） | 必要に応じて |

### 8.2.3 ペネトレーションテストの流れ

```
【ペネトレーションテストのフェーズ】

  ① 事前準備（Planning）
     ├── テスト範囲と目標の合意
     ├── ルールの策定（禁止事項、緊急連絡先）
     └── 実施時期の調整

  ② 情報収集（Reconnaissance）
     ├── 公開情報の収集（OSINT）
     │   DNS情報、Whois、SNS、求人情報等
     ├── ポートスキャン
     └── サービスの特定

  ③ 脆弱性の特定（Vulnerability Analysis）
     ├── 発見したサービスの脆弱性調査
     ├── 設定不備の確認
     └── 攻撃シナリオの検討

  ④ 攻撃の実行（Exploitation）
     ├── 脆弱性の悪用を試行
     ├── 権限昇格の試行
     └── 横展開（ラテラルムーブメント）の試行

  ⑤ 報告（Reporting）
     ├── 発見事項のまとめ
     ├── リスク評価と優先度
     └── 対策の推奨事項
```

### 8.2.4 テストの種類

| 種類 | 説明 | 事前情報 |
|------|------|---------|
| **ブラックボックステスト** | テスト対象の情報を事前に与えない | なし |
| **ホワイトボックステスト** | ネットワーク構成、ソースコード等を提供 | 全て |
| **グレーボックステスト** | 一部の情報（アカウント情報等）を提供 | 一部 |

## 8.3 OWASP Top 10

### 8.3.1 OWASP Top 10（2021年版）

OWASP（Open Web Application Security Project）が定期的に発表する、Webアプリケーションの最も深刻なセキュリティリスクのランキングです。

| 順位 | カテゴリ | 概要 | 主な対策 |
|------|---------|------|---------|
| **A01** | アクセス制御の不備 | 認可チェックの欠如、IDOR | 全APIで認可チェック、デフォルトDeny |
| **A02** | 暗号化の失敗 | 平文通信、弱い暗号の使用 | TLS必須、強力な暗号アルゴリズム |
| **A03** | インジェクション | SQL、XSS、コマンドインジェクション | パラメータ化クエリ、出力エスケープ |
| **A04** | 安全でない設計 | 設計段階のセキュリティ不足 | 脅威モデリング、セキュアデザインパターン |
| **A05** | セキュリティの設定ミス | デフォルト設定、不要な機能 | セキュリティ設定の自動化、最小権限 |
| **A06** | 脆弱で古いコンポーネント | 既知脆弱性のあるライブラリ | 依存関係の定期更新、SCA |
| **A07** | 識別と認証の失敗 | 弱いパスワード、セッション管理不備 | MFA、パスワードポリシー強化 |
| **A08** | ソフトウェアとデータの整合性の不備 | 安全でないCI/CD、デシリアライゼーション | 署名検証、整合性チェック |
| **A09** | セキュリティログと監視の不備 | ログ不足、異常検知の欠如 | 包括的なログ記録、アラート |
| **A10** | SSRF | サーバに内部リソースへのリクエストを強制 | 入力検証、ネットワーク分離 |

## 8.4 セキュアコーディング

### 8.4.1 セキュアコーディングの基本原則

```
【セキュアコーディング 10の原則】

  ① 入力を検証する（Input Validation）
  ② 出力をエンコード/エスケープする（Output Encoding）
  ③ 認証と認可を正しく実装する
  ④ セッション管理を適切に行う
  ⑤ アクセス制御を実装する
  ⑥ 暗号を正しく使用する
  ⑦ エラー処理とログ記録を適切に行う
  ⑧ データを保護する
  ⑨ 通信を保護する
  ⑩ システム構成を安全にする
```

### 8.4.2 入力検証の原則

```
【入力検証の基本】

  すべてのユーザ入力は信頼できない

  検証の方針:
  ・ホワイトリスト方式（推奨）: 許可する値のみ受け入れる
  ・ブラックリスト方式（非推奨）: 禁止する値を除外する
    → 漏れが発生しやすい

  検証すべき項目:
  ├── データ型（数値、文字列等）
  ├── 文字種（英数字のみ等）
  ├── 長さ（最小・最大）
  ├── 範囲（数値の上限・下限）
  ├── 形式（メールアドレス、日付等）
  └── エンコーディング（UTF-8等）
```

### 8.4.3 安全なコーディングの例

**SQLインジェクション対策**

```python
# NG: 文字列結合によるSQL構築
query = f"SELECT * FROM users WHERE id = {user_input}"

# OK: パラメータ化クエリ
query = "SELECT * FROM users WHERE id = ?"
cursor.execute(query, (user_input,))
```

**XSS対策**

```python
# NG: エスケープなしの出力
return f"<p>ようこそ、{username}さん</p>"

# OK: HTMLエスケープ
from markupsafe import escape
return f"<p>ようこそ、{escape(username)}さん</p>"
```

**パスワードの安全な保存**

```python
# NG: 平文で保存
password_hash = password  # 絶対にやってはいけない

# NG: MD5やSHA-1でハッシュ化
password_hash = hashlib.md5(password.encode()).hexdigest()

# OK: bcrypt等の適切なハッシュ関数を使用
import bcrypt
password_hash = bcrypt.hashpw(
    password.encode(), bcrypt.gensalt()
)
```

### 8.4.4 セキュリティ上避けるべきプラクティス

| 避けるべきこと | 理由 | 対策 |
|-------------|------|------|
| 認証情報のハードコード | ソースコードから漏洩する | 環境変数や秘密管理サービスを使用 |
| 自作の暗号アルゴリズム | 専門家でないと安全に作れない | 標準的なライブラリ（AES等）を使用 |
| 詳細なエラーメッセージの表示 | 攻撃者に情報を与える | ユーザには一般的なメッセージ、詳細はログに記録 |
| 古いライブラリの使用 | 既知の脆弱性が存在 | 定期的なアップデート |
| eval()等の動的コード実行 | 任意のコード実行のリスク | 使用を避ける。必要な場合は厳格な入力検証 |

## 8.5 DevSecOps と セキュリティのシフトレフト

### 8.5.1 DevSecOps

DevSecOps は、開発（Development）、セキュリティ（Security）、運用（Operations）を統合し、開発ライフサイクルの全段階でセキュリティを組み込むアプローチです。

```
【従来の開発 vs DevSecOps】

  従来: セキュリティはリリース前に後付けで検査
  ──────────────────────────────────→
  設計 → 開発 → テスト → セキュリティ検査 → リリース
                              ↑ ここで初めてセキュリティ
                                問題が多数発覚（コスト大）

  DevSecOps: 全段階にセキュリティを組み込み
  ──────────────────────────────────→
  設計 → 開発 → テスト → リリース → 運用
   ↑       ↑       ↑        ↑       ↑
  脅威   SAST    DAST     設定    監視
  モデリング コードレビュー  検査    監査   ログ分析
```

### 8.5.2 シフトレフト

「シフトレフト」とは、セキュリティ活動を開発ライフサイクルの早い段階（左側）に移すことです。早期に発見するほど修正コストが低く、品質が向上します。

| 開発フェーズ | セキュリティ活動 |
|------------|----------------|
| **要件定義** | セキュリティ要件の定義、脅威モデリング |
| **設計** | セキュアデザインレビュー、アーキテクチャレビュー |
| **実装** | セキュアコーディング、SAST（静的解析）、コードレビュー |
| **テスト** | DAST（動的解析）、ペネトレーションテスト |
| **リリース** | セキュリティ設定の検証、脆弱性スキャン |
| **運用** | ログ監視、インシデント対応、パッチ管理 |

### 8.5.3 脅威モデリング

設計段階でシステムの脅威を特定し、対策を検討するプロセスです。

```
【STRIDE モデル（脅威の分類）】

  S: Spoofing（なりすまし）      → 認証で対策
  T: Tampering（改ざん）         → 完全性チェックで対策
  R: Repudiation（否認）         → ログ記録・デジタル署名で対策
  I: Information Disclosure      → 暗号化・アクセス制御で対策
     （情報漏洩）
  D: Denial of Service           → 可用性対策（冗長化等）で対策
     （サービス拒否）
  E: Elevation of Privilege      → 認可・最小権限で対策
     （権限昇格）
```

## 8.6 まとめ

### ポイントまとめ

1. **脆弱性診断** はシステムの弱点を網羅的に発見する活動で、SAST（静的解析）とDAST（動的解析）を組み合わせます
2. **ペネトレーションテスト** は攻撃者の視点から実際の侵入可否を検証するテストです
3. **OWASP Top 10** はWebアプリケーションの主要なセキュリティリスクのランキングで、定期的に確認すべきです
4. **セキュアコーディング** では入力検証、出力エスケープ、パラメータ化クエリが基本です
5. **DevSecOps** は開発ライフサイクル全体にセキュリティを組み込むアプローチです
6. **シフトレフト** により、セキュリティの問題を早期に発見・修正し、コストを削減します

### 試験対策キーワード

- 脆弱性診断、ペネトレーションテスト
- SAST（静的解析）、DAST（動的解析）
- CVSS、CVE、CWE
- OWASP Top 10
- セキュアコーディング、入力検証、出力エスケープ、パラメータ化クエリ
- DevSecOps、シフトレフト、脅威モデリング、STRIDE

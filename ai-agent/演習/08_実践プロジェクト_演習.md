# 第8章 演習：実践プロジェクト

この演習では、AIエージェントプロジェクトの要件定義からアーキテクチャ設計、実装計画、テスト、運用までを一貫して学びます。これまでの章で学んだ知識を総合的に活用してください。

## 基本問題

### 問題1：プロジェクト要件定義

以下のビジネス要件から、AIエージェントプロジェクトの要件定義書を作成してください。

**ビジネス要件：**
```
あなたの会社では、社内ITヘルプデスクへの問い合わせが月間500件あり、
対応に平均4名のスタッフが従事しています。問い合わせの約60%は
パスワードリセット、VPN接続方法、ソフトウェアインストール手順など、
定型的な内容です。経営層はAIエージェントを導入して、
定型問い合わせの自動対応率を70%以上にしたいと考えています。
```

**作成すべき内容：**
1. 機能要件（5つ以上）
2. 非機能要件（4つ以上）
3. 成功指標（KPI）（3つ以上）
4. スコープ外の定義（対応しないこと）

**期待される出力例：**

```
=== ITヘルプデスクエージェント 要件定義書 ===

【機能要件】
FR-001: パスワードリセット手順の自動案内
  - ユーザーの質問を理解し、該当するパスワードリセット手順を提示する
  ...

【非機能要件】
NFR-001: 応答時間
  - 95%のリクエストに対して5秒以内に応答する
  ...

【成功指標（KPI）】
1. 定型問い合わせの自動対応率：70%以上
  ...

【スコープ外】
- ...
```

<details>
<summary>ヒント</summary>

- 機能要件は「エージェントが何をできるか」を具体的に列挙します。
- 非機能要件は「性能」「セキュリティ」「可用性」「拡張性」の観点で考えましょう。
- KPIはビジネス目標と直結する測定可能な指標にしてください。
- スコープ外を定義することで、プロジェクトの範囲を明確にし、期待値を調整します。
</details>

<details>
<summary>解答例</summary>

```
=== ITヘルプデスクエージェント 要件定義書 ===

【機能要件】

FR-001: パスワードリセット手順の自動案内
  - 対象システム（メール、VPN、社内ポータル等）を特定し、
    該当するパスワードリセット手順をステップバイステップで案内する

FR-002: VPN接続トラブルシューティング
  - OS（Windows/Mac）とエラー内容を確認し、
    対応する解決手順を提示する
  - 解決しない場合はエスカレーションする

FR-003: ソフトウェアインストール手順の案内
  - 承認済みソフトウェア一覧から該当ソフトを特定し、
    インストール手順を案内する
  - 未承認ソフトウェアの場合は申請手順を案内する

FR-004: 有人対応へのエスカレーション
  - エージェントが対応できない問題を検出した場合、
    会話履歴を含めて有人対応に引き継ぐ
  - ユーザーが明示的に人間の対応を希望した場合も即座にエスカレーション

FR-005: FAQ検索と回答生成
  - 社内FAQデータベースを検索し、質問に対する回答を生成する
  - 回答の根拠となるFAQ記事へのリンクを提示する

FR-006: 対応履歴の記録
  - 全ての対話ログをチケットシステムに自動記録する
  - 対応結果（解決/エスカレーション）を分類して記録する

【非機能要件】

NFR-001: 応答時間
  - 95パーセンタイルで5秒以内に応答する

NFR-002: 可用性
  - 稼働率99.5%以上（月間ダウンタイム3.6時間以内）
  - 営業時間（9:00-18:00）は99.9%以上

NFR-003: セキュリティ
  - 社内認証システム（SSO）と連携し、認証済みユーザーのみ利用可能
  - パスワードなどの機密情報をログに記録しない
  - 社内ネットワーク内でのみアクセス可能

NFR-004: 拡張性
  - 新しいFAQ記事の追加が管理画面から可能（コード変更不要）
  - 対応カテゴリの追加が設定変更のみで可能

NFR-005: 多言語対応
  - 日本語を主言語とし、英語での問い合わせにも基本対応する

【成功指標（KPI）】

1. 定型問い合わせ自動対応率：70%以上
   - 計算方法：エージェントのみで解決した定型問い合わせ数 / 全定型問い合わせ数
   - 測定頻度：週次

2. ユーザー満足度：4.0以上（5段階）
   - 計算方法：対話終了後のアンケート結果の平均
   - 測定頻度：月次

3. 平均対応時間の短縮：50%以上削減
   - 計算方法：導入前の平均対応時間 vs 導入後の平均対応時間
   - 測定頻度：月次

4. ヘルプデスクスタッフの工数削減：30%以上
   - 計算方法：定型対応に費やす時間の削減率
   - 測定頻度：月次

【スコープ外】

- ハードウェア障害への対応（物理的な対応が必要なもの）
- ネットワークインフラの設定変更（管理者権限が必要な操作）
- セキュリティインシデントへの対応（専門チームが対応）
- 社外ユーザーからの問い合わせ対応
- 新入社員向けのオンボーディング（別プロジェクトとして実施）
```
</details>

### 問題2：アーキテクチャ選定判断

以下の3つのユースケースそれぞれに対して、最適なエージェントアーキテクチャを選定し、その理由を説明してください。

**選択肢：**
- A. シングルエージェント（ReActパターン）
- B. マルチエージェント（階層型オーケストレーション）
- C. マルチエージェント（水平型コラボレーション）
- D. ワークフロー型（DAG実行）

**ユースケース：**

```
ケース1：レストラン予約ボット
  - ユーザーの希望（日時・人数・エリア・料理ジャンル）を聞き取り
  - 予約サイトAPIで空き状況を検索
  - 予約を確定

ケース2：企業のM&Aデューデリジェンス支援
  - 財務分析（財務諸表の分析、リスク評価）
  - 法務分析（契約書レビュー、コンプライアンス確認）
  - 技術分析（特許調査、技術スタック評価）
  - 総合レポート作成

ケース3：EC注文処理パイプライン
  - 注文受付 → 在庫確認 → 決済処理 → 配送手配 → 通知送信
  - 各ステップの成功/失敗に応じた分岐処理
  - 失敗時のリトライと補償処理
```

**期待される出力例：**

```
ケース1：A（シングルエージェント・ReActパターン）
  理由：...

ケース2：B or C（マルチエージェント）
  理由：...

ケース3：D（ワークフロー型）
  理由：...
```

<details>
<summary>ヒント</summary>

- シングルエージェントは、タスクが比較的単純で1つの会話フローで完結する場合に適しています。
- マルチエージェント（階層型）は、専門分野が異なる複数のサブタスクを統括する必要がある場合に適しています。
- マルチエージェント（水平型）は、対等な専門エージェントが協力して成果物を作る場合に適しています。
- ワークフロー型は、実行順序が明確で、各ステップの入出力が定義されている場合に適しています。
</details>

<details>
<summary>解答例</summary>

```
ケース1：A（シングルエージェント・ReActパターン）

理由：
- タスクの複雑さが低く、1つのエージェントで十分に処理可能です。
- 会話の流れ（希望聞き取り→検索→予約確定）が直線的です。
- 使用するツールが少なく（予約API、検索API程度）、
  ツール選択の判断も単純です。
- マルチエージェントにするとオーバーヘッドが増え、
  応答速度が低下するため不適切です。
- ReActパターンにより「考える→行動する→観察する」の
  サイクルで柔軟に対話できます。

ケース2：B（マルチエージェント・階層型オーケストレーション）

理由：
- 財務・法務・技術という異なる専門領域の分析が必要です。
- 各領域のエージェントには専門的なプロンプト、ツール、
  知識ベースが必要であり、1つのエージェントでは
  プロンプトが複雑になりすぎます。
- オーケストレーター（統括エージェント）が各専門エージェントに
  タスクを配分し、結果を統合してレポートを作成する構造が最適です。
- 水平型ではなく階層型を選ぶ理由は、最終的に統合レポートを
  作成する「まとめ役」が必要なためです。
- 各専門エージェントは並列実行可能で、処理時間を短縮できます。

ケース3：D（ワークフロー型・DAG実行）

理由：
- 処理の順序が明確に定義されており（注文→在庫→決済→配送→通知）、
  LLMの判断による柔軟なルーティングは不要です。
- 各ステップの入出力が明確に定義されています。
- 失敗時のリトライ・補償処理（例：決済失敗時の在庫戻し）は
  ワークフローエンジンの機能として実装するのが信頼性が高いです。
- LLMを使わないステップ（決済処理、在庫確認）が多く、
  エージェントパターンを使うメリットがありません。
- トランザクションの一貫性と監査証跡が重要な業務であり、
  決定論的なワークフローが適しています。
```
</details>

### 問題3：技術スタック選定

以下の要件に基づいて、エージェント開発の技術スタックを選定してください。

**プロジェクト要件：**
```
- チーム：バックエンドエンジニア3名（Python経験あり）
- 用途：社内ドキュメント検索 + Q&Aエージェント
- 規模：社員500名が利用、1日あたり約200件の質問
- 予算：月額$500以内（LLM API費用含む）
- 期間：3ヶ月で本番リリース
- 要件：社内データのセキュリティ確保が最重要
```

**選定すべき項目：**
1. LLMプロバイダー・モデル
2. エージェントフレームワーク
3. ベクトルDB（RAG用）
4. デプロイ環境
5. 監視・ログ基盤

各選定に対して理由も記載してください。

**期待される出力例：**

```
【技術スタック選定】

1. LLMプロバイダー・モデル
   選定：○○○
   理由：...
   月額コスト見積もり：$XXX

2. エージェントフレームワーク
   選定：○○○
   理由：...
...
```

<details>
<summary>ヒント</summary>

- 予算制約（月額$500以内）とセキュリティ要件を最優先に考えましょう。
- チームのスキルセット（Python経験）に合った技術を選びましょう。
- 3ヶ月という期間を考えると、学習コストの低い技術が有利です。
- 1日200件の質問は比較的小規模です。過剰な設計は避けましょう。
</details>

<details>
<summary>解答例</summary>

```
【技術スタック選定】

1. LLMプロバイダー・モデル
   選定：OpenAI GPT-4o-mini（メイン）+ GPT-4o（フォールバック）
   理由：
   - GPT-4o-miniはコストパフォーマンスに優れ、
     日常的なQ&A対応に十分な性能があります
   - 複雑な質問のみGPT-4oにルーティングすることで
     コストを抑えつつ品質を維持できます
   - OpenAIはエンタープライズ向けのデータ保護ポリシーがあり、
     API経由のデータはモデル学習に使用されません
   月額コスト見積もり：約$100-150
     （200件/日 × 30日 × 平均2000トークン/件）

2. エージェントフレームワーク
   選定：LangChain（Python版）
   理由：
   - チーム全員がPython経験者であり、学習コストが低いです
   - RAG構築のためのツールチェーンが充実しています
   - ドキュメントとコミュニティが豊富で、問題解決が容易です
   - 3ヶ月の開発期間で十分に習熟可能です

3. ベクトルDB（RAG用）
   選定：Chroma（開発・小規模）またはPgvector（PostgreSQL拡張）
   理由：
   - Chromaはセットアップが容易で、プロトタイピングに最適です
   - 本番環境ではPgvectorを推奨します。既存のPostgreSQLに
     拡張として追加でき、運用の複雑さを軽減できます
   - 500名規模のドキュメント量であれば十分な性能です
   - 追加コスト：$0（既存DB環境を活用）

4. デプロイ環境
   選定：社内オンプレミスサーバー or AWS VPC（プライベートサブネット）
   理由：
   - セキュリティ最重要のため、社内データが外部に出ない構成にします
   - 社内サーバーがある場合はDockerコンテナでデプロイ
   - クラウドの場合はVPC内にデプロイし、
     LLM APIへのアクセスのみ外部通信を許可
   月額コスト見積もり：$100-200（クラウドの場合）

5. 監視・ログ基盤
   選定：LangSmith（トレーシング）+ Grafana + Loki（ログ）
   理由：
   - LangSmithはLangChainとの統合が容易で、
     LLM呼び出しのトレーシングに特化しています
   - Grafana + Lokiはオープンソースで追加コスト不要です
   - ダッシュボードで応答時間、エラー率、コストを可視化できます
   月額コスト見積もり：$50（LangSmith）+ $0（Grafana/Loki）

【月額コスト合計見積もり】
- LLM API：$100-150
- デプロイ環境：$100-200
- 監視基盤：$50
- 合計：$250-400（予算$500以内に収まる）
```
</details>

## 応用問題

### 問題4：カスタマーサポートエージェントの詳細設計

ECサイトのカスタマーサポートエージェントを詳細設計してください。以下の3つの要素を具体的に定義してください。

**エージェントの対応範囲：**
- 注文状況の確認
- 返品・交換の受付
- 商品に関する質問への回答
- クーポン・ポイントの確認
- クレーム対応（有人エスカレーション含む）

**設計すべき内容：**

#### Part A：ツール定義（5つ以上）

各ツールについて以下を定義してください。
- ツール名
- 説明（description）
- 入力パラメータ（型と説明）
- 出力形式
- エラーケース

#### Part B：対話フロー

「返品リクエスト」の対話フローを設計してください。
- 開始条件
- 各ステップの処理内容
- 分岐条件
- 終了条件

#### Part C：エラー処理設計

以下のエラーシナリオへの対処方法を設計してください。
- ツールのAPI呼び出しタイムアウト
- ユーザーの意図が不明確
- 返品期限切れ
- システム全体の障害

**期待される出力例：**

```
【Part A：ツール定義】

ツール1: get_order_status
  説明：注文IDに基づいて注文の現在のステータスを取得する
  入力：
    - order_id: string（必須）- 注文番号
  出力：
    { status: string, items: array, shipped_date: string | null }
  エラー：
    - ORDER_NOT_FOUND: 注文が見つからない場合
...

【Part B：返品リクエストの対話フロー】
...

【Part C：エラー処理設計】
...
```

<details>
<summary>ヒント</summary>

- ツール定義は「エージェントがLLMに提示するツール仕様」として設計します。型情報は明確に記述しましょう。
- 対話フローはフローチャートのように条件分岐を明確にしましょう。
- エラー処理では「ユーザーへの影響を最小化する」ことを最優先に考えましょう。
</details>

<details>
<summary>解答例</summary>

```
【Part A：ツール定義】

ツール1: get_order_status
  説明：注文IDに基づいて注文の現在のステータスと詳細情報を取得する
  入力：
    - order_id: string（必須）- 注文番号（例："ORD-20260326-001"）
  出力：
    {
      "status": "processing" | "shipped" | "delivered" | "cancelled",
      "items": [{ "name": string, "quantity": number, "price": number }],
      "order_date": string,
      "shipped_date": string | null,
      "delivery_estimate": string | null,
      "tracking_number": string | null
    }
  エラー：
    - ORDER_NOT_FOUND: 注文IDが存在しない
    - PERMISSION_DENIED: 他のユーザーの注文

ツール2: search_orders_by_user
  説明：ユーザーIDに基づいて注文履歴を検索する
  入力：
    - user_id: string（必須）- ユーザーID
    - limit: number（任意、デフォルト10）- 取得件数
    - status_filter: string（任意）- ステータスでフィルタ
  出力：
    { "orders": [{ "order_id": string, "status": string, "order_date": string, "total": number }] }
  エラー：
    - USER_NOT_FOUND: ユーザーが存在しない

ツール3: create_return_request
  説明：返品リクエストを作成する
  入力：
    - order_id: string（必須）- 注文番号
    - item_ids: string[]（必須）- 返品対象の商品ID
    - reason: string（必須）- 返品理由
    - reason_category: "defective" | "wrong_item" | "not_as_described" | "changed_mind"（必須）
  出力：
    {
      "return_id": string,
      "status": "accepted" | "rejected",
      "rejection_reason": string | null,
      "return_label_url": string | null,
      "refund_estimate": number
    }
  エラー：
    - RETURN_PERIOD_EXPIRED: 返品期限を超過
    - ITEM_NOT_RETURNABLE: 返品不可商品
    - ALREADY_RETURNED: 既に返品済み

ツール4: search_product_faq
  説明：商品FAQデータベースを検索して回答を取得する
  入力：
    - query: string（必須）- 検索クエリ
    - product_id: string（任意）- 特定商品に限定する場合
    - top_k: number（任意、デフォルト3）- 取得件数
  出力：
    { "results": [{ "question": string, "answer": string, "relevance_score": number }] }
  エラー：
    - NO_RESULTS: 関連するFAQが見つからない

ツール5: get_user_points_and_coupons
  説明：ユーザーのポイント残高と利用可能なクーポンを取得する
  入力：
    - user_id: string（必須）- ユーザーID
  出力：
    {
      "points_balance": number,
      "expiring_points": { "amount": number, "expiry_date": string },
      "coupons": [{ "code": string, "description": string, "expiry_date": string, "min_purchase": number }]
    }
  エラー：
    - USER_NOT_FOUND: ユーザーが存在しない

ツール6: escalate_to_human
  説明：有人対応にエスカレーションする
  入力：
    - user_id: string（必須）- ユーザーID
    - category: "complaint" | "complex_issue" | "user_request"（必須）
    - summary: string（必須）- 問い合わせの要約
    - priority: "low" | "medium" | "high" | "urgent"（必須）
    - conversation_history: string（必須）- これまでの会話履歴
  出力：
    {
      "ticket_id": string,
      "assigned_agent": string | null,
      "estimated_wait_time": number
    }
  エラー：
    - ALL_AGENTS_BUSY: 全オペレーターが対応中
    - OUTSIDE_BUSINESS_HOURS: 営業時間外

【Part B：返品リクエストの対話フロー】

[開始条件]
  ユーザーの発言が返品・交換の意図を含む場合に開始

[ステップ1：注文の特定]
  エージェント：「返品のご依頼ですね。注文番号はおわかりですか？」
  ├─ 注文番号あり → get_order_status(order_id) → ステップ2へ
  ├─ 注文番号なし → search_orders_by_user(user_id) で最近の注文を表示
  │   └─ ユーザーが選択 → ステップ2へ
  └─ 注文が見つからない → 「注文が見つかりませんでした。
       注文番号をご確認ください」→ リトライ or 終了

[ステップ2：返品対象商品の確認]
  エージェント：「以下の商品が含まれています。返品したい商品はどれですか？」
  └─ 商品を一覧表示し、ユーザーに選択させる → ステップ3へ

[ステップ3：返品理由の確認]
  エージェント：「返品の理由を教えてください」
  ├─ 不良品（defective）→ ステップ4へ（送料無料）
  ├─ 商品違い（wrong_item）→ ステップ4へ（送料無料）
  ├─ 説明と異なる（not_as_described）→ ステップ4へ（送料無料）
  └─ 気が変わった（changed_mind）→ ステップ4へ（送料ユーザー負担を案内）

[ステップ4：返品リクエスト作成]
  create_return_request(order_id, item_ids, reason, reason_category)
  ├─ 成功（accepted）→ ステップ5へ
  ├─ RETURN_PERIOD_EXPIRED → 「申し訳ございませんが、返品期限を過ぎています。
  │   詳しくはスタッフにご相談ください」→ エスカレーション提案
  ├─ ITEM_NOT_RETURNABLE → 「この商品は返品対象外です」→ 理由説明
  └─ ALREADY_RETURNED → 「この商品は既に返品処理済みです」→ 状況確認

[ステップ5：返品手続きの案内]
  エージェント：
  「返品リクエストを受け付けました（返品番号：RET-XXXXX）。
   返品ラベルをメールでお送りしました。
   商品を梱包し、最寄りの配送業者にお持ちください。
   返金予定額：¥XXXXX（返品確認後5営業日以内）」

[終了条件]
  - 返品リクエストが正常に作成された
  - ユーザーが返品をキャンセルした
  - エスカレーションが完了した

【Part C：エラー処理設計】

1. ツールのAPI呼び出しタイムアウト
   対処方針：リトライ + ユーザーへの状況説明
   詳細：
   - 1回目のタイムアウト：3秒後に自動リトライ
   - 2回目のタイムアウト：5秒後に自動リトライ
   - 3回目のタイムアウト：ユーザーに以下を案内
     「現在システムの応答に時間がかかっております。
      しばらく待ってから再度お試しいただくか、
      オペレーターにおつなぎすることも可能です。」
   - 最大リトライ回数：3回
   - リトライ間隔：指数バックオフ（3秒、5秒、10秒）

2. ユーザーの意図が不明確
   対処方針：段階的な確認質問
   詳細：
   - 信頼度60%未満：「確認させてください。○○についてのご質問でしょうか？」
     と最も可能性の高い意図を提示
   - 信頼度40%未満：「以下のどちらに近いですか？」
     と上位2-3の候補を選択肢として提示
   - 3回確認しても不明：「お手数ですが、もう少し詳しく
     教えていただけますか？または、オペレーターにおつなぎしましょうか？」
   - ユーザーが frustration を示す言葉を使った場合：
     即座にエスカレーション提案

3. 返品期限切れ
   対処方針：柔軟な対応オプションの提示
   詳細：
   - まず期限切れであることを丁寧に説明
   - 代替案を提示：
     (a)「特別な事情がある場合はオペレーターにご相談いただけます」
     (b)「交換であれば対応可能な場合があります」
     (c)「修理サービスをご利用いただける場合があります」
   - ユーザーが希望する場合はエスカレーション（priority: medium）

4. システム全体の障害
   対処方針：グレースフルデグラデーション
   詳細：
   - ヘルスチェック機能で障害を検出
   - 障害発生時のフォールバック応答を事前に定義：
     「現在システムメンテナンス中のため、一部機能がご利用いただけません。
      お急ぎの場合はお電話（0120-XXX-XXX）でお問い合わせください。
      ご不便をおかけし申し訳ございません。」
   - FAQ検索など、外部API不要な機能は継続提供
   - 障害中の問い合わせはキューに保存し、復旧後にフォローアップ
```

## チャレンジ問題

### 問題5：エンドツーエンドのエージェントプロジェクト企画書

あなたは「社内ナレッジマネジメントエージェント」のプロジェクトリーダーです。以下の背景を踏まえて、エンドツーエンドのプロジェクト企画書を作成してください。

**背景：**
```
従業員2,000名の製造業企業。社内に蓄積された技術ドキュメント、
議事録、マニュアル、報告書が複数のシステム（SharePoint、
Confluence、社内Wiki、ファイルサーバー）に分散しており、
必要な情報を見つけるのに平均30分かかっている。
年間の情報検索にかかる工数は推定12,000時間（人件費換算で約6,000万円）。
経営層はAIエージェントの導入により、情報検索時間を70%削減したい。
```

**企画書に含めるべき内容：**

1. **要件定義**（機能要件・非機能要件・制約条件）
2. **アーキテクチャ設計**（全体構成図をテキストで表現、コンポーネント説明）
3. **実装計画**（フェーズ分け、各フェーズの成果物とマイルストーン）
4. **テスト計画**（テストレベル別の計画、受入基準）
5. **運用計画**（監視項目、運用体制、インシデント対応）
6. **コスト見積もり**（初期費用・ランニングコスト・ROI分析）

**期待される出力例：**

```

====================================================

...

...

...

...
```

<details>
<summary>ヒント</summary>

- 要件定義では「4つのデータソースからの統合検索」が最重要機能です。
- アーキテクチャではRAG（検索拡張生成）パターンが基本になります。
- 実装計画は一気に全機能を実装するのではなく、MVP（最小実行可能製品）から段階的に拡張しましょう。
- テスト計画では「回答の正確性」をどう検証するかが最も重要です。
- 運用計画では「ドキュメントの追加・更新への対応」を忘れずに。
- ROI分析では「6,000万円の70%削減」を基準にコストと比較しましょう。

■ 機能要件

FR-001: 統合検索
  - SharePoint、Confluence、社内Wiki、ファイルサーバーの
    4つのデータソースを横断的に検索する
  - 自然言語での質問に対して関連ドキュメントを取得する

FR-002: 回答生成
  - 検索されたドキュメントを基に質問への回答を生成する
  - 回答の根拠となるドキュメントへのリンクを必ず提示する

FR-005: フィードバック機能

  - フィードバックデータを品質改善に活用

FR-006: ドキュメント更新の自動反映
  - データソースの変更を定期的にクロールし、インデックスを更新する
  - 新規ドキュメントの追加を自動検出する

NFR-002: 可用性 - 営業時間内99.5%以上

NFR-004: セキュリティ - 社内ネットワーク内のみアクセス可能、SSO連携

  対象：各コネクタ、チャンク分割、エンベディング生成、プロンプト処理
  手法：pytest による自動テスト
  基準：カバレッジ80%以上

■ 統合テスト
  対象：RAGパイプライン全体（検索→取得→生成）
  手法：100件の質問セットで自動評価
  基準：
  - 検索適合率（Precision）：80%以上
  - 回答正確性：85%以上（人間評価）

■ E2Eテスト
  対象：UIからの質問→回答表示→フィードバック
  手法：Playwrightによる自動UIテスト + シナリオテスト
  基準：主要シナリオ20件が全て正常完了

■ 性能テスト
  対象：同時接続時の応答時間

- 日次：KPIダッシュボード確認、アラート対応
- 週次：品質レポートレビュー、低評価回答の分析
- 月次：プロンプト改善、インデックス最適化、コストレビュー
- 四半期：ユーザーアンケート、大規模な改善計画策定

- レベル3（高）：サービス停止 → 30分以内に復旧、翌日振り返り
- レベル4（緊急）：情報漏洩の疑い → 即座にサービス停止、セキュリティチーム連携

|---|---|

| インフラ費用（サーバー・DB） | 240万円 |
| 運用人件費（0.5人工） | 400万円 |
| 監視ツールライセンス | 60万円 |
| **ランニングコスト合計** | **1,060万円/年** |

■ ROI分析

現状のコスト：
  - 情報検索にかかる年間工数：12,000時間
  - 人件費換算：6,000万円/年

導入後の削減効果（目標70%削減）：
  - 削減工数：8,400時間/年
  - 削減金額：4,200万円/年

ROI計算：
  - 初年度：
    削減効果4,200万円 − ランニングコスト1,060万円 = 3,140万円の効果
    初期投資2,780万円を考慮すると、
    初年度純効果 = 3,140万円 − 2,780万円 = +360万円
  - 2年目以降：
    年間純効果 = 4,200万円 − 1,060万円 = 3,140万円/年

  投資回収期間：約11ヶ月（初年度中に回収完了）
  3年間の累計ROI：(360 + 3,140 + 3,140) / 2,780 = 239%

■ リスクと対策

| リスク | 影響 | 対策 |
|---|---|---|
| 回答精度が目標未達 | 利用率低下 | MVP段階で品質検証、段階的展開 |
| LLM API費用の高騰 | ランニングコスト増 | キャッシング導入、モデル切替の準備 |
| ユーザー定着率の低下 | ROI未達 | 定期的なトレーニング、UI改善 |
| データソースAPI変更 | 機能停止 | コネクタの抽象化、変更監視 |

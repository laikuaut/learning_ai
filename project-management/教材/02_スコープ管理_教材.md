## 2.1 スコープ管理とは

スコープ管理とは、**プロジェクトで何を作るか（含むもの）と何を作らないか（含まないもの）を明確に定義し、管理する**こと。

スコープが曖昧だと：
- 「これも入ると思っていた」という認識齟齬が発生する
- 作業量が膨らみ、スケジュールが遅延する
- チームが何に集中すべきか分からなくなる

---

## 2.2 要件収集（Requirements Gathering）

### 要件の種類

| 種類 | 説明 | 例 |
|------|------|-----|
| **機能要件** | システムが「何をするか」 | ユーザー登録ができる、商品検索ができる |
| **非機能要件** | システムが「どのように動くか」 | レスポンス200ms以内、99.9%可用性 |
| **ビジネス要件** | ビジネス上の目標 | 売上20%向上、問い合わせ50%削減 |

### 要件収集の手法

#### 1. ユーザーインタビュー

```
対象：実際のユーザーや業務担当者
方法：1対1またはグループでのヒアリング
ポイント：
  - オープンクエスチョンを使う
  - 「何をしたいか」だけでなく「なぜしたいか」を聞く
  - 現状の課題・不満を具体的に引き出す
```

#### 2. ユースケース分析

```
アクター：顧客
ユースケース：商品を購入する
前提条件：ユーザーがログイン済みであること
基本フロー：
  1. 商品一覧を表示する
  2. 商品を選択する
  3. カートに追加する
  4. 購入手続きに進む
  5. 支払い情報を入力する
  6. 注文を確定する
代替フロー：
  3a. 在庫がない場合 → 「在庫切れ」を表示
  5a. 支払い失敗 → エラーメッセージを表示
```

#### 3. プロトタイピング

- ワイヤーフレームやモックアップを作成して確認する
- Figma、Adobe XD等のツールを活用
- 早い段階で認識を合わせられる

#### 4. 既存システムの分析

- 既存システムがある場合、現行機能の棚卸しを行う
- 「現行踏襲」だけでなく不要な機能の洗い出しも重要

## 2.3 WBS（Work Breakdown Structure）

### WBSとは

プロジェクトの成果物を**階層的に分解**した構造図。プロジェクト全体を管理可能な単位に分割する。

### WBSの作成手順

1. **プロジェクト全体**を最上位に置く
2. **主要な成果物**（フェーズまたは機能）に分解する
3. さらに**作業パッケージ**レベルまで分解する
4. 各作業パッケージが**1人で1～5日**で完了できるサイズになるまで分解する

### WBSの例：ECサイト構築

```
ECサイト構築
├── 1. ユーザー管理
│   ├── 1.1 会員登録機能
│   │   ├── 1.1.1 登録フォーム作成
│   │   ├── 1.1.2 メール認証実装
│   │   └── 1.1.3 バリデーション実装
│   ├── 1.2 ログイン機能
│   │   ├── 1.2.1 ログインフォーム作成
│   │   ├── 1.2.2 認証処理実装
│   │   └── 1.2.3 パスワードリセット
│   └── 1.3 プロフィール管理
│       ├── 1.3.1 プロフィール表示
│       └── 1.3.2 プロフィール編集
├── 2. 商品管理
│   ├── 2.1 商品一覧
│   ├── 2.2 商品詳細
│   ├── 2.3 商品検索
│   └── 2.4 カテゴリ管理
├── 3. カート・決済
│   ├── 3.1 カート機能
│   ├── 3.2 決済処理
│   └── 3.3 注文履歴
└── 4. インフラ
    ├── 4.1 環境構築
    ├── 4.2 CI/CDパイプライン
    └── 4.3 監視設定
```

### WBS作成のポイント

- **100%ルール**: WBSの下位レベルの合計が上位レベルの100%をカバーすること
- **相互排他**: 各要素が重複しないこと
- **適切な粒度**: 細かすぎず粗すぎない（1〜5日/人が目安）
- **成果物ベース**: 作業ではなく「何ができるか」で分解する

## 2.4 スコープクリープ

### スコープクリープとは

プロジェクト進行中に、**承認プロセスを経ずにスコープが徐々に拡大していく**現象。

### よくあるパターン

```
「ついでにこの機能も入れてほしい」
  → 小さな追加が積み重なる

「ここまで作ったなら、もう少し」
  → 成果物を見て欲望が膨らむ

「当初の要件に含まれていたはず」
  → 曖昧な要件定義が原因
```

### スコープクリープの防止策

1. **スコープを文書化する** - 含む/含まないを明記
2. **変更管理プロセスを設ける** - 変更はすべて正式な手続きを経る
3. **影響分析を行う** - 変更による時間・コストへの影響を明示
4. **ステークホルダーの合意を得る** - トレードオフを理解してもらう
5. **バックログで管理する** - 新しい要望はバックログに追加し、優先度をつける

### 変更管理の流れ

```
変更要求 → 影響分析 → レビュー → 承認/却下 → 計画更新
               ↓
         時間：+2日
         コスト：+○○円
         リスク：他機能への影響
```

## 2.5 MoSCoW優先度付け

### MoSCoWとは

要件の優先度を4段階に分類するフレームワーク。

| 分類 | 意味 | 説明 |
|------|------|------|
| **M**ust have | 必須 | これがないとリリースできない |
| **S**hould have | 重要 | 強く望まれるが、なくてもリリースは可能 |
| **C**ould have | あると嬉しい | 余裕があれば対応する |
| **W**on't have | 今回はやらない | 将来検討するが今回は対象外 |

### 適用例：ECサイトの要件

| 要件 | 優先度 | 理由 |
|------|--------|------|
| 商品一覧・詳細表示 | Must | コア機能 |
| カート・決済 | Must | コア機能 |
| ユーザー登録・ログイン | Must | 購入に必要 |
| 商品検索 | Should | UX上重要だが初期はカテゴリで代替可 |
| レビュー機能 | Could | あると嬉しいが必須ではない |
| SNSシェア | Could | マーケティング的にプラス |
| AIレコメンド | Won't | 今回は対象外、フェーズ2で検討 |

### 優先度付けのコツ

- **Must は全体の60%以下** にする（全部Mustにしない）
- ステークホルダーと一緒に優先度を決める
- **ビジネス価値 × 実装コスト** のマトリクスも併用する

```
高 | Quick Win    | 大物
ビ |（優先的に     |（計画的に
ジ |  やる）       |  やる）
ネ |--------------|-------------
ス | やらない      | 検討
価 |（コストに     |（本当に必要か
値 |  見合わない）  |  確認）
低 |              |
   +--------------+-------------
          低           高
         実装コスト
```

## 2.6 ユーザーストーリー

### ユーザーストーリーとは

ユーザー視点で要件を記述する手法。アジャイル開発で広く使われる。

### フォーマット

```
[ユーザー種別]として、
[達成したいこと]をしたい。
なぜなら[理由・価値]だからだ。
```

英語では：
```
As a [user type],
I want to [goal],
so that [benefit].
```

### 良いユーザーストーリーの例

```
顧客として、
キーワードで商品を検索したい。
なぜなら、目的の商品を素早く見つけたいからだ。
```

```
管理者として、
注文のステータスを一覧で確認したい。
なぜなら、出荷遅延を早期に発見したいからだ。
```

### 悪いユーザーストーリーの例

```
× ユーザーとして、データベースをPostgreSQLにしたい。
  → 技術的な実装の話であり、ユーザー価値が不明

× ユーザーとして、すべての機能を使いたい。
  → 曖昧すぎて何も定義できていない
```

### INVESTの原則

良いユーザーストーリーの基準：

| 基準 | 意味 | 説明 |
|------|------|------|
| **I**ndependent | 独立している | 他のストーリーに依存しない |
| **N**egotiable | 交渉可能 | 詳細は会話で詰める余地がある |
| **V**aluable | 価値がある | ユーザーやビジネスに価値をもたらす |
| **E**stimable | 見積もり可能 | 作業量を見積もれる程度に具体的 |
| **S**mall | 小さい | 1スプリントで完了できるサイズ |
| **T**estable | テスト可能 | 完了を検証できる |

## 2.7 受け入れ基準（Acceptance Criteria）

### 受け入れ基準とは

ユーザーストーリーが「完了」したとみなすための**具体的な条件**。

### Given-When-Then 形式

```
Given（前提条件）: [特定の状態であるとき]
When（操作）: [ある操作を行ったら]
Then（期待結果）: [こうなるべき]
```

### 例：商品検索のユーザーストーリーの受け入れ基準

```markdown
## ユーザーストーリー
顧客として、キーワードで商品を検索したい。

## 受け入れ基準

### AC1: 基本検索
- Given: 商品が100件登録されている
- When: 「スニーカー」で検索する
- Then: 商品名に「スニーカー」を含む商品が一覧表示される

### AC2: 検索結果なし
- Given: 商品が登録されている
- When: 存在しないキーワードで検索する
- Then: 「該当する商品が見つかりませんでした」と表示される

### AC3: 部分一致
- Given: 「ランニングシューズ」という商品がある
- When: 「ランニング」で検索する
- Then: 「ランニングシューズ」が検索結果に含まれる

### AC4: パフォーマンス
- Given: 商品が10,000件登録されている
- When: 検索を実行する
- Then: 2秒以内に結果が表示される
```

### 受け入れ基準を書くポイント

- **具体的な数値**を入れる（「速い」ではなく「2秒以内」）
- **正常系と異常系**の両方を書く
- **テスト可能**な形で書く
- 多すぎない（1ストーリーあたり3〜8個程度）

## 2.8 スコープ管理の実践フロー

```
1. ビジョン・目標の設定
   ↓
2. 要件収集（インタビュー、プロトタイプ等）
   ↓
3. 要件の整理・優先度付け（MoSCoW）
   ↓
4. ユーザーストーリーの作成
   ↓
5. 受け入れ基準の定義
   ↓
6. WBSの作成
   ↓
7. スコープベースラインの承認
   ↓
8. 変更管理（スコープクリープの防止）
```

---

## まとめ

| 概念 | ポイント |
|------|---------|
| 要件収集 | 機能要件・非機能要件・ビジネス要件を区別する |
| WBS | 成果物を階層的に分解、1〜5日/人の粒度まで |
| スコープクリープ | 変更管理プロセスで防止する |
| MoSCoW | Must/Should/Could/Won'tで優先度を明確に |
| ユーザーストーリー | ユーザー視点で「誰が・何を・なぜ」を記述 |
| 受け入れ基準 | Given-When-Thenで完了条件を具体化 |

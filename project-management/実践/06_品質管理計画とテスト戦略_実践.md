# 実践課題06：品質管理計画とテスト戦略 ★3

> **難易度**: ★★★☆☆（中級）
> **前提知識**: 第2章（スコープ管理）、第6章（品質管理）
> **課題の種類**: 設計課題
> **学習目標**: 品質目標の設定、テスト戦略の策定、品質メトリクスの定義を一貫して行い、プロジェクトの品質管理計画書を作成できるようになる

---

## 完成イメージ

以下のような品質管理計画書を作成します。

```
=============================================
  品質管理計画書
  プロジェクト: 受注管理システム
  版数: 1.0 / 作成日: 2026-04-06
=============================================

1. 品質目標
   - 本番リリース後1ヶ月以内の重大障害: 0件
   - 受入テスト合格率: 95%以上
   - ユーザー満足度: 4.0/5.0以上

2. テスト戦略
   - 単体テスト: カバレッジ80%以上
   - 結合テスト: 全API仕様の100%カバー
   - システムテスト: 全ユースケースの検証
   - 受入テスト: クライアント立会い

3. 品質メトリクス
   - バグ密度: 0.5件/KLOC以下（リリース時）
   - テストケース消化率: リリース判定基準100%
   - 障害対応時間: 重大障害4時間以内
```

---

## 課題の要件

以下のシナリオを読み、品質管理計画書を作成してください。

### シナリオ

あなたはオンライン予約システム（飲食店向け）の開発プロジェクトのPMです。

**プロジェクト概要：**
- 飲食チェーン（20店舗）の予約管理をWebシステムに統合
- 顧客がスマートフォンから予約・変更・キャンセルを行える
- 店舗スタッフが予約状況をリアルタイムで確認できる
- 期間：6ヶ月、開発者4名、テスター1名
- 非機能要件：
  - 同時接続100ユーザーに対応
  - レスポンスタイム: 2秒以内
  - 稼働率: 99.5%以上
  - 個人情報（氏名、電話番号）を扱う

### 作成すべきもの

1. **品質目標**: 機能品質・非機能品質それぞれ3つ以上の目標値
2. **テスト戦略**: テストピラミッドに基づく各レベルのテスト方針
3. **品質メトリクス**: 開発中・リリース時・リリース後のメトリクス定義
4. **レビュー計画**: コードレビュー・設計レビューの方針
5. **リリース判定基準**: Go/No-Go判定の具体的条件

## ステップガイド

<details>
<summary>ステップ1：品質目標を設定する</summary>

品質目標は「測定可能」で「達成可能」なものを設定します。

```
品質目標の設定フレームワーク：

【機能品質（正しく動くか）】
  - テスト合格率: ○%以上
  - 未解決バグ: 重大○件以下、中○件以下
  - 受入テスト合格率: ○%以上

【非機能品質（どのように動くか）】
  - 性能: レスポンスタイム○秒以内
  - 可用性: 稼働率○%以上
  - セキュリティ: 脆弱性診断でCritical/High指摘0件

【プロセス品質（どのように作るか）】
  - コードレビュー実施率: ○%
  - テストカバレッジ: ○%以上
  - CI/CDパイプラインのビルド成功率: ○%以上
```

</details>

<details>
<summary>ステップ2：テスト戦略を策定する</summary>

テストピラミッドに基づいて各レベルのテスト方針を決めます。

```
テストピラミッド：

        /\
       /  \      E2Eテスト（少）
      /    \     → 主要シナリオのみ
     /------\
    /        \   統合テスト（中）
   /          \  → API間の連携、DB操作
  /------------\
 /              \ 単体テスト（多）
/________________\ → ビジネスロジック中心
```

**各レベルで決めること：**
- 対象範囲: 何をテストするか
- 担当者: 誰が実施するか
- ツール: 何を使うか
- 基準: どこまでやるか（カバレッジ等）
- タイミング: いつ実施するか

</details>

<details>
<summary>ステップ3：品質メトリクスを定義する</summary>

メトリクスは「いつ」「何を」「誰が」「どうやって」測定するかを明確にします。

```
メトリクス定義テンプレート：

  メトリクス名: バグ密度
  定義: 発見されたバグ数 ÷ コード行数（KLOC）
  目標値: 0.5件/KLOC以下
  測定タイミング: 毎週金曜
  測定者: テスター
  測定方法: バグ管理ツール + ソースコード行数から算出
  しきい値: 1.0件/KLOCを超えたら品質会議を開催
```

</details>

## 解答例

<details>
<summary>解答例（初心者向け）</summary>

```
=============================================
  品質管理計画書
  プロジェクト: オンライン予約システム（飲食店向け）
  版数: 1.0 / 作成日: 2026-04-06
=============================================

──────────────────────────────
1. 品質目標
──────────────────────────────

【機能品質目標】
  Q1: 受入テスト（UAT）合格率 95%以上
  Q2: リリース時の未解決バグ: 重大(Critical) 0件、重要(Major) 0件
  Q3: 予約の二重登録が発生しないこと（データ整合性100%）

【非機能品質目標】
  Q4: レスポンスタイム 2秒以内（95パーセンタイル）
  Q5: 同時接続100ユーザー時も正常動作
  Q6: 月間稼働率 99.5%以上
  Q7: セキュリティ診断 Critical/High指摘 0件

──────────────────────────────
2. テスト戦略
──────────────────────────────

【単体テスト】
  対象: ビジネスロジック（予約登録、空席計算、時間帯管理）
  担当: 各開発者
  基準: カバレッジ80%以上
  タイミング: 開発と同時（テスト駆動開発を推奨）

【結合テスト】
  対象: API間の連携、DB操作、外部連携
  担当: 開発者 + テスター
  基準: 全APIエンドポイントの正常系・異常系を網羅
  タイミング: 各機能の開発完了後

【システムテスト】
  対象: 全機能を通した業務シナリオ
  担当: テスター
  テストケース例:
    - 顧客が予約 → 店舗に通知 → 店舗が確認 → 顧客にメール
    - 予約変更 → 元の時間帯が解放 → 新しい時間帯が確保
    - 同時刻に複数予約が入った場合の排他制御
  タイミング: 結合テスト完了後

【性能テスト】
  対象: 同時接続100ユーザー、レスポンスタイム
  担当: テスター + 開発リーダー
  シナリオ: ピーク時（金曜18:00〜20:00相当）の負荷を再現
  タイミング: システムテスト後

【受入テスト（UAT）】
  対象: クライアント（飲食チェーン本部 + パイロット2店舗）
  担当: クライアント（テスター支援）
  基準: クライアントの業務シナリオで問題なく運用できること
  タイミング: リリース2週間前

──────────────────────────────
3. 品質メトリクス
──────────────────────────────

【開発中メトリクス】
  - コードカバレッジ: 週次計測、80%以上を維持
  - バグ検出数: 週次集計、傾向を監視
  - コードレビュー指摘件数: 週次集計

【リリース時メトリクス】
  - バグ密度: 0.5件/KLOC以下
  - テストケース消化率: 100%
  - 未解決バグ: Critical 0件、Major 0件

【リリース後メトリクス】
  - 本番障害件数: 月1件以下
  - 障害対応時間: 重大障害は4時間以内に暫定対応
  - ユーザー満足度: 導入1ヶ月後アンケートで4.0/5.0以上

──────────────────────────────
4. レビュー計画
──────────────────────────────

【コードレビュー】
  - 全コード変更にプルリクエスト必須
  - 最低1名のレビュー承認で マージ可能
  - レビュー観点: 機能正確性、セキュリティ、可読性

【設計レビュー】
  - DB設計、API設計は全員でレビュー
  - チェックリストを事前配布

──────────────────────────────
5. リリース判定基準（Go/No-Go）
──────────────────────────────

  以下のすべてを満たした場合にリリースGoとする:

  □ テストケース消化率: 100%
  □ 未解決バグ: Critical 0件、Major 0件
  □ 性能テスト: レスポンス2秒以内（95パーセンタイル）
  □ セキュリティ診断: Critical/High指摘 0件
  □ 受入テスト: クライアント承認済み
  □ 運用手順書: 作成・レビュー済み
  □ ロールバック手順: 作成・確認済み
```

</details>

<details>
<summary>解答例（改良版 ─ 品質ゲートとエスカレーション付き）</summary>

改良版では、フェーズごとの品質ゲート（通過基準）とメトリクスの閾値管理を追加します。

```
=============================================
  品質管理計画書（改良版）
  プロジェクト: オンライン予約システム（飲食店向け）
  版数: 2.0 / 作成日: 2026-04-06
=============================================

──────────────────────────────
1. 品質ゲート（フェーズ通過基準）
──────────────────────────────

各フェーズの完了時に品質ゲートレビューを実施し、
基準を満たさない場合は次のフェーズに進まない。

  ゲート1: 要件定義完了
    □ 全要件にMoSCoW優先度が付与されていること
    □ 全Must要件に受入基準（Acceptance Criteria）があること
    □ 非機能要件が数値で定義されていること
    □ クライアント署名済みの要件定義書が存在すること

  ゲート2: 設計完了
    □ 設計レビュー完了、指摘事項ゼロ（または合意済み）
    □ API仕様書がSwagger/OpenAPIで記述されていること
    □ ER図がレビュー済みであること
    □ セキュリティ設計（認証・認可・暗号化方式）が明文化されていること

  ゲート3: 開発完了
    □ 単体テストカバレッジ 80%以上
    □ 静的解析ツールのCritical指摘 0件
    □ 全コードにコードレビュー済みの記録があること

  ゲート4: テスト完了（= リリース判定）
    □ 全テストケース消化率 100%
    □ 未解決バグ: Critical 0件、Major 0件
    □ 性能テスト合格（2秒以内、100同時接続）
    □ セキュリティ診断合格
    □ UAT合格（クライアント承認）
    □ 運用マニュアル・ロールバック手順 完備

──────────────────────────────
2. メトリクス閾値管理
──────────────────────────────

各メトリクスに3段階の閾値を設け、自動でアラートする。

  メトリクス        緑(正常)      黄(注意)        赤(警告)
  ────────────────────────────────────────────────────
  バグ密度          <0.5/KLOC     0.5〜1.0/KLOC   >1.0/KLOC
  カバレッジ        >80%          70〜80%          <70%
  テスト合格率      >95%          90〜95%          <90%
  レビュー指摘件数  <5件/PR       5〜10件/PR       >10件/PR
  ビルド成功率      >95%          90〜95%          <90%

  黄: チームリーダーに報告 → 原因分析 → 改善アクション
  赤: PMに即報告 → 品質緊急会議 → 是正措置の実施

──────────────────────────────
3. テスト工数見積もり
──────────────────────────────

  テストレベル    工数(人日)  テストケース数(概算)  備考
  ─────────────────────────────────────────────────
  単体テスト      10         約200件            開発者が実施
  結合テスト       8         約80件             テスター主導
  システムテスト   5         約50件             業務シナリオ
  性能テスト       3         5シナリオ          ツール使用
  セキュリティ     2         -                 外部診断
  UAT              5         約30件             クライアント
  ─────────────────────────────────────────────────
  合計            33人日      約365件

  ※テスト工数は全体工数の約25〜30%が目安。
    今回の開発工数90人日に対し33人日（37%）はやや多めだが、
    個人情報を扱うシステムであることを考慮し妥当と判断。

──────────────────────────────
4. 欠陥管理プロセス
──────────────────────────────

  バグ発見 → 登録 → 分類 → 対応 → 検証 → クローズ

  重要度分類と対応期限：
    Critical(致命的): 業務停止レベル → 即日対応
    Major(重要):      主要機能の障害 → 2営業日以内
    Minor(軽微):      操作上の不便   → 次スプリントで対応
    Trivial(微細):    表記揺れ等     → バックログに積む

  エスカレーション基準：
    - Critical バグが1件でも検出 → PM + クライアントに即報告
    - Major バグが同一機能で3件以上 → 設計見直しを検討
    - バグ検出速度が収束しない → テスト期間の延長を検討
```

**初心者向けとの違い：**
- **品質ゲート**: フェーズ間に「関所」を設けることで、品質問題を後工程に持ち越さない仕組みです
- **閾値管理**: メトリクスに段階的な閾値を設けることで、問題の早期検知と迅速な対応が可能になります
- **テスト工数見積もり**: テストにも工数がかかることを明示し、スケジュールに組み込みます
- **欠陥管理プロセス**: バグの重要度分類と対応期限を決めておくことで、混乱なく対処できます

</details>

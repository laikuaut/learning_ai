# 第6章 システム開発 演習問題

## 基本問題

### 問1（四択・要件定義）
非機能要件に該当するものはどれか。

ア 商品の検索機能を提供すること
イ 注文データを月次で集計しレポート出力すること
ウ ログイン時に多要素認証を使用すること
エ 画面の応答時間を3秒以内とすること

<details><summary>解答・解説</summary>

**正解: エ**

- ア: 機能要件（システムが提供する機能）
- イ: 機能要件（レポート出力機能）
- ウ: 機能要件に近い（認証方式の指定だが、セキュリティ要件として非機能要件とも言える）
- エ: 非機能要件（性能に関する要件）

非機能要件は「システムがどのように動作するか」を定義するもので、性能、可用性、セキュリティ、運用性などが含まれます。応答時間の指定は性能要件であり、代表的な非機能要件です。

</details>

### 問2（四択・モジュール結合度）
モジュール結合度が最も弱い（最も望ましい）ものはどれか。

ア 共通結合　　イ データ結合　　ウ 制御結合　　エ スタンプ結合

<details><summary>解答・解説</summary>

**正解: イ データ結合**

結合度の弱い順（望ましい順）:
データ結合 < スタンプ結合 < 制御結合 < 外部結合 < 共通結合 < 内容結合

データ結合は必要なデータ項目のみを引数として受け渡す方式で、最も独立性が高いです。

</details>

### 問3（四択・テスト）
境界値分析の対象として最も適切なテストケースの組合せはどれか。入力値の有効範囲が1から100の整数の場合。

ア 0, 1, 50, 100, 101
イ -1, 0, 50, 100, 200
ウ 1, 25, 50, 75, 100
エ 0, 1, 99, 100, 101

<details><summary>解答・解説</summary>

**正解: ア 0, 1, 50, 100, 101**

境界値分析では、有効範囲の境界とその前後の値をテストします。
- 下限の境界: 0（無効）, 1（有効）
- 上限の境界: 100（有効）, 101（無効）
- 中間値: 50（有効）

エも境界値を含んでいますが、アの方が中間値も含み、より典型的な境界値分析のテストケースです。

</details>

### 問4（四択・カバレッジ）
プログラムの全ての分岐について、真と偽の両方を少なくとも1回ずつ実行するテストカバレッジ基準はどれか。

ア 命令網羅（C0）　　イ 分岐網羅（C1）　　ウ 条件網羅（C2）　　エ パス網羅

<details><summary>解答・解説</summary>

**正解: イ 分岐網羅（C1）**

- 命令網羅（C0）: 全命令を最低1回実行
- 分岐網羅（C1）: 全分岐の真/偽を最低1回実行（判定条件網羅とも呼ぶ）
- 条件網羅（C2）: 各条件式の真/偽を最低1回実行
- パス網羅: 全実行パスを実行

</details>

### 問5（四択・CMMI）
CMMIの成熟度レベルで、組織標準プロセスが定義され全プロジェクトで使用されているレベルはどれか。

ア レベル2（管理された）
イ レベル3（定義された）
ウ レベル4（定量的に管理された）
エ レベル5（最適化している）

<details><summary>解答・解説</summary>

**正解: イ レベル3（定義された）**

- レベル2: プロジェクト単位での基本的な管理
- レベル3: 組織標準プロセスの定義と全プロジェクトでの適用
- レベル4: プロセスの定量的な測定・管理
- レベル5: 継続的なプロセス改善

</details>

## 応用問題

### 問6（四択・レビュー技法）
ソフトウェアレビュー技法のうち、モデレータが主導し、チェックリストを使って欠陥を検出する最も厳密な手法はどれか。

ア ウォークスルー　　イ インスペクション　　ウ パスアラウンド　　エ アドホックレビュー

<details><summary>解答・解説</summary>

**正解: イ インスペクション**

インスペクションは最も厳密な公式レビュー手法で、以下の特徴があります:
- モデレータが全体を管理・進行
- チェックリストに基づく体系的な検査
- 役割が明確（モデレータ、読み手、記録係、レビューア）
- 計画→準備→会議→修正→確認の5ステップ

ウォークスルーは作成者主導のレビューで、インスペクションより非公式です。

</details>

### 問7（四択・FP法）
ファンクションポイント法の入力項目に含まれないものはどれか。

ア 外部入力（EI）　　イ 外部出力（EO）　　ウ コード行数（LOC）　　エ 内部論理ファイル（ILF）

<details><summary>解答・解説</summary>

**正解: ウ コード行数（LOC）**

FP法の5つの入力項目:
1. 外部入力（EI）
2. 外部出力（EO）
3. 外部照会（EQ）
4. 内部論理ファイル（ILF）
5. 外部インタフェースファイル（EIF）

LOC（コード行数）はFP法とは別の規模見積り手法です。FP法は言語に依存しない規模見積りが特徴です。

</details>

### 問8（記述・テスト設計）
次の仕様に基づき、デシジョンテーブルを作成せよ。

**仕様:**
- 会員区分（一般会員/プレミアム会員）
- 購入金額（5,000円以上/5,000円未満）
- 割引ルール:
  - プレミアム会員かつ5,000円以上: 15%割引
  - プレミアム会員かつ5,000円未満: 10%割引
  - 一般会員かつ5,000円以上: 5%割引
  - 一般会員かつ5,000円未満: 割引なし

<details><summary>解答・解説</summary>

**デシジョンテーブル:**

| | ルール1 | ルール2 | ルール3 | ルール4 |
|---|:---:|:---:|:---:|:---:|
| **条件** | | | | |
| プレミアム会員か | Y | Y | N | N |
| 5,000円以上か | Y | N | Y | N |
| **動作** | | | | |
| 15%割引 | X | | | |
| 10%割引 | | X | | |
| 5%割引 | | | X | |
| 割引なし | | | | X |

テストケース数: 条件が2つで各2値なので 2² = 4パターン。全パターンを網羅しています。

テストケースの具体例:
1. プレミアム会員、購入金額6,000円 → 15%割引（900円引き → 5,100円）
2. プレミアム会員、購入金額3,000円 → 10%割引（300円引き → 2,700円）
3. 一般会員、購入金額8,000円 → 5%割引（400円引き → 7,600円）
4. 一般会員、購入金額2,000円 → 割引なし（2,000円）

</details>

## チャレンジ問題

### 問9（記述・午後試験形式）
E社の新システム開発プロジェクトにおけるテスト計画について、以下の設問に答えなさい。

E社は受発注管理システムをリプレースする計画を進めている。テスト工程において、単体テスト、結合テスト、システムテストを実施する予定である。

**設問1:** V字モデルに基づき、単体テスト、結合テスト、システムテストがそれぞれ対応する設計工程を述べ、各テストの目的を説明せよ。

**設問2:** 結合テストで使用するスタブとドライバの違いを説明し、トップダウンテストとボトムアップテストのそれぞれで使用するものを述べよ。

**設問3:** テスト中にバグ曲線が直線的に増加し続ける場合、考えられる原因と対応策を2つ述べよ。

<details><summary>解答・解説</summary>

**設問1の解答:**

| テスト工程 | 対応する設計工程 | 目的 |
|-----------|----------------|------|
| 単体テスト | 内部設計（詳細設計） | 個々のモジュールが内部設計通りに正しく動作するか検証する |
| 結合テスト | 外部設計（基本設計） | 複数のモジュールを組み合わせた際のインタフェースや連携が外部設計通りに正しく動作するか検証する |
| システムテスト | 要件定義 | システム全体が要件定義の機能要件・非機能要件を満たしているか検証する |

**設問2の解答:**

- **スタブ（Stub）:** テスト対象モジュールが呼び出す下位モジュールの代替として使用するダミーモジュール。テスト対象が必要とするデータを返す
- **ドライバ（Driver）:** テスト対象モジュールを呼び出す上位モジュールの代替として使用するダミーモジュール。テスト対象にデータを渡して実行する

| テスト方式 | 使用するもの | 理由 |
|-----------|------------|------|
| トップダウンテスト | スタブ | 上位モジュールから順にテストするため、未完成の下位モジュールの代わりにスタブが必要 |
| ボトムアップテスト | ドライバ | 下位モジュールから順にテストするため、未完成の上位モジュールの代わりにドライバが必要 |

**設問3の解答:**

原因1: **ソフトウェアの品質が根本的に低い**
- 設計段階での問題が多く、実装にも多数の欠陥が含まれている
- 対応策: テストを一旦中断し、設計レビューを再実施する。重要なモジュールはコードレビュー（インスペクション）を行い、根本的な品質改善を図る

原因2: **テストの網羅性が不十分で、テスト範囲を拡大している**
- テスト設計が不十分で、テスト実施中にテストケースを追加し続けているため、新しい領域のバグが継続的に発見されている
- 対応策: テスト計画を見直し、カバレッジ基準を明確にしたテスト設計を行う。テスト対象の範囲と完了基準を再定義し、計画的にテストを実施する

</details>

### 問10（記述・午後試験形式）
アジャイル開発（スクラム）の導入を検討しているF社について、以下の設問に答えなさい。

F社はこれまでウォーターフォールモデルで開発を行ってきたが、顧客の要件変更が頻繁に発生し、納期遅延が問題となっている。

**設問1:** ウォーターフォールモデルとスクラムのそれぞれの特徴を比較し、F社の課題に対してスクラムが適している理由を述べよ。

**設問2:** スクラムの3つの役割（プロダクトオーナー、スクラムマスター、開発チーム）の責務をそれぞれ説明せよ。

<details><summary>解答・解説</summary>

**設問1の解答:**

| 項目 | ウォーターフォール | スクラム |
|------|----------------|---------|
| 要件の確定 | プロジェクト初期に全要件を確定 | スプリントごとに要件を詳細化・変更可能 |
| 変更への対応 | 変更コストが大きい（手戻りが発生） | 変更を前提とした仕組み |
| 成果物の提供 | プロジェクト終了時にまとめて | スプリントごとに動作する成果物を提供 |
| 顧客フィードバック | 最終段階で確認 | スプリントレビューで定期的に確認 |

F社の課題は「顧客の要件変更が頻繁に発生する」ことです。スクラムでは1-4週間のスプリントごとに優先度の高い機能から開発し、スプリントレビューで顧客のフィードバックを得られるため、要件変更を自然にプロセスに取り込むことができます。また、動作するソフトウェアを早期に提供できるため、顧客の期待とのずれを早期に検出・修正できます。

**設問2の解答:**

| 役割 | 責務 |
|------|------|
| プロダクトオーナー | プロダクトバックログ（開発すべき機能の一覧）の管理と優先順位の決定を行う。プロダクトの価値を最大化する責任を持ち、ステークホルダーと開発チームの間の窓口となる |
| スクラムマスター | スクラムプロセスが正しく実施されるように促進し、開発チームの障害を取り除くサーバントリーダーの役割を担う。チームの自己組織化を支援する |
| 開発チーム | スプリント内で計画した機能を開発・テストし、動作するインクリメント（成果物）を作成する。自己組織化チームとして、スプリント目標の達成方法を自ら決定する |

</details>

### 問題1：ER図の読み取り

以下のER図を読み取り、質問に答えてください。

```
┌──────────┐       ┌──────────┐       ┌──────────┐
│ students │       │enrollments│      │ courses  │
├──────────┤       ├──────────┤       ├──────────┤
│*id       │──1:N─→│*id       │←─N:1─│*id       │
│ name     │       │ student_id│      │ name     │
│ email    │       │ course_id │      │ teacher_id│
│ grade    │       │ enrolled_at│     │ capacity │
└──────────┘       │ score     │      └────┬─────┘
                   └──────────┘           │ N:1
                                          ▼
                                    ┌──────────┐
                                    │ teachers │
                                    ├──────────┤
                                    │*id       │
                                    │ name     │
                                    │ department│
                                    └──────────┘
```

1. studentsとcoursesの関係は何ですか（1対1、1対多、多対多）？
2. enrollmentsテーブルの役割は何ですか？
3. 1つのコースに対して講師は何人割り当てられますか？
4. あるStudentが受講しているコースの一覧を取得するSQLを書いてください。

<details>
<summary>解答例</summary>

```
1. 多対多（Many-to-Many）
   → 1人の学生が複数のコースを受講でき、
     1つのコースに複数の学生が登録できる。

2. 中間テーブル（Junction Table / 関連テーブル）
   → studentsとcoursesの多対多の関係を実現するためのテーブル。
     どの学生がどのコースに登録しているかを管理する。
     追加情報として登録日（enrolled_at）と成績（score）を持つ。

3. 1人
   → coursesテーブルにteacher_idがあり、N:1の関係なので、
     1つのコースには1人の講師が割り当てられる。
     （1人の講師は複数のコースを担当できる）

4. SQL:
   SELECT c.id, c.name
   FROM courses c
   INNER JOIN enrollments e ON c.id = e.course_id
   WHERE e.student_id = 1;

   -- または
   SELECT c.id, c.name, e.enrolled_at, e.score
   FROM courses c
   INNER JOIN enrollments e ON c.id = e.course_id
   INNER JOIN students s ON s.id = e.student_id
   WHERE s.id = 1;
```

</details>

### 問題2：テーブル定義書の作成

以下の要件に基づいて、テーブル定義書（カラム定義）を作成してください。

**要件：** 書籍管理テーブル（books）

- 書籍ID（自動採番の主キー）
- 書籍タイトル（最大200文字、必須）
- 著者名（最大100文字、必須）
- ISBN（13桁、一意制約、必須）
- 出版日（日付型）
- 価格（整数、0以上）
- カテゴリID（外部キー、categoriesテーブルを参照）
- 在庫数（整数、0以上、デフォルト0）
- 作成日時・更新日時

テーブル定義書のフォーマットで記述してください。

<details>
<summary>ヒント</summary>

各カラムのデータ型（VARCHAR, INTEGER, DATE, TIMESTAMP等）、NOT NULL制約、デフォルト値、制約（CHECK, UNIQUE, FOREIGN KEY）を考えましょう。

</details>

<details>
<summary>解答例</summary>

```markdown
## テーブル：books（書籍）

### カラム定義

| No | カラム名 | 論理名 | データ型 | NOT NULL | デフォルト | 説明 |
|----|---------|--------|---------|----------|-----------|------|
| 1 | id | 書籍ID | SERIAL | ◯ | 自動採番 | 主キー |
| 2 | title | 書籍タイトル | VARCHAR(200) | ◯ | - | 書籍のタイトル |
| 3 | author | 著者名 | VARCHAR(100) | ◯ | - | 著者の氏名 |
| 4 | isbn | ISBN | CHAR(13) | ◯ | - | 国際標準図書番号。一意制約 |
| 5 | published_date | 出版日 | DATE | × | NULL | 出版された日付 |
| 6 | price | 価格 | INTEGER | × | NULL | 税抜価格（円）。0以上 |
| 7 | category_id | カテゴリID | INTEGER | × | NULL | categoriesテーブルへの外部キー |
| 8 | stock | 在庫数 | INTEGER | ◯ | 0 | 現在の在庫数。0以上 |
| 9 | created_at | 作成日時 | TIMESTAMP | ◯ | CURRENT_TIMESTAMP | レコード作成日時 |
| 10 | updated_at | 更新日時 | TIMESTAMP | ◯ | CURRENT_TIMESTAMP | レコード更新日時 |

### 制約

| 制約名 | 種類 | カラム | 説明 |
|--------|------|--------|------|
| books_pkey | PRIMARY KEY | id | 主キー |
| books_isbn_key | UNIQUE | isbn | ISBNの一意制約 |
| books_category_id_fkey | FOREIGN KEY | category_id | categories(id)を参照 |
| books_price_check | CHECK | price | price >= 0 |
| books_stock_check | CHECK | stock | stock >= 0 |

### DDL

CREATE TABLE books (
    id             SERIAL       PRIMARY KEY,
    title          VARCHAR(200) NOT NULL,
    author         VARCHAR(100) NOT NULL,
    isbn           CHAR(13)     NOT NULL UNIQUE,
    published_date DATE,
    price          INTEGER      CHECK (price >= 0),
    category_id    INTEGER      REFERENCES categories(id),
    stock          INTEGER      NOT NULL DEFAULT 0 CHECK (stock >= 0),
    created_at     TIMESTAMP    NOT NULL DEFAULT CURRENT_TIMESTAMP,
    updated_at     TIMESTAMP    NOT NULL DEFAULT CURRENT_TIMESTAMP
);

CREATE INDEX books_category_id_idx ON books (category_id);
CREATE INDEX books_author_idx ON books (author);
CREATE INDEX books_published_date_idx ON books (published_date);
```

### 問題3：正規化

以下の非正規形のテーブルを第3正規形まで正規化してください。

**非正規形テーブル：受注データ**

| 受注ID | 受注日 | 顧客名 | 顧客住所 | 商品名1 | 単価1 | 数量1 | 商品名2 | 単価2 | 数量2 |
|--------|--------|--------|---------|---------|-------|-------|---------|-------|-------|
| 1001 | 2026-04-01 | 山田太郎 | 東京都... | りんご | 200 | 3 | みかん | 150 | 5 |
| 1002 | 2026-04-02 | 佐藤花子 | 大阪府... | りんご | 200 | 2 | - | - | - |

<details>
<summary>ヒント</summary>

- 第1正規形：繰り返しグループ（商品1、商品2...）を排除する
- 第2正規形：部分関数従属を排除する
- 第3正規形：推移的関数従属を排除する

それぞれのステップで何をどう分割するか考えましょう。

</details>

<details>
<summary>解答例</summary>

```
■ 第1正規形（繰り返しグループの排除）

商品の繰り返し（商品名1,2...）を別の行に分ける。

受注テーブル（1NF）:
| 受注ID | 受注日 | 顧客名 | 顧客住所 | 商品名 | 単価 | 数量 |
|--------|--------|--------|---------|--------|------|------|
| 1001 | 2026-04-01 | 山田太郎 | 東京都... | りんご | 200 | 3 |
| 1001 | 2026-04-01 | 山田太郎 | 東京都... | みかん | 150 | 5 |
| 1002 | 2026-04-02 | 佐藤花子 | 大阪府... | りんご | 200 | 2 |

■ 第2正規形（部分関数従属の排除）

複合キー（受注ID, 商品名）の一部にだけ依存するカラムを分離。
- 顧客名、顧客住所、受注日は受注IDだけで決まる → 受注テーブルへ
- 単価は商品名だけで決まる → 商品テーブルへ

受注テーブル:
| *受注ID | 受注日 | 顧客名 | 顧客住所 |

受注明細テーブル:
| *受注ID | *商品ID | 数量 |

商品テーブル:
| *商品ID | 商品名 | 単価 |

■ 第3正規形（推移的関数従属の排除）

受注テーブル内で、顧客住所は顧客名で決まる（推移的関数従属）。
→ 顧客テーブルに分離。

受注テーブル:
| *受注ID | 受注日 | 顧客ID |

顧客テーブル:
| *顧客ID | 顧客名 | 顧客住所 |

受注明細テーブル:
| *受注ID | *商品ID | 数量 |

商品テーブル:
| *商品ID | 商品名 | 単価 |

■ 最終結果のER図:

  顧客 ──1:N─→ 受注 ──1:N─→ 受注明細 ←─N:1── 商品
```

</details>

## 応用問題

### 問題4：インデックス設計

以下のテーブルとクエリパターンに基づいて、必要なインデックスを設計してください。

**テーブル：** orders（注文テーブル）

```sql
CREATE TABLE orders (
    id          SERIAL PRIMARY KEY,
    user_id     INTEGER NOT NULL,
    status      VARCHAR(20) NOT NULL,
    total_amount INTEGER NOT NULL,
    ordered_at  TIMESTAMP NOT NULL DEFAULT CURRENT_TIMESTAMP,
    created_at  TIMESTAMP NOT NULL DEFAULT CURRENT_TIMESTAMP
);
```

**よく実行されるクエリ：**

```sql
-- Q1: 特定ユーザーの注文一覧（注文日の新しい順）
SELECT * FROM orders WHERE user_id = ? ORDER BY ordered_at DESC;

-- Q2: ステータスで絞り込み
SELECT * FROM orders WHERE status = 'pending';

-- Q3: 期間指定で注文を検索
SELECT * FROM orders WHERE ordered_at BETWEEN '2026-04-01' AND '2026-04-30';

-- Q4: 特定ユーザーの特定ステータスの注文
SELECT * FROM orders WHERE user_id = ? AND status = ?;

-- Q5: 売上集計（月次）
SELECT DATE_TRUNC('month', ordered_at) AS month, SUM(total_amount)
FROM orders GROUP BY month;
```

各クエリに適したインデックスを提案し、理由を説明してください。

<details>
<summary>解答例</summary>

```markdown
## インデックス設計

### 提案するインデックス

| No | インデックス名 | カラム | 対応クエリ |
|----|---------------|--------|-----------|
| 1 | orders_user_ordered_idx | (user_id, ordered_at DESC) | Q1, Q4 |
| 2 | orders_status_idx | (status) | Q2 |
| 3 | orders_ordered_at_idx | (ordered_at) | Q3, Q5 |

### 各インデックスの理由

1. orders_user_ordered_idx (user_id, ordered_at DESC)
   → Q1を最も効率的に処理できる複合インデックス。
     user_idで絞り込み、ordered_atでソート済みのデータを返せる。
     Q4のuser_id + status検索にも、user_idの部分で利用可能。

   CREATE INDEX orders_user_ordered_idx
   ON orders (user_id, ordered_at DESC);

2. orders_status_idx (status)
   → Q2のステータス検索に使用。
     ただし、statusのカーディナリティ（種類数）が低い場合
     （例：3-5種類）は効果が限定的。
     レコード数が多い場合に有効。

   CREATE INDEX orders_status_idx ON orders (status);

3. orders_ordered_at_idx (ordered_at)
   → Q3の範囲検索とQ5の集計に使用。
     B-Treeインデックスは範囲検索に適している。

   CREATE INDEX orders_ordered_at_idx ON orders (ordered_at);

### 作成しないインデックスとその理由

- (user_id, status) の複合インデックス
  → Q4には有効だが、Q1用のインデックスのuser_id部分で
    ある程度対応できるため、インデックスの増加を避ける。
    Q4の頻度が非常に高い場合は追加を検討する。

- total_amount のインデックス
  → 現在のクエリパターンでは金額での検索がないため不要。
```

</details>

### 問題5：データベース設計（総合問題）

以下のシステムのデータベース設計を行ってください。

**システム：** レストラン予約管理システム

**要件：**
- レストラン情報（名前、住所、電話番号、席数、営業時間）
- メニュー（料理名、価格、カテゴリ、写真URL）
- 顧客情報（名前、電話番号、メールアドレス）
- 予約情報（どの顧客が、どのレストランに、いつ、何名で予約）
- 予約時にコースメニューを選択可能

以下の3つを作成してください：
1. ER図（テキストベース）
2. テーブル一覧
3. 主要テーブル2つのカラム定義

<details>
<summary>ヒント</summary>

予約とメニューの関係は多対多になります（1つの予約で複数のコースを選択でき、1つのコースは複数の予約で選ばれる）。中間テーブルが必要です。

</details>

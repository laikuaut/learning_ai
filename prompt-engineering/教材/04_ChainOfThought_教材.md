# 第4章：Chain of Thought（思考の連鎖）

## この章のゴール

- Chain of Thought（CoT）プロンプティングの仕組みと効果を理解する
- Zero-Shot CoTとFew-Shot CoTの使い分けを習得する
- 段階的推論の適用方法を学ぶ
- 自己整合性（Self-Consistency）の考え方を理解する
- CoTが効果的なタスクと限界を把握する

---

## 4.1 Chain of Thoughtとは

### 基本概念

Chain of Thought（CoT: 思考の連鎖）は、LLMに**回答に至るまでの推論過程を明示的に出力させる**テクニックです。2022年にGoogleの研究チームが発表し、特に推論・計算・論理タスクで大幅な精度向上が確認されました。

```
通常のプロンプティング：
  質問 → 回答（推論過程が見えない）

Chain of Thought：
  質問 → 思考ステップ1 → 思考ステップ2 → ... → 回答
  （推論過程を明示的に出力）
```

### CoTの効果

```
■ CoTなし：
Q: 「カフェで3人がコーヒーを注文しました。コーヒーは1杯480円です。
    さらに2人がケーキ（350円）も追加しました。合計はいくらですか？」
A: 「2,140円です」  ← 正しいが、どう計算したか不明。間違いの検証も困難

■ CoTあり：
Q:（同上）
A: 「ステップバイステップで考えます。

    1. コーヒーの合計：480円 × 3人 = 1,440円
    2. ケーキの合計：350円 × 2人 = 700円
    3. 総合計：1,440円 + 700円 = 2,140円

    合計は2,140円です。」
    ← 推論過程が見えるため、検証可能
```

## 4.2 Zero-Shot CoT

### 最もシンプルなCoT

「**ステップバイステップで考えてください**」と一文を追加するだけで、LLMの推論能力が向上します。これがZero-Shot CoTです。

```
Zero-Shot CoTの基本形：

「[質問内容]

 ステップバイステップで考えてください。」
```

### 効果的なトリガーフレーズ

| フレーズ | 用途 |
|---------|------|
| 「ステップバイステップで考えてください」 | 汎用的。最も一般的 |
| 「段階を追って推論してください」 | 論理的な分析タスク |
| 「順を追って説明してください」 | 教育的な説明タスク |
| 「まず問題を分解してから解いてください」 | 複雑な問題 |
| "Let's think step by step." | 英語での利用。原著論文のフレーズ |

### Zero-Shot CoTの実例

```
Q: あるプロジェクトでは、5人のエンジニアが20日間で完了する
   作業量があります。途中で2人が別プロジェクトに異動し、
   残り3人で作業を続ける場合、合計何日かかりますか？

   ステップバイステップで考えてください。

A: ステップバイステップで考えます。

   ステップ1：全体の作業量を計算する
   5人 × 20日 = 100人日（全体の作業量）

   ステップ2：状況を整理する
   最初は5人で作業を開始。途中で2人が異動し、3人になる。
   ただし「途中で」のタイミングが明示されていないため、
   開始時から2人が異動したと仮定します。

   ステップ3：3人で作業した場合の日数
   100人日 ÷ 3人 ≈ 33.3日

   ステップ4：結論
   3人で作業する場合、約34日（切り上げ）かかります。

   ※ 「途中で」異動のタイミングにより結果は変わります。
      例えば10日後に異動した場合：
      最初の10日：5人 × 10日 = 50人日消化
      残り50人日 ÷ 3人 ≈ 17日
      合計 = 10 + 17 = 27日
```

> **ポイントまとめ**: 「ステップバイステップで考えてください」の一文を追加するだけで推論精度が上がります。コストが低く、最初に試すべきテクニックです。

## 4.3 Few-Shot CoT

### Few-Shot CoTとは

Zero-Shot CoTは「考えて」と指示するだけですが、Few-Shot CoTでは**推論過程の例も含めて**提示します。LLMにどのように考えるべきかの「型」を示すイメージです。

```
Few-Shot CoTの構造：

例1：[問題] → [推論過程] → [回答]
例2：[問題] → [推論過程] → [回答]
本番：[問題] → ???
```

### Few-Shot CoTの実例

```
以下の問題を、推論過程を示しながら解いてください。

### 例題1
問題：サーバーAの応答時間が200ms、サーバーBの応答時間が300msです。
ロードバランサーがA:Bに2:1の割合でリクエストを振り分ける場合、
平均応答時間はいくらですか？

推論：
1. サーバーAへのリクエスト割合は 2/3
2. サーバーBへのリクエスト割合は 1/3
3. 加重平均 = 200ms × (2/3) + 300ms × (1/3)
4. = 133.3ms + 100ms
5. = 233.3ms

回答：平均応答時間は約233msです。

### 例題2
問題：あるAPIのSLAが99.9%の可用性を保証しています。
年間のダウンタイムは最大何分ですか？

1. 年間の総時間 = 365日 × 24時間 × 60分 = 525,600分
2. 稼働率99.9%なので、ダウンタイム率は0.1%
3. 最大ダウンタイム = 525,600分 × 0.001
4. = 525.6分
5. = 約8時間45分

回答：年間の最大ダウンタイムは約526分（約8時間45分）です。
```

## 4.4 段階的推論の応用

### 問題分解（Decomposition）

大きな問題を小さなサブ問題に分解してから解くアプローチです。

```
以下の設計判断を段階的に検討してください。

問題：ECサイトのバックエンドをモノリスからマイクロサービスに
移行すべきかどうか判断してください。

ステップ1：現状の課題を整理してください
ステップ2：モノリスのメリット・デメリットを列挙してください
ステップ3：マイクロサービスのメリット・デメリットを列挙してください
ステップ4：移行コストと期間を概算してください
ステップ5：判断基準を定義してください
ステップ6：上記を総合して推奨を述べてください
```

### 逆方向推論（Working Backwards）

ゴールから逆算して推論するアプローチです。

```
以下のデバッグ問題を、逆方向に推論して原因を特定してください。

症状：本番環境でユーザーのログインが10秒以上かかっている。
開発環境では1秒以内で完了する。

ステップ1：最終的な症状は何か？
  → ログインに10秒以上かかる

ステップ2：直接的な原因は何が考えられるか？
  → （回答させる）

ステップ3：それぞれの原因の更に上流の原因は？
  → （回答させる）

ステップ4：開発環境と本番環境の差異から、最も可能性の高い
原因を特定してください
  → （回答させる）
```

### 多角的推論

同じ問題を複数の観点から分析させます。

```
以下の技術的判断について、3つの異なる観点から分析してください。

判断：新しいプロジェクトでTypeScriptを採用すべきか

観点1（開発速度の視点）：TypeScriptの導入は開発速度にどう影響するか？
観点2（品質の視点）：TypeScriptは品質にどう寄与するか？
観点3（チームの視点）：チームのスキルセットや学習コストはどうか？

それぞれの観点で推論し、最後に総合的な判断を述べてください。
```

> **ポイントまとめ**: 問題分解・逆方向推論・多角的推論を使い分け、複雑な問題に対するLLMの推論精度を上げましょう。

## 4.5 自己整合性（Self-Consistency）

### 自己整合性とは

同じ質問に対して**複数の推論パスを生成し、最も多く出現した回答を採用する**手法です。1回の推論で間違えても、複数回推論すれば正しい答えが多数派になるという考え方です。

```
自己整合性の流れ：

質問
 ├── 推論パス1 → 回答A
 ├── 推論パス2 → 回答B
 ├── 推論パス3 → 回答A
 ├── 推論パス4 → 回答A
 └── 推論パス5 → 回答B

多数決 → 回答A（3/5 = 60%）を採用
```

### プロンプトでの活用

```
以下の問題を3つの異なるアプローチで解いてください。
それぞれの推論過程を示し、最後に最も信頼性の高い回答を
選択してください。

問題：あるWebサービスのレスポンスタイムが突然悪化した。
最も可能性の高い原因は何か？

アプローチ1（インフラの観点）：
[推論]

アプローチ2（アプリケーションの観点）：
[推論]

アプローチ3（外部要因の観点）：
[推論]

総合判断：3つの推論を比較し、最も可能性の高い原因を特定してください。
```

## 4.6 CoTが効果的なタスクと限界

### 効果的なタスク

| タスク | 例 | CoTの効果 |
|--------|-----|---------|
| **数学・計算** | 複利計算、統計計算 | 高 |
| **論理推論** | 条件分岐の多い判断 | 高 |
| **コード分析** | バグの原因特定 | 高 |
| **意思決定** | 技術選定、設計判断 | 中〜高 |
| **問題解決** | デバッグ、障害分析 | 高 |

### 効果が限定的なタスク

| タスク | 理由 |
|--------|------|
| 単純な事実の検索 | 推論不要で、知識の有無が問題 |
| 創造的な文章生成 | 論理的推論より感性が重要 |
| 翻訳 | 言語知識が主で、推論の恩恵が小さい |
| 感情分析（単純なもの） | Few-Shotで十分 |

> **ポイントまとめ**: CoTは推論・計算・論理タスクで特に効果的です。単純なタスクでは不要なトークンを消費するだけなので、使い分けが大切です。

## まとめ

```
Chain of Thoughtの重要ポイント：

✅ CoTは「推論過程を明示的に出力させる」テクニック
✅ Zero-Shot CoT: 「ステップバイステップで考えてください」を追加
✅ Few-Shot CoT: 推論過程の例を含めて提示する
✅ 問題分解・逆方向推論・多角的推論で複雑な問題に対応
✅ 自己整合性: 複数の推論パスで信頼性を向上
✅ 数学・論理・コード分析に特に効果的
✅ 単純なタスクには不要（トークンの無駄遣い）
```

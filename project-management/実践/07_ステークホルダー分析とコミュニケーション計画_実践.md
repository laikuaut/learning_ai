# 実践課題07：ステークホルダー分析とコミュニケーション計画 ★3

> **難易度**: ★★★☆☆（中級）
> **前提知識**: 第1章（PM概論）、第7章（コミュニケーション管理）
> **課題の種類**: 分析演習
> **学習目標**: ステークホルダーの関心と影響力を分析し、それに基づいた効果的なコミュニケーション計画を策定できるようになる

## 課題の要件

以下のシナリオを読み、ステークホルダー分析とコミュニケーション計画を作成してください。

### シナリオ

あなたは大手小売企業のIT部門のPMです。以下のプロジェクトを任されました。

**プロジェクト概要：**
- 内容：社内の在庫管理システムのリプレース（レガシー → クラウド移行）
- 期間：12ヶ月
- 予算：5,000万円
- 影響範囲：全国30店舗 + 物流倉庫3拠点 + 本社

**登場するステークホルダー：**

| # | ステークホルダー | 組織 | 役割の概要 |
|---|----------------|------|-----------|
| 1 | 取締役CTO | 経営層 | IT投資の最終承認者。コスト意識が高い |
| 2 | IT部門長 | IT部門 | あなたの上司。プロジェクトのスポンサー |
| 3 | 店舗運営部長 | 営業部門 | 30店舗の運営責任者。現行システムに不満あり |
| 4 | 物流センター長 | 物流部門 | 倉庫3拠点の責任者。システム変更に慎重 |
| 5 | 経理部長 | 経理部門 | 在庫金額の管理に関心。正確なデータを求める |
| 6 | 情報セキュリティ担当 | IT部門 | クラウド移行のセキュリティを審査する立場 |
| 7 | 店舗スタッフ（代表3名） | 店舗 | 日常的にシステムを使うエンドユーザー |
| 8 | 開発チーム（5名） | IT部門/外部 | 設計・開発を担当 |
| 9 | クラウドベンダー営業担当 | 外部 | クラウド基盤を提供するベンダー |
| 10 | 現行システム保守ベンダー | 外部 | レガシーシステムの保守を担当（移行後は契約終了） |

### 作成すべきもの

1. **ステークホルダー分析表**: 各ステークホルダーの関心事・影響力・関心度・現在の姿勢（支持/中立/抵抗）
2. **影響力×関心度マトリクス**: 4象限にマッピング
3. **コミュニケーション計画**: 対象者ごとの頻度・手段・内容・責任者
4. **対立（コンフリクト）が予想される場面と対処方針**

## ステップガイド

<details>
<summary>ステップ1：各ステークホルダーの分析</summary>

各ステークホルダーについて以下を整理します。

```
分析の観点：

【関心事】何を気にしているか？
  - コスト？ スケジュール？ 品質？ 使いやすさ？ セキュリティ？

【影響力】プロジェクトにどれだけ影響を与えられるか？
  - 高: 予算の承認権、プロジェクトの中止判断ができる
  - 中: 要件や方針に影響を与えられる
  - 低: 直接的な意思決定権はないが、協力が必要

【関心度】プロジェクトにどれだけ関心を持っているか？
  - 高: 日常業務に直結する、自分の評価に関わる
  - 中: 間接的に影響がある
  - 低: ほとんど影響がない

【姿勢】プロジェクトに対する現在の態度
  - 支持: 積極的に協力してくれる
  - 中立: 特に賛成も反対もしていない
  - 抵抗: 変化に消極的、反対の可能性がある
```

**ヒント：物流センター長は「慎重」とあるので、「抵抗」寄りの可能性があります。なぜ慎重なのかを考えましょう。**

</details>

<details>
<summary>ステップ2：マトリクスにマッピングする</summary>

4象限の対応方針を参考にマッピングします。

```
A象限（影響力高×関心度高）: 緊密に管理する
  → 頻繁なコミュニケーション、意思決定に参加してもらう

B象限（影響力低×関心度高）: 情報を提供する
  → 定期的に情報を共有し、安心感を与える

C象限（影響力高×関心度低）: 満足させる
  → 必要な情報を簡潔に提供し、不満を持たせない

D象限（影響力低×関心度低）: 監視する
  → 大きな変化があったときのみ情報提供
```

</details>

<details>
<summary>ステップ3：コミュニケーション計画を策定する</summary>

各象限の方針に基づいて、具体的なコミュニケーション手段を決めます。

```
計画で決めること：
  - Who（誰に）: 対象者
  - What（何を）: 報告内容
  - When（いつ）: 頻度・タイミング
  - How（どうやって）: 手段（対面、メール、報告書等）
  - Who（誰が）: 発信者・責任者
```

</details>

### 4. 想定されるコンフリクトと対処

```
場面1: 物流センター長がシステム移行に反対する
  原因: 繁忙期に業務が止まることへの不安
  対処:
  - 移行スケジュールを閑散期に設定する
  - 段階的移行（並行稼働期間を設ける）を提案する
  - 移行リハーサルを実施し、安全性を実証する

場面2: 店舗運営部長の追加要望が次々と出る
  原因: 現行システムへの不満が大きく、期待値が高い
  対処:
  - MoSCoW優先度付けで第1期のスコープを明確にする
  - 「第2期で対応」リストを作り、要望を受け止める
  - 変更管理プロセスを通じて正式に管理する

場面3: 保守ベンダーの協力が得られない
  原因: 契約終了後のビジネスがなくなるため、モチベーションが低い
  対処:
  - 引き継ぎ作業を契約に明記する
  - 引き継ぎ完了のマイルストーンと支払い条件を紐づける
  - 良好な関係を維持し、将来の別案件の可能性を示唆する
```

改良版では、各ステークホルダーの「現在の姿勢」と「望ましい姿勢」のギャップを分析し、エンゲージメント戦略を策定します。

### ステークホルダーエンゲージメント評価

```
ステークホルダー    現在     望ましい   ギャップ  アクション
────────────────────────────────────────────────────────
① 取締役CTO       中立  →  支持      小        ROIレポート提出
② IT部門長        支持  →  支持      なし      現状維持
③ 店舗運営部長    支持  →  支持      なし      期待値の管理
④ 物流センター長  抵抗  →  中立      大 ★     個別フォロー必須
⑤ 経理部長        中立  →  支持      小        データ品質の保証提示
⑥ セキュリティ担当 中立  →  支持      小        早期セキュリティレビュー
⑦ 店舗スタッフ    中立  →  支持      中        研修＋ヒアリング参加
⑧ 開発チーム      支持  →  支持      なし      技術的チャレンジを確保
⑨ クラウドベンダー 支持  →  支持      なし      定期レビュー
⑩ 保守ベンダー    抵抗  →  中立      大 ★     契約上の合意確保

★ = 重点対応が必要
```

### 物流センター長への個別エンゲージメント戦略

```
【現状分析】
  ・過去にシステム移行で業務が3日間止まった経験がある
  ・年末商戦期（10-12月）は絶対にシステムを変更したくない
  ・ITリテラシーは高くなく、技術的な説明だけでは不安が解消されない

【アプローチ】
  Phase 1（1-2ヶ月目）: 信頼構築
    - 現場訪問し、現行システムの困り事を丁寧にヒアリング
    - 「物流の業務を理解している」ことを示す
    - 今回の移行で物流にとってどんなメリットがあるかを整理

  Phase 2（3-4ヶ月目）: 巻き込み
    - 移行計画のドラフトをレビューしてもらう
    - 「繁忙期を避ける」スケジュールを一緒に作る
    - パイロット倉庫を選定してもらう（当事者意識の醸成）

  Phase 3（5-8ヶ月目）: 安心感の提供
    - パイロット移行の結果を共有
    - 並行稼働中の切り戻し手順を説明
    - 物流スタッフへの研修を実施

  Phase 4（9-12ヶ月目）: 成功体験
    - 段階的に全拠点に展開
    - 移行後の改善効果をデータで示す
```

**初心者向けとの違い：**
- **エンゲージメント評価**: 現在と望ましい姿勢のギャップを可視化し、重点対応すべき人を明確にしています
- **個別エンゲージメント戦略**: 「抵抗」のステークホルダーに対して、段階的なアプローチを計画しています
- **イベント駆動コミュニケーション**: 定期的なものだけでなく、特定の出来事が起きたときの対応も事前に決めておきます
- **RACI**: 情報の責任者を明確にすることで「誰に聞けばいいか分からない」状態を防ぎます

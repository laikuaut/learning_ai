## 課題の要件

1. **社内ドキュメント検索システム**を題材に、RAGパイプラインを設計すること
   - 対象ドキュメント: 社内マニュアル（PDF）、FAQ（Markdown）、議事録（テキスト）
   - 想定ドキュメント数: 約1,000件、合計約500ページ分
2. 以下の**インデックス構築フロー**（オフライン処理）を設計すること
   - ドキュメント取り込み（ingestion）
   - チャンク分割（chunking）戦略
   - エンベディング（embedding）生成
   - ベクトルDB（vector database）への格納
3. 以下の**検索・回答生成フロー**（オンライン処理）を設計すること
   - ユーザークエリの処理
   - ベクトル検索（vector search）
   - リランキング（re-ranking）
   - LLMによる回答生成
4. 各コンポーネントの**技術選定と理由**を記述すること
5. **HyDE（Hypothetical Document Embeddings）** の適用可否を検討すること
6. パイプライン全体の**評価指標**を3つ以上定義すること

## ステップガイド

<details>
<summary>ステップ1：対象ドキュメントの特性を分析する</summary>

RAGの設計は対象ドキュメントの特性に大きく依存します。以下を整理しましょう。

| 特性 | 社内マニュアル | FAQ | 議事録 |
|------|-------------|-----|--------|
| 形式 | PDF | Markdown | テキスト |
| 構造 | 章・節あり | Q&A形式 | 非構造 |
| 更新頻度 | 月1回程度 | 週1回程度 | 毎日 |
| 平均長さ | 20〜50ページ | 1ページ/件 | 2〜5ページ |

これらの特性に合わせてチャンク分割戦略を変える必要があります。

</details>

<details>
<summary>ステップ2：チャンク分割戦略を決める</summary>

チャンク分割は RAG の精度に最も影響するコンポーネントの1つです。

考慮すべきポイント:
- **チャンクサイズ**: 大きすぎるとノイズが多い、小さすぎると文脈が失われる
- **オーバーラップ**: 隣接チャンク間で重複を持たせるか
- **分割単位**: 固定長 / 段落 / 意味的境界

**よくある間違い**: 全ドキュメントに同じチャンク分割戦略を適用してしまうこと。PDF のマニュアルと FAQ では最適な分割方法が異なります。

</details>

<details>
<summary>ステップ3：検索フローを設計する</summary>

ユーザーのクエリが入力されてから回答が返るまでの流れを設計します。

```
ユーザークエリ
    ↓
クエリ前処理（意図解析、キーワード抽出）
    ↓
ベクトル検索（上位20件を取得）
    ↓
リランキング（上位5件に絞り込み）
    ↓
プロンプト構築（検索結果 + クエリ + 指示）
    ↓
LLM回答生成
    ↓
回答 + 参照元の表示
```

各ステップで「何をするか」「なぜ必要か」「どんなパラメータがあるか」を記述しましょう。

</details>

<details>
<summary>ステップ4：HyDEの適用を検討する</summary>

HyDE（Hypothetical Document Embeddings）は、ユーザーのクエリをそのまま検索するのではなく、LLMに「仮の回答文書」を生成させ、その文書のエンベディングで検索する手法です。

```
通常:     ユーザークエリ → Embedding → 検索
HyDE:     ユーザークエリ → LLMで仮回答生成 → 仮回答のEmbedding → 検索
```

**メリット**: クエリと文書の言い回しのギャップを埋められる
**デメリット**: LLM呼び出しが1回増えるため、レイテンシが増加する

この社内ドキュメント検索システムに HyDE が有効かどうか、根拠とともに判断してください。

</details>

### 1. パイプライン全体図

```
【インデックス構築フロー（オフライン）】

  PDF/Markdown/テキスト
       ↓
  ドキュメントローダー（形式ごとにパーサーを切替）
       ↓
  前処理（メタデータ抽出、クリーニング）
       ↓
  チャンク分割
       ↓
  Embeddingモデル（text-embedding-3-small）
       ↓
  ベクトルDB（Chroma）に格納


【検索・回答生成フロー（オンライン）】

  ユーザークエリ
       ↓
  Embeddingモデル（同じモデル）
       ↓
  ベクトル検索（コサイン類似度、上位10件）
       ↓
  プロンプト構築
       ↓
  LLM回答生成（Claude / GPT-4）
       ↓
  回答 + 参照元ドキュメント
```

### 2. コンポーネント設計

#### ドキュメント取り込み

| ドキュメント形式 | パーサー | 備考 |
|----------------|---------|------|
| PDF | PyPDF2 / pdfplumber | 表やレイアウトが多い場合は pdfplumber を推奨 |
| Markdown | markdown-it | 見出し構造を保持 |
| テキスト | 標準テキスト処理 | 文字コード判定を含む |

#### チャンク分割戦略

| ドキュメント種別 | 分割戦略 | チャンクサイズ | オーバーラップ |
|----------------|---------|-------------|-------------|
| 社内マニュアル | 見出し（章・節）で分割 | 500〜1000トークン | 100トークン |
| FAQ | 1つのQ&Aペアを1チャンク | 可変（Q&A単位） | なし |
| 議事録 | 固定長で分割 | 500トークン | 100トークン |

#### メタデータ

各チャンクに以下のメタデータを付与します。

```json
{
  "source": "manual/it_security_guide.pdf",
  "doc_type": "manual",
  "department": "IT",
  "page_number": 15,
  "section": "3.2 パスワードポリシー",
  "last_updated": "2025-12-01",
  "chunk_index": 3
}
```

#### 技術選定

| コンポーネント | 選定技術 | 選定理由 |
|-------------|---------|---------|
| Embeddingモデル | text-embedding-3-small | コスパが良く、日本語対応。1,000件程度ならコスト問題なし |
| ベクトルDB | Chroma | 軽量で導入が容易。小規模（〜数万件）に適する |
| LLM | Claude 3.5 Sonnet | 日本語の精度が高く、長いコンテキストに対応 |

### 4. HyDEの検討結果

**結論: この用途では HyDE の導入は見送りが妥当**

理由:
- 社内ドキュメントの語彙は比較的統一されており、クエリと文書の言い回しのギャップが小さい
- レイテンシの増加（LLM呼び出し1回分）がユーザー体験を損なう
- 1,000件程度のドキュメント規模であれば、通常のベクトル検索で十分な精度が期待できる

### 5. 評価指標

| 指標 | 定義 | 目標値 |
|------|------|-------|
| 回答精度（Answer Accuracy） | 正解と一致する回答の割合 | 80%以上 |
| 検索適合率（Retrieval Precision） | 検索結果のうち関連文書の割合 | 70%以上 |
| 応答時間（Latency） | クエリから回答までの時間 | 5秒以内 |

```python
# 擬似コード: 階層的チャンキング

def hierarchical_chunk(document, doc_type):
    """ドキュメントを階層的にチャンク分割する"""

    if doc_type == "faq":
        # FAQ: Q&Aペアをそのまま1チャンクとする
        chunks = split_by_qa_pairs(document)
        return [Chunk(content=c, parent=None) for c in chunks]

    elif doc_type == "manual":
        # マニュアル: 見出しで親チャンク、段落で子チャンク
        sections = split_by_headings(document)
        all_chunks = []
        for section in sections:
            parent = Chunk(
                content=section.full_text,
                size="1000 tokens",
                metadata={"section": section.heading}
            )
            children = split_fixed_size(
                section.full_text,
                size=200,
                overlap=50
            )
            for child in children:
                child.parent_id = parent.id
                all_chunks.append(child)
            all_chunks.append(parent)
        return all_chunks

    elif doc_type == "minutes":
        # 議事録: 固定長で分割、発言者の切れ目を優先
        return split_by_speaker_then_fixed(
            document,
            size=500,
            overlap=100
        )
```

### 3. ハイブリッド検索の実装

```python
# 擬似コード: ハイブリッド検索 + RRF

def hybrid_search(query, top_k=5):
    """ベクトル検索とキーワード検索を組み合わせる"""

    # 1. ベクトル検索
    query_embedding = embed(query)
    vector_results = vector_db.search(
        query_embedding,
        top_k=20,
        filter={"department": user.department}  # アクセス制御
    )

    # 2. キーワード検索（BM25）
    keywords = extract_keywords(query)
    bm25_results = fulltext_index.search(
        keywords,
        top_k=20
    )

    # 3. Reciprocal Rank Fusion（RRF）でスコア統合
    merged = reciprocal_rank_fusion(
        vector_results,
        bm25_results,
        k=60  # RRFのハイパーパラメータ
    )

    # 4. リランキング（Cross-Encoder）
    reranked = cross_encoder_rerank(
        query=query,
        documents=merged[:20],
        top_k=top_k
    )

    # 5. 親チャンク展開
    expanded = []
    for chunk in reranked:
        if chunk.parent_id:
            parent = vector_db.get(chunk.parent_id)
            expanded.append(parent)
        else:
            expanded.append(chunk)

    return deduplicate(expanded)
```

### 4. HyDEの検討（改良版）

**結論: 条件付きで HyDE を導入する**

```python
# 擬似コード: 条件付き HyDE

def search_with_conditional_hyde(query):
    """クエリの種類に応じてHyDEを適用するか判断する"""

    query_type = classify_query(query)

    if query_type == "specific_fact":
        # 「〇〇の申請期限はいつ？」→ HyDE不要（キーワード検索で十分）
        return hybrid_search(query)

    elif query_type == "conceptual":
        # 「セキュリティポリシーに違反しないデータの扱い方は？」
        # → HyDE有効（抽象的な質問と具体的な文書のギャップを埋める）
        hypothetical_doc = llm.generate(
            f"以下の質問に対する社内マニュアルの回答を想像して書いてください: {query}"
        )
        hyde_embedding = embed(hypothetical_doc)
        return vector_db.search(hyde_embedding, top_k=20)

    elif query_type == "procedural":
        # 「VPNの設定手順を教えて」→ HyDE不要（手順書がそのまま検索される）
        return hybrid_search(query)
```

**判断根拠**:
- 具体的な事実を問う質問（全体の約60%と推定）では HyDE は不要でレイテンシが無駄になる
- 概念的・抽象的な質問（約20%）では、クエリと文書の語彙ギャップが大きいため HyDE が有効
- 条件分岐により、HyDE のメリットを享受しつつレイテンシの悪化を最小限に抑える

### 5. 評価フレームワーク（改良版）

#### コンポーネント別評価

| 段階 | 評価指標 | 測定方法 | 目標値 |
|------|---------|---------|-------|
| 検索 | Recall@10 | 正解文書が上位10件に含まれる割合 | 90%以上 |
| 検索 | MRR（Mean Reciprocal Rank） | 正解文書の順位の逆数の平均 | 0.7以上 |
| 検索 | Precision@5 | リランキング後の上位5件の適合率 | 80%以上 |
| 回答 | 回答正確性 | 人手評価（正解/部分正解/不正解） | 正解率80%以上 |
| 回答 | 幻覚率（Hallucination Rate） | 参照元に記載のない情報を含む回答の割合 | 5%以下 |
| 回答 | 参照元一致率 | 回答の根拠が参照元と一致する割合 | 90%以上 |
| 全体 | 応答時間 | クエリ入力から回答表示まで | 3秒以内 |

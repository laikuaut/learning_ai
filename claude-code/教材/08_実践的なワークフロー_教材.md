# 第8章：実践的なワークフロー

> **この章のゴール**: Claude Codeを使った実践的な開発ワークフローを習得し、日常の開発作業に活用できるようになること

---

## 8.1 バグ修正ワークフロー

バグ修正は開発者の日常的なタスクです。Claude Codeを活用した効率的なバグ修正の流れを紹介します。

### ステップバイステップ

```
┌──────────────────────────────────────────────────┐
│           バグ修正ワークフロー                     │
│                                                   │
│  Step 1: バグの報告・再現                         │
│    │                                              │
│    ▼                                              │
│  Step 2: 原因の調査                               │
│    │     Claude Codeにエラーログを共有             │
│    ▼                                              │
│  Step 3: 修正の実施                               │
│    │     Claude Codeに修正を依頼                  │
│    ▼                                              │
│  Step 4: テストの確認                             │
│    │     既存テスト実行＋回帰テスト追加            │
│    ▼                                              │
│  Step 5: コミット・PR作成                         │
│                                                   │
└──────────────────────────────────────────────────┘
```

### 実践例

```
# Step 1: バグの説明
> ユーザーログイン時に、メールアドレスに大文字が含まれると
> 「ユーザーが見つかりません」エラーになります。
> エラーログは以下です：
> Error: User not found for email: John@Example.com

# Step 2: Claude Codeが自動的に調査
# → 関連するファイルを検索し、原因を特定

# Step 3: 修正を確認
> その修正で進めてください

# Step 4: テスト
> テストを実行して、問題がないか確認してください

# Step 5: コミット
> /commit
```

### ポイントまとめ
- エラーメッセージやログを具体的に共有する
- Claude Codeに調査→修正→テストの流れを依頼
- 修正内容を確認してからコミット

## 8.2 機能開発ワークフロー

### 計画→実装→テスト→コミット

新機能を開発する際の推奨ワークフローです：

```
# Step 1: 計画（設計を相談）
> ユーザープロフィール画像のアップロード機能を追加したいです。
> どのような設計が良いか提案してください。
> 現在の技術スタック：Express + PostgreSQL + S3

# Step 2: 実装
> では、提案してくれた設計で実装を進めてください。
> まずはAPIエンドポイントから始めましょう。

# Step 3: テスト
> 実装したAPIのユニットテストを書いてください。
> 正常系と異常系（ファイルサイズ超過、不正な形式）を
> カバーしてください。

# Step 4: 動作確認
> テストを実行してください

# Step 5: コミット・PR
> /commit
```

### 大きな機能の分割

大きな機能は段階的に実装するのが効果的です：

```
# 全体の計画を立てる
> ユーザー管理機能を追加したいです。以下の機能が必要です：
> 1. ユーザー登録
> 2. ログイン・ログアウト
> 3. プロフィール編集
> 4. パスワードリセット
> どの順番で実装すべきか、計画を立ててください。

# 各機能を順番に実装
> では、まずユーザー登録機能から始めましょう。

# 各機能完了ごとにコミット
> /commit
```

### よくある間違い
> **一度にすべてを実装しようとする**: 大きな機能は小さなステップに分割しましょう。各ステップでテストとコミットを行うことで、問題が発生しても戻しやすくなります。

### ポイントまとめ
- 計画→実装→テスト→コミットのサイクルで進める
- 大きな機能は小さなステップに分割
- 各ステップでテストとコミットを実施

## 8.3 コードレビュー・リファクタリング

### コードレビューの依頼

```
> src/services/ ディレクトリのコードをレビューして、
> 以下の観点で問題点を指摘してください：
> - セキュリティ上の懸念
> - パフォーマンスの問題
> - エラーハンドリングの不備
> - コードの重複
```

### リファクタリングの実施

```
# 具体的なリファクタリング指示
> UserService クラスが500行を超えています。
> 認証関連のメソッドを AuthService クラスに分離してください。

# 段階的なリファクタリング
> まず、分離対象のメソッドを特定して一覧を見せてください
（確認後）
> では、AuthService クラスを作成して、メソッドを移動してください
（確認後）
> 移動後のインポートパスをすべて更新してください
（確認後）
> テストを実行して、既存の機能が壊れていないか確認してください
```

### ポイントまとめ
- レビュー観点を明示すると精度の高いレビューが得られる
- リファクタリングは段階的に進め、各段階でテストを実行
- 大規模な変更は影響範囲を確認してから実施

## 8.4 テスト駆動開発（TDD）

### Claude CodeでTDDを実践

テスト駆動開発（TDD：Test-Driven Development）の流れをClaude Codeで実践できます：

```
┌──────────────────────────────────────────────┐
│              TDD サイクル                      │
│                                               │
│     ① テストを書く（Red）                    │
│        │                                      │
│        ▼                                      │
│     ② 実装する（Green）                      │
│        │                                      │
│        ▼                                      │
│     ③ リファクタリング（Refactor）            │
│        │                                      │
│        └──→ ① に戻る                        │
│                                               │
└──────────────────────────────────────────────┘
```

### 実践例

```
# Step 1: テストを先に書く
> calculateDiscount 関数のテストを先に書いてください。
> 仕様：
> - 合計金額が1000円以上で5%割引
> - 合計金額が5000円以上で10%割引
> - 合計金額が10000円以上で15%割引
> - 割引後の金額は小数点以下切り捨て

# Step 2: テストが失敗することを確認
> テストを実行してください（まだ実装がないので失敗するはず）

# Step 3: 最小限の実装
> テストがすべて通る最小限の実装を書いてください

# Step 4: テスト実行
> テストを実行してください

# Step 5: リファクタリング
> 実装をリファクタリングして、可読性を改善してください
> テストが引き続き通ることを確認してください
```

### ポイントまとめ
- TDDの Red → Green → Refactor サイクルをClaude Codeで実践できる
- テストの仕様を明確に伝えることが重要
- リファクタリング後は必ずテストを再実行

## 8.5 ドキュメント生成

### コードからドキュメントを生成

```
# APIドキュメントの生成
> src/controllers/ のすべてのAPIエンドポイントから
> API仕様書をMarkdown形式で生成してください。
> 各エンドポイントに対して、メソッド、パス、
> リクエストパラメータ、レスポンス形式を記載してください。

# JSDoc/TSDocの追加
> src/services/UserService.ts のすべてのpublicメソッドに
> TSDocコメントを追加してください

# READMEの生成
> このプロジェクトのREADME.mdを作成してください。
> セットアップ手順、使用方法、API一覧を含めてください。
```

### ポイントまとめ
- コードを分析してドキュメントを自動生成できる
- API仕様書、JSDoc、READMEなど様々な形式に対応
- 生成されたドキュメントは必ず人間がレビューする

## 8.6 大規模コードベースの操作

### 効果的なナビゲーション

大規模なコードベースでは、Claude Codeの検索能力を活用します：

```
# プロジェクト全体の把握
> このプロジェクトの全体構成を説明してください。
> 主要なモジュールとその依存関係を図で示してください。

# 特定の機能の追跡
> ユーザー登録のリクエストが処理される流れを、
> エントリポイントから順に追跡してください。

# 影響範囲の分析
> User モデルの email フィールドを変更した場合、
> 影響を受けるファイルを一覧で示してください。
```

### Agentツールの活用

複雑な調査タスクでは、Claude Codeは内部的に**サブエージェント**（sub-agent）を起動して並列処理を行います：

```
> このプロジェクトのセキュリティ上の問題点を
> 包括的に調査してください

# Claude Codeは複数のサブエージェントを起動:
# - Agent 1: SQLインジェクションの調査
# - Agent 2: XSSの調査
# - Agent 3: 認証・認可の調査
# → 結果を統合して報告
```

### ポイントまとめ
- 大規模コードベースでも検索機能で効率的にナビゲーション
- 処理の流れの追跡や影響範囲の分析が可能
- 複雑な調査はサブエージェントで並列処理される

## 8.7 IDE連携と利用環境

### VS Code拡張

Claude CodeはVS Code拡張（extension）として利用でき、エディタ内のパネルで直接操作できます：

```
┌──────────────────────────────────────────────┐
│  VS Code                                      │
│  ┌────────────────────┬─────────────────────┐│
│  │                    │                     ││
│  │  エディタ           │  Claude Code        ││
│  │  （コード編集）     │  パネル             ││
│  │                    │                     ││
│  │                    │  > バグを修正して    ││
│  │                    │                     ││
│  │                    │  修正を適用中...     ││
│  │                    │                     ││
│  └────────────────────┴─────────────────────┘│
└──────────────────────────────────────────────┘
```

### JetBrains拡張

IntelliJ IDEA、WebStorm、PyCharmなどのJetBrains IDEでも利用可能です。

### Web版（claude.ai/code）

ブラウザからClaude Codeを利用できます。ローカル環境のセットアップが不要なため、すぐに使い始められます。

### デスクトップアプリ

Mac/Windows向けのデスクトップアプリケーション（desktop application）も提供されています。

### 利用環境の選び方

```
┌──────────────────┬──────────────────────────────┐
│ 利用環境          │ おすすめの場面               │
├──────────────────┼──────────────────────────────┤
│ ターミナルCLI    │ 最も柔軟。自動化やCI連携向き │
│ VS Code拡張      │ VS Codeユーザー。コードと並行│
│ JetBrains拡張    │ JetBrainsユーザー            │
│ Web版            │ 環境構築不要。お試し向き     │
│ デスクトップアプリ│ 独立したウィンドウで使いたい │
└──────────────────┴──────────────────────────────┘
```

### ポイントまとめ
- Claude CodeはCLI、IDE拡張、Web版、デスクトップアプリで利用可能
- 用途に応じて最適な環境を選択できる
- ターミナルCLIが最も柔軟で機能が豊富

## 8.8 自動化とCI連携

### ワンショットモードでの自動化

`-p`フラグを使ったワンショットモードは、スクリプトやCI/CDパイプラインに組み込めます：

```bash
# コードレビューの自動化
claude -p "src/ のコードをレビューして問題点を報告して" > review.md

# テスト生成の自動化
claude -p "src/services/新しいファイル.ts のテストを生成して" > test.ts

# コミットメッセージの自動生成
MESSAGE=$(claude -p "git diff --staged の内容からコミットメッセージを生成して" --output-format text)
git commit -m "$MESSAGE"
```

### パイプラインへの統合

```bash
# CI/CDでの活用例
# 1. PRの変更内容を分析
claude -p "$(git diff main...HEAD)" --output-format json

# 2. エラーログの分析
cat build.log | claude -p "このビルドエラーの原因と修正方法を教えて"
```

### よくある間違い
> **自動化で無制限に実行する**: Claude CodeはAPI呼び出しごとに課金されます。CI/CDに組み込む場合は、`--max-turns`で最大ターン数を制限し、コストが際限なく増えないようにしましょう。

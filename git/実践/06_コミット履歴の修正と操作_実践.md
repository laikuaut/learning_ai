> **難易度**: ★★★☆☆（中級）
> **前提知識**: 第2章（基本操作）、第3章（ブランチとマージ）、第8章（高度なGit操作）
> **課題の種類**: ミニプロジェクト
> **学習目標**: amend・reset・revert・stashなど履歴操作コマンドを状況に応じて使い分けられるようにする

## 完成イメージ

```
$ git log --oneline
d4e5f6a (HEAD -> main) Revert "不要な機能を追加"
c3d4e5f 不要な機能を追加
b2c3d4e 正しいコミットメッセージに修正済み
a1b2c3d 初回コミット

$ git stash list
stash@{0}: WIP on main: 作業中の変更を一時退避
```

---

## 課題の要件

以下の5つのシナリオをそれぞれ実践してください。

1. **amend**: 直前のコミットメッセージを修正する
2. **amend**: 直前のコミットにファイルを追加する
3. **reset**: ステージングを取り消す（ファイルは残す）
4. **revert**: 過去のコミットを打ち消す新しいコミットを作成する
5. **stash**: 作業中の変更を一時退避して、別の作業をする

---

## シナリオ1：コミットメッセージの修正（amend）

<details>
<summary>課題と手順</summary>

タイポのあるコミットメッセージを修正します。

```bash
mkdir history-practice
cd history-practice
git init

# わざとタイポしたメッセージでコミット
echo "最初のファイル" > file1.txt
git add file1.txt
git commit -m "初回コミト"  # 「コミット」が「コミト」になっている

# 履歴を確認（タイポがある）
git log --oneline

# amend でメッセージを修正
git commit --amend -m "初回コミット"

# 修正されたことを確認
git log --oneline
```

**注意**: `--amend` はコミットハッシュが変わります。既にプッシュ済みのコミットには使わないでください。

</details>

## シナリオ2：直前のコミットにファイルを追加する（amend）

<details>
<summary>課題と手順</summary>

コミットし忘れたファイルを、直前のコミットに追加します。

```bash
# 2つのファイルをコミットするつもりが、1つ忘れた
echo "設定ファイル" > config.txt
echo "ヘルパー" > helper.txt

git add config.txt
git commit -m "設定ファイルとヘルパーを追加"

# helper.txt を忘れたことに気づいた！
git status
# Untracked files: helper.txt

# 忘れたファイルをステージングして amend
git add helper.txt
git commit --amend --no-edit  # メッセージは変えない

# 確認：1つのコミットに両方のファイルが含まれている
git show --stat HEAD
```

</details>

## シナリオ3：ステージングの取り消し（reset）

<details>
<summary>課題と手順</summary>

間違えてステージングしたファイルを取り消します。

```bash
# 3つのファイルを作成
echo "公開してよいファイル" > public.txt
echo "これも公開してよい" > data.txt
echo "API_KEY=secret12345" > .env  # これは間違えてステージしたくない

# うっかり全部ステージング
git add .

# 状態確認
git status
# Changes to be committed:
#   new file:   .env
#   new file:   data.txt
#   new file:   public.txt

# .env だけステージングから外す
git reset HEAD .env
# または: git restore --staged .env

# 確認：.env が Untracked に戻っている
git status

# .env を .gitignore に追加して、今後追跡しないようにする
echo ".env" > .gitignore
git add .gitignore public.txt data.txt
git commit -m "公開ファイルを追加（.envは除外）"
```

</details>

## シナリオ4：コミットの打ち消し（revert）

<details>
<summary>課題と手順</summary>

過去のコミットで入れた変更を取り消します。`revert` は新しいコミットを作って打ち消すため、履歴が残り安全です。

```bash
# 3つのコミットを作成
echo "機能A" > feature_a.txt
git add feature_a.txt
git commit -m "機能Aを追加"

echo "不要な機能" > unnecessary.txt
git add unnecessary.txt
git commit -m "不要な機能を追加"

echo "機能B" > feature_b.txt
git add feature_b.txt
git commit -m "機能Bを追加"

# 履歴を確認
git log --oneline
# c3d4e5f 機能Bを追加
# b2c3d4e 不要な機能を追加    ← これを取り消したい
# a1b2c3d 機能Aを追加

# 2つ前のコミットを revert（新しい打ち消しコミットが作られる）
git revert HEAD~1

# 確認：unnecessary.txt が削除されている
ls
git log --oneline
# d4e5f6a Revert "不要な機能を追加"
# c3d4e5f 機能Bを追加
# b2c3d4e 不要な機能を追加
# a1b2c3d 機能Aを追加
```

</details>

## シナリオ5：作業の一時退避（stash）

<details>
<summary>課題と手順</summary>

開発中に急なバグ修正が入った場合のシナリオです。

```bash
# 機能開発中（まだ完成していない）
echo "新機能の開発中..." > new_feature.txt
git add new_feature.txt

# この時点で「急ぎのバグ修正が必要！」と言われた

# 作業中の変更を一時退避
git stash save "新機能の開発途中"

# 作業ディレクトリがきれいになったことを確認
git status
# nothing to commit, working tree clean
ls  # new_feature.txt が消えている

# 急ぎのバグ修正
echo "バグ修正完了" > bugfix.txt
git add bugfix.txt
git commit -m "緊急バグ修正"

# 退避していた作業を復帰
git stash pop

# new_feature.txt が戻ってきたことを確認
ls
git status
# new_feature.txt がステージングされた状態で戻る

# 開発を続ける
echo "新機能の開発完了！" > new_feature.txt
git add new_feature.txt
git commit -m "新機能を追加"
```

</details>

## 解答例

<details>
<summary>解答例（全シナリオの一括実行スクリプト）</summary>

```bash
#!/bin/bash
# コミット履歴操作の練習スクリプト

# ===== 準備 =====
rm -rf /tmp/history-practice
mkdir /tmp/history-practice
cd /tmp/history-practice
git init

# ===== シナリオ1: amend（メッセージ修正） =====
echo "=== シナリオ1: amend（メッセージ修正）==="
echo "最初のファイル" > file1.txt
git add file1.txt
git commit -m "初回コミト"  # タイポ
git log --oneline           # タイポを確認
git commit --amend -m "初回コミット"
git log --oneline           # 修正を確認

# ===== シナリオ2: amend（ファイル追加） =====
echo ""
echo "=== シナリオ2: amend（ファイル追加）==="
echo "設定ファイル" > config.txt
echo "ヘルパー" > helper.txt
git add config.txt
git commit -m "設定ファイルとヘルパーを追加"
git add helper.txt
git commit --amend --no-edit
git show --stat HEAD        # 2ファイル含まれることを確認

# ===== シナリオ3: reset（ステージング取り消し） =====
echo ""
echo "=== シナリオ3: reset（ステージング取り消し）==="
echo "公開ファイル" > public.txt
echo "API_KEY=secret" > .env
git add .
git status                  # 両方ステージされている
git reset HEAD .env
git status                  # .env が Untracked に戻る
echo ".env" > .gitignore
git add .gitignore public.txt
git commit -m "公開ファイルを追加（.envは除外）"

# ===== シナリオ4: revert（コミット打ち消し） =====
echo ""
echo "=== シナリオ4: revert（コミット打ち消し）==="
echo "機能A" > feature_a.txt
git add feature_a.txt
git commit -m "機能Aを追加"

echo "不要な機能" > unnecessary.txt
git add unnecessary.txt
git commit -m "不要な機能を追加"

echo "機能B" > feature_b.txt
git add feature_b.txt
git commit -m "機能Bを追加"

git revert HEAD~1 --no-edit
git log --oneline
ls                          # unnecessary.txt が消えている

# ===== シナリオ5: stash（一時退避） =====
echo ""
echo "=== シナリオ5: stash（一時退避）==="
echo "開発中..." > new_feature.txt
git add new_feature.txt
git stash save "新機能の開発途中"
git status                  # クリーンな状態

echo "バグ修正" > bugfix.txt
git add bugfix.txt
git commit -m "緊急バグ修正"

git stash list              # 退避一覧を確認
git stash pop               # 退避を復帰

echo "新機能完成！" > new_feature.txt
git add new_feature.txt
git commit -m "新機能を追加"

# ===== 最終確認 =====
echo ""
echo "=== 最終履歴 ==="
git log --oneline
```

</details>

<details>
<summary>解答例（改良版 ─ reset の3つのモード比較）</summary>

`git reset` には `--soft`、`--mixed`（デフォルト）、`--hard` の3つのモードがあります。

```bash
mkdir /tmp/reset-comparison
cd /tmp/reset-comparison
git init

# 3つのコミットを作成
echo "v1" > file.txt && git add file.txt && git commit -m "v1"
echo "v2" > file.txt && git add file.txt && git commit -m "v2"
echo "v3" > file.txt && git add file.txt && git commit -m "v3"

git log --oneline
# c3c3c3c v3
# b2b2b2b v2
# a1a1a1a v1

# ===== --soft: コミットだけ戻す（ステージングは維持）=====
git reset --soft HEAD~1
git status
# Changes to be committed: file.txt (v3の内容がステージされている)
git log --oneline
# b2b2b2b v2
# a1a1a1a v1

# 元に戻す
git commit -m "v3"

# ===== --mixed: コミットとステージングを戻す（ファイルは維持）=====
git reset --mixed HEAD~1
# または単に: git reset HEAD~1
git status
# Changes not staged: file.txt (v3の内容がワーキングツリーに残っている)
git log --oneline
# b2b2b2b v2
# a1a1a1a v1

# 元に戻す
git add file.txt && git commit -m "v3"

# ===== --hard: すべて戻す（ファイルの内容も戻る）=====
# 【注意】--hard は変更が完全に失われます！
git reset --hard HEAD~1
git status
# nothing to commit, working tree clean
cat file.txt
# v2 （ファイルも戻っている）
git log --oneline
# b2b2b2b v2
# a1a1a1a v1

# ===== 比較表 =====
# | モード   | コミット | ステージング | ファイル |
# |----------|----------|------------|----------|
# | --soft   | 戻る     | 維持       | 維持     |
# | --mixed  | 戻る     | 戻る       | 維持     |
# | --hard   | 戻る     | 戻る       | 戻る     |
```

**初心者向けとの違い:**
- `reset` の3モードを実際に比較して、違いを体感する
- `--hard` の危険性を理解した上で、適切な場面でのみ使う判断力を養う
- 実務では `--soft` や `--mixed` を使うことが多い（変更を失わずに履歴を整理）

</details>

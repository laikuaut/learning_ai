## 読解対象ファイル

### ファイル1: `CLAUDE.md`

```markdown
# TaskFlow - プロジェクト管理SaaS

## 概要
チーム向けプロジェクト管理ツール。カンバンボード、ガントチャート、
タイムトラッキング機能を提供するSaaS。

## 技術スタック
- Frontend: React 18 + TypeScript + Vite
- Backend: Node.js + Express + TypeScript
- Database: PostgreSQL 15 + Prisma
- Cache: Redis
- Queue: BullMQ
- Test: Vitest (frontend), Jest (backend)
- CI/CD: GitHub Actions

## モノレポ構成
```
packages/
├── frontend/     # React SPA
├── backend/      # Express API
├── shared/       # 共有型定義・ユーティリティ
└── e2e/          # Playwright E2Eテスト
```

## 開発コマンド
- `pnpm dev` : 全パッケージの開発サーバー起動
- `pnpm test` : 全テスト実行
- `pnpm test:frontend` : フロントエンドテストのみ
- `pnpm test:backend` : バックエンドテストのみ
- `pnpm test:e2e` : E2Eテスト（Playwright）
- `pnpm lint` : 全パッケージのリント
- `pnpm build` : プロダクションビルド
- `pnpm db:migrate` : DBマイグレーション
- `pnpm db:seed` : テストデータ投入

## コーディング規約
- 命名: camelCase (変数/関数), PascalCase (コンポーネント/クラス/型)
- frontend: 関数コンポーネント + Hooks、classコンポーネント禁止
- backend: Controller → Service → Repository の3層
- shared: 破壊的変更はメジャーバージョンアップで行う
- コミット: Conventional Commits 必須（feat:, fix:, chore:, docs:）
- PR: 必ず1つ以上のレビュー承認を得てからマージ

## 重要なルール
- `packages/shared/` の変更は影響範囲が広いため、必ずfrontendとbackend両方のテストを実行すること
- `.env.production` は絶対に変更しない（インフラチームが管理）
- DBマイグレーションは必ず `pnpm db:migrate` で行い、手動SQLは禁止
- パフォーマンスに影響する変更（N+1クエリ等）はPRで明記すること
```

### ファイル3: `ci-claude.sh`（CI/CDでClaude Codeを使うスクリプト）

```bash
#!/bin/bash
set -euo pipefail

# GitHub ActionsからClaude Codeを使ってPRレビューを自動化するスクリプト

PR_NUMBER=${1:?'PR番号を指定してください'}
REPO="taskflow-team/taskflow"

echo "=== PR #${PR_NUMBER} の自動レビュー ==="

# PRの差分を取得
gh pr diff "$PR_NUMBER" --repo "$REPO" > /tmp/pr_diff.txt

# PRの情報を取得
PR_TITLE=$(gh pr view "$PR_NUMBER" --repo "$REPO" --json title -q '.title')
PR_BODY=$(gh pr view "$PR_NUMBER" --repo "$REPO" --json body -q '.body')

# Claude Codeでレビュー実行
REVIEW=$(cat /tmp/pr_diff.txt | claude -p "
以下のPRをレビューしてください。
タイトル: ${PR_TITLE}
説明: ${PR_BODY}

レビュー観点：
1. コーディング規約への準拠
2. セキュリティ上の問題
3. パフォーマンスへの影響
4. テストの十分性
5. sharedパッケージへの変更がある場合の影響範囲

差分:
$(cat /tmp/pr_diff.txt)
" --output-format text --max-turns 3)

# レビュー結果をPRにコメント
gh pr comment "$PR_NUMBER" --repo "$REPO" --body "## 🤖 自動レビュー結果

${REVIEW}

---
*このレビューはClaude Codeによる自動生成です。人間のレビューも必ず受けてください。*"

echo "=== レビュー完了 ==="
```

### 問4（理解）

## 解答例

<details>
<summary>問1の解答</summary>

パッケージマネージャーは **pnpm** です。開発コマンドに `pnpm dev`、`pnpm test` 等が記載されています。

個別パッケージのテスト実行コマンド：
- `pnpm test:frontend` ─ フロントエンドテスト（Vitest）
- `pnpm test:backend` ─ バックエンドテスト（Jest）
- `pnpm test:e2e` ─ E2Eテスト（Playwright）

</details>

<details>
<summary>問2の解答</summary>

1. **github**: GitHub MCPサーバー。GitHub APIと連携し、PR操作、Issue管理、リポジトリ情報の取得などをClaude Codeから直接行えるようにします。

2. **postgres**: PostgreSQL MCPサーバー。開発用データベース（taskflow_dev）に接続し、スキーマ情報の取得やクエリ実行をClaude Codeから行えるようにします。

</details>

<details>
<summary>問3の解答</summary>

**1つ目（Bashマッチャー）:**
- `rm -rf`: ファイルの再帰的な強制削除を防止
- `DROP TABLE`: データベーステーブルの削除を防止
- `TRUNCATE`: テーブル内データの全削除を防止
- `ALTER TABLE...DROP`: カラムやインデックスの削除を防止

これらはすべて**破壊的で取り消しが困難な操作**です。

**2つ目（Write|Editマッチャー）:**
- `.env.production`ファイルの変更を防止
- CLAUDE.mdの「.env.productionは絶対に変更しない」というルールを**フックで強制**しています
- CLAUDE.mdの指示だけではLLMが従わない可能性がありますが、フックなら確実にブロックできます

</details>

<details>
<summary>問4の解答</summary>

`--filter` オプションは、pnpmのモノレポでリントを実行するパッケージを限定するために使われています。

`grep -oE 'packages/[^/]+'` は、ファイルパスから**パッケージ名の部分**を抽出します。

例:
- `packages/frontend/src/App.tsx` → `packages/frontend`
- `packages/backend/src/index.ts` → `packages/backend`

これにより、変更されたファイルが属するパッケージのリントだけが実行され、全パッケージのリントを毎回走らせるよりも**高速**に動作します。

</details>

<details>
<summary>問5の解答</summary>

PostToolUseに設定されている理由：
- **ファイルの変更自体はブロックしない**（shared変更が必要な場合もある）
- 変更が完了した**後に警告を出す**ことで、開発者にテスト実行を促す
- 「ブロックではなく通知」という設計方針

PreToolUseに設定した場合の違い：
- `exit 2` で終了するとsharedパッケージへの変更が**一切できなくなる**
- `exit 0` で終了すると変更は許可されるが、警告が変更**前**に表示されるため、まだ変更内容が確定していない段階での警告になる
- PostToolUseの方が「変更を許可しつつ、注意喚起する」という目的に合っている

</details>

<details>
<summary>問6の解答</summary>

`git diff --stat HEAD` は、現在のワーキングディレクトリとHEADコミットの差分をファイル名と変更行数で表示します。

Stopフックに設定することで、Claude Codeのセッション終了時に**そのセッションで行われた未コミットの変更の概要**が表示されます。

これにより：
- コミットし忘れている変更がないか確認できる
- どのファイルがどの程度変更されたかの概要が一目でわかる
- セッションの作業結果のサマリーとして機能する

</details>

<details>
<summary>問7の解答</summary>

`--max-turns 3` がない場合、Claude Codeがレビューのために多数のツール呼び出し（ファイル読み込み、検索など）を行い、**APIの使用量（コスト）が想定以上に膨らむ**可能性があります。

CI/CDパイプラインでは：
- 1回のPRレビューに対するコストを予測可能にする必要がある
- 無制限のターン数だと、大きなPRでClaude Codeが長時間実行し続ける可能性がある
- CIのタイムアウトに引っかかる可能性もある
- `--max-turns 3` で上限を設けることで、コストと実行時間を制御している

</details>

<details>
<summary>問8の解答</summary>

Conventional Commitsを強制するStopフックの設定例：

```json
{
  "Stop": [
    {
      "matcher": "",
      "command": "cd \"$CLAUDE_PROJECT_DIR\" && LAST_MSG=$(git log -1 --pretty=%s 2>/dev/null) && if echo \"$LAST_MSG\" | grep -qvE '^(feat|fix|chore|docs|style|refactor|perf|test|ci|build|revert)(\\(.+\\))?!?:'; then echo \"WARNING: 直近のコミットメッセージがConventional Commits形式ではありません: $LAST_MSG\"; fi"
    }
  ]
}
```

別のアプローチとして、Git自体のcommit-msgフック（`.git/hooks/commit-msg`）で検証する方法もあります。Claude Code側のフックとGit側のフックは別のものです。

</details>

<details>
<summary>問9の解答</summary>

**セキュリティ上の問題:**
- `DATABASE_URL`（データベース接続文字列）にユーザー名とパスワードが含まれている
- `.claude/settings.json`はGitにコミットされる可能性がある（チーム共有のプロジェクトスコープ設定）
- リポジトリを閲覧できるすべてのメンバーにDB認証情報が公開される

**より安全な設定方法:**

1. `.claude/settings.local.json` に移動する（Gitにコミットしない）
```json
{
  "mcpServers": {
    "postgres": {
      "env": {
        "DATABASE_URL": "postgresql://dev:dev@localhost:5432/taskflow_dev"
      }
    }
  }
}
```

2. 環境変数を参照する形にする
```json
{
  "mcpServers": {
    "postgres": {
      "command": "npx",
      "args": ["-y", "@modelcontextprotocol/server-postgres"],
      "env": {
        "DATABASE_URL": "${DATABASE_URL}"
      }
    }
  }
}
```

</details>

<details>
<summary>問10の解答</summary>

**よくできている点:**

1. **CLAUDE.mdとフックの連携**: 「.env.productionは変更しない」というCLAUDE.mdのルールがPreToolUseフックで技術的に強制されている。CLAUDE.mdの指示だけに頼らず、確実にルールを守らせる設計。

2. **モノレポ対応のリント**: PostToolUseフックで変更されたパッケージのみリントを実行する仕組み。全パッケージを毎回リントするよりも効率的で、開発速度を損なわない。

3. **sharedパッケージの変更警告**: 影響範囲が広いsharedパッケージの変更に対して適切なレベル（ブロックではなく警告）で注意喚起している。

**改善すべき点:**

1. **DATABASE_URLの直接記載**: settings.jsonにDB認証情報が直接書かれている。settings.local.jsonに移動すべき。

2. **Conventional Commitsの強制がない**: CLAUDE.mdではConventional Commits必須と記載されているが、それを検証するフックがない。ルールの実効性が低い。

3. **CI連携スクリプトのエラーハンドリング**: `ci-claude.sh`で`claude -p`の実行が失敗した場合のフォールバック処理がない。APIエラーやタイムアウト時にCIが失敗する可能性がある。

</details>

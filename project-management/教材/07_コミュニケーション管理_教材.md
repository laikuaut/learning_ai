## 7.1 コミュニケーション管理の基本

### なぜコミュニケーション管理が重要か

プロジェクトの失敗原因の多くは、技術的な問題ではなく**コミュニケーションの問題**です。PMI（Project Management Institute）の調査によると、プロジェクトマネージャーは業務時間の約90%をコミュニケーションに費やしているとされています。

```
プロジェクト失敗の主な原因（コミュニケーション関連）：

1. 要件の認識齟齬          ← ステークホルダーとの会話不足
2. 進捗状況の共有不足       ← 報告体制の未整備
3. 課題のエスカレーション遅れ ← 報告ルートの不明確
4. チーム内の情報格差       ← 情報共有の仕組み不足
5. 変更の伝達漏れ          ← 変更管理プロセスの欠如
```

### コミュニケーション管理の3つのプロセス

```
1. コミュニケーション計画策定
   → 誰に、何を、いつ、どの方法で伝えるかを決める
   ↓
2. コミュニケーションの実行
   → 計画に基づいて情報を配信・収集する
   ↓
3. コミュニケーションの監視
   → 情報伝達が適切に行われているかを確認する
```

### コミュニケーションの種類

| 種類 | 説明 | 例 |
|------|------|-----|
| **公式・書面** | 正式な文書による伝達 | プロジェクト計画書、議事録、契約書 |
| **公式・口頭** | 正式な場での口頭伝達 | ステアリングコミッティ、プレゼンテーション |
| **非公式・書面** | カジュアルな文書伝達 | Slackメッセージ、メモ |
| **非公式・口頭** | カジュアルな口頭伝達 | 雑談、立ち話、1on1 |

### コミュニケーションチャネル数の計算

チーム人数が増えると、コミュニケーションの経路（チャネル）は爆発的に増加します。

```
コミュニケーションチャネル数 = n × (n - 1) / 2

n = メンバー数

例：
  5人チーム  → 5 × 4 / 2 = 10 チャネル
  10人チーム → 10 × 9 / 2 = 45 チャネル
  20人チーム → 20 × 19 / 2 = 190 チャネル
  50人チーム → 50 × 49 / 2 = 1,225 チャネル
```

> **現場でのポイント**: チームが大きくなるほどコミュニケーションコストは二次関数的に増大します。これがアジャイルで「少人数チーム（5〜9名）」が推奨される理由の一つです。

## 7.2 ステークホルダー分析

### ステークホルダーの特定

プロジェクトに関わるすべての人や組織を洗い出し、それぞれの特性を把握します。

```
ステークホルダー一覧の例：

| ステークホルダー | 役割           | 関心事               | 影響力 | 関心度 |
|----------------|---------------|---------------------|--------|--------|
| CTO            | スポンサー      | 技術戦略との整合性     | 高     | 中     |
| PO（田中さん）  | プロダクトオーナー | 機能の優先順位        | 高     | 高     |
| 開発チーム      | 実装担当       | 技術的実現性、納期     | 中     | 高     |
| デザイナー      | UI/UX設計      | ユーザー体験          | 中     | 中     |
| カスタマーサポート | 利用者の声     | 操作性、問い合わせ削減  | 低     | 高     |
| 法務部          | コンプライアンス | 個人情報保護、利用規約  | 高     | 低     |
```

### 権力/関心度マトリクス

ステークホルダーを影響力と関心度の2軸で分類し、適切な対応方針を決めます。

```
  高 |  B: 満足させる       |  A: 重点管理
影  |  (Keep Satisfied)    |  (Manage Closely)
響  |                      |
力  |  対応：定期報告       |  対応：密な連携
    |  例：法務部、経営層    |  例：PO、CTO
    |----------------------|----------------------
  低 |  D: 最小限の対応      |  C: 情報提供
    |  (Monitor)           |  (Keep Informed)
    |                      |
    |  対応：必要時のみ      |  対応：進捗共有
    |  例：他部門の担当者    |  例：カスタマーサポート
    +----------------------+----------------------
           低                      高
                  関心度
```

### ステークホルダーエンゲージメント評価

各ステークホルダーの現在の態度と、望ましい態度を把握します。

| レベル | 説明 | 対応方針 |
|--------|------|---------|
| **不認識** | プロジェクトの存在を知らない | 情報を提供する |
| **抵抗的** | プロジェクトに反対している | 原因を理解し、対話する |
| **中立的** | 賛成でも反対でもない | メリットを伝え、巻き込む |
| **支持的** | プロジェクトを支持している | 関係を維持する |
| **主導的** | 積極的に推進している | リーダーシップを活かす |

> **現場でのポイント**: 抵抗的なステークホルダーを無視せず、早期に対話の場を設けることが重要です。反対意見の中にプロジェクトの改善ヒントが含まれていることも少なくありません。

## 7.3 コミュニケーション計画

### コミュニケーション計画書の要素

| 要素 | 説明 | 例 |
|------|------|-----|
| **対象者** | 誰に伝えるか | PO、開発チーム、経営層 |
| **内容** | 何を伝えるか | 進捗状況、課題、リスク |
| **頻度** | いつ伝えるか | 毎日、毎週、マイルストーン時 |
| **方法** | どう伝えるか | 会議、メール、Slack、ダッシュボード |
| **責任者** | 誰が伝えるか | PM、テックリード |
| **エスカレーション** | 問題発生時の報告ルート | チームリーダー → PM → スポンサー |

### コミュニケーション計画の例

```
ソフトウェア開発プロジェクトのコミュニケーション計画：

| 会議/報告           | 対象者       | 頻度     | 方法        | 責任者 |
|--------------------|-------------|---------|------------|--------|
| デイリースタンドアップ | 開発チーム    | 毎日     | 対面/Zoom   | SM     |
| スプリントレビュー    | PO+開発チーム | 隔週     | 対面/Zoom   | SM     |
| 進捗報告書           | 経営層       | 月次     | メール+資料  | PM     |
| ステアリングコミッティ | スポンサー    | 月次     | 対面       | PM     |
| 技術共有会           | 開発チーム    | 隔週     | 社内Wiki   | TL     |
| 障害報告             | 全関係者     | 発生時   | Slack+メール | PM     |
| リリースノート        | 全社        | リリース時 | メール      | PM     |
```

### エスカレーションルート

問題の深刻度に応じて、報告先を段階的に上げていく仕組みです。

```
レベル1：チーム内で解決
  └→ 担当者 → チームリーダー
      対応時間目標：当日中

レベル2：PM判断が必要
  └→ チームリーダー → PM
      対応時間目標：翌営業日まで

レベル3：スポンサー判断が必要
  └→ PM → スポンサー/ステアリングコミッティ
      対応時間目標：3営業日以内

レベル4：経営判断が必要
  └→ スポンサー → CTO/経営層
      対応時間目標：1週間以内
```

## 7.4 効果的な報告

### 報告の種類

| 報告の種類 | 目的 | 対象 | 頻度 |
|-----------|------|------|------|
| **ステータスレポート** | 現在の進捗を報告 | ステークホルダー | 定期（週次/月次） |
| **進捗レポート** | 計画と実績の比較 | PM、スポンサー | 定期（週次/月次） |
| **予測レポート** | 今後の見通しを報告 | 経営層 | 月次/マイルストーン時 |
| **例外レポート** | 異常事態を報告 | 即時判断が必要な人 | 発生時 |

### 週次進捗報告のテンプレート

```markdown
# 週次進捗報告（Week 12: 2026/03/20 - 2026/03/26）

## サマリー
- 全体進捗：計画比 +2日 ✅
- 今週の完了タスク：5件
- ブロッカー：1件（外部API連携の仕様未確定）

## 今週の実績
| タスク         | 担当   | 状態   | 備考                |
|---------------|--------|--------|---------------------|
| ユーザー認証    | 佐藤   | 完了 ✅ |                    |
| 検索API実装    | 田中   | 完了 ✅ |                    |
| 管理画面UI     | 鈴木   | 進行中  | 80%完了、来週完了予定 |
| 外部API連携    | 高橋   | ブロック | 仕様確認待ち        |

## リスク・課題
| 項目                    | 影響度 | 対応状況           |
|------------------------|--------|-------------------|
| 外部API仕様未確定       | 高     | 先方に確認中（3/22） |
| テスト環境のリソース不足 | 中     | インフラチームに依頼済 |

## 来週の予定
- 管理画面UI完了
- 結合テスト開始
- パフォーマンステスト準備
```

### 報告のベストプラクティス

1. **結論を先に述べる**（ピラミッド原則）
2. **数値で語る**（「順調です」ではなく「計画比+2日」）
3. **悪いニュースほど早く伝える**（問題の隠蔽は最悪の結果を招く）
4. **アクションアイテムを明確にする**（誰が、何を、いつまでに）
5. **読み手に合わせた粒度で報告する**（経営層にはサマリー、チームには詳細）

> **現場でのポイント**: 「問題ありません」という報告が一番危険です。何をもって問題がないと判断しているか、根拠を示すようにしましょう。

## 7.5 会議の運営とファシリテーション

### 会議の種類（ソフトウェア開発）

| 会議名 | 目的 | 参加者 | 時間目安 |
|--------|------|--------|---------|
| **デイリースタンドアップ** | 日次の同期 | 開発チーム | 15分 |
| **スプリントプランニング** | スプリント計画 | PO+チーム | 2〜4時間 |
| **スプリントレビュー** | 成果物デモ | PO+チーム+SH | 1〜2時間 |
| **レトロスペクティブ** | 振り返り | チーム | 1〜1.5時間 |
| **バックログリファインメント** | 要件の詳細化 | PO+チーム | 1〜2時間 |
| **設計レビュー** | 技術設計の合意 | 開発チーム | 1時間 |
| **ステアリングコミッティ** | 経営層への報告 | PM+経営層 | 1時間 |

### 効果的な会議のルール

```
会議の前（準備）：
  ✅ アジェンダを事前に共有する
  ✅ 参加者を必要最小限にする
  ✅ 事前に資料を配布し、読んでおくよう依頼する
  ✅ 会議の目的とゴールを明確にする

会議中（運営）：
  ✅ 時間を厳守する（タイムボックス）
  ✅ 議論が脱線したら「パーキングロット」に記録して本題に戻る
  ✅ 全員に発言の機会を作る
  ✅ 決定事項とアクションアイテムを記録する

会議の後（フォロー）：
  ✅ 議事録を24時間以内に共有する
  ✅ アクションアイテムの期限と担当を確認する
  ✅ 次回会議の日程を確定する
```

### パーキングロットとは

会議中に出た「今は議論すべきでないが重要なトピック」を一時的に保管する場所です。

```
パーキングロット：
┌─────────────────────────────────┐
│ 1. テスト自動化ツールの選定       │ → 別途技術検討会で議論
│ 2. 来期の採用計画               │ → EM との 1on1 で議論
│ 3. モニタリングダッシュボードの改善 │ → インフラチームに相談
└─────────────────────────────────┘
```

### ファシリテーションのテクニック

| テクニック | 説明 | 使いどころ |
|-----------|------|-----------|
| **ラウンドロビン** | 順番に全員が発言 | 全員の意見を聞きたいとき |
| **ドットボーティング** | 各自が投票で優先順位付け | 多数の選択肢から絞るとき |
| **タイムボックス** | 議題ごとに時間制限を設定 | 議論が長引きがちなとき |
| **5 Whys** | 「なぜ？」を5回繰り返す | 根本原因を突き止めるとき |
| **フィスト・オブ・ファイブ** | 5段階で同意度を示す | チームの合意度を確認するとき |

> **現場でのポイント**: 「会議が多くて仕事が進まない」という声が出たら、会議の棚卸しを行いましょう。目的が曖昧な定例会議は廃止し、必要な情報共有は非同期（Slack、ドキュメント）で代替できないか検討します。

## 7.6 変更管理

### 変更管理とは

プロジェクト進行中に発生する**スコープ、スケジュール、コスト等の変更を、体系的に管理する**プロセスです。変更を禁止するのではなく、変更の影響を適切に評価し、合意のうえで実施することが目的です。

### 変更管理プロセス

```
1. 変更の要求
   → 誰でも変更要求（Change Request）を起票できる
   ↓
2. 変更の記録
   → 変更管理台帳に記録する
   ↓
3. 影響分析
   → スコープ、スケジュール、コスト、品質への影響を評価する
   ↓
4. 変更の審査
   → CCB（Change Control Board）または承認者が判断する
   ↓
5. 承認 or 却下
   → 承認された変更のみ実施する
   ↓
6. 変更の実施
   → 計画を更新し、変更を反映する
   ↓
7. 変更の検証
   → 変更が正しく実施されたことを確認する
```

### 変更要求書のテンプレート

```
変更要求書（Change Request）
─────────────────────────────
変更ID      ：CR-2026-015
起票日      ：2026/03/20
起票者      ：田中（PO）
カテゴリ    ：スコープ変更
優先度      ：高

【変更内容】
管理画面にCSVエクスポート機能を追加する。

【変更理由】
営業部門から月次レポート作成のためにデータ出力機能が
必要との要望があった。

【影響分析】
- スコープ ：API 2本、UI 1画面の追加
- スケジュール：+3日（開発2日 + テスト1日）
- コスト   ：約40万円（人件費）
- 品質     ：既存機能への影響なし（新規追加のため）
- リスク   ：大量データ出力時の性能リスク

【推奨対応】
承認。ただしスプリント13ではなく14で対応する。

【承認】
□ 承認  □ 却下  □ 保留
承認者：_________  日付：________
```

### 変更管理のベストプラクティス

1. **すべての変更を記録する**（口頭だけの合意は避ける）
2. **影響分析を必ず行う**（「簡単だから」で受けない）
3. **承認プロセスを飛ばさない**（緊急時でも事後承認は必須）
4. **変更のトレーサビリティを確保する**（なぜ変更したか追跡可能に）
5. **変更の累積影響を監視する**（小さな変更の積み重ねがスコープクリープに）

> **現場でのポイント**: 「ちょっとした修正だから」と変更管理を通さないケースが最も危険です。小さな変更の積み重ねが、気づいたときには大幅なスケジュール遅延やコスト超過を引き起こします。

## 7.7 リモートチームのコミュニケーション

### リモートワーク特有の課題

| 課題 | 説明 | 対策 |
|------|------|------|
| **情報の非対称性** | オフィスにいる人だけが情報を持つ | すべての情報をオンラインに残す |
| **孤立感** | チームとのつながりが薄くなる | 定期的な雑談の時間を設ける |
| **タイムゾーン差** | 同期的なコミュニケーションが困難 | 重複する時間帯を活用する |
| **意思決定の遅延** | 確認に時間がかかる | 意思決定の権限を委譲する |
| **非言語情報の欠如** | 表情やジェスチャーが見えない | カメラONを推奨する |

### 非同期コミュニケーションのベストプラクティス

```
効果的な非同期コミュニケーションのルール：

1. ドキュメント・ファースト
   → 口頭で決めたことも必ず文書化する

2. コンテキストを含める
   → 「あの件ですが」ではなく「CR-015のCSVエクスポートの件ですが」

3. 期限と期待を明確にする
   → 「確認お願いします」ではなく「3/25までにレビューをお願いします」

4. 適切なツールを使う
   → 緊急：電話/Slack（即時通知）
   → 日常：Slack チャネル
   → 記録：Wiki/ドキュメント
   → 意思決定：メール/チケット

5. レスポンスタイムの期待値を合意する
   → 例：Slackは4時間以内、メールは翌営業日まで
```

## 7.8 コンフリクト（対立）の管理

### コンフリクトの原因

ソフトウェア開発プロジェクトでよくある対立の原因です。

| 原因 | 例 |
|------|-----|
| **優先順位の不一致** | POとエンジニアで実装順序が異なる |
| **リソースの競合** | 複数プロジェクトで同じエンジニアを取り合う |
| **技術的見解の相違** | アーキテクチャの選択でチームが分裂 |
| **スケジュールのプレッシャー** | 無理な納期に対する不満 |
| **コミュニケーション不足** | 意図が正しく伝わっていない |

### コンフリクト解決の5つのスタイル

```
          高
    ├──────────────────────┤
    │  競争        │  協調  │
自  │ (Competing)  │(Collaborating)
分  │ Win-Lose     │ Win-Win │
の  │──────────────┼────────│
関  │  回避        │  妥協   │
心  │ (Avoiding)   │(Compromising)
の  │ Lose-Lose    │ 部分的Win│
充  │──────────────┼────────│
足  │  適応               │
度  │ (Accommodating)     │
    │ Lose-Win            │
  低├──────────────────────┤
    低    相手の関心の充足度    高
```

| スタイル | 説明 | 使いどころ |
|---------|------|-----------|
| **協調** | 双方の要求を満たす解決策を模索 | 重要な課題で時間がある場合 |
| **妥協** | 双方が少しずつ譲歩する | 時間が限られ、暫定的な解決が必要な場合 |
| **競争** | 自分の主張を押し通す | 緊急時や重大な安全問題の場合 |
| **適応** | 相手の意見を受け入れる | 関係維持が重要で、論点が小さい場合 |
| **回避** | 対立を先送りにする | 冷却期間が必要な場合 |

> **現場でのポイント**: PMBOKでは**協調**が最も望ましいコンフリクト解決手法とされています。ただし、常に協調が可能とは限りません。状況に応じて適切なスタイルを選ぶことが重要です。

✅ コンフリクトは協調的解決を目指すが、状況に応じて使い分ける

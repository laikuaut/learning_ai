## 4.1 コスト管理の基本

### コスト管理とは

コスト管理とは、**プロジェクトを承認された予算内で完了させる**ための一連のプロセスです。単にお金を節約することではなく、投入するコストに対して最大の価値を生み出すことが目的です。

### コスト管理の4つのプロセス

```
1. コスト見積もり（Estimate Costs）
   → 各作業にどれだけのコストがかかるかを予測する
   ↓
2. 予算の策定（Determine Budget）
   → 見積もりを集約し、コストベースラインを確定する
   ↓
3. コストのコントロール（Control Costs）
   → 実績と計画の差異を監視し、是正措置を講じる
   ↓
4. コストの報告（Report Costs）
   → ステークホルダーにコスト状況を報告する
```

### ソフトウェア開発におけるコスト構造

ソフトウェアプロジェクトのコストは、ハードウェア製造業とは大きく異なります。最大のコスト要因は**人件費**です。

| コスト項目 | 内訳例 | 全体に占める割合（目安） |
|-----------|--------|------------------------|
| **人件費** | エンジニア、デザイナー、PM、QA | 60〜80% |
| **インフラ費** | クラウド（AWS/GCP/Azure）、SaaS | 10〜20% |
| **ツール・ライセンス費** | IDE、CI/CDサービス、監視ツール | 3〜5% |
| **外注費** | 外部委託、フリーランス | 0〜20% |
| **間接費** | オフィス、通信、教育研修 | 5〜10% |

> **現場でのポイント**: ソフトウェア開発では「人月（にんげつ）」がコスト計算の基本単位になります。1人月 = 1人のエンジニアが1ヶ月フルタイムで働くコストです。例えば月額単価80万円のエンジニアが5名で3ヶ月なら、人件費は 80万 x 5人 x 3ヶ月 = 1,200万円です。

## 4.2 見積もり手法

### 4.2.1 アナロジー見積もり（類推見積もり）

過去の類似プロジェクトの実績データを基に、現在のプロジェクトのコストを推定する手法です。

```
過去の類似プロジェクト実績
  └→ ECサイト構築：8名で6ヶ月 = 4,800万円

今回のプロジェクト
  └→ 同規模のECサイト構築
  └→ 追加要件あり（+20%と見積もる）
  └→ 推定コスト：4,800万 × 1.2 = 5,760万円
```

**メリット：**
- 少ない情報でも早期に見積もれる
- 過去の実績に基づくため、根拠がある
- 短時間で実施できる

**デメリット：**
- 過去のプロジェクトとの類似度が低いと精度が落ちる
- 過去データの蓄積が必要
- 新技術を使う場合には適用しにくい

> **よくある間違い**: 「前回と同じくらいでしょ」と安易に見積もること。チームメンバーの経験、使用技術、要件の複雑さなど、差分を丁寧に分析する必要があります。

### 4.2.2 パラメトリック見積もり

統計的なモデルや数式を使ってコストを算出する手法です。プロジェクトの特性（パラメータ）と過去データの関係から推定します。

```
基本式：
  コスト = 単位コスト × 数量 × 調整係数

例1：画面数ベース
  単価：150万円/画面
  画面数：20画面
  複雑度係数：1.3（複雑な業務ロジックあり）
  推定コスト = 150万 × 20 × 1.3 = 3,900万円

例2：ファンクションポイント（FP）ベース
  FP数：500FP
  生産性：10FP/人月
  人月単価：80万円
  必要人月 = 500 ÷ 10 = 50人月
  推定コスト = 50 × 80万 = 4,000万円
```

**メリット：**
- 再現性が高い（同じパラメータなら同じ結果）
- 大規模プロジェクトの概算に適している
- 組織として知見を蓄積しやすい

**デメリット：**
- 信頼できる統計データが必要
- パラメータの選定が難しい
- 技術的な不確実性を反映しにくい

### 4.2.3 ボトムアップ見積もり

WBS（Work Breakdown Structure）の最下位レベルの各作業パッケージについて個別にコストを見積もり、積み上げて全体コストを算出する手法です。

```
プロジェクト全体
├── 要件定義フェーズ
│   ├── ユーザーヒアリング    : 2人月 = 160万円
│   ├── 要件書作成            : 1.5人月 = 120万円
│   └── 要件レビュー          : 0.5人月 = 40万円
│                             小計: 320万円
├── 設計フェーズ
│   ├── アーキテクチャ設計    : 2人月 = 160万円
│   ├── DB設計                : 1人月 = 80万円
│   ├── API設計               : 1.5人月 = 120万円
│   └── UI/UXデザイン         : 2人月 = 160万円
│                             小計: 520万円
├── 実装フェーズ
│   ├── バックエンド実装      : 6人月 = 480万円
│   ├── フロントエンド実装    : 5人月 = 400万円
│   └── インフラ構築          : 2人月 = 160万円
│                             小計: 1,040万円
├── テストフェーズ
│   ├── 単体テスト            : 2人月 = 160万円
│   ├── 結合テスト            : 2人月 = 160万円
│   └── システムテスト        : 1.5人月 = 120万円
│                             小計: 440万円
└── PM・管理コスト            : 4人月 = 320万円
                               ─────────
                          合計: 2,640万円
                    + 予備費(10%): 264万円
                        総計: 2,904万円
```

**メリット：**
- 最も精度が高い
- 各作業の担当者が見積もるので現実的
- コスト超過の原因を特定しやすい

**デメリット：**
- 時間がかかる（WBSが必要）
- 早期段階では実施困難
- 個々の見積もり精度に依存する

### 4.2.4 三点見積もり（PERT見積もり）

第3章で学んだPERT法をコストにも適用します。楽観値・最頻値・悲観値からコストの期待値を算出します。

```
期待値 = (楽観値 + 4×最頻値 + 悲観値) ÷ 6

例：API実装のコスト見積もり
  楽観値（O）: 300万円（すべて順調な場合）
  最頻値（M）: 480万円（通常の場合）
  悲観値（P）: 900万円（大きな問題が発生した場合）

  期待値 = (300 + 4×480 + 900) ÷ 6
         = (300 + 1,920 + 900) ÷ 6
         = 3,120 ÷ 6
         = 520万円

  標準偏差 = (900 - 300) ÷ 6 = 100万円

  → 520万円 ± 100万円（68%の確率でこの範囲に収まる）
  → 520万円 ± 200万円（95%の確率でこの範囲に収まる）
```

### 見積もり手法の使い分け

| 手法 | 精度 | 必要情報 | 適用時期 | 所要時間 |
|------|------|---------|---------|---------|
| アナロジー | 低〜中 | 過去の類似実績 | 企画・提案段階 | 短い |
| パラメトリック | 中 | 統計データ・パラメータ | 企画〜計画段階 | 中程度 |
| ボトムアップ | 高 | 詳細なWBS | 計画段階 | 長い |
| 三点見積もり | 中〜高 | 各タスクの3値 | 計画段階 | 中〜長い |

> **現場でのベストプラクティス**: プロジェクトの進行に合わせて見積もり手法を変えていきます。初期はアナロジーで概算を出し、計画フェーズに入ったらボトムアップで精緻化するのが一般的です。これを「ローリングウェーブ計画法」と呼びます。

## 4.3 EVM（アーンドバリューマネジメント）

### EVMとは

EVM（Earned Value Management）は、プロジェクトの**スケジュールとコストの両方を統合的に管理**するための手法です。「計画に対してどれだけ進んでいて、どれだけのコストを使っているか」を定量的に測定できます。

### EVMの3つの基本指標

| 指標 | 英語名 | 意味 | 質問 |
|------|--------|------|------|
| **PV** | Planned Value（計画価値） | 現時点までに完了しているべき作業の計画コスト | 「今日までにいくら分の仕事をする予定だったか？」 |
| **EV** | Earned Value（出来高） | 現時点で実際に完了した作業の計画コスト | 「今日までに実際にいくら分の仕事が完了したか？」 |
| **AC** | Actual Cost（実コスト） | 現時点で実際に使ったコスト | 「今日までに実際にいくら使ったか？」 |

### EVMの図解

```
コスト
  ↑
  │        ／ BAC（完了時総予算）
  │       ／
  │      ／ ← PV（計画価値）の曲線
  │     ／
  │    ★ ← 現時点
  │   ╱  ╲
  │  AC    EV
  │ (実コスト) (出来高)
  │
  └──────────────────→ 時間
```

### EVMの分析指標

**差異分析（Variance Analysis）：**

| 指標 | 計算式 | 意味 | 正の場合 | 負の場合 |
|------|--------|------|---------|---------|
| **CV（コスト差異）** | EV - AC | コストの差異 | 予算内 | 予算超過 |
| **SV（スケジュール差異）** | EV - PV | スケジュールの差異 | 前倒し | 遅延 |

**効率指標（Performance Index）：**

| 指標 | 計算式 | 意味 | >1.0 | <1.0 |
|------|--------|------|------|------|
| **CPI（コスト効率指数）** | EV ÷ AC | コスト効率 | 効率的 | 非効率 |
| **SPI（スケジュール効率指数）** | EV ÷ PV | スケジュール効率 | 前倒し | 遅延 |

### EVM計算の具体例

あるWebアプリ開発プロジェクトの3ヶ月目の状況を見てみましょう。

```
プロジェクト概要：
  総予算（BAC）: 3,000万円
  期間: 6ヶ月
  現在: 3ヶ月目の末日

EVMデータ：
  PV（計画価値）    : 1,500万円（3ヶ月目までに完了予定の計画コスト）
  EV（出来高）      : 1,200万円（実際に完了した作業の計画コスト）
  AC（実コスト）    : 1,400万円（実際に使ったコスト）
```

**ステップ1：差異を計算する**

```
CV = EV - AC = 1,200 - 1,400 = -200万円
→ 200万円の予算超過

SV = EV - PV = 1,200 - 1,500 = -300万円
→ 300万円分のスケジュール遅延
```

**ステップ2：効率指数を計算する**

```
CPI = EV ÷ AC = 1,200 ÷ 1,400 = 0.857
→ 1円あたり0.857円分の価値しか生み出していない（非効率）

SPI = EV ÷ PV = 1,200 ÷ 1,500 = 0.800
→ 予定の80%しか進んでいない（遅延）
```

**ステップ3：完了時の予測を行う**

```
EAC（完了時コスト予測）= BAC ÷ CPI
  = 3,000 ÷ 0.857
  = 3,501万円

→ このペースだと最終的に約3,500万円かかる見込み（500万円の超過）

ETC（残作業コスト予測）= EAC - AC
  = 3,501 - 1,400
  = 2,101万円

→ 残りの作業に約2,100万円必要

VAC（完了時コスト差異）= BAC - EAC
  = 3,000 - 3,501
  = -501万円

→ 約500万円の予算超過で終わる見込み
```

### EVMの判断基準

```
CPI と SPI の組み合わせによる判断：

CPI > 1.0 かつ SPI > 1.0  → 理想的（予算内＆前倒し）
CPI > 1.0 かつ SPI < 1.0  → コストOK、スケジュール遅延
CPI < 1.0 かつ SPI > 1.0  → 予算超過、スケジュールOK
CPI < 1.0 かつ SPI < 1.0  → 危機的状況（両方悪い）
```

> **現場でのベストプラクティス**: CPIが0.8を下回ったら赤信号です。PMBOKの統計によると、プロジェクトの20%完了時点でCPIが0.8以下の場合、挽回できる確率は非常に低いとされています。早期の対策が不可欠です。

## 4.4 予算の策定と管理

### 予算策定のステップ

```
1. 各作業パッケージの見積もりを集約する
   ↓
2. マネジメント予備費（Management Reserve）を加える
   → 想定外のリスクに備える予備費（通常5〜15%）
   ↓
3. コンティンジェンシー予備費（Contingency Reserve）を加える
   → 特定済みリスクに備える予備費（通常10〜20%）
   ↓
4. コストベースラインを確定する
   → 進捗管理の基準となる予算の基準線
```

### 予算の構造

```
┌─────────────────────────────────────────┐
│          プロジェクト総予算               │
│                                          │
│  ┌───────────────────────────────────┐   │
│  │      コストベースライン            │   │
│  │                                    │   │
│  │  ┌─────────────────────────┐      │   │
│  │  │  作業見積もりの合計      │      │   │
│  │  │  （各WBSの積み上げ）    │      │   │
│  │  └─────────────────────────┘      │   │
│  │  + コンティンジェンシー予備        │   │
│  │    （特定済みリスク対応）          │   │
│  └───────────────────────────────────┘   │
│  + マネジメント予備                      │
│    （未知のリスク対応）                  │
└─────────────────────────────────────────┘
```

### 具体例：予算策定

```
Webアプリケーション開発プロジェクト

作業見積もり合計：          2,640万円
+ コンティンジェンシー予備（15%）: 396万円
= コストベースライン：      3,036万円
+ マネジメント予備（10%）:    304万円
= プロジェクト総予算：      3,340万円

内訳：
  人件費：          2,100万円
  インフラ費：        280万円
  外注費：           200万円
  ツール・ライセンス：  60万円
  予備費合計：        700万円
```

### 予算消化の監視

定期的に予算消化状況を監視し、異常を早期に検知します。

```
月次予算消化レポートの例：

月    | 計画予算  | 実績    | 差異    | 消化率
------|----------|---------|---------|-------
1月   | 400万円  | 380万円 | +20万  | 95%
2月   | 450万円  | 470万円 | -20万  | 104%
3月   | 500万円  | 550万円 | -50万  | 110%  ← 要注意
4月   | 500万円  | ?       |         |
5月   | 450万円  |         |         |
6月   | 340万円  |         |         |
予備  | 700万円  |         |         |
合計  | 3,340万円|         |         |

3月時点の累計：
  計画：1,350万円
  実績：1,400万円
  差異：-50万円（3.7%超過）
```

## 4.5 ソフトウェア開発特有のコスト課題

### 人月の神話

フレデリック・ブルックスの著書にちなんだ概念です。ソフトウェア開発では、**人を増やしても単純に生産性が比例して上がるわけではない**ことを示しています。

```
理想（線形スケール）：
  5人で6ヶ月の作業 → 10人なら3ヶ月？

現実（コミュニケーションコスト増大）：
  5人のコミュニケーションパス：5×4÷2 = 10本
  10人のコミュニケーションパス：10×9÷2 = 45本

  → コミュニケーションコストが4.5倍に！
  → 10人にしても3ヶ月では終わらない可能性が高い
```

### 技術的負債のコスト

技術的負債（Technical Debt）は、短期的な速度を優先した結果、将来的に追加コストが発生するものです。

```
技術的負債の具体例：
  ・テストが書かれていないコード → バグ修正コスト増
  ・古いフレームワーク → セキュリティ対応コスト増
  ・ドキュメント不足 → オンボーディングコスト増
  ・密結合なアーキテクチャ → 変更コスト増

コスト影響のイメージ：
  Year 1: 技術的負債の返済コスト 0%
  Year 2: 開発速度が10〜20%低下
  Year 3: 開発速度が30〜50%低下
  → 放置すると複利的にコストが膨らむ
```

### 隠れたコスト

ソフトウェア開発には見積もりから漏れやすい隠れたコストがあります。

| 隠れたコスト | 説明 | 対策 |
|-------------|------|------|
| **コミュニケーションコスト** | 会議、チャット、コードレビュー | WBSに明示的に含める |
| **環境構築コスト** | 開発環境、CI/CD、ステージング環境 | インフラ費として計上 |
| **学習コスト** | 新技術のキャッチアップ | バッファに含める |
| **手戻りコスト** | 仕様変更、バグ修正 | 予備費に含める |
| **移行コスト** | データ移行、既存システム連携 | 独立した作業として見積もる |
| **運用開始後のサポート** | 問い合わせ対応、初期不具合対応 | 保守フェーズとして予算化 |

## 4.6 コスト超過への対処

### コスト超過の早期警告サイン

以下の兆候が見られたら、コスト超過のリスクが高まっています。

```
警告サイン：
  ・CPIが1.0を下回り始めた
  ・計画外の残業が増加している
  ・スコープが曖昧なまま拡大している（スコープクリープ）
  ・外部ベンダーからの追加請求が発生している
  ・予備費の消化が進捗を上回っている
  ・技術的な問題の解決に想定以上の時間がかかっている
```

### 対処法

| 対処法 | 内容 | メリット | デメリット |
|--------|------|---------|----------|
| **スコープ削減** | 優先度の低い機能を後回し | 品質を保てる | ステークホルダーの合意が必要 |
| **リソース最適化** | チーム構成の見直し | コスト効率改善 | 調整に時間がかかる |
| **プロセス改善** | 無駄な作業の排除 | 持続的な改善 | 即効性は低い |
| **技術的解決** | ライブラリ活用、自動化 | 長期的な効率化 | 初期コストがかかる |
| **予備費の活用** | 予備費から充当 | 迅速に対応可能 | 残りの予備が減る |
| **予算の再承認** | ステークホルダーに追加予算を申請 | 現実的な予算になる | 信頼を損なうリスク |

> **よくある間違い**: コスト超過の事実を隠したり、報告を遅らせたりすること。早期に透明性を持って報告し、対策を講じることが最も重要です。報告が遅れるほど、取れる選択肢が減っていきます。

## 4.7 コスト管理のツールとテンプレート

### コスト見積もりシートの例

```
┌────────────────────────────────────────────────────┐
│ プロジェクト名：ECサイトリニューアル                │
│ 作成日：2025/04/01  作成者：PM太郎                 │
```

## まとめ

| 概念 | ポイント |
|------|---------|
| コスト構造 | ソフトウェア開発は人件費が60〜80%を占める |
| アナロジー見積もり | 過去の類似実績から概算する（早期段階向き） |
| ボトムアップ見積もり | WBSから積み上げる（最も精度が高い） |
| EVM | PV・EV・ACでスケジュールとコストを統合管理 |
| CPI/SPI | 1.0を基準に効率を測る（<1.0は危険信号） |
| 予備費 | コンティンジェンシーとマネジメント予備を分けて設定 |
| 人月の神話 | 人を増やしてもコミュニケーションコストが増大する |
| 早期対処 | コスト超過は早期発見・透明性のある報告が最重要 |

## 課題の要件

以下のシナリオを読み、WBSを作成してください。

### シナリオ

あなたのチーム（4名）は、飲食チェーン（10店舗）向けの**メニュー管理Webアプリ**を開発することになりました。

**プロジェクトの概要：**
- 各店舗のメニュー（料理名、価格、カテゴリ、写真）を一元管理するWebアプリ
- 本部でメニュー情報を登録・更新し、各店舗のタブレットに表示する
- 季節メニューの追加・削除が簡単にできるようにする
- 期間は4ヶ月、予算は300万円

### 作成ルール

1. **第1階層**: プロジェクトの大きなフェーズ（5〜7個）
2. **第2階層**: 各フェーズの主要作業（各フェーズに3〜5個）
3. **第3階層**: 具体的なワークパッケージ（必要に応じて）
4. **100%ルール**: 上位の作業 = 下位の作業の合計（漏れなし、ダブりなし）
5. 各ワークパッケージは「成果物」が明確であること

## ステップガイド

<details>
<summary>ステップ1：第1階層（フェーズ）を洗い出す</summary>

ソフトウェア開発プロジェクトでは、一般的に以下のフェーズがあります。

```
典型的なフェーズ構成：
  1. プロジェクト管理  ← プロジェクト全体を通して行う活動
  2. 要件定義        ← 何を作るかを決める
  3. 設計            ← どう作るかを決める
  4. 開発（実装）     ← 実際に作る
  5. テスト          ← 正しく動くか確認する
  6. 導入・移行      ← 本番環境に展開する
```

シナリオ固有の要素（メニューデータ移行、店舗展開など）も考慮しましょう。

</details>

<details>
<summary>ステップ2：第2階層を分解する</summary>

各フェーズを「成果物ベース」で分解します。

**分解のコツ：**
- 「〜する」ではなく「〜を作る」「〜を完了する」で考える
- 1つのワークパッケージは1人が1〜2週間で完了できる粒度が目安
- 「レビュー」や「承認」も忘れずに含める

**例：要件定義フェーズの場合**
```
2. 要件定義
├── 2.1 店舗スタッフへのヒアリング
├── 2.2 要件定義書の作成
├── 2.3 画面ワイヤーフレーム作成
└── 2.4 要件レビュー・承認
```

</details>

<details>
<summary>ステップ3：漏れがないか確認する</summary>

以下のチェックリストで確認しましょう。

```
□ プロジェクト管理活動（計画、進捗管理、報告）は含まれているか？
□ 各フェーズにレビュー・承認のステップがあるか？
□ テスト工程は十分に分解されているか？
□ データ移行（既存メニュー情報の取り込み）は含まれているか？
□ ユーザー教育・研修は含まれているか？
□ 環境構築（開発環境、本番環境）は含まれているか？
□ ドキュメント作成（マニュアルなど）は含まれているか？
```

</details>

<summary>解答例（改良版 ─ 成果物と担当を明記）</summary>

改良版では、各ワークパッケージに「成果物」と「担当」を付けてより実務的にします。

```
メニュー管理Webアプリ開発プロジェクト
│
├── 1. プロジェクト管理
│   ├── 1.1 プロジェクト計画書作成

│   │     成果物: 議事録            担当: PM
│   ├── 1.3 課題・リスク管理
│   │     成果物: 課題管理表         担当: PM
│   └── 1.4 プロジェクト完了報告
│         成果物: 完了報告書         担当: PM
│
├── 2. 要件定義（2週間）
│   ├── 2.1 ステークホルダーヒアリング
│   │   ├── 2.1.1 本部担当者（メニュー企画部）
│   │   └── 2.1.2 店舗スタッフ（2店舗、各30分）
│   │     成果物: ヒアリング議事録    担当: PM + メンバーA
│   ├── 2.2 要件定義書作成

│   ├── 2.4 非機能要件の定義
│   │     成果物: 非機能要件一覧     担当: メンバーB
```

**初心者向けとの違い：**
- **第3階層まで分解**: ヒアリングや画面設計を具体的に分けることで、作業漏れを防ぎます
- **成果物と担当の明記**: 「誰が何を作るか」が明確になり、責任が曖昧になりません
- **非機能要件を独立**: パフォーマンスやセキュリティの要件定義を忘れがちですが、独立して管理します
- **保守引き渡しフェーズの追加**: 開発して終わりではなく、運用に引き渡すまでがプロジェクトです
- **概算期間の記載**: WBSと期間を紐づけると、後のガントチャート作成がスムーズになります

</details>

## 8.1 アジャイル開発の基本

### アジャイルマニフェスト（2001年）

アジャイル開発は、2001年に17名のソフトウェア開発者が「アジャイルソフトウェア開発宣言」を発表したことが出発点です。

```
アジャイルソフトウェア開発宣言（Agile Manifesto）

私たちは以下を重視する：

  プロセスやツール        よりも  個人と対話
  包括的なドキュメント    よりも  動くソフトウェア
  契約交渉              よりも  顧客との協調
  計画に従うこと         よりも  変化への対応

左側にも価値はあるが、右側により大きな価値を置く。
```

### アジャイルの12の原則（要約）

| # | 原則 | ソフトウェア開発での意味 |
|---|------|----------------------|
| 1 | 顧客満足を最優先する | 早期かつ継続的にソフトウェアを届ける |
| 2 | 要求の変更を歓迎する | 変化を競争力の源泉と考える |
| 3 | 動くソフトウェアを頻繁に提供する | 数週間〜数ヶ月のサイクルで |
| 4 | ビジネスと開発が日々協働する | POとチームの密なコミュニケーション |
| 5 | 意欲ある個人を中心に構成する | チームを信頼し、環境を整える |
| 6 | 対話は直接話すのが最善 | フェイス・トゥ・フェイスを重視 |
| 7 | 動くソフトウェアが進捗の最も重要な指標 | ドキュメントではなく成果物で測る |
| 8 | 持続可能なペースを維持する | 無理な残業は避ける |
| 9 | 技術的卓越性に継続的に注意を払う | 技術的負債を放置しない |
| 10 | シンプルさを追求する | やらないことを決める |
| 11 | 最良のアーキテクチャは自己組織化チームから生まれる | チームに権限を委譲する |
| 12 | 定期的に振り返り、やり方を改善する | レトロスペクティブの実施 |

### アジャイル開発のフレームワーク比較

| フレームワーク | 特徴 | 適したケース |
|-------------|------|------------|
| **スクラム** | 役割・イベント・成果物が明確に定義 | 多くのソフトウェア開発プロジェクト |
| **カンバン** | 作業の可視化とフロー最適化に注力 | 運用・保守、継続的な改善 |
| **XP** | 技術プラクティスを重視 | 品質を重視するプロジェクト |
| **リーン** | 無駄の排除と価値の最大化 | スタートアップ、新規事業 |

## 8.2 スクラム（Scrum）

### スクラムの全体像

スクラムは、複雑な問題に対応するための軽量なフレームワークです。3つのロール、5つのイベント、3つの成果物で構成されます。

```
スクラムの流れ：

プロダクトバックログ
       │
       ▼
┌─── スプリント（1〜4週間）─────────────────┐
│                                           │
│  スプリント     デイリー      スプリント    │
│  プランニング → スクラム  →  レビュー      │
│                 （毎日）                   │
│                              ↓            │
│                         レトロスペクティブ  │
│                                           │
│  [スプリントバックログ] → [インクリメント]   │
└───────────────────────────────────────────┘
       │
       ▼
    リリース
```

### 3つのロール

| ロール | 責務 | ソフトウェア開発での例 |
|--------|------|---------------------|
| **プロダクトオーナー（PO）** | プロダクトの価値を最大化する。バックログの管理と優先順位付け | 事業部門のマネージャー、プロダクトマネージャー |
| **スクラムマスター（SM）** | スクラムの導入と運用を支援。チームの障害を除去する | チームリーダー、アジャイルコーチ |
| **開発チーム** | スプリントごとにインクリメント（動くソフトウェア）を作成する | エンジニア、デザイナー、QA |

```
ロール間の関係：

  プロダクトオーナー（PO）
    │
    │ 「何を作るか」を決める
    │ バックログを管理する
    │
    ▼
  開発チーム（3〜9名）
    │
    │ 「どう作るか」を決める
    │ インクリメントを作成する
    │
    ◀── スクラムマスター（SM）
         プロセスを支援する
         障害を除去する
```

> **よくある間違い**: スクラムマスターは「チームの管理者」ではありません。SMの役割はサーバントリーダーシップ（奉仕型リーダーシップ）であり、チームが自律的に働ける環境を整えることです。

### 5つのイベント

#### 1. スプリントプランニング

スプリントの最初に行う計画会議です。

```
スプリントプランニングの流れ：

Part 1: 「何をやるか」（What）
  ・POがバックログの優先順位を説明する
  ・チームがスプリントに取り込むアイテムを選ぶ
  ・スプリントゴール（このスプリントで達成する目標）を設定する

Part 2: 「どうやるか」（How）
  ・選んだアイテムをタスクに分解する
  ・各タスクの工数を見積もる
  ・スプリントバックログを作成する

時間の目安：
  2週間スプリント → 最大4時間
  1週間スプリント → 最大2時間
```

#### 2. デイリースクラム

毎日同じ時間・場所で行う15分のミーティングです。

```
デイリースクラムの3つの質問：

1. 昨日やったことは？
   → 「ユーザー認証のAPI実装を完了しました」

2. 今日やることは？
   → 「認証のフロントエンド部分に取りかかります」

3. 障害（ブロッカー）はある？
   → 「テスト環境がダウンしていて、結合テストができません」
```

> **よくある間違い**: デイリースクラムは「PMへの進捗報告会」ではありません。チームメンバーが互いに同期し、協力して障害を解決するための場です。

#### 3. スプリントレビュー

スプリントの成果物をステークホルダーにデモし、フィードバックを得る場です。

```
スプリントレビューの流れ：
  1. スプリントゴールの達成状況を説明する
  2. 完成したインクリメントをデモする
  3. ステークホルダーからフィードバックを受ける
  4. バックログを調整する（新しい発見を反映）

時間の目安：2週間スプリント → 最大2時間
```

#### 4. レトロスペクティブ（振り返り）

チームのプロセスを振り返り、改善点を特定する場です（詳細は8.5節で解説）。

#### 5. バックログリファインメント

将来のスプリントに向けてバックログアイテムを詳細化する活動です。

```
リファインメントで行うこと：
  ・ユーザーストーリーの詳細化
  ・受け入れ基準の明確化
  ・見積もり（ストーリーポイント）の付与
  ・大きすぎるストーリーの分割
  ・依存関係の確認

目安：スプリントの10%程度の時間を充てる
```

### 3つの成果物

| 成果物 | 説明 | 管理者 |
|--------|------|--------|
| **プロダクトバックログ** | 製品に必要な機能や改善の一覧（優先順位付き） | PO |
| **スプリントバックログ** | 今回のスプリントで取り組むアイテムとタスク | 開発チーム |
| **インクリメント** | スプリントの成果物（動くソフトウェア） | 開発チーム |

```
プロダクトバックログの例：

| 優先度 | ID    | ユーザーストーリー                         | SP  | 状態   |
|--------|-------|-----------------------------------------|-----|--------|
| 1      | US-01 | ユーザーとして、メールでログインしたい        | 5   | 完了   |
| 2      | US-02 | ユーザーとして、商品を検索したい             | 8   | 進行中 |
| 3      | US-03 | 管理者として、売上レポートを見たい           | 13  | 未着手 |
| 4      | US-04 | ユーザーとして、お気に入りに登録したい        | 3   | 未着手 |
| 5      | US-05 | ユーザーとして、SNSでシェアしたい            | 5   | 未着手 |

SP = ストーリーポイント（相対見積もり）
```

## 8.3 カンバン（Kanban）

### カンバンの基本

カンバンは、作業を可視化し、フローを最適化するための手法です。スクラムのようにスプリントという固定期間を設けず、継続的に作業を流します。

### カンバンの4つの原則

1. **作業の可視化** - すべての作業をボード上で見える化する
2. **WIP（仕掛かり作業）制限** - 同時並行の作業数を制限する
3. **フローの管理** - 作業の流れを計測し、ボトルネックを特定する
4. **継続的改善** - プロセスを定期的に見直し、改善する

### カンバンボードの例

```
┌──────────┬──────────┬──────────┬──────────┬──────────┐
│ Backlog  │  To Do   │ In Dev   │ Review   │  Done    │
│          │  (3)     │  (2)     │  (2)     │          │
│          │ WIP: 3   │ WIP: 2   │ WIP: 2   │          │
├──────────┼──────────┼──────────┼──────────┼──────────┤
│ ┌──────┐ │ ┌──────┐ │ ┌──────┐ │ ┌──────┐ │ ┌──────┐ │
│ │US-08 │ │ │US-05 │ │ │US-03 │ │ │US-02 │ │ │US-01 │ │
│ └──────┘ │ ├──────┤ │ ├──────┤ │ └──────┘ │ ├──────┤ │
│ ┌──────┐ │ │US-06 │ │ │US-04 │ │          │ │US-09 │ │
│ │US-10 │ │ └──────┘ │ └──────┘ │          │ └──────┘ │
│ └──────┘ │ ┌──────┐ │          │          │          │
│ ┌──────┐ │ │US-07 │ │          │          │          │
│ │US-11 │ │ └──────┘ │          │          │          │
│ └──────┘ │          │          │          │          │
└──────────┴──────────┴──────────┴──────────┴──────────┘
             WIP制限を超えていたら新しい作業を引かない
```

### WIP制限（Work In Progress制限）

各列（ステージ）で同時に進行できる作業数の上限を設定します。

```
WIP制限の効果：

WIP制限なし：                    WIP制限あり：
┌───┐ ┌───┐ ┌───┐ ┌───┐        ┌───┐ ┌───┐
│ A │ │ B │ │ C │ │ D │        │ A │ │ B │  ← 2つに集中
└───┘ └───┘ └───┘ └───┘        └───┘ └───┘

→ コンテキストスイッチが多い      → 集中してタスクを完了
→ 全部が少しずつ進む              → 1つずつ確実に完了
→ 完了までの時間が長い            → リードタイムが短い
```

> **現場でのポイント**: WIP制限は「サボっていい」という意味ではありません。制限に達したら、滞留しているタスクを助ける（ペアプロ、コードレビュー等）ことが期待されます。

### スクラムとカンバンの比較

| 項目 | スクラム | カンバン |
|------|---------|---------|
| タイムボックス | スプリント（固定期間） | なし（継続的フロー） |
| ロール | PO、SM、開発チーム | 特に定義なし |
| 計画 | スプリントプランニング | 必要に応じて |
| 変更 | スプリント中は変更しない | いつでも追加可能 |
| 指標 | ベロシティ | リードタイム、スループット |
| 適した状況 | 新規開発 | 運用・保守、サポート |

## 8.4 スプリント計画とベロシティ

### ストーリーポイントとは

タスクの**相対的な大きさ**を表す単位です。時間ではなく、複雑さ・量・不確実性を総合的に評価します。

```
ストーリーポイントの基準（フィボナッチ数列）：

1  - とても簡単（ボタンのラベル変更）
2  - 簡単（既存APIにフィールド追加）
3  - やや簡単（新しいフォーム画面作成）
5  - 中程度（CRUD機能一式の実装）
8  - やや複雑（外部API連携機能）
13 - 複雑（検索エンジンの実装）
21 - とても複雑（決済システムの実装）→ 分割を検討
```

### プランニングポーカー

チーム全員で見積もりを行う手法です。

```
プランニングポーカーの流れ：

1. POがユーザーストーリーを説明する
2. チームが質問し、理解を深める
3. 全員が同時にカードを出す（1, 2, 3, 5, 8, 13...）
4. 見積もりが揃ったら → 採用
5. ばらつきがあったら → 最高値と最低値の人が理由を説明
6. 再度カードを出す
7. 合意に至るまで繰り返す（最大3回程度）

例：
  Aさん: 5   Bさん: 5   Cさん: 8   Dさん: 5
  → Cさん「外部APIの認証が複雑になる可能性がある」
  → チーム「確かに。ただし既存の認証基盤を使える」
  → 再投票 → 全員5 → 採用
```

### ベロシティ

チームが1スプリントで完了できるストーリーポイントの量です。

```
ベロシティの推移例：

SP
40 |
35 |              ■        ■
30 |         ■         ■        ■   ← 平均ベロシティ: 30 SP
25 |    ■
20 | ■                               ← 立ち上げ期は低い
15 |
10 |
 0 +-----+----+----+----+----+----→ スプリント
   SP1   SP2  SP3  SP4  SP5  SP6
   20    25   30   35   30   35
```

### ベロシティを使ったスプリント計画

```
チームの平均ベロシティ：30 SP/スプリント

プロダクトバックログの残り：120 SP

残りスプリント数の予測：
  楽観的（ベロシティ35）：120 / 35 ≈ 3.4 → 4スプリント
  現実的（ベロシティ30）：120 / 30 = 4.0 → 4スプリント
  悲観的（ベロシティ25）：120 / 25 = 4.8 → 5スプリント

→ 4〜5スプリント（2〜2.5ヶ月）で完了見込み
```

> **現場でのポイント**: ベロシティは**チーム間で比較してはいけません**。見積もりの基準がチームごとに異なるため、「AチームのベロシティはBチームの2倍だから優秀」という比較は無意味です。

## 8.5 レトロスペクティブ（振り返り）

### レトロスペクティブとは

スプリントの終わりに行う振り返りで、チームのプロセスを改善するための場です。**何がうまくいったか、何がうまくいかなかったか、どう改善するか**を議論します。

### 代表的な振り返り手法

#### 1. KPT（Keep / Problem / Try）

```
┌──────────────────┬──────────────────┬──────────────────┐
│      Keep        │     Problem      │       Try        │
│  （続けること）    │  （問題点）       │  （試すこと）     │
├──────────────────┼──────────────────┼──────────────────┤
│ ・朝会が15分以内  │ ・コードレビュー  │ ・レビューの      │
│   で終わっている  │   に2日かかる    │   SLA を24時間   │
│                  │                  │   に設定する      │
│ ・ペアプロが      │ ・テスト環境が    │ ・テスト環境を    │
│   効果的だった    │   不安定         │   刷新する        │
│                  │                  │   （チケット起票） │
│ ・PO との連携が   │ ・ドキュメントが  │ ・ADR（設計決定  │
│   スムーズ        │   更新されない   │   記録）を導入    │
└──────────────────┴──────────────────┴──────────────────┘
```

#### 2. Start / Stop / Continue

```
┌──────────────────┬──────────────────┬──────────────────┐
│     Start        │      Stop        │    Continue      │
│ （始めること）    │ （やめること）     │ （続けること）    │
├──────────────────┼──────────────────┼──────────────────┤
│ ・モブプロ週1回   │ ・全員参加の      │ ・デプロイ自動化  │
│ ・技術共有会      │   長時間会議     │ ・スプリント      │
│ ・自動テスト      │ ・手動デプロイ    │   レビューデモ   │
└──────────────────┴──────────────────┴──────────────────┘
```

#### 3. Fun / Done / Learn

```
          Fun（楽しかった）
           /          \
          /            \
  ┌──────┐    ┌──────┐
  │ Fun+ │    │ Fun+ │
  │ Done │    │Learn │
  └──────┘    └──────┘
        \      /
     ┌───────────┐
     │  Fun+Done │
     │  +Learn   │  ← 理想：楽しく、成果があり、学びがあった
     └───────────┘
        /      \
  ┌──────┐    ┌──────┐
  │ Done │    │Done+ │
  │ のみ  │    │Learn │
  └──────┘    └──────┘
          \            /
           \          /
         Done（達成した）  ─── Learn（学んだ）
```

### レトロスペクティブのベストプラクティス

```
効果的なレトロスペクティブのルール：

1. 安全な場を作る
   → 「失敗しても責めない」というグラウンドルールを設定する
   → ラスベガスルール（この場で話したことはこの場限り）

2. 全員が参加する
   → 発言の少ないメンバーにも意見を求める
   → 付箋に書く → グルーピング の手法が効果的

3. アクションアイテムを決める
   → 「次のスプリントで試すこと」を具体的に決める
   → 担当者と期限を明確にする

4. 前回のアクションを確認する
   → 前回のTryが実行されたかフォローアップする

5. マンネリ化を防ぐ
   → 手法を定期的に変える（KPT、Start/Stop/Continue、4Ls等）
```

> **現場でのポイント**: レトロスペクティブで最も重要なのは「安心して話せる場」を作ることです。問題を指摘した人が責められるような雰囲気では、本当の問題が表面化しません。

## 8.6 アジャイルとウォーターフォールのハイブリッド

### ハイブリッドアプローチとは

多くの現場では、純粋なアジャイルでも純粋なウォーターフォールでもなく、両者の良いところを組み合わせた**ハイブリッドアプローチ**が採用されています。

```
ハイブリッドの例（ウォーターアジャイル）：

要件定義（WF）→ 基本設計（WF）→ 開発・テスト（アジャイル）→ リリース（WF）

├── ウォーターフォール ──┤├── アジャイル ──┤├── WF ──┤

  ・全体計画はWFで策定     ・開発はスプリントで   ・リリースは
  ・要件はWF的に確定       ・デイリースクラム実施    計画的に実行
  ・マイルストーン設定      ・レトロスペクティブ
```

### どんなときにハイブリッドを使うか

| 状況 | ハイブリッドの形 |
|------|----------------|
| 官公庁・大企業の受託開発 | 契約はWF、開発はアジャイル |
| 基幹システム刷新 | 要件定義はWF、実装はアジャイル |
| 複数チームの大規模開発 | 全体計画はWF、チーム単位でスクラム |
| 規制産業（金融・医療） | ドキュメントはWF基準、開発プロセスはアジャイル |

## 8.7 スケーリングフレームワーク

大規模な組織でアジャイルを適用するためのフレームワークです。

### 主なスケーリングフレームワーク

| フレームワーク | 特徴 | チーム規模 |
|-------------|------|----------|
| **SAFe** (Scaled Agile Framework) | 最も包括的。ポートフォリオレベルまで対応 | 50〜数千名 |
| **LeSS** (Large-Scale Scrum) | スクラムをシンプルにスケーリング | 2〜8チーム |
| **Nexus** | スクラムの拡張。3〜9チーム向け | 3〜9チーム |
| **Spotify Model** | トライブ、スクワッド、チャプター、ギルドで構成 | 組織全体 |

### SAFeの概要

```
SAFeの4つのレベル：

ポートフォリオレベル
  └→ 企業全体の戦略とテーマを管理
      │
ラージソリューションレベル
  └→ 大規模なソリューションの調整
      │
プログラムレベル（ART: Agile Release Train）
  └→ 5〜12チームが連携してリリースを行う
      │
チームレベル
  └→ 個々のスクラムチームが開発を行う
```

> **現場でのポイント**: スケーリングフレームワークは万能ではありません。まずは1チームでスクラムを正しく実践し、その経験を基にスケーリングを検討しましょう。いきなりSAFeを導入すると、プロセスの複雑さに圧倒されるリスクがあります。

## ポイントまとめ

```
アジャイル開発の重要ポイント：

✅ アジャイルマニフェストの4つの価値観と12の原則を理解する
✅ スクラムの3ロール（PO、SM、開発チーム）の責務を明確にする
✅ 5つのイベント（プランニング、デイリー、レビュー、レトロ、リファインメント）
✅ カンバンはWIP制限で作業の流れを最適化する
✅ ストーリーポイントは相対見積もり。チーム間比較は無意味
✅ ベロシティはチームの能力の指標。計画の精度を上げるために使う
✅ レトロスペクティブは安全な場を作り、具体的なアクションに落とす
✅ 現場ではハイブリッドアプローチが多い。教条的にならない
✅ スケーリングは1チームの成功の後に検討する
```

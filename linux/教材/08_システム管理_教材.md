# 第8章：システム管理

## この章のゴール

- systemctl を使ったサービスの起動・停止・管理ができるようになること
- journalctl を使ったログの閲覧・検索ができるようになること
- cron を使ったタスクのスケジューリングができるようになること
- ログファイルの管理とローテーションの仕組みを理解すること
- ディスク管理の基本コマンド（df, du, mount）を使いこなせるようになること

## 8.1 systemctl - サービス管理

`systemctl` は、systemd が管理するサービス（デーモン）を制御するコマンドです。現代のほとんどのLinuxディストリビューションで使われています。

### systemd とは

```
┌─────────────────────────────────────────────────────┐
│                    systemd                            │
│                                                      │
│  Linux の init システム（PID 1 として動作）            │
│  サービスの起動・停止・依存関係の管理を行う             │
│                                                      │
│  管理対象（Unit）:                                     │
│    .service  → デーモン（nginx, sshd, mysqld等）      │
│    .timer    → タイマー（cron的な定期実行）            │
│    .socket   → ソケット                              │
│    .mount    → マウントポイント                       │
│    .target   → 複数のUnitをグループ化                 │
└─────────────────────────────────────────────────────┘
```

### サービスの基本操作

```bash
# サービスの状態確認
$ sudo systemctl status nginx
● nginx.service - A high performance web server
     Loaded: loaded (/lib/systemd/system/nginx.service; enabled; vendor preset: enabled)
     Active: active (running) since Wed 2024-03-20 10:00:00 JST; 5h ago
   Main PID: 900 (nginx)
      Tasks: 3 (limit: 4915)
     Memory: 8.5M
        CPU: 1.234s
     CGroup: /system.slice/nginx.service
             ├─900 "nginx: master process /usr/sbin/nginx"
             ├─901 "nginx: worker process"
             └─902 "nginx: worker process"

# サービスの起動
$ sudo systemctl start nginx

# サービスの停止
$ sudo systemctl stop nginx

# サービスの再起動
$ sudo systemctl restart nginx

# 設定のリロード（プロセスを停止せずに設定を再読み込み）
$ sudo systemctl reload nginx

# 自動起動の有効化（起動時に自動で開始される）
$ sudo systemctl enable nginx
Created symlink /etc/systemd/system/multi-user.target.wants/nginx.service → /lib/systemd/system/nginx.service.

# 自動起動の無効化
$ sudo systemctl disable nginx
Removed /etc/systemd/system/multi-user.target.wants/nginx.service.

# 有効化と同時に起動
$ sudo systemctl enable --now nginx
```

### status 出力の読み方

```
Active: active (running)    → 正常に動作中
Active: inactive (dead)     → 停止中
Active: failed              → 起動に失敗した
Active: activating (start)  → 起動処理中

Loaded: loaded (...; enabled; ...)   → 自動起動が有効
Loaded: loaded (...; disabled; ...)  → 自動起動が無効
```

### サービスの一覧表示

```bash
# 実行中のサービス一覧
$ systemctl list-units --type=service --state=running
UNIT                     LOAD   ACTIVE SUB     DESCRIPTION
cron.service             loaded active running Regular background program
nginx.service            loaded active running A high performance web server
sshd.service             loaded active running OpenBSD Secure Shell server

# 全サービスの状態を表示
$ systemctl list-units --type=service

# 自動起動が有効なサービスの一覧
$ systemctl list-unit-files --type=service --state=enabled
UNIT FILE                  STATE   ENABLED
cron.service               enabled enabled
nginx.service              enabled enabled
sshd.service               enabled enabled

# 起動に失敗したサービスを確認
$ systemctl --failed
```

### よく管理するサービス

| サービス名 | 説明 |
|-----------|------|
| `sshd.service` | SSHサーバー |
| `nginx.service` | Webサーバー（nginx） |
| `apache2.service` | Webサーバー（Apache） |
| `mysqld.service` | MySQL データベース |
| `postgresql.service` | PostgreSQL データベース |
| `cron.service` | タスクスケジューラ |
| `firewalld.service` | ファイアウォール |
| `NetworkManager.service` | ネットワーク管理 |

### ポイントまとめ

| 操作 | コマンド |
|------|---------|
| 状態確認 | `systemctl status サービス名` |
| 起動 | `systemctl start サービス名` |
| 停止 | `systemctl stop サービス名` |
| 再起動 | `systemctl restart サービス名` |
| 設定リロード | `systemctl reload サービス名` |
| 自動起動有効化 | `systemctl enable サービス名` |
| 自動起動無効化 | `systemctl disable サービス名` |
| 一覧表示 | `systemctl list-units --type=service` |
| 失敗確認 | `systemctl --failed` |

## 8.2 journalctl - ログの閲覧

`journalctl` は、systemd が収集したログ（ジャーナル）を閲覧するコマンドです。従来の `/var/log/syslog` に代わる統合的なログ管理の仕組みです。

### 基本的な使い方

```bash
# 全ログを表示（ページャーで表示）
$ journalctl

# 最新のログをリアルタイムで追跡（tail -f のような動き）
$ journalctl -f

# 最新N件のみ表示
$ journalctl -n 20

# ページャーなしで出力（パイプで処理するとき便利）
$ journalctl --no-pager -n 10
```

### 特定のサービスのログ

```bash
# nginx のログのみ表示
$ journalctl -u nginx.service

# nginx のログをリアルタイムで追跡
$ journalctl -u nginx.service -f

# 複数のサービスのログを同時に表示
$ journalctl -u nginx.service -u php-fpm.service
```

### 時間で絞り込み

```bash
# 今日のログのみ表示
$ journalctl --since today

# 特定の日時以降のログ
$ journalctl --since "2024-03-20 10:00:00"

# 期間を指定
$ journalctl --since "2024-03-20" --until "2024-03-21"

# 直近1時間のログ
$ journalctl --since "1 hour ago"

# 直近30分のログ
$ journalctl --since "30 min ago"
```

### 優先度（ログレベル）で絞り込み

```bash
# エラー以上のログのみ表示
$ journalctl -p err

# 警告以上のログのみ表示
$ journalctl -p warning
```

| 優先度 | 数値 | 意味 |
|--------|------|------|
| `emerg` | 0 | システム使用不可 |
| `alert` | 1 | 即時対応が必要 |
| `crit` | 2 | 重大な状態 |
| `err` | 3 | エラー |
| `warning` | 4 | 警告 |
| `notice` | 5 | 通知 |
| `info` | 6 | 情報 |
| `debug` | 7 | デバッグ |

### 出力フォーマットの変更

```bash
# JSON形式で出力（プログラムから処理する場合に便利）
$ journalctl -u nginx.service -o json-pretty -n 1

# 短縮形式
$ journalctl -o short-iso -n 5
2024-03-20T15:30:00+0900 myserver nginx[900]: access from 192.168.1.50
```

### カーネルログ

```bash
# カーネルログの表示（dmesg の代替）
$ journalctl -k

# 前回の起動時のログ
$ journalctl -b -1

# 起動一覧
$ journalctl --list-boots
-2 abc123... Wed 2024-03-18 10:00:00 JST
-1 def456... Thu 2024-03-19 10:00:00 JST
 0 ghi789... Fri 2024-03-20 10:00:00 JST
```

### ディスク使用量の確認と管理

```bash
# ジャーナルのディスク使用量を確認
$ journalctl --disk-usage
Archived and active journals take up 256.0M in the file system.

# 古いログを削除（サイズを制限）
$ sudo journalctl --vacuum-size=100M

# 古いログを削除（日数で制限）
$ sudo journalctl --vacuum-time=7d
```

### ポイントまとめ

- `journalctl -u サービス名` で特定サービスのログを閲覧
- `journalctl -f` でリアルタイム追跡
- `--since` / `--until` で時間フィルタリング
- `-p err` でエラー以上の重要なログだけ表示
- `--disk-usage` と `--vacuum-*` でログのディスク管理

## 8.3 cron - タスクのスケジューリング

`cron` は、指定した時間に自動的にコマンドやスクリプトを実行するスケジューラです。定期的なバックアップ、ログの整理、監視スクリプトの実行などに使います。

### crontab の基本構文

```
┌───────────── 分（0-59）
│ ┌───────────── 時（0-23）
│ │ ┌───────────── 日（1-31）
│ │ │ ┌───────────── 月（1-12）
│ │ │ │ ┌───────────── 曜日（0-7、0と7は日曜）
│ │ │ │ │
* * * * * コマンド
```

### crontab の操作

```bash
# 現在のcrontabを表示
$ crontab -l

# crontabを編集
$ crontab -e

# crontabを削除（注意！）
$ crontab -r

# 特定のユーザーのcrontabを表示（root権限が必要）
$ sudo crontab -u tanaka -l
```

### スケジュールの設定例

```bash
# 毎分実行
* * * * * /home/tanaka/scripts/check.sh

# 毎時0分に実行
0 * * * * /home/tanaka/scripts/hourly.sh

# 毎日午前3時に実行
0 3 * * * /home/tanaka/scripts/daily_backup.sh

# 毎週月曜の午前9時に実行
0 9 * * 1 /home/tanaka/scripts/weekly_report.sh

# 毎月1日の午前0時に実行
0 0 1 * * /home/tanaka/scripts/monthly.sh

# 平日（月〜金）の午前8時30分に実行
30 8 * * 1-5 /home/tanaka/scripts/weekday.sh

# 5分ごとに実行
*/5 * * * * /home/tanaka/scripts/monitor.sh

# 毎日午前2時と午後2時に実行
0 2,14 * * * /home/tanaka/scripts/twice_daily.sh
```

### 特殊な記法

| 記法 | 意味 | 等価な設定 |
|------|------|-----------|
| `@reboot` | 起動時に1回 | ― |
| `@yearly` | 年に1回 | `0 0 1 1 *` |
| `@monthly` | 月に1回 | `0 0 1 * *` |
| `@weekly` | 週に1回 | `0 0 * * 0` |
| `@daily` | 毎日1回 | `0 0 * * *` |
| `@hourly` | 毎時1回 | `0 * * * *` |

```bash
# 起動時にスクリプトを実行
@reboot /home/tanaka/scripts/startup.sh

# 毎日実行
@daily /home/tanaka/scripts/cleanup.sh
```

### cron で注意すべきこと

```bash
# 1. 環境変数が限定的（PATHが通常のシェルと異なる）
# → コマンドはフルパスで書く
0 3 * * * /usr/bin/python3 /home/tanaka/scripts/backup.py

# 2. 出力を適切にリダイレクトする
# → 出力を残さないと問題発生時に原因がわからない
0 3 * * * /home/tanaka/scripts/backup.sh >> /var/log/backup.log 2>&1

# 3. 実行権限を忘れない
$ chmod +x /home/tanaka/scripts/backup.sh

# 4. 正しいcrontabの記述例（ヘッダーに環境変数を設定）
SHELL=/bin/bash
PATH=/usr/local/sbin:/usr/local/bin:/usr/sbin:/usr/bin:/sbin:/bin
MAILTO=admin@example.com

# 毎日3時にバックアップ
0 3 * * * /home/tanaka/scripts/backup.sh >> /var/log/backup.log 2>&1
```

### cron のログ確認

```bash
# cron の実行ログを確認
$ grep CRON /var/log/syslog
Mar 20 03:00:01 myserver CRON[12345]: (tanaka) CMD (/home/tanaka/scripts/backup.sh)

# journalctl で確認
$ journalctl -u cron.service --since today
```

### ポイントまとめ

- `crontab -e` で編集、`crontab -l` で確認
- スケジュール設定は「分 時 日 月 曜日」の5フィールド
- コマンドはフルパスで記述する
- 出力は必ずログファイルにリダイレクトする
- `@daily` などの特殊記法も活用できる

## 8.4 ログ管理

### 主要なログファイル

Linuxシステムのログは `/var/log/` ディレクトリに格納されます。

| ファイル | 内容 |
|---------|------|
| `/var/log/syslog` | システム全般のログ（Debian系） |
| `/var/log/messages` | システム全般のログ（RHEL系） |
| `/var/log/auth.log` | 認証関連のログ（ログイン、sudo等） |
| `/var/log/kern.log` | カーネルのログ |
| `/var/log/dmesg` | 起動時のカーネルメッセージ |
| `/var/log/nginx/access.log` | nginx のアクセスログ |
| `/var/log/nginx/error.log` | nginx のエラーログ |
| `/var/log/apt/history.log` | パッケージの操作履歴 |
| `/var/log/cron.log` | cron の実行ログ |

### ログの閲覧方法

```bash
# ログファイルの末尾を確認
$ tail -20 /var/log/syslog

# リアルタイムで監視
$ tail -f /var/log/syslog

# エラーを検索
$ grep -i "error" /var/log/syslog

# 特定の日時のログを検索
$ grep "Mar 20 15:" /var/log/syslog

# 認証失敗のログを確認
$ grep "Failed password" /var/log/auth.log
Mar 20 10:15:23 myserver sshd[12345]: Failed password for root from 203.0.113.50 port 54321 ssh2
```

### logrotate - ログのローテーション

`logrotate` は、ログファイルが大きくなりすぎないように、定期的にローテーション（古いログをアーカイブ・削除）する仕組みです。

```bash
# logrotate の設定ファイル
$ cat /etc/logrotate.conf
# デフォルト設定
weekly          # 毎週ローテーション
rotate 4        # 4世代分保持
create          # ローテーション後に新しいファイルを作成
compress        # 古いログをgzip圧縮

# 個別サービスの設定例
$ cat /etc/logrotate.d/nginx
/var/log/nginx/*.log {
    daily               # 毎日ローテーション
    missingok           # ファイルがなくてもエラーにしない
    rotate 14           # 14世代分保持
    compress            # gzip圧縮
    delaycompress       # 1世代目は圧縮しない
    notifempty          # 空のログはローテーションしない
    create 0640 www-data adm    # 新しいファイルのパーミッション
    postrotate
        /usr/sbin/nginx -s reopen    # ローテーション後にnginxにシグナル送信
    endscript
}
```

```bash
# logrotate の手動実行（テスト）
$ sudo logrotate -d /etc/logrotate.conf    # ドライラン（実際には実行しない）
$ sudo logrotate -f /etc/logrotate.conf    # 強制実行
```

### ポイントまとめ

- ログファイルは `/var/log/` に集約される
- `tail -f` でリアルタイム監視、`grep` で検索
- `logrotate` でログが肥大化しないように管理する
- 重要なログは `auth.log`（認証）、`syslog`（全般）

## 8.5 ディスク管理

### df - ディスク使用量の確認

```bash
# ディスク使用量を人が読める形式で表示
$ df -h
Filesystem      Size  Used Avail Use% Mounted on
/dev/sda1        50G   23G   25G  48% /
/dev/sda2       200G  156G   34G  82% /home
tmpfs           3.9G  1.2M  3.9G   1% /tmp

# 特定のファイルシステムのみ表示
$ df -h /home
Filesystem      Size  Used Avail Use% Mounted on
/dev/sda2       200G  156G   34G  82% /home

# inode の使用状況を確認
$ df -i
Filesystem      Inodes  IUsed   IFree IUse% Mounted on
/dev/sda1      3276800 234567 3042233    8% /
```

### 出力の読み方

```
Filesystem      Size  Used Avail Use% Mounted on
│               │     │    │     │    │
│               │     │    │     │    └─ マウントポイント
│               │     │    │     └─ 使用率
│               │     │    └─ 空き容量
│               │     └─ 使用済み容量
│               └─ 総容量
└─ ファイルシステム（パーティション）
```

### du - ディレクトリサイズの確認

```bash
# 指定ディレクトリのサイズを表示
$ du -sh /var/log
256M    /var/log

# サブディレクトリごとのサイズを表示
$ du -sh /var/log/*
4.0K    /var/log/alternatives.log
12K     /var/log/apt
128M    /var/log/journal
64M     /var/log/nginx
32M     /var/log/syslog

# サイズ順にソートしてトップ10を表示
$ du -sh /var/* 2>/dev/null | sort -rh | head -10
512M    /var/log
256M    /var/cache
128M    /var/lib
 32M    /var/backups
```

### 主要なオプション

| オプション | df | du | 説明 |
|-----------|----|----|------|
| `-h` | ○ | ○ | 人が読みやすい形式（K, M, G） |
| `-s` | ― | ○ | サマリー（合計のみ） |
| `-a` | ○ | ○ | すべてのファイルシステム/ファイルを表示 |
| `-i` | ○ | ― | inode情報を表示 |

### mount / umount - ファイルシステムのマウント

```bash
# マウントされているファイルシステムの一覧
$ mount
/dev/sda1 on / type ext4 (rw,relatime)
/dev/sda2 on /home type ext4 (rw,relatime)
tmpfs on /tmp type tmpfs (rw,nosuid,nodev)

# USBドライブをマウント
$ sudo mount /dev/sdb1 /mnt/usb

# アンマウント
$ sudo umount /mnt/usb

# /etc/fstab の確認（起動時の自動マウント設定）
$ cat /etc/fstab
# <file system>  <mount point>  <type>  <options>       <dump>  <pass>
/dev/sda1        /              ext4    defaults        0       1
/dev/sda2        /home          ext4    defaults        0       2
```

### lsblk - ブロックデバイスの一覧

```bash
$ lsblk
NAME   MAJ:MIN RM   SIZE RO TYPE MOUNTPOINTS
sda      8:0    0   250G  0 disk
├─sda1   8:1    0    50G  0 part /
├─sda2   8:2    0   200G  0 part /home
sdb      8:16   0    16G  0 disk
└─sdb1   8:17   0    16G  0 part /mnt/usb
```

### ポイントまとめ

- `df -h` でファイルシステム全体の使用率を確認
- `du -sh` でディレクトリのサイズを確認
- `mount` / `umount` でファイルシステムのマウント操作
- `lsblk` でブロックデバイスの構成を確認
- ディスク使用率80%を超えたら対策が必要

## 8.6 メモリとCPUの確認

### free - メモリ使用量

```bash
$ free -h
               total        used        free      shared  buff/cache   available
Mem:           7.8Gi       2.1Gi       1.2Gi       256Mi       4.5Gi       5.2Gi
Swap:          2.0Gi          0B       2.0Gi
```

各フィールドの意味：

```
total:      物理メモリの総容量
used:       使用中のメモリ
free:       完全に未使用のメモリ
shared:     共有メモリ
buff/cache: バッファとキャッシュ（必要に応じて解放される）
available:  新しいプロセスが使用可能なメモリ（実質的な空き容量）
```

> **重要**: Linux はメモリを積極的にキャッシュに使うため、`free` が少なくても `available` が十分あれば問題ありません。`available` の値を確認してください。

### uptime - 稼働時間と負荷

```bash
$ uptime
 15:30:00 up 45 days,  3:22,  2 users,  load average: 0.25, 0.18, 0.15
                                         │             │     │     │
                                         │             │     │     └─ 15分平均
                                         │             │     └─ 5分平均
                                         │             └─ 1分平均
                                         └─ ロードアベレージ
```

ロードアベレージの目安：
- CPUコア数以下 : 正常
- CPUコア数の2倍以上 : 高負荷（要調査）

```bash
# CPUコア数の確認
$ nproc
4

# CPU情報の確認
$ lscpu | grep "CPU(s):"
CPU(s):                          4
```

### ポイントまとめ

- `free -h` でメモリ使用量を確認（`available` が重要）
- `uptime` でロードアベレージを確認
- ロードアベレージはCPUコア数と比較して判断する

## 8.7 パッケージ管理

### apt（Debian / Ubuntu系）

```bash
# パッケージリストの更新
$ sudo apt update

# パッケージのアップグレード
$ sudo apt upgrade

# パッケージのインストール
$ sudo apt install nginx

# パッケージの削除
$ sudo apt remove nginx

# 設定ファイルも含めて完全削除
$ sudo apt purge nginx

# パッケージの検索
$ apt search nginx

# パッケージの情報表示
$ apt show nginx

# 不要なパッケージの自動削除
$ sudo apt autoremove
```

### yum / dnf（RHEL / CentOS系）

```bash
# パッケージのインストール
$ sudo dnf install nginx

# パッケージのアップデート
$ sudo dnf update

# パッケージの削除
$ sudo dnf remove nginx

# パッケージの検索
$ dnf search nginx

# パッケージの情報表示
$ dnf info nginx
```

### ポイントまとめ

- Debian系は `apt`、RHEL系は `dnf`（旧 `yum`）
- `apt update` → `apt upgrade` の順でシステムを更新
- インストール前に `apt search` で正確なパッケージ名を確認

## 8.8 よくある間違い

### 間違い1: cron で相対パスを使う

```bash
# 【誤り】cron の実行環境では PATH が限定的
0 3 * * * backup.sh

# 【正解】フルパスで指定する
0 3 * * * /home/tanaka/scripts/backup.sh
```

### 間違い2: cron の出力をリダイレクトしない

```bash
# 【誤り】出力がメールで送信される（大量のメールが溜まる原因）
*/5 * * * * /home/tanaka/scripts/check.sh

# 【正解】ログファイルにリダイレクトする
*/5 * * * * /home/tanaka/scripts/check.sh >> /var/log/check.log 2>&1
```

### 間違い3: df の出力で free だけ見る

```bash
# 【誤り】free が少なくてもキャッシュが解放されるので問題ない場合がある

# 【正解】Use% を確認する
$ df -h /
Filesystem      Size  Used Avail Use% Mounted on
/dev/sda1        50G   23G   25G  48% /      # ← Use% が80%超えたら要注意
```

## 8.9 現場での運用Tips

### 1. サーバー起動後のチェックリスト

```bash
# 1. サービスの状態確認
$ systemctl --failed

# 2. ディスク使用率の確認
$ df -h

# 3. メモリの確認
$ free -h

# 4. ロードアベレージの確認
$ uptime

# 5. ログのエラー確認
$ journalctl -p err --since "1 hour ago"
```

### 2. 障害調査の手順

```bash
# Step 1: サービスの状態を確認
$ sudo systemctl status nginx

# Step 2: ログを確認
$ journalctl -u nginx.service --since "10 min ago" --no-pager

# Step 3: ポートの確認
$ sudo ss -tlnp | grep :80

# Step 4: ディスクとメモリの確認
$ df -h && free -h

# Step 5: プロセスの確認
$ ps aux | grep nginx
```

## 5.1 リスク管理の基本

### リスクとは

リスクとは、**プロジェクトの目標に対してプラスまたはマイナスの影響を与える不確実な事象**のことです。多くの人はリスクを「悪いこと」だけと考えがちですが、PMBOKではプラスのリスク（好機：Opportunity）とマイナスのリスク（脅威：Threat）の両方を管理対象としています。

```
リスクの分類：

マイナスのリスク（脅威）        プラスのリスク（好機）
├── 主要メンバーの離脱          ├── 新技術で工数削減
├── 要件の大幅変更              ├── 類似案件のコード再利用
├── 外部API提供遅延             ├── チームの習熟による生産性向上
├── セキュリティ脆弱性の発覚    └── 先行リリースによる早期フィードバック
└── 性能要件の未達
```

### リスクと課題の違い

| 項目 | リスク（Risk） | 課題（Issue） |
|------|---------------|--------------|
| 時制 | 将来起きるかもしれない | 既に起きている |
| 対応 | 事前に予防策・対応策を準備 | 即座に解決が必要 |
| 管理ツール | リスク登録簿 | 課題管理表 |
| 例 | 「メンバーが退職するかもしれない」 | 「メンバーが退職した」 |

> **現場でのポイント**: リスクが顕在化すると「課題」に変わります。リスク管理の目的は、できるだけ多くのリスクを課題になる前に対処することです。

### リスク管理の6つのプロセス

```
1. リスク管理計画策定
   → どのようにリスク管理を行うかを決める
   ↓
2. リスクの特定
   → どんなリスクがあるかを洗い出す
   ↓
3. リスクの定性的分析
   → リスクの優先順位をつける
   ↓
4. リスクの定量的分析
   → リスクの影響を数値化する（必要に応じて）
   ↓
5. リスク対応策の計画
   → 各リスクへの対応方法を決める
   ↓
6. リスクの監視
   → 継続的にリスクを監視し、新たなリスクを特定する
```

## 5.2 リスクの特定

### 5.2.1 ブレインストーミング

チーム全体でリスクを自由に洗い出す手法です。最も一般的で、多様な視点からリスクを特定できます。

```
ブレインストーミングのルール：
  1. 批判しない（どんな意見も歓迎）
  2. 自由奔放に（突飛な意見もOK）
  3. 質より量（まずは数を出す）
  4. 結合・改善OK（他の人のアイデアに乗る）

進め方：
  ・時間を区切る（15〜30分）
  ・カテゴリ別に整理する
  ・全員が発言できるよう配慮する
  ・付箋やホワイトボードを活用する
```

### 5.2.2 チェックリスト法

過去のプロジェクトの経験や業界の知見をチェックリスト化し、漏れなくリスクを特定する手法です。

**ソフトウェア開発プロジェクトのリスクチェックリスト：**

| カテゴリ | チェック項目 | リスクの例 |
|---------|-------------|-----------|
| **技術** | 未経験の技術を使うか？ | 学習コスト増、実装遅延 |
| **技術** | 外部APIに依存するか？ | API仕様変更、提供遅延 |
| **技術** | 性能要件は明確か？ | 性能チューニングの追加工数 |
| **要件** | 要件は確定しているか？ | スコープクリープ |
| **要件** | ステークホルダーの合意は取れているか？ | 仕様の手戻り |
| **人員** | 必要なスキルを持つメンバーがいるか？ | 採用・教育コスト |
| **人員** | キーパーソンの離脱リスクはないか？ | 知識の喪失、遅延 |
| **外部** | 外部ベンダーの品質は信頼できるか？ | 成果物の品質不足 |
| **外部** | 法規制の変更はあり得るか？ | 追加対応コスト |
| **組織** | 経営層のコミットメントはあるか？ | プロジェクト中止 |
| **組織** | 他プロジェクトとのリソース競合はないか？ | リソース不足 |

### 5.2.3 SWOT分析

組織やプロジェクトの内部要因と外部要因を分析し、リスクと機会を特定します。

```
┌─────────────────────┬─────────────────────┐
│    強み (Strengths)  │   弱み (Weaknesses) │
│                      │                      │
│  ・経験豊富なチーム   │  ・フロントエンドの   │
│  ・類似案件の実績     │    経験が浅い         │
│  ・CI/CD基盤が整備   │  ・ドキュメント不足   │
│    済み              │  ・テスト自動化が     │
│                      │    未整備             │
├─────────────────────┼─────────────────────┤
│   機会 (Opportunities)│   脅威 (Threats)     │
│                      │                      │
│  ・新技術で差別化可能 │  ・競合他社の先行     │
│  ・市場ニーズの拡大   │  ・法規制の変更       │
│  ・パートナー企業     │  ・為替変動による     │
│    との連携           │    コスト増           │
└─────────────────────┴─────────────────────┘
       内部要因                外部要因
```

### 5.2.4 前提条件分析

プロジェクト計画の前提条件を列挙し、それが崩れた場合のリスクを特定します。

```
前提条件とリスクの対応：

前提条件                          崩れた場合のリスク
───────────────────────────────  ────────────────────────
クラウドサービスの稼働率99.9%    → サービス停止で開発遅延
チームメンバー5名が全期間参画    → 離脱による工数不足
外部APIが4月末に提供される      → 提供遅延で結合テスト遅延
要件は初期フェーズで確定する     → 要件変更によるスコープクリープ
```

## 5.3 リスクの定性的分析

### 確率・影響度マトリクス

リスクの**発生確率**と**影響度**を評価し、優先順位をつけます。

**確率の基準：**

| レベル | 確率 | 説明 |
|--------|------|------|
| 極低 | 1〜10% | ほとんど起きない |
| 低 | 11〜30% | 起きにくい |
| 中 | 31〜50% | 起きる可能性がある |
| 高 | 51〜70% | 起きやすい |
| 極高 | 71〜90% | ほぼ確実に起きる |

**影響度の基準：**

| レベル | スケジュール影響 | コスト影響 | 品質影響 |
|--------|-----------------|-----------|---------|
| 極低 | 1日以内の遅延 | 1%未満の増加 | ほぼ影響なし |
| 低 | 1〜5日の遅延 | 1〜5%の増加 | 軽微な品質低下 |
| 中 | 1〜2週間の遅延 | 5〜10%の増加 | 一部機能に影響 |
| 高 | 2〜4週間の遅延 | 10〜20%の増加 | 主要機能に影響 |
| 極高 | 1ヶ月以上の遅延 | 20%以上の増加 | プロダクト全体に影響 |

### 確率・影響度マトリクスの例

```
              影響度
         極低   低    中    高   極高
        ┌─────┬─────┬─────┬─────┬─────┐
極高    │  中  │  高  │ 極高│ 極高│ 極高│
        ├─────┼─────┼─────┼─────┼─────┤
高      │  低  │  中  │  高 │ 極高│ 極高│
確      ├─────┼─────┼─────┼─────┼─────┤
率  中  │  低  │  中  │  中 │  高 │ 極高│
        ├─────┼─────┼─────┼─────┼─────┤
低      │ 極低 │  低  │  中 │  中 │  高 │
        ├─────┼─────┼─────┼─────┼─────┤
極低    │ 極低 │ 極低 │  低 │  低 │  中 │
        └─────┴─────┴─────┴─────┴─────┘
```

### 分析の具体例

```
リスク：主要なバックエンドエンジニアが退職する可能性がある

確率の評価：
  ・本人から転職活動の兆候がある → 「高」（60%）

影響度の評価：
  ・当該エンジニアは核心的なアーキテクチャを設計した
  ・代替要員のキャッチアップに1ヶ月以上かかる
  → 「高」（3〜4週間の遅延、15%のコスト増）

リスクスコア：確率「高」× 影響度「高」= 極高
→ 最優先で対応策を講じる必要がある
```

## 5.4 リスクの定量的分析

### 期待金額価値（EMV: Expected Monetary Value）

リスクの発生確率と金額的影響を掛け合わせて、リスクの金銭的な期待値を算出します。

```
EMV = 発生確率 × 金額的影響

例：3つのリスクのEMV計算

リスクA: メンバー離脱
  確率: 40%、影響: -500万円（追加採用・教育コスト）
  EMV = 0.4 × (-500万) = -200万円

リスクB: 外部API遅延
  確率: 30%、影響: -300万円（待機コスト・回避策開発）
  EMV = 0.3 × (-300万) = -90万円

リスクC: 技術検証で予想以上の成果（好機）
  確率: 25%、影響: +200万円（工数削減）
  EMV = 0.25 × (+200万) = +50万円

合計EMV = -200 + (-90) + 50 = -240万円
→ リスクに備えて少なくとも240万円の予備費が必要
```

### デシジョンツリー分析

複数の選択肢がある場合に、各選択肢のEMVを比較して意思決定を行います。

```
選択肢A: 自社開発
  ├── 成功（70%）: +1,000万円
  └── 失敗（30%）: -500万円
  EMV = 0.7×1,000 + 0.3×(-500) = 700 - 150 = +550万円

選択肢B: 外部委託
  ├── 高品質（60%）: +800万円
  └── 低品質（40%）: +200万円
  EMV = 0.6×800 + 0.4×200 = 480 + 80 = +560万円

→ EMVはほぼ同等だが、選択肢Aのほうがばらつきが大きい
→ リスク許容度に応じて判断する
```

### モンテカルロシミュレーション（概念）

各タスクの見積もりに確率分布を設定し、何千回もシミュレーションを実行して、プロジェクト全体の完了日やコストの確率分布を求める手法です。

```
シミュレーション結果の例：

プロジェクト完了コスト:
  最小: 2,400万円
  最頻: 2,800万円
  最大: 3,500万円

確率分布:
  2,600万円以下で完了する確率: 20%
  2,800万円以下で完了する確率: 50%
  3,000万円以下で完了する確率: 75%
  3,200万円以下で完了する確率: 90%

→ 90%の信頼度で完了させたいなら、3,200万円の予算が必要
```

> **現場でのベストプラクティス**: 中小規模のプロジェクトでは、定性的分析とEMV計算で十分なことが多いです。モンテカルロシミュレーションは大規模プロジェクトや高い精度が求められる場合に検討しましょう。

## 5.5 リスク対応戦略

### マイナスのリスク（脅威）への対応

| 戦略 | 説明 | 具体例 |
|------|------|--------|
| **回避（Avoid）** | リスクの原因を排除する | 不安定な新技術の採用を見送る |
| **軽減（Mitigate）** | 確率や影響度を下げる | PoC（概念実証）を先行実施する |
| **転嫁（Transfer）** | リスクを第三者に移す | 保険に加入する、外注する |
| **受容（Accept）** | リスクを受け入れる | 発生時に対応する（予備費で対応） |
| **エスカレーション** | 上位の管理層に対応を委ねる | 組織レベルのリスクを経営層に報告 |

### プラスのリスク（好機）への対応

| 戦略 | 説明 | 具体例 |
|------|------|--------|
| **活用（Exploit）** | 好機を確実に実現する | 優秀なメンバーを確保して品質向上 |
| **強化（Enhance）** | 好機の確率や影響を高める | 早期に技術検証を行い効果を確認 |
| **共有（Share）** | 第三者と好機を分かち合う | パートナー企業と共同開発 |
| **受容（Accept）** | 好機が来たら利用する | 特に対策は打たず、実現を待つ |

### 対応策の選定基準

```
リスク対応策を選ぶ際の判断基準：

1. コスト効果
   対応策のコスト < リスクのEMV であるか？

2. 実現可能性
   対応策を実行するリソースがあるか？

3. 副作用
   対応策が新たなリスクを生まないか？

4. タイミング
   対応策は間に合うか？

5. ステークホルダーの合意
   対応策はステークホルダーに受け入れられるか？
```

### 具体例：リスク対応策の検討

```
リスク：本番環境でデータベースの性能が不足する可能性がある

回避策：
  → 性能要件が不確実な機能を削除する（スコープを縮小）
  → コスト：機能が失われる

軽減策（選択）：
  → 早期に負荷テスト環境を構築し、性能検証を行う
  → コスト：50万円（環境構築 + テスト工数）
  → 効果：問題の早期発見で手戻りコスト削減

転嫁策：
  → マネージドDBサービスを利用し、スケーリングをクラウドベンダーに委ねる
  → コスト：月額10万円の追加（オンプレ比較）

受容策：
  → 性能問題が発生したら対処する（予備費から拠出）
  → コスト：0円（事前）、最大300万円（事後）

→ 軽減策を採用：50万円のコストで最大300万円のリスクを軽減できる
```

## 5.6 リスク登録簿

### リスク登録簿とは

リスク登録簿（Risk Register）は、特定されたリスク、分析結果、対応策を一元的に管理するドキュメントです。プロジェクト期間を通じて継続的に更新されます。

### リスク登録簿のテンプレート

```
┌────┬──────────┬──────┬────┬────┬──────┬──────────┬────────┬──────┬──────┐
│ ID │ リスク名  │カテゴリ│確率│影響│スコア│ 対応戦略  │対応策   │担当者│状態  │
├────┼──────────┼──────┼────┼────┼──────┼──────────┼────────┼──────┼──────┤
│R-01│メンバー  │ 人員  │ 高 │ 高 │ 極高 │ 軽減     │ ナレッジ│鈴木  │監視中│
│    │離脱      │      │    │    │      │          │共有推進 │      │      │
├────┼──────────┼──────┼────┼────┼──────┼──────────┼────────┼──────┼──────┤
│R-02│外部API   │ 技術  │ 中 │ 高 │  高  │ 軽減     │モック   │田中  │監視中│
│    │提供遅延  │      │    │    │      │          │API作成 │      │      │
├────┼──────────┼──────┼────┼────┼──────┼──────────┼────────┼──────┼──────┤
│R-03│性能要件  │ 技術  │ 中 │ 中 │  中  │ 軽減     │早期負荷 │佐藤  │対応中│
│    │未達      │      │    │    │      │          │テスト  │      │      │
├────┼──────────┼──────┼────┼────┼──────┼──────────┼────────┼──────┼──────┤
│R-04│要件の    │ 要件  │ 高 │ 中 │  高  │ 回避     │要件凍結 │山田  │監視中│
│    │追加変更  │      │    │    │      │          │ルール  │      │      │
├────┼──────────┼──────┼────┼────┼──────┼──────────┼────────┼──────┼──────┤
│R-05│クラウド  │外部   │ 低 │ 高 │  中  │ 転嫁     │マルチ  │伊藤  │監視中│
│    │障害      │      │    │    │      │          │AZ構成  │      │      │
├────┼──────────┼──────┼────┼────┼──────┼──────────┼────────┼──────┼──────┤
│R-06│セキュリティ│ 技術 │ 低 │極高│  高  │ 軽減     │脆弱性  │佐藤  │監視中│
│    │脆弱性    │      │    │    │      │          │スキャン│      │      │
└────┴──────────┴──────┴────┴────┴──────┴──────────┴────────┴──────┴──────┘
```

### リスク登録簿の運用ルール

```
更新頻度と方法：

  ・週次：リスクオーナーがステータスを更新
  ・隔週：チーム全体でリスクレビュー会議を実施
  ・都度：新たなリスクが特定されたら随時追加
  ・月次：リスク対応策の有効性を評価

ステータスの種類：
  ・未対応：対応策が未定
  ・対応中：対応策を実行中
  ・監視中：対応策を講じ、発生を監視中
  ・顕在化：リスクが現実になった（課題に移行）
  ・終結：リスクが消滅、または対応完了
```

## 5.7 ソフトウェア開発特有のリスク

### 技術リスク

```
よくある技術リスクと対応策：

リスク：新しいフレームワークの採用による不確実性
対応：PoC（Proof of Concept）を初期フェーズで実施

リスク：マイクロサービス間の整合性が取れない
対応：API契約テスト（Contract Test）の導入

リスク：レガシーシステムとの連携で想定外の制約
対応：早期にインテグレーションテストを実施

リスク：スケーラビリティ要件の未達
対応：負荷テストを継続的に実施（CI/CDに組み込む）
```

### 人的リスク

```
よくある人的リスクと対応策：

リスク：バス係数（Bus Factor）が1のメンバーがいる
  → 特定の1人しか理解していない領域がある
対応：
  ・ペアプログラミング、モブプログラミング
  ・ドキュメンテーション強化
  ・コードレビューの徹底

リスク：チーム内のスキルギャップ
対応：
  ・技術勉強会の定期開催
  ・メンタリング制度の導入
  ・段階的にタスクの難易度を上げる
```

### プロセスリスク

```
よくあるプロセスリスクと対応策：

リスク：スコープクリープ（範囲の肥大化）
対応：変更管理プロセスの厳格化、変更要求書の必須化

リスク：コミュニケーション不足による認識のズレ
対応：定例会議、デモの実施、チャットツールの活用

リスク：テスト不足によるリリース品質の低下
対応：テスト自動化、品質ゲートの設定
```

---

## 5.8 リスクの継続的監視

### リスク監視のポイント

```
監視すべき指標：

1. リスクトリガー
   → リスクが顕在化する前兆となる指標
   例：メンバーの残業時間が増えている → バーンアウトリスク

2. 残存リスク
   → 対応策を実施した後も残るリスク
   例：軽減策を講じたが、完全には排除できないリスク

3. 二次リスク
   → 対応策を実施したことで新たに生じるリスク
   例：外注に切り替えたことで、コミュニケーションコストが増加

4. リスクの再評価
   → 確率や影響度が変化していないか定期的に見直す
```

### リスクバーンダウンチャート

プロジェクト全体のリスク露出度（総EMV）の推移を可視化します。

```
リスク露出度
(万円)
 600│ ★
    │  ＼
 500│   ＼
    │    ＼★
 400│      ＼
    │       ＼
 300│        ＼★
    │         ╲
 200│          ╲★
    │           ╲
 100│            ╲★
    │             ╲
   0│──────────────★─→ 時間
    M1  M2  M3  M4  M5  M6

→ プロジェクトの進行に伴い、リスク露出度は減少していくべき
→ 途中で増加する場合は、新たなリスクの発生や対応の不足を示す
```

> **よくある間違い**: プロジェクト開始時にリスク登録簿を作成して、その後更新しないこと。リスク管理は継続的なプロセスです。新しいリスクは常に出現し、既存のリスクの確率や影響度は変化します。

## まとめ

| 概念 | ポイント |
|------|---------|
| リスクの定義 | 目標に影響を与える不確実な事象（脅威と好機の両方） |
| リスク特定 | ブレインストーミング、チェックリスト、SWOT等で網羅的に |
| 定性的分析 | 確率×影響度でリスクの優先順位をつける |
| 定量的分析 | EMV計算で金銭的な影響を定量化する |
| 対応戦略（脅威） | 回避・軽減・転嫁・受容から選択 |
| 対応戦略（好機） | 活用・強化・共有・受容から選択 |
| リスク登録簿 | リスク情報を一元管理し、継続的に更新する |
| 継続的監視 | トリガー、残存リスク、二次リスクを定期的にレビュー |

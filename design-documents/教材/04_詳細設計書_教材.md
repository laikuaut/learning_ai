# 第4章：詳細設計書（内部設計）

## この章のゴール

- 詳細設計書の目的と基本設計書との違いを理解する
- クラス図の読み方・書き方を習得する
- シーケンス図の作成方法を学ぶ
- 処理フロー（フローチャート）の書き方を把握する
- モジュール分割と設計原則を理解する

---

## 4.1 詳細設計書とは

### 目的

詳細設計書（内部設計書）は、**各モジュールの内部処理ロジックを開発者が実装できるレベルで定義するドキュメント**です。

### 基本設計書との違い

| 項目 | 基本設計書 | 詳細設計書 |
|------|-----------|-----------|
| **視点** | ユーザー・システム全体 | 開発者・モジュール内部 |
| **粒度** | 機能単位 | クラス・メソッド単位 |
| **読者** | 顧客・開発チーム全体 | 実装担当の開発者 |
| **内容** | 画面・API・データの概要 | 処理ロジック・アルゴリズム |
| **図表** | システム構成図、画面遷移図 | クラス図、シーケンス図 |

```
基本設計書                    詳細設計書
「注文機能がある」      →   「OrderServiceクラスの
                            createOrderメソッドが
                            在庫確認→決済→注文確定を
                            順に実行する」
```

### 詳細設計書の構成

```markdown
1. モジュール構成
   1.1 モジュール一覧
   1.2 モジュール関連図
2. クラス設計
   2.1 クラス図
   2.2 各クラスの詳細
3. 処理設計
   3.1 シーケンス図
   3.2 処理フロー
   3.3 状態遷移図
4. エラーハンドリング設計
5. ログ設計
```

## 4.2 モジュール構成

### レイヤードアーキテクチャの例

典型的なWebアプリケーションのモジュール構成を示します。

```
┌─────────────────────────────────────────────┐
│              プレゼンテーション層              │
│  Controller / Router / Middleware            │
│  リクエストの受付、レスポンスの返却            │
├─────────────────────────────────────────────┤
│              アプリケーション層               │
│  Service / UseCase                          │
│  ビジネスロジックの実行、トランザクション管理    │
├─────────────────────────────────────────────┤
│                ドメイン層                     │
│  Entity / ValueObject / DomainService       │
│  ビジネスルール、ドメインロジック              │
├─────────────────────────────────────────────┤
│             インフラストラクチャ層             │
│  Repository / External API Client           │
│  データの永続化、外部サービスとの通信           │
└─────────────────────────────────────────────┘
```

### モジュール一覧の例

| モジュール名 | レイヤー | 責務 | 依存先 |
|------------|---------|------|--------|
| OrderController | プレゼンテーション | 注文APIのエンドポイント | OrderService |
| OrderService | アプリケーション | 注文処理のビジネスロジック | OrderRepository, PaymentService |
| Order | ドメイン | 注文エンティティ | OrderItem |
| OrderItem | ドメイン | 注文明細エンティティ | Product |
| OrderRepository | インフラ | 注文データの永続化 | Database |
| PaymentService | インフラ | 決済処理の外部連携 | Stripe API |

## 4.3 クラス図

### クラス図とは

クラス図は、**クラスの属性・メソッドと、クラス間の関係**を図示したUML図です。

### クラス図の記法

```
┌─────────────────────┐
│     <<クラス名>>      │
├─────────────────────┤
│ - privateField       │  ← 属性（フィールド）
│ # protectedField     │     - : private
│ + publicField        │     # : protected
├─────────────────────┤     + : public
│ + publicMethod()     │  ← メソッド
│ - privateMethod()    │
└─────────────────────┘
```

### 関連の種類

```
関連（Association）：
  ClassA ──── ClassB        「AはBを知っている」

依存（Dependency）：
  ClassA - - -> ClassB      「AはBを一時的に使う」

集約（Aggregation）：
  ClassA ◇── ClassB         「AはBを持つ（Bは独立して存在可）」

コンポジション（Composition）：
  ClassA ◆── ClassB         「AはBを持つ（AなしにBは存在しない）」

継承（Generalization）：
  ClassA ──▷ ClassB         「AはBを継承する」

実装（Realization）：
  ClassA --▷ InterfaceB     「AはBを実装する」
```

### 注文機能のクラス図の例

```
┌─────────────────────┐        ┌─────────────────────┐
│   OrderController    │        │   <<interface>>      │
├─────────────────────┤        │   IOrderService      │
│                     │        ├─────────────────────┤
├─────────────────────┤        │ + createOrder()      │
│ + create(req, res)  │───────→│ + getOrder()         │
│ + index(req, res)   │  uses  │ + cancelOrder()      │
│ + show(req, res)    │        └──────────▲──────────┘
└─────────────────────┘                   │ implements
                                          │
                               ┌──────────┴──────────┐
                               │   OrderService       │
                               ├─────────────────────┤
                               │ - orderRepo          │
                               │ - paymentService     │
                               ├─────────────────────┤
                               │ + createOrder()      │
                               │ + getOrder()         │
                               │ + cancelOrder()      │
                               │ - validateOrder()    │
                               └──────────┬──────────┘
                                          │ uses
                                    ┌─────┴─────┐
                                    │           │
                          ┌─────────▼───┐ ┌─────▼─────────┐
                          │OrderRepository│ │PaymentService │
                          ├─────────────┤ ├───────────────┤
                          │+ save()     │ │+ charge()     │
                          │+ findById() │ │+ refund()     │
                          │+ findAll()  │ └───────────────┘
                          └─────────────┘
```

### クラスの詳細定義

各クラスの属性・メソッドを詳細に記載します。

```markdown
## OrderService

### 概要
注文に関するビジネスロジックを担当するサービスクラス。

### 属性
| 属性名 | 型 | アクセス修飾子 | 説明 |
|--------|-----|-------------|------|
| orderRepo | IOrderRepository | private | 注文リポジトリ |
| paymentService | IPaymentService | private | 決済サービス |
| inventoryService | IInventoryService | private | 在庫サービス |

### メソッド

#### createOrder(userId: string, items: CartItem[]): Order
注文を作成する。

**引数：**
| 引数名 | 型 | 説明 |
|--------|-----|------|
| userId | string | ユーザーID |
| items | CartItem[] | カート内商品のリスト |

**戻り値：** Order - 作成された注文オブジェクト

**例外：**
| 例外 | 条件 |
|------|------|
| InsufficientStockError | 在庫が不足している場合 |
| PaymentFailedError | 決済が失敗した場合 |
| InvalidOrderError | カートが空の場合 |

**処理概要：**
1. カート内容のバリデーション
2. 各商品の在庫確認
3. 合計金額の計算
4. 決済処理の実行
5. 注文レコードの作成・保存
6. 在庫の引き当て
7. 注文確認メールの送信
```

## 4.4 シーケンス図

### シーケンス図とは

シーケンス図は、**オブジェクト間のメッセージのやり取りを時系列で**表現するUML図です。処理の流れを視覚的に理解できます。

### シーケンス図の記法

```
  オブジェクトA    オブジェクトB    オブジェクトC
      │               │               │
      │──メッセージ──→│               │  同期メッセージ
      │               │──メッセージ──→│
      │               │←─戻り値──────│  応答メッセージ
      │←─戻り値───────│               │
      │               │               │
      │= = メッセージ =>│              │  非同期メッセージ
```

### 注文処理のシーケンス図

```
 ユーザー    Controller    OrderService    InventoryService   PaymentService    OrderRepo
   │            │              │                │                │               │
   │──POST───→│              │                │                │               │
   │ /orders   │              │                │                │               │
   │            │──createOrder→│                │                │               │
   │            │              │                │                │               │
   │            │              │──checkStock──→│                │               │
   │            │              │               │                │               │
   │            │              │←─在庫OK───────│                │               │
   │            │              │                │                │               │
   │            │              │──charge─────────────────────→│               │
   │            │              │                │               │               │
   │            │              │←─決済成功────────────────────│               │
   │            │              │                │               │               │
   │            │              │──save────────────────────────────────────→│
   │            │              │                │               │              │
   │            │              │←─注文データ──────────────────────────────│
   │            │              │                │               │              │
   │            │←─Order───────│                │               │              │
   │            │              │                │               │              │
   │←─201──────│              │                │               │              │
   │  Created   │              │                │               │              │
```

### 例外フローのシーケンス図

正常系だけでなく、例外（エラー）が発生した場合の処理も記述します。

```
 ユーザー    Controller    OrderService    PaymentService
   │            │              │                │
   │──POST───→│              │                │
   │            │──createOrder→│                │
   │            │              │──charge──────→│
   │            │              │                │
   │            │              │←─決済失敗──────│
   │            │              │                │
   │            │              │  PaymentFailedError
   │            │              │  を送出
   │            │              │                │
   │            │←─Error───────│                │
   │            │              │                │
   │←─402──────│  エラーレスポンス               │
   │  Payment   │  を生成                       │
```

## 4.5 処理フロー（フローチャート）

### フローチャートの基本記号

```
  ┌─────────┐
  │ 開始/終了 │  端子（開始・終了）
  └─────────┘

  ┌─────────┐
  │  処理    │  処理（プロセス）
  └─────────┘

      ◇
     / \
    / 判断 \      判断（分岐）
    \     /
     \ ◇ /

  ┌─────────┐
  │ 入出力   │  データの入出力
  └─────────┘
```

```
        ┌─────────┐
        │  開始    │
        └────┬────┘
             ▼

        │カート内容│
        │を取得    │
        └────┬────┘
             ▼
           ◇◇◇
          ◇カートは◇─── Yes ──→┌─────────┐
          ◇ 空か？ ◇           │エラー返却│
           ◇◇◇                └─────────┘
             │ No
             ▼
        ┌──────────┐
        │各商品の   │
        │在庫を確認  │
        └────┬─────┘
             ▼
           ◇◇◇
          ◇在庫は ◇─── No ───→┌──────────┐
          ◇足りるか◇           │在庫不足   │
           ◇◇◇                │エラー返却  │
             │ Yes              └──────────┘

        ┌──────────┐

        ┌──────────┐
        │決済処理   │
```

## 4.6 状態遷移図

### 状態遷移図とは

状態遷移図は、**オブジェクトがどのような状態を持ち、どのイベントで状態が変化するか**を表現する図です。

### 注文ステータスの状態遷移図

```
                    注文確定
    ┌──────┐  ──────────→  ┌──────┐
    │ 下書き │               │ 確定  │
    └──────┘               └──┬───┘
                               │
              ┌────────────────┼────────────────┐
              │ 出荷指示       │ キャンセル       │
              ▼                ▼                │
         ┌──────┐        ┌──────────┐          │
         │ 出荷中 │        │ キャンセル │          │
         └──┬───┘        └──────────┘          │
            │                                   │
            │ 配達完了                            │
            ▼                                   │
         ┌──────┐        ┌──────┐              │
         │ 配達済 │───────→│ 返品  │──────────────┘
         └──────┘  返品   └──────┘     返金処理
                   申請
```

### 状態遷移表

状態遷移図を表形式で整理すると、漏れなく設計できます。

| 現在の状態 | イベント | 次の状態 | 処理 |
|-----------|---------|---------|------|
| 下書き | 注文確定 | 確定 | 決済実行、在庫引当 |
| 確定 | 出荷指示 | 出荷中 | 配送業者にデータ連携 |
| 確定 | キャンセル | キャンセル | 決済取消、在庫戻し |
| 出荷中 | 配達完了 | 配達済 | 配達完了通知送信 |
| 配達済 | 返品申請 | 返品 | 返品受付処理 |
| 返品 | 返金処理 | キャンセル | 返金実行 |

## 4.7 エラーハンドリング設計

### エラー分類

| エラー分類 | 説明 | HTTPステータス | 対応 |
|-----------|------|--------------|------|
| **バリデーションエラー** | 入力値が不正 | 400 | エラー内容をレスポンスに含める |
| **認証エラー** | 未認証 | 401 | ログインを促す |
| **認可エラー** | 権限不足 | 403 | アクセス拒否を通知 |
| **リソース未存在** | 指定リソースが存在しない | 404 | 該当なしを通知 |
| **ビジネスルール違反** | 在庫不足等 | 409/422 | 具体的なエラー理由を返す |
| **外部サービスエラー** | 決済失敗等 | 502 | リトライ or ユーザーに通知 |
| **内部エラー** | 予期しないエラー | 500 | ログ記録、一般的なメッセージを返す |

### エラーレスポンスの形式

```json
{
  "error": {
    "code": "INSUFFICIENT_STOCK",
    "message": "商品の在庫が不足しています",
    "details": [
      {
        "field": "items[0].quantity",
        "message": "商品ID: P-001 の在庫は残り3個です（要求: 5個）"
      }
    ]
  }
}
```

## 4.8 ログ設計

### ログレベル

| レベル | 用途 | 例 |
|--------|------|-----|
| **ERROR** | システムエラー、即時対応が必要 | DB接続失敗、外部API障害 |
| **WARN** | 想定内の問題、監視が必要 | リトライ発生、閾値超過 |
| **INFO** | 正常な業務処理の記録 | 注文作成、ユーザー登録 |
| **DEBUG** | 開発・デバッグ用の詳細情報 | リクエスト/レスポンス詳細 |

### ログフォーマット

```json
{
  "timestamp": "2026-04-01T10:30:00.000Z",
  "level": "INFO",
  "service": "order-service",
  "traceId": "abc-123-def",
  "userId": "user-456",
  "action": "createOrder",
  "message": "注文を作成しました",
  "metadata": {
    "orderId": "order-789",
    "amount": 15000,
    "itemCount": 3
  }
}
```

> **実務でのポイント**: ログに個人情報（メールアドレス、氏名、住所等）を出力しないよう注意してください。マスキング処理を設計段階で考慮します。

## まとめ

| 概念 | ポイント |
|------|---------|
| 詳細設計書 | モジュール内部の処理ロジックを開発者向けに定義する |
| クラス図 | クラスの属性・メソッドと関連をUMLで図示する |
| シーケンス図 | オブジェクト間のメッセージを時系列で表現する |
| フローチャート | 処理の流れと分岐を視覚的に表現する |
| 状態遷移図 | オブジェクトの状態変化とイベントを表現する |
| エラーハンドリング | エラーを分類し、適切なHTTPステータスとメッセージを定義する |
| ログ設計 | ログレベル・フォーマットを統一し、個人情報の取り扱いに注意する |

# 第6章：HTTP 演習問題

## 基本問題

### 問題6-1：HTTPメソッド
以下の操作に適切なHTTPメソッドを答えてください。

1. Webページを閲覧する
2. フォームでデータを送信する
3. ユーザ情報を全体更新する
4. ユーザを削除する
5. レスポンスヘッダだけ取得する（ボディは不要）
6. CORSのプリフライトリクエスト

<details>
<summary>解答</summary>

1. **GET**：リソースの取得に使用。安全で冪等
2. **POST**：データの送信・リソースの作成に使用。冪等ではない
3. **PUT**：リソースの全体置換（更新）に使用。冪等
4. **DELETE**：リソースの削除に使用。冪等
5. **HEAD**：GETと同じだがレスポンスボディを返さない。帯域の節約やリソースの存在確認に使用
6. **OPTIONS**：利用可能なメソッドの確認。CORSのプリフライトリクエストで使用

</details>

### 問題6-2：ステータスコード
以下のステータスコードの意味を答えてください。

1. 200
2. 201
3. 301
4. 302
5. 304
6. 400
7. 401
8. 403
9. 404
10. 500
11. 503

<details>
<summary>解答</summary>

1. **200 OK**：リクエスト成功
2. **201 Created**：リソースが新たに作成された（POST成功時等）
3. **301 Moved Permanently**：恒久的な移転。ブックマークやSEOの更新が必要
4. **302 Found**：一時的なリダイレクト。元のURLは有効なまま
5. **304 Not Modified**：キャッシュが有効。レスポンスボディなし
6. **400 Bad Request**：リクエストの構文が不正
7. **401 Unauthorized**：認証が必要（未認証状態）
8. **403 Forbidden**：アクセス拒否（認証済みだが権限なし）
9. **404 Not Found**：リソースが存在しない
10. **500 Internal Server Error**：サーバ内部で予期しないエラーが発生
11. **503 Service Unavailable**：サーバが一時的にサービス提供不可（メンテナンス、過負荷）

</details>

### 問題6-3：HTTPリクエストの構造
以下のHTTPリクエストの各部分の名称と役割を答えてください。

```
POST /api/users HTTP/1.1          ... (A)
Host: api.example.com             ... (B)
Content-Type: application/json    ... (C)
Authorization: Bearer abc123      ... (D)
                                  ... (E)
{"name": "Taro", "age": 25}      ... (F)
```

<details>
<summary>解答</summary>

- **(A) リクエスト行**：HTTPメソッド（POST）、リクエストURI（/api/users）、HTTPバージョン（HTTP/1.1）を指定
- **(B) Hostヘッダ**：接続先のホスト名。HTTP/1.1で必須。1つのIPアドレスで複数のサイトをホストする（バーチャルホスト）場合に重要
- **(C) Content-Typeヘッダ**：リクエストボディのMIMEタイプ。application/json はJSON形式であることを示す
- **(D) Authorizationヘッダ**：認証情報。Bearer トークンによる認証を使用
- **(E) 空行**：ヘッダとボディの区切りを示す
- **(F) メッセージボディ**：サーバに送信するデータ（JSON形式のユーザ情報）

</details>

### 問題6-4：GETとPOSTの違い
GETとPOSTの違いを以下の5つの観点から説明してください。

1. パラメータの送信方法
2. ブックマーク
3. キャッシュ
4. データサイズ
5. セキュリティ

<details>
<summary>解答</summary>

| 観点 | GET | POST |
|------|-----|------|
| パラメータの送信方法 | URLのクエリ文字列（?key=value） | メッセージボディ |
| ブックマーク | 可能（URLにパラメータが含まれる） | 不可（ボディは保存されない） |
| キャッシュ | キャッシュされる | キャッシュされない |
| データサイズ | URL長の制限あり（約2,000文字） | 制限が緩い（サーバ設定に依存） |
| セキュリティ | URLに表示され、ログに残る（機密情報に不向き） | ボディに含まれURL上は見えない（ただしHTTPSでないと盗聴可能） |

</details>

## 応用問題

### 問題6-5：Cookieのセキュリティ
以下のSet-Cookieヘッダを解釈し、各属性の意味を説明してください。

```
Set-Cookie: session_id=abc123; Path=/; HttpOnly; Secure; SameSite=Lax; Max-Age=3600
```

<details>
<summary>解答</summary>

| 属性 | 値 | 意味 |
|------|-----|------|
| **session_id** | abc123 | Cookieの名前と値 |
| **Path** | / | サイト全体のパスでCookieが有効 |
| **HttpOnly** | （フラグ） | JavaScriptからのCookieアクセスを禁止。XSS攻撃によるCookie窃取を防止 |
| **Secure** | （フラグ） | HTTPS通信時のみCookieを送信。HTTP通信での盗聴を防止 |
| **SameSite** | Lax | クロスサイトリクエストでのCookie送信を制限。トップレベルナビゲーション（GETのリンククリック）のみCookieを送信。CSRF攻撃の緩和 |
| **Max-Age** | 3600 | Cookieの有効期間は3600秒（1時間）。期限切れ後は自動削除 |

セキュリティのベストプラクティス：セッションCookieには **HttpOnly + Secure + SameSite** を必ず設定しましょう。

</details>

### 問題6-6：HTTPS
以下の問いに答えてください。

(1) HTTPSでは何を暗号化しますか。URLやヘッダも暗号化されますか。
(2) TLSハンドシェイクで使用される2種類の暗号方式を答えてください。
(3) サーバ証明書のDV、OV、EVの違いを説明してください。

<details>
<summary>解答</summary>

**(1)** HTTPSではHTTPの通信内容（リクエスト行、ヘッダ、ボディ全て）がTLSにより暗号化されます。ただし、接続先のホスト名（SNI: Server Name Indication）はTLS 1.2までは平文で送信されます（TLS 1.3のEncrypted Client Helloで暗号化可能に）。また、IPアドレスとポート番号は暗号化されません。

**(2)**
- **公開鍵暗号方式（非対称鍵暗号）**：鍵交換とサーバ認証に使用（例：RSA、ECDHE）。処理が重い
- **共通鍵暗号方式（対称鍵暗号）**：実際のデータ通信の暗号化に使用（例：AES）。高速
- ハンドシェイクで公開鍵暗号を使って共通鍵を安全に交換し、以降の通信は高速な共通鍵暗号で行う（ハイブリッド暗号）

**(3)**
| 種類 | 検証レベル | 取得の難易度 | 信頼性 |
|------|-----------|-------------|--------|
| **DV（ドメイン認証）** | ドメインの所有権のみ確認 | 容易（自動取得可能、Let's Encrypt等） | 低（フィッシングサイトでも取得可能） |
| **OV（企業認証）** | 組織の実在性を書類等で確認 | 中程度 | 中 |
| **EV（Extended Validation）** | 厳格な組織審査（登記簿等） | 困難（審査に時間がかかる） | 高 |

</details>

### 問題6-7：HTTP/2 と HTTP/3
HTTP/1.1 の問題点を2つ挙げ、HTTP/2 と HTTP/3 でそれぞれどのように解決されたか説明してください。

<details>
<summary>解答</summary>

**HTTP/1.1 の問題点と解決策：**

**問題1：Head-of-Line（HoL）Blocking**
- HTTP/1.1 では1つのTCP接続上でリクエストが直列処理される。前のリクエストのレスポンスが遅いと後続が待たされる
- **HTTP/2 の解決策**：ストリーム多重化。1つのTCP接続上で複数のリクエスト/レスポンスを並列処理。ただしTCPレベルのHoLブロッキングは残る
- **HTTP/3 の解決策**：QUIC（UDP上のプロトコル）を使用。ストリームごとに独立しているため、1つのストリームのパケットロスが他のストリームに影響しない

**問題2：ヘッダの冗長性**
- HTTP/1.1 ではリクエストごとに同じヘッダ（Cookie、User-Agent等）を繰り返し送信するため、オーバーヘッドが大きい
- **HTTP/2 の解決策**：HPACK圧縮。ヘッダテーブルを使って重複するヘッダを効率的に圧縮
- **HTTP/3 の解決策**：QPACK圧縮。HoLブロッキングを避けつつヘッダを圧縮

</details>

## チャレンジ問題

### 問題6-8：RESTful API設計
ユーザ管理APIを設計する場合、以下の操作に対するHTTPメソッド、パス、ステータスコードの組み合わせを考えてください。

1. ユーザ一覧の取得
2. 特定ユーザの取得
3. 新規ユーザの作成
4. ユーザ情報の更新
5. ユーザの削除

<details>
<summary>解答</summary>

| 操作 | メソッド | パス | 成功時ステータス | 説明 |
|------|---------|------|----------------|------|
| ユーザ一覧取得 | GET | /api/users | 200 OK | 全ユーザの配列を返す |
| 特定ユーザ取得 | GET | /api/users/{id} | 200 OK | 指定IDのユーザを返す（存在しなければ404） |
| 新規ユーザ作成 | POST | /api/users | 201 Created | リクエストボディにユーザ情報。Locationヘッダに新リソースのURLを含める |
| ユーザ情報更新 | PUT | /api/users/{id} | 200 OK | リクエストボディに更新後のユーザ情報（全体置換） |
| ユーザ削除 | DELETE | /api/users/{id} | 204 No Content | ボディなしで成功を返す（存在しなければ404） |

RESTful APIの設計原則：
- URLはリソースを名詞で表す（動詞は使わない）
- HTTPメソッドで操作を表現する
- 適切なステータスコードを返す

</details>

### 問題6-9：トラブルシューティング
Webアプリケーションで以下の問題が発生した場合、原因として考えられることと調査方法を述べてください。

(1) ブラウザに「ERR_CERT_DATE_INVALID」と表示される
(2) APIリクエストで「Access-Control-Allow-Origin」エラーが発生する
(3) ページ表示が非常に遅い（タイムアウトに近い）

<details>
<summary>解答</summary>

**(1) ERR_CERT_DATE_INVALID**
- **原因**：SSL/TLS証明書の有効期限が切れている、またはまだ有効期間に入っていない。あるいはクライアントPCの時計がずれている
- **調査方法**：
  - ブラウザで証明書の詳細を確認し、有効期間を確認
  - クライアントPCの日時設定を確認
  - サーバ側で証明書の更新状況を確認

**(2) Access-Control-Allow-Origin エラー（CORSエラー）**
- **原因**：フロントエンドとAPIサーバのオリジン（スキーム+ホスト+ポート）が異なり、サーバ側でCORSヘッダが正しく設定されていない
- **調査方法**：
  - ブラウザの開発者ツールのネットワークタブでプリフライトリクエスト（OPTIONS）の応答を確認
  - サーバ側の `Access-Control-Allow-Origin`、`Access-Control-Allow-Methods`、`Access-Control-Allow-Headers` の設定を確認
  - フロントエンドのオリジンが許可リストに含まれているか確認

**(3) ページ表示が非常に遅い**
- **原因の可能性**：DNS解決の遅延、サーバの応答遅延、大量のリソース読み込み、帯域不足など
- **調査方法**：
  - ブラウザの開発者ツールのネットワークタブで各リクエストの所要時間を確認
  - DNS解決時間、TCP接続時間、TLSハンドシェイク時間、サーバ応答時間（TTFB）を分析
  - サーバ側のアクセスログとアプリケーションログを確認
  - `traceroute` でネットワーク経路の遅延を確認

</details>

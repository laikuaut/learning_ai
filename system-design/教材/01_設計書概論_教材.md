# 第1章：設計書概論

## この章のゴール

- 設計書（Design Document）とは何か、なぜ必要なのかを理解する
- システム開発における設計工程の位置づけを把握する
- 設計書の種類と全体像を理解する
- 良い設計書と悪い設計書の違いを見分けられるようになる
- 設計書作成の基本原則を身につける

---

## 1.1 設計書とは何か

### 設計書の定義

設計書（Design Document）とは、**システムを「どのように作るか」を記述した文書**です。建築における設計図と同じく、ソフトウェア開発においても「作る前に設計する」ことが重要です。

```
【建築との対比】

建築                          ソフトウェア開発
─────────────────────────────────────────────
施主の要望ヒアリング    →    要件定義
建築設計図の作成        →    基本設計・詳細設計
施工（建設工事）        →    実装（コーディング）
完了検査                →    テスト
引き渡し                →    リリース・納品
```

### なぜ設計書が必要なのか

設計書を書かずにいきなりコーディングを始めることは、設計図なしに建物を建て始めるようなものです。

| 設計書がない場合の問題 | 具体例 |
|---|---|
| **手戻りの発生** | 実装後に「そもそもこの機能は不要だった」と判明する |
| **認識のズレ** | 開発者ごとに異なる解釈でコードを書いてしまう |
| **コミュニケーションコスト増大** | 仕様の確認に毎回口頭で聞く必要がある |
| **品質のばらつき** | 基準がないため、担当者によって品質が大きく異なる |
| **保守の困難** | 後から参加したメンバーがシステム全体を把握できない |

> **実務でのポイント**: 「設計書を書く時間がもったいない」と言う現場もありますが、設計書がないことで発生する手戻りや認識齟齬のコストの方が、はるかに大きくなります。特に複数人で開発する場合、設計書は「共通言語」として機能します。

## 1.2 システム開発の工程と設計書の位置づけ

### ウォーターフォールモデルにおける設計工程

システム開発の工程は、一般的に以下のV字モデル（V-Model）で表されます。設計工程とテスト工程が対応関係にあることが特徴です。

```
要件定義 ─────────────────────────────── 受入テスト
  │                                         │
  ▼                                         ▲
  基本設計 ───────────────────────── 結合テスト
    │                                     │
    ▼                                     ▲
    詳細設計 ─────────────────── 単体テスト
      │                               │
      ▼                               ▲
      ┌──────────────────────┐
      │    実装（コーディング）    │
      └──────────────────────┘
```

### 各工程で作成する設計書

| 工程 | 作成する設計書 | 主な内容 |
|---|---|---|
| **要件定義** | 要件定義書 | 何を作るか（What） |
| **基本設計** | 基本設計書（外部設計書） | システム全体の構造、画面、API |
| **詳細設計** | 詳細設計書（内部設計書） | モジュールの内部ロジック |
| **実装** | （コード・コメント） | 実際のプログラム |
| **テスト** | テスト設計書 | テストの計画・ケース |

### アジャイル開発における設計書

アジャイル開発では「動くソフトウェアを重視する」ため、設計書の位置づけが異なります。

```
【ウォーターフォール】                【アジャイル】
設計書 → 実装 → テスト               設計 → 実装 → テスト
（全機能を一括で設計）               （1機能ずつ繰り返す）

┌────────────────────┐          ┌─────────┐
│要件定義書（全量）     │          │ Sprint 1 │→ 設計→実装→テスト
│基本設計書（全量）     │          ├─────────┤
│詳細設計書（全量）     │          │ Sprint 2 │→ 設計→実装→テスト
│テスト設計書（全量）   │          ├─────────┤
└────────────────────┘          │ Sprint 3 │→ 設計→実装→テスト
                                └─────────┘
```

| 観点 | ウォーターフォール | アジャイル |
|---|---|---|
| 設計書の量 | 多い（網羅的） | 少ない（必要最小限） |
| 設計書の更新頻度 | 少ない（最初に確定） | 高い（スプリントごとに更新） |
| 設計書の形式 | フォーマルな文書 | 軽量（Wiki、チケット、ADR等） |
| 設計書の目的 | 契約・承認の根拠 | チーム内の共有・記録 |

> **実務でのポイント**: アジャイル開発だから設計書が不要というわけではありません。形式が軽量になるだけで、「設計を考える」行為自体は必ず必要です。ADR（Architecture Decision Record）のように、設計判断の記録を残すことが重要視されています。

## 1.3 設計書の種類と全体像

### 設計書の分類マップ

システム開発で作成する主な設計書を、工程と対象領域で整理します。

```
                    ┌─────────────────────────────────────────┐
                    │              設計書の全体像                │
                    └─────────────────────────────────────────┘
                                      │
          ┌───────────────┬───────────┼───────────┬──────────────┐
          ▼               ▼           ▼           ▼              ▼
    ┌──────────┐  ┌──────────┐ ┌──────────┐ ┌──────────┐ ┌──────────┐
    │ 要件定義書 │  │ 基本設計書 │ │ 詳細設計書 │ │ テスト   │ │ 運用設計書 │
    │          │  │          │ │          │ │ 設計書   │ │          │
    └──────────┘  └──────────┘ └──────────┘ └──────────┘ └──────────┘
         │              │           │           │              │
         │         ┌────┼────┐      │      ┌────┼────┐         │
         │         ▼    ▼    ▼      │      ▼    ▼    ▼         │
         │      画面  API  DB       │    単体 結合 受入       移行
         │      設計  設計  設計     │    テスト             手順書
         │      書    書    書       │
         ▼                          ▼
    機能要件                   モジュール設計
    非機能要件                 クラス設計
    業務フロー                 シーケンス図
```

### 主要な設計書の一覧

| 設計書名 | 作成工程 | 主な読者 | 概要 |
|---|---|---|---|
| **要件定義書** | 要件定義 | 顧客・PM・開発者 | システムに求める機能・性能の定義 |
| **基本設計書** | 基本設計 | PM・開発者・レビュアー | システム全体のアーキテクチャ・構造 |
| **画面設計書** | 基本設計 | デザイナー・フロント開発者 | 画面レイアウト・遷移・操作仕様 |
| **API設計書** | 基本設計 | バックエンド・フロント開発者 | APIのエンドポイント・リクエスト・レスポンス |
| **データベース設計書** | 基本設計 | バックエンド開発者・DBA | テーブル定義・ER図・インデックス |
| **詳細設計書** | 詳細設計 | 開発者 | モジュール内部のロジック・処理フロー |
| **テスト設計書** | テスト計画 | テスター・開発者・PM | テスト方針・ケース・期待結果 |

## 1.4 良い設計書の条件

### 良い設計書の5つの条件

```
良い設計書の条件

┌─────────────────────────────────────────────────┐
│  1. 正確性    ─ 実際のシステムと一致している       │
│  2. 明確性    ─ 曖昧な表現がなく、解釈が一意       │
│  3. 完全性    ─ 必要な情報が漏れなく記載されている   │
│  4. 一貫性    ─ 他の設計書と矛盾がない             │
│  5. 保守性    ─ 変更があった際に更新しやすい        │
└─────────────────────────────────────────────────┘
```

### 良い設計書と悪い設計書の比較

**悪い例：曖昧な記述**

```
機能概要：ユーザーが商品を検索できる。
処理内容：検索条件に基づいてデータベースから商品を取得し、一覧で表示する。
```

**良い例：具体的な記述**

```
機能名：商品検索機能（FUNC-001）
概要  ：ユーザーが入力した検索条件に基づき、商品テーブルから該当商品を
        取得し、検索結果一覧画面に表示する。

検索条件：
  - 商品名（部分一致、最大50文字）
  - カテゴリ（完全一致、プルダウン選択）
  - 価格帯（最小値〜最大値、整数）
  - 並び順（価格昇順/降順、新着順）

表示件数：1ページあたり20件（ページネーション対応）
検索結果0件の場合：「該当する商品が見つかりませんでした」と表示

レスポンス要件：検索結果の表示まで3秒以内
```

### よくある「悪い設計書」のパターン

| パターン | 問題点 | 改善策 |
|---|---|---|
| **曖昧語の多用** | 「適切に処理する」「必要に応じて」 | 具体的な条件・処理を明記する |
| **図がない** | テキストだけでは構造が伝わらない | 図表・アスキーアートを活用する |
| **コードのコピペ** | 設計書がコード解説になっている | 設計意図と仕様を書く |
| **更新されていない** | 実装と設計書が乖離している | 変更管理プロセスを設ける |
| **読者を想定していない** | 技術者以外が読めない | 読者に合わせた粒度で書く |

> **実務でのポイント**: 設計書に「等」「適宜」「必要に応じて」「適切に」といった曖昧な言葉が出てきたら要注意です。これらは後工程で必ず問題になります。設計書では、曖昧な表現を排除し、誰が読んでも同じ解釈になる記述を心がけましょう。

## 1.5 設計書作成の基本原則

### MECE（ミーシー）の原則

設計書の内容は、**Mutually Exclusive, Collectively Exhaustive（漏れなく、ダブりなく）**であることが重要です。

```
【MECEの例：ユーザー種別の定義】

❌ 悪い例（MECEでない）          ✅ 良い例（MECE）
─────────────────────────────────────────────
・一般ユーザー                   ・一般ユーザー
・管理者                         ・店舗管理者
・ゲスト                         ・システム管理者
                                 ・ゲスト（未認証ユーザー）

→ 「管理者」が曖昧               → 管理者の種類が明確
→ 他の権限がないか不明            → 全ユーザー種別を網羅
```

### 番号体系の設計

設計書のすべての要素には、一意な識別番号を付与します。これにより、設計書間のトレーサビリティ（追跡可能性）を確保できます。

```
【番号体系の例】

要件定義書    基本設計書      詳細設計書       テスト設計書
─────────────────────────────────────────────────────
REQ-001  →  BD-001      →  DD-001       →  TC-001
REQ-002  →  BD-002      →  DD-002       →  TC-002
         →  BD-003      →  DD-003       →  TC-003
                         →  DD-004       →  TC-004

REQ: Requirement（要件）
BD : Basic Design（基本設計）
DD : Detail Design（詳細設計）
TC : Test Case（テストケース）
```

### 変更管理

設計書は「生きた文書」です。変更があった場合は、変更履歴を記録し、関連する設計書も合わせて更新します。

```
【変更履歴の記載例】

┌──────┬────────────┬────────────────────────┬──────┬──────┐
│ 版数 │ 更新日      │ 変更内容                │ 更新者 │ 承認者 │
├──────┼────────────┼────────────────────────┼──────┼──────┤
│ 1.0  │ 2026/01/15 │ 初版作成                │ 田中  │ 鈴木  │
│ 1.1  │ 2026/02/03 │ REQ-005 検索条件追加    │ 田中  │ 鈴木  │
│ 2.0  │ 2026/03/10 │ API仕様の大幅変更       │ 佐藤  │ 鈴木  │
└──────┴────────────┴────────────────────────┴──────┴──────┘
```

## 1.6 設計書のフォーマットとツール

### よく使われるフォーマット

| フォーマット | 特徴 | 適するケース |
|---|---|---|
| **Word/Google Docs** | フォーマルな体裁、印刷向き | SIer、官公庁向けプロジェクト |
| **Excel** | テーブル形式が得意、一覧管理向き | テーブル定義書、テスト仕様書 |
| **Markdown** | 軽量、バージョン管理しやすい | アジャイル開発、GitHubベース |
| **Wiki（Confluence等）** | 共同編集、リンクで関連づけ | チーム開発、社内ドキュメント |
| **専用ツール** | 図の作成・管理に特化 | UML図、ER図、画面設計 |

### 設計書で使う代表的な図

| 図の種類 | 用途 | 使用する設計書 |
|---|---|---|
| **システム構成図** | システム全体の物理・論理構成 | 基本設計書 |
| **ER図** | データベースのテーブル関連 | データベース設計書 |
| **画面遷移図** | 画面間の遷移関係 | 画面設計書 |
| **シーケンス図** | 処理の時系列の流れ | 詳細設計書 |
| **フローチャート** | 処理の分岐・ループ | 詳細設計書 |
| **クラス図** | クラス間の関係 | 詳細設計書 |
| **ユースケース図** | ユーザーとシステムの関わり | 要件定義書 |

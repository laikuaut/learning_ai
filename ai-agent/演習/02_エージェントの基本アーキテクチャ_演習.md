### 問題1：ReActパターンの穴埋め（基本）

ReAct（Reasoning + Acting）パターンは、AIエージェントの基本的な動作パターンです。以下のReActトレースの空欄（A）〜（F）を埋めてください。

**タスク：** 「東京の明日の天気を調べて、傘が必要か教えて」

```
Thought 1: （A）
Action 1:  （B）
Observation 1: （C）
Thought 2: （D）
Action 2:  （E）
Observation 2: （F）
Thought 3: 降水確率が60%なので、傘を持っていくことを推奨すべきです。
            最終回答をユーザーに返します。
Final Answer: 明日の東京の天気は「くもり時々雨」で、降水確率は60%です。
              傘を持っていくことをおすすめします。
```

**期待される出力例：**
```
（A）ユーザーは東京の明日の天気を知りたい。まず天気予報APIで天気情報を取得する必要がある。
（B）weather_api.get_forecast(location="東京", date="tomorrow")
（C）{"location": "東京", "date": "2026-03-27", "weather": "くもり時々雨",
      "high_temp": 15, "low_temp": 8, "precipitation_chance": 60}
（D）天気は「くもり時々雨」で降水確率60%と分かった。傘が必要かどうか判断するために、
     降水確率の基準を確認する。一般的に40%以上なら傘を推奨する。
（E）なし（追加のツール呼び出しは不要。内部の推論で判断可能）
（F）なし（ツールを使わなかったため観察結果なし）
```

<details>
<summary>ヒント</summary>

ReActパターンは以下の3ステップの繰り返しです。
- **Thought（思考）**：現在の状況を分析し、次に何をすべきか考える
- **Action（行動）**：ツールを呼び出すなど、具体的な行動を実行する
- **Observation（観察）**：行動の結果を受け取り、確認する

このサイクルを、タスクが完了するまで繰り返します。なお、Thought内で判断が完了する場合、追加のAction/Observationは不要になることもあります。

</details>

<details>
<summary>解答例</summary>

```
（A）ユーザーは東京の明日の天気を知りたがっている。まず天気予報APIを使って
     東京の明日の天気情報を取得する必要がある。

（B）weather_api.get_forecast(location="東京", date="tomorrow")

（C）{"location": "東京", "date": "2026-03-27", "weather": "くもり時々雨",
      "high_temp": 15, "low_temp": 8, "precipitation_chance": 60}

（D）天気情報を取得できた。天気は「くもり時々雨」で降水確率が60%だ。
     一般的に降水確率が40%以上であれば傘を持参することが推奨される。
     追加の情報取得は不要で、この情報だけで判断できる。

（E）なし（追加のツール呼び出しは不要。Thought 2の推論で結論が出た）

（F）なし（Action 2でツールを使用しなかったため、観察結果はなし）
```

<!-- 解説：
- ReActの核心は「考えてから行動する」という点です。
  闇雲にツールを呼び出すのではなく、Thoughtで次の行動を計画します。
- Thought 2のように、既に得られた情報で判断できる場合は
  追加のAction/Observationが不要になることもあります。
- 実際のエージェント実装では、LLMがThought/Action/Observationの
  各ステップを生成し、Actionの実行結果がObservationとしてフィードバックされます。
-->

</details>

### 問題2：メモリの種類と特徴（基本）

AIエージェントのメモリは複数の種類に分類されます。以下の説明が、どのメモリの種類に該当するかを答えてください。

**メモリの種類：**
- 短期メモリ（Short-term Memory / Working Memory）
- 長期メモリ（Long-term Memory）
- エピソード記憶（Episodic Memory）
- 意味記憶（Semantic Memory）
- 手続き記憶（Procedural Memory）

**説明：**
1. 「以前にユーザーAさんの問い合わせで返金処理を行った、その時の会話の流れと結果」
2. 「現在の会話で、3つ前のメッセージでユーザーが言った条件」
3. 「"Pythonではインデントがブロック構造を表す"という一般的な知識」
4. 「コードレビューを行う際の手順（ファイルを読む→問題を特定→改善案を提示）」
5. 「過去1年分の対応履歴から学んだ、よくある問い合わせパターンのデータベース」

**期待される出力例：**
```
1. エピソード記憶（特定の出来事・経験の記録）
2. 短期メモリ（現在のセッション内の情報保持）
3. 意味記憶（一般的な知識・事実）
4. 手続き記憶（手順・やり方の知識）
5. 長期メモリ（永続的に蓄積されたデータ）/ 意味記憶（パターンとして汎化された知識）
```

<details>
<summary>ヒント</summary>

人間の記憶の分類と対応させて考えると理解しやすいです。
- 短期メモリ：「今の会話を覚えている」（ワーキングメモリ）
- 長期メモリ：「ずっと前のことも覚えている」
  - エピソード記憶：「いつ、どこで、何があったか」の具体的な出来事
  - 意味記憶：「〜とは何か」という一般知識
  - 手続き記憶：「〜のやり方」という手順の知識

</details>

<details>
<summary>解答例</summary>

```
1. エピソード記憶（Episodic Memory）
   → 特定のユーザーとの特定の出来事（返金処理を行った経験）の記録です。
     「いつ・誰と・何をしたか」という具体的な経験が保存されています。

2. 短期メモリ（Short-term Memory / Working Memory）
   → 現在の会話セッション内で保持されている情報です。
     セッションが終了すると通常は失われます。
     LLMの場合、コンテキストウィンドウ内の情報がこれに相当します。

3. 意味記憶（Semantic Memory）
   → 特定の経験に紐づかない一般的な知識・事実です。
     LLMの事前学習で獲得された知識や、RAGで参照される
     ナレッジベースの情報がこれに相当します。

4. 手続き記憶（Procedural Memory）
   → 「どうやるか」という手順・方法の知識です。
     エージェントのシステムプロンプトやツールの使用手順がこれに相当します。

5. 長期メモリ（Long-term Memory）/ 意味記憶（Semantic Memory）
   → 長期間蓄積されたデータベースであり、具体的な個別事例というよりも
     パターンとして汎化された知識です。
     ベクトルDBに保存された対応パターンのデータベースなどが実装例です。
```

<!-- 解説：
- 現代のLLMベースエージェントでの実装対応：
  - 短期メモリ → コンテキストウィンドウ内の会話履歴
  - 長期メモリ → ベクトルDB、外部データベース
  - エピソード記憶 → 過去の実行ログ・会話ログの保存と検索
  - 意味記憶 → RAGのナレッジベース、LLMの事前学習知識
  - 手続き記憶 → システムプロンプト、ツール定義、Few-shot例
- メモリ設計はエージェントの性能に大きく影響する重要な要素です。
-->

</details>

### 問題3：ツール定義のJSON Schema作成（基本）

AIエージェントがツールを利用するには、ツールの機能をJSON Schemaで定義する必要があります。以下の3つのツールについて、それぞれのJSON Schema定義を作成してください。

1. **天気取得ツール**：指定された都市の天気情報を返す
2. **メール送信ツール**：指定された宛先にメールを送信する
3. **計算ツール**：数式（文字列）を受け取り、計算結果を返す

**期待される出力例（ツール1の例）：**
```json
{
  "name": "get_weather",
  "description": "指定された都市の現在の天気情報を取得します",
  "parameters": {
    "type": "object",
    "properties": {
      "city": {
        "type": "string",
        "description": "天気を取得する都市名（例：東京、大阪）"
      },
      "unit": {
        "type": "string",
        "enum": ["celsius", "fahrenheit"],
        "description": "温度の単位（デフォルト：celsius）"
      }
    },
    "required": ["city"]
  }
}
```

<details>
<summary>ヒント</summary>

JSON Schemaでツールを定義する際のポイントです。
- `name`：ツールの識別名（英語のスネークケースが一般的）
- `description`：ツールが何をするかの説明（LLMがツール選択の判断に使います）
- `parameters`：ツールに渡すパラメータの定義
  - `type`：パラメータのデータ型（string, number, boolean, array, object等）
  - `description`：各パラメータの説明（具体例を含めると効果的）
  - `required`：必須パラメータのリスト
  - `enum`：選択肢が限定される場合の値リスト

LLMはこの定義を読んで「どのツールをいつ使うか」を判断するため、descriptionは明確かつ具体的に書くことが重要です。

</details>

<details>
<summary>解答例</summary>

```json
// ツール1：天気取得ツール
{
  "name": "get_weather",
  "description": "指定された都市の現在の天気情報を取得します。天気、気温、湿度、降水確率を返します。",
  "parameters": {
    "type": "object",
    "properties": {
      "city": {
        "type": "string",
        "description": "天気を取得する都市名（例：東京、大阪、名古屋）"
      },
      "unit": {
        "type": "string",
        "enum": ["celsius", "fahrenheit"],
        "description": "温度の単位。デフォルトはcelsius（摂氏）"
      }
    },
    "required": ["city"]
  }
}

// ツール2：メール送信ツール
{
  "name": "send_email",
  "description": "指定された宛先にメールを送信します。件名、本文、オプションでCC・添付ファイルを指定できます。",
  "parameters": {
    "type": "object",
    "properties": {
      "to": {
        "type": "string",
        "description": "送信先のメールアドレス（例：user@example.com）"
      },
      "subject": {
        "type": "string",
        "description": "メールの件名"
      },
      "body": {
        "type": "string",
        "description": "メールの本文"
      },
      "cc": {
        "type": "array",
        "items": { "type": "string" },
        "description": "CCに追加するメールアドレスのリスト（オプション）"
      }
    },
    "required": ["to", "subject", "body"]
  }
}

// ツール3：計算ツール
{
  "name": "calculate",
  "description": "数式を文字列として受け取り、計算結果を返します。四則演算、べき乗、平方根等に対応しています。",
  "parameters": {
    "type": "object",
    "properties": {
      "expression": {
        "type": "string",
        "description": "計算する数式（例：'(10 + 20) * 3', 'sqrt(144)', '2 ** 10'）"
      }
    },
    "required": ["expression"]
  }
}
```

<!-- 解説：
- JSON Schemaによるツール定義は、OpenAIのFunction Calling、
  AnthropicのTool Use、LangChainのToolクラスなど、
  主要なフレームワークで共通の考え方です。
- description の質がツール選択の精度を大きく左右します。
  「何をするか」だけでなく「いつ使うべきか」が分かる説明が理想的です。
- requiredフィールドを適切に設定し、必須パラメータと
  オプションパラメータを明確に区別することが重要です。
- enumを使って値を制限することで、LLMの誤入力を防げます。
-->

### 問題4：エージェントループの理解（基本）

以下は、AIエージェントの基本的な動作ループを示した擬似コードです。空欄（A）〜（E）を埋めて、ループを完成させてください。

```
function agent_loop(user_input):
    messages = [{"role": "user", "content": user_input}]

    while True:
        // ステップ1：（A）
        response = llm.generate(messages)

        // ステップ2：（B）
        if response.type == "final_answer":
            return response.content

        // ステップ3：（C）
        tool_name = response.tool_call.name
        tool_args = response.tool_call.arguments

        // ステップ4：（D）
        tool_result = execute_tool(tool_name, tool_args)

        // ステップ5：（E）
        messages.append({"role": "assistant", "content": response})
        messages.append({"role": "tool", "content": tool_result})
```

**期待される出力例：**
```
（A）LLMに現在のメッセージ履歴を渡して、次の行動を推論させる
（B）LLMの応答が最終回答であれば、ループを終了して結果を返す
（C）LLMがツール呼び出しを要求している場合、呼び出すツール名と引数を取得する
（D）指定されたツールを実際に実行し、結果を受け取る
（E）LLMの応答とツールの実行結果をメッセージ履歴に追加し、次のループに備える
```

<details>
<summary>ヒント</summary>

エージェントループは以下の流れで動作します。
1. LLMに状況を伝えて判断を仰ぐ
2. LLMの判断を確認する（タスク完了？それともまだやることがある？）
3. ツール呼び出しの指示があれば、それを解析する
4. 実際にツールを動かす
5. 結果を記録して、次のサイクルに入る

この「考える → 行動する → 結果を確認する → また考える」のサイクルが、エージェントの基本動作です。

</details>

<details>
<summary>解答例</summary>

```
（A）LLMに現在のメッセージ履歴を渡して、次の行動（ツール呼び出し or 最終回答）を推論させる
    → これがエージェントの「思考」フェーズです。LLMは会話の全履歴を見て、
      次に何をすべきかを判断します。

（B）LLMの応答が最終回答（final_answer）であれば、ループを終了してユーザーに結果を返す
    → これがループの「終了条件」です。LLMがタスク完了と判断した場合に
      ループから抜け出します。

（C）LLMがツール呼び出しを要求している場合、呼び出すべきツールの名前と引数を取得する
    → LLMの応答をパースして、どのツールをどんなパラメータで呼ぶかを
      構造化データとして取り出します。

（D）指定されたツールを実際に実行し、その結果を受け取る
    → エージェントの「行動」フェーズです。外部APIの呼び出しや
      ファイル操作など、実際の処理がここで行われます。

（E）LLMの応答（ツール呼び出しの指示）とツールの実行結果をメッセージ履歴に追加する
    → 次のループでLLMが「何をやって、何が返ってきたか」を把握できるようにします。
      これがエージェントの「短期メモリ」の実体です。
```

<!-- 解説：
- このエージェントループは、現代のほぼすべてのLLMエージェントの基盤です。
- 重要なポイント：
  1. ループの各反復でLLMは「全履歴」を見て判断する（短期メモリの役割）
  2. 終了条件がないと無限ループになるリスクがある（最大ステップ数の制限が実務では必須）
  3. ツール実行結果をメッセージに追加することで、LLMがフィードバックを受け取れる
  4. このシンプルなループが、ReActパターンの実装の本質です
- 実際の実装では、エラーハンドリング、タイムアウト、最大反復回数の制限なども必要です。
-->

</details>

### 問題5：具体的タスクのReActトレース作成（応用）

以下のタスクに対する完全なReActトレースを作成してください。Thought/Action/Observationの各ステップを省略せず、タスクが完了するまで記述してください。

**タスク：** 「Pythonのrequestsライブラリの最新バージョンを調べて、前のバージョンからの主な変更点を3つ教えて」

**利用可能なツール：**
- `web_search(query: string)` ：Web検索を実行し、結果のリストを返す
- `read_webpage(url: string)` ：指定URLのWebページの内容を取得する

**期待される出力例：**
```
Thought 1: Pythonのrequestsライブラリの最新バージョンを調べる必要がある。
           まずWeb検索で最新バージョンを確認しよう。
Action 1:  web_search("Python requests library latest version 2026")
Observation 1: [
  {"title": "requests 2.32.3 - PyPI", "url": "https://pypi.org/project/requests/", ...},
  {"title": "Release History - requests", "url": "https://pypi.org/project/requests/#history", ...},
  ...
]
Thought 2: PyPIのページで最新バージョンが2.32.3であることが分かった。
           変更履歴を確認するためにリリースページを読んでみよう。
Action 2:  read_webpage("https://pypi.org/project/requests/#history")
Observation 2: ...（リリース履歴の内容）...
Thought 3: ...
...（続く）
```

<details>
<summary>ヒント</summary>

ReActトレースを作成する際のポイントです。

1. **Thought（思考）** では以下を含めましょう。
   - 現在分かっていること / まだ分かっていないこと
   - 次にどのツールを使うべきかの判断理由

2. **Action（行動）** では以下を含めましょう。
   - ツール名と具体的な引数
   - 検索クエリは情報が得やすいように工夫する

3. **Observation（観察）** では以下を含めましょう。
   - ツールの実行結果（実際のAPI応答をイメージ）
   - 必要な情報が含まれているかどうか

</details>

Thought 3: v2.32.4のリリースノートが確認できた。前バージョン（v2.32.3）からの

  2. **セッションCookieハンドリングの修正**：リダイレクト時にセッションの

  3. **コネクションプール管理の改善**：接続プールの管理が最適化され、

```

<!-- 解説：

        "parameters": {

    def __init__(self):

    def add_step(self, step_number, thought, action, observation, duration):

def execute_tool(tool_name, tool_input):
    """
    指定されたツールを実行する。
    リトライロジック付き。
    """
    # ツール名のバリデーション
    if tool_name not in TOOLS:
        return {
            "success": False,
            "error": f"不明なツール: {tool_name}。利用可能: {list(TOOLS.keys())}"
        }

    tool = TOOLS[tool_name]

    # 必須パラメータのバリデーション
    for param_name, param_def in tool["parameters"].items():
        if param_def.get("required") and param_name not in tool_input:
            return {
                "success": False,
                "error": f"必須パラメータが不足: {param_name}"
            }

    # リトライ付きでツール実行
    for attempt in range(1, MAX_RETRIES + 1):
        try:
            result = tool["function"](**tool_input)
            return {"success": True, "result": result}
        except Exception as e:
            if attempt < MAX_RETRIES:
                log(f"ツール実行失敗（{attempt}/{MAX_RETRIES}回目）: {e}")
                wait(1 * attempt)  # エクスポネンシャルバックオフ
            else:
                return {
                    "success": False,
                    "error": f"ツール実行が{MAX_RETRIES}回失敗: {e}"
                }

    elif "Action:" in response_text:
        result["action"] = extract_section(response_text, "Action")
        result["action_input"] = parse_json(
            extract_section(response_text, "Action Input")
        )

    return result

def agent_run(user_input):
    """
    ReActパターンに基づくメインのエージェントループ。

    引数:
        user_input: ユーザーからの入力テキスト

    戻り値:
        最終回答のテキスト
    """
    # 初期化
    messages = [
        {"role": "system", "content": SYSTEM_PROMPT},
        {"role": "user", "content": user_input}
    ]
    execution_log = ExecutionLog()

    log(f"エージェント開始: {user_input}")

    # === メインループ ===
    for step in range(1, MAX_STEPS + 1):
        log(f"--- ステップ {step}/{MAX_STEPS} ---")
        step_start = now()

        # ステップ1：LLMに推論させる
        llm_response = llm.generate(
            model=MODEL,
            messages=messages
        )

        # ステップ2：応答をパースする
        parsed = parse_llm_response(llm_response.content)

        log(f"Thought: {parsed['thought']}")

        # ステップ3：最終回答かどうかを判定する
        if parsed["final_answer"] is not None:
            log(f"最終回答: {parsed['final_answer']}")
            execution_log.add_step(
                step_number=step,
                thought=parsed["thought"],
                action=None,
                observation=None,
                duration=now() - step_start
            )

            # 実行ログのサマリーを出力
            log(f"実行完了: {execution_log.get_summary()}")

            return parsed["final_answer"]

        # ステップ4：ツールを実行する
        tool_name = parsed["action"]
        tool_input = parsed["action_input"]

        log(f"Action: {tool_name}({tool_input})")

        tool_result = execute_tool(tool_name, tool_input)

        if tool_result["success"]:
            observation = tool_result["result"]
            log(f"Observation: {observation}")
        else:
            observation = f"エラー: {tool_result['error']}"
            log(f"Observation (エラー): {observation}")

        # ステップ5：結果をメッセージ履歴に追加する
        messages.append({
            "role": "assistant",
            "content": llm_response.content
        })
        messages.append({
            "role": "tool",
            "name": tool_name,
            "content": str(observation)
        })

        # ステップ6：実行ログに記録する
        execution_log.add_step(
            step_number=step,
            thought=parsed["thought"],
            action={"tool": tool_name, "input": tool_input},
            observation=observation,
            duration=now() - step_start
        )

    # === 最大ステップ数に到達 ===
    log(f"警告: 最大ステップ数 ({MAX_STEPS}) に到達しました")

    # 最後にLLMに強制的に最終回答を要求
    messages.append({
        "role": "user",
        "content": "最大ステップ数に到達しました。現在までの情報で最終回答を作成してください。"
    })

    final_response = llm.generate(model=MODEL, messages=messages)
    final_parsed = parse_llm_response(final_response.content)

    return final_parsed.get("final_answer", "タスクを完了できませんでした。")


# --- 実行例 ---
if __name__ == "__main__":
    user_query = "日本の人口と面積から人口密度を計算して、結果をファイルに保存して"

    result = agent_run(user_query)
    print(f"結果: {result}")

    # 期待される実行の流れ：
    # ステップ1: Thought→日本の人口と面積を調べる必要がある
    #            Action→web_search("日本 人口 面積 最新")
    # ステップ2: Thought→人口と面積が分かった。人口密度を計算する
    #            Action→calculate("125000000 / 377975")
    # ステップ3: Thought→計算結果が出た。ファイルに保存する
    #            Action→save_file("population_density.txt", "日本の人口密度: ...")
    # ステップ4: Thought→全て完了した。最終回答を返す
    #            Final Answer→日本の人口密度は約330.7人/km²です。結果をファイルに保存しました。
```

### 問題9：アーキテクチャパターンの比較と選定（チャレンジ）

以下の3つのエージェントアーキテクチャパターンについて、それぞれの動作フローを図示（テキストベース）し、特徴・長所・短所を比較した上で、各パターンが最適なユースケースを提案してください。

1. **ReAct（Reasoning + Acting）**
2. **Plan-and-Execute**
3. **LLM Compiler（並列実行最適化）**

<details>
<summary>ヒント</summary>

3つのパターンの核心的な違いは「いつ計画するか」です。

- **ReAct**：1ステップずつ「考えて→行動して→観察して→また考える」（逐次的・適応的）
- **Plan-and-Execute**：「最初に全体計画を立てて→順番に実行する」（計画先行・構造的）
- **LLM Compiler**：「タスクの依存関係を分析して→並列実行できるものは同時に実行する」（並列最適化）

それぞれのトレードオフ（柔軟性 vs 効率性 vs 速度）に注目して比較しましょう。

===== 1. ReAct（Reasoning + Acting） =====

【動作フロー】

  ユーザー入力
    │
    ▼
  ┌─────────────────────┐
  │  Thought（思考）      │ ◄───────────────┐

       └────┬─────┘                        │
            │ No                            │
            ▼                              │
  ┌─────────────────────┐                  │
  │  Action（行動）       │                  │
  │  「ツールを1つ実行」  │                  │
  └──────────┬──────────┘                  │
             │                              │
             ▼                              │
  ┌─────────────────────┐                  │
  │  Observation（観察）  │                  │
  │  「実行結果を受信」   │──────────────────┘
  └─────────────────────┘

【特徴】
  - 1ステップずつ逐次的に判断・実行するインクリメンタルなアプローチ
  - 各ステップでLLMが状況を再評価し、次の行動を動的に決定
  - 最もシンプルで広く使われているパターン

【長所】
  - 柔軟性が高い（途中で方針変更が容易）
  - 予期しない状況にも適応できる
  - 実装がシンプル
  - 1つのステップの失敗が全体に波及しにくい

【短所】
  - 全体の効率が悪い（1つずつ逐次実行）
  - LLM呼び出し回数が多くなりがち（コスト増）
  - 全体最適な計画が立てられない（近視眼的になる可能性）
  - 同じ試行錯誤を繰り返す場合がある

【最適なユースケース】
  - 対話型のタスク（カスタマーサポート、コーディング支援）
  - 探索的なタスク（調査・リサーチ）
  - タスクの全体像が事前に分からない場合
  - 中間結果に応じて方針を変える必要がある場合

  ユーザー入力
    │
    ▼
  ┌─────────────────────┐

  │  Step 2: ○○         │
  │  Step 3: ○○         │
  │  ...                  │
  └──────────┬──────────┘
             │
             ▼
  ┌─────────────────────┐
  │  Executor（実行）     │ ◄──── 計画の各ステップ

【特徴】
  - 最初に全体計画を立ててから、各ステップを順次実行する
  - 計画（Plan）と実行（Execute）が明確に分離されている
  - 計画の修正（Re-planning）を組み込むことも可能

【長所】
  - 全体を見通した効率的な計画が可能
  - 計画の段階で漏れや矛盾を検出できる
  - 各ステップの役割が明確（デバッグしやすい）
  - ユーザーに事前に計画を提示して承認を得られる

【短所】
  - 初期計画が間違っていた場合の修正コストが大きい
  - 予期しない状況への適応が難しい
  - 計画の策定自体にLLM呼び出しが必要（オーバーヘッド）
  - タスクが不明確な場合、良い計画が立てられない

【最適なユースケース】
  - 手順が明確なタスク（レポート作成、データ処理パイプライン）
  - 複数ステップの業務フロー自動化
  - 事前にユーザーの承認を得たい場合
  - 再現性が求められるタスク

【特徴】
  - タスク間の依存関係を分析し、独立したタスクを並列実行する
  - コンパイラの最適化パスに着想を得たアプローチ
  - 実行時間の最適化に重点を置く

【長所】
  - 実行時間の大幅な短縮（並列実行による）
  - 依存関係の明示的な管理
  - スループットの最大化
  - 独立したタスクの失敗が他のタスクに影響しない（部分的）

【短所】
  - 依存関係の分析が複雑（LLMの判断に依存）
  - 並列実行の管理が実装上難しい
  - タスク間で情報共有が必要な場合に対応しにくい
  - 依存関係の誤認識が致命的な結果を招く可能性

【最適なユースケース】
  - 複数の独立した情報収集（複数ソースからのデータ取得）
  - バッチ処理（大量のデータの並列処理）
  - 時間制約が厳しいタスク
  - 明確に分離可能なサブタスクの集合

【結論】
  3つのパターンはトレードオフの関係にあり、タスクの特性に応じて選択すべきです。

  - 探索的・対話的なタスク → ReAct
  - 構造的・手順的なタスク → Plan-and-Execute
  - 並列化可能なタスク     → LLM Compiler

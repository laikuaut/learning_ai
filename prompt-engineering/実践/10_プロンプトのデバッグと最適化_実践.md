## 課題の要件

### 5つの問題ケース

以下のプロンプトは、それぞれ特定の問題を抱えています。問題を特定し、修正してください。

---

**ケース1：出力が長すぎる問題**

プロンプト：
```
あなたはマーケティングの専門家です。

新しいフィットネスアプリのキャッチコピーを作成してください。
ターゲットは20代の働く女性です。
健康意識が高く、忙しい日常の中でも運動を習慣化したいと考えています。
アプリの特徴は、AIが個人の体力レベルに合わせたトレーニングメニューを
自動生成する点です。

10文字以内のキャッチコピーを5つ提案してください。
```

症状：キャッチコピーが10文字を大幅に超える。各コピーに長い説明文が付く。

---

**ケース2：出力の一貫性がない問題**

プロンプト：
```
以下の商品レビューの感情を分析してください。

レビュー1：「届くのが早くて助かった」
レビュー2：「思ったより小さかった」
レビュー3：「価格相応かな」
レビュー4：「プレゼントに最適でした！」
レビュー5：「2回目の購入です」
```

症状：実行するたびに分類ラベルが変わる（ポジティブ/肯定的/好意的 等）。フォーマットもばらつく。

---

**ケース3：ハルシネーションが発生する問題**

プロンプト：
```
あなたはIT業界の専門記者です。

2025年に日本で最も注目されたAIスタートアップ企業のトップ5を
具体的な売上金額と従業員数を含めてランキング形式で紹介してください。
各企業について、創業者の経歴と主力製品の詳細スペックも記載してください。
```

症状：実在しない企業名や架空の売上データが生成される。

---

**ケース4：指示の一部が無視される問題**

プロンプト：
```
以下の会議の議事録を作成してください。

参加者：田中（プロジェクトマネージャー）、山田（開発リーダー）、鈴木（デザイナー）

議題1：来月のリリーススケジュールについて
- 田中：「現在の進捗は80%。テスト期間を考えると、リリースは15日が妥当。」
- 山田：「15日は厳しい。バグ修正に時間がかかっている。20日に延期を提案。」
- 鈴木：「デザインは完了済み。どちらの日程でも対応可能。」
- 結論：20日にリリースを延期。田中がクライアントに連絡。

議題2：新機能の優先順位について
- 田中：「クライアントからダッシュボード機能の要望が多い。」
- 山田：「実装工数は約2週間。」
- 鈴木：「UIデザインのプロトタイプは来週完成予定。」
- 結論：ダッシュボード機能を次期バージョンの最優先にする。

出力条件：
1. 議事録テンプレートに従って整形
2. 各議題のアクションアイテムを明確に
3. 参加者ごとの発言を箇条書きで整理
4. 次回会議の推奨日程を提案
5. 議事録全体を英語でも作成
```

症状：条件5（英語版の作成）が無視される、またはアクションアイテムが曖昧になる。

---

**ケース5：出力トーンが不適切な問題**

プロンプト：
```
あなたはカスタマーサポートの担当者です。

以下のクレームに対する返信メールを作成してください。

「先週購入した商品が壊れていました。3日前にサポートに電話しましたが
繋がりませんでした。もう御社の製品は二度と買いません。」

返信メールを書いてください。
```

症状：謝罪が形式的すぎる、または過度に平謝りで具体的な対応策がない。

---

### 各ケースで実施すること

1. **症状の詳細記録**: 期待する出力と実際の出力の差異を具体的に記述
2. **原因の仮説**: プロンプトのどの部分が問題を引き起こしているか分析
3. **デバッグ手法の選択**: 下記のデバッグ手法から適切なものを選んで適用
4. **修正版プロンプトの作成**: 原因に対処した改善版を作成
5. **検証**: 修正版で問題が解消されたか確認

### デバッグ手法一覧

| 手法 | 概要 | 適用場面 |
|------|------|---------|
| **制約強化法** | 出力の制約をより明確・厳密に指定 | 出力が長すぎる、形式が乱れる |
| **例示法** | Few-Shotで期待する出力の具体例を提示 | 一貫性がない、形式がばらつく |
| **分割法** | 1つの大きなタスクを複数に分割 | 指示の一部が無視される |
| **否定制約法** | 「○○しないでください」と明示的に禁止 | 不要な内容が出力される |
| **グラウンディング法** | 回答の根拠の明示を要求 | ハルシネーションが発生する |
| **トーン指定法** | トーンの具体例や禁止表現を明示 | トーンが不適切 |

## ステップガイド

<details>
<summary>ステップ1：症状を正確に記録する</summary>

デバッグの第一歩は「何が問題か」を正確に記録することです。

**記録テンプレート**:

```
ケース番号：[番号]
プロンプト：[対象プロンプトを記載]

【期待する出力】
[こうなってほしいという具体的な出力イメージ]

【実際の出力】
[実際にLLMが返した出力を記録]

【差異の詳細】
- 差異1：[具体的な差異]
- 差異2：[具体的な差異]
- ...

【再現性】
- 同じプロンプトで何回試したか：[回数]
- 毎回同じ問題が起きるか：[はい/いいえ/部分的に]
```

ケース1を例に：
- 期待：「10文字以内のキャッチコピー × 5つ（コピーのみ）」
- 実際：「15〜30文字のコピー + 各コピーに100文字以上の解説」
- 差異：文字数制約の違反、不要な解説の付加

</details>

<details>
<summary>ステップ2：原因を仮説立てる</summary>

各ケースの原因仮説を立ててみましょう。

**ケース1の原因仮説**:
- 仮説A：「10文字以内」の指示が文脈の中に埋もれている（構造的問題）
- 仮説B：LLMが「提案」と聞くと説明を付ける傾向がある（モデルの特性）
- 仮説C：コピーの後に説明を追加しないという否定制約がない

**ケース3の原因仮説**:
- 仮説A：「具体的な売上金額」を要求しているが、正確なデータがLLMの学習データにない
- 仮説B：「紹介してください」という指示が、事実の報告ではなく創作を許容している
- 仮説C：「回答できない場合は正直に伝える」という制約がない

</details>

<details>
<summary>ステップ3：デバッグ手法を適用する</summary>

各ケースに最適なデバッグ手法を選んで適用しましょう。

**ケース1 → 制約強化法 + 否定制約法**:
```
（修正のポイント）
- 「10文字以内」を冒頭に移動し、強調する
- 出力フォーマットを厳密に指定する
- 「解説は不要です」と明示的に禁止する
```

**ケース2 → 例示法（Few-Shot）**:
```
（修正のポイント）
- 分類ラベルを明確に定義する
- 出力フォーマットの具体例を1つ提示する
```

**ケース3 → グラウンディング法**:
```
（修正のポイント）
- 「確実に把握している情報のみ」と制約する
- 「不確実な場合は推定であることを明記」と指示する
- そもそも正確なデータを要求するプロンプト設計自体を見直す
```

</details>

<details>
<summary>ステップ4：修正版を作成し検証する</summary>

修正版を作成したら、以下の手順で検証します。

1. 修正版プロンプトをLLMに入力する
2. 出力が期待に沿っているか確認する
3. 同じプロンプトを3回実行し、一貫性を確認する
4. 元の問題が解消されているか、ステップ1の記録と比較する
5. 新たな問題が発生していないか確認する

**検証記録テンプレート**:
```
【修正版プロンプト】
[修正版を記載]

【適用したデバッグ手法】
[手法名]

【検証結果】
- 元の問題は解消されたか：[はい/部分的/いいえ]
- 3回の実行で一貫性はあるか：[はい/いいえ]
- 新たな問題は発生したか：[なし/あり（内容：...）]
- 総合評価：[成功/部分的成功/失敗]
```

</details>

## 解答例

<details>
<summary>解答例（初心者向け）</summary>

### ケース1：出力が長すぎる問題

**原因分析**:
- 「10文字以内」の制約が文の中盤にあり、LLMが見落としやすい
- 「提案してください」が説明付きの提案を誘発している
- 出力フォーマットが未定義

**修正版プロンプト**:
```
【重要】キャッチコピーのみを出力してください。説明・解説は不要です。

フィットネスアプリのキャッチコピーを5つ作成してください。

制約条件：
- 1つのコピーは10文字以内（厳守）
- ターゲット：20代の働く女性
- アプリの特徴：AI自動トレーニングメニュー生成

出力フォーマット（この形式に厳密に従ってください）：
1. ○○○○○
2. ○○○○○
3. ○○○○○
4. ○○○○○
5. ○○○○○
```

**適用手法**: 制約強化法 + 否定制約法

**検証結果**: 10文字以内のコピーが5つ出力され、余分な説明は付かなくなりました。

---

### ケース2：出力の一貫性がない問題

**原因分析**:
- 分類ラベルが定義されていない（LLMが自由に表現を選んでしまう）
- 出力フォーマットが未指定

**修正版プロンプト**:
```
以下の商品レビューの感情を分析してください。

分類ラベル（必ず以下の3つのいずれかを使用してください）：
- ポジティブ
- ネガティブ
- ニュートラル

出力フォーマット（この形式に厳密に従ってください）：
レビュー: [レビュー原文]
感情: [ラベル]
根拠: [判断理由を1文で]

---

例：
レビュー: とても気に入りました。色もデザインも最高です。
感情: ポジティブ
根拠: 商品への高い満足度を示す表現が複数含まれているため。

---

以下のレビューを分析してください。

レビュー1：「届くのが早くて助かった」
レビュー2：「思ったより小さかった」
レビュー3：「価格相応かな」
レビュー4：「プレゼントに最適でした！」
レビュー5：「2回目の購入です」
```

**適用手法**: 例示法（Few-Shot）+ 制約強化法

**検証結果**: 3回実行してすべて同じラベルと形式で出力されました。

---

### ケース3：ハルシネーションが発生する問題

**原因分析**:
- 「具体的な売上金額と従業員数」をLLMの事前知識から正確に回答するのは困難
- 「紹介してください」が創作的な出力を許容してしまう
- 回答の限界を認める指示がない

**修正版プロンプト**:
```
あなたはIT業界の専門記者です。

日本のAIスタートアップ業界の動向について、以下の形式で記事を作成してください。

重要な注意事項：
- 確実に正しいと分かる情報のみを記載してください
- 具体的な売上金額や従業員数は、情報源を確認できない場合は
  「非公開」または「推定：○○規模」と記載してください
- 不確実な情報には「※推定値」と注記をつけてください

記事の構成：
1. 日本のAIスタートアップ業界の全体動向（200文字）
2. 注目される分野（3つ、各50文字で概要）
3. 今後の展望（100文字）

※ 具体的な企業名を挙げる場合は、一般的に広く知られている企業に限定し、
   正確性に自信がない情報は含めないでください。
```

**適用手法**: グラウンディング法

**検証結果**: 架空のデータが消え、「推定」「非公開」の注記が適切に使用されるようになりました。ただし、プロンプトの目的自体を「ランキング作成」から「業界動向の紹介」に変更しています。

### ケース4：指示の一部が無視される問題（改良版）

**原因分析（詳細）**:

```
プロンプトの構造分析：

[大量の入力データ（議事録の内容）]
   ↕ ← 入力データと出力条件が混在
[5つの出力条件]
   │
   ├─ 条件1〜3：一般的な議事録フォーマットの要求 → 処理されやすい
   ├─ 条件4：推論が必要（新しい情報の生成） → やや処理されにくい
   └─ 条件5：全く別のタスク（翻訳） → 無視されやすい
```

**根本原因**:
1. **タスク過負荷**: 5つの条件の中に、性質の異なるタスク（整形 vs 推論 vs 翻訳）が混在
2. **優先順位の不在**: どの条件が最も重要か示されていない
3. **Lost in the Middle**: 入力データが長いため、末尾の条件への注意が薄れる

**修正戦略**: 分割法を使い、3段階のチェーンに分割する

**修正版プロンプト（ステップ1：議事録作成）**:
```
以下の会議情報から議事録を作成してください。

【会議情報】
参加者：田中（PM）、山田（開発リーダー）、鈴木（デザイナー）
{議題と発言内容}

【出力フォーマット】
## 議事録
- 日時：[日時を記載]
- 参加者：[参加者一覧]

### 議題1：[議題名]
- 発言要旨：
  - [参加者名]：[発言内容を1文で要約]
- 結論：[結論を1文で]
- アクションアイテム：
  - [ ] [担当者]：[具体的なタスク] / 期限：[期限]

（議題2も同様）
```

**修正版プロンプト（ステップ2：次回会議提案）**:
```
以下の議事録に基づいて、次回会議の日程と議題を提案してください。

【議事録】
{ステップ1の出力}

提案条件：
- 今回の会議から1〜2週間後の日程
- アクションアイテムの進捗確認を必ず含める
- 所要時間の見積もりも記載
```

**修正版プロンプト（ステップ3：英語版作成）**:
```
以下の日本語の議事録を英語に翻訳してください。

翻訳ルール：
- ビジネス英語のフォーマルなトーン
- 人名はローマ字表記（田中→Tanaka）
- 専門用語は英語の一般的な表記を使用

【議事録】
{ステップ1の出力}
```

### ケース5：出力トーンが不適切な問題（改良版）

**原因分析（詳細）**:

```
問題の層別分析：

トーンの問題は3つのレベルで発生しうる：

Level 1: 語彙レベル
  → 使用する言葉が不適切（「残念です」vs「大変申し訳ございません」）

Level 2: 構造レベル
  → 謝罪→共感→対応策の流れが不適切

Level 3: 姿勢レベル
  → 企業としての責任感・誠意が伝わらない
```

**修正版プロンプト**:
```
あなたは大手ECサイトのカスタマーサポート責任者です。
顧客満足度の回復を最優先に考えて対応してください。

以下のクレームに対する返信メールを作成してください。

【クレーム内容】
「先週購入した商品が壊れていました。3日前にサポートに電話しましたが
繋がりませんでした。もう御社の製品は二度と買いません。」

【返信メールの構成（この順序を厳守）】
1. 件名（お詫びであることが分かる件名）
2. 冒頭の謝罪（商品の不具合 + 電話が繋がらなかった件の両方に触れる）
3. 問題の認識（お客様が経験された具体的な不便を言い換えて示す）
4. 具体的な対応策（以下から選択して提案）
   - 商品交換（新品を送料無料で発送）
   - 全額返金
   - 交換 + お詫びクーポン
5. 電話サポートの改善について（具体的な対策を1つ述べる）
6. 結びの言葉（今後も利用してもらいたいという誠意）

【トーンの指針】
- 形式的な謝罪文は避け、お客様の状況に寄り添った表現を使う
- 「お客様にはご不便をおかけし」のような定型句は最小限に
- 具体的な対応策を先に提示し、誠意を行動で示す
- 全体で300文字以内

【禁止表現】
- 「通常は〜ですが」（言い訳に聞こえる）
- 「ご理解ください」（一方的な要求に聞こえる）
- 「お客様の方で〜」（責任転嫁に聞こえる）
```

### デバッグフローチャート（改良版）

```
問題の発生
│
├─ Step1: 症状を分類する
│   ├─ 出力の内容が問題 ──→ A
│   ├─ 出力の形式が問題 ──→ B
│   └─ 出力の品質が問題 ──→ C
│
├─ A: 内容の問題
│   ├─ 事実と異なる → グラウンディング法
│   │   └─ プロンプトの設計自体を見直すべきか検討
│   ├─ 指示の一部が無視 → 分割法
│   │   └─ タスクの種類が異なる部分で分割
│   └─ 不要な内容が付加 → 否定制約法
│       └─ 「○○は含めないでください」を追加
│
├─ B: 形式の問題
│   ├─ フォーマットが不安定 → 例示法（Few-Shot）
│   │   └─ 期待する出力形式の具体例を提示
│   ├─ 長さが制御不能 → 制約強化法
│   │   └─ 文字数制約を冒頭に配置 + 否定制約
│   └─ ラベルが不統一 → 制約強化法 + 例示法
│       └─ ラベルの選択肢を明示的に列挙
│
└─ C: 品質の問題
    ├─ トーンが不適切 → トーン指定法
    │   └─ 具体例 + 禁止表現の明示
    ├─ 一貫性がない → 例示法 + 制約強化法
    │   └─ Few-Shot + 出力フォーマット固定
    └─ 深さが不足 → CoT + 構造化指示
        └─ 「ステップバイステップで」+ 各ステップの出力形式指定
```

- 問題の根本原因を「タスク過負荷」「Lost in the Middle」等の概念で分析

- デバッグフローチャートを「症状の分類」から始まる体系的な構造に

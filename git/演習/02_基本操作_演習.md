## 基本問題

### 問題1: git add と git commit の基本

練習用リポジトリを作成し、`hello.txt`（内容:「Hello, Git!」）をコミットしてください。

期待される出力例：
```
[main (root-commit) abc1234] 最初のファイルを追加
 1 file changed, 1 insertion(+)
```

<details>
<summary>ヒント</summary>

`git add <ファイル>` → `git commit -m "メッセージ"` の順です。
</details>

<details>
<summary>解答</summary>

```bash
$ mkdir practice && cd practice
$ git init -b main
$ echo "Hello, Git!" > hello.txt
$ git add hello.txt                    # ステージングエリアに登録
$ git commit -m "最初のファイルを追加"   # リポジトリに記録
```

- 「変更 → add → commit」がGitの基本サイクルです
</details>

### 問題2: git status の3つの状態

以下の3状態を再現し、`git status` の出力の違いを確認してください。

1. `memo.txt` を新規作成した直後（未追跡）
2. `git add` した後（ステージ済み）
3. ステージ後にさらに編集した状態（ステージ済み＋未ステージ）

期待される出力例（状態3）：
```
Changes to be committed:
        new file:   memo.txt
Changes not staged for commit:
        modified:   memo.txt
```

<details>
<summary>ヒント</summary>

同じファイルが「ステージ済み」と「未ステージ」の両方に表示されることがあります。
</details>

<details>
<summary>解答</summary>

```bash
$ echo "メモ1" > memo.txt && git status    # Untracked files に表示
$ git add memo.txt && git status           # Changes to be committed に表示
$ echo "メモ2" >> memo.txt && git status   # 両方のセクションに表示
# → このままコミットすると「メモ1」だけが記録されます
```
</details>

### 問題3: git diff で差分を確認する

1. `profile.txt`（内容:「名前: 田中」）をコミットする
2. 「名前: 田中太郎」に変更し、`git diff` / `git diff --staged` の違いを確認する

期待される出力例：
```diff
-名前: 田中
+名前: 田中太郎
```

<details>
<summary>ヒント</summary>

`git diff` は未ステージの差分、`git diff --staged` はステージ済みの差分です。
</details>

<details>
<summary>解答</summary>

```bash
$ echo "名前: 田中" > profile.txt
$ git add profile.txt && git commit -m "プロフィールを追加"
$ echo "名前: 田中太郎" > profile.txt
$ git diff              # add 前の差分が表示される
$ git add profile.txt
$ git diff              # 何も表示されない
$ git diff --staged     # add 済みの差分が表示される
```

- `-` が削除行、`+` が追加行です
- `--staged` と `--cached` は同じ意味です
</details>

## 応用問題

### 問題4: 複数ファイルを段階的にコミットする

4ファイルを論理単位ごとに3回に分けてコミットしてください。

| コミット | ファイル | メッセージ |
|---|---|---|
| 1回目 | `README.md` | `docs: プロジェクト説明を追加` |
| 2回目 | `index.html`, `style.css` | `feat: HTMLとCSSの雛形を追加` |
| 3回目 | `app.js` | `feat: JavaScript初期ファイルを追加` |

期待される出力例（`git log --oneline`）：
```
ccc3333 feat: JavaScript初期ファイルを追加
bbb2222 feat: HTMLとCSSの雛形を追加
aaa1111 docs: プロジェクト説明を追加
```

<details>
<summary>ヒント</summary>

`git add` でファイルを個別に指定すれば、コミット単位を自由に制御できます。
</details>

<details>
<summary>解答</summary>

```bash
# ファイルをすべて作成
$ echo '# Website' > README.md
$ echo '<html><body>Hello</body></html>' > index.html
$ echo 'body { margin: 0; }' > style.css
$ echo 'console.log("start");' > app.js

# 論理単位ごとにコミット
$ git add README.md && git commit -m "docs: プロジェクト説明を追加"
$ git add index.html style.css && git commit -m "feat: HTMLとCSSの雛形を追加"
$ git add app.js && git commit -m "feat: JavaScript初期ファイルを追加"
```

- `git add .` は意図しないファイルが含まれるリスクがあるため、明示的な指定を推奨します
</details>

### 問題5: コミット履歴の確認と検索

問題4の続きで `git log` の5つのオプションを試してください。

1. `--oneline`（1行表示）
2. `-2`（直近2件）
3. `--stat`（変更ファイルと変更量）
4. `--grep="feat"`（メッセージ検索）
5. `--graph --all`（グラフ表示）

期待される出力例（4）：
```
ccc3333 feat: JavaScript初期ファイルを追加
bbb2222 feat: HTMLとCSSの雛形を追加
```

<details>
<summary>ヒント</summary>

オプションは組み合わせ可能です。例: `git log --oneline --graph --all`
</details>

<details>
<summary>解答</summary>

```bash
$ git log --oneline                  # 1行表示
$ git log --oneline -2               # 直近2件
$ git log --oneline --stat           # 変更ファイルと変更量
$ git log --oneline --grep="feat"    # メッセージ検索（正規表現も可）
$ git log --oneline --graph --all    # グラフ表示
```

- 現場では `git log --oneline --graph --all` をエイリアスに登録して使うことが多いです
</details>

## チャレンジ問題

### 問題6: 変更の取り消しとコミットメッセージ規約

ステージングの取り消しとワーキングツリーの復元を総合的に練習してください。

1. `app.py` に `greet` 関数を書いてコミットする
2. `farewell` 関数を追加して `git add` する
3. **ステージングを取り消す**（変更は残す）→ `git status` で確認
4. **変更自体を破棄**してコミット時点に戻す
5. 改めて `farewell` 関数を追加し、Conventional Commits 形式でコミットする

期待される出力例（`git log --oneline`）：
```
bbb2222 feat: farewell関数を追加
aaa1111 feat: greet関数を追加
```

<details>
<summary>ヒント</summary>

- ステージング取り消し: `git restore --staged <ファイル>`
- 変更の破棄: `git restore <ファイル>`（元に戻せないので注意）
</details>

<details>
<summary>解答</summary>

```bash
# 1. greet 関数をコミット
$ cat <<'EOF' > app.py
def greet(name):
    return f"Hello, {name}!"
EOF
$ git add app.py && git commit -m "feat: greet関数を追加"

# 2-3. farewell 追加 → add → ステージング取り消し
$ echo -e '\ndef farewell(name):\n    return f"Goodbye, {name}!"' >> app.py
$ git add app.py
$ git restore --staged app.py     # ステージング取り消し（変更は残る）
$ git status                      # → "Changes not staged" に表示

# 4. 変更を破棄（コミット時点に戻る）
$ git restore app.py

# 5. 改めて追加してコミット
$ echo -e '\ndef farewell(name):\n    return f"Goodbye, {name}!"' >> app.py
$ git add app.py && git commit -m "feat: farewell関数を追加"
```

- `git restore --staged` は add の取り消し、`git restore` は変更の破棄です
- Git 2.23以降の推奨コマンドで、従来の `git reset HEAD` / `git checkout --` に代わります
</details>

## 3.1 スケジュール管理の基本

スケジュール管理とは、**プロジェクトの作業を時間軸に配置し、期限内に完了させる**ための管理活動。

### スケジュール管理のステップ

```
1. アクティビティの定義（何をするか）
   ↓
2. 順序の設定（どの順番でやるか）
   ↓
3. 所要時間の見積もり（どれくらいかかるか）
   ↓
4. スケジュールの作成（いつやるか）
   ↓
5. スケジュールのコントロール（計画通りか）
```

## 3.2 ガントチャート

### ガントチャートとは

作業を横棒（バー）で表し、時間軸に沿って配置した図表。プロジェクト全体の見通しを視覚的に把握できる。

### ガントチャートの例

```
タスク              | Week1 | Week2 | Week3 | Week4 | Week5 | Week6
--------------------|-------|-------|-------|-------|-------|------
要件定義            | ████  |       |       |       |       |
DB設計              |       | ████  |       |       |       |
API設計             |       | ████  |       |       |       |
フロントエンド設計   |       |  ███  |       |       |       |
API実装             |       |       | ████  | ██    |       |
フロントエンド実装   |       |       |  ███  | ████  |       |
結合テスト          |       |       |       |       | ████  |
リリース準備        |       |       |       |       |   ██  | ██
★ MVPリリース       |       |       |       |       |       | ◆
```

### ガントチャートの利点と限界

**利点：**
- 全体像が一目で分かる
- 期間の重なりが見える
- ステークホルダーへの説明に便利
- マイルストーンが明確

**限界：**
- タスク間の依存関係が見えにくい
- アジャイルの反復的な開発には合わない場合がある
- メンテナンスに手間がかかる

### ツール

- **GitHub Projects** - タイムラインビュー
- **Mermaid記法** - ドキュメント内でガントチャートを記述
- **Notion** - タイムラインビュー
- **Microsoft Project** - 本格的なガントチャート
- **Miro / FigJam** - ビジュアルコラボレーション

## 3.3 マイルストーン

### マイルストーンとは

プロジェクトの**重要な節目や達成点**。作業そのものではなく、特定の状態に到達したことを示す。

### マイルストーンの設定例

```
フェーズ1：基盤構築
  ◆ M1: 技術選定完了（4/15）
  ◆ M2: 開発環境構築完了（4/22）

フェーズ2：MVP開発
  ◆ M3: コア機能実装完了（5/31）
  ◆ M4: 内部レビュー完了（6/7）

フェーズ3：品質担保
  ◆ M5: E2Eテスト完了（6/21）
  ◆ M6: ステージング検証完了（6/28）

フェーズ4：リリース
  ◆ M7: 本番リリース（7/1）
  ◆ M8: リリース後安定稼働確認（7/15）
```

### 良いマイルストーンの条件

- **明確な完了条件**がある（曖昧でない）
- **検証可能**である（達成したかどうか判定できる）
- **ステークホルダーにとって意味がある**
- **適切な間隔**で設定されている（2〜4週間に1つが目安）

## 3.4 タスクの依存関係

### 依存関係の4タイプ

| タイプ | 意味 | 例 |
|--------|------|-----|
| **FS（Finish to Start）** | Aが終わったらBを開始 | DB設計 → API実装 |
| **SS（Start to Start）** | Aが始まったらBも開始可能 | API実装 & フロントエンド実装 |
| **FF（Finish to Finish）** | Aが終わるときBも終わる | コーディング & コードレビュー |
| **SF（Start to Finish）** | Aが始まったらBを終了 | めったに使わない |

### 実務で最もよく使うパターン

```
FS（最も一般的）:
  要件定義 ──FS──→ 設計 ──FS──→ 実装

SS（並行作業）:
  API実装 ──SS──→ フロントエンド実装
  （APIの一部ができたらフロント実装を開始）

FF:
  実装 ──FF──→ 単体テスト
  （実装の完了とテストの完了を同期）
```

### 依存関係の種類

| 種類 | 説明 | 例 |
|------|------|-----|
| **必須依存** | 技術的に避けられない | DB構築後にデータ投入 |
| **任意依存** | ベストプラクティスだが必須ではない | 設計レビュー後に実装開始 |
| **外部依存** | 外部の成果物に依存 | 外部APIの提供を待つ |

## 3.5 クリティカルパス

### クリティカルパスとは

プロジェクトの開始から完了までの**最長経路**。クリティカルパス上のタスクが1日遅れると、プロジェクト全体が1日遅れる。

### クリティカルパスの算出例

```
        ┌─── B(3日) ───┐
        │               │
A(2日)──┤               ├── E(2日) ── F(1日)
        │               │
        └─── C(5日) ───┘
                │
                └─── D(3日) ──┘

パス1: A → B → E → F = 2+3+2+1 = 8日
パス2: A → C → E → F = 2+5+2+1 = 10日 ★クリティカルパス
パス3: A → C → D → F = 2+5+3+1 = 11日 ★★最長パス（真のクリティカルパス）
```

### クリティカルパスの活用

- クリティカルパス上のタスクに**最も注意**を払う
- クリティカルパスを**短縮**する方法を検討する
  - タスクの並列化
  - リソースの追加投入
  - スコープの縮小
- **フロート（余裕時間）**のあるタスクは多少遅れても全体に影響しない

### フロート（スラック）

```
クリティカルパスでないタスクの余裕時間：

タスクB: フロート = 10日 - 8日 = 2日
→ Bは2日遅れてもプロジェクト全体には影響しない
```

## 3.6 見積もり手法

### PERT（Program Evaluation and Review Technique）

3点見積もり法。楽観値・最頻値・悲観値から期待値を算出する。

```
期待値 = (楽観値 + 4×最頻値 + 悲観値) ÷ 6
```

#### 例：ログイン機能の実装

```
楽観値（Optimistic）: 2日（すべて順調なら）
最頻値（Most Likely）: 4日（通常なら）
悲観値（Pessimistic）: 10日（問題が発生したら）

期待値 = (2 + 4×4 + 10) ÷ 6 = 28 ÷ 6 ≈ 4.7日

標準偏差 = (悲観値 - 楽観値) ÷ 6 = (10 - 2) ÷ 6 ≈ 1.3日
```

### ストーリーポイント

タスクの**相対的な大きさ・複雑さ**を表す数値。時間ではなく、労力を相対的に評価する。

```
フィボナッチ数列を使うことが多い：
1, 2, 3, 5, 8, 13, 21, ...

基準（例）：
  1ポイント: 単純なテキスト変更、設定変更
  2ポイント: 小さなバグ修正、単純な機能追加
  3ポイント: 標準的な機能実装
  5ポイント: やや複雑な機能、複数コンポーネントにまたがる
  8ポイント: 複雑な機能、不確実性がある
  13ポイント: 非常に複雑、分割を検討すべき
```

### プランニングポーカー

チーム全員で見積もりを行う手法。

```
手順：
1. ストーリーを説明する
2. 各メンバーがカード（ポイント）を伏せて出す
3. 一斉にオープンする
4. 最大と最小の人が理由を説明する
5. 議論後、再度見積もる
6. 合意に達するまで繰り返す
```

**ポイント：**
- 最初に声に出すと「アンカリング効果」で偏るので、同時にオープンする
- 完全な一致を目指す必要はない（近い値なら大きい方を採用等）
- 見積もりは「正確さ」より「チームの認識合わせ」が目的

### Tシャツサイズ

ストーリーポイントよりざっくりした見積もり。初期段階のロードマップ作成に便利。

```
XS: 数時間
S:  1〜2日
M:  3〜5日
L:  1〜2週間
XL: 2週間以上（分割を検討）
```

## 3.7 ベロシティ

### ベロシティとは

チームが**1スプリントで完了できるストーリーポイントの量**。過去の実績から算出する。

### ベロシティの計測

```
Sprint 1: 完了 18ポイント
Sprint 2: 完了 22ポイント
Sprint 3: 完了 20ポイント
Sprint 4: 完了 24ポイント
Sprint 5: 完了 21ポイント

平均ベロシティ = (18+22+20+24+21) ÷ 5 = 21ポイント/Sprint
```

### ベロシティの活用

```
残りのバックログ: 126ポイント
平均ベロシティ: 21ポイント/Sprint
スプリント期間: 2週間

予想残りスプリント数: 126 ÷ 21 = 6スプリント
予想完了日: 約12週間後
```

### ベロシティに関する注意点

- **チーム間で比較しない**（チームごとに基準が違う）
- **ベロシティを目標にしない**（数字を膨らませるインセンティブが生まれる）
- **安定するまで3〜4スプリントかかる**（初期は参考値）
- **チーム構成が変わると変動する**

## 3.8 バッファ管理

### バッファの種類

| 種類 | 説明 | 目安 |
|------|------|------|
| **タスクバッファ** | 各タスクに含める余裕 | 見積もりの10〜20% |
| **プロジェクトバッファ** | プロジェクト末尾に設ける余裕 | 全体の20〜30% |
| **合流バッファ** | 並行作業の合流点に設ける余裕 | クリティカルパスの50% |

### CCPM（クリティカルチェーンプロジェクトマネジメント）

個々のタスクからバッファを取り除き、プロジェクト全体にまとめてバッファを設ける手法。

```
従来：各タスクにバッファを含む
  A(5日) → B(8日) → C(4日) = 17日
  （各タスクに1〜2日のバッファを含む）

CCPM：バッファを集約
  A(3日) → B(6日) → C(3日) + プロジェクトバッファ(5日) = 17日
  （個々のタスクは最頻値で見積もり、バッファを末尾に集約）
```

### バッファ消費率の監視

```
プロジェクト進捗: 50%完了
バッファ消費: 30%

→ 順調（進捗 > バッファ消費）

プロジェクト進捗: 50%完了
バッファ消費: 70%

→ 危険信号（バッファ消費が進捗を大幅に上回る）
```

## 3.9 スケジュール遅延への対処

### 遅延の早期発見

- **デイリースタンドアップ**で進捗を確認
- **バーンダウンチャート**で傾向を把握
- **ブロッカー**を早期に検知

### 遅延時の対処法

| 手法 | 説明 | リスク |
|------|------|--------|
| **クラッシング** | リソースを追加投入して短縮 | コスト増、人月の神話 |
| **ファストトラッキング** | 順序的だったタスクを並列化 | 手戻りリスク |
| **スコープ削減** | 優先度の低い機能を削る | ステークホルダーの合意が必要 |
| **品質調整** | テスト範囲の絞り込み等 | 品質リスク（最終手段） |

> **人月の神話（ブルックスの法則）**: 遅延しているプロジェクトに人を追加しても、コミュニケーションコストが増え、さらに遅延する可能性がある。

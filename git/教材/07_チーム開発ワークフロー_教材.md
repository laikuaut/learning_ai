# 第7章：チーム開発ワークフロー

## この章のゴール

- 代表的なブランチ戦略（Git Flow、GitHub Flow、GitLab Flow、トランクベース開発）を理解する
- プロジェクトの規模や性質に応じて適切なワークフローを選択できるようになる
- コードレビューやコミットメッセージの規約など、チーム開発の実践的なルールを習得する

---

## 7.1 Git Flow

Git Flow（ギットフロー）は、Vincent Driessen 氏が2010年に提唱したブランチ戦略（branching strategy）です。リリースサイクルが明確なプロジェクトに適しています。

### Git Flow のブランチ構成

```
main (本番リリース用)
  │
  │  hotfix/urgent-fix ──→ main & develop にマージ
  │         │
  ├─────────┤
  │         │
  │   ┌─────┴──────────────────────────────────┐
  │   │           develop (開発用メイン)          │
  │   │              │                           │
  │   │    feature/   │    feature/              │
  │   │    login ─────┤    search ───────┐       │
  │   │              │                  │       │
  │   │              │    release/      │       │
  │   │              ├──→ v1.0 ───┐    │       │
  │   │              │            │    │       │
  │   │              │     バグ修正 │    │       │
  │   │              │            │    │       │
  │   └──────────────┴────────────┴────┴───────┘
  │                                │
  ├────────────────────────────────┘
  │  ← release ブランチから main にマージ（タグ付け）
  ▼
```

### 5つのブランチの役割

| ブランチ | 寿命 | 役割 |
|---|---|---|
| `main` | 永続 | 本番リリース済みのコードを管理 |
| `develop` | 永続 | 開発中の最新コードを管理 |
| `feature/*` | 一時的 | 新機能の開発用 |
| `release/*` | 一時的 | リリース準備用（バグ修正のみ） |
| `hotfix/*` | 一時的 | 本番の緊急バグ修正用 |

### Git Flow の操作手順

```bash
# --- feature ブランチの作成と開発 ---
# develop から feature ブランチを作成
git checkout develop
git checkout -b feature/login

# 機能を開発してコミット
git add .
git commit -m "feat: ログイン画面を作成"
git commit -m "feat: ログイン認証処理を実装"

# develop にマージ
git checkout develop
git merge --no-ff feature/login
git branch -d feature/login

# --- release ブランチの作成 ---
# develop から release ブランチを作成
git checkout develop
git checkout -b release/v1.0.0

# バージョン番号の更新やバグ修正のみ行う
git commit -m "chore: バージョンを 1.0.0 に更新"

# main にマージしてタグを付ける
git checkout main
git merge --no-ff release/v1.0.0
git tag -a v1.0.0 -m "バージョン 1.0.0 リリース"

# develop にもマージ
git checkout develop
git merge --no-ff release/v1.0.0
git branch -d release/v1.0.0

# --- hotfix ブランチ（緊急修正） ---
git checkout main
git checkout -b hotfix/fix-crash

git commit -m "fix: アプリ起動時のクラッシュを修正"

# main と develop の両方にマージ
git checkout main
git merge --no-ff hotfix/fix-crash
git tag -a v1.0.1 -m "緊急修正 v1.0.1"

git checkout develop
git merge --no-ff hotfix/fix-crash
git branch -d hotfix/fix-crash
```

> **よくある間違い：** `--no-ff`（no fast-forward）を忘れると、マージコミットが作成されず、どのブランチからマージされたかの情報が失われます。Git Flow では必ず `--no-ff` を使いましょう。

### ポイントまとめ

- Git Flow はリリースサイクルが明確なプロジェクトに適しています
- 5つのブランチをルールに従って運用します
- ブランチが多く管理が複雑なため、小規模プロジェクトにはオーバースペックな場合があります

## 7.2 GitHub Flow

GitHub Flow（ギットハブフロー）は、GitHub社が提唱したシンプルなブランチ戦略です。Webアプリケーションなど、頻繁にデプロイするプロジェクトに適しています。

### GitHub Flow の流れ

```
main (常にデプロイ可能な状態)
  │
  ├── feature/add-login ──→ PR ──→ レビュー ──→ マージ ──→ デプロイ
  │
  ├── fix/header-style ───→ PR ──→ レビュー ──→ マージ ──→ デプロイ
  │
  ├── feature/search ─────→ PR ──→ レビュー ──→ マージ ──→ デプロイ
  │
  ▼
```

### GitHub Flow の6つのルール

1. **main ブランチは常にデプロイ可能**な状態を保つ
2. **作業は main から新しいブランチを作成**して行う
3. **定期的にリモートにプッシュ**して進捗を共有する
4. **プルリクエストを作成**してレビューを依頼する
5. **レビュー後にマージ**する
6. **マージしたらすぐにデプロイ**する

### GitHub Flow の操作手順

```bash
# 1. main から作業ブランチを作成
git checkout main
git pull origin main
git checkout -b feature/add-search

# 2. 開発してコミット
git add .
git commit -m "feat: 検索機能を追加"

# 3. リモートにプッシュ
git push origin feature/add-search

# 4. GitHub 上で Pull Request を作成
# 5. レビューを受けて修正があれば追加コミット
git add .
git commit -m "fix: レビュー指摘の修正"
git push origin feature/add-search

# 6. レビュー承認後、GitHub 上でマージ
# 7. ローカルのブランチを削除
git checkout main
git pull origin main
git branch -d feature/add-search
```

### Git Flow との比較

| 項目 | Git Flow | GitHub Flow |
|---|---|---|
| ブランチ数 | 5種類 | 2種類（main + feature） |
| 複雑さ | 高い | 低い |
| デプロイ頻度 | リリース単位 | マージごと |
| 適したプロジェクト | パッケージソフト、モバイルアプリ | Webアプリ、SaaS |

### ポイントまとめ

- GitHub Flow はシンプルで理解しやすいワークフローです
- main ブランチを常にデプロイ可能な状態に保つことが最重要です
- 小〜中規模のWebアプリケーション開発に最適です

## 7.3 GitLab Flow

GitLab Flow（ギットラブフロー）は、GitHub Flow に「環境ブランチ（environment branch）」の概念を加えたワークフローです。

### 環境ブランチを使うパターン

```
main (開発用メイン)
  │
  ├── feature/xxx ──→ MR ──→ main にマージ
  │
  ▼
main ──→ staging (ステージング環境)
            │
            ▼
         staging ──→ production (本番環境)

マージの方向: main → staging → production（上流から下流へ）
```

### リリースブランチを使うパターン

モバイルアプリのように複数バージョンを同時にサポートする場合に使います。

```
main (開発用メイン)
  │
  ├──→ release/1.0 (v1.0系のメンテナンス)
  │       │
  │       ├── cherry-pick でバグ修正を取り込む
  │       │
  ├──→ release/2.0 (v2.0系のメンテナンス)
  │       │
  │       ├── cherry-pick でバグ修正を取り込む
  │       │
  ▼
```

### GitLab Flow の操作手順

```bash
# 1. main で開発
git checkout main
git pull origin main
git checkout -b feature/notification

git add .
git commit -m "feat: 通知機能を追加"
git push origin feature/notification
# MR を作成して main にマージ

# 2. staging にデプロイ
git checkout staging
git merge main
git push origin staging
# staging 環境で動作確認

# 3. production にデプロイ
git checkout production
git merge staging
git push origin production
# 本番環境にデプロイ
```

### ポイントまとめ

- GitLab Flow は環境ごとにブランチを持つことで、デプロイの流れを明確にします
- 上流（main）から下流（production）への一方向のマージが基本です
- 複数環境を管理するプロジェクトに適しています

## 7.4 トランクベース開発（Trunk-based Development）

トランクベース開発（Trunk-based Development）は、全員が1つのメインブランチ（trunk = main）に直接、または非常に短命なブランチを通じてコミットする手法です。Google や Facebook などの大規模企業で採用されています。

### トランクベース開発の流れ

```
main (trunk)
  │
  ├─ 直接コミット（小さな変更の場合）
  │
  ├─ feature/xxx ──→ 1〜2日以内にマージ
  │
  ├─ 直接コミット
  │
  ├─ feature/yyy ──→ 1日以内にマージ
  │
  ▼
  （常にリリース可能な状態を維持）
```

### トランクベース開発のルール

1. **ブランチの寿命は最大1〜2日**（長くても3日）
2. **フィーチャーフラグ（feature flag）** で未完成の機能を隠す
3. **自動テスト**が充実していることが前提
4. **小さな単位**でこまめにマージする

### フィーチャーフラグの例

```python
# フィーチャーフラグ（feature flag）の例
# 未完成の機能をコードに含めつつ、ユーザーには見せない

FEATURE_FLAGS = {
    "new_search": False,      # 開発中：まだ有効にしない
    "dark_mode": True,        # リリース済み：有効
    "notification": False,    # 開発中：まだ有効にしない
}

def show_search():
    if FEATURE_FLAGS["new_search"]:
        return new_search_page()
    else:
        return old_search_page()
```

### ポイントまとめ

- トランクベース開発はブランチの寿命を極力短くする手法です
- フィーチャーフラグと自動テストが前提条件です
- 高いCI/CDの成熟度が求められますが、マージコンフリクトが減るメリットがあります

## 7.5 各ワークフローの比較と使い分け

### 比較表

| 項目 | Git Flow | GitHub Flow | GitLab Flow | トランクベース |
|---|---|---|---|---|
| 複雑さ | 高 | 低 | 中 | 低 |
| ブランチ寿命 | 長い | 中程度 | 中程度 | 非常に短い |
| デプロイ頻度 | 低 | 高 | 中〜高 | 非常に高 |
| チーム規模 | 中〜大 | 小〜中 | 中〜大 | 中〜大 |
| 適したプロジェクト | パッケージソフト | Webアプリ | 複数環境管理 | 大規模Web |
| 前提条件 | リリース計画 | CI/CD | CI/CD | 高度なCI/CD |

### 選択のフローチャート

```
リリースサイクルが明確か？
├── はい → バージョン管理が重要か？
│          ├── はい → Git Flow
│          └── いいえ → GitLab Flow
└── いいえ → チームのCI/CD成熟度は？
              ├── 高い → トランクベース開発
              └── 普通 → GitHub Flow
```

## 7.6 コードレビューのベストプラクティス

コードレビュー（code review）は、チーム開発で品質を担保するための重要なプロセスです。

### レビュアーのチェックリスト

| カテゴリ | チェック項目 |
|---|---|
| 機能 | 要件を満たしているか |
| バグ | エッジケースの考慮漏れはないか |
| 可読性 | コードが読みやすいか |
| 命名 | 変数名・関数名が適切か |
| テスト | テストが十分に書かれているか |
| セキュリティ | 脆弱性がないか |
| パフォーマンス | 明らかな性能問題がないか |

### レビューコメントの良い例と悪い例

```
【悪い例】
「ここ、ダメです」
「なんでこう書いたんですか？」

【良い例】
「この部分、null チェックを追加すると安全性が向上すると思います。
 例: if (user != null) { ... }」

「この関数、処理が30行を超えているので、
 validateInput() と processData() に分割すると
 読みやすくなりそうです。いかがでしょうか？」
```

### レビュー対応側のマナー

- 指摘には感謝の気持ちで対応する
- 修正した場合は「修正しました」とコメントで報告する
- 意見が合わない場合は冷静に理由を説明する

### ポイントまとめ

- コードレビューは品質向上だけでなく、チームの知識共有にも貢献します
- 指摘は理由とともに、提案形式で書きましょう
- レビューは「攻撃」ではなく「協力」です

## 7.7 コミットメッセージの規約（Conventional Commits）

Conventional Commits（コンベンショナルコミッツ）は、コミットメッセージの書き方を標準化する規約です。

### フォーマット

```
<type>(<scope>): <description>

[optional body]

[optional footer]
```

### type の種類

| type | 意味 | 例 |
|---|---|---|
| `feat` | 新機能 | feat: ユーザー登録機能を追加 |
| `fix` | バグ修正 | fix: ログイン時のエラーを修正 |
| `docs` | ドキュメント | docs: README を更新 |
| `style` | コードスタイル | style: インデントを修正 |
| `refactor` | リファクタリング | refactor: 認証ロジックを整理 |
| `test` | テスト | test: ログインのテストを追加 |
| `chore` | ビルドやツール | chore: ESLint の設定を更新 |
| `ci` | CI設定 | ci: GitHub Actions を追加 |
| `perf` | パフォーマンス改善 | perf: クエリを最適化 |

### 実践例

```bash
# 新機能の追加
git commit -m "feat(auth): ソーシャルログイン機能を追加

Google と GitHub アカウントでのログインに対応。
OAuth2.0 を使用した認証フローを実装。

Closes #34"

# バグ修正
git commit -m "fix(cart): 数量が0のとき合計金額が NaN になる問題を修正"

# 破壊的変更（BREAKING CHANGE）
git commit -m "feat(api)!: レスポンス形式を JSON:API 仕様に変更

BREAKING CHANGE: レスポンスの data フィールドの構造が変更されます。
既存のクライアントは更新が必要です。"
```

### ポイントまとめ

- Conventional Commits はコミット履歴の可読性を大幅に向上させます
- `type` を統一することで、変更の種類が一目でわかります
- 自動的な CHANGELOG 生成やバージョニングにも活用できます

## 7.8 チーム開発の実践的なルール

### ブランチ命名規則

```
feature/   ← 新機能開発
fix/       ← バグ修正
hotfix/    ← 緊急修正
docs/      ← ドキュメント
refactor/  ← リファクタリング
test/      ← テスト追加

例:
  feature/add-user-registration
  fix/login-error-on-mobile
  docs/update-api-reference
```

### PR / MR の粒度

```
【良い例：小さく分割】
PR #1: データベースのスキーマを追加
PR #2: API エンドポイントを実装
PR #3: フロントエンドの画面を実装
PR #4: テストを追加

【悪い例：巨大な PR】
PR #1: ユーザー管理機能をすべて実装（変更3000行）
  → レビューに時間がかかり、品質が下がる
```

> **よくある間違い：** ブランチを長期間マージせずに放置すると、main との差分が大きくなり、マージ時に大量のコンフリクトが発生します。ブランチの寿命は短く保ちましょう（目安: 数日以内）。

## 7.9 プロテクトブランチ（Protected Branch）の設定

プロテクトブランチ（protected branch）は、重要なブランチへの直接変更を制限する機能です。

### GitHub での設定

Settings → Branches → Add branch protection rule で設定します。

```
推奨設定:
  ✅ Require a pull request before merging
    ✅ Require approvals: 1人以上
  ✅ Require status checks to pass before merging
    ✅ Require branches to be up to date before merging
  ✅ Do not allow bypassing the above settings
```

### GitLab での設定

Settings → Repository → Protected branches で設定します。

```
推奨設定:
  Branch: main
  Allowed to merge: Maintainers
  Allowed to push and merge: No one（直接 push を禁止）
  Allowed to force push: No（強制 push を禁止）
```

## 7.10 CODEOWNERS ファイル

CODEOWNERS（コードオーナーズ）ファイルは、コードの各部分の責任者を定義するファイルです。PRやMRで変更されたファイルに応じて、自動的にレビュアーがアサインされます。

### CODEOWNERS の書き方

```bash
# .github/CODEOWNERS（GitHub の場合）
# .gitlab/CODEOWNERS（GitLab の場合）

# デフォルトのオーナー（全ファイル）
* @tech-lead

# フロントエンドのコード
/src/frontend/ @frontend-team
*.css @frontend-team
*.tsx @frontend-team

# バックエンドのコード
/src/backend/ @backend-team
*.py @backend-team

# インフラ関連
/terraform/ @infra-team
Dockerfile @infra-team
docker-compose.yml @infra-team

# ドキュメント
/docs/ @docs-team
*.md @docs-team

# セキュリティ関連（セキュリティチームのレビュー必須）
/src/auth/ @security-team
/src/crypto/ @security-team
```

### ポイントまとめ

- CODEOWNERS でコードの責任者を明確にしましょう
- 変更箇所に応じて適切なレビュアーが自動アサインされます
- セキュリティに関わるコードは専門チームのレビューを必須にしましょう

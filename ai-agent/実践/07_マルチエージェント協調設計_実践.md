## 課題の要件

1. **「技術ブログ記事の自動作成システム」** を題材に、マルチエージェントシステムを設計すること
   - ユーザーがトピックを指定すると、調査→執筆→レビュー→修正→公開の一連のフローを自動実行
2. 以下の**エージェント群**を設計すること（最低4体）
   - オーケストレータ（進行管理・タスク割り振り）
   - リサーチャー（情報収集・調査）
   - ライター（記事執筆）
   - レビュアー（品質チェック・フィードバック）
3. 各エージェントの**ロール定義**を作成すること（名前、責任範囲、使えるツール、入出力）
4. エージェント間の**通信プロトコル**を定義すること（メッセージ形式、送受信ルール）
5. **ワークフロー全体図**をアスキーアートで作成すること（分岐・ループを含む）
6. **エラーハンドリング**（エージェントがタスクに失敗した場合の対応）を設計すること

## ステップガイド

<details>
<summary>ステップ1：エージェントの役割を定義する</summary>

各エージェントについて以下を明確にしましょう。

| 項目 | 記述内容 |
|------|---------|
| 名前 | エージェントの識別名 |
| 責任範囲 | 何をするか / 何をしないか |
| 入力 | 他のエージェントから何を受け取るか |
| 出力 | 他のエージェントに何を渡すか |
| ツール | 使えるツールの一覧 |
| 判断基準 | どうなったら「完了」とみなすか |

**よくある間違い**: エージェントの責任範囲が曖昧で重複してしまうこと。例えば「リサーチャーが記事の下書きまで書いてしまう」「レビュアーが大幅に書き直してしまう」など。各エージェントの「やること」と「やらないこと」を明確にしましょう。

</details>

<details>
<summary>ステップ2：通信プロトコルを設計する</summary>

エージェント間でやりとりするメッセージの形式を統一しましょう。

基本的なメッセージ形式の例:

```json
{
  "from": "researcher",
  "to": "orchestrator",
  "type": "task_complete",
  "payload": {
    "task_id": "research_001",
    "result": "...",
    "confidence": 0.85
  },
  "timestamp": "2026-04-05T10:30:00Z"
}
```

通信パターン:
- **リクエスト-レスポンス**: オーケストレータがタスクを割り振り、エージェントが結果を返す
- **パブリッシュ-サブスクライブ**: 特定のイベントを全エージェントに通知
- **パイプライン**: 前のエージェントの出力が次のエージェントの入力になる

</details>

<details>
<summary>ステップ3：ワークフローを設計する</summary>

ブログ記事作成の全体フローを設計します。

重要なのは「レビュー → フィードバック → 修正」のループがあることです。

```
開始 → リサーチ → 執筆 → レビュー → 修正 → 再レビュー → 完成
                                ↑        ↓
                                └────────┘（ループ、最大3回）
```

ループの終了条件（例: レビューOK or 最大ループ回数到達）も定義しましょう。

</details>

<details>
<summary>ステップ4：エラーハンドリングを設計する</summary>

各ステップで発生しうるエラーと対応を整理しましょう。

| エラーケース | 影響 | 対応策 |
|------------|------|--------|
| リサーチャーが情報を見つけられない | 執筆の質が低下 | 検索クエリを変えてリトライ（最大3回）。それでもダメなら、情報不足の旨をユーザーに報告 |
| ライターが長文を生成してしまう | レビュー負荷の増大 | 文字数制限をプロンプトで設定。超過したら自動で要約を依頼 |
| レビュアーが常に不合格にする | 無限ループ | レビューループは最大3回。3回目は「要改善点リスト付きで公開」とする |
| エージェントがタイムアウトする | 全体フローの停止 | タイムアウト検知 → オーケストレータが再割り当て |

</details>

<summary>解答例（初心者向け）</summary>

### 1. エージェント定義

#### オーケストレータ（Orchestrator）

| 項目 | 内容 |
|------|------|
| 名前 | orchestrator |
| 責任範囲 | 全体の進行管理、タスクの割り振り、進捗監視、エラーハンドリング |
| 入力 | ユーザーからのトピック指定、各エージェントからのタスク完了報告 |
| 出力 | 各エージェントへのタスク指示、ユーザーへの進捗報告 |
| ツール | なし（他エージェントの調整のみ） |
| 判断基準 | 全ステップが完了し、最終レビューがOKとなったら完了 |

#### リサーチャー（Researcher）

| 項目 | 内容 |
|------|------|
| 名前 | researcher |
| 責任範囲 | トピックに関する情報収集・整理。情報の要約・構造化 |
| 入力 | トピック名、調査の観点（オーケストレータから） |
| 出力 | 調査結果レポート（事実、数値データ、参考URL） |
| ツール | web_search, read_url, summarize |
| 判断基準 | 十分な情報量（5件以上の情報源）が集まったら完了 |

#### ライター（Writer）

| 項目 | 内容 |
|------|------|
| 名前 | writer |
| 責任範囲 | 調査結果に基づくブログ記事の執筆・修正 |
| 入力 | 調査結果レポート（リサーチャーから）、レビューコメント（レビュアーから） |
| 出力 | ブログ記事（Markdown形式） |
| ツール | なし（LLMの生成能力のみ） |
| 判断基準 | 記事が完成し、文字数が1,000〜3,000字の範囲内 |

#### レビュアー（Reviewer）

| 項目 | 内容 |
|------|------|
| 名前 | reviewer |
| 責任範囲 | 記事の品質チェック（正確性、読みやすさ、SEO） |
| 入力 | ブログ記事（ライターから）、調査結果（参照用） |
| 出力 | レビュー結果（合格/要修正 + 修正コメント） |
| ツール | check_grammar, check_facts |
| 判断基準 | 全チェック項目を確認し、合否を判定したら完了 |

### 2. 通信プロトコル

```json
{
  "message_format": {
    "id": "msg_001",
    "from": "エージェント名",
    "to": "エージェント名",
    "type": "task_request | task_complete | feedback | error",
    "payload": {},
    "timestamp": "ISO 8601形式"
  }
}
```

**メッセージタイプ一覧**:

| タイプ | 送信元 | 送信先 | 内容 |
|--------|--------|--------|------|
| task_request | orchestrator | 各エージェント | タスクの割り振り |
| task_complete | 各エージェント | orchestrator | タスク完了報告 |
| feedback | reviewer | writer | 修正コメント |
| error | 各エージェント | orchestrator | エラー報告 |

### 3. ワークフロー図

```
ユーザー: 「AIエージェントの最新動向」で記事を書いて
    │
    ▼
[orchestrator] タスク分解
    │
    ├──→ [researcher] へ task_request
    │        │
    │        ▼ Web検索・情報収集
    │        │
    │    ◄── task_complete（調査結果レポート）
    │
    ├──→ [writer] へ task_request + 調査結果
    │        │
    │        ▼ 記事執筆
    │        │
    │    ◄── task_complete（記事ドラフト）
    │
    ├──→ [reviewer] へ task_request + 記事 + 調査結果
    │        │
    │        ▼ 品質チェック
    │        │
    │    ◄── task_complete（レビュー結果）
    │
    ├── レビュー結果の判定
    │    ├── 合格 → 完了（ユーザーに記事を返す）
    │    └── 要修正 → [writer] にフィードバックを送信
    │                    │
    │                    ▼ 記事修正
    │                    │
    │                ◄── task_complete（修正記事）
    │                    │
    │              ※ レビューループ（最大3回）
    │
    ▼
ユーザーに最終記事を返す
```

### 4. エラーハンドリング

| エラー | 検知方法 | 対応 |
|--------|---------|------|
| リサーチ失敗 | 結果が0件 | クエリを変えてリトライ（最大3回） |
| 記事が文字数超過 | payload内の文字数チェック | ライターに「N文字以内に要約して」と再依頼 |
| レビュー不合格ループ | ループカウンター ≥ 3 | 現状の記事に「要確認」タグを付けてユーザーに返す |
| タイムアウト | 60秒以内にレスポンスなし | エラーログを記録し、ユーザーに状況を通知 |

### 2. 状態管理マシン（State Machine）

```python
# 擬似コード: ワークフロー状態管理

class WorkflowState:
    INITIALIZED = "initialized"
    RESEARCHING = "researching"
    WRITING = "writing"
    EDITING = "editing"
    REVIEWING = "reviewing"
    REVISING = "revising"
    COMPLETED = "completed"
    FAILED = "failed"

class BlogWorkflow:
    def __init__(self, topic, user_id):
        self.topic = topic
        self.user_id = user_id
        self.state = WorkflowState.INITIALIZED
        self.review_count = 0
        self.max_reviews = 3
        self.artifacts = {}  # 各ステップの成果物を保持

    # 状態遷移テーブル
    TRANSITIONS = {
        "initialized":  {"start":            "researching"},
        "researching":  {"research_done":    "writing",
                         "research_failed":  "researching"},  # リトライ
        "writing":      {"draft_done":       "editing"},
        "editing":      {"edit_done":        "reviewing"},
        "reviewing":    {"review_passed":    "completed",
                         "review_failed":    "revising"},
        "revising":     {"revision_done":    "editing"},      # 再度エディターへ
    }

    def transition(self, event):
        """状態遷移を実行する"""
        current = self.state
        if event not in self.TRANSITIONS.get(current, {}):
            raise InvalidTransition(f"{current} → {event} は無効な遷移です")

        if event == "review_failed":
            self.review_count += 1
            if self.review_count >= self.max_reviews:
                self.state = WorkflowState.COMPLETED  # 強制完了
                self.artifacts["warning"] = "レビュー上限に達しました"
                return

        self.state = self.TRANSITIONS[current][event]
```

### 5. エラーハンドリング（改良版）

```python
# 擬似コード: エラーハンドリング with リトライ・フォールバック

class OrchestratorErrorHandler:
    def handle_error(self, error, workflow):
        """エラーの種類に応じて適切な対応を取る"""

        if error.type == "agent_timeout":
            # タイムアウト → リトライ（最大2回）
            if error.retry_count < 2:
                return Action(
                    type="retry",
                    target=error.agent,
                    delay_seconds=5
                )
            else:
                return Action(
                    type="notify_user",
                    message=f"{error.agent}が応答しません。手動介入が必要です。"
                )

        elif error.type == "quality_below_threshold":
            # 品質不足 → フィードバック付きリトライ
            if workflow.review_count < workflow.max_reviews:
                return Action(
                    type="send_feedback",
                    target="writer",
                    feedback=error.review_comments
                )
            else:
                return Action(
                    type="complete_with_warning",
                    message="品質基準を完全には満たしていませんが、"
                            "レビューの上限に達しました。"
                )

        elif error.type == "research_insufficient":
            # 調査不足 → クエリを変えてリトライ
            return Action(
                type="retry_with_alternative",
                target="researcher",
                alternative_queries=generate_alternative_queries(
                    workflow.topic
                )
            )
```

**プロの知見**: マルチエージェントシステムの最大の落とし穴は「デバッグの困難さ」です。エージェントが増えると、問題がどこで発生したのかの特定が非常に難しくなります。そのため、(1) すべてのメッセージをログに記録する、(2) 各エージェントの入出力を明確に型定義する、(3) まず2体のエージェント構成で動かしてから段階的にエージェントを追加する、という手順を踏むことが重要です。

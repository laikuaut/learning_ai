## 基本問題

### 問題1：要件の分類

以下の要件を「機能要件」「非機能要件」「ビジネス要件」に分類してください。

1. ユーザーがメールアドレスでアカウント登録できる
2. ページの表示速度が2秒以内であること
3. 月間売上を前年比20%向上させる
4. 管理者が商品の価格を変更できる
5. システムの稼働率が99.9%以上であること
6. カスタマーサポートへの問い合わせを30%削減する
7. パスワードはbcryptでハッシュ化して保存する
8. ユーザーが購入履歴を一覧で確認できる

**期待される出力例：**

```
機能要件：1, 4, 8
非機能要件：2, 5, 7
ビジネス要件：3, 6
```

<details>
<summary>ヒント</summary>

- 機能要件：システムが「何をするか」（ユーザーの操作や機能）
- 非機能要件：システムが「どのように動くか」（性能、セキュリティ、可用性）
- ビジネス要件：ビジネス上の目標（売上、コスト削減等の数値目標）

</details>

<details>
<summary>解答例</summary>

```
■ 機能要件（システムが何をするか）
  1. ユーザーがメールアドレスでアカウント登録できる
  4. 管理者が商品の価格を変更できる
  8. ユーザーが購入履歴を一覧で確認できる

■ 非機能要件（システムがどのように動くか）
  2. ページの表示速度が2秒以内（性能要件）
  5. システムの稼働率が99.9%以上（可用性要件）
  7. パスワードはbcryptでハッシュ化（セキュリティ要件）

■ ビジネス要件（ビジネス上の目標）
  3. 月間売上を前年比20%向上
  6. カスタマーサポートへの問い合わせを30%削減
```

</details>

### 問題2：MoSCoW優先度付け

あなたは社内タスク管理ツールのPMです。以下の要件にMoSCoW（Must / Should / Could / Won't）の優先度をつけてください。初回リリースまで3ヶ月、エンジニア3名です。

1. タスクの作成・編集・削除
2. タスクへの担当者アサイン
3. 期限の設定とリマインダー通知
4. カンバンボード表示
5. ガントチャート表示
6. Slack連携（通知）
7. AIによるタスク自動分類
8. ファイル添付機能
9. ユーザー認証（ログイン）
10. ダークモード対応

**期待される出力例：**

```
Must:  1, 2, 9
Should: 3, 4
Could:  6, 8
Won't:  5, 7, 10
```

<details>
<summary>ヒント</summary>

Mustは「これがないとリリースできない」コア機能です。Mustは全体の60%以下に抑えましょう。3ヶ月・3名のリソースで実現可能かも考慮してください。

</details>

<details>
<summary>解答例</summary>

```
Must（必須 - これがないとタスク管理ツールとして成立しない）
  1. タスクの作成・編集・削除  → コア機能
  2. タスクへの担当者アサイン  → チームで使うための必須機能
  9. ユーザー認証（ログイン）  → セキュリティ上必須

Should（重要 - 強く望まれるがMVPでは妥協可能）
  3. 期限の設定とリマインダー通知  → タスク管理の基本だが、期限のみ先行実装し
                                     リマインダーは後から追加可能
  4. カンバンボード表示  → 視覚的な管理に重要だが、一覧表示で代替可能

Could（あると嬉しい - 余裕があれば対応）
  6. Slack連携  → 利便性向上だが必須ではない
  8. ファイル添付機能  → あると便利だがコメントにURL記載で代替可能

Won't（今回はやらない - 将来検討）
  5. ガントチャート表示  → 実装コストが高く、初回リリースでは不要
  7. AIによるタスク自動分類  → 技術的に不確実性が高く、フェーズ2以降
  10. ダークモード対応  → UXの改善だが優先度は低い
```

</details>

### 問題3：ユーザーストーリーの作成

以下のシステム要件を、ユーザーストーリーの形式（「[ユーザー種別]として、[達成したいこと]をしたい。なぜなら[理由・価値]だからだ。」）で書き直してください。

1. 商品レビュー投稿機能
2. 管理者向けダッシュボード
3. パスワードリセット機能

**期待される出力例：**

```
1. 購入者として、商品にレビューを投稿したい。
   なぜなら、他の購入者に自分の体験を共有したいからだ。
```

<details>
<summary>解答例</summary>

```
1. 購入者として、購入した商品にレビューと星評価を投稿したい。
   なぜなら、商品の使用感を他の購入検討者に共有し、
   購入判断の参考にしてもらいたいからだ。

2. 店舗管理者として、売上・注文数・人気商品をダッシュボードで
   一覧確認したい。
   なぜなら、日々の売上状況を素早く把握し、
   販売戦略の意思決定に活かしたいからだ。

3. 登録ユーザーとして、パスワードを忘れた場合にメール経由で
   リセットしたい。
   なぜなら、パスワードを忘れてもアカウントに
   アクセスできる状態を維持したいからだ。
```

**よくある間違い：**
```
× ユーザーとして、データベースをPostgreSQLにしたい。
  → 技術的な実装詳細であり、ユーザー価値が不明

× ユーザーとして、すべての機能を使いたい。
  → 漠然としすぎて、具体的なアクションに落とせない
```

</details>

### 問題4：WBSの作成

以下のプロジェクトについて、WBSを作成してください。第2レベル（主要カテゴリ）と第3レベル（作業パッケージ）まで分解してください。

**プロジェクト：** 社内向けFAQシステムの構築

要件：
- 社員がFAQを検索・閲覧できる
- 管理者がFAQを登録・編集・削除できる
- カテゴリによるFAQの分類
- 「役に立った」ボタンによる評価機能

<details>
<summary>ヒント</summary>

WBSは成果物ベースで分解します。「フロントエンド」「バックエンド」「インフラ」のような技術軸ではなく、「FAQ検索機能」「FAQ管理機能」のような機能軸で第2レベルを作ると分かりやすくなります。

</details>

<details>
<summary>解答例</summary>

### 問題5：スコープクリープへの対応

以下のシナリオを読んで、PMとしてどう対応するか述べてください。

**シナリオ：** あなたはECサイト構築プロジェクトのPMです。開発フェーズに入って2週間が経過した時点で、以下の追加要望が届きました。

- 営業部長：「ポイント還元機能を入れてほしい。競合がやっている。」
- デザイナー：「トップページのアニメーションをもっとリッチにしたい。」
- 開発リーダー：「今のうちにマイクロサービス化した方がいい。将来のために。」

**期待される出力例：**

```
各要望について、影響分析を行い、変更管理プロセスに乗せるか
判断する。すべてを無条件に受け入れるのではなく、
スコープ・スケジュール・コストへの影響を明確にした上で
ステークホルダーと合意する。
```

<details>
<summary>解答例</summary>

```
【対応方針】
すべての要望を変更管理プロセスに乗せ、影響を分析した上で判断する。

■ 営業部長：ポイント還元機能
  影響分析：
  ・スコープ：DB設計変更、ポイント管理API、UI変更が必要
  ・スケジュール：+3〜4週間
  ・コスト：+約500万円
  対応：
  → 変更要求書を起票し、ビジネス価値（競合対抗）と
    コスト・スケジュール影響をPOに提示する。
  → 推奨：フェーズ2に含める。理由は開発途中での
    DB設計変更はリスクが高いため。

■ デザイナー：トップページのアニメーション強化
  影響分析：
  ・スコープ：フロントエンドの実装変更
  ・スケジュール：+2〜3日
  ・コスト：軽微
  ・リスク：パフォーマンスへの影響（表示速度低下の可能性）
  対応：
  → 影響が小さいため、現スプリントで対応可能か検討。
  → ただし性能要件を満たすことが条件。
  → 性能テストの結果次第で採否を判断。

■ 開発リーダー：マイクロサービス化
  影響分析：
  ・スコープ：アーキテクチャの根本的変更
  ・スケジュール：+2〜3ヶ月
  ・コスト：+数千万円規模
  ・リスク：プロジェクト全体の大幅な遅延
  対応：
  → 現時点での採用は不適切。プロジェクトの目的は
    ECサイトの立ち上げであり、アーキテクチャ刷新ではない。
  → モノリスで先にリリースし、トラフィック増加に応じて
    段階的にマイクロサービス化する計画を別途策定する。

【共通の対応】
・すべての要望をバックログに記録する
・優先度と影響度を評価し、ステークホルダーに可視化する
・「やる/やらない」ではなく「いつやるか」のフレームで整理する
```

</details>

## チャレンジ問題

### 問題6：受け入れ基準（Acceptance Criteria）の作成

以下のユーザーストーリーに対して、受け入れ基準を3〜5個作成してください。Given-When-Then形式で記述してください。

**ユーザーストーリー：**
「ECサイトの顧客として、商品をキーワードで検索したい。なぜなら、目的の商品を素早く見つけたいからだ。」

<details>
<summary>ヒント</summary>

Given-When-Then形式：
- Given（前提条件）：〜の状態で
- When（操作）：〜をしたとき
- Then（期待結果）：〜となること

正常系だけでなく、異常系（検索結果0件の場合等）も含めましょう。

</details>

<details>
<summary>解答例</summary>

```
受け入れ基準（Acceptance Criteria）：

AC-1: 商品名での検索
  Given: 商品「ワイヤレスイヤホン」が登録されている状態で
  When: 検索ボックスに「ワイヤレス」と入力して検索ボタンを押したとき
  Then: 「ワイヤレスイヤホン」を含む商品が一覧に表示されること

AC-2: 検索結果0件の場合
  Given: 「存在しない商品名XYZ」が登録されていない状態で
  When: 検索ボックスに「存在しない商品名XYZ」と入力して検索したとき
  Then: 「検索結果が見つかりませんでした」というメッセージが表示されること

AC-3: 検索結果の表示速度
  Given: 商品が10,000件登録されている状態で
  When: 任意のキーワードで検索したとき
  Then: 検索結果が2秒以内に表示されること

AC-4: 空文字での検索防止
  Given: 検索画面が表示されている状態で
  When: 検索ボックスに何も入力せずに検索ボタンを押したとき
  Then: 「キーワードを入力してください」というバリデーションメッセージが
        表示され、検索は実行されないこと

AC-5: 部分一致検索
  Given: 商品「Bluetooth ワイヤレスイヤホン Pro」が登録されている状態で
  When: 検索ボックスに「イヤホン」と入力して検索したとき
  Then: 商品名に「イヤホン」を含む商品がすべて表示されること
```

</details>

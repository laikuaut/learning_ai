# 第1章 演習：MCP概論

## 基本問題

### 問題1-1：MCPの定義（穴埋め）
以下の文章の空欄（ア）〜（オ）に当てはまる語句を答えてください。

> MCP（Model Context Protocol）とは、（ア）が外部の（イ）やツールに接続するための（ウ）なプロトコルです。
> Anthropic社が（エ）年に発表し、AIアプリケーションと外部データソースの間の（オ）問題を解決することを目的としています。

**期待される出力例：**
```
（ア）AIモデル（LLM）
（イ）データソース
（ウ）オープン（標準的な）
（エ）2024
（オ）N×M（統合の複雑性）
```

<details>
<summary>ヒント</summary>

- MCPは特定の企業に縛られない「オープン」な仕様です
- N×M問題とは、N個のAIアプリケーションとM個のツール/データソースの組み合わせが爆発的に増える問題です
- USBがさまざまな周辺機器との接続を統一したのと同じ考え方です

</details>

<details>
<summary>解答例</summary>

```
（ア）AIモデル（LLM）
（イ）データソース
（ウ）オープン（標準的な）
（エ）2024
（オ）N×M（統合の複雑性）
```

**解説：**
- MCPは「AIアプリケーションのUSBポート」と例えられます
- N×M問題とは、N個のAIアプリとM個のツール/データソースがそれぞれ個別に統合すると N×M 通りのコネクタが必要になる問題です
- MCPという共通プロトコルを挟むことで、各AIアプリはMCPクライアントを1つ実装するだけ、各ツールはMCPサーバーを1つ実装するだけで済み、N+M の実装で相互接続が実現します

</details>

### 問題1-2：Host / Client / Serverの役割分類
以下の説明が、MCPの3層構造のうち「Host」「Client」「Server」のどれに該当するかを答えてください。

1. Claude DesktopやVS Code（Copilot）などのAIアプリケーション本体
2. MCPサーバーとの1対1の接続を管理し、プロトコルレベルの通信を処理する
3. 外部のデータベースやAPI、ファイルシステムなどのリソースへのアクセスを提供する
4. ユーザーとの対話インターフェースを提供し、AIモデルの推論を実行する
5. Tools、Resources、Promptsなどの機能（ケーパビリティ）を公開する
6. Host内部に存在し、セキュリティ境界を維持する

**期待される出力例：**
```
1. Host
2. Client
3. Server
4. Host
5. Server
6. Client
```

<details>
<summary>ヒント</summary>

- Hostはエンドユーザーが直接操作するアプリケーション（AIアシスタント等）です
- Clientは仲介役で、HostとServerの間の通信を管理します
- Serverは外部リソースやツールの窓口となるプログラムです
- 1つのHostが複数のClientを持ち、各Clientは1つのServerに接続します

</details>

<details>
<summary>解答例</summary>

```
1. Host   ← ユーザー向けAIアプリケーションそのものです
2. Client ← プロトコル通信の管理はClientの役割です
3. Server ← 外部リソースへのアクセス提供はServerの責務です
4. Host   ← UIの提供とAI推論の実行はHostの役割です
5. Server ← ケーパビリティの公開はServerが行います
6. Client ← Clientはセキュリティ境界を維持しながらHost内に存在します
```

**3層構造の図：**
```
┌──────────────────────────────┐
│  Host（例：Claude Desktop）    │
│  ┌────────┐  ┌────────┐     │
│  │ Client │  │ Client │     │
│  └───┬────┘  └───┬────┘     │
└──────┼───────────┼──────────┘
       │           │
  ┌────┴────┐ ┌────┴────┐
  │ Server  │ │ Server  │
  │(GitHub) │ │(DB)     │
  └─────────┘ └─────────┘
```

</details>

### 問題1-3：Tools / Resources / Promptsの使い分け
以下のユースケースが、MCPの3つの機能「Tools」「Resources」「Prompts」のどれに該当するかを答えてください。

1. GitHubにプルリクエストを作成する
2. データベースのテーブル一覧を読み取り専用で取得する
3. 「コードレビューしてください」という定型的な指示テンプレートを提供する
4. Slackチャンネルにメッセージを投稿する
5. 社内ドキュメントの最新版を参照する
6. 「この文章を要約してください」という再利用可能なプロンプトテンプレートを提供する
7. 外部APIを呼び出してデータを更新する
8. ログファイルの内容をストリーミングで取得する

**期待される出力例：**
```
1. Tools
2. Resources
3. Prompts
4. Tools
5. Resources
6. Prompts
7. Tools
8. Resources
```

<details>
<summary>ヒント</summary>

それぞれの機能の特徴を思い出しましょう：

- **Tools**：モデルが呼び出すアクション。外部に対して何かを「実行」します（副作用あり）
- **Resources**：アプリケーションが参照するデータ。「読み取り専用」で情報を取得します
- **Prompts**：ユーザーが選択する対話テンプレート。再利用可能な「指示の型」を提供します

</details>

<details>
<summary>解答例</summary>

```
1. Tools      ← PRの作成は外部への「アクション実行」です
2. Resources  ← テーブル一覧の読み取り専用取得は「データ参照」です
3. Prompts    ← 定型的な指示テンプレートは「プロンプトテンプレート」です
4. Tools      ← メッセージ投稿は外部への「アクション実行」です
5. Resources  ← ドキュメント参照は「データ参照」です
6. Prompts    ← 再利用可能なプロンプトは「プロンプトテンプレート」です
7. Tools      ← データ更新は外部への「アクション実行」です
8. Resources  ← ログ取得は「データ参照」です
```

**判断のポイント：**
- 「実行する」「作成する」「投稿する」「更新する」→ **Tools**
- 「読み取る」「参照する」「取得する」（読み取り専用）→ **Resources**
- 「テンプレート」「定型指示」「再利用可能なプロンプト」→ **Prompts**

</details>

### 問題1-4：N×M問題の理解
AIアプリケーションが3つ（App A、App B、App C）、外部ツール/データソースが4つ（Tool 1〜4）ある状況を考えます。

(1) MCPを使わない場合、個別の統合（コネクタ）は最大何通り必要ですか。
(2) MCPを導入した場合、必要な実装数の合計はいくつになりますか。
(3) アプリケーションが10個、ツールが20個に増えた場合、それぞれの数を求めてください。

**期待される出力例：**
```
(1) 12通り（3×4）
(2) 7つ（3+4）
(3) MCP未使用：200通り（10×20）、MCP使用：30（10+20）
```

<details>
<summary>解答例</summary>

```
(1) 12通り（3 × 4 = 12）
(2) 7つ（3 + 4 = 7：MCPクライアント3つ + MCPサーバー4つ）
(3) MCP未使用：200通り（10 × 20 = 200）
    MCP使用：30（10 + 20 = 30）
```

**解説：**
- MCPなしの場合、各アプリケーションが各ツールへの専用コネクタを個別に実装するため、N × M 通りの統合が必要です
- MCPありの場合、各アプリはMCPクライアントを1つ、各ツールはMCPサーバーを1つ実装するだけなので N + M で済みます
- スケールするほど効果は顕著です：10×20=200 vs 10+20=30（約85%の削減）

</details>

### 問題1-6：Function CallingとMCPの比較分析
OpenAIのFunction Calling（Tool Use）とMCPを比較し、以下の観点から違いを説明してください。

(1) 標準化の範囲（どこまでをカバーするか）
(2) ツール定義の管理方法
(3) 複数のAIプロバイダー間での互換性
(4) それぞれが適しているユースケース

**期待される出力例（一部）：**
```
(1) 標準化の範囲：
    Function Calling: モデルへのツール定義の渡し方と、モデルからの呼び出し形式を標準化
    MCP: ツールの発見、接続、認証、実行までのエンドツーエンドを標準化
...
```

<details>
<summary>ヒント</summary>

- Function Callingは「AIモデルが関数を呼ぶ仕組み」、MCPは「AIアプリと外部ツールを繋ぐプロトコル」です
- Function Callingではツール定義をAPIリクエストに毎回含める必要があります
- MCPではサーバーが自身のケーパビリティを動的に公開します

</details>

<details>
<summary>解答例</summary>

| 観点 | Function Calling | MCP |
|---|---|---|
| **(1) 標準化の範囲** | モデルAPIレベル（ツール定義の渡し方、モデルの呼び出し形式）に限定 | アプリケーションレベル（接続確立、ケーパビリティ交換、ツール実行、リソース取得まで）をエンドツーエンドで標準化 |
| **(2) ツール定義の管理** | 開発者がAPIリクエストごとにJSON Schemaでツール定義を埋め込む（静的） | MCPサーバーが自身のケーパビリティを動的に公開し、クライアントが実行時に発見する（動的） |
| **(3) 互換性** | 各プロバイダー（OpenAI, Anthropic, Google等）で仕様が微妙に異なる | プロバイダー非依存のオープンプロトコル。MCP対応であれば任意のHost/Serverが相互接続可能 |
| **(4) 適するユースケース** | 単一アプリ内で少数のツールを使う場合、プロトタイピング | 多数のツールやデータソースを複数のAIアプリで共有する場合、エンタープライズ環境 |

**補足：**
- Function CallingとMCPは排他的ではありません。MCPのToolsが呼び出される際、Host内部ではFunction Callingの仕組みを使ってモデルがツール選択を行います
- MCPはFunction Callingの「上位レイヤー」として機能し、ツールの発見・接続・管理を標準化する役割を担います

</details>

### 問題1-8：MCPエコシステムの将来予測
2024年にMCPが発表されて以降、エコシステムは急速に拡大しています。以下の問いについて、あなたの考えを論述してください。

(1) MCPが広く普及するために必要な条件を3つ挙げ、それぞれの理由を説明してください。
(2) MCPが普及した場合、ソフトウェア開発のワークフローはどのように変化すると予測しますか。具体例を2つ挙げて説明してください。
(3) MCPの普及において想定されるリスクや課題を2つ挙げ、対策を提案してください。

**期待される出力例（一部）：**
```
(1) 普及に必要な条件：
  条件1：主要AIプロバイダーの対応
    理由：OpenAI、Google、Anthropic等が対応することで事実上の標準になる...

(2) ワークフローの変化：
  変化1：「ツール統合」が開発プロジェクトの独立したフェーズになる
    具体例：...
```

<details>
<summary>ヒント</summary>

- 過去のプロトコル標準化の成功事例（HTTP、USB、OAuth等）を参考にしましょう
- エコシステムの成長には「供給側」（MCPサーバーの充実）と「需要側」（MCP対応ホストの普及）の両方が必要です
- セキュリティや認証の標準化は普及の重要な要素です

</details>

<details>
<summary>解答例</summary>

**(1) 普及に必要な条件：**

**条件1：主要AIプロバイダーの対応**
OpenAI、Google DeepMind、Meta AI等の主要プロバイダーがMCPをサポートすることで、事実上の業界標準（デファクトスタンダード）となります。HTTP がWebの標準になったように、複数のプレイヤーが採用して初めてプロトコルは普及します。

**条件2：MCPサーバーのエコシステム充実**
npmやPyPIのように、すぐに使えるMCPサーバーが豊富に公開されるレジストリやマーケットプレイスが必要です。開発者が簡単にサーバーを作成・公開できるSDKやツールチェーンの整備も重要です。

**条件3：セキュリティと認証の標準化**
企業が安心して導入できるよう、OAuth 2.0ベースの認証、アクセス制御（どのツールにどの権限でアクセスできるか）、監査ログの標準仕様が必要です。

---

**(2) ワークフローの変化：**

**変化1：「MCP統合エンジニア」という新しい役割の登場**
社内システムのMCPサーバー化を担当する専門的な役割が生まれます。APIエンジニアがREST APIを設計するように、MCPサーバーのTools/Resources/Promptsを適切に設計・運用するスキルが求められるようになります。

**変化2：開発フローのAI駆動化**
例えば、PRを作成すると自動的にMCP経由でコードレビュー、テスト実行、ドキュメント更新が連鎖的に行われるワークフローが実現します。開発者は「何をしたいか」を伝えるだけで、AIがMCPを通じて複数のツールを横断的に操作します。

---

**(3) リスクと対策：**

**リスク1：セキュリティ上の脅威（悪意のあるMCPサーバー）**
信頼できないMCPサーバーに接続した場合、データ漏洩やシステムへの不正操作が発生する可能性があります。
- **対策：** MCPサーバーの署名検証・認証制度の導入、サンドボックス環境での実行、ユーザーの明示的な承認フローの必須化

**リスク2：仕様のフラグメンテーション（断片化）**
拡張仕様やプロバイダー独自実装が乱立し、「MCPに対応しているが実際には互換性がない」状態に陥る可能性があります。
- **対策：** コンプライアンステストスイートの提供、バージョン管理の厳格化、コア仕様と拡張仕様の明確な分離

</details>

# 実践課題08：スコープ変更管理演習 ★3

> **難易度**: ★★★☆☆（中級）
> **前提知識**: 第2章（スコープ管理）、第3章（スケジュール管理）、第4章（コスト管理）、第7章（コミュニケーション管理）
> **課題の種類**: ケーススタディ
> **学習目標**: スコープ変更要求を受けた際に、影響分析を行い、ステークホルダーに対して合理的な判断材料を提示できるようになる

---

## 完成イメージ

以下のような変更管理の一連のドキュメントを作成します。

```
============================================
  変更要求書（Change Request）
  CR番号: CR-2026-003
============================================

■ 要求内容: 多言語対応機能の追加（英語・中国語）
■ 要求者: 営業部長
■ 理由: 訪日外国人客の増加に伴い、多言語対応が急務

■ 影響分析:
  スケジュール: +3週間（全体12週 → 15週）
  コスト:       +120万円（翻訳＋開発工数）
  スコープ:     画面テキストの外部化、言語切替UI追加
  リスク:       翻訳品質、テスト工数の増加

■ 判定: 承認 / 却下 / 延期
■ 推奨: 第2フェーズに延期（MoSCoW: Could）
```

---

## 課題の要件

以下のシナリオで発生する3つの変更要求について、それぞれ影響分析と対応方針を作成してください。

### プロジェクト背景

**プロジェクト概要：**
- 内容：社内ナレッジ共有システム（Wiki型Webアプリ）の開発
- 期間：12週間（現在4週目、要件定義完了・設計フェーズ中）
- 予算：600万円（現時点で150万円消化済み）
- チーム：PM 1名、開発者 3名
- 当初スコープ：記事作成・編集、検索、タグ管理、ユーザー管理

### 変更要求1（小規模）

> **要求者**: エンドユーザー代表
> **内容**: 「記事にファイルを添付できるようにしてほしい。Word、Excel、PDFを想定。」
> **理由**: 「マニュアルや手順書をWikiの記事と一緒に管理したい。」

### 変更要求2（中規模）

> **要求者**: 事業部長（スポンサー）
> **内容**: 「コメント機能とリアクション（いいね）機能を追加してほしい。」
> **理由**: 「一方通行の情報共有ではなく、チーム内の対話を促進したい。」

### 変更要求3（大規模）

> **要求者**: 経営企画部
> **内容**: 「AIによる記事の自動要約・関連記事の推薦機能がほしい。」
> **理由**: 「生成AIを活用して、社内ナレッジの活用度を飛躍的に高めたい。」

### 各変更要求で分析すべきこと

1. **影響分析**: スケジュール・コスト・スコープ・品質・リスクへの影響
2. **優先度評価**: MoSCoW分析での位置づけ
3. **選択肢の提示**: 承認・却下・延期のそれぞれの場合のメリット/デメリット
4. **推奨案**: PMとしての推奨と根拠

## ステップガイド

<details>
<summary>ステップ1：変更の規模を見積もる</summary>

各変更要求の技術的な実装工数を見積もります。

```
見積もりの観点：

【設計への影響】
  - DB変更は必要か？（テーブル追加、カラム追加）
  - 画面の追加・変更は必要か？
  - APIの追加・変更は必要か？

【開発工数】
  - 新規開発: 何人日かかるか？
  - 既存コードの修正: 何人日かかるか？

【テスト工数】
  - 追加テストケース数は？
  - 既存機能への影響テスト（回帰テスト）は必要か？
```

**参考：変更要求1（ファイル添付）の場合**
- DB: ファイル管理テーブルの追加
- 画面: 記事編集画面にアップロードUI追加
- API: ファイルアップロード/ダウンロードAPI追加
- ストレージ: ファイル保存先の設計

</details>

<details>
<summary>ステップ2：トレードオフを分析する</summary>

PMの三角形（スコープ・時間・コスト）のトレードオフを明確にします。

```
変更を受け入れる場合の選択肢：

パターンA: スケジュール延長
  変更を追加し、期限を延ばす
  → メリット: スコープを維持、品質も確保
  → デメリット: ビジネス上の機会損失

パターンB: コスト追加
  人員を追加投入して期限内に収める
  → メリット: スコープ・期限を維持
  → デメリット: 予算超過、新メンバーの立ち上げコスト

パターンC: スコープのトレードオフ
  別の機能を削減して入れ替える
  → メリット: 期限・予算を維持
  → デメリット: 当初予定の機能が削られる

パターンD: 延期
  次のフェーズで対応する
  → メリット: 現在のプロジェクトに影響なし
  → デメリット: 要求者の期待に応えられない
```

</details>

<details>
<summary>ステップ3：推奨案を作成する</summary>

以下の観点で推奨案を判断します。

```
推奨判断のチェックリスト：
  □ ビジネス価値: この変更はユーザーにとって本当に必要か？
  □ 緊急性: 今のリリースに入れないと意味がないか？
  □ リスク: 変更を入れることで品質リスクが高まらないか？
  □ 実現性: 残りの期間・予算で実現可能か？
  □ 影響範囲: 既存の設計・開発への手戻りはどの程度か？
```

</details>

## 解答例

<details>
<summary>解答例（初心者向け）</summary>

### 変更要求1：ファイル添付機能

```
============================================
  変更要求書 CR-001
============================================
■ 要求内容: 記事へのファイル添付機能（Word/Excel/PDF）
■ 要求者: エンドユーザー代表
■ 受付日: 2026-04-06（4週目）

【影響分析】
  スケジュール: +1週間
    - 設計変更: 1人日
    - 開発: 3人日（API + 画面）
    - テスト: 1人日
  コスト: +25万円（5人日 × 5万円/日）
  品質: 低リスク（独立した機能のため既存への影響小）
  技術: ファイルストレージの設計が必要

【MoSCoW評価】
  Should（あるべき）
  → ナレッジ共有システムとしてファイル添付は自然な機能
  → ないと運用で「別の場所にファイルを置いてリンクを貼る」運用になる

【選択肢】
  A. 承認（現フェーズに追加）
     メリット: ユーザーの利便性が大幅向上
     デメリット: 1週間の遅延、25万円の追加コスト

  B. 延期（第2フェーズ）
     メリット: 現在のスケジュール・予算に影響なし
     デメリット: リリース時の利便性が下がる

【推奨】A. 承認
  理由: 工数が小さく、ナレッジ共有の本質的な機能であるため。
  1週間の遅延は許容範囲内。ただし、第1期では
  ファイルサイズ上限（10MB）とファイル種別制限を設ける。
```

### 変更要求2：コメント・リアクション機能

```
============================================
  変更要求書 CR-002
============================================
■ 要求内容: コメント機能 + リアクション（いいね）機能
■ 要求者: 事業部長（スポンサー）
■ 受付日: 2026-04-06（4週目）

【影響分析】
  スケジュール: +2〜3週間
    - 設計変更: 2人日（DBスキーマ、API設計）
    - 開発: 8人日（コメントCRUD + リアクション + 通知）
    - テスト: 3人日
  コスト: +65万円（13人日 × 5万円/日）
  品質: 中リスク（記事画面の大幅変更、通知機能の複雑性）
  技術: リアルタイム通知の実装方式を検討する必要あり

【MoSCoW評価】
  Should〜Could
  → コメント機能はShould（対話促進はビジネス価値が高い）
  → リアクション機能はCould（あれば良いが必須ではない）

【選択肢】
  A. 全部承認
     スケジュール: +3週間、コスト: +65万円

  B. コメント機能のみ承認、リアクションは延期
     スケジュール: +1.5週間、コスト: +40万円

  C. 全部延期
     スケジュール: 影響なし

【推奨】B. コメント機能のみ承認、リアクションは第2フェーズ
  理由: スポンサーの要求の本質は「対話の促進」であり、
  コメント機能だけで目的の大部分は達成できる。
  リアクション機能は実装コストに対して効果が限定的なため、
  第2フェーズでユーザーの声を聞いてから判断しても遅くない。
```

### 変更要求3：AI自動要約・関連記事推薦

```
============================================
  変更要求書 CR-003
============================================
■ 要求内容: AIによる記事の自動要約 + 関連記事推薦
■ 要求者: 経営企画部
■ 受付日: 2026-04-06（4週目）

【影響分析】
  スケジュール: +6〜8週間
    - 技術調査: 5人日（AI API選定、精度検証）
    - 設計: 3人日
    - 開発: 15人日
    - テスト: 5人日（AIの出力品質テスト含む）
  コスト: +200万円以上（開発工数 + AI API利用料）
  品質: 高リスク（AI出力の品質保証が困難）
  技術: 外部AI APIへの依存、プロンプト設計の専門性が必要

【MoSCoW評価】
  Won't（今回は対象外）
  → 技術的な不確実性が高く、現在のスコープとは独立した大きな機能
  → 本来のプロジェクト（ナレッジ共有基盤）の完成が最優先

【選択肢】
  A. 承認 → 非推奨
     プロジェクト期間がほぼ倍に。予算も大幅超過。
     プロジェクトの性質が変わってしまう。

  B. 延期（第2フェーズまたは別プロジェクト）
     本体の完成後、AI活用を別途検討。

  C. 最小限のPoC（概念実証）を実施
     2人日程度でAI APIの検証だけ行い、第2フェーズの判断材料にする。

【推奨】B. 延期 + C. PoC（併用）
  理由: 現在のプロジェクトスコープに含めると、リスクが大幅に
  増大し、本来の目的（ナレッジ共有基盤の構築）が危険にさらされる。
  ただし、経営企画部の関心に応えるため、2人日程度のPoCを
  実施してフィージビリティレポートを作成し、第2フェーズの
  計画材料とすることを提案する。
```

</details>

<details>
<summary>解答例（改良版 ─ 変更管理プロセス全体の設計）</summary>

改良版では、個別の変更要求への対応に加えて、変更管理プロセス自体を設計します。

### 変更管理プロセス定義

```
=============================================
  変更管理プロセス定義書
  プロジェクト: 社内ナレッジ共有システム
=============================================

1. 変更要求の受付
   ─────────────
   - 変更要求は所定のフォーマット（CR票）で受け付ける
   - 口頭のみの依頼は正式な変更要求として受け付けない
   - 受付後24時間以内にCR番号を採番し、要求者に受領を通知する

2. 影響分析
   ─────────────
   分析担当: PM + 開発リーダー
   分析期間: 小規模(2営業日) / 中規模(5営業日) / 大規模(10営業日)

   規模判定基準:
     小規模: 追加工数 5人日以下、既存設計の変更なし
     中規模: 追加工数 6〜20人日、設計変更あり
     大規模: 追加工数 20人日超、アーキテクチャへの影響あり

3. 承認プロセス
   ─────────────
   小規模: PM判断で承認可（事後報告）
   中規模: PM + スポンサーの合意で承認
   大規模: ステアリング委員会での審議・承認

4. 判定基準
   ─────────────
   以下のすべてを検討した上で判定する:
   □ ビジネス価値（ROI）は十分か？
   □ 残り期間で実現可能か？
   □ 残り予算内に収まるか？（マネジメントリザーブの活用可否）
   □ 品質リスクは許容範囲か？
   □ 他の進行中タスクへの影響は？

5. 変更ログ
   ─────────────
   CR番号  受付日      要求内容              規模  判定    判定日
   CR-001  2026-04-06  ファイル添付          小    承認    2026-04-08
   CR-002  2026-04-06  コメント/リアクション  中    一部承認 2026-04-13
   CR-003  2026-04-06  AI要約/推薦           大    延期    2026-04-20
```

### 累積影響サマリ

```
変更承認後のプロジェクト状況：

                  変更前    CR-001  CR-002   累計影響
  期間（週）       12       +1      +1.5     14.5（+20.8%）
  予算（万円）     600      +25     +40      665（+10.8%）
  スコープ         基本4機能  +1機能  +1機能   6機能

  マネジメントリザーブ（当初60万円 = 10%）:
    CR-001: -25万円
    CR-002: -40万円
    残額: -5万円（リザーブ枯渇、追加予算が必要）

  ★ これ以上の変更承認は予算追加なしでは不可能
  → スポンサーに状況を報告し、追加予算の確保または
    スコープの再優先順位付けを依頼する
```

### ステークホルダーへの報告テンプレート

```
件名: 【報告】変更要求 CR-001〜003 の影響分析結果

○○部長（スポンサー）

お疲れ様です。PMの○○です。
今週受領した3件の変更要求について、影響分析を完了しましたので
ご報告いたします。

■ サマリ
  CR-001（ファイル添付）: 承認推奨。+1週、+25万円。
  CR-002（コメント）:     コメントのみ承認推奨。+1.5週、+40万円。
  CR-003（AI要約）:       延期推奨。フィージビリティ調査のみ実施。

■ 全体影響
  期間: 12週 → 14.5週（+2.5週）
  予算: 600万 → 665万（+65万、リザーブ超過5万）

■ ご判断いただきたい事項
  1. CR-001, CR-002の承認可否
  2. 追加予算（65万円）の確保可否
  3. 追加予算が困難な場合、スコープ削減の候補選定

ご都合のよい日時にお打ち合わせをお願いできますでしょうか。
```

**初心者向けとの違い：**
- **プロセス定義**: 個別対応ではなく、再現可能なプロセスとして変更管理を設計しています
- **規模判定基準**: 変更の大きさによって承認プロセスを分けることで、小さな変更を迅速に処理できます
- **累積影響の可視化**: 個々の変更は小さくても、積み重なるとプロジェクトに大きな影響を与えます（スコープクリープ）
- **マネジメントリザーブの管理**: 予備費の残額を追跡し、枯渇前に手を打つのがプロのやり方です
- **報告テンプレート**: 上位者には「判断材料」と「判断事項」を明確に提示します

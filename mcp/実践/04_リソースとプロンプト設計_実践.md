<details>
<summary>ステップ1：3つの機能の判断基準を整理する</summary>

分類の判断基準は以下のフローチャートで考えると明確になります。

```
その機能は外部データを変更するか？
  │
  ├── はい → Tool（ツール）
  │         例：タスク作成、ステータス更新、削除
  │
  └── いいえ → データを読み取るだけか？
                │
                ├── はい → Resource（リソース）
                │         例：設定情報、一覧表示、詳細表示
                │
                └── LLMへの指示テンプレートか？
                    │
                    ├── はい → Prompt（プロンプト）
                    │         例：レポート生成、レビュー依頼文
                    │
                    └── 再検討が必要
```

**補足ルール**:
- **Tool**: 副作用（side effect）がある操作。データの作成・更新・削除
- **Resource**: 読み取り専用のデータ提供。URIでアクセスする
- **Prompt**: LLMに対する指示のテンプレート。ユーザーが明示的に選択する

</details>

<details>
<summary>ステップ2：リソースのURI設計をする</summary>

MCPリソースのURIは、一意にリソースを特定できる形式で設計します。

```
スキーム://パス

例：
  project://config           → プロジェクト設定（静的リソース）
  project://tasks            → 全タスク一覧（動的リソース）
  project://tasks/{task_id}  → 特定タスクの詳細（テンプレートリソース）
  project://members          → メンバー一覧（静的リソース）
```

FastMCPでは2種類のリソースを定義できます。

```python
# 静的リソース（固定URI）
@mcp.resource("project://config")
def get_config() -> str:
    ...

# テンプレートリソース（URIにパラメータを含む）
@mcp.resource("project://tasks/{task_id}")
def get_task(task_id: str) -> str:
    ...
```

</details>

<summary>ステップ3：プロンプトを設計する</summary>

MCPプロンプトは、LLMへの指示テンプレートです。ユーザーが明示的に選択して使います。

```python
from mcp.server.fastmcp import FastMCP
from mcp.server.fastmcp.prompts import base

@mcp.prompt()
def daily_report() -> list[base.Message]:
    """日次レポートを生成するプロンプトです。"""
    return [
        base.UserMessage(
            content="プロジェクトの本日の進捗をもとに日次レポートを作成してください。"
        )
    ]

# 引数付きプロンプト
@mcp.prompt()
def sprint_summary(sprint_number: int) -> list[base.Message]:
    """指定スプリントの振り返りまとめを生成します。"""
    return [
        base.UserMessage(
            content=f"スプリント{sprint_number}の完了タスク、未完了タスク、"
                    f"振り返りポイントをまとめてください。"
        )
    ]
```

### 2. 実装コード

```python
"""
プロジェクト管理MCPサーバー（初心者向け）
学べること：Tools / Resources / Prompts の使い分けと実装
実行方法：uv run project_server.py
"""
from mcp.server.fastmcp import FastMCP
from mcp.server.fastmcp.prompts import base

mcp = FastMCP("project-manager")

# ── サンプルデータ ──
config = {
    "project_name": "ECサイトリニューアル",
    "start_date": "2026-04-01",
    "end_date": "2026-06-30",
    "team_size": 5,
}

members = [
    {"name": "田中", "role": "リーダー"},
    {"name": "佐藤", "role": "フロントエンド"},
    {"name": "鈴木", "role": "バックエンド"},
    {"name": "高橋", "role": "デザイナー"},
    {"name": "伊藤", "role": "QA"},
]

tasks: list[dict] = [
    {"id": 1, "title": "要件定義書作成", "assignee": "田中",
     "status": "完了", "priority": "高"},
    {"id": 2, "title": "DB設計", "assignee": "鈴木",
     "status": "進行中", "priority": "高"},
    {"id": 3, "title": "トップページデザイン", "assignee": "高橋",
     "status": "進行中", "priority": "中"},
]
next_id = 4


# ── Tools（副作用のある操作） ──

@mcp.tool()
def create_task(title: str, assignee: str, priority: str = "中") -> str:
    """タスクを新規作成します。

    Args:
        title: タスクのタイトル
        assignee: 担当者名
        priority: 優先度（"高" / "中" / "低"）
    """
    global next_id
    task = {
        "id": next_id,
        "title": title,
        "assignee": assignee,
        "status": "未着手",
        "priority": priority,
    }
    tasks.append(task)
    next_id += 1
    return f"タスク「{title}」を作成しました（ID: {task['id']}）"


@mcp.tool()
def update_status(task_id: int, status: str) -> str:
    """タスクのステータスを変更します。

    Args:
        task_id: タスクID
        status: 新しいステータス（"未着手" / "進行中" / "完了"）
    """
    for task in tasks:
        if task["id"] == task_id:
            old_status = task["status"]
            task["status"] = status
            return f"タスク「{task['title']}」: {old_status} → {status}"
    return f"エラー: ID {task_id} のタスクが見つかりません"


@mcp.tool()
def delete_task(task_id: int) -> str:
    """指定IDのタスクを削除します。

    Args:
        task_id: 削除するタスクのID
    """
    for i, task in enumerate(tasks):
        if task["id"] == task_id:
            deleted = tasks.pop(i)
            return f"タスク「{deleted['title']}」を削除しました"
    return f"エラー: ID {task_id} のタスクが見つかりません"


# ── Resources（読み取り専用データ） ──

@mcp.resource("project://config")
def get_config() -> str:
    """プロジェクトの設定情報です。"""
    lines = [
        f"プロジェクト名: {config['project_name']}",
        f"期間: {config['start_date']} 〜 {config['end_date']}",
        f"チーム人数: {config['team_size']}名",
    ]
    return "\n".join(lines)


@mcp.resource("project://tasks")
def get_all_tasks() -> str:
    """全タスクの一覧です。"""
    if not tasks:
        return "タスクはまだ登録されていません"
    lines = [f"タスク一覧（{len(tasks)}件）:"]
    for t in tasks:
        lines.append(
            f"  [{t['id']}] {t['title']} "
            f"（{t['assignee']}、{t['status']}、優先度:{t['priority']}）"
        )
    return "\n".join(lines)


@mcp.resource("project://tasks/{task_id}")
def get_task_detail(task_id: str) -> str:
    """特定タスクの詳細情報です。"""
    tid = int(task_id)
    for t in tasks:
        if t["id"] == tid:
            return (
                f"タスク詳細:\n"
                f"  ID: {t['id']}\n"
                f"  タイトル: {t['title']}\n"
                f"  担当者: {t['assignee']}\n"
                f"  ステータス: {t['status']}\n"
                f"  優先度: {t['priority']}"
            )
    return f"ID {task_id} のタスクは見つかりません"


@mcp.resource("project://members")
def get_members() -> str:
    """チームメンバー一覧です。"""
    lines = [f"チームメンバー（{len(members)}名）:"]
    for m in members:
        lines.append(f"  ・{m['name']}（{m['role']}）")
    return "\n".join(lines)


@mcp.resource("project://progress")
def get_progress() -> str:
    """プロジェクトの進捗サマリーです。"""
    total = len(tasks)
    done = sum(1 for t in tasks if t["status"] == "完了")
    in_progress = sum(1 for t in tasks if t["status"] == "進行中")
    todo = sum(1 for t in tasks if t["status"] == "未着手")
    rate = (done / total * 100) if total > 0 else 0
    return (
        f"進捗サマリー:\n"
        f"  全タスク: {total}件\n"
        f"  完了: {done}件 / 進行中: {in_progress}件 / 未着手: {todo}件\n"
        f"  進捗率: {rate:.0f}%"
    )


# ── Prompts（LLMへの指示テンプレート） ──

@mcp.prompt()
def daily_report() -> list[base.Message]:
    """日次レポートを生成するテンプレートです。"""
    return [
        base.UserMessage(
            content=(
                "以下の情報をもとに日次レポートを作成してください：\n"
                "1. 本日の完了タスク\n"
                "2. 進行中のタスク（進捗状況を含む）\n"
                "3. ブロッカーや課題\n"
                "4. 明日の予定\n\n"
                "レポートは箇条書きで簡潔にまとめてください。"
            )
        )
    ]


@mcp.prompt()
def code_review_request() -> list[base.Message]:
    """コードレビュー依頼文を生成するテンプレートです。"""
    return [
        base.UserMessage(
            content=(
                "以下の項目を含むコードレビュー依頼文を作成してください：\n"
                "1. 変更概要（何を、なぜ変更したか）\n"
                "2. 変更ファイル一覧\n"
                "3. テスト方法\n"
                "4. レビューで特に見てほしいポイント\n"
                "5. 影響範囲"
            )
        )
    ]


@mcp.prompt()
def sprint_summary(sprint_number: int) -> list[base.Message]:
    """スプリントの振り返りまとめを生成します。

    Args:
        sprint_number: スプリント番号
    """
    return [
        base.UserMessage(
            content=(
                f"スプリント{sprint_number}の振り返りまとめを作成してください：\n"
                f"1. 完了したタスク一覧と成果\n"
                f"2. 未完了のタスクと理由\n"
                f"3. Good（良かったこと）\n"
                f"4. Problem（問題点）\n"
                f"5. Try（次スプリントで改善すること）"

        )
    ]


@mcp.prompt()
def bug_report_template() -> list[base.Message]:
    """バグ報告テンプレートを生成します。"""
    return [
        base.UserMessage(

                "以下のテンプレートに沿ってバグ報告を作成してください：\n\n"
```

<summary>解答例（改良版）</summary>

改良版では、分類の判断が難しいグレーゾーンのケースについて、設計判断の根拠を深く考察しています。

### グレーゾーンの考察

実務では「Tool か Resource か」の判断が難しいケースがあります。

```
■ ケース：「全タスク一覧の表示」

  Resource派の主張:
    ・データを変更しない読み取り専用の操作
    ・URI (project://tasks) で自然にアクセスできる
    ・LLMがコンテキストとして自動的に読める

  Tool派の主張:
    ・フィルタ条件（ステータス別、担当者別）が必要になりそう
    ・引数を渡す必要があるならToolの方が自然

  → 結論：基本はResourceで提供し、フィルタ付き検索が必要なら
    別途 search_tasks Tool を追加する
```

```
■ ケース：「進捗サマリーの表示」

  Resource派の主張:
    ・集計結果の読み取りなので副作用はない
    ・project://progress というURIで自然

  Prompt派の主張:
    ・「まとめてください」という指示が含まれる？
    → いいえ、生データの集計であり指示テンプレートではない

  → 結論：集計データの提供なのでResource
    LLMに「分析してください」と頼むテンプレートならPrompt
```

### 設計判断フローチャート（改良版）

```
機能を分類する
│
├─ 外部データに変更を加えるか？
│  └─ はい → Tool
│     例：create, update, delete, send, execute
│
├─ データ/情報を提供するだけか？
│  ├─ URIで自然に表現できるか？
│  │  └─ はい → Resource
│  │     例：config, task一覧, メンバー情報
│  │
│  └─ 引数によって大きく結果が変わるか？
│     └─ はい → Tool（検索系）
│        例：search_tasks(status="進行中")
│
├─ LLMへの「指示の型」を定義したいか？
│  └─ はい → Prompt
│     例：レポート生成、レビュー依頼、テンプレート
│
└─ 判断に迷ったら？
   ├─ 副作用がある → Tool（安全側に倒す）
   └─ 副作用がない → Resource（シンプルに保つ）
```

### 改良版コード（追加部分）

```python
# ── フィルタ付きタスク検索はToolとして提供 ──

@mcp.tool()
def search_tasks(
    status: str | None = None,
    assignee: str | None = None,
    priority: str | None = None,
) -> str:
    """条件を指定してタスクを検索します。

    全件取得にはリソース project://tasks を使ってください。
    フィルタリングが必要な場合にこのツールを使います。

    Args:
        status: ステータスでフィルタ（"未着手" / "進行中" / "完了"）
        assignee: 担当者名でフィルタ
        priority: 優先度でフィルタ（"高" / "中" / "低"）
    """
    results = list(tasks)

    if status:
        results = [t for t in results if t["status"] == status]
    if assignee:
        results = [t for t in results if t["assignee"] == assignee]
    if priority:
        results = [t for t in results if t["priority"] == priority]

    if not results:
        return "条件に一致するタスクはありません"

    lines = [f"検索結果: {len(results)}件"]
    for t in results:
        lines.append(
            f"  [{t['id']}] {t['title']} "
            f"（{t['assignee']}、{t['status']}、優先度:{t['priority']}）"
        )
    return "\n".join(lines)


# ── コンテキスト付きプロンプト（改良版） ──

@mcp.prompt()
def daily_report_with_context() -> list[base.Message]:
    """本日の進捗データを含む日次レポート生成テンプレートです。"""
    # プロンプト内でリソースの内容を動的に埋め込む
    progress = get_progress()
    task_list = get_all_tasks()

    return [
        base.UserMessage(
            content=(
                f"以下のプロジェクトデータをもとに日次レポートを作成してください。\n\n"
                f"【進捗データ】\n{progress}\n\n"
                f"【タスク一覧】\n{task_list}\n\n"
                f"レポートには以下を含めてください：\n"
                f"1. 本日の完了タスク\n"
                f"2. 進行中のタスクと進捗\n"
                f"3. ブロッカーや課題\n"
                f"4. 明日の予定"
            )
        )
    ]
```

### 初心者向けとの違い

| ポイント | 初心者向け | 改良版 |
|---------|-----------|--------|
| 分類判断 | 明確なケースのみ | グレーゾーンの考察を含む |
| フローチャート | シンプルな3分岐 | 検索系Toolを含む詳細な分岐 |
| プロンプト | 静的テンプレートのみ | リソースデータを動的に埋め込む |
| Tool/Resource併用 | 明確に分離 | 同じデータを用途別に提供（一覧=Resource、検索=Tool） |

**実務のポイント**: 「ResourceかToolか」で迷ったら、**読み取り専用ならResource、副作用があるならTool**という原則に立ち返りましょう。迷うケースの多くは「フィルタ付き読み取り」で、これはToolとして別途提供するのがベストプラクティスです。

## よくある間違い

| 間違い | 正しい理解 |
|--------|-----------|
| 全機能をToolにしてしまう | 読み取り専用のデータはResourceとして提供しましょう。LLMが自動的にコンテキストとして利用できます |
| ResourceにURIを設計しない | リソースにはURI（例: `project://tasks/{id}`）が必須です。RESTful APIのURL設計と同じ考え方です |
| PromptとToolを混同する | Promptは「LLMへの指示テンプレート」であり、外部処理は行いません。Toolは「外部処理を実行」します |
| Promptに副作用のある処理を入れる | Promptはテキストを返すだけです。データの変更が必要な場合はToolを使いましょう |
| リソースの動的部分をハードコードする | テンプレートリソース（`{task_id}` 等）を使い、パラメータ化しましょう |

### 問題5：APIキーのセキュリティ管理

以下のAPIキーの管理方法について、それぞれの安全性を「安全」「注意が必要」「危険」で評価し、理由を説明してください。

1. 環境変数 `ANTHROPIC_API_KEY` にシェルの設定ファイル（`.bashrc`）で設定する
2. プロジェクトの `.env` ファイルに記載し、`.gitignore` に `.env` を追加する
3. ソースコード内にハードコードする
4. プロジェクトの `README.md` にAPIキーを記載して共有する
5. OSのキーチェーン（macOS）やシークレットマネージャーで管理する

**期待される出力例：**
```
1. 注意が必要 - 他のユーザーが同じマシンを使う場合にキーが見える可能性がある
2. 注意が必要 - .gitignoreの設定忘れでリポジトリに混入するリスクがある
3. 危険 - ソースコードと一緒にバージョン管理に入り、漏洩リスクが非常に高い
4. 危険 - 公開リポジトリの場合、誰でもキーを取得できてしまう
5. 安全 - 暗号化されて保管され、アクセス制御も行われる
```

<details>
<summary>ヒント</summary>

APIキーの管理では「漏洩リスク」と「アクセス制御」の観点で評価します。ソースコードに含まれるものは特に危険です。環境変数は便利ですが、マルチユーザー環境では注意が必要です。

</details>

<details>
<summary>解答例</summary>

```
1. 注意が必要
   → .bashrc自体がバージョン管理やdotfiles共有に含まれる場合に漏洩リスクがあります。
   → 単一ユーザーのローカル環境では比較的安全ですが、共有マシンでは注意が必要です。

2. 注意が必要
   → .gitignoreの設定が正しければ安全ですが、設定忘れや、git addで
     強制追加してしまうと漏洩します。
   → CI/CDパイプラインでは別途シークレット管理が必要です。

3. 危険
   → ソースコードと一緒にリポジトリにコミットされ、履歴にも残るため、
     後から削除しても完全には消えません。
   → 最も一般的で最も危険なアンチパターンです。

4. 危険
   → READMEは通常最初に読まれる公開ファイルであり、リポジトリが
     公開された時点で即座にキーが漏洩します。
   → ボットによる自動スキャンの対象にもなります。

5. 安全
   → OSレベルの暗号化とアクセス制御により保護されます。
   → プログラムからは安全なAPIで取得でき、平文で保存されません。
```

<!-- 解説：
- APIキーの漏洩はクラウド利用料の不正請求や、不正アクセスにつながります。
- ベストプラクティス：
  1. OSのシークレットマネージャーやキーチェーンを使う（最も推奨）
  2. .envファイル + .gitignore（チーム開発では標準的な手法）
  3. CI/CDではGitHub SecretsやAWS Secrets Managerなどを利用する
- 万が一漏洩した場合は、即座にAnthropicのダッシュボードでキーを無効化してください。
-->

</details>

### 問題6：トラブルシューティング

以下のClaude Code使用時のエラーメッセージについて、考えられる原因と対処法をそれぞれ説明してください。

**エラーケース1：**
```
Error: ANTHROPIC_API_KEY is not set
```

**エラーケース2：**
```
Error: Could not connect to Anthropic API
```

**エラーケース3：**
```
npm ERR! engine Unsupported engine
npm ERR! notsup Required: {"node":">=18"}
npm ERR! notsup Actual:  {"node":"v16.20.0"}
```

**エラーケース4：**
```
Error: Rate limit exceeded. Please try again later.
```

**期待される出力例：**
```
エラーケース1：
  原因：APIキーが環境変数に設定されていない
  対処法：export ANTHROPIC_API_KEY="sk-..." を実行するか、.bashrcに追記する

エラーケース2：
  原因：ネットワーク接続の問題、またはプロキシ設定の問題
  対処法：インターネット接続を確認し、プロキシ環境の場合は適切な設定を行う

（以下省略）
```

<details>
<summary>ヒント</summary>

エラーメッセージを読み解く際は、以下の観点で分析してみましょう。
- 認証に関するエラーか？
- ネットワークに関するエラーか？
- 環境（Node.jsバージョンなど）に関するエラーか？
- API利用制限に関するエラーか？

</details>

<details>
<summary>解答例</summary>

```
エラーケース1：ANTHROPIC_API_KEY is not set
  原因：
    - 環境変数 ANTHROPIC_API_KEY が設定されていない
    - シェルの設定ファイルに記載したが、ターミナルを再起動していない
    - 設定ファイルを source していない
  対処法：
    1. export ANTHROPIC_API_KEY="sk-ant-..." を実行する
    2. 永続化するには ~/.bashrc や ~/.zshrc に追記し、source で再読み込みする
    3. echo $ANTHROPIC_API_KEY で正しく設定されているか確認する

エラーケース2：Could not connect to Anthropic API
  原因：
    - インターネット接続が切れている
    - ファイアウォールやプロキシがAPI通信をブロックしている
    - Anthropic APIサーバーが一時的にダウンしている
    - DNS解決に失敗している
  対処法：
    1. curl https://api.anthropic.com で接続確認する
    2. プロキシ環境の場合は HTTP_PROXY / HTTPS_PROXY 環境変数を設定する
    3. VPN利用時は接続先を確認する
    4. Anthropicのステータスページで障害情報を確認する

エラーケース3：npm ERR! engine Unsupported engine
  原因：
    - Node.jsのバージョンが18未満（この例ではv16.20.0）
    - Claude Codeが要求するNode.jsの最低バージョン要件を満たしていない
  対処法：
    1. Node.jsを最新のLTS版にアップデートする
    2. nvm（Node Version Manager）を使用している場合：nvm install 18 && nvm use 18
    3. 公式サイトから最新版をダウンロードしてインストールする

エラーケース4：Rate limit exceeded
  原因：
    - 短時間に大量のAPIリクエストを送信し、レート制限に達した
    - APIプランの利用上限に達した
  対処法：
    1. 数分待ってから再試行する
    2. リクエストの頻度を下げる（大きなタスクを小分けにする）
    3. Anthropicのダッシュボードで利用状況を確認する
    4. 必要に応じて上位プランへのアップグレードを検討する
```

<!-- 解説：
- エラーメッセージを正確に読むことがトラブルシューティングの第一歩です。
- 環境構築時のエラーは大きく4種類に分類できます：
  1. 認証エラー → APIキーの設定を確認
  2. 接続エラー → ネットワーク環境を確認
  3. 環境エラー → Node.jsなどの前提条件を確認
  4. 制限エラー → 利用プランと使用量を確認
- どのエラーも焦らず、メッセージを読んで原因を特定することが重要です。
-->

</details>

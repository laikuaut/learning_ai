# 第1章：SSHの基礎知識

## この章のゴール
- SSH（Secure Shell）の概要と歴史を理解する
- SSHが解決する問題と、なぜ必要なのかを説明できる
- SSHプロトコルの基本的な仕組みを理解する
- SSHのバージョン（SSH-1とSSH-2）の違いを把握する

## 1.1 SSHとは

**SSH（Secure Shell）** は、ネットワーク上のコンピュータに安全にリモート接続するためのプロトコル（通信規約）です。暗号化された通信路を通じて、リモートサーバーへのログインやコマンド実行、ファイル転送を行えます。

### SSHの主な用途

| 用途 | 説明 |
|------|------|
| リモートログイン | 遠隔地のサーバーにログインして操作する |
| コマンド実行 | リモートサーバー上でコマンドを実行する |
| ファイル転送 | SCP/SFTPによる安全なファイル転送 |
| ポートフォワーディング | SSHトンネルを通じて他のサービスに安全にアクセスする |
| Git操作 | GitHubやGitLabとのSSH接続 |

### SSHが使われる場面

```
+------------------+                    +------------------+
|   自分のPC       |   SSH（暗号化）    |   リモートサーバー |
|  (ローカル)      | =================> |  (クラウド/社内)  |
|                  |   ポート22番       |                  |
+------------------+                    +------------------+

例：
- クラウドサーバー（AWS EC2, GCP, Azure）への接続
- 社内開発サーバーへのログイン
- GitHubへのコード push/pull
- VPSの管理・運用
```

## 1.2 SSHの歴史と背景

### SSH以前の世界

SSHが登場する前は、以下のプロトコルがリモート接続に使われていました。

| プロトコル | 問題点 |
|-----------|--------|
| Telnet | 通信が**平文（暗号化なし）**で、パスワードが盗聴される |
| rlogin | 認証が弱く、IPアドレス偽装に脆弱 |
| rsh | 通信が平文、信頼ベースの認証で安全でない |
| FTP | パスワードとデータが平文で流れる |

### 盗聴のリスク（Telnetの場合）

```
+--------+                           +--------+
| クライアント | --- パスワード: abc123 ---> | サーバー |
+--------+         ↑ 平文！          +--------+
                   |
             +----------+
             | 盗聴者    |
             | abc123を  |
             | 取得！    |
             +----------+

Telnetでは通信内容がそのままネットワークを流れるため、
途中で盗み見ることが容易でした。
```

### SSH の誕生

- **1995年**: フィンランドのタトゥ・ウルネン（Tatu Ylönen）が SSH-1 を開発
  - ヘルシンキ工科大学のネットワークでパスワード盗聴事件が発生したことがきっかけ
- **1996年**: SSH Communications Security 社を設立し、商用化
- **2006年**: SSH-2 が RFC 4251〜4256 として標準化
- **現在**: OpenSSH が事実上の標準実装として広く利用されている

## 1.3 SSHプロトコルの仕組み

SSHプロトコルは3つのレイヤーで構成されています。

### SSHの3層構造

```
+------------------------------------------+
|     コネクション層（SSH-CONNECT）         |
|     チャネル多重化、ポートフォワーディング |
+------------------------------------------+
|     認証層（SSH-USERAUTH）                |
|     パスワード認証、公開鍵認証            |
+------------------------------------------+
|     トランスポート層（SSH-TRANS）          |
|     暗号化、完全性検証、鍵交換            |
+------------------------------------------+
|     TCP/IP（ポート22番）                  |
+------------------------------------------+
```

### 各レイヤーの役割

#### トランスポート層（SSH-TRANS）
最下層で、暗号化と通信の完全性を担当します。

```
役割:
1. サーバー認証（接続先が正しいサーバーか確認）
2. 暗号アルゴリズムの合意（どの暗号方式を使うか決める）
3. 鍵交換（通信を暗号化するための共通鍵を安全に生成）
4. データの暗号化・復号
5. データの完全性検証（改ざんされていないか確認）
```

#### 認証層（SSH-USERAUTH）
ユーザーが正しい利用者であることを確認します。

```
主な認証方式:
- パスワード認証: ユーザー名とパスワードで認証（非推奨）
- 公開鍵認証:    鍵ペアによる認証（推奨）
- キーボードインタラクティブ: チャレンジ/レスポンス方式
- GSSAPI:       Kerberos等を使った統合認証
```

#### コネクション層（SSH-CONNECT）
認証済みの暗号化通信路上で、複数の論理チャネルを多重化します。

```
1つのSSH接続上で複数のチャネルを同時利用:
+-- SSH接続（1本のTCPコネクション）--+
|                                    |
|  チャネル1: シェルセッション        |
|  チャネル2: ポートフォワーディング  |
|  チャネル3: SFTPファイル転送        |
|                                    |
+------------------------------------+
```

## 1.4 SSH接続の流れ

SSH接続がどのように確立されるか、ステップごとに見ていきましょう。

### 接続確立の手順

```
クライアント                              サーバー
    |                                        |
    |  1. TCP接続（ポート22番）               |
    |--------------------------------------->|
    |                                        |
    |  2. プロトコルバージョンの交換           |
    |<-------------------------------------->|
    |   "SSH-2.0-OpenSSH_9.6"               |
    |                                        |
    |  3. 鍵交換（Key Exchange）              |
    |<-------------------------------------->|
    |   暗号アルゴリズムの合意                |
    |   共通鍵の生成（Diffie-Hellman等）      |
    |                                        |
    |  4. サーバー認証                        |
    |<-------------------------------------->|
    |   サーバーのホスト鍵を確認              |
    |                                        |
    |  === ここから暗号化通信 ===             |
    |                                        |
    |  5. ユーザー認証                        |
    |--------------------------------------->|
    |   パスワード or 公開鍵                  |
    |                                        |
    |  6. セッション開始                      |
    |<-------------------------------------->|
    |   シェル、コマンド実行など               |
    |                                        |
```

### 初回接続時のホスト鍵確認

初めてサーバーに接続するとき、以下のようなメッセージが表示されます。

```bash
$ ssh user@192.168.1.100
The authenticity of host '192.168.1.100 (192.168.1.100)' can't be established.
ED25519 key fingerprint is SHA256:xxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxx.
This key is not listed in any known_hosts file.
Are you sure you want to continue connecting (yes/no/[fingerprint])? yes
Warning: Permanently added '192.168.1.100' (ED25519) to the list of known hosts.
```

この仕組みは **TOFU（Trust On First Use）** と呼ばれ、初回接続時にサーバーの鍵を信頼し、`~/.ssh/known_hosts` に記録します。2回目以降は記録された鍵と照合し、変更があれば警告を表示します。

## 1.5 暗号技術の基本

SSHが使用する暗号技術の基本を理解しておきましょう。

### 共通鍵暗号（対称鍵暗号）

送信者と受信者が**同じ鍵**で暗号化・復号を行います。

```
暗号化:  平文 + 共通鍵 → 暗号文
復号:    暗号文 + 共通鍵 → 平文

+--------+    共通鍵で暗号化     +--------+
| 送信者  | ===================> | 受信者  |
+--------+    共通鍵で復号       +--------+
      ↑                              ↑
   共通鍵A                         共通鍵A
   （同じ鍵）                     （同じ鍵）

代表的なアルゴリズム:
- AES（Advanced Encryption Standard）← SSHで最も使われる
- ChaCha20-Poly1305 ← モバイル環境で高速
```

### 公開鍵暗号（非対称鍵暗号）

**公開鍵**と**秘密鍵**のペアを使用します。

```
+--------+     公開鍵で暗号化    +--------+
| 送信者  | ===================> | 受信者  |
+--------+     秘密鍵で復号      +--------+
      ↑                              ↑
   公開鍵B                         秘密鍵B
（誰でも持てる）               （受信者だけが持つ）

特徴:
- 公開鍵で暗号化 → 秘密鍵でしか復号できない
- 秘密鍵で署名 → 公開鍵で検証できる

代表的なアルゴリズム:
- RSA     ← 長い歴史、2048bit以上推奨
- Ed25519 ← 現在の推奨、高速で安全
- ECDSA   ← 楕円曲線暗号
```

### 鍵交換アルゴリズム

安全でない通信路上で共通鍵を安全に生成する方法です。

```
Diffie-Hellman 鍵交換のイメージ:

クライアント                     サーバー
秘密値 a を選ぶ                  秘密値 b を選ぶ
A = g^a mod p を計算             B = g^b mod p を計算
     |                               |
     |  A を送る                      |
     |------------------------------>|
     |                               |
     |  B を送る                      |
     |<------------------------------|
     |                               |
K = B^a mod p を計算             K = A^b mod p を計算
     |                               |
     +----- 同じ共通鍵 K が得られる ----+

※ 盗聴者が A と B を知っても、a や b を知らない限り K を計算できない
```

## 1.6 SSH-1 と SSH-2 の違い

| 項目 | SSH-1 | SSH-2 |
|------|-------|-------|
| 公開年 | 1995年 | 2006年（RFC標準化） |
| 暗号方式 | 限定的 | 多数の暗号アルゴリズムに対応 |
| 鍵交換 | RSAベースのみ | Diffie-Hellman、ECDH等 |
| 完全性検証 | CRC-32（脆弱） | HMAC（安全） |
| チャネル多重化 | 限定的 | 完全対応 |
| セキュリティ | 既知の脆弱性あり | 現在も安全 |
| 現在の状態 | **使用禁止** | **標準** |

> **重要**: SSH-1 は既知の脆弱性があり、現在は**絶対に使用してはいけません**。現在のSSHといえばSSH-2を指します。

## 1.7 OpenSSH

### OpenSSHとは

**OpenSSH（Open Secure Shell）** は、SSHプロトコルの最も広く使われているオープンソース実装です。

```
OpenSSH に含まれる主なツール:

クライアント側:
  ssh        - リモートログイン
  scp        - ファイルコピー（レガシー）
  sftp       - 安全なファイル転送
  ssh-keygen - 鍵ペアの生成・管理
  ssh-agent  - 認証エージェント
  ssh-add    - エージェントに鍵を追加
  ssh-copy-id - 公開鍵をサーバーに登録

サーバー側:
  sshd       - SSHデーモン（サーバープロセス）
```

### バージョン確認

```bash
# クライアントのバージョン
$ ssh -V
OpenSSH_9.6p1, OpenSSL 3.2.1 30 Jan 2024

# サーバーのバージョン（接続先で確認）
$ sshd -V
# ※ 一部の環境では使えないため、以下でも確認可能
$ ssh -v localhost 2>&1 | grep "Remote protocol version"
```

### 主要なOS別のSSH対応状況

| OS | SSHクライアント | SSHサーバー |
|----|----------------|-------------|
| Linux | OpenSSH（標準搭載） | OpenSSH（インストール可） |
| macOS | OpenSSH（標準搭載） | OpenSSH（設定で有効化） |
| Windows 10/11 | OpenSSH（標準搭載） | OpenSSH（オプション機能） |

Windows での確認：
```powershell
# PowerShellでSSHクライアントの確認
> ssh -V
OpenSSH_for_Windows_9.5p1, LibreSSL 3.8.2

# OpenSSHが入っていない場合のインストール
> Add-WindowsCapability -Online -Name OpenSSH.Client~~~~0.0.1.0
```

## 1.8 SSHのポート番号

SSHはデフォルトで **TCPポート22番** を使用します。

```bash
# SSHが使用するポートの確認
$ grep ssh /etc/services
ssh     22/tcp    # SSH Remote Login Protocol
ssh     22/udp

# 接続中のSSHセッションの確認
$ ss -tnp | grep :22
ESTAB  0  0  192.168.1.10:54321  192.168.1.100:22  users:(("ssh",pid=1234,fd=3))
```

> **よくある間違い**: SSHのポート番号をセキュリティ目的で変更する（例：2222番にする）方法は「セキュリティ・スルー・オブスキュリティ（隠蔽によるセキュリティ）」と呼ばれ、それ自体はセキュリティ対策として十分ではありません。ただし、自動化されたブルートフォース攻撃の軽減には一定の効果があります。

## ポイントまとめ

```
1. SSHは暗号化されたリモート接続プロトコル（ポート22番）
2. Telnet等の平文プロトコルの問題を解決するために生まれた
3. SSHプロトコルは3層構造（トランスポート層・認証層・コネクション層）
4. 共通鍵暗号と公開鍵暗号を組み合わせて安全な通信を実現
5. SSH-1は脆弱 → SSH-2を使うこと
6. OpenSSHが事実上の標準実装
7. 初回接続時のホスト鍵確認（TOFU）を正しく理解する
```

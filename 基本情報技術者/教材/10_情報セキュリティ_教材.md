# 第10章：情報セキュリティ

## この章のゴール
- 情報セキュリティの3要素（CIA）を説明できる
- 暗号化方式（共通鍵・公開鍵・ハイブリッド）の仕組みと違いを理解する
- デジタル署名と認証局（CA）の仕組みを説明できる
- 主要な攻撃手法（マルウェア、DoS、SQLインジェクション等）を把握する
- ファイアウォール、IDS/IPSなどのセキュリティ対策技術を理解する
- ISMS（情報セキュリティマネジメントシステム）の概要を説明できる

## 試験での出題傾向
- 午前試験：暗号化方式の特徴、デジタル署名の手順、攻撃手法の名称と内容、ISMSの基本概念が頻出です。
- 午後試験：セキュリティインシデントへの対応手順、暗号化通信の仕組みに関する読解問題が出題されます。

## 10.1 情報セキュリティの基本概念

### 情報セキュリティの3要素（CIA）

情報セキュリティの基本となる3つの性質です。

| 要素 | 英語 | 説明 |
|------|------|------|
| 機密性 | Confidentiality | 許可された者だけが情報にアクセスできること |
| 完全性 | Integrity | 情報が改ざんされていないこと |
| 可用性 | Availability | 必要なときに情報にアクセスできること |

> **暗記ポイント：** 「CIA」の頭文字で覚えましょう。これは試験で非常に頻繁に問われます。

### 追加の4要素

| 要素 | 英語 | 説明 |
|------|------|------|
| 真正性 | Authenticity | 利用者や情報が本物であること |
| 責任追跡性 | Accountability | 行動を追跡・特定できること |
| 否認防止 | Non-repudiation | 行為を後から否認できないこと |
| 信頼性 | Reliability | システムが意図した通りに動作すること |

### 脅威の分類

| 分類 | 説明 | 例 |
|------|------|-----|
| 物理的脅威 | 物理的な被害 | 地震、火災、盗難 |
| 技術的脅威 | 技術を利用した攻撃 | マルウェア、不正アクセス |
| 人的脅威 | 人による脅威 | 誤操作、内部不正、ソーシャルエンジニアリング |

### リスクマネジメント

```
リスク = 脅威 × 脆弱性 × 資産価値
```

| リスク対応 | 説明 | 例 |
|-----------|------|-----|
| リスク回避 | リスクのある活動をやめる | 危険なサービスを停止する |
| リスク低減 | 対策を講じて被害を小さくする | ファイアウォールを導入する |
| リスク移転 | リスクを第三者に移す | 保険に加入する、外部委託する |
| リスク受容 | リスクを受け入れる | 対策コストが被害額を上回る場合 |

### ポイントまとめ
- セキュリティの3要素＝機密性・完全性・可用性（CIA）
- リスク＝脅威 × 脆弱性 × 資産価値
- リスク対応は回避・低減・移転・受容の4種類

## 10.2 暗号化技術

### 暗号化の基本用語

| 用語 | 説明 |
|------|------|
| 平文（ひらぶん） | 暗号化前の元のデータ |
| 暗号文 | 暗号化後のデータ |
| 暗号化 | 平文を暗号文に変換すること |
| 復号（ふくごう） | 暗号文を平文に戻すこと |
| 鍵 | 暗号化・復号に使う秘密の情報 |

### 10.2.1 共通鍵暗号方式（対称鍵暗号方式）

暗号化と復号に**同じ鍵**を使う方式です。

```
送信者                              受信者
 平文 ─→ [暗号化] ─→ 暗号文 ─→ [復号] ─→ 平文
           ↑                        ↑
         共通鍵                    共通鍵（同じ鍵）
```

**代表的なアルゴリズム：**

| アルゴリズム | 鍵長 | 特徴 |
|------------|------|------|
| AES（Advanced Encryption Standard） | 128/192/256ビット | 現在の標準。高速で安全 |
| DES（Data Encryption Standard） | 56ビット | 旧標準。現在は安全でない |
| 3DES（Triple DES） | 112/168ビット | DESを3回適用。DESより安全だが低速 |

**メリット・デメリット：**

| メリット | デメリット |
|---------|-----------|
| 暗号化・復号が高速 | 鍵の配送が困難（鍵配送問題） |
| 大量データの暗号化に適する | 通信相手ごとに異なる鍵が必要 |

**鍵の数の計算：**
n人が互いに通信する場合に必要な鍵の数：

```
鍵の数 = n(n-1) / 2

例：10人の場合 → 10 × 9 / 2 = 45個
```

### 10.2.2 公開鍵暗号方式（非対称鍵暗号方式）

暗号化と復号に**異なる鍵のペア**（公開鍵と秘密鍵）を使う方式です。

```
送信者                              受信者
 平文 ─→ [暗号化] ─→ 暗号文 ─→ [復号] ─→ 平文
           ↑                        ↑
      受信者の公開鍵            受信者の秘密鍵
     （誰でも入手可能）       （受信者だけが持つ）
```

**代表的なアルゴリズム：**

| アルゴリズム | 基盤となる問題 | 用途 |
|------------|--------------|------|
| RSA | 素因数分解の困難さ | 暗号化、デジタル署名 |
| 楕円曲線暗号（ECC） | 楕円曲線上の離散対数問題 | 短い鍵長で高い安全性 |

**メリット・デメリット：**

| メリット | デメリット |
|---------|-----------|
| 鍵配送が容易（公開鍵は公開してよい） | 暗号化・復号が低速 |
| 通信相手が増えても鍵管理が容易 | 大量データの暗号化には不向き |

**鍵の数の計算：**
n人が互いに通信する場合に必要な鍵の数：

```
鍵の数 = 2n（各人が公開鍵と秘密鍵のペアを1組持つ）

例：10人の場合 → 2 × 10 = 20個
```

### 10.2.3 ハイブリッド暗号方式

共通鍵暗号と公開鍵暗号の長所を組み合わせた方式です。SSL/TLSなど実際の暗号化通信で広く使われています。

```
1. 送信者が共通鍵（セッション鍵）をランダムに生成
2. その共通鍵を受信者の公開鍵で暗号化して送信（鍵の配送）
3. 受信者は自分の秘密鍵で共通鍵を復号
4. 以降、共通鍵でデータを暗号化して高速に通信

┌────────────────────────────────────────────┐
│ 鍵交換：公開鍵暗号（安全だが遅い）          │
│ データ通信：共通鍵暗号（高速）              │
└────────────────────────────────────────────┘
```

> **試験頻出ポイント：** ハイブリッド方式は「共通鍵を公開鍵暗号で安全に配送し、データ本体は共通鍵暗号で高速に暗号化する」という仕組みです。

### 10.2.4 ハッシュ関数

任意の長さのデータから、固定長のハッシュ値（メッセージダイジェスト）を生成する一方向関数です。

**特徴：**
- 同じ入力からは必ず同じハッシュ値が生成される
- ハッシュ値から元のデータを復元できない（一方向性）
- 異なる入力から同じハッシュ値が生成されにくい（衝突耐性）

| アルゴリズム | ハッシュ長 | 安全性 |
|------------|----------|--------|
| MD5 | 128ビット | 脆弱（衝突が見つかっている） |
| SHA-1 | 160ビット | 非推奨（脆弱性が指摘） |
| SHA-256 | 256ビット | 現在の標準 |

**用途：**
- パスワードの保存（ハッシュ化して保存）
- データの改ざん検知
- デジタル署名

### ポイントまとめ
- 共通鍵暗号：同じ鍵で暗号化・復号。高速だが鍵配送が課題
- 公開鍵暗号：公開鍵で暗号化、秘密鍵で復号。鍵配送が容易だが低速
- ハイブリッド方式：実用的な暗号化通信の基盤（SSL/TLS）
- ハッシュ関数：一方向関数。改ざん検知やパスワード保存に使用

## 10.3 認証技術

### 10.3.1 デジタル署名（電子署名）

送信者が「確かに自分が作成した」ことと「改ざんされていない」ことを証明する仕組みです。

**手順：**

```
【署名の作成（送信者側）】
1. 送信データのハッシュ値を計算
2. ハッシュ値を送信者の秘密鍵で暗号化 → これが「デジタル署名」
3. データ + デジタル署名を送信

【署名の検証（受信者側）】
1. 受信データのハッシュ値を計算 → ハッシュ値A
2. デジタル署名を送信者の公開鍵で復号 → ハッシュ値B
3. ハッシュ値A と ハッシュ値B を比較
   - 一致 → 改ざんなし＆送信者本人であると確認
   - 不一致 → 改ざんされたか、別人が署名
```

```
送信者                                    受信者
 データ ──→ ハッシュ関数 → ハッシュ値       ハッシュ値A ←── ハッシュ関数 ←── データ
                            │                               比較
                     送信者の秘密鍵で暗号化                     │
                            ↓                               ↓
                      デジタル署名 ──────→ 送信者の公開鍵で復号 → ハッシュ値B
```

> **よくある間違い：** デジタル署名では「秘密鍵で暗号化、公開鍵で復号」です。暗号化通信の「公開鍵で暗号化、秘密鍵で復号」とは鍵の使い方が逆です。

### 10.3.2 認証局（CA: Certificate Authority）とPKI

**PKI（Public Key Infrastructure / 公開鍵基盤）：**

公開鍵の正当性を保証する仕組みです。「この公開鍵は確かにこの人のものである」ことを第三者（認証局）が証明します。

**デジタル証明書（電子証明書）：**

認証局が発行する証明書で、以下の情報を含みます。
- 証明書の所有者情報
- 所有者の公開鍵
- 認証局のデジタル署名
- 有効期限

```
1. サーバーが認証局に公開鍵を登録
2. 認証局がデジタル証明書を発行（認証局の秘密鍵で署名）
3. クライアントがサーバーの証明書を受け取る
4. クライアントは認証局の公開鍵で証明書を検証
5. 正当であれば、サーバーの公開鍵を信頼して使用
```

### 10.3.3 認証方式

| 認証方式 | 説明 | 例 |
|---------|------|-----|
| 知識認証（Something you know） | 本人だけが知っている情報 | パスワード、PIN、秘密の質問 |
| 所有物認証（Something you have） | 本人だけが持っている物 | ICカード、ワンタイムパスワードトークン |
| 生体認証（Something you are） | 本人の身体的特徴 | 指紋、虹彩、顔、声紋 |

**多要素認証（MFA: Multi-Factor Authentication）：**
上記の異なる種類を2つ以上組み合わせる認証方式です。

```
例：パスワード（知識） + スマートフォンのワンタイムパスワード（所有物）
```

> **注意：** 「パスワード + 秘密の質問」は両方とも知識認証なので、多要素認証ではなく「多段階認証」です。異なる「要素」を組み合わせることが多要素認証の条件です。

### 10.3.4 生体認証の評価指標

| 指標 | 説明 |
|------|------|
| 本人拒否率（FRR: False Rejection Rate） | 本人を誤って拒否する割合 |
| 他人受入率（FAR: False Acceptance Rate） | 他人を誤って受け入れる割合 |

FRRとFARはトレードオフの関係にあります。認証の閾値を厳しくするとFRRが上がり（本人も拒否されやすい）、緩くするとFARが上がります（他人も通りやすい）。

### ポイントまとめ
- デジタル署名：秘密鍵で署名、公開鍵で検証（暗号化通信と逆）
- 認証局（CA）が公開鍵の正当性を証明
- 多要素認証：知識・所有物・生体の異なる種類を2つ以上組み合わせ
- 生体認証はFRR（本人拒否率）とFAR（他人受入率）がトレードオフ

## 10.4 攻撃手法

### 10.4.1 マルウェア（Malware）

悪意のあるソフトウェアの総称です。

| 種類 | 説明 |
|------|------|
| コンピュータウイルス | 他のプログラムに寄生して感染する。自己増殖する |
| ワーム | 独立して動作し、ネットワーク経由で自己増殖する |
| トロイの木馬 | 有用なプログラムを装って侵入し、不正な動作をする |
| ランサムウェア | データを暗号化し、復号のために身代金を要求する |
| スパイウェア | ユーザーの情報を密かに収集して外部に送信する |
| キーロガー | キーボード入力を記録して情報を盗む |
| ボット | 外部から遠隔操作される。ボットネットによるDDoS攻撃に利用 |
| ルートキット | 不正アクセスの痕跡を隠し、バックドアを設置する |

### 10.4.2 ネットワークへの攻撃

| 攻撃手法 | 説明 |
|---------|------|
| DoS攻撃（Denial of Service） | 大量のリクエストを送ってサービスを妨害する |
| DDoS攻撃（Distributed DoS） | 多数のコンピュータ（ボットネット）から一斉にDoS攻撃を行う |
| 中間者攻撃（Man-in-the-Middle） | 通信を傍受し、改ざんやなりすましを行う |
| DNSキャッシュポイズニング | DNSの応答を偽装し、偽のWebサイトに誘導する |
| ARPスプーフィング | 偽のARP応答を送り、通信を傍受する |

### 10.4.3 Webアプリケーションへの攻撃

| 攻撃手法 | 説明 | 対策 |
|---------|------|------|
| SQLインジェクション | 入力欄にSQL文を挿入し、DBを不正操作 | プレースホルダ（バインド機構）の使用 |
| クロスサイトスクリプティング（XSS） | 悪意のあるスクリプトをWebページに埋め込む | 出力時のエスケープ処理 |
| クロスサイトリクエストフォージェリ（CSRF） | 利用者の意図しない操作をWebサイトに実行させる | トークンの検証 |
| ディレクトリトラバーサル | パス文字列を操作してアクセス権のないファイルを参照 | 入力値の検証 |
| セッションハイジャック | セッションIDを盗み、なりすます | セッションIDの推測困難化 |

**SQLインジェクションの例：**

```
ログイン画面のユーザー名入力欄に以下を入力：
' OR '1'='1' --

実行されるSQL：
SELECT * FROM users WHERE name = '' OR '1'='1' --' AND password = '...'
                                ↑ 常に真になる   ↑ 以降がコメントアウト

→ 全ユーザーのデータが取得できてしまう
```

### 10.4.4 パスワードへの攻撃

| 攻撃手法 | 説明 |
|---------|------|
| 総当たり攻撃（ブルートフォース） | 全ての組み合わせを試す |
| 辞書攻撃 | よく使われる単語のリストを使って試す |
| パスワードリスト攻撃 | 他サービスから流出したID・パスワードの組を使って試す |
| レインボーテーブル攻撃 | ハッシュ値と元の文字列の対応表を使う |

**対策：**
- 十分な長さのパスワードを設定する
- パスワードのソルト（ランダムな文字列の付加）付きハッシュ化
- アカウントロック（一定回数の失敗でロック）
- 多要素認証の導入

### 10.4.5 ソーシャルエンジニアリング

技術的な手段ではなく、人間の心理的な隙を突いて情報を盗む手法です。

| 手法 | 説明 |
|------|------|
| フィッシング | 偽のメールやWebサイトで情報を詐取する |
| ショルダーハッキング | 肩越しに画面を覗き見る |
| トラッシング（スキャベンジング） | ゴミ箱を漁って情報を得る |
| なりすまし電話 | 管理者や関係者になりすまして情報を聞き出す |

### ポイントまとめ
- マルウェア：ウイルス（寄生）、ワーム（自己増殖）、トロイの木馬（偽装）
- SQLインジェクション対策：プレースホルダの使用
- XSS対策：出力時のエスケープ処理
- ソーシャルエンジニアリングは人間の心理を突く非技術的攻撃

## 10.5 セキュリティ対策技術

### 10.5.1 ファイアウォール（Firewall）

ネットワークの境界に設置し、通信を監視・制御するシステムです。

**種類：**

| 種類 | 動作層 | 説明 |
|------|--------|------|
| パケットフィルタリング | ネットワーク層/トランスポート層 | IPアドレス、ポート番号でフィルタリング |
| ステートフルインスペクション | トランスポート層 | 通信の状態（コネクション情報）を管理して判断 |
| アプリケーションゲートウェイ（プロキシ） | アプリケーション層 | アプリケーションレベルで中継・検査 |
| WAF（Web Application Firewall） | アプリケーション層 | Webアプリケーション特有の攻撃（SQL注入、XSS等）を防御 |

### 10.5.2 IDS / IPS

| システム | 正式名称 | 機能 |
|---------|---------|------|
| IDS（Intrusion Detection System） | 侵入検知システム | 不正アクセスを**検知**して管理者に通知する |
| IPS（Intrusion Prevention System） | 侵入防止システム | 不正アクセスを**検知し、自動的に遮断**する |

**検知方式：**

| 方式 | 説明 | 特徴 |
|------|------|------|
| シグネチャ型（ミスユース検知） | 既知の攻撃パターンと照合 | 未知の攻撃は検知できない |
| アノマリ型（異常検知） | 正常な通信パターンとの差異を検知 | 未知の攻撃も検知可能。誤検知が多い |

### 10.5.3 DMZ（DeMilitarized Zone / 非武装地帯）

外部ネットワーク（インターネット）と内部ネットワーク（社内LAN）の間に設けられる緩衝領域です。

```
インターネット ──[FW]── DMZ ──[FW]── 内部ネットワーク
                       │
                  Webサーバー
                  メールサーバー
                  DNSサーバー
```

外部に公開するサーバーをDMZに配置することで、万一サーバーが侵害されても内部ネットワークへの影響を最小限にします。

### 10.5.4 VPN（Virtual Private Network）

公衆ネットワーク（インターネット）上に仮想的なプライベートネットワークを構築する技術です。

| プロトコル | 説明 |
|-----------|------|
| IPsec | ネットワーク層で暗号化。拠点間VPNに使用 |
| SSL/TLS VPN | トランスポート層以上で暗号化。リモートアクセスVPNに使用 |

### 10.5.5 SSL/TLS

Web通信を暗号化するプロトコルです（HTTPS）。ハイブリッド暗号方式を使用します。

**SSL/TLS通信の流れ：**

```
1. クライアントがサーバーに接続要求
2. サーバーがデジタル証明書（公開鍵を含む）を送信
3. クライアントが証明書を検証（認証局の公開鍵で署名を確認）
4. クライアントが共通鍵（セッション鍵）を生成
5. サーバーの公開鍵で共通鍵を暗号化して送信
6. サーバーが秘密鍵で共通鍵を復号
7. 以降、共通鍵で暗号化通信
```

### ポイントまとめ
- ファイアウォール：パケットフィルタリング、WAFはWeb攻撃対策
- IDS=検知のみ、IPS=検知＋自動遮断
- DMZ：外部公開サーバーの配置場所。内部LANとの緩衝地帯
- SSL/TLS：証明書の検証 → 鍵交換 → 暗号化通信

## 10.6 ISMS（情報セキュリティマネジメントシステム）

### ISMSとは

ISMS（Information Security Management System）とは、組織が情報セキュリティを体系的に管理するための仕組みです。国際規格 **ISO/IEC 27001** で要求事項が定められています。

### PDCAサイクル

ISMSの運用はPDCAサイクルで継続的に改善します。

```
Plan（計画）→ Do（実施）→ Check（点検）→ Act（改善）→ Plan（計画）→ ...
```

| 段階 | 内容 |
|------|------|
| Plan（計画） | セキュリティポリシーの策定、リスクアセスメントの実施 |
| Do（実施） | セキュリティ対策の実施、教育・訓練 |
| Check（点検） | 監視、監査、レビュー |
| Act（改善） | 問題点の是正、改善策の実施 |

### 情報セキュリティポリシー

組織の情報セキュリティに関する方針や規則を文書化したものです。

| 文書 | 説明 |
|------|------|
| 基本方針（セキュリティポリシー） | 経営層が策定する基本的な方針 |
| 対策基準（スタンダード） | 具体的な遵守事項や基準 |
| 実施手順（プロシージャ） | 各対策の具体的な作業手順 |

### リスクアセスメント

情報資産に対するリスクを特定・分析・評価するプロセスです。

```
リスク特定 → リスク分析 → リスク評価 → リスク対応
```

### 不正のトライアングル

内部不正が発生する条件を示した理論です。3つの要因がそろうと不正が発生しやすくなります。

```
     動機（Motivation）
        /  \
       /    \
  機会（Opportunity） ── 正当化（Rationalization）
```

| 要因 | 説明 | 対策例 |
|------|------|--------|
| 動機 | 不正を行う理由 | 適切な人事評価、相談窓口の設置 |
| 機会 | 不正が可能な環境 | アクセス制御、監視カメラ、ログ管理 |
| 正当化 | 不正を正当化する心理 | コンプライアンス教育、倫理研修 |

### 主要なセキュリティ関連法規

| 法律・規格 | 概要 |
|-----------|------|
| 個人情報保護法 | 個人情報の適切な取り扱いを義務付ける |
| 不正アクセス禁止法 | 不正アクセス行為を犯罪として罰する |
| サイバーセキュリティ基本法 | 国のサイバーセキュリティに関する基本理念 |
| ISO/IEC 27001 | ISMSの要求事項の国際規格 |
| ISO/IEC 27002 | 情報セキュリティ管理策の実践規範 |

### ポイントまとめ
- ISMS＝情報セキュリティを体系的に管理する仕組み（ISO 27001）
- PDCAサイクルで継続的に改善
- 不正のトライアングル：動機・機会・正当化
- リスクアセスメントで脅威と脆弱性を評価

## まとめ

| 項目 | 重要ポイント |
|------|-------------|
| CIA | 機密性・完全性・可用性の3要素 |
| 共通鍵暗号 | 同じ鍵で暗号化・復号。高速。鍵数=n(n-1)/2 |
| 公開鍵暗号 | 公開鍵で暗号化、秘密鍵で復号。鍵数=2n |
| ハイブリッド暗号 | 共通鍵を公開鍵で配送。SSL/TLSで使用 |
| デジタル署名 | 秘密鍵で署名、公開鍵で検証（暗号化通信と逆） |
| SQLインジェクション | SQL文の挿入攻撃。対策はプレースホルダ |
| ファイアウォール | パケットフィルタリング、WAFはWeb攻撃防御 |
| IDS vs IPS | IDS=検知通知、IPS=検知+自動遮断 |
| ISMS | ISO 27001。PDCAサイクルで運用 |

## 確認問題

1. 情報セキュリティの3要素（CIA）を日本語で述べよ。
2. 共通鍵暗号方式と公開鍵暗号方式の違いを、鍵の使い方と速度の観点から説明せよ。
3. デジタル署名において、署名の作成と検証にそれぞれどの鍵を使うか述べよ。
4. SQLインジェクションとは何か、またその対策を1つ述べよ。
5. 10人が共通鍵暗号方式で互いに通信する場合、必要な鍵の数を求めよ。

<details>
<summary>解答</summary>

1. 機密性（Confidentiality）：許可された者だけが情報にアクセスできること。完全性（Integrity）：情報が改ざんされていないこと。可用性（Availability）：必要なときに情報にアクセスできること。

2. 共通鍵暗号は暗号化と復号に同じ鍵を使う。処理が高速だが、通信相手に安全に鍵を渡す必要がある（鍵配送問題）。公開鍵暗号は暗号化に公開鍵、復号に秘密鍵を使う。鍵配送が容易だが、処理が低速。

3. 署名の作成には送信者の「秘密鍵」を使い、署名の検証には送信者の「公開鍵」を使う。これは暗号化通信（公開鍵で暗号化、秘密鍵で復号）と鍵の使い方が逆である。

4. SQLインジェクションとは、Webアプリケーションの入力欄にSQL文を挿入して、データベースを不正に操作する攻撃。対策として、プレースホルダ（バインド機構）を使用し、入力値をSQL文に直接埋め込まないようにする。

5. n(n-1)/2 = 10 × 9 / 2 = 45個

</details>

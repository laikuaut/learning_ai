# 第4章：プロセス管理

## この章のゴール

- Linuxにおける「プロセス」の概念を理解すること
- 実行中のプロセスを確認・監視するコマンドを使いこなせるようになること
- プロセスの停止・強制終了を安全に行えるようになること
- systemctlによるサービス管理の基本を習得すること
- cronによるタスクのスケジューリングができるようになること

## 4.1 プロセスとは

Linuxにおいて、**プロセス**とは実行中のプログラムのことです。コマンドを実行するたびに、カーネルはそのプログラムに対してプロセスを作成します。すべてのプロセスには一意の**PID（Process ID）**が割り振られます。

### プロセスの階層構造

Linuxのプロセスは親子関係を持つツリー構造になっています。すべてのプロセスの最上位には `systemd`（または `init`）プロセスが存在します。

```
systemd (PID=1)
├── sshd (PID=850)
│   └── sshd (PID=12345)
│       └── bash (PID=12346)
│           ├── vim (PID=12400)
│           └── grep (PID=12401)
├── nginx (PID=900)
│   ├── nginx worker (PID=901)
│   └── nginx worker (PID=902)
├── cron (PID=950)
└── mysqld (PID=1000)
```

### プロセスの状態

プロセスには以下の状態があります。

| 状態 | 記号 | 説明 |
|------|------|------|
| 実行中（Running） | `R` | CPUで実行中、または実行待ち |
| スリープ（Sleeping） | `S` | 入出力やイベントを待機中（割り込み可能） |
| 深いスリープ | `D` | 入出力待機中（割り込み不可能、ディスクI/O等） |
| 停止（Stopped） | `T` | Ctrl+Z やシグナルで停止中 |
| ゾンビ（Zombie） | `Z` | 終了したが親が回収していない状態 |

```bash
# プロセスツリーを表示
$ pstree
systemd─┬─accounts-daemon───2*[{accounts-daemon}]
        ├─cron
        ├─networkd-dispat
        ├─nginx───2*[nginx]
        ├─sshd───sshd───sshd───bash───pstree
        └─systemd-journal

# PID付きで表示
$ pstree -p
systemd(1)─┬─cron(950)
            ├─nginx(900)─┬─nginx(901)
            │             └─nginx(902)
            └─sshd(850)───sshd(12345)───bash(12346)───pstree(12500)
```

## 4.2 ps - プロセスの表示

`ps`（process status）コマンドは、実行中のプロセスの情報を表示します。

### 書式

```
ps [オプション]
```

### 主要なオプション

| オプション | 説明 |
|-----------|------|
| `aux` | 全ユーザーの全プロセスを詳細表示（BSD形式） |
| `-ef` | 全プロセスをフル表示（System V形式） |
| `-u ユーザー名` | 指定ユーザーのプロセスのみ表示 |
| `-p PID` | 指定PIDのプロセスを表示 |
| `--forest` | ツリー形式で表示 |
| `-o フィールド` | 表示するフィールドを指定 |

### 基本的な使い方

```bash
# 現在のターミナルのプロセスのみ
$ ps
    PID TTY          TIME CMD
  12346 pts/0    00:00:00 bash
  12500 pts/0    00:00:00 ps

# 全プロセスを詳細表示（最もよく使うパターン）
$ ps aux
USER         PID %CPU %MEM    VSZ   RSS TTY      STAT START   TIME COMMAND
root           1  0.0  0.3 169436 12048 ?        Ss   Feb15   2:30 /sbin/init
root         850  0.0  0.1  15420  5640 ?        Ss   Feb15   0:05 sshd: /usr/sbin/sshd
www-data     901  0.0  0.2  55272  8192 ?        S    Feb15   1:23 nginx: worker process
tanaka     12346  0.0  0.1  21432  5120 pts/0    Ss   10:00   0:00 -bash
mysql       1000  0.5  5.2 1852340 213000 ?      Sl   Feb15  45:12 /usr/sbin/mysqld
root       12345  0.0  0.1  15804  6020 ?        Ss   10:00   0:00 sshd: tanaka [priv]
```

### ps aux の各フィールド

```
USER         PID %CPU %MEM    VSZ   RSS TTY      STAT START   TIME COMMAND
│            │    │    │      │     │   │        │    │       │    │
│            │    │    │      │     │   │        │    │       │    └─ コマンド名
│            │    │    │      │     │   │        │    │       └────── CPU使用時間
│            │    │    │      │     │   │        │    └────────────── 開始時刻
│            │    │    │      │     │   │        └─────────────────── 状態
│            │    │    │      │     │   └──────────────────────────── 端末
│            │    │    │      │     └──────────────────────────────── 物理メモリ使用量(KB)
│            │    │    │      └────────────────────────────────────── 仮想メモリ使用量(KB)
│            │    │    └───────────────────────────────────────────── メモリ使用率(%)
│            │    └────────────────────────────────────────────────── CPU使用率(%)
│            └─────────────────────────────────────────────────────── プロセスID
└──────────────────────────────────────────────────────────────────── 実行ユーザー
```

### STATフィールドの読み方

```
STAT の1文字目（プロセスの状態）:
  R = 実行中
  S = スリープ（割り込み可能）
  D = ディスクI/O待ち（割り込み不可）
  T = 停止
  Z = ゾンビ

STAT の2文字目以降（付加情報）:
  s = セッションリーダー
  l = マルチスレッド
  + = フォアグラウンドプロセスグループ
  < = 高い優先度
  N = 低い優先度
```

### 実用的な ps コマンドの使い方

```bash
# 特定のプロセスを検索（grepと組み合わせ）
$ ps aux | grep nginx
root         900  0.0  0.1  55272  4096 ?        Ss   Feb15   0:00 nginx: master process
www-data     901  0.0  0.2  55272  8192 ?        S    Feb15   1:23 nginx: worker process
www-data     902  0.0  0.2  55272  8192 ?        S    Feb15   1:20 nginx: worker process
tanaka     12501  0.0  0.0  17668   740 pts/0    S+   15:10   0:00 grep --color=auto nginx

# grep自身を除外するテクニック
$ ps aux | grep [n]ginx
root         900  0.0  0.1  55272  4096 ?        Ss   Feb15   0:00 nginx: master process
www-data     901  0.0  0.2  55272  8192 ?        S    Feb15   1:23 nginx: worker process
www-data     902  0.0  0.2  55272  8192 ?        S    Feb15   1:20 nginx: worker process

# CPU使用率の高い順に表示
$ ps aux --sort=-%cpu | head -10

# メモリ使用率の高い順に表示
$ ps aux --sort=-%mem | head -10

# 特定ユーザーのプロセスのみ表示
$ ps -u www-data
    PID TTY          TIME CMD
    901 ?        00:01:23 nginx
    902 ?        00:01:20 nginx

# ツリー形式で表示
$ ps auxf
root         850  0.0  0.1  15420  5640 ?        Ss   Feb15   0:05 sshd: /usr/sbin/sshd

# 表示フィールドをカスタマイズ
$ ps -eo pid,ppid,user,%cpu,%mem,stat,cmd --sort=-%cpu | head -10
    PID    PPID USER     %CPU %MEM STAT CMD
   1000       1 mysql     0.5  5.2 Sl   /usr/sbin/mysqld
    901     900 www-data  0.1  0.2 S    nginx: worker process
    902     900 www-data  0.1  0.2 S    nginx: worker process
```

### top の操作キー

| キー | 操作 |
|------|------|
| `q` | topを終了 |
| `h` | ヘルプ表示 |
| `P` | CPU使用率でソート（デフォルト） |
| `M` | メモリ使用率でソート |
| `T` | CPU時間でソート |
| `k` | プロセスをkill（PIDを入力） |
| `r` | nice値を変更 |
| `1` | CPUコア別の表示を切り替え |
| `c` | コマンドの全パスを表示 |
| `f` | 表示フィールドの選択 |
| `d` | 更新間隔の変更 |

### htop（高機能版top）

`htop` はtopの高機能版で、カラフルで操作しやすいインターフェースを持ちます。

```bash
# インストール
$ sudo apt install htop    # Ubuntu/Debian
$ sudo yum install htop    # CentOS/RHEL

# 起動
$ htop
```

htopの主な利点：
- マウスでプロセスを選択できる
- カラー表示で見やすい
- プロセスをスクロールで探せる
- CPU/メモリのバーグラフ表示
- F9キーでシグナル送信メニュー
- ツリー表示（F5キー）が見やすい

## 4.4 kill - プロセスの終了

### シグナルの概念

Linuxではプロセスに**シグナル**を送信してプロセスを制御します。`kill` コマンドはプロセスにシグナルを送るコマンドです（名前に反して「殺す」だけではありません）。

### 主要なシグナル

| シグナル | 番号 | 説明 | 用途 |
|---------|------|------|------|
| `SIGHUP` | 1 | ハングアップ | 設定ファイルの再読み込み |
| `SIGINT` | 2 | 割り込み | Ctrl+C と同じ |
| `SIGQUIT` | 3 | 終了（コアダンプ付き） | デバッグ用 |
| `SIGKILL` | 9 | 強制終了（無視できない） | 最終手段 |
| `SIGTERM` | 15 | 終了要求（デフォルト） | 正常終了 |
| `SIGSTOP` | 19 | 停止（無視できない） | プロセスの一時停止 |
| `SIGCONT` | 18 | 続行 | 停止したプロセスを再開 |
| `SIGUSR1` | 10 | ユーザー定義1 | アプリ固有の動作 |
| `SIGUSR2` | 12 | ユーザー定義2 | アプリ固有の動作 |

### kill コマンドの書式

```
kill [オプション] PID
```

### 使用例

```bash
# プロセスに終了要求を送る（SIGTERM = デフォルト）
$ kill 12345
# → プロセスは後片付けをしてから終了する

# シグナル番号を指定
$ kill -15 12345    # SIGTERMを送信（上と同じ）
$ kill -9 12345     # SIGKILLを送信（強制終了）

# シグナル名で指定
$ kill -SIGTERM 12345
$ kill -SIGKILL 12345

# 設定ファイルの再読み込み（nginx等で使用）
$ kill -HUP 900
# → nginx はプロセスを再起動せずに設定を再読み込みする

# 利用可能なシグナル一覧
$ kill -l
 1) SIGHUP       2) SIGINT       3) SIGQUIT      4) SIGILL
 5) SIGTRAP      6) SIGABRT      7) SIGBUS       8) SIGFPE
 9) SIGKILL     10) SIGUSR1     11) SIGSEGV     12) SIGUSR2
13) SIGPIPE     14) SIGALRM     15) SIGTERM     ...
```

### プロセスの正しい止め方

```
┌──────────────────────────────────────────────────────┐
│            プロセスを止める手順                        │
│                                                      │
│  Step 1: SIGTERM（15）を送る                          │
│          → プロセスが正常に終了処理を行う              │
│          → 数秒待つ                                   │
│                                                      │
│  Step 2: まだ動いているなら SIGKILL（9）を送る         │
│          → 強制的に即座に停止する                      │
│          → 終了処理は行われない                        │
│                                                      │
│  ※ 最初から SIGKILL を送るのは避けるべき              │
│    （データが壊れたり、一時ファイルが残る可能性）       │
└──────────────────────────────────────────────────────┘
```

```bash
# Step 1: まず正常終了を試みる
$ kill 12345

# 数秒待ってプロセスが終了したか確認
$ ps aux | grep 12345

# Step 2: まだ動いていたら強制終了
$ kill -9 12345
```

### killall - プロセス名で終了

```bash
# プロセス名で全プロセスに SIGTERM を送信
$ killall nginx

# 強制終了
$ killall -9 python3

# 特定ユーザーのプロセスのみ
$ killall -u tanaka python3
```

### pkill - パターンマッチで終了

```bash
# プロセス名のパターンで終了
$ pkill -f "python3 app.py"

# シグナルを指定
$ pkill -9 -f "node server.js"

# 特定ユーザーのプロセス
$ pkill -u www-data
```

## 4.5 フォアグラウンドとバックグラウンド

### フォアグラウンドとバックグラウンドの違い

```
┌──────────────────────────────────────────────────┐
│ ターミナル                                        │
│                                                  │
│  フォアグラウンド:                                │
│  ┌──────────────────────────────────────┐        │
│  │ $ ./long_process.sh                  │        │
│  │ 処理中... (ターミナルを占有)          │        │
│  │ (Ctrl+C で中断、他のコマンド打てない) │        │
│  └──────────────────────────────────────┘        │
│                                                  │
│  バックグラウンド:                                │
│  ┌──────────────────────────────────────┐        │
│  │ $ ./long_process.sh &               │        │
│  │ [1] 12500                           │        │
│  │ $ ← すぐに入力できる                 │        │
│  │ $ ls  ← 他のコマンドも実行できる     │        │
│  └──────────────────────────────────────┘        │
└──────────────────────────────────────────────────┘
```

### バックグラウンド実行

```bash
# コマンドの末尾に & を付けるとバックグラウンドで実行される
$ ./backup.sh &
[1] 12500    # [ジョブ番号] PID

# ターミナルはすぐに使えるようになる
$ ls
file1.txt  file2.txt

# バックグラウンドジョブの一覧
$ jobs
[1]+  Running                 ./backup.sh &

# 出力を捨ててバックグラウンドで実行
$ ./noisy_script.sh > /dev/null 2>&1 &
[2] 12501
```

### ジョブ制御

```bash
# フォアグラウンドのプロセスを一時停止（Ctrl+Z）
$ ./long_process.sh
^Z
[1]+  Stopped                 ./long_process.sh

# 停止したプロセスをバックグラウンドで再開
$ bg %1
[1]+ ./long_process.sh &

# バックグラウンドのプロセスをフォアグラウンドに戻す
$ fg %1
./long_process.sh

# ジョブ一覧
$ jobs
[1]-  Running                 ./backup.sh &
[2]+  Running                 ./long_process.sh &
```

### nohup - ログアウト後も実行を継続

通常のバックグラウンドプロセスはログアウトすると終了しますが、`nohup` を使うとログアウト後も実行を継続できます。

```bash
# nohup でバックグラウンド実行（ログアウトしても続く）
$ nohup ./long_task.sh &
[1] 12600
nohup: ignoring input and appending output to 'nohup.out'

# 出力先を指定
$ nohup ./long_task.sh > /var/log/task.log 2>&1 &
[1] 12601

# 実行中か確認
$ ps aux | grep long_task
tanaka   12601  0.5  0.2  12345  8192 ?  S  15:30  0:02 /bin/bash ./long_task.sh
```

## 4.6 systemctl - サービス管理

### systemdとは

`systemd` は現代のLinuxディストリビューションで標準的に使われているサービス管理システム（init system）です。`systemctl` コマンドを通じてサービスの起動・停止・状態確認を行います。

### 書式

```
systemctl [サブコマンド] [サービス名]
```

### 主要なサブコマンド

| サブコマンド | 説明 |
|-------------|------|
| `start` | サービスを起動 |
| `stop` | サービスを停止 |
| `restart` | サービスを再起動 |
| `reload` | 設定ファイルを再読み込み（サービスを停止しない） |
| `status` | サービスの状態を表示 |
| `enable` | 自動起動を有効にする |
| `disable` | 自動起動を無効にする |
| `is-active` | 動作中か確認 |
| `is-enabled` | 自動起動が有効か確認 |
| `list-units` | ユニット一覧を表示 |
| `daemon-reload` | ユニットファイルの変更を反映 |

### サービスの操作

```bash
# サービスの状態を確認
$ sudo systemctl status nginx

     Loaded: loaded (/lib/systemd/system/nginx.service; enabled; vendor preset: enabled)
     Active: active (running) since Thu 2024-02-15 09:00:00 JST; 1 month 3 days ago
       Docs: man:nginx(8)
    Process: 898 ExecStartPre=/usr/sbin/nginx -t -q -g daemon on; master_process on; (code=exited, status=0/SUCCESS)
   Main PID: 900 (nginx)
      Tasks: 3 (limit: 4915)
     Memory: 10.2M
        CPU: 2min 15.200s
     CGroup: /system.slice/nginx.service
             ├─900 nginx: master process /usr/sbin/nginx -g daemon on; master_process on;
             ├─901 nginx: worker process
             └─902 nginx: worker process

Mar 20 09:00:00 server systemd[1]: Starting A high performance web server...
Mar 20 09:00:00 server systemd[1]: Started A high performance web server.

# サービスの起動
$ sudo systemctl start nginx

# サービスの停止
$ sudo systemctl stop nginx

# サービスの再起動
$ sudo systemctl restart nginx

# 設定ファイルの再読み込み（接続を切断しない）
$ sudo systemctl reload nginx

# 自動起動の有効化（OS起動時にサービスも自動で起動する）
$ sudo systemctl enable nginx

# 自動起動の無効化
$ sudo systemctl disable nginx

# 有効化と起動を同時に行う
$ sudo systemctl enable --now nginx
```

## 4.7 cron - タスクのスケジューリング

### cronとは

`cron` は指定した日時に自動的にコマンドを実行するデーモンです。定期的なバックアップ、ログのローテーション、監視スクリプトの実行などに使います。

### crontab の書式

```
分  時  日  月  曜日  コマンド
│   │   │   │    │
│   │   │   │    └── 曜日（0-7、0と7は日曜、または Sun-Sat）
│   │   │   └─────── 月（1-12、または Jan-Dec）
│   │   └─────────── 日（1-31）
│   └─────────────── 時（0-23）
└─────────────────── 分（0-59）
```

### crontab の特殊記号

| 記号 | 意味 | 例 |
|------|------|-----|
| `*` | すべての値 | `* * * * *`（毎分） |
| `,` | 複数の値を指定 | `0,30 * * * *`（0分と30分） |
| `-` | 範囲 | `9-17 * * * *`（9時〜17時の毎時） |
| `/` | 間隔 | `*/5 * * * *`（5分ごと） |

### crontab の操作

```bash
# 現在のcrontabを表示
$ crontab -l
no crontab for tanaka

# crontabを編集（エディタが開く）
$ crontab -e

# 他のユーザーのcrontabを表示（root権限が必要）
$ sudo crontab -u www-data -l

# crontabを削除（注意！全てのジョブが消える）
$ crontab -r
```

### crontab の設定例

```bash
# crontab -e で以下のように記述する

# ──────────── 分  時  日  月  曜日  コマンド ──────────────

# 毎日午前3時にバックアップを実行
0 3 * * * /home/tanaka/scripts/backup.sh

# 5分ごとにサーバーの死活監視
*/5 * * * * /home/tanaka/scripts/health_check.sh

# 毎週月曜の午前9時にレポートを生成
0 9 * * 1 /home/tanaka/scripts/weekly_report.sh

# 毎月1日の午前0時にログを整理
0 0 1 * * /home/tanaka/scripts/log_cleanup.sh

# 平日の午前9時〜午後6時まで、30分ごとにデータを同期
*/30 9-18 * * 1-5 /home/tanaka/scripts/sync_data.sh

# 毎日午前2時にログをローテーション（出力をログに保存）
0 2 * * * /home/tanaka/scripts/rotate_logs.sh >> /var/log/cron_rotate.log 2>&1

# 毎分実行（テスト用、本番では使わない）
* * * * * echo "cron test $(date)" >> /tmp/cron_test.log
```

### cronの定型表現

| 表現 | 意味 | 同等のcron式 |
|------|------|-------------|
| `@reboot` | 起動時に1回 | - |
| `@yearly` | 年に1回 | `0 0 1 1 *` |
| `@monthly` | 月に1回 | `0 0 1 * *` |
| `@weekly` | 週に1回 | `0 0 * * 0` |
| `@daily` | 日に1回 | `0 0 * * *` |
| `@hourly` | 毎時 | `0 * * * *` |

```bash
# 起動時に1回だけ実行
@reboot /home/tanaka/scripts/startup.sh

# 毎日0時に実行
@daily /home/tanaka/scripts/daily_task.sh
```

### cronのデバッグ

```bash
# cronのログを確認（Ubuntu）
$ grep CRON /var/log/syslog
Mar 20 15:00:01 server CRON[12700]: (tanaka) CMD (/home/tanaka/scripts/backup.sh)

# CentOS/RHEL の場合
$ grep CRON /var/log/cron

# cronデーモンが動いているか確認
$ systemctl status cron
● cron.service - Regular background program processing daemon
     Active: active (running)

# crontabの注意点：
# 1. 環境変数がログインシェルと異なる（PATHが限定的）
# 2. コマンドはフルパスで書くのが安全
# 3. 出力はメールで送信される（不要なら /dev/null へ）
```

### cronの環境変数の注意

```bash
# crontab内で環境変数を設定できる
PATH=/usr/local/sbin:/usr/local/bin:/usr/sbin:/usr/bin:/sbin:/bin
SHELL=/bin/bash
MAILTO=admin@example.com

# コマンドはフルパスで記述するのが安全
0 3 * * * /usr/bin/python3 /home/tanaka/scripts/cleanup.py

# 出力を捨てたい場合
0 3 * * * /home/tanaka/scripts/backup.sh > /dev/null 2>&1

# 出力をファイルに保存したい場合
0 3 * * * /home/tanaka/scripts/backup.sh >> /var/log/backup.log 2>&1
```

## 4.8 よくある間違い

### 間違い1: いきなり kill -9 を使う

```bash
# 【誤り】最初から強制終了
$ kill -9 12345    # データが壊れる可能性！

# 【正解】まずSIGTERMで正常終了を試みる
$ kill 12345       # SIGTERM（デフォルト）
$ sleep 5          # 数秒待つ
$ kill -9 12345    # まだ動いていたら強制終了
```

### 間違い2: cronでPATHが通っていない

```bash
# 【誤り】cronではPATHが限定的
*/5 * * * * python3 /home/tanaka/app.py    # python3 が見つからない！

# 【正解】フルパスで指定
*/5 * * * * /usr/bin/python3 /home/tanaka/app.py
```

### 間違い3: nohup を忘れてSSH切断でプロセスが終了

```bash
# 【誤り】SSH接続を切ると終了してしまう
$ ./long_task.sh &

# 【正解】nohupを使う
$ nohup ./long_task.sh > /var/log/task.log 2>&1 &
```

## 4.9 現場での注意点

### サーバーの負荷確認手順

```bash
# 1. まずロードアベレージを確認
$ uptime
 15:30:00 up 30 days,  5:53,  2 users,  load average: 8.52, 5.35, 3.28

# 2. topでCPU/メモリを消費しているプロセスを特定
$ top -bn1 | head -20

# 3. 特定のプロセスの詳細を確認
$ ps aux | grep mysql

# 4. 必要に応じてプロセスを再起動
$ sudo systemctl restart mysql
```

### サービス再起動時の注意

```bash
# restart（停止→起動）ではなく reload を使えるなら reload を優先する
# reload はダウンタイムなしで設定を反映できる

$ sudo systemctl reload nginx    # 接続を切断しない（推奨）
$ sudo systemctl restart nginx   # 一瞬接続が切れる
```

## 4.10 まとめ

# 第5章 演習：Tools・Resources・Promptsの詳細設計

MCPが提供する3つの主要機能（Tools・Resources・Prompts）を正しく理解し、実際のサーバー設計に活用できる力を身につけましょう。

---

## 基本問題

### 問題5-1：Tool・Resource・Promptの判断

以下の10個のユースケースについて、MCPの「Tool」「Resource」「Prompt」のどれに該当するかを答えてください。また、その理由も簡潔に説明してください。

1. データベースのテーブル一覧を取得する
2. 新しいGitHubイシューを作成する
3. 「コードレビューをして」という定型指示テンプレートを提供する
4. 天気APIから現在の天気情報を取得する
5. プロジェクトのREADME.mdの内容を読み取る
6. ファイルをS3バケットにアップロードする
7. 「SQLクエリを最適化して」という手順付きのプロンプトテンプレートを提供する
8. ユーザーのプロフィール情報（`user://{user_id}/profile`）を参照する
9. Slackチャンネルにメッセージを送信する
10. アプリケーションのログファイルの最新100行を読み取る

<details>
<summary>ヒント</summary>

以下の基準で判断しましょう。

- **Tool**：外部システムに対して**副作用のある操作**（作成・更新・削除・送信など）を行うもの、または計算処理を実行するもの
- **Resource**：データや情報を**読み取り専用で提供**するもの。URIでアクセスできる静的・動的なデータソース
- **Prompt**：LLMに対する**指示テンプレート**。ユーザーの意図を構造化して伝えるためのもの
</details>

<details>
<summary>解答例</summary>

1. **Resource** — テーブル一覧は読み取り専用のデータです。`db://tables` のようなURIで公開します
2. **Tool** — イシューの「作成」は副作用のある操作です。`create_issue` のようなToolとして定義します
3. **Prompt** — 定型の指示テンプレートであり、LLMへの入力を構造化するものです
4. **Tool** — 外部API呼び出しを伴う動的な処理です。結果はリアルタイムに変わるため、Toolとして定義するのが適切です
5. **Resource** — ファイルの内容を読み取るだけなので、`file://project/README.md` のようなResourceです
6. **Tool** — ファイルのアップロードは副作用のある操作です
7. **Prompt** — 手順付きのプロンプトテンプレートであり、LLMの振る舞いを制御するものです
8. **Resource** — ユーザー情報の参照です。URIテンプレート `user://{user_id}/profile` で動的にアクセスできるResourceです
9. **Tool** — メッセージの送信は副作用のある操作です
10. **Resource** — ログファイルの読み取りは参照系の操作です。`logs://app/latest?lines=100` のようなResourceとして公開します

</details>

### 問題5-2：Toolのdescriptionを改善する

以下のToolのdescription（説明文）には問題があります。それぞれ何が問題かを指摘し、改善版を作成してください。

```json
{
  "name": "search",
  "description": "検索する"
}
```

```json
{
  "name": "send_email",
  "description": "この関数はメールを送信します。引数にto、subject、bodyを取ります。toはメールアドレスの文字列、subjectは件名、bodyは本文です。"
}
```

```json
{
  "name": "delete_file",
  "description": "Delete a file from the system"
}
```

期待される解答の観点：
- 何を検索するのかが分かるか
- LLMがいつこのToolを使うべきか判断できるか
- 引数の説明はdescriptionではなくinputSchemaに書くべきか
- 副作用や注意点が明示されているか

<details>
<summary>ヒント</summary>

良いdescriptionの条件を思い出しましょう。

- LLMが「このToolをいつ使うべきか」判断できる
- 何に対して何をするのかが明確
- 副作用がある場合は注意点を記載する
- 引数の説明はinputSchemaに委ねる
- 日本語・英語どちらでも可だが、一貫性を保つ
</details>

<details>
<summary>解答例</summary>

**1つ目の問題点と改善：**

問題点：「検索する」だけでは、何を検索するのか不明です。LLMはこのToolをいつ使うべきか判断できません。

```json
{
  "name": "search_documents",
  "description": "プロジェクト内のドキュメントをキーワードで全文検索します。ファイル名・本文の両方が検索対象です。"
}
```

**2つ目の問題点と改善：**

問題点：引数の説明をdescriptionに含めてしまっています。引数の型や意味はinputSchemaで定義すべきです。

```json
{
  "name": "send_email",
  "description": "指定した宛先にメールを送信します。送信後の取り消しはできません。",
  "inputSchema": {
    "type": "object",
    "properties": {
      "to": { "type": "string", "description": "送信先メールアドレス" },
      "subject": { "type": "string", "description": "件名" },
      "body": { "type": "string", "description": "本文（プレーンテキスト）" }
    },
    "required": ["to", "subject", "body"]
  }
}
```

**3つ目の問題点と改善：**

問題点：ファイル削除は破壊的な操作ですが、その危険性が記載されていません。また、どの範囲のファイルが対象かも不明です。

```json
{
  "name": "delete_file",
  "description": "指定パスのファイルを完全に削除します。【注意】この操作は取り消せません。ワークスペース内のファイルのみ対象です。"
}
```

</details>

### 問題5-3：Resourceの基本理解

以下の質問に答えてください。

1. MCPにおけるResourceの読み取りメソッドは何ですか？
2. Resourceが返すレスポンスには何が含まれますか？主要なフィールドを3つ挙げてください。
3. 静的リソースと動的リソース（リソーステンプレート）の違いを説明してください。
4. リソースのURIスキームとして適切なものを以下から全て選んでください。
   - `file://`
   - `http://`
   - `db://`
   - `custom-app://`

<details>
<summary>解答例</summary>

1. **`resources/read`** メソッドを使用します。リソースの一覧取得には `resources/list` を使います。

2. Resourceのレスポンスに含まれる主要フィールド：
   - **`uri`**：リソースを一意に識別するURI
   - **`mimeType`**：リソースのMIMEタイプ（例：`text/plain`、`application/json`）
   - **`text`** または **`blob`**：リソースの内容（テキストデータまたはBase64エンコードされたバイナリデータ）

3. 違い：
   - **静的リソース**：URIが固定で、`resources/list` で列挙されます。例：`file://config.json`
   - **動的リソース（リソーステンプレート）**：URIにパラメータを含み、`resources/templates/list` で列挙されます。例：`user://{user_id}/profile`。RFC 6570のURIテンプレート構文を使用します

4. **全て適切**です。MCPのリソースURIは任意のスキームを使用できます。標準的なスキーム（`file://`、`http://`）も、カスタムスキーム（`db://`、`custom-app://`）も有効です。アプリケーションの用途に合わせて自由に定義できます。

</details>

### 問題5-4：Promptの基本理解

MCPのPromptsについて、以下の空欄を埋めてください。

```
Promptsは、LLMに対する（ ① ）を再利用可能な形で定義する仕組みです。
サーバーは prompts/（ ② ） で利用可能なプロンプトの一覧を公開し、
クライアントは prompts/（ ③ ） でプロンプトの内容を取得します。

プロンプトには（ ④ ）を定義でき、ユーザーから値を受け取って
プロンプト内容を動的に組み立てることができます。

プロンプトが返す messages 配列の各要素には（ ⑤ ）と content が含まれます。
```

<details>
<summary>解答例</summary>

1. **指示テンプレート**（またはプロンプトテンプレート）
2. **list**
3. **get**
4. **arguments（引数）**
5. **role**（`"user"` または `"assistant"`）

補足：Promptsの定義例は以下のようになります。

```json
{
  "name": "code_review",
  "description": "コードレビューを依頼するプロンプト",
  "arguments": [
    {
      "name": "code",
      "description": "レビュー対象のコード",
      "required": true
    },
    {
      "name": "language",
      "description": "プログラミング言語",
      "required": false
    }
  ]
}
```

</details>

### 問題5-5：既存APIをMCPサーバー化する設計

あなたは「タスク管理アプリ」のREST APIをMCPサーバーとして公開する設計を任されました。以下のAPIエンドポイントを、Tools・Resources・Promptsに適切に振り分けてください。

**既存のREST API一覧：**

| メソッド | エンドポイント | 説明 |
|---------|--------------|------|
| GET | `/tasks` | タスク一覧を取得 |
| GET | `/tasks/{id}` | タスク詳細を取得 |
| POST | `/tasks` | 新規タスクを作成 |
| PUT | `/tasks/{id}` | タスクを更新 |
| DELETE | `/tasks/{id}` | タスクを削除 |
| GET | `/tasks/{id}/comments` | タスクのコメントを取得 |
| POST | `/tasks/{id}/comments` | コメントを追加 |
| GET | `/projects` | プロジェクト一覧を取得 |
| GET | `/projects/{id}/stats` | プロジェクトの統計情報を取得 |

さらに、以下のPromptも設計してください。
- タスクの優先度を判断するプロンプト
- 週次レポートを生成するプロンプト

期待される出力例：

```
【Resources】
- tasks://list — タスク一覧（GET /tasks に対応）
- ...

【Tools】
- create_task — 新規タスクを作成（POST /tasks に対応）
- ...

【Prompts】
- prioritize_task — タスクの優先度判断
- ...
```

<details>
<summary>ヒント</summary>

振り分けの基準：
- **GET（読み取り）→ Resource** が基本ですが、パラメータ付きのGETは場合によってToolにもなり得ます
- **POST/PUT/DELETE（副作用あり）→ Tool**
- **LLMへの指示テンプレート → Prompt**
- 動的なリソース（IDを含むもの）はリソーステンプレートを使いましょう
</details>

<details>
<summary>解答例</summary>

```
【Resources】
- tasks://list
    説明：全タスクの一覧を返します
    対応API：GET /tasks
    MIMEタイプ：application/json

- tasks://{task_id}
    説明：指定IDのタスク詳細を返します（リソーステンプレート）
    対応API：GET /tasks/{id}
    MIMEタイプ：application/json

- tasks://{task_id}/comments
    説明：指定タスクのコメント一覧を返します（リソーステンプレート）
    対応API：GET /tasks/{id}/comments
    MIMEタイプ：application/json

- projects://list
    説明：プロジェクト一覧を返します
    対応API：GET /projects
    MIMEタイプ：application/json

- projects://{project_id}/stats
    説明：プロジェクトの統計情報を返します（リソーステンプレート）
    対応API：GET /projects/{id}/stats
    MIMEタイプ：application/json

【Tools】
- create_task
    説明：新しいタスクを作成します
    対応API：POST /tasks
    inputSchema: { title: string, description?: string, project_id?: string }

- update_task
    説明：既存のタスクを更新します
    対応API：PUT /tasks/{id}
    inputSchema: { task_id: string, title?: string, status?: string, description?: string }

- delete_task
    説明：タスクを削除します。この操作は取り消せません
    対応API：DELETE /tasks/{id}
    inputSchema: { task_id: string }

- add_comment
    説明：タスクにコメントを追加します
    対応API：POST /tasks/{id}/comments
    inputSchema: { task_id: string, body: string }

【Prompts】
- prioritize_task
    説明：タスクの内容と状況から優先度を判断します
    arguments:
      - task_description (required): タスクの説明
      - deadline (optional): 期限
      - dependencies (optional): 依存タスク
    messages:
      - role: user
        content: |
          以下のタスクの優先度を「高・中・低」で判断してください。
          判断理由も説明してください。

          タスク内容：{task_description}
          期限：{deadline}
          依存タスク：{dependencies}

- weekly_report
    説明：今週の進捗から週次レポートを生成します
    arguments:
      - project_name (required): プロジェクト名
    messages:
      - role: user
        content: |
          以下のプロジェクトの週次レポートを作成してください。
          完了タスク、進行中タスク、ブロッカーを整理してまとめてください。

          プロジェクト：{project_name}
```

<!-- 設計ポイント：
     - GETエンドポイントはResourceとして公開し、URIテンプレートで動的パラメータを扱う
     - POST/PUT/DELETEはToolとして公開し、inputSchemaで引数を定義する
     - LLMに判断や生成を依頼するパターンはPromptとして定義する
     - Resourceはキャッシュやサブスクリプションの恩恵を受けられる -->

</details>

### 問題5-6：リソーステンプレートの設計

以下の要件を満たすリソーステンプレートを設計してください。JSON形式で `resources/templates/list` のレスポンスとして記述してください。

**要件：**
1. ECサイトの商品情報を、カテゴリとページ番号で絞り込んで取得する
2. ユーザーの注文履歴を、年月で絞り込んで取得する
3. ブログ記事を、タグで絞り込んで取得する

期待される出力例：

```json
{
  "resourceTemplates": [
    {
      "uriTemplate": "shop://products/{category}?page={page}",
      "name": "...",
      "description": "...",
      "mimeType": "application/json"
    }
  ]
}
```

<details>
<summary>ヒント</summary>

- URIテンプレートはRFC 6570の構文に従います
- `{変数名}` でパラメータを埋め込みます
- 各テンプレートには `name`、`description`、`uriTemplate`、`mimeType` を含めましょう
</details>

<details>
<summary>解答例</summary>

```json
{
  "resourceTemplates": [
    {
      "uriTemplate": "shop://products/{category}?page={page}",
      "name": "商品一覧（カテゴリ別）",
      "description": "指定カテゴリの商品一覧をページ単位で取得します。categoryはカテゴリスラッグ（例：electronics）、pageは1始まりのページ番号です。",
      "mimeType": "application/json"
    },
    {
      "uriTemplate": "shop://users/{user_id}/orders/{year}/{month}",
      "name": "注文履歴（月別）",
      "description": "指定ユーザーの注文履歴を年月で絞り込んで取得します。yearは4桁の西暦、monthは1〜12の数値です。",
      "mimeType": "application/json"
    },
    {
      "uriTemplate": "blog://posts?tag={tag}",
      "name": "ブログ記事（タグ別）",
      "description": "指定タグに一致するブログ記事の一覧を取得します。tagはタグのスラッグ（例：python、web-development）です。",
      "mimeType": "application/json"
    }
  ]
}
```

<!-- 設計ポイント：
     - URIのパス部分に必須パラメータ、クエリ部分にオプションパラメータを配置するのが一般的
     - descriptionにはパラメータの具体的な値の例を含めると、LLMが正しくURIを組み立てやすくなる
     - mimeTypeは返すデータの形式に合わせて設定する -->

</details>

### 問題5-7：Toolの入力スキーマ設計

以下のToolについて、`inputSchema`（JSON Schema形式）を設計してください。バリデーションを適切に設定し、descriptionも付けてください。

**Tool：`search_flights`（フライト検索）**

要件：
- 出発地の空港コード（3文字の英大文字、必須）
- 到着地の空港コード（3文字の英大文字、必須）
- 出発日（YYYY-MM-DD形式、必須）
- 帰着日（YYYY-MM-DD形式、任意・片道の場合は不要）
- 乗客数（1〜9の整数、デフォルト1）
- 座席クラス（economy / business / first のいずれか、デフォルト economy）

<details>
<summary>ヒント</summary>

JSON Schemaで使えるバリデーション：
- `pattern`：正規表現による文字列パターンの制約
- `enum`：許可される値の列挙
- `minimum` / `maximum`：数値の範囲制約
- `default`：デフォルト値
- `format`：日付などの標準フォーマット指定
</details>

<details>
<summary>解答例</summary>

```json
{
  "name": "search_flights",
  "description": "出発地・到着地・日程を指定してフライトを検索します。片道・往復の両方に対応しています。",
  "inputSchema": {
    "type": "object",
    "properties": {
      "origin": {
        "type": "string",
        "pattern": "^[A-Z]{3}$",
        "description": "出発地の空港コード（IATA 3レターコード。例：NRT、HND）"
      },
      "destination": {
        "type": "string",
        "pattern": "^[A-Z]{3}$",
        "description": "到着地の空港コード（IATA 3レターコード。例：LAX、SFO）"
      },
      "departure_date": {
        "type": "string",
        "format": "date",
        "description": "出発日（YYYY-MM-DD形式。例：2026-04-15）"
      },
      "return_date": {
        "type": "string",
        "format": "date",
        "description": "帰着日（YYYY-MM-DD形式）。片道の場合は省略してください"
      },
      "passengers": {
        "type": "integer",
        "minimum": 1,
        "maximum": 9,
        "default": 1,
        "description": "乗客数（1〜9名）"
      },
      "seat_class": {
        "type": "string",
        "enum": ["economy", "business", "first"],
        "default": "economy",
        "description": "座席クラス"
      }
    },
    "required": ["origin", "destination", "departure_date"]
  }
}
```

<!-- ポイント：
     - patternで空港コードの形式を制約し、不正な値を防止する
     - format: "date" でISO 8601の日付形式を指定する
     - requiredに必須項目のみを含め、任意項目にはdefaultを設定する
     - descriptionに具体例を含めると、LLMが正しい値を生成しやすい -->

</details>

## チャレンジ問題

### 問題5-8：総合MCPサーバー設計書の作成

あなたは「社内ナレッジベースシステム」をMCPサーバーとして設計することになりました。以下の要件をもとに、Tools・Resources・Promptsの3機能を全て活用した設計書を作成してください。

**システム概要：**
- 社内のドキュメント（技術記事、議事録、マニュアル）を管理する
- ドキュメントの検索・閲覧・作成・編集ができる
- ドキュメントにタグやカテゴリを付与できる
- ドキュメントへのコメント機能がある
- よくある質問をドキュメントから自動回答する機能が求められている

**設計書に含めるべき内容：**

1. **サーバーの基本情報**
   - サーバー名、バージョン、capabilities

2. **Resources（最低5つ）**
   - 静的リソースとリソーステンプレートの両方を含めること
   - URI設計、name、description、mimeTypeを記載

3. **Tools（最低5つ）**
   - name、description、inputSchema を記載
   - 副作用のあるToolには注意書きを含めること

4. **Prompts（最低3つ）**
   - name、description、arguments、messagesの構造を記載
   - 少なくとも1つは複数メッセージ（multi-turn）のPromptにすること

5. **設計の判断理由**
   - なぜResourceではなくToolにしたか（またはその逆）の根拠を2つ以上記載

期待される出力形式：

```markdown
# MCPサーバー設計書：社内ナレッジベース

## 1. サーバー基本情報
...

## 2. Resources
### 2-1. 静的リソース
...
### 2-2. リソーステンプレート
...

## 3. Tools
...

## 4. Prompts
...

## 5. 設計判断の根拠
...
```

<details>
<summary>ヒント</summary>

設計のコツ：

- **読み取り系はResource**：ドキュメントの内容、一覧、統計情報など
- **書き込み系はTool**：作成、編集、削除、タグ付与など
- **LLMの知性を活かすものはPrompt**：要約、Q&A生成、レビューなど
- **検索は判断が分かれるところ**：単純な全文検索はToolが適切（クエリの組み立てが必要なため）。一方、固定的なフィルタリングはResourceテンプレートでも対応可能
- capabilitiesでは、このサーバーがどの機能をサポートするかを宣言します
</details>

<details>

```markdown
# MCPサーバー設計書：社内ナレッジベース

## 1. サーバー基本情報

サーバー名：knowledge-base-server
バージョン：1.0.0
capabilities：
  - resources（リソース提供、サブスクリプション対応）
  - tools（ツール提供）
  - prompts（プロンプト提供）

## 2. Resources

### 2-1. 静的リソース

| URI | name | description | mimeType |
|-----|------|-------------|----------|
| kb://categories | カテゴリ一覧 | ドキュメントのカテゴリ一覧と各カテゴリのドキュメント数を返します | application/json |
| kb://tags | タグ一覧 | 登録されている全タグの一覧を返します | application/json |
| kb://stats | 統計情報 | ドキュメント総数、カテゴリ別件数、最近の更新状況などの統計を返します | application/json |

### 2-2. リソーステンプレート

| URIテンプレート | name | description | mimeType |
|----------------|------|-------------|----------|
| kb://docs/{doc_id} | ドキュメント詳細 | 指定IDのドキュメント本文・メタ情報を返します | application/json |
| kb://docs/{doc_id}/comments | コメント一覧 | 指定ドキュメントのコメント一覧を返します | application/json |
| kb://categories/{category}/docs | カテゴリ別ドキュメント | 指定カテゴリのドキュメント一覧を返します | application/json |
| kb://tags/{tag}/docs | タグ別ドキュメント | 指定タグが付いたドキュメント一覧を返します | application/json |

## 3. Tools

### create_document
- description：新しいドキュメントを作成します。作成後はドキュメントIDが返されます
- inputSchema：
  - title (string, required)：ドキュメントタイトル
  - body (string, required)：本文（Markdown形式）
  - category (string, required)：カテゴリ
  - tags (string[], optional)：タグの配列

### update_document
- description：既存ドキュメントの内容を更新します。指定したフィールドのみ更新されます
- inputSchema：
  - doc_id (string, required)：対象ドキュメントID
  - title (string, optional)：新しいタイトル
  - body (string, optional)：新しい本文
  - category (string, optional)：新しいカテゴリ

### delete_document
- description：ドキュメントを削除します。【注意】この操作は取り消せません。関連するコメントも全て削除されます
- inputSchema：
  - doc_id (string, required)：削除対象のドキュメントID

### search_documents
- description：キーワードでドキュメントを全文検索します。タイトルと本文の両方が検索対象です
- inputSchema：
  - query (string, required)：検索キーワード
  - category (string, optional)：カテゴリで絞り込み
  - tags (string[], optional)：タグで絞り込み
  - limit (integer, optional, default: 20)：最大取得件数

### add_comment
- description：ドキュメントにコメントを追加します
- inputSchema：
  - doc_id (string, required)：対象ドキュメントID
  - body (string, required)：コメント本文

### manage_tags
- description：ドキュメントのタグを追加・削除します
- inputSchema：
  - doc_id (string, required)：対象ドキュメントID
  - add_tags (string[], optional)：追加するタグ
  - remove_tags (string[], optional)：削除するタグ

## 4. Prompts

### answer_question
- description：ナレッジベースの情報をもとに質問に回答します
- arguments：
  - question (string, required)：ユーザーの質問
  - context_docs (string, optional)：参考ドキュメントのID（カンマ区切り）
- messages：
  - role: user
    content: |
      社内ナレッジベースの情報をもとに、以下の質問に回答してください。
      回答の根拠となるドキュメントがあれば、そのタイトルとIDを引用してください。
      情報が見つからない場合は「該当する情報が見つかりませんでした」と回答してください。

      質問：{question}

### summarize_document
- description：ドキュメントの内容を要約します
- arguments：
  - doc_id (string, required)：要約対象のドキュメントID
  - length (string, optional)：要約の長さ（short / medium / long）
- messages：
  - role: user
    content: |
      以下のドキュメントを{length}で要約してください。
      箇条書きで要点を整理し、最後に一言でまとめてください。

### review_and_improve（multi-turn）
- description：ドキュメントのレビューと改善提案を対話的に行います
- arguments：
  - doc_id (string, required)：レビュー対象のドキュメントID
  - focus (string, optional)：重点レビュー観点（accuracy / readability / completeness）
- messages：
  - role: user
    content: |
      以下のドキュメントをレビューしてください。
      観点：{focus}
      問題点を指摘し、具体的な改善案を提示してください。
  - role: assistant
    content: |
      ドキュメントをレビューしました。以下が指摘事項です。
      （ここにレビュー結果が入ります）
      修正を反映しますか？追加の観点はありますか？
  - role: user
    content: |
      指摘事項を踏まえて、改善版のドキュメントを作成してください。

## 5. 設計判断の根拠

### 判断1：検索をResourceではなくToolにした理由
検索（search_documents）はGET的な操作ですが、以下の理由でToolとして設計しました。
- 検索クエリの組み立てが動的で複雑（キーワード＋カテゴリ＋タグの組み合わせ）
- LLMがユーザーの意図を解釈して適切な検索パラメータを生成する必要がある
- リソーステンプレートでは表現しきれない柔軟なクエリが求められる

### 判断2：カテゴリ別・タグ別一覧をToolではなくResourceにした理由
カテゴリ別ドキュメント一覧（kb://categories/{category}/docs）は、以下の理由でResourceとして設計しました。
- 単一パラメータによる単純なフィルタリングであり、URIテンプレートで十分表現可能
- Resourceにすることでサブスクリプション（変更通知）の恩恵を受けられる
- キャッシュが効きやすく、パフォーマンス面で有利
```

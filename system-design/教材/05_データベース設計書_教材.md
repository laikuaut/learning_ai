## 5.2 命名規約

### テーブル・カラムの命名規約

設計の一貫性を保つために、命名規約を定めます。

```
【命名規約（例）】

■ テーブル名
  ・英小文字のスネークケース（snake_case）
  ・複数形を使用する
  ・例：users, products, order_items

■ カラム名
  ・英小文字のスネークケース
  ・テーブル名のプレフィックスは付けない
  ・例：id, name, created_at（✗ user_id_pk）

■ 主キー
  ・カラム名は "id" とする
  ・型は BIGINT AUTO_INCREMENT またはUUID

■ 外部キー
  ・参照先テーブル名の単数形 + "_id"
  ・例：user_id, product_id, category_id

■ 日時カラム
  ・"_at" サフィックスを使用する
  ・例：created_at, updated_at, deleted_at

■ フラグカラム
  ・"is_" プレフィックスを使用する
  ・例：is_active, is_deleted, is_published

■ 予約語
  ・SQLの予約語は使用しない
  ・例：✗ order → ○ orders（テーブル名）
        ✗ select → ○ selection（カラム名）
```

### 共通カラム設計

全テーブルに共通で設定するカラムを定義します。

```
【共通カラム】

| カラム名     | データ型                | NULL | 説明              |
|-------------|------------------------|------|-------------------|
| id          | BIGINT AUTO_INCREMENT  | NO   | 主キー             |
| created_at  | TIMESTAMP              | NO   | レコード作成日時    |
| updated_at  | TIMESTAMP              | NO   | レコード更新日時    |
| deleted_at  | TIMESTAMP              | YES  | 論理削除日時（NULL=有効）|
```

> **実務でのポイント**: 論理削除（soft delete）は `deleted_at` カラムを使い、NULLなら有効、値が入っていれば削除済みとする方式が一般的です。物理削除（DELETE文）は行わず、削除フラグで管理することで、誤削除からの復元やデータ分析が可能になります。

## 5.3 ER図（Entity-Relationship Diagram）

### ER図の基本要素

```
【ER図の記法（IE記法 / Crow's Foot記法）】

┌──────────┐         ┌──────────┐
│ エンティティA│─────────│ エンティティB│
└──────────┘         └──────────┘

リレーションシップの記号:

 ──│──  1（ちょうど1つ）
 ──○──  0または1（任意）
 ──┤├── 1以上（1つ以上）
 ──○┤── 0以上（0個以上）

組み合わせ例:
 ──│──○┤──  1対多（1つのAに対して0個以上のB）
 ──│──┤├──  1対多（1つのAに対して1個以上のB）
 ──○┤──○┤──  多対多（中間テーブルで実現）
```

### ER図の例：ECサイト

```
【ER図：ECサイト（主要テーブル）】

┌─────────────┐     ┌──────────────┐     ┌───────────────┐
│   users      │     │   orders      │     │  order_items   │
├─────────────┤     ├──────────────┤     ├───────────────┤
│ PK id        │──┐  │ PK id         │──┐  │ PK id          │
│    name      │  │  │ FK user_id    │  │  │ FK order_id    │
│    email     │  │  │    status     │  │  │ FK product_id  │
│    password  │  └─→│    total_amount│  └─→│    quantity     │
│    created_at│     │    ordered_at  │     │    unit_price   │
└─────────────┘     └──────────────┘     │    subtotal     │
                                          └───────────────┘
       │1                │1                      │*
       │                 │                       │
       │*                │*                      │
┌─────────────┐     ┌──────────────┐     ┌───────────────┐
│  addresses   │     │  payments     │     │   products     │
├─────────────┤     ├──────────────┤     ├───────────────┤
│ PK id        │     │ PK id         │     │ PK id          │
│ FK user_id   │     │ FK order_id   │     │    name        │
│    zip_code  │     │    method     │     │    price       │
│    address   │     │    amount     │     │    stock       │
│    is_default│     │    status     │     │ FK category_id │
└─────────────┘     │    paid_at    │     │    description │
                     └──────────────┘     └───────────────┘
                                                 │*
                                                 │
                                                 │1
                                          ┌───────────────┐
                                          │  categories    │
                                          ├───────────────┤
                                          │ PK id          │
                                          │    name        │
                                          │    parent_id   │
                                          └───────────────┘

【リレーションシップの説明】
・users 1:N orders     → 1人のユーザーは複数の注文を持つ
・orders 1:N order_items → 1つの注文は複数の注文明細を持つ
・products 1:N order_items → 1つの商品は複数の注文明細に含まれる
・users 1:N addresses  → 1人のユーザーは複数の配送先を持つ
・orders 1:1 payments  → 1つの注文は1つの決済情報を持つ
・categories 1:N products → 1つのカテゴリに複数の商品が属する
```

### テーブル定義書の書き方

```

| No | カラム名        | カラム和名   | データ型        | PK | FK | NOT NULL | デフォルト      | 説明                  |
|----|----------------|------------|----------------|----|----|----------|---------------|-----------------------|
| 1  | id             | ユーザーID  | BIGINT         | ○  |    | ○        | AUTO_INCREMENT| 主キー                |
| 2  | name           | 氏名       | VARCHAR(100)   |    |    | ○        |               | ユーザーの氏名         |
| 3  | email          | メールアドレス| VARCHAR(255)   |    |    | ○        |               | ログインID（一意）     |
| 4  | password_hash  | パスワード  | VARCHAR(255)   |    |    | ○        |               | bcryptハッシュ値      |
| 5  | role           | ロール     | ENUM           |    |    | ○        | 'general'     | general/admin         |
| 6  | is_active      | 有効フラグ  | BOOLEAN        |    |    | ○        | true          | false=無効化          |
```

## 5.5 正規化

### 正規化とは

正規化（Normalization）は、**データの重複を排除し、整合性を保ちやすい構造にする**ための手法です。

### 第1正規形（1NF）

**ルール：** すべてのカラムが原子値（これ以上分割できない値）であること。

```
【第1正規形に違反する例】

❌ 非正規形:
| 注文ID | 商品名              | 数量      |
|-------|---------------------|----------|
| 1     | りんご, バナナ, みかん | 2, 3, 1  |

→ 1つのカラムに複数の値が入っている（繰り返し項目）

✅ 第1正規形:
| 注文ID | 商品名  | 数量 |
|-------|--------|-----|
| 1     | りんご  | 2   |
| 1     | バナナ  | 3   |
| 1     | みかん  | 1   |

→ 1つのカラムに1つの値（原子値）のみ
```

### 第2正規形（2NF）

**ルール：** 第1正規形 + すべての非キーカラムが主キー全体に完全関数従属すること。

```
【第2正規形に違反する例】

❌ 部分関数従属がある:
| 注文ID | 商品ID | 商品名  | 数量 |
|-------|-------|--------|-----|
| 1     | P001  | りんご  | 2   |
| 1     | P002  | バナナ  | 3   |
| 2     | P001  | りんご  | 1   |

主キー: (注文ID, 商品ID)
商品名は商品IDだけで決まる → 部分関数従属

✅ 第2正規形:
【注文明細】                    【商品】
| 注文ID | 商品ID | 数量 |      | 商品ID | 商品名  |
|-------|-------|-----|       |-------|--------|
| 1     | P001  | 2   |       | P001  | りんご  |
| 1     | P002  | 3   |       | P002  | バナナ  |
| 2     | P001  | 1   |

→ 商品名を商品テーブルに分離
```

### 第3正規形（3NF）

**ルール：** 第2正規形 + 非キーカラム間の推移的関数従属がないこと。

```
【第3正規形に違反する例】

❌ 推移的関数従属がある:
| 社員ID | 部署ID | 部署名    |
|-------|-------|----------|
| E001  | D01   | 開発部    |
| E002  | D01   | 開発部    |
| E003  | D02   | 営業部    |

社員ID → 部署ID → 部署名（推移的関数従属）
部署名が変更されると全行を更新する必要がある

✅ 第3正規形:
【社員】                  【部署】
| 社員ID | 部署ID |        | 部署ID | 部署名  |
|-------|-------|         |-------|--------|
| E001  | D01   |         | D01   | 開発部  |
| E002  | D01   |         | D02   | 営業部  |
| E003  | D02   |

→ 部署名を部署テーブルに分離
→ 部署名の変更は1箇所の更新で済む
```

## 5.6 インデックス設計

### インデックスとは

インデックス（Index）は、データベースの検索を高速化する仕組みです。本の索引と同じ役割を持ちます。

```
【インデックスの効果イメージ】

インデックスなし:
┌────────────────────────────────────────┐
│ テーブル全体を先頭から順に探す（フルスキャン）│
│ 100万行 → 100万行すべてを確認             │
│ 処理時間：数秒〜数十秒                     │
└────────────────────────────────────────┘

インデックスあり:
┌────────────────────────────────────────┐
│ インデックスのツリー構造で素早く特定        │
│ 100万行 → 約20回の比較で特定              │
│ 処理時間：数ミリ秒                        │
└────────────────────────────────────────┘
```

### インデックス設計の指針

```
【インデックスを張るべきカラム】

✅ 張るべき:
  ・WHERE句で頻繁に使われるカラム
  ・JOIN条件に使われるカラム（外部キー）
  ・ORDER BY / GROUP BYに使われるカラム
  ・一意性制約が必要なカラム（UNIQUE INDEX）

❌ 張らない方が良い:
  ・カーディナリティが低いカラム（例：性別、フラグ）
  ・頻繁に更新されるカラム（インデックス更新コスト）
  ・テーブルのレコード数が少ない場合（1000行以下）
  ・ほとんど使われない検索条件のカラム
```

### インデックス設計書の例

```
【インデックス設計書】

テーブル: orders

| インデックス名            | カラム              | 種別    | 用途                      |
|-------------------------|--------------------|---------|-----------------------------|
| PRIMARY                 | id                 | PRIMARY | 主キー                       |
| idx_orders_user_id      | user_id            | INDEX   | ユーザー別注文一覧取得        |
| idx_orders_status       | status             | INDEX   | ステータス別検索              |
| idx_orders_ordered_at   | ordered_at         | INDEX   | 日時範囲検索（月別売上集計等） |
| uq_orders_order_number  | order_number       | UNIQUE  | 注文番号の一意性保証          |
| idx_orders_user_status  | user_id, status    | INDEX   | ユーザー×ステータスの複合検索  |

※ 複合インデックス idx_orders_user_status は
   「特定ユーザーの未完了注文」など、よく使われるクエリパターン用
```

## 5.7 容量見積もり

### データ容量の見積もり方法

```
【容量見積もり（例：ordersテーブル）】

1レコードあたりの容量:
  id(8B) + user_id(8B) + order_number(20B) + status(20B) +
  total_amount(5B) + tax_amount(5B) + shipping_fee(5B) +
  address_id(8B) + ordered_at(8B) + shipped_at(8B) +
  delivered_at(8B) + cancelled_at(8B) + created_at(8B) +
  updated_at(8B) + オーバーヘッド(約50B)
  ≒ 約180B/レコード

年間レコード数:
  1日平均注文数: 100件
  年間: 100 × 365 = 36,500件

年間データ量:
  36,500 × 180B ≒ 6.6MB/年

5年間のデータ量:
  6.6MB × 5年 ≒ 33MB（インデックス含め約50MB）

→ 結論：容量は問題なし。パーティション分割は不要。
```

## 8.1 ログ管理

### ログフォーマットのカスタマイズ

```nginx
http {
    # デフォルトのログフォーマット
    log_format main '$remote_addr - $remote_user [$time_local] '
                    '"$request" $status $body_bytes_sent '
                    '"$http_referer" "$http_user_agent"';

    # JSON形式のログフォーマット（解析しやすい）
    log_format json escape=json '{'
        '"time": "$time_iso8601",'
        '"remote_addr": "$remote_addr",'
        '"method": "$request_method",'
        '"uri": "$uri",'
        '"status": $status,'
        '"body_bytes_sent": $body_bytes_sent,'
        '"request_time": $request_time,'
        '"upstream_response_time": "$upstream_response_time",'
        '"user_agent": "$http_user_agent"'
    '}';

    # サイトごとにログファイルを分離
    server {
        server_name example.com;
        access_log /var/log/nginx/example.com.access.log json;
        error_log  /var/log/nginx/example.com.error.log warn;
    }
}
```

### エラーログのレベル

```
【エラーログレベル（重大度順）】

emerg   → 緊急事態（nginxが動作不能）
alert   → 即座の対応が必要
crit    → 深刻な問題
error   → エラー（デフォルト）
warn    → 警告
notice  → 通知
info    → 情報
debug   → デバッグ（大量に出力される）

設定例:
  error_log /var/log/nginx/error.log warn;  # warn以上を記録
  error_log /var/log/nginx/debug.log debug; # デバッグ時のみ
```

### ログローテーション

```bash
# /etc/logrotate.d/nginx（一般的なデフォルト設定）
/var/log/nginx/*.log {
    daily           # 日次でローテーション
    missingok       # ログファイルがなくてもエラーにしない
    rotate 14       # 14世代保持
    compress        # 古いログを圧縮
    delaycompress   # 1世代前は圧縮しない
    notifempty      # 空のログはローテーションしない
    create 0640 www-data adm   # 新しいログファイルの権限
    sharedscripts
    postrotate
        # ローテーション後にnginxのログを再オープン
        if [ -f /var/run/nginx.pid ]; then
            kill -USR1 `cat /var/run/nginx.pid`
        fi
    endscript
}
```

> **実務でのポイント**: JSON形式のログフォーマットを使うと、Elasticsearch + Kibana（ELK Stack）やCloudWatch Logs、Datadog などのログ分析ツールで簡単にパースできます。

## 8.2 パフォーマンスチューニング

### ワーカープロセスと接続数の最適化

```nginx
# メインコンテキスト
worker_processes auto;            # CPUコア数に自動設定
worker_rlimit_nofile 65535;       # ワーカーのファイルディスクリプタ上限

events {
    worker_connections 4096;      # 1ワーカーあたりの最大接続数
    multi_accept on;              # 一度に複数の接続を受け付ける
    use epoll;                    # Linux: epoll を使用（高効率）
}
```

```
【最大同時接続数の計算】

最大接続数 = worker_processes × worker_connections

例: worker_processes 4, worker_connections 4096
    → 最大 16,384 同時接続

※ リバースプロキシの場合、1クライアント接続につき
  バックエンドへの接続も使うため、実質は半分程度
```

### gzip圧縮

```nginx
http {
    gzip on;                      # gzip圧縮を有効化
    gzip_vary on;                 # Varyヘッダーを付与
    gzip_proxied any;             # プロキシ経由のレスポンスも圧縮
    gzip_comp_level 6;            # 圧縮レベル（1-9、6が推奨バランス）
    gzip_min_length 256;          # 256バイト未満は圧縮しない
    gzip_types
        text/plain
        text/css
        text/javascript
        application/json
        application/javascript
        application/xml
        application/xml+rss
        image/svg+xml;
    # 注意: 画像(jpg, png)やすでに圧縮済み(gz, zip)は対象外
}
```

### 静的ファイルのキャッシュ設定

```nginx
server {
    # ブラウザキャッシュの設定
    location ~* \.(css|js)$ {
        expires 30d;
        add_header Cache-Control "public, immutable";
    }

    location ~* \.(jpg|jpeg|png|gif|ico|svg|webp)$ {
        expires 365d;
        add_header Cache-Control "public, immutable";
    }

    location ~* \.(woff|woff2|ttf|eot)$ {
        expires 365d;
        add_header Cache-Control "public, immutable";
    }
}
```

### ファイル転送の最適化

```nginx
http {
    sendfile on;          # カーネルレベルのファイル転送（高速化）
    tcp_nopush on;        # sendfile使用時にヘッダとボディをまとめて送信
    tcp_nodelay on;       # 小さなパケットを即座に送信（Keep-Alive時）

    # Keep-Alive 設定
    keepalive_timeout 65;       # クライアント接続のタイムアウト
    keepalive_requests 100;     # 1接続あたりの最大リクエスト数
}
```

## 8.3 プロキシキャッシュ

### バックエンドレスポンスのキャッシュ

```nginx
http {
    # キャッシュの保存先とパラメータを定義
    proxy_cache_path /var/cache/nginx
                     levels=1:2           # ディレクトリ階層
                     keys_zone=my_cache:10m  # キャッシュゾーン名とメモリ
                     max_size=1g          # 最大ディスク使用量
                     inactive=60m         # 60分アクセスなしで削除
                     use_temp_path=off;   # 一時ファイルを使わない

    server {
        location /api/ {
            proxy_cache my_cache;
            proxy_cache_valid 200 10m;      # 200レスポンスを10分キャッシュ
            proxy_cache_valid 404 1m;       # 404レスポンスを1分キャッシュ
            proxy_cache_use_stale error timeout updating;  # エラー時に古いキャッシュを使用
            add_header X-Cache-Status $upstream_cache_status;  # デバッグ用

            proxy_pass http://localhost:3000;
        }

        # キャッシュしない条件
        location /api/user/ {
            proxy_cache my_cache;
            proxy_cache_bypass $http_authorization;  # 認証ヘッダーがある場合はキャッシュ不使用
            proxy_no_cache $http_authorization;      # 認証レスポンスはキャッシュしない

            proxy_pass http://localhost:3000;
        }
    }
}
```

## 8.4 よくあるトラブルと解決方法

### トラブルシューティングの手順

```
【トラブル発生時の調査フロー】

1. 設定ファイルのチェック
   $ sudo nginx -t

2. エラーログの確認
   $ sudo tail -f /var/log/nginx/error.log

3. アクセスログの確認
   $ sudo tail -f /var/log/nginx/access.log

4. プロセスの確認
   $ ps aux | grep nginx
   $ sudo ss -tlnp | grep nginx

5. 権限の確認
   $ ls -la /var/www/html/
   $ namei -l /var/www/html/index.html

6. SELinux / AppArmor の確認
   $ getenforce           # SELinux
   $ sudo aa-status       # AppArmor
```

### よくあるエラーと対処法

```
【エラー別の対処法】

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
403 Forbidden
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
原因1: ファイルのパーミッション不足

原因2: ドキュメントルートにインデックスファイルがない
  → index.html を配置するか autoindex on; を設定
原因3: SELinuxがアクセスをブロック
  → sudo setsebool -P httpd_read_user_content on

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

  → バックエンドのログを確認

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

  → proxy_read_timeout を延長
  → バックエンドの処理を最適化

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

  → client_max_body_size を増やす

原因: ポート80が他のプロセスで使用中
  → sudo ss -tlnp | grep :80 で確認
  → Apache等が起動していれば停止する
```

## 8.5 監視とステータス確認

### stub_status モジュール

```nginx
server {
    listen 8080;   # 監視用のポート

    location /nginx_status {
        stub_status;
        allow 127.0.0.1;     # ローカルのみ
        allow 10.0.0.0/8;    # 監視サーバー
        deny all;
    }
}
```

```bash
# ステータスの確認
$ curl http://localhost:8080/nginx_status

Active connections: 15
server accepts handled requests
 1234 1234 5678
Reading: 2 Writing: 5 Waiting: 8
```

```
【ステータス値の意味】

Active connections: 15  → 現在のアクティブ接続数
accepts: 1234          → 受け付けた接続の総数
handled: 1234          → 処理した接続の総数（accepts と同じなら正常）
requests: 5678         → 処理したリクエストの総数
Reading: 2             → リクエストヘッダーを読み込み中
Writing: 5             → レスポンスを書き込み中
Waiting: 8             → Keep-Alive で待機中
```

## 8.6 設定変更のベストプラクティス

### 安全な設定変更の手順

```
【設定変更の安全な手順】

1. バックアップ
   $ sudo cp /etc/nginx/sites-enabled/example.conf \
              /etc/nginx/sites-enabled/example.conf.bak

2. 設定ファイルの編集
   $ sudo vim /etc/nginx/sites-enabled/example.conf

3. 文法チェック（必須！）
   $ sudo nginx -t
   nginx: the configuration file syntax is ok
   nginx: configuration file test is successful

4. 設定の反映（reload = ダウンタイムなし）
   $ sudo systemctl reload nginx

5. 動作確認
   $ curl -I https://example.com
   $ tail -f /var/log/nginx/error.log

6. 問題があればロールバック
   $ sudo cp /etc/nginx/sites-enabled/example.conf.bak \
              /etc/nginx/sites-enabled/example.conf
   $ sudo nginx -t && sudo systemctl reload nginx
```

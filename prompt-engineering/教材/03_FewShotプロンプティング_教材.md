# 第3章：Few-Shotプロンプティング

## この章のゴール

- Zero-Shot、One-Shot、Few-Shotの違いを理解する
- Few-Shotプロンプティングの効果と仕組みを学ぶ
- 効果的な例示の選び方を習得する
- フォーマット統一の重要性を理解する
- Few-Shotが特に効果的なタスクを把握する

---

## 3.1 Zero-Shot / One-Shot / Few-Shotとは

### 3つのアプローチ

LLMへの指示方法は、提供する「例」の数によって3つに分類されます。

```
Zero-Shot（例なし）：
  指示のみで回答を求める。
  → 「以下のテキストの感情を判定してください」

One-Shot（1例）：
  1つの例を示してから回答を求める。
  → 「例：『最高の映画だった』→ ポジティブ」

Few-Shot（複数例）：
  複数の例を示してから回答を求める。
  → 「例1：... → ポジティブ、例2：... → ネガティブ、例3：...」
```

### 比較表

| アプローチ | 例の数 | 精度 | トークン消費 | 適したタスク |
|-----------|--------|------|------------|------------|
| Zero-Shot | 0 | 低〜中 | 少ない | 一般的なタスク、LLMが得意な領域 |
| One-Shot | 1 | 中 | 少ない | フォーマットの指定 |
| Few-Shot | 2〜10 | 高 | 多い | 特殊なフォーマット、分類タスク |

## 3.2 Few-Shotプロンプティングの基本

### なぜFew-Shotが効果的か

LLMは「パターン認識」が得意です。具体的な入出力の例を示すことで、LLMは以下を暗黙的に学習します。

```
Few-Shotから学習される情報：

1. タスクの種類（何をすべきか）
2. 出力フォーマット（どう返すべきか）
3. 判断基準（どう判定すべきか）
4. トーンと文体（どう表現すべきか）
5. 詳細度（どの程度詳しく返すべきか）
```

### 基本的なFew-Shotの構造

```
あなたは商品レビューの感情分析を行います。
レビューテキストを「ポジティブ」「ネガティブ」「ニュートラル」に
分類してください。

### 例1
レビュー：「この商品は期待以上でした！毎日使っています。」
分類：ポジティブ

### 例2
レビュー：「サイズが合わなくて返品しました。品質も微妙。」
分類：ネガティブ

### 例3
レビュー：「普通に使えます。可もなく不可もなく。」
分類：ニュートラル

### 分析対象
レビュー：「デザインは気に入っているけど、バッテリーがすぐ切れる。」
分類：
```

> **ポイント**: 例の後に同じフォーマットで入力を与えると、LLMはパターンに従って回答します。

## 3.3 効果的な例の選び方

### 例の選び方の原則

| 原則 | 説明 | 具体例 |
|------|------|--------|
| **多様性** | 異なるパターンをカバーする | ポジティブ、ネガティブ、ニュートラルを各1つ |
| **代表性** | 典型的なケースを選ぶ | 極端な例ではなく、よくあるパターンを優先 |
| **境界例** | 判断が難しいケースを含める | 「良い点と悪い点が混在する」ような例 |
| **一貫性** | 例同士で矛盾しない | 同じ基準で分類されていること |

### 例の数の目安

```
推奨する例の数：

タスクの複雑度   推奨例数   説明
─────────────  ────────  ────────────────────
単純（感情分析）   2〜3     パターンが少ないため少数で十分
中程度（分類）     3〜5     カテゴリ数に応じて調整
複雑（構造化抽出） 3〜7     多様な入力パターンをカバー
```

> **注意**: 例を増やしすぎるとトークンを消費し、コストと応答時間が増加します。「質の高い例を少数」が「質の低い例を多数」より効果的です。

### 悪い例の選び方

```
❌ 悪いFew-Shot（偏った例）：

例1：「すごく良かった！」→ ポジティブ
例2：「とても気に入りました」→ ポジティブ
例3：「最高です！」→ ポジティブ

→ ポジティブの例しかないため、LLMはすべてを
   ポジティブと判定する傾向になる

✅ 良いFew-Shot（バランスの取れた例）：

例1：「すごく良かった！」→ ポジティブ
例2：「期待はずれでした」→ ネガティブ
例3：「普通です」→ ニュートラル
```

> **ポイントまとめ**: 例は多様で代表的なものを選び、分類のバランスを取りましょう。

### フォーマットが統一されていないと

## 3.6 Few-Shotの注意点と限界

### 注意点

| 注意点 | 説明 | 対策 |
|--------|------|------|
| **トークン消費** | 例が多いほどコスト増 | 必要最小限の例数にする |
| **例の品質** | 悪い例はLLMを誤誘導 | 正確で代表的な例を選ぶ |
| **順序の影響** | 最後の例に引っ張られやすい | 多様な例をバランスよく配置 |
| **過学習** | 例に似たパターンしか扱えない | 境界例や例外パターンも含める |

### Few-Shotが効きにくいケース

```
・推論や計算が主体のタスク
  → Chain of Thought（第4章）が有効

・知識が必要なタスク
  → RAG（第6章）が有効

・非常に創造的なタスク
  → 例に引っ張られすぎる可能性あり
```

> **ポイントまとめ**: Few-Shotは分類・変換・フォーマット統一に強力です。ただしトークン消費に注意し、質の高い例を厳選しましょう。

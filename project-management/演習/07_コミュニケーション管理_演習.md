### 問題1：コミュニケーションチャネル数の計算

以下の各チーム規模について、コミュニケーションチャネル数を計算してください。

1. 4人チーム
2. 7人チーム
3. 12人チーム
4. 25人チーム

また、7人チームに3人追加して10人チームになった場合、チャネル数はどれだけ増加しますか？

**期待される出力例：**

```
1. 4人チーム → 6チャネル
```

<details>
<summary>ヒント</summary>

コミュニケーションチャネル数 = n x (n - 1) / 2

</details>

<details>
<summary>解答例</summary>

```
公式：チャネル数 = n × (n - 1) / 2

1.  4人チーム →  4 ×  3 / 2 =   6 チャネル
2.  7人チーム →  7 ×  6 / 2 =  21 チャネル
3. 12人チーム → 12 × 11 / 2 =  66 チャネル
4. 25人チーム → 25 × 24 / 2 = 300 チャネル

■ 7人 → 10人への変化：
  7人: 21チャネル
  10人: 10 × 9 / 2 = 45チャネル
  増加: 45 - 21 = 24チャネル

→ 3人追加しただけで、チャネル数は21→45と2倍以上に増加。
  これがチームサイズを小さく保つべき理由の一つ。
```

</details>

### 問題2：ステークホルダー分析

以下のプロジェクトのステークホルダーについて、エンゲージメントレベル（不認識 / 抵抗的 / 中立的 / 支持的 / 主導的）を判定し、望ましいレベルと対応方針を述べてください。

**プロジェクト：** 社内の紙ベースの経費精算をデジタル化するシステムの導入

| ステークホルダー | 現在の反応 |
|----------------|-----------|
| A: CFO（最高財務責任者） | 「コスト削減につながるなら進めてくれ」 |
| B: 経理部マネージャー | 「今のやり方で問題ないのに、なぜ変えるの？」 |
| C: 一般社員（申請者） | 「便利になるなら嬉しい」 |
| D: IT部門マネージャー | 「セキュリティが心配だが、技術的には可能」 |
| E: 総務部 | このプロジェクトの存在を知らない |

<details>
<summary>解答例</summary>

```
| SH  | 現在レベル | 望ましいレベル | ギャップ | 対応方針                         |
|-----|-----------|-------------|---------|----------------------------------|
| A   | 支持的     | 主導的       | 小      | 定期報告で関心を維持し、           |
|     |           |             |         | 意思決定への参加を促す             |
| B   | 抵抗的     | 支持的       | 大      | 現行業務の課題（時間、ミス）を     |
|     |           |             |         | データで示す。導入後の業務フローを  |
|     |           |             |         | 一緒に設計し、当事者意識を持たせる  |
| C   | 支持的     | 支持的       | なし    | 操作性の良いUIを提供し、           |
|     |           |             |         | 満足度を維持する                   |
| D   | 中立的     | 支持的       | 中      | セキュリティ要件を初期から共同で    |
|     |           |             |         | 定義し、技術的な懸念を解消する      |
| E   | 不認識     | 中立的       | 中      | プロジェクトの概要と影響を説明する   |
|     |           |             |         | メールやブリーフィングで情報提供     |

■ 最も注意が必要なのはB（経理部マネージャー）
  理由：
  ・抵抗的→支持的への変化は最もギャップが大きい
  ・経理部は主要な利用部門であり、協力が不可欠
  ・対策：
    - 1on1ミーティングで不安や懸念を丁寧にヒアリング
    - 現行プロセスの問題点（紙の紛失、承認の遅延等）をデータで提示
    - パイロット導入で小さな成功体験を作る
    - 「経理部の意見を反映したシステム」という当事者意識を醸成
```

</details>

### 問題3：エスカレーション判断

以下の状況について、適切なエスカレーションレベルを判断し、報告先と報告内容を記述してください。

1. スプリント内でバグが見つかり、修正に半日かかる見込み
2. 外部APIベンダーが納期を2週間延期すると通知してきた
3. チームメンバー間で技術的なアプローチについて意見が対立し、2日間進捗がない
4. 本番環境でデータ漏洩の疑いがある

<details>
<summary>解答例</summary>

```
1. バグ修正に半日 → レベル1（チーム内で解決）
   報告先：チームリーダー
   報告内容：バグの内容、修正見込み時間（半日）、
   スプリント内の他タスクへの影響なし
   対応：デイリースクラムで共有し、チーム内で完結

2. 外部API納期2週間延期 → レベル3（スポンサー判断が必要）
   報告先：PM → スポンサー / ステアリングコミッティ
   報告内容：
   ・影響：プロジェクト全体のスケジュールに2週間の遅延リスク
   ・代替案：スタブAPIで開発を先行 / スコープ調整 / 期限延長
   ・判断を仰ぐ内容：どの対策を採るか、追加コストの承認
   対応：即日で影響分析を行い、翌営業日までに報告

3. 技術的意見の対立で2日間停滞 → レベル2（PM判断が必要）
   報告先：チームリーダー → PM
   報告内容：
   ・対立の内容（例：REST vs GraphQL）
   ・各選択肢のメリット・デメリット
   ・チームでの議論の経緯
   対応：PMが技術リーダーと協議し、判断基準を明確にして決定。
   必要に応じてアーキテクチャレビュー会議を設定

4. データ漏洩の疑い → レベル4（緊急・経営判断が必要）
   報告先：PM → CTO → CISO / 経営層 → 法務部
   報告内容：
   ・発覚した状況と影響範囲
   ・暫定対処（該当機能の停止等）
   ・調査計画
   対応：即時対応。セキュリティインシデント対応手順に従い、
   影響範囲の特定、証拠の保全、関係機関への通報を検討。
   個人情報保護委員会への報告義務も確認する。
```

</details>

### 問題5：変更要求書の作成

以下のシナリオの変更要求書を作成してください。

**シナリオ：** ECサイトリニューアルプロジェクトの開発フェーズ中、マーケティング部から「商品ページにSNSシェアボタン（X、Facebook、LINE）を追加してほしい」という要望がありました。

<details>
<summary>解答例</summary>

```
変更要求書（Change Request）
─────────────────────────────
変更ID      ：CR-2026-003
起票日      ：2026/04/10
起票者      ：マーケティング部 佐藤
カテゴリ    ：スコープ追加
優先度      ：中

【変更内容】
商品詳細ページにSNSシェアボタンを追加する。
対象SNS：X（旧Twitter）、Facebook、LINE
各ボタンから商品名、商品画像、商品ページURLを含む
投稿が作成されること。

【変更理由】
・リニューアルに伴い、SNS経由の流入を増やしたい
・競合ECサイトはすべてシェアボタンを実装済み
・マーケティング部の今期KPI「SNS経由流入20%増」に貢献

【影響分析】
- スコープ：フロントエンド1コンポーネント追加、
            OGPメタタグの動的生成
- スケジュール：+2日（実装1日 + テスト1日）
- コスト：約16万円（0.2人月）
- 品質：既存機能への影響なし（新規追加のため）
- リスク：各SNSのシェアAPI仕様変更への追従が
         将来の運用コストになる可能性（低）

【推奨対応】
承認を推奨。実装コストが小さく、ビジネス価値が高い。
現スプリントではなく次スプリント（Sprint 6）で対応する。

【承認】
□ 承認  □ 却下  □ 保留
承認者：＿＿＿＿＿＿  日付：＿＿＿＿＿＿
```

</details>

### 問題6：コンフリクト解決シミュレーション

以下の対立状況について、最適なコンフリクト解決スタイルを選び、PMとしての対応を具体的に記述してください。

**状況1：** フロントエンドエンジニアのAさんは「React」を、バックエンドエンジニアのBさんは「Vue.js」を主張し、フレームワーク選定で2日間議論が平行線。

**状況2：** POが「このスプリントにあと2つ機能を追加してほしい」と言っているが、チームは「品質が下がる」と反対している。

**状況3：** チームメンバーのCさんとDさんが個人的な不和で口をきかなくなり、ペアプログラミングを拒否している。

<details>
<summary>解答例</summary>

```
■ 状況1：フレームワーク選定の対立 → 協調（Collaborating）

  理由：技術選定はプロジェクト全体に長期的な影響を与える重要な判断。
  双方の主張に技術的な根拠があるはずで、Win-Winの解決を目指すべき。

  PMの対応：
  1. 両者に「選定基準」を先に合意させる
     ・チームの既存スキル
     ・プロジェクトの要件（SSR、パフォーマンス等）
     ・エコシステムの充実度
     ・採用への影響
  2. 各基準で両フレームワークを客観的に評価する表を作成
  3. プロトタイプを両方で作成し、実際に比較（技術検証）
  4. 評価結果に基づいてチーム全体で投票・決定
  5. 決定後は全員がその判断を尊重するよう合意

■ 状況2：POとチームのスコープ対立 → 妥協（Compromising）

  理由：POのビジネス要求とチームの品質懸念の両方に正当性がある。
  時間が限られるスプリント内で、双方が譲歩する必要がある。

  PMの対応：
  1. チームに「2機能追加した場合の具体的なリスク」を説明してもらう
     ・どのテストが省略されるか
     ・どの程度品質が下がるか
  2. POに「2機能のうち、よりビジネス価値が高い方」を1つ選んでもらう
  3. 妥協案を提示：
     ・今スプリント：1機能を追加（品質基準を維持）
     ・次スプリント：残り1機能を最優先で実装
  4. 品質基準は交渉不可であることをPOに理解してもらう
  5. ベロシティのデータを示し、客観的にキャパシティを説明

■ 状況3：個人的な不和 → 協調を目指すが、短期的には適応

  理由：個人間の対立はチーム全体の生産性に影響する。
  根本解決（協調）を目指しつつ、短期的にはペア変更で対応。

  PMの対応：
  1. 短期対応（適応/回避）：
     ・ペアの組み合わせを変更し、CさんとDさんが直接組まない配置にする
     ・プロジェクトの進行を優先する

  2. 根本解決（協調）：
     ・CさんとDさんそれぞれと1on1ミーティングを実施
     ・何が問題の原因かを個別にヒアリング
     ・両者の合意のもとで3者面談を設定
     ・グラウンドルール（相手を尊重する、仕事に集中する）を設定
     ・必要に応じてEM（エンジニアリングマネージャー）に相談

  3. 予防策：
     ・チームの行動規範（Working Agreement）を策定
     ・レトロスペクティブで「チームの健全性」を定期的に確認
```

</details>

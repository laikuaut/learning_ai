### 問題1：正規化

以下の非正規形テーブルを第3正規形まで正規化してください。

```
【受注テーブル（非正規形）】
| 受注ID | 顧客名   | 顧客住所       | 商品名1 | 単価1 | 数量1 | 商品名2 | 単価2 | 数量2 |
|-------|---------|---------------|--------|------|------|--------|------|------|
| 1     | 田中太郎 | 東京都新宿区    | りんご  | 100  | 3    | バナナ  | 80   | 5    |
| 2     | 鈴木花子 | 大阪府大阪市    | みかん  | 120  | 2    |        |      |      |
| 3     | 田中太郎 | 東京都新宿区    | バナナ  | 80   | 10   | みかん  | 120  | 3    |
```

**期待される出力例：**

```
【第1正規形】
繰り返し項目を排除...

【第2正規形】
部分関数従属を排除...

【第3正規形】
推移的関数従属を排除...
```

<details>
<summary>ヒント</summary>

1. 第1正規形：繰り返し項目（商品名1, 商品名2...）を排除する
2. 第2正規形：主キーの一部にだけ依存するカラムを分離する
3. 第3正規形：非キーカラム間の依存関係を分離する

</details>

<details>
<summary>解答例</summary>

```
【第1正規形】繰り返し項目を排除
```

### 問題2：テーブル定義書の作成

ブログシステムの「記事（articles）」テーブルのテーブル定義書を作成してください。

以下の要件を満たすこと：
- 記事にはタイトル、本文、著者（ユーザー）がある
- 公開/下書きのステータスを管理できる
- カテゴリを設定できる（1記事1カテゴリ）
- 公開日時を指定できる（予約投稿対応）
- 論理削除に対応する

**期待される出力例：**

```
テーブル名：articles
| No | カラム名 | カラム和名 | データ型 | PK | FK | NOT NULL | デフォルト | 説明 |
...
```

<details>
<summary>ヒント</summary>

共通カラム（id, created_at, updated_at, deleted_at）を忘れずに含めてください。外部キーは users テーブルと categories テーブルへの参照が必要です。

</details>

### 問題3：ER図の作成

以下のシステムのER図を作成してください。

**システム：** 図書館の貸出管理
- ユーザー（利用者）が書籍を借りる
- 1人のユーザーは同時に最大5冊まで借りられる
- 書籍にはカテゴリ（小説、技術書、漫画等）がある
- 1冊の書籍は1つのカテゴリに属する
- 貸出記録（誰が、何を、いつ借りて、いつ返したか）を管理する

**期待される出力例：**

```
┌──────────┐     ┌──────────┐
│  users    │     │  books    │
├──────────┤     ├──────────┤
│ PK id     │     │ PK id     │
...
```

<details>
<summary>ヒント</summary>

必要なテーブルは以下の4つです。
1. users（利用者）
2. books（書籍）
3. categories（カテゴリ）
4. borrowings（貸出記録）- usersとbooksの関連テーブル

</details>

<details>
<summary>解答例</summary>

```
【ER図：図書館貸出管理】

┌──────────────┐        ┌──────────────┐
│   users       │        │  categories   │
├──────────────┤        ├──────────────┤
│ PK id         │        │ PK id         │
│    name       │        │    name       │
│    email      │        │    description│
│    phone      │        │    created_at │
│    is_active  │        └───────┬──────┘
│    created_at │                │1
│    updated_at │                │
└───────┬──────┘                │N
        │1                ┌─────▼────────┐
        │                 │    books      │
        │                 ├──────────────┤
        │N                │ PK id         │
┌───────▼──────┐         │    title      │
│  borrowings   │         │    author     │
├──────────────┤         │    isbn       │
│ PK id         │         │ FK category_id│
│ FK user_id   ─┼────────→│    stock      │
│ FK book_id   ─┼────────→│    is_available│
│    borrowed_at│         │    created_at │
│    due_date   │         └──────────────┘
│    returned_at│
│    status     │
│    created_at │
└──────────────┘

【リレーションシップ】
・users 1:N borrowings
  → 1人のユーザーは複数の貸出記録を持つ
  → ※ ビジネスルール：同時貸出は5冊まで

・books 1:N borrowings
  → 1冊の書籍は複数の貸出記録を持つ（過去の貸出含む）

・categories 1:N books
  → 1つのカテゴリに複数の書籍が属する

【テーブル定義概要】

users:
  id, name, email, phone, is_active, created_at, updated_at

categories:
  id, name, description, created_at

books:
  id, title, author, isbn(UNIQUE), category_id(FK),
  stock(蔵書数), is_available(貸出可能フラグ), created_at

borrowings:
  id, user_id(FK), book_id(FK), borrowed_at(貸出日),
  due_date(返却期限), returned_at(実返却日/NULL=未返却),
  status(borrowing/returned/overdue), created_at

【ビジネスルールの実装】
・同時貸出5冊制限：
  borrowingsテーブルで user_id ごとに
  returned_at IS NULL のレコード数が5未満であることを
  アプリケーション層でチェックする
・返却期限：
  borrowed_at + 14日 を due_date に設定する
・延滞チェック：
  due_date < 現在日時 AND returned_at IS NULL のレコードを
  バッチ処理で検出し、statusを 'overdue' に更新する
```

</details>

## 応用問題

### 問題4：インデックス設計

以下のクエリパターンに基づいて、`orders` テーブルのインデックスを設計してください。

**テーブル定義：**
```
orders: id, user_id, status, total_amount, ordered_at, created_at
```

**クエリパターン：**
1. 特定ユーザーの注文一覧を取得（user_id指定、ordered_at降順）
2. ステータスが「shipped」の注文一覧を取得
3. 特定期間の注文を合計金額の降順で取得
4. 特定ユーザーの「未完了」注文を取得（user_id + status）
5. 注文番号で注文を1件取得

**期待される出力例：**

```
| インデックス名 | カラム | 用途 |
...
```

<details>
<summary>ヒント</summary>

- 複合インデックスは左端のカラムから使われます
- クエリ4のような複合条件は複合インデックスが有効です
- ORDER BYのカラムもインデックスに含めるとソートが効率化されます

</details>

<details>
<summary>解答例</summary>

```
【インデックス設計】

| No | インデックス名                  | カラム                     | 種別    | 対応クエリ |
|----|-------------------------------|---------------------------|---------|----------|
| 1  | PRIMARY                       | id                        | PRIMARY | -        |
| 2  | idx_orders_user_ordered       | user_id, ordered_at DESC  | INDEX   | クエリ1,4|
| 3  | idx_orders_status             | status                    | INDEX   | クエリ2  |
| 4  | idx_orders_ordered_amount     | ordered_at, total_amount  | INDEX   | クエリ3  |
| 5  | idx_orders_user_status        | user_id, status           | INDEX   | クエリ4  |

【各インデックスの解説】

クエリ1: SELECT * FROM orders
         WHERE user_id = ? ORDER BY ordered_at DESC
  → idx_orders_user_ordered が最適。
    user_idでフィルタ後、ordered_atでソート済みのため
    追加ソート不要。

クエリ2: SELECT * FROM orders WHERE status = 'shipped'
  → idx_orders_status を使用。
    ただし statusのカーディナリティが低い場合は
    効果が限定的。データ量が多ければ有効。

クエリ3: SELECT * FROM orders
         WHERE ordered_at BETWEEN ? AND ?
         ORDER BY total_amount DESC
  → idx_orders_ordered_at で範囲検索は可能だが、
    total_amountのソートにはインデックスが効かない。
    データ量が多い場合はアプリ側でソートするか、
    別途対策を検討する。

クエリ4: SELECT * FROM orders
         WHERE user_id = ? AND status != 'delivered'
  → idx_orders_user_status の複合インデックスが最適。
    user_idで絞り込み→statusでフィルタ。
    idx_orders_user_ordered でもuser_id部分は使える。

クエリ5: SELECT * FROM orders WHERE order_number = ?
  → uq_orders_order_number（UNIQUE INDEX）を使用。
    1件取得なので最も効率的。

【注意事項】
・インデックスを増やすとINSERT/UPDATEの性能が低下する
・実際のクエリ頻度とデータ量を考慮して取捨選択する
・EXPLAIN文で実行計画を確認し、効果を検証する
```

</details>

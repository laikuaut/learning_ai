# 第2章：基本操作（add・commit・status・log）

## この章のゴール

- `git status` でファイルの状態を確認できるようになる
- ファイルのライフサイクル（Untracked → Staged → Committed）を理解する
- `git add` と `git commit` を使いこなせるようになる
- `git log` でコミット履歴を効率的に確認できるようになる
- `git diff` で変更内容を確認できるようになる
- 良いコミットメッセージの書き方を身につける

---

## 1. git status でファイル状態を確認する

`git status` は、現在のリポジトリの状態を確認する最も基本的なコマンドです。Git を使う上で最も頻繁に実行するコマンドと言っても過言ではありません。

```bash
# 現在の状態を確認
git status
```

### 出力の読み方

```bash
$ git status
On branch main                        # ← 現在のブランチ

Changes to be committed:              # ← ステージング済み（次のコミットに含まれる）
  (use "git restore --staged <file>..." to unstage)
        modified:   index.html

Changes not staged for commit:        # ← 変更済みだが未ステージング
  (use "git add <file>..." to update what will be committed)
  (use "git restore <file>..." to discard changes)
        modified:   style.css

Untracked files:                      # ← Git が追跡していない新規ファイル
  (use "git add <file>..." to include in what will be committed)
        script.js
```

### 短縮表示

`-s` オプションを使うと、短縮形式で表示できます。

```bash
$ git status -s
M  index.html    # M（左側）= ステージング済みの変更
 M style.css     # M（右側）= 未ステージングの変更
?? script.js     # ?? = 未追跡ファイル
```

| 記号 | 意味 |
|------|------|
| `??` | 未追跡（Untracked） |
| `A ` | 新規追加でステージング済み |
| `M ` | 変更してステージング済み（左側の M） |
| ` M` | 変更して未ステージング（右側の M） |
| `MM` | ステージング後にさらに変更あり |
| `D ` | 削除してステージング済み |

### ポイントまとめ

- `git status` はこまめに実行して、現在の状態を把握しましょう
- `-s` オプションで簡潔な表示が得られます
- 出力を読めるようになると、次に何をすべきかがわかります

## 2. ファイルのライフサイクル

Git で管理されるファイルには、いくつかの状態（state）があります。

```
┌──────────────────────────────────────────────────────────────┐
│                  ファイルのライフサイクル                       │
│                                                              │
│                        git add                               │
│  ┌───────────┐  ──────────────▶  ┌───────────┐              │
│  │ Untracked │                   │  Staged   │              │
│  │ (未追跡)   │                   │(ステージ済)│              │
│  └───────────┘  ◀──────────────  └─────┬─────┘              │
│       ▲          git rm --cached       │                     │
│       │                            git commit                │
│       │                                │                     │
│       │                                ▼                     │
│       │          git rm          ┌───────────┐              │
│       └───────────────────────── │ Committed │              │
│                                  │(コミット済)│              │
│                                  └─────┬─────┘              │
│                                        │                     │
│                                   ファイルを編集              │
│                                        │                     │
│                                        ▼                     │
│                                  ┌───────────┐              │
│                     git add      │ Modified  │              │
│              ┌───────────────── │ (変更済み) │              │
│              │                   └───────────┘              │
│              ▼                                               │
│        ┌───────────┐                                        │
│        │  Staged   │ ──── git commit ────▶ Committed        │
│        └───────────┘                                        │
└──────────────────────────────────────────────────────────────┘
```

| 状態 | 説明 |
|------|------|
| Untracked（未追跡） | Git が存在を認識しているが、追跡していないファイル |
| Staged（ステージ済み） | 次のコミットに含まれるよう登録されたファイル |
| Committed（コミット済み） | リポジトリに安全に保存されたファイル |
| Modified（変更済み） | コミット済みのファイルに変更を加えた状態 |

### 実際に確認してみる

```bash
# 新しいリポジトリで試す
mkdir lifecycle-demo && cd lifecycle-demo
git init

# 1. ファイルを作成 → Untracked
echo "Hello" > hello.txt
git status
# Untracked files:
#         hello.txt

# 2. ステージング → Staged
git add hello.txt
git status
# Changes to be committed:
#         new file:   hello.txt

# 3. コミット → Committed
git commit -m "hello.txt を追加"
git status
# nothing to commit, working tree clean

# 4. ファイルを編集 → Modified
echo "World" >> hello.txt
git status
# Changes not staged for commit:
#         modified:   hello.txt
```

### ポイントまとめ

- ファイルは Untracked → Staged → Committed → Modified のサイクルを繰り返します
- 各状態を `git status` で確認できます
- この流れを理解することが Git 操作の基盤になります

## 3. git add でステージングする

`git add` は、ワーキングツリーの変更をステージングエリアに追加するコマンドです。

### さまざまな指定方法

```bash
# 特定のファイルをステージング
git add hello.txt

# 複数のファイルをステージング
git add hello.txt world.txt

# 特定の拡張子をすべてステージング
git add *.py

# 特定のディレクトリ以下をすべてステージング
git add src/

# 変更のあるすべてのファイルをステージング（新規ファイルも含む）
git add -A

# カレントディレクトリ以下の変更をステージング
git add .
```

### git add . と git add -A の違い

> **よくある間違い：** `git add .` と `git add -A` は似ていますが、動作が異なる場合があります。

| コマンド | 新規ファイル | 変更ファイル | 削除ファイル | 対象範囲 |
|----------|:---:|:---:|:---:|------|
| `git add .` | 追加 | 追加 | 追加 | カレントディレクトリ以下 |
| `git add -A` | 追加 | 追加 | 追加 | リポジトリ全体 |

現在のディレクトリがリポジトリのルートであれば同じ動作になりますが、サブディレクトリにいる場合は対象範囲が異なります。

```bash
# 例：サブディレクトリにいる場合
cd src/

git add .     # src/ 以下の変更のみステージング
git add -A    # リポジトリ全体の変更をステージング
```

### ステージングの取り消し

間違えてステージングした場合は、`git restore --staged` で取り消せます。

```bash
# 特定のファイルのステージングを取り消す
git restore --staged hello.txt

# すべてのステージングを取り消す
git restore --staged .
```

### git add -p で部分的にステージング

ファイルの変更の一部だけをステージングしたい場合は、`-p`（patch）オプションを使います。

```bash
# 変更を確認しながら選択的にステージング
git add -p hello.txt
# y = この変更をステージング（yes）
# n = この変更をスキップ（no）
# s = この変更をさらに分割（split）
# q = 終了（quit）
```

これにより、1つのファイルの変更を複数のコミットに分けることができます。現場では非常によく使われるテクニックです。

### ポイントまとめ

- `git add` でステージングエリアに変更を追加します
- ファイル指定、ディレクトリ指定、パターン指定など柔軟に使えます
- `git add -p` で変更の一部だけをステージングできます
- 間違えたら `git restore --staged` で取り消せます

## 4. git commit で変更を記録する

`git commit` は、ステージングエリアの変更をリポジトリに記録するコマンドです。コミット（commit）は Git の最も基本的な単位で、ある時点のスナップショットを保存します。

### 基本的な使い方

```bash
# メッセージ付きでコミット（最もよく使う形式）
git commit -m "ユーザー登録機能を追加"

# エディタでメッセージを書いてコミット
git commit
# → 設定されたエディタが開く

# ステージングとコミットを同時に行う（追跡済みファイルのみ）
git commit -am "ヘッダーのスタイルを修正"
```

> **よくある間違い：** `git commit -am` は **追跡済みファイル** の変更のみをコミットします。新規ファイル（Untracked）は含まれません。新規ファイルは必ず先に `git add` してください。

### コミットの中身

各コミットには以下の情報が含まれます。

```
┌─────────────────────────────────────────┐
│ commit a1b2c3d4e5f6...                  │  ← SHA-1 ハッシュ（一意のID）
│                                         │
│ Author: Taro Yamada <taro@example.com>  │  ← 作者
│ Date:   Thu Mar 27 10:00:00 2026 +0900  │  ← 日時
│                                         │
│ ユーザー登録機能を追加                     │  ← コミットメッセージ
│                                         │
│ Parent: 9f8e7d6c5b4a...                 │  ← 親コミット（前のコミット）
│ Tree:   1a2b3c4d5e6f...                 │  ← スナップショット
└─────────────────────────────────────────┘
```

### 良いコミットメッセージの書き方

コミットメッセージは、将来の自分やチームメンバーが変更内容を理解するための重要な情報です。

**悪い例：**
```
fix
修正
update
あれを直した
WIP
```

**良い例：**
```
ユーザー登録時のメールバリデーションを追加
ログインページのCSRF対策を修正
README にインストール手順を追記
不要な console.log を削除
```

### コミットメッセージの慣例（Conventional Commits）

多くのプロジェクトで採用されている書き方です。

```
<type>: <description>

feat: ユーザー登録機能を追加
fix: ログイン時のパスワード検証エラーを修正
docs: README にセットアップ手順を追記
style: コードのインデントを統一
refactor: ユーザーモデルのバリデーションを整理
test: ユーザー登録のテストを追加
chore: 依存パッケージを更新
```

| type | 用途 |
|------|------|
| `feat` | 新機能の追加 |
| `fix` | バグ修正 |
| `docs` | ドキュメントのみの変更 |
| `style` | コードの意味に影響しない変更（フォーマット等） |
| `refactor` | バグ修正でも新機能でもないコード変更 |
| `test` | テストの追加・修正 |
| `chore` | ビルドプロセスやツールの変更 |

### ポイントまとめ

- `git commit -m "メッセージ"` で変更をリポジトリに記録します
- コミットメッセージは「何をしたか」が明確にわかるように書きましょう
- 1つのコミットには1つの論理的な変更をまとめるのがベストプラクティスです
- Conventional Commits を使うと履歴が読みやすくなります

## 5. git log でコミット履歴を確認する

`git log` は、コミット履歴を表示するコマンドです。

### 基本的な使い方

```bash
# 基本的な履歴表示
git log
# 出力:
# commit a1b2c3d (HEAD -> main)
# Author: Taro Yamada <taro@example.com>
# Date:   Thu Mar 27 10:05:00 2026 +0900
#
#     ユーザー登録機能を追加
#
# commit 9f8e7d6
# Author: Taro Yamada <taro@example.com>
# Date:   Thu Mar 27 09:30:00 2026 +0900
#
#     最初のコミット: READMEを追加
```

### よく使うオプション

```bash
# 1行表示（コンパクト）
git log --oneline
# 出力:
# a1b2c3d (HEAD -> main) ユーザー登録機能を追加
# 9f8e7d6 最初のコミット: READMEを追加

# ブランチの分岐・マージをグラフ表示
git log --oneline --graph
# 出力:
# * a1b2c3d (HEAD -> main) Merge branch 'feature/login'
# |\
# | * e5f6a7b feat: ログイン機能を追加
# |/
# * 9f8e7d6 最初のコミット: READMEを追加

# 全ブランチの履歴をグラフ表示（全体像の把握に便利）
git log --oneline --graph --all

# 表示件数を制限
git log --oneline -5

# 特定のファイルの変更履歴
git log -- README.md

# 変更内容（差分）も表示
git log -p

# 変更されたファイルの統計情報を表示
git log --stat
```

### 便利な検索オプション

```bash
# コミットメッセージでフィルタ
git log --grep="バグ修正"

# 特定の著者のコミットのみ表示
git log --author="Taro"

# 日付で絞り込み
git log --since="2026-03-01" --until="2026-03-31"

# 特定の文字列の変更を含むコミットを検索（pickaxe）
git log -S "function login"
```

### ポイントまとめ

- `git log --oneline` がコンパクトで普段使いに便利です
- `--graph --all` でブランチの全体像を可視化できます
- `--grep`、`--author`、`--since` で履歴を絞り込めます

## 6. git diff で変更内容を確認する

`git diff` は、ファイルの変更内容（差分）を表示するコマンドです。コミットやステージングの前に、変更内容を確認する習慣をつけましょう。

### 3つのエリア間の比較

```
  ワーキング          ステージング          リポジトリ
  ツリー              エリア              (HEAD)
  ┌────────┐         ┌────────┐         ┌────────┐
  │        │         │        │         │        │
  │  編集中 │         │ add済み │         │commit済│
  │        │         │        │         │        │
  └────────┘         └────────┘         └────────┘
       │                  │                  │
       │◀─ git diff ────▶│                  │
       │   (デフォルト)    │                  │
       │                  │◀─ git diff ────▶│
       │                  │   --staged       │
       │                                     │
       │◀──────── git diff HEAD ───────────▶│
```

```bash
# ワーキングツリー vs ステージングエリア（未ステージの変更を表示）
git diff

# ステージングエリア vs 最新コミット（次のコミットに含まれる変更を表示）
git diff --staged
# または
git diff --cached

# ワーキングツリー vs 最新コミット（すべての変更を表示）
git diff HEAD

# 特定のファイルだけの差分
git diff hello.txt

# コミット間の差分
git diff a1b2c3d 9f8e7d6
```

### diff の読み方

```diff
$ git diff
diff --git a/hello.txt b/hello.txt
index ce01362..a042389 100644
--- a/hello.txt                    # ← 変更前のファイル
+++ b/hello.txt                    # ← 変更後のファイル
@@ -1,2 +1,3 @@                   # ← 変更箇所の位置情報
 Hello                             # ← 変更なしの行（空白で始まる）
-World                             # ← 削除された行（- で始まる）
+Beautiful World                   # ← 追加された行（+ で始まる）
+Welcome to Git!                   # ← 追加された行
```

### 実践例

```bash
# 試してみましょう
mkdir diff-demo && cd diff-demo
git init

# ファイルを作成してコミット
cat << 'EOF' > hello.txt
Hello
World
EOF
git add hello.txt
git commit -m "hello.txt を作成"

# ファイルを編集
cat << 'EOF' > hello.txt
Hello
Beautiful World
Welcome to Git!
EOF

# 未ステージの変更を確認
git diff
# -World
# +Beautiful World
# +Welcome to Git!

# ステージング
git add hello.txt

# ステージ済みの変更を確認
git diff --staged
# -World
# +Beautiful World
# +Welcome to Git!

# 未ステージの変更はもうない
git diff
# （何も表示されない）
```

### ポイントまとめ

- `git diff` は未ステージの変更を表示します
- `git diff --staged` はステージ済みの変更を表示します
- `git diff HEAD` はすべての変更を表示します
- diff の出力で `-` が削除行、`+` が追加行です

## 7. git show でコミット内容を確認する

`git show` は、特定のコミットの詳細情報を表示するコマンドです。

```bash
# 最新のコミットの詳細を表示
git show

# 特定のコミットを指定して表示
git show 9f8e7d6

# 特定のコミット内の特定ファイルの内容を表示
git show a1b2c3d:hello.txt

# 統計情報のみ表示
git show --stat
```

### ポイントまとめ

- `git show` で最新コミットの詳細を確認できます
- コミットハッシュを指定して、任意のコミットの内容を見ることができます

## 8. .gitignore の詳細

第1章で基本を学んだ `.gitignore` について、より詳細なパターン記法を紹介します。

### パターン記法の詳細

```bash
# === 基本パターン ===

# 特定のファイル名
secret.txt

# ワイルドカード（*）: 任意の文字列にマッチ
*.log
*.tmp

# 疑問符（?）: 任意の1文字にマッチ
temp?.txt        # temp1.txt, tempA.txt にマッチ

# 角括弧: 指定した文字のいずれかにマッチ
file[0-9].txt    # file0.txt 〜 file9.txt にマッチ

# === ディレクトリ ===

# 末尾の / はディレクトリを意味する
logs/            # logs ディレクトリ全体を無視

# 先頭の / はリポジトリルートからの相対パス
/TODO.txt        # ルートの TODO.txt のみ無視（サブディレクトリの TODO.txt は追跡）

# === 再帰パターン ===

# ** は任意の階層にマッチ
**/logs          # どの階層の logs ディレクトリも無視
logs/**          # logs 以下のすべてを無視
**/logs/**       # どの階層の logs 以下のすべてを無視

# === 否定パターン ===

# !（感嘆符）で例外を指定
*.log
!important.log   # important.log だけは追跡する
```

> **よくある間違い：** 親ディレクトリが無視されている場合、その中のファイルを `!` で例外にすることはできません。

```bash
# これは効かない
build/
!build/result.txt   # ← build/ が先に無視されるため無効

# 正しいやり方
build/*              # build 直下のファイルを無視（ディレクトリ自体は無視しない）
!build/result.txt    # ← これなら有効
```

### グローバル .gitignore

OS 固有のファイルなど、すべてのプロジェクトで共通して無視したいパターンはグローバル `.gitignore` に設定できます。

```bash
# グローバル .gitignore を設定
git config --global core.excludesfile ~/.gitignore_global

# グローバル .gitignore を作成
cat << 'EOF' > ~/.gitignore_global
# macOS
.DS_Store

# Windows
Thumbs.db
desktop.ini

# エディタ
*.swp
*~
.idea/
.vscode/
EOF
```

### ポイントまとめ

- `*`、`?`、`[]`、`**` などのパターン記法を使いこなしましょう
- `/` の位置でルートからの相対パスかどうかが変わります
- `!` で例外を指定できますが、親ディレクトリが無視されていると効きません
- グローバル `.gitignore` で OS 固有のファイルを一括管理できます

## 基本問題

### 問題1：リスクと課題の区別

以下の項目が「リスク」か「課題」かを判定し、理由を述べてください。

1. テスト環境のサーバーが昨日ダウンし、テストが中断している
2. 主要なバックエンドエンジニアが来月退職する可能性がある
3. 外部APIの仕様書がまだ届いていない
4. 新しいフレームワークの採用により、学習コストが発生するかもしれない
5. 本番デプロイ時にDBマイグレーションが失敗し、ロールバック中

**期待される出力例：**

```
1. 課題 - すでに発生しており、即座の対応が必要
2. リスク - まだ発生しておらず、将来起きるかもしれない事象
```

<details>
<summary>解答例</summary>

```
1. 課題（Issue）
   理由：サーバーダウンは既に発生した事象。即座に復旧対応が必要。

2. リスク（Risk）
   理由：退職は「可能性がある」段階で、まだ発生していない。
   事前に対策（知識の引き継ぎ、代替要員の確保）を準備できる。

3. 課題（Issue）
   理由：仕様書が届いていないのは現在の問題。
   催促や代替手段の検討など即座のアクションが必要。

4. リスク（Risk）
   理由：学習コストが「発生するかもしれない」という不確実な事象。
   事前に研修計画を立てるなどの予防策が取れる。

5. 課題（Issue）
   理由：デプロイ失敗は既に発生した事象。
   ロールバック対応が進行中であり、現在進行形の問題。
```

</details>

### 問題2：リスクの確率・影響度マトリクス

以下のリスクについて、確率（極低〜極高）と影響度（極低〜極高）を評価し、優先度をつけてください。

| リスク | 説明 |
|--------|------|
| R1 | 使用する外部APIが提供遅延する可能性がある（先方のスケジュールが不透明） |
| R2 | 台風でオフィスが使えなくなる可能性がある（9月リリースのプロジェクト） |
| R3 | 要件の追加変更が発生する可能性が高い（POが頻繁に方針を変える傾向） |
| R4 | 使用するOSSライブラリに重大な脆弱性が発見される可能性がある |
| R5 | チームメンバーの1人が育児休暇を取得する可能性がある |

**期待される出力例：**

```
R1: 確率 中、影響度 高 → 優先度：高
```

<details>
<summary>解答例</summary>

```
| リスク | 確率  | 影響度 | 優先度 | 理由                                    |
|--------|-------|--------|--------|-----------------------------------------|
| R1     | 中    | 高     | 高     | 先方のスケジュールが不透明。遅延すると   |
|        |       |        |        | 結合テストが実施できず全体に影響大        |
| R2     | 低    | 中     | 中     | 9月は台風シーズンだが、リモートワークで  |
|        |       |        |        | 代替可能であれば影響は限定的              |
| R3     | 高    | 高     | 極高   | POの傾向から変更発生は高確率。           |
|        |       |        |        | スコープクリープにつながり影響度も大      |
| R4     | 低    | 高     | 中     | 脆弱性発見の確率は低いが、発見された     |
|        |       |        |        | 場合は緊急対応が必要で影響度は高い        |
| R5     | 中    | 中     | 中     | 育児休暇は予測可能。事前に引き継ぎ計画   |
|        |       |        |        | を立てておけば影響は軽減できる            |

優先順位：R3 > R1 > R4 = R5 = R2
→ R3（要件変更）とR1（外部API遅延）を最優先で対策を立てる
```

</details>

### 問題3：リスク対応戦略の選択

以下のリスクに対して、最適な対応戦略（回避・軽減・転嫁・受容）を選び、具体的な対策を述べてください。

1. 新技術（GraphQL）の採用により、実装工数が増加するリスク
2. データセンターの自然災害によるデータ損失リスク
3. テスト自動化ツールのライセンス費が値上がりするリスク
4. プロジェクト後半で軽微なUI修正が必要になるリスク

<details>
<summary>ヒント</summary>

- 回避：リスクの原因を除去する
- 軽減：発生確率または影響度を下げる
- 転嫁：リスクを第三者に移す（保険、外部委託等）
- 受容：リスクを認識した上で、発生したら対処する

</details>

<details>
<summary>解答例</summary>

```
1. 新技術（GraphQL）の実装工数増加リスク → 軽減

   対策：
   ・事前にチーム内でGraphQLのトレーニングを実施する（1週間）
   ・プロトタイプを作成し、技術的な不確実性を早期に解消する
   ・GraphQL経験者のアドバイザリー契約を結ぶ
   ・見積もりに学習コスト分のバッファ（+20%）を含める

   ※ 回避（GraphQLを使わない）も選択肢だが、技術的なメリットが
   大きい場合は軽減策で対応する方が合理的。

2. 自然災害によるデータ損失リスク → 転嫁 + 軽減

   対策（転嫁）：
   ・クラウドサービス（AWS、GCP）を利用し、データの保全を
     クラウドベンダーのSLAで担保する
   ・サイバー保険に加入する

   対策（軽減）：
   ・マルチリージョンでのバックアップを設定する
   ・DR（ディザスターリカバリー）計画を策定する
   ・定期的にバックアップの復元テストを実施する

3. ライセンス費の値上がりリスク → 受容（能動的受容）

   対策：
   ・予算に予備費（コンティンジェンシーリザーブ）を確保しておく
   ・値上がりが確認された時点で代替ツールを検討するトリガーを設定
   ・現時点では発生確率・影響度ともに低いため、監視を継続する

4. 軽微なUI修正リスク → 受容（能動的受容）

   対策：
   ・UI修正はプロジェクトの通常の範囲内として予測されるもの
   ・スプリントのバッファとして毎回10%程度の余裕を持たせる
   ・発生した場合はバックログに追加し、優先度に応じて対応する
```

</details>

## チャレンジ問題

### 問題5：期待金額価値（EMV）分析

以下のリスク一覧について、期待金額価値（EMV: Expected Monetary Value）を計算し、プロジェクトのコンティンジェンシーリザーブ（予備費）を算出してください。

| リスク | 発生確率 | 発生時の影響額 |
|--------|---------|-------------|
| 外部API連携の遅延 | 40% | 200万円（追加工数） |
| 要件追加によるスコープ拡大 | 60% | 300万円（追加開発費） |
| テスト環境の障害 | 20% | 50万円（復旧費用） |
| メンバー1名の1ヶ月離脱 | 15% | 120万円（代替要員費） |
| 性能問題の発覚 | 30% | 150万円（チューニング費） |

**期待される出力例：**

```
EMV合計（コンティンジェンシーリザーブ）= ???万円
```

<details>
<summary>ヒント</summary>

EMV = 発生確率 x 影響額
コンティンジェンシーリザーブ = 全リスクのEMVの合計

</details>

<details>
<summary>解答例</summary>

```
■ 期待金額価値（EMV）の計算

| リスク                    | 確率  | 影響額   | EMV                |
|--------------------------|-------|---------|-------------------|
| 外部API連携の遅延         | 40%   | 200万円  | 0.40 × 200 = 80万円 |
| 要件追加によるスコープ拡大  | 60%   | 300万円  | 0.60 × 300 = 180万円|
| テスト環境の障害           | 20%   | 50万円   | 0.20 × 50 = 10万円  |
| メンバー1名の1ヶ月離脱    | 15%   | 120万円  | 0.15 × 120 = 18万円 |
| 性能問題の発覚            | 30%   | 150万円  | 0.30 × 150 = 45万円 |
|                          |       |         |                    |
| **合計**                  |       |         | **333万円**         |

■ コンティンジェンシーリザーブ = 333万円

これはプロジェクトの見積もりに加算すべき予備費の金額です。

■ 注意点
・EMVは統計的な期待値であり、実際にこの金額が発生するわけではない
・すべてのリスクが同時に発生する可能性は低い
・しかし、最悪のケース（全リスク発生）では
  200 + 300 + 50 + 120 + 150 = 820万円が必要
・現実的にはEMV合計（333万円）〜最悪ケースの80%（656万円）の
  範囲で予備費を設定するのが一般的

■ 予算への反映例
  プロジェクト見積もり     : 2,000万円
  + コンティンジェンシー   :   333万円
  ─────────────────
  コストベースライン       : 2,333万円
  + マネジメントリザーブ(5%):   117万円
  ─────────────────
  プロジェクト予算          : 2,450万円
```

</details>

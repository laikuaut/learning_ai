# 第3章：パーミッションとユーザー管理

## この章のゴール

- Linuxのパーミッション（アクセス権）の仕組みを理解し、適切に設定できるようになること
- ユーザーとグループの概念を理解し、管理操作ができるようになること

## 3.1 Linuxのユーザーとグループ

Linuxはマルチユーザーシステムです。複数のユーザーが同時にシステムを利用できる設計になっており、セキュリティの根幹として「ユーザー」と「グループ」という概念があります。

### ユーザーの種類

```
┌─────────────────────────────────────────────────────┐
│                  Linuxのユーザー                      │
├──────────────┬──────────────┬───────────────────────┤
│  rootユーザー │  一般ユーザー │  システムユーザー       │
│  (UID=0)     │  (UID≧1000)  │  (UID=1〜999)         │
│              │              │                       │
│  全権限を持つ │  制限された   │  サービス実行用        │
│  管理者      │  権限を持つ   │  (nginx, mysql等)     │
└──────────────┴──────────────┴───────────────────────┘
```

| 種類 | 説明 | UID範囲 | 例 |
|------|------|---------|-----|
| rootユーザー | システム管理者。全てのファイルにアクセス可能 | 0 | root |
| 一般ユーザー | 通常の作業を行うユーザー | 1000以上 | tanaka, suzuki |
| システムユーザー | サービスやデーモンの実行用 | 1〜999 | www-data, mysql, nobody |

### ユーザー情報の確認

```bash
# 現在のユーザー名を確認
$ whoami
tanaka

# 現在のユーザーのUID、GID、所属グループを確認
$ id
uid=1000(tanaka) gid=1000(tanaka) groups=1000(tanaka),27(sudo),33(www-data)

# 特定のユーザーの情報
$ id suzuki
uid=1001(suzuki) gid=1001(suzuki) groups=1001(suzuki)

# 現在ログインしているユーザー一覧
$ who
tanaka   pts/0        2024-03-20 10:00 (192.168.1.10)
suzuki   pts/1        2024-03-20 11:30 (192.168.1.20)

# より詳細なログイン情報
$ w
 15:00:01 up 30 days,  5:23,  2 users,  load average: 0.15, 0.10, 0.08
USER     TTY      FROM             LOGIN@   IDLE   JCPU   PCPU WHAT
tanaka   pts/0    192.168.1.10     10:00    0.00s  0.05s  0.00s w
suzuki   pts/1    192.168.1.20     11:30    1:30m  0.02s  0.02s vim report.txt
```

### /etc/passwd ファイル

すべてのユーザー情報は `/etc/passwd` ファイルに記録されています。

```bash
$ cat /etc/passwd
root:x:0:0:root:/root:/bin/bash
daemon:x:1:1:daemon:/usr/sbin:/usr/sbin/nologin
www-data:x:33:33:www-data:/var/www:/usr/sbin/nologin
tanaka:x:1000:1000:Tanaka Taro:/home/tanaka:/bin/bash
suzuki:x:1001:1001:Suzuki Hanako:/home/suzuki:/bin/bash
```

各フィールドの意味：

```
tanaka:x:1000:1000:Tanaka Taro:/home/tanaka:/bin/bash
  │    │  │    │       │           │           │
  │    │  │    │       │           │           └─ ログインシェル
  │    │  │    │       │           └───────────── ホームディレクトリ
  │    │  │    │       └───────────────────────── コメント（氏名等）
  │    │  │    └───────────────────────────────── GID（プライマリグループID）
  │    │  └────────────────────────────────────── UID（ユーザーID）
  │    └───────────────────────────────────────── パスワード（xは/etc/shadowに格納）
  └────────────────────────────────────────────── ユーザー名
```

### グループの概念

グループは複数のユーザーをまとめる仕組みです。ファイルのアクセス権をグループ単位で管理できます。

```bash
# 自分が所属しているグループを確認
$ groups
tanaka sudo www-data

# 特定ユーザーの所属グループを確認
$ groups suzuki
suzuki : suzuki developers

# グループ一覧は /etc/group に記録されている
$ cat /etc/group
root:x:0:
sudo:x:27:tanaka
www-data:x:33:tanaka
tanaka:x:1000:
developers:x:1002:suzuki,yamada
```

グループファイルの各フィールド：

```
developers:x:1002:suzuki,yamada
    │       │  │      │
    │       │  │      └─ メンバー（カンマ区切り）
    │       │  └──────── GID（グループID）
    │       └─────────── パスワード（通常は使わない）
    └─────────────────── グループ名
```

## 3.2 パーミッション（アクセス権）の基本

Linuxでは、すべてのファイルとディレクトリに「誰が」「何をできるか」を制御するパーミッションが設定されています。これはLinuxのセキュリティモデルの根幹です。

### パーミッションの表示

```bash
$ ls -l
-rw-r--r-- 1 tanaka developers  4096 Mar 20 15:00 report.txt
drwxrwxr-x 2 tanaka developers  4096 Mar 20 14:00 shared/
-rwxr-x--- 1 root   root        8192 Mar 18 09:00 backup.sh
lrwxrwxrwx 1 tanaka tanaka        15 Mar 15 10:00 link.txt -> original.txt
```

### パーミッション文字列の読み方

```
-rwxr-xr--
│├─┤├─┤├─┤
│ │   │  │
│ │   │  └─ その他のユーザー（Other）: r-- = 読み取りのみ
│ │   └──── グループ（Group）: r-x = 読み取り + 実行
│ └──────── 所有者（Owner/User）: rwx = 読み取り + 書き込み + 実行
└────────── ファイルタイプ
```

### ファイルタイプの記号

| 記号 | 種類 | 説明 |
|------|------|------|
| `-` | 通常ファイル | テキスト、バイナリ等 |
| `d` | ディレクトリ | フォルダ |
| `l` | シンボリックリンク | ショートカットのようなもの |
| `c` | キャラクタデバイス | /dev/tty 等 |
| `b` | ブロックデバイス | /dev/sda 等 |
| `s` | ソケット | プロセス間通信用 |
| `p` | 名前付きパイプ | FIFO |

### 3つの権限（rwx）

| 権限 | 文字 | 数値 | ファイルの場合 | ディレクトリの場合 |
|------|------|------|----------------|-------------------|
| 読み取り | `r` | 4 | ファイルの内容を読める | ディレクトリ内のファイル一覧を取得できる |
| 書き込み | `w` | 2 | ファイルの内容を変更できる | ディレクトリ内にファイルを作成・削除できる |
| 実行 | `x` | 1 | プログラムとして実行できる | ディレクトリに `cd` で移動できる |

> **重要**: ディレクトリの場合、`x`（実行権限）がないとそのディレクトリに `cd` できません。`r` があっても `x` がなければ、ファイル名の一覧は取得できますが中身へのアクセスはできません。これは初心者がよく混乱するポイントです。

### 3つの対象

| 対象 | 記号 | 説明 |
|------|------|------|
| 所有者（User） | `u` | ファイルの所有者 |
| グループ（Group） | `g` | ファイルが属するグループのメンバー |
| その他（Other） | `o` | 上記以外の全ユーザー |
| 全員（All） | `a` | u + g + o のすべて |

### 数値表記（8進数）

パーミッションは数値でも表現できます。実務ではこちらを使うことが多いです。

```
rwx = 4 + 2 + 1 = 7
rw- = 4 + 2 + 0 = 6
r-x = 4 + 0 + 1 = 5
r-- = 4 + 0 + 0 = 4
--- = 0 + 0 + 0 = 0
```

よく使われるパーミッションの組み合わせ：

| 数値 | 文字表記 | 意味 | 用途 |
|------|----------|------|------|
| `755` | `rwxr-xr-x` | 所有者は全権限、他は読み取り+実行 | スクリプト、公開ディレクトリ |
| `644` | `rw-r--r--` | 所有者は読み書き、他は読み取りのみ | 設定ファイル、HTMLファイル |
| `700` | `rwx------` | 所有者のみ全権限 | 秘密鍵のディレクトリ (.ssh) |
| `600` | `rw-------` | 所有者のみ読み書き | 秘密鍵ファイル、パスワードファイル |
| `775` | `rwxrwxr-x` | 所有者とグループは全権限 | 共有ディレクトリ |
| `664` | `rw-rw-r--` | 所有者とグループは読み書き | 共有ファイル |
| `444` | `r--r--r--` | 全員が読み取りのみ | 重要な設定ファイル（変更防止） |

## 3.3 chmod - パーミッションの変更

`chmod`（change mode）コマンドでファイルやディレクトリのパーミッションを変更できます。

### 書式

```
chmod [オプション] モード ファイル名
```

### 主要なオプション

| オプション | 説明 |
|-----------|------|
| `-R` | ディレクトリ以下を再帰的に変更 |
| `-v` | 変更内容を表示 |
| `--reference=FILE` | 指定したファイルと同じパーミッションにする |

### 数値モードでの変更（最もよく使う方法）

```bash
# スクリプトに実行権限を付与（所有者:全権限、他:読み+実行）
$ chmod 755 deploy.sh
$ ls -l deploy.sh
-rwxr-xr-x 1 tanaka tanaka 2048 Mar 20 15:00 deploy.sh

# 設定ファイルの標準的なパーミッション
$ chmod 644 config.yaml
$ ls -l config.yaml
-rw-r--r-- 1 tanaka tanaka 512 Mar 20 15:00 config.yaml

# SSH秘密鍵のパーミッション（必須：600でないとSSHが拒否する）
$ chmod 600 ~/.ssh/id_rsa
$ ls -l ~/.ssh/id_rsa
-rw------- 1 tanaka tanaka 3243 Mar 10 09:00 /home/tanaka/.ssh/id_rsa

# .sshディレクトリのパーミッション（必須：700）
$ chmod 700 ~/.ssh
$ ls -ld ~/.ssh
drwx------ 2 tanaka tanaka 4096 Mar 10 09:00 /home/tanaka/.ssh

# ディレクトリ以下を再帰的に変更
$ chmod -R 755 /var/www/html/
```

### 記号モードでの変更

記号モードは「誰に」「何を」「どうするか」を直感的に指定できます。

```bash
# 書式: chmod [対象][操作][権限] ファイル
#   対象: u(所有者), g(グループ), o(その他), a(全員)
#   操作: +(追加), -(削除), =(設定)
#   権限: r(読み取り), w(書き込み), x(実行)

# 所有者に実行権限を追加
$ chmod u+x script.sh
$ ls -l script.sh
-rwxr--r-- 1 tanaka tanaka 1024 Mar 20 15:00 script.sh

# グループとその他から書き込み権限を削除
$ chmod go-w shared.txt

# 全員に読み取り権限を設定（既存の権限を上書き）
$ chmod a=r readonly.txt
$ ls -l readonly.txt
-r--r--r-- 1 tanaka tanaka 256 Mar 20 15:00 readonly.txt

# 所有者に読み書き実行、グループに読み実行、その他に権限なし
$ chmod u=rwx,g=rx,o= private_script.sh
$ ls -l private_script.sh
-rwxr-x--- 1 tanaka tanaka 512 Mar 20 15:00 private_script.sh

# 全員に実行権限を追加（よく使うパターン）
$ chmod +x run.sh

# 複数の操作を同時に
$ chmod u+x,g-w,o-rwx important.sh
```

### 実行結果の確認

```bash
# 変更前の確認
$ ls -l myfile.txt
-rw-rw-r-- 1 tanaka developers 1024 Mar 20 15:00 myfile.txt

# パーミッションを変更
$ chmod 640 myfile.txt

# 変更後の確認
$ ls -l myfile.txt
-rw-r----- 1 tanaka developers 1024 Mar 20 15:00 myfile.txt
```

### stat コマンドで詳細を確認

```bash
$ stat myfile.txt
  File: myfile.txt
  Size: 1024            Blocks: 8          IO Block: 4096   regular file
Device: 801h/2049d      Inode: 262148      Links: 1
Access: (0640/-rw-r-----)  Uid: ( 1000/  tanaka)   Gid: ( 1002/developers)
Access: 2024-03-20 15:00:00.000000000 +0900
Modify: 2024-03-20 15:00:00.000000000 +0900
Change: 2024-03-20 15:05:00.000000000 +0900
 Birth: 2024-03-20 14:00:00.000000000 +0900
```

## 3.4 chown - 所有者の変更

`chown`（change owner）コマンドでファイルの所有者やグループを変更できます。

### 書式

```
chown [オプション] [所有者][:グループ] ファイル名
```

### 主要なオプション

| オプション | 説明 |
|-----------|------|
| `-R` | ディレクトリ以下を再帰的に変更 |
| `-v` | 変更内容を表示 |
| `--reference=FILE` | 指定ファイルと同じ所有者/グループにする |

### 使用例

```bash
# 所有者を変更（root権限が必要）
$ sudo chown suzuki report.txt
$ ls -l report.txt
-rw-r--r-- 1 suzuki tanaka 1024 Mar 20 15:00 report.txt

# 所有者とグループを同時に変更
$ sudo chown suzuki:developers report.txt
$ ls -l report.txt
-rw-r--r-- 1 suzuki developers 1024 Mar 20 15:00 report.txt

# グループのみ変更（所有者の前のコロンだけ指定）
$ sudo chown :developers report.txt
$ ls -l report.txt
-rw-r--r-- 1 tanaka developers 1024 Mar 20 15:00 report.txt

# ディレクトリ以下を再帰的に変更（Webサーバーの公開ディレクトリなど）
$ sudo chown -R www-data:www-data /var/www/html/
$ ls -l /var/www/html/
total 8
-rw-r--r-- 1 www-data www-data 4096 Mar 20 15:00 index.html
drwxr-xr-x 2 www-data www-data 4096 Mar 20 15:00 images/
```

### chgrp - グループのみ変更

```bash
# グループのみを変更する専用コマンド
$ sudo chgrp developers shared_file.txt
$ ls -l shared_file.txt
-rw-rw-r-- 1 tanaka developers 512 Mar 20 15:00 shared_file.txt

# 再帰的にグループ変更
$ sudo chgrp -R developers /opt/project/
```

## 3.5 ユーザーの管理

### useradd - ユーザーの追加

```bash
# 基本的なユーザー追加
$ sudo useradd yamada

# ホームディレクトリを自動作成して追加（推奨）
$ sudo useradd -m yamada
$ ls /home/
tanaka  suzuki  yamada

# 各種オプションを指定して追加
$ sudo useradd -m -s /bin/bash -G sudo,developers -c "Yamada Jiro" yamada
# -m: ホームディレクトリを作成
# -s: ログインシェルを指定
# -G: 追加グループを指定（カンマ区切り）
# -c: コメント（氏名等）

# 実行結果の確認
$ id yamada
uid=1002(yamada) gid=1002(yamada) groups=1002(yamada),27(sudo),1003(developers)

$ grep yamada /etc/passwd
yamada:x:1002:1002:Yamada Jiro:/home/yamada:/bin/bash
```

### adduser - 対話的なユーザー追加（Debian/Ubuntu系）

```bash
# 対話的にユーザーを追加（より親切なインターフェース）
$ sudo adduser yamada
Adding user `yamada' ...
Adding new group `yamada' (1002) ...
Adding new user `yamada' (1002) with group `yamada' ...
Creating home directory `/home/yamada' ...
Copying files from `/etc/skel' ...
New password:
Retype new password:
passwd: password updated successfully
Changing the user information for yamada
Enter the new value, or press ENTER for the default
    Full Name []: Yamada Jiro
    Room Number []:
    Work Phone []:
    Home Phone []:
    Other []:
Is the information correct? [Y/n] Y
```

> **Debian/Ubuntu系では `adduser` を推奨**: `useradd` は低レベルコマンドで、ホームディレクトリの作成やパスワード設定を手動で行う必要があります。`adduser` は対話的に必要な設定をすべて行ってくれます。

### passwd - パスワードの設定・変更

```bash
# 自分のパスワードを変更
$ passwd
Changing password for tanaka.
Current password:
New password:
Retype new password:
passwd: password updated successfully

# 他のユーザーのパスワードを設定（root権限が必要）
$ sudo passwd yamada
New password:
Retype new password:
passwd: password updated successfully

# パスワードの状態を確認
$ sudo passwd -S yamada
yamada P 03/20/2024 0 99999 7 -1
# P=パスワード設定済, L=ロック, NP=パスワードなし

# アカウントをロック
$ sudo passwd -l yamada

# アカウントのロックを解除
$ sudo passwd -u yamada
```

### usermod - ユーザー情報の変更

```bash
# ユーザーを追加グループに追加（-a -G が重要）
$ sudo usermod -aG developers yamada
$ groups yamada
yamada : yamada developers

# 【注意】-a なしで -G を使うと、既存のグループから外れてしまう！
# 悪い例（絶対にやらないこと）:
# $ sudo usermod -G newgroup yamada  ← 他のグループから外れる！

# ログインシェルの変更
$ sudo usermod -s /bin/zsh yamada

# ホームディレクトリの変更（ファイルも移動する場合は -m を付ける）
$ sudo usermod -d /home/yamada_new -m yamada

# コメント（氏名）の変更
$ sudo usermod -c "Yamada Jiro (Lead)" yamada

# ユーザー名の変更
$ sudo usermod -l yamada_new yamada
```

### userdel - ユーザーの削除

```bash
# ユーザーを削除（ホームディレクトリは残る）
$ sudo userdel yamada

# ホームディレクトリごと削除
$ sudo userdel -r yamada
$ ls /home/
tanaka  suzuki    # yamada のディレクトリが消えている
```

dev-team:x:1002:suzuki,yamada

```bash
# グループを削除

```

### ユーザーをグループに追加・削除

```bash

# グループからユーザーを削除
$ sudo gpasswd -d tanaka developers
Removing user tanaka from group developers
$ groups tanaka
tanaka : tanaka sudo www-data
```

## 3.7 sudo - 管理者権限の実行

### sudoとは

`sudo`（superuser do）は、一般ユーザーが一時的にroot権限でコマンドを実行するための仕組みです。直接rootアカウントでログインするよりも安全です。

```
┌────────────────────────────────────────────────────────┐
│                  なぜ sudo を使うのか                    │
├────────────────────────────────────────────────────────┤
│                                                        │
│  root で直接ログイン          sudo を使う               │
│  ┌──────────────────┐      ┌──────────────────┐       │
│  │ ・常に全権限      │      │ ・必要な時だけ    │       │
│  │ ・操作ミスが致命的│      │ ・誰が何を実行    │       │
│  │ ・誰が操作したか  │      │   したかログに残る│       │
│  │   わからない      │      │ ・パスワード認証  │       │
│  │ ・リモートから     │      │ ・権限を細かく    │       │
│  │   攻撃されやすい  │      │   制御できる      │       │
│  └──────────────────┘      └──────────────────┘       │
│        ✗ 非推奨                  ◎ 推奨               │
└────────────────────────────────────────────────────────┘
```

### sudo の基本的な使い方

```bash
# root権限でコマンドを実行
$ sudo apt update
[sudo] password for tanaka:    # 自分のパスワードを入力
Hit:1 http://archive.ubuntu.com/ubuntu jammy InRelease
...

# sudoなしでは権限エラー
$ cat /etc/shadow
cat: /etc/shadow: Permission denied

# sudoを付けると読める
$ sudo cat /etc/shadow
root:$6$...:19400:0:99999:7:::
...

# rootユーザーとしてシェルを起動
$ sudo -i
root@server:~#    # rootのプロンプトになる
root@server:~# exit    # rootシェルから抜ける

# 特定のユーザーとしてコマンドを実行
$ sudo -u www-data whoami
www-data

# 直前のsudoコマンドを繰り返す
$ sudo !!
```

### sudo の設定（/etc/sudoers）

`/etc/sudoers` ファイルで sudo の権限を管理します。編集は必ず `visudo` コマンドを使います（構文チェックを行ってくれるため）。

```bash
# visudoで安全に編集（構文エラーを防止）
$ sudo visudo
```

```
# /etc/sudoers の主な設定例

# ユーザーにsudo権限を付与
tanaka  ALL=(ALL:ALL) ALL
# 書式: ユーザー ホスト=(実行ユーザー:実行グループ) コマンド

# グループにsudo権限を付与（%でグループ指定）
%sudo   ALL=(ALL:ALL) ALL
%admin  ALL=(ALL:ALL) ALL

# パスワードなしでsudoを許可（自動化スクリプト用）
deploy  ALL=(ALL) NOPASSWD: ALL

# 特定のコマンドのみ許可
operator ALL=(ALL) NOPASSWD: /usr/bin/systemctl restart nginx, /usr/bin/systemctl restart php-fpm
```

### sudo のログ確認

```bash
# sudo の実行ログは /var/log/auth.log（Ubuntu）または /var/log/secure（CentOS）に記録される
$ sudo grep sudo /var/log/auth.log
Mar 20 15:00:01 server sudo: tanaka : TTY=pts/0 ; PWD=/home/tanaka ; USER=root ; COMMAND=/usr/bin/apt update
Mar 20 15:05:00 server sudo: suzuki : TTY=pts/1 ; PWD=/home/suzuki ; USER=root ; COMMAND=/bin/cat /etc/shadow

# CentOS/RHEL の場合
$ sudo grep sudo /var/log/secure
```

## 3.8 特殊なパーミッション

通常の rwx 以外にも、特殊なパーミッションがあります。

### SUID（Set User ID）

ファイルの実行時に、実行者ではなくファイル所有者の権限で動作します。

```bash
# passwdコマンドはSUIDが設定されている
$ ls -l /usr/bin/passwd
-rwsr-xr-x 1 root root 68208 Mar  1  2024 /usr/bin/passwd
#   ^
#   s = SUID（所有者の実行権限の位置に s が表示される）

# 一般ユーザーでも passwd コマンドを実行すると、
# root 権限で /etc/shadow を更新できる

# SUIDの設定
$ sudo chmod u+s executable
$ sudo chmod 4755 executable    # 4000 がSUIDbビット
```

### SGID（Set Group ID）

ファイルの場合、実行時にファイルのグループ権限で動作します。ディレクトリの場合、その中に作成されたファイルのグループが自動的にディレクトリのグループになります。

```bash
# ディレクトリにSGIDを設定（共有ディレクトリでよく使う）
$ sudo chmod g+s /opt/shared/
$ ls -ld /opt/shared/
drwxrwsr-x 2 root developers 4096 Mar 20 15:00 /opt/shared/
#      ^
#      s = SGID

# このディレクトリ内に作成したファイルは自動的に developers グループになる
$ cd /opt/shared/
$ touch newfile.txt
$ ls -l newfile.txt
-rw-rw-r-- 1 tanaka developers 0 Mar 20 15:00 newfile.txt
#                    ^^^^^^^^^^
#                    自動的に developers グループになった

# 数値では 2000
$ sudo chmod 2775 /opt/shared/
```

### スティッキービット（Sticky Bit）

ディレクトリに設定すると、そのディレクトリ内のファイルは所有者とrootのみが削除できるようになります。`/tmp` ディレクトリに設定されています。

```bash
$ ls -ld /tmp
drwxrwxrwt 15 root root 4096 Mar 20 15:00 /tmp
#        ^
#        t = スティッキービット

# /tmp では誰でもファイルを作成できるが、
# 他人のファイルは削除できない

# スティッキービットの設定
$ sudo chmod +t /opt/shared_temp/
$ sudo chmod 1777 /opt/shared_temp/    # 1000 がスティッキービット
```

### 特殊パーミッションまとめ

| パーミッション | 数値 | 記号 | 対象 | 用途 |
|---------------|------|------|------|------|
| SUID | 4000 | `s`（u） | ファイル | 所有者権限で実行（passwd等） |
| SGID | 2000 | `s`（g） | ディレクトリ | 新規ファイルのグループを継承 |
| Sticky Bit | 1000 | `t` | ディレクトリ | 所有者のみ削除可能（/tmp等） |

## 3.9 umask - デフォルトパーミッションの制御

新しいファイルやディレクトリを作成したとき、デフォルトのパーミッションは `umask` によって決まります。

### umask の仕組み

```
ファイルの場合:   666 - umask = デフォルトパーミッション
ディレクトリの場合: 777 - umask = デフォルトパーミッション
```

```bash
# 現在のumaskを確認
$ umask
0022

# 数値の意味
# ファイル:     666 - 022 = 644 (rw-r--r--)
# ディレクトリ: 777 - 022 = 755 (rwxr-xr-x)

# 実際に確認
$ touch test_file
$ mkdir test_dir
$ ls -l
-rw-r--r-- 1 tanaka tanaka    0 Mar 20 15:00 test_file      # 644
drwxr-xr-x 2 tanaka tanaka 4096 Mar 20 15:00 test_dir/      # 755

# umaskを変更（セッション内のみ有効）
$ umask 0077
$ touch secret_file
$ mkdir secret_dir
$ ls -l
-rw------- 1 tanaka tanaka    0 Mar 20 15:01 secret_file    # 600
drwx------ 2 tanaka tanaka 4096 Mar 20 15:01 secret_dir/    # 700
```

### umaskの永続設定

```bash
# ~/.bashrc に追加すると、ログイン時に自動適用される
$ echo 'umask 0027' >> ~/.bashrc
# 0027 の場合:
#   ファイル:     666 - 027 = 640 (rw-r-----)
#   ディレクトリ: 777 - 027 = 750 (rwxr-x---)
```

## 3.10 よくある間違い

### 間違い1: usermod -G でグループを上書きしてしまう

```bash
# 【誤り】既存のグループ所属が全て消える！
$ sudo usermod -G newgroup tanaka

# 【正解】-a（append）を付ける
$ sudo usermod -aG newgroup tanaka
```

### 間違い2: chmod 777 を安易に使う

```bash
# 【誤り】全員に全権限を与えてしまう（セキュリティホール）
$ chmod 777 /var/www/html/config.php    # 絶対にダメ！

# 【正解】最小限の権限を付与
$ chmod 640 /var/www/html/config.php
$ sudo chown root:www-data /var/www/html/config.php
```

### 間違い3: SSH鍵のパーミッションを間違える

```bash
# SSHは鍵ファイルのパーミッションを厳密にチェックする
# パーミッションが緩いと接続を拒否される

# 【正しい設定】
$ chmod 700 ~/.ssh
$ chmod 600 ~/.ssh/id_rsa          # 秘密鍵
$ chmod 644 ~/.ssh/id_rsa.pub      # 公開鍵
$ chmod 600 ~/.ssh/authorized_keys  # 認証済み鍵リスト
$ chmod 644 ~/.ssh/config          # SSH設定ファイル
```

### 間違い4: chown -R を間違ったディレクトリに実行

```bash
# 【危険】ルートディレクトリに対して実行するとシステム崩壊
# $ sudo chown -R tanaka:tanaka /    ← 絶対にやらないこと！

# 【安全】対象を確認してから実行
$ ls -la /var/www/mysite/
$ sudo chown -R www-data:www-data /var/www/mysite/
```

## 3.11 現場での注意点

### Webサーバーのパーミッション設定

```bash
# Nginx/Apache のドキュメントルート
$ sudo chown -R www-data:www-data /var/www/html/
$ sudo find /var/www/html/ -type d -exec chmod 755 {} \;    # ディレクトリは755
$ sudo find /var/www/html/ -type f -exec chmod 644 {} \;    # ファイルは644

# アプリケーションが書き込みが必要なディレクトリ（ログ、キャッシュ等）
$ sudo chmod 775 /var/www/html/storage/
$ sudo chmod 775 /var/www/html/cache/
```

### 本番サーバーでの原則

1. **最小権限の原則**: 必要最小限の権限のみを付与する
2. **rootログイン禁止**: SSH経由のrootログインは無効にする
3. **sudo ログの監視**: 誰が何を実行したか定期的に確認する
4. **パスワードポリシー**: 強力なパスワードを強制する
5. **不要なユーザーの削除**: 退職者のアカウントは速やかに削除・ロックする

```bash
# rootのSSHログインを無効にする
$ sudo vim /etc/ssh/sshd_config
# PermitRootLogin no

# パスワード認証を無効にし、鍵認証のみにする
# PasswordAuthentication no

# SSHサービスを再起動して設定を反映
$ sudo systemctl restart sshd
```

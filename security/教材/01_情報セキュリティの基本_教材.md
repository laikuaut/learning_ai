# 第1章：情報セキュリティの基本

## 学習目標
- 情報セキュリティの基本概念（CIA三要素）を理解する
- 脅威・脆弱性・リスクの違いを説明できる
- 主要なセキュリティ脅威を分類できる
- セキュリティポリシーの重要性を理解する

---

## 1.1 情報セキュリティとは

情報セキュリティとは、情報の **機密性**、**完全性**、**可用性** を維持し、情報資産を様々な脅威から保護することである。

組織にとって情報は最も重要な資産の一つであり、その保護は事業継続の根幹をなす。

---

## 1.2 CIA三要素（情報セキュリティの3大要素）

### 1.2.1 機密性（Confidentiality）

**定義**: 許可された者だけが情報にアクセスできる状態を確保すること。

**具体例**:
- アクセス制御（認証・認可）
- データの暗号化
- 物理的なセキュリティ（施錠、入退室管理）
- 情報の分類（極秘/秘密/社外秘/公開）

**機密性が破られた例**:
- 顧客情報の漏洩
- 社内メールの盗聴
- 不正アクセスによるデータ窃取

### 1.2.2 完全性（Integrity）

**定義**: 情報が正確であり、不正に改ざんされていない状態を確保すること。

**具体例**:
- ハッシュ値による改ざん検知
- デジタル署名
- アクセスログの保護
- バージョン管理

**完全性が破られた例**:
- Webサイトの改ざん
- データベースの不正変更
- 通信経路上でのデータ改ざん（中間者攻撃）

### 1.2.3 可用性（Availability）

**定義**: 許可された者が必要な時に情報やシステムにアクセスできる状態を確保すること。

**具体例**:
- 冗長化（サーバー、ネットワーク、電源）
- バックアップと復旧手順
- DDoS対策
- BCP（事業継続計画）

**可用性が破られた例**:
- DDoS攻撃によるサービス停止
- ランサムウェアによるシステム利用不能
- 自然災害によるサーバー停止

### 1.2.4 追加の要素

近年は以下の要素も重視される（ISO 27001等）:

| 要素 | 説明 |
|------|------|
| **真正性（Authenticity）** | 情報や利用者が本物であることを保証 |
| **責任追跡性（Accountability）** | 行動を追跡・特定できること |
| **否認防止（Non-repudiation）** | 行為を後から否定できないこと |
| **信頼性（Reliability）** | システムが意図した通りに動作すること |

## 1.3 脅威・脆弱性・リスクの関係

### 1.3.1 用語の定義

```
リスク = 脅威 × 脆弱性 × 情報資産の価値
```

| 用語 | 定義 | 例 |
|------|------|-----|
| **脅威（Threat）** | 情報資産に損害を与える可能性のある事象 | マルウェア、自然災害、内部不正 |
| **脆弱性（Vulnerability）** | 脅威に対する弱点 | ソフトウェアのバグ、設定ミス、教育不足 |
| **リスク（Risk）** | 脅威が脆弱性を突いて損害が発生する可能性 | 個人情報漏洩のリスク |
| **情報資産（Asset）** | 保護すべき情報やシステム | 顧客DB、メールサーバー |

### 1.3.2 脅威の分類

```
脅威
├── 意図的脅威（人為的・故意）
│   ├── 外部攻撃（ハッキング、マルウェア）
│   └── 内部不正（情報持ち出し、権限悪用）
├── 偶発的脅威（人為的・過失）
│   ├── 操作ミス
│   ├── 設定ミス
│   └── 紛失・置き忘れ
└── 環境的脅威
    ├── 自然災害（地震、台風、洪水）
    ├── 設備障害（停電、空調故障）
    └── 経年劣化
```

### 1.3.3 脆弱性の分類

| 分類 | 例 |
|------|-----|
| 技術的脆弱性 | ソフトウェアバグ、暗号の脆弱性、プロトコルの欠陥 |
| 運用上の脆弱性 | パッチ未適用、弱いパスワードポリシー |
| 物理的脆弱性 | 入退室管理の不備、監視カメラの死角 |
| 人的脆弱性 | セキュリティ教育不足、ソーシャルエンジニアリング耐性 |

## 1.4 情報セキュリティ10大脅威（IPA）

IPA（情報処理推進機構）が毎年発表する「情報セキュリティ10大脅威」は、日本における最新の脅威動向を把握する上で重要である。

### 組織向け脅威（主要なもの）

| 順位 | 脅威 | 説明 |
|------|------|------|
| 1 | **ランサムウェアによる被害** | データを暗号化し身代金を要求。二重脅迫型が増加 |
| 2 | **サプライチェーンの弱点を悪用した攻撃** | 取引先・委託先を踏み台にした攻撃 |
| 3 | **標的型攻撃による機密情報の窃取** | 特定組織を狙った巧妙な攻撃（APT） |
| 4 | **内部不正による情報漏洩** | 従業員・元従業員による意図的な情報持ち出し |
| 5 | **テレワーク等のニューノーマルを狙った攻撃** | VPN脆弱性、自宅環境の脆弱性を悪用 |
| 6 | **修正プログラムの公開前を狙う攻撃（ゼロデイ攻撃）** | パッチ提供前の脆弱性を悪用 |
| 7 | **ビジネスメール詐欺（BEC）** | 取引先や上司を装い送金指示 |
| 8 | **脆弱性対策の公開に伴う悪用増加** | 脆弱性情報公開後の攻撃 |
| 9 | **不注意による情報漏洩等の被害** | メール誤送信、設定ミス |
| 10 | **犯罪のビジネス化（アンダーグラウンドサービス）** | RaaS等の攻撃サービス化 |

### 個人向け脅威（主要なもの）

| 脅威 | 説明 |
|------|------|
| フィッシングによる個人情報詐取 | 偽サイトでID/パスワード等を騙し取る |
| ネット上の誹謗・中傷・デマ | SNS等での名誉毀損 |
| メール・SMS等を使った脅迫・詐欺 | 架空請求、偽警告 |
| クレジットカード情報の不正利用 | EC サイト改ざん等による情報窃取 |
| スマホ決済の不正利用 | アカウント乗っ取り |

## 1.5 攻撃者の分類

### 攻撃者のタイプ

| 分類 | 動機 | スキル | 例 |
|------|------|--------|-----|
| **スクリプトキディ** | 愉快犯、自己顕示 | 低（既存ツール利用） | 公開ツールでの攻撃 |
| **ハクティビスト** | 政治的・社会的主張 | 中〜高 | Anonymous等 |
| **組織犯罪グループ** | 金銭目的 | 高 | ランサムウェアグループ |
| **国家支援型攻撃者（APT）** | 諜報・破壊活動 | 非常に高 | 国家支援のハッカー集団 |
| **内部脅威者** | 金銭、怨恨 | 変動 | 不満を持つ従業員 |
| **産業スパイ** | 競争優位の獲得 | 高 | 企業秘密の窃取 |

### 攻撃の動機（MICE）

- **M（Money）**: 金銭的利益
- **I（Ideology）**: 思想・信条
- **C（Coercion/Compromise）**: 脅迫・弱み
- **E（Ego）**: 自己顕示欲

## 1.6 セキュリティポリシー

### 1.6.1 情報セキュリティポリシーの階層構造

```
┌─────────────────────────────────┐
│    基本方針（ポリシー）           │  ← 経営層が策定
│    なぜ保護するか                │
├─────────────────────────────────┤
│    対策基準（スタンダード）        │  ← 管理者が策定
│    何を守るか                    │
├─────────────────────────────────┤
│    実施手順（プロシージャ）        │  ← 担当者が策定
│    どう守るか                    │
└─────────────────────────────────┘
```

### 1.6.2 基本方針に含めるべき内容

1. **目的**: セキュリティポリシーの目的と適用範囲
2. **定義**: 用語の定義
3. **基本原則**: 情報セキュリティに対する組織の基本的な考え方
4. **体制**: 管理体制と責任の明確化
5. **遵守義務**: 従業員の義務と違反時の措置
6. **見直し**: 定期的な見直しの方針

### 1.6.3 対策基準の例

| 分野 | 対策基準例 |
|------|-----------|
| アクセス管理 | パスワードは12文字以上、多要素認証必須 |
| ネットワーク | 外部接続はVPN経由、許可されたポートのみ開放 |
| 物理セキュリティ | ICカードによる入退室管理、重要区域は2名以上 |
| インシデント対応 | 発見から24時間以内に報告、初動対応手順の遵守 |
| 教育 | 年1回以上のセキュリティ教育受講必須 |

## 1.7 防御の基本原則

### 多層防御（Defense in Depth）

```
┌─────────────────────────────────────┐
│ ポリシー・教育（人的対策）            │
│ ┌─────────────────────────────────┐ │
│ │ 物理セキュリティ（入退室管理）     │ │
│ │ ┌─────────────────────────────┐ │ │
│ │ │ ネットワーク（FW/IDS/IPS）    │ │ │
│ │ │ ┌─────────────────────────┐ │ │ │
│ │ │ │ ホスト（OS/アンチマルウェア）│ │ │ │
│ │ │ │ ┌─────────────────────┐ │ │ │ │
│ │ │ │ │ アプリケーション      │ │ │ │ │
│ │ │ │ │ ┌─────────────────┐ │ │ │ │ │
│ │ │ │ │ │ データ（暗号化）  │ │ │ │ │ │
│ │ │ │ │ └─────────────────┘ │ │ │ │ │
│ │ │ │ └─────────────────────┘ │ │ │ │
│ │ │ └─────────────────────────┘ │ │ │
│ │ └─────────────────────────────┘ │ │
│ └─────────────────────────────────┘ │
└─────────────────────────────────────┘
```

### 最小権限の原則（Principle of Least Privilege）

- 業務に必要最小限の権限のみを付与する
- 不要な権限は速やかに削除する
- 定期的な権限の見直しを行う

### その他の基本原則

| 原則 | 説明 |
|------|------|
| **Need to Know** | 知る必要のある者だけに情報を開示 |
| **職務分離** | 重要な操作を一人で完結させない |
| **フェイルセーフ** | 障害時に安全側に倒れる設計 |
| **フェイルセキュア** | 障害時にセキュリティが確保される設計 |
| **ゼロトラスト** | 「信頼しない、常に検証する」 |

## 1.8 セキュリティ対策の分類

### 機能別分類

| 分類 | 説明 | 例 |
|------|------|-----|
| **抑止的対策** | 攻撃の動機を減らす | 監視カメラ、警告表示 |
| **予防的対策** | 攻撃を事前に防ぐ | ファイアウォール、パッチ適用 |
| **検知的対策** | 攻撃を検知する | IDS、ログ監視 |
| **是正的対策** | 攻撃後に復旧する | バックアップ、インシデント対応 |

### 手段別分類

| 分類 | 例 |
|------|-----|
| **技術的対策** | 暗号化、アクセス制御、ファイアウォール |
| **物理的対策** | 入退室管理、施錠、監視カメラ |
| **人的対策** | セキュリティ教育、規程策定、運用手順 |
| **組織的対策** | 体制整備、監査、ISMS構築 |

---

## 1.9 まとめ

### 重要ポイント

1. **CIA三要素**は情報セキュリティの根幹であり、すべての対策はこの3要素の維持を目的とする
2. **リスク = 脅威 × 脆弱性 × 資産価値** の関係を理解する
3. 脅威は意図的・偶発的・環境的に分類される
4. セキュリティポリシーは **基本方針 → 対策基準 → 実施手順** の階層構造を持つ
5. **多層防御**により、単一の対策が破られても被害を最小化する

### 試験対策キーワード

- CIA三要素、真正性、責任追跡性、否認防止
- 脅威、脆弱性、リスク、情報資産
- 多層防御、最小権限の原則、Need to Know
- 情報セキュリティポリシー（基本方針/対策基準/実施手順）
- IPA 情報セキュリティ10大脅威

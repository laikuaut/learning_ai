## 課題の要件

1. 以下のビジネスシナリオに対して、生成AI APIを活用した自動化ワークフローを設計する
   - **シナリオ**: 顧客からの問い合わせメールを自動分類し、カテゴリ別に要約してSlackチャンネルに通知、回答案を自動生成する
2. ワークフローの全体設計を行う
   - トリガー条件（何をきっかけに処理を開始するか）
   - 処理ステップ（前処理 → API呼び出し → 後処理）
   - 出力先（通知先、保存先）
3. API呼び出し部分の詳細設計
   - 使用するAPIとエンドポイント（endpoint）
   - リクエスト/レスポンスの形式
   - リトライ戦略（Retry Strategy）とエラーハンドリング
4. コスト管理設計
   - 月間コストの試算
   - コスト削減策（キャッシュ、モデル選択、プロンプト最適化）
5. 監視・運用設計
   - 異常検知の仕組み
   - ログ設計
   - アラート基準

## ステップガイド

<details>
<summary>ステップ1：ワークフローの全体像を設計する</summary>

まず、エンドツーエンド（End-to-End）の処理フローを描きましょう。

```
トリガー → 前処理 → AI処理 → 後処理 → 出力
```

各ステップで以下を明確にします。
- **入力**: 何を受け取るか
- **処理**: 何をするか
- **出力**: 何を次のステップに渡すか
- **異常系**: エラー時にどうするか

**よくある間違い**: 正常系だけ設計して異常系を考えないこと。API呼び出しは「失敗することがある」前提で設計する必要があります。

</details>

<details>
<summary>ステップ2：API呼び出しの詳細を設計する</summary>

以下の観点でAPI呼び出しを設計しましょう。

- **モデル選択**: タスクの複雑さに応じて使い分け（簡単な分類は軽量モデル、回答生成は高性能モデル）
- **リクエスト設計**: プロンプト、パラメータ（temperature、max_tokens等）
- **レスポンス処理**: 出力の構造化（JSON化）、バリデーション
- **リトライ戦略**: 指数バックオフ（Exponential Backoff）、最大リトライ回数
- **レートリミット対応**: API制限に合わせたリクエスト制御

</details>

<details>
<summary>ステップ3：コスト試算と最適化を行う</summary>

以下のパラメータでコストを試算しましょう。

- 1日あたりの問い合わせメール件数
- 1件あたりの平均トークン数（入力+出力）
- 使用するモデルの料金単価
- 月間稼働日数

コスト削減策として以下を検討します。
- 類似質問のキャッシュ
- 分類タスクへの軽量モデル使用
- プロンプトの短縮

</details>

<details>
<summary>ステップ4：監視・運用設計を行う</summary>

以下の観点で監視設計を行いましょう。

- **メトリクス**: レスポンスタイム、成功率、コスト推移
- **ログ**: 何をどこに記録するか
- **アラート**: どんな条件でアラートを出すか
- **ダッシュボード**: 何を可視化するか

</details>

### 2. API呼び出し詳細設計

#### ステップ1：メール分類（軽量モデル）

**プロンプト:**
```
以下の顧客メールを分類してください。

【分類カテゴリ】
- technical（技術的な質問・障害報告）
- billing（請求・料金に関する問い合わせ）
- general（一般的な質問）
- complaint（クレーム・苦情）

【緊急度】
- high（サービス停止、データ損失等）
- medium（業務に支障があるが回避策あり）
- low（一般的な質問、機能リクエスト）

【出力形式】JSONのみで回答してください
{"category": "...", "urgency": "...", "reason": "..."}

【メール】
件名：{subject}
本文：{body}
```

**APIリクエスト設計:**
```
エンドポイント: POST /v1/messages
モデル: claude-3-haiku（軽量・高速・低コスト）
パラメータ:
  - max_tokens: 100
  - temperature: 0（分類タスクなので確定的に）
```

#### ステップ2：要約＋回答案生成（高性能モデル）

**プロンプト:**
```
あなたはカスタマーサポートの専門家です。
以下の顧客メールについて、要約と回答案を作成してください。

【メール情報】
カテゴリ: {category}
緊急度: {urgency}
件名: {subject}
本文: {body}

【出力ルール】
1. 要約：メールの要点を3行以内にまとめる
2. 回答案：丁寧で具体的な回答を200文字以内で作成
3. 回答案に「確認が必要な点」がある場合は [要確認] タグを付ける

【出力形式】
{
  "summary": "...",
  "draft_reply": "...",
  "needs_review": ["要確認事項1", "要確認事項2"]
}
```

**APIリクエスト設計:**
```
エンドポイント: POST /v1/messages
モデル: claude-3-sonnet（高性能・バランス型）
パラメータ:
  - max_tokens: 500
  - temperature: 0.3（多少の創造性を許容）
```

### 3. リトライ戦略

```
リクエスト送信
     │
     ├─ 成功（200）→ 処理続行
     │
     ├─ レートリミット（429）→ Retry-Afterヘッダの秒数待って再試行
     │
     ├─ サーバーエラー（500/503）→ 指数バックオフで再試行
     │    ├─ 1回目: 1秒後に再試行
     │    ├─ 2回目: 2秒後に再試行
     │    ├─ 3回目: 4秒後に再試行
     │    └─ 4回目以降: エラーとして記録、手動対応キューへ
     │
     └─ クライアントエラー（400/401）→ リトライせずエラーログ記録
```

### 4. コスト試算

**前提条件:**
- 1日あたり問い合わせメール：50件
- 月間稼働日数：22日
- 月間処理件数：1,100件

**分類タスク（Haiku）:**
```
入力：平均500トークン/件
出力：平均50トークン/件
単価：入力 $0.25/100万トークン、出力 $1.25/100万トークン

月間コスト：
  入力: 500 × 1,100 = 550,000トークン → $0.14
  出力: 50 × 1,100 = 55,000トークン → $0.07
  小計: 約 $0.21/月
```

**要約＋回答案（Sonnet）:**
```
入力：平均800トークン/件
出力：平均300トークン/件
単価：入力 $3/100万トークン、出力 $15/100万トークン

月間コスト：
  入力: 800 × 1,100 = 880,000トークン → $2.64
  出力: 300 × 1,100 = 330,000トークン → $4.95
  小計: 約 $7.59/月
```

**合計: 約 $7.80/月（約1,200円）**

| 監視項目 | 正常値 | アラート条件 | 通知先 |
|---------|--------|-------------|--------|
| API成功率 | 99%以上 | 95%未満が5分間継続 | 運用チームSlack |
| レスポンスタイム | 3秒以内 | 10秒超が3回連続 | 運用チームSlack |
| 日次コスト | $0.35以下 | $1.00超 | 管理者メール |
| 未処理キュー | 0件 | 10件以上滞留 | 運用チームSlack |

### 改良版の特徴

- **キャッシュ戦略**でコストを50%以上削減
- **フォールバック（Fallback）設計**で障害時も稼働継続
- **A/Bテスト基盤**で継続的なプロンプト改善
- **セキュリティ設計**を追加

---

#### キャッシュ戦略

```
問い合わせメール
     │
     ▼
類似質問キャッシュを検索
（エンベディングの類似度 > 0.95）
     │
     ├─ キャッシュヒット → キャッシュされた回答案を使用
     │   （API呼び出しスキップ → コスト$0）
     │
     └─ キャッシュミス → AI API呼び出し
         │
         ▼
     回答をキャッシュに保存
     （TTL: 30日、最大1000エントリ）
```

**コスト削減効果の試算：**
- FAQ的な定型質問が全体の40%と仮定
- キャッシュヒット率40%で、月間APIコスト $7.80 → $4.68（40%削減）

---

#### フォールバック設計

```
プライマリAPI（Claude）
     │
     ├─ 成功 → 通常処理
     │
     └─ 失敗（3回リトライ後）
          │
          ├─ セカンダリAPI（GPT-4o-mini）で代替処理
          │    │
          │    ├─ 成功 → 処理続行（ログに「フォールバック使用」を記録）
          │    │
          │    └─ 失敗 → 最終フォールバック
          │
          └─ 最終フォールバック
               │
               ├─ 分類タスク → ルールベース分類（キーワードマッチング）
               └─ 回答生成 → テンプレート回答 + 手動対応キュー
```

**ポイント**: 生成AI APIに100%依存しない設計にすること。APIが全て停止しても、最低限の分類とテンプレート回答で業務を継続できます。

---

#### A/Bテスト基盤

```
問い合わせメール
     │
     ▼
ランダムに振り分け（50:50）
     │
     ├─ グループA: 現行プロンプト
     │    └─ 回答案 + メトリクス記録
     │
     └─ グループB: 新プロンプト（候補）
          └─ 回答案 + メトリクス記録

     │
     ▼
1週間後に比較分析
     ├─ 回答採用率（担当者が修正なしで使った割合）
     ├─ 顧客満足度スコア
     └─ 処理時間

     │
     ▼
優れた方を本番採用
```

---

#### セキュリティ設計

```
【APIキー管理】
- 環境変数またはシークレットマネージャー（AWS Secrets Manager等）で管理
- コードにハードコードしない
- 定期ローテーション（90日ごと）

【データの取り扱い】
- 顧客の個人情報（PII）はマスキング後にAPIに送信
  例：「田中太郎」→「[顧客名]」、「03-1234-5678」→「[電話番号]」
- APIログには個人情報を含めない
- API提供元のデータ利用ポリシーを確認（学習に使用されないことを確認）

【アクセス制御】
- API呼び出し元のIPアドレスを制限
- APIキーごとに呼び出し上限を設定
- 管理画面へのアクセスは多要素認証（MFA）必須
```

**マスキングの実装例（プロンプト）：**
```
以下のメール本文から個人情報をマスキングしてからAPIに送信します。

マスキングルール：
- 氏名 → [顧客名]
- 電話番号 → [電話番号]
- メールアドレス → [メール]
- 住所 → [住所]
- クレジットカード番号 → [カード番号]
```

**初心者向けとの違い**:
- キャッシュ戦略でコストを大幅削減する実践的な手法を導入
- マルチプロバイダのフォールバック設計で可用性を向上
- A/Bテスト基盤でプロンプトを継続的に改善する仕組みを構築
- セキュリティ設計（PIIマスキング、キー管理、アクセス制御）を追加

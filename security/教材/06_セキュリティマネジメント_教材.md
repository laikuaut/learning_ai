# 第6章：セキュリティマネジメント

## この章のゴール

- ISMS（情報セキュリティマネジメントシステム）の概念とPDCAサイクルを理解すること
- リスクアセスメント（リスク特定・分析・評価）の手法を説明できること
- リスク対応の4つの選択肢を把握すること
- インシデント対応の流れと体制（CSIRT）を理解すること
- 情報セキュリティ監査の目的と種類を説明できること

---

## 6.1 ISMS（Information Security Management System）

### 6.1.1 ISMSとは

ISMS（情報セキュリティマネジメントシステム）は、組織における情報セキュリティを体系的に管理するための枠組みです。国際規格 **ISO/IEC 27001** に基づき、技術的対策だけでなく、人的・組織的・物理的対策を含めた包括的なセキュリティ管理を実現します。

```
【ISMSの基本概念】

  ISMSは「仕組み」であり「製品」ではない

  技術的対策だけではセキュリティは確保できない
  ┌──────────────────────────────────┐
  │ ISMS = 技術 + 人 + 組織 + 物理  │
  │                                  │
  │  技術的対策: FW, 暗号化, IDS等   │
  │  人的対策:   教育, 訓練, 規程    │
  │  組織的対策: 体制, 監査, CSIRT   │
  │  物理的対策: 入退室管理, 施錠    │
  └──────────────────────────────────┘
```

### 6.1.2 ISMSのPDCAサイクル

ISMSは **PDCAサイクル** により継続的に改善されます。

```
【ISMSのPDCAサイクル】

         ┌──────────┐
         │  Plan     │ ← ISMSの確立
         │ (計画)    │   リスクアセスメント、対策の計画
         └────┬─────┘
              │
         ┌────▼─────┐
         │  Do       │ ← ISMSの導入・運用
         │ (実行)    │   対策の実施、教育・訓練
         └────┬─────┘
              │
         ┌────▼─────┐
         │  Check    │ ← ISMSの監視・レビュー
         │ (点検)    │   内部監査、有効性の評価
         └────┬─────┘
              │
         ┌────▼─────┐
         │  Act      │ ← ISMSの維持・改善
         │ (改善)    │   是正処置、予防処置
         └────┬─────┘
              │
              └───→ Plan に戻る（継続的改善）
```

### 6.1.3 ISO/IEC 27001 と 27002

| 規格 | 内容 |
|------|------|
| **ISO/IEC 27001** | ISMS の要求事項（認証規格）。「何をすべきか」を規定 |
| **ISO/IEC 27002** | 情報セキュリティ管理策の実践規範。「どのようにすべきか」のガイドライン |

## 6.2 リスクマネジメント

### 6.2.1 リスクマネジメントの全体像

```
【リスクマネジメントのプロセス】

  ┌──────────────────────────────────────┐
  │         リスクマネジメント              │
  │                                       │
  │  ① リスクアセスメント                  │
  │     ├── リスク特定（何がリスクか）      │
  │     ├── リスク分析（どの程度深刻か）    │
  │     └── リスク評価（対応の優先順位）    │
  │                                       │
  │  ② リスク対応                          │
  │     ├── リスク低減（軽減）             │
  │     ├── リスク回避                     │
  │     ├── リスク移転（共有）             │
  │     └── リスク受容（保有）             │
  │                                       │
  │  ③ リスクコミュニケーション             │
  │     関係者との情報共有                  │
  │                                       │
  │  ④ リスクモニタリング                   │
  │     継続的な監視と見直し                │
  └──────────────────────────────────────┘
```

### 6.2.2 リスクアセスメント

**リスク特定**

保護すべき情報資産を洗い出し、それぞれに対する脅威と脆弱性を特定します。

```
  情報資産       脅威                脆弱性
  顧客DB    ← 不正アクセス     ← パッチ未適用
  メールサーバ ← マルウェア感染  ← ウイルス対策の不備
  社員PC    ← 盗難・紛失       ← 暗号化未実施
  紙文書    ← 持ち出し         ← 施錠管理の不備
```

**リスク分析**

リスクの大きさを定量的または定性的に評価します。

```
  リスク値 = 情報資産の価値 × 脅威の発生可能性 × 脆弱性の程度

  定量的分析:
    年間予想損失額（ALE）= 1回あたりの損失額（SLE）× 年間発生頻度（ARO）
    例: 情報漏洩1回の損失 = 1億円、年間発生確率 = 5%
        ALE = 1億円 × 0.05 = 500万円

  定性的分析:
    リスクを「高・中・低」等のランクで評価
    ┌───────────────────┐
    │    脅威の発生可能性  │
    │    高   中   低     │
    │ 高  A    A    B     │ ← 情報資産の
    │ 中  A    B    C     │    価値
    │ 低  B    C    C     │
    └───────────────────┘
    A: 即時対策必要  B: 計画的に対策  C: 受容可能
```

**リスク評価**

分析結果に基づき、対応の優先順位を決定します。リスク受容基準（組織が許容できるリスクのレベル）と比較して判断します。

### 6.2.3 リスク対応の4つの選択肢

| 対応策 | 説明 | 例 |
|--------|------|-----|
| **リスク低減（軽減）** | 対策を実施してリスクを減らす | ファイアウォール導入、暗号化実施 |
| **リスク回避** | リスクの原因となる活動を停止 | クラウドサービスの利用を中止 |
| **リスク移転（共有）** | リスクを第三者に移す | サイバー保険への加入、アウトソーシング |
| **リスク受容（保有）** | リスクがある状態を受け入れる | 対策コスト > 被害額 の場合 |

> **現場の知識**: 全てのリスクをゼロにすることは不可能であり、コストとのバランスが重要です。リスク受容の場合でも、経営層の承認を得て文書化しておくことが必要です。

## 6.3 インシデント対応

### 6.3.1 セキュリティインシデントとは

情報セキュリティに関する事故や事象のことです。マルウェア感染、情報漏洩、不正アクセス、サービス停止等が含まれます。

### 6.3.2 インシデント対応の流れ

```
【インシデント対応のフェーズ】

  ① 準備（Preparation）
     ├── インシデント対応計画の策定
     ├── CSIRTの設置
     ├── ツール・環境の整備
     └── 訓練の実施

  ② 検知と分析（Detection & Analysis）
     ├── 異常の検知（IDS、ログ監視等）
     ├── インシデントの判定
     ├── 影響範囲の特定
     └── 優先度の決定（トリアージ）

  ③ 封じ込め（Containment）
     ├── 短期的封じ込め（被害拡大の防止）
     │   例: 感染端末のネットワーク隔離
     └── 長期的封じ込め
         例: 脆弱なシステムの一時停止

  ④ 根絶（Eradication）
     ├── マルウェアの駆除
     ├── 侵入経路の特定と遮断
     └── 脆弱性の修正（パッチ適用等）

  ⑤ 復旧（Recovery）
     ├── システムの復元
     ├── サービスの再開
     └── 監視の強化

  ⑥ 事後対応（Lessons Learned）
     ├── インシデント報告書の作成
     ├── 原因分析と再発防止策
     └── 対応手順の改善
```

### 6.3.3 CSIRT（Computer Security Incident Response Team）

**CSIRT**（シーサート）は、情報セキュリティインシデントに対応する専門チームです。

| 役割 | 説明 |
|------|------|
| インシデント対応 | 検知・分析・封じ込め・復旧を実施 |
| 脆弱性情報の収集 | 新たな脅威・脆弱性情報を収集・分析 |
| 社内への注意喚起 | セキュリティ情報の周知 |
| 外部組織との連携 | JPCERT/CC、IPA、他組織のCSIRTとの情報共有 |
| 教育・訓練 | セキュリティ意識向上の活動 |

## 6.4 情報セキュリティ監査

### 6.4.1 監査の種類

| 種類 | 実施者 | 目的 |
|------|--------|------|
| **内部監査** | 組織内の監査部門 | ISMSの運用状況を自己確認 |
| **外部監査（第三者監査）** | 独立した認証機関 | ISMS認証（ISO 27001）の取得・更新 |
| **情報セキュリティ監査** | 情報セキュリティ監査人 | 経済産業省の「情報セキュリティ監査基準」に基づく |

### 6.4.2 監査の2つの基準

| 基準 | 説明 |
|------|------|
| **情報セキュリティ管理基準** | 監査で確認すべき管理策の一覧（何を確認するか） |
| **情報セキュリティ監査基準** | 監査の実施方法と報告の基準（どう監査するか） |

### 6.4.3 情報セキュリティ教育

| 対象 | 内容 | 頻度 |
|------|------|------|
| 全従業員 | セキュリティポリシーの周知、フィッシング対策 | 年1回以上 |
| IT部門 | 技術的な脅威動向、インシデント対応手順 | 四半期ごと |
| 経営層 | リスクマネジメント、法的責任 | 年1回以上 |
| 新入社員 | 基本的なセキュリティルール | 入社時 |

## 6.5 BCP / BCM（事業継続）

### 6.5.1 BCPとBCM

| 用語 | 正式名称 | 説明 |
|------|---------|------|
| **BCP** | Business Continuity Plan | 事業継続計画。災害やインシデント発生時に事業を継続するための計画 |
| **BCM** | Business Continuity Management | 事業継続マネジメント。BCPの策定・運用・改善を管理する活動全体 |
| **BIA** | Business Impact Analysis | ビジネス影響分析。事業中断の影響度を分析 |

### 6.5.2 BIA（ビジネス影響分析）の指標

| 指標 | 説明 |
|------|------|
| **RTO（Recovery Time Objective）** | 目標復旧時間。許容できる最大停止時間 |
| **RPO（Recovery Point Objective）** | 目標復旧時点。どの時点のデータまで復旧させるか |
| **MTPD（Maximum Tolerable Period of Disruption）** | 最大許容停止時間 |

```
【RTOとRPOの関係】

  ←─── RPO ───→│ インシデント発生 │←── RTO ──→│
                  │                 │             │
  バックアップ時点  │  データ喪失      │ 復旧完了    │
                  │  許容範囲        │ 目標時間    │
  ────────────────┼─────────────────┼─────────────→ 時間
```

## 6.6 まとめ

### ポイントまとめ

1. **ISMS** は技術・人・組織・物理の包括的なセキュリティ管理体制で、PDCAサイクルで継続的に改善します
2. **リスクアセスメント** は リスク特定 → リスク分析 → リスク評価 の3段階です
3. **リスク対応** には 低減・回避・移転・受容 の4つの選択肢があります
4. **インシデント対応** は 準備 → 検知 → 封じ込め → 根絶 → 復旧 → 事後対応 の流れで行います
5. **CSIRT** はインシデント対応の専門チームです
6. **BCP/BCM** で事業継続性を確保し、RTO/RPOを定義して復旧計画を策定します

### 試験対策キーワード

- ISMS、ISO 27001、ISO 27002、PDCA
- リスクアセスメント（リスク特定、リスク分析、リスク評価）
- リスク対応（低減、回避、移転、受容）
- インシデント対応（準備、検知、封じ込め、根絶、復旧、事後対応）
- CSIRT、JPCERT/CC、IPA
- BCP、BCM、BIA、RTO、RPO

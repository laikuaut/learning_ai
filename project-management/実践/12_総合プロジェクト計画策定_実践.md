## 課題の要件

以下のシナリオを読み、プロジェクト計画書一式を作成してください。これは本コースの総まとめとなる課題です。

### シナリオ

あなたは受託開発企業のPM（経験3年目）です。以下の案件を新規受注し、プロジェクト計画を策定する段階です。

**クライアント概要：**
- 企業：中堅の人材派遣会社（従業員300名、派遣スタッフ2,000名）
- IT担当：情報システム部 3名（部長1名、担当2名）

**案件概要：**
- 内容：派遣スタッフ向けのポータルサイト新規構築
- 目的：派遣スタッフの満足度向上と、社内業務の効率化

**ポータルサイトに求められる機能：**

| # | 機能 | 詳細 |
|---|------|------|
| F1 | 勤怠入力 | 派遣スタッフがスマホ/PCから勤怠を入力・修正 |
| F2 | 給与明細閲覧 | 月次の給与明細をPDFで確認 |
| F3 | 契約情報確認 | 自分の契約期間、就業条件、派遣先情報を確認 |
| F4 | お知らせ配信 | 全体・個別のお知らせを配信・閲覧 |
| F5 | 有給申請 | 有給休暇の申請・承認ワークフロー |
| F6 | 各種届出 | 住所変更、口座変更などの届出フォーム |
| F7 | FAQ・問い合わせ | よくある質問の閲覧と、個別問い合わせフォーム |
| F8 | 管理者ダッシュボード | 管理者向けの利用状況・勤怠集計画面 |

**制約条件：**
- 予算：2,500万円（税別）
- 期限：2026年10月末の本番稼働（6ヶ月後）
- 技術：React + Node.js + PostgreSQL（クライアント指定）
- 連携：既存の人事給与システム（パッケージ製品）とCSV連携
- セキュリティ：個人情報（住所、口座情報）を扱うため、高いセキュリティが必要
- モバイル対応：スマートフォンからの利用が主（レスポンシブデザイン必須）

**あなたのチーム：**
- PM: あなた（1名）
- フロントエンドリーダー: 1名（React経験5年）
- バックエンドリーダー: 1名（Node.js経験3年）
- フロントエンド開発者: 1名（React経験1年）
- バックエンド開発者: 1名（Node.js経験2年）
- テスター: 1名（兼務、月の50%をこのプロジェクトに充当）
- デザイナー: 1名（兼務、UI/UXデザインのみ参加）

## 作成すべきドキュメント

以下の9つのドキュメントを作成してください。各ドキュメントは独立して回答しても、一つの文書にまとめても構いません。

### ドキュメント1：プロジェクト憲章

- プロジェクトの目的、ビジネス上の理由、成功基準
- ハイレベルスコープ（IN/OUT）
- 前提条件と制約条件
- 概算スケジュールと予算

### ドキュメント2：ステークホルダー分析

- ステークホルダー一覧（影響力、関心度、姿勢）
- 影響力×関心度マトリクス
- エンゲージメント方針

### ドキュメント3：スコープ定義書（WBS付き）

- 機能要件の詳細化（各機能の受入基準）
- 非機能要件の定義
- WBS（第3階層まで）
- MoSCoW優先度付け

### ドキュメント4：スケジュール

- ガントチャート（テキスト形式）
- クリティカルパスの特定
- マイルストーン一覧

### ドキュメント5：コスト計画

- 工数見積もり（三点見積もりまたはボトムアップ）
- 予算配分（フェーズ別）
- マネジメントリザーブの設定

### ドキュメント6：リスク管理計画

- リスク登録簿（最低8件）
- 確率×影響度マトリクス
- 上位リスクの対応策

### ドキュメント7：品質管理計画

- 品質目標
- テスト戦略
- リリース判定基準

### ドキュメント8：コミュニケーション計画

- 定期コミュニケーション一覧
- 報告テンプレート
- エスカレーション基準

### ドキュメント9：変更管理計画

- 変更要求の受付〜承認プロセス
- 規模別の承認権限
- スコープフリーズのタイミング

<summary>ステップ2：スコープの優先順位付け（20分）</summary>

8機能のMoSCoW分類を行います。

```
判断基準：
  Must:   初日から必要。これがないと本番稼働できない
  Should: 稼働後1ヶ月以内に必要。遅れても致命的ではない
  Could:  あれば嬉しいが、なくても運用は可能
  Won't:  今回は対象外

ヒント：
  - F1（勤怠入力）は派遣スタッフの日常業務に直結 → Must
  - F2（給与明細）は月次で必要 → Must
  - F8（ダッシュボード）は管理者向け → Shouldの可能性
  - 全機能がMustになりそうな場合、段階リリースを検討
```

### ドキュメント1：プロジェクト憲章

```
============================================================
  プロジェクト憲章
  プロジェクト名: 派遣スタッフポータルサイト構築
  文書番号: PC-2026-PSP-001  Ver.1.0
  作成日: 2026-04-06
============================================================

1. プロジェクトの目的
   派遣スタッフ（2,000名）向けのWebポータルを新規構築し、
   勤怠管理・給与明細確認・各種届出のオンライン化による
   スタッフの利便性向上と、社内管理業務の効率化を実現する。

2. ビジネス上の理由
   - 派遣スタッフの定着率低下（前年比-5%）への対策
   - 紙ベースの届出処理に月間40時間の事務工数
   - 給与明細の郵送コスト（年間約150万円）の削減
   - 競合他社はすでにスタッフ向けポータルを導入済み

3. 成功基準
   | 指標                     | 目標値          |
   |--------------------------|----------------|
   | 派遣スタッフの利用率      | 80%以上（3ヶ月後）|
   | 勤怠入力のオンライン化率  | 90%以上         |
   | 届出処理の事務工数        | 月40時間→月10時間 |
   | スタッフ満足度            | 4.0/5.0以上     |

4. ハイレベルスコープ
   【IN】
   - 勤怠入力、給与明細閲覧、契約情報確認
   - お知らせ配信、有給申請、各種届出
   - FAQ・問い合わせ、管理者ダッシュボード
   - 人事給与システムとのCSV連携
   - スマートフォン対応（レスポンシブデザイン）

   【OUT】
   - 人事給与システム本体の改修
   - 派遣先企業への機能提供
   - チャット機能
   - ネイティブアプリ（iOS/Android）

5. 制約条件
   - 予算: 2,500万円（税別）
   - 期限: 2026年10月末本番稼働
   - 技術: React + Node.js + PostgreSQL

6. 前提条件
   - 人事給与システムのCSV出力仕様が5月中に提供される
   - クライアント側でSSL証明書・ドメインを準備する
   - テスト環境用のサンプルデータが提供される

7. 概算スケジュール
   5月: 要件定義 + UIデザイン
   6月: 設計
   7-8月: 開発
   9月: テスト
   10月: 移行・研修・本番稼働
```

**初心者向けとの違い：**
- **段階リリース戦略**: 予算・期間の制約が厳しい場合、全機能を一括リリースするのではなく段階的に出すのが現実的な判断です
- **セキュリティ対策の具体化**: 個人情報を扱うシステムでは、セキュリティ対策を設計段階から計画に組み込む必要があります
- **予算リスクの段階的対策**: 問題が起きてから慌てるのではなく、段階的な対策を事前に準備しておきます
- **運用引き渡し計画**: プロジェクトは「リリースして終わり」ではなく、運用に引き渡すまでがPMの責任です

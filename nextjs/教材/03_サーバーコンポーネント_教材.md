# 第3章: サーバーコンポーネントとクライアントコンポーネント

## 3.1 React Server Components (RSC) とは

React Server Components は、**サーバー上でのみレンダリングされるコンポーネント**である。Next.js の App Router では、**すべてのコンポーネントがデフォルトでサーバーコンポーネント**として扱われる。

### サーバーコンポーネントのメリット

1. **バンドルサイズの削減**: サーバーコンポーネントの JavaScript はクライアントに送信されない
2. **直接的なデータアクセス**: データベースや API に直接アクセスできる
3. **セキュリティ**: API キーやトークンがクライアントに露出しない
4. **初回読み込みの高速化**: HTML がサーバーで生成されるため、FCP（First Contentful Paint）が速い
5. **SEO 最適化**: 検索エンジンがコンテンツをクロールしやすい

---

## 3.2 サーバーコンポーネント vs クライアントコンポーネント

### 比較表

| 特徴 | サーバーコンポーネント | クライアントコンポーネント |
|------|----------------------|--------------------------|
| デフォルト | はい（App Router） | `'use client'` が必要 |
| レンダリング場所 | サーバーのみ | サーバー + クライアント |
| JavaScript バンドル | 含まれない | 含まれる |
| `useState` / `useEffect` | 使用不可 | 使用可能 |
| イベントハンドラ | 使用不可 | 使用可能 |
| ブラウザ API | 使用不可 | 使用可能 |
| データベース直接アクセス | 可能 | 不可 |
| `async/await`（コンポーネント） | 可能 | 不可 |
| 環境変数（`process.env`） | すべて参照可能 | `NEXT_PUBLIC_` のみ |

### サーバーコンポーネントの例

```tsx
// app/users/page.tsx — サーバーコンポーネント（デフォルト）
// 'use client' がないのでサーバーコンポーネント

// データベースに直接アクセスできる
import { db } from '@/lib/database'

export default async function UsersPage() {
  // サーバー上で実行される
  const users = await db.user.findMany()

  return (
    <div>
      <h1>ユーザー一覧</h1>
      <ul>
        {users.map(user => (
          <li key={user.id}>{user.name}</li>
        ))}
      </ul>
    </div>
  )
}
```

### クライアントコンポーネントの例

```tsx
// app/components/Counter.tsx — クライアントコンポーネント
'use client' // この宣言が必要

import { useState } from 'react'

export default function Counter() {
  const [count, setCount] = useState(0)

  return (
    <div>
      <p>カウント: {count}</p>
      <button onClick={() => setCount(count + 1)}>
        増やす
      </button>
    </div>
  )
}
```

## 3.3 "use client" ディレクティブ

### 基本ルール

```tsx
'use client' // ファイルの先頭に配置する（import よりも前）

import { useState } from 'react'
// ...
```

### "use client" の境界

`'use client'` を宣言したコンポーネントと、そこから**インポートされるすべてのコンポーネント**がクライアントコンポーネントになる。

```
サーバーコンポーネント (デフォルト)
  └── 'use client' を宣言したコンポーネント ← ここが境界
        ├── インポートしたコンポーネント A → クライアント
        └── インポートしたコンポーネント B → クライアント
              └── インポートしたコンポーネント C → クライアント
```

> **重要**: `'use client'` は「このファイル以下がクライアント」という境界を設定するものであり、「このコンポーネントをクライアントで実行する」という単純な指示ではない。

## 3.4 使い分けのガイドライン

### サーバーコンポーネントを使うべき場合

| ユースケース | 理由 |
|-------------|------|
| データ取得 | サーバーで直接 DB/API にアクセスできる |
| 静的コンテンツの表示 | JS バンドルに含まれず軽量 |
| SEO が必要なコンテンツ | サーバーで HTML 生成される |
| 秘密情報へのアクセス | API キーが露出しない |
| 大きな依存関係の使用 | クライアントバンドルに影響しない |

### クライアントコンポーネントを使うべき場合

| ユースケース | 理由 |
|-------------|------|
| インタラクティブな UI | `onClick`, `onChange` 等が必要 |
| 状態管理 | `useState`, `useReducer` が必要 |
| ライフサイクル / 副作用 | `useEffect` が必要 |
| ブラウザ API | `window`, `localStorage` 等が必要 |
| カスタムフック（状態系） | React Hooks が必要 |
| サードパーティライブラリ | 多くのUIライブラリはクライアント用 |

### 実践的な判断フロー

```
コンポーネントを作る
  ↓
イベントハンドラ or useState/useEffect が必要？
  → YES → 'use client' を追加
  → NO → サーバーコンポーネントのまま
```

## 3.5 コンポジションパターン

### パターン 1: サーバー → クライアントの子として渡す

```tsx
// app/page.tsx（サーバーコンポーネント）
import InteractiveSection from './InteractiveSection'

export default async function Page() {
  const data = await fetchData() // サーバーで取得

  return (
    <div>
      <h1>データ一覧</h1>
      {/* サーバーで取得したデータをクライアントに渡す */}
      <InteractiveSection items={data} />
    </div>
  )
}
```

```tsx
// app/InteractiveSection.tsx（クライアントコンポーネント）
'use client'

import { useState } from 'react'

type Props = {
  items: { id: number; name: string }[]
}

export default function InteractiveSection({ items }: Props) {
  const [filter, setFilter] = useState('')

  const filtered = items.filter(item =>
    item.name.includes(filter)
  )

  return (
    <div>
      <input
        value={filter}
        onChange={(e) => setFilter(e.target.value)}
        placeholder="検索..."
      />
      <ul>
        {filtered.map(item => (
          <li key={item.id}>{item.name}</li>
        ))}
      </ul>
    </div>
  )
}
```

### パターン 2: children パターン（サーバーコンポーネントをクライアントの中に配置）

```tsx
// app/providers.tsx（クライアントコンポーネント）
'use client'

import { ThemeProvider } from 'next-themes'

export default function Providers({ children }: { children: React.ReactNode }) {
  return (
    <ThemeProvider attribute="class" defaultTheme="system">
      {children}
    </ThemeProvider>
  )
}
```

```tsx
// app/layout.tsx（サーバーコンポーネント）
import Providers from './providers'

export default function RootLayout({
  children,
}: {
  children: React.ReactNode
}) {
  return (
    <html lang="ja">
      <body>
        <Providers>
          {/* children はサーバーコンポーネントのまま */}
          {children}
        </Providers>
      </body>
    </html>
  )
}
```

> **重要**: `children` として渡されたサーバーコンポーネントは、クライアントコンポーネント内に配置されても**サーバーコンポーネントのまま**である。これは props として渡された React ノードがすでにサーバーでレンダリング済みだからである。

### パターン 3: クライアント境界を最小化する

```tsx
// 悪い例: ページ全体を 'use client' にしてしまう
'use client'
export default function Page() {
  const [open, setOpen] = useState(false)
  // ... 大量の静的コンテンツ ...
  return (
    <div>
      <StaticContent /> {/* これもクライアントに含まれてしまう */}
      <button onClick={() => setOpen(!open)}>開く</button>
      {open && <Modal />}
    </div>
  )
}

// 良い例: インタラクティブな部分だけを分離する
// app/page.tsx（サーバーコンポーネント）
import ToggleModal from './ToggleModal'

export default function Page() {
  return (
    <div>
      <StaticContent /> {/* サーバーコンポーネントのまま */}
      <ToggleModal />   {/* この部分だけクライアント */}
    </div>
  )
}

// app/ToggleModal.tsx（クライアントコンポーネント）
'use client'
import { useState } from 'react'

export default function ToggleModal() {
  const [open, setOpen] = useState(false)
  return (
    <>
      <button onClick={() => setOpen(!open)}>開く</button>
      {open && <div className="modal">モーダル内容</div>}
    </>
  )
}
```

## 3.6 ストリーミングと Suspense

### ストリーミング SSR

App Router はストリーミング SSR をサポートしており、ページの一部を先にクライアントに送信し、残りの部分を非同期で送信する。

```tsx
// app/dashboard/page.tsx（サーバーコンポーネント）
import { Suspense } from 'react'

export default function DashboardPage() {
  return (
    <div>
      <h1>ダッシュボード</h1>

      {/* 即座に表示される */}
      <WelcomeMessage />

      {/* データ取得中はフォールバックを表示 */}
      <Suspense fallback={<p>統計を読み込み中...</p>}>
        <Statistics />
      </Suspense>

      <Suspense fallback={<p>最近の活動を読み込み中...</p>}>
        <RecentActivity />
      </Suspense>
    </div>
  )
}

// 重い処理を含むサーバーコンポーネント
async function Statistics() {
  // この取得に3秒かかるとする
  const stats = await fetch('https://api.example.com/stats', {
    cache: 'no-store',
  }).then(res => res.json())

  return (
    <div>
      <h2>統計</h2>
      <p>ユーザー数: {stats.users}</p>
      <p>売上: {stats.revenue}</p>
    </div>
  )
}

async function RecentActivity() {
  // この取得に2秒かかるとする
  const activities = await fetch('https://api.example.com/activities', {
    cache: 'no-store',
  }).then(res => res.json())

  return (
    <div>
      <h2>最近の活動</h2>
      <ul>
        {activities.map((a: any) => (
          <li key={a.id}>{a.description}</li>
        ))}
      </ul>
    </div>
  )
}
```

### ストリーミングの流れ

1. サーバーが HTML のシェル（レイアウト + Suspense のフォールバック）を即座に送信
2. `RecentActivity` の取得が完了 → 2秒後にその部分が送信される
3. `Statistics` の取得が完了 → 3秒後にその部分が送信される
4. 各部分は取得完了順にクライアントに反映される（順序は保証されない）

## 3.7 サーバーコンポーネントの制約と注意点

### やってはいけないこと

```tsx
// サーバーコンポーネントでやってはいけないこと

// 1. useState / useEffect は使えない
import { useState } from 'react' // エラー！

// 2. イベントハンドラは使えない
<button onClick={() => {}}>クリック</button> // エラー！

// 3. ブラウザ API は使えない
const width = window.innerWidth // エラー！

// 4. クラスコンポーネントは使えない
class MyComponent extends React.Component {} // 非推奨
```

### シリアライズ可能な Props

サーバーコンポーネントからクライアントコンポーネントに渡す Props は**シリアライズ可能**でなければならない。

```tsx
// OK: プリミティブ値、配列、プレーンオブジェクト
<ClientComponent
  name="太郎"
  count={42}
  items={['a', 'b', 'c']}
  data={{ id: 1, name: 'test' }}
/>

// NG: 関数、クラスインスタンス、Date オブジェクト
<ClientComponent
  onClick={() => {}}       // エラー！関数は渡せない
  date={new Date()}         // エラー！
  element={<ServerComp />}  // OK: React エレメントは渡せる
/>
```

## 3.8 サードパーティライブラリとの共存

多くの既存ライブラリはクライアントコンポーネント前提で作られている。

### ラッパーパターン

```tsx
// app/components/ChartWrapper.tsx
'use client'

// Chart.js はクライアントでしか動かない
import { Chart } from 'chart.js'
import { Line } from 'react-chartjs-2'

type Props = {
  data: number[]
  labels: string[]
}

export default function ChartWrapper({ data, labels }: Props) {
  return (
    <Line
      data={{
        labels,
        datasets: [{ data, label: '売上' }],
      }}
    />
  )
}
```

```tsx
// app/analytics/page.tsx（サーバーコンポーネント）
import ChartWrapper from '@/components/ChartWrapper'

export default async function AnalyticsPage() {
  // サーバーでデータを取得
  const salesData = await fetchSalesData()

  return (
    <div>
      <h1>売上分析</h1>
      {/* データだけ渡してクライアントで描画 */}
      <ChartWrapper
        data={salesData.values}
        labels={salesData.months}
      />
    </div>
  )
}
```

## 3.9 パフォーマンス最適化のベストプラクティス

### 1. クライアントコンポーネントはリーフに配置する

コンポーネントツリーの**できるだけ末端（リーフ）**にクライアントコンポーネントを配置し、バンドルサイズを最小化する。

### 2. `'use client'` の境界を意識する

```
ルートレイアウト（サーバー）
  ├── ナビゲーション（サーバー）
  │     └── 検索バー（クライアント）← 境界
  ├── メインコンテンツ（サーバー）
  │     ├── 記事一覧（サーバー）
  │     └── いいねボタン（クライアント）← 境界
  └── フッター（サーバー）
```

### 3. 動的インポートで遅延読み込み

```tsx
import dynamic from 'next/dynamic'

// 重いクライアントコンポーネントを遅延読み込み
const HeavyEditor = dynamic(() => import('./HeavyEditor'), {
  loading: () => <p>エディタを読み込み中...</p>,
  ssr: false, // サーバーサイドレンダリングを無効化
})
```

## 3.10 まとめ

本章で学んだこと:

1. App Router では**すべてのコンポーネントがデフォルトでサーバーコンポーネント**
2. **`'use client'`** ディレクティブでクライアントコンポーネントを宣言する
3. サーバーコンポーネントは**直接データベースにアクセス**でき、**JS バンドルに含まれない**
4. クライアントコンポーネントは**インタラクティブな UI**（状態、イベント、ブラウザ API）に使う
5. **`children` パターン**でサーバーコンポーネントをクライアントコンポーネント内に配置できる
6. **ストリーミング + Suspense** で段階的なページ表示が可能
7. クライアント境界を**最小化**することがパフォーマンス最適化の鍵

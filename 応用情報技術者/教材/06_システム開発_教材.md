# 第6章 システム開発

> ソフトウェア開発の上流工程から下流工程まで、要件定義・設計手法・テスト技法・品質メトリクス・レビュー技法・CMMIモデルを体系的に学ぶ。午後試験の「システム開発」分野に直結する重要章。

---

## 6.1 要件定義

### 6.1.1 要件定義の位置づけ

```
企画プロセス
  │
  ▼
要件定義プロセス  ← 本節の対象
  │
  ├── 業務要件定義
  ├── 機能要件定義
  └── 非機能要件定義
  │
  ▼
開発プロセス（設計→実装→テスト）
```

### 6.1.2 機能要件と非機能要件

| 種別 | 説明 | 例 |
|------|------|-----|
| 機能要件 | システムが「何をするか」 | 注文登録、在庫照会、帳票出力 |
| 非機能要件 | システムが「どのように動くか」 | 性能、可用性、セキュリティ |

**非機能要件の分類（IPA非機能要件グレード）:**

| カテゴリ | 内容 | 例 |
|---------|------|-----|
| 可用性 | 稼働率、目標復旧時間 | 稼働率99.9%、RTO 4時間 |
| 性能・拡張性 | レスポンスタイム、同時接続数 | 応答3秒以内、1000同時接続 |
| 運用・保守性 | バックアップ、監視、パッチ適用 | 日次バックアップ、24時間監視 |
| 移行性 | 旧システムからの移行方式 | 段階移行、一括移行 |
| セキュリティ | 認証、暗号化、監査ログ | 多要素認証、TLS暗号化 |
| 環境・エコロジー | 設置環境、消費電力 | サーバ室温度25度以下 |

### 6.1.3 要件定義の技法

| 技法 | 説明 |
|------|------|
| ヒアリング | ステークホルダーへのインタビュー |
| ユースケース分析 | アクターとシステムの対話を記述 |
| プロトタイピング | 試作品で要件を確認 |
| ブレインストーミング | 自由なアイデア出し |
| KJ法 | アイデアをグループ化して整理 |

**ポイントまとめ:**
- 機能要件と非機能要件の区別は基本中の基本
- 非機能要件の6カテゴリを覚える
- 午後試験では要件の抽出・整理を問われることが多い

## 6.2 設計手法

### 6.2.1 構造化設計

**構造化分析の主要図法:**

| 図法 | 用途 |
|------|------|
| DFD（データフロー図） | データの流れを表現 |
| ER図 | データ構造（エンティティと関連）を表現 |
| 状態遷移図 | システムの状態変化を表現 |
| 決定表（デシジョンテーブル） | 条件と動作の組合せを表現 |

**DFDの記号:**
```
○ : プロセス（処理）
→ : データフロー
＝ : データストア
□ : 外部エンティティ（源泉/吸収）

例:
□顧客 → 注文データ → ○注文処理 → 在庫データ → ＝在庫DB
```

### 6.2.2 オブジェクト指向設計

**UMLの主要なダイアグラム:**

| 分類 | ダイアグラム | 用途 |
|------|------------|------|
| 構造図 | クラス図 | クラスの属性・操作・関連 |
| 構造図 | コンポーネント図 | ソフトウェアの構成要素 |
| 構造図 | 配置図 | ハードウェアとソフトウェアの配置 |
| 振舞い図 | ユースケース図 | 機能とアクターの関係 |
| 振舞い図 | シーケンス図 | オブジェクト間のメッセージ交換（時系列） |
| 振舞い図 | アクティビティ図 | 処理の流れ（フローチャートに近い） |
| 振舞い図 | 状態マシン図 | オブジェクトの状態遷移 |

### 6.2.3 設計原則

| 原則 | 説明 |
|------|------|
| モジュール結合度 | モジュール間の依存度（低いほど良い） |
| モジュール凝集度 | モジュール内の機能の関連度（高いほど良い） |
| 情報隠蔽 | 内部実装を隠し、インタフェースのみ公開 |
| 開放閉鎖原則 | 拡張に開き、修正に閉じる |

**モジュール結合度（弱い順 = 良い順）:**
```
データ結合       ← 最も弱い（最良）
  ↓ データ項目のみを受け渡し
スタンプ結合
  ↓ データ構造を受け渡し
制御結合
  ↓ 制御フラグを受け渡し
外部結合
  ↓ 外部データを共有
共通結合
  ↓ グローバルデータを共有
内容結合         ← 最も強い（最悪）
    他モジュールの内部を直接参照
```

**モジュール凝集度（強い順 = 良い順）:**
```
機能的凝集       ← 最も強い（最良）
  ↓ 単一機能を実現
情報的凝集
  ↓ 同じデータ構造を操作する複数機能
連絡的凝集
  ↓ 同じデータを使う逐次的な処理
手順的凝集
  ↓ 関連する手順の集まり
時間的凝集
  ↓ 同時期に実行される処理の集まり
論理的凝集
  ↓ 論理的に類似した処理の集まり
偶発的凝集       ← 最も弱い（最悪）
    無関係な処理の寄せ集め
```

**ポイントまとめ:**
- 結合度は低いほど良い、凝集度は高いほど良い
- 結合度・凝集度の各レベルの順序と具体例を覚える
- 午前試験ではレベルの並び替え問題が定番

## 6.3 テスト技法

### 6.3.1 テストの工程

```
要件定義 ←──────────────────→ システムテスト（受入テスト）
  │                              ↑ 対応関係
外部設計 ←──────────────────→ 結合テスト
  │                              ↑
内部設計 ←──────────────────→ 単体テスト
  │
コーディング
```

この対応関係を**V字モデル**と呼びます。

```
要件定義 ─────────────────────────── システムテスト
   ╲                                ╱
    外部設計 ──────────────── 結合テスト
       ╲                     ╱
        内部設計 ──── 単体テスト
            ╲       ╱
           コーディング
```

### 6.3.2 ブラックボックステスト

内部構造を意識せず、仕様に基づいてテストケースを作成する手法です。

**同値分割法:**
```
入力: 1〜100の整数
有効同値クラス: 1〜100
無効同値クラス: 0以下、101以上

テストケース: 各同値クラスから代表値を選ぶ
  有効: 50
  無効: -1, 150
```

**境界値分析:**
```
入力: 1〜100の整数
境界値: 0, 1, 100, 101

テストケース: 境界とその前後の値
  0(無効), 1(有効), 100(有効), 101(無効)
```

**デシジョンテーブルテスト:**
```
条件/動作の全組合せを網羅的にテスト

条件1: 会員か    Y Y Y Y N N N N
条件2: 3000円以上 Y Y N N Y Y N N
条件3: クーポンあり Y N Y N Y N Y N
──────────────────────────────
動作: 割引率     20 15 10 5  10 5  0  0
```

**原因結果グラフ:**
入力条件（原因）と出力（結果）の論理関係を分析し、デシジョンテーブルを効率的に作成する技法です。

### 6.3.3 ホワイトボックステスト

プログラムの内部構造に基づいてテストケースを作成する手法です。

**カバレッジ基準:**

| カバレッジ | 基準 | 網羅性 |
|-----------|------|--------|
| 命令網羅（C0） | 全命令を最低1回実行 | 低 |
| 分岐網羅（C1） | 全分岐の真/偽を最低1回実行 | 中 |
| 条件網羅（C2） | 全条件の真/偽を最低1回実行 | 中〜高 |
| 複合条件網羅 | 条件の全組合せを実行 | 高 |
| パス網羅 | 全実行パスを実行 | 最高（現実的でない場合が多い） |

```
例: if (A && B) の場合

命令網羅(C0): A=true, B=true の1ケースで達成
分岐網羅(C1): A=true,B=true + A=false,B=false の2ケース
条件網羅(C2): A=true,B=false + A=false,B=true の2ケース
             （分岐網羅を満たさない場合がある）
```

### 6.3.4 テストの種類

| テスト | 目的 |
|--------|------|
| 回帰テスト（リグレッションテスト） | 変更が既存機能に影響していないか確認 |
| 性能テスト（負荷テスト） | 想定負荷での性能を確認 |
| ストレステスト | 限界を超えた負荷での動作を確認 |
| セキュリティテスト | 脆弱性がないか確認 |
| ユーザビリティテスト | 使いやすさの確認 |

**ポイントまとめ:**
- V字モデルの対応関係を理解する
- 同値分割と境界値分析のテストケース作成ができるようにする
- カバレッジの強さの順序（C0 < C1 < C2 < 複合 < パス）を覚える

## 6.4 品質メトリクス

### 6.4.1 ソフトウェア品質特性（ISO/IEC 25010）

| 品質特性 | 説明 |
|---------|------|
| 機能適合性 | 必要な機能が正しく提供されるか |
| 性能効率性 | 時間・資源の効率性 |
| 互換性 | 他システムとの共存・相互運用性 |
| 使用性 | 使いやすさ、学習しやすさ |
| 信頼性 | 障害なく動作し続けるか |
| セキュリティ | 不正からの保護 |
| 保守性 | 修正・改善のしやすさ |
| 移植性 | 他環境への移植のしやすさ |

### 6.4.2 バグ管理のメトリクス

**バグ曲線（信頼度成長曲線）:**
```
累積
バグ数
  ↑        ＿＿＿＿＿＿ ← 収束（バグが出尽くした）
  │      ╱
  │    ╱
  │  ╱
  │╱
  └──────────────→ テスト時間

  S字カーブを描くのが理想
  ・直線的に増加し続ける → テスト不十分 or 品質が悪い
  ・早期に収束 → テストが不十分な可能性
```

**バグ密度:**
```
バグ密度 = 検出バグ数 / プログラム規模（KLOC）

例: 50件のバグ / 10KLOC = 5件/KLOC
業界平均: 1〜10件/KLOC程度
```

### 6.4.3 生産性のメトリクス

| メトリクス | 計算式 | 用途 |
|-----------|--------|------|
| FP（ファンクションポイント） | 機能の種類と複雑度で算出 | 言語非依存の規模見積り |
| COCOMO | 規模から工数を算出 | コスト見積り |
| LOC（Lines of Code） | コード行数 | 単純な規模指標 |

**FP法の入力項目:**
```
外部入力(EI): 外部からのデータ入力
外部出力(EO): 外部へのデータ出力
外部照会(EQ): 外部からの問合せ
内部論理ファイル(ILF): 内部で管理するデータ
外部インタフェースファイル(EIF): 外部から参照するデータ

各項目に複雑度（低/中/高）の重みを掛けて合計 = 未調整FP
未調整FP × 調整係数 = 調整済みFP
```

**ポイントまとめ:**
- ISO/IEC 25010の8つの品質特性を覚える
- 信頼度成長曲線の形状から品質状況を判断できるようにする
- FP法の5つの入力項目は午前試験で頻出

## 6.5 レビュー技法

### 6.5.1 レビューの種類

| レビュー | 特徴 |
|---------|------|
| インスペクション | 最も厳密。モデレータが主導、チェックリスト使用、欠陥記録 |
| ウォークスルー | 作成者が説明、参加者が指摘。インスペクションより軽量 |
| テクニカルレビュー | 技術的な観点で専門家がレビュー |
| ラウンドロビン | 参加者が順番に説明役を担当 |
| パスアラウンド | ドキュメントを回覧し、個別にコメント |
| アドホックレビュー | 非公式、机上で気軽に確認 |

### 6.5.2 インスペクションの手順

```
1. 計画: レビュー対象・参加者・スケジュール決定
2. 準備: 参加者が個別にドキュメントを精査
3. 会議: モデレータ主導で欠陥を議論・記録
4. 修正: 作成者が欠陥を修正
5. 確認: モデレータが修正結果を確認
```

**役割:**
| 役割 | 責務 |
|------|------|
| モデレータ | レビュー全体の管理・進行 |
| 作成者 | レビュー対象物の作成者 |
| 読み手 | ドキュメントを読み上げる |
| 記録係 | 欠陥・議論内容を記録 |
| レビューア | 欠陥を発見・指摘 |

**ポイントまとめ:**
- インスペクションが最も厳密なレビュー手法
- インスペクションの手順と各役割を覚える
- ウォークスルーは作成者主導、インスペクションはモデレータ主導

## 6.6 CMMIモデル

### 6.6.1 CMMIとは

**CMMI（Capability Maturity Model Integration: 能力成熟度モデル統合）** は、組織のプロセス改善のための成熟度モデルです。

### 6.6.2 成熟度レベル

| レベル | 名称 | 特徴 |
|--------|------|------|
| レベル1 | 初期 | プロセスが場当たり的、属人的 |
| レベル2 | 管理された | プロジェクト単位で基本的な管理がされている |
| レベル3 | 定義された | 組織標準プロセスが定義され、全プロジェクトで使用 |
| レベル4 | 定量的に管理された | プロセスを定量的に測定・管理 |
| レベル5 | 最適化している | 継続的なプロセス改善が実施されている |

```
レベル5: 最適化    ← 継続的改善
  ↑
レベル4: 定量管理  ← 統計的管理
  ↑
レベル3: 定義     ← 組織標準
  ↑
レベル2: 管理     ← プロジェクト管理
  ↑
レベル1: 初期     ← 属人的・場当たり的
```

**ポイントまとめ:**
- CMMIの5段階の成熟度レベルの名称と特徴を覚える
- レベル3は「組織標準プロセスの定義」がキーワード
- レベル4は「定量的管理」、レベル5は「継続的改善」

## 6.7 開発プロセスモデル

### 6.7.1 代表的なプロセスモデル

| モデル | 特徴 | 適用場面 |
|--------|------|---------|
| ウォーターフォール | 各工程を順番に完了 | 要件が明確な大規模開発 |
| スパイラル | リスク分析→プロトタイプを反復 | リスクの高い開発 |
| アジャイル(スクラム) | 短いイテレーションで反復開発 | 要件変更が多い開発 |
| DevOps | 開発と運用の統合、CI/CD | 継続的デリバリーが必要な開発 |

### 6.7.2 アジャイル開発（スクラム）

```
┌──────────────────────────────────────┐
│          スクラムのフレームワーク        │
│                                       │
│  プロダクトバックログ                    │
│  │                                    │
│  ├→ スプリント計画                      │
│  │   │                                │
│  │   ├→ スプリント(1-4週間)             │
│  │   │   ├── デイリースクラム(毎日15分)  │
│  │   │   ├── 開発作業                   │
│  │   │   └── スプリントバックログ        │
│  │   │                                │
│  │   ├→ スプリントレビュー(成果物確認)   │
│  │   └→ スプリントレトロスペクティブ     │
│  │       (振り返り)                     │
│  └→ 次のスプリントへ                    │
└──────────────────────────────────────┘
```

**スクラムの3つの役割:**
| 役割 | 責務 |
|------|------|
| プロダクトオーナー | プロダクトバックログの管理、優先順位決定 |
| スクラムマスター | スクラムプロセスの促進、障害の除去 |
| 開発チーム | スプリント内の開発作業を実施（自己組織化） |

## 基本問題

### 問題1：QAとQCの区別

以下の活動が「品質保証（QA）」か「品質管理（QC）」かを判定してください。

1. コーディング規約を策定する
2. 単体テストを実行し、バグを検出する
3. CI/CDパイプラインにリンターを組み込む
4. リリース前の受入テストでバグを報告する
5. コードレビュー制度を導入する
6. ペネトレーションテストで脆弱性を発見する
7. 開発プロセスの監査を実施する
8. テストカバレッジを測定する

**期待される出力例：**

```
QA: 1, 3, 5, 7（プロセスの改善・予防活動）
QC: 2, 4, 6, 8（成果物の検査・検出活動）
```

<details>
<summary>解答例</summary>

```
■ QA（品質保証）- プロセスの改善・品質問題の予防
  1. コーディング規約を策定する
     → プロセスの標準化により品質問題を予防する活動
  3. CI/CDパイプラインにリンターを組み込む
     → 自動化により品質を継続的に保証する仕組みの構築
  5. コードレビュー制度を導入する
     → 制度（プロセス）を整備して品質を保証する活動
  7. 開発プロセスの監査を実施する
     → プロセスが正しく運用されているかの確認

■ QC（品質管理）- 成果物の検査・品質問題の検出
  2. 単体テストを実行し、バグを検出する
     → 成果物（コード）を検査して不良を検出する活動
  4. リリース前の受入テストでバグを報告する
     → 成果物の最終検査
  6. ペネトレーションテストで脆弱性を発見する
     → 成果物のセキュリティを検査する活動
  8. テストカバレッジを測定する
     → 成果物の品質指標を測定する活動
```

</details>

### 問題2：テストピラミッドの適用

以下の機能に対して、どのレベルのテスト（単体テスト / 統合テスト / E2Eテスト）で検証すべきか判断してください。

1. 消費税の計算ロジック（税率10%、軽減税率8%の判定）
2. ユーザー登録 → メール認証 → ログインの一連のフロー
3. APIとデータベースの間でデータが正しく保存・取得されること
4. パスワードのバリデーション（8文字以上、英数字混在）
5. 商品一覧画面の表示 → 商品をカートに追加 → 購入完了の全フロー

**期待される出力例：**

```
1. 単体テスト - 純粋な計算ロジックであり、外部依存なし
```

<details>
<summary>解答例</summary>

```
1. 単体テスト
   理由：消費税の計算は純粋なビジネスロジック。入力値（金額、品目）に
   対する出力値（税込み価格）が決まっている。外部依存がなく、
   高速に多数のパターンを検証できる。

2. E2Eテスト
   理由：登録→メール認証→ログインは複数のシステム（DB、メール送信、
   認証サービス）をまたぐ一連のユーザーフロー。
   エンドツーエンドで正しく動作することを確認する必要がある。

3. 統合テスト
   理由：APIとデータベースという2つのコンポーネントの連携を検証する。
   単体テストではモック化してしまう部分を実際に接続して確認する。

4. 単体テスト
   理由：バリデーションロジックは入力文字列に対する真偽値を返す
   純粋な関数。境界値（7文字/8文字、英字のみ/英数字混在）を
   多数テストすべきであり、単体テストが最適。

5. E2Eテスト
   理由：ユーザー視点での購入フロー全体を検証する。
   UI操作→API→DB→決済→確認画面の一連の動作を確認。
   ただし、主要な購入シナリオに絞って実施する（数を絞る）。
```

</details>

### 問題3：品質メトリクスの読み取り

以下のプロジェクトの品質メトリクスを見て、問題点と改善提案を述べてください。

```
品質メトリクス（スプリント10時点）：

| メトリクス              | 目標   | 実績   | 評価 |
|------------------------|--------|--------|------|
| テストカバレッジ        | 80%    | 62%    | ×    |
| バグ検出率（テスト工程） | 90%    | 75%    | ×    |
| 本番障害件数/月         | 2件以下 | 5件    | ×    |
| コードレビュー実施率    | 100%   | 85%    | △    |
| 平均バグ修正時間        | 4時間   | 3時間  | ○    |
| リリース後1週間の       | 0件    | 2件    | ×    |
|   重大バグ件数          |        |        |      |
```

<details>
<summary>解答例</summary>

```
■ 問題点の分析

1. テストカバレッジ（目標80% → 実績62%）
   問題：18%のギャップがあり、テストされていないコードが多い。
   影響：未テストのコードからバグが本番に流出するリスク。

2. バグ検出率（目標90% → 実績75%）
   問題：テスト工程で検出すべきバグの25%が漏れている。
   影響：本番障害の増加に直結している。

3. 本番障害件数（目標2件以下 → 実績5件）
   問題：目標の2.5倍。上記2つの問題が原因と推測される。

4. コードレビュー実施率（目標100% → 実績85%）
   問題：15%のPRがレビューなしでマージされている。
   影響：品質の門番が機能していない箇所がある。

■ 根本原因の推測

テストカバレッジ不足 → バグ検出率低下 → 本番障害増加
コードレビューの漏れ → 品質問題の見逃し → 本番障害増加

スケジュールプレッシャーにより、テストとレビューが
省略されている可能性が高い。

■ 改善提案

1. テストカバレッジの向上
   ・CIで「カバレッジ80%未満のPRはマージ不可」のゲートを設定
   ・各スプリントで「テスト負債解消」のタスクを10%確保
   ・新規コードは必ずテストを書くルールを徹底

2. コードレビューの徹底
   ・ブランチ保護ルールで「1名以上のApprove必須」を設定
   ・レビュー待ちが長い場合のエスカレーションルールを策定
   ・レビューのSLA（24時間以内）を設定

3. テスト戦略の見直し
   ・テスト自動化の範囲を拡大（特に回帰テスト）
   ・リスクベースドテスト：重要機能を優先的にテスト
   ・本番障害の根本原因分析（RCA）を実施し、テスト観点に反映

4. プロセスの改善
   ・スプリントの速度を下げてでも品質を優先する方針をPOと合意
   ・品質メトリクスをスプリントレビューで毎回確認する
```

</details>

## 応用問題

### 問題4：テスト計画書の作成

以下のプロジェクトについて、テスト計画の概要を作成してください。

**プロジェクト：** オンライン予約システム（飲食店向け）

主要機能：
- 顧客：店舗検索、予約登録、予約変更・キャンセル
- 店舗管理者：予約一覧確認、座席管理、通知設定
- システム管理者：店舗登録、マスタ管理

テスト計画に以下を含めてください：
- テスト対象と範囲
- テストの種類と方針
- 合格基準
- リスクと対策

<details>
<summary>解答例</summary>

```markdown
# テスト計画書（概要）：オンライン予約システム

## 1. テスト対象と範囲

【テスト対象】
- 顧客機能：店舗検索、予約登録、予約変更・キャンセル
- 店舗管理者機能：予約一覧、座席管理、通知設定
- システム管理者機能：店舗登録、マスタ管理
- 非機能要件：性能、セキュリティ、可用性

【テスト対象外】
- 決済機能（外部サービスを利用、先方の責任範囲）
- SEO対策（テスト対象外だが、別途チェック）

## 2. テストの種類と方針

| テスト種類     | 対象                    | ツール      | 担当    |
|-------------- |------------------------|------------|---------|
| 単体テスト     | ビジネスロジック全般      | Jest       | 開発者  |
| 統合テスト     | API + DB連携            | Jest       | 開発者  |
| E2Eテスト      | 主要ユーザーシナリオ5本   | Playwright | QA     |
| 性能テスト     | 予約API（同時100件）     | k6         | SRE    |
| セキュリティ   | 認証・認可、入力値       | OWASP ZAP  | セキュリティ|
| 受入テスト     | 全機能                  | 手動       | PO     |

## 3. 合格基準

| 指標                     | 基準                       |
|-------------------------|---------------------------|
| テストカバレッジ          | 80%以上（ステートメント）    |
| 重大バグ（Sev1/Sev2）    | 0件                        |
| 中程度バグ（Sev3）        | 5件以下                    |
| 軽微バグ（Sev4）          | 制限なし（バックログに記録） |
| E2Eテスト合格率           | 100%（5シナリオすべて合格）  |
| 予約API応答時間           | P95で500ms以下             |
| 受入テスト                | POの承認                    |

## 4. リスクと対策

| リスク                        | 影響  | 対策                       |
|------------------------------ |------|---------------------------|
| テスト環境の不安定さ           | 高   | Docker化で環境を再現可能に  |
| テスト工数の不足              | 中   | テスト自動化で回帰テスト効率化|
| 予約の同時実行競合             | 高   | 負荷テストでダブルブッキング |
|                               |      | がないことを検証            |
| 外部通知サービスの障害         | 中   | モックを使ったテストも並行実施|
```

## チャレンジ問題

### 問題5：コードレビューの改善

あなたのチームでは、以下の問題が発生しています。

- コードレビューに平均3日かかっている（目標は1日以内）
- レビューの品質にばらつきがある（表面的なレビューが多い）
- レビュー依頼が特定のシニアエンジニアに集中している

この問題を解決するためのコードレビュー改善計画を作成してください。プロセス、ルール、ツール活用の観点で提案してください。

<details>
<summary>解答例</summary>

```
■ コードレビュー改善計画

【課題の根本原因分析】
1. レビュー遅延（3日）→ 特定メンバーへの集中、優先度が低い扱い
2. 品質のばらつき → レビュー観点が標準化されていない
3. 負荷の偏り → レビュワーのアサインルールが不明確

【改善策】

■ 1. プロセスの改善

  (a) レビュワーの自動アサイン
    ・GitHubのCODEOWNERSを設定し、変更ファイルに応じて
      適切なレビュワーを自動指定する
    ・ラウンドロビンでレビュワーを均等に分散させる

  (b) レビューSLAの設定
    ・小規模PR（100行未満）：4時間以内にレビュー開始
    ・中規模PR（100〜300行）：8時間以内にレビュー開始
    ・大規模PR（300行超）：分割を推奨。やむを得ない場合は24時間以内
    ・SLAに達したらSlackで自動リマインド

  (c) PRサイズの制限
    ・1PRあたり300行以下を目標とする
    ・大きなPRは機能単位で分割する
    ・CI で行数チェックし、超えた場合は警告

■ 2. レビュー品質の標準化

  (a) レビューチェックリストの導入
    □ ビジネスロジックは要件を満たしているか
    □ エラーハンドリングは適切か
    □ テストは十分に書かれているか
    □ セキュリティ上の問題はないか
    □ パフォーマンスに影響する変更はないか
    □ コーディング規約に沿っているか

  (b) 自動チェックの強化
    ・リンター（ESLint、Prettier）による自動フォーマット
    ・静的解析ツール（SonarQube）による品質チェック
    ・テストカバレッジの自動計測
    → 機械でチェックできることは人がやらない

  (c) レビューコメントの分類
    ・[must] 必ず修正（バグ、セキュリティ問題）
    ・[should] 修正を推奨（可読性、設計改善）
    ・[nit] 細かい指摘（命名、スタイル）
    ・[question] 理解のための質問

■ 3. 文化の醸成

  ・ジュニアメンバーもレビュワーに参加する
    （学習機会として、シニアとペアレビュー）
  ・レビューは「批判」ではなく「チームの品質向上」が目的と周知
  ・良いレビューコメントを週次で共有する

■ 効果測定

| 指標                    | 現状  | 目標      |
|------------------------|-------|----------|
| 平均レビュー時間         | 3日   | 1日以内   |
| レビュワーの偏り（ジニ係数）| 高   | 均等に近い |
| PR平均サイズ            | 500行 | 200行以下 |
| 本番障害件数/月          | 5件   | 2件以下   |
```

</details>

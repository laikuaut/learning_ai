# 第4章：詳細設計書（内部設計書）

## この章のゴール

- 詳細設計書（Detailed Design Document）の目的と構成を理解する
- フローチャートの読み書きができるようになる
- シーケンス図で処理の流れを記述できるようになる
- クラス設計の基本を理解する
- 擬似コード（Pseudocode）で処理ロジックを記述できるようになる

---

## 4.1 詳細設計書とは

### 詳細設計書の位置づけ

詳細設計書（内部設計書）は、**モジュール内部の処理ロジックを具体的に記述する文書**です。基本設計で定めた「システムの構造」を、「実際にどう処理するか」のレベルまで落とし込みます。

```
【基本設計と詳細設計の違い】

基本設計（外部設計）          詳細設計（内部設計）
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
「何をするか」を定義          「どう処理するか」を定義

・機能の構成・分割            ・処理フローチャート
・画面レイアウト              ・条件分岐の詳細ロジック
・APIのインタフェース         ・データの変換・加工処理
・データベースのテーブル構造    ・エラーハンドリングの詳細
・外部システム連携方式         ・擬似コード

粒度：機能・モジュール単位    粒度：関数・メソッド単位
読者：PM・開発者・レビュアー  読者：開発者
```

### 詳細設計書の構成

```
詳細設計書の目次（例）
━━━━━━━━━━━━━━━━━━━━━━━
1. モジュール概要
   1.1 モジュールID・名称
   1.2 機能概要
   1.3 関連する基本設計書

2. 処理設計
   2.1 処理フロー（フローチャート）
   2.2 処理詳細（擬似コード）
   2.3 シーケンス図

3. データ設計
   3.1 入力データ
   3.2 出力データ
   3.3 内部データ（変数・定数）

4. エラー処理設計
   4.1 エラー一覧
   4.2 エラー処理方式

5. 制約・注意事項
━━━━━━━━━━━━━━━━━━━━━━━
```

## 4.3 シーケンス図

### シーケンス図とは

シーケンス図は、**オブジェクト間のメッセージのやり取りを時系列で表現する図**です。処理の順序と、どのコンポーネントが何をするかが一目でわかります。

## 4.4 擬似コード（Pseudocode）

### 擬似コードとは

擬似コードは、**プログラミング言語に依存しない形で処理ロジックを記述する方法**です。実際のコードに近い粒度で、処理の流れを自然言語と制御構造で表現します。

### 擬似コードの例：月次勤怠集計処理

```
関数: 月次勤怠集計(対象年月)
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

入力: 対象年月（YYYYMM形式）
出力: 集計結果リスト
前提: 対象月の全打刻データがDBに保存済み

処理:
  1. 対象年月の営業日カレンダーを取得する
  2. 全社員リストを取得する
  3. 集計結果リスト = 空リスト

  4. 各社員について繰り返す:
     4.1 社員の対象月の打刻データを全件取得する
     4.2 集計レコード = 新規作成（社員ID、対象年月）

     4.3 各営業日について繰り返す:
         4.3.1 当日の出勤打刻と退勤打刻を取得する

         4.3.2 もし 出勤打刻と退勤打刻が両方存在する場合:
               ・実労働時間 = 退勤時刻 - 出勤時刻 - 休憩時間
               ・もし 実労働時間 > 所定労働時間(8h) の場合:
                   残業時間 = 実労働時間 - 所定労働時間
                   もし 退勤時刻 > 22:00 の場合:
                       深夜残業時間 = 退勤時刻 - 22:00
               ・もし 出勤時刻 > コアタイム開始(10:00) の場合:
                   遅刻回数 += 1

         4.3.3 もし 出勤打刻がない場合:
               ・有給休暇申請が承認済みか確認する
               ・もし 承認済みの場合:
                   有給取得日数 += 1（半休の場合は0.5）
               ・そうでない場合:
                   欠勤日数 += 1

     4.4 集計レコードに以下を設定する:
         ・所定労働日数 = 営業日数
         ・実労働日数
         ・所定労働時間 = 営業日数 × 8
         ・実労働時間（合計）
         ・残業時間（合計）
         ・深夜残業時間（合計）
         ・有給取得日数
         ・遅刻回数
         ・欠勤日数

     4.5 集計結果リストに集計レコードを追加する

  5. 集計結果リストをDBに保存する
  6. 集計結果リストを返す

例外処理:
  ・DBアクセスエラーの場合: 処理を中断し、エラーログを記録する
  ・打刻データ不整合の場合: 該当社員をスキップし、警告ログを記録する
```

## 4.5 エラー処理設計

### エラー一覧の設計

```
【エラー一覧（例：注文処理モジュール）】

| エラーID  | エラー名         | 発生条件                  | エラーメッセージ              | 対処                    |
|----------|-----------------|--------------------------|----------------------------|------------------------|
| E-ORD-001| 在庫不足         | 在庫数 < 注文数量          | 「在庫が不足しています」       | 注文を受け付けない        |
| E-ORD-002| 決済失敗         | 決済APIがエラーを返した     | 「決済に失敗しました」        | 在庫引当を戻し、再試行を促す|
| E-ORD-003| カート期限切れ    | カート作成から30日超過      | 「カートの有効期限が切れました」| カートを無効化する        |
| E-ORD-004| 注文上限超過     | 1日の注文数が10件超過      | 「本日の注文上限に達しました」 | 注文を受け付けない        |
| E-ORD-005| 配送先不正       | 配送先住所のバリデーション失敗| 「配送先を確認してください」  | 入力画面に戻す           |
| E-ORD-006| DB保存失敗       | DB接続エラー・タイムアウト  | 「システムエラーが発生しました」| リトライ後、エラーログ記録 |
```

### エラーハンドリング方針

```
【エラーハンドリングの階層】

┌────────────────────────────────────────────────┐
│ レイヤー1：バリデーションエラー（入力値チェック）    │
│                                                │
│ ・フロントエンドで事前チェック（即時フィードバック）  │
│ ・バックエンドでも必ず再チェック（セキュリティ）     │
│ ・HTTPステータス：400 Bad Request                 │
├────────────────────────────────────────────────┤
│ レイヤー2：ビジネスロジックエラー                   │
│                                                │
│ ・在庫不足、残高不足、権限不足など                  │
│ ・ユーザーに具体的なエラーメッセージを表示           │
│ ・HTTPステータス：409 Conflict / 403 Forbidden    │
├────────────────────────────────────────────────┤
│ レイヤー3：システムエラー                          │
│                                                │
│ ・DB接続エラー、外部API障害、メモリ不足など          │
│ ・ユーザーには汎用メッセージ、詳細はログに記録       │
│ ・HTTPステータス：500 Internal Server Error       │
│ ・アラートで運用チームに通知                       │
└────────────────────────────────────────────────┘
```

## 4.6 クラス設計

### クラス図の基本

```
【クラス図の表記】

┌──────────────────────┐
│     <<クラス名>>       │
├──────────────────────┤
│ - privateField: 型    │  属性（フィールド）
│ + publicField: 型     │  - : private
│ # protectedField: 型  │  + : public
├──────────────────────┤  # : protected
│ + publicMethod(): 戻り値│ 操作（メソッド）
│ - privateMethod(): void│
└──────────────────────┘
```

### クラス設計の例：注文管理

```
┌────────────────────────┐     ┌──────────────────────────┐
│        Order           │     │       OrderItem          │
├────────────────────────┤     ├──────────────────────────┤
│ - orderId: string      │     │ - itemId: string         │
│ - userId: string       │ 1  *│ - productId: string      │
│ - status: OrderStatus  │◆────│ - quantity: number       │
│ - totalAmount: number  │     │ - unitPrice: number      │
│ - orderedAt: DateTime  │     │ - subtotal: number       │
├────────────────────────┤     ├──────────────────────────┤
│ + create(): Order      │     │ + calcSubtotal(): number │
│ + cancel(): void       │     └──────────────────────────┘
│ + calcTotal(): number  │
└────────┬───────────────┘
         │ 使用
         ▼
┌────────────────────────┐
│     OrderService       │
├────────────────────────┤
│ - orderRepo: Repository│
│ - stockService         │
│ - paymentService       │
├────────────────────────┤
│ + placeOrder(): Order  │
│ + cancelOrder(): void  │
│ + getOrderHistory()    │
└────────────────────────┘

【関連の種類】
◆──── : コンポジション（OrderがOrderItemを所有。Orderが消えるとItemも消える）
1  *  : 多重度（1つのOrderに複数のOrderItem）
──→   : 依存（OrderServiceがOrderを使用）
```

## 4.7 詳細設計のレビューポイント

### チェックリスト

```
【詳細設計レビューのチェックリスト】

□ 処理フローに漏れがないか
  ├─ □ 正常系のフローが網羅されているか
  ├─ □ 異常系（エラー時）のフローが定義されているか
  └─ □ 境界値（0件、最大件数）の処理が考慮されているか

□ 基本設計との整合性
  ├─ □ 基本設計で定義した機能がすべて詳細化されているか
  ├─ □ API仕様と処理ロジックが矛盾していないか
  └─ □ DB設計のテーブル定義と整合しているか

□ 実装可能性
  ├─ □ 擬似コードが具体的で実装に落とし込めるか
  ├─ □ 使用するライブラリ・フレームワークの制約を考慮しているか
  └─ □ パフォーマンス要件を満たせる設計か

□ 保守性
  ├─ □ 処理の責務が適切に分離されているか
  ├─ □ 共通処理が適切に抽出されているか
  └─ □ 変更が発生しやすい箇所が局所化されているか
```

```
│  2. フローチャートで処理の分岐・ループを視覚化する             │
│     → 正常系だけでなく異常系も必ず描く                       │
│                                                             │
│  3. シーケンス図でコンポーネント間の連携を時系列で表現する      │
│     → 補償トランザクション（エラー時の戻し処理）も設計する     │
│                                                             │
│  4. 擬似コードで処理ロジックを言語非依存で記述する             │
│     → 入力・出力・例外処理を明記する                         │
│                                                             │
│  5. エラー処理は3階層（バリデーション・ビジネス・システム）     │
│     で設計する                                               │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

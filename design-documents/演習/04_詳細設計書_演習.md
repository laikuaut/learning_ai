### 問題1：クラス図の読み取り

以下のクラス図を読み取り、質問に答えてください。

```
┌────────────────────┐
│     <<interface>>   │
│   IUserRepository   │
├────────────────────┤
│ + findById(id): User│
│ + findByEmail(email)│
│ + save(user): User  │
│ + delete(id): void  │
└────────▲───────────┘
         │ implements
┌────────┴───────────┐        ┌────────────────────┐
│ UserRepository      │        │   UserService       │
├────────────────────┤        ├────────────────────┤
│ - db: Database      │        │ - userRepo          │
├────────────────────┤        │ - passwordHasher    │
│ + findById(id)      │◄───── │ ├────────────────────┤
│ + findByEmail(email)│  uses  │ + register(dto)     │
│ + save(user)        │        │ + login(email, pw)  │
│ + delete(id)        │        │ + updateProfile(dto)│
└────────────────────┘        └────────────────────┘
```

1. `IUserRepository` は何ですか（クラス/インターフェース）？
2. `UserRepository` と `IUserRepository` の関係は何ですか？
3. `UserService` は `UserRepository` をどのように利用していますか？
4. `UserRepository` の `db` フィールドのアクセス修飾子は何ですか？
5. この設計にはどのような利点がありますか？

<details>
<summary>解答例</summary>

```
1. インターフェース
   → <<interface>> と表記されており、メソッドの定義のみを持つ。

2. 実装関係（implements）
   → UserRepositoryはIUserRepositoryインターフェースを実装している。
     IUserRepositoryで定義された全メソッドの具体的な実装を提供する。

3. 依存関係（uses）
   → UserServiceはUserRepositoryを利用している。
     具体的にはIUserRepository型のフィールド(userRepo)として保持し、
     ユーザーデータの取得・保存等を委譲している。

4. private（-記号）
   → 「- db」の「-」はprivateを意味する。
     クラス外部から直接アクセスできない。

5. 利点：
   ・依存性逆転の原則（DIP）に従っている
     → UserServiceはインターフェース(IUserRepository)に依存しており、
       具体的な実装(UserRepository)には依存していない。
   ・テスト容易性が高い
     → テスト時にIUserRepositoryのモック実装を注入できる。
   ・実装の差し替えが容易
     → 例えばDBをPostgreSQLからMongoDBに変更する場合、
       新しいMongoUserRepositoryを作成するだけで対応可能。
```

</details>

### 問題2：シーケンス図の作成

以下の処理について、シーケンス図を作成してください。

**処理：** ユーザーがログインする

**登場オブジェクト：**
- ユーザー（ブラウザ）
- AuthController
- AuthService
- UserRepository
- PasswordHasher

**処理の流れ：**
1. ユーザーがメールアドレスとパスワードを送信
2. Controllerがリクエストを受け取り、Serviceに処理を委譲
3. Serviceがメールアドレスでユーザーを検索
4. パスワードのハッシュ値を比較
5. 認証成功ならJWTトークンを生成して返す
6. 認証失敗なら401エラーを返す

テキストベースでシーケンス図を描いてください。

</details>

### 問題3：状態遷移の設計

以下のエンティティの状態遷移図と状態遷移表を作成してください。

**エンティティ：** タスク（プロジェクト管理ツール）

**ステータス一覧：**
- 未着手（Open）
- 進行中（In Progress）
- レビュー中（In Review）
- 完了（Done）
- 保留（On Hold）

**ルール：**
- 未着手 → 進行中、保留 が可能
- 進行中 → レビュー中、未着手（差し戻し）、保留 が可能
- レビュー中 → 完了、進行中（差し戻し）が可能
- 保留 → 未着手 に戻せる
- 完了からは遷移不可（完了後に再開する場合は新しいタスクを作成）

<details>
<summary>ヒント</summary>

まず状態遷移図を描き、次に状態遷移表（現在の状態 × イベント → 次の状態）で整理すると漏れなく設計できます。

</details>

<details>
<summary>解答例</summary>

```
## 状態遷移図

                    着手
   ┌──────┐  ──────────→  ┌──────┐
   │ 未着手 │               │ 進行中 │
   │(Open) │◄──────────── │(In    │
   └──┬───┘   差し戻し    │Progress)│
      │                    └──┬───┘
      │ 保留                   │
      ▼                       │ レビュー依頼
   ┌──────┐               ┌──▼─────┐
   │ 保留  │               │レビュー中│
   │(On   │               │(In     │
   │ Hold) │               │ Review)│
   └──┬───┘               └──┬───┘
      │ 再開                   │
      │                       ├── 承認 ──→ ┌──────┐
      └──→ 未着手に戻る       │            │ 完了  │
                              │            │(Done) │
                              └── 差し戻し  └──────┘
                                  → 進行中に戻る

## 状態遷移表

| 現在の状態 | イベント | 次の状態 | 処理 |
|-----------|---------|---------|------|
| 未着手 | 着手 | 進行中 | 担当者をアサイン、開始日を記録 |
| 未着手 | 保留 | 保留 | 保留理由を記録 |
| 進行中 | レビュー依頼 | レビュー中 | レビューアに通知 |
| 進行中 | 差し戻し | 未着手 | 差し戻し理由を記録 |
| 進行中 | 保留 | 保留 | 保留理由を記録 |
| レビュー中 | 承認 | 完了 | 完了日を記録、担当者に通知 |
| レビュー中 | 差し戻し | 進行中 | 修正コメントを記録、担当者に通知 |
| 保留 | 再開 | 未着手 | 保留解除日を記録 |
| 完了 | - | -（遷移不可） | 再開が必要な場合は新タスクを作成 |
```

### 問題4：処理フローの設計

以下の処理について、フローチャートを作成してください。

**処理：** クーポン適用処理

**ルール：**
- クーポンコードの存在チェック
- クーポンの有効期限チェック
- クーポンの使用回数上限チェック（1ユーザー1回まで）
- 注文金額が最低利用金額以上であることのチェック
- 全チェックOKなら割引を適用

## チャレンジ問題

### 問題5：詳細設計の総合問題

以下の機能について、詳細設計を行ってください。

**機能：** パスワード変更

**要件：**
- ログイン済みユーザーが現在のパスワードと新しいパスワードを入力
- 現在のパスワードの照合
- 新しいパスワードのポリシーチェック（8文字以上、英数字混在）
- 変更成功後、全デバイスのセッションを無効化

以下の3つを作成してください：
1. クラスの詳細定義（メソッドの入出力・例外・処理概要）
2. シーケンス図
3. エラーハンドリング一覧

<details>
<summary>解答例</summary>

```markdown
## 1. クラスの詳細定義

### AuthService.changePassword

#### 引数
| 引数名 | 型 | 説明 |
|--------|-----|------|
| userId | string | ログイン中のユーザーID |
| currentPassword | string | 現在のパスワード |
| newPassword | string | 新しいパスワード |

#### 戻り値
void（成功時は例外を送出しない）

#### 例外
| 例外 | 条件 |
|------|------|
| UserNotFoundError | ユーザーが存在しない |
| InvalidPasswordError | 現在のパスワードが不正 |
| PasswordPolicyError | 新パスワードがポリシー違反 |
| SamePasswordError | 新旧パスワードが同一 |

#### 処理概要
1. userIdでユーザーを取得する
2. currentPasswordとDBのハッシュ値を照合する
3. newPasswordのポリシーチェックを行う
   - 8文字以上
   - 英字を1文字以上含む
   - 数字を1文字以上含む
4. 現在のパスワードと新パスワードが異なることを確認する
5. newPasswordをbcryptでハッシュ化する
6. パスワードハッシュを更新する
7. 該当ユーザーの全セッション（現在のセッション以外）を無効化する
8. パスワード変更通知メールを送信する
```

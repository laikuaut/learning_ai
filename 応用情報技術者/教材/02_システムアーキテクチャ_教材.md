# 第2章 システムアーキテクチャ

> システムの性能評価・構成・設計に関する知識を体系的に学ぶ。午前・午後共に頻出の重要分野。

---

## 2.1 性能評価

### 2.1.1 性能指標

| 指標 | 説明 | 単位例 |
|------|------|--------|
| スループット | 単位時間あたりの処理量 | 件/秒, トランザクション/秒(TPS) |
| レスポンスタイム | 要求を送信してから応答を受信するまでの時間 | 秒, ミリ秒 |
| ターンアラウンドタイム | ジョブの投入から結果の出力完了までの全時間 | 秒, 分 |
| CPU利用率 | CPUが実際に処理を行っている時間の割合 | % |
| メモリ使用率 | 物理メモリの使用割合 | % |

**レスポンスタイムの構成:**
```
レスポンスタイム = CPU処理時間 + I/O待ち時間 + ネットワーク遅延 + 待ち行列時間
```

### 2.1.2 ベンチマーク

**代表的なベンチマーク:**

| ベンチマーク | 対象 | 内容 |
|-------------|------|------|
| SPEC CPU | CPU性能 | 整数/浮動小数点演算 |
| TPC-C | OLTP性能 | オンライントランザクション処理 |
| TPC-H | 意思決定支援 | 複雑なクエリ処理性能 |
| Linpack | 科学技術計算 | 連立一次方程式の求解速度 |
| SPECjbb | Javaサーバ | ビジネスアプリケーション性能 |

**午前ポイント:** TPC-CはOLTP、TPC-HはDWH(データウェアハウス)向け。

### 2.1.3 アムダールの法則

システムの一部を高速化したときの全体の高速化率を計算する法則。

```
全体の高速化率 = 1 / {(1 - P) + P/S}

P: 高速化できる部分の割合
S: その部分の高速化倍率
```

**例題:**
プログラムの80%を占める処理を4倍に高速化した場合、全体の高速化率は？

```
P = 0.8, S = 4
全体の高速化率 = 1 / {(1 - 0.8) + 0.8/4}
              = 1 / {0.2 + 0.2}
              = 1 / 0.4 = 2.5倍
```

**教訓:** 全体の80%を4倍にしても、全体は2.5倍にしかならない。

## 2.2 キャパシティプランニング

### 2.2.1 目的と手順

1. **現状分析**: 現在のリソース使用状況を把握
2. **需要予測**: 将来のトランザクション量・データ量を予測
3. **性能モデリング**: 待ち行列理論等で必要リソースを算出
4. **容量設計**: ピーク時にも耐えられるリソースを確保
5. **監視・見直し**: 継続的なモニタリングと計画の修正

### 2.2.2 性能計算の基本

**MIPS（Million Instructions Per Second）:**
```
処理時間 = 命令数 / (MIPS × 10⁶)
```

**例題:**
100MIPSのCPUで、1トランザクションあたり50万命令が必要。
1秒あたり最大何トランザクション処理できるか？

```
1トランザクションの処理時間 = 500,000 / (100 × 1,000,000) = 0.005秒
最大TPS = 1 / 0.005 = 200 TPS
```

### 2.2.3 ピーク率の考慮

```
ピーク時TPS = 平均TPS × ピーク率
必要処理能力 = ピーク時TPS / 目標CPU利用率
```

**例題:**
平均100TPS、ピーク率3倍、CPU利用率を70%以下にしたい場合:
```
ピーク時TPS = 100 × 3 = 300 TPS
必要処理能力 = 300 / 0.7 ≈ 429 TPS
```

## 2.3 システム構成

### 2.3.1 クライアントサーバシステム

**2層C/S:**
```
┌──────────┐      ┌──────────┐
│ クライアント │◄────►│  サーバ   │
│(UI＋業務) │      │  (DB)    │
└──────────┘      └──────────┘
```
- クライアント側に業務ロジックが存在
- クライアントの保守・更新が大変（ファットクライアント）

**3層C/S（3層アーキテクチャ）:**
```
┌────────┐    ┌──────────┐    ┌────────┐
│ プレゼン │    │  アプリ   │    │  データ  │
│ テーション│◄──►│ ケーション │◄──►│  ベース  │
│   層    │    │    層    │    │   層   │
└────────┘    └──────────┘    └────────┘
  (ブラウザ)    (APサーバ)      (DBサーバ)
```
- 業務ロジックをAPサーバに集約
- クライアントはWebブラウザ（シンクライアント化）
- スケーラビリティが高い

### 2.3.2 マイクロサービスアーキテクチャ

**従来のモノリシックとの比較:**

```
【モノリシック】              【マイクロサービス】
┌───────────────┐      ┌─────┐ ┌─────┐ ┌─────┐
│               │      │ ユーザ│ │ 注文 │ │ 在庫 │
│   全機能が     │      │ 管理 │ │ 管理 │ │ 管理 │
│   1つのアプリ  │      │サービス│ │サービス│ │サービス│
│               │      └──┬──┘ └──┬──┘ └──┬──┘
│               │         │       │       │
└───────┬───────┘      ┌──┴──┐ ┌──┴──┐ ┌──┴──┐
        │              │ DB  │ │ DB  │ │ DB  │
   ┌────┴────┐         └─────┘ └─────┘ └─────┘
   │  共通DB  │
   └─────────┘
```

**マイクロサービスの特徴:**
- 各サービスが独立してデプロイ可能
- サービス間はAPIで通信（REST, gRPC, メッセージキュー）
- 各サービスが独自のDBを持つことが可能
- 技術スタックを各サービスで選択可能
- 障害の影響範囲を限定できる

**課題:**
- 分散システムの複雑さ（ネットワーク障害、データ整合性）
- サービス間通信のオーバーヘッド
- 運用・監視の複雑さ

### 2.3.3 サーバレスアーキテクチャ

- FaaS（Function as a Service）が中心
- 実行時のみリソースを消費（従量課金）
- スケーリングはプラットフォームが自動管理
- 例: AWS Lambda, Azure Functions, Google Cloud Functions

## 2.4 クラウドコンピューティング

### 2.4.1 サービスモデル

```
┌────────────────────────────────────────┐
│        アプリケーション                   │← SaaS
├────────────────────────────────────────┤
│      ミドルウェア・ランタイム               │← PaaS
├────────────────────────────────────────┤
│        OS・ネットワーク                   │← IaaS
├────────────────────────────────────────┤
│        ハードウェア                       │← すべてプロバイダ管理
└────────────────────────────────────────┘
```

| モデル | ユーザ管理範囲 | 例 |
|--------|---------------|-----|
| IaaS | OS以上全て | AWS EC2, Azure VM |
| PaaS | アプリケーションのみ | Heroku, AWS Elastic Beanstalk |
| SaaS | 設定・データのみ | Gmail, Salesforce, Microsoft 365 |

### 2.4.2 デプロイメントモデル

| モデル | 説明 |
|--------|------|
| パブリッククラウド | 不特定多数向け、共有インフラ |
| プライベートクラウド | 特定組織専用、セキュリティ重視 |
| ハイブリッドクラウド | パブリック＋プライベートの組み合わせ |
| マルチクラウド | 複数のパブリッククラウドを併用 |

### 2.4.3 クラウドの主要サービス

**コンピューティング:** EC2, Azure VM, GCE
**コンテナ:** ECS, EKS, AKS, GKE
**ストレージ:** S3, Azure Blob, GCS
**データベース:** RDS, DynamoDB, Cloud SQL
**ネットワーク:** VPC, CloudFront, Azure CDN

## 2.5 仮想化技術

### 2.5.1 サーバ仮想化

**ハイパーバイザ型:**
```
┌─────┐ ┌─────┐ ┌─────┐
│VM 1 │ │VM 2 │ │VM 3 │
│ OS  │ │ OS  │ │ OS  │
├─────┴─┴─────┴─┴─────┤
│   ハイパーバイザ       │
├───────────────────────┤
│    ハードウェア         │
└───────────────────────┘
```

**Type1（ベアメタル）:** ハードウェア上に直接ハイパーバイザ
  - 例: VMware ESXi, Hyper-V, KVM

**Type2（ホスト型）:** ホストOS上にハイパーバイザ
  - 例: VirtualBox, VMware Workstation

### 2.5.2 コンテナ技術

```
┌─────┐ ┌─────┐ ┌─────┐
│App 1│ │App 2│ │App 3│
│ Lib │ │ Lib │ │ Lib │
├─────┴─┴─────┴─┴─────┤
│    コンテナエンジン     │
│      (Docker等)       │
├───────────────────────┤
│       ホストOS         │
├───────────────────────┤
│     ハードウェア        │
└───────────────────────┘
```

**VMとコンテナの比較:**

| 項目 | 仮想マシン | コンテナ |
|------|-----------|---------|
| 起動速度 | 分単位 | 秒単位 |
| リソース消費 | 大きい（OSごと） | 小さい（カーネル共有） |
| 分離レベル | 高い（完全分離） | 中程度（カーネル共有） |
| ポータビリティ | 高い | 非常に高い |
| 代表技術 | VMware, Hyper-V | Docker, Podman |

### 2.5.3 コンテナオーケストレーション

**Kubernetes (K8s):**
- コンテナの自動配置・スケーリング・管理
- Pod: 最小デプロイ単位（1つ以上のコンテナ）
- Service: Podへのアクセスを抽象化
- Deployment: Podのレプリカ数やアップデート戦略を管理

## 2.6 冗長化構成

### 2.6.1 システム構成パターン

**シンプレックスシステム:**
```
┌──────┐
│サーバ1│  ← 1台のみ（冗長性なし）
└──────┘
```

**デュプレックスシステム:**
```
┌──────┐
│現用系 │  ← 通常稼働
└──┬───┘
   │ 障害時切替
┌──┴───┐
│待機系 │  ← 障害時に稼働
└──────┘
```

待機系の稼働状態による分類:

| 方式 | 待機系の状態 | 切替時間 | コスト |
|------|------------|---------|--------|
| ホットスタンバイ | 起動済み・データ同期 | 短い（秒～分） | 高い |
| ウォームスタンバイ | 起動済み・定期同期 | 中程度 | 中程度 |
| コールドスタンバイ | 停止状態 | 長い（時間単位） | 低い |

**デュアルシステム:**
```
┌──────┐    ┌──────┐
│系統1  │◄──►│系統2  │  ← 両方が同時に同じ処理を実行
└──────┘    └──────┘
     │           │
     ▼           ▼
  結果を照合（クロスチェック）
```
- 両系統で同じ処理を実行し、結果を照合
- 高信頼性が求められるシステム（金融、航空管制）

### 2.6.2 信頼性の計算

**直列システム:**
```
→ [A] → [B] → [C] →
全体の稼働率 = Ra × Rb × Rc
```

**並列システム:**
```
    ┌→ [A] →┐
→ ──┤        ├── →
    └→ [B] →┘
全体の稼働率 = 1 - (1-Ra)(1-Rb)
```

**例題:**
稼働率0.9のサーバを2台並列にした場合の稼働率:
```
1 - (1-0.9)(1-0.9) = 1 - 0.1×0.1 = 1 - 0.01 = 0.99
```

### 2.6.3 RAID

| RAIDレベル | 方式 | 最低台数 | 利用可能容量 | 耐障害性 |
|-----------|------|---------|------------|---------|
| RAID 0 | ストライピング | 2 | N台分 | なし |
| RAID 1 | ミラーリング | 2 | N/2台分 | 1台故障OK |
| RAID 5 | パリティ分散 | 3 | (N-1)台分 | 1台故障OK |
| RAID 6 | 二重パリティ | 4 | (N-2)台分 | 2台故障OK |
| RAID 10 | ミラー＋ストライプ | 4 | N/2台分 | 各ペア1台故障OK |

## 2.7 可用性の指標

### 2.7.1 RASIS

| 指標 | 英語 | 意味 |
|------|------|------|
| 信頼性 | Reliability | 故障しにくさ（MTBF） |
| 可用性 | Availability | 使いたいときに使える度合い |
| 保守性 | Serviceability | 修理のしやすさ（MTTR） |
| 保全性 | Integrity | データの完全性・一貫性 |
| 安全性 | Security | 不正アクセスからの保護 |

### 2.7.2 可用性の計算

```
可用性(稼働率) = MTBF / (MTBF + MTTR)

MTBF: 平均故障間隔（Mean Time Between Failures）
MTTR: 平均修復時間（Mean Time To Repair）
```

**例題:**
MTBFが4,000時間、MTTRが100時間のシステムの稼働率:
```
稼働率 = 4000 / (4000 + 100) = 4000 / 4100 ≈ 0.9756 (97.56%)
```

### 2.7.3 可用性の目標レベル

| レベル | 年間ダウンタイム | 用途例 |
|--------|---------------|--------|
| 99% (2ナイン) | 約3.65日 | 社内システム |
| 99.9% (3ナイン) | 約8.76時間 | 一般的業務システム |
| 99.99% (4ナイン) | 約52.6分 | 金融系、EC |
| 99.999% (5ナイン) | 約5.26分 | 通信インフラ |

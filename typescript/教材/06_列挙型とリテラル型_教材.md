# 第6章: 列挙型（Enum）とリテラル型

## この章のゴール

- `enum` の仕組みと種類（数値enum、文字列enum、const enum）を理解する
- `as const` を使ったモダンな定数定義パターンを習得する
- リテラル型とテンプレートリテラル型を実務レベルで使いこなせるようになる
- enum と `as const` のそれぞれの長所・短所を理解し、場面に応じて使い分けられるようになる

---

## 6.1 列挙型（Enum）の基礎

### enum とは何か

`enum`（イーナム）は、**関連する定数値の集合に名前を付ける**仕組みです。マジックナンバーやマジックストリングをコードから排除し、型安全に定数を管理できます。

```typescript
// JavaScriptでの問題点: マジックナンバー
function getStatusText(status: number): string {
  if (status === 0) return "下書き";     // 0って何？
  if (status === 1) return "公開中";     // 1って何？
  if (status === 2) return "アーカイブ"; // 2って何？
  return "不明";
}

// TypeScriptのenum: 意味のある名前を付けられる
enum ArticleStatus {
  Draft = 0,
  Published = 1,
  Archived = 2,
}

function getStatusText2(status: ArticleStatus): string {
  switch (status) {
    case ArticleStatus.Draft:
      return "下書き";
    case ArticleStatus.Published:
      return "公開中";
    case ArticleStatus.Archived:
      return "アーカイブ";
  }
}
```

### 数値 enum

数値enumは、各メンバーに数値が割り当てられます。初期値を省略すると、0から順にインクリメントされます。

```typescript
// 自動インクリメント（0から始まる）
enum Direction {
  Up,     // 0
  Down,   // 1
  Left,   // 2
  Right,  // 3
}

// 初期値を指定
enum HttpStatus {
  OK = 200,
  Created = 201,
  BadRequest = 400,
  Unauthorized = 401,
  Forbidden = 403,
  NotFound = 404,
  InternalError = 500,
}

// 途中から指定すると、そこからインクリメント
enum Priority {
  Low = 1,
  Medium,  // 2
  High,    // 3
  Urgent,  // 4
}
```

**数値enumの問題点: 逆引き（リバースマッピング）**

数値enumには「逆引き」機能がありますが、これが混乱の原因になることがあります。

```typescript
enum Color {
  Red = 0,
  Green = 1,
  Blue = 2,
}

// 正引き
console.log(Color.Red);   // 0
console.log(Color.Green); // 1

// 逆引き（数値 → 名前）
console.log(Color[0]); // "Red"
console.log(Color[1]); // "Green"

// コンパイル後のJavaScriptを見ると...
// var Color;
// (function (Color) {
//     Color[Color["Red"] = 0] = "Red";
//     Color[Color["Green"] = 1] = "Green";
//     Color[Color["Blue"] = 2] = "Blue";
// })(Color || (Color = {}));
```

> **注意:** 数値enumは型安全性に穴があります。任意の数値を代入できてしまう場合があるためです。文字列enumの使用を推奨します。

### 文字列 enum（推奨）

文字列enumは、各メンバーに文字列リテラルを割り当てます。**デバッグしやすく、型安全性も高い**ため、実務では文字列enumが推奨されます。

```typescript
enum LogLevel {
  Debug = "DEBUG",
  Info = "INFO",
  Warn = "WARN",
  Error = "ERROR",
  Fatal = "FATAL",
}

function log(level: LogLevel, message: string): void {
  console.log(`[${level}] ${message}`);
}

log(LogLevel.Info, "アプリケーション起動");
// [INFO] アプリケーション起動

log(LogLevel.Error, "データベース接続失敗");
// [ERROR] データベース接続失敗

// ❌ 文字列enumは逆引きができない（型安全！）
// console.log(LogLevel["INFO"]); // エラー！

// ❌ 任意の文字列を渡すこともできない
// log("INFO", "test"); // エラー！LogLevel型が必要
```

### const enum

`const enum` は、コンパイル時に完全にインライン化されるenumです。実行時のオーバーヘッドがゼロになります。

```typescript
const enum Fruit {
  Apple = "APPLE",
  Banana = "BANANA",
  Cherry = "CHERRY",
}

const myFruit = Fruit.Apple;
// コンパイル後: const myFruit = "APPLE";
// enumオブジェクト自体は生成されない → バンドルサイズが小さくなる
```

**const enum の注意点:**
- `Object.keys()` や `Object.values()` でメンバーを列挙できない
- `--isolatedModules` オプション有効時（Vite, Next.js等）は使えないことがある
- 外部ライブラリからエクスポートすると問題が起きやすい

## 6.2 リテラル型の詳細

### リテラル型とは

リテラル型は、**特定の値だけを許容する型**です。第4章でも軽く触れましたが、ここではさらに深掘りします。

```typescript
// 文字列リテラル型
type Theme = "light" | "dark" | "system";

let theme: Theme = "light"; // OK
// theme = "blue"; // エラー！

// 数値リテラル型
type Port = 80 | 443 | 8080 | 3000;

let port: Port = 443; // OK
// port = 9999; // エラー！

// booleanリテラル型
type Enabled = true;

const isEnabled: Enabled = true; // OK
// const isDisabled: Enabled = false; // エラー！
```

### const と型推論の関係

`let` と `const` では型推論の結果が異なります。これを理解することは非常に重要です。

```typescript
let mutableStr = "hello";     // string型（変更可能なので広い型に推論）
const immutableStr = "hello"; // "hello"型（リテラル型に推論）

let mutableNum = 42;          // number型
const immutableNum = 42;      // 42型

// なぜこの違いが重要か？
type HttpMethod = "GET" | "POST" | "PUT" | "DELETE";

function request(method: HttpMethod, url: string): void {
  // ...
}

const method1 = "GET";    // "GET"型 → HttpMethodに代入可能
let method2 = "GET";      // string型 → HttpMethodに代入不可能

request(method1, "/api"); // OK
// request(method2, "/api"); // エラー！string型はHttpMethodに代入できない

// 解決策1: 型注釈を付ける
let method3: HttpMethod = "GET";
request(method3, "/api"); // OK

// 解決策2: as const を使う
let method4 = "GET" as const; // "GET"型
request(method4, "/api"); // OK
```

```
┌────────────────────────────────────────────────┐
│          let vs const の型推論                   │
│                                                  │
│  const x = "hello"    →  型は "hello"           │
│  let   x = "hello"    →  型は string            │
│                                                  │
│  const x = 42         →  型は 42                │
│  let   x = 42         →  型は number            │
│                                                  │
│  const x = true       →  型は true              │
│  let   x = true       →  型は boolean           │
│                                                  │
│  理由: letは再代入可能なので、広い型に推論される │
│        constは不変なので、リテラル型に推論される │
└────────────────────────────────────────────────┘
```

## 6.3 as const（const assertion）

### as const とは

`as const` は、値を**最も狭いリテラル型**として推論させるアサーションです。オブジェクトや配列に適用すると、全てのプロパティが `readonly` かつリテラル型になります。

```typescript
// as const なし
const config = {
  host: "localhost",
  port: 3000,
  debug: true,
};
// 型: { host: string; port: number; debug: boolean; }

// as const あり
const config2 = {
  host: "localhost",
  port: 3000,
  debug: true,
} as const;
// 型: { readonly host: "localhost"; readonly port: 3000; readonly debug: true; }

// 配列にas const
const colors = ["red", "green", "blue"] as const;
// 型: readonly ["red", "green", "blue"]（タプルになる）

// 通常の配列
const colors2 = ["red", "green", "blue"];
// 型: string[]
```

### as const で enum の代替を作る

モダンなTypeScriptでは、`as const` を使ってenumの代わりにする方法が広く採用されています。

```typescript
// enumの代替: as const オブジェクト
const LogLevel = {
  Debug: "DEBUG",
  Info: "INFO",
  Warn: "WARN",
  Error: "ERROR",
  Fatal: "FATAL",
} as const;

// メンバーの値の型をユニオン型として抽出
type LogLevel = (typeof LogLevel)[keyof typeof LogLevel];
// "DEBUG" | "INFO" | "WARN" | "ERROR" | "FATAL"

// 使用例
function log(level: LogLevel, message: string): void {
  console.log(`[${level}] ${message}`);
}

log(LogLevel.Info, "起動");   // OK
log("INFO", "起動");          // OK（文字列も受け付ける）
// log("VERBOSE", "起動");    // エラー！
```

### enum vs as const の比較

| 特徴 | `enum` | `as const` オブジェクト |
|------|--------|------------------------|
| 型安全性 | ○ | ○ |
| ランタイムオブジェクト | 生成される | 通常のオブジェクト |
| Tree-shaking | △（使われなくても残る場合あり） | ○（通常のオブジェクトとして最適化される） |
| `Object.keys()` で列挙 | ○（ただし数値enumは注意） | ○ |
| 文字列代入 | ×（enum型が必要） | ○（リテラル文字列も代入可能） |
| `--isolatedModules` | △（const enumは非推奨） | ○（問題なし） |
| IDE補完 | ○ | ○ |

> **現場の使い分け:** 厳密な型の区別が必要な場面（同じ文字列でも別のenumなら区別したい場合）は `enum` が適しています。それ以外の場面では `as const` の方が柔軟で扱いやすいです。

## 6.4 テンプレートリテラル型

TypeScript 4.1で導入されたテンプレートリテラル型は、文字列リテラル型をテンプレートで合成する仕組みです。

### 基本構文

```typescript
// 基本的なテンプレートリテラル型
type Greeting = `Hello, ${string}`;

const g1: Greeting = "Hello, 太郎";    // OK
const g2: Greeting = "Hello, World";   // OK
// const g3: Greeting = "Hi, 太郎";    // エラー！"Hello, "で始まっていない

// ユニオン型と組み合わせ
type Color = "red" | "green" | "blue";
type Size = "small" | "medium" | "large";

type ColorSize = `${Color}-${Size}`;
// "red-small" | "red-medium" | "red-large" |
// "green-small" | "green-medium" | "green-large" |
// "blue-small" | "blue-medium" | "blue-large"

const cs: ColorSize = "red-small";     // OK
// const cs2: ColorSize = "red-tiny";  // エラー！
```

### 実践例: CSSユニット型

```typescript
type CSSUnit = "px" | "em" | "rem" | "%" | "vh" | "vw";
type CSSValue = `${number}${CSSUnit}`;

const width: CSSValue = "100px";    // OK
const height: CSSValue = "50vh";    // OK
const margin: CSSValue = "1.5rem";  // OK
// const bad: CSSValue = "auto";    // エラー！
```

### 実践例: イベントハンドラの型

```typescript
type EventType = "click" | "focus" | "blur" | "change";
type EventHandler = `on${Capitalize<EventType>}`;
// "onClick" | "onFocus" | "onBlur" | "onChange"

// 組み込みの文字列操作型
type A = Uppercase<"hello">;    // "HELLO"
type B = Lowercase<"HELLO">;    // "hello"
type C = Capitalize<"hello">;   // "Hello"
type D = Uncapitalize<"Hello">; // "hello"
```

### 実践例: APIエンドポイントの型

```typescript
type Resource = "users" | "posts" | "comments";
type ApiEndpoint = `/api/v1/${Resource}` | `/api/v1/${Resource}/${number}`;

const endpoint1: ApiEndpoint = "/api/v1/users";      // OK
const endpoint2: ApiEndpoint = "/api/v1/posts/42";    // OK
// const endpoint3: ApiEndpoint = "/api/v2/users";    // エラー！
```

### ポイントまとめ

- テンプレートリテラル型はバッククォート内で型を合成する
- ユニオン型と組み合わせると、全ての組み合わせが自動生成される
- `Uppercase`, `Lowercase`, `Capitalize`, `Uncapitalize` で文字列型を変換できる

## 6.5 判別ユニオン型とリテラル型の連携

リテラル型は、判別ユニオン型（Discriminated Unions）で最も真価を発揮します。

```typescript
// イベントシステムの型定義
type AppEvent =
  | { type: "LOGIN"; payload: { userId: string; timestamp: Date } }
  | { type: "LOGOUT"; payload: { userId: string } }
  | { type: "PAGE_VIEW"; payload: { path: string; referrer?: string } }
  | { type: "PURCHASE"; payload: { productId: string; amount: number } };

function handleEvent(event: AppEvent): void {
  switch (event.type) {
    case "LOGIN":
      console.log(`ログイン: ${event.payload.userId}`);
      // event.payload は { userId: string; timestamp: Date }
      break;
    case "LOGOUT":
      console.log(`ログアウト: ${event.payload.userId}`);
      break;
    case "PAGE_VIEW":
      console.log(`ページ閲覧: ${event.payload.path}`);
      break;
    case "PURCHASE":
      console.log(`購入: ${event.payload.productId} ¥${event.payload.amount}`);
      break;
  }
}
```

### 型安全なアクションクリエーター

```typescript
const ActionTypes = {
  ADD_ITEM: "ADD_ITEM",
  REMOVE_ITEM: "REMOVE_ITEM",
  UPDATE_QUANTITY: "UPDATE_QUANTITY",
  CLEAR_CART: "CLEAR_CART",
} as const;

type ActionType = (typeof ActionTypes)[keyof typeof ActionTypes];

// アクションクリエーター
function addItem(item: { id: string; name: string; price: number }) {
  return { type: ActionTypes.ADD_ITEM, payload: item } as const;
}

function removeItem(id: string) {
  return { type: ActionTypes.REMOVE_ITEM, payload: { id } } as const;
}

function updateQuantity(id: string, quantity: number) {
  return { type: ActionTypes.UPDATE_QUANTITY, payload: { id, quantity } } as const;
}

function clearCart() {
  return { type: ActionTypes.CLEAR_CART } as const;
}

// アクションの型を自動導出
type CartAction =
  | ReturnType<typeof addItem>
  | ReturnType<typeof removeItem>
  | ReturnType<typeof updateQuantity>
  | ReturnType<typeof clearCart>;
```

## 6.6 よくある間違い

### 1. 数値enumに任意の数値を代入してしまう

```typescript
enum Status {
  Active = 1,
  Inactive = 2,
}

// ❌ 数値enumは型チェックが甘い
const s: Status = 999; // エラーにならない！（TypeScriptの既知の問題）

// ✅ 文字列enumならこの問題は起きない
enum StatusStr {
  Active = "ACTIVE",
  Inactive = "INACTIVE",
}

// const s2: StatusStr = "UNKNOWN"; // エラー！
```

### 2. as const を忘れる

```typescript
const routes = {
  home: "/",
  about: "/about",
  contact: "/contact",
};

// ❌ routes.home は string型 — 型安全でない
type Route = (typeof routes)[keyof typeof routes]; // string

const routes2 = {
  home: "/",
  about: "/about",
  contact: "/contact",
} as const;

// ✅ routes2.home は "/"型 — 型安全
type Route2 = (typeof routes2)[keyof typeof routes2]; // "/" | "/about" | "/contact"
```

### 3. enum を比較演算子で間違って使う

```typescript
enum Color {
  Red = "RED",
  Blue = "BLUE",
}

const userColor = Color.Red;

// ❌ 文字列と直接比較（動くが推奨されない）
if (userColor === "RED") { /* ... */ }

// ✅ enumメンバーと比較
if (userColor === Color.Red) { /* ... */ }
```

## 6.7 現場でのベストプラクティス

### 1. 新規プロジェクトでは as const を優先する

```typescript
// ✅ モダンな定数定義
export const API_METHODS = {
  GET: "GET",
  POST: "POST",
  PUT: "PUT",
  DELETE: "DELETE",
} as const;

export type ApiMethod = (typeof API_METHODS)[keyof typeof API_METHODS];
```

### 2. 定数配列からユニオン型を導出する

```typescript
// 配列として定数を定義
const SUPPORTED_LANGUAGES = ["ja", "en", "zh", "ko"] as const;

// 配列の要素の型をユニオン型として取得
type SupportedLanguage = (typeof SUPPORTED_LANGUAGES)[number];
// "ja" | "en" | "zh" | "ko"

// 配列自体もランタイムで使える
function isSupported(lang: string): lang is SupportedLanguage {
  return (SUPPORTED_LANGUAGES as readonly string[]).includes(lang);
}
```

### 3. マジックナンバー・マジックストリングを排除する

```typescript
// ❌ マジックナンバー
if (response.status === 200) { /* ... */ }
if (user.role === "admin") { /* ... */ }

// ✅ 名前付き定数
const HTTP_STATUS = {
  OK: 200,
  CREATED: 201,
  BAD_REQUEST: 400,
  NOT_FOUND: 404,
  INTERNAL_ERROR: 500,
} as const;

const USER_ROLES = {
  ADMIN: "admin",
  EDITOR: "editor",
  VIEWER: "viewer",
} as const;

if (response.status === HTTP_STATUS.OK) { /* ... */ }
if (user.role === USER_ROLES.ADMIN) { /* ... */ }
```

## 6.8 まとめ

| トピック | 内容 |
|---------|------|
| 数値enum | メンバーに数値を割り当て。逆引き可能だが型安全性に穴あり |
| 文字列enum | メンバーに文字列を割り当て。型安全で推奨 |
| const enum | コンパイル時にインライン化。バンドルサイズ削減 |
| as const | オブジェクト/配列をリテラル型+readonlyに変換 |
| テンプレートリテラル型 | 文字列リテラル型をテンプレートで合成 |
| enum vs as const | 新規プロジェクトでは as const が主流 |

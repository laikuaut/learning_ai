# 第7章: ミドルウェアと認証

## この章のゴール

- Next.js のミドルウェア（`middleware.ts`）の仕組みと用途を理解する
- 認証（Authentication）と認可（Authorization）の違いを明確に区別できるようになる
- NextAuth.js（Auth.js）を使った認証フローを実装できるようになる
- セッション管理、JWT、Cookie ベースの認証パターンを実務レベルで設計できるようになる

---

## 7.1 なぜミドルウェアが必要なのか

### ミドルウェアとは

ミドルウェアは、**リクエストが完了する前に実行されるコード**です。受信したリクエストに基づいて、レスポンスの書き換え、リダイレクト、ヘッダーの修正、あるいはリクエストの中断などを行えます。

```
クライアント → [ミドルウェア] → ルート (page.tsx / route.ts)
                   ↓
            ・認証チェック
            ・リダイレクト
            ・ヘッダー追加
            ・リクエストの書き換え
            ・レスポンスの変更
```

### 従来の方法との比較

ミドルウェアがなかった時代は、各ページやAPIで個別に認証チェックを行う必要がありました。

```tsx
// 従来の方法（各ページで個別に認証チェック）
export default async function DashboardPage() {
  const session = await getSession()
  if (!session) {
    redirect('/login') // 毎回このコードを書く必要がある
  }
  // ...
}

// ミドルウェアを使えば一箇所で管理できる
// middleware.ts
export function middleware(request: NextRequest) {
  const session = request.cookies.get('session')
  if (!session && request.nextUrl.pathname.startsWith('/dashboard')) {
    return NextResponse.redirect(new URL('/login', request.url))
  }
}
```

## 7.2 middleware.ts の基本

### ファイルの配置

`middleware.ts` は**プロジェクトのルートディレクトリ**（`app` と同じ階層）に配置します。

```
my-app/
├── middleware.ts      ← ここに配置
├── app/
│   ├── layout.tsx
│   ├── page.tsx
│   └── dashboard/
│       └── page.tsx
├── public/
└── next.config.ts
```

> **注意**: `middleware.ts` はプロジェクトに**1つだけ**配置できます。`app/middleware.ts` ではなく、ルート直下に配置してください。

### 基本構文

```tsx
// middleware.ts
import { NextRequest, NextResponse } from 'next/server'

export function middleware(request: NextRequest) {
  // リクエストの情報を取得
  const pathname = request.nextUrl.pathname
  const method = request.method
  const headers = request.headers
  const cookies = request.cookies

  // そのまま次の処理に進める
  return NextResponse.next()
}

// ミドルウェアを適用するパスを指定
export const config = {
  matcher: [
    // 静的ファイルと _next を除外するパターン
    '/((?!api|_next/static|_next/image|favicon.ico).*)',
  ],
}
```

### matcher（パスのフィルタリング）

```tsx
export const config = {
  matcher: [
    // 特定のパスのみに適用
    '/dashboard/:path*',
    '/admin/:path*',
    '/api/protected/:path*',
  ],
}

// または、条件式でフィルタリング
export const config = {
  matcher: [
    // 静的ファイルと Next.js 内部パスを除外
    {
      source: '/((?!_next/static|_next/image|favicon.ico).*)',
      missing: [
        { type: 'header', key: 'next-router-prefetch' },
        { type: 'header', key: 'purpose', value: 'prefetch' },
      ],
    },
  ],
}
```

## 7.3 ミドルウェアの主要パターン

### パターン1: 認証チェックとリダイレクト

```tsx
// middleware.ts
import { NextRequest, NextResponse } from 'next/server'

// 認証が必要なパス
const protectedPaths = ['/dashboard', '/settings', '/profile']

// 認証済みユーザーがアクセスすべきでないパス
const authPaths = ['/login', '/register']

export function middleware(request: NextRequest) {
  const { pathname } = request.nextUrl
  const sessionToken = request.cookies.get('session-token')?.value

  // 保護されたパスに未認証でアクセス → ログインにリダイレクト
  const isProtected = protectedPaths.some(path =>
    pathname.startsWith(path)
  )
  if (isProtected && !sessionToken) {
    const loginUrl = new URL('/login', request.url)
    loginUrl.searchParams.set('callbackUrl', pathname)
    return NextResponse.redirect(loginUrl)
  }

  // ログイン済みで認証ページにアクセス → ダッシュボードにリダイレクト
  const isAuthPage = authPaths.some(path =>
    pathname.startsWith(path)
  )
  if (isAuthPage && sessionToken) {
    return NextResponse.redirect(new URL('/dashboard', request.url))
  }

  return NextResponse.next()
}

export const config = {
  matcher: ['/dashboard/:path*', '/settings/:path*', '/profile/:path*', '/login', '/register'],
}
```

### パターン2: リクエストヘッダーの追加

```tsx
// middleware.ts
import { NextRequest, NextResponse } from 'next/server'

export function middleware(request: NextRequest) {
  // リクエストヘッダーにカスタム情報を追加
  const requestHeaders = new Headers(request.headers)
  requestHeaders.set('x-pathname', request.nextUrl.pathname)
  requestHeaders.set('x-request-id', crypto.randomUUID())

  const response = NextResponse.next({
    request: {
      headers: requestHeaders,
    },
  })

  // レスポンスヘッダーの設定
  response.headers.set('x-middleware-cache', 'no-cache')

  return response
}
```

### パターン3: リライト（URL の書き換え）

```tsx
// middleware.ts
import { NextRequest, NextResponse } from 'next/server'

export function middleware(request: NextRequest) {
  const hostname = request.headers.get('host') || ''

  // サブドメインに基づくルーティング
  // blog.example.com → /blog ルートに書き換え
  if (hostname.startsWith('blog.')) {
    return NextResponse.rewrite(
      new URL(`/blog${request.nextUrl.pathname}`, request.url)
    )
  }

  // api.example.com → /api ルートに書き換え
  if (hostname.startsWith('api.')) {
    return NextResponse.rewrite(
      new URL(`/api${request.nextUrl.pathname}`, request.url)
    )
  }

  return NextResponse.next()
}
```

### パターン4: レート制限（Rate Limiting）

```tsx
// middleware.ts
import { NextRequest, NextResponse } from 'next/server'

// 簡易的なインメモリレート制限（本番環境では Redis 等を使用）
const rateLimitMap = new Map<string, { count: number; lastReset: number }>()
const RATE_LIMIT = 100        // リクエスト数上限
const WINDOW_MS = 60 * 1000   // 1分間

export function middleware(request: NextRequest) {
  if (!request.nextUrl.pathname.startsWith('/api')) {
    return NextResponse.next()
  }

  const ip = request.headers.get('x-forwarded-for') || 'unknown'
  const now = Date.now()
  const record = rateLimitMap.get(ip)

  if (!record || now - record.lastReset > WINDOW_MS) {
    rateLimitMap.set(ip, { count: 1, lastReset: now })
    return NextResponse.next()
  }

  if (record.count >= RATE_LIMIT) {
    return NextResponse.json(
      { error: 'レート制限を超えました。しばらくしてからお試しください。' },
      { status: 429 }
    )
  }

  record.count++
  return NextResponse.next()
}
```

### パターン5: 国際化（i18n）リダイレクト

```tsx
// middleware.ts
import { NextRequest, NextResponse } from 'next/server'

const locales = ['ja', 'en', 'ko']
const defaultLocale = 'ja'

function getLocale(request: NextRequest): string {
  const acceptLanguage = request.headers.get('accept-language') || ''

  // Accept-Language ヘッダーから最適なロケールを判定
  for (const locale of locales) {
    if (acceptLanguage.includes(locale)) {
      return locale
    }
  }

  return defaultLocale
}

export function middleware(request: NextRequest) {
  const { pathname } = request.nextUrl

  // パスにロケールが含まれているか確認
  const hasLocale = locales.some(
    locale => pathname.startsWith(`/${locale}/`) || pathname === `/${locale}`
  )

  if (hasLocale) return NextResponse.next()

  // ロケールが含まれていない場合、リダイレクト
  const locale = getLocale(request)
  return NextResponse.redirect(
    new URL(`/${locale}${pathname}`, request.url)
  )
}

export const config = {
  matcher: ['/((?!_next|api|favicon.ico).*)'],
}
```

## 7.4 認証の基本概念

### 認証（Authentication）と認可（Authorization）

| 概念 | 英語 | 質問 | 例 |
|------|------|------|-----|
| 認証 | Authentication | 「あなたは誰ですか？」 | ログイン、パスワード確認 |
| 認可 | Authorization | 「あなたは何ができますか？」 | 管理者のみ、有料会員のみ |

```
ユーザーのリクエスト
  ↓
[認証] ログインしているか？
  → NO → ログインページへリダイレクト
  → YES
    ↓
[認可] この操作の権限があるか？
  → NO → 403 Forbidden
  → YES → リクエストを処理
```

### 認証方式の比較

| 方式 | 仕組み | メリット | デメリット |
|------|--------|---------|-----------|
| セッション + Cookie | サーバーにセッションを保存 | サーバー側で無効化可能 | サーバーの状態管理が必要 |
| JWT (JSON Web Token) | トークンに情報を含める | ステートレス、スケーラブル | トークンの無効化が困難 |
| OAuth 2.0 | 外部プロバイダに認証を委託 | パスワード管理不要 | 実装が複雑 |

## 7.5 NextAuth.js（Auth.js）による認証

### NextAuth.js とは

NextAuth.js（v5 以降は Auth.js）は、Next.js に特化した認証ライブラリです。OAuth プロバイダ（Google、GitHub 等）やメール認証、資格情報認証を簡単に実装できます。

### セットアップ

```bash
npm install next-auth@beta
```

### Auth.js の設定

```tsx
// auth.ts（プロジェクトルート）
import NextAuth from 'next-auth'
import GitHub from 'next-auth/providers/github'
import Google from 'next-auth/providers/google'
import Credentials from 'next-auth/providers/credentials'

export const { handlers, auth, signIn, signOut } = NextAuth({
  providers: [
    // GitHub OAuth
    GitHub({
      clientId: process.env.AUTH_GITHUB_ID,
      clientSecret: process.env.AUTH_GITHUB_SECRET,
    }),

    // Google OAuth
    Google({
      clientId: process.env.AUTH_GOOGLE_ID,
      clientSecret: process.env.AUTH_GOOGLE_SECRET,
    }),

    // メール + パスワード認証
    Credentials({
      name: 'credentials',
      credentials: {
        email: { label: 'メールアドレス', type: 'email' },
        password: { label: 'パスワード', type: 'password' },
      },
      async authorize(credentials) {
        // データベースでユーザーを検証
        const user = await findUserByEmail(credentials.email as string)

        if (!user) return null

        const isValid = await verifyPassword(
          credentials.password as string,
          user.hashedPassword
        )

        if (!isValid) return null

        return {
          id: user.id,
          name: user.name,
          email: user.email,
        }
      },
    }),
  ],

  // セッション戦略
  session: {
    strategy: 'jwt', // 'jwt' or 'database'
  },

  // カスタムページ
  pages: {
    signIn: '/login',
    error: '/auth/error',
  },

  // コールバック
  callbacks: {
    // JWT に追加情報を含める
    async jwt({ token, user }) {
      if (user) {
        token.role = user.role
      }
      return token
    },

    // セッションに追加情報を含める
    async session({ session, token }) {
      if (session.user) {
        session.user.id = token.sub as string
        session.user.role = token.role as string
      }
      return session
    },

    // 認可チェック
    async authorized({ auth, request: { nextUrl } }) {
      const isLoggedIn = !!auth?.user
      const isProtected = nextUrl.pathname.startsWith('/dashboard')

      if (isProtected && !isLoggedIn) {
        return false // ログインページにリダイレクト
      }

      return true
    },
  },
})
```

## 7.7 認可（Authorization）の実装

### ロールベースのアクセス制御（RBAC）

```tsx
// lib/auth-utils.ts
import { auth } from '@/auth'

type Role = 'user' | 'editor' | 'admin'

const roleHierarchy: Record<Role, number> = {
  user: 1,
  editor: 2,
  admin: 3,
}

export async function checkRole(requiredRole: Role) {
  const session = await auth()

  if (!session?.user) {
    return { authorized: false, reason: '未認証' }
  }

  const userRole = (session.user.role || 'user') as Role
  const hasPermission = roleHierarchy[userRole] >= roleHierarchy[requiredRole]

  return {
    authorized: hasPermission,
    reason: hasPermission ? null : '権限が不足しています',
    user: session.user,
  }
}
```

```tsx
// app/admin/page.tsx
import { checkRole } from '@/lib/auth-utils'
import { redirect } from 'next/navigation'

export default async function AdminPage() {
  const { authorized, user } = await checkRole('admin')

  if (!authorized) {
    redirect('/unauthorized')
  }

  return (
    <div>
      <h1>管理者ページ</h1>
      <p>管理者: {user?.name}</p>
    </div>
  )
}
```

### Server Actions での認可チェック

```tsx
// app/actions/admin.ts
```

## 7.8 セッション管理のパターン

### JWT ベースのセッション

```
┌──────────┐     ログイン      ┌──────────┐
│ クライアント │ ──────────→   │ サーバー   │
│          │              │          │
│          │ ← JWT 返却    │          │
│          │              └──────────┘
│          │
│ Cookie に │    以降のリクエスト
│ JWT 保存  │ ─────────────→ JWT を検証
└──────────┘
```

- **メリット**: サーバー側でセッションストアが不要（ステートレス）
- **デメリット**: トークンの即時無効化が困難

### データベースセッション

```
┌──────────┐    ログイン     ┌──────────┐    ┌────────┐
│ クライアント │ ──────────→ │ サーバー   │ ──→ │ DB     │
│          │             │          │    │ セッション│
│          │ ← セッション ID │          │    │ テーブル │
│          │             └──────────┘    └────────┘
│ Cookie に │   以降のリクエスト
│ ID 保存   │ ─────────────→ DB でセッション検証
└──────────┘
```

- **メリット**: サーバー側で即座にセッションを無効化できる
- **デメリット**: リクエストごとに DB アクセスが発生

### NextAuth.js でのセッション設定

```tsx
// auth.ts
export const { handlers, auth, signIn, signOut } = NextAuth({
  // JWT ベース（デフォルト）
  session: {
    strategy: 'jwt',
    maxAge: 30 * 24 * 60 * 60, // 30日
  },

  // または データベースベース
  // session: {
  //   strategy: 'database',
  //   maxAge: 30 * 24 * 60 * 60,
  // },
  // adapter: PrismaAdapter(prisma),
})
```

## 7.9 セキュリティのベストプラクティス

### 1. CSRF 対策

Server Actions は自動的に CSRF トークンを使用するため、追加の対策は不要です。Route Handlers を使う場合は、手動で対策が必要になることがあります。

### 2. Cookie のセキュリティ設定

```tsx
// セキュアな Cookie の設定例
const cookieOptions = {
  httpOnly: true,     // JavaScript からアクセス不可
  secure: true,       // HTTPS のみ
  sameSite: 'lax',    // CSRF 対策
  path: '/',
  maxAge: 60 * 60 * 24 * 7, // 7日
}
```

### 3. 環境変数の管理

```bash
# .env.local
AUTH_SECRET=your-secret-key-here  # 必須: 256ビット以上のランダム文字列
AUTH_GITHUB_ID=your-github-client-id
AUTH_GITHUB_SECRET=your-github-client-secret

# 生成方法:
# openssl rand -base64 33
```

> **重要**: `AUTH_SECRET` は本番環境で必ずランダムな値を設定してください。推測可能な値を使うと、セッションが偽造されるリスクがあります。

### 4. パスワードのハッシュ化

```tsx
// lib/password.ts
import bcrypt from 'bcryptjs'

const SALT_ROUNDS = 12

export async function hashPassword(password: string): Promise<string> {
  return bcrypt.hash(password, SALT_ROUNDS)
}

export async function verifyPassword(
  password: string,
  hashedPassword: string
): Promise<boolean> {
  return bcrypt.compare(password, hashedPassword)
}
```

### 5. 入力値のサニタイズ

```tsx
// Server Actions での入力値チェック
'use server'
import { z } from 'zod'

const loginSchema = z.object({
  email: z.string().email('有効なメールアドレスを入力してください'),
  password: z
    .string()
    .min(8, 'パスワードは8文字以上必要です')
    .max(100, 'パスワードが長すぎます'),
})

export async function login(formData: FormData) {
  const result = loginSchema.safeParse({
    email: formData.get('email'),
    password: formData.get('password'),
  })

  if (!result.success) {
    return { error: result.error.flatten().fieldErrors }
  }

  // 認証処理...
}
```

## 7.10 よくある間違い

### 間違い1: ミドルウェアで重い処理を行う

```tsx
// 悪い例: ミドルウェアでデータベースにアクセスする
export async function middleware(request: NextRequest) {
  // ミドルウェアは Edge Runtime で動作するため、
  // 多くの Node.js API やデータベースドライバが使えない
  const user = await prisma.user.findUnique({ /* ... */ }) // エラー！
}

// 良い例: JWT の検証のみ行う
export function middleware(request: NextRequest) {
  const token = request.cookies.get('session-token')?.value
  // トークンの存在チェックや簡易的なデコードのみ
  if (!token) {
    return NextResponse.redirect(new URL('/login', request.url))
  }
  return NextResponse.next()
}
```

### 間違い2: クライアント側だけで認証チェックする

```tsx
// 悪い例: クライアントコンポーネントでのみチェック
'use client'
export default function Dashboard() {
  const { data: session } = useSession()
  if (!session) return <Redirect to="/login" />
  // → サーバーサイドのデータは保護されない
}

// 良い例: サーバー側で認証チェック
export default async function Dashboard() {
  const session = await auth()
  if (!session) redirect('/login')
  // → サーバーでチェックするので確実
}
```

### 間違い3: ミドルウェアの matcher を設定し忘れる

```tsx
// 悪い例: matcher がない → 全リクエストに適用される
// → 静的ファイル（画像、CSS）のリクエストにも適用されパフォーマンスが低下
export function middleware(request: NextRequest) {
  // ...
}

// 良い例: 必要なパスのみに matcher を設定
export const config = {
  matcher: ['/dashboard/:path*', '/api/protected/:path*'],
}
```

## 7.11 現場でのベストプラクティス

### 1. 認証レイヤーの設計

```
┌─────────────────────────────────────────────┐
│ middleware.ts                                 │
│ → セッショントークンの存在チェック              │
│ → 未認証の場合はリダイレクト                    │
│ （軽量な処理のみ）                             │
├─────────────────────────────────────────────┤
│ layout.tsx / page.tsx                        │
│ → auth() でセッション情報の詳細取得            │
│ → ロールベースの認可チェック                    │
│ （データベースアクセスを含む処理）               │
├─────────────────────────────────────────────┤
│ Server Actions / Route Handlers              │
│ → データ変更時の認証・認可チェック              │
│ → 入力値のバリデーション                       │
└─────────────────────────────────────────────┘
```

### 2. 環境変数の設計

```bash
# .env.local（ローカル開発用）
AUTH_SECRET=dev-secret-key
AUTH_URL=http://localhost:3000
AUTH_GITHUB_ID=dev-github-id
AUTH_GITHUB_SECRET=dev-github-secret

# .env.production（本番用 — Vercel の環境変数で設定）
AUTH_SECRET=production-random-secret
AUTH_URL=https://myapp.example.com
```

### 3. 認証フローの全体像

```
1. ユーザーがログインページにアクセス
2. OAuth プロバイダまたは資格情報で認証
3. 認証成功 → JWT/セッション を Cookie に保存
4. 以降のリクエスト:
   ├── middleware.ts: トークンの存在チェック
   ├── layout.tsx: セッション情報の取得・表示
   ├── page.tsx: 権限に応じたコンテンツ表示
   └── Server Actions: 操作ごとの認可チェック
5. ログアウト → Cookie を削除
```

## 7.12 ポイントまとめ

本章で学んだこと:

1. **middleware.ts** はプロジェクトルートに1つだけ配置し、リクエストの前処理を行います
2. `matcher` で適用するパスを制限し、**静的ファイルへの不要な処理を防ぐ**ことが重要です
3. **認証（Authentication）** は「誰か」を確認し、**認可（Authorization）** は「何ができるか」を制御します
4. **NextAuth.js（Auth.js）** を使うと、OAuth やクレデンシャル認証を簡単に実装できます
5. ミドルウェアでは**軽量な処理のみ**を行い、詳細な認可チェックはサーバーコンポーネントや Server Actions で行います
6. セキュリティの基本（CSRF 対策、Cookie 設定、パスワードハッシュ化、入力値バリデーション）を常に意識しましょう
7. JWT セッションとデータベースセッションの**特性を理解し、要件に応じて使い分ける**ことが大切です

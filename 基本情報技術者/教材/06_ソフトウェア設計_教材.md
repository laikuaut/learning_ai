# 第6章：ソフトウェア設計

## この章のゴール
- UML（統一モデリング言語）の主要な図を読み書きできる
- ソフトウェア開発プロセス（ウォーターフォール、アジャイル等）の違いを理解する
- オブジェクト指向の基本概念（カプセル化・継承・多態性）を説明できる
- テスト手法（ホワイトボックス・ブラックボックス等）を使い分けられる
- レビュー手法の種類と特徴を把握する

## 試験での出題傾向
- 午前試験：UMLの各図の名称と特徴、開発モデルの比較、テスト技法の名称が頻出です。
- 午後試験：クラス図やシーケンス図の読み取り、テストケースの設計が出題されます。

## 6.1 ソフトウェア開発プロセス

### 6.1.1 ウォーターフォールモデル（Waterfall Model）

各工程を順番に進める、最も古典的な開発モデルです。前の工程に戻ることを原則として行いません。

```
要件定義 → 外部設計 → 内部設計 → プログラミング → テスト → 運用・保守
```

| メリット | デメリット |
|---------|-----------|
| 工程ごとに成果物が明確 | 前工程への手戻りコストが高い |
| 進捗管理がしやすい | 要件変更に弱い |
| 大規模プロジェクトに適する | 完成まで動くものが見えない |

### 6.1.2 プロトタイピングモデル（Prototyping Model）

開発の早期段階で試作品（プロトタイプ）を作成し、ユーザーの確認を得ながら進めるモデルです。

- 要件が不明確な場合に有効
- ユーザーの要望を早期に反映できる

### 6.1.3 スパイラルモデル（Spiral Model）

設計・実装・テストを小さな単位で繰り返しながら、段階的にシステムを成長させるモデルです。各繰り返しでリスク分析を行います。

### 6.1.4 アジャイル開発（Agile Development）

短い反復期間（イテレーション、通常1〜4週間）を繰り返し、動くソフトウェアを少しずつ作り上げていく手法の総称です。

**代表的なアジャイル手法：**

| 手法 | 特徴 |
|------|------|
| スクラム（Scrum） | スプリント（2〜4週間の反復）で開発。デイリースクラム（毎日の短い会議）を行う |
| XP（Extreme Programming） | ペアプログラミング、テスト駆動開発（TDD）、リファクタリングを重視 |
| カンバン（Kanban） | タスクを可視化し、WIP（仕掛中の作業）を制限する |

**スクラムの用語：**
| 用語 | 意味 |
|------|------|
| プロダクトオーナー | 製品の方向性を決める人 |
| スクラムマスター | チームの障害を取り除く人 |
| スプリント | 1〜4週間の開発サイクル |
| プロダクトバックログ | 開発すべき機能の優先順位付きリスト |
| デイリースクラム | 毎日15分程度の進捗確認ミーティング |

### 6.1.5 DevOps

開発（Development）と運用（Operations）を統合し、ソフトウェアの構築・テスト・リリースを迅速に行う文化・手法です。CI/CD（継続的インテグレーション/継続的デリバリー）が核となります。

### 開発モデルの比較

| モデル | 要件変更への対応 | 向いている案件 |
|--------|----------------|---------------|
| ウォーターフォール | 困難 | 要件が明確な大規模案件 |
| プロトタイピング | やや容易 | 要件が不明確な案件 |
| スパイラル | 容易 | リスクが高い案件 |
| アジャイル | 非常に容易 | 変化が多い案件 |

## 6.2 オブジェクト指向

### 基本概念

| 概念 | 英語 | 説明 |
|------|------|------|
| クラス | Class | オブジェクトの設計図（ひな型） |
| オブジェクト（インスタンス） | Object / Instance | クラスから生成された実体 |
| 属性 | Attribute | オブジェクトが持つデータ |
| メソッド（操作） | Method / Operation | オブジェクトが持つ処理 |

### オブジェクト指向の三大要素

**1. カプセル化（Encapsulation）**

データ（属性）と処理（メソッド）を一体化し、外部から直接データにアクセスできないようにすることです。情報隠蔽（Information Hiding）ともいいます。

```
┌───────────────┐
│   クラス: 社員  │
│───────────────│
│ - 氏名         │  ← private（外部からアクセス不可）
│ - 給与         │
│───────────────│
│ + 氏名を取得()  │  ← public（外部からアクセス可能）
│ + 給与を設定()  │
└───────────────┘
```

**2. 継承（Inheritance）**

既存のクラス（親クラス、スーパークラス）の属性やメソッドを、新しいクラス（子クラス、サブクラス）が引き継ぐことです。

```
     ┌──────────┐
     │  動物     │  ← スーパークラス（親クラス）
     │──────────│
     │ 名前     │
     │ 鳴く()   │
     └────┬─────┘
       ┌──┴──┐
  ┌────┴───┐ ┌─┴──────┐
  │  犬     │ │  猫     │  ← サブクラス（子クラス）
  │────────│ │────────│
  │ お手()  │ │ 丸くなる()│
  └────────┘ └────────┘
```

**3. 多態性（ポリモーフィズム、Polymorphism）**

同じメッセージ（メソッド呼び出し）に対して、オブジェクトの型によって異なる処理を行うことです。

```
動物.鳴く() を呼び出すと：
- 犬オブジェクト → 「ワン」
- 猫オブジェクト → 「ニャー」
```

### その他の重要概念

| 概念 | 説明 |
|------|------|
| 抽象クラス | インスタンス化できないクラス。共通の枠組みを定義する |
| インタフェース | メソッドの仕様（シグネチャ）だけを定義する |
| オーバーライド | 親クラスのメソッドを子クラスで再定義する |
| オーバーロード | 同名で引数の異なるメソッドを複数定義する |
| コンポジション | 「全体と部品」の関係（強い所有。部品は全体なしに存在しない） |
| 集約 | 「全体と部品」の関係（弱い所有。部品は単独で存在可能） |

## 6.3 UML（Unified Modeling Language / 統一モデリング言語）

### UMLとは

オブジェクト指向の分析・設計で使う図の標準的な記法です。OMG（Object Management Group）が策定しています。

### 主要なUML図

| 分類 | 図の名称 | 目的 |
|------|---------|------|
| 構造図 | クラス図（Class Diagram） | クラスの構造と関係を表す |
| 構造図 | オブジェクト図（Object Diagram） | 具体的なインスタンスの状態を表す |
| 振る舞い図 | ユースケース図（Use Case Diagram） | システムの機能と利用者の関係を表す |
| 振る舞い図 | シーケンス図（Sequence Diagram） | オブジェクト間のメッセージの時間的順序を表す |
| 振る舞い図 | アクティビティ図（Activity Diagram） | 処理の流れ（フローチャート的） |
| 振る舞い図 | 状態遷移図（State Machine Diagram） | オブジェクトの状態変化を表す |
| 振る舞い図 | コミュニケーション図（Communication Diagram） | オブジェクト間のメッセージのやりとりを表す |

### クラス図の読み方

```
┌─────────────────┐
│     <<abstract>> │
│       商品        │
│─────────────────│
│ # 商品ID: String │  ← 属性
│ # 商品名: String │
│ # 価格: int      │
│─────────────────│
│ + 価格を取得(): int│  ← メソッド
│ + 表示(): void    │
└─────────────────┘

可視性の記号：
+ public（公開）
- private（非公開）
# protected（サブクラスのみ公開）
~ package（パッケージ内のみ公開）
```

**クラス間の関係：**

| 関係 | 記号 | 意味 |
|------|------|------|
| 関連（Association） | ─── | クラス間のつながり |
| 集約（Aggregation） | ◇── | 「全体と部品」（弱い所有） |
| コンポジション（Composition） | ◆── | 「全体と部品」（強い所有） |
| 汎化（Generalization） | ──▷ | 継承（is-a関係） |
| 実現（Realization） | --▷ | インタフェースの実装 |
| 依存（Dependency） | --→ | 一時的な利用関係 |

### ユースケース図

```
┌──────────────────────────────────┐
│          図書館システム              │
│                                    │
│    ┌────────────┐                 │
│    │  本を借りる  │←── 利用者（棒人間）│
│    └────────────┘                 │
│    ┌────────────┐                 │
│    │  本を返す   │←── 利用者       │
│    └────────────┘                 │
│    ┌────────────┐                 │
│    │  蔵書管理   │←── 図書館員     │
│    └────────────┘                 │
└──────────────────────────────────┘
```

- **アクター（Actor）：** システムの外部の利用者（棒人間で表す）
- **ユースケース（Use Case）：** システムが提供する機能（楕円で表す）

### シーケンス図

オブジェクト間のメッセージのやりとりを時間軸に沿って表します。

```
 利用者      :画面      :業務ロジック   :DB
  │          │           │            │
  │─ログイン→│           │            │
  │          │─認証要求──→│            │
  │          │           │─問合せ───→│
  │          │           │←結果──────│
  │          │←認証結果──│            │
  │←画面表示─│           │            │
```

## 6.4 設計手法

### 6.4.1 構造化設計

プログラムを「機能」に分割し、階層的に構成する手法です。

- **モジュール分割の原則：**
  - 高凝集（High Cohesion）：1つのモジュールは1つの機能に集中する
  - 低結合（Low Coupling）：モジュール間の依存を少なくする

**モジュール結合度（弱い順）：**

| 結合度 | 名称 | 説明 |
|--------|------|------|
| 弱い | データ結合 | 必要なデータのみ受け渡す |
| ↓ | スタンプ結合 | 構造体全体を受け渡す |
| ↓ | 制御結合 | 制御情報（フラグ）を受け渡す |
| ↓ | 外部結合 | グローバル変数を共有する |
| ↓ | 共通結合 | 共通データ領域を共有する |
| 強い | 内容結合 | 他モジュールの内部を直接参照する |

> **暗記ポイント：** 結合度は弱いほど良い設計です。「データ → スタンプ → 制御 → 外部 → 共通 → 内容」の順に強くなります。

**モジュール凝集度（強い順）：**

| 凝集度 | 名称 | 説明 |
|--------|------|------|
| 強い | 機能的凝集 | 1つの機能のみを行う |
| ↓ | 情報的凝集 | 同じデータ構造を扱う複数機能 |
| ↓ | 連絡的凝集 | 同じデータを使う逐次的処理 |
| ↓ | 手順的凝集 | 実行順序に関連する処理 |
| ↓ | 時間的凝集 | 同じ時間に実行する処理 |
| ↓ | 論理的凝集 | 論理的に似た処理をまとめた |
| 弱い | 偶発的凝集 | たまたままとめた |

> **暗記ポイント：** 凝集度は強いほど良い設計です。

### 6.4.2 デザインパターン

よくある設計上の問題に対する、再利用可能な解決策のカタログです。GoF（Gang of Four）が提唱した23のパターンが有名です。

| カテゴリ | パターン例 | 説明 |
|---------|-----------|------|
| 生成 | Singleton | インスタンスが1つだけ存在することを保証 |
| 構造 | Adapter | インタフェースの違いを吸収 |
| 振る舞い | Observer | 状態変化を通知する仕組み |
| 振る舞い | Strategy | アルゴリズムを切り替え可能にする |

## 6.5 テスト手法

### 6.5.1 テストの工程（V字モデル）

開発の各工程に対応するテスト工程があるという考え方です。

```
要件定義 ─────────────────── システムテスト（総合テスト）
    外部設計 ───────────── 結合テスト
        内部設計 ─────── 単体テスト
            プログラミング
```

| テスト工程 | 対応する開発工程 | テスト内容 |
|-----------|----------------|-----------|
| 単体テスト（Unit Test） | 内部設計 | モジュール単位の動作確認 |
| 結合テスト（Integration Test） | 外部設計 | モジュール間の結合が正しいか |
| システムテスト（System Test） | 要件定義 | システム全体が要件を満たすか |
| 受入テスト（Acceptance Test） | ユーザー要件 | ユーザーが承認するか |

### 6.5.2 ホワイトボックステスト

プログラムの内部構造（ソースコード）を把握した上で行うテストです。主に単体テストで使用します。

**網羅基準（カバレッジ）：**

| 基準 | 説明 | 網羅率 |
|------|------|--------|
| 命令網羅（Statement Coverage） | すべての命令を最低1回実行 | C0 |
| 分岐網羅（Branch Coverage） | すべての分岐（真/偽）を最低1回実行 | C1 |
| 条件網羅（Condition Coverage） | すべての条件の真/偽を最低1回実行 | C2 |

**例：**
```
if (A > 0 AND B > 0):
    処理1
else:
    処理2
```

- 命令網羅（C0）：処理1と処理2の両方を実行するテストケース
- 分岐網羅（C1）：条件が真（処理1）と偽（処理2）の両方を実行するテストケース
- 条件網羅（C2）：A>0が真/偽、B>0が真/偽のすべての組み合わせ

### 6.5.3 ブラックボックステスト

プログラムの内部構造を考慮せず、入力と出力の関係のみでテストする手法です。

| 技法 | 説明 |
|------|------|
| 同値分割法 | 入力を「有効な範囲」と「無効な範囲」に分類し、各グループから代表値を選んでテスト |
| 境界値分析 | 有効範囲と無効範囲の境界（境界値とその前後）をテスト |
| 原因-結果グラフ | 入力条件（原因）と出力（結果）の関係を図で整理し、テストケースを導出 |
| デシジョンテーブル | 条件と動作の組み合わせを表で整理 |

**同値分割法と境界値分析の例：**

入力値が 1 〜 100 の場合：

```
同値分割：
  有効同値クラス：1〜100（例：50）
  無効同値クラス：0以下（例：-1）、101以上（例：200）

境界値分析：
  境界値：0, 1, 100, 101
  → テストケース：0（無効）、1（有効境界）、100（有効境界）、101（無効）
```

### 6.5.4 結合テストの手法

| 手法 | 説明 |
|------|------|
| トップダウンテスト | 上位モジュールから下位へ順にテスト。未完成の下位モジュールは**スタブ**で代替 |
| ボトムアップテスト | 下位モジュールから上位へ順にテスト。未完成の上位モジュールは**ドライバ**で代替 |
| ビッグバンテスト | すべてのモジュールを一度に結合してテスト |

> **暗記ポイント：** スタブは下位の代替、ドライバは上位の代替です。「スタブは下」「ドライバは上」と覚えましょう。

### 6.5.5 その他のテスト

| テスト | 説明 |
|--------|------|
| 回帰テスト（リグレッションテスト） | 修正が他に影響していないことを確認する |
| 性能テスト | 処理速度や応答時間を確認する |
| 負荷テスト | 大量のアクセスやデータに対する動作を確認する |
| セキュリティテスト | 脆弱性がないことを確認する |

## 6.6 レビュー手法

### レビューの種類

| 種類 | 説明 |
|------|------|
| インスペクション | 事前準備のもと、モデレータ（進行役）の主導で体系的に行う公式レビュー |
| ウォークスルー | 作成者が説明しながらチームメンバーと確認する非公式なレビュー |
| ラウンドロビン | 参加者が持ち回りで説明者とレビュアを務める |
| ピアレビュー | 同じ立場の技術者同士で行うレビュー |
| デスクチェック | 作成者本人が机上で確認する |

> **試験頻出ポイント：** インスペクションは最も体系的・公式なレビューであり、モデレータ（進行役）が存在することが特徴です。

## まとめ

| 項目 | 重要ポイント |
|------|-------------|
| ウォーターフォール | 順序通りに進む。手戻りが困難 |
| アジャイル | 短期反復。変化に強い。スクラムが代表的 |
| カプセル化 | データと処理を一体化し外部から隠蔽 |
| 継承 | 親クラスの機能を子クラスが引き継ぐ |
| 多態性 | 同じメッセージで異なる処理を実行 |
| 結合度 | 弱いほど良い。データ結合が最善 |
| 凝集度 | 強いほど良い。機能的凝集が最善 |
| ホワイトボックス | 内部構造を見てテスト（命令・分岐・条件網羅） |
| ブラックボックス | 入出力のみでテスト（同値分割・境界値分析） |
| スタブとドライバ | スタブ＝下位代替、ドライバ＝上位代替 |

## 確認問題

1. ウォーターフォールモデルとアジャイル開発の最大の違いを述べよ。
2. オブジェクト指向の三大要素を挙げよ。
3. 命令網羅（C0）と分岐網羅（C1）の違いを説明せよ。
4. スタブとドライバの違いを述べよ。
5. 結合度が最も弱い結合の名称を答えよ。

<details>
<summary>解答</summary>

1. ウォーターフォールは工程を順番に進めて前に戻らない。アジャイルは短い反復期間（イテレーション）で設計・実装・テストを繰り返し、変化に柔軟に対応する。
2. カプセル化（Encapsulation）、継承（Inheritance）、多態性（Polymorphism）。
3. 命令網羅（C0）はすべての命令文を最低1回実行すること。分岐網羅（C1）はすべての分岐の真と偽の両方を最低1回実行すること。
4. スタブはトップダウンテストにおいて未完成の下位モジュールの代わりに使う仮モジュール。ドライバはボトムアップテストにおいて未完成の上位モジュールの代わりに使う仮モジュール。
5. データ結合。

</details>

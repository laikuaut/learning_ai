## 課題の要件

1. MCPの3層構造（ホスト・クライアント・サーバー）をアスキーアートまたは図表で描く
2. 各コンポーネント（ホスト、クライアント、サーバー）の役割を1〜2文で説明する
3. MCPの3つの機能（ツール（Tools）、リソース（Resources）、プロンプト（Prompts））がどの層に属するかを図に示す
4. 通信プロトコル（JSON-RPC 2.0）とトランスポート（stdio / Streamable HTTP）の関係を説明する
5. MCPがない場合（N×M問題）とある場合の比較図を描く
6. 具体的なMCPホスト（Claude Desktop、Cursor等）とMCPサーバー（ファイルシステム、GitHub等）の例を3つ以上挙げる

## ステップガイド

<details>
<summary>ステップ1：N×M問題の比較図を描く</summary>

まず、MCPがない世界を描きましょう。AIアプリ3つ、ツール4つを線で結びます。

```
MCPがない場合（N×M = 12通りの個別実装）

  AIアプリ              ツール
  ┌──────┐         ┌──────────┐
  │App A ├─┬─┬─┬─→│ GitHub   │
  ├──────┤ │ │ │   ├──────────┤
  │App B ├─┼─┼─┼─→│ Slack    │
  ├──────┤ │ │ │   ├──────────┤
  │App C ├─┼─┼─┼─→│ DB       │
  └──────┘ │ │ │   ├──────────┤
           └─┴─┴─→│ ファイル  │
                   └──────────┘
  → すべての組み合わせに個別の接続コードが必要
```

次に、MCPがある世界を描きます。

```
MCPがある場合（N+M = 7通りの実装）

  AIアプリ         MCP        ツール
  ┌──────┐    ┌────────┐   ┌──────────┐
  │App A ├───→│  MCP   │←──┤ GitHub   │
  ├──────┤    │共通     │   ├──────────┤
  │App B ├───→│プロト   │←──┤ Slack    │
  ├──────┤    │コル     │   ├──────────┤
  │App C ├───→│        │←──┤ DB       │
  └──────┘    └────────┘   ├──────────┤
                           │ ファイル  │
                           └──────────┘
  → 各アプリ・ツールはMCPだけ実装すればOK
```

</details>

<details>
<summary>ステップ2：3層構造を描く</summary>

MCP の3層は「ホスト」「クライアント」「サーバー」です。

- **ホスト（Host）**: ユーザーが直接操作するアプリケーション。MCPクライアントを内包する
- **クライアント（Client）**: サーバーと1対1で通信する中間層。ホストの内部に存在する
- **サーバー（Server）**: ツール・リソース・プロンプトを外部に公開するプロセス

ポイントは「1つのホストが複数のクライアントを持つ」ことです。クライアントとサーバーは1対1対応で、サーバーごとにクライアントが生成されます。

</details>

<details>
<summary>ステップ3：3つの機能をサーバー層に配置する</summary>

MCPサーバーが提供する3つの機能を整理しましょう。

| 機能 | 役割 | 例 |
|------|------|-----|
| ツール（Tools） | LLMが呼び出す「アクション」 | ファイル作成、DB検索、API呼び出し |
| リソース（Resources） | LLMが読み取る「データ」 | 設定ファイル、ログ、DBテーブル一覧 |
| プロンプト（Prompts） | 定型的な「指示テンプレート」 | コードレビュー、要約、翻訳のテンプレート |

これらはすべて**サーバー側**で定義し、クライアントを通じてホストに公開されます。

</details>

<details>
<summary>ステップ4：通信プロトコルとトランスポートを説明する</summary>

MCPの通信は2つのレイヤーで構成されます。

```
┌──────────────────────────────────┐
│ メッセージ形式：JSON-RPC 2.0     │  ← 何を送るか
├──────────────────────────────────┤
│ トランスポート：stdio または      │  ← どうやって送るか
│                Streamable HTTP   │
└──────────────────────────────────┘
```

- **JSON-RPC 2.0**: メッセージの形式。リクエスト・レスポンス・通知の3種類がある
- **stdio**: 標準入出力を使ったローカル通信（同一マシン上）
- **Streamable HTTP**: HTTP上のストリーミング通信（ネットワーク越し）

</details>

### 3. 各コンポーネントの説明

| コンポーネント | 説明 |
|---------------|------|
| ホスト（Host） | ユーザーが直接操作するAIアプリケーションであり、MCPクライアントのライフサイクルを管理し、セキュリティポリシーを適用する |
| クライアント（Client） | ホストの内部に存在する通信モジュールであり、1つのMCPサーバーと1対1で JSON-RPC 2.0 メッセージをやり取りする |
| サーバー（Server） | ツール・リソース・プロンプトを定義して公開するプロセスであり、外部データソースやAPIへのアクセスを仲介する |
| 通信プロトコル | MCPは JSON-RPC 2.0 プロトコルで、リクエスト・レスポンス・通知の3種類の形式のメッセージを使う |

### 4. 通信プロトコルの整理

```
┌────────────────────────────────────────────────┐
│         MCP の通信スタック                       │
│                                                │
│  ┌──────────────────────────────────────────┐  │
│  │ アプリケーション層                         │  │
│  │ MCPメッセージ（initialize, tools/call 等） │  │
│  ├──────────────────────────────────────────┤  │
│  │ メッセージ形式                             │  │
│  │ JSON-RPC 2.0                              │  │
│  │ ・リクエスト: id + method + params         │  │
│  │ ・レスポンス: id + result / error          │  │
│  │ ・通知:       method + params（idなし）     │  │
│  ├──────────────────────────────────────────┤  │
│  │ トランスポート層                           │  │
│  │ ・stdio: 標準入出力（ローカル向け）         │  │
│  │ ・Streamable HTTP: HTTP通信（リモート向け） │  │
│  └──────────────────────────────────────────┘  │
└────────────────────────────────────────────────┘
```

## よくある間違い

| 間違い | 正しい理解 |
|--------|-----------|
| クライアントとサーバーが多対多で通信する | クライアントとサーバーは**1対1**で通信する。複数サーバーに接続するには、ホストが複数のクライアントを生成する |
| MCPサーバーがLLMに直接リクエストを送る | サーバーはLLMと直接通信しない。ホストを経由してLLMと連携する |
| stdioとStreamable HTTPは排他的 | 同じサーバーが両方のトランスポートをサポートすることも可能 |
| MCPはHTTP上の独自プロトコル | MCPのメッセージ形式はJSON-RPC 2.0。トランスポートとしてHTTPを使う場合がある |

## 基本問題

### 問題1：通信パターンの識別

以下の4つの通信パターンについて、それぞれの特徴を簡潔に説明してください。

1. 直接通信（Direct Communication）
2. メッセージパッシング（Message Passing）
3. 共有メモリ（Shared Memory）
4. ブラックボード（Blackboard）

**期待される出力例：**

```
1. 直接通信：エージェント同士が1対1で直接やり取りする方式。
   シンプルだが、エージェント数が増えると接続が複雑になる。
```

<details>
<summary>解答例</summary>

```
1. 直接通信（Direct Communication）
   エージェント同士が1対1で直接メッセージを送受信する方式。
   実装がシンプルで遅延が少ないが、エージェント数が増えると
   接続数がO(n²)で増加し管理が複雑になる。

2. メッセージパッシング（Message Passing）
   メッセージキューやブローカーを介して非同期にメッセージを交換する方式。
   送信側と受信側が疎結合になり、スケーラビリティが高い。
   メッセージの順序保証や配信保証の設計が必要になる。

3. 共有メモリ（Shared Memory）
   全エージェントがアクセスできる共有データ領域を通じて情報を交換する方式。
   高速なデータ共有が可能だが、競合状態（Race Condition）の管理や
   排他制御が必要になる。

4. ブラックボード（Blackboard）
   共有のデータストア（ブラックボード）にエージェントが情報を書き込み・
   読み取りする方式。各エージェントは自分が処理できる情報があれば
   自律的に取得して処理する。柔軟だが、全体の進行管理が難しい。
```

</details>

### 問題2：通信パターンの適用場面判断

以下のシナリオに対して、最適な通信パターン（直接通信/メッセージパッシング/共有メモリ/ブラックボード）を選び、理由を説明してください。

| シナリオ | 説明 |
|----------|------|
| S1 | 2つのエージェントが交互に質問と回答を繰り返すQ&Aシステム |
| S2 | 10個以上のエージェントが異なるタイミングで分析結果を提出し、統合する調査システム |
| S3 | リアルタイムに株価データを複数の分析エージェントが同時参照する金融システム |
| S4 | 注文受付→在庫確認→発送指示の順で処理が流れるECシステム |
| S5 | 複数の専門家エージェントが、未解決の問題リストから自分の専門に合うものを選んで回答する知識ベースシステム |

**期待される出力例：**

```
S1: 直接通信
理由：エージェントが2つだけで、1対1の同期的なやり取りが中心のため。
```

<details>
<summary>解答例</summary>

```
S1: 直接通信（Direct Communication）
理由：エージェントが2つだけで、交互に同期的なやり取りを行う構造。
直接通信が最もシンプルで低遅延。仲介を挟む必要がない。

S2: ブラックボード（Blackboard）
理由：エージェントの数が多く、それぞれ異なるタイミングで成果物を提出する。
ブラックボードに分析結果を書き込むことで、非同期かつ柔軟な統合が可能。
各エージェントは他のエージェントの存在を知る必要がない。

S3: 共有メモリ（Shared Memory）
理由：リアルタイムデータを複数エージェントが同時参照する必要があるため、
共有メモリによる高速な読み取りが適している。
書き込みは株価データソースのみなので、競合も最小限。

S4: メッセージパッシング（Message Passing）
理由：処理が順番に流れるパイプライン構造であり、各段階のエージェントは
前段階の完了通知を受けて処理を開始する。メッセージキューを使うことで
疎結合かつ信頼性のある順次処理が実現できる。

S5: ブラックボード（Blackboard）
理由：未解決問題リストが共有データとなり、各専門家エージェントが
自律的に自分の得意分野の問題を取得して処理する構造。
ブラックボードパターンの典型的なユースケース。
```

</details>

### 問題3：協調パターンの分類

以下の協調パターンを「競争型」「協力型」「階層型」のいずれかに分類し、それぞれの特徴を説明してください。

1. マスター・スレーブ型
2. ディベート型
3. 合議型（投票ベース）
4. パイプライン型
5. オーケストレーション型
6. オークション型

**期待される出力例：**

```
1. マスター・スレーブ型 → 階層型
   マスターが指示を出し、スレーブが実行する上下関係の構造。
```

<details>
<summary>解答例</summary>

```
1. マスター・スレーブ型 → 階層型
   マスターエージェントがタスクを分割して各スレーブに指示を出し、
   結果を集約する。制御が明確だがマスターがボトルネックになりうる。

2. ディベート型 → 競争型
   複数のエージェントが異なる立場から主張を行い、議論を通じて
   最良の結論を導き出す。多様な視点による品質向上が期待できる。

3. 合議型（投票ベース） → 協力型
   各エージェントが独立して判断を下し、多数決や加重投票で
   最終結論を決定する。単一エージェントより頑健な判断が可能。

4. パイプライン型 → 協力型
   各エージェントが処理の一段階を担当し、順番に成果物を
   次のエージェントに渡す。分業による効率化が利点。

5. オーケストレーション型 → 階層型
   オーケストレーターが全体のワークフローを管理し、
   各エージェントの実行順序やデータの流れを制御する。
   複雑なワークフローの管理に適している。

6. オークション型 → 競争型
   タスクに対して複数のエージェントが「入札」し、最も適した
   エージェントがタスクを獲得する。動的なタスク割り当てに有効。
```

</details>

### 問題4：役割設計の基本原則

マルチエージェントシステムにおけるエージェントの役割設計について、以下の問いに答えてください。

(1) 「単一責任の原則」をマルチエージェントの役割設計に適用すると、どのような設計になりますか？

(2) 以下のタスクを1つのエージェントに任せた場合の問題点を指摘し、適切な役割分割を提案してください。

```
タスク：「Webの情報を検索し、内容を要約し、日本語に翻訳し、
         品質をチェックし、最終レポートとしてフォーマットする」
```

**期待される出力例：**

```
(1) 各エージェントが1つの明確な責務のみを持つ設計。
(2) 問題点：...
    分割案：...
```

<details>
<summary>解答例</summary>

```
(1) 単一責任の原則をマルチエージェントに適用した設計：
    各エージェントが1つの明確な責務（検索、分析、生成など）のみを持ち、
    その責務に特化したプロンプト・ツール・知識を与える設計です。
    これにより以下のメリットが得られます：
    - 各エージェントの品質が向上する（専門特化による精度向上）
    - デバッグが容易になる（問題の切り分けが明確）
    - 個別にスケール・改善が可能（1つのエージェントだけ改善できる）
    - 再利用性が高まる（他のシステムでも同じエージェントを使える）

(2) 問題点：
    - プロンプトが長大かつ複雑になり、各ステップの品質が低下する
    - どのステップで問題が起きたか特定しにくい
    - 一部のステップだけ改善・変更することが困難
    - コンテキストウィンドウを圧迫し、後半の処理精度が下がる

    分割案（5エージェント構成）：
    ┌──────────────────────────────────────────────────┐
    │  検索エージェント → 要約エージェント →             │
    │  翻訳エージェント → 品質チェックエージェント →     │
    │  フォーマットエージェント                          │
    └──────────────────────────────────────────────────┘
    - 検索エージェント：Web検索ツールを使い、関連情報を収集
    - 要約エージェント：収集した情報を構造化・要約
    - 翻訳エージェント：英語の内容を日本語に翻訳
    - 品質チェックエージェント：翻訳の正確性・一貫性を検証
    - フォーマットエージェント：最終レポートの体裁を整える
```

</details>

### 問題5：コンテンツ制作マルチエージェントの設計

あなたはブログ記事を自動生成するマルチエージェントシステムを設計します。以下の3つのエージェントを含むシステムの全体設計を行ってください。

- **ライターエージェント**：記事の執筆を担当
- **編集者エージェント**：記事の校正・改善を担当
- **イラストレーターエージェント**：記事に合った画像プロンプトの生成を担当

以下の項目を設計してください。

1. 各エージェントのシステムプロンプト（概要）
2. エージェント間の通信フロー（どの順番でどんなデータをやり取りするか）
3. 使用する通信パターンとその理由
4. エラーハンドリング（編集者がリジェクトした場合のフロー）

**期待される出力例：**

```
1. システムプロンプト：
   ライター：「あなたはプロのWebライターです。SEOを意識した...」
   編集者：「あなたは経験豊富な編集者です。以下の観点で記事を...」
   イラストレーター：「あなたは画像生成AIのプロンプトエンジニアです...」

2. 通信フロー：
   [テーマ入力] → ライター → 編集者 → (OK/NG分岐) → イラストレーター → [最終出力]

3. 通信パターン：オーケストレーション + メッセージパッシング
   理由：...

4. エラーハンドリング：
   編集者がNGを返した場合 → フィードバック付きでライターに差し戻し（最大3回）
```

<details>
<summary>ヒント</summary>

- 各エージェントは独立して動作し、前のエージェントの出力を入力として受け取ります
- 編集者はライターの成果物に対して「承認」か「修正依頼」を返す判断が必要です
- イラストレーターは記事の内容を理解した上で画像プロンプトを作成します
- 無限ループを防ぐためのリトライ上限を設けましょう

</details>

<details>
<summary>解答例</summary>

```
【コンテンツ制作マルチエージェントシステム設計】

1. 各エージェントのシステムプロンプト：

   ■ ライターエージェント
   「あなたはプロのWebライターです。与えられたテーマに基づき、
   SEOを意識した2000〜3000字のブログ記事を執筆してください。
   以下の構成で書いてください：
   - キャッチーなタイトル
   - 導入（読者の関心を引く問いかけ）
   - 本文（H2見出し3〜5個、各セクションに具体例を含む）
   - まとめ（アクションアイテム付き）
   編集者からフィードバックがある場合は、その指摘を反映して修正してください。」

   ■ 編集者エージェント
   「あなたは経験豊富なコンテンツ編集者です。
   提出された記事を以下の観点で評価してください：
   - 文法・誤字脱字のチェック
   - 論理構成の一貫性
   - ターゲット読者への適切さ
   - SEO観点での改善点
   評価結果を以下の形式で返してください：
   { "status": "approved" | "revision_needed",
     "score": 1-10,
     "feedback": "具体的な改善点のリスト" }」

   ■ イラストレーターエージェント
   「あなたは画像生成AIのプロンプトエンジニアです。
   記事の内容を読み、各セクションに適した画像プロンプトを
   英語で生成してください。以下を含めてください：
   - アイキャッチ画像のプロンプト（1枚）
   - セクションごとの挿絵プロンプト（各1枚）
   スタイルは統一感のあるフラットイラスト調にしてください。」

2. 通信フロー：

   ┌─────────┐     記事草稿     ┌─────────┐
   │ ライター │───────────────→│ 編集者  │
   │         │←───────────────│         │
   └─────────┘  フィードバック   └────┬────┘
        ↑                             │ approved
        │ revision_needed             ↓
        │                     ┌──────────────────┐
        └─(最大3回ループ)     │イラストレーター    │
                              └────────┬─────────┘
                                       ↓
                               ┌──────────────┐
                               │ 最終出力      │
                               │（記事+画像    │
                               │  プロンプト） │
                               └──────────────┘

   データフロー詳細：
   Step 1: ユーザーがテーマを入力
   Step 2: ライターが記事草稿を生成
   Step 3: 編集者が記事をレビュー
   Step 4a: approved → イラストレーターに記事を渡す
   Step 4b: revision_needed → フィードバック付きでライターに差し戻し
   Step 5: イラストレーターが画像プロンプトを生成
   Step 6: 記事＋画像プロンプトを統合して最終出力

3. 通信パターン：オーケストレーション型 + メッセージパッシング

   理由：
   - オーケストレーターがワークフロー全体（順序・分岐・リトライ）を管理
   - 各エージェント間のデータ受け渡しはメッセージパッシングで非同期対応
   - 編集者の判断による条件分岐があるため、単純なパイプラインでは不十分
   - オーケストレーターが状態（リトライ回数等）を一元管理できる

4. エラーハンドリング：

   ■ 編集者リジェクト時のフロー：
   - 編集者が revision_needed を返した場合：
     → フィードバック内容をライターに渡し、修正を依頼
     → リトライカウンターをインクリメント（初期値0）
     → 最大3回まで修正ループを許可
   - 3回修正してもスコアが基準（7/10）未満の場合：
     → 「最善の版」（最もスコアが高かった版）を採用し、
       編集者のコメントを注記として添付して次のステップへ進む
   - ライターエージェントがエラー（タイムアウト等）の場合：
     → 1回リトライし、それでも失敗したらユーザーに通知
```

<!-- 設計のポイント：
     - 編集者の判断基準を明確にし、主観的な判断を減らす
     - リトライ上限を設けて無限ループを防止する
     - 「最善の版を採用」というフォールバック戦略で品質を担保する
     - 各エージェントは他のエージェントの内部実装を知らない（疎結合） -->

### 問題6：ディベート型エージェントの設計

「リモートワークは企業にとって有益か？」というテーマについて議論するディベート型マルチエージェントシステムを設計してください。

以下の要素を含めてください。

1. 必要なエージェントの一覧と各役割
2. 議論の進行フロー（ラウンド制の設計）
3. 最終的な結論の導出方法
4. このシステムが単一エージェントより優れている点

**期待される出力例：**

```
1. エージェント一覧：
   - 肯定派エージェント：リモートワークの利点を主張
   - 否定派エージェント：リモートワークの問題点を主張
   - 司会者エージェント：議論を整理し結論を導く
   ...

2. 進行フロー：
   ラウンド1（開始陳述）→ ラウンド2（反論）→ ...
```

<details>
<summary>ヒント</summary>

- ディベートには「肯定側」「否定側」「判定者」の役割が基本です
- 各ラウンドには明確な目的（主張提示、反論、最終弁論など）を持たせましょう
- 結論は単なる勝敗ではなく、両方の論点を統合した建設的なものにしましょう
- ファクトチェック（事実確認）の役割があるとさらに品質が向上します

</details>

<details>
<summary>解答例</summary>

```
【ディベート型マルチエージェントシステム設計】

1. エージェント一覧と役割：

   ┌────────────────────────────────────────────────────┐
   │ エージェント名       │ 役割                         │
   ├────────────────────────────────────────────────────┤
   │ 肯定派エージェント   │ リモートワークの利点を主張。  │
   │                      │ 生産性向上、人材獲得、コスト │
   │                      │ 削減等の観点から論じる        │
   ├────────────────────────────────────────────────────┤
   │ 否定派エージェント   │ リモートワークの問題点を主張。│
   │                      │ コミュニケーション低下、      │
   │                      │ セキュリティ、管理困難等      │
   ├────────────────────────────────────────────────────┤
   │ ファクトチェッカー   │ 各主張に含まれるデータや      │
   │                      │ 統計の正確性を検証する        │
   ├────────────────────────────────────────────────────┤
   │ 司会者エージェント   │ 議論の進行管理、論点整理、    │
   │                      │ 各ラウンドの要約を行う        │
   ├────────────────────────────────────────────────────┤
   │ 判定者エージェント   │ 両者の主張を客観的に評価し、  │
   │                      │ バランスの取れた結論を導出    │
   └────────────────────────────────────────────────────┘

2. 議論の進行フロー（4ラウンド制）：

   ■ ラウンド1：開始陳述（各3分相当）
     肯定派 → 主要論点を3つ提示（根拠・データ付き）
     否定派 → 主要論点を3つ提示（根拠・データ付き）
     ファクトチェッカー → 提示されたデータの検証結果を報告
     司会者 → 両者の論点を整理してまとめる

   ■ ラウンド2：反論（各2分相当）
     肯定派 → 否定派の各論点への反論
     否定派 → 肯定派の各論点への反論
     ファクトチェッカー → 反論に含まれる新データの検証
     司会者 → 論点の対立構造を整理

   ■ ラウンド3：再反論と補強（各2分相当）
     肯定派 → 反論への再反論＋新たな根拠の追加
     否定派 → 反論への再反論＋新たな根拠の追加
     司会者 → 合意できている点と対立点を明確化

   ■ ラウンド4：最終弁論（各1分相当）
     肯定派 → 最も強い論点に絞った最終主張
     否定派 → 最も強い論点に絞った最終主張

3. 最終的な結論の導出方法：

   判定者エージェントが以下の基準で評価：
   - 論拠の強さ（データの信頼性、論理の一貫性）：40%
   - 反論の的確さ（相手の論点を正確に捉えているか）：30%
   - 実用性（実際の企業にとって実行可能か）：30%

   出力形式：
   {
     "winning_side": "肯定派/否定派",
     "score": { "肯定派": 75, "否定派": 68 },
     "key_insights": [
       "リモートワークは生産性向上に寄与するが、職種による差が大きい",
       "ハイブリッド型が多くの企業にとって最適解となりうる",
       "成功にはコミュニケーションツールと評価制度の整備が不可欠"
     ],
     "balanced_conclusion": "完全なリモートワークではなく、
       職種・業務内容に応じたハイブリッド型の導入が推奨される。
       ただし、適切なインフラ・制度設計が前提条件となる。"
   }

4. 単一エージェントより優れている点：
```

### 問題8：マルチエージェントのアンチパターン分析

以下のマルチエージェントシステム設計には問題があります。それぞれの問題点を指摘し、改善案を提示してください。

**設計A：全知全能型**

```
エージェント構成：
- スーパーエージェント：全てのタスク（検索・分析・執筆・翻訳・校正）を1つで処理
- ロガーエージェント：スーパーエージェントの出力をログに記録するだけ
```

```
エージェント構成（5エージェント）：
- 全エージェントが他の全エージェントと双方向通信
- 中央の制御機構なし
- 各エージェントが独自に判断して他のエージェントにメッセージを送信
```

**期待される出力例：**


<details>
<summary>解答例</summary>

```
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
設計A：全知全能型 の分析

1. マルチエージェントにする意味がない
   → 実質的に単一エージェントと同じ。ロガーは受動的で協調に寄与しない。
2. スーパーエージェントのプロンプトが肥大化し、各タスクの品質が低下する
3. 障害の単一点（Single Point of Failure）
   → スーパーエージェントが失敗すると全体が停止する
4. スケーラビリティがない
   → 特定タスクだけ強化・改善することが不可能

改善案：
- 機能ごとに専門エージェントに分割する
  （検索エージェント、分析エージェント、執筆エージェント等）
- オーケストレーターエージェントを導入して全体のフロー制御を行う
- 各エージェントに適切なツールと専門特化したプロンプトを付与する
- ロガーは独立エージェントではなく、システム基盤の機能として実装する

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
設計B：過剰分割型 の分析
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

問題点：
1. 通信オーバーヘッドが非常に大きい
   → 11エージェント間のデータ受け渡しでレイテンシが累積する
2. 各エージェントの処理が軽微すぎる
   → 「件名読取」「署名追加」はエージェント（LLM呼び出し）にする
     必要がなく、通常のプログラム処理で十分
3. デバッグが困難
   → 問題発生時に11段階のどこで起きたか追跡が大変
4. コストの無駄
   → 各ステップでLLM APIを呼び出すため、コストが不必要に増大

改善案：
- 論理的にまとめられる処理をグループ化して3〜4エージェントに統合
  例：
  ・分析エージェント（件名+本文の読取、感情・意図分析、キーワード抽出）
  ・返信生成エージェント（方針決定、本文生成、挨拶・署名付与）
  ・品質チェックエージェント（敬語チェック、全体整合性の確認）
- LLM呼び出しが不要な処理（件名読取、署名追加等）はツール関数として実装
- 各エージェントの粒度を「独立した専門判断が必要な単位」に設定する

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
設計C：無秩序通信型 の分析
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

問題点：
1. 通信経路が爆発的に増加
   → 5エージェントの全対全通信はC(5,2)×2=20本の通信経路
   → エージェント追加時にO(n²)で増加し、スケールしない
2. メッセージの無限ループリスク
   → AがBに送信→BがCに送信→CがAに送信...の循環が発生しうる
3. デバッグ・トレースが極めて困難
   → メッセージの流れが予測不能で、問題の原因特定が困難
4. 全体の進捗管理ができない
   → 誰が何をしているか、タスクが完了したかを把握する仕組みがない
5. 競合状態のリスク
   → 複数エージェントが同時に同じリソースを変更する可能性がある

改善案：
- オーケストレーターパターンの導入
  → 1つのエージェントが全体のフロー制御を担当
- 通信をハブ&スポーク型に変更
  → 各エージェントはオーケストレーターとのみ通信する
- メッセージにIDとタイムスタンプを付与し、トレーサビリティを確保
- 各エージェントの状態（待機中/処理中/完了/エラー）を一元管理する
  状態管理の仕組みを導入する
```

# 第7章：プロジェクト管理とCLAUDE.md

> **この章のゴール**: CLAUDE.mdファイルの役割と書き方を理解し、プロジェクトに合わせたClaude Codeの設定管理ができるようになること

---

## 7.1 CLAUDE.mdとは

CLAUDE.mdは、**プロジェクト固有の指示書**です。このファイルに書かれた内容は、Claude Codeがそのプロジェクトで作業する際に常に参照されます。

### CLAUDE.mdの役割

```
┌─────────────────────────────────────────────────┐
│               CLAUDE.md の役割                   │
│                                                  │
│  ┌─────────────────────────────────────────┐    │
│  │ プロジェクトの概要・構成                  │    │
│  │ コーディング規約・スタイルガイド          │    │
│  │ テスト・ビルドの実行方法                  │    │
│  │ 使用技術・フレームワーク                  │    │
│  │ 特別なルールや注意事項                    │    │
│  └─────────────────────────────────────────┘    │
│                    │                             │
│                    ▼                             │
│          Claude Code がこれを読み、              │
│          プロジェクトに合った対応をする           │
│                                                  │
└─────────────────────────────────────────────────┘
```

### どのような情報を書くべきか

| カテゴリ | 記載例 |
|---|---|
| **プロジェクト概要** | このプロジェクトはECサイトのバックエンドAPIです |
| **技術スタック** | TypeScript, Express, PostgreSQL, Jest |
| **ディレクトリ構成** | src/配下の各ディレクトリの役割 |
| **コーディング規約** | 命名規則、インポート順、エラーハンドリング方針 |
| **テスト方法** | `npm test`でユニットテスト実行 |
| **ビルド方法** | `npm run build`でビルド |
| **特別なルール** | 「○○ディレクトリは変更しないこと」等 |

### ポイントまとめ
- CLAUDE.mdはプロジェクト固有の指示書
- Claude Codeの挙動をプロジェクトに合わせてカスタマイズできる
- プロジェクトルートに配置する

## 7.2 CLAUDE.mdの書き方

### 基本的な構成

```markdown
# プロジェクト名

## 概要
ECサイトのバックエンドAPI。REST APIを提供し、
フロントエンド（Next.js）と連携します。

## 技術スタック
- 言語: TypeScript 5.x
- フレームワーク: Express 4.x
- データベース: PostgreSQL 15
- ORM: Prisma
- テスト: Jest + Supertest

## ディレクトリ構成
```
src/
├── controllers/  # APIエンドポイント
├── services/     # ビジネスロジック
├── models/       # データモデル（Prisma）
├── middleware/    # ミドルウェア
├── utils/        # ユーティリティ
└── tests/        # テストファイル
```

## 開発コマンド
- `npm run dev` - 開発サーバー起動
- `npm test` - テスト実行
- `npm run build` - ビルド
- `npm run lint` - リンター実行

## コーディング規約
- 関数名はキャメルケースを使用
- ファイル名はケバブケースを使用
- エラーはカスタムエラークラスを使って投げる
- 全てのAPI関数にJSDocコメントを付ける

## 注意事項
- src/generated/ は自動生成ファイルなので編集しないこと
- 環境変数は .env.example を参照すること
- PRはmainブランチに対して作成すること
```

### 効果的なCLAUDE.mdを書くコツ

```
┌────────────────────────────────────────────────────┐
│       効果的な CLAUDE.md のポイント                  │
├────────────────────────────────────────────────────┤
│                                                    │
│  ✓ 具体的に書く（曖昧な表現を避ける）             │
│  ✓ 実行可能なコマンドを正確に記載する              │
│  ✓ 「やってはいけないこと」も明記する              │
│  ✓ 定期的に更新する                                │
│  ✓ チーム全員がレビューする                        │
│                                                    │
│  ✕ 長すぎる説明（簡潔に）                         │
│  ✕ 一般的すぎる内容（プロジェクト固有の情報を）    │
│  ✕ 古い情報の放置                                  │
│                                                    │
└────────────────────────────────────────────────────┘
```

### よくある間違い
> **CLAUDE.mdに一般的なプログラミング知識を書く**: 「変数とは値を格納する箱です」のような一般知識は不要です。Claude Codeはすでにプログラミングの知識を持っています。プロジェクト**固有**の情報を書きましょう。

### ポイントまとめ
- プロジェクト概要、技術スタック、コマンド、規約を記載
- 具体的かつ簡潔に書く
- 禁止事項や注意事項も明記する

## 7.3 設定ファイルの階層

Claude Codeの設定は複数のレベルで管理されます。

### 設定の優先順位

```
優先度: 高 ──────────────────────────────────────→ 低

┌──────────────┐  ┌──────────────┐  ┌──────────────┐
│   直接の指示   │  │ プロジェクト  │  │ ユーザー     │
│  （プロンプト）│  │ 設定         │  │ グローバル   │
│              │  │              │  │ 設定         │
│              │  │ CLAUDE.md    │  │              │
│              │  │ .claude/     │  │ ~/.claude/   │
│              │  │  settings.   │  │  settings.   │
│              │  │  json        │  │  json        │
└──────────────┘  └──────────────┘  └──────────────┘
```

### 各設定ファイルの場所と役割

| ファイル | 場所 | 役割 |
|---|---|---|
| **CLAUDE.md** | プロジェクトルート | プロジェクト固有の指示（Git管理対象） |
| **.claude/settings.json** | プロジェクト内 | プロジェクトの技術的設定（Git管理対象） |
| **.claude/settings.local.json** | プロジェクト内 | 個人的な設定（Git管理対象外） |
| **~/.claude/settings.json** | ホームディレクトリ | グローバル設定（全プロジェクト共通） |

### .claude/settings.json の例

```json
{
  "permissions": {
    "allow": [
      "Read",
      "Glob",
      "Grep",
      "Bash(npm test)",
      "Bash(npm run lint)"
    ],
    "deny": [
      "Bash(rm -rf *)"
    ]
  },
  "mcpServers": {
    "github": {
      "command": "npx",
      "args": ["-y", "@modelcontextprotocol/server-github"]
    }
  }
}
```

### .claude/settings.local.json の例

```json
{
  "permissions": {
    "allow": [
      "Bash(docker compose up)"
    ]
  },
  "env": {
    "DEBUG": "true"
  }
}
```

### よくある間違い
> **settings.local.jsonをGitにコミットする**: `.claude/settings.local.json`は個人的な設定です。`.gitignore`に追加して、リポジトリにコミットしないようにしましょう。APIキーやローカル環境固有の設定をここに書けます。

### ポイントまとめ
- 設定は直接の指示 > プロジェクト設定 > グローバル設定の優先順位
- CLAUDE.mdはチームで共有、settings.local.jsonは個人用
- .gitignoreにsettings.local.jsonを追加する

## 7.4 チーム開発でのClaude Code活用

### チームでの設定共有

```
┌─────────────────────────────────────────────────┐
│          チーム開発での設定管理                    │
│                                                  │
│  Git管理する（チーム共有）:                       │
│    ├── CLAUDE.md                                 │
│    └── .claude/settings.json                     │
│                                                  │
│  Git管理しない（個人設定）:                       │
│    └── .claude/settings.local.json               │
│                                                  │
│  .gitignore に追加:                              │
│    .claude/settings.local.json                   │
│                                                  │
└─────────────────────────────────────────────────┘
```

### チーム向けCLAUDE.mdの記載項目

チーム開発では、以下の情報を CLAUDE.md に含めることを推奨します：

1. **オンボーディング情報**: 新メンバーが最初に知るべきこと
2. **ブランチ戦略**: ブランチ命名規則、マージフロー
3. **レビュー基準**: PRで確認すべきポイント
4. **禁止操作**: 「mainに直接pushしない」等
5. **デプロイ手順**: デプロイ関連のコマンドや注意事項

### 実践例：チーム向けCLAUDE.md

```markdown
# チーム開発ルール

## ブランチ戦略
- main: 本番環境（直接pushは禁止）
- develop: 開発環境
- feature/*: 機能開発ブランチ
- fix/*: バグ修正ブランチ

## PR作成ルール
- PRはdevelopブランチに対して作成すること
- タイトルは「[種別] 概要」形式（例：[feat] ユーザー登録機能追加）
- テストが通ることを確認してからPRを作成すること

## コミットメッセージ規約
Conventional Commits形式を使用:
- feat: 新機能
- fix: バグ修正
- docs: ドキュメント
- refactor: リファクタリング
- test: テスト追加・修正
```

### ポイントまとめ
- CLAUDE.mdとsettings.jsonはチームで共有してGit管理
- settings.local.jsonは個人設定として.gitignoreに追加
- チーム開発ではブランチ戦略やPRルールも記載する

## 7.5 複数プロジェクトの管理

### プロジェクトごとのCLAUDE.md

各プロジェクトに専用のCLAUDE.mdを作成することで、Claude Codeの挙動をプロジェクトごとにカスタマイズできます：

```
~/projects/
├── frontend/
│   ├── CLAUDE.md          ← React/TypeScriptの規約
│   └── .claude/settings.json
├── backend/
│   ├── CLAUDE.md          ← Python/FastAPIの規約
│   └── .claude/settings.json
└── mobile/
    ├── CLAUDE.md          ← React Nativeの規約
    └── .claude/settings.json
```

### グローバル設定の活用

全プロジェクト共通の設定は`~/.claude/settings.json`に書きます：

```json
{
  "permissions": {
    "allow": [
      "Read",
      "Glob",
      "Grep"
    ]
  }
}
```

### よくある間違い
> **CLAUDE.mdをプロジェクトルート以外に置く**: Claude CodeはプロジェクトのルートディレクトリにあるCLAUDE.mdを読み込みます。サブディレクトリに置いても認識されません。

## 7.6 CLAUDE.mdの更新と運用

### 定期的な見直し

CLAUDE.mdはプロジェクトの進化に合わせて更新する必要があります：

```
┌────────────────────────────────────────────────┐
│        CLAUDE.md 更新タイミング                 │
├────────────────────────────────────────────────┤
│                                                │
│  ✓ 新しい技術やライブラリを導入したとき        │
│  ✓ ディレクトリ構成を変更したとき              │
│  ✓ コーディング規約を変更したとき              │
│  ✓ ビルド・テスト手順が変わったとき            │
│  ✓ 新しいチームメンバーが困った点があったとき  │
│                                                │
└────────────────────────────────────────────────┘
```

### Claude CodeにCLAUDE.mdを更新させる

```
> CLAUDE.md を更新して、最近追加した認証モジュールの
> 情報を追記してください
```

Claude Code自身にCLAUDE.mdの更新を依頼することもできます。ただし、内容は必ず人間がレビューしてください。

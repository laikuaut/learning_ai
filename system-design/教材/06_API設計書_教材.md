## 6.1 API設計書とは

### API設計書の位置づけ

API設計書は、**フロントエンドとバックエンドの「契約書」**です。APIのエンドポイント、リクエスト形式、レスポンス形式を定義し、フロントエンド開発者とバックエンド開発者が並行して開発できるようにします。

```
【API設計書の位置づけ】

フロントエンド開発者         バックエンド開発者
       │                        │
       │    API設計書（契約）     │
       │◀══════════════════════▶│
       │                        │
       │  この仕様に従って       │  この仕様に従って
       │  APIを呼び出す         │  APIを実装する
       │                        │
       ▼                        ▼
    画面の実装              APIの実装
  （並行して開発可能）     （並行して開発可能）
```

### API設計書の構成

```
API設計書の目次（例）
━━━━━━━━━━━━━━━━━━━━━━━━
1. API設計方針
   1.1 アーキテクチャスタイル（REST）
   1.2 ベースURL
   1.3 認証方式
   1.4 共通ヘッダー
   1.5 共通エラーレスポンス

2. API一覧

3. 各APIの詳細仕様
   （エンドポイントごとに記述）

4. 認証API
5. ユーザーAPI
6. 商品API
7. 注文API
...
━━━━━━━━━━━━━━━━━━━━━━━━
```

## 6.2 REST APIの設計原則

### RESTとは

REST（Representational State Transfer）は、Web APIの設計スタイルです。以下の原則に従います。

```
【REST APIの基本原則】

1. リソース指向
   → URL はリソース（名詞）を表す
   → 例：/users, /products, /orders

2. HTTPメソッドで操作を表す
   → GET（取得）、POST（作成）、PUT（更新）、DELETE（削除）

3. ステートレス
   → サーバーはクライアントの状態を保持しない
   → 認証情報は毎回リクエストに含める

4. 統一インタフェース
   → 一貫したURL設計とレスポンス形式
```

### HTTPメソッドとCRUD操作

```
【HTTPメソッドの対応表】

| HTTPメソッド | CRUD操作 | 説明         | べき等性 | 安全性 |
|-------------|---------|-------------|---------|-------|
| GET         | Read    | リソースの取得 | ○       | ○     |
| POST        | Create  | リソースの作成 | ✗       | ✗     |
| PUT         | Update  | リソースの全体更新| ○    | ✗     |
| PATCH       | Update  | リソースの部分更新| ✗    | ✗     |
| DELETE      | Delete  | リソースの削除 | ○       | ✗     |

※ べき等性：同じリクエストを何度送っても同じ結果になる
※ 安全性：リソースの状態を変更しない
```

### エンドポイントの命名規則

```
【URL設計のルール】

✅ 良い設計:
  GET    /users           → ユーザー一覧取得
  GET    /users/123       → ユーザー詳細取得
  POST   /users           → ユーザー作成
  PUT    /users/123       → ユーザー更新
  DELETE /users/123       → ユーザー削除
  GET    /users/123/orders → ユーザーの注文一覧

❌ 悪い設計:
  GET    /getUsers         → 動詞を使わない
  POST   /createUser       → HTTPメソッドで操作を表す
  GET    /user/list        → 複数形にする
  POST   /users/delete/123 → DELETEメソッドを使う

【命名規則のまとめ】
  ・リソース名は複数形の名詞（users, products, orders）
  ・小文字のケバブケース（order-items）
  ・動詞は使わない（HTTPメソッドで表現）
  ・階層はネスト（/users/123/orders）
  ・深いネストは避ける（3階層まで）
```

## 6.5 共通仕様の設計

### 認証方式

```
【認証方式：Bearer Token（JWT）】

リクエストヘッダーに以下を含める:
  Authorization: Bearer eyJhbGciOiJIUzI1NiIs...

トークンの種類:
┌──────────────┬────────────┬───────────────────────┐
│ トークン種別   │ 有効期限    │ 用途                   │
├──────────────┼────────────┼───────────────────────┤
│ アクセストークン│ 1時間      │ API認証に使用           │
│ リフレッシュ   │ 30日       │ アクセストークンの再発行  │
│ トークン      │            │                       │
└──────────────┴────────────┴───────────────────────┘

トークン更新フロー:
  1. アクセストークンが期限切れ（401レスポンス）
  2. リフレッシュトークンで POST /auth/refresh を呼ぶ
  3. 新しいアクセストークンを取得
  4. 元のリクエストを再送信
```

### 共通エラーレスポンス

```
【エラーレスポンスの統一形式】

{
  "error": {
    "code": "OUT_OF_STOCK",
    "message": "在庫が不足しています",
    "details": [
      {
        "field": "items[0].quantity",
        "message": "商品ID:1の在庫が3個しかありません"
      }
    ]
  }
}

| フィールド     | 型       | 必須 | 説明                    |
|--------------|---------|-----|------------------------|
| error.code   | string  | ○   | エラーコード（機械判読用）  |
| error.message| string  | ○   | エラーメッセージ（人間用）  |
| error.details| array   | -   | フィールド別のエラー詳細   |

【HTTPステータスコードの使い分け】

| ステータス | 意味              | 使用場面                       |
|----------|------------------|-------------------------------|
| 200      | OK               | 正常に処理が完了（GET/PUT/PATCH）|
| 201      | Created          | リソースの作成成功（POST）       |
| 204      | No Content       | 正常完了、レスポンスボディなし（DELETE）|
| 400      | Bad Request      | リクエストの形式不正             |
| 401      | Unauthorized     | 認証が必要 / トークン無効        |
| 403      | Forbidden        | 認証済みだが権限不足             |
| 404      | Not Found        | リソースが存在しない             |
| 409      | Conflict         | ビジネスロジックの競合           |
| 422      | Unprocessable    | バリデーションエラー             |
| 429      | Too Many Requests| レートリミット超過               |
| 500      | Internal Error   | サーバー内部エラー               |
```

### ページネーション

```
【ページネーションの方式】

■ オフセットベース（一般的）
  GET /products?page=2&per_page=20

  レスポンス:
  {
    "pagination": {
      "current_page": 2,
      "per_page": 20,
      "total_count": 150,
      "total_pages": 8
    }
  }

  利点：ページ番号でジャンプ可能
  欠点：データが多いとOFFSET計算が遅い

■ カーソルベース（大量データ向け）
  GET /products?cursor=eyJpZCI6MTAwfQ&limit=20

  レスポンス:
  {
    "pagination": {
      "next_cursor": "eyJpZCI6MTIwfQ",
      "has_next": true
    }
  }

  利点：大量データでも高速
  欠点：ページ番号ジャンプ不可
```

## 6.6 APIバージョニング

### バージョニングの方法

```
【APIバージョニングの方式】

方式1：URLパス方式（最も一般的）
  GET /v1/products
  GET /v2/products

方式2：ヘッダー方式
  Accept: application/vnd.myapp.v1+json

方式3：クエリパラメータ方式
  GET /products?version=1

【推奨：URLパス方式】
  ・直感的でわかりやすい
  ・ブラウザで直接テスト可能
  ・キャッシュが容易

ベースURL例：
  開発環境：https://dev-api.example.com/v1
  本番環境：https://api.example.com/v1
```

## 6.7 API設計のベストプラクティス

### チェックリスト

```
【API設計のチェックリスト】

□ URL設計
  ├─ □ リソース名は複数形の名詞か
  ├─ □ HTTPメソッドを正しく使い分けているか
  ├─ □ ネストが深すぎないか（3階層以内）
  └─ □ クエリパラメータの命名が統一されているか

□ リクエスト
  ├─ □ 必須/任意パラメータが明記されているか
  ├─ □ データ型・制約が明記されているか
  └─ □ リクエスト例が記載されているか

□ レスポンス
  ├─ □ 成功時のレスポンス形式が定義されているか
  ├─ □ エラー時のレスポンス形式が統一されているか
  ├─ □ HTTPステータスコードが適切か
  └─ □ レスポンス例が記載されているか

□ セキュリティ
  ├─ □ 認証が必要なAPIに認証が設定されているか
  ├─ □ 認可（権限チェック）が考慮されているか
  └─ □ レートリミットが設定されているか

□ ページネーション
  ├─ □ 一覧APIにページネーションがあるか
  └─ □ 最大件数の上限が設定されているか
```

> **実務でのポイント**: API設計書は、OpenAPI（Swagger）仕様で記述することが現代では一般的です。OpenAPIで定義すると、ドキュメント自動生成、モックサーバー作成、クライアントコード自動生成が可能になり、開発効率が大幅に向上します。

## 3.1 正規化理論

### 3.1.1 関数従属性

正規化の理論的基盤となるのが**関数従属性（Functional Dependency: FD）**です。

```
X → Y  「XがYを関数的に決定する」
        「Xの値が決まればYの値が一意に決まる」
```

**例:**
```
社員番号 → 社員名       （社員番号が決まれば社員名が決まる）
社員番号 → 部署コード   （社員番号が決まれば部署コードが決まる）
部署コード → 部署名     （部署コードが決まれば部署名が決まる）
```

**完全関数従属と部分関数従属:**

```
完全関数従属: {X, Y} → Z で、XだけでもYだけでもZは決まらない
部分関数従属: {X, Y} → Z で、XだけでZが決まる（Yなしでも決まる）
```

**推移的関数従属:**
```
X → Y かつ Y → Z のとき、X → Z が推移的に成り立つ
（ただし Y → X は成り立たない）
```

### 3.1.2 正規形の体系

| 正規形 | 条件 | 解消する問題 |
|--------|------|-------------|
| 第1正規形（1NF） | 全属性が原子値（繰り返し項目なし） | 繰り返しグループ |
| 第2正規形（2NF） | 1NF ＋ 部分関数従属の排除 | 複合キーの一部への依存 |
| 第3正規形（3NF） | 2NF ＋ 推移的関数従属の排除 | 非キー属性間の依存 |
| ボイス・コッド正規形（BCNF） | 全ての関数従属 X→Y で X が候補キー | 候補キー以外からの関数従属 |
| 第4正規形（4NF） | BCNF ＋ 多値従属性の排除 | 独立した多値属性 |
| 第5正規形（5NF） | 4NF ＋ 結合従属性の排除 | 3つ以上のリレーションの結合 |

### 3.1.3 正規化の実例

**非正規形（繰り返しあり）:**
```
注文(注文番号, 顧客名, {商品コード, 商品名, 数量, 単価})
```

**第1正規形（1NF）: 繰り返しを排除**
```
注文明細(注文番号, 顧客名, 商品コード, 商品名, 数量, 単価)
主キー: {注文番号, 商品コード}
```

**第2正規形（2NF）: 部分関数従属を排除**
```
関数従属性の分析:
  {注文番号, 商品コード} → 数量        ← 完全関数従属 ✓
  注文番号 → 顧客名                    ← 部分関数従属 ✗
  商品コード → 商品名, 単価            ← 部分関数従属 ✗

分解:
  注文(注文番号, 顧客名)
  商品(商品コード, 商品名, 単価)
  注文明細(注文番号, 商品コード, 数量)
```

**第3正規形（3NF）: 推移的関数従属を排除**

もし「顧客名 → 顧客住所」のような推移的従属があれば:
```
注文番号 → 顧客コード → 顧客住所  （推移的関数従属）

分解:
  注文(注文番号, 顧客コード)
  顧客(顧客コード, 顧客名, 顧客住所)
```

### 3.1.4 正規化と性能のトレードオフ

正規化を進めるほど:
- **メリット:** データの冗長性排除、更新異常の防止、一貫性の維持
- **デメリット:** テーブル結合（JOIN）が増加し、検索性能が低下

**非正規化（デノーマライゼーション）** を行う場合:
- 読み取り性能が重要なシステム（DWH、レポート用DB）
- 結合コストが高い大量データ処理
- 冗長性を許容できる場合

**ポイントまとめ:**
- 正規化は関数従属性の分析が基本
- 午前試験では「第何正規形か」を判定する問題が頻出
- 午後試験では正規化の過程を記述する問題が出る
- 実務では第3正規形まで正規化し、必要に応じて非正規化する

## 3.2 トランザクション制御

### 3.2.1 トランザクションとは

**トランザクション（Transaction）** とは、データベースに対する一連の操作をまとめた論理的な作業単位です。

```
【銀行振込の例】
BEGIN TRANSACTION
  (1) A口座の残高を10,000円減らす
  (2) B口座の残高を10,000円増やす
COMMIT（またはROLLBACK）
```

(1)だけ成功して(2)が失敗すると、お金が消えてしまいます。これを防ぐために、両方まとめて成功か失敗にする必要があります。

### 3.2.2 ACID特性

トランザクションが満たすべき4つの性質:

| 特性 | 英語 | 説明 |
|------|------|------|
| 原子性 | Atomicity | 全て実行されるか、全て取り消されるか（All or Nothing） |
| 一貫性 | Consistency | トランザクション前後でDBの整合性が保たれる |
| 独立性 | Isolation | 複数トランザクションが互いに影響しない |
| 耐久性 | Durability | コミット済みデータは障害があっても失われない |

```
┌─────────────────────────────────────────────┐
│              ACID特性                         │
│                                              │
│  Atomicity ─── 全か無か                      │
│  Consistency ── 整合性の維持                  │
│  Isolation ──── 他のトランザクションと独立     │
│  Durability ─── コミット後は永続的            │
└─────────────────────────────────────────────┘
```

### 3.2.3 コミットとロールバック

```
BEGIN TRANSACTION
  ├── 操作1 (成功)
  ├── 操作2 (成功)
  ├── 操作3 (成功)
  └── COMMIT → 全操作を確定

BEGIN TRANSACTION
  ├── 操作1 (成功)
  ├── 操作2 (失敗!)
  └── ROLLBACK → 全操作を取り消し
```

**セーブポイント（SAVEPOINT）:**
```sql
BEGIN TRANSACTION;
  INSERT INTO 注文 VALUES (...);
  SAVEPOINT sp1;
  INSERT INTO 注文明細 VALUES (...);  -- ここでエラー
  ROLLBACK TO sp1;                     -- sp1まで戻る（注文は残る）
COMMIT;
```

**ポイントまとめ:**
- ACIDの各特性の意味と具体例を説明できるようにする
- 午前試験では「ACID特性のうち〇〇を保証するものはどれか」が定番
- トランザクションは「All or Nothing」が基本原則

## 3.3 同時実行制御（排他制御）

### 3.3.1 同時実行時の問題

複数のトランザクションが同時にデータを操作すると、以下の問題が発生します。

**ダーティリード（Dirty Read）:**
```
T1: X=100を読む → X=200に更新
T2:                 X=200を読む（未コミットのデータ）
T1:                              ROLLBACK（X=100に戻る）
→ T2は存在しないデータ200を読んでしまった
```

**ノンリピータブルリード（Non-repeatable Read）:**
```
T1: X=100を読む
T2:              X=200に更新 → COMMIT
T1:                            X=200を読む（同じデータが変わった!）
```

**ファントムリード（Phantom Read）:**
```
T1: WHERE 部署='営業' で3件取得
T2:                        INSERT（営業部の新社員）→ COMMIT
T1:                        WHERE 部署='営業' で4件取得（件数が変わった!）
```

### 3.3.2 トランザクション分離レベル

| 分離レベル | ダーティリード | ノンリピータブルリード | ファントムリード |
|-----------|:---:|:---:|:---:|
| READ UNCOMMITTED | 発生する | 発生する | 発生する |
| READ COMMITTED | 防止 | 発生する | 発生する |
| REPEATABLE READ | 防止 | 防止 | 発生する |
| SERIALIZABLE | 防止 | 防止 | 防止 |

```
分離レベルが上がるほど:
  安全性: 高い  ←─────────→  低い
  性能:   低い  ←─────────→  高い

  SERIALIZABLE > REPEATABLE READ > READ COMMITTED > READ UNCOMMITTED
```

### 3.3.3 ロック方式

**共有ロック（Sロック / 読取りロック）:**
- 読み取り時に獲得
- 他のトランザクションもSロックを獲得可能
- 他のトランザクションのXロック獲得を阻止

**排他ロック（Xロック / 書込みロック）:**
- 更新時に獲得
- 他のトランザクションはS/Xロックとも獲得不可

| | Sロック保持中 | Xロック保持中 |
|---|:---:|:---:|
| Sロック要求 | 許可 | 拒否 |
| Xロック要求 | 拒否 | 拒否 |

### 3.3.4 2相ロッキングプロトコル（2PL）

トランザクションのロック操作を2つのフェーズに分ける方式です。

```
  ロック数
    ↑
    │    ┌──────┐
    │   ╱        ╲
    │  ╱          ╲
    │ ╱            ╲
    │╱              ╲
    ├───────┬────────┬──→ 時間
    │ 成長相 │ 縮退相  │
    │(ロック │(ロック  │
    │ 獲得)  │ 解放)  │
```

- **成長相（Growing Phase）:** ロックの獲得のみ行う（解放しない）
- **縮退相（Shrinking Phase）:** ロックの解放のみ行う（獲得しない）

2PLを守ると**直列可能性（Serializability）** が保証されます。

### 3.3.5 デッドロック

2つ以上のトランザクションが互いにロック解放を待ち合う状態です。

```
T1: テーブルAをロック → テーブルBをロック要求 → 待ち状態
T2: テーブルBをロック → テーブルAをロック要求 → 待ち状態

  T1 ──待ち──→ T2
  ↑              │
  └──── 待ち ────┘

  → デッドロック発生!
```

**デッドロックの対処法:**
1. **タイムアウト:** 一定時間待っても取得できなければロールバック
2. **デッドロック検出:** 待ちグラフ（Wait-for Graph）で循環を検出し、一方をロールバック
3. **デッドロック予防:** ロック順序を統一する（テーブルA→B→Cの順など）

### 3.3.6 楽観的排他制御と悲観的排他制御

| 方式 | 考え方 | 適用場面 |
|------|--------|---------|
| 悲観的排他制御 | 競合は頻繁に起きる→事前にロック | 更新が多い業務系システム |
| 楽観的排他制御 | 競合はめったに起きない→更新時にチェック | 読み取り中心のWebアプリ |

**楽観的排他制御の仕組み:**
```
1. データ読み取り時にバージョン番号を記録（version=5）
2. 更新時: UPDATE ... WHERE id=1 AND version=5
3. 他のトランザクションが先に更新していたら version≠5 で更新失敗
4. 失敗時はリトライ
```

**ポイントまとめ:**
- 3つの読み取り異常（ダーティ/ノンリピータブル/ファントム）を区別する
- 分離レベルとそれぞれで防止できる問題の対応を暗記する
- デッドロックの発生条件と対処法を説明できるようにする
- 2PLの成長相・縮退相の意味を理解する

## 3.4 障害回復

### 3.4.1 障害の種類

| 障害種類 | 説明 | 例 |
|---------|------|-----|
| トランザクション障害 | 個別トランザクションの失敗 | デッドロック、制約違反 |
| システム障害 | DBMS全体の停止（メモリ内容消失） | OS障害、電源断 |
| メディア障害 | 記憶装置の物理的破損 | ディスク故障 |

### 3.4.2 ログ（ジャーナル）

トランザクションの操作履歴を記録するファイルです。

```
ログレコードの内容:
┌──────────────────────────────────────────┐
│ トランザクションID | 操作種別 | 対象 | 更新前値 | 更新後値 │
├──────────────────────────────────────────┤
│ T1 | UPDATE | X | 100 | 200 │
│ T1 | UPDATE | Y | 300 | 250 │
│ T1 | COMMIT |   |     |     │
│ T2 | UPDATE | X | 200 | 150 │
│ T2 | UPDATE | Z | 500 | 600 │
└──────────────────────────────────────────┘
```

### 3.4.3 WAL（Write-Ahead Logging）プロトコル

**ログ先書きプロトコル:** データベースへの変更を書き込む**前に**、必ずログを書き込む。

```
  ログファイル          データベース
     │                    │
  ①ログ書込み    ─→   ②データ更新
     │                    │
  ③COMMIT記録            │
```

これにより、システム障害が発生しても、ログからデータベースを復旧できます。

### 3.4.4 チェックポイント

メモリ上の更新データを定期的にディスクに書き出すタイミングです。

```
時間軸 ──────────────────────────────────────→
        │             │              │
     チェック       チェック        障害発生
     ポイント1     ポイント2
        │             │              │
 T1: ───┼──COMMIT─────┼──────────────┤  → 回復不要
 T2: ───┼─────────────┼──COMMIT──────┤  → REDO(やり直し)
 T3: ───┼─────────────┼──────────────┤  → UNDO(取り消し)
 T4:    │         開始─┼──────────────┤  → UNDO(取り消し)
```

### 3.4.5 回復処理

**REDO（ロールフォワード / やり直し）:**
- コミット済みだがディスク未反映のトランザクションを再実行
- ログの**更新後値**を使用

**UNDO（ロールバック / 取り消し）:**
- 未コミットのトランザクションの操作を取り消す
- ログの**更新前値**を使用

```
障害回復の手順:
1. チェックポイント以降のログを走査
2. コミット済みトランザクション → REDO
3. 未コミットトランザクション → UNDO
```

### 3.4.6 メディア障害への対応

- **バックアップ**: 定期的にデータベースの全体コピーを取得
- **アーカイブログ**: バックアップ以降のログを保存
- **復旧手順**: バックアップ復元 → アーカイブログの適用（REDO）

```
バックアップ   アーカイブログ
 時点           │
  │    ログ1  ログ2  ログ3
  ├────►├────►├────►│
  │                  │
  復元        REDO適用
              → 障害直前まで復旧
```

**ポイントまとめ:**
- WAL（ログ先書き）の原則を理解する
- REDO/UNDOの対象の判定（コミット済み/未コミット）ができるようにする
- チェックポイントとの関係で回復対象を判断する問題が頻出
- メディア障害はバックアップ＋アーカイブログで対応

## 3.5 分散データベース

### 3.5.1 分散DBの透過性

分散データベースがユーザに対して隠蔽すべき性質:

| 透過性 | 説明 |
|--------|------|
| 位置透過性 | データの物理的な格納場所を意識しない |
| 分割透過性 | データが分割されていることを意識しない |
| 複製透過性 | データが複製されていることを意識しない |
| 障害透過性 | 一部の障害が全体に影響しない |
| 移動透過性 | データの移動を意識しない |

### 3.5.2 2相コミット（Two-Phase Commit: 2PC）

分散環境で全サイトの一貫性を保証するプロトコルです。

```
          調整者(Coordinator)
             │
      ┌──────┼──────┐
      │      │      │
    参加者1 参加者2 参加者3

【第1相: 投票フェーズ】
調整者 → 参加者: 「PREPARE（準備せよ）」
参加者 → 調整者: 「OK（準備完了）」or「NG（拒否）」

【第2相: 決定フェーズ】
全員OK → 調整者 → 参加者: 「COMMIT」
1人でもNG → 調整者 → 参加者: 「ROLLBACK」
```

**2PCの課題:**
- 調整者が障害を起こすとブロッキングが発生する可能性
- 通信コストが大きい（分散環境の性能ボトルネック）
- **3相コミット（3PC）** はブロッキングを防ぐが、さらに複雑

### 3.5.3 CAP定理

分散システムでは以下の3つを同時に満たすことはできない（最大2つ）:

```
       Consistency（一貫性）
          ╱╲
         ╱  ╲
        ╱ CA ╲
       ╱──────╲
      ╱   CP    ╲
     ╱────────────╲
Availability      Partition Tolerance
（可用性）        （分断耐性）
         AP
```

| 組合せ | 特徴 | 代表例 |
|--------|------|--------|
| CA | 一貫性＋可用性（分断耐性なし） | 従来のRDBMS |
| CP | 一貫性＋分断耐性（可用性を犠牲） | HBase, MongoDB(設定次第) |
| AP | 可用性＋分断耐性（一貫性を犠牲） | DynamoDB, Cassandra |

### 3.5.4 レプリケーション

データを複数のサイトに複製して可用性・性能を向上させる技術です。

| 方式 | 説明 | 特徴 |
|------|------|------|
| 同期レプリケーション | 全レプリカに即時反映 | 一貫性◎、性能△ |
| 非同期レプリケーション | 遅延を許容して反映 | 性能◎、一貫性△ |
| マスタ・スレーブ | 書込みはマスタ、読取りはスレーブ | 読取りスケーラビリティ◎ |
| マルチマスタ | 複数ノードで書込み可能 | 可用性◎、競合管理が複雑 |

**ポイントまとめ:**
- 分散DBの透過性の種類を覚える
- 2PCの2つのフェーズの流れを説明できるようにする
- CAP定理の3要素と「同時に2つまで」のルールを理解する

## 3.6 NoSQLデータベース

### 3.6.1 NoSQLとは

**NoSQL（Not Only SQL）** は、RDBMSの制約を緩和し、大規模データ・高速アクセスに特化したデータベースの総称です。

### 3.6.2 NoSQLの分類

| 種類 | データモデル | 代表例 | 適用場面 |
|------|------------|--------|---------|
| キーバリューストア | キーと値のペア | Redis, DynamoDB | キャッシュ、セッション管理 |
| ドキュメント指向 | JSON/BSON文書 | MongoDB, CouchDB | Webアプリ、CMS |
| カラム指向 | 列ファミリ | Cassandra, HBase | 大量データの集計・分析 |
| グラフDB | ノードとエッジ | Neo4j, Amazon Neptune | SNS、推薦システム |

### 3.6.3 RDBMSとNoSQLの比較

| 項目 | RDBMS | NoSQL |
|------|-------|-------|
| データモデル | テーブル（固定スキーマ） | 柔軟（スキーマレス） |
| スケーリング | 垂直（スケールアップ） | 水平（スケールアウト） |
| 一貫性 | ACID（強い一貫性） | BASE（結果整合性） |
| クエリ | SQL（標準化） | 製品固有 |
| 適用場面 | 業務系（銀行、受発注） | Web系（SNS、IoT） |

### 3.6.4 BASE特性

NoSQLが採用する一貫性モデル:

| 特性 | 英語 | 説明 |
|------|------|------|
| BA | Basically Available | 基本的に利用可能 |
| S | Soft state | 状態が常に一貫しているとは限らない |
| E | Eventual consistency | 最終的に一貫性が保たれる |

```
ACID（厳密な一貫性） ←──────────→ BASE（結果整合性）
  銀行の口座管理              SNSのタイムライン
  在庫管理                    アクセスカウンタ
```

**ポイントまとめ:**
- NoSQLの4分類とそれぞれの代表例・適用場面を覚える
- RDBMSとNoSQLの使い分けの判断基準を理解する
- ACID vs BASEの違いは午前試験で頻出

## 3.7 データベース設計

### 3.7.1 ER図（Entity-Relationship Diagram）

```
┌──────┐    1    N   ┌──────┐    M    N   ┌──────┐
│ 部署  │────────────│ 社員  │────────────│ プロジェクト│
│      │            │      │            │          │
│部署ID │            │社員ID │            │プロジェクトID│
│部署名 │            │社員名 │            │プロジェクト名│
└──────┘            │部署ID │            └──────────┘
                     └──────┘
```

**カーディナリティ（多重度）:**
- 1:1 → 一人の社長に一つの会社
- 1:N → 一つの部署に複数の社員
- M:N → 社員とプロジェクトの多対多関係

**M:N関係の解消:**
```
社員 ──M:N── プロジェクト

↓ 連関エンティティ（中間テーブル）で解消

社員 ──1:N── 社員プロジェクト ──N:1── プロジェクト
              (社員ID, プロジェクトID, 役割, 参加日)
```

### 3.7.2 論理設計と物理設計

| 段階 | 内容 |
|------|------|
| 概念設計 | ER図の作成、エンティティと関連の洗い出し |
| 論理設計 | ER図をリレーショナルモデルに変換、正規化 |
| 物理設計 | インデックス設計、パーティション、物理配置 |

### 3.7.3 インデックス設計

| インデックス種類 | 特徴 |
|----------------|------|
| B木インデックス | 範囲検索に強い、最も一般的 |
| ハッシュインデックス | 等値検索が高速、範囲検索は不可 |
| ビットマップインデックス | カーディナリティが低い列向け（性別等） |
| 関数インデックス | 式や関数の結果にインデックスを作成 |
| 複合インデックス | 複数列を組み合わせたインデックス |

**インデックスの設計指針:**
- WHERE句やJOIN条件に頻繁に使われる列に作成
- カーディナリティが高い列ほど効果的
- 更新頻度が高いテーブルでは作りすぎに注意（更新コスト増大）

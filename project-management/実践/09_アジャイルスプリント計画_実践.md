## 課題の要件

以下のシナリオを読み、アジャイルスプリント計画を作成してください。

### シナリオ

あなたはスクラムマスターです。社内の経費精算システムをアジャイル（スクラム）で開発します。

**プロジェクト概要：**
- スプリント期間: 2週間
- チーム: 開発者3名（フロント1名、バック1名、フルスタック1名）
- 過去の実績ベロシティ: なし（初回プロジェクト、暫定15SP）
- 総期間: 6スプリント（3ヶ月）
- プロダクトオーナー（PO）: 経理部のリーダー

**プロダクトオーナーからの要求一覧（未整理）：**

| # | 要求内容 |
|---|---------|
| 1 | 社員が経費を申請できる（日付、金額、カテゴリ、領収書画像） |
| 2 | 上長が申請を承認/却下できる |
| 3 | 経理部が承認済み申請を一括で集計できる |
| 4 | 社員が自分の申請履歴を確認できる |
| 5 | ログイン/ログアウトができる |
| 6 | 月次の経費レポート（部署別・カテゴリ別）を出力できる |
| 7 | 申請時にOCRで領収書を自動読み取りできる |
| 8 | Slackに承認通知が飛ぶ |
| 9 | CSVで経費データをエクスポートできる |
| 10 | ダッシュボードで経費の傾向をグラフ表示できる |
| 11 | 申請にコメントを付けられる（差し戻し理由など） |
| 12 | 経費カテゴリのマスタを管理画面で編集できる |
| 13 | 予算上限を設定し、超過時にアラートを出す |
| 14 | モバイル対応（スマホで申請・承認できる） |

### 作成すべきもの

1. **ユーザーストーリーへの変換**: 各要求を「〜として、〜したい。なぜなら〜。」の形式に変換
2. **MoSCoW優先度付け**: Must / Should / Could / Won't に分類
3. **ストーリーポイント見積もり**: フィボナッチ数列（1, 2, 3, 5, 8, 13）で見積もり
4. **プロダクトバックログ**: 優先順位でソートした一覧
5. **Sprint 1〜3の計画**: 各スプリントに割り当てるストーリーとタスク分解

## ステップガイド

<details>
<summary>ステップ1：ユーザーストーリーに変換する</summary>

ユーザーストーリーの書式に従って変換します。

```
フォーマット:
  [ユーザーの種類]として、
  [達成したいこと]したい。
  なぜなら[理由・価値]だからだ。

例：要求1の場合
  社員として、
  経費を申請したい。
  なぜなら、立て替えた費用を速やかに精算したいからだ。
```

**受入基準（Acceptance Criteria）も書きましょう：**
```
  受入基準:
  - 日付、金額、カテゴリ、メモを入力して申請できること
  - 領収書画像を1枚以上添付できること
  - 入力内容のバリデーションが行われること
  - 申請後に確認メールが送信されること
```

</details>

<details>
<summary>ステップ2：MoSCoW優先度を付ける</summary>

POの視点で優先度を判断します。

```
判断基準：

Must（なければリリースできない）:
  - システムの基本機能として必須
  - これがないと業務が回らない
  → ログイン、申請、承認、集計

Should（重要だが、なくてもリリースは可能）:
  - ユーザー体験を大幅に向上させる
  - 次のスプリントで対応しても許容できる
  → 申請履歴、CSV出力、コメント

Could（あれば嬉しい）:
  - 付加価値だが、ないことによる業務影響は小さい
  → ダッシュボード、Slack通知

Won't（今回は対象外）:
  - 技術的に複雑、または今回のスコープから外れる
  → OCR読み取り（技術的不確実性が高い）
```

</details>

<details>
<summary>ステップ3：ストーリーポイントを見積もる</summary>

相対見積もりの考え方でポイントを付けます。

```
基準ストーリー:
  「ログインできる」= 2SP（シンプルなCRUD + 認証）

これを基準に他のストーリーを相対的に見積もる:

  1SP: 非常にシンプル（設定変更、UI微修正程度）
  2SP: シンプル（単一のCRUD操作）
  3SP: やや複雑（複数画面、バリデーション付き）
  5SP: 複雑（複数テーブル連携、ワークフロー）
  8SP: 非常に複雑（外部連携、複雑なロジック）
  13SP: 巨大（分割を検討すべき）
```

**13以上のストーリーは分割しましょう。** 大きすぎるストーリーは見積もりの精度が低く、1スプリントで完了しない可能性があります。

</details>

<details>
<summary>ステップ4：スプリントに割り当てる</summary>

```
割り当ての原則：
  1. ベロシティ（15SP）を超えない
  2. Must → Should → Could の順で入れる
  3. 依存関係を考慮する（例: ログインが先、申請が後）
  4. 各スプリントで「動くソフトウェア」をリリースできること
  5. スプリントゴール（そのスプリントで達成する1文の目標）を設定
```

</details>

```

  □ 全受入基準を満たすこと
  □ コードレビュー済み
  □ 単体テスト・結合テストがパスすること
```

**初心者向けとの違い：**
- **受入基準とDoD**: ストーリーが「完了」とみなされる条件を明文化することで、品質のブレを防ぎます
- **スクラムイベントの具体的設計**: イベントの時間配分と内容を事前に決めておくことで、チームの生産性が上がります
- **ベロシティ追跡と予測**: 実績データに基づいてリリース可能なスコープを予測し、早期に調整判断ができます
- **3つのシナリオ分析**: 楽観・現実・悲観の3パターンでリリース計画を検討するのが実務的なアプローチです

# 第6章 演習：フックとカスタマイズ

## 基本問題

### 問題6-1：フックの種類（穴埋め）
以下の文章の空欄（ア）〜（オ）に当てはまる語句を答えてください。

> Claude Codeのフック（Hooks）とは、特定の（ア）に自動的に実行される（イ）のことです。
> フックには主に3つのタイプがあります。（ウ）はClaude Codeがユーザーに応答を返す前に実行されます。
> （エ）はClaude Codeがツールを呼び出す前に実行されます。（オ）はツールの実行が完了した後に実行されます。

**期待される出力例：**
```
（ア）イベント（タイミング）
（イ）シェルコマンド（カスタムスクリプト）
（ウ）PreToolUse
（エ）PostToolUse
（オ）Notification
```

<details>
<summary>ヒント</summary>

- フックはClaude Codeの動作の特定のポイントで介入するための仕組みです
- フックの名前はその実行タイミングを表しています
- 「Pre」は「前」、「Post」は「後」を意味します

</details>

<details>
<summary>解答例</summary>

```
（ア）イベント（タイミング）
（イ）シェルコマンド（カスタムスクリプト）
（ウ）PreToolUse
（エ）PostToolUse
（オ）Notification
```

**解説：**
Claude Codeのフックには以下の種類があります：

| フック種類 | 実行タイミング | 主な用途 |
|---|---|---|
| PreToolUse | ツール実行前 | 実行前のバリデーション、ブロック |
| PostToolUse | ツール実行後 | 結果の検証、ログ記録 |
| Notification | 通知時 | ユーザーへの入力待ちなどの通知 |

- フックはシェルコマンドとして実行されるため、任意のスクリプトを呼び出せます
- フックの終了コードによって動作を制御できます（例：非ゼロで操作をブロック）

</details>

### 問題6-2：フックのタイプ分類
以下のユースケースに最適なフックタイプを「PreToolUse」「PostToolUse」「Notification」から選んでください。

1. Claude Codeがファイルを書き換える前にバックアップを取る
2. コマンド実行後にログファイルに記録する
3. Claude Codeが入力を待っている時にデスクトップ通知を送る
4. 特定のディレクトリへの書き込みをブロックする
5. ファイル編集後にリンターを自動実行する
6. 長時間処理の完了時にSlackに通知する

**期待される出力例：**
```
1. PreToolUse
2. PostToolUse
3. Notification
4. PreToolUse
5. PostToolUse
6. Notification
```

<details>
<summary>ヒント</summary>

- 「実行前に何かする」「ブロックする」→ PreToolUse
- 「実行後に何かする」「結果を検証する」→ PostToolUse
- 「ユーザーに知らせる」→ Notification
- PreToolUseで非ゼロの終了コードを返すと、ツール実行がブロックされます

</details>

<details>
<summary>解答例</summary>

```
1. PreToolUse   ← ファイル書き換え「前」にバックアップを取る処理です
2. PostToolUse  ← コマンド実行「後」のログ記録です
3. Notification ← 入力待ちの通知はNotificationフックの典型的な用途です
4. PreToolUse   ← 書き込みを「ブロック」するにはPreToolUseで非ゼロを返します
5. PostToolUse  ← ファイル編集「後」にリンターを実行する処理です
6. Notification ← 完了通知はNotificationフックで行います
```

**フックタイプの使い分け：**
```
┌─────────────────────────────────────────────┐
│ PreToolUse  : ツール実行を「許可/拒否」する  │
│               実行前の「準備処理」を行う      │
│                                              │
│ PostToolUse : 実行結果の「検証」を行う        │
│               実行後の「後処理」を行う        │
│                                              │
│ Notification: ユーザーへの「通知」を行う      │
│               外部サービスへの「通知」を行う  │
└─────────────────────────────────────────────┘
```

</details>

### 問題6-3：フック設定の読解
以下のフック設定を読み、各問いに答えてください。

```json
{
  "hooks": {
    "PreToolUse": [
      {
        "matcher": "Edit|Write",
        "command": "bash /scripts/backup.sh \"$CLAUDE_FILE_PATH\""
      }
    ],
    "PostToolUse": [
      {
        "matcher": "Bash",
        "command": "echo \"$(date): Bash command executed\" >> /tmp/claude-audit.log"
      }
    ],
    "Notification": [
      {
        "matcher": "",
        "command": "notify-send 'Claude Code' \"$CLAUDE_NOTIFICATION\""
      }
    ]
  }
}
```

(1) PreToolUseフックはどのツールが使われるときに実行されますか。
(2) PreToolUseフックは何をしますか。
(3) PostToolUseフックはどのツールの後に実行されますか。
(4) PostToolUseフックは何をしますか。
(5) Notificationフックの `matcher` が空文字列である意味は何ですか。

**期待される出力例：**
```
(1) Edit または Write ツール
(2) ファイルのバックアップスクリプトを実行する
(3) Bash ツール
(4) 監査ログにタイムスタンプ付きで記録する
(5) すべての通知に対して実行される（フィルタなし）
```

<details>
<summary>ヒント</summary>

- `matcher` は正規表現パターンで、対象となるツール名をフィルタします
- `|` は正規表現の「または」を意味します
- `matcher` が空文字列の場合は全てのイベントにマッチします
- 環境変数 `$CLAUDE_FILE_PATH` にはツールが操作するファイルのパスが入ります

</details>

<details>
<summary>解答例</summary>

```
(1) Edit（ファイル編集）または Write（ファイル書き込み）ツール
    matcher の "Edit|Write" は正規表現で「EditまたはWrite」を意味します

(2) 対象ファイルのバックアップスクリプト（/scripts/backup.sh）を実行します
    環境変数 $CLAUDE_FILE_PATH でファイルパスが渡されます

(3) Bash ツール（シェルコマンド実行）の後に実行されます

(4) /tmp/claude-audit.log に日時付きで「Bashコマンドが実行された」という
    ログを追記します（監査ログの記録）

(5) matcher が空文字列の場合、すべての通知イベントに対して
    フックが実行されます（フィルタリングなし）
```

**環境変数の一覧：**

| 環境変数 | 説明 |
|---|---|
| `$CLAUDE_FILE_PATH` | ツールが操作するファイルのパス |
| `$CLAUDE_NOTIFICATION` | 通知メッセージの内容 |
| `$CLAUDE_TOOL_NAME` | 実行されるツールの名前 |
| `$CLAUDE_TOOL_INPUT` | ツールに渡される入力（JSON） |

</details>

### 問題6-4：設定ファイルの階層
Claude Codeの設定ファイルには複数の階層があります。以下の設定ファイルを優先度が高い順に並べてください。

- (A) `~/.claude/settings.json`（ユーザー設定）
- (B) `.claude/settings.json`（プロジェクト設定）
- (C) `.claude/settings.local.json`（プロジェクトローカル設定）
- (D) エンタープライズポリシー設定

**期待される出力例（優先度が高い順）：**
```
1. (D) エンタープライズポリシー設定
2. (C) .claude/settings.local.json
3. (B) .claude/settings.json
4. (A) ~/.claude/settings.json
```

<details>
<summary>ヒント</summary>

- エンタープライズポリシーは組織全体のルールなので最優先です
- ローカル設定（`.local.json`）はGitにコミットしない個人設定です
- プロジェクト設定はチーム共有、ユーザー設定は個人全体の設定です
- より具体的・限定的なスコープの設定が優先されます

</details>

<details>
<summary>解答例</summary>

```
優先度（高い順）：
1. (D) エンタープライズポリシー設定    ← 最優先（組織のルール）
2. (C) .claude/settings.local.json    ← プロジェクト内の個人設定
3. (B) .claude/settings.json          ← プロジェクト共有設定
4. (A) ~/.claude/settings.json        ← ユーザーのグローバル設定
```

**設定ファイルの階層図：**
```
┌──────────────────────────────────────┐
│ エンタープライズポリシー             │ ← 最優先（上書き不可）
├──────────────────────────────────────┤
│ .claude/settings.local.json          │ ← Git管理外の個人設定
├──────────────────────────────────────┤
│ .claude/settings.json                │ ← Git管理のチーム共有設定
├──────────────────────────────────────┤
│ ~/.claude/settings.json              │ ← ユーザーのグローバル設定
└──────────────────────────────────────┘
```

**各設定ファイルの役割：**
- **エンタープライズポリシー**：セキュリティや利用制限などの組織ルール
- **settings.local.json**：個人のAPIキーやローカル固有の設定（`.gitignore` で除外）
- **settings.json（プロジェクト）**：チームで共有するプロジェクト標準設定
- **settings.json（ユーザー）**：全プロジェクト共通の個人設定

</details>

### 問題6-5：フック設定の作成
以下の要件を満たすフック設定JSONを作成してください。

**要件：**
1. ファイル編集時（Edit、Writeツール）に、対象ファイルが `src/` ディレクトリ以外の場合はブロックする
2. Bashツール実行後に、`npm test` を自動で実行する
3. 通知時にmacOSの通知センターに通知を送る

**期待される出力例：**
```json
{
  "hooks": {
    "PreToolUse": [
      {
        "matcher": "Edit|Write",
        "command": "..."
      }
    ],
    "PostToolUse": [...],
    "Notification": [...]
  }
}
```

<details>
<summary>ヒント</summary>

- PreToolUseフックでブロックするには、非ゼロの終了コードを返します
- `$CLAUDE_FILE_PATH` でファイルパスを取得できます
- macOSの通知には `osascript` コマンドが使えます
- bashの条件文で `[[ ! "$path" =~ ^src/ ]]` のようにパスを検証できます

</details>

<details>
<summary>解答例</summary>

```json
{
  "hooks": {
    "PreToolUse": [
      {
        "matcher": "Edit|Write",
        "command": "bash -c '[[ \"$CLAUDE_FILE_PATH\" =~ ^src/ ]] || (echo \"src/ ディレクトリ以外への書き込みはブロックされました: $CLAUDE_FILE_PATH\" && exit 1)'"
      }
    ],
    "PostToolUse": [
      {
        "matcher": "Bash",
        "command": "cd \"$CLAUDE_PROJECT_DIR\" && npm test 2>/dev/null || true"
      }
    ],
    "Notification": [
      {
        "matcher": "",
        "command": "osascript -e 'display notification \"$CLAUDE_NOTIFICATION\" with title \"Claude Code\"'"
      }
    ]
  }
}
```

**各フックの解説：**

1. **PreToolUse（ファイル書き込みのブロック）**
   - `$CLAUDE_FILE_PATH` がsrc/で始まるかチェックします
   - 条件に合わない場合は `exit 1` で非ゼロを返し、ツール実行をブロックします
   - エラーメッセージを出力してユーザーに理由を伝えます

2. **PostToolUse（テスト自動実行）**
   - Bashツール実行後にプロジェクトディレクトリで `npm test` を実行します
   - `2>/dev/null || true` でテスト失敗時もフック自体はエラーにしません

3. **Notification（デスクトップ通知）**
   - macOSの `osascript` を使って通知センターに通知を表示します
   - Linuxの場合は `notify-send` コマンドを使います

**Linux版の通知フック：**
```json
{
  "matcher": "",
  "command": "notify-send 'Claude Code' \"$CLAUDE_NOTIFICATION\""
}
```

</details>

### 問題6-6：マッチャーパターンの設計
以下の要件に対応する `matcher` の正規表現パターンを作成してください。

1. すべてのツールにマッチする
2. ファイル操作系ツール（Edit、Write、Read）にマッチする
3. Bashツールのみにマッチする
4. Bash以外のすべてのツールにマッチする
5. 名前に「File」を含むツールにマッチする

**期待される出力例：**
```
1. ""（空文字列）
2. "Edit|Write|Read"
3. "^Bash$"
4. "^(?!Bash$)"
5. "File"
```

<details>
<summary>ヒント</summary>

- 空文字列はすべてにマッチします
- `|` は「または」の正規表現演算子です
- `^` は先頭、`$` は末尾にマッチする正規表現アンカーです
- `(?!...)` は否定先読み（negative lookahead）です

</details>

<details>
<summary>解答例</summary>

```
1. ""           ← 空文字列は全てのツールにマッチします
2. "Edit|Write|Read" ← パイプ（|）でOR条件を記述します
3. "^Bash$"     ← 完全一致で "Bash" のみにマッチします
4. "^(?!Bash$)" ← 否定先読みで "Bash" 以外にマッチします
5. "File"       ← 部分一致で "File" を含むツール名にマッチします
```

**matcherパターンの考え方：**

| パターン | 意味 | マッチ例 |
|---|---|---|
| `""` | 全て | すべてのツール |
| `"Bash"` | 部分一致 | Bash |
| `"^Bash$"` | 完全一致 | Bash のみ |
| `"Edit\|Write"` | OR | Edit, Write |
| `"^(?!Bash)"` | 否定 | Bash以外 |
| `".*File.*"` | 含む | ReadFile, WriteFile 等 |

**注意事項：**
- matcherは正規表現としてツール名に対して評価されます
- 部分一致でよい場合はシンプルに文字列を書けば十分です
- 完全一致が必要な場合は `^` と `$` のアンカーを使いましょう

</details>

## チャレンジ問題

### 問題6-7：実践的なフック構成の設計
以下のチーム開発プロジェクトの要件を満たすフック構成を設計してください。

**プロジェクトの状況：**
- フロントエンド（React）+ バックエンド（Node.js）のWebアプリ
- ESLint + Prettier でコード品質を管理
- テストは Jest で実行
- チーム5人で開発、コーディング規約あり

**要件：**
1. ファイル編集後に ESLint を自動実行し、エラーがあれば Claude にフィードバックする
2. `node_modules/`、`dist/`、`.env` への書き込みをブロックする
3. Bashツールで `rm -rf` を含むコマンドの実行をブロックする
4. 通知をSlack Webhookに送信する

**回答に含めること：**
- 完全なフック設定JSON
- 各フックの説明
- チーム共有すべき設定と個人設定の区分け

<details>
<summary>ヒント</summary>

- ESLintの自動実行はPostToolUseフックで行います
- ファイルパスの検証はPreToolUseフックで行います
- `$CLAUDE_TOOL_INPUT` からコマンド内容を取得できます
- Slack Webhookへの通知は `curl` コマンドで行えます
- プロジェクトスコープにはSlack WebhookのURLを含めないようにしましょう

</details>

<details>
<summary>解答例</summary>

**プロジェクトスコープ（`.claude/settings.json`）に入れる設定：**

```json
{
  "hooks": {
    "PreToolUse": [
      {
        "matcher": "Edit|Write",
        "command": "bash -c 'BLOCKED_PATHS=\"node_modules/|dist/|\\.env\"; if echo \"$CLAUDE_FILE_PATH\" | grep -qE \"$BLOCKED_PATHS\"; then echo \"保護されたパスへの書き込みはブロックされました: $CLAUDE_FILE_PATH\" && exit 1; fi'"
      },
      {
        "matcher": "Bash",
        "command": "bash -c 'if echo \"$CLAUDE_TOOL_INPUT\" | grep -q \"rm -rf\"; then echo \"rm -rf コマンドの実行はブロックされました\" && exit 1; fi'"
      }
    ],
    "PostToolUse": [
      {
        "matcher": "Edit|Write",
        "command": "bash -c 'if [[ \"$CLAUDE_FILE_PATH\" =~ \\.(js|jsx|ts|tsx)$ ]]; then npx eslint \"$CLAUDE_FILE_PATH\" 2>&1 || true; fi'"
      }
    ]
  }
}
```

**ユーザースコープ（`~/.claude/settings.json`）に入れる設定：**

```json
{
  "hooks": {
    "Notification": [
      {
        "matcher": "",
        "command": "curl -s -X POST -H 'Content-Type: application/json' -d '{\"text\": \"Claude Code: '\"$CLAUDE_NOTIFICATION\"'\"}' $SLACK_WEBHOOK_URL"
      }
    ]
  }
}
```

**各フックの説明：**

| # | フック | 対象 | 動作 | スコープ |
|---|---|---|---|---|
| 1 | PreToolUse | Edit/Write | 保護パスへの書き込みブロック | project |
| 2 | PreToolUse | Bash | rm -rf コマンドのブロック | project |
| 3 | PostToolUse | Edit/Write | JS/TSファイルにESLint実行 | project |
| 4 | Notification | 全て | Slack Webhook通知 | user |

**設計のポイント：**
- セキュリティ関連のフック（ブロック系）はプロジェクトスコープで全員に適用します
- Slack WebhookのURLは個人の認証情報なのでユーザースコープに置きます
- ESLintのフック結果はClaude Codeにフィードバックされ、自動的に修正を試みます
- `|| true` を付けることで、ESLintエラー時もフック自体は成功扱いにしています

**補足（環境変数の管理）：**
- `SLACK_WEBHOOK_URL` は `~/.zshrc` や `~/.bashrc` で環境変数として設定します
- 設定ファイルにURLを直接書かないことでセキュリティを確保します

</details>

### 問題6-8：フックのデバッグと改善
以下のフック設定にはいくつかの問題があります。問題点を指摘し、改善版を作成してください。

```json
{
  "hooks": {
    "PreToolUse": [
      {
        "matcher": "Edit",
        "command": "cp $CLAUDE_FILE_PATH /backups/"
      }
    ],
    "PostToolUse": [
      {
        "matcher": "",
        "command": "npm test && npm run lint && npm run build"
      }
    ],
    "Notification": [
      {
        "matcher": "Bash",
        "command": "echo $CLAUDE_NOTIFICATION > /tmp/notify.txt"
      }
    ]
  }
}
```

**問いかけ：**
(1) 各フックの問題点を指摘してください（最低5つ）
(2) 改善版の設定JSONを作成してください
(3) 改善理由をそれぞれ説明してください

<details>
<summary>ヒント</summary>

- 環境変数をクォートしないとスペースを含むパスで問題が起きます
- 全ツールに対して重い処理を毎回実行するのはパフォーマンスに影響します
- バックアップ先ディレクトリの存在確認をしていません
- Notificationフックのmatcherはツール名ではなく通知のフィルタです
- `&&` でチェインすると前の処理が失敗した時点で後続が実行されません

</details>

<details>
<summary>解答例</summary>

**(1) 問題点の指摘：**

| # | 問題点 | 影響 |
|---|---|---|
| 1 | `$CLAUDE_FILE_PATH` がクォートされていない | スペースを含むパスでコマンドが壊れる |
| 2 | `/backups/` ディレクトリの存在確認がない | ディレクトリがなければ cp が失敗する |
| 3 | PostToolUseの matcher が空（全ツールに適用） | 全ツール実行後にtest/lint/buildが走り、非常に重い |
| 4 | `npm test && npm run lint && npm run build` を毎回実行 | ビルドまで毎回実行するのは過剰で時間がかかる |
| 5 | Notificationの matcher に "Bash" が指定されている | Notificationのmatcherはツール名フィルタではなく用途が不適切 |
| 6 | `echo $CLAUDE_NOTIFICATION` がクォートされていない | 特殊文字を含む通知で問題が起きる |
| 7 | Editのみでバックアップし、Writeは対象外 | Writeでファイルを作成・上書きする場合はバックアップされない |

**(2) 改善版：**

```json
{
  "hooks": {
    "PreToolUse": [
      {
        "matcher": "Edit|Write",
        "command": "bash -c 'if [ -f \"$CLAUDE_FILE_PATH\" ]; then mkdir -p /tmp/claude-backups && cp \"$CLAUDE_FILE_PATH\" \"/tmp/claude-backups/$(basename \"$CLAUDE_FILE_PATH\").$(date +%Y%m%d%H%M%S).bak\"; fi'"
      }
    ],
    "PostToolUse": [
      {
        "matcher": "Edit|Write",
        "command": "bash -c 'if [[ \"$CLAUDE_FILE_PATH\" =~ \\.(js|jsx|ts|tsx)$ ]]; then npx eslint \"$CLAUDE_FILE_PATH\" 2>&1 || true; fi'"
      }
    ],
    "Notification": [
      {
        "matcher": "",
        "command": "bash -c 'echo \"$(date): $CLAUDE_NOTIFICATION\" >> /tmp/claude-notifications.log'"
      }
    ]
  }
```

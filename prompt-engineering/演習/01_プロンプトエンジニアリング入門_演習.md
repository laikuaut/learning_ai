## 基本問題

### 問題1：LLMの特性の理解

以下の状況がLLMの4つの特性（確率的生成 / 知識のカットオフ / ハルシネーション / コンテキスト依存）のどれに該当するか答えてください。

1. 同じプロンプトを2回送ったが、微妙に異なる回答が返ってきた
2. 「2026年3月の最新ニュースを教えて」と聞いたが、2025年5月までの情報しか返ってこなかった
3. 実在しない論文タイトルと著者名をもっともらしく生成した
4. 「短く教えて」と言ったらそっけない回答が返り、「詳しく教えて」と言ったら丁寧な説明が返った

**期待される出力例：**
```
1. 確率的生成
2. 知識のカットオフ
3. ハルシネーション
4. コンテキスト依存
```

<details>
<summary>解答例</summary>

```
1. 確率的生成
   → LLMは次のトークンを確率的に選択するため、同じ入力でも
     異なる出力が生成される場合がある。temperatureパラメータで
     制御可能（低くすると安定、高くすると多様性が増す）。

2. 知識のカットオフ
   → LLMの学習データには時点の制限がある。
     カットオフ以降の情報を回答するにはRAG等で外部情報を提供する必要がある。

3. ハルシネーション
   → LLMがもっともらしいが事実に基づかない情報を生成する現象。
     特に固有名詞（人名、論文名等）で発生しやすい。

4. コンテキスト依存
   → プロンプトの書き方により、出力の品質・長さ・トーンが大きく変わる。
     これがプロンプトエンジニアリングが重要な理由。
```
</details>

### 問題2：コンテキストウィンドウの計算

以下の状況で、コンテキストウィンドウを効率的に使えているか判断してください。

使用モデル：Claude 3.5 Sonnet（コンテキストウィンドウ：200Kトークン）

| ケース | 入力トークン | 出力トークン | 合計 | 効率的？ |
|--------|-----------|-----------|------|---------|
| A | 1,000 | 500 | 1,500 | ？ |
| B | 190,000 | 5,000 | 195,000 | ？ |
| C | 150,000 | 60,000 | 210,000 | ？ |
| D | 50,000 | 2,000 | 52,000 | ？ |

<details>
<summary>解答例</summary>

```
A: 効率的 ✅
   合計1,500トークンで200Kの0.75%しか使っていない。
   シンプルな質問には十分。

B: 効率的（ただし注意が必要）⚠️
   200Kの97.5%を使用。動作するが、Lost in the Middle問題
   （長文の中間部分の情報が無視されやすい）に注意。
   本当にすべての情報が必要か検討すべき。

C: エラー ❌
   合計210,000トークンで、コンテキストウィンドウ（200K）を超過。
   入力の削減か、出力の制限が必要。

D: 効率的 ✅
   200Kの26%を使用。長文のドキュメントを入力しつつ、
   十分な出力の余地がある。
```
</details>

### 問題3：temperatureの使い分け

以下のタスクに適したtemperature設定（低：0〜0.3 / 中：0.4〜0.7 / 高：0.8〜1.0）を選んでください。

1. コードのバグ修正
2. キャッチコピーのアイデア出し
3. 法律文書の要約
4. 創作小説の書き出し
5. CSVデータのJSON変換

<details>
<summary>解答例</summary>

```
1. コードのバグ修正 → 低（0〜0.3）
   理由：正確性が最重要。毎回同じ正しい修正を返してほしい。

2. キャッチコピーのアイデア出し → 高（0.8〜1.0）
   理由：多様で創造的なアイデアが欲しい。
   毎回異なるバリエーションが出る方が有用。

3. 法律文書の要約 → 低（0〜0.3）
   理由：正確性と一貫性が重要。原文に忠実な要約が必要。

4. 創作小説の書き出し → 高（0.8〜1.0）
   理由：創造性と表現の多様性が求められる。

5. CSVデータのJSON変換 → 低（0〜0.2）
   理由：決定的な変換タスク。正確に変換されることが必須。
   temperature 0が望ましい。
```
</details>

## 応用問題

### 問題4：プロンプトの改善

以下の「悪いプロンプト」を改善してください。改善理由も述べてください。

```
悪いプロンプト1：
「プログラミングについて教えてください」

悪いプロンプト2：
「このエラーを直してください」

悪いプロンプト3：
「レポートを書いてください」
```

<details>
<summary>解答例</summary>

```
■ プロンプト1の改善

改善前：「プログラミングについて教えてください」
改善後：「Python初心者がWebアプリ開発を始めるために、
最初に学ぶべきトピックを優先度順に5つ挙げてください。
各トピックについて、なぜ重要か、学習にかかる目安時間も
教えてください。」

改善理由：
・対象言語（Python）を特定した
・対象者（初心者）を明示した
・目的（Webアプリ開発）を具体化した
・出力形式（5つ、優先度順）を指定した
・必要な情報（理由、時間）を具体的に要求した

■ プロンプト2の改善

改善前：「このエラーを直してください」
改善後：「以下のPythonコードを実行すると
TypeError: 'NoneType' object is not subscriptable
というエラーが発生します。

コード：
```python
（対象のコード）
```

エラーの原因と修正方法を教えてください。
修正後のコードも提供してください。」

改善理由：
・エラーメッセージを具体的に記載した
・対象のコードを提示した
・何を求めているか（原因、修正方法、修正後コード）を明示した

■ プロンプト3の改善

改善前：「レポートを書いてください」
改善後：「2026年Q1の営業チーム（10名）の活動実績レポートを
作成してください。

含める内容：
1. エグゼクティブサマリー（3行以内）
2. 主要KPIの達成状況（表形式）
3. 成功事例と課題（各2〜3個）
4. Q2への提言

対象読者：経営会議メンバー
文量：A4で2ページ以内」

改善理由：
・レポートのテーマと対象を特定した
・構成を具体的に指定した
・対象読者を明示した
・文量の制約を設定した
```
</details>

## チャレンジ問題

### 問題5：プロンプトの設計

あなたの会社では、エンジニアの技術面接を自動化したいと考えています。LLMを使った面接アシスタントのシステムプロンプトを設計してください。

以下の要件を満たすプロンプトを作成してください：
- 面接官の役割を設定する
- 候補者のレベル（ジュニア/ミドル/シニア）に応じた質問ができる
- 回答の評価基準を含める
- 不適切な質問（個人情報、差別的な質問）を避ける制約を含める

<details>
<summary>解答例</summary>

```
あなたは技術面接のアシスタントです。
ソフトウェアエンジニアの候補者に対して技術的な質問を行い、
回答を評価してください。

## 面接の進め方
1. 候補者のレベル（ジュニア/ミドル/シニア）を確認する
2. レベルに応じた質問を3〜5問出題する
3. 各回答を評価基準に基づいてスコアリングする
4. 総合評価とフィードバックを提供する

## レベル別の質問領域

ジュニア（経験0〜2年）：
- プログラミング基礎（変数、関数、条件分岐）
- データ構造の基本（配列、リスト、辞書）
- バージョン管理（Gitの基本操作）

ミドル（経験3〜5年）：
- 設計原則（SOLID、デザインパターン）
- テスト戦略（単体テスト、TDD）
- データベース設計（正規化、インデックス）

シニア（経験6年以上）：
- システム設計（スケーラビリティ、可用性）
- アーキテクチャ判断（トレードオフの説明）
- チームリーダーシップ（コードレビュー、メンタリング）

## 評価基準（各質問5点満点）
5点：正確で深い理解を示し、実務経験に基づく具体例がある
4点：概ね正確で、理解度が高い
3点：基本的な理解はあるが、深さが不足
2点：部分的な理解のみ
1点：理解が不十分

## 制約
- 年齢、性別、国籍、家族構成に関する質問は絶対にしない
- 候補者の回答に対して否定的な言葉を使わない
- 回答が不十分な場合はヒントを出して掘り下げる
- 技術的な正確性のみを評価し、言葉遣いや話し方は評価しない
```
</details>

### 3つのシナリオ

以下の3つのシナリオについて、プロンプトチェーンを設計してください。

| シナリオ | タスク | 想定ステップ数 |
|---------|--------|:------------:|
| 1 | ブログ記事の企画から完成まで | 4〜5ステップ |
| 2 | 市場調査レポートの作成 | 5〜6ステップ |
| 3 | 教育カリキュラムの設計 | 4〜5ステップ |

### 各シナリオで設計すること

1. **チェーン全体のフロー図**: 各ステップの入出力関係を図示
2. **各ステップのプロンプト**: 完全な形で記述
3. **ステップ間の受け渡しルール**: 前のステップの出力をどう次に渡すか
4. **品質チェックポイント**: 各ステップの出力を検証する基準
5. **分岐処理**: 条件によってフローが変わるケースの処理方法
6. **実行テスト**: 実際に全ステップを通して実行し、結果を記録

## ステップガイド

<details>
<summary>ステップ1：プロンプトチェーンの基本概念を理解する</summary>

プロンプトチェーンとは、複数のプロンプトを順番に実行し、前のステップの出力を次のステップの入力として使うパターンです。

**単一プロンプト vs プロンプトチェーン**:

```
【単一プロンプト】
「ブログ記事を書いてください。テーマはAI活用で、
 構成を考えて、2000文字で、最後に校正もしてください」
→ すべてを1回で処理。品質管理が困難。

【プロンプトチェーン】
Step1: テーマの深掘り → テーマと切り口を決定
Step2: アウトラインの作成 → 構成案を策定
Step3: 各セクションの執筆 → 本文を生成
Step4: 校正と改善 → 最終版を完成
→ 各ステップで品質確認が可能。出力の制御性が高い。
```

**チェーンが有効な場面**:
- タスクが複数の異なるスキルを必要とする（企画力 + 文章力 + 校正力）
- 中間成果物のレビューや修正が必要
- 最終出力が長文・複雑な構造を持つ
- ステップごとに異なるペルソナやパターンが最適

ステップ間でデータを受け渡す際のルールを定義しましょう。

**受け渡しフォーマットの例**:

```
各ステップの出力は以下のフォーマットで出力してください。
次のステップへの入力として使用します。

---OUTPUT_START---
## ステップ名: [現在のステップ名]

### 成果物
[メインの出力内容]

### メタ情報
- 品質自己評価: [1-5]
- 次のステップへの申し送り事項: [任意のメモ]
- 懸念点: [あれば記載]
---OUTPUT_END---
```

**分岐処理の例**:

```
Step2のアウトラインが承認されなかった場合：
→ Step2を修正フィードバック付きで再実行

Step3の本文が文字数条件を満たさない場合：
→ 不足セクションのみStep3を再実行
```

<details>
<summary>ステップ4：品質チェックポイントを設計する</summary>

各ステップの出力に対して、次のステップに進む前に確認すべきチェックポイントを設計します。

| ステップ | チェックポイント | 判定基準 |
|---------|----------------|---------|
| Step1 | テーマが具体的か | キーワードが3つ以上含まれる |
| Step1 | ターゲット読者が明確か | 年齢層・職種・関心が定義されている |
| Step2 | 見出しの数は適切か | 3〜7セクション |
| Step2 | 論理的な流れがあるか | 導入→展開→まとめの構成 |
| Step3 | 文字数は目標範囲内か | 目標の±10%以内 |
| Step3 | 各セクションにコード例/具体例があるか | 1セクション1つ以上 |
| Step4 | 誤字脱字がないか | 校正ツールでの確認 |
| Step4 | SEOキーワードが適切に含まれるか | タイトルと見出しに含まれる |

</details>

### シナリオ1：ブログ記事作成パイプライン

#### Step1：企画プロンプト

```
あなたはIT系メディアのコンテンツストラテジストです。

以下のキーワードから、技術ブログの記事テーマを企画してください。

キーワード：プロンプトエンジニアリング、初心者、実務

以下の形式で出力してください：
1. 記事タイトル案（3つ）
2. 推奨タイトル（1つ選択し、理由を1文で）
3. ターゲット読者（年齢層、職種、知識レベル）
4. 記事のゴール（読者が記事を読んだ後にできるようになること）
5. 想定文字数
```

**出力例**:
```
1. 記事タイトル案
   - 「プロンプトエンジニアリング入門：明日から仕事で使える5つのテクニック」
   - 「非エンジニアでもできる！プロンプトエンジニアリング実践ガイド」
   - 「ChatGPTの回答が劇的に変わる！プロンプト設計の基本」

2. 推奨タイトル
   「非エンジニアでもできる！プロンプトエンジニアリング実践ガイド」
   理由：ターゲットの「初心者」に最も訴求力があり、敷居の低さを伝えられるため。

3. ターゲット読者
   - 年齢層：25〜40歳
   - 職種：非エンジニア（マーケター、営業、人事など）
   - 知識レベル：ChatGPT等を使ったことはあるが、プロンプト設計は意識していない

4. 記事のゴール
   プロンプトの基本構造（役割・指示・文脈・出力形式）を理解し、
   自分の業務で効果的なプロンプトを設計できるようになること。

5. 想定文字数：3,000〜4,000文字
```

#### Step2：構成プロンプト

```
あなたはIT系メディアの編集者です。

以下の記事企画に基づいて、ブログ記事のアウトラインを作成してください。

【記事企画（Step1の出力を貼り付け）】
{Step1の出力}

以下の形式で出力してください：
- 見出し構成（H2、H3レベル）
- 各セクションの概要（2〜3文）
- 各セクションの想定文字数
- セクション間の論理的つながりの説明（1文ずつ）
```

#### Step3：執筆プロンプト

```
あなたはIT系ブログの専門ライターです。
読者に親しみやすく、かつ実践的な内容を心がけています。

以下のアウトラインに基づいて、セクションごとに本文を執筆してください。

【アウトライン（Step2の出力を貼り付け）】
{Step2の出力}

執筆ルール：
- です・ます調で統一
- 各セクションに最低1つの具体例（プロンプトの実例）を含める
- 専門用語は初出時に日本語と英語を併記
- 1文は60文字以内を目安
- 各セクションの文字数はアウトラインの指定に従う
```

#### Step4：校正プロンプト

```
あなたはプロの校正者であり、SEOの知識も持っています。

以下のブログ記事を校正し、改善してください。

【記事本文（Step3の出力を貼り付け）】
{Step3の出力}

以下の2段階で処理してください：

段階1：校正
- 誤字脱字の修正
- 文法の修正
- 表現の改善（冗長な表現の簡潔化）
- 論理的なつながりの確認

段階2：SEO改善
- タイトルにメインキーワードが含まれているか確認
- 見出しにサブキーワードが含まれているか確認
- メタディスクリプション（120文字以内）の作成

出力形式：
1. 修正箇所一覧（修正前 → 修正後）
2. 校正済み完成記事
3. SEO改善提案
4. メタディスクリプション
```

### シナリオ2：市場調査レポートの作成（改良版）

改良版では、**並列処理**と**統合ステップ**を含む高度なチェーン設計を行います。

#### チェーン全体のフロー図

```
[入力: 調査対象の製品/サービス]
       │
       ▼
┌──────────────────┐
│ Step1: 調査設計   │
│ → 調査項目の定義  │
└──────┬───────────┘
       │
       ├─────────────┬─────────────┐   ← 並列処理
       ▼             ▼             ▼
┌──────────┐  ┌──────────┐  ┌──────────┐
│Step2a:    │  │Step2b:    │  │Step2c:    │
│市場規模   │  │競合分析   │  │顧客分析   │
│分析       │  │           │  │           │
└──────┬───┘  └──────┬───┘  └──────┬───┘
       │             │             │
       └─────────────┼─────────────┘
                     ▼
              ┌──────────────┐
              │ Step3: 統合   │ ← 並列結果を統合
              │ → SWOT分析   │
              └──────┬───────┘
                     │
                     ▼
              ┌──────────────┐
              │ Step4: 提言   │
              │ → 戦略提案   │
              └──────┬───────┘
                     │
                     ▼
              ┌──────────────┐
              │ Step5: 校閲   │
              │ → 最終版     │
```

#### Step1：調査設計プロンプト

```
あなたは市場調査の専門コンサルタントです。

以下の製品について、市場調査レポートの調査項目を設計してください。

製品：生成AIを活用した議事録自動作成ツール
想定市場：日本国内の中小企業（従業員50〜300名）

以下の形式で出力してください：

## 調査フレームワーク
1. 市場規模分析の調査項目（5つ）
2. 競合分析の調査項目（5つ）
3. 顧客分析の調査項目（5つ）

## 各項目の定義
項目ごとに「何を調べるか」「なぜ重要か」「想定されるデータソース」を記載

## 注意事項
レポートの読者は経営層です。意思決定に直結する項目を優先してください。
```

#### Step2a：市場規模分析プロンプト（並列ステップの1つ）

```
あなたは市場規模分析の専門アナリストです。

以下の調査設計に基づいて、市場規模の分析を行ってください。

【調査設計（Step1の出力から市場規模分析の部分を抽出）】
{Step1の市場規模分析の調査項目}

【対象製品】
生成AIを活用した議事録自動作成ツール

分析に使用する情報：
- 一般的に知られている市場データや業界動向に基づいて推定してください
- 推定の根拠と前提条件を必ず明記してください
- 数値はすべて概算であることを注記してください

出力形式：
1. 市場規模の推定（TAM / SAM / SOM）
2. 市場成長率の推定
3. 主要ドライバー（市場を押し上げる要因）3つ
4. 主要リスク（市場成長を阻害する要因）3つ
5. 分析の前提条件と限界
```

#### Step3：統合・SWOT分析プロンプト

```
あなたは経営コンサルタントです。

以下の3つの分析結果を統合し、SWOT分析を行ってください。

【市場規模分析（Step2aの出力）】
{Step2aの出力}

【競合分析（Step2bの出力）】
{Step2bの出力}

【顧客分析（Step2cの出力）】
{Step2cの出力}

以下の形式で出力してください：

## SWOT分析

| | プラス要因 | マイナス要因 |
|---|---|---|
| 内部環境 | 強み(S): | 弱み(W): |
| 外部環境 | 機会(O): | 脅威(T): |

## クロスSWOT分析
- SO戦略（強み × 機会）: 
- WO戦略（弱み × 機会）: 
- ST戦略（強み × 脅威）: 
- WT戦略（弱み × 脅威）: 

## 3分析の統合サマリー（300文字以内）
3つの分析から導かれる最も重要なインサイトを記述してください。
```

#### 品質チェックポイント（改良版）

```
┌─────────────────────────────────────────────────────────┐
│         品質ゲート（Quality Gate）の設計                   │
├─────────────────────────────────────────────────────────┤
│                                                         │
│  Gate 1（Step1後）: 調査項目の網羅性チェック             │

│  ├─ 3つの分析結果に矛盾がないか                       │

│  └─ 不合格 → 矛盾のある分析ステップのみ再実行        │

│  Gate 4（Step5後）: 最終品質チェック                     │

│  ├─ データの出典が明記されているか                    │
│  └─ 不合格 → Step5を再実行                            │
│                                                         │
└─────────────────────────────────────────────────────────┘
```

#### 並列処理の設計ルール

```
並列処理を使う条件：
1. 各並列ステップが独立して実行可能（相互依存がない）
2. 最終的に統合ステップで結合される
3. 並列ステップの出力フォーマットが統一されている

並列処理のメリット：
- 実行時間の短縮（APIで並列呼び出し可能な場合）
- 各分析の独立性が保たれ、バイアスが入りにくい
- 1つの分析が失敗しても他に影響しない

並列処理の注意点：
- 統合ステップのプロンプトが複雑になる（複数の入力を処理する必要がある）
- コンテキストウィンドウの消費が大きい
- 並列結果間の用語や基準を統一する仕組みが必要
```

**初心者向けとの違い**:
- 並列処理ステップを導入し、複雑なチェーン構造を設計
- 品質ゲートの概念を導入し、各ステップの通過基準と不合格時の処理を定義
- SWOT分析とクロスSWOTによる統合分析手法を活用
- 並列処理の設計ルールとトレードオフを明文化

# 第4章：TCP と UDP 演習問題

## 基本問題

### 問題4-1：TCPとUDPの比較
TCPとUDPの違いを以下の観点から比較してください。

1. 接続方式
2. 信頼性
3. 順序制御
4. 速度
5. 代表的な使用プロトコル

<details>
<summary>解答</summary>

| 観点 | TCP | UDP |
|------|-----|-----|
| 接続方式 | コネクション型（3ウェイハンドシェイク） | コネクションレス型 |
| 信頼性 | 高い（ACK、再送制御） | 低い（確認応答なし） |
| 順序制御 | あり（シーケンス番号） | なし（到着順序は保証されない） |
| 速度 | UDPより遅い（オーバーヘッドあり） | 高速（ヘッダが小さく軽量） |
| 代表プロトコル | HTTP, HTTPS, FTP, SMTP, SSH | DNS(通常), NTP, DHCP, RIP, VoIP |

</details>

### 問題4-2：3ウェイハンドシェイク
TCPの3ウェイハンドシェイクの3つのステップを、フラグ名とともに説明してください。

<details>
<summary>解答</summary>

1. **SYN（同期）**：クライアント → サーバ
   - クライアントが接続要求を送信
   - SYNフラグをセット、初期シーケンス番号（ISN）を通知

2. **SYN + ACK（同期確認応答）**：サーバ → クライアント
   - サーバが接続受諾を送信
   - SYNフラグとACKフラグをセット
   - サーバの初期シーケンス番号を通知、クライアントのシーケンス番号+1を確認応答

3. **ACK（確認応答）**：クライアント → サーバ
   - クライアントが確認応答を送信
   - ACKフラグをセット、サーバのシーケンス番号+1を確認応答
   - この時点でコネクションが確立される

```
クライアント                    サーバ
    │── SYN (seq=100) ────────→│
    │←─ SYN+ACK (seq=300, ack=101) ─│
    │── ACK (seq=101, ack=301) ──→│
    │      コネクション確立        │
```

</details>

### 問題4-3：ポート番号の分類
ポート番号の範囲による分類を3つ挙げ、それぞれの範囲と用途を説明してください。

<details>
<summary>解答</summary>

| 分類 | 範囲 | 用途 |
|------|------|------|
| **ウェルノウンポート** | 0 〜 1023 | サーバ側の標準サービスに予約（HTTP=80、HTTPS=443等） |
| **登録済みポート** | 1024 〜 49151 | IANAに登録されたアプリケーション用（MySQL=3306等） |
| **動的/エフェメラルポート** | 49152 〜 65535 | クライアント側が一時的に使用するポート（OSが自動割当） |

</details>

### 問題4-4：TCPフラグ
以下のTCPフラグの役割を説明してください。

1. SYN
2. ACK
3. FIN
4. RST
5. PSH
6. URG

<details>
<summary>解答</summary>

1. **SYN（Synchronize）**：コネクション確立の要求。初期シーケンス番号の同期
2. **ACK（Acknowledgment）**：確認応答。確認応答番号フィールドが有効であることを示す
3. **FIN（Finish）**：コネクション終了の要求。送信完了を通知
4. **RST（Reset）**：コネクションの強制リセット。異常時に使用
5. **PSH（Push）**：バッファリングせずに即座にアプリケーションにデータを渡すよう要求
6. **URG（Urgent）**：緊急データの存在を示す。緊急ポインタフィールドが有効

</details>

## 応用問題

### 問題4-5：フロー制御とウィンドウサイズ
TCPのウィンドウ制御（スライディングウィンドウ）の仕組みを説明してください。ウィンドウサイズが小さい場合と大きい場合でどのような違いがあるか述べてください。

<details>
<summary>解答</summary>

**ウィンドウ制御（スライディングウィンドウ）：**

ウィンドウサイズは、受信側がACKなしで受信できるデータ量を示します。送信側はウィンドウサイズ分のデータをACKを待たずに連続送信できます。

```
ウィンドウサイズ = 3 の場合:
送信側 ──→ セグメント1 ──→
送信側 ──→ セグメント2 ──→  ACKを待たずに連続送信
送信側 ──→ セグメント3 ──→
       ←── ACK(1,2,3) ←──
送信側 ──→ セグメント4 ──→  ウィンドウがスライド
...
```

**ウィンドウサイズが小さい場合：**
- ACKの待ち時間が増え、スループットが低下
- 受信側の処理能力が低い場合に使われる

**ウィンドウサイズが大きい場合：**
- 一度に多くのデータを送信でき、スループットが向上
- 遅延の大きいネットワーク（WAN等）で効果が大きい
- ただし、受信側のバッファを超えるとデータが失われる可能性がある

受信側はTCPヘッダのWindowフィールドで自身の受信可能サイズを送信側に通知し、送信量を動的に制御します。

</details>

### 問題4-6：TCPの接続切断
TCPのコネクション切断（4ウェイハンドシェイク）の流れを説明してください。

<details>
<summary>解答</summary>

```
クライアント                    サーバ
    │── ① FIN ─────────────→│  クライアントが切断要求
    │←─ ② ACK ─────────────│  サーバが確認応答
    │   （サーバは残りのデータ送信可能）
    │←─ ③ FIN ─────────────│  サーバも切断要求
    │── ④ ACK ─────────────→│  クライアントが確認応答
    │                          │
    │  TIME_WAIT状態           │
    │  （一定時間後に完全切断）  │
```

1. **FIN**（クライアント→サーバ）：クライアントが送信完了を通知
2. **ACK**（サーバ→クライアント）：サーバがFINを確認
3. **FIN**（サーバ→クライアント）：サーバも送信完了を通知
4. **ACK**（クライアント→サーバ）：クライアントがFINを確認

**TIME_WAIT 状態**：最後のACKが失われた場合にサーバがFINを再送することに備え、一定時間（通常60秒〜4分）待機する。この間、同じソケットの再利用はできない。

</details>

### 問題4-7：netstat / ss コマンド
以下の `netstat` の出力から、各接続の状態を説明してください。

```
Proto  Local Address       Foreign Address      State
TCP    0.0.0.0:80          0.0.0.0:0           LISTENING
TCP    192.168.1.10:80     203.0.113.50:52341  ESTABLISHED
TCP    192.168.1.10:80     203.0.113.51:52890  TIME_WAIT
TCP    192.168.1.10:22     10.0.0.100:49152    ESTABLISHED
```

<details>
<summary>解答</summary>

1. **0.0.0.0:80 - LISTENING**
   - Webサーバ（ポート80）が全IPアドレスで接続待ち受け中
   - クライアントからの接続要求を待っている状態

2. **192.168.1.10:80 ↔ 203.0.113.50:52341 - ESTABLISHED**
   - 外部クライアント（203.0.113.50）とWebサーバ（ポート80）の間で接続が確立済み
   - 通信中の状態

3. **192.168.1.10:80 ↔ 203.0.113.51:52890 - TIME_WAIT**
   - 外部クライアント（203.0.113.51）との接続が切断された直後
   - 最後のACKの再送に備えて待機中（一定時間後に消える）

4. **192.168.1.10:22 ↔ 10.0.0.100:49152 - ESTABLISHED**
   - 内部クライアント（10.0.0.100）とSSHサーバ（ポート22）の間で接続が確立済み
   - SSH接続が活動中

</details>

## チャレンジ問題

### 問題4-8：プロトコル選択の理由
以下のアプリケーションがTCPまたはUDPのどちらを使用するか答え、その理由を説明してください。

1. Webブラウジング（HTTP/HTTPS）
2. DNS名前解決
3. VoIP（音声通話）
4. オンラインゲーム（リアルタイム）
5. ファイル転送（FTP）
6. 動画ストリーミング

<details>
<summary>解答</summary>

1. **HTTP/HTTPS → TCP**：Webページのデータは完全性が重要。一部でも欠けるとページが正しく表示されないため、信頼性のあるTCPを使用

2. **DNS → 主にUDP**：問い合わせと応答が小さなデータ（通常512バイト以下）で完結するため、軽量なUDPが適切。ただしゾーン転送や大きな応答にはTCPを使用

3. **VoIP → UDP**：リアルタイム性が最重要。再送によるデータの遅延は音質の劣化につながるため、多少のパケットロスを許容しても低遅延なUDPを使用

4. **オンラインゲーム → UDP（多くの場合）**：位置情報等のリアルタイムデータは最新の状態が重要で、古いデータの再送は不要。アプリケーション層で独自の信頼性制御を実装することが多い

5. **FTP → TCP**：ファイルは1バイトでも欠損すると使えないため、データの完全性を保証するTCPが必須

6. **動画ストリーミング → UDP（RTP等）またはTCP（HTTP経由）**：従来のストリーミングはUDP（RTP）だが、近年はHTTP（TCP）ベースの適応型ストリーミング（HLS、DASH）が主流。バッファリングにより遅延を吸収できるため

</details>

### 問題4-9：輻輳制御
TCPの輻輳制御について、以下の用語を使って動作を説明してください。
- スロースタート
- 輻輳ウィンドウ（cwnd）
- 輻輳回避
- ssthresh（スロースタート閾値）

<details>
<summary>解答</summary>

**TCPの輻輳制御の流れ：**

1. **スロースタート**：
   - 接続開始時、輻輳ウィンドウ（cwnd）を1セグメントから開始
   - ACKを受信するたびに cwnd を**指数的**に増加（1→2→4→8...）
   - cwnd が ssthresh に達するまで続ける

2. **輻輳回避**：
   - cwnd が ssthresh を超えたら、**線形的**に増加（1ずつ）
   - ネットワークの容量を慎重に探る

3. **パケットロス検出時**：
   - タイムアウトの場合：ssthresh を cwnd の半分に設定、cwnd を1に戻してスロースタートからやり直し
   - 重複ACK3回の場合（高速再転送）：ssthresh を cwnd の半分に設定、cwnd を ssthresh に設定して輻輳回避フェーズから再開

```
cwnd
 ↑
 │          ×パケットロス
 │        ╱  ssthresh = cwnd/2
 │      ╱    cwnd = 1（タイムアウトの場合）
 │    ╱  輻輳回避（線形増加）
 │  ╱───
 │╱ ssthresh
 │ スロースタート
 │ （指数的増加）
 └────────────────→ 時間
```

</details>

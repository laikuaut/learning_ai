## 基本問題

### 問題1：機能要件と非機能要件の分類

以下の要件が機能要件か非機能要件かを分類し、その理由を述べてください。

1. ユーザーが商品をカートに追加できること
2. 画面の応答時間が3秒以内であること
3. ユーザーが注文履歴を一覧で確認できること
4. システムの月間稼働率が99.9%以上であること
5. パスワードはハッシュ化して保存すること
6. 管理者が商品情報を登録・編集できること
7. 同時に500ユーザーがアクセスしても正常に動作すること
8. 注文確定時に確認メールが送信されること

**期待される出力例：**

```
1. 機能要件 - 「カートに追加する」というシステムの動作を定義している
2. 非機能要件 - 応答時間という品質特性を定義している
```

<details>
<summary>ヒント</summary>

機能要件は「システムが何をするか」、非機能要件は「どの程度のレベルで実現するか」です。判断に迷ったら、「その要件がなくてもシステムは動作するか？」を考えてみてください。

</details>

<details>
<summary>解答例</summary>

```
1. 機能要件
   理由：「カートに追加する」というシステムの具体的な振る舞いを定義している。

2. 非機能要件（性能要件）
   理由：画面応答の速度という品質特性を定義しており、
         機能自体の動作定義ではない。

3. 機能要件
   理由：「注文履歴を一覧表示する」というシステムの具体的な機能を
         定義している。

4. 非機能要件（可用性要件）
   理由：システムの稼働率という可用性の品質特性を定義している。

5. 非機能要件（セキュリティ要件）
   理由：データの保存方式というセキュリティの品質特性を定義している。
         ※「パスワードを設定できる」は機能要件だが、
           「ハッシュ化して保存する」は実現方法の品質に関する要件。

6. 機能要件
   理由：「商品情報を登録・編集する」というシステムの具体的な
         機能を定義している。

7. 非機能要件（性能要件）
   理由：同時接続数というキャパシティの品質特性を定義している。

8. 機能要件
   理由：「確認メールを送信する」というシステムの具体的な
         振る舞いを定義している。
```

</details>

### 問題2：ユースケース記述の作成

以下の機能について、ユースケース記述を作成してください。

**機能：** ECサイトで、ユーザーが商品をカートに追加する

以下のテンプレートに従って記述してください。

```
ユースケースID ：
ユースケース名 ：
アクター       ：
事前条件       ：
事後条件       ：

【基本フロー（正常系）】

【代替フロー】

【ビジネスルール】
```

**期待される出力例：**

```
ユースケースID ：UC-010
ユースケース名 ：商品をカートに追加する
アクター       ：会員ユーザー
...
```

<details>
<summary>ヒント</summary>

以下の観点を考慮してください。
- 事前条件：この操作を行うために必要な状態は？
- 基本フロー：正常にカートに追加される手順は？
- 代替フロー：在庫切れ、数量上限超過、未ログイン時など
- ビジネスルール：カートの上限数、同一商品の扱いなど

</details>

<details>
<summary>解答例</summary>

```
ユースケースID ：UC-010
ユースケース名 ：商品をカートに追加する
アクター       ：会員ユーザー
事前条件       ：ユーザーがログイン済みであること
                商品詳細画面を表示していること
事後条件       ：選択した商品がカートに追加され、
                カート内の商品数が更新されること

【基本フロー（正常系）】
  1. ユーザーが商品詳細画面で数量を選択する（デフォルト：1）
  2. ユーザーが「カートに追加」ボタンを押す
  3. システムが在庫数を確認する
  4. システムがカート内の同一商品の有無を確認する
  5. システムが商品をカートに追加する
  6. システムがヘッダーのカートアイコンの数量を更新する
  7. システムが「カートに追加しました」のメッセージを表示する
     （「カートを見る」「買い物を続ける」のリンク付き）

【代替フロー】
  3a. 在庫が0の場合
      → 「この商品は現在在庫切れです」と表示する
      → 「入荷通知を受け取る」ボタンを表示する
      → 基本フローを終了する

  3b. 選択数量が在庫数を超える場合
      → 「在庫数は残り○個です」と表示する
      → 数量を在庫数に自動修正する
      → ステップ2に戻る

  4a. 同一商品が既にカートにある場合
      → カート内の既存商品の数量に加算する
      → ステップ6に進む

  4b. 加算後の数量がカート上限（99個）を超える場合
      → 「この商品のカート上限は99個です」と表示する
      → 基本フローを終了する

  1a. 未ログインの場合
      → ログイン画面にリダイレクトする
      → ログイン後、元の商品詳細画面に戻る

【ビジネスルール】
  ・1商品あたりのカート内最大数量：99個
  ・カートに入れられる商品種類の上限：50種類
  ・カート内商品の保持期間：30日間（超過で自動削除）
  ・在庫確認はカート追加時にリアルタイムで行う
  ・予約商品と通常商品は同一カートで扱える
```

</details>

### 問題3：スコープの定義

あなたはオンライン図書館システムの要件定義を担当しています。以下の機能候補から、スコープ内とスコープ外を分類し、その理由を記述してください。

**プロジェクトの前提：** 開発期間4ヶ月、開発者3名、初期リリース

**機能候補：**
1. 書籍の検索・閲覧
2. 書籍の貸出・返却管理
3. 利用者登録・認証
4. AIによる書籍レコメンド機能
5. 延滞通知メール送信
6. 書籍のレビュー・評価機能
7. 蔵書の在庫管理
8. 他図書館との相互貸借システム連携
9. 新着図書のお知らせ機能
10. 読書記録・統計ダッシュボード

**期待される出力例：**

```
【スコープ内】
1. 書籍の検索・閲覧 - 図書館の基本機能として必須
...

【スコープ外】
4. AIによる書籍レコメンド機能 - 初期リリースでは優先度が低い
...
```

<details>
<summary>ヒント</summary>

以下の観点で判断してください。
- 開発期間4ヶ月・3名で実現可能か？
- 図書館システムとして最低限必要な機能か？
- 初期リリースに必須か、後のフェーズでも良いか？

</details>

<details>
<summary>解答例</summary>

```
【スコープ内（Must / Should）】

1. 書籍の検索・閲覧（Must）
   理由：図書館システムの最も基本的な機能。
         これがないとシステムとして成立しない。

2. 書籍の貸出・返却管理（Must）
   理由：図書館業務の中核機能。
         貸出・返却ができないと業務が回らない。

3. 利用者登録・認証（Must）
   理由：貸出管理の前提として、利用者を識別する
         仕組みが必要。

5. 延滞通知メール送信（Should）
   理由：延滞管理は図書館の重要業務。
         メール通知は実装コストが比較的低い。

7. 蔵書の在庫管理（Must）
   理由：書籍の所在・状態を管理する基本機能。
         貸出管理と密接に関連する。


【スコープ外（Could / Won't）】

4. AIによる書籍レコメンド機能（Won't）
   理由：AI機能は開発コストが高く、4ヶ月の工期では困難。
         フェーズ2以降で検討する。

6. 書籍のレビュー・評価機能（Could → フェーズ2）
   理由：あれば便利だが、図書館の基本業務には不要。
         余裕があればフェーズ2で対応。

8. 他図書館との相互貸借システム連携（Won't）
   理由：外部システムとの連携は仕様調整に時間がかかり、
         初期リリースには間に合わない。

9. 新着図書のお知らせ機能（Could → フェーズ2）
   理由：利便性向上の機能だが、初期リリースでは必須ではない。
         蔵書管理の延長で後から追加しやすい。

10. 読書記録・統計ダッシュボード（Could → フェーズ2）
    理由：付加価値機能であり、基本機能の安定稼働を
          優先すべき。
```

</details>

## 応用問題

### 問題4：非機能要件の具体化

以下のシステムについて、非機能要件を「可用性」「性能」「セキュリティ」の3つの観点で具体的に定義してください。

**システム：** 従業員500名が利用する社内経費精算システム（Webアプリケーション）

**期待される出力例：**

```
【可用性要件】
| 項目       | 要件                          |
|-----------|-------------------------------|
| 稼働率     | ...                           |
```

<details>
<summary>ヒント</summary>

以下の観点で考えてください。
- 可用性：いつ使えるか、止まったときどうするか
- 性能：どの程度の速度・処理量が必要か
- セキュリティ：誰がアクセスでき、データをどう守るか

社内システムなので、24時間365日の可用性は不要かもしれません。利用パターンを考慮しましょう。

</details>

<details>
<summary>解答例</summary>

```
【可用性要件】

| 項目           | 要件                                      |
|---------------|------------------------------------------|
| サービス時間    | 平日 8:00〜22:00                          |
| 稼働率         | サービス時間内で月間99.0%以上               |
|               |（月間ダウンタイム約4.3時間以内）             |
| 計画停止       | 月1回、土曜22:00〜日曜6:00（最大8時間）     |
| 障害時の復旧    | RTO: 4時間以内、RPO: 前日バックアップ時点    |
| データバックアップ| 日次フルバックアップ（7世代保持）            |

※ 社内システムのため、夜間・休日は計画停止を許容する。
  月末（25日〜末日）は経費申請の締切期間のため、
  計画停止を禁止する。

【性能要件】

| 項目             | 要件                                   |
|-----------------|---------------------------------------|
| 画面応答時間     | 3秒以内（95パーセンタイル）               |
| 一覧表示        | 経費明細1,000件の表示が5秒以内            |
| ファイルアップロード| 領収書画像（5MB以下）のアップロードが10秒以内|
| 同時接続数       | 最大100ユーザー（ピーク時：月末締切前）     |
| バッチ処理       | 月次集計処理が30分以内に完了               |
| データ量想定     | 年間経費申請件数：約60,000件              |
|                 |（500名 × 月平均10件 × 12ヶ月）          |

※ 月末の締切前にアクセスが集中するため、
  ピーク時の同時接続数を考慮する。

【セキュリティ要件】

| 項目           | 要件                                      |
|---------------|------------------------------------------|
| 認証方式       | 社内Active Directoryと連携（SSO）           |
| 認可          | 一般社員、部門承認者、経理部、管理者の4ロール  |
| 通信暗号化     | HTTPS（TLS 1.2以上）                       |
| 操作ログ       | 申請・承認・差戻し・金額変更の全操作を記録     |
| セッション管理  | 60分操作なしでタイムアウト                    |
| アクセス制限   | 社内ネットワークからのみアクセス可能            |
|               | VPN経由でのリモートアクセスを許可              |
| データ保持期間  | 経費データは税法に基づき7年間保持              |
| 個人情報       | 口座番号等はマスキング表示（権限者のみ閲覧可）  |
```

</details>

## チャレンジ問題

### 問題5：要件定義書の作成（総合演習）

以下のプロジェクトについて、要件定義書の「プロジェクト概要」と「機能一覧」セクションを作成してください。

**プロジェクト概要：**
ある中小企業（従業員50名）が、社内のナレッジ（技術文書・議事録・マニュアル等）をチームで共有・管理するためのWebアプリケーションを新規開発したいと考えています。

現状は、ナレッジがファイルサーバー上の個人フォルダ、メール、チャットに散在しており、必要な情報を見つけるのに時間がかかっています。

**期待される出力例：**

```
1. プロジェクト概要
   1.1 背景・目的
   1.2 スコープ
   1.3 前提条件・制約条件
   1.4 用語定義

2. 機能一覧
   機能ID | 機能名 | 概要 | 優先度
   ...
```

<details>
<summary>ヒント</summary>

以下の点を考慮してください。
- 背景・目的：現状の課題を具体的に記述する
- スコープ：初期リリースに含める範囲を絞る
- 機能一覧：ナレッジ管理に必要な機能を洗い出す
  （作成・閲覧・検索・編集・タグ付け・権限管理など）
- 優先度：MoSCoW法で分類する

</details>

<summary>解答例</summary>

  社内ナレッジ管理システム 要件定義書（抜粋）
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

1. プロジェクト概要

1.1 背景・目的

【背景】
現在、社内のナレッジ（技術文書・議事録・マニュアル等）が以下の場所に
散在しており、情報の検索・活用に課題が生じている。

  ・ファイルサーバー上の個人フォルダ（約60%の文書）
  ・メール添付ファイル（約20%の文書）
  ・チャットツールの投稿（約15%の文書）
  ・紙の資料（約5%の文書）

この状況により、以下の問題が発生している。

  課題1：必要な情報の検索に1件あたり平均30分を要している
  課題2：同じ質問が月平均15件繰り返されている（属人化）
  課題3：退職者のナレッジが引き継がれず消失している
  課題4：ファイルのバージョン管理ができず、古い情報を参照してしまう

【目的】
上記課題を解決するため、社内ナレッジ管理Webアプリケーションを
構築する。導入により以下の効果を見込む。

  効果1：全文検索により情報検索時間を5分以内に短縮する
  効果2：ナレッジの一元管理により属人化を解消する
  効果3：バージョン管理により常に最新の情報を参照できる
  効果4：退職・異動時のナレッジ移管を容易にする


1.2 スコープ

【スコープ内（初期リリース）】
  ・Markdown形式でのナレッジ記事の作成・編集・削除
  ・全文検索機能
  ・タグ・カテゴリによる分類
  ・閲覧権限の管理（公開・部署限定・非公開）
  ・記事のバージョン管理（変更履歴の保持）
  ・ユーザー認証（社内ディレクトリ連携）
  ・ファイル添付（画像・PDF・Office文書）

【スコープ外】
  ・AIによる自動要約・タグ付け（フェーズ2で検討）
  ・チャットツールとの連携通知（フェーズ2で検討）
  ・既存ファイルサーバーからの自動インポート
    （手動での移行手順を提供する）
  ・モバイルアプリ（ブラウザでの利用を前提とする）
  ・多言語対応（日本語のみ）


1.3 前提条件・制約条件

【前提条件】
  1. 全従業員（50名）が社用PCとメールアドレスを保持している
  2. 社内にActive Directoryが運用されている
  3. 従業員はWebブラウザ（Chrome最新版）を使用できる
  4. ナレッジの大部分はテキストベースの文書である

【制約条件】
  1. 開発予算は500万円以内とする
  2. 開発期間は3ヶ月（2026年7月リリース）
  3. 既存のAWS環境（会社アカウント）にデプロイすること
  4. 社内ネットワーク + VPN経由でのみアクセス可能とすること
  5. 社内Active Directoryとの認証連携必須

| コントリビューター| 記事を作成・編集する権限を持つユーザー       |

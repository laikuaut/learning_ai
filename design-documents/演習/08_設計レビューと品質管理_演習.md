## 基本問題

### 問題1：設計書の問題点の指摘

以下の設計書の記述には問題があります。問題点を指摘し、改善案を提示してください。

**記述1：テーブル定義**
```
テーブル名: user
カラム: id, name, mail, pw, addr, tel, flg
```

**記述2：API仕様**
```
POST /api/deleteUser
リクエスト: { userId: "123" }
レスポンス: { result: "OK" }
```

**記述3：画面設計**
```
エラーが発生した場合は適切なメッセージを表示する。
```

**記述4：処理設計**
```
データを取得して画面に表示する。
エラーの場合はエラー処理を行う。
```

<details>
<summary>ヒント</summary>

命名規則、RESTful設計、具体性、曖昧さの排除の観点からチェックしましょう。

</details>

<details>
<summary>解答例</summary>

```
■ 記述1の問題点と改善

問題点：
  ・テーブル名が単数形（user → users にすべき）
  ・カラム名が略語（pw, addr, tel, flg は不明確）
  ・カラムのデータ型、NOT NULL、制約が未定義

改善案：
  テーブル名: users
  | カラム名 | データ型 | NOT NULL | 説明 |
  |---------|---------|----------|------|
  | id | SERIAL | ◯ | 主キー |
  | name | VARCHAR(100) | ◯ | 氏名 |
  | email | VARCHAR(255) | ◯ | メールアドレス（UNIQUE） |
  | password_hash | VARCHAR(255) | ◯ | パスワードハッシュ |
  | address | VARCHAR(255) | × | 住所 |
  | phone | VARCHAR(20) | × | 電話番号 |
  | is_active | BOOLEAN | ◯ | 有効フラグ |

■ 記述2の問題点と改善

問題点：
  ・POSTメソッドで削除を行っている（DELETEを使うべき）
  ・URLに動詞が含まれている（deleteUser）
  ・リクエストボディでIDを渡している（URLパスパラメータにすべき）
  ・レスポンスが不明確

改善案：
  DELETE /api/v1/users/123
  レスポンス: 204 No Content（ボディなし）

  エラー時:
  404 Not Found → { "error": { "code": "USER_NOT_FOUND", "message": "..." } }

■ 記述3の問題点と改善

問題点：
  ・「適切なメッセージ」が曖昧で、実装者によって解釈が異なる
  ・どのようなエラーでどのメッセージを表示するか不明

改善案：
  | エラー種別 | 表示位置 | メッセージ |
  |-----------|---------|-----------|
  | バリデーションエラー | 該当項目の下 | 各項目のエラーメッセージ |
  | 認証エラー | 画面上部 | 「セッションが切れました。再度ログインしてください」 |
  | サーバーエラー | 画面上部 | 「システムエラーが発生しました。時間をおいて再度お試しください」 |

■ 記述4の問題点と改善

問題点：
  ・「データ」が何のデータか不明
  ・「取得して表示」の処理の詳細が不明
  ・「エラー処理」の内容が不明

改善案：
  1. GET /api/v1/products でカテゴリID=3の商品一覧を取得する
  2. レスポンスのdataを商品カード一覧として表示する
  3. pagination情報をもとにページネーションを表示する
  4. 404エラーの場合「該当する商品がありません」を表示する
  5. 500エラーの場合「データの取得に失敗しました」を表示する
```

</details>

### 問題2：レビューチェックリストの適用

以下の設計書（抜粋）に対して、レビューチェックリストを適用し、指摘事項を挙げてください。

```markdown
## ユーザー登録API

POST /api/users

リクエスト:
{
  "name": "山田太郎",
  "email": "yamada@example.com",
  "password": "pass1234"
}

レスポンス（成功）:
{
  "id": 1,
  "name": "山田太郎",
  "email": "yamada@example.com",
  "password": "pass1234"
}
```

最低3つの指摘事項を挙げてください。

<details>
<summary>ヒント</summary>

セキュリティ、API設計のベストプラクティス、レスポンスの適切性の観点からチェックしましょう。

</details>

<details>
<summary>解答例</summary>

```
指摘1：【重大度：高】レスポンスにパスワードが含まれている
  → セキュリティ上の重大な問題。パスワード（平文・ハッシュ問わず）を
    レスポンスに含めてはいけない。
  → 改善：レスポンスからpasswordフィールドを削除する。

指摘2：【重大度：高】パスワードのバリデーションが未定義
  → "pass1234"のような弱いパスワードが許容されてしまう。
  → 改善：パスワードポリシー（最低文字数、英数字混在等）を
    リクエスト仕様に明記する。

指摘3：【重大度：中】APIバージョンがURLに含まれていない
  → /api/users ではなく /api/v1/users にすべき。
  → 将来のAPI変更に対応するため。

指摘4：【重大度：中】エラーケースが未定義
  → メールアドレス重複時、バリデーションエラー時の
    レスポンスが定義されていない。
  → 改善：400, 409等のエラーレスポンスを追加する。

指摘5：【重大度：中】成功時のHTTPステータスコードが未記載
  → 新規作成なので201 Createdが適切。明記すべき。

指摘6：【重大度：低】リクエストパラメータの型・制約が未記載
  → name, emailの最大文字数、emailの形式バリデーション等を
    明記すべき。
```

</details>

### 問題3：指摘事項の重大度分類

以下の設計レビューの指摘事項を重大度（高・中・低）に分類し、理由を述べてください。

1. ER図にcreated_at, updated_atカラムが記載されていない
2. 決済APIでクレジットカード番号がログに出力される設計になっている
3. API設計書の日付が「2025年」のまま更新されていない
4. 注文テーブルにuser_idのインデックスが設計されていない
5. 画面設計書のボタンの文言が「OK」と「はい」で統一されていない
6. ユーザーの入力値がサーバーサイドでバリデーションされていない設計
7. シーケンス図で非同期処理の待ち合わせが考慮されていない

<details>
<summary>解答例</summary>

```
■ 高（リリース前に必ず修正）

2. 決済APIでクレジットカード番号がログ出力される
   → PCI DSS違反の可能性がある重大なセキュリティ問題。
     カード番号のマスキングまたはログ出力除外が必須。

6. サーバーサイドのバリデーション未実施
   → フロントのみのバリデーションはバイパス可能。
     SQLインジェクションやXSS等のセキュリティリスクに直結する。

7. 非同期処理の待ち合わせが未考慮
   → 実装時にデータの不整合やレースコンディションが発生する可能性。
     設計段階で解決しないと手戻りが大きい。

■ 中（可能な限り修正）

1. created_at, updated_atカラムがない
   → データの追跡・デバッグに必要な基本的なカラム。
     後から追加するとマイグレーションが必要になる。

4. user_idのインデックスがない
   → ユーザーの注文一覧取得で性能問題が発生する可能性。
     データ量が増えると顕著になる。

■ 低（次回更新時に修正）

3. 設計書の日付が古い
   → 内容の正確性には影響しないが、ドキュメント管理の観点で更新すべき。

5. ボタン文言の不統一
   → 機能への影響はないが、UI/UXの一貫性のために統一すべき。
```

</details>

### 問題4：設計レビューの実施

以下の基本設計書（抜粋）に対して設計レビューを行い、指摘事項一覧を作成してください。

```markdown
## タスク管理API

### POST /api/tasks（タスク作成）
リクエスト:
{
  "title": "設計書レビュー",
  "description": "基本設計書のレビューを行う",
  "assigned_to": "yamada",
  "due_date": "2026-04-15"
}

レスポンス:
{
  "task_id": 1,
  "title": "設計書レビュー",
  "status": "open"
}

### GET /api/getTasks（タスク一覧取得）
レスポンス:
{
  "tasks": [
    { "task_id": 1, "title": "設計書レビュー", "status": "open" }
  ]
}

### POST /api/tasks/1/delete（タスク削除）
レスポンス:
{ "result": "success" }
```

指摘事項一覧表（No、対象、指摘内容、重大度、改善案）を作成してください。最低5つの指摘を挙げてください。

<details>
<summary>解答例</summary>

```markdown
## 指摘事項一覧

| No | 対象 | 指摘内容 | 重大度 | 改善案 |
|----|------|---------|--------|--------|
| 1 | GET /api/getTasks | URLに動詞が含まれている。RESTful設計に違反 | 中 | GET /api/v1/tasks に変更 |
| 2 | POST .../delete | 削除にPOSTを使用し、URLに動詞が含まれている | 中 | DELETE /api/v1/tasks/:id に変更 |
| 3 | 全API | URLにバージョンが含まれていない | 中 | /api/v1/ をベースURLに追加 |
| 4 | POST /api/tasks | assigned_toが文字列。ユーザーIDを使うべき | 中 | assigned_to_user_id (integer)に変更 |
| 5 | POST /api/tasks | リクエストパラメータの型・必須・制約が未定義 | 中 | 各フィールドの型、必須/任意、文字数制限を追記 |
| 6 | POST /api/tasks | 成功時のHTTPステータスが未記載 | 低 | 201 Created を明記 |
| 7 | 全API | エラーレスポンスの形式が未定義 | 中 | 共通エラー形式を定義し、各APIのエラーケースを列挙 |
| 8 | GET /api/getTasks | ページネーションがない | 中 | page, per_page パラメータとpaginationレスポンスを追加 |
| 9 | 削除API | レスポンスが独自形式 | 低 | 204 No Content（ボディなし）に変更 |
| 10 | POST /api/tasks | 認証の要否が未記載 | 中 | 各APIに認証の要否を明記 |
```

</details>

## チャレンジ問題

### 問題5：設計書の品質改善（総合問題）

あなたは新しいプロジェクト（社内の経費精算システム）のリードエンジニアです。チームメンバーが作成した以下の設計書について、改善すべき点を洗い出し、改善後の設計書を作成してください。

**チームメンバーが作成した設計書：**

```markdown
## 経費精算API

### 精算申請
POST /api/expenses
{
  "user": "suzuki",
  "amount": 5000,
  "type": "交通費",
  "receipt": "base64エンコードされた画像データ"
}

→ OK

### 承認
POST /api/approve
{
  "expense_id": 1
}

→ OK

### 一覧
GET /api/expenses
→ 全精算データを返す

### 削除
GET /api/expenses/delete/1
→ OK
```

以下を回答してください：
1. 問題点を最低6つ指摘する
2. 改善後のAPI一覧表を作成する
3. 精算申請API（POST）の改善後の詳細仕様を作成する

<details>
<summary>解答例</summary>

```markdown
## 1. 問題点

1. 【高】receiptにbase64画像を直接送信する設計はリクエストサイズが巨大になる
   → ファイルアップロードAPIを分離すべき

2. 【中】削除にGETメソッドを使用している（RESTful違反）
   → DELETEメソッドを使用する

3. 【中】承認APIのURL設計が不適切（POST /api/approve）
   → PATCH /api/v1/expenses/:id/approve が適切

4. 【中】一覧APIにページネーション・フィルタがない
   → page, per_page, status等のパラメータが必要

5. 【中】レスポンスが「OK」のみで仕様が不明確
   → 適切なHTTPステータスコードと詳細なレスポンスを定義すべき

6. 【中】userが文字列名。ログインユーザーのIDを使うべき
   → 認証トークンからユーザーを特定する

7. 【中】エラーレスポンスが未定義
8. 【低】APIバージョンがない

## 2. 改善後のAPI一覧

| No | メソッド | エンドポイント | 認証 | 説明 |
|----|---------|--------------|------|------|
| 1 | GET | /api/v1/expenses | 必要 | 精算一覧取得 |
| 2 | POST | /api/v1/expenses | 必要 | 精算申請作成 |
| 3 | GET | /api/v1/expenses/:id | 必要 | 精算詳細取得 |
| 4 | PATCH | /api/v1/expenses/:id | 必要 | 精算申請修正 |
| 5 | DELETE | /api/v1/expenses/:id | 必要 | 精算申請取消 |
| 6 | PATCH | /api/v1/expenses/:id/approve | 必要(管理者) | 承認 |
| 7 | PATCH | /api/v1/expenses/:id/reject | 必要(管理者) | 却下 |
| 8 | POST | /api/v1/expenses/:id/receipts | 必要 | 領収書アップロード |

## 3. 改善後の精算申請API

### POST /api/v1/expenses

#### 認証
必要（Bearer Token）。申請者はトークンから特定。

#### リクエストボディ
| フィールド | 型 | 必須 | 制約 | 説明 |
|-----------|-----|------|------|------|
| amount | integer | ◯ | 1以上、1,000,000以下 | 金額（円） |
| category | string | ◯ | enum: transportation, meal, supplies, other | 経費区分 |
| description | string | ◯ | 1〜500文字 | 利用目的・内容 |
| expense_date | string | ◯ | YYYY-MM-DD形式、過去1年以内 | 経費発生日 |

#### リクエスト例
{
  "amount": 5000,
  "category": "transportation",
  "description": "顧客訪問のための新幹線代（東京→大阪）",
  "expense_date": "2026-03-28"
}

#### レスポンス（201 Created）
{
  "data": {
    "id": 101,
    "user_id": 45,
    "user_name": "鈴木 一郎",
    "amount": 5000,
    "category": "transportation",
    "description": "顧客訪問のための新幹線代（東京→大阪）",
    "expense_date": "2026-03-28",
    "status": "pending",
    "created_at": "2026-04-01T10:00:00Z"
  }
}

#### エラーケース
| ステータス | エラーコード | 条件 |
|-----------|------------|------|
| 400 | VALIDATION_ERROR | 入力値が不正 |
| 401 | UNAUTHORIZED | 認証トークンが無効 |
| 400 | INVALID_DATE | 経費発生日が1年以上前 |
```

</details>

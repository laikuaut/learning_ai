> **難易度**: ★★★☆☆（中級）
> **前提知識**: 第2章（テキスト生成AIの活用）、第4章（コード生成AIの活用）
> **課題の種類**: 設計課題
> **学習目標**: コード生成AIを活用してコードレビュー（Code Review）を効率化するためのプロンプト体系を設計し、レビュー品質を向上させる手法を身につける

## 課題の要件

1. 以下の5つのレビュー観点それぞれについて、専用のプロンプトを設計する
   - バグ検出（Bug Detection）
   - セキュリティ脆弱性（Security Vulnerability）
   - コードの可読性（Readability）
   - パフォーマンス（Performance）
   - 設計品質（Design Quality）
2. 各プロンプトには以下を含める
   - レビュー対象コードの入力方法
   - チェック項目の明示
   - 出力形式の指定（深刻度、該当箇所、改善案）
3. サンプルコード（Python、50行程度）を1つ用意し、5つのプロンプトすべてで実際にレビューする
4. 5つの個別レビューを統合する「総合レビューテンプレート」を作成する
5. AI支援コードレビューの運用ガイドライン（注意点・限界）をまとめる

import pickle

def get_user_data(user_id):
    # ユーザーデータをファイルから読み込む

### 運用ガイドライン

1. **AIレビューは第一段階のスクリーニングとして使う**: 人間のレビューを代替するものではない
2. **セキュリティ指摘は必ず人間が検証する**: AIの偽陽性（false positive）と偽陰性（false negative）の両方がある
3. **ビジネスロジックの正しさはAIに判断できない**: 仕様との整合性は人間が確認する
4. **定期的にプロンプトを更新する**: 新しい脆弱性パターンやベストプラクティスを反映する

### 改良版の特徴

- **プロンプトチェーン（Prompt Chaining）方式**を導入（段階的にレビューを深掘り）
- **レビュー結果の自動分類スキーム**を追加
- **修正後のコード全体**を生成するプロンプトを追加

---

#### プロンプトチェーン方式

1回のプロンプトですべてをレビューするのではなく、段階的に深掘りします。

```
【ステップ1：全体概要レビュー】
以下のコードの全体的な品質を、バグ・セキュリティ・可読性・
パフォーマンス・設計の5観点で、各1文でサマリーしてください。
深刻な問題がある観点には🔴、軽微な問題は🟡、問題なしは🟢を付けてください。

【ステップ2：深刻な観点の詳細分析】
ステップ1で🔴と判定された「セキュリティ」について、
詳細なレビューを実施してください。
（以下、詳細プロンプト）

【ステップ3：修正コードの生成】
ステップ2で指摘されたすべての問題を修正した
コード全体を生成してください。
修正箇所にはコメントで「# FIXED: ...」と記載してください。
```

---

#### 修正後コード生成プロンプト

```
以下のレビュー指摘に基づいて、修正後のコード全体を生成してください。

【指摘一覧】
1. pickle.loadをJSONに変更（セキュリティ）
2. パストラバーサル対策を追加（セキュリティ）
3. with文でファイル操作を安全化（バグ防止）
4. 例外処理を追加（バグ防止）
5. f-stringを使用（可読性）
6. for文をPythonicに変更（可読性）

【ルール】
- 修正箇所に「# FIXED: 理由」コメントを付ける
- 元のコードの機能は変えない
- 型ヒント（type hints）を追加する
```

**生成される修正コード:**
```python
import json
import os
import re
from pathlib import Path

# FIXED: pickleをjsonに変更（安全でないデシリアライゼーション対策）
# FIXED: 型ヒントを追加

def get_user_data(user_id: str) -> dict:
    """ユーザーデータをJSONファイルから読み込む"""
    # FIXED: パストラバーサル対策（user_idのバリデーション）
    if not re.match(r'^[a-zA-Z0-9_-]+$', user_id):
        raise ValueError(f"不正なユーザーID: {user_id}")

    base_dir = Path("/data/users")
    file_path = base_dir / f"{user_id}.json"

    # FIXED: パスが基準ディレクトリ外を参照していないか確認
    if not file_path.resolve().is_relative_to(base_dir.resolve()):
        raise ValueError(f"不正なファイルパス: {file_path}")

    # FIXED: with文でリソースリーク防止 + 例外処理追加
    try:
        with open(file_path, "r", encoding="utf-8") as f:
            return json.load(f)
    except FileNotFoundError:
        raise ValueError(f"ユーザーが見つかりません: {user_id}")
    except json.JSONDecodeError:
        raise ValueError(f"データファイルが破損しています: {user_id}")


def calculate_total(items: list[dict]) -> float:
    """注文商品の合計金額を計算する"""
    if not items:
        return 0.0
    # FIXED: Pythonicなイテレーション + 内包表記
    return sum(item["price"] * item["quantity"] for item in items)


def process_order(order_data: dict) -> str:
    """注文を処理し、結果を返す"""
    user = get_user_data(order_data["user_id"])
    total = calculate_total(order_data["items"])

    if total > 10000:
        discount = total * 0.1
        total -= discount

    # FIXED: f-stringで可読性向上
    result = f"User: {user['name']}, Total: {total:.0f}"
    print(result)
    return result
```

#### レビュー結果の自動分類スキーム

```
レビュー指摘
   │
   ├─ 【Must Fix（必須修正）】
   │    ├─ セキュリティ脆弱性（Critical/High）
   │    └─ 確実なバグ（高深刻度）
   │
   ├─ 【Should Fix（推奨修正）】
   │    ├─ 潜在的なバグ（中深刻度）
   │    ├─ パフォーマンス問題
   │    └─ 可読性の大幅な問題
   │
   └─ 【Nice to Have（改善提案）】
        ├─ コーディングスタイル
        ├─ 軽微な可読性改善
        └─ 設計の改善提案
```

この分類により、レビュー指摘の優先順位が明確になり、効率的に修正を進められます。

**初心者向けとの違い**:
- プロンプトチェーン方式で段階的に深掘りする手法を導入
- 修正後のコード全体を生成するプロンプトを追加し、「指摘して終わり」ではなく「修正まで完了」する流れを提示
- レビュー結果の自動分類スキームで、指摘の優先度を体系的に管理
- 型ヒントやdocstring等、モダンなPythonのベストプラクティスも反映

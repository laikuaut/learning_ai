# 第7章：セキュリティとヘッダー設定

## この章のゴール

- nginxのセキュリティ強化の基本方針を理解する
- セキュリティヘッダー（Security Headers）の設定方法を学ぶ
- アクセス制御（IP制限、Basic認証）を設定できるようになる
- レートリミット（Rate Limiting）で過剰なリクエストを抑制する方法を学ぶ
- DDoS対策やBot対策の基本を理解する

---

## 7.1 セキュリティの基本方針

### 最小権限の原則

```
【nginxセキュリティの基本方針】

1. バージョン情報を隠す        → 攻撃者への情報漏洩を防ぐ
2. セキュリティヘッダーを付与   → ブラウザのセキュリティ機能を活用
3. 不要なアクセスを制限        → IP制限、認証、レートリミット
4. SSL/TLSを適切に設定        → 通信の暗号化（第6章で解説済み）
5. ログを適切に記録            → インシデント対応の基盤
```

### バージョン情報の非表示

```nginx
http {
    # nginxのバージョン番号をレスポンスヘッダーから削除
    server_tokens off;
}
```

```
【server_tokens off の効果】

変更前のレスポンスヘッダー:
  Server: nginx/1.24.0     ← バージョンが露出

変更後のレスポンスヘッダー:
  Server: nginx             ← バージョン非表示
```

> **実務でのポイント**: バージョン情報が公開されていると、そのバージョンの既知の脆弱性を狙った攻撃を受ける可能性があります。`server_tokens off;` は必ず設定しましょう。

## 7.2 セキュリティヘッダー

### 主要なセキュリティヘッダー

```nginx
server {
    # --- XSS対策 ---
    # ブラウザの XSS フィルターを有効化
    add_header X-XSS-Protection "1; mode=block" always;

    # --- コンテンツタイプの偽装防止 ---
    # ブラウザにMIMEタイプのスニッフィングを禁止
    add_header X-Content-Type-Options "nosniff" always;

    # --- クリックジャッキング対策 ---
    # iframe での埋め込みを禁止
    add_header X-Frame-Options "SAMEORIGIN" always;

    # --- リファラー情報の制御 ---
    add_header Referrer-Policy "strict-origin-when-cross-origin" always;

    # --- Content Security Policy ---
    # 読み込み可能なリソースの制限
    add_header Content-Security-Policy "default-src 'self'; script-src 'self'; style-src 'self' 'unsafe-inline';" always;

    # --- HSTS（第6章参照）---
    add_header Strict-Transport-Security "max-age=63072000; includeSubDomains" always;

    # --- Permissions Policy ---
    # ブラウザ機能へのアクセスを制限
    add_header Permissions-Policy "camera=(), microphone=(), geolocation=()" always;
}
```

```
【各ヘッダーの役割まとめ】

┌──────────────────────────┬──────────────────────────────────┐
│ ヘッダー                  │ 防ぐ攻撃・目的                    │
├──────────────────────────┼──────────────────────────────────┤
│ X-XSS-Protection         │ クロスサイトスクリプティング（XSS）  │
│ X-Content-Type-Options   │ MIMEタイプスニッフィング            │
│ X-Frame-Options          │ クリックジャッキング                │
│ Referrer-Policy          │ リファラー情報の漏洩                │
│ Content-Security-Policy  │ 不正なリソース読み込み              │
│ Strict-Transport-Security│ HTTPダウングレード攻撃              │
│ Permissions-Policy       │ 不要なブラウザ機能の無効化          │
└──────────────────────────┴──────────────────────────────────┘
```

### snippets に切り出して再利用

```nginx
# /etc/nginx/snippets/security-headers.conf
add_header X-XSS-Protection "1; mode=block" always;
add_header X-Content-Type-Options "nosniff" always;
add_header X-Frame-Options "SAMEORIGIN" always;
add_header Referrer-Policy "strict-origin-when-cross-origin" always;
add_header Permissions-Policy "camera=(), microphone=(), geolocation=()" always;

# 各 server ブロックで include
server {
    listen 443 ssl;
    server_name example.com;
    include snippets/security-headers.conf;
    # ...
}
```

## 7.3 アクセス制御

### IPアドレスによるアクセス制限

```nginx
# 特定のIPのみ許可（ホワイトリスト）
location /admin/ {
    allow 192.168.1.0/24;    # 社内ネットワーク
    allow 10.0.0.5;          # 特定のIP
    deny all;                # それ以外を拒否
}

# 特定のIPを拒否（ブラックリスト）
location / {
    deny 203.0.113.50;       # 特定の攻撃元IP
    deny 198.51.100.0/24;    # 特定のネットワーク
    allow all;               # それ以外を許可
}
```

> **よくある間違い**: `allow` と `deny` の順序が重要です。nginx は上から順に評価し、最初にマッチしたルールを適用します。`deny all;` を `allow` の前に書くと、すべてのアクセスが拒否されます。

### Basic認証

```bash
# パスワードファイルの作成（htpasswd コマンド）
sudo apt install apache2-utils  # htpasswd を使うためにインストール
sudo htpasswd -c /etc/nginx/.htpasswd admin
# パスワードの入力を求められる
```

```nginx
# Basic認証の設定
location /admin/ {
    auth_basic "Restricted Area";
    auth_basic_user_file /etc/nginx/.htpasswd;
}

# 特定のlocationだけ認証を無効化
location /admin/health {
    auth_basic off;  # ヘルスチェック用に認証不要
}
```

## 7.4 レートリミット

### リクエストレートの制限

```nginx
http {
    # レートリミットゾーンの定義
    # $binary_remote_addr: クライアントIPごとに制限
    # zone=api:10m: "api" という名前で10MBのメモリを確保
    # rate=10r/s: 1秒あたり10リクエストまで
    limit_req_zone $binary_remote_addr zone=api:10m rate=10r/s;

    # ログインページ用（より厳しい制限）
    limit_req_zone $binary_remote_addr zone=login:10m rate=1r/s;

    server {
        # API全体に適用
        location /api/ {
            limit_req zone=api burst=20 nodelay;
            # burst=20: バースト時に20リクエストまで許容
            # nodelay: バースト分を即座に処理（遅延なし）
            proxy_pass http://localhost:3000;
        }

        # ログインには厳しく制限
        location /api/login {
            limit_req zone=login burst=5;
            # burst=5: 5リクエストまでキューに入れる
            # nodelay なし: 超過分は遅延処理
            proxy_pass http://localhost:3000;
        }
    }
}
```

```
【レートリミットの動作イメージ（rate=10r/s, burst=20）】

時間 →
リクエスト: ||||||||||||||||||||||||||||||||
             ├─ 10r/s ─┤
             │ 通常処理  │← burst=20 →│ 拒否(503) │
             │ (即座に)  │ (キュー/即座)│           │
```

### 接続数の制限

```nginx
http {
    # 同時接続数の制限ゾーン
    limit_conn_zone $binary_remote_addr zone=conn:10m;

    server {
        # 1つのIPから同時10接続まで
        limit_conn conn 10;

        # ダウンロードディレクトリは同時3接続まで
        location /downloads/ {
            limit_conn conn 3;
            limit_rate 500k;   # 帯域制限：500KB/秒
        }
    }
}
```

## 7.5 不要な情報の隠蔽とアクセス制限

### 隠しファイルへのアクセス拒否

```nginx
server {
    # .で始まるファイル（.git, .env, .htaccess 等）へのアクセス拒否
    location ~ /\. {
        deny all;
        access_log off;
        log_not_found off;
    }

    # 特定のファイルタイプへのアクセス拒否
    location ~* \.(bak|config|sql|fla|psd|ini|log|sh|inc|swp|dist)$ {
        deny all;
    }
}
```

### リクエストサイズの制限

```nginx
server {
    # リクエストボディの最大サイズ（デフォルト: 1MB）
    client_max_body_size 10m;    # 10MBまで許可

    # ファイルアップロード用のlocationでは緩める
    location /upload/ {
        client_max_body_size 100m;  # 100MBまで
    }
}
```

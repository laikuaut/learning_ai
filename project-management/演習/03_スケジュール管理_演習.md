## 基本問題

### 問題1：タスクの依存関係

以下のタスク一覧について、依存関係の種類（FS / SS / FF）を判定してください。

1. DB設計 → API実装
2. API実装の開始 → フロントエンド実装の開始（APIの一部ができ次第着手）
3. 実装の完了 → コードレビューの完了（同時に終わる）
4. 要件定義 → テスト計画策定
5. バックエンド実装の開始 → インフラ構築の開始（同時に着手可能）

**期待される出力例：**

```
1. FS（Finish to Start）
2. SS（Start to Start）
3. FF（Finish to Finish）
4. FS（Finish to Start）
5. SS（Start to Start）
```

<details>
<summary>ヒント</summary>

- FS：前のタスクが「終わったら」次が「始まる」
- SS：前のタスクが「始まったら」次も「始められる」
- FF：前のタスクが「終わるとき」次も「終わる」

</details>

<details>
<summary>解答例</summary>

```
1. FS（Finish to Start）
   → DB設計が完了してからAPI実装を開始する。必須依存。

2. SS（Start to Start）
   → API実装が始まったら、フロントエンド実装も開始可能。
     すべてのAPIを待たず、できた部分から着手できる。

3. FF（Finish to Finish）
   → 実装が完了するタイミングでコードレビューも完了する。
     並行して進み、同時に終わる関係。

4. FS（Finish to Start）
   → 要件定義が完了してからテスト計画を策定する。
     要件が確定しないとテスト観点が定まらない。

5. SS（Start to Start）
   → バックエンド実装とインフラ構築は同時に着手可能。
     互いに独立した作業として並行できる。
```

</details>

### 問題2：クリティカルパスの算出

以下のネットワーク図からクリティカルパスを求め、プロジェクト全体の最短完了日数を答えてください。

```
タスク一覧：
A: 3日（先行タスクなし）
B: 4日（先行タスク: A）
C: 6日（先行タスク: A）
D: 2日（先行タスク: B）
E: 3日（先行タスク: C）
F: 1日（先行タスク: D, E）
```

**期待される出力例：**

```
クリティカルパス：A → C → E → F
最短完了日数：13日
```

<details>
<summary>ヒント</summary>

すべてのパスを列挙し、それぞれの合計日数を計算してください。最も日数が多いパスがクリティカルパスです。

</details>

<details>
<summary>解答例</summary>

```
■ すべてのパスを列挙：

パス1: A → B → D → F = 3 + 4 + 2 + 1 = 10日
パス2: A → C → E → F = 3 + 6 + 3 + 1 = 13日

■ クリティカルパス: A → C → E → F（13日）

■ 最短完了日数: 13日

■ フロート（余裕時間）の計算：
パス1のフロート = 13 - 10 = 3日
→ タスクBまたはDが合計3日まで遅れても全体には影響しない

■ 図で表すと：
         ┌── B(4日) ── D(2日) ──┐
         │                       │
  A(3日)─┤                       ├── F(1日)
         │                       │
         └── C(6日) ── E(3日) ──┘
              ↑          ↑
         クリティカルパス上のタスク
```

</details>

### 問題3：PERT見積もり

以下のタスクについて、PERT法で期待値と標準偏差を求めてください。

| タスク | 楽観値 | 最頻値 | 悲観値 |
|--------|--------|--------|--------|
| ユーザー認証実装 | 3日 | 5日 | 13日 |
| 商品検索API | 2日 | 4日 | 8日 |
| 決済連携 | 5日 | 10日 | 21日 |

**期待される出力例：**

```
ユーザー認証実装：期待値 = 5.7日、標準偏差 = 1.7日
```

<details>
<summary>ヒント</summary>

計算式：
- 期待値 = (楽観値 + 4 x 最頻値 + 悲観値) / 6
- 標準偏差 = (悲観値 - 楽観値) / 6

</details>

<details>
<summary>解答例</summary>

```
■ ユーザー認証実装
  期待値 = (3 + 4×5 + 13) / 6 = (3 + 20 + 13) / 6 = 36 / 6 = 6.0日
  標準偏差 = (13 - 3) / 6 = 10 / 6 ≈ 1.7日

■ 商品検索API
  期待値 = (2 + 4×4 + 8) / 6 = (2 + 16 + 8) / 6 = 26 / 6 ≈ 4.3日
  標準偏差 = (8 - 2) / 6 = 6 / 6 = 1.0日

■ 決済連携
  期待値 = (5 + 4×10 + 21) / 6 = (5 + 40 + 21) / 6 = 66 / 6 = 11.0日
  標準偏差 = (21 - 5) / 6 = 16 / 6 ≈ 2.7日

■ まとめ
  | タスク           | 期待値  | 標準偏差 | 68%信頼区間     |
  |-----------------|---------|---------|----------------|
  | ユーザー認証実装  | 6.0日   | 1.7日   | 4.3 〜 7.7日   |
  | 商品検索API      | 4.3日   | 1.0日   | 3.3 〜 5.3日   |
  | 決済連携         | 11.0日  | 2.7日   | 8.3 〜 13.7日  |

※ 68%信頼区間 = 期待値 ± 標準偏差（1σ）
※ 95%信頼区間 = 期待値 ± 2×標準偏差（2σ）
```

</details>

### 問題4：ガントチャートの作成

以下のタスク一覧を基に、アスキーアートでガントチャートを作成してください。

| タスク | 所要期間 | 先行タスク | 担当 |
|--------|---------|-----------|------|
| 要件定義 | 2週間 | なし | PM |
| DB設計 | 1週間 | 要件定義 | バックエンド |
| API設計 | 1週間 | 要件定義 | バックエンド |
| UI設計 | 1週間 | 要件定義 | フロントエンド |
| API実装 | 2週間 | DB設計, API設計 | バックエンド |
| UI実装 | 2週間 | UI設計 | フロントエンド |
| 結合テスト | 1週間 | API実装, UI実装 | QA |
| リリース | 0.5週間 | 結合テスト | 全員 |

<details>
<summary>ヒント</summary>

DB設計とAPI設計は要件定義の後に並行して開始できます。UI設計も同様です。API実装はDB設計とAPI設計の両方が完了してから開始します。

</details>

<details>
<summary>解答例</summary>

```
タスク          | W1   | W2   | W3   | W4   | W5   | W6   | W7   |
----------------|------|------|------|------|------|------|------|
要件定義        | ████ | ████ |      |      |      |      |      |
DB設計          |      |      | ████ |      |      |      |      |
API設計         |      |      | ████ |      |      |      |      |
UI設計          |      |      | ████ |      |      |      |      |
API実装         |      |      |      | ████ | ████ |      |      |
UI実装          |      |      |      | ████ | ████ |      |      |
結合テスト      |      |      |      |      |      | ████ |      |
リリース        |      |      |      |      |      |      | ██   |
◆ リリース日    |      |      |      |      |      |      | ◆    |

クリティカルパス：
  要件定義(2w) → DB設計(1w) → API実装(2w) → 結合テスト(1w) → リリース(0.5w)
  = 6.5週間

並行作業：
  ・DB設計 / API設計 / UI設計 は並行実施（W3）
  ・API実装 / UI実装 は並行実施（W4〜W5）

プロジェクト全体：約7週間（約1.5ヶ月）
```

## チャレンジ問題

### 問題5：スケジュール遅延への対処

以下のシナリオを読んで、PMとしてどのような対策を講じるか検討してください。

**シナリオ：** 6週間のプロジェクトで3週目に入ったところ、以下の状況が判明しました。

- API実装がクリティカルパス上にあり、当初3日の見積もりだったが、外部APIの仕様が想定より複雑で、あと5日かかる見込み（2日の遅延）
- フロントエンド実装はフロート2日で順調に進行中
- テストフェーズは1週間の予定
- リリース日は変更不可（顧客との契約上の期限）

少なくとも3つの対策案を提示し、それぞれのメリット・デメリットを述べてください。

<details>
<summary>解答例</summary>

```
【状況分析】
・クリティカルパス上のタスクが2日遅延
・リリース日は変更不可
・フロントエンドにはフロート2日あり

【対策案】

■ 対策1：クラッシング（リソース追加投入）
  内容：API実装にもう1名エンジニアを追加し、作業を分担する。
  メリット：スケジュールの遅延を直接的に回復できる。
  デメリット：
  ・追加コストが発生する
  ・新メンバーの立ち上がりに時間がかかる（オンボーディングコスト）
  ・「ブルックスの法則」- 遅れているプロジェクトに人を追加すると
    さらに遅れる場合がある
  適用条件：分担可能な作業がある場合のみ有効

■ 対策2：ファストトラッキング（並行作業化）
  内容：API実装が一部完了した段階で結合テストを並行して開始する。
  メリット：追加コストなしでスケジュールを短縮できる。
  デメリット：
  ・手戻りのリスクが増える（後から仕様変更があると影響大）
  ・並行作業の管理が複雑になる
  ・テスト環境の準備が必要
  適用条件：完了したAPI部分から段階的にテスト可能な場合

■ 対策3：スコープの調整
  内容：テストフェーズの内容を見直し、自動テストでカバーできる範囲を
  拡大して手動テスト期間を短縮する。または、一部機能をフェーズ2に
  移動する。
  メリット：リリース日を確実に守れる。
  デメリット：
  ・テスト範囲縮小は品質リスクを伴う
  ・機能削減はステークホルダーの合意が必要
  適用条件：POの承認が得られる場合

■ 対策4：残業・休日出勤
  内容：チームに時間外労働をお願いする。
  メリット：短期間なら効果的。
  デメリット：
  ・メンバーのモチベーション低下
  ・長期化すると生産性が落ちる
  ・持続可能ではない
  適用条件：最後の手段として短期間に限定

【推奨対応】
対策2（ファストトラッキング）を主軸とし、API実装が完了した
モジュールから段階的にテストを開始する。
それでも間に合わない場合は対策3（スコープ調整）をPOに提案する。
```

</details>

## 6.1 品質管理の基本

### ソフトウェア品質とは

ソフトウェアにおける品質とは、**利用者の要求や期待を満たす度合い**のことです。単に「バグがない」ことだけではなく、使いやすさ、性能、セキュリティ、保守性など多面的な要素を含みます。

### ISO/IEC 25010 品質モデル

ソフトウェア品質の国際標準であるISO/IEC 25010では、8つの品質特性が定義されています。

| 品質特性 | 説明 | ソフトウェアでの例 |
|---------|------|------------------|
| **機能適合性** | 要求された機能を正確に提供するか | 検索が正しい結果を返す |
| **性能効率性** | 適切な応答速度・リソース効率か | ページ読み込みが2秒以内 |
| **互換性** | 他のシステムと共存・連携できるか | 異なるブラウザで正しく動作する |
| **使用性** | 利用者にとって使いやすいか | 直感的なUI、エラーメッセージが分かりやすい |
| **信頼性** | 障害なく安定して動作するか | 99.9%の稼働率 |
| **セキュリティ** | 不正なアクセスからデータを保護するか | SQLインジェクション対策 |
| **保守性** | 修正・改善が容易か | モジュール化されたコード |
| **移植性** | 異なる環境に移行できるか | コンテナ化されたデプロイ |

### QA（品質保証）とQC（品質管理）

| 項目 | QA（Quality Assurance） | QC（Quality Control） |
|------|------------------------|----------------------|
| 目的 | 品質問題の**予防** | 品質問題の**検出** |
| タイミング | プロセス全体に渡って実施 | 成果物の検査時に実施 |
| 対象 | プロセス（やり方） | プロダクト（成果物） |
| 手法 | プロセス監査、標準策定、教育 | テスト、レビュー、インスペクション |
| 例 | コーディング規約の策定 | 単体テストの実行 |
| 例 | コードレビュー制度の導入 | バグの検出と報告 |
| 例 | CI/CDパイプラインの構築 | リリース前の受入テスト |

```
品質管理の関係性：

  品質マネジメント（QM: Quality Management）
  ├── 品質計画（Quality Planning）
  │   └── 品質基準、テスト戦略の策定
  ├── 品質保証（QA: Quality Assurance）
  │   └── プロセスの改善と予防活動
  └── 品質管理（QC: Quality Control）
      └── 成果物の検査と不良の検出
```

> **現場でのポイント**: 「QA＝テストする人」という認識は不正確です。QAの本質は品質を「保証」すること、つまりプロセスを整備して品質問題を予防することです。テストはQCの活動に分類されます。

## 6.2 テスト戦略

### テストピラミッド

テストピラミッドは、テストの種類ごとの**理想的な量と速度のバランス**を示すモデルです。下位のテストほど数を多く、上位ほど数を少なくするのが基本方針です。

```
          /\
         /  \        E2Eテスト（少数）
        / E2E\       ・実際のブラウザで全体を通して検証
       /------\      ・遅い、壊れやすい、高コスト
      /        \
     / 統合テスト \    統合テスト（中程度）
    /  Integration\   ・複数モジュールの連携を検証
   /--------------\   ・中程度の速度、中程度のコスト
  /                \
 /   単体テスト     \  単体テスト（多数）
/    Unit Tests     \  ・関数やクラス単位で検証
/────────────────────\  ・高速、安定、低コスト
```

### テストの種類

| テスト種類 | 対象範囲 | 実行速度 | 維持コスト | 実施者 |
|-----------|---------|---------|-----------|--------|
| **単体テスト** | 関数・メソッド | 非常に速い | 低い | 開発者 |
| **統合テスト** | モジュール間連携 | 速い | 中程度 | 開発者 |
| **E2Eテスト** | システム全体 | 遅い | 高い | QA/開発者 |
| **性能テスト** | 応答時間・スループット | 遅い | 高い | QA/SRE |
| **セキュリティテスト** | 脆弱性 | 中程度 | 中程度 | セキュリティ担当 |
| **受入テスト** | ビジネス要件の充足 | 遅い | 中程度 | PO/ユーザー |

### 各テスト種類の詳細

**単体テスト（Unit Test）：**

```
目的：個々の関数やメソッドが正しく動作するかを検証

テスト対象の例：
  ・計算ロジック（税込み価格の計算が正しいか）
  ・バリデーション（メールアドレスの形式チェック）
  ・条件分岐（ユーザーの権限に応じた処理の分岐）

カバレッジの目安：
  ・ステートメントカバレッジ: 80%以上
  ・ブランチカバレッジ: 70%以上
  ・クリティカルな処理: 100%近く目指す
```

**統合テスト（Integration Test）：**

```
目的：複数のモジュールを組み合わせた時に正しく動作するかを検証

テスト対象の例：
  ・APIとデータベースの連携
  ・複数のマイクロサービス間の通信
  ・外部APIとの連携
  ・認証フローの一連の処理

注意点：
  ・外部サービスはモックやスタブを使うことが多い
  ・テスト環境の構築と維持にコストがかかる
```

**E2Eテスト（End-to-End Test）：**

```
目的：ユーザーの操作シナリオに沿って、システム全体が正しく動作するかを検証

テスト対象の例：
  ・ユーザー登録 → ログイン → 商品購入 → 確認メール受信
  ・管理画面でのデータ登録 → フロントでの表示確認

ツール例：
  ・Playwright
  ・Cypress
  ・Selenium

注意点：
  ・最も壊れやすい（UI変更の影響を受ける）
  ・実行時間が長い
  ・主要なシナリオに絞って実施する
```

## 6.3 テスト計画

### テスト計画書に含める要素

```
テスト計画書の構成：

1. テスト方針・目的
   → 何のためにテストするか、品質目標は何か

2. テスト対象・範囲
   → テストする機能としない機能を明確にする

3. テスト種類とレベル
   → 単体、統合、E2E、性能等のどれを実施するか

4. テスト環境
   → 開発環境、ステージング環境、本番同等環境

5. テストスケジュール
   → 各テストフェーズの期間と担当

6. 合格基準（Exit Criteria）
   → テスト完了の判断基準

7. リスクと対策
   → テスト活動に関するリスク

8. ツール・インフラ
   → テストに使用するツールと環境
```

### 合格基準の設定例

```
テスト完了の合格基準：

  ・全テストケースの実行完了率: 100%
  ・テスト合格率: 95%以上
  ・重大（Critical）バグ: 0件
  ・高（High）バグ: 未修正0件
  ・中（Medium）バグ: 未修正5件以下
  ・カバレッジ: ステートメント80%以上
  ・性能テスト: 全APIが応答時間200ms以内
  ・セキュリティ: OWASP Top 10の脆弱性0件
```

### バグの重要度分類

| 重要度 | 定義 | 例 | 対応期限 |
|--------|------|-----|---------|
| **Critical** | システムが使用不可 | アプリが起動しない、データ消失 | 即時対応 |
| **High** | 主要機能が使用不可 | 決済ができない、ログインできない | 24時間以内 |
| **Medium** | 機能に不具合があるが回避策あり | 検索結果の並び順が不正 | 次のリリースまで |
| **Low** | 軽微な不具合 | 表示の微調整、タイポ | 余裕があれば対応 |

## 6.4 コードレビュー

### コードレビューの目的

コードレビューは、**バグの早期発見だけでなく、チーム全体のコード品質とスキル向上を目的とした活動**です。

```
コードレビューの4つの目的：

1. バグ・欠陥の早期発見
   → テストでは見つけにくい論理的な誤りを検出

2. コード品質の維持
   → コーディング規約の遵守、設計パターンの適用

3. 知識の共有
   → チーム内でのノウハウの伝搬

4. 集団的コード所有権
   → 「誰が書いたか」ではなく「チームのコード」という意識
```

### コードレビューの手法

| 手法 | 説明 | 適用場面 |
|------|------|---------|
| **プルリクエストレビュー** | PRベースでオンラインレビュー | 日常的な開発（最も一般的） |
| **ペアプログラミング** | 2人で同時にコーディング | 複雑な機能、オンボーディング |
| **モブプログラミング** | チーム全員で1つの画面を見ながら | 設計判断、困難な問題 |
| **フォーマルインスペクション** | 正式な手順に従った検査 | 重要なモジュール、規制対応 |

### プルリクエストレビューのベストプラクティス

**レビュアー（レビューする側）：**

```
良いレビューのポイント：

1. コードの意図を理解してからコメントする
   × 「なぜこの変数名？」
   ○ 「この変数名だと○○と混同しそうです。△△はいかがでしょう？」

2. 具体的な改善案を提示する
   × 「この実装は良くない」
   ○ 「N+1クエリが発生しています。joinを使うと改善できます」

3. 良い点も指摘する
   ○ 「このヘルパー関数の抽出、再利用性が高くて良いですね」

4. 質問形式で指摘する
   ○ 「ここでnullチェックがないようですが、意図的でしょうか？」

5. 重要度を明示する
   ○ 「[MUST] セキュリティ上の問題があります」
   ○ 「[NIT] 好みの問題ですが、こちらの書き方もあります」
   ○ 「[QUESTION] この設計の意図を教えてください」
```

**レビュイー（レビューを受ける側）：**

```
良いPRの作り方：

1. PRのサイズを小さくする
   → 目安：変更行数300行以内、1つの論理的な変更単位

2. PRの説明を充実させる
   → 何を変えたか、なぜ変えたか、テスト方法

3. セルフレビューを先に行う
   → 自分で一度見直してからレビュー依頼

4. テストが通っている状態で出す
   → CI/CDが緑の状態でレビュー依頼

PR テンプレートの例：
  ## 概要
  ユーザー認証機能にMFA（多要素認証）を追加

  ## 変更内容
  - TOTP認証の実装
  - 認証画面のUI追加
  - 関連するテストの追加

  ## テスト方法
  - 単体テスト: `npm test -- --grep "MFA"`
  - 手動テスト: ログイン画面でQRコードをスキャン

  ## スクリーンショット
  （UIの変更がある場合）

  ## チェックリスト
  - [ ] テストが追加/更新されている
  - [ ] ドキュメントが更新されている
  - [ ] セキュリティの考慮がされている
```

> **よくある間違い**: レビューで人格を攻撃してしまうこと。「このコードはひどい」ではなく「このコードにはN+1問題があります」のように、コードに対して指摘しましょう。また、スタイルの好みの違い（タブ vs スペースなど）はリンターに任せ、レビューではロジックと設計に集中しましょう。

## 6.5 品質メトリクス

### ソフトウェア品質の測定指標

| メトリクス | 説明 | 目標値の例 |
|-----------|------|-----------|
| **バグ密度** | コード1,000行あたりのバグ数 | <1.0 bugs/KLOC |
| **テストカバレッジ** | テストで実行されるコードの割合 | >80% |
| **MTBF（平均故障間隔）** | 障害と障害の間の平均時間 | >720時間（30日） |
| **MTTR（平均復旧時間）** | 障害発生から復旧までの平均時間 | <1時間 |
| **リリース後バグ数** | リリース後に発見されたバグの数 | 0件（Critical） |
| **技術的負債比率** | 技術的負債の解消にかかる時間/開発時間 | <5% |
| **コード複雑度** | 循環的複雑度（Cyclomatic Complexity） | <10/関数 |
| **PRレビュー時間** | PRが作成されてからマージされるまで | <24時間 |

### バグ収束曲線

テスト期間中のバグ発見数の推移を追跡し、品質の収束を判断します。

```
バグ発見数
（件/週）
  30│ ★
    │  ★
  25│
    │   ★
  20│
    │    ★
  15│
    │      ★
  10│        ★
    │          ★
   5│            ★   ★
    │                   ★  ★
   0│─────────────────────────→ 週
    W1  W2  W3  W4  W5  W6  W7  W8

→ バグ発見数が減少傾向にある = 品質が収束に向かっている
→ バグ発見数が横ばいや増加傾向 = 品質が不安定（テスト強化が必要）
```

### 品質ダッシュボードの例

```
┌───────────────────────────────────────────────┐
│          品質ダッシュボード（月次）            │
├───────────────────────────────────────────────┤
│                                               │
│  テストカバレッジ: 83.2% [=========>  ]       │
│  バグ密度:         0.7 bugs/KLOC  (良好)      │
│                                               │
│  未解決バグ:                                  │
│    Critical: 0件  High: 2件                   │
│    Medium: 8件    Low: 15件                   │
│                                               │
│  今月のバグ修正率: 92%                        │
│  平均PRレビュー時間: 18時間                   │
│  デプロイ頻度: 12回/月                        │
│  デプロイ失敗率: 8%                           │
│                                               │
│  MTBF: 820時間  MTTR: 45分                    │
│                                               │
└───────────────────────────────────────────────┘
```

## 6.6 継続的な品質改善

### シフトレフト（Shift Left）

品質活動を開発プロセスの**より早い段階（左側）に移す**考え方です。バグは発見が遅くなるほど修正コストが指数的に増大します。

```
バグ修正コストの増大（相対値）：

フェーズ          | 修正コスト
──────────────────|──────────
要件定義で発見     | 1x
設計で発見        | 3〜6x
実装で発見        | 10x
テストで発見      | 15〜40x
本番で発見        | 30〜100x

→ テストで見つけるよりも、要件定義や設計の段階で
   見つける方が圧倒的にコストが低い
```

### シフトレフトの実践方法

```
要件定義段階：
  ・要件レビュー、ウォークスルー
  ・受入基準（Acceptance Criteria）の明確化
  ・テスト可能な要件の記述

設計段階：
  ・設計レビュー（アーキテクチャレビュー）
  ・脅威モデリング（セキュリティ設計レビュー）
  ・テスト設計の先行実施（BDD/TDD）

実装段階：
  ・テスト駆動開発（TDD）
  ・ペアプログラミング
  ・静的解析ツールの活用
  ・コードレビューの徹底

CI/CD段階：
  ・自動テストの実行
  ・品質ゲートの設定
  ・セキュリティスキャンの自動化
```

### CI/CDにおける品質ゲート

```
プルリクエスト時の品質ゲート：

  ┌──────────────────────────────┐
  │ PR作成                       │
  │  ↓                          │
  │ 1. リンター/フォーマッター   │ → 不合格なら自動修正
  │  ↓                          │
  │ 2. 単体テスト実行            │ → 1件でも失敗したらブロック
  │  ↓                          │
  │ 3. テストカバレッジ確認      │ → 80%未満ならブロック
  │  ↓                          │
  │ 4. 静的解析（SonarQube等）   │ → Critical問題があればブロック
  │  ↓                          │
  │ 5. セキュリティスキャン      │ → 脆弱性があればブロック
  │  ↓                          │
  │ 6. コードレビュー承認        │ → 1名以上の承認が必要
  │  ↓                          │
  │ マージ許可                   │
  └──────────────────────────────┘
```

### 品質改善のPDCAサイクル

```
Plan（計画）:
  ・品質目標の設定（例：バグ密度を0.5以下にする）
  ・改善策の立案（例：コードレビューチェックリストの導入）

Do（実行）:
  ・改善策の実施
  ・チームへの展開

Check（確認）:
  ・品質メトリクスの測定
  ・目標との差異分析

Act（改善）:
  ・効果があれば標準化
  ・効果が不十分なら改善策を修正
  ・次のサイクルへ
```

> **現場でのベストプラクティス**: 品質は一朝一夕には改善しません。チームの文化として品質意識を根付かせることが最も重要です。「品質は全員の責任」という意識を持ち、定期的なレトロスペクティブで品質に関する議論を行いましょう。

## 6.7 品質管理のアンチパターン

避けるべき品質管理のアンチパターンを紹介します。

| アンチパターン | 問題点 | 改善策 |
|--------------|--------|--------|
| **テストを後回しにする** | バグの発見・修正コストが増大 | テストファーストの文化を醸成 |
| **カバレッジ至上主義** | 意味のないテストが増える | 重要なパスのカバレッジを重視 |
| **手動テストに依存** | 回帰テストに時間がかかる | 自動テストの比率を高める |
| **レビューの形骸化** | 「LGTM」だけで承認してしまう | レビュー観点のチェックリスト化 |
| **品質をQAだけの責任にする** | 開発者の品質意識が低下 | 品質は全員の責任であることを共有 |
| **リリース前に集中テスト** | テスト期間が圧迫される | 継続的テストの実施 |

## まとめ

| 概念 | ポイント |
|------|---------|
| ソフトウェア品質 | 機能面だけでなく、性能、セキュリティ、保守性等を含む多面的な概念 |
| QA vs QC | QAはプロセスの改善（予防）、QCは成果物の検査（検出） |
| テストピラミッド | 単体テストを厚く、E2Eテストは薄くするのが原則 |
| コードレビュー | バグ発見だけでなく、知識共有とコード品質向上が目的 |
| シフトレフト | 品質活動をより早い段階に移すことで修正コストを削減 |
| 品質ゲート | CI/CDに組み込んだ自動的な品質チェック |
| 品質メトリクス | バグ密度、カバレッジ、MTBF/MTTR等で定量的に測定 |
| PDCA | 品質は継続的に改善していくもの |

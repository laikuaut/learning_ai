## 課題の要件

### 設計するテンプレート（8本）

以下の8つの業務シーンに対応するプロンプトテンプレートを設計してください。

| 番号 | カテゴリ | テンプレート名 | 難易度 |
|:----:|---------|--------------|:------:|
| 1 | 営業 | 提案書のドラフト作成 | 中 |
| 2 | 営業 | クライアント向けフォローアップメール生成 | 低 |
| 3 | マーケティング | 競合製品の比較分析 | 高 |
| 4 | 開発 | コードレビュー指摘事項の生成 | 中 |
| 5 | 開発 | 技術仕様書のドラフト作成 | 高 |
| 6 | 管理 | 会議の議事録作成 | 低 |
| 7 | 管理 | プロジェクト企画書の構成案作成 | 中 |
| 8 | 共通 | 長文ドキュメントの要約 | 低 |

### 各テンプレートに含めるもの

1. **テンプレートID**: 一意の識別子（例：SALES-001）
2. **利用場面の説明**: いつ、誰が、何のために使うか
3. **入力パラメータ**: ユーザーが埋める変数（`{変数名}`形式）
4. **プロンプト本体**: 変数を含む完全なプロンプト
5. **出力仕様**: 期待する出力の形式・長さ・品質基準
6. **使用例**: 変数を埋めた具体的な使用例1つ
7. **注意事項**: 使用時の注意点やよくある間違い

### 追加要件

- **テンプレート設計ガイドライン**: 新しいテンプレートを作成する際の標準ルール
- **バージョン管理方針**: テンプレートの更新・改善のプロセス

<details>
<summary>ステップ2：入力パラメータを設計する</summary>

各テンプレートの入力パラメータを設計する際のポイントです。

**パラメータ設計の原則**:
1. **必須パラメータは最小限に**: 入力の手間を減らす
2. **オプションパラメータで柔軟性を確保**: 詳細を追加したい場合に対応
3. **デフォルト値を設定**: 空欄の場合のフォールバック（fallback）を定義
4. **入力例を必ず付ける**: 何を入れるべきか迷わないようにする

**テンプレート6（議事録作成）のパラメータ例**:

| パラメータ | 必須 | 説明 | デフォルト値 | 例 |
|-----------|:----:|------|-------------|-----|
| `{会議名}` | ○ | 会議の名称 | - | 週次定例ミーティング |
| `{参加者}` | ○ | 参加者の名前と役職 | - | 田中（PM）、山田（開発） |
| `{議題}` | ○ | 議題の一覧 | - | 進捗確認、次期計画 |
| `{発言記録}` | ○ | 各参加者の発言内容 | - | （発言内容を箇条書きで） |
| `{出力言語}` | × | 出力する言語 | 日本語 | 日本語、英語 |
| `{詳細度}` | × | 議事録の詳細度 | 標準 | 簡潔、標準、詳細 |

</details>

```
- 長さ: 200〜300文字

```

**ポイント**:

<details>
<summary>ステップ4：テンプレート設計ガイドラインを作成する</summary>

新しいテンプレートを作成する際のルールを定めましょう。

以下の観点を含めてください：

1. **命名規則**: テンプレートIDの採番ルール
2. **パラメータの命名規則**: `{日本語名}`で統一するか `{english_name}` で統一するか
3. **品質基準**: テンプレートが満たすべき最低要件
4. **レビュープロセス**: テンプレートを公開する前の承認フロー
5. **バージョニング**: 更新時のバージョン番号の付け方
6. **廃止ルール**: 使われなくなったテンプレートの扱い

</details>

### テンプレート1：提案書のドラフト作成

```
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
テンプレートカード
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

ID: SALES-001  バージョン: v1.0
名称: 提案書ドラフト作成
対象者: 営業担当者、アカウントマネージャー
最終更新: 2026年4月
```

**利用場面**:
クライアントへの新規提案や既存契約の更新時に、提案書の初稿を迅速に作成したい場合に使用します。営業担当者が商談前の準備として利用することを想定しています。

**入力パラメータ**:

| パラメータ | 必須 | 説明 | 例 |
|-----------|:----:|------|-----|
| `{クライアント名}` | ○ | 提案先の企業名 | 株式会社ABC商事 |
| `{課題}` | ○ | クライアントが抱える課題 | カスタマーサポートの応答時間が長い |
| `{提案内容}` | ○ | 提案するソリューションの概要 | AIチャットボットの導入 |
| `{予算感}` | × | 想定予算の範囲 | 月額30〜50万円 |
| `{競合情報}` | × | 競合他社の情報 | B社が既に提案済み |
| `{期限}` | × | 提案書の提出期限 | 2026年4月末 |

**プロンプト本体**:

```
あなたはIT企業のソリューション営業の専門家です。

以下の情報に基づいて、提案書のドラフトを作成してください。

## クライアント情報
- 企業名: {クライアント名}
- 抱えている課題: {課題}

## 提案内容
- ソリューション: {提案内容}
- 想定予算: {予算感}
- 競合状況: {競合情報}

## 提案書の構成（以下の構成に従ってください）
1. エグゼクティブサマリー（100文字以内）
2. クライアントの課題整理（箇条書き3〜5個）
3. 提案するソリューションの概要（200文字以内）
4. 導入効果の期待値（定量的な指標を含む）
5. 概算費用とスケジュール
6. 次のステップの提案

## 出力条件
- ビジネスフォーマルなトーン
- 合計1,000〜1,500文字
- 専門用語は最小限に（経営層が読むことを想定）
```

**使用例**:

```
あなたはIT企業のソリューション営業の専門家です。

以下の情報に基づいて、提案書のドラフトを作成してください。

## クライアント情報
- 企業名: 株式会社ABC商事
- 抱えている課題: カスタマーサポートの応答時間が平均15分と長く、顧客満足度が低下している

## 提案内容
- ソリューション: AIチャットボット「SmartSupport」の導入
- 想定予算: 月額30〜50万円
- 競合状況: B社のチャットボットサービスが先に提案済みだが、カスタマイズ性に不満あり

（以下省略）
```

**注意事項**:
- 具体的な数値（ROI、導入効果）はAIの推定値です。必ず自社の実績データで裏付けを取ってください
- 競合情報は機密性に注意し、外部のAIサービスに入力する際は社内規程を確認してください
- ドラフトはあくまで初稿です。必ず上長のレビューを経てからクライアントに提出してください

**注意事項**:

- 長すぎるドキュメント（1万文字超）はセクションごとに分割して要約してください

改良版では、**条件分岐**と**段階的な詳細化**を含むテンプレートを設計します。

```
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

名称: 技術仕様書ドラフト作成
対象者: テックリード、シニアエンジニア
最終更新: 2026年4月
```

**入力パラメータ**:

| パラメータ | 必須 | 説明 | デフォルト値 | 例 |
|-----------|:----:|------|-------------|-----|
| `{機能名}` | ○ | 実装する機能の名称 | - | ユーザー認証API |
| `{機能概要}` | ○ | 機能の説明（3〜5文） | - | JWTを使用した... |
| `{技術スタック}` | ○ | 使用する技術 | - | Node.js, PostgreSQL |
| `{非機能要件}` | × | 性能、セキュリティ等 | 特に指定なし | レスポンス200ms以内 |
| `{既存システム}` | × | 連携する既存システム | なし | ユーザー管理DB |
| `{詳細度}` | × | 仕様書の詳細レベル | 標準 | 概要/標準/詳細 |

**プロンプト本体（条件分岐付き）**:

```
あなたはシニアソフトウェアアーキテクトです。

以下の情報に基づいて、技術仕様書のドラフトを作成してください。

## 基本情報
- 機能名: {機能名}
- 機能概要: {機能概要}
- 技術スタック: {技術スタック}
- 非機能要件: {非機能要件}
- 既存システム連携: {既存システム}

## 仕様書の構成

### 必須セクション（すべての詳細度で含める）
1. 機能概要（目的、背景、スコープ）
2. APIエンドポイント一覧（メソッド、パス、概要を表形式で）
3. データモデル（主要なエンティティと属性を表形式で）

### 詳細度が「標準」以上の場合、追加で含める
4. シーケンス図（テキストで主要フローを記述）
5. エラーハンドリング一覧（エラーコード、原因、対応を表形式で）
6. セキュリティ要件（認証、認可、データ保護）

### 詳細度が「詳細」の場合、さらに追加で含める
7. パフォーマンス要件と測定方法
8. テスト戦略（単体テスト、結合テスト、E2Eテストの範囲）
9. デプロイメント手順
10. 移行計画（既存システムからの移行がある場合）

## 出力条件
- 技術者が読むことを前提とした正確な表現
- 曖昧な表現は避け、数値や具体的な仕様で記述
- 「TBD（To Be Determined: 要検討）」と明記すべき未決定事項があれば記載
```

```
│     - メジャー更新（構造変更）: v1.0 → v2.0            │

│                                                         │
│  6. 廃止ルール                                           │
│     - 3ヶ月間使用実績がないテンプレートを棚卸し対象     │
```

### バージョン管理と改善プロセス（改良版）

```
改善サイクル（月次）：

1. 利用状況の収集
   ├─ 各テンプレートの使用回数
   ├─ ユーザーからのフィードバック
   └─ 出力品質のサンプルチェック（月5件）

2. 改善候補の特定
   ├─ 使用頻度が高いが評価が低いテンプレート → 優先改善
   ├─ 新しい業務ニーズ → 新規テンプレートの作成
   └─ LLMのアップデートに伴う調整 → 既存テンプレートの再テスト

3. 改善の実施
   ├─ A/Bテストで改善効果を検証
   ├─ レビュープロセスを経て更新
   └─ 変更ログを記録

4. 展開
   ├─ 更新内容をチーム全体に通知
   ├─ 旧バージョンとの違いを説明
   └─ 必要に応じてトレーニングを実施
```

**初心者向けとの違い**:
- 条件分岐を含む柔軟なテンプレート設計（詳細度パラメータによる出力制御）
- テンプレート設計ガイドラインを体系的に文書化
- レビュープロセスと品質基準を組織運用レベルで設計
- バージョン管理と月次改善サイクルを含む運用プロセスを定義

# 第7章：画面設計書

## この章のゴール

- 画面設計書の目的と構成を理解する

## 7.1 画面設計書とは

### 画面設計書の位置づけ

画面設計書は、**ユーザーが操作する画面のレイアウト、遷移、項目を定義する文書**です。基本設計の一部として作成され、フロントエンド開発とUIデザインの基盤となります。

```
【画面設計のプロセス】

画面一覧     画面遷移図     ワイヤーフレーム    画面項目定義書
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
どの画面が    画面間の      各画面の         各項目の
必要か？      つながりは？   レイアウトは？     仕様は？
```

### 画面設計書の構成

```
画面設計書の目次（例）
━━━━━━━━━━━━━━━━━━━━━━━
1. 画面設計方針
   1.1 対応デバイス・画面サイズ
   1.2 UIフレームワーク
   1.3 共通レイアウト

2. 画面一覧

3. 画面遷移図

4. 各画面の設計
   4.1 ワイヤーフレーム
   4.2 画面項目定義書
   4.3 画面アクション定義
━━━━━━━━━━━━━━━━━━━━━━━
```

## 7.4 ワイヤーフレーム

### ワイヤーフレームとは

ワイヤーフレームは、画面のレイアウトを**骨格レベル**で表現した図です。色やデザインの詳細は含めず、要素の配置と情報の構造を伝えることが目的です。

## 7.5 画面項目定義書

### 画面項目定義書の書き方

画面上の各入力項目・表示項目について、詳細な仕様を定義します。

```
【画面項目定義書（例）：SCR-003 ログイン画面】

画面ID  ：SCR-003
画面名  ：ログイン

■ 入力項目

| No | 項目名        | 項目ID    | 種別        | 必須 | 初期値 | 最大長 | バリデーション                | エラーメッセージ                   |
|----|-------------|----------|------------|------|-------|-------|----------------------------|---------------------------------|
| 1  | メールアドレス | email    | テキスト入力 | ○    | 空     | 255   | メール形式（RFC 5322）       | 「正しいメールアドレスを入力してください」|
| 2  | パスワード    | password | パスワード  | ○    | 空     | 100   | 1文字以上                    | 「パスワードを入力してください」       |
| 3  | パスワード表示 | show_pw  | チェックボックス| -  | OFF   | -     | -                          | -                               |

■ ボタン・リンク

| No | 項目名             | 項目ID      | 種別           | アクション                    |
|----|-------------------|------------|---------------|------------------------------|
| 4  | ログイン           | btn_login  | プライマリボタン | POST /auth/login を呼び出す   |
| 5  | パスワードをお忘れの方| link_reset | テキストリンク  | パスワードリセット画面に遷移    |
| 6  | Googleでログイン    | btn_google | ボタン         | Google OAuth認証を開始        |
| 7  | 新規会員登録        | link_signup| テキストリンク  | 会員登録画面に遷移             |

■ アクション定義

【ログインボタン押下時の処理】

1. フロントエンドバリデーション
   ・メールアドレスが空 → エラーメッセージ表示
   ・メールアドレスの形式不正 → エラーメッセージ表示
   ・パスワードが空 → エラーメッセージ表示

2. バリデーションOKの場合
   ・ログインボタンをローディング状態にする
   ・POST /auth/login を呼び出す

3. API成功時（200）
   ・トークンをlocalStorageに保存する
   ・リダイレクト元の画面に遷移する
   ・リダイレクト元がない場合はトップページに遷移する

4. API失敗時（401）
   ・「メールアドレスまたはパスワードが正しくありません」を表示
   ・パスワード入力欄をクリアする
   ・ログインボタンを元に戻す
```

## 7.7 ユーザビリティの考慮点

### 画面設計のチェックリスト

```
【ユーザビリティのチェックリスト】

□ 視認性
  ├─ □ 文字サイズは十分か（本文16px以上推奨）
  ├─ □ コントラスト比は十分か（WCAG AA 4.5:1以上）
  └─ □ 重要な情報が目立つ位置にあるか

□ 操作性
  ├─ □ ボタンやリンクの領域は十分か（44px×44px以上）
  ├─ □ フォームのラベルと入力欄が紐づいているか
  ├─ □ Tabキーで論理的な順序でフォーカスが移動するか
  └─ □ エラー時にどの項目がエラーか明確か

□ フィードバック
  ├─ □ ボタン押下後にローディング表示があるか
  ├─ □ 処理成功時にメッセージが表示されるか
  ├─ □ エラー時にわかりやすいメッセージが表示されるか
  └─ □ 確認が必要な操作（削除等）にダイアログがあるか

□ 一貫性
  ├─ □ ボタンの色・配置が全画面で統一されているか
  ├─ □ プライマリアクションが右、キャンセルが左か
  └─ □ 用語が全画面で統一されているか
```

> **実務でのポイント**: 画面設計書では「何を表示するか」だけでなく、「エラー時にどうなるか」「ローディング中はどう見えるか」「データが0件の場合はどう表示するか」といった**状態のバリエーション**を定義することが重要です。これを「ステートバリエーション」と呼びます。

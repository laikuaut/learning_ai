### 問題1：セキュリティ脆弱性の発見（基本）

以下のMCPサーバーのコードには複数のセキュリティ上の問題があります。脆弱性を全て指摘し、それぞれの修正案を示してください。

```typescript
import { McpServer } from "@modelcontextprotocol/sdk/server/mcp.js";
import { z } from "zod";

const server = new McpServer({ name: "file-server", version: "1.0.0" });
const API_KEY = "sk-abc123secret456key789";  // (A)

server.tool(
  "read_file",
  { path: z.string() },                      // (B)
  async ({ path }) => {
    const fs = await import("fs/promises");
    const content = await fs.readFile(path, "utf-8");  // (C)
    return {
      content: [{ type: "text", text: content }],
    };
  }
);

server.tool(
  "run_query",
  { table: z.string(), filter: z.string() },  // (D)
  async ({ table, filter }) => {
    const query = `SELECT * FROM ${table} WHERE ${filter}`;  // (E)
    const result = await db.execute(query);
    return {
      content: [{ type: "text", text: JSON.stringify(result) }],
    };
  }
);

server.tool(
  "execute_command",
  { command: z.string() },                     // (F)
  async ({ command }) => {
    const { exec } = await import("child_process");
    return new Promise((resolve) => {
      exec(command, (error, stdout) => {        // (G)
        resolve({
          content: [{ type: "text", text: stdout || error?.message || "" }],
        });
      });
    });
  }
);
```

**期待される出力例：**
```
=== セキュリティ脆弱性レビュー結果 ===

脆弱性1 (A): APIキーのハードコード
  リスク: 高
  説明: ソースコードにAPIキーが直接記述されている
  修正: 環境変数から読み取る（process.env.API_KEY）
  ...

脆弱性2 (B)(C): パストラバーサル
  リスク: 致命的
  説明: ファイルパスのバリデーションがなく、任意のファイルを読み取れる
  ...
```

<summary>ヒント</summary>

コードの各ポイント (A)〜(G) に注目してください。主なセキュリティリスクのカテゴリは以下の通りです。
- 機密情報の管理（ハードコード vs 環境変数）
- パストラバーサル（ディレクトリトラバーサル攻撃）
- SQLインジェクション
- コマンドインジェクション（OSコマンドの任意実行）
- 入力バリデーションの不足

</details>

```
=== セキュリティ脆弱性レビュー結果 ===

脆弱性1 (A): APIキーのハードコード
  リスク: 高
  説明: ソースコード内にAPIキーが平文で記述されています。
        ソースコードがバージョン管理されている場合、履歴からも
        キーが漏洩するリスクがあります。
  修正案:
    // 修正前
    const API_KEY = "sk-abc123secret456key789";

    // 修正後: 環境変数から読み取る
    const API_KEY = process.env.API_KEY;
    if (!API_KEY) {
      throw new Error("API_KEY environment variable is required");
    }

    // さらに .gitignore に .env を追加すること
    // 既にコミットされたキーは即座にローテーション（再発行）すること

----------------------------------------------

脆弱性2 (B)(C): パストラバーサル攻撃
  リスク: 致命的
  説明: path パラメータに "../../../etc/passwd" のような値を渡すと、
        サーバー上の任意のファイルを読み取ることができます。
        機密ファイル（設定ファイル、秘密鍵など）が漏洩する恐れがあります。
  修正案:
    import path from "path";

    // 許可するベースディレクトリを定義
    const ALLOWED_BASE_DIR = "/app/documents";

    server.tool(
      "read_file",
      {
        path: z.string()
          .min(1)
          .max(500)
          .refine((p) => !p.includes(".."), {
            message: "パスに '..' は使用できません",
          }),
      },
      async ({ path: filePath }) => {
        const fs = await import("fs/promises");

        // パスを正規化してベースディレクトリ内か確認
        const resolvedPath = path.resolve(ALLOWED_BASE_DIR, filePath);
        if (!resolvedPath.startsWith(ALLOWED_BASE_DIR)) {
          return {
            content: [{ type: "text", text: "アクセスが拒否されました" }],
            isError: true,
          };
        }

        const content = await fs.readFile(resolvedPath, "utf-8");
        return {
          content: [{ type: "text", text: content }],
        };
      }
    );

----------------------------------------------

脆弱性3 (D)(E): SQLインジェクション
  リスク: 致命的
  説明: table と filter がそのままSQL文に組み込まれており、
        攻撃者が任意のSQLを実行できます。
        例: filter = "1=1; DROP TABLE users; --"
        これによりデータの漏洩・改ざん・削除が発生します。
  修正案:
    // テーブル名はホワイトリストで制限
    const ALLOWED_TABLES = ["products", "categories", "reviews"];

    server.tool(
      "run_query",
      {
        table: z.enum(["products", "categories", "reviews"]),
        column: z.string().regex(/^[a-zA-Z_]+$/),  // カラム名を制限
        value: z.string().max(200),                  // 値を分離
        operator: z.enum(["=", "LIKE", ">", "<"]).default("="),
      },
      async ({ table, column, value, operator }) => {
        // プリペアドステートメントを使用
        const query = `SELECT * FROM ${table} WHERE ${column} ${operator} ?`;
        const result = await db.execute(query, [value]);
        return {
          content: [{ type: "text", text: JSON.stringify(result) }],
        };
      }
    );

    // ポイント:
    // - テーブル名は z.enum() で許可リストに限定
    // - カラム名は正規表現で英字とアンダースコアのみに制限
    // - 値はプリペアドステートメントのプレースホルダ(?)で渡す
    // - 演算子もenumで限定する

----------------------------------------------

脆弱性4 (F)(G): コマンドインジェクション（OSコマンド実行）
  リスク: 致命的（最高レベル）
  説明: 任意のOSコマンドが実行可能です。
        例: command = "rm -rf /" でサーバー全体が破壊されます。
        例: command = "cat /etc/shadow" でパスワードハッシュが漏洩します。
        このツールの存在自体が重大なリスクです。
  修正案:
    // 方針1（推奨）: このツール自体を削除する
    // 任意のコマンド実行ツールはMCPサーバーに含めるべきではありません

    // 方針2: やむを得ない場合、許可コマンドをホワイトリスト化
    const ALLOWED_COMMANDS = {
      "disk_usage": ["df", "-h"],
      "memory_info": ["free", "-m"],
      "process_list": ["ps", "aux"],
    } as const;

    server.tool(
      "system_info",
      {
        command_name: z.enum(["disk_usage", "memory_info", "process_list"])
          .describe("実行するシステム情報コマンド"),
      },
      async ({ command_name }) => {
        const { execFile } = await import("child_process");
        const [cmd, ...args] = ALLOWED_COMMANDS[command_name];

        // execFile は exec と異なりシェルを経由しないため安全
        return new Promise((resolve) => {
          execFile(cmd, args, { timeout: 5000 }, (error, stdout) => {
            resolve({
              content: [{
                type: "text",
                text: stdout || error?.message || "",
              }],
            });
          });
        });
      }
    );

    // ポイント:
    // - exec ではなく execFile を使用（シェル解釈を回避）
    // - コマンドはホワイトリスト方式で限定
    // - タイムアウトを設定して無限実行を防止
    // - ユーザー入力をコマンド引数に直接渡さない

----------------------------------------------

脆弱性5: エラー情報の漏洩
  リスク: 中
  説明: エラー発生時にスタックトレースや内部情報がそのまま返される
        可能性があります。攻撃者にシステムの内部構造を推測させる材料になります。
  修正案:
    // エラーハンドリングのラッパー
    function safeError(message: string): ToolResult {
      // 内部エラーの詳細はログに記録し、ユーザーには汎用メッセージを返す
      return {
        content: [{ type: "text", text: JSON.stringify({ error: message }) }],
        isError: true,
      };
    }

    // ツール内での使い方
    try {
      // 処理
    } catch (error) {
      console.error("Internal error:", error);  // 詳細はサーバーログのみ
      return safeError("処理中にエラーが発生しました");  // ユーザーには汎用メッセージ
    }

# まとめ:
# - APIキーは環境変数で管理し、コードにハードコードしない
# - ファイルパスは正規化してベースディレクトリ内に限定する
# - SQL文はプリペアドステートメントを必ず使う
# - OSコマンド実行は原則として提供しない。必要な場合はホワイトリスト方式
# - エラーの詳細情報はログに記録し、外部には汎用メッセージを返す
```

### 問題2：入力バリデーションの実装（基本）

以下のMCPツールに対して、適切な入力バリデーションをZodスキーマで実装してください。各フィールドにはセキュリティを考慮した制約を付けてください。

**要件：**
- ユーザープロフィール更新ツール
- 更新可能フィールド: 表示名、メールアドレス、自己紹介文、Webサイト URL、年齢
- XSS（クロスサイトスクリプティング）対策を含めること
- 適切なエラーメッセージを日本語で提供すること

**期待される出力例：**
```typescript
server.tool(
  "update_profile",
  {
    display_name: z.string()
      .min(1, "表示名は1文字以上で入力してください")
      .max(50, "表示名は50文字以内で入力してください")
      .regex(/^[^<>&"']*$/, "使用できない文字が含まれています"),
    // ... 他のフィールド
  },
  async (params) => { /* 処理 */ }
);
```

<details>
<summary>ヒント</summary>

Zodバリデーションで考慮すべきポイントは以下の通りです。
- 文字列長の上限・下限（DoS対策にもなります）
- 正規表現による形式チェック（メール、URL）
- HTMLタグやスクリプトタグの排除（XSS対策）
- URLのプロトコル制限（http/httpsのみ許可、javascript:スキーム排除）
- 数値の範囲制限
- `.optional()` で任意フィールドを明示
- `.describe()` でLLMにフィールドの意味を伝達

</details>

<details>
<summary>解答例</summary>

```typescript
import { z } from "zod";

// --- XSS対策用のカスタムバリデーション ---

// HTMLタグを検出する正規表現
const noHtmlTags = (value: string) => !/<[^>]*>/g.test(value);

// 危険な文字列パターンを検出
const noScriptPatterns = (value: string) => {
  const dangerous = [
    /javascript:/i,
    /on\w+\s*=/i,       // onload=, onclick= など
    /data:\s*text\/html/i,
    /vbscript:/i,
  ];
  return !dangerous.some((pattern) => pattern.test(value));
};

// HTMLエンティティへのサニタイズ関数
function sanitizeHtml(input: string): string {
  return input
    .replace(/&/g, "&amp;")
    .replace(/</g, "&lt;")
    .replace(/>/g, "&gt;")
    .replace(/"/g, "&quot;")
    .replace(/'/g, "&#x27;");
}

// --- ツール定義 ---

server.tool(
  "update_profile",
  "ユーザープロフィールを更新します。変更したいフィールドのみ指定してください",
  {
    // 表示名: 1〜50文字、HTMLタグ禁止
    display_name: z.string()
      .min(1, "表示名は1文字以上で入力してください")
      .max(50, "表示名は50文字以内で入力してください")
      .refine(noHtmlTags, "HTMLタグは使用できません")
      .refine(noScriptPatterns, "使用できない文字パターンが含まれています")
      .optional()
      .describe("ユーザーの表示名"),

    // メールアドレス: 形式チェック + 長さ制限
    email: z.string()
      .email("有効なメールアドレスを入力してください")
      .max(254, "メールアドレスが長すぎます")  // RFC 5321の上限
      .toLowerCase()  // 正規化: 小文字に統一
      .optional()
      .describe("メールアドレス"),

    // 自己紹介文: 最大500文字、HTMLタグ禁止
    bio: z.string()
      .max(500, "自己紹介文は500文字以内で入力してください")
      .refine(noHtmlTags, "HTMLタグは使用できません")
      .refine(noScriptPatterns, "使用できない文字パターンが含まれています")
      .transform(sanitizeHtml)  // 保存前にサニタイズ
      .optional()
      .describe("自己紹介文（500文字以内）"),

    // WebサイトURL: http/httpsのみ許可
    website: z.string()
      .url("有効なURLを入力してください")
      .max(2000, "URLが長すぎます")
      .refine(
        (url) => {
          try {
            const parsed = new URL(url);
            // http と https のみ許可（javascript: や data: を排除）
            return ["http:", "https:"].includes(parsed.protocol);
          } catch {
            return false;
          }
        },
        "URLはhttp://またはhttps://で始まる必要があります"
      )
      .optional()
      .describe("個人サイトやブログのURL"),

    // 年齢: 0〜150の整数
    age: z.number()
      .int("年齢は整数で入力してください")
      .min(0, "年齢は0以上で入力してください")
      .max(150, "年齢の値が不正です")
      .optional()
      .describe("年齢"),
  },
  async ({ display_name, email, bio, website, age }) => {
    // 少なくとも1つのフィールドが指定されていることを確認
    if (!display_name && !email && !bio && !website && age === undefined) {
      return {
        content: [{
          type: "text" as const,
          text: JSON.stringify({
            error: "更新するフィールドを1つ以上指定してください",
          }),
        }],
        isError: true,
      };
    }

    // 更新処理（実際にはDB更新）
    const updates: Record<string, unknown> = {};
    if (display_name) updates.display_name = display_name;
    if (email) updates.email = email;
    if (bio) updates.bio = bio;
    if (website) updates.website = website;
    if (age !== undefined) updates.age = age;

    // await db.updateProfile(userId, updates);

    return {
      content: [{
        type: "text" as const,
        text: JSON.stringify({
          status: "updated",
          updated_fields: Object.keys(updates),
        }),
      }],
    };
  }
);

// バリデーションのポイント:
// 1. 全フィールドにmin/maxの長さ制限をつける（DoS対策）
// 2. HTMLタグとスクリプトパターンを検出・排除する（XSS対策）
// 3. URLはプロトコルをホワイトリストで制限する
// 4. メールアドレスはRFC準拠の長さ制限を設ける
// 5. .transform() で保存前にサニタイズを行う（多層防御）
// 6. .describe() でLLMにフィールドの意味を伝える
// 7. 日本語のエラーメッセージでユーザビリティを向上させる
```

### 問題3：既存MCPサーバーコードのセキュリティ監査（応用）

以下の「メモ管理MCPサーバー」のコード全体をセキュリティ監査してください。脆弱性の一覧、リスク評価（致命的/高/中/低）、修正後のコードを提示してください。

```typescript
import { McpServer } from "@modelcontextprotocol/sdk/server/mcp.js";
import { StdioServerTransport } from "@modelcontextprotocol/sdk/server/stdio.js";
import { z } from "zod";
import fs from "fs/promises";
import path from "path";

const server = new McpServer({ name: "memo-server", version: "1.0.0" });
const DATA_DIR = process.env.DATA_DIR || "./data";

// メモを保存
server.tool("save_memo", {
  filename: z.string(),
  content: z.string(),
}, async ({ filename, content }) => {
  const filePath = path.join(DATA_DIR, filename);
  await fs.writeFile(filePath, content);
  return { content: [{ type: "text", text: `Saved: ${filePath}` }] };
});

// メモを読み込み
server.tool("read_memo", {
  filename: z.string(),
}, async ({ filename }) => {
  const filePath = path.join(DATA_DIR, filename);
  const content = await fs.readFile(filePath, "utf-8");
  return { content: [{ type: "text", text: content }] };
});

// メモを削除
server.tool("delete_memo", {
  filename: z.string(),
}, async ({ filename }) => {
  const filePath = path.join(DATA_DIR, filename);
  await fs.unlink(filePath);
  return { content: [{ type: "text", text: `Deleted: ${filePath}` }] };
});

// メモ一覧
server.tool("list_memos", {}, async () => {
  const files = await fs.readdir(DATA_DIR);
  return { content: [{ type: "text", text: JSON.stringify(files) }] };
});

// メモを検索
server.tool("search_memos", {
  keyword: z.string(),
}, async ({ keyword }) => {
  const files = await fs.readdir(DATA_DIR);
  const results = [];
  for (const file of files) {
    const content = await fs.readFile(path.join(DATA_DIR, file), "utf-8");
    if (content.includes(keyword)) {
      results.push({ file, snippet: content.substring(0, 200) });
    }
  }
  return { content: [{ type: "text", text: JSON.stringify(results) }] };
});

async function main() {
  const transport = new StdioServerTransport();
  await server.connect(transport);
}

main();
```

<details>
<summary>ヒント</summary>

セキュリティ監査のチェックリストを以下に示します。

1. **入力バリデーション**: ファイル名の形式・長さ・禁止文字
2. **パストラバーサル**: `../` によるディレクトリ脱出
3. **リソース消費**: 大きなファイルの読み書き、大量ファイルの全走査
4. **エラーハンドリング**: 例外時の情報漏洩
5. **ファイルタイプ制限**: 任意の拡張子で保存可能な点
6. **並行性**: 同時書き込み時のレースコンディション
7. **ログ・監査**: 操作ログが存在しない

</details>

<details>
<summary>解答例</summary>

```
=== セキュリティ監査レポート ===

監査対象: memo-server v1.0.0
監査日: 2026-03-26
総合リスク評価: 致命的（即時対応が必要）

【脆弱性一覧】

V01 | リスク: 致命的 | パストラバーサル
  対象: save_memo, read_memo, delete_memo
  説明:
    filename に "../../../etc/passwd" のようなパスを指定することで、
    DATA_DIR外の任意のファイルを読み書き・削除できます。
    path.join() はパスを結合するだけで、トラバーサルを防止しません。
    例: path.join("./data", "../../../etc/passwd")
        → "./etc/passwd" ではなく "../../../etc/passwd" に解決される

V02 | リスク: 高 | ファイル名バリデーション不足
  対象: save_memo
  説明:
    - ファイル名に制限がなく、任意の拡張子(.sh, .exe等)で保存可能
    - ファイル名の長さ制限がなく、非常に長い名前でファイルシステム障害の可能性
    - 特殊文字（ヌルバイト、制御文字）を含むファイル名を拒否していない

V03 | リスク: 高 | コンテンツサイズ無制限
  対象: save_memo
  説明:
    content の長さ制限がなく、数GBのデータを書き込むことが可能です。
    ディスクを枯渇させるDoS攻撃に使われるリスクがあります。

V04 | リスク: 中 | search_memos の全ファイル走査
  対象: search_memos
  説明:
    全ファイルを読み込んで検索するため、ファイル数やサイズが増えると
    メモリ不足やCPU負荷の問題が発生します。
    大量の検索リクエストによるDoS攻撃のリスクがあります。

V05 | リスク: 中 | エラーハンドリングの欠如
  対象: 全ツール
  説明:
    try-catchがなく、ファイルが存在しない場合や権限エラー時に
    スタックトレースが含まれるエラーがそのまま返されます。
    これにより内部のディレクトリ構造やパスが外部に漏洩します。

V06 | リスク: 中 | list_memos の情報漏洩
  対象: list_memos
```

```typescript
// --- ログ・監査ミドルウェア ---

// ログレベルの定義
const LOG_LEVELS = {
  DEBUG: 0,
  INFO: 1,
  WARN: 2,
  ERROR: 3,
} as const;

type LogLevel = keyof typeof LOG_LEVELS;

// 現在のログレベル（環境変数で制御）
const CURRENT_LOG_LEVEL: LogLevel =
  (process.env.LOG_LEVEL as LogLevel) || "INFO";

// マスク対象のフィールド名パターン
const SENSITIVE_FIELDS = [
  "password",
  "token",
  "secret",
  "api_key",
  "apikey",
  "authorization",
  "credential",
  "private_key",
];

// --- ログ出力関数 ---
function log(level: LogLevel, data: Record<string, unknown>): void {
  // ログレベルのフィルタリング
  if (LOG_LEVELS[level] < LOG_LEVELS[CURRENT_LOG_LEVEL]) {
    return;
  }

  const entry = {
    level,
    timestamp: new Date().toISOString(),
    ...data,
  };

  // 構造化ログをstderrに出力（stdoutはMCP通信に使用されるため）
  console.error(JSON.stringify(entry));
}

// --- 機密情報のマスク処理（再帰対応） ---
function maskSensitiveData(data: unknown, depth: number = 0): unknown {
  // 無限再帰の防止
  if (depth > 10) return "[MAX_DEPTH]";

  // null / undefined / プリミティブ型はそのまま返す
  if (data === null || data === undefined) return data;
  if (typeof data !== "object") return data;

  // 配列の場合
  if (Array.isArray(data)) {
    return data.map((item) => maskSensitiveData(item, depth + 1));
  }

  // オブジェクトの場合
  const masked: Record<string, unknown> = {};
  for (const [key, value] of Object.entries(data as Record<string, unknown>)) {
    // フィールド名が機密パターンに一致するかチェック
    const isSensitive = SENSITIVE_FIELDS.some((field) =>
      key.toLowerCase().includes(field)
    );

    if (isSensitive) {
      masked[key] = "***MASKED***";
    } else if (typeof value === "object" && value !== null) {
      masked[key] = maskSensitiveData(value, depth + 1);
    } else {
      masked[key] = value;
    }
  }

  return masked;
}

// --- 監査付きツール登録ラッパー ---
// 既存の server.tool をラップして、自動的にログを記録する

type ToolHandler = (params: Record<string, unknown>) => Promise<{
  content: Array<{ type: string; text: string }>;
  isError?: boolean;
}>;

function createAuditedTool(
  server: McpServer,
  name: string,
  description: string,
  schema: Record<string, z.ZodType>,
  handler: ToolHandler
): void {
  server.tool(name, description, schema, async (params: any) => {
    const startTime = Date.now();
    const maskedParams = maskSensitiveData(params);

    // 開始ログ
    log("DEBUG", {
      event: "tool_call_start",
      tool: name,
      params: maskedParams,
    });

    try {
      // 実際のハンドラを実行
      const result = await handler(params);
      const duration = Date.now() - startTime;

      // 完了ログ
      log("INFO", {
        event: "tool_call_complete",
        tool: name,
        params: maskedParams,
        duration_ms: duration,
        success: !result.isError,
      });

      // 処理が遅い場合は警告
      if (duration > 5000) {
        log("WARN", {
          event: "slow_tool_call",
          tool: name,
          duration_ms: duration,
          message: `ツール呼び出しが${duration}msかかりました`,
        });
      }

      return result;
    } catch (error) {
      const duration = Date.now() - startTime;

      // エラーログ
      log("ERROR", {
        event: "tool_call_error",
        tool: name,
        params: maskedParams,
        duration_ms: duration,
        success: false,
        error: error instanceof Error ? error.message : String(error),
        // スタックトレースはDEBUGレベルのみ
        ...(CURRENT_LOG_LEVEL === "DEBUG" && error instanceof Error
          ? { stack: error.stack }
          : {}),
      });

      return {
        content: [{
          type: "text" as const,
          text: JSON.stringify({
            error: "内部エラーが発生しました",
          }),
        }],
        isError: true,
      };
    }
  });
}

  "create_user",

    email: z.string().email(),

  },
  async ({ name, email, password }) => {
    // ユーザー作成処理（passwordはログにマスクされて記録される）
    return {
      content: [{
        type: "text" as const,
        text: JSON.stringify({ status: "created", name, email }),
      }],
    };
  }
);
```

```
================================================================

■ 2.1 ツール命名規則

  [必須] ツール名は snake_case で命名すること
         - 例: create_document, search_users, update_config
         - 禁止: createDocument, CreateDocument, create-document

  [必須] ツール名は「動詞_名詞」の形式にすること
         - 動詞: create, get, list, update, delete, search, export, import
         - 名詞: 操作対象のリソース名（単数形 or 複数形を統一）

  [必須] ツールのdescriptionを日本語で記述すること
         - LLMが適切に選択できるよう、具体的な説明を書く
         - 悪い例: 「ユーザーを管理する」
         - 良い例: 「指定した条件に合うユーザーを検索し、一覧を返します」

■ 2.2 粒度基準

  [必須] 1ツール = 1アクションとすること
         - CRUD操作は個別のツールに分割する
         - action パラメータで動作を切り替えるパターンは禁止

  [必須] MCPサーバーあたりのツール数は20個以下とすること
         - 20個を超える場合はドメインごとにサーバーを分割する
         - 例: user-server, document-server, notification-server

  [推奨] 関連するフィールドの更新はオプショナルパラメータでまとめること
         - フィールドごとの個別ツールは作成しない

■ 2.3 スキーマ設計

  [必須] 全パラメータに .describe() で説明を付けること
  [必須] オプショナルパラメータにはデフォルト値を設定すること
  [推奨] レスポンスは一貫した形式にすること
         成功時: { status: "...", data: {...} }
         失敗時: { error: "メッセージ" }（isError: true）

================================================================
【3. エラーハンドリング標準】
================================================================

■ 3.1 エラー分類

  [必須] エラーを以下の4カテゴリに分類して処理すること

  カテゴリ1: バリデーションエラー（入力不正）
    - Zodスキーマが自動で検出
    - isError: true で返す
    - ユーザーに修正方法を提示する

  カテゴリ2: ビジネスロジックエラー（権限不足、リソース未発見など）
    - isError: true で返す
    - 具体的なエラーメッセージを返す
    - 例: 「指定されたドキュメントが見つかりません」

  カテゴリ3: 外部サービスエラー（API障害、タイムアウトなど）
    - リトライポリシーに従って再試行
    - 最終的に失敗した場合は isError: true で返す
    - 例: 「外部サービスに接続できません。しばらく後に再試行してください」

  カテゴリ4: 内部エラー（予期しない例外）
    - 詳細はログに記録し、ユーザーには汎用メッセージを返す
    - スタックトレースは本番環境では非表示

■ 3.2 リトライポリシー

  [必須] 外部API呼び出しには以下のリトライポリシーを適用すること
    - 最大リトライ回数: 3回
    - バックオフ方式: 指数バックオフ（1秒、2秒、4秒）
    - リトライ対象: 5xxエラーおよびタイムアウトのみ
    - 4xxエラー（クライアントエラー）はリトライしない

  [必須] タイムアウトを必ず設定すること
    - 外部API呼び出し: 10秒
    - DB操作: 5秒
    - ファイルI/O: 30秒

================================================================
【4. ログ・監視基準】
================================================================

■ 4.1 ログフォーマット

  [必須] 構造化ログ（JSON形式）をstderrに出力すること
         ※ stdoutはMCPプロトコル通信に使用されるため禁止

  [必須] 以下のフィールドを全ログに含めること
    {
      "level": "INFO",              // DEBUG/INFO/WARN/ERROR
      "timestamp": "ISO8601形式",    // 例: 2026-03-26T10:00:00.000Z
      "service": "サーバー名",       // 例: document-server
      "event": "イベント名",         // 例: tool_call_complete
      "tool": "ツール名",           // 例: search_documents
      "duration_ms": 45,            // 実行時間（ミリ秒）
      "success": true               // 成功/失敗
    }

  [必須] ログレベルの使い分け
    - DEBUG: パラメータの詳細、SQLクエリ（開発環境のみ）
    - INFO: ツール呼び出しの完了、正常な操作
    - WARN: 遅いレスポンス（>5秒）、レート制限接近、非推奨機能の使用
    - ERROR: ツール実行の失敗、外部サービスエラー

■ 4.2 メトリクス（監視項目）

  [必須] 以下のメトリクスを収集すること
    - ツール呼び出し回数（ツール別・成否別）
    - レスポンス時間（p50, p95, p99）
    - エラー率（全体・ツール別）
    - アクティブ接続数

■ 4.3 アラートルール

  [必須] 以下の条件でアラートを発報すること
    - 致命的: エラー率 > 10% が5分間継続
    - 警告: レスポンス時間 p95 > 3秒
    - 警告: エラー率 > 5% が10分間継続
    - 情報: 新しいバージョンのデプロイ完了

================================================================
【5. テスト要件】
================================================================

■ 5.1 カバレッジ基準

  [必須] 単体テストのコードカバレッジ: 80%以上
  [必須] 全ツールのハンドラ関数に対するテストを作成すること
  [推奨] ブランチカバレッジ: 70%以上

■ 5.2 テスト種類

  [必須] 単体テスト
    - 各ツールのハンドラ関数のテスト
    - バリデーションの正常系・異常系テスト
    - ビジネスロジックのテスト
    - テストフレームワーク: vitest または jest

  [必須] 統合テスト
    - MCP Client を使用したツール呼び出しテスト
    - 一連のワークフロー（作成→更新→検索→削除）のテスト
    - エラーハンドリングの検証

  [推奨] E2Eテスト
    - 実際のLLMクライアントからの利用シナリオ
    - MCP Inspector を使用した手動テスト

  [推奨] セキュリティテスト
    - パストラバーサル攻撃の試行
    - SQLインジェクションの試行
    - 大量リクエストによる負荷テスト

■ 5.3 CI統合

  [必須] プルリクエスト時に以下を自動実行すること
    1. 静的解析（ESLint + TypeScript型チェック）
    2. 単体テスト + カバレッジレポート
    3. 統合テスト

  [必須] カバレッジが基準を下回った場合、マージをブロックすること

【6. デプロイ・運用ルール】

■ 6.1 環境分離

  [必須] 開発・ステージング・本番の3環境を用意すること
    - 開発環境: stdio接続、ローカルDB
    - ステージング: SSE/HTTP接続、本番同等のインフラ
    - 本番環境: HTTP接続（Streamable HTTP推奨）

  [必須] 環境固有の設定は環境変数で管理すること
         コードに環境名のハードコードは禁止

  [必須] 本番データをステージング以下の環境にコピーしないこと

■ 6.2 リリースフロー

  [必須] 以下のフローでリリースすること
    1. 開発ブランチでの実装・テスト
    2. プルリクエスト → コードレビュー（1名以上の承認）
    3. CI全テスト通過
    4. ステージング環境へデプロイ → 動作確認
    5. 本番環境へデプロイ（カナリアリリース推奨）
    6. デプロイ後の動作確認（ヘルスチェック + スモークテスト）

  [必須] 本番デプロイは平日10:00〜16:00に限定すること
         ※ 緊急修正は除く

  [必須] ロールバック手順を事前に準備・テストすること

■ 6.3 インシデント対応

  [必須] インシデント発生時の対応フローを以下とすること
    1. 検知: アラート受信 or ユーザー報告
    2. トリアージ: 影響範囲の特定（5分以内）
       - Sev1（致命的）: 全サービス停止
       - Sev2（重大）: 主要機能の障害
       - Sev3（軽微）: 一部機能の不具合
    3. 対応: 一次対応の実施
       - Sev1: 即座にロールバック、全チームに通知
       - Sev2: 30分以内に修正またはロールバック
       - Sev3: 次回リリースで修正
    4. 振り返り: インシデントレポートの作成（3営業日以内）
       - 原因分析（Why×5）
       - 再発防止策の策定
       - ガイドラインへの反映
```

## まとめ

この章の演習では、MCPサーバーの運用とセキュリティに関する以下のスキルを確認しました。

- **基本問題**: 脆弱なコードからセキュリティ問題を発見する力、Zodを使った入力バリデーションの実装
- **応用問題**: 実際のコードベースに対するセキュリティ監査の実施、ログ・監査機能のミドルウェア設計
- **チャレンジ問題**: エンタープライズ環境で通用する運用ガイドラインの策定

MCPサーバーはLLMとシステムを接続するゲートウェイです。セキュリティの不備はLLMを介して増幅されるリスクがあるため、多層防御（Defense in Depth）の考え方が特に重要です。

## 問題1：基本的なリバースプロキシ設定（基本）

ポート3000で動作するNode.jsアプリケーションに対して、`app.example.com` というドメインでアクセスできるリバースプロキシ設定を作成してください。

**要件：**
- ポート80で待ち受け
- すべてのリクエストを `127.0.0.1:3000` に転送
- 基本的なプロキシヘッダー（Host, X-Real-IP, X-Forwarded-For）を設定

**期待される動作：**
```
http://app.example.com/ → http://127.0.0.1:3000/
http://app.example.com/users → http://127.0.0.1:3000/users
```

<details>
<summary>ヒント</summary>

`proxy_pass` で転送先を指定し、`proxy_set_header` で各ヘッダーを設定します。変数 `$host`, `$remote_addr`, `$proxy_add_x_forwarded_for` を使います。

</details>

<details>
<summary>解答例</summary>

```nginx
server {
    listen 80;
    server_name app.example.com;

    location / {
        proxy_pass http://127.0.0.1:3000;

        # プロキシヘッダーの設定
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $scheme;
    }
}
```

</details>

## 問題2：proxy_pass の末尾スラッシュ（基本）

以下の2つの設定で、リクエスト `/app/hello` がバックエンドにどのようなパスで届くか答えてください。

**設定A:**
```nginx
location /app/ {
    proxy_pass http://127.0.0.1:3000;
}
```

**設定B:**
```nginx
location /app/ {
    proxy_pass http://127.0.0.1:3000/;
}
```

<details>
<summary>ヒント</summary>

`proxy_pass` にURIパス（末尾スラッシュを含む）が指定されている場合、location で指定したパスが置き換えられます。URIパスがない場合は、元のパスがそのまま渡されます。

</details>

<details>
<summary>解答例</summary>

**設定A（末尾スラッシュなし）:**
```
リクエスト: /app/hello
バックエンド: /app/hello
→ パスがそのまま渡される
```

**設定B（末尾スラッシュあり）:**
```
リクエスト: /app/hello
バックエンド: /hello
→ /app/ が取り除かれ、proxy_pass のパス（/）に置き換わる
```

`proxy_pass` にURIコンポーネント（`/` も含む）が含まれている場合、Nginxは location にマッチした部分を proxy_pass のURIに置き換えます。含まれていない場合は、リクエストURIがそのまま転送されます。

</details>

## 問題3：静的ファイルとプロキシの分離（基本）

以下の要件を満たすリバースプロキシ設定を作成してください。

- `/static/` 以下の静的ファイルはNginxが直接配信（`/var/www/app/static/` から）
- `/media/` 以下のファイルもNginxが直接配信（`/var/www/app/media/` から）
- それ以外のリクエストはポート8000のバックエンドに転送

**期待される動作：**
```
/static/css/style.css → Nginxが /var/www/app/static/css/style.css を配信
/media/images/photo.jpg → Nginxが /var/www/app/media/images/photo.jpg を配信
/api/users → http://127.0.0.1:8000/api/users に転送
```

<details>
<summary>ヒント</summary>

static と media には `alias` ディレクティブを使い、それ以外は `proxy_pass` で転送します。静的ファイルには `expires` でキャッシュヘッダーを付けるとパフォーマンスが向上します。

</details>

<details>
<summary>解答例</summary>

```nginx
server {
    listen 80;
    server_name example.com;

    # 静的ファイル（Nginxが直接配信）
    location /static/ {
        alias /var/www/app/static/;
        expires 30d;
        add_header Cache-Control "public, immutable";
    }

    # メディアファイル（Nginxが直接配信）
    location /media/ {
        alias /var/www/app/media/;
        expires 7d;
    }

    # その他のリクエストはバックエンドに転送
    location / {
        proxy_pass http://127.0.0.1:8000;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $scheme;
    }
}
```

**補足**: 静的ファイルをNginxが直接配信すると、バックエンドサーバーの負荷が大幅に軽減されます。Nginxは静的ファイル配信に非常に優れたパフォーマンスを発揮します。

</details>

## 問題4：プロキシヘッダーの共通化（応用）

3つの異なるバックエンドサービスにプロキシする設定を作成してください。プロキシヘッダーは共通ファイルに分離してください。

| パス | 転送先 | 用途 |
|---|---|---|
| `/` | `127.0.0.1:3000` | フロントエンド |
| `/api/` | `127.0.0.1:8000` | API サーバー |
| `/admin/` | `127.0.0.1:9000` | 管理画面 |

**要件：**
- プロキシヘッダー設定を `/etc/nginx/proxy_params` に分離
- 各 location で `include` で読み込む

<details>
<summary>ヒント</summary>

共通のプロキシヘッダー設定を別ファイルに記述し、各 location ブロック内で `include proxy_params;` で読み込みます。`/api/` と `/admin/` では `proxy_pass` の末尾スラッシュに注意してください。

</details>

<details>
<summary>解答例</summary>

**/etc/nginx/proxy_params:**
```nginx
proxy_set_header Host $host;
proxy_set_header X-Real-IP $remote_addr;
proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
proxy_set_header X-Forwarded-Proto $scheme;
```

**server ブロック:**
```nginx
server {
    listen 80;
    server_name example.com;

    # フロントエンド
    location / {
        proxy_pass http://127.0.0.1:3000;
        include proxy_params;
    }

    # APIサーバー（/api/ のパスをそのまま渡す）
    location /api/ {
        proxy_pass http://127.0.0.1:8000;
        include proxy_params;
    }

    # 管理画面（/admin/ のパスをそのまま渡す）
    location /admin/ {
        proxy_pass http://127.0.0.1:9000;
        include proxy_params;
    }
}
```

**補足**: 共通設定をファイルに分離することで、ヘッダー変更時に1箇所を修正するだけで全てに反映されます。保守性が大幅に向上する実務的なテクニックです。

</details>

## 問題5：WebSocket 対応のプロキシ設定（応用）

チャットアプリケーション用のリバースプロキシ設定を作成してください。

**要件：**
- 通常のHTTPリクエストは `127.0.0.1:3000` に転送
- WebSocket（`/ws/`）も `127.0.0.1:3000` に転送（WebSocket対応設定）
- WebSocketの接続は24時間維持できるようにする

<details>
<summary>ヒント</summary>

WebSocket対応には以下が必要です。
- `proxy_http_version 1.1;`（HTTP/1.1 が必須）
- `proxy_set_header Upgrade $http_upgrade;`
- `proxy_set_header Connection "upgrade";`
- `proxy_read_timeout` を長時間に設定

</details>

<details>
<summary>解答例</summary>

```nginx
upstream chat_app {
    server 127.0.0.1:3000;
}

server {
    listen 80;
    server_name chat.example.com;

    # 通常のHTTPリクエスト
    location / {
        proxy_pass http://chat_app;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $scheme;
    }

    # WebSocket接続
    location /ws/ {
        proxy_pass http://chat_app;

        # HTTP/1.1 が必須（WebSocketはHTTP/1.1のUpgradeを使用）
        proxy_http_version 1.1;

        # WebSocketアップグレード用ヘッダー
        proxy_set_header Upgrade $http_upgrade;
        proxy_set_header Connection "upgrade";

        # 標準のプロキシヘッダー
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;

        # 24時間（86400秒）のタイムアウト設定
        proxy_read_timeout 86400s;
        proxy_send_timeout 86400s;
    }
}
```

</details>

## 問題6：タイムアウトの使い分け（応用）

以下の3つのエンドポイントに対して、それぞれ適切なタイムアウト設定をしてください。

| エンドポイント | 用途 | 特徴 |
|---|---|---|
| `/api/` | 通常のAPI | レスポンスは数秒以内 |
| `/api/upload/` | ファイルアップロード | 大きなファイル（最大50MB）を受け付ける |
| `/api/reports/` | レポート生成 | 処理に最大5分かかる場合がある |

<details>
<summary>ヒント</summary>

- 通常のAPI: 短めのタイムアウト（5〜30秒）
- ファイルアップロード: `proxy_send_timeout` と `client_max_body_size` を大きく設定
- レポート生成: `proxy_read_timeout` を長めに設定
- `proxy_connect_timeout` はどれも短め（5秒程度）で問題ありません

</details>

<details>
<summary>解答例</summary>

```nginx
upstream api_backend {
    server 127.0.0.1:8000;
}

server {
    listen 80;
    server_name api.example.com;

    # 通常のAPI（短めのタイムアウト）
    location /api/ {
        proxy_pass http://api_backend;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;

        proxy_connect_timeout 5s;
        proxy_read_timeout 30s;
        proxy_send_timeout 30s;
    }

    # ファイルアップロード（大きなファイル対応）
    location /api/upload/ {
        proxy_pass http://api_backend;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;

        proxy_connect_timeout 5s;
        proxy_read_timeout 120s;     # 処理待ち
        proxy_send_timeout 300s;     # アップロード時間を確保

        client_max_body_size 50m;    # 最大50MBのアップロードを許可
    }

    # レポート生成（長時間処理対応）
    location /api/reports/ {
        proxy_pass http://api_backend;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;

        proxy_connect_timeout 5s;
        proxy_read_timeout 300s;     # 最大5分の処理待ち
        proxy_send_timeout 60s;
    }
}
```

**補足**: Nginxの location マッチングでは、より長いプレフィックスが優先されます。`/api/upload/` は `/api/` より長いため、アップロードリクエストは正しく `/api/upload/` のブロックにマッチします。

</details>

## 問題7：UNIXソケットを使ったプロキシ（応用）

Gunicornが UNIXソケット `/run/gunicorn/myapp.sock` で動作しているDjangoアプリケーションに対するリバースプロキシ設定を作成してください。

**要件：**
- 静的ファイル（`/static/`）は `/var/www/myapp/staticfiles/` からNginxが直接配信
- メディアファイル（`/media/`）は `/var/www/myapp/media/` からNginxが直接配信
- その他のリクエストはGunicornに転送
- アップロード上限は10MB

<details>
<summary>ヒント</summary>

UNIXソケットへの接続は `server unix:/path/to/socket;` の形式で upstream に記述します。Djangoでは `collectstatic` で静的ファイルを1箇所に集約します。

</details>

<details>
<summary>解答例</summary>

```nginx
upstream django_app {
    # UNIXソケットによる接続（同一マシンではTCPより高速）
    server unix:/run/gunicorn/myapp.sock;
}

server {
    listen 80;
    server_name myapp.example.com;

    access_log /var/log/nginx/myapp.access.log;
    error_log  /var/log/nginx/myapp.error.log;

    client_max_body_size 10m;

    # Django の静的ファイル（collectstatic で収集されたもの）
    location /static/ {
        alias /var/www/myapp/staticfiles/;
        expires 30d;
        add_header Cache-Control "public, immutable";
        access_log off;  # 静的ファイルのログは不要
    }

    # ユーザーがアップロードしたメディアファイル
    location /media/ {
        alias /var/www/myapp/media/;
        expires 7d;
    }

    # その他のリクエストをGunicornに転送
    location / {
        proxy_pass http://django_app;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $scheme;

        proxy_connect_timeout 5s;
        proxy_read_timeout 60s;
        proxy_send_timeout 60s;
    }
}
```

</details>

## 問題8：SPA + API のリバースプロキシ（チャレンジ）

React SPA（Single Page Application）とAPIサーバーを1つのドメインで運用するためのNginx設定を作成してください。

**要件：**
- SPAのビルド済みファイルは `/var/www/frontend/build/` にある
- SPAのルーティング（`/about`, `/dashboard` 等）はすべて `index.html` にフォールバック
- `/api/` 以下のリクエストはポート8000のAPIサーバーに転送
- `/ws/` 以下はWebSocket対応でポート8000に転送
- 静的アセット（CSS, JS, 画像）にはキャッシュヘッダーを設定

<details>
<summary>ヒント</summary>

- SPAルーティングには `try_files $uri $uri/ /index.html;` を使います
- 静的アセットには正規表現 location `~* \.(css|js|png|jpg|jpeg|gif|ico|svg|woff2?)$` でマッチさせます
- location の記述順序に注意してください（prefix match と regex match の優先度）

</details>

<details>
<summary>解答例</summary>

```nginx
upstream api_server {
    server 127.0.0.1:8000;
}

server {
    listen 80;
    server_name example.com;

    root /var/www/frontend/build;
    index index.html;

    access_log /var/log/nginx/example.com.access.log;
    error_log  /var/log/nginx/example.com.error.log;

    # APIリクエストをバックエンドに転送
    location /api/ {
        proxy_pass http://api_server;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $scheme;

        proxy_connect_timeout 5s;
        proxy_read_timeout 60s;
    }

    # WebSocket接続
    location /ws/ {
        proxy_pass http://api_server;
        proxy_http_version 1.1;
        proxy_set_header Upgrade $http_upgrade;
        proxy_set_header Connection "upgrade";
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_read_timeout 86400s;
    }

    # 静的アセットにキャッシュヘッダーを設定
    location ~* \.(css|js|png|jpg|jpeg|gif|ico|svg|woff2?)$ {
        expires 30d;
        add_header Cache-Control "public, immutable";
        access_log off;
        # ファイルが見つからなければ404
        try_files $uri =404;
    }

    # SPAルーティング：すべてのパスを index.html にフォールバック
    location / {
        try_files $uri $uri/ /index.html;
    }
}
```

**補足**: `try_files $uri $uri/ /index.html;` は、React Router や Vue Router などのクライアントサイドルーティングを使うSPAでは必須の設定です。存在しないパスへのリクエストを `index.html` に渡すことで、JavaScriptが適切なページをレンダリングします。

</details>

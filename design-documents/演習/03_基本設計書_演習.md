## 基本問題

### 問題1：システム構成図の読み取り

以下のシステム構成図を読み取り、質問に答えてください。

```
  ユーザー
    │ HTTPS
    ▼
  CloudFront (CDN)
    │
    ├── 静的ファイル → S3
    │
    └── API → ALB
                │
          ┌─────┴─────┐
          │           │
        ECS (App1)  ECS (App2)
          │           │
          └─────┬─────┘
                │
          ┌─────┴─────┐
          │           │
        RDS         ElastiCache
      (PostgreSQL)   (Redis)
```

1. ユーザーからのリクエストを最初に受け取るコンポーネントは何ですか？
2. 静的ファイル（CSS、画像等）はどこに保存されていますか？
3. アプリケーションサーバーは何台ありますか？冗長化されていますか？
4. データベースに使われている技術は何ですか？
5. キャッシュに使われている技術は何ですか？

<details>
<summary>解答例</summary>

```
1. CloudFront（CDN）
   → ユーザーからのHTTPSリクエストを最初に受け取り、
     静的ファイルはCDNから配信し、APIリクエストはALBに転送する。

2. Amazon S3
   → CloudFrontから静的ファイルのリクエストがS3に転送される。

3. 2台（ECS App1, App2）。冗長化されている。
   → ALB（Application Load Balancer）が2台に負荷分散し、
     1台がダウンしてももう1台で処理を継続できる。

4. PostgreSQL（Amazon RDS）
   → マネージドリレーショナルデータベースサービスで運用。

5. Redis（Amazon ElastiCache）
   → インメモリキャッシュとして使用。セッション管理や
     頻繁にアクセスされるデータのキャッシュに活用される。
```

</details>

### 問題2：技術スタックの選定理由

以下のプロジェクトに対して、技術スタック（フロントエンド、バックエンド、データベース）を選定し、その理由を述べてください。

**プロジェクト：** 社内の勤怠管理Webアプリケーション
- 利用者：従業員200名
- 機能：出退勤打刻、月次勤務表表示、有給申請・承認
- 要件：社内ネットワークからのみアクセス、モバイル対応必要

<details>
<summary>ヒント</summary>

技術選定では「チームのスキルセット」「要件との適合性」「保守性」「コミュニティの活発さ」を考慮します。正解は1つではありません。

</details>

<details>
<summary>解答例</summary>

```markdown
## 技術スタック

### フロントエンド
| 技術 | 選定理由 |
|------|---------|
| React + TypeScript | コンポーネントベースで再利用性が高く、型安全で保守しやすい |
| Tailwind CSS | ユーティリティファーストで開発効率が高い |

### バックエンド
| 技術 | 選定理由 |
|------|---------|
| Python + FastAPI | チームにPython経験者が多く、学習コストが低い。OpenAPI自動生成が便利 |

### データベース
| 技術 | 選定理由 |
|------|---------|
| PostgreSQL | 日時計算が豊富。勤怠データの集計に適している |

### インフラ
| 技術 | 選定理由 |
|------|---------|
| Docker + Docker Compose | 社内ネットワーク内のサーバーで運用。コンテナで環境統一 |

※これは一例です。チームのスキルセットや既存環境に応じて
Node.js + Express、Ruby on Rails等も有力な選択肢です。
重要なのは「なぜその技術を選んだか」の理由を明記することです。
```

</details>

## 応用問題

### 問題4：機能設計の作成

以下の機能について、機能設計（入力・出力・処理概要・ビジネスルール）を作成してください。

**機能：** 商品レビューの投稿

**前提条件：**
- ログイン済みユーザーのみ投稿可能
- 評価（1〜5の星）とコメント（テキスト）を投稿
- 1つの商品に対して1ユーザー1レビューまで
- 購入した商品のみレビュー可能

<details>
<summary>ヒント</summary>

入力項目、出力（レスポンス）、処理の手順、ビジネスルール（制約条件）の4つを整理しましょう。エラーケースも忘れずに。

</details>

<details>
<summary>解答例</summary>

```markdown
## F-050 商品レビュー投稿

### 概要
ログイン済みユーザーが購入した商品にレビュー（評価＋コメント）を投稿する。

### 入力
| 項目 | 型 | 必須 | 制約 | 説明 |
|------|-----|------|------|------|
| product_id | integer | ◯ | 存在する商品ID | レビュー対象の商品 |
| rating | integer | ◯ | 1〜5 | 星評価 |
| comment | string | ◯ | 10〜1000文字 | レビュー本文 |

### 出力
| 項目 | 型 | 説明 |
|------|-----|------|
| id | integer | 作成されたレビューのID |
| product_id | integer | 商品ID |
| user_name | string | 投稿者名 |
| rating | integer | 星評価 |
| comment | string | レビュー本文 |
| created_at | datetime | 投稿日時 |

### 処理概要
1. リクエストのバリデーション（rating: 1-5, comment: 10-1000文字）
2. ユーザーの認証状態を確認
3. 対象商品の存在を確認
4. ユーザーが対象商品を購入済みか確認（ordersテーブルを参照）
5. ユーザーが対象商品に既にレビューを投稿していないか確認
6. レビューレコードを作成・保存
7. 商品の平均評価を再計算・更新
8. 作成されたレビュー情報を返却

### ビジネスルール
- 購入済み商品のみレビュー可能（注文ステータスが「配達済み」の商品）
- 1ユーザー1商品につき1レビューまで
- レビュー投稿後の編集は可能、削除は不可
- 不適切な表現のフィルタリングは行わない（フェーズ2で対応）

### エラーケース
| エラー | HTTPステータス | メッセージ |
|--------|--------------|-----------|
| 未ログイン | 401 | ログインが必要です |
| 商品未存在 | 404 | 指定された商品が見つかりません |
| 未購入 | 403 | 購入した商品のみレビューできます |
| 既にレビュー済み | 409 | この商品には既にレビューを投稿しています |
| バリデーションエラー | 400 | 入力内容に誤りがあります |
```

</details>

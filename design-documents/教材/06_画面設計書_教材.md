# 第6章：画面設計書・UI設計

## この章のゴール

- 画面設計書の目的と構成要素を理解する
- ワイヤーフレームの作成方法を習得する
- 画面レイアウト定義の書き方を学ぶ
- 入力バリデーション定義の設計方法を把握する
- UIデザインの基本原則を理解する

## 6.1 画面設計書とは

### 目的

画面設計書は、**ユーザーが操作する画面のレイアウト・要素・動作を定義するドキュメント**です。フロントエンド開発者、デザイナー、顧客の間で画面仕様を共有するために作成します。

### 画面設計書の位置づけ

```
要件定義（ユースケース）
  │  「ユーザーが商品を検索できる」
  ▼
画面設計書
  │  「検索画面のレイアウト・要素・動作」
  ▼
UIデザイン（Figma等）
  │  「実際のビジュアルデザイン」
  ▼
フロントエンド実装
```

### 画面設計書の構成

```markdown
1. 画面一覧
2. 画面遷移図
3. 共通レイアウト
   3.1 ヘッダー
   3.2 フッター
   3.3 サイドバー
4. 各画面の詳細
   4.1 画面レイアウト（ワイヤーフレーム）
   4.2 画面項目定義
   4.3 入力バリデーション
   4.4 画面アクション定義
5. レスポンシブ対応方針
6. アクセシビリティ方針
```

## 6.2 ワイヤーフレーム

### ワイヤーフレームとは

ワイヤーフレームは、**画面の構成要素の配置を大まかに示した設計図**です。色やデザインの詳細は含まず、レイアウトと情報の配置に焦点を当てます。

### ワイヤーフレームの粒度

| レベル | 名称 | 目的 | ツール |
|--------|------|------|--------|
| Lo-Fi | ラフスケッチ | 大まかな構成の検討 | 手書き、ホワイトボード |
| Mid-Fi | ワイヤーフレーム | 要素の配置と構成の確定 | draw.io、Figma |
| Hi-Fi | モックアップ | 実際のビジュアルに近い | Figma、Adobe XD |

### テキストベースのワイヤーフレーム

設計書にはアスキーアートでワイヤーフレームを記述できます。

## 6.3 画面項目定義

### 画面項目定義とは

画面上の各要素（入力欄、ボタン、表示項目等）の仕様を一覧にしたものです。

### ログイン画面の画面項目定義

| No | 項目名 | 種類 | 必須 | 初期値 | 説明 |
|----|--------|------|------|--------|------|
| 1 | メールアドレス | テキスト入力 | ◯ | 空 | ログイン用メールアドレス |
| 2 | パスワード | パスワード入力 | ◯ | 空 | パスワード（マスク表示） |
| 3 | ログイン状態を保持する | チェックボックス | - | OFF | ONの場合、セッションを30日延長 |
| 4 | ログインボタン | ボタン | - | - | 認証処理を実行する |
| 5 | パスワードを忘れた方はこちら | リンク | - | - | パスワードリセット画面へ遷移 |
| 6 | 新規登録 | リンク | - | - | ユーザー登録画面へ遷移 |

### 商品一覧画面の画面項目定義

| No | 項目名 | 種類 | 説明 |
|----|--------|------|------|
| 1 | パンくずリスト | テキスト | 現在の階層を表示 |
| 2 | カテゴリ名 | テキスト | 表示中のカテゴリ名と件数 |
| 3 | 価格帯フィルター | チェックボックス群 | 価格帯で絞り込み |
| 4 | ブランドフィルター | チェックボックス群 | ブランドで絞り込み |
| 5 | フィルタークリア | ボタン | フィルター条件をリセット |
| 6 | フィルター適用 | ボタン | フィルター条件を適用して再検索 |
| 7 | 並び替え | ドロップダウン | 新着順/価格安い順/価格高い順/評価順 |
| 8 | 商品カード | カード要素 | 画像、商品名、価格、評価を表示 |
| 9 | ページネーション | リンク群 | 1ページ20件、ページ番号とNext/Prev |

## 6.4 入力バリデーション定義

### バリデーションの種類

| 種類 | 説明 | 実施場所 |
|------|------|---------|
| **フォーマット** | 形式チェック（メール形式、電話番号等） | フロント＋サーバー |
| **必須** | 入力が空でないことの確認 | フロント＋サーバー |
| **文字数** | 最小・最大文字数の確認 | フロント＋サーバー |
| **範囲** | 数値の範囲チェック | フロント＋サーバー |
| **整合性** | 他の項目との関連チェック | サーバー |
| **一意性** | 重複チェック（メールアドレス等） | サーバー |

> **実務でのポイント**: フロントエンドのバリデーションはUX向上のため（即座にエラーを表示）、サーバーサイドのバリデーションはセキュリティのために必ず実施します。フロントのバリデーションだけでは不十分です。

### ユーザー登録画面のバリデーション定義

| No | 項目名 | ルール | エラーメッセージ |
|----|--------|--------|----------------|
| 1 | 氏名 | 必須 | 氏名を入力してください |
| 1 | 氏名 | 1〜100文字 | 氏名は100文字以内で入力してください |
| 2 | メールアドレス | 必須 | メールアドレスを入力してください |
| 2 | メールアドレス | メール形式 | 正しいメールアドレスの形式で入力してください |
| 2 | メールアドレス | 一意（未登録） | このメールアドレスは既に登録されています |
| 3 | パスワード | 必須 | パスワードを入力してください |
| 3 | パスワード | 8文字以上 | パスワードは8文字以上で入力してください |
| 3 | パスワード | 英数字混在 | パスワードは英字と数字を含めてください |
| 4 | パスワード（確認） | 必須 | パスワード（確認）を入力してください |
| 4 | パスワード（確認） | パスワードと一致 | パスワードが一致しません |

## 6.5 画面アクション定義

### アクション定義とは

画面上のユーザー操作に対するシステムの動作を定義します。

### ログイン画面のアクション定義

```markdown
## アクション: ログインボタン押下

### トリガー
ログインボタンをクリック

### 事前チェック
1. メールアドレスが入力されていること
2. パスワードが入力されていること
→ 未入力の場合、該当項目のエラーメッセージを表示

### 処理
1. ローディング表示を開始する
2. POST /api/auth/login を呼び出す
   - リクエスト: { email, password, remember_me }
3. レスポンスに応じて処理を分岐する

### 正常系（200 OK）
1. JWTトークンをlocalStorageに保存する
2. ダッシュボード画面（/dashboard）に遷移する

### 異常系
| ステータス | 処理 |
|-----------|------|
| 401 Unauthorized | 「メールアドレスまたはパスワードが正しくありません」を表示 |
| 429 Too Many Requests | 「ログイン試行回数が制限を超えました。しばらく待ってからお試しください」を表示 |
| 500 Internal Server Error | 「システムエラーが発生しました。時間をおいて再度お試しください」を表示 |
```

## 6.6 レスポンシブ対応方針

### ブレークポイントの定義

| デバイス | ブレークポイント | レイアウト |
|---------|----------------|-----------|
| モバイル | 〜767px | 1カラム、ハンバーガーメニュー |
| タブレット | 768px〜1023px | 2カラム |
| デスクトップ | 1024px〜 | 3カラム、サイドバー表示 |

### レスポンシブ対応の方針例

```markdown
## 商品一覧画面のレスポンシブ対応

### デスクトップ（1024px〜）
- サイドバーにフィルターを常時表示
- 商品カードは3列グリッド

### タブレット（768px〜1023px）
- フィルターは画面上部に折りたたみ表示
- 商品カードは2列グリッド

### モバイル（〜767px）
- フィルターはモーダルで表示（フィルターボタンで開閉）
- 商品カードは1列リスト表示
- ヘッダーのナビはハンバーガーメニュー
```

## 6.7 UIデザインの基本原則

### ニールセンの10ヒューリスティクス

| No | 原則 | 説明 | 具体例 |
|----|------|------|--------|
| 1 | **システム状態の可視性** | 現在の状態を常にユーザーに知らせる | ローディング表示、進捗バー |
| 2 | **実世界との一致** | ユーザーの言葉や概念を使う | 「カートに入れる」（技術用語を避ける） |
| 3 | **ユーザーの自由と制御** | 間違った操作を簡単に戻せる | 「元に戻す」ボタン、確認ダイアログ |
| 4 | **一貫性と標準** | 同じ操作は同じ見た目・動作にする | ボタンのスタイル統一 |
| 5 | **エラーの防止** | エラーが起きにくい設計にする | 入力フォーマットのプレースホルダー |
| 6 | **記憶より認識** | 情報を見えるようにする | ドロップダウンで選択肢を提示 |
| 7 | **柔軟性と効率** | 初心者にも上級者にも使いやすく | ショートカットキー、オートコンプリート |
| 8 | **美的で最小限のデザイン** | 不要な情報を排除する | 余白の活用、情報の優先順位付け |
| 9 | **エラーの認識・診断・回復** | エラーは分かりやすく、対処法を示す | 「パスワードは8文字以上で入力してください」 |
| 10 | **ヘルプとドキュメント** | 必要な場合にヘルプを提供する | ツールチップ、FAQリンク |

## まとめ

| 概念 | ポイント |
|------|---------|
| 画面設計書 | 画面のレイアウト・要素・動作を定義するドキュメント |
| ワイヤーフレーム | 画面構成を大まかに示す設計図（Lo-Fi→Hi-Fi） |
| 画面項目定義 | 各要素の種類・必須・初期値・説明を一覧化する |
| バリデーション定義 | 入力チェックルールとエラーメッセージを定義する |
| アクション定義 | ユーザー操作に対するシステム動作を定義する |
| レスポンシブ対応 | デバイスごとのレイアウト変更方針を定める |

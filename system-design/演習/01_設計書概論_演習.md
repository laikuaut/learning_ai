# 第1章 演習：設計書概論

## 基本問題

### 問題1：設計書の必要性

以下のシナリオを読んで、設計書がないことで発生しうる問題を3つ以上挙げてください。

**シナリオ：** 5人のチームで社内の勤怠管理システムを開発することになりました。プロジェクトマネージャーは「スピード重視で行こう。設計書は後から書けばいい」と言い、すぐにコーディングを開始しました。

**期待される出力例：**

```
1. 各開発者が異なる解釈で実装し、機能間の整合性が取れない
2. 顧客が求めていた機能と実装された機能にズレが発生する
3. 後から参加したメンバーがシステム全体を把握できない
```

<details>
<summary>ヒント</summary>

設計書がない場合の問題を、「チーム内」「顧客との関係」「将来の保守」の3つの観点で考えてみましょう。

</details>

<details>
<summary>解答例</summary>

```
1. 認識のズレ：
   各開発者が「勤怠入力画面」の仕様を異なる解釈で実装してしまう。
   例えば、あるメンバーは時刻入力を15分単位にし、
   別のメンバーは1分単位にするなど。

2. 手戻りの発生：
   実装後に顧客から「残業申請機能が必要」と言われ、
   データベース構造から作り直す必要が生じる。

3. 保守の困難：
   後から参加したメンバーが「なぜこの設計なのか」を理解できず、
   既存の設計意図に反する変更をしてしまう。

4. テストの不備：
   テスト基準が曖昧なため、何をもって「正しく動作する」とするかが
   担当者によって異なる。

5. 品質のばらつき：
   エラーハンドリングやバリデーションの基準がなく、
   画面ごとに対応レベルが異なる。
```

</details>

### 問題2：V字モデルの対応関係

以下のV字モデルの空欄（A）〜（C）を埋めてください。

```
要件定義 ─────────────────────────── （ A ）テスト
  │                                       │
  ▼                                       ▲
  基本設計 ───────────────────── （ B ）テスト
    │                                   │
    ▼                                   ▲
    詳細設計 ─────────────── （ C ）テスト
      │                             │
      ▼                             ▲
         実装（コーディング）
```

**期待される出力例：**

```
A: 受入
B: 結合
C: 単体
```

<details>
<summary>ヒント</summary>

各設計工程で定義した内容を検証するのが、対応するテスト工程です。要件を検証するテスト、システム全体の連携を検証するテスト、個々のモジュールを検証するテストをそれぞれ考えてください。

</details>

<details>
<summary>解答例</summary>

```
A: 受入（受入テスト / Acceptance Test）
   → 要件定義で定めた要件を満たしているかを、
     顧客の視点で検証する。

B: 結合（結合テスト / Integration Test）
   → 基本設計で定めたモジュール間の連携が
     正しく動作するかを検証する。

C: 単体（単体テスト / Unit Test）
   → 詳細設計で定めた個々のモジュールが
     仕様通りに動作するかを検証する。
```

**補足：** V字モデルの左側（設計工程）で定義した内容を、右側（テスト工程）で検証するという対応関係が重要です。この対応を意識することで、テスト漏れを防ぐことができます。

</details>

### 問題3：曖昧な記述の改善

以下の設計書の記述には曖昧な表現が含まれています。それぞれ具体的な記述に書き直してください。

1. 「ユーザーが適切に認証されること」
2. 「エラー発生時は適切にハンドリングする」
3. 「大量データにも対応できるパフォーマンスとする」
4. 「必要に応じてキャッシュを利用する」

**期待される出力例：**

```
1. 「ユーザーはメールアドレスとパスワードで認証され、
    認証トークン（JWT）の有効期限は24時間とする」

2. 「HTTP 400系エラーの場合はエラーメッセージを画面に表示し、
    HTTP 500系エラーの場合はエラー画面に遷移しログを記録する」
```

<details>
<summary>ヒント</summary>

曖昧な表現を具体化するには、以下の観点で考えましょう。
- 「適切に」→ 何が適切なのか、具体的な条件・基準は？
- 「大量」→ 何件？何MB？
- 「必要に応じて」→ どのような条件で？

</details>

<details>
<summary>解答例</summary>

```
1. 原文：「ユーザーが適切に認証されること」
   改善：「ユーザーはメールアドレス（RFC 5322準拠）と
          パスワード（8文字以上、英数記号混合）で認証する。
          認証方式はJWTトークン方式とし、
          アクセストークンの有効期限は1時間、
          リフレッシュトークンの有効期限は30日とする。
          認証失敗は5回連続でアカウントを30分間ロックする。」

2. 原文：「エラー発生時は適切にハンドリングする」
   改善：「エラーハンドリングは以下の通りとする。
          ・バリデーションエラー（400）：該当項目に赤枠と
            エラーメッセージを表示する。
          ・認証エラー（401）：ログイン画面にリダイレクトする。
          ・権限エラー（403）：権限不足メッセージを表示する。
          ・サーバーエラー（500）：汎用エラー画面を表示し、
            エラーログを記録する。エラーIDを画面に表示する。」

3. 原文：「大量データにも対応できるパフォーマンスとする」
   改善：「以下のパフォーマンス要件を満たすこと。
          ・商品一覧表示：100万件のデータに対して、
            1ページ（20件）の取得が1秒以内。
          ・検索機能：100万件のデータに対して、
            検索結果の表示が3秒以内。
          ・同時接続ユーザー数：500ユーザー。
          ・ピーク時のスループット：100リクエスト/秒。」

4. 原文：「必要に応じてキャッシュを利用する」
   改善：「以下のデータにキャッシュを適用する。
          ・商品カテゴリマスタ：Redis、TTL 1時間。
          ・商品一覧（検索結果）：Redis、TTL 5分。
          ・ユーザーセッション：Redis、TTL 24時間。
          キャッシュの無効化は、対象データの更新時に
          即時実行する。」
```

</details>

## 応用問題

### 問題4：設計書の種類の選定

あなたはECサイトの開発プロジェクトに参画しています。以下の情報をどの設計書に記載すべきか、理由とともに答えてください。

1. 「商品テーブルのカラム定義（商品ID、商品名、価格、在庫数）」
2. 「ユーザーが商品をカートに追加する際のAPI仕様」
3. 「注文処理のループ・分岐を含む詳細な処理フロー」
4. 「ログイン画面のレイアウトとボタン配置」
5. 「システムの月間稼働率99.9%以上という要件」

**期待される出力例：**

```
1. データベース設計書 - テーブル定義はDB設計書に記載する
```

<details>
<summary>ヒント</summary>

各情報の「粒度」と「対象領域」を考えましょう。
- テーブルのカラム定義 → データに関する設計
- API仕様 → 外部インタフェースの設計
- 詳細な処理フロー → モジュール内部の設計
- 画面レイアウト → UIの設計
- 稼働率 → 非機能要件

</details>

<details>
<summary>解答例</summary>

```
1. データベース設計書
   理由：テーブルのカラム名・データ型・制約など、データベースの
   物理的な構造を定義する情報はデータベース設計書に記載する。

2. API設計書
   理由：APIのエンドポイント、HTTPメソッド、リクエスト/レスポンスの
   形式など、外部インタフェースの仕様はAPI設計書に記載する。

3. 詳細設計書
   理由：モジュール内部の処理ロジック（ループ、分岐、例外処理など）
   は詳細設計書に記載する。フローチャートやシーケンス図を用いる。

4. 画面設計書
   理由：画面のレイアウト、UI部品の配置、操作仕様などは
   画面設計書に記載する。ワイヤーフレームやモックアップも含む。

5. 要件定義書（非機能要件）
   理由：稼働率・応答時間・セキュリティなどの非機能要件は、
   要件定義書の非機能要件セクションに記載する。
   基本設計書のインフラ設計にも反映される。
```

</details>

### 問題5：番号体系の設計

以下の要件から、設計書間のトレーサビリティを確保する番号体系を設計してください。

**要件：**
- ユーザー登録機能（REQ-001）
- ユーザーログイン機能（REQ-002）
- 商品検索機能（REQ-003）

各要件について、基本設計・詳細設計・テストケースの番号を付与してください。

**期待される出力例：**

```
REQ-001 ユーザー登録機能
  ├─ BD-001-01 ユーザー登録画面設計
  ├─ BD-001-02 ユーザー登録API設計
  ...
```

<details>
<summary>ヒント</summary>

番号体系のポイントは以下の3つです。
1. 上位の設計書の番号を含める（トレーサビリティ）
2. 1つの要件から複数の設計項目が派生しうる
3. プレフィックスで設計書の種類がわかるようにする

</details>

<details>
<summary>解答例</summary>

```
REQ-001 ユーザー登録機能
  ├─ BD-001-01 ユーザー登録画面設計
  ├─ BD-001-02 ユーザー登録API設計
  ├─ BD-001-03 ユーザーテーブル設計
  │   ├─ DD-001-01 入力バリデーション処理
  │   ├─ DD-001-02 パスワードハッシュ化処理
  │   ├─ DD-001-03 メール送信処理
  │   │   ├─ TC-001-01 正常系：必須項目を全て入力して登録
  │   │   ├─ TC-001-02 異常系：メールアドレス重複時のエラー
  │   │   ├─ TC-001-03 異常系：必須項目未入力時のエラー
  │   │   └─ TC-001-04 異常系：パスワード要件未達時のエラー

REQ-002 ユーザーログイン機能
  ├─ BD-002-01 ログイン画面設計
  ├─ BD-002-02 認証API設計
  │   ├─ DD-002-01 認証処理（JWT生成）
  │   ├─ DD-002-02 アカウントロック処理
  │   │   ├─ TC-002-01 正常系：正しい認証情報でログイン
  │   │   ├─ TC-002-02 異常系：パスワード誤り
  │   │   ├─ TC-002-03 異常系：5回連続失敗でアカウントロック
  │   │   └─ TC-002-04 異常系：存在しないユーザーでログイン

REQ-003 商品検索機能
  ├─ BD-003-01 商品検索画面設計
  ├─ BD-003-02 商品検索API設計
  ├─ BD-003-03 商品テーブル・インデックス設計
  │   ├─ DD-003-01 検索クエリ組み立て処理
  │   ├─ DD-003-02 ページネーション処理
  │   ├─ DD-003-03 検索結果キャッシュ処理
  │   │   ├─ TC-003-01 正常系：商品名で検索し結果が返る
  │   │   ├─ TC-003-02 正常系：複数条件での絞り込み検索
  │   │   ├─ TC-003-03 正常系：検索結果0件の表示
  │   │   └─ TC-003-04 非機能：100万件での応答時間が3秒以内
```

**補足：** この番号体系により、テストケースTC-001-02が失敗した場合、
対応する詳細設計DD-001-01〜03と基本設計BD-001-01〜03を
すぐに特定でき、原因調査を効率化できます。

</details>

## チャレンジ問題

### 問題6：設計書作成計画の立案

あなたは以下のプロジェクトの設計リーダーに任命されました。どの設計書をどの順番で作成すべきか、作成計画を立ててください。

**プロジェクト概要：**
- 中規模の社内タスク管理システムの新規開発
- ウォーターフォールモデルで開発
- チーム構成：PM 1名、設計者 2名、開発者 5名、テスター 2名
- 開発期間：6ヶ月
- 主な機能：ユーザー管理、タスクCRUD、ダッシュボード、通知機能

**期待される出力例：**

```
【設計書作成計画】

Phase 1（第1月）：要件定義
  - 要件定義書（機能要件・非機能要件）

Phase 2（第2〜3月）：基本設計
  - 基本設計書（システム構成・アーキテクチャ）
  ...
```

<details>
<summary>ヒント</summary>

以下の点を考慮してください。
1. 工程の順序（要件定義→基本設計→詳細設計）
2. 基本設計の中での依存関係（例：DB設計がAPI設計の前提になる）
3. 設計者2名での並行作業の可能性
4. レビューと承認のタイミング

</details>

<details>
<summary>解答例</summary>

```
【設計書作成計画】

Phase 1（第1月）：要件定義工程
  担当：設計者A + 設計者B（共同作成）
  ─────────────────────────────────
  [Week 1-2] 要件定義書（機能要件）
    - ユーザー管理、タスクCRUD、ダッシュボード、通知の
      機能要件を整理
  [Week 3]   要件定義書（非機能要件）
    - パフォーマンス、可用性、セキュリティ要件を定義
  [Week 4]   レビュー・承認
    - PM・ステークホルダーによるレビュー
    - 要件定義書の承認

Phase 2（第2〜3月）：基本設計工程
  担当：設計者A（画面・API）、設計者B（DB・インフラ）
  ─────────────────────────────────
  [Week 5-6] 基本設計書（システム構成・アーキテクチャ）
    - 全体アーキテクチャ、技術スタック選定
    ※ 設計者A・B共同で作成

  [Week 7-8] 並行作業
    設計者A：画面設計書
      - ワイヤーフレーム、画面遷移図
    設計者B：データベース設計書
      - ER図、テーブル定義

  [Week 9-10] 並行作業
    設計者A：API設計書
      - エンドポイント定義、リクエスト/レスポンス仕様
      ※ DB設計書を参照して作成
    設計者B：基本設計書（インフラ・セキュリティ）
      - サーバー構成、ネットワーク設計

  [Week 11-12] レビュー・承認
    - 開発者を含めたレビュー（実装可能性の確認）
    - 基本設計書一式の承認

Phase 3（第3〜4月）：詳細設計工程
  担当：設計者A + 設計者B（機能を分担）
  ─────────────────────────────────
  [Week 13-16] 詳細設計書
    設計者A：ユーザー管理、ダッシュボードの詳細設計
    設計者B：タスクCRUD、通知機能の詳細設計
    - モジュール設計、シーケンス図、フローチャート

  [Week 17-18] テスト設計書
    設計者A + B：テスト計画、テストケース作成
    - 単体テスト・結合テスト・受入テストのケース

  [Week 18] レビュー・承認
    - 詳細設計書・テスト設計書の承認
    - 開発着手のGo/No-Go判定

【依存関係の整理】

  要件定義書 ──→ 基本設計書 ──→ 詳細設計書
                    │
           ┌───────┼───────┐
           ▼       ▼       ▼
         画面     DB      API    ──→ テスト設計書
         設計書   設計書   設計書
                    │
                    └──→ API設計書（DB設計が前提）
```

**補足：**
- DB設計書がAPI設計書より先に完成する必要がある
  （テーブル構造がAPIのレスポンス形式に影響するため）
- 画面設計書とDB設計書は並行作業が可能
  （依存関係が薄い）
- テスト設計書は基本設計・詳細設計の両方を参照するため、
  最後に作成する

</details>

### 問題7：ウォーターフォールとアジャイルの設計書比較

同じ「ECサイトの商品検索機能」を開発する場合、ウォーターフォール開発とアジャイル開発で、それぞれどのような設計ドキュメントを作成するか比較してください。

**期待される出力例：**

```
【ウォーターフォール】
1. 要件定義書に検索機能の要件を記載
2. ...

【アジャイル】
1. ユーザーストーリーをバックログに追加
2. ...
```

<details>
<summary>ヒント</summary>

以下の観点で比較してみましょう。
- 作成するドキュメントの種類と形式
- ドキュメントの詳細度
- 作成タイミング
- 更新・メンテナンスの方法

</details>

<details>
<summary>解答例</summary>

```
【ウォーターフォール開発の場合】

1. 要件定義書
   - 検索機能の要件を詳細に定義
   - 検索条件（商品名、カテゴリ、価格帯、キーワード）
   - 非機能要件（応答時間3秒以内、100万件対応）
   - 形式：Word/Excel、フォーマル

2. 基本設計書
   - 検索画面のワイヤーフレーム
   - 検索APIのインタフェース定義
   - 検索用インデックスの設計
   - 画面遷移図に検索結果画面を追加
   - 形式：Word/Excel + 設計ツール

3. 詳細設計書
   - 検索クエリ組み立てロジックのフローチャート
   - 全文検索のアルゴリズム設計
   - キャッシュ戦略の詳細
   - 形式：Word/Excel

4. テスト設計書
   - 検索機能のテストケース（50件以上）
   - パフォーマンステスト計画
   - 形式：Excel

→ 合計ドキュメント量：50〜100ページ程度
→ 作成期間：機能着手前に全量完成

------------------------------------------------------

【アジャイル開発の場合】

1. ユーザーストーリー（バックログ）
   - 「ユーザーとして、商品名で検索できる」
   - 「ユーザーとして、カテゴリで絞り込みできる」
   - 「ユーザーとして、価格帯で絞り込みできる」
   - 形式：JIRAチケット / GitHub Issue

2. 受け入れ条件（Acceptance Criteria）
   - 各ユーザーストーリーに紐づく
   - 「商品名の部分一致で検索できること」
   - 「検索結果は20件/ページで表示されること」
   - 形式：チケットの記述

3. 技術設計メモ（ADR / Design Doc）
   - 検索エンジンの技術選定（SQL LIKE vs Elasticsearch）
   - 判断理由と代替案の記録
   - 形式：Markdown（リポジトリ内 or Wiki）

4. API仕様
   - OpenAPI（Swagger）で定義
   - コードと連動して自動生成・更新
   - 形式：YAML / 自動生成ドキュメント

5. テスト
   - 自動テストコードが仕様書を兼ねる
   - E2Eテストで受け入れ条件を検証
   - 形式：テストコード

→ 合計ドキュメント量：10〜20ページ程度
→ 作成期間：スプリント内で段階的に作成・更新
```

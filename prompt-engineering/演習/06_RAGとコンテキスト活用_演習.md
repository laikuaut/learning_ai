## 基本問題

### 問題1：RAGの必要性の判断

以下のタスクについて、RAGが必要かどうか判断し、理由を述べてください。

1. Pythonの基本的な文法について質問する
2. 社内の勤怠管理ルール（独自の就業規則）について質問する
3. 今日のニュースを要約する
4. 一般的なデザインパターンについて解説する
5. 自社製品のFAQに回答する

<details>
<summary>解答例</summary>

```
1. RAG不要 ✅
   理由：Pythonの基本文法はLLMの学習データに十分含まれている。
   一般的な知識の範囲内。

2. RAG必要 ✅
   理由：社内独自の就業規則はLLMが学習していない情報。
   就業規則のドキュメントをコンテキストとして提供する必要がある。

3. RAG必要 ✅
   理由：LLMの知識にはカットオフがあり、今日のニュースは
   学習データに含まれていない。ニュースAPIや最新データの提供が必要。

4. RAG不要 ✅
   理由：GoFのデザインパターン等の一般的な知識は
   LLMの学習データに十分含まれている。

5. RAG必要 ✅
   理由：自社製品のFAQは組織固有の情報。
   FAQデータベースから関連情報を取得して提供する必要がある。

■ 判断基準まとめ
  RAG不要：一般的・公知の知識 → LLMが直接回答可能
  RAG必要：組織固有の情報、最新情報、LLMが学習していないデータ
```
</details>

以下のシナリオで、RAGプロンプトのコンテキスト部分を設計してください。

**シナリオ：** 社内の技術ドキュメントから取得した以下の3つのチャンクを使って、「本番デプロイの手順を教えてください」という質問に回答する。

チャンク1（関連度: 0.95）：デプロイ手順書の「本番デプロイ」セクション
チャンク2（関連度: 0.72）：障害対応マニュアルの「ロールバック手順」セクション
チャンク3（関連度: 0.45）：新入社員向け開発環境セットアップガイド

<details>
<summary>解答例</summary>

```
■ コンテキスト設計

まず、関連度のフィルタリング：
- チャンク1（0.95）：採用 ← 直接関連
- チャンク2（0.72）：採用 ← 補足情報として有用
- チャンク3（0.45）：除外 ← 関連度が低い（閾値0.5未満）

■ RAGプロンプト

あなたは社内DevOpsアシスタントです。
以下のドキュメントの情報に基づいて質問に回答してください。
ドキュメントに記載されていない情報は回答に含めないでください。

### 参照ドキュメント

<document source="デプロイ手順書 - 本番デプロイ" relevance="0.95">
（チャンク1の内容）
</document>

<document source="障害対応マニュアル - ロールバック手順" relevance="0.72">
（チャンク2の内容）
</document>

### 質問
本番デプロイの手順を教えてください。

### 回答ルール
- デプロイ手順書の内容を中心に回答する
- ロールバック手順も補足情報として含める
- 手順は番号付きリストで記述する
- 出典のドキュメント名を末尾に記載する

■ 設計のポイント
1. 関連度の低いチャンク3を除外（ノイズ削減）
2. 関連度の高いチャンクを先に配置（重要情報を先頭に）
3. ドキュメントのソースを明記（出典の透明性）
4. LLMに「ドキュメント外の情報を使わない」制約を課している
```

### 問題3：チャンク分割の判断

以下のドキュメントの種類に対して、最適なチャンク分割手法を選んでください。

1. APIリファレンス（エンドポイントごとにセクションが分かれている）
2. 議事録（時系列で記述されている長文）
3. FAQ（質問と回答がペアになっている）
4. 技術ブログ記事（見出しで構造化されている）
5. ログファイル（1行ずつ独立したエントリ）

<details>
<summary>解答例</summary>

```
1. APIリファレンス → セクション分割
   理由：エンドポイントごとに独立した意味のまとまりがある。
   各エンドポイントを1チャンクとすると検索精度が高い。
   例：/api/users のGET/POST/PUT/DELETEを1チャンクに。

2. 議事録 → オーバーラップ付き固定長分割
   理由：時系列の長文では、文脈の切れ目が明確でない。
   固定長（500トークン）+ オーバーラップ（15%）で
   議論の流れが途切れないようにする。

3. FAQ → Q&Aペア単位の分割
   理由：質問と回答は必ずペアで意味を成す。
   各Q&Aペアを1チャンクとする（セマンティック分割）。
   質問のみのチャンクや回答のみのチャンクは意味がない。

4. 技術ブログ記事 → セクション分割（見出しベース）
   理由：H2/H3見出しで意味的に区切られている。
   各セクションを1チャンクとし、見出しをメタデータとして保持。
   大きすぎるセクションはH3レベルで分割。

5. ログファイル → 固定行数分割 + 時間ベースの区切り
   理由：各行が独立しているが、近い時間帯のログは
   関連していることが多い。時間ウィンドウ（例：1分間）で
   グルーピングするか、固定行数（50〜100行）で分割。
```
</details>

## 応用問題

### 問題4：システムプロンプトの設計

社内ヘルプデスクのチャットボット用のシステムプロンプトを設計してください。

要件：
- 人事・総務関連の質問に回答する
- 社内ドキュメント（就業規則、福利厚生ガイド、各種申請マニュアル）をRAGで参照
- 回答できない質問は適切な部署を案内する
- 個人情報に関する質問は回答しない

<details>
<summary>解答例</summary>

```
あなたは「HRアシスタント」という社内ヘルプデスクのAIチャットボットです。

## 目的
社員の人事・総務に関する質問に対して、社内ドキュメントに
基づいて正確かつ迅速に回答することです。

## 参照可能なドキュメント
- 就業規則
- 福利厚生ガイド
- 各種申請マニュアル（有給、経費精算、出張申請等）
- 社内制度説明資料

## 回答ルール

### 基本ルール
- 提供されたドキュメントの内容のみに基づいて回答する
- ドキュメントに記載がない場合は正直に「この情報は
  手元のドキュメントに見つかりませんでした」と伝える
- 敬語（です・ます調）で丁寧に回答する
- 回答は簡潔に（3〜5文以内を目安）
- 根拠となるドキュメント名を必ず記載する

### 回答できないケース
以下の場合は回答せず、適切な対応を案内する：
- 個人情報に関する質問
  → 「個人情報に関するお問い合わせは人事部（内線：1234）に
     直接ご連絡ください」
- 法的判断が必要な質問
  → 「法的な判断が必要な内容です。法務部にご相談ください」
- ドキュメントに記載がない質問
  → 「この件については[該当部署名]にお問い合わせください。
     連絡先：[内線番号]」

### 回答の形式
1. 質問への直接的な回答
2. 根拠：参照したドキュメント名とセクション
3. 補足情報（あれば）
4. 不明点がある場合の問い合わせ先

## 禁止事項
- 個人の給与情報、評価情報に言及しない
- ドキュメントにない情報を推測で回答しない
- 他の社員に関する情報を開示しない
- 退職や転職に関するアドバイスをしない
- 医療に関するアドバイスをしない
```
</details>

## チャレンジ問題

### 問題5：RAGパイプラインの設計

あなたの会社では、エンジニア向けの技術ドキュメント検索システムをRAGで構築しようとしています。以下の情報源を基に、RAGパイプラインの設計書を作成してください。

情報源：
- GitHub上のREADME.md（200リポジトリ分）
- Confluenceの技術ドキュメント（500ページ）
- 社内技術ブログ（100記事）
- APIドキュメント（Swagger/OpenAPI形式）

設計書に含める内容：
1. ドキュメント取り込みパイプライン
2. チャンク分割戦略（情報源ごと）
3. 検索・回答のフロー
4. プロンプト設計
5. 品質改善の仕組み

<details>
<summary>解答例</summary>

```
# 技術ドキュメント検索RAGシステム 設計書

## 1. ドキュメント取り込みパイプライン

情報源 → 取得 → 前処理 → チャンク分割 → ベクトル化 → DB格納

| 情報源        | 取得方法           | 更新頻度   |
|------------- |-------------------|-----------|
| GitHub README | GitHub API         | 日次       |
| Confluence    | Confluence REST API| リアルタイム|
| 技術ブログ     | RSS/API           | 日次       |
| APIドキュメント | CI/CDで自動生成    | デプロイ時  |

## 2. チャンク分割戦略

| 情報源         | 分割手法            | サイズ目安    | オーバーラップ |
|-------------- |-------------------|-------------|-------------|
| GitHub README  | セクション分割（H2）  | 300〜800トークン | 10%   |
| Confluence     | 見出しベース分割     | 500〜1000トークン| 15%   |
| 技術ブログ      | セクション分割（H2/H3）| 300〜500トークン | 10%   |
| APIドキュメント  | エンドポイント単位    | 200〜500トークン | なし   |

メタデータ：
- ソース（GitHub/Confluence/ブログ/API）
- ドキュメント名・URL
- 最終更新日
- 著者
- タグ・カテゴリ

## 3. 検索・回答フロー

質問
  → クエリの前処理（正規化、キーワード抽出）
  → ベクトル検索（上位10件取得）
  → リランキング（関連度でソート）
  → フィルタリング（関連度0.6以上を採用、最大5件）
  → プロンプト構築
  → LLM回答生成
  → 出典付きの回答を返却

## 4. プロンプト設計

あなたは社内の技術ドキュメント検索アシスタントです。
提供されたドキュメントに基づいて、エンジニアの技術的な
質問に回答してください。

ルール：
- ドキュメントの情報のみ使用する
- コード例は原文を正確に引用する
- 複数のドキュメントが該当する場合、すべて言及する
- 情報がない場合は「該当するドキュメントが見つかりません。
  Confluenceで「○○」と検索するか、#tech-helpチャンネルで
  質問してみてください」と案内する

## 5. 品質改善の仕組み

フィードバック収集：
  - 各回答に「役立った / 役立たなかった」ボタン
  - 「役立たなかった」場合に自由記述フィードバック

改善サイクル：
  - 週次で低評価の回答を分析
  - チャンク分割の改善（分割が不適切な箇所を修正）
  - プロンプトの改善（指示の追加・修正）
  - 検索クエリの改善（類義語辞書の更新）

メトリクス：
  - 回答満足度（Good率）: 目標80%以上
  - 回答不能率: 目標10%以下
  - 平均レスポンス時間: 目標3秒以内
```
</details>

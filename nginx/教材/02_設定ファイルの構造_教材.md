# 第2章：設定ファイルの構造

## この章のゴール

- nginx設定ファイルの基本文法（ディレクティブとブロック）を理解する
- コンテキスト（main, events, http, server, location）の階層構造を理解する
- nginx.conf のデフォルト設定を読み解けるようになる
- location ブロックのマッチングルールを理解する
- include ディレクティブによる設定の分割管理を学ぶ

## 2.1 設定ファイルの基本文法

### ディレクティブ（Directive）

nginx の設定はすべて**ディレクティブ**で記述します。ディレクティブには2種類あります。

```nginx
# 1. 単純ディレクティブ（Simple Directive）
#    名前 パラメータ;（セミコロンで終わる）
worker_processes  auto;
error_log  /var/log/nginx/error.log;

# 2. ブロックディレクティブ（Block Directive）
#    名前 { ... }（波括弧で囲む）
events {
    worker_connections 1024;
}
```

```
【ディレクティブの基本構文】

単純ディレクティブ:
  ┌─────────────────────────────────────┐
  │  ディレクティブ名  値1 値2 ... ;      │
  └─────────────────────────────────────┘
  例: listen 80;
      server_name example.com www.example.com;

ブロックディレクティブ:
  ┌─────────────────────────────────────┐
  │  ディレクティブ名  [パラメータ] {     │
  │      子ディレクティブ1;               │
  │      子ディレクティブ2;               │
  │  }                                   │
  └─────────────────────────────────────┘
  例: server {
          listen 80;
          server_name example.com;
      }
```

### コメント

```nginx
# シャープ記号から行末までがコメント
worker_processes auto;  # CPUコア数に自動設定
```

> **よくある間違い**: ディレクティブの末尾のセミコロン `;` を忘れると、`nginx -t` で文法エラーになります。これは初心者が最もよく遭遇するエラーです。

## 2.2 コンテキスト（Context）の階層構造

### 5つの主要コンテキスト

nginx の設定はコンテキスト（Context）と呼ばれる階層構造で組織されます。

```
【コンテキストの階層構造】

nginx.conf
├── main（トップレベル）
│   ├── worker_processes auto;
│   ├── error_log /var/log/nginx/error.log;
│   │
│   ├── events {                    ← eventsコンテキスト
│   │   └── worker_connections 1024;
│   │   }
│   │
│   └── http {                      ← httpコンテキスト
│       ├── include mime.types;
│       ├── default_type application/octet-stream;
│       │
│       ├── server {                ← serverコンテキスト
│       │   ├── listen 80;
│       │   ├── server_name example.com;
│       │   │
│       │   ├── location / {       ← locationコンテキスト
│       │   │   └── root /var/www/html;
│       │   │   }
│       │   │
│       │   └── location /api {    ← locationコンテキスト
│       │       └── proxy_pass http://localhost:3000;
│       │       }
│       │   }
│       │
│       └── server {                ← 別のserverコンテキスト
│           └── ...
│           }
│       }
```

### 各コンテキストの役割

| コンテキスト | 役割 | 主なディレクティブ |
|---|---|---|
| **main** | グローバル設定 | worker_processes, error_log, pid |
| **events** | 接続処理の設定 | worker_connections, use |
| **http** | HTTP関連の共通設定 | include, log_format, gzip |
| **server** | 仮想サーバー（バーチャルホスト） | listen, server_name, root |
| **location** | URI パスごとの処理 | root, proxy_pass, try_files |

## 2.3 デフォルトの nginx.conf を読み解く

### 基本的な nginx.conf

```nginx
# === main コンテキスト ===
user www-data;                     # ワーカープロセスの実行ユーザー
worker_processes auto;              # ワーカープロセス数（auto=CPUコア数）
pid /run/nginx.pid;                 # PIDファイルの場所
error_log /var/log/nginx/error.log; # エラーログ

# === events コンテキスト ===
events {
    worker_connections 1024;        # 1ワーカーあたりの最大同時接続数
                                    # 全体の最大接続数 = worker_processes × worker_connections
}

# === http コンテキスト ===
http {
    # --- MIMEタイプ ---
    include       /etc/nginx/mime.types;   # 拡張子→Content-Typeの対応表
    default_type  application/octet-stream; # 未知の拡張子のデフォルト

    # --- ログ ---
    log_format main '$remote_addr - $remote_user [$time_local] '
                    '"$request" $status $body_bytes_sent '
                    '"$http_referer" "$http_user_agent"';
    access_log /var/log/nginx/access.log main;

    # --- パフォーマンス ---
    sendfile        on;   # カーネルレベルのファイル転送（高速化）
    tcp_nopush      on;   # sendfile使用時にヘッダとボディをまとめて送信
    keepalive_timeout 65;  # Keep-Alive接続のタイムアウト（秒）

    # --- 圧縮 ---
    gzip on;               # gzip圧縮を有効化
    gzip_types text/plain text/css application/json application/javascript;

    # --- サーバー設定の読み込み ---
    include /etc/nginx/conf.d/*.conf;          # conf.d内の全.confファイル
    include /etc/nginx/sites-enabled/*;        # sites-enabled内の全ファイル
}
```

## 2.4 locationブロックのマッチング

### location の修飾子

location ブロックは、リクエストURIに基づいて処理を振り分けます。

```nginx
# 1. プレフィックスマッチ（修飾子なし）
location /images/ {
    # /images/ で始まるURIにマッチ
    # 例: /images/logo.png, /images/photos/cat.jpg
}

# 2. 完全一致（=）
location = /favicon.ico {
    # /favicon.ico に完全一致する場合のみマッチ
}

# 3. 優先プレフィックスマッチ（^~）
location ^~ /static/ {
    # /static/ で始まるURIにマッチ（正規表現より優先）
}

# 4. 正規表現マッチ（~：大文字小文字区別あり）
location ~ \.(jpg|png|gif)$ {
    # .jpg, .png, .gif で終わるURIにマッチ
}

# 5. 正規表現マッチ（~*：大文字小文字区別なし）
location ~* \.(jpg|png|gif)$ {
    # .JPG, .Png なども含めてマッチ
}
```

### マッチングの優先順位

```
【locationのマッチング優先順位】

優先度 高
  │
  │  1. = （完全一致）           → 最初にチェック。一致したら即確定
  │  2. ^~ （優先プレフィックス）  → 最長一致のプレフィックスが ^~ なら確定
  │  3. ~ / ~* （正規表現）      → 設定ファイルに書かれた順に評価
  │  4. （プレフィックスマッチ）   → 最長一致のプレフィックスを採用
  │
優先度 低


【マッチング例：リクエスト URI = "/images/photo.jpg"】

location = /images/photo.jpg { A }   ← 完全一致 → A に決定
location ^~ /images/ { B }            ← 優先プレフィックス
location ~ \.(jpg|png)$ { C }        ← 正規表現
location /images/ { D }               ← プレフィックス

= がなければ:  ^~ /images/ → B に決定
^~ もなければ:  ~ \.(jpg|png)$ → C に決定
正規表現もなければ: /images/ → D に決定
```

### よく使う location パターン

```nginx
server {
    listen 80;
    server_name example.com;

    # ルート（トップページ）
    location / {
        root /var/www/html;
        index index.html;
    }

    # 静的ファイル（キャッシュ設定付き）
    location ~* \.(css|js|jpg|png|gif|ico|svg|woff2?)$ {
        root /var/www/html;
        expires 30d;                 # 30日間キャッシュ
        add_header Cache-Control "public, immutable";
    }

    # APIへのプロキシ
    location /api/ {
        proxy_pass http://localhost:3000;
    }

    # favicon.ico のログを抑制
    location = /favicon.ico {
        log_not_found off;
        access_log off;
    }

    # robots.txt
    location = /robots.txt {
        log_not_found off;
        access_log off;
    }
}
```

> **実務でのポイント**: location ブロックの優先順位を理解していないと、意図したルールが適用されず、「設定したのに動かない」というトラブルの原因になります。特に正規表現マッチ（`~`）とプレフィックスマッチの優先関係に注意してください。

## 2.5 include ディレクティブによる設定の分割

### 設定ファイルの分割管理

```nginx
# nginx.conf のhttpブロック内
http {
    # 共通設定
    include /etc/nginx/conf.d/*.conf;

    # サイト別設定
    include /etc/nginx/sites-enabled/*;
}
```

```
【推奨するファイル分割パターン】

/etc/nginx/
├── nginx.conf                  # グローバル設定のみ
├── conf.d/
│   ├── gzip.conf              # gzip圧縮の共通設定
│   ├── security-headers.conf  # セキュリティヘッダー
│   └── logging.conf           # ログフォーマット
├── snippets/
│   ├── ssl-params.conf        # SSL共通パラメータ
│   └── proxy-params.conf      # プロキシ共通パラメータ
└── sites-enabled/
    ├── example.com.conf       # サイトA
    └── api.example.com.conf   # サイトB
```

### snippets による再利用

```nginx
# /etc/nginx/snippets/ssl-params.conf
ssl_protocols TLSv1.2 TLSv1.3;
ssl_prefer_server_ciphers on;
ssl_ciphers 'ECDHE-ECDSA-AES128-GCM-SHA256:ECDHE-RSA-AES128-GCM-SHA256';
ssl_session_cache shared:SSL:10m;
ssl_session_timeout 1d;
add_header Strict-Transport-Security "max-age=63072000" always;

# サイト設定で include する
server {
    listen 443 ssl;
    ssl_certificate /etc/letsencrypt/live/example.com/fullchain.pem;
    ssl_certificate_key /etc/letsencrypt/live/example.com/privkey.pem;
    include snippets/ssl-params.conf;  # SSL共通設定を読み込み
}
```

## 2.6 変数

### 組み込み変数

nginx には多くの組み込み変数があります。

```
【よく使う組み込み変数】

| 変数名              | 内容                          | 例                    |
|---------------------|-------------------------------|----------------------|
| $host               | リクエストの Host ヘッダー      | example.com          |
| $uri                | 正規化されたリクエストURI       | /index.html          |
| $request_uri        | 元のリクエストURI（クエリ付き）  | /search?q=nginx      |
| $remote_addr        | クライアントのIPアドレス        | 192.168.1.10         |
| $request_method     | HTTPメソッド                   | GET, POST            |
| $scheme             | プロトコル                     | http, https          |
| $server_name        | server_name の値              | example.com          |
| $args               | クエリ文字列                   | q=nginx&page=1       |
| $http_user_agent    | User-Agent ヘッダー           | Mozilla/5.0 ...      |
| $status             | レスポンスステータスコード      | 200, 404             |
```

```nginx
# 変数の使用例：カスタムログフォーマット
log_format detailed '$remote_addr - [$time_local] '
                    '"$request_method $uri" $status '
                    'Host: $host '
                    'UA: "$http_user_agent"';
```

## 2.7 try_files ディレクティブ

### try_files の基本

`try_files` は、指定した順序でファイルの存在を確認し、最初に見つかったものを返します。

```nginx
# 基本構文
try_files ファイル1 ファイル2 ... 最終処理;

# 例：静的ファイル → index.html → 404
location / {
    root /var/www/html;
    try_files $uri $uri/ =404;
    # 1. $uri（リクエストされたファイル）を探す
    # 2. $uri/（ディレクトリとして index を探す）
    # 3. どちらもなければ 404 を返す
}

# 例：SPA（Single Page Application）用
location / {
    root /var/www/app;
    try_files $uri $uri/ /index.html;
    # ファイルが見つからなければ /index.html を返す
    # → React, Vue 等のクライアントサイドルーティングに対応
}
```

# 第8章: ユーティリティ型

## この章のゴール

- TypeScriptが標準で提供するユーティリティ型を理解し、使いこなせるようになる
- `Partial`, `Required`, `Pick`, `Omit`, `Record` など主要なユーティリティ型を実務で活用できるようになる
- 各ユーティリティ型の内部実装を理解し、独自のユーティリティ型を作れるようになる
- 複数のユーティリティ型を組み合わせて、複雑な型変換を表現できるようになる

---

## 8.1 ユーティリティ型とは

### 既存の型を変換するツール

ユーティリティ型（Utility Types）は、TypeScriptが標準で提供する**型変換のためのジェネリック型**です。既存の型をベースにして、プロパティをオプショナルにしたり、特定のプロパティだけを抽出したりできます。

なぜユーティリティ型が必要なのでしょうか。実務では、同じデータモデルでも場面によって必要な型が異なります。

```typescript
// ユーザーの完全な型
interface User {
  id: number;
  name: string;
  email: string;
  age: number;
  role: "admin" | "user";
  createdAt: Date;
}

// 場面ごとに異なる型が必要:
// - 新規作成時: id と createdAt は不要
// - 更新時: 全フィールドがオプショナル
// - 一覧表示: id, name, role のみ必要
// - プロフィール: email, age, role だけ更新可能
```

これらを全て手動で定義すると、重複が大量に発生し、元の型を変更した際に全ての派生型を修正する必要があります。ユーティリティ型を使えば、元の型からの**自動変換**でこれらの型を定義できます。

```
┌─────────────────────────────────────────────────┐
│          ユーティリティ型のイメージ                │
│                                                   │
│  User ──→ Partial<User>     全プロパティを?に     │
│       ──→ Required<User>    全プロパティを必須に   │
│       ──→ Pick<User, "id" | "name">  一部を抽出  │
│       ──→ Omit<User, "createdAt">    一部を除外  │
│       ──→ Readonly<User>    全プロパティを読取専用 │
│                                                   │
│  元の型(User)を変更すれば、派生型も自動で追従     │
└─────────────────────────────────────────────────┘
```

## 8.2 Partial<T> -- 全プロパティをオプショナルに

### 基本的な使い方

`Partial<T>` は、`T` の全てのプロパティをオプショナル（`?`）にした型を生成します。

```typescript
interface User {
  id: number;
  name: string;
  email: string;
  age: number;
}

type PartialUser = Partial<User>;
// 等価:
// {
//   id?: number;
//   name?: string;
//   email?: string;
//   age?: number;
// }
```

### 実践例: 更新処理

最も典型的な使い方は、エンティティの部分更新です。

```typescript
interface User {
  id: number;
  name: string;
  email: string;
  age: number;
}

// 更新関数: 変更したいフィールドだけ指定できる
function updateUser(id: number, updates: Partial<User>): User {
  const currentUser = getUserById(id); // 現在のユーザーを取得
  return { ...currentUser, ...updates };
}

// 名前だけ更新
updateUser(1, { name: "新しい名前" });

// メールとageを更新
updateUser(1, { email: "new@example.com", age: 30 });

// 空の更新も可能（何も変わらない）
updateUser(1, {});
```

### 内部実装

```typescript
// Partial<T> の実装（TypeScript標準ライブラリ）
type MyPartial<T> = {
  [P in keyof T]?: T[P];
};
// keyof T で全てのキーを列挙し、各プロパティに ? を付加
```

## 8.3 Required<T> -- 全プロパティを必須に

### 基本的な使い方

`Required<T>` は、`T` の全てのオプショナルプロパティを必須にした型を生成します。`Partial<T>` の逆です。

```typescript
interface Config {
  host?: string;
  port?: number;
  debug?: boolean;
  timeout?: number;
}

type RequiredConfig = Required<Config>;
// 等価:
// {
//   host: string;
//   port: number;
//   debug: boolean;
//   timeout: number;
// }
```

### 実践例: デフォルト値の適用

```typescript
interface AppConfig {
  host?: string;
  port?: number;
  debug?: boolean;
  logLevel?: "debug" | "info" | "warn" | "error";
}

const DEFAULT_CONFIG: Required<AppConfig> = {
  host: "localhost",
  port: 3000,
  debug: false,
  logLevel: "info",
};

function createConfig(userConfig: AppConfig): Required<AppConfig> {
  return { ...DEFAULT_CONFIG, ...userConfig };
}

// ユーザーが一部だけ指定
const config = createConfig({ port: 8080, debug: true });
// { host: "localhost", port: 8080, debug: true, logLevel: "info" }
// 戻り値の型は Required<AppConfig> なので、全プロパティが確実に存在
```

### 内部実装

```typescript
// Required<T> の実装
type MyRequired<T> = {
  [P in keyof T]-?: T[P];
};
// -? で オプショナル修飾子(?)を除去
```

## 8.4 Readonly<T> -- 全プロパティを読み取り専用に

### 基本的な使い方

`Readonly<T>` は、`T` の全てのプロパティを `readonly` にした型を生成します。

```typescript
interface Todo {
  id: number;
  title: string;
  completed: boolean;
}

const todo: Readonly<Todo> = {
  id: 1,
  title: "TypeScriptを学ぶ",
  completed: false,
};

// todo.completed = true; // エラー！readonly プロパティに代入できない
```

### 実践例: 不変データの保護

```typescript
// 状態管理: stateを直接変更させない
interface AppState {
  users: User[];
  selectedUserId: number | null;
  isLoading: boolean;
}

function getState(): Readonly<AppState> {
  return currentState; // 読み取り専用として返す
}

const state = getState();
// state.isLoading = true; // エラー！
// state.users.push(newUser); // 注意: 浅いreadonly（配列の中身は変更できてしまう）
```

> **注意:** `Readonly<T>` は**浅い**（shallow）変換です。ネストしたオブジェクトや配列の中身は保護されません。深いReadonlyが必要な場合は、自作するかライブラリを使います。

### 内部実装

```typescript
// Readonly<T> の実装
type MyReadonly<T> = {
  readonly [P in keyof T]: T[P];
};
```

## 8.5 Pick<T, K> -- 特定のプロパティだけ抽出

### 基本的な使い方

`Pick<T, K>` は、`T` から指定したキー `K` のプロパティだけを抽出した型を生成します。

```typescript
interface User {
  id: number;
  name: string;
  email: string;
  age: number;
  role: "admin" | "user";
  createdAt: Date;
}

// 一覧表示用の型
type UserListItem = Pick<User, "id" | "name" | "role">;
// 等価:
// {
//   id: number;
//   name: string;
//   role: "admin" | "user";
// }

// プロフィール表示用の型
type UserProfile = Pick<User, "name" | "email" | "age">;
// 等価:
// {
//   name: string;
//   email: string;
//   age: number;
// }
```

### 実践例: APIレスポンスの型

```typescript
interface Product {
  id: number;
  name: string;
  description: string;
  price: number;
  stock: number;
  category: string;
  imageUrl: string;
  createdAt: Date;
  updatedAt: Date;
}

// カート表示に必要な情報だけ
type CartItem = Pick<Product, "id" | "name" | "price"> & {
  quantity: number;
};

// 検索結果に表示する情報
type SearchResult = Pick<Product, "id" | "name" | "price" | "category" | "imageUrl">;
```

### 内部実装

```typescript
// Pick<T, K> の実装
type MyPick<T, K extends keyof T> = {
  [P in K]: T[P];
};
// K は T のキーのサブセットでなければならない（extends keyof T）
```

## 8.6 Omit<T, K> -- 特定のプロパティを除外

### 基本的な使い方

`Omit<T, K>` は、`T` から指定したキー `K` のプロパティを除外した型を生成します。`Pick` の逆です。

```typescript
interface User {
  id: number;
  name: string;
  email: string;
  age: number;
  createdAt: Date;
  updatedAt: Date;
}

// 新規作成用: id, createdAt, updatedAt はサーバーで自動生成
type CreateUserInput = Omit<User, "id" | "createdAt" | "updatedAt">;
// 等価:
// {
//   name: string;
//   email: string;
//   age: number;
// }
```

### Pick vs Omit の使い分け

```
┌────────────────────────────────────────────────┐
│          Pick vs Omit の選び方                  │
│                                                  │
│  プロパティが多い型から少しだけ使いたい          │
│  → Pick を使う（必要なものを列挙）              │
│                                                  │
│  プロパティが多い型から少しだけ除きたい          │
│  → Omit を使う（不要なものを列挙）              │
│                                                  │
│  例: 10個のプロパティがある型から...             │
│  ・3個だけ使いたい → Pick<T, "a" | "b" | "c">  │
│  ・2個だけ除きたい → Omit<T, "x" | "y">        │
└────────────────────────────────────────────────┘
```

### 実践例: データベースのCRUD型

```typescript
interface Entity {
  id: number;
  createdAt: Date;
  updatedAt: Date;
}

interface User extends Entity {
  name: string;
  email: string;
  age: number;
  role: "admin" | "user";
}

// 作成時: id, createdAt, updatedAt は自動生成
type CreateUser = Omit<User, keyof Entity>;
// { name: string; email: string; age: number; role: "admin" | "user" }

// 更新時: 作成入力のPartial
type UpdateUser = Partial<CreateUser>;
// { name?: string; email?: string; age?: number; role?: "admin" | "user" }

// 一覧表示: idと基本情報のみ
type UserSummary = Pick<User, "id" | "name" | "role">;
```

### 内部実装

```typescript
// Omit<T, K> の実装
type MyOmit<T, K extends keyof any> = Pick<T, Exclude<keyof T, K>>;
// Exclude<keyof T, K> で T のキーから K を除外し、Pick で残りを抽出
```

## 8.7 Record<K, V> -- キーと値の型を指定したオブジェクト

### 基本的な使い方

`Record<K, V>` は、キーの型が `K`、値の型が `V` であるオブジェクト型を生成します。

```typescript
// 基本
type StringRecord = Record<string, number>;
// { [key: string]: number }

const scores: StringRecord = {
  math: 85,
  english: 92,
  science: 78,
};

// リテラル型をキーに使う
type Role = "admin" | "editor" | "viewer";

type RolePermissions = Record<Role, string[]>;

const permissions: RolePermissions = {
  admin: ["read", "write", "delete", "manage"],
  editor: ["read", "write"],
  viewer: ["read"],
};
// 全てのRoleに対応するエントリーが必要
```

### 実践例: ルックアップテーブル

```typescript
// ステータスコードのメッセージ
type HttpStatusCode = 200 | 201 | 400 | 401 | 403 | 404 | 500;

const HTTP_MESSAGES: Record<HttpStatusCode, string> = {
  200: "OK",
  201: "Created",
  400: "Bad Request",
  401: "Unauthorized",
  403: "Forbidden",
  404: "Not Found",
  500: "Internal Server Error",
};

// 言語別の翻訳
type Language = "ja" | "en" | "zh";

type Translations = Record<Language, Record<string, string>>;

const translations: Translations = {
  ja: { greeting: "こんにちは", farewell: "さようなら" },
  en: { greeting: "Hello", farewell: "Goodbye" },
  zh: { greeting: "你好", farewell: "再见" },
};
```

### 実践例: マップ/辞書の型

```typescript
// IDをキーにしたエンティティマップ
type EntityMap<T extends { id: number }> = Record<number, T>;

interface User {
  id: number;
  name: string;
}

const userMap: EntityMap<User> = {
  1: { id: 1, name: "太郎" },
  2: { id: 2, name: "花子" },
  3: { id: 3, name: "次郎" },
};

// 正規化されたストア（Redux等で使用）
interface NormalizedState<T extends { id: number }> {
  byId: Record<number, T>;
  allIds: number[];
}

const usersState: NormalizedState<User> = {
  byId: {
    1: { id: 1, name: "太郎" },
    2: { id: 2, name: "花子" },
  },
  allIds: [1, 2],
};
```

### 内部実装

```typescript
// Record<K, T> の実装
type MyRecord<K extends keyof any, T> = {
  [P in K]: T;
};
```

## 8.8 Exclude<T, U> と Extract<T, U>

### Exclude: ユニオン型から特定の型を除外

```typescript
type AllColors = "red" | "green" | "blue" | "yellow" | "purple";

// "red" と "blue" を除外
type WarmColors = Exclude<AllColors, "red" | "blue">;
// "green" | "yellow" | "purple"

// 実践例: イベントの種類をフィルタリング
type AllEvents = "click" | "focus" | "blur" | "keydown" | "keyup" | "scroll";
type KeyboardEvents = Extract<AllEvents, "keydown" | "keyup">;
// "keydown" | "keyup"

type NonKeyboardEvents = Exclude<AllEvents, "keydown" | "keyup">;
// "click" | "focus" | "blur" | "scroll"
```

### Extract: ユニオン型から特定の型を抽出

```typescript
type Mixed = string | number | boolean | null | undefined;

// string | number だけを抽出
type Primitives = Extract<Mixed, string | number>;
// string | number

// null | undefined だけを抽出
type Nullish = Extract<Mixed, null | undefined>;
// null | undefined
```

### 内部実装

```typescript
// Exclude<T, U> の実装
type MyExclude<T, U> = T extends U ? never : T;

// Extract<T, U> の実装
type MyExtract<T, U> = T extends U ? T : never;
```

## 8.9 NonNullable<T>

`NonNullable<T>` は、`T` から `null` と `undefined` を除外します。

```typescript
type MaybeString = string | null | undefined;

type DefiniteString = NonNullable<MaybeString>;
// string

// 実践例
interface FormData {
  name: string | null;
  email: string | null;
  age: number | null;
}

// バリデーション済みのフォームデータ
type ValidatedFormData = {
  [K in keyof FormData]: NonNullable<FormData[K]>;
};
// { name: string; email: string; age: number; }
```

### 内部実装

```typescript
// NonNullable<T> の実装
type MyNonNullable<T> = T extends null | undefined ? never : T;
```

## 8.10 ReturnType<T> と Parameters<T>

### ReturnType: 関数の戻り値の型を取得

```typescript
function createUser(name: string, age: number) {
  return {
    id: Math.random(),
    name,
    age,
    createdAt: new Date(),
  };
}

// 関数の戻り値から型を取得
type User = ReturnType<typeof createUser>;
// { id: number; name: string; age: number; createdAt: Date; }

// 使い場面: ライブラリの関数の戻り値の型が知りたいとき
// 型定義ファイルを探すよりも、ReturnType で直接取得できる
```

### Parameters: 関数の引数の型を取得

```typescript
function sendEmail(to: string, subject: string, body: string): void {
  // ...
}

type EmailParams = Parameters<typeof sendEmail>;
// [to: string, subject: string, body: string]

// 個別のパラメータの型を取得
type FirstParam = Parameters<typeof sendEmail>[0]; // string
type SecondParam = Parameters<typeof sendEmail>[1]; // string
```

### 実践例: ラッパー関数

```typescript
// 元の関数の型に合わせてラッパーを定義
function originalFetch(url: string, options?: RequestInit): Promise<Response> {
  return fetch(url, options);
}

// パラメータと戻り値の型を自動的に合わせる
function wrappedFetch(
  ...args: Parameters<typeof originalFetch>
): ReturnType<typeof originalFetch> {
  console.log(`Fetching: ${args[0]}`);
  return originalFetch(...args);
}
```

## 8.11 ユーティリティ型の組み合わせ

実務では、複数のユーティリティ型を組み合わせて使うことが多いです。

### パターン1: 一部のプロパティだけオプショナル

```typescript
// Tの中でKに指定したプロパティだけをオプショナルにする
type PartialBy<T, K extends keyof T> = Omit<T, K> & Partial<Pick<T, K>>;

interface User {
  id: number;
  name: string;
  email: string;
  age: number;
}

// email と age だけオプショナル
type UserWithOptionalContact = PartialBy<User, "email" | "age">;
// { id: number; name: string; email?: string; age?: number; }
```

### パターン2: 一部のプロパティだけ必須

```typescript
// Tの中でKに指定したプロパティだけを必須にする
type RequiredBy<T, K extends keyof T> = T & Required<Pick<T, K>>;

interface Config {
  host?: string;
  port?: number;
  debug?: boolean;
  apiKey?: string;
}

// apiKeyだけ必須にする
type ProductionConfig = RequiredBy<Config, "apiKey">;
// { host?: string; port?: number; debug?: boolean; apiKey: string; }
```

### パターン3: CRUD操作の型セット

```typescript
interface Entity {
  id: number;
  createdAt: Date;
  updatedAt: Date;
}

// CRUD操作に必要な型を自動生成するユーティリティ
type CRUDTypes<T extends Entity> = {
  Full: T;
  Create: Omit<T, keyof Entity>;
  Update: Partial<Omit<T, keyof Entity>>;
  Summary: Pick<T, "id"> & Partial<Omit<T, "id">>;
};

interface Product extends Entity {
  name: string;
  price: number;
  stock: number;
  category: string;
}

type ProductCRUD = CRUDTypes<Product>;
// ProductCRUD["Full"]    → Product
// ProductCRUD["Create"]  → { name: string; price: number; stock: number; category: string; }
// ProductCRUD["Update"]  → { name?: string; price?: number; stock?: number; category?: string; }
// ProductCRUD["Summary"] → { id: number; name?: string; ... }
```

### パターン4: 読み取り専用のディープ変換

```typescript
// DeepReadonly: ネストしたオブジェクトも再帰的にreadonly化
type DeepReadonly<T> = {
  readonly [P in keyof T]: T[P] extends object
    ? T[P] extends Function
      ? T[P]
      : DeepReadonly<T[P]>
    : T[P];
};

interface NestedConfig {
  server: {
    host: string;
    port: number;
  };
  database: {
    connection: {
      url: string;
      pool: number;
    };
  };
}

type FrozenConfig = DeepReadonly<NestedConfig>;
// 全てのネストしたプロパティがreadonly
```

## 8.12 よくある間違い

### 1. Omit のキーを間違える

```typescript
interface User {
  id: number;
  name: string;
  email: string;
}

// ❌ 存在しないキーを指定してもエラーにならない
type WithoutPhone = Omit<User, "phone">; // User と同じ型になる
// TypeScriptはOmitのキーの存在チェックをしない

// ✅ 安全なOmitを自作する
type StrictOmit<T, K extends keyof T> = Omit<T, K>;
// type Bad = StrictOmit<User, "phone">; // エラー！"phone"はUserのキーではない
```

### 2. Partial と undefined の混同

```typescript
interface User {
  name: string;
  age: number;
}

type PartialUser = Partial<User>;
// { name?: string; age?: number; }

// Partial はプロパティを「省略可能」にする
// 「値がundefinedかもしれない」とは微妙に異なる

const user1: PartialUser = {};        // OK: プロパティ自体が存在しない
const user2: PartialUser = { name: undefined }; // OK: undefinedも代入可能
```

### 3. Readonly の浅さを忘れる

```typescript
interface State {
  users: string[];
  config: { debug: boolean };
}

const state: Readonly<State> = {
  users: ["太郎"],
  config: { debug: false },
};

// state.users = [];  // エラー！readonly
state.users.push("花子");    // ⚠️ エラーにならない！配列の中身は変更可能
state.config.debug = true;   // ⚠️ エラーにならない！ネストしたオブジェクトは変更可能
```

## 8.13 現場でのベストプラクティス

### 1. 型の再利用を意識する

```typescript
// ❌ 重複した型定義
interface CreateUserInput {
  name: string;
  email: string;
  age: number;
}

interface UpdateUserInput {
  name?: string;
  email?: string;
  age?: number;
}

// ✅ ユーティリティ型で派生させる
interface User {
  id: number;
  name: string;
  email: string;
  age: number;
  createdAt: Date;
}

type CreateUserInput2 = Omit<User, "id" | "createdAt">;
type UpdateUserInput2 = Partial<CreateUserInput2>;
```

### 2. ユーティリティ型の一覧を把握する

| ユーティリティ型 | 用途 |
|-----------------|------|
| `Partial<T>` | 全プロパティをオプショナルに |
| `Required<T>` | 全プロパティを必須に |
| `Readonly<T>` | 全プロパティを読み取り専用に |
| `Pick<T, K>` | 指定したプロパティだけ抽出 |
| `Omit<T, K>` | 指定したプロパティを除外 |
| `Record<K, V>` | キーと値の型を指定したオブジェクト |
| `Exclude<T, U>` | ユニオンから特定の型を除外 |
| `Extract<T, U>` | ユニオンから特定の型を抽出 |
| `NonNullable<T>` | null と undefined を除外 |
| `ReturnType<T>` | 関数の戻り値の型を取得 |
| `Parameters<T>` | 関数のパラメータの型を取得 |
| `Awaited<T>` | Promiseの中身の型を取得 |
| `ConstructorParameters<T>` | コンストラクタの引数の型 |
| `InstanceType<T>` | クラスのインスタンスの型 |

### 3. プロジェクト共通のユーティリティ型を定義する

```typescript
// types/utility.ts - プロジェクト共通のユーティリティ型
export type Nullable<T> = T | null;
export type Optional<T> = T | undefined;
export type Maybe<T> = T | null | undefined;

export type PartialBy<T, K extends keyof T> = Omit<T, K> & Partial<Pick<T, K>>;
export type RequiredBy<T, K extends keyof T> = T & Required<Pick<T, K>>;

export type StrictOmit<T, K extends keyof T> = Omit<T, K>;

export type DeepPartial<T> = {
  [P in keyof T]?: T[P] extends object ? DeepPartial<T[P]> : T[P];
};
```

## 8.14 まとめ

| トピック | 内容 |
|---------|------|
| Partial<T> | 全プロパティをオプショナルに（更新処理に最適） |
| Required<T> | 全プロパティを必須に（デフォルト値適用後の型に） |
| Readonly<T> | 全プロパティを読み取り専用に（浅い変換に注意） |
| Pick<T, K> | 必要なプロパティだけ抽出 |
| Omit<T, K> | 不要なプロパティを除外 |
| Record<K, V> | キーと値の型を指定したオブジェクト |
| Exclude / Extract | ユニオン型のフィルタリング |
| ReturnType / Parameters | 関数の型情報を取得 |
| 組み合わせ | 複数のユーティリティ型を合成して複雑な型変換を実現 |

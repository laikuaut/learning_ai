# 第2章：要件定義書

## この章のゴール

- 要件定義書（Requirements Definition Document）の目的と構成を理解する
- 機能要件（Functional Requirements）と非機能要件（Non-Functional Requirements）の違いを理解する
- 業務フロー図の読み書きができるようになる
- ユースケース記述の書き方を身につける
- 要件の優先順位付けと合意形成の方法を理解する

---

## 2.1 要件定義書とは

### 要件定義書の位置づけ

要件定義書は、**「何を作るか（What）」を定義する文書**です。「どう作るか（How）」を記述する設計書の前段階に位置します。

```
【要件定義書の位置づけ】

顧客の要望          要件定義書           基本設計書
（曖昧・口頭）  →  （明確・文書化）  →  （技術的な設計）

「商品を検索      「商品名・カテゴリ・    「GET /api/products
 できるように      価格帯で検索可能。      ?name=xxx&category=yyy
 してほしい」      1ページ20件表示。       レスポンス: JSON
                   応答3秒以内」           キャッシュ: Redis TTL 5分」
```

### 要件定義書の構成

一般的な要件定義書は以下の構成で作成します。

```
要件定義書の目次（例）
━━━━━━━━━━━━━━━━━━━━
1. プロジェクト概要
   1.1 背景・目的
   1.2 スコープ
   1.3 前提条件・制約条件
   1.4 用語定義

2. 業務要件
   2.1 現行業務フロー（As-Is）
   2.2 新業務フロー（To-Be）
   2.3 業務ルール

3. 機能要件
   3.1 機能一覧
   3.2 ユースケース記述
   3.3 画面一覧
   3.4 帳票一覧

4. 非機能要件
   4.1 性能要件
   4.2 可用性要件
   4.3 セキュリティ要件
   4.4 運用・保守要件
   4.5 移行要件

5. 外部インタフェース要件
   5.1 外部システム連携
   5.2 データ連携

6. 制約事項・リスク
━━━━━━━━━━━━━━━━━━━━
```

## 2.2 プロジェクト概要の記述

### 背景・目的

なぜこのシステムを作るのか、ビジネス上の背景と目的を明記します。

**悪い例：**

```
目的：勤怠管理システムを構築する。
```

**良い例：**

```
【背景】
現在、従業員300名の勤怠管理をExcelで運用している。
以下の課題が発生しており、業務効率の改善が急務である。

  課題1：月末の集計作業に担当者2名×3日間を要している
  課題2：入力ミスによる給与計算の誤りが月平均5件発生している
  課題3：リモートワーク社員の打刻手段がなく、メール報告に依存している

【目的】
上記課題を解決するため、Webベースの勤怠管理システムを構築する。
導入により以下の効果を見込む。

  効果1：月末集計作業を自動化し、工数を90%削減する
  効果2：入力バリデーションにより、入力ミスをゼロにする
  効果3：PC・スマートフォンからの打刻を可能にする
```

### スコープの定義

スコープ（Scope）とは、このプロジェクトで**対応する範囲と対応しない範囲**を明確にすることです。

```
【スコープ定義の例】

┌─────────────────────────────────────────────┐
│ スコープ内（対応する）                         │
│                                             │
│  ・出退勤の打刻（PC・スマートフォン）          │
│  ・勤怠データの集計・レポート出力              │
│  ・有給休暇の申請・承認ワークフロー            │
│  ・残業申請・承認ワークフロー                  │
│  ・管理者向けダッシュボード                    │
├─────────────────────────────────────────────┤
│ スコープ外（対応しない）                       │
│                                             │
│  ・給与計算（既存の給与システムに連携する）     │
│  ・人事評価機能                               │
│  ・経費精算機能                               │
│  ・シフト管理機能（フェーズ2で検討）           │
└─────────────────────────────────────────────┘
```

> **実務でのポイント**: スコープ外の定義は、スコープ内と同じくらい重要です。「これは対応しない」と明記しておくことで、後からの追加要求（スコープクリープ）を防止できます。

### 前提条件と制約条件

| 種類 | 定義 | 例 |
|---|---|---|
| **前提条件** | 真であると仮定する事項 | 「全社員がメールアドレスを持っている」 |
| **制約条件** | 変更できない制限事項 | 「既存の給与システム（SAP）と連携必須」 |

```
【前提条件】
1. 全従業員（300名）が社用メールアドレスを保持している
2. オフィスにはWi-Fi環境が整備されている
3. 従業員の80%以上がスマートフォンを所持している

【制約条件】
1. 既存の給与システム（SAP HR）にCSV形式でデータ連携すること
2. 社内のAWSアカウントを使用すること（オンプレミス不可）
3. 開発予算は1,500万円以内とする
4. 2026年10月の運用開始に間に合わせること
```

### 用語定義

プロジェクト内で使用する用語を定義し、認識のズレを防ぎます。

```
【用語定義（例）】

| 用語         | 定義                                          |
|-------------|----------------------------------------------|
| 打刻         | 出勤・退勤時刻を記録する行為                    |
| コアタイム    | 全社員が勤務する必要がある時間帯（10:00〜15:00） |
| フレックス   | 始業・終業時刻を自由に設定できる勤務制度         |
| 所定労働時間  | 1日あたりの標準勤務時間（8時間）                 |
| 法定外残業    | 1日8時間または週40時間を超える労働時間           |
```

## 2.3 機能要件の定義

### 機能要件とは

機能要件（Functional Requirements）は、**システムが「何をするか」を定義**するものです。

### 機能一覧の作成

まず、システムが提供する機能を一覧化します。

```
【機能一覧（例：勤怠管理システム）】

┌────────┬──────────────┬───────────────────────────┬──────┐
│ 機能ID │ 機能名        │ 概要                       │ 優先度│
├────────┼──────────────┼───────────────────────────┼──────┤
│ F-001  │ 出退勤打刻    │ PC/スマホから出退勤を記録   │ 必須  │
│ F-002  │ 打刻修正      │ 誤った打刻データを修正する  │ 必須  │
│ F-003  │ 有給休暇申請  │ 有給休暇の申請と承認       │ 必須  │
│ F-004  │ 残業申請      │ 残業の事前申請と承認       │ 必須  │
│ F-005  │ 勤怠集計      │ 月次の勤怠データを自動集計  │ 必須  │
│ F-006  │ レポート出力  │ 部署別・個人別レポート出力  │ 重要  │
│ F-007  │ ダッシュボード │ 管理者向け勤怠状況表示     │ 重要  │
│ F-008  │ アラート通知  │ 残業超過や未打刻の通知     │ 希望  │
│ F-009  │ データ連携    │ 給与システムへCSVエクスポート│ 必須  │
│ F-010  │ ユーザー管理  │ 社員マスタの管理           │ 必須  │
└────────┴──────────────┴───────────────────────────┴──────┘
```

### ユースケース記述

各機能の詳細を、ユースケース記述として文書化します。

```
【ユースケース記述（例）】

ユースケースID ：UC-001
ユースケース名 ：出退勤を打刻する
アクター       ：一般社員
事前条件       ：ユーザーがログイン済みであること
事後条件       ：打刻データが保存され、一覧に表示されること

【基本フロー（正常系）】
  1. ユーザーがトップ画面の「出勤」ボタンを押す
  2. システムが現在時刻を取得する
  3. システムが打刻確認ダイアログを表示する
     「9:00 に出勤を記録しますか？」
  4. ユーザーが「OK」を押す
  5. システムが打刻データをデータベースに保存する
  6. システムが「出勤を記録しました」のメッセージを表示する
  7. トップ画面に戻り、出勤時刻が表示される

【代替フロー】
  3a. 同日に既に出勤打刻がある場合
      → システムが「本日は既に出勤済みです」と表示する
      → 基本フローを終了する

  5a. データベースへの保存に失敗した場合
      → システムが「打刻の記録に失敗しました。再度お試しください」
        と表示する
      → ステップ1に戻る

【ビジネスルール】
  ・打刻時刻は15分単位に丸める（切り捨て）
  ・コアタイム（10:00）より遅い出勤は「遅刻」フラグを付与する
  ・1日に出勤打刻は1回のみ許可する
```

## 2.4 非機能要件の定義

### 非機能要件とは

非機能要件（Non-Functional Requirements）は、**システムの品質特性**を定義するものです。「何をするか」ではなく「どの程度のレベルで実現するか」を定めます。

### 非機能要件の分類

IPA（情報処理推進機構）の「非機能要求グレード」に基づく6つの分類があります。

```
【非機能要件の6分類】

┌──────────────────────────────────────────────┐
│                非機能要件                      │
├──────────┬──────────┬──────────┬─────────────┤
│ 可用性    │ 性能     │ 拡張性    │ 運用・保守性 │
│          │          │          │             │
│ 稼働率    │ 応答時間  │ 利用者増加│ 監視        │
│ 障害復旧  │ スループ │ データ増加│ バックアップ │
│ 冗長化    │ ット     │ 機能追加  │ ログ管理    │
├──────────┼──────────┴──────────┴─────────────┤
│ セキュリティ│ 移行性                            │
│           │                                   │
│ 認証・認可 │ データ移行                         │
│ 暗号化    │ システム切替                        │
│ 監査ログ  │ 並行運用                            │
└──────────┴───────────────────────────────────┘
```

### 非機能要件の記述例

```
【可用性要件】

| 項目         | 要件                              |
|-------------|----------------------------------|
| 稼働率       | 月間99.5%以上（月間ダウンタイム3.6時間以内） |
| サービス時間 | 24時間365日（計画停止除く）          |
| 計画停止     | 月1回、日曜深夜2:00〜5:00（最大3時間）  |
| 障害復旧時間 | RTO（目標復旧時間）4時間以内          |
| データ復旧   | RPO（目標復旧時点）1時間以内          |

【性能要件】

| 項目             | 要件                          |
|-----------------|------------------------------|
| 画面応答時間     | 3秒以内（95パーセンタイル）      |
| バッチ処理時間   | 月次集計処理：30分以内           |
| 同時接続数       | 最大300ユーザー                 |
| スループット     | 100リクエスト/秒               |
| データ量         | 初期10万件、年間増加12万件       |

【セキュリティ要件】

| 項目         | 要件                                  |
|-------------|--------------------------------------|
| 認証方式     | ID/パスワード認証 + 二要素認証（任意）   |
| パスワード   | 8文字以上、英大小文字・数字・記号を含む   |
| 通信暗号化   | HTTPS（TLS 1.2以上）                   |
| データ暗号化 | 個人情報はAES-256で暗号化して保存        |
| 監査ログ     | ログイン・データ変更・権限変更を記録      |
| セッション   | 30分操作なしでタイムアウト               |
```

> **実務でのポイント**: 非機能要件は後から追加すると大きな手戻りが発生します。特に性能要件とセキュリティ要件は、アーキテクチャ全体に影響するため、要件定義の段階で確定させることが重要です。

## 2.5 業務フロー図

### 業務フロー図とは

業務フロー図は、業務の流れをスイムレーン（担当者ごとのレーン）で表現した図です。As-Is（現状）とTo-Be（あるべき姿）の両方を描き、改善点を明確にします。

```
【業務フロー図の例：有給休暇申請（To-Be）】

  社員           上長            人事部          システム
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
  │               │              │              │
  │ 申請画面入力   │              │              │
  ├──────────────→│              │              │
  │               │              │       ┌──────┤
  │               │              │       │保存  │
  │               │              │       └──────┤
  │               │              │              │
  │          承認メール           │              │
  │         ┌─────┤              │              │
  │         │通知  │              │              │
  │         └─────┤              │              │
  │               │              │              │
  │           ◇判断              │              │
  │          ╱   ╲              │              │
  │        承認   差戻し          │              │
  │         │      │             │              │
  │         │      ├────→ 社員に差戻し通知       │
  │         │      │             │              │
  │         ├──────────→│         │              │
  │         │         │ 有給残日数│              │
  │         │         │ 更新     │              │
  │         │         │         ├──────→│更新  │
  │         │         │         │       │      │
  │    完了通知        │         │       │      │
  │←────────┤         │         │       │      │
  │         │         │         │       │      │
  ▼         ▼         ▼         ▼       ▼      ▼
```

### 業務ルールの定義

業務フローに付随するビジネスルールを明文化します。

```
【業務ルール一覧（例：有給休暇申請）】

| ルールID | ルール名           | 内容                                    |
|---------|--------------------|-----------------------------------------|
| BR-001  | 申請期限           | 有給休暇は取得希望日の3営業日前までに申請  |
| BR-002  | 承認権限           | 直属の上長が承認権限を持つ                |
| BR-003  | 連続取得制限       | 5日以上の連続取得は部長承認が必要          |
| BR-004  | 有給残日数チェック  | 残日数が0の場合、申請を受け付けない        |
| BR-005  | 年間取得義務       | 年間5日以上の取得が必要（労基法準拠）      |
| BR-006  | 半休対応           | 午前半休（AM）と午後半休（PM）を選択可能   |
```

## 2.6 要件の優先順位付け

### MoSCoW法

要件の優先度を4段階で分類する方法です。

```
【MoSCoW法】

┌──────────────────────────────────────────────────┐
│ Must have（必須）                                  │
│  なければシステムとして成立しない要件                │
│  例：出退勤打刻、勤怠集計、給与システム連携         │
├──────────────────────────────────────────────────┤
│ Should have（重要）                                │
│  なくても最低限は動くが、強く求められる要件          │
│  例：ダッシュボード、レポート出力                   │
├──────────────────────────────────────────────────┤
│ Could have（あれば良い）                           │
│  余裕があれば対応したい要件                         │
│  例：アラート通知、スマホアプリ対応                 │
├──────────────────────────────────────────────────┤
│ Won't have（今回は対応しない）                      │
│  今回のスコープからは除外する要件                   │
│  例：シフト管理、人事評価連携                       │
└──────────────────────────────────────────────────┘
```

### 要件のトレーサビリティマトリクス

要件と設計書・テストの対応関係を管理する表です。

```
【要件トレーサビリティマトリクス（RTM）】

| 要件ID  | 要件名       | 優先度 | 基本設計  | 詳細設計  | テストケース | 状態   |
|--------|-------------|-------|----------|----------|------------|--------|
| REQ-001| 出退勤打刻   | Must  | BD-001   | DD-001   | TC-001〜005| 承認済 |
| REQ-002| 打刻修正     | Must  | BD-002   | DD-002   | TC-006〜008| 承認済 |
| REQ-003| 有給休暇申請 | Must  | BD-003   | DD-003   | TC-009〜015| レビュー中|
| REQ-004| 残業申請     | Must  | BD-004   | DD-004   | TC-016〜020| 未着手 |
| REQ-005| 勤怠集計     | Must  | BD-005   | DD-005   | TC-021〜025| 未着手 |
| REQ-006| レポート出力 | Should| BD-006   | DD-006   | TC-026〜030| 未着手 |
| REQ-007| ダッシュボード| Should| BD-007   | DD-007   | TC-031〜035| 未着手 |
| REQ-008| アラート通知 | Could | -        | -        | -          | 保留   |
```

## 2.7 要件定義の合意形成

### レビューと承認プロセス

```
【要件定義書の承認フロー】

  作成者         レビュアー       承認者（顧客）
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
  │               │              │
  │ 要件定義書    │              │
  │ ドラフト作成   │              │
  ├──────────────→│              │
  │               │ レビュー      │
  │               │ 指摘事項      │
  │←──────────────┤              │
  │               │              │
  │ 修正・反映     │              │
  ├──────────────→│              │
  │               │ 再レビュー   │
  │               │ OK           │
  │               ├─────────────→│
  │               │              │ 最終確認
  │               │              │ 承認
  │               │←─────────────┤
  │←──────────────┤              │
  │               │              │
  │  ★ 要件確定（ベースライン化）  │
  │               │              │
```

### よくある要件定義の失敗パターン

| パターン | 問題 | 対策 |
|---|---|---|
| **顧客が要件を決められない** | いつまでも要件が確定しない | 選択肢を提示し、期限を設けて合意する |
| **暗黙の要件** | 「当然できると思っていた」 | チェックリストで網羅的に確認する |
| **スコープクリープ** | 後から要件が際限なく追加される | 変更管理プロセスを設ける |
| **技術用語の壁** | 顧客と技術者で認識がズレる | 用語定義を作り、平易な言葉で説明する |
| **非機能要件の欠落** | 性能・セキュリティが未定義 | 非機能要件チェックリストを使う |

```
│     → 基本フロー・代替フロー・ビジネスルールを網羅する         │
│                                                             │
│  3. 非機能要件は6分類（可用性・性能・拡張性・                  │
│     運用保守性・セキュリティ・移行性）で整理する               │
│     → 後から追加すると手戻りが大きいため、早期に確定する       │
│                                                             │
│  4. スコープ外の定義がスコープクリープを防止する               │
│     → 「対応しない」と明記することが重要                      │
│                                                             │
│  5. MoSCoW法で要件の優先順位を付け、                         │
│     トレーサビリティマトリクスで追跡可能にする                 │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

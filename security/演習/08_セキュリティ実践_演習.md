# 第8章：セキュリティ実践 演習問題

## 基本問題

### 問題8-1：脆弱性診断の種類
以下の脆弱性診断手法を説明してください。

1. SAST（静的解析）
2. DAST（動的解析）
3. プラットフォーム診断
4. Webアプリケーション診断

<details>
<summary>解答</summary>

1. **SAST（Static Application Security Testing / 静的解析）**：ソースコードを実行せずに解析し、脆弱性を発見する手法。開発段階で早期に脆弱性を発見できる。バッファオーバーフロー、ハードコードされた認証情報等を検出

2. **DAST（Dynamic Application Security Testing / 動的解析）**：実行中のアプリケーションに対して外部から攻撃的なリクエストを送信し、脆弱性を発見する手法。実際の動作環境での脆弱性を発見できる

3. **プラットフォーム診断**：OS、ネットワーク機器、ミドルウェア等のインフラレベルの脆弱性を検査する。既知の脆弱性（CVE）の有無、不要なサービスの公開、設定不備等を確認

4. **Webアプリケーション診断**：Webアプリケーション固有の脆弱性（SQLインジェクション、XSS、CSRF等）を検査する。自動ツール診断と手動診断を組み合わせる

</details>

### 問題8-2：脆弱性診断とペネトレーションテスト
脆弱性診断とペネトレーションテストの違いを3つの観点から比較してください。

<details>
<summary>解答</summary>

| 観点 | 脆弱性診断 | ペネトレーションテスト |
|------|-----------|---------------------|
| **目的** | 脆弱性の網羅的な発見と一覧化 | 実際に侵入が可能かどうかの検証 |
| **手法** | スキャナツール + 手動検査で広範囲を検査 | 攻撃者と同じ手法で特定の目標に対して侵入を試行 |
| **成果物** | 脆弱性の一覧と各脆弱性のリスク評価・対策案 | 侵入経路の実証、影響範囲の検証、攻撃シナリオの報告 |

</details>

### 問題8-3：CVSS
CVSSスコアの深刻度レベルを答えてください。

| スコア範囲 | 深刻度レベル |
|-----------|-------------|
| 9.0 - 10.0 | ? |
| 7.0 - 8.9 | ? |
| 4.0 - 6.9 | ? |
| 0.1 - 3.9 | ? |

<details>
<summary>解答</summary>

| スコア範囲 | 深刻度レベル | 対応の目安 |
|-----------|-------------|-----------|
| 9.0 - 10.0 | **緊急（Critical）** | 即時対応が必要 |
| 7.0 - 8.9 | **重要（High）** | 速やかに対応 |
| 4.0 - 6.9 | **警告（Medium）** | 計画的に対応 |
| 0.1 - 3.9 | **注意（Low）** | リスク受容も検討 |

</details>

### 問題8-4：OWASP Top 10
OWASP Top 10（2021年版）の上位3つのカテゴリを挙げ、それぞれの概要と主な対策を述べてください。

<details>
<summary>解答</summary>

| 順位 | カテゴリ | 概要 | 主な対策 |
|------|---------|------|---------|
| **A01** | アクセス制御の不備 | 認可チェックの欠如により、権限のない操作やデータへのアクセスが可能 | 全APIで認可チェック実施、デフォルトDeny、IDOR対策 |
| **A02** | 暗号化の失敗 | 機密データの平文通信・保存、弱い暗号アルゴリズムの使用 | TLS必須化、パスワードはbcrypt等で保存、AES-256以上の使用 |
| **A03** | インジェクション | SQLインジェクション、XSS、コマンドインジェクション等 | パラメータ化クエリ、出力エスケープ、入力検証 |

</details>

## 応用問題

### 問題8-5：セキュアコーディング
以下のコードにあるセキュリティ上の問題を全て指摘し、修正方針を述べてください。

```python
import os
import hashlib

# ユーザ登録処理
def register_user(username, password, email):
    # パスワードをハッシュ化して保存
    password_hash = hashlib.md5(password.encode()).hexdigest()

    # データベースに保存
    query = f"INSERT INTO users (name, pass, email) VALUES ('{username}', '{password_hash}', '{email}')"
    db.execute(query)

    # 確認メール送信
    os.system(f"sendmail {email} 'Registration complete'")

    return {"status": "ok", "password_hash": password_hash}
```

<details>
<summary>解答</summary>

**問題1：MD5によるパスワードハッシュ化**
- MD5は暗号学的に破られており、レインボーテーブル攻撃に脆弱
- 修正：bcrypt、Argon2等のパスワード専用ハッシュ関数を使用し、ソルトを自動付加する
```python
import bcrypt
password_hash = bcrypt.hashpw(password.encode(), bcrypt.gensalt())
```

**問題2：SQLインジェクション**
- ユーザ入力を文字列結合でSQL文に組み込んでいる
- 修正：パラメータ化クエリを使用する
```python
query = "INSERT INTO users (name, pass, email) VALUES (?, ?, ?)"
db.execute(query, (username, password_hash, email))
```

**問題3：コマンドインジェクション**
- `os.system()` にユーザ入力（email）を直接渡している。email に `; rm -rf /` 等を入力されると任意のコマンドが実行される
- 修正：`subprocess` をshell=Falseで使用するか、メール送信専用ライブラリを使用する

**問題4：レスポンスにパスワードハッシュを含めている**
- パスワードハッシュをクライアントに返すべきではない
- 修正：レスポンスからpassword_hashを除去する

**問題5：入力検証の欠如**
- username、email等の入力値が検証されていない
- 修正：メールアドレスの形式検証、ユーザ名の文字種・長さの検証を追加する

</details>

### 問題8-6：DevSecOps
以下の問いに答えてください。

(1) DevSecOpsとは何ですか。従来の開発方法との違いを説明してください。
(2) 「シフトレフト」とは何ですか。なぜ重要ですか。
(3) 開発ライフサイクルの各フェーズで実施すべきセキュリティ活動を挙げてください。

<details>
<summary>解答</summary>

**(1) DevSecOps**
DevSecOpsは、開発（Dev）、セキュリティ（Sec）、運用（Ops）を統合し、開発ライフサイクルの全段階にセキュリティを組み込むアプローチです。

従来の開発では、セキュリティチェックはリリース前に行われ、多数の脆弱性が発見されて手戻りが発生していました。DevSecOpsでは、開発の初期段階からセキュリティを考慮し、自動化されたツールでCI/CDパイプラインにセキュリティチェックを統合します。

**(2) シフトレフト**
セキュリティ活動を開発ライフサイクルの早い段階（左側）に移すことです。重要な理由は、脆弱性の発見が遅いほど修正コストが指数関数的に増加するためです。設計段階で発見・修正するコストは、本番環境で発見した場合の1/100とも言われています。

**(3) 各フェーズのセキュリティ活動**

| フェーズ | セキュリティ活動 |
|---------|----------------|
| 要件定義 | セキュリティ要件の定義、脅威モデリング |
| 設計 | セキュアデザインレビュー、アーキテクチャレビュー |
| 実装 | セキュアコーディング、SAST、コードレビュー |
| テスト | DAST、ペネトレーションテスト |
| リリース | セキュリティ設定の検証、脆弱性スキャン |
| 運用 | ログ監視、インシデント対応、パッチ管理 |

</details>

### 問題8-7：脅威モデリング
STRIDEモデルの6つの脅威カテゴリを挙げ、それぞれの対策を答えてください。

<details>
<summary>解答</summary>

| カテゴリ | 脅威 | 対策 |
|---------|------|------|
| **S: Spoofing（なりすまし）** | 他人になりすましてシステムにアクセス | 認証の強化（MFA等） |
| **T: Tampering（改ざん）** | データの不正な変更 | ハッシュによる完全性検証、デジタル署名 |
| **R: Repudiation（否認）** | 行為の否定 | 監査ログの記録、デジタル署名 |
| **I: Information Disclosure（情報漏洩）** | 機密情報の不正な取得 | 暗号化、アクセス制御 |
| **D: Denial of Service（サービス拒否）** | サービスの妨害 | 冗長化、レート制限、DDoS対策 |
| **E: Elevation of Privilege（権限昇格）** | 本来持たない権限の取得 | 最小権限の原則、認可チェック |

</details>

## チャレンジ問題

### 問題8-8：総合セキュリティ設計
新規Webアプリケーションの開発プロジェクトにおいて、セキュリティを確保するための計画を策定してください。

要件：
- ユーザ登録・ログイン機能あり
- 個人情報（氏名、住所、クレジットカード情報）を扱う
- REST APIでフロントエンドとバックエンドが分離

(1) 開発の各フェーズで実施すべきセキュリティ活動を挙げてください。
(2) 実装時に特に注意すべきセキュリティ対策を5つ挙げてください。
(3) リリース前に実施すべきセキュリティテストの種類と内容を述べてください。

<details>
<summary>解答例</summary>

**(1) 各フェーズのセキュリティ活動**

- **要件定義**：セキュリティ要件の定義（パスワードポリシー、暗号化要件等）、個人情報保護法・PCI DSSへの対応要件の確認
- **設計**：脅威モデリング（STRIDE）、認証・認可のアーキテクチャ設計、データフロー図の作成とリスク箇所の特定
- **実装**：セキュアコーディングガイドラインの遵守、SAST（静的解析）をCI/CDに統合、コードレビューでのセキュリティチェック
- **テスト**：DAST（動的解析）、ペネトレーションテスト
- **リリース**：セキュリティヘッダの設定確認、本番環境の設定レビュー
- **運用**：脆弱性情報の監視とパッチ適用、ログ監視、定期的な脆弱性診断

**(2) 実装時に特に注意すべき対策**

1. **パラメータ化クエリ**：全てのSQL文でプレースホルダを使用し、SQLインジェクションを防止
2. **出力エスケープ**：HTMLへの出力時にエスケープ処理を行い、XSSを防止
3. **パスワードの安全な保存**：bcrypt/Argon2でハッシュ化、ソルト付き
4. **クレジットカード情報の扱い**：PCI DSS準拠。自社で保存せず決済代行サービスを利用することを強く推奨
5. **全APIで認証・認可チェック**：JWTの検証、リソースへのアクセス権限チェック（IDOR対策）

**(3) リリース前のセキュリティテスト**

| テスト種類 | 内容 |
|-----------|------|
| **SAST（静的解析）** | ソースコードの脆弱性スキャン。ハードコードされた認証情報、安全でない関数の使用等を検出 |
| **DAST（動的解析）** | 実行中のアプリケーションに対してSQLインジェクション、XSS等の攻撃パターンを自動送信 |
| **SCA（ソフトウェア構成分析）** | 使用しているオープンソースライブラリに既知の脆弱性がないかチェック |
| **ペネトレーションテスト** | セキュリティ専門家による手動の侵入テスト。ビジネスロジックの脆弱性も検証 |
| **セキュリティヘッダ/設定の確認** | HSTS、CSP、X-Frame-Options等のセキュリティヘッダが正しく設定されているか確認 |

</details>

### 問題8-9：ペネトレーションテストのシナリオ
ある企業のWebアプリケーションに対するペネトレーションテストを計画する場合、以下の問いに答えてください。

(1) テスト前に合意すべき事項を5つ挙げてください。
(2) ブラックボックステストとホワイトボックステストの違いを述べ、それぞれのメリットを挙げてください。
(3) テスト結果の報告書に含めるべき内容を挙げてください。

<details>
<summary>解答</summary>

**(1) テスト前に合意すべき事項**
1. **テスト範囲（スコープ）**：対象システム、URL、IPアドレス範囲
2. **テスト期間**：開始日時と終了日時
3. **禁止事項**：DoS攻撃の禁止、データの削除・変更の禁止等
4. **緊急連絡先**：重大な脆弱性発見時やシステム障害時の連絡先
5. **テスト手法の合意**：使用するツール、攻撃手法の範囲

**(2)**
| テスト種類 | 事前情報 | メリット |
|-----------|---------|---------|
| **ブラックボックス** | なし | 実際の外部攻撃者に近い視点で検証。現実的な脅威の評価 |
| **ホワイトボックス** | 全て（ソースコード、構成情報等） | 効率的にテストでき、網羅性が高い。内部構造の問題も発見可能 |

**(3) 報告書に含めるべき内容**
- **エグゼクティブサマリー**：経営層向けの概要（全体のリスクレベル）
- **テスト条件**：実施期間、スコープ、使用ツール、テスト手法
- **発見事項一覧**：各脆弱性の詳細、CVSSスコア、証拠（スクリーンショット等）
- **攻撃シナリオ**：実際に成功した侵入経路と影響範囲の説明
- **リスク評価**：各脆弱性のビジネスへの影響度
- **推奨対策**：優先度付きの具体的な修正方法

</details>

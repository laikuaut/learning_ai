## 問題1：文法エラーの修正（基本）

以下のnginx設定には文法エラーが3つあります。すべて見つけて修正してください。

```nginx
server {
    listen 80
    server_name example.com;

    location / {
        root /var/www/html
        index index.html;
    }

    location /api {
        proxy_pass http://localhost:3000;
}
```

<details>
<summary>ヒント</summary>

nginx の文法ルール:
1. 単純ディレクティブは必ずセミコロン `;` で終わる
2. ブロックディレクティブは `{` と `}` の対応が必要

</details>

<details>
<summary>解答例</summary>

```nginx
server {
    listen 80;                          # ← セミコロン追加
    server_name example.com;

    location / {
        root /var/www/html;             # ← セミコロン追加
        index index.html;
    }

    location /api {
        proxy_pass http://localhost:3000;
    }
}                                       # ← 閉じ波括弧追加
```

エラー一覧:
1. `listen 80` → `listen 80;` （セミコロン欠落）
2. `root /var/www/html` → `root /var/www/html;` （セミコロン欠落）
3. server ブロックの閉じ `}` が欠落

</details>

## 問題2：コンテキストの理解（基本）

以下の各ディレクティブが配置されるべきコンテキストを答えてください。

1. `worker_processes auto;`
2. `worker_connections 1024;`
3. `listen 80;`
4. `proxy_pass http://localhost:3000;`
5. `gzip on;`
6. `server_name example.com;`

<details>
<summary>解答例</summary>

```
1. worker_processes auto;      → main コンテキスト
   ワーカープロセス数はグローバル設定

2. worker_connections 1024;    → events コンテキスト
   接続処理に関する設定

3. listen 80;                  → server コンテキスト
   リスンポートは仮想サーバーごとに設定

4. proxy_pass http://...;      → location コンテキスト
   URIパスごとのプロキシ先はlocationで指定

5. gzip on;                    → http コンテキスト
   （server や location でも設定可能だが、通常は http で共通設定）

6. server_name example.com;    → server コンテキスト
   ドメイン名は仮想サーバーごとに設定
```

</details>

## 問題3：locationのマッチング（基本）

以下のlocation設定に対して、各リクエストURIがどのlocationにマッチするか答えてください。

```nginx
server {
    location = / { A }
    location / { B }
    location /images/ { C }
    location ^~ /images/icons/ { D }
    location ~* \.(png|jpg)$ { E }
}
```

1. `GET /`
2. `GET /about`
3. `GET /images/photo.jpg`
4. `GET /images/icons/home.png`
5. `GET /docs/manual.png`

<details>
<summary>ヒント</summary>

優先順位: `=`（完全一致）> `^~`（優先プレフィックス）> `~`/`~*`（正規表現）> 通常プレフィックス（最長一致）

</details>

<details>
<summary>解答例</summary>

```
1. GET /           → A（完全一致）
   = / が完全に一致するため、最優先で A が選ばれる。

2. GET /about      → B（プレフィックスマッチ）
   / にプレフィックスマッチする B が選ばれる。
   = / は完全一致なので /about にはマッチしない。

3. GET /images/photo.jpg → E（正規表現マッチ）
   プレフィックスでは C（/images/）が最長一致。
   しかし C は ^~ ではないため、正規表現もチェックされる。
   ~* \.(png|jpg)$ にマッチするので E が選ばれる。

4. GET /images/icons/home.png → D（優先プレフィックスマッチ）
   ^~ /images/icons/ が最長プレフィックスマッチで、
   ^~ 修飾子により正規表現チェックを行わず D に確定。

5. GET /docs/manual.png → E（正規表現マッチ）
   プレフィックスでは B（/）が最長一致。
   正規表現 ~* \.(png|jpg)$ にマッチするので E が選ばれる。
```

</details>

## 問題4：try_files の動作理解（基本）

以下の設定で、各リクエストに対してnginxがどのファイルを返すか答えてください。

```nginx
location / {
    root /var/www/html;
    try_files $uri $uri/ /index.html;
}
```

ファイルシステムの状態：
```
/var/www/html/
├── index.html
├── about.html
└── css/
    └── style.css
```

1. `GET /about.html`
2. `GET /css/style.css`
3. `GET /contact`
4. `GET /products/item1`

<details>
<summary>解答例</summary>

```
1. GET /about.html → /var/www/html/about.html を返す
   $uri = /about.html → ファイルが存在する → そのまま返す

2. GET /css/style.css → /var/www/html/css/style.css を返す
   $uri = /css/style.css → ファイルが存在する → そのまま返す

3. GET /contact → /var/www/html/index.html を返す
   $uri = /contact → ファイルなし
   $uri/ = /contact/ → ディレクトリなし
   /index.html → フォールバック先として index.html を返す

4. GET /products/item1 → /var/www/html/index.html を返す
   $uri = /products/item1 → ファイルなし
   $uri/ = /products/item1/ → ディレクトリなし
   /index.html → フォールバック先として index.html を返す

※ 3, 4 はSPA（React, Vue等）のルーティングで重要なパターン。
  クライアントサイドのルーターが /contact や /products/item1 を処理する。
```

</details>

## 問題5：設定ファイルの作成（応用）

以下の要件を満たすserver ブロックを作成してください。

**要件：**
- ドメイン: `mysite.example.com`
- ポート: 80
- ドキュメントルート: `/var/www/mysite`
- インデックスファイル: `index.html`
- 静的ファイル（.css, .js, .jpg, .png）は30日間キャッシュ
- `/api/` へのリクエストは `http://localhost:8080` にプロキシ
- それ以外のリクエストはSPA用に `index.html` を返す

<details>
<summary>ヒント</summary>

3つの location ブロックが必要です。
1. 静的ファイル用（正規表現マッチ `~*`）
2. API用（プレフィックスマッチ `/api/`）
3. SPA用（ルート `/` + `try_files`）

</details>

<details>
<summary>解答例</summary>

```nginx
server {
    listen 80;
    server_name mysite.example.com;

    root /var/www/mysite;
    index index.html;

    # 静的ファイル（30日キャッシュ）
    location ~* \.(css|js|jpg|png)$ {
        expires 30d;
        add_header Cache-Control "public, immutable";
    }

    # API へのプロキシ
    location /api/ {
        proxy_pass http://localhost:8080;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
    }

    # SPA 用フォールバック
    location / {
        try_files $uri $uri/ /index.html;
    }
}
```

</details>

## 問題6：設定ファイルの分割（応用）

以下の一つの大きな nginx.conf を、適切なファイルに分割してください。どのファイルにどの設定を入れるか、ファイル名とディレクトリ構成を示してください。

```nginx
user www-data;
worker_processes auto;

events {
    worker_connections 1024;
}

http {
    include mime.types;
    gzip on;
    gzip_types text/plain text/css application/json application/javascript;

    ssl_protocols TLSv1.2 TLSv1.3;
    ssl_prefer_server_ciphers on;

    server {
        listen 80;
        server_name site-a.com;
        root /var/www/site-a;
        index index.html;
        location / {
            try_files $uri $uri/ =404;
        }
    }

    server {
        listen 80;
        server_name site-b.com;
        root /var/www/site-b;
        index index.html;
        location / {
            try_files $uri $uri/ =404;
        }
    }
}
```

<details>
<summary>解答例</summary>

```
【ディレクトリ構成】

/etc/nginx/
├── nginx.conf
├── conf.d/
│   └── gzip.conf
├── snippets/
│   └── ssl-params.conf
└── sites-enabled/
    ├── site-a.com.conf
    └── site-b.com.conf


【nginx.conf（メイン設定のみ）】
```

```nginx
user www-data;
worker_processes auto;

events {
    worker_connections 1024;
}

http {
    include mime.types;
    include /etc/nginx/conf.d/*.conf;
    include /etc/nginx/sites-enabled/*.conf;
}
```

```
【/etc/nginx/conf.d/gzip.conf】
```

```nginx
gzip on;
gzip_types text/plain text/css application/json application/javascript;
```

```
【/etc/nginx/snippets/ssl-params.conf】
```

```nginx
ssl_protocols TLSv1.2 TLSv1.3;
ssl_prefer_server_ciphers on;
```

```
【/etc/nginx/sites-enabled/site-a.com.conf】
```

```nginx
server {
    listen 80;
    server_name site-a.com;
    root /var/www/site-a;
    index index.html;

    location / {
        try_files $uri $uri/ =404;
    }
}
```

```
【/etc/nginx/sites-enabled/site-b.com.conf】
```

```nginx
server {
    listen 80;
    server_name site-b.com;
    root /var/www/site-b;
    index index.html;

    location / {
        try_files $uri $uri/ =404;
    }
}
```

分割のメリット:
- 各サイトの設定を独立して管理できる
- 共通設定（gzip, SSL）を一箇所で管理
- サイトの有効/無効の切り替えが容易

</details>

## 問題7：総合問題（チャレンジ）

以下の要件を満たす完全なnginx設定を作成してください。

**要件：**
- メインサイト `www.example.com`（ポート80）
  - ドキュメントルート: `/var/www/main`
  - SPA対応（React アプリ）
- APIサーバー `api.example.com`（ポート80）
  - `/v1/` → `http://localhost:3001` にプロキシ
  - `/v2/` → `http://localhost:3002` にプロキシ
- 共通設定
  - gzip圧縮を有効化
  - アクセスログとエラーログを分離
  - favicon.ico と robots.txt のログを抑制

<details>
<summary>解答例</summary>

```nginx
# === /etc/nginx/sites-enabled/www.example.com.conf ===

server {
    listen 80;
    server_name www.example.com;

    root /var/www/main;
    index index.html;

    access_log /var/log/nginx/www.example.com.access.log;
    error_log  /var/log/nginx/www.example.com.error.log;

    # gzip
    gzip on;
    gzip_types text/plain text/css application/json
               application/javascript text/xml;

    # favicon / robots.txt のログ抑制
    location = /favicon.ico {
        log_not_found off;
        access_log off;
    }
    location = /robots.txt {
        log_not_found off;
        access_log off;
    }

    # 静的ファイルのキャッシュ
    location ~* \.(css|js|jpg|png|gif|ico|svg|woff2?)$ {
        expires 30d;
        add_header Cache-Control "public, immutable";
    }

    # SPA 用フォールバック
    location / {
        try_files $uri $uri/ /index.html;
    }
}


# === /etc/nginx/sites-enabled/api.example.com.conf ===

server {
    listen 80;
    server_name api.example.com;

    access_log /var/log/nginx/api.example.com.access.log;
    error_log  /var/log/nginx/api.example.com.error.log;

    # 共通プロキシヘッダー
    proxy_set_header Host $host;
    proxy_set_header X-Real-IP $remote_addr;
    proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
    proxy_set_header X-Forwarded-Proto $scheme;

    # API v1
    location /v1/ {
        proxy_pass http://localhost:3001;
    }

    # API v2
    location /v2/ {
        proxy_pass http://localhost:3002;
    }

    # ルートへのアクセスはAPIドキュメントにリダイレクト
    location = / {
        return 301 /v2/docs;
    }
}
```

</details>

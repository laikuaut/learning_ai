# 第3章：ブランチとマージ

## この章のゴール

- ブランチ（branch）の概念を理解する
- ブランチの作成・切り替え・削除ができるようになる
- マージ（merge）の仕組みと種類を理解する
- コンフリクト（conflict）を解決できるようになる
- リベース（rebase）の基本を理解する
- 実務で使われるブランチ命名規則を知る

---

## 1. ブランチとは何か

ブランチ（branch）は、開発の「枝分かれ」を作る機能です。Git におけるブランチは、特定のコミットを指す軽量な「ポインタ（pointer）」にすぎません。これが Git のブランチが高速で軽量な理由です。

```
【ブランチのイメージ】

main ブランチ（本流）
  ○──○──○──○──○        ← 安定版のコード
              │
              └──○──○    ← feature/login ブランチ（機能開発）
                          新しいログイン機能を開発中

※ ○ はコミットを表します
```

### なぜブランチが必要か

ブランチがないと、以下のような問題が起きます。

- 開発中の不完全なコードが本番環境に影響する
- 複数人が同じコードを同時に変更して混乱する
- 新機能の開発中にバグ修正が必要になったとき対応できない

ブランチを使えば、メインのコードに影響を与えずに、安全に新機能の開発やバグ修正ができます。

### ブランチの正体：ポインタ

Git のブランチは、実際にはコミットを指すポインタです。新しいコミットを作ると、ブランチのポインタが自動的に前に進みます。

```
main が指すコミットの変化:

初期状態:
  main
   ↓
   ○

コミット後:
       main
        ↓
   ○───○

さらにコミット:
            main
             ↓
   ○───○───○
```

### HEAD とは

HEAD（ヘッド）は、「今自分がどのブランチにいるか」を示す特別なポインタです。

```
         main    feature/login
          ↓         ↓
  ○──○──○──○      ○──○
                      ↑
                     HEAD  ← 今は feature/login ブランチにいる
```

### ポイントまとめ

- ブランチはコミットを指す軽量なポインタです
- ブランチを使って安全に並行開発ができます
- HEAD は「今いるブランチ」を指すポインタです

## 2. ブランチの基本操作

### ブランチの一覧表示

```bash
# ローカルブランチの一覧
git branch
# 出力:
# * main                ← * がついているのが現在のブランチ
#   feature/login

# リモートブランチも含めて一覧
git branch -a
# 出力:
# * main
#   feature/login
#   remotes/origin/main
#   remotes/origin/feature/login

# 各ブランチの最新コミットも表示
git branch -v
```

### ブランチの作成

```bash
# ブランチを作成（切り替えはしない）
git branch feature/login

# ブランチを作成して切り替える（推奨）
git switch -c feature/login

# 旧式の書き方（git switch が推奨されるが、こちらも動作する）
git checkout -b feature/login
```

### ブランチの切り替え

```bash
# ブランチを切り替える（モダンな方法、推奨）
git switch main

# 旧式の書き方
git checkout main
```

> **よくある間違い：** `git checkout` はブランチの切り替え以外にもファイルの復元などにも使われ、意味が多義的です。Git 2.23 以降では、ブランチ切り替え専用の `git switch` と、ファイル復元専用の `git restore` が推奨されています。

```
git checkout の役割が分離:

  git checkout <branch>  →  git switch <branch>     （ブランチ切り替え）
  git checkout -- <file> →  git restore <file>       （ファイル復元）
  git checkout -b <new>  →  git switch -c <new>      （ブランチ作成＋切り替え）
```

### ブランチの削除

```bash
# マージ済みのブランチを削除
git branch -d feature/login

# マージされていないブランチを強制削除
git branch -D feature/login
```

### ブランチの名前変更

```bash
# 現在のブランチの名前を変更
git branch -m new-name

# 指定したブランチの名前を変更
git branch -m old-name new-name
```

### ポイントまとめ

- `git branch` でブランチの一覧を確認できます
- `git switch -c <name>` でブランチの作成と切り替えを同時に行えます
- `git switch` は `git checkout` のモダンな代替コマンドです
- `git branch -d` でマージ済みブランチを安全に削除できます

## 3. マージ（merge）

マージ（merge）は、あるブランチの変更を別のブランチに統合する操作です。

### Fast-forward マージ

分岐先のブランチが進んでいて、分岐元がそのまま（新しいコミットがない）の場合、ポインタを前に進めるだけで統合できます。これを Fast-forward（早送り）マージと呼びます。

```
【マージ前】
  main
   ↓
   ○──○──○
            \
             ○──○  ← feature
                ↑
               HEAD

【Fast-forward マージ後（git merge feature）】
                main
                 ↓
   ○──○──○──○──○
                ↑
               HEAD

※ main のポインタが feature の先端まで移動しただけ
※ マージコミットは作られない
```

```bash
# Fast-forward マージの実践
git switch main
git merge feature/login
# 出力: Fast-forward
```

### 3-way マージ

両方のブランチにコミットがある場合、Git は共通の祖先から両方の変更を統合する「3-way マージ」を行い、新しいマージコミット（merge commit）を作成します。

```
【マージ前】
            main
             ↓
   ○──○──○──○
         \
          ○──○  ← feature

【3-way マージ後（git merge feature）】
                  main
                   ↓
   ○──○──○──○──M    ← M はマージコミット
         \      /
          ○──○
              ↑
            feature

※ M（マージコミット）は2つの親を持つ特別なコミット
```

```bash
# 3-way マージの実践
git switch main
git merge feature/login
# エディタが開いてマージコミットのメッセージを入力
# デフォルト: "Merge branch 'feature/login'"
```

### --no-ff オプション

Fast-forward が可能な場合でも、マージコミットを作りたい場合は `--no-ff` オプションを使います。これにより、「このブランチでまとまった機能を開発した」という記録が残ります。

```bash
git merge --no-ff feature/login
```

```
【--no-ff マージ】
                  main
                   ↓
   ○──○──○──────M    ← 必ずマージコミットを作る
            \   /
             ○──○
```

### ポイントまとめ

- Fast-forward マージはポインタを進めるだけの軽量な操作です
- 3-way マージは両方の変更を統合し、マージコミットを作ります
- `--no-ff` でマージコミットを強制的に作り、ブランチの履歴を残せます

## 4. コンフリクト（conflict）の解決

2つのブランチで同じファイルの同じ箇所を変更した場合、Git は自動でマージできず、コンフリクト（競合）が発生します。

### コンフリクトが起きる例

```bash
# main ブランチでファイルを作成
git switch main
cat << 'EOF' > greeting.txt
Hello
Good morning
Goodbye
EOF
git add greeting.txt
git commit -m "greeting.txt を作成"

# feature ブランチを作成して変更
git switch -c feature/update-greeting
cat << 'EOF' > greeting.txt
Hello
Good evening
Goodbye
EOF
git add greeting.txt
git commit -m "Good morning を Good evening に変更"

# main ブランチに戻って別の変更
git switch main
cat << 'EOF' > greeting.txt
Hello
Good afternoon
Goodbye
EOF
git add greeting.txt
git commit -m "Good morning を Good afternoon に変更"

# マージを試みる → コンフリクト発生！
git merge feature/update-greeting
# 出力:
# Auto-merging greeting.txt
# CONFLICT (content): Merge conflict in greeting.txt
# Automatic merge failed; fix conflicts and then commit the result.
```

### コンフリクトの確認

```bash
git status
# 出力:
# Unmerged paths:
#   (use "git add <file>..." to mark resolution)
#         both modified:   greeting.txt
```

### コンフリクトマーカーの読み方

コンフリクトが発生したファイルを開くと、以下のようなマーカーが挿入されています。

```
Hello
<<<<<<< HEAD
Good afternoon
=======
Good evening
>>>>>>> feature/update-greeting
Goodbye
```

```
<<<<<<< HEAD            ← 現在のブランチ（main）の変更
Good afternoon          ← main での変更内容
=======                 ← 区切り線
Good evening            ← マージしようとしたブランチの変更内容
>>>>>>> feature/...     ← マージ元のブランチ名
```

### コンフリクトの解決手順

1. コンフリクトマーカーを含むファイルを開く
2. どちらの変更を採用するか（または両方を組み合わせるか）を決める
3. コンフリクトマーカー（`<<<<<<<`、`=======`、`>>>>>>>`）を削除する
4. 修正したファイルをステージングしてコミットする

```bash
# 1. ファイルを編集してコンフリクトを解決
cat << 'EOF' > greeting.txt
Hello
Good afternoon and Good evening
Goodbye
EOF

# 2. 解決したファイルをステージング
git add greeting.txt

# 3. マージコミットを作成
git commit -m "greeting.txt のコンフリクトを解決"
```

### マージの中止

コンフリクトが複雑で、マージ自体をやめたい場合は以下のコマンドで中止できます。

```bash
# マージを中止して、マージ前の状態に戻す
git merge --abort
```

> **よくある間違い：** コンフリクトが発生するとパニックになりがちですが、落ち着いて対処すれば大丈夫です。`git merge --abort` でいつでもマージ前の状態に戻せます。焦って適当にファイルを編集しないようにしましょう。

### ポイントまとめ

- コンフリクトは同じファイルの同じ箇所を変更したときに発生します
- コンフリクトマーカー（`<<<<<<<`、`=======`、`>>>>>>>`）を確認して手動で解決します
- 解決後は `git add` → `git commit` で完了です
- `git merge --abort` でマージを中止できます

## 5. リベース（rebase）の基本

リベース（rebase）は、ブランチの分岐点を変更する操作です。コミット履歴を直線的に保つために使います。

### rebase の仕組み

```
【rebase 前】
            main
             ↓
   ○──○──○──D
         \
          A──B  ← feature

【git rebase main 実行後（feature ブランチ上で）】
                  main
                   ↓
   ○──○──○──D
                \
                 A'──B'  ← feature（コミットが再作成される）

※ A', B' は内容は同じだが、新しいコミットとして再作成される
```

```bash
# feature ブランチで main の最新を取り込む
git switch feature/login
git rebase main
```

### merge vs rebase の比較

| 特徴 | merge | rebase |
|------|-------|--------|
| 履歴の見た目 | 分岐・合流が残る | 直線的になる |
| コミットの変更 | 既存コミットは変わらない | コミットが再作成される |
| コンフリクト解決 | 1回で済む | コミットごとに解決が必要な場合がある |
| 安全性 | 安全（非破壊的） | 注意が必要（コミットを書き換える） |
| 適した場面 | チームでの共有ブランチ | ローカルでの個人作業 |

```
【merge の履歴】            【rebase の履歴】
* Merge branch 'feature'    * feat: ログイン機能を追加
|\                           * feat: ユーザーモデルを作成
| * feat: ログイン機能追加     * fix: バリデーションエラーを修正
| * feat: ユーザーモデル作成   * 最初のコミット
|/
* fix: バリデーションエラー
* 最初のコミット
```

### rebase の注意点

> **よくある間違い：** すでにリモートに push したブランチを rebase してはいけません。他の開発者が混乱し、コミット履歴が壊れる原因になります。

```
【rebase のゴールデンルール】

  ✅ ローカルでまだ push していないブランチ → rebase OK
  ❌ すでにリモートに push したブランチ   → rebase NG（merge を使う）
```

### rebase 中のコンフリクト解決

```bash
# rebase 中にコンフリクトが発生した場合
git rebase main
# CONFLICT が発生

# 1. コンフリクトを解決してファイルを編集
# 2. 解決したファイルをステージング
git add greeting.txt

# 3. rebase を続行
git rebase --continue

# rebase を中止したい場合
git rebase --abort
```

### ポイントまとめ

- rebase はコミット履歴を直線的に整理できます
- merge は安全で非破壊的、rebase は履歴がきれいになる
- **push 済みのブランチを rebase しない**のが鉄則です
- チームでのルールを事前に決めておくことが重要です

## 6. ブランチ命名規則

実務では、ブランチ名に規則を設けることが一般的です。以下はよく使われる命名規則です。

### プレフィックスによる分類

| プレフィックス | 用途 | 例 |
|---------------|------|-----|
| `feature/` | 新機能の開発 | `feature/user-registration` |
| `bugfix/` | バグ修正 | `bugfix/login-validation` |
| `hotfix/` | 緊急のバグ修正 | `hotfix/security-patch` |
| `release/` | リリース準備 | `release/v2.0.0` |
| `docs/` | ドキュメント更新 | `docs/api-guide` |
| `refactor/` | リファクタリング | `refactor/auth-module` |

### 命名のベストプラクティス

```bash
# 良い例
feature/user-registration
bugfix/fix-login-error
hotfix/critical-security-patch

# 悪い例
new-feature          # 何の機能かわからない
fix                  # 何を修正したかわからない
taro-branch          # 個人名は避ける
test123              # 意味がない
```

### Git Flow のブランチ戦略

実務で広く使われているブランチ戦略の1つです。

```
main（本番環境）
  │
  ├── develop（開発の統合ブランチ）
  │     │
  │     ├── feature/login（機能開発）
  │     ├── feature/signup（機能開発）
  │     │
  │     └── release/v1.0（リリース準備）
  │
  └── hotfix/critical-bug（緊急修正）
```

| ブランチ | 役割 |
|----------|------|
| `main` | 本番環境にデプロイされるコード |
| `develop` | 開発中の統合ブランチ |
| `feature/*` | 個別の機能開発 |
| `release/*` | リリース準備（テスト・微調整） |
| `hotfix/*` | 本番の緊急バグ修正 |

### ポイントまとめ

- ブランチ名にプレフィックスを付けて目的を明確にしましょう
- 英語の小文字とハイフンで統一するのが一般的です
- チームでブランチ戦略を決めておくと、開発がスムーズになります

## 章のポイントまとめ

| コマンド | 用途 |
|----------|------|
| `git branch` | ブランチの一覧表示 |
| `git branch <name>` | ブランチの作成 |
| `git switch <name>` | ブランチの切り替え |
| `git switch -c <name>` | ブランチの作成と切り替え |
| `git branch -d <name>` | ブランチの削除 |
| `git merge <branch>` | 指定ブランチを現在のブランチに統合 |
| `git merge --no-ff` | マージコミットを必ず作成 |
| `git merge --abort` | マージを中止 |
| `git rebase <branch>` | ブランチの分岐点を変更 |
| `git rebase --abort` | リベースを中止 |

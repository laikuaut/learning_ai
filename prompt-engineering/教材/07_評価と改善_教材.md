# 第7章：評価と改善

## この章のゴール

- プロンプトの品質を評価する基準を理解する
- A/Bテストの方法とメトリクスを学ぶ
- イテレーション（反復改善）のプロセスを習得する
- プロンプトのデバッグ手法を学ぶ
- 評価の自動化とベンチマークの考え方を理解する

---

## 7.1 プロンプトの評価基準

### 5つの評価軸

プロンプトの出力品質を評価する際に使える基準です。

```
プロンプト評価の5軸：

┌─────────────┬──────────────────────────────────────┐
│ 正確性       │ 回答は事実に基づいて正しいか？          │
│ (Accuracy)   │ ハルシネーションはないか？              │
├─────────────┼──────────────────────────────────────┤
│ 関連性       │ 質問に対して的確に回答しているか？       │
│ (Relevance)  │ 余計な情報が混入していないか？          │
├─────────────┼──────────────────────────────────────┤
│ 完全性       │ 必要な情報が漏れなく含まれているか？     │
│ (Completeness)│ 重要な観点が欠けていないか？           │
├─────────────┼──────────────────────────────────────┤
│ 一貫性       │ 同じ質問に対して安定した回答が返るか？   │
│ (Consistency) │ フォーマットは指定通りか？             │
├─────────────┼──────────────────────────────────────┤
│ 有用性       │ ユーザーの目的を達成できる回答か？       │
│ (Usefulness)  │ 実際にアクションにつながるか？         │
└─────────────┴──────────────────────────────────────┘
```

### タスク別の重要度

| タスク | 最重要な評価軸 | 理由 |
|--------|-------------|------|
| コード生成 | 正確性 | 動作しないコードは無価値 |
| 要約 | 関連性 + 完全性 | 重要なポイントの漏れが致命的 |
| Q&Aチャットボット | 正確性 + 一貫性 | 誤情報の提供は信頼を損なう |
| ブレインストーミング | 有用性 | 実行可能なアイデアであることが重要 |
| データ分析 | 正確性 + 完全性 | 分析の正確さと網羅性が命 |

> **ポイントまとめ**: 評価基準はタスクによって優先度が変わります。「何のためのプロンプトか」に応じて重点を置く軸を決めましょう。

## 7.2 A/Bテスト

### プロンプトのA/Bテストとは

2つ以上のプロンプトのバリエーションを同じ入力で比較し、どちらがより良い出力を生成するか検証する手法です。

```
A/Bテストの流れ：

1. 仮説を立てる
   「ペルソナ設定を追加すると、回答の専門性が上がるのでは？」

2. バリエーションを作成する
   プロンプトA：ペルソナ設定なし
   プロンプトB：ペルソナ設定あり

3. 同じ入力セットで実行する
   テスト質問10〜20件を両方のプロンプトで実行

4. 評価基準に基づいてスコアリングする
   各回答を5段階で評価

5. 結果を比較して判断する
   スコアの平均値を比較し、有意な差があるか確認
```

### A/Bテストの記録テンプレート

```
■ テスト記録

テスト名：ペルソナ設定の効果検証
日付：2026/03/25
モデル：Claude 4 Sonnet
テスト質問数：10件

| # | 質問             | プロンプトA | プロンプトB | 勝者 |
|---|-----------------|-----------|-----------|------|
| 1 | Pythonのデコレータ | 3/5      | 4/5       | B    |
| 2 | REST API設計     | 3/5      | 5/5       | B    |
| 3 | Git rebaseの使い方| 4/5      | 4/5       | 引分 |
| ...                                                    |

結果：
  プロンプトA 平均スコア：3.2/5
  プロンプトB 平均スコア：4.1/5
  → プロンプトBを採用

学び：
  ペルソナ設定を追加すると、特に実務寄りの質問で
  回答の具体性が向上した。
```

### 効果的なテスト質問セットの作り方

| 種類 | 説明 | 例 |
|------|------|-----|
| 典型的な質問 | よくある問い合わせ | 「有給休暇の申請方法は？」 |
| 境界的な質問 | 判断が難しいケース | 「病気で休んだ場合は有給になる？」 |
| 想定外の質問 | 対応範囲外の質問 | 「転職活動の相談に乗って」 |
| 曖昧な質問 | 解釈が分かれる質問 | 「休みの取り方を教えて」 |

> **ポイントまとめ**: A/Bテストは小規模でも効果的です。10件程度のテスト質問で十分な知見が得られます。

## 7.3 イテレーション（反復改善）

### イテレーションのプロセス

プロンプトは一度で完成しません。段階的に改善していくプロセスが重要です。

```
プロンプト改善サイクル：

    ┌──→ 設計 ──→ テスト ──→ 評価 ──→ 改善 ──┐
    │                                          │
    └──────────────────────────────────────────┘
                  （繰り返す）

各ステップの内容：

1. 設計：プロンプトを作成（または改修）する
2. テスト：複数の入力で実行し、出力を収集する
3. 評価：評価基準に基づいてスコアリングする
4. 改善：問題点を特定し、プロンプトを修正する
```

### イテレーションの具体例

```
■ イテレーション1：初版
プロンプト：「このコードのバグを見つけてください」
問題：見つけたバグの説明が冗長で、修正案がない

■ イテレーション2：出力形式を追加
プロンプト：「このコードのバグを見つけ、以下の形式で報告してください。
- バグの内容（1文）
- 影響（1文）
- 修正案（コード付き）」
問題：軽微な問題も重大なバグも同列に報告される

■ イテレーション3：重要度の分類を追加
プロンプト：「このコードのバグを見つけ、重要度（高/中/低）と
ともに報告してください。重要度の高いものから順に記述し...」
問題：セキュリティ関連のバグを見逃すことがある

■ イテレーション4：ペルソナとチェックリストを追加
プロンプト：「あなたはセキュリティに精通したシニアエンジニアです。
以下のチェックリストに基づいてコードをレビューしてください：
□ SQLインジェクション  □ XSS  □ 認証・認可  □ ...」
結果：セキュリティ観点のレビュー品質が大幅に向上
```

### 改善のための質問リスト

プロンプトの出力に問題がある場合、以下の質問で原因を特定します。

```
出力が不正確 →
  □ 文脈（コンテキスト）は十分に提供したか？
  □ 事実確認のための情報源を指定したか？
  □ 「分からない場合は分からないと言う」と指示したか？

出力が的外れ →
  □ 指示は具体的か？曖昧な表現はないか？
  □ タスクの目的を明示したか？
  □ 出力形式を指定したか？

出力の品質がばらつく →
  □ temperatureの設定は適切か？（低くすると安定）
  □ Few-Shotの例は十分か？
  □ 制約条件は明確か？

出力が冗長 →
  □ 文字数や項目数の制限を指定したか？
  □ 「簡潔に」「要点のみ」と指示したか？
  □ 不要な情報の除外を指示したか？
```

> **ポイントまとめ**: プロンプトは1回で完成しません。テスト→評価→改善のサイクルを回し、段階的に品質を上げていきましょう。

## 7.4 プロンプトのデバッグ

### よくある問題と対処法

| 症状 | 原因候補 | 対処法 |
|------|---------|--------|
| 指示を無視する | 指示が長すぎる/曖昧 | 指示を簡潔にする、箇条書きにする |
| ハルシネーション | 知識にない情報を聞かれた | RAGで根拠を提供、「分からない」と言わせる |
| フォーマットが崩れる | 出力形式の指定が不十分 | Few-Shotで例を示す |
| 回答が浅い | 十分な深さを求めていない | 「詳細に」「根拠を示して」と追加 |
| 繰り返しが多い | コンテキストが長すぎる | 不要な情報を削減する |

### デバッグのテクニック

```
テクニック1：LLMに推論過程を見せてもらう
「回答を出す前に、まず自分の思考過程を説明してください。
 その後で最終的な回答を述べてください。」
→ どこで推論がおかしくなっているかが分かる

テクニック2：段階的に指示を追加する
最初はシンプルな指示から始め、問題が出た箇所だけ修正する。
一度に多くの変更を加えると、何が効いたか分からなくなる。

テクニック3：メタプロンプト
「以下のプロンプトの問題点を指摘してください」
とLLMに自分のプロンプトを評価させる。
```

## 7.5 評価の自動化

### 自動評価の考え方

大量のプロンプトを手動で評価するのは限界があります。自動評価の仕組みを構築することで、効率的な改善サイクルを実現できます。

```
自動評価のアプローチ：

1. ルールベース評価
   ・出力が指定フォーマット（JSON等）に沿っているか
   ・必須キーワードが含まれているか
   ・文字数制限を守っているか

2. LLMベース評価（LLM-as-Judge）
   ・別のLLMに出力を評価させる
   ・評価基準と評価スケールを明示
   ・人間の評価との相関を検証

3. 回帰テスト
   ・過去の質問と期待回答のセットを用意
   ・プロンプト変更後に全テストを再実行
   ・スコアが低下した項目を特定
```

### LLM-as-Judge のプロンプト例

```
あなたは品質評価担当です。
以下の質問と回答のペアを評価してください。

## 評価基準
1. 正確性（1-5）：事実として正しいか
2. 関連性（1-5）：質問に適切に回答しているか
3. 完全性（1-5）：必要な情報が漏れていないか

## 質問
「Pythonで辞書のキーが存在するか確認する方法を教えてください」

## 回答
（評価対象の回答）

## 評価
以下の形式で評価してください：
- 正確性：X/5 - 理由
- 関連性：X/5 - 理由
- 完全性：X/5 - 理由
- 総合スコア：X/5
- 改善提案：
```

> **ポイントまとめ**: 小規模なら手動評価、大規模ならLLM-as-Judgeや回帰テストで効率化しましょう。

## まとめ

```
評価と改善の重要ポイント：

✅ 評価基準は正確性・関連性・完全性・一貫性・有用性の5軸
✅ タスクに応じて重視する評価軸を決める
✅ A/Bテストで改善効果を定量的に検証する
✅ イテレーション（設計→テスト→評価→改善）を繰り返す
✅ 出力の問題は質問リストで原因を特定する
✅ LLM-as-Judgeで評価を効率化できる
✅ プロンプトは一度で完成しない。継続的な改善が重要
```

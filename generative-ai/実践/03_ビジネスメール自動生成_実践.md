# 実践課題03：ビジネスメール自動生成 ★2

> **難易度**: ★★☆☆☆（基礎）
> **前提知識**: 第2章（テキスト生成AIの活用）
> **課題の種類**: ミニプロジェクト
> **学習目標**: ビジネスシーンに応じた適切なメール文面を生成AIで作成するためのプロンプト設計力を身につける

---

## 課題の要件

1. 以下の5つのビジネスシナリオそれぞれに対して、メール生成用プロンプトを設計する
   - シナリオ1：商談後のお礼メール
   - シナリオ2：納期遅延のお詫びメール
   - シナリオ3：会議日程の調整依頼メール
   - シナリオ4：新サービスの案内・提案メール
   - シナリオ5：プロジェクト完了の報告メール
2. 各プロンプトは「テンプレート変数方式」で設計する（{宛先名}、{日付}などの変数を使い回せる形式）
3. 実際にAIで生成した結果を記録し、品質チェックリストで評価する
4. 各シナリオで「よくある失敗パターン」を1つ以上記載する
5. 最後に全シナリオ共通の「メール生成プロンプト設計のベストプラクティス」を5項目以上まとめる

---

## ステップガイド

<details>
<summary>ステップ1：シナリオの詳細を整理する</summary>

各シナリオについて、以下の情報を整理しましょう。

- **送信者の立場**: 自分はどんな役職・関係性か
- **受信者の情報**: 相手の役職、関係性、期待する反応
- **メールの目的**: 何を伝え、相手にどう行動してほしいか
- **トーン**: フォーマル度合い（最大フォーマル〜ややカジュアル）
- **含めるべき情報**: 日付、金額、固有名詞など

**よくある間違い**: シナリオの背景を十分に整理せずにプロンプトを書き始めること。事前の情報整理が出力品質を大きく左右します。

</details>

<details>
<summary>ステップ2：テンプレート変数方式のプロンプトを設計する</summary>

以下のような形式で、変数を使ったプロンプトを設計します。

```
あなたはビジネスメールの専門ライターです。
以下の情報に基づいて、{メール種別}を作成してください。

【送信者情報】
- 名前：{送信者名}
- 役職：{送信者役職}
- 会社名：{送信者会社名}

【受信者情報】
- 名前：{受信者名}
- 役職：{受信者役職}
- 関係性：{取引先/社内/初回連絡}

【メール要件】
- 目的：{目的を1文で}
- トーン：{フォーマル/セミフォーマル}
- 文字数目安：{200〜300文字}
- 必ず含める情報：{箇条書きで列挙}

【制約】
- 敬語の間違いがないこと
- 過度にへりくだらないこと
- 件名も生成すること
```

</details>

<details>
<summary>ステップ3：生成結果を品質チェックリストで評価する</summary>

以下のチェックリストで各生成結果を評価しましょう。

```
□ 件名が内容を適切に表しているか
□ 宛名の敬称が正しいか（様/部長 等）
□ 用件が冒頭3行以内で明確か
□ 敬語・丁寧語に誤りがないか
□ 具体的な日付・数字が正しく含まれているか
□ 結びの言葉が適切か
□ 署名欄があるか
□ 文字数が指定範囲内か
□ トーンが場面に合っているか
□ 次のアクション（CTA（Call to Action））が明確か
```

</details>

<details>
<summary>ステップ4：失敗パターンとベストプラクティスをまとめる</summary>

5つのシナリオを通して見えてきた以下のパターンを整理しましょう。

**失敗パターンの例：**
- 敬語が過剰で不自然（「させていただきます」の連発）
- 件名が長すぎる / 抽象的すぎる
- 用件が本文の後半にならないと分からない

**ベストプラクティスの観点：**
- プロンプトに含めるべき必須情報は何か
- トーンの調整で効果的だった指示は何か
- 出力品質を安定させるコツは何か

</details>

### シナリオ1：商談後のお礼メール

**プロンプト:**
```
あなたはビジネスメールの専門ライターです。
以下の情報に基づいて、商談後のお礼メールを作成してください。

【送信者】山田太郎（ABC株式会社 営業部 主任）
【受信者】田中一郎 部長（XYZ株式会社 購買部）
【状況】昨日（4月4日）に弊社にてAI導入に関する商談を実施
【目的】商談のお礼 + 次回打ち合わせ（4月15日）の確認
【トーン】フォーマル
【文字数】200〜300文字
【必須要素】商談日、議題、次回日程、添付資料への言及

件名も含めて生成してください。
敬語は正確に、ただし過度にへりくだらないでください。
```

**生成結果:**
```
件名：4月4日 AI導入に関するお打ち合わせの御礼

XYZ株式会社
購買部 田中一郎 部長

いつもお世話になっております。
ABC株式会社 営業部の山田太郎です。

昨日はご多忙の中、AI導入に関するお打ち合わせのお時間を
いただき、誠にありがとうございました。

ご質問いただいた導入コストと運用体制につきまして、
補足資料を添付いたしましたのでご確認ください。

次回のお打ち合わせは4月15日（火）14時にて
承っておりますが、ご都合に変更がございましたら
お知らせいただけますと幸いです。

引き続きどうぞよろしくお願いいたします。

──────────────
ABC株式会社 営業部
主任 山田太郎
TEL: 03-XXXX-XXXX
Email: yamada@abc.co.jp
──────────────
```

**品質チェック結果:**
- [x] 件名が内容を適切に表しているか → ○
- [x] 宛名の敬称が正しいか → ○
- [x] 用件が冒頭3行以内で明確か → ○
- [x] 敬語に誤りがないか → ○
- [x] 日付が正しく含まれているか → ○
- [x] トーンが場面に合っているか → ○

**よくある失敗パターン:**
「お忙しいところ恐縮ではございますが〜させていただければと存じます」のように、過剰な敬語が連続して読みにくくなるケース。プロンプトに「過度にへりくだらない」と指示することで防止できます。

### シナリオ2：納期遅延のお詫びメール

**プロンプト:**
```
あなたはビジネスメールの専門ライターです。
以下の情報に基づいて、納期遅延のお詫びメールを作成してください。

【送信者】鈴木花子（ABC株式会社 プロジェクト管理部 課長）
【受信者】佐藤様（XYZ株式会社 情報システム部 部長）
【状況】Webシステム開発プロジェクトの納品が当初予定（4月10日）
       から1週間遅延し、4月17日になる見込み
【遅延理由】テスト工程で重大なバグが発見され、修正に時間を要している
【目的】お詫び + 新納期の提示 + 再発防止策の提示
【トーン】フォーマル（謝罪の誠意を込める）
【文字数】250〜400文字

注意：
- 言い訳がましくならないこと
- 具体的な対策を示すこと
- 件名にも配慮すること
```

**よくある失敗パターン:**
遅延理由を長々と説明して言い訳がましくなるケース。プロンプトに「言い訳がましくならないこと」と制約を入れることが重要です。

### メール生成プロンプト設計のベストプラクティス

1. **送信者と受信者の関係性を明示する**: 上司宛、取引先宛、初回連絡かで敬語レベルが変わる
2. **メールの目的を1文で書く**: 「お礼 + 次回日程確認」のように簡潔に
3. **必須要素を箇条書きで列挙する**: 含めるべき固有名詞、日付、数字を明示
4. **トーンを具体的に指定する**: 「フォーマル」だけでなく「謝罪の誠意を込める」等の補足
5. **やってはいけないことも指定する**: 「言い訳がましくしない」「過度にへりくだらない」等の制約

### 改良版の特徴

改良版では以下を追加しています。

- **シナリオごとにバリエーション（2パターン）を生成**して比較
- **プロンプトをモジュール化**（共通部分と可変部分を分離）
- **生成AIの出力を人間がレビューする際のチェックフロー**を追加

---

#### モジュール化されたプロンプト構造

```
【ベースプロンプト（全シナリオ共通）】
あなたはビジネスコミュニケーションの専門家です。
日本のビジネスマナーに精通し、TPOに応じた適切なメールを
作成できます。以下の要件に基づいてメールを生成してください。

生成ルール：
- 件名は30文字以内で内容が一目でわかるものにする
- 宛名→挨拶→用件→詳細→結び→署名 の構成を守る
- 「させていただく」は1メール中2回まで
- 1文は60文字以内を目安にする
- 結びの文言はメールの種別に合わせて適切に選ぶ

【シナリオ別パラメータ（JSONライク形式）】
{
  "メール種別": "商談後のお礼",
  "送信者": {
    "名前": "山田太郎",
    "役職": "営業部 主任",
    "会社": "ABC株式会社"
  },
  "受信者": {
    "名前": "田中一郎",
    "役職": "購買部 部長",
    "会社": "XYZ株式会社"
  },
  "状況": "4月4日にAI導入に関する商談を実施",
  "目的": "お礼と次回打ち合わせ（4/15）の確認",
  "トーン": "フォーマル",
  "文字数": "200〜300",
  "必須要素": ["商談日", "議題", "次回日程", "添付資料への言及"],
  "NG表現": ["過度な謙譲", "曖昧な表現"]
}
```

この方式のメリット：
- 共通のベースプロンプトを使い回せるため、品質が安定する
- パラメータ部分だけを変更すれば新しいシナリオに即対応できる
- JSON形式のため、将来的にシステム化（API連携）しやすい

---

#### 生成結果のレビューフロー

```
生成結果
   │
   ▼
[自動チェック]
   ├─ 文字数が範囲内か？ ──→ NG: 再生成
   ├─ 必須要素が含まれるか？ → NG: 追加指示
   └─ NG表現がないか？ ────→ NG: 修正指示
   │
   ▼
[人間レビュー]
   ├─ 敬語は自然か？
   ├─ トーンは適切か？
   ├─ 事実関係に誤りはないか？ ← 最重要！
   └─ 送信して問題ないか？
   │
   ▼
送信 or 修正依頼
```

**よくある間違い**: AIの生成結果をそのまま送信すること。特に日付、金額、固有名詞などの事実情報は必ず人間が確認する必要があります。これを「ヒューマン・イン・ザ・ループ（Human in the Loop）」と呼びます。

---

### メール生成プロンプト設計のベストプラクティス（拡張版）

1. **送信者と受信者の関係性を明示する**
2. **メールの目的を1文で書く**
3. **必須要素を箇条書きで列挙する**
4. **トーンを具体的に指定する**
5. **やってはいけないことも指定する（NG表現リスト）**
6. **プロンプトをモジュール化する**（共通部分と可変部分を分離）
7. **生成結果は必ず人間がレビューする**（特に事実情報）
8. **バリエーションを2パターン生成して選ぶ**（1回の生成で満足しない）

**初心者向けとの違い**:
- プロンプトをモジュール化し、再利用性と保守性を向上
- JSON形式のパラメータで、システム連携を視野に入れた設計
- 生成結果のレビューフローを明示し、実務での運用を意識
- ヒューマン・イン・ザ・ループの概念を導入し、AI任せにしない運用を強調

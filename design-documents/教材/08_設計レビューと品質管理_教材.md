# 第8章：設計レビューと品質管理

## この章のゴール

- 設計レビューの目的と種類を理解する
- 設計レビューの進め方を習得する
- レビューチェックリストの活用方法を学ぶ
- よくある設計の問題点とその対策を把握する
- 設計ドキュメントの保守・運用方法を理解する

---

## 8.1 設計レビューとは

### 目的

設計レビューは、**設計書の品質を第三者の目で確認し、問題点を早期に発見・修正するプロセス**です。

### なぜ設計レビューが重要なのか

```
バグ修正コストの法則（1:10:100の法則）

  設計段階で発見    →  修正コスト:   1
  実装段階で発見    →  修正コスト:  10
  テスト段階で発見  →  修正コスト:  30
  リリース後に発見  →  修正コスト: 100
```

設計の問題を設計段階で見つけることが、最もコスト効率の良い品質向上手段です。

### 設計レビューの種類

| 種類 | 参加者 | 目的 | 形式 |
|------|--------|------|------|
| **セルフレビュー** | 設計者本人 | 自分で見直して問題を発見 | 個人作業 |
| **ピアレビュー** | 同僚のエンジニア | 技術的な問題を発見 | 1対1またはオンライン |
| **ウォークスルー** | 設計者 + チーム | 設計内容を説明し、疑問を解消 | 会議形式 |
| **インスペクション** | チーム + モデレーター | 体系的にチェックリストで検証 | 公式な会議 |
| **ステークホルダーレビュー** | 顧客・PM | ビジネス要件との整合性を確認 | 報告・承認形式 |

## 8.2 設計レビューの進め方

### レビュープロセス

```
1. 準備
   │  ・レビュー対象の設計書を事前配布
   │  ・レビューアはチェックリストに沿って事前確認
   │
2. レビュー実施
   │  ・設計者が設計内容を説明
   │  ・レビューアが指摘事項を挙げる
   │  ・指摘事項を記録する
   │
3. 指摘事項の分類
   │  ・重大度（高・中・低）で分類
   │  ・対応方針を決定
   │
4. 修正
   │  ・設計者が指摘事項に対応
   │
5. 確認
      ・修正内容をレビューアが確認
      ・必要に応じて再レビュー
```

### レビュー実施のポイント

**レビューアの心構え:**

| やるべきこと | やってはいけないこと |
|------------|-------------------|
| 設計書の内容を事前に読んでおく | 会議中に初めて読む |
| 具体的な問題点を指摘する | 「なんか気持ち悪い」で終わる |
| 代替案を提案する | 批判するだけで代案なし |
| 設計の良い点も伝える | 問題点だけを指摘する |
| 設計書の内容を対象にする | 設計者個人を攻撃する |

**設計者の心構え:**

| やるべきこと | やってはいけないこと |
|------------|-------------------|
| 指摘を素直に受け止める | 防御的になる |
| 設計の意図を説明する | 「言われた通りにします」 |
| 不明点は質問する | 曖昧なまま進める |
| 指摘事項を記録する | 記録せず忘れる |

## 8.3 設計レビューチェックリスト

### 共通チェックリスト

| No | チェック項目 | 確認内容 |
|----|------------|---------|
| C-01 | 目的の明確性 | ドキュメントの目的と対象読者が明記されているか |
| C-02 | 用語の統一 | 用語が統一されているか、用語集があるか |
| C-03 | 要件との整合性 | 要件定義書の全要件がカバーされているか |
| C-04 | 文書間の整合性 | 他の設計書と矛盾がないか |
| C-05 | 完全性 | 必要な情報に漏れがないか |
| C-06 | 曖昧さの排除 | 「適切に」「必要に応じて」等の曖昧な表現がないか |
| C-07 | 図表の正確性 | 図表の内容が本文と一致しているか |
| C-08 | 変更履歴 | バージョン、更新日、更新者が記録されているか |

### 基本設計レビューチェックリスト

| No | チェック項目 | 確認内容 |
|----|------------|---------|
| B-01 | システム構成 | 全コンポーネントが図示され、通信方式が明記されているか |
| B-02 | 技術選定 | 技術スタックと選定理由が記載されているか |
| B-03 | 画面設計 | 全画面のワイヤーフレームと画面遷移が定義されているか |
| B-04 | 機能網羅 | 要件定義の全機能が機能一覧に含まれているか |
| B-05 | 非機能要件 | 性能・セキュリティ・可用性の設計方針があるか |
| B-06 | 外部IF | 外部システムとの連携方式が定義されているか |
| B-07 | セキュリティ | 認証・認可・暗号化の方針が定義されているか |
| B-08 | 運用設計 | 監視・バックアップ・障害対応の方針があるか |

### 詳細設計レビューチェックリスト

| No | チェック項目 | 確認内容 |
|----|------------|---------|
| D-01 | クラス設計 | クラスの責務が明確で、単一責任の原則に沿っているか |
| D-02 | メソッド設計 | 入出力・例外・処理概要が定義されているか |
| D-03 | エラーハンドリング | 例外の種類と対処が網羅されているか |
| D-04 | ログ設計 | ログレベル・出力内容が定義されているか |
| D-05 | シーケンス図 | 主要な処理フローのシーケンス図があるか |
| D-06 | 状態遷移 | ステータスを持つエンティティの状態遷移が定義されているか |
| D-07 | 実装可能性 | この設計書を見れば実装できるレベルの詳細度があるか |

### DB設計レビューチェックリスト

| No | チェック項目 | 確認内容 |
|----|------------|---------|
| DB-01 | 正規化 | 第3正規形まで正規化されているか（非正規化は意図的か） |
| DB-02 | 命名規則 | テーブル名・カラム名が命名規則に沿っているか |
| DB-03 | データ型 | カラムのデータ型が適切か（VARCHAR長は十分か等） |
| DB-04 | NOT NULL | NULL許可の判断が適切か |
| DB-05 | インデックス | 検索パターンに基づくインデックスが設計されているか |
| DB-06 | 外部キー | テーブル間のリレーションシップが正しく定義されているか |
| DB-07 | 共通カラム | created_at, updated_at等の共通カラムがあるか |
| DB-08 | 拡張性 | 将来の要件変更に対応しやすい構造か |

### API設計レビューチェックリスト

| No | チェック項目 | 確認内容 |
|----|------------|---------|
| API-01 | RESTful | URL設計がRESTの原則に沿っているか |
| API-02 | 認証 | 認証・認可の仕組みが定義されているか |
| API-03 | リクエスト | パラメータの型・必須・バリデーションが定義されているか |
| API-04 | レスポンス | 正常系・異常系のレスポンス形式が定義されているか |
| API-05 | エラー | HTTPステータスコードが適切に使い分けられているか |
| API-06 | ページネーション | 一覧APIにページネーションがあるか |
| API-07 | バージョニング | APIバージョンの方針があるか |

## 8.4 よくある設計の問題点

### 要件定義でよくある問題

| 問題 | 例 | 対策 |
|------|-----|------|
| 曖昧な要件 | 「使いやすいシステム」 | 具体的な指標を定義する |
| 非機能要件の欠如 | 性能要件が未定義 | NFRチェックリストで漏れ確認 |
| 要件の矛盾 | 画面AとBで異なる仕様 | 要件のトレーサビリティマトリクス |
| スコープクリープ | 開発中に要件が膨張 | 変更管理プロセスの確立 |

### 設計でよくある問題

| 問題 | 例 | 対策 |
|------|-----|------|
| 過剰な設計 | 不要なデザインパターンの適用 | YAGNI原則（必要になるまで作らない） |
| 過少な設計 | エラーケースの考慮不足 | チェックリストでの網羅性確認 |
| 密結合 | モジュール間の依存が多すぎる | インターフェースの適切な定義 |
| 図と文書の不一致 | ER図と本文の記述が矛盾 | レビュー時の整合性チェック |
| 古い設計書 | 実装と設計書がずれている | 設計書も実装と同時に更新 |

### 設計の品質指標

| 原則 | 説明 | チェックポイント |
|------|------|----------------|
| **単一責任** | 1つのモジュールは1つの責務のみ | クラスの説明が「○○と△△を行う」になっていないか |
| **開放閉鎖** | 拡張に開き、修正に閉じる | 新機能追加時に既存コードの修正が必要か |
| **依存性逆転** | 上位モジュールが下位に依存しない | インターフェースを介して依存しているか |
| **DRY** | 同じ情報を2箇所に書かない | 設計書間でコピペされた記述がないか |
| **KISS** | シンプルに保つ | 必要以上に複雑な設計になっていないか |

## 8.5 指摘事項の管理

### 指摘事項の記録テンプレート

```markdown
## 設計レビュー 指摘事項一覧

| No | 対象 | 指摘内容 | 重大度 | 対応方針 | 担当 | 期限 | 状態 |
|----|------|---------|--------|---------|------|------|------|
| 1 | DB設計書 3.2節 | usersテーブルのemail にUNIQUE制約がない | 高 | UNIQUE制約を追加 | 山田 | 4/5 | 完了 |
| 2 | API設計書 P-001 | 商品一覧のページネーション上限が未定義 | 中 | per_page最大100を追加 | 佐藤 | 4/5 | 完了 |
| 3 | 画面設計書 SC-020 | カート画面のエラー表示位置が未定義 | 低 | フォーム上部に表示と追記 | 田中 | 4/8 | 対応中 |
| 4 | 詳細設計書 4.3節 | OrderServiceの同時注文の排他制御が未考慮 | 高 | 楽観的ロックを追加 | 山田 | 4/5 | 対応中 |
```

### 重大度の基準

| 重大度 | 基準 | 対応 |
|--------|------|------|
| **高** | 要件の未充足、セキュリティリスク、データ不整合の可能性 | リリース前に必ず修正 |
| **中** | 品質上の懸念、ベストプラクティスからの逸脱 | 可能な限り修正 |
| **低** | ドキュメントの誤字、表現の改善 | 次回更新時に修正 |

## 8.6 設計ドキュメントの保守

### ドキュメントの陳腐化を防ぐ

設計書が「作って終わり」になると、実装と乖離して価値がなくなります。

```markdown
## ドキュメント保守ルール

### 更新タイミング
- 機能追加・変更時に、該当する設計書を同時に更新する
- プルリクエストに設計書の更新も含める
- スプリントレビュー時にドキュメントの鮮度を確認する

### 責任者
- 各設計書にオーナーを設定する
- オーナーは設計書の正確性に責任を持つ

### レビュー
- 四半期に一度、全設計書の棚卸しを行う
- 実装との乖離がないか確認する
```

### ドキュメント管理のベストプラクティス

| プラクティス | 説明 |
|------------|------|
| **Gitで管理** | ソースコードと同じリポジトリで設計書を管理する |
| **Markdownで記述** | テキストベースで差分が見やすい |
| **PRに含める** | コード変更と設計書変更を同じPRで行う |
| **テンプレート化** | 設計書のテンプレートを用意し、品質を均一化する |
| **自動生成の活用** | OpenAPIからAPIドキュメントを自動生成する |
| **リンクの整備** | 設計書間の相互参照を整備する |

## 8.7 設計工程全体の振り返り

### 設計ドキュメントの全体像

本教材で学んだ設計ドキュメントの関係を整理します。

```
要件定義書（第2章）
  │  「何を作るか」を定義
  │
  ├──→ 基本設計書（第3章）
  │      │  「システム全体の構造」を定義
  │      │
  │      ├──→ 画面設計書（第6章）
  │      │      「ユーザーが触れるUI」を定義
  │      │
  │      ├──→ API設計書（第7章）
  │      │      「フロントとバックの契約」を定義
  │      │
  │      └──→ データベース設計書（第5章）
  │             「データの構造」を定義
  │
  └──→ 詳細設計書（第4章）
         「各モジュールの内部ロジック」を定義

  設計レビュー（第8章）
    全ドキュメントの品質を担保
```

### 各ドキュメントの作成順序

```
1. 要件定義書       ← 最初に作る（基盤）
2. システム構成図    ← 技術方針を決める
3. 画面設計書       ← ユーザー体験を先に設計
4. ER図・テーブル定義 ← データ構造を決める
5. API設計書        ← 画面とデータをつなぐ
6. 詳細設計書       ← 最後に内部ロジックを決める
7. 設計レビュー      ← 全体の品質を確認
```

> **実務でのポイント**: 実際のプロジェクトでは、これらすべてのドキュメントを作る必要はありません。プロジェクトの規模・複雑さ・チーム体制に応じて、必要なドキュメントを選択します。小規模プロジェクトではAPI設計書とDB設計書だけ、といった運用もよくあります。

## まとめ

| 概念 | ポイント |
|------|---------|
| 設計レビュー | 問題の早期発見が最もコスト効率が良い |
| レビューの種類 | セルフ→ピア→ウォークスルー→インスペクション |
| チェックリスト | 分野別のチェックリストで漏れなく確認する |
| 指摘事項管理 | 重大度・対応方針・担当・期限・状態を記録する |
| ドキュメント保守 | Gitで管理し、コード変更と同時に更新する |
| 全体像 | プロジェクトの規模に応じて必要なドキュメントを選択する |

### システム構成の見直しに関する次の記述を読んで、設問に答えよ。

A社は従業員500名の中堅製造業で、社内業務システムをオンプレミスで運用している。現在のシステム構成は以下のとおりである。

**【現行システム構成】**

```
インターネット
     │
  [FW(ファイアウォール)]
     │
  ┌──┴──────────────────┐
  │      DMZ            │
  │  [Webサーバ(1台)]    │
  │  [メールサーバ(1台)] │
  └──┬──────────────────┘
     │
  [FW(内部)]
     │
  ┌──┴──────────────────────────────────┐
  │         内部ネットワーク              │
  │  [業務APサーバ(1台)]                 │
  │  [DBサーバ(1台)]                     │
  │  [ファイルサーバ(1台)]               │
  │  [Active Directoryサーバ(1台)]       │
  │  [クライアントPC(500台)]             │
  └─────────────────────────────────────┘
```

**【運用上の課題】**

A社の情報システム部門は、以下の課題を認識している。

(1) 昨年、DBサーバのハードディスク障害でデータベースが8時間停止した。バックアップからの復旧に時間がかかり、その間の受発注業務が完全に停止した。

(2) 業務APサーバのCPU使用率が月末の締め処理時に常時90%を超え、レスポンスが著しく低下する。通常時のCPU使用率は30%程度である。

(3) Webサーバへの外部からのアクセスが増加傾向にあり、現在のピーク時の同時接続数は500件である。今後2年で1,500件に増加すると予測されている。

(4) リモートワークの導入に伴い、社外から社内システムへのセキュアなアクセス手段が必要になった。現在はVPNを使用しているが、全社員が同時接続するとVPN装置の処理能力が不足する。

(5) 現在のバックアップは日次で取得しているが、RPO（Recovery Point Objective: 目標復旧時点）が24時間であることが経営層から問題視されている。目標RPOは1時間以内、RTO（Recovery Time Objective: 目標復旧時間）は2時間以内に改善したい。

**【検討中の対策案】**

情報システム部門のB主任は、以下の対策を検討している。

- 対策a: DBサーバをアクティブ・スタンバイ構成に変更する。
- 対策b: 業務APサーバにスケールアウト構成を導入し、ロードバランサを配置する。
- 対策c: Webサーバをクラウドに移行し、オートスケーリングを導入する。
- 対策d: VPNに代えて、（  ア  ）の考え方に基づくリモートアクセス基盤を導入する。
- 対策e: バックアップ方式をレプリケーション方式に変更し、RPOを改善する。

## 設問1（知識問題）

空欄（ア）に入る最も適切な用語を答えてください。また、従来のVPNとの違いを3つの観点から説明してください。

<details>
<summary>ヒント</summary>

「ネットワークの内外を区別せず、すべてのアクセスを検証する」セキュリティモデルを考えてください。「信頼しない」がキーワードです。

</details>

<details>
<summary>解答</summary>

**（ア）: ゼロトラスト（Zero Trust）**

VPNとゼロトラストの違い:

| 観点 | VPN | ゼロトラスト |
|------|-----|------------|
| 信頼モデル | 社内ネットワークに入れば信頼する（境界型防御） | すべてのアクセスを常に検証する（Never Trust, Always Verify） |
| 認証 | VPN接続時の1回のみ | アクセスのたびに認証・認可（継続的検証） |
| アクセス制御 | ネットワーク単位（接続すれば社内全体にアクセス可能） | リソース単位（必要最小限のアクセスのみ許可、マイクロセグメンテーション） |

ゼロトラストの主な構成要素:
- ID管理（IAM）と多要素認証（MFA）
- デバイスの健全性チェック（エンドポイント検証）
- マイクロセグメンテーション（細かいアクセス制御）
- 継続的な監視とログ分析

</details>

## 設問2（計算問題）

以下の条件でシステムの稼働率を計算してください。

**条件:**
- DBサーバ単体の稼働率: 0.99
- 業務APサーバ単体の稼働率: 0.99
- ロードバランサの稼働率: 0.999

(1) 現行構成（DBサーバ1台、APサーバ1台の直列構成）の稼働率を計算してください。

(2) 対策a・b適用後（DBサーバ2台並列、APサーバ2台並列＋ロードバランサの直列構成）の稼働率を計算してください。

(3) 対策適用前後で年間ダウンタイムがどれだけ改善されるか計算してください。

<details>
<summary>ヒント</summary>

```
現行: DB(直列) × AP(直列)
対策後: DB並列 × LB(直列) × AP並列

並列2台: 1 - (1-A)²
年間時間: 365日 × 24時間 = 8,760時間
年間ダウンタイム = (1 - 稼働率) × 8,760
```

</details>

<details>
<summary>解答</summary>

**(1) 現行構成の稼働率**

```
A_現行 = A_DB × A_AP
       = 0.99 × 0.99
       = 0.9801
```

**(2) 対策後の稼働率**

```
DB並列: A_DB並列 = 1 - (1 - 0.99)² = 1 - 0.0001 = 0.9999
AP並列: A_AP並列 = 1 - (1 - 0.99)² = 1 - 0.0001 = 0.9999

対策後（直列）: A_対策後 = A_DB並列 × A_LB × A_AP並列
              = 0.9999 × 0.999 × 0.9999
              = 0.9988
```

**(3) 年間ダウンタイムの改善**

```
現行:   (1 - 0.9801) × 8,760 = 0.0199 × 8,760 ≈ 174.3時間/年
対策後: (1 - 0.9988) × 8,760 = 0.0012 × 8,760 ≈ 10.5時間/年

改善幅: 174.3 - 10.5 ≈ 163.8時間/年の削減
```

年間約164時間（約7日間分）のダウンタイムが削減されます。

</details>

## 設問3（分析問題）

課題(5)のバックアップに関して、以下の問いに答えてください。

(1) 現在の日次バックアップ方式でRPO=24時間となる理由を説明してください。

(2) 対策eのレプリケーション方式を採用した場合、同期レプリケーションと非同期レプリケーションのそれぞれについて、RPOとメリット・デメリットを説明してください。

(3) RTO=2時間以内を達成するために、対策a（アクティブ・スタンバイ）と対策e（レプリケーション）をどのように組み合わせるべきか提案してください。

<details>
<summary>解答</summary>

**(1) RPO=24時間の理由**

日次バックアップでは、最大で前日のバックアップ取得時点から障害発生時点までの約24時間分のデータが失われる可能性があります。例えば、午前0時にバックアップを取得し、午後11時に障害が発生した場合、約23時間分のデータが消失します。

**(2) 同期と非同期レプリケーションの比較**

| 項目 | 同期レプリケーション | 非同期レプリケーション |
|------|-------------------|---------------------|
| RPO | ほぼ0（データロスなし） | 数秒〜数分（レプリケーション遅延分） |
| 仕組み | 書込完了をプライマリ・セカンダリ両方で確認 | プライマリの書込完了後、非同期でセカンダリに転送 |
| メリット | データ完全性が高い | パフォーマンスへの影響が小さい |
| デメリット | 書込性能が低下する（レイテンシ増加） | 障害時に若干のデータロスが発生する |
| 適用距離 | 近距離（同一データセンタ内）が望ましい | 遠距離（異なるデータセンタ間）でも可能 |

**(3) 組み合わせ提案**

```
[プライマリDBサーバ] ←同期レプリケーション→ [スタンバイDBサーバ]
     (アクティブ)                          (スタンバイ)
         │
         │ 非同期レプリケーション
         ↓
[DR用DBサーバ（遠隔地）]
```

- **同一サイト内**: アクティブ・スタンバイ＋同期レプリケーションでRPO≈0、フェイルオーバーでRTO数分を実現
- **DR（災害対策）**: 遠隔地への非同期レプリケーションで、サイト障害にも対応
- スタンバイサーバへの自動フェイルオーバーを設定することで、RTO=2時間以内を十分に達成可能

</details>

## 設問4（設計課題）

課題(3)のWebサーバのクラウド移行について、以下の構成図の空欄を埋め、クラウド移行後のシステム構成を完成させてください。

```
インターネット
     │
  [  (a)  ]  ← クラウド側のDDoS対策・CDN
     │
  [  (b)  ]  ← 負荷分散
     │
  ┌──┴──────────────┐
  │ [Webサーバ] × N台  │  ← (c) による自動増減
  └──┬──────────────┘
     │
  [  (d)  ]  ← クラウドと社内の接続
     │
  ┌──┴──────────────────────────────────┐
  │         社内ネットワーク              │
  │  [業務APサーバ]  [DBサーバ]           │
  └─────────────────────────────────────┘
```

<details>
<summary>ヒント</summary>

- (a): クラウドベンダが提供するコンテンツ配信＋DDoS対策サービス
- (b): L7レベルの負荷分散を行うクラウドサービス
- (c): 負荷に応じてインスタンスを自動的に増減する機能
- (d): クラウドとオンプレミスを安全に接続する方法

</details>

<details>
<summary>解答</summary>

```
(a): CDN（Content Delivery Network）/ WAF（Web Application Firewall）
     ← CloudFront + AWS WAF、Azure CDN + Front Door など

(b): クラウドロードバランサ（ALB: Application Load Balancer 等）
     ← L7負荷分散、パスベースルーティング

(c): オートスケーリング（Auto Scaling）
     ← CPU使用率やリクエスト数に応じてインスタンスを自動増減

(d): VPN接続 または 専用線（AWS Direct Connect、Azure ExpressRoute 等）
     ← ハイブリッドクラウド構成を実現する接続方式
```

**完成した構成図:**

```
インターネット
     │
  [CDN / WAF]  ← DDoS対策、静的コンテンツのキャッシュ、WAFによる攻撃防御
     │
  [ALB（Application Load Balancer）]  ← L7負荷分散、ヘルスチェック
     │
  ┌──┴──────────────────────┐
  │ [Webサーバ] × 2〜10台     │  ← オートスケーリング
  │ ・最小2台（通常時）        │     （CPU使用率70%超で1台追加）
  │ ・最大10台（ピーク時）     │     （同時接続1,500件に対応）
  └──┬──────────────────────┘
     │
  [Site-to-Site VPN / 専用線]  ← 暗号化された安全な接続
     │
  ┌──┴──────────────────────────────────┐
  │         社内ネットワーク              │
  │  [業務APサーバ群]  [DBサーバ(冗長)]  │
  └─────────────────────────────────────┘
```

この構成のメリット:
1. Webサーバがオートスケーリングにより同時接続数1,500件に自動対応
2. CDN/WAFにより外部からの攻撃を軽減
3. 社内の業務APサーバとDBサーバはオンプレミスに残すことで、データの機密性を確保（ハイブリッドクラウド）
4. 段階的なクラウド移行が可能

</details>

## 検証用Pythonコード

<details>
<summary>設問2の計算を検証するPythonコード</summary>

```python
# 午後問題 設問2の計算検証
# 実行方法：python architecture_calc.py

def parallel(a, n=2):
    """並列n台の稼働率"""
    return 1 - (1 - a) ** n

def serial(*args):
    """直列の稼働率"""
    result = 1.0
    for a in args:
        result *= a
    return result

def annual_downtime(availability):
    """年間ダウンタイム（時間）"""
    return (1 - availability) * 365 * 24

# 条件
a_db = 0.99
a_ap = 0.99
a_lb = 0.999

# (1) 現行構成
current = serial(a_db, a_ap)
print(f"(1) 現行構成の稼働率: {current:.4f}")

# (2) 対策後
db_parallel = parallel(a_db)
ap_parallel = parallel(a_ap)
improved = serial(db_parallel, a_lb, ap_parallel)
print(f"\n(2) 対策後の稼働率:")
print(f"  DB並列: {db_parallel:.4f}")
print(f"  AP並列: {ap_parallel:.4f}")
print(f"  全体:   {improved:.4f}")

# (3) 年間ダウンタイム
dt_current = annual_downtime(current)
dt_improved = annual_downtime(improved)
print(f"\n(3) 年間ダウンタイム:")
print(f"  現行:   {dt_current:.1f}時間/年")
print(f"  対策後: {dt_improved:.1f}時間/年")
print(f"  改善:   {dt_current - dt_improved:.1f}時間/年の削減")
```

</details>

# 実践課題10：プロジェクト状況レビュー演習 ★4

> **難易度**: ★★★★☆（上級）
> **前提知識**: 第1章〜第7章（PM概論〜コミュニケーション管理）
> **課題の種類**: レビュー演習
> **学習目標**: 問題を抱えるプロジェクトの状況報告を読み解き、根本原因を特定し、具体的な立て直し策を提案できるようになる

---

## 課題の説明

以下は、あるプロジェクトの5ヶ月目時点での状況報告書です。この報告書を読んで、**10個の設問**に答えてください。

**この課題は「コードリーディング」のPM版です。他人が書いたプロジェクト資料を批判的に読み解く力を鍛えます。**

---

## 設問

以下の設問に答えてください。各設問は独立して回答可能ですが、すべてを通して回答することで、プロジェクト立て直しの全体像が見えてきます。

### 設問1（分析）
EVM指標からSV、CV、SPI、CPI、EAC、ETCを計算し、プロジェクトの健全性を数値で診断してください。

### 設問2（分析）
課題#1と#4から、このプロジェクトで発生している問題の名前（PMの専門用語）を答えてください。また、なぜこの問題が起きたのか、根本原因を3つ挙げてください。

### 設問3（分析）
課題#2が起きた根本原因は何だと考えますか？要件定義フェーズの進め方にどんな問題があったかを指摘してください。

### 設問4（分析）
「リスク登録簿: 作成されていない」ことで、どのようなリスクが見逃された可能性がありますか？ 少なくとも3つのリスクを具体的に挙げてください。

### 設問5（分析）
コミュニケーション体制の問題点を3つ以上指摘してください。

### 設問6（判断）
「PMのコメント」にはどのような問題がありますか？PMとしてあるべき姿勢と比較して論じてください。

### 設問7（計画）
このプロジェクトを立て直すとしたら、最初の1週間で何をしますか？優先順位を付けてアクションプランを作成してください。

### 設問8（判断）
10月末の期限に間に合わせるための現実的な選択肢を3つ提示し、それぞれのメリット/デメリットを整理してください。

### 設問9（設計）
今後の変更管理プロセスを設計してください。A社からの追加要求にどう対応するルールを作りますか？

### 設問10（総合）
このプロジェクトの状況を経営層（自社の部長クラス）に報告する想定で、報告メール（A4 1枚相当）を書いてください。現状の問題、影響、対策案を含めてください。

<details>
<summary>設問1の解答：EVM分析</summary>

```
EVM計算結果：

  BAC = 3,000万円
  PV  = 2,200万円
  EV  = 1,350万円
  AC  = 2,100万円

  SV = EV - PV = 1,350 - 2,200 = -850万円（大幅なスケジュール遅延）
  CV = EV - AC = 1,350 - 2,100 = -750万円（大幅なコスト超過）
  SPI = EV / PV = 1,350 / 2,200 = 0.61（計画の61%しか進んでいない）
  CPI = EV / AC = 1,350 / 2,100 = 0.64（1円あたり0.64円の成果）

  EAC = BAC / CPI = 3,000 / 0.64 = 4,688万円（予測総コスト）
  ETC = EAC - AC = 4,688 - 2,100 = 2,588万円（残り必要コスト）
  VAC = BAC - EAC = 3,000 - 4,688 = -1,688万円（予測予算超過額）

  診断:
  ・SPI 0.61 → 計画の約6割しか進んでおらず、深刻な遅延状態
  ・CPI 0.64 → 効率が極めて悪く、投入した費用に見合う成果が出ていない
  ・このままのペースでは総額4,688万円（+56%超過）が予測される
  ・残り2ヶ月で2,588万円が必要だが、通常ペースでは不可能
  ★ 結論: プロジェクトは「赤信号」であり、即時の介入が必要
```

</details>

<summary>設問2の解答：スコープクリープの分析</summary>

```
問題の名前: スコープクリープ（Scope Creep）

スコープクリープとは、プロジェクトのスコープ（作業範囲）が、
正式な変更管理プロセスを経ずに、少しずつ拡大していく現象です。

根本原因：

1. 変更管理プロセスが存在しない
   - 追加要求17件 + 5件 = 計22件がすべて無条件で受け入れられている
   - 各変更の影響分析（工数・コスト・スケジュール）が行われていない
   - 「受け入れるかどうかの判断基準」が定義されていない

2. スコープのベースラインが不明確
   - 「含むもの/含まないもの」の合意文書がない（あったとしても機能していない）
   - 要件定義書の承認プロセスが形骸化している可能性
   - A社側に「要件は確定した」という認識がない

3. PMがクライアントの要求を断れていない
   - PM初経験で、要求への「ノー」の言い方が分からない
   - 「顧客の言うことはすべて聞くべき」と誤解している
   - トレードオフ（追加するなら何かを削る/延ばす/追加費用）を
     提示できていない
```

</details>

<summary>設問3の解答：設計レビュー失敗の原因</summary>

```
課題#2「画面イメージが違う」の根本原因：

1. 要件定義時にプロトタイプ/モックアップを使っていなかった
   - テキストベースの要件定義書だけでは、画面のイメージが共有できない
   - A社側は「自分の頭の中のイメージ」で理解し、開発側は「自分の解釈」で設計した
   - 認識の齟齬が設計レビューまで発見されなかった

2. レビューのタイミングが遅すぎた
   - 基本設計が完了してからレビューしたため、手戻りのコストが大きい
   - 要件定義の段階でワイヤーフレームを見せて合意すべきだった
   - インクリメンタルに確認していれば、3回のやり直しは防げた

3. レビュー参加者の問題
   - A社側のレビュー参加者が適切だったか不明
   - 実際にシステムを使う現場の営業担当が参加していたか？
   - 意思決定者（営業部長）が画面イメージを事前に確認していたか？

対策案:
  - 次フェーズ以降はFigma等でプロトタイプを作り、実機で確認してもらう
  - 2週間ごとにデモを実施し、早期にフィードバックを得る
  - レビュー時の「承認」を文書で記録する
```

</details>

<summary>設問5の解答：コミュニケーションの問題</summary>

```
コミュニケーション体制の問題点：

1. A社との定例が月1回は少なすぎる
   - 7ヶ月のプロジェクトで月1回では、問題の発見・対応が遅れる
   - 週次（最低でも隔週）にすべき
   - 定例の間に要求が溜まり、一度に大量に出てくる原因にもなっている

2. チーム内会議が「不定期」で構造化されていない
   - 日次のスタンドアップ（朝会）を実施すべき
   - 課題・障害の共有が遅れ、問題が潜在化する原因
   - SE離脱（課題#3）の兆候も早期に察知できた可能性

3. 上長への報告が月次報告書のみ
   - 遅延が拡大している状況で、月1回の書面報告では対応が遅すぎる
   - エスカレーション（上位者への緊急報告）の基準が設けられていない
   - 上長は5ヶ月目の報告で初めて深刻さを知る可能性がある

4. A社内のステークホルダー構造が不明
   - 要求を出しているのは誰か？ 承認権限を持つのは誰か？
   - 営業部長が追加要求を出しているのに、定例の場で止められていない
   - RACI（責任分担表）が定義されていない

5. 変更要求のコミュニケーション経路が未定義
   - 誰がどのルートで要求を出すのか、ルールがない
   - 「思いついたら直接開発者に言う」状態の可能性がある
```

<summary>設問9の解答：変更管理プロセスの設計</summary>

```
変更管理プロセス（A社向け）

1. 受付ルール
   - 変更要求はCR票（所定フォーマット）で提出
   - 口頭依頼はCR票に記載後に正式受付
   - 開発者への直接依頼は禁止（必ずPM経由）

2. 影響分析（PM + 開発リーダー、3営業日以内）
   - 工数影響、スケジュール影響、コスト影響
   - 既存機能への影響（回帰テストの必要性）

3. 承認基準
   小規模（3人日以下）: PM + A社担当者で判断
   中規模（3-10人日）:  PM + A社プロジェクト責任者で判断
   大規模（10人日超）:  ステアリング委員会で審議

4. トレードオフの原則
   「追加するなら、以下のいずれかを選択してください」
   (a) 他の機能を同等工数分だけ削減する
   (b) 期限を延長する
   (c) 追加費用を負担する

5. スコープフリーズ
   テスト開始2週間前以降は、原則として変更要求を受け付けない
   （緊急の障害修正を除く）
```

# 第6章：RAGとコンテキスト活用

## この章のゴール

- RAG（Retrieval-Augmented Generation：検索拡張生成）の仕組みを理解する
- コンテキスト設計の原則を学ぶ
- チャンク分割の手法と最適化を習得する
- 効果的なシステムプロンプトの設計方法を学ぶ
- RAGの限界と改善方法を理解する

---

## 6.1 RAGとは

### 基本概念

RAG（Retrieval-Augmented Generation）は、LLMに**外部の知識ベースから取得した情報を追加で提供し、回答の精度と最新性を向上させる**手法です。

```
従来のLLM：
  質問 → LLM（学習済みの知識のみ） → 回答
  問題：古い情報、組織固有の情報を知らない

RAG：
  質問 → 検索エンジン → 関連ドキュメント取得
       → LLM（質問 + 取得ドキュメント） → 回答
  メリット：最新情報、組織固有の情報を活用できる
```

### RAGの処理フロー

```
┌─────────┐     ┌──────────┐     ┌─────────────┐
│ ユーザー │     │ 検索      │     │ 知識ベース   │
│ の質問   │────→│ エンジン  │────→│（ドキュメント│
│         │     │          │←────│ DB、Wiki等） │
└─────────┘     └────┬─────┘     └─────────────┘
                     │
                     │ 関連ドキュメント
                     ▼
              ┌──────────────┐
              │   LLM        │
              │              │
              │ プロンプト：  │
              │ ・質問       │
              │ ・取得した    │
              │   ドキュメント│
              │ ・指示       │
              └──────┬───────┘
                     │
                     ▼
              ┌──────────────┐
              │   回答       │
              │（根拠に基づく）│
              └──────────────┘
```

### なぜRAGが必要か

| LLMの限界 | RAGによる解決 |
|----------|-------------|
| 知識のカットオフ | 最新ドキュメントを検索して提供 |
| ハルシネーション | 根拠となる文書を明示 |
| 組織固有の知識がない | 社内ドキュメントを知識ベースに |
| 機密情報を学習に使えない | ランタイムでのみ参照（学習不要） |

> **ポイントまとめ**: RAGはLLMの知識の限界を補完する仕組みです。ファインチューニング（再学習）と異なり、知識ベースを更新するだけで最新情報に対応できます。

## 6.2 コンテキスト設計

### コンテキストの構造

LLMに提供する情報（コンテキスト）の設計は、RAGの品質を大きく左右します。

```
効果的なコンテキストの構造：

┌──────────────────────────────────────┐
│ システムプロンプト（役割・制約）        │
├──────────────────────────────────────┤
│ 取得したドキュメント（検索結果）        │
│  ├── ドキュメント1（関連度: 0.95）     │
│  ├── ドキュメント2（関連度: 0.87）     │
│  └── ドキュメント3（関連度: 0.82）     │
├──────────────────────────────────────┤
│ ユーザーの質問                         │
├──────────────────────────────────────┤
│ 回答の指示（出力形式・制約）            │
└──────────────────────────────────────┘
```

### コンテキスト設計の原則

| 原則 | 説明 | 具体例 |
|------|------|--------|
| **関連性** | 質問に関連する情報のみ提供 | 無関係なドキュメントは除外 |
| **簡潔性** | 必要最小限の情報に絞る | 全文ではなく関連部分を抽出 |
| **構造化** | 情報を整理して提供 | タグやセクションで区切る |
| **優先順位** | 重要な情報を先頭に配置 | 関連度の高いものを上に |
| **出典の明示** | 情報のソースを明確に | ドキュメント名、ページ番号 |

### RAGプロンプトのテンプレート

```
あなたは社内FAQアシスタントです。
以下のドキュメントの内容のみに基づいて質問に回答してください。
ドキュメントに記載がない情報については「この情報は社内ドキュメントに
見つかりませんでした」と回答してください。

### 参照ドキュメント

有給休暇は入社6ヶ月経過後に10日付与される。
勤続年数に応じて最大20日まで増加する。
有給休暇の取得は原則として3営業日前までに申請する。
</document>

<document source="福利厚生ガイド 2026年版">
リモートワーク手当は月額5,000円。
通勤手当は実費支給（上限月額50,000円）。
</document>

### 質問
有給休暇は入社後いつから取得できますか？

### 回答形式
- 簡潔に回答する（1〜3文）
- 根拠となるドキュメント名を記載する
- 補足情報があれば追記する
```

> **ポイントまとめ**: コンテキストには関連する情報のみを構造的に提供し、出典を明示することで回答の信頼性が向上します。

## 6.3 チャンク分割

### チャンク分割とは

ドキュメントをLLMに提供するために適切なサイズの断片（チャンク）に分割する処理です。チャンクの品質がRAG全体の品質を決めると言っても過言ではありません。

```
元のドキュメント（10,000トークン）
        ↓ チャンク分割
┌────────────┐
│ チャンク1   │ 500トークン
│ (導入部分)  │
├────────────┤
│ チャンク2   │ 500トークン
│ (基本機能)  │
├────────────┤
│ チャンク3   │ 500トークン
│ (詳細設定)  │
├────────────┤
│    ...      │
└────────────┘
```

### チャンク分割の手法

| 手法 | 説明 | メリット | デメリット |
|------|------|---------|-----------|
| **固定長分割** | 一定のトークン数で分割 | 実装が簡単 | 文脈が途中で切れる |
| **セクション分割** | 見出しやセクション単位で分割 | 意味的なまとまりを保持 | チャンクサイズが不均一 |
| **オーバーラップ分割** | 前後のチャンクと一部重複させる | 文脈の欠落を防ぐ | トークン消費が増える |
| **セマンティック分割** | 意味の区切りで分割 | 最も自然な分割 | 実装が複雑 |

### チャンクサイズの目安

```
チャンクサイズと検索精度の関係：

小さすぎるチャンク（100トークン未満）
  ↓
  ・文脈が失われる
  ・断片的で意味が取りにくい
  ・検索ノイズが増える

────────────────────
最適なチャンクサイズ（200〜1000トークン）
  ↓
  ・十分な文脈を保持
  ・検索精度が高い
  ・トークン消費のバランスが良い
────────────────────

大きすぎるチャンク（2000トークン以上）
  ↓
  ・関連しない情報が混入する
  ・検索精度が低下する
  ・トークン消費が多い
```

### オーバーラップの効果

```
オーバーラップなし：
  [チャンク1: A B C] [チャンク2: D E F] [チャンク3: G H I]
  → CとDの間の文脈が失われる

オーバーラップあり（20%）：
  [チャンク1: A B C]
       [チャンク2: C D E F]
            [チャンク3: F G H I]
  → 重複部分で文脈がつながる

推奨オーバーラップ率：10〜20%
```

> **ポイントまとめ**: チャンクサイズは200〜1000トークンが目安。セクション単位での分割にオーバーラップを組み合わせると効果的です。

## 6.4 システムプロンプトの設計

### システムプロンプトとは

LLMの動作を全体的に制御する初期設定のプロンプトです。APIを使用する場合、systemロールのメッセージとして設定します。

```
システムプロンプトの位置：

┌───────────────────────┐
│ System: (システムプロンプト)│ ← 最初に設定
│  役割、制約、振る舞い     │
├───────────────────────┤
│ User: (ユーザーの質問)     │ ← 各会話で変わる
├───────────────────────┤
│ Assistant: (LLMの回答)    │ ← LLMが生成
└───────────────────────┘
```

### システムプロンプトの設計テンプレート

```
あなたは[役割]です。

## 目的
[何のために存在するか]

## 制約
- [やるべきこと]
- [やってはいけないこと]
- [回答のトーン]

## 知識ベース
提供されるドキュメントの情報のみに基づいて回答してください。
ドキュメントに記載がない場合は「分かりません」と正直に伝えてください。

## 出力形式
[デフォルトの出力形式]
```

### 具体例：社内チャットボット

```
あなたは社内ヘルプデスクのAIアシスタント「サポートくん」です。

## 目的
社員の業務上の質問に対して、社内ドキュメントに基づいて
迅速かつ正確に回答することです。

## 制約
- 社内ドキュメントの情報のみに基づいて回答する
- ドキュメントに記載がない場合は「担当部署にお問い合わせください」と案内する
- 個人情報や機密情報を含む回答はしない
- 法的アドバイスや医療アドバイスは行わない
- 敬語（です・ます調）で回答する
- 回答は簡潔に（3〜5文程度）

## 回答形式
1. 質問への直接的な回答
2. 根拠となるドキュメント名
3. 追加で参考になる情報（あれば）
4. 不明点がある場合の問い合わせ先
```

> **ポイントまとめ**: システムプロンプトでLLMの振る舞いの大枠を制御し、各ユーザープロンプトで個別のタスクを指示します。

## 6.5 RAGの改善手法

### よくある問題と対策

| 問題 | 原因 | 対策 |
|------|------|------|
| 関係ない情報が取得される | 検索の精度が低い | 検索クエリの最適化、リランキング |
| 回答に根拠がない | LLMが検索結果を無視 | 「提供されたドキュメントのみ使用」と強調 |
| 情報が古い | 知識ベースが更新されていない | 定期的なドキュメント更新 |
| 回答が断片的 | チャンクが小さすぎる | チャンクサイズの調整 |
| 回答が冗長 | コンテキストが多すぎる | 関連度でフィルタリング |

### クエリ変換テクニック

ユーザーの質問をそのまま検索するのではなく、検索に適した形に変換します。

```
ユーザーの質問：
「有給って入社してすぐ使えるの？」

変換パターン：

1. 正規化：
「有給休暇の付与条件 入社後の取得開始時期」

2. 拡張：
「有給休暇 付与 条件 入社 期間 取得 開始」

3. HyDE（仮の回答を生成して検索）：
「入社後6ヶ月経過した時点で有給休暇が10日付与されます...」
→ この文で類似検索を行う
```

## 6.6 実践例：社内ドキュメント検索システム

### 設計の全体像

```
社内ドキュメント検索システムの構成：

1. ドキュメント準備
   ・社内Wiki、規程、マニュアルを収集
   ・チャンク分割（セクション単位、500トークン、オーバーラップ10%）
   ・ベクトル化してベクトルDBに格納

2. 検索・回答フロー
   ・ユーザー質問を受け取る
   ・質問をベクトル化し、類似チャンクを上位5件取得
   ・関連度スコアでフィルタリング（0.7以上のみ使用）
   ・プロンプトを構築してLLMに投入
   ・回答を生成し、出典を付けて返す

3. フィードバックループ
   ・ユーザーの「役に立った/立たなかった」の評価を収集
   ・低評価の質問を分析し、チャンク分割やプロンプトを改善
```

> **ポイントまとめ**: RAGは「検索の品質 x プロンプトの品質 x チャンクの品質」で決まります。どれか一つが低いと全体の品質が下がるため、バランスよく改善することが重要です。

## まとめ

```
RAGとコンテキスト活用の重要ポイント：

✅ RAGはLLMの知識の限界を外部データで補完する手法
✅ コンテキストは関連性・簡潔性・構造化・優先順位を意識して設計
✅ チャンク分割は200〜1000トークンが目安、オーバーラップ10〜20%
✅ システムプロンプトでLLMの基本的な振る舞いを制御
✅ ドキュメントに基づく回答を徹底し、ハルシネーションを防ぐ
✅ クエリ変換で検索精度を向上させる
✅ フィードバックループで継続的に品質を改善する
```

## 基本問題

### 問題1: SSHの特徴
SSHの特徴として正しいものをすべて選んでください。

a) 通信が暗号化される
b) デフォルトでTCPポート22番を使用する
c) リモートサーバーへのログインのみに使用される
d) ファイル転送にも利用できる
e) 通信内容は平文で送信される

<details>
<summary>解答</summary>

**a, b, d** が正しいです。

```
a) ○ - SSHはすべての通信を暗号化します
b) ○ - デフォルトでTCPポート22番を使用します
c) × - ログインだけでなく、ファイル転送やポートフォワーディングも可能です
d) ○ - SCP/SFTPによるファイル転送が可能です
e) × - SSHの通信は暗号化されます。平文で送信するのはTelnetです
```

</details>

### 問題2: SSH以前のプロトコル
SSH登場以前に使われていたTelnetの最大の問題点は何ですか？また、SSHはその問題をどのように解決しましたか？

<details>
<summary>解答</summary>

**最大の問題点**: 通信が**平文（暗号化なし）**で行われるため、パスワードやコマンドの内容がネットワーク上で盗聴される危険がありました。

**SSHの解決方法**: すべての通信を**暗号化**して送受信することで、たとえ通信が傍受されても内容を読み取れないようにしました。

```
Telnet: クライアント ---[平文: password123]--→ サーバー
                         ↑ 盗聴可能！

SSH:    クライアント ---[暗号化データ]--→ サーバー
                         ↑ 盗聴しても解読不能
```

</details>

### 問題3: SSHプロトコルの3層構造
SSHプロトコルの3つのレイヤーの名前と、それぞれの役割を答えてください。

<details>
<summary>ヒント</summary>

最下層から「暗号化」「認証」「チャネル管理」の順に積み重なっています。

</details>

<details>
<summary>解答</summary>

```
1. トランスポート層（SSH-TRANS）
   役割: 暗号化、完全性検証、鍵交換、サーバー認証

2. 認証層（SSH-USERAUTH）
   役割: ユーザー認証（パスワード認証、公開鍵認証など）

3. コネクション層（SSH-CONNECT）
   役割: チャネルの多重化、ポートフォワーディング、セッション管理
```

</details>

### 問題4: 暗号技術の分類
以下の暗号技術を「共通鍵暗号」と「公開鍵暗号」に分類してください。

- AES
- RSA
- Ed25519
- ChaCha20-Poly1305
- ECDSA

<details>
<summary>解答</summary>

```
【共通鍵暗号（対称鍵暗号）】
- AES                 ← SSHでデータの暗号化に最も使われる
- ChaCha20-Poly1305   ← モバイル環境で高速

【公開鍵暗号（非対称鍵暗号）】
- RSA                 ← 長い歴史を持つ、2048bit以上推奨
- Ed25519             ← 現在の推奨、高速で安全
- ECDSA               ← 楕円曲線暗号
```

</details>

|------|-------|-------|


### 問題6: OpenSSHのツール
以下のOpenSSHツールの役割をそれぞれ答えてください。

1. `ssh`
2. `sshd`
3. `ssh-keygen`
4. `ssh-agent`
5. `ssh-copy-id`

<details>
<summary>解答</summary>

```
1. ssh        - SSHクライアント。リモートサーバーへのログインやコマンド実行を行う
2. sshd       - SSHデーモン（サーバープロセス）。SSH接続を受け付ける
3. ssh-keygen - SSH鍵ペアの生成・管理を行うツール
4. ssh-agent  - SSH秘密鍵をメモリに保持し、パスフレーズの再入力を省略する認証エージェント
5. ssh-copy-id - 公開鍵をリモートサーバーのauthorized_keysに登録するツール
```

</details>

### 問題7: TOFU
SSHの初回接続時に表示される以下のメッセージについて、(1) このメッセージが表示される理由、(2) yes と答えた場合に何が起きるか を説明してください。

```
The authenticity of host 'example.com (93.184.216.34)' can't be established.
ED25519 key fingerprint is SHA256:xxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxx.
Are you sure you want to continue connecting (yes/no/[fingerprint])?
```

<details>
<summary>解答</summary>

**(1) 表示される理由:**
初めて接続するサーバーのため、`~/.ssh/known_hosts` にホスト鍵の記録がなく、接続先が本当に正しいサーバーかどうかを確認できないためです。この仕組みは **TOFU（Trust On First Use: 初回利用時の信頼）** と呼ばれます。

**(2) yes と答えた場合:**
サーバーのホスト鍵（ED25519フィンガープリント）が `~/.ssh/known_hosts` ファイルに記録されます。2回目以降の接続では、記録された鍵と照合し、一致すれば自動的に接続されます。鍵が変わっていれば警告が表示されます。

</details>

### 問題8: 鍵交換の仕組み
Diffie-Hellman鍵交換では、クライアントとサーバーが安全でない通信路上で共通鍵を生成できます。この仕組みについて、なぜ盗聴者が共通鍵を計算できないのか、簡潔に説明してください。

<details>
<summary>ヒント</summary>

盗聴者が知ることができる情報と、知ることができない情報を整理してみましょう。

</details>

<details>
<summary>解答</summary>

Diffie-Hellman鍵交換では、クライアントとサーバーはそれぞれ**秘密値**（a, b）を選び、その秘密値から計算した**公開値**（A, B）を交換します。

```
盗聴者が知れる情報: 公開値 A と B、パラメータ g と p
盗聴者が知れない情報: 秘密値 a と b

共通鍵 K = B^a mod p = A^b mod p

盗聴者が K を計算するには a または b が必要ですが、
A = g^a mod p から a を逆算する問題（離散対数問題）は
計算量的に困難であるため、実用上は解けません。
```

つまり、「公開値から秘密値を逆算するのが数学的に困難」であることがセキュリティの根拠です。

</details>

## チャレンジ問題

### 問題9: SSHの接続確立手順
SSH接続が確立される手順を正しい順番に並べ替えてください。

a) ユーザー認証（パスワードまたは公開鍵）
b) TCP接続（ポート22番）
c) セッション開始（シェル、コマンド実行）
d) 鍵交換と暗号アルゴリズムの合意
e) プロトコルバージョンの交換
f) サーバー認証（ホスト鍵の確認）

<details>
<summary>解答</summary>

正しい順番: **b → e → d → f → a → c**

```
1. b) TCP接続（ポート22番）
   → まずTCPレベルで接続を確立する

2. e) プロトコルバージョンの交換
   → "SSH-2.0-OpenSSH_9.6" のようなバージョン文字列を交換

3. d) 鍵交換と暗号アルゴリズムの合意
   → 使用する暗号方式を決め、共通鍵を生成

4. f) サーバー認証（ホスト鍵の確認）
   → 接続先が正しいサーバーか検証

   ===== ここから暗号化通信 =====

5. a) ユーザー認証（パスワードまたは公開鍵）
   → ユーザーが正しい利用者か確認

6. c) セッション開始（シェル、コマンド実行）
   → 認証成功後、実際の作業を開始
```

</details>

### 問題10: 総合理解
あなたが新しいクラウドサーバーを契約し、初めてSSH接続する場面を想像してください。接続時に起きることを、SSHプロトコルの3層構造に対応づけて、時系列で説明してください。

<details>
<summary>解答</summary>

```
【1. トランスポート層での処理】
① TCP接続を確立（ポート22番）
② プロトコルバージョンを交換（SSH-2.0-OpenSSH_9.6）
③ 暗号アルゴリズムを合意（例: AES-256-GCM、curve25519-sha256）
④ Diffie-Hellman鍵交換で共通鍵を生成
⑤ サーバーのホスト鍵を確認
   → 初回接続のため "Are you sure you want to continue connecting?" が表示
   → yes と答えるとホスト鍵がknown_hostsに記録される
⑥ ここから先は暗号化された通信路が確立

【2. 認証層での処理】
⑦ サーバーが対応する認証方式を提示
⑧ クライアントが認証を試行
   - 公開鍵認証: 秘密鍵で署名し、サーバーが公開鍵で検証
   - パスワード認証: パスワードを暗号化して送信（非推奨）
⑨ 認証成功

【3. コネクション層での処理】
⑩ セッションチャネルを開設
⑪ シェルが起動し、コマンドを実行できる状態になる
⑫ 必要に応じてポートフォワーディング等の追加チャネルを開設
```

</details>

# 第4章: データフェッチング

## この章のゴール

- サーバーコンポーネントにおける `fetch` の使い方とキャッシュ戦略を理解する

- 静的生成（SSG）、サーバーサイドレンダリング（SSR）、ISR の使い分けを実務レベルで判断できるようになる
- Server Actions を使ったデータ変更（ミューテーション）の基本を知る

## 4.1 なぜ Next.js のデータフェッチングは特別なのか

### 従来の React（SPA）でのデータ取得

従来の React アプリケーションでは、コンポーネントがブラウザ上でマウントされた後に `useEffect` でデータを取得していました。この方式にはいくつかの問題があります。

```tsx
// 従来の React（SPA）— ブラウザでデータを取得
'use client'
import { useState, useEffect } from 'react'

export default function PostList() {
  const [posts, setPosts] = useState([])
  const [loading, setLoading] = useState(true)

  useEffect(() => {
    // 問題1: ブラウザからAPIにリクエスト（ウォーターフォール）
    // 問題2: ローディング状態の管理が煩雑
    // 問題3: 初回表示時にコンテンツが空（SEOに不利）
    fetch('/api/posts')
      .then(res => res.json())
      .then(data => {
        setPosts(data)
        setLoading(false)
      })
  }, [])

  if (loading) return <p>読み込み中...</p>

  return (
    <ul>
      {posts.map((post: any) => (
        <li key={post.id}>{post.title}</li>
      ))}
    </ul>
  )
}
```

この従来方式には次の問題がありました。

| 問題 | 説明 |
|------|------|
| ウォーターフォール | HTML → JavaScript → データ取得 の順に直列で処理される |
| 空の初期表示 | JavaScript が実行されるまでコンテンツが表示されない |
| SEO の問題 | 検索エンジンがコンテンツをクロールできない |
| API キーの露出 | クライアントから外部 API を呼ぶと認証情報が漏洩するリスク |
| ローディング管理 | 各コンポーネントで `loading` / `error` 状態を個別に管理する必要がある |

### Next.js App Router でのデータ取得

App Router では、**サーバーコンポーネント内で直接 `async/await`** を使ってデータを取得できます。これにより上記の問題がすべて解決されます。

```tsx
// app/posts/page.tsx — サーバーコンポーネント（デフォルト）
export default async function PostList() {
  // サーバー上で直接データを取得
  // → SEO 対応済み（HTML に含まれる）
  // → API キーが漏洩しない
  // → ウォーターフォールが発生しない
  const res = await fetch('https://api.example.com/posts')
  const posts = await res.json()

  return (
    <ul>
      {posts.map((post: any) => (
        <li key={post.id}>{post.title}</li>
      ))}
    </ul>
  )
}
```

> **ポイント**: サーバーコンポーネントは `async` 関数として定義できるため、`useEffect` や `useState` を使わずにデータを取得できます。これは React の従来のパラダイムからの大きな転換です。

## 4.2 fetch API の拡張

Next.js は Web 標準の `fetch` API を拡張し、キャッシュとリバリデーション（再検証）のオプションを追加しています。

### 基本構文

```tsx
const res = await fetch(url, {
  cache: 'force-cache' | 'no-store',   // キャッシュ戦略
  next: {
    revalidate: number,                 // 再検証間隔（秒）
    tags: string[],                     // キャッシュタグ
  },
})
```

### キャッシュオプションの比較

| オプション | 動作 | 用途 |
|-----------|------|------|
| `cache: 'force-cache'` | レスポンスをキャッシュする（デフォルト） | 頻繁に変わらないデータ |
| `cache: 'no-store'` | キャッシュしない（毎回取得） | リアルタイムデータ |
| `next: { revalidate: 60 }` | 60秒ごとに再検証 | 適度に更新されるデータ |
| `next: { tags: ['posts'] }` | タグでキャッシュを管理 | オンデマンド再検証 |

### 実例: 各パターンの使い分け

```tsx
// パターン1: 静的データ（ビルド時に取得、変更なし）
// → ブログの固定ページ、会社概要など
const staticData = await fetch('https://api.example.com/about', {
  cache: 'force-cache',  // デフォルトなので省略可能
})

// パターン2: 動的データ（毎回最新を取得）
// → ユーザーのダッシュボード、在庫情報など
const dynamicData = await fetch('https://api.example.com/stock', {
  cache: 'no-store',
})

// パターン3: ISR（一定間隔で再検証）
// → ブログ記事一覧、商品カタログなど
const isrData = await fetch('https://api.example.com/posts', {
  next: { revalidate: 3600 }, // 1時間ごとに再検証
})

// パターン4: タグベースの再検証
// → CMS のコンテンツなど（更新時にオンデマンドで再検証）
const taggedData = await fetch('https://api.example.com/posts', {
  next: { tags: ['posts'] },
})
```

## 4.3 キャッシュの仕組み（4つのレイヤー）

Next.js には4つのキャッシュレイヤーがあります。それぞれの役割を正しく理解することが、パフォーマンス最適化の鍵です。

### キャッシュレイヤー一覧

```
ブラウザ
  ↓ リクエスト
┌─────────────────────────────────────────┐
│  ① Router Cache（クライアント側）         │ ← ブラウザメモリ上
│    └ 訪問済みルートの RSC ペイロード       │
├─────────────────────────────────────────┤
│  ② Full Route Cache（サーバー側）         │ ← ビルド時に生成
│    └ 静的ルートの HTML + RSC ペイロード   │
├─────────────────────────────────────────┤
│  ③ Data Cache（サーバー側）               │ ← fetch 結果の永続キャッシュ
│    └ fetch レスポンスのキャッシュ          │
├─────────────────────────────────────────┤
│  ④ Request Memoization（サーバー側）      │ ← 同一リクエスト内の重複排除
│    └ 同一 URL + オプションの fetch を統合  │
└─────────────────────────────────────────┘
  ↓
外部データソース（API、DB）
```

### ① Request Memoization（リクエストメモ化）

同一のレンダリングリクエスト内で、同じ URL とオプションの `fetch` が複数回呼ばれた場合、**実際のリクエストは1回だけ**実行されます。

```tsx
// app/layout.tsx
async function getUser() {
  const res = await fetch('https://api.example.com/user')
  return res.json()
}

export default async function Layout({ children }: { children: React.ReactNode }) {
  const user = await getUser() // ← 1回目の fetch
  return (
    <html>
      <body>
        <nav>ようこそ、{user.name}さん</nav>
        {children}
      </body>
    </html>
  )
}

// app/page.tsx
async function getUser() {
  const res = await fetch('https://api.example.com/user')
  return res.json()
}

export default async function Page() {
  const user = await getUser() // ← 2回目の fetch（実際にはメモ化される）
  return <p>プロフィール: {user.email}</p>
}
```

この場合、`getUser()` が2か所で呼ばれていますが、実際の HTTP リクエストは **1回だけ** 発生します。これにより、データ取得関数を prop で渡す必要がなく、必要な場所で自由に呼び出せます。

> **注意**: Request Memoization は `GET` リクエストの `fetch` にのみ適用されます。`POST` や `DELETE` には適用されません。

### ② Data Cache（データキャッシュ）

`fetch` のレスポンスをサーバー側で永続的にキャッシュします。リクエストをまたいで再利用されるため、複数のユーザーが同じデータにアクセスする場合に高速です。

```tsx
// Data Cache に保存される（デフォルト動作）
const res = await fetch('https://api.example.com/posts')

// Data Cache をスキップ（毎回外部に問い合わせる）
const res = await fetch('https://api.example.com/posts', {
  cache: 'no-store',
})

// 60秒間はキャッシュを使い、60秒経過後に再検証
const res = await fetch('https://api.example.com/posts', {
  next: { revalidate: 60 },
})
```

### ③ Full Route Cache（フルルートキャッシュ）

ビルド時に静的に生成されたルートの HTML と RSC ペイロードをキャッシュします。`cache: 'no-store'` を使ったり、動的関数（`cookies()`、`headers()` など）を使ったりすると、このキャッシュは無効になります。

### ④ Router Cache（ルーターキャッシュ）

クライアント側（ブラウザ）で訪問済みルートの RSC ペイロードをキャッシュします。ブラウザの「戻る」「進む」ボタンが即座に動作するのはこのキャッシュのおかげです。

### キャッシュの無効化方法

| メソッド | 対象 | 用途 |
|---------|------|------|
| `revalidatePath('/posts')` | 特定パスのキャッシュ | ページ更新時 |
| `revalidateTag('posts')` | タグに紐づくキャッシュ | データ更新時 |
| `router.refresh()` | Router Cache | クライアントからの手動更新 |
| `cache: 'no-store'` | Data Cache | 動的ルートの指定 |

## 4.4 静的生成（SSG）と動的レンダリング（SSR）

### レンダリングの種類

Next.js では、ルートごとに以下のレンダリング方式が自動的に選択されます。

```
┌───────────────────────────────────────────────┐
│          静的レンダリング（Static）              │
│  ビルド時に HTML を生成。CDN から配信。           │
│  → 最も高速                                     │
├───────────────────────────────────────────────┤
│          動的レンダリング（Dynamic）              │
│  リクエストごとに HTML を生成。                   │
│  → リアルタイムデータに最適                       │
├───────────────────────────────────────────────┤
│          ISR（Incremental Static Regeneration） │
│  静的に生成し、一定間隔で再検証。                 │
│  → 静的と動的のハイブリッド                       │
└───────────────────────────────────────────────┘
```

### 静的レンダリング（デフォルト）

何も指定しなければ、Next.js はルートを**静的に生成**しようとします。

```tsx
// app/about/page.tsx — 完全に静的なページ
export default function AboutPage() {
  return (
    <div>
      <h1>会社概要</h1>
      <p>この会社は2020年に設立されました。</p>
    </div>
  )
}
```

### 動的レンダリングへの切り替え

以下のいずれかを使うと、ルートは**動的レンダリング**に切り替わります。

```tsx
import { cookies, headers } from 'next/headers'

export default async function DashboardPage() {
  // 方法1: cookies() を使う → 動的に
  const cookieStore = await cookies()
  const token = cookieStore.get('session')

  // 方法2: headers() を使う → 動的に
  const headerList = await headers()
  const userAgent = headerList.get('user-agent')

  // 方法3: searchParams を使う → 動的に
  // (searchParams は page の props から受け取る)

  // 方法4: fetch で cache: 'no-store' を指定 → 動的に
  const res = await fetch('https://api.example.com/data', {
    cache: 'no-store',
  })

  return <div>ダッシュボード</div>
}
```

### 明示的な動的ルート設定

```tsx
// ルートセグメント設定でレンダリング方式を明示する
export const dynamic = 'force-dynamic'  // 常に動的
// export const dynamic = 'force-static'  // 常に静的
// export const dynamic = 'auto'           // 自動判定（デフォルト）
// export const dynamic = 'error'          // 動的関数使用時にエラー

export default async function Page() {
  // ...
}
```

### ISR（Incremental Static Regeneration）

ISR は「静的に生成するが、一定時間ごとにバックグラウンドで再生成する」方式です。

```tsx
// app/blog/page.tsx — 60秒間隔で再検証される ISR ページ
export const revalidate = 60 // このルート全体のリバリデーション間隔

export default async function BlogPage() {
  const res = await fetch('https://api.example.com/posts')
  const posts = await res.json()

  return (
    <div>
      <h1>ブログ一覧</h1>
      <ul>
        {posts.map((post: any) => (
          <li key={post.id}>{post.title}</li>
        ))}
      </ul>
    </div>
  )
}
```

ISR の動作フロー:

1. 初回アクセス時にページが生成され、キャッシュに保存される
2. 60秒以内の再アクセスにはキャッシュされたページが返される
3. 60秒経過後のアクセスでは、キャッシュされたページを返しつつ、バックグラウンドで再生成
4. 再生成が完了すると、次のリクエストから新しいページが返される

> **ポイント**: ISR は「古いデータを見せつつ、裏で更新する」という stale-while-revalidate パターンです。ユーザーは常に即座にページを閲覧でき、データの鮮度も保たれます。

## 4.5 動的ルートの静的生成 — generateStaticParams

動的ルート（`[slug]` など）でも、ビルド時にパラメータの一覧を提供することで静的生成が可能です。

```tsx
// app/blog/[slug]/page.tsx

// ビルド時に生成するパスの一覧を返す
export async function generateStaticParams() {
  const res = await fetch('https://api.example.com/posts')
  const posts = await res.json()

  return posts.map((post: any) => ({
    slug: post.slug,
  }))
  // 例: [{ slug: 'hello-world' }, { slug: 'next-js-guide' }]
}

export default async function BlogPost({
  params,
}: {
  params: Promise<{ slug: string }>
}) {
  const { slug } = await params
  const res = await fetch(`https://api.example.com/posts/${slug}`)
  const post = await res.json()

  return (
    <article>
      <h1>{post.title}</h1>
      <div>{post.content}</div>
    </article>
  )
}
```

### dynamicParams の制御

```tsx
// generateStaticParams で生成されなかったパスの扱い
export const dynamicParams = true  // デフォルト: オンデマンドで生成
// export const dynamicParams = false // 生成されなかったパスは 404

export async function generateStaticParams() {
  // ...
}
```

| `dynamicParams` | 動作 |
|-----------------|------|
| `true`（デフォルト） | 未生成のパスはリクエスト時に動的生成し、キャッシュ |
| `false` | 未生成のパスは 404 を返す |

## 4.6 データベースへの直接アクセス

サーバーコンポーネントでは、`fetch` を使わずにデータベースに直接アクセスすることもできます。

### Prisma の例

```tsx
// lib/prisma.ts — Prisma クライアントの初期化
import { PrismaClient } from '@prisma/client'

const globalForPrisma = globalThis as unknown as {
  prisma: PrismaClient | undefined
}

export const prisma = globalForPrisma.prisma ?? new PrismaClient()

if (process.env.NODE_ENV !== 'production') globalForPrisma.prisma = prisma
```

```tsx
// app/users/page.tsx — データベースに直接アクセス
import { prisma } from '@/lib/prisma'

export default async function UsersPage() {
  const users = await prisma.user.findMany({
    orderBy: { createdAt: 'desc' },
    take: 20,
  })

  return (
    <div>
      <h1>ユーザー一覧</h1>
      <ul>
        {users.map(user => (
          <li key={user.id}>
            {user.name} ({user.email})
          </li>
        ))}
      </ul>
    </div>
  )
}
```

> **注意**: データベースに直接アクセスする場合、Next.js の `fetch` キャッシュは適用されません。キャッシュが必要な場合は、React の `cache` 関数や `unstable_cache`（Next.js 提供）を使用します。

### React の cache 関数によるメモ化

```tsx
import { cache } from 'react'
import { prisma } from '@/lib/prisma'

// 同一リクエスト内でのメモ化（Request Memoization と同等）
export const getUser = cache(async (id: string) => {
  const user = await prisma.user.findUnique({
    where: { id },
  })
  return user
})
```

```tsx
// layout.tsx と page.tsx の両方で getUser(id) を呼んでも、
// 実際のDBクエリは1回だけ実行される
```

## 4.7 並列データ取得

複数のデータを取得する際は、**逐次取得（ウォーターフォール）を避け、並列取得** を心がけましょう。

### 悪い例: ウォーターフォール

```tsx
// 悪い例: 直列にデータを取得（合計 3秒）
export default async function Page() {
  const user = await fetchUser()        // 1秒
  const posts = await fetchPosts()      // 1秒
  const comments = await fetchComments() // 1秒
  // 合計: 3秒かかる

  return <div>...</div>
}
```

### 良い例: 並列取得

```tsx
// 良い例: Promise.all で並列取得（最大 1秒）
export default async function Page() {
  const [user, posts, comments] = await Promise.all([
    fetchUser(),       // 1秒
    fetchPosts(),      // 1秒（同時に開始）
    fetchComments(),   // 1秒（同時に開始）
  ])
  // 合計: 最も遅いものに合わせて約1秒

  return <div>...</div>
}
```

### Suspense を使った段階的表示

完全な並列取得ではなく、**取得完了順に表示する**パターンも強力です。

```tsx
import { Suspense } from 'react'

export default function DashboardPage() {
  return (
    <div>
      <h1>ダッシュボード</h1>

      {/* 軽いデータ: すぐ表示 */}
      <Suspense fallback={<p>概要を読み込み中...</p>}>
        <Summary />
      </Suspense>

      {/* 重いデータ: 取得完了後に表示 */}
      <Suspense fallback={<p>グラフを読み込み中...</p>}>
        <Charts />
      </Suspense>

      {/* さらに重いデータ */}
      <Suspense fallback={<p>詳細を読み込み中...</p>}>
        <DetailedReport />
      </Suspense>
    </div>
  )
}

async function Summary() {
  const data = await fetch('https://api.example.com/summary')
  const summary = await data.json()
  return <div>合計: {summary.total}</div>
}

async function Charts() {
  const data = await fetch('https://api.example.com/charts')
  const charts = await data.json()
  return <div>グラフ表示: {charts.length}件</div>
}
```

> **ポイント**: Suspense を使うと、各データの取得が独立して行われ、完了した順番にストリーミングされます。ユーザーは待ち時間が最小化された体験を得られます。

## 4.8 オンデマンド再検証

ISR の時間ベースの再検証に加えて、**特定のイベント（データ更新など）をトリガーにキャッシュを無効化** できます。

### revalidatePath — パスベースの再検証

```tsx
// app/actions.ts
'use server'

import { revalidatePath } from 'next/cache'

export async function createPost(formData: FormData) {
  // データベースに記事を保存
  await savePostToDatabase(formData)

  // /blog ページのキャッシュを無効化
  revalidatePath('/blog')

  // 特定の記事ページも無効化
  revalidatePath('/blog/[slug]', 'page')

  // レイアウトを含めて無効化
  revalidatePath('/blog', 'layout')
}
```

### revalidateTag — タグベースの再検証

```tsx
// データ取得時にタグを付与
const res = await fetch('https://api.example.com/posts', {
  next: { tags: ['posts'] },
})

const res2 = await fetch(`https://api.example.com/posts/${id}`, {
  next: { tags: ['posts', `post-${id}`] },
})
```

```tsx
// app/actions.ts
'use server'

import { revalidateTag } from 'next/cache'

export async function updatePost(id: string) {
  // データベースを更新
  await updatePostInDatabase(id)

  // 'posts' タグのキャッシュをすべて無効化
  revalidateTag('posts')

  // または特定の記事のキャッシュだけ無効化
  revalidateTag(`post-${id}`)
}
```

> **ベストプラクティス**: タグベースの再検証は、パスベースよりも**きめ細かくキャッシュを制御**できます。特にCMSと連携する場合は、コンテンツの種類ごとにタグを設計すると効率的です。

## 4.9 エラーハンドリング

データ取得時のエラーは、適切にハンドリングすることが重要です。

### fetch のエラーハンドリング

```tsx
// app/posts/page.tsx
export default async function PostsPage() {
  try {
    const res = await fetch('https://api.example.com/posts')

    if (!res.ok) {
      // HTTP エラー（4xx, 5xx）
      throw new Error(`API エラー: ${res.status}`)
    }

    const posts = await res.json()
    return (
      <ul>
        {posts.map((post: any) => (
          <li key={post.id}>{post.title}</li>
        ))}
      </ul>
    )
  } catch (error) {
    // error.tsx にエラーを伝播させる
    throw error
  }
}
```

### not-found と組み合わせる

```tsx
import { notFound } from 'next/navigation'

export default async function PostPage({
  params,
}: {
  params: Promise<{ slug: string }>
}) {
  const { slug } = await params
  const res = await fetch(`https://api.example.com/posts/${slug}`)

  if (res.status === 404) {
    notFound() // not-found.tsx を表示
  }

  if (!res.ok) {
    throw new Error('データの取得に失敗しました')
  }

  const post = await res.json()
  return <article>{post.title}</article>
}
```

## 4.10 よくある間違い

### 間違い1: クライアントコンポーネントで fetch してしまう

```tsx
// 悪い例
'use client'
import { useEffect, useState } from 'react'

export default function Posts() {
  const [posts, setPosts] = useState([])
  useEffect(() => {
    fetch('/api/posts').then(r => r.json()).then(setPosts)
  }, [])
  // → SEO に不利、ウォーターフォール発生
}

// 良い例: サーバーコンポーネントで取得
export default async function Posts() {
  const res = await fetch('https://api.example.com/posts')
  const posts = await res.json()
  return <ul>{posts.map(/* ... */)}</ul>
}
```

### 間違い2: 不必要に cache: 'no-store' を使う

```tsx
// 悪い例: すべてのfetchで no-store
const data = await fetch(url, { cache: 'no-store' })
// → すべてのリクエストで外部APIにアクセスし、遅くなる

// 良い例: データの性質に応じて使い分ける
// 変更頻度が低いデータは ISR
const posts = await fetch(url, { next: { revalidate: 3600 } })
// リアルタイム性が必要なデータのみ no-store
const stock = await fetch(url, { cache: 'no-store' })
```

### 間違い3: ウォーターフォールに気づかない

```tsx
// 悪い例: 順番に await している
const user = await getUser()
const posts = await getPostsByUser(user.id) // user に依存
const tags = await getAllTags()              // user に依存しない！

// 良い例: 独立したデータは並列取得
const userPromise = getUser()
const tagsPromise = getAllTags() // user に依存しないので並列化
const user = await userPromise
const [posts, tags] = await Promise.all([
  getPostsByUser(user.id), // user に依存するのでここで await
  tagsPromise,
])
```

## 4.11 現場でのベストプラクティス

### 1. キャッシュ戦略の設計表を作る

プロジェクト開始時に、データの種類ごとにキャッシュ戦略を決めておくと統一感のある設計になります。

| データ | 更新頻度 | 戦略 | revalidate |
|--------|---------|------|-----------|
| 会社概要 | ほぼなし | 静的生成 | - |
| ブログ記事一覧 | 週数回 | ISR | 3600（1時間） |
| 商品在庫 | リアルタイム | 動的 (no-store) | - |
| ユーザープロフィール | ログイン時 | 動的 (cookies) | - |
| カテゴリマスタ | 月1回 | ISR | 86400（1日） |

### 2. データ取得関数を集約する

```
lib/
├── api/
│   ├── posts.ts    ← 記事関連のデータ取得
│   ├── users.ts    ← ユーザー関連のデータ取得
│   └── products.ts ← 商品関連のデータ取得
```

```tsx
// lib/api/posts.ts
export async function getPosts() {
  const res = await fetch('https://api.example.com/posts', {
    next: { tags: ['posts'], revalidate: 3600 },
  })
  if (!res.ok) throw new Error('記事の取得に失敗しました')
  return res.json()
}

export async function getPost(slug: string) {
  const res = await fetch(`https://api.example.com/posts/${slug}`, {
    next: { tags: ['posts', `post-${slug}`] },
  })
  if (!res.ok) return null
  return res.json()
}
```

### 3. 環境変数で API URL を管理する

```bash
# .env.local
API_BASE_URL=https://api.example.com
```

```tsx
// lib/api/client.ts
const BASE_URL = process.env.API_BASE_URL

export async function apiFetch(path: string, options?: RequestInit) {
  const res = await fetch(`${BASE_URL}${path}`, options)
  if (!res.ok) {
    throw new Error(`API Error: ${res.status} ${res.statusText}`)
  }
  return res.json()
}
```

## 4.12 ポイントまとめ

本章で学んだこと:

1. **サーバーコンポーネント**では `async/await` で直接データを取得でき、`useEffect` は不要です
2. Next.js の `fetch` は**キャッシュとリバリデーション**が拡張されています
3. **4つのキャッシュレイヤー**（Request Memoization、Data Cache、Full Route Cache、Router Cache）を理解することが重要です
4. **静的生成（SSG）**、**動的レンダリング（SSR）**、**ISR** はルートの設定により自動的に切り替わります
5. `generateStaticParams` で動的ルートも静的に生成できます
6. データの**並列取得**と **Suspense** を組み合わせることで、最適なユーザー体験を実現できます
7. `revalidatePath` / `revalidateTag` でオンデマンドにキャッシュを無効化できます

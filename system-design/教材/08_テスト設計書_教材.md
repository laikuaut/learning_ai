# 第8章：テスト設計書

## この章のゴール

- テスト設計書の目的と構成を理解する
- テストレベル（単体・結合・システム・受入）の違いを理解する
- テストケースの書き方を身につける
- テスト技法（同値分割・境界値分析・デシジョンテーブル）を理解する
- テスト設計と要件・設計書のトレーサビリティを理解する

---

## 8.1 テスト設計書とは

### テスト設計書の位置づけ

テスト設計書は、**「何を、どのようにテストするか」を定義する文書**です。V字モデルの右側に位置し、各設計工程に対応するテストを計画します。

```
【V字モデルとテスト設計の対応】

要件定義書 ──────────────────────── 受入テスト設計
  │        テスト設計書に基づく          │
  ▼                                     ▲
  基本設計書 ─────────────────── 結合テスト設計
    │                                   │
    ▼                                   ▲
    詳細設計書 ──────────────── 単体テスト設計
      │                               │
      ▼                               ▲
      ┌──────────────────────┐
      │    実装（コーディング）    │
      └──────────────────────┘
```

### テスト設計書の構成

```
テスト設計書の目次（例）
━━━━━━━━━━━━━━━━━━━━━━━━
1. テスト計画
   1.1 テスト目的
   1.2 テスト範囲（スコープ）
   1.3 テストレベル
   1.4 テスト環境
   1.5 テストスケジュール
   1.6 合格基準（Exit Criteria）

2. テスト設計
   2.1 テスト観点
   2.2 テスト技法

3. テストケース
   3.1 単体テストケース
   3.2 結合テストケース
   3.3 システムテストケース

4. テストデータ

5. 不具合管理
━━━━━━━━━━━━━━━━━━━━━━━━
```

## 8.2 テストレベル

### 4つのテストレベル

```
【テストレベルの階層】

┌──────────────────────────────────────────────────┐
│ レベル4：受入テスト（Acceptance Test）              │
│  実施者：顧客・エンドユーザー                       │
│  目的  ：ビジネス要件を満たしているか検証            │
│  対象  ：要件定義書の要件                           │
├──────────────────────────────────────────────────┤
│ レベル3：システムテスト（System Test）               │
│  実施者：テスターチーム                             │
│  目的  ：システム全体が仕様通りに動作するか検証       │
│  対象  ：基本設計書の仕様（性能、セキュリティ含む）   │
├──────────────────────────────────────────────────┤
│ レベル2：結合テスト（Integration Test）              │
│  実施者：開発チーム                                │
│  目的  ：モジュール間の連携が正しく動作するか検証     │
│  対象  ：基本設計書のインタフェース仕様              │
├──────────────────────────────────────────────────┤
│ レベル1：単体テスト（Unit Test）                     │
│  実施者：開発者                                    │
│  目的  ：個々のモジュールが仕様通りに動作するか検証   │
│  対象  ：詳細設計書の処理ロジック                    │
└──────────────────────────────────────────────────���
```

### 各テストレベルの特徴

| 項目 | 単体テスト | 結合テスト | システムテスト | 受入テスト |
|---|---|---|---|---|
| **テスト対象** | 関数・メソッド | モジュール間連携 | システム全体 | 業務シナリオ |
| **実施者** | 開発者 | 開発チーム | テスター | 顧客 |
| **テスト技法** | ホワイトボックス | グレーボックス | ブラックボックス | ブラックボックス |
| **自動化** | 高い | 中〜高 | 中 | 低 |
| **実行速度** | 高速 | 中速 | 低速 | 低速 |
| **検出バグ** | コーディングミス | IF不整合 | 仕様漏れ | 要件漏れ |

## 8.3 テスト観点

### テスト観点の整理

テスト観点とは、「何の視点でテストするか」を整理したものです。

```
【テスト観点の分類】

┌───────────────────────────────────────────────┐
│ 機能テスト                                     │
│  ├─ 正常系：仕様通りの入力で期待通り動作するか   │
│  ├─ 異常系：不正入力やエラー時に正しく対処するか  │
│  ├─ 境界値：境界条件の前後で正しく動作するか      │
│  └─ 組み合わせ：複数条件の組み合わせで正しいか    │
├───────────────────────────────────────────────┤
│ 非機能テスト                                    │
│  ├─ 性能：応答時間、スループット                  │
│  ├─ 負荷：高負荷時の挙動                        │
│  ├─ セキュリティ：脆弱性、認証・認可              │
│  ├─ ユーザビリティ：操作性、わかりやすさ          │
│  └─ 互換性：ブラウザ、OS、デバイス               │
├───────────────────────────────────────────────┤
│ 回帰テスト                                     │
│  └─ 既存機能への影響：変更後も既存機能が正常か     │
└───────────────────────────────────────────────┘
```

## 8.5 テスト技法

### 同値分割法

入力値を「同じ結果になるグループ（同値クラス）」に分割し、各グループから代表値をテストする技法です。

```
【同値分割法の例：年齢入力（0〜150歳が有効）】

       無効          有効            無効
  ◀──────────┤───────────────────┤──────────▶
        -1    0                 150   151

テストケース:
  ・無効クラス1：-1（0未満）
  ・有効クラス：75（0〜150の範囲内の代表値）
  ・無効クラス2：151（150超過）
```

### 境界値分析

同値クラスの境界（端の値）を重点的にテストする技法です。バグは境界付近で発生しやすいためです。

```
【境界値分析の例：年齢入力（0〜150歳が有効）】

       無効      │  有効          │  無効
  ◀─────────────┤────────────────┤──────────▶
            -1  0  1         149 150 151

テストケース:
  ・-1（境界の直前：無��）
  ・ 0（境界値：有効）
  ・ 1（境界の直後：有効）
  ・149（境界の直前：有効）
  ・150（境界値：有効）
  ・151（境界の直後：無効）
```

### デシジョンテーブル

複数の条件の組み合わせと、それに対する動作を表にまとめる技法です。

```
【デシジョンテーブルの例：ECサイトの送料計算】

条件:
  ・会員ランク（ゴールド / 一般）
  ・注文金額（5,000円以上 / 5,000円未満）
  ・離島フラグ（離島 / 離島以外）

┌───────────────┬────┬────┬────┬────┬────┬────┬────┬────┐
│ 条件           │ R1 │ R2 │ R3 │ R4 │ R5 │ R6 │ R7 │ R8 │
├───────────────┼────┼────┼────┼────┼────┼────┼────┼────┤
│ ゴールド会員    │ Y  │ Y  │ Y  │ Y  │ N  │ N  │ N  │ N  │
│ 5,000円以上    │ Y  │ Y  │ N  │ N  │ Y  │ Y  │ N  │ N  │
��� 離島          │ Y  │ N  │ Y  │ N  │ Y  │ N  │ Y  │ N  │
├───────────────┼────┼────┼────┼────┼────┼────┼────┼────┤
│ 送料無料       │    │ ○  │    │ ○  │    │ ○  │    │    │
│ 送料500円      │    │    │    │    │    │    │ ○  │    │
│ 離島追加料金   │ ○  │    │ ○  │    │ ○  │    │    │    │
│ 送料500円+離島 │    │    │    │    │    │    │    │    │
├───────────────┼────┼────┼────┼────┼────┼────┼────┼────┤
│ 結果の送料     │離島 │ 0円│離島 │ 0円│離島 │ 0円│500 │500 │
│               │のみ │    │のみ │    │のみ │    │円  │円  │
└───────────────┴────┴────┴────┴────┴────┴────┴────┴────┘

R1: ゴールド + 5000円以上 + 離島 → 離島追加料金の��
R2: ゴールド + 5000円以上 + 離島以外 → 送料無料
R3: ゴールド + 5000円未満 + 離島 → 離島追加料金のみ
R4: ゴールド + 5000円未満 + 離島以外 → 送料無料
R5: 一般 + 5000円以上 + 離島 → 離島追加料金のみ
R6: 一般 + 5000円以上 + 離島以外 → 送料無料
R7: 一般 + 5000円未満 + 離島 → 送料500円（離島追加なし※要確認）
R8: 一般 + 5000円未満 + 離島以外 → 送料500円
```

> **実務でのポイント**: デシジョンテーブルを作ると、仕様の曖昧な部分や考慮漏れが明確になります。上の例で「一般会員 + 5000円未満 + 離島」のケース（R7）は、送料500円に離島追加料金を足すのか、足さないのかが曖昧です。テスト設計の段階でこうした問題を発見できることが、テスト技法の価値です。

## 8.7 合格基準と不具合管理

### テストの合格基準

```
【合格基準（Exit Criteria）の例】

┌──────────────────────────────────────────────────┐
│ 単体テストの合格基準                               │
│  ・テストケース消化率：100%                         │
│  ・テスト成功率：100%（全ケースパス）               │
│  ・コードカバレッジ：80%以上（C1カバレッジ）         │
├──────────────────────────────────────────────────┤
│ 結合テストの合格基準                               │
│  ・テストケース消化率：100%                         │
│  ・重大バグ（Critical/Major）：0件                 │
│  ・未解決バグ：Minor以下のみ                        │
├──────────────────────────────────────────────────┤
│ システムテストの合格基準                            │
│  ・テストケース消化率：100%                         │
│  ・重大バグ（Critical/Major）：0件                 │
│  ・性能要件の達成：全項目合格                       │
│  ・セキュリティテスト：重大脆弱性0件                │
├──────────────────────────────────────────────────┤
│ 受入テストの合格基準                               │
│  ・全業務シナリオの実行完了                         │
│  ・顧客承認                                       │
└──────────────────────────────────────────────────┘
```

### 不具合の重大度分類

```
【不具合の重大度（Severity）】

| 重大度     | 定義                                    | 例                          |
|-----------|----------------------------------------|-----------------------------|
| Critical  | システムが停止する、データ損失の可能性     | サーバーダウン、データ消失     |
| Major     | 主要機能が使用不可、代替手段なし          | ログインできない、注文できない |
| Minor     | 機能が一部制限されるが、代替手段あり       | 検索のソートが効かない        |
| Trivial   | 見た目の問題、使用に支障なし              | 文字の誤字、レイアウトのズレ  |
```

### 不具合チケットの記述例

```
【不具合チケット（例）】

バグID     ：BUG-042
タイトル   ：商品検索で価格のソートが降順にならない
重大度     ：Minor
テストケース：TC-SRCH-005
発見日     ：2026-04-01
報告者     ：テスター 佐藤

【再現手順】
1. 商品一覧画面（SCR-004）を表示する
2. 並び替えプルダウンで「価格が高い順」を選択する
3. 商品一覧が表示される

【期待結果】
価格の降順（高い → 安い）で商品が並ぶ

【実際の結果】
価格の昇順（安い → 高い）で商品が並ぶ
（「価格が安い順」と同じ結果になっている）

【環境】
ブラウザ：Chrome 120
OS：Windows 11
テスト環境：staging
```

## 基本問題

### 問題1：Zero-Shot / One-Shot / Few-Shotの判別

以下のプロンプトがZero-Shot、One-Shot、Few-Shotのどれか判定してください。

A:
```
以下のテキストを感情分析してください。
テキスト：「この映画は期待はずれだった」
```

B:
```
テキストの感情を分析します。
例：「最高の一日だった！」→ ポジティブ
分析対象：「この映画は期待はずれだった」→
```

C:
```
テキストの感情を分析します。
例1：「最高の一日だった！」→ ポジティブ
例2：「最悪の体験だった」→ ネガティブ
例3：「普通でした」→ ニュートラル
分析対象：「この映画は期待はずれだった」→
```

<details>
<summary>解答例</summary>

```
A: Zero-Shot
   → 例が0個。指示のみで回答を求めている。

B: One-Shot
   → 例が1個。1つの入出力例を示してパターンを伝えている。

C: Few-Shot
   → 例が3個。複数の入出力例でパターンを示している。
     3つの感情カテゴリすべてをカバーしている点が良い。
```
</details>

### 問題2：Few-Shotプロンプトの作成（分類タスク）

以下のカテゴリでGitコミットメッセージを分類するFew-Shotプロンプトを作成してください。

カテゴリ：feature / bugfix / refactor / docs / test

各カテゴリに1つずつ例を作成し、最後に分類対象のコミットメッセージを含めてください。

<details>
<summary>解答例</summary>

```
以下のGitコミットメッセージをカテゴリに分類してください。
カテゴリ：feature / bugfix / refactor / docs / test

### 例1
コミット：「Add user registration with email verification」
カテゴリ：feature

### 例2
コミット：「Fix null pointer exception in payment processing」
カテゴリ：bugfix

### 例3
コミット：「Extract validation logic into separate module」
カテゴリ：refactor

### 例4
コミット：「Update API documentation for v2 endpoints」
カテゴリ：docs

### 例5
コミット：「Add unit tests for cart calculation service」
カテゴリ：test

### 分類対象
コミット：「Rename variables for consistency across auth module」
カテゴリ：

期待される回答：refactor
（変数名の一貫性のためのリネームはリファクタリングに分類される）
```
</details>

### 問題3：例の品質改善

以下のFew-Shotプロンプトの問題点を指摘し、改善してください。

```
問い合わせメールを分類してください。

例1：「システムがエラーで動きません」→ 技術サポート
例2：「ログインできません」→ 技術サポート
例3：「画面が白くなります」→ 技術サポート

分類対象：「来月の請求書を送ってほしい」→
```

<details>
<summary>解答例</summary>

```
■ 問題点
1. 例がすべて「技術サポート」に偏っている（多様性の欠如）
2. 他のカテゴリの例がないため、LLMはすべてを「技術サポート」に
   分類する傾向になる
3. カテゴリの全体像が示されていない

■ 改善後

問い合わせメールを以下のカテゴリに分類してください。
カテゴリ：技術サポート / 請求・契約 / 機能要望 / 使い方の質問

例1：「システムがエラーで動きません」
→ 技術サポート

例2：「先月の請求額が想定より高いのですが」
→ 請求・契約

例3：「ダークモードに対応してほしいです」
→ 機能要望

例4：「CSVエクスポートはどこからできますか？」
→ 使い方の質問

分類対象：「来月の請求書を送ってほしい」
→

■ 改善ポイント
- 全カテゴリを網羅する例を配置した（多様性）
- カテゴリの一覧を事前に明示した
- 各カテゴリの典型的な例を選んだ（代表性）
```
</details>

### 問題5：文体変換のFew-Shot設計

技術的なリリースノートを、一般ユーザー向けの「新機能のお知らせ」に変換するFew-Shotプロンプトを設計してください。

要件：
- 技術用語を排除する
- ユーザーにとってのメリットを中心に記述する
- 親しみやすいトーンにする
- 3つの例を含める

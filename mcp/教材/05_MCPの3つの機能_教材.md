# 第5章：MCPの3つの機能（Tools・Resources・Prompts）

## 学習目標

- MCPが提供する3つの機能（Tools・Resources・Prompts）の違いを理解する
- 各機能の設計パターンとベストプラクティスを習得する
- 3つの機能を組み合わせた実用的なサーバーを構築できる
- 「この機能はTool? Resource? Prompt?」を正しく判断できる

---

## 5.1 3つの機能の使い分け

MCPサーバーは、AIに対して3種類の機能を公開できます。それぞれ**誰が起動するか**と**何をするか**が異なります。

### 機能比較表

| 特性 | Tools（ツール） | Resources（リソース） | Prompts（プロンプト） |
|---|---|---|---|
| **誰が使うか** | AIが自動で呼び出す | AIが自動で読み取る | ユーザーが明示的に選ぶ |
| **役割** | 動的な操作・副作用あり | データの読み取り | 会話テンプレート |
| **例え** | リモコンのボタン | 本棚の本 | 料理のレシピ |
| **具体例** | DB更新、API呼び出し | ファイル内容、DB参照 | コードレビュー依頼 |

### アーキテクチャ図

```
┌─────────────────────────────────────────────┐
│              ホストアプリ（Claude等）          │
│  ┌───────────────────────────────────────┐   │
│  │           MCPクライアント              │   │
│  │                                       │   │
│  │  ① Tools    → AIが判断して呼び出す     │   │
│  │  ② Resources → AIが判断して読み込む    │   │
│  │  ③ Prompts  → ユーザーが選択して起動   │   │
│  └───────────┬───────────────────────────┘   │
└──────────────┼───────────────────────────────┘
               │ JSON-RPC
┌──────────────┼───────────────────────────────┐
│  MCPサーバー │                                │
│  ┌───────────┴───────────────────────────┐   │
│  │  @mcp.tool()      → 動的操作          │   │
│  │  @mcp.resource()  → データ提供        │   │
│  │  @mcp.prompt()    → テンプレート提供  │   │
│  └───────────────────────────────────────┘   │
└──────────────────────────────────────────────┘
```

### 判断の基本原則

- **何かを「実行」する** → Tools
- **何かを「読む」だけ** → Resources
- **会話の「型」を提供する** → Prompts

> **ポイントまとめ**
> - Tools = AIが呼ぶ「操作」、Resources = AIが読む「データ」、Prompts = ユーザーが選ぶ「テンプレート」
> - 3つの機能は排他的ではなく、1つのサーバーで組み合わせて使うのが一般的です

## 5.2 Toolsの詳細設計

ツール（Tools）は、AIが判断して呼び出す**動的な操作**です。ファイル作成、API呼び出し、データベース更新など、副作用を伴う処理に使います。

### 基本的なToolの定義

```python
from mcp.server.fastmcp import FastMCP

mcp = FastMCP("task-manager")

@mcp.tool()
def create_task(title: str, priority: str = "medium") -> str:
    """新しいタスクを作成します。

    Args:
        title: タスクのタイトル（必須）
        priority: 優先度。"high", "medium", "low" のいずれか（デフォルト: medium）
    """
    # 実際にはDBに保存する処理
    task_id = "task_001"
    return f"タスク '{title}'（優先度: {priority}）を作成しました。ID: {task_id}"
```

### description の書き方 ―― AIの判断を左右する

`description`（docstring）は、AIがツールを呼ぶかどうかを判断する最重要情報です。

```python
# 悪い例：説明が曖昧
@mcp.tool()
def process(data: str) -> str:
    """データを処理します。"""  # AIはいつ使うべきか判断できない
    pass

# 良い例：具体的で明確
@mcp.tool()
def search_products(query: str, max_results: int = 10) -> str:
    """商品データベースから商品を検索します。

    商品名、カテゴリ、説明文に対してキーワード検索を行います。
    ユーザーが商品を探している場合に使用してください。

    Args:
        query: 検索キーワード（部分一致）
        max_results: 最大取得件数（1〜100、デフォルト: 10）
    """
    pass
```

### 入力スキーマ設計のベストプラクティス

型アノテーションを活用すると、MCPが自動的にJSON Schemaを生成します。

```python
from typing import Literal

@mcp.tool()
def update_task(
    task_id: str,
    status: Literal["todo", "in_progress", "done"],
    assignee: str | None = None,
) -> str:
    """タスクのステータスを更新します。

    Args:
        task_id: 更新対象のタスクID（例: "task_001"）
        status: 新しいステータス
        assignee: 担当者名（省略可）
    """
    return f"タスク {task_id} を {status} に更新しました"
```

上記は以下のJSON Schemaに変換されます。

```json
{
  "name": "update_task",
  "inputSchema": {
    "type": "object",
    "properties": {
      "task_id": { "type": "string" },
      "status": { "enum": ["todo", "in_progress", "done"] },
      "assignee": { "type": "string", "nullable": true }
    },
    "required": ["task_id", "status"]
  }
}
```

### 冪等性（Idempotency）と副作用の管理

冪等性（idempotency）とは、同じ操作を何度実行しても結果が変わらない性質のことです。

```python
# 冪等なツール：何度呼んでも安全
@mcp.tool()
def set_user_email(user_id: str, email: str) -> str:
    """ユーザーのメールアドレスを設定します。"""
    # 同じ値で何度呼んでも結果は同じ
    db.users.update(user_id, {"email": email})
    return f"ユーザー {user_id} のメールを {email} に設定しました"

# 冪等でないツール：呼ぶたびに結果が変わる
@mcp.tool()
def send_email(to: str, subject: str, body: str) -> str:
    """メールを送信します。

    注意: このツールは冪等ではありません。呼び出すたびにメールが送信されます。
    重複送信を避けるため、確認してから実行してください。
    """
    # 呼ぶたびにメールが送信される
    mailer.send(to=to, subject=subject, body=body)
    return "メールを送信しました"
```

### 成功/エラーの返し方（isError フラグ）

ツールの戻り値でエラーを明示すると、AIが適切にリカバリーできます。

```python
@mcp.tool()
def delete_task(task_id: str) -> str:
    """タスクを削除します。"""
    task = db.find_task(task_id)
    if task is None:
        # エラーを返す ―― 例外を投げるのではなくメッセージで返す
        raise mcp.ToolError(f"タスク {task_id} が見つかりません")

    db.delete_task(task_id)
    return f"タスク {task_id} を削除しました"
```

MCPプロトコルでは、ツールの応答に `isError: true` を含めることで、AIにエラーを伝えます。FastMCPでは例外を発生させるとこのフラグが自動的に設定されます。

```
成功時の応答:
  { "content": [{"type": "text", "text": "削除しました"}], "isError": false }

エラー時の応答:
  { "content": [{"type": "text", "text": "見つかりません"}], "isError": true }
```

### よくある間違い

| 間違い | 正しい方法 |
|---|---|
| descriptionを空にする | AIが判断できるよう具体的に書く |
| 全パラメータを必須にする | 省略可能なものはデフォルト値を設定する |
| エラー時にPython例外だけ投げる | `ToolError` で明示的にエラーメッセージを返す |
| 1つのツールに多機能を詰め込む | 1ツール1責任で分割する |

> **ポイントまとめ**
> - descriptionはAIへの指示書。具体的・明確に書くことが最重要
> - 型アノテーションでJSON Schemaが自動生成される
> - 冪等でないツールはdescriptionに明記し、AIが慎重に扱えるようにする
> - エラーは `isError` フラグで返し、AIのリカバリーを助ける

## 5.3 Resourcesの詳細設計

リソース（Resources）は、AIが読み取る**データソース**です。URI形式でアクセスし、ファイル内容やデータベースの参照結果などを提供します。

### 静的リソース

URIが固定のリソースです。

```python
from mcp.server.fastmcp import FastMCP

mcp = FastMCP("docs-server")

@mcp.resource("config://app/settings")
def get_app_settings() -> str:
    """アプリケーションの現在の設定を返します。"""
    import json
    settings = {
        "app_name": "MyApp",
        "version": "2.1.0",
        "debug": False,
        "max_connections": 100,
    }
    return json.dumps(settings, indent=2, ensure_ascii=False)
```

### 動的リソース（テンプレート）

URIにパラメータを含むリソースです。`{変数名}` でテンプレートを定義します。

```python
@mcp.resource("users://{user_id}/profile")
def get_user_profile(user_id: str) -> str:
    """指定ユーザーのプロフィールを返します。"""
    # 実際にはDBから取得
    profiles = {
        "u001": {"name": "田中太郎", "role": "engineer"},
        "u002": {"name": "佐藤花子", "role": "designer"},
    }
    profile = profiles.get(user_id)
    if profile is None:
        return f"ユーザー {user_id} が見つかりません"
    import json
    return json.dumps(profile, ensure_ascii=False, indent=2)
```

### MIMEタイプの指定

リソースが返すデータの種類をMIMEタイプで宣言できます。

```python
import json

# テキストデータ（デフォルト）
@mcp.resource("reports://monthly", mime_type="text/plain")
def get_monthly_report() -> str:
    """月次レポートをテキストで返します。"""
    return "2026年3月 月次レポート\n売上: 1,200万円\n..."

# JSONデータ
@mcp.resource("data://products", mime_type="application/json")
def get_products() -> str:
    """商品一覧をJSONで返します。"""
    products = [
        {"id": 1, "name": "ノートPC", "price": 98000},
        {"id": 2, "name": "マウス", "price": 3500},
    ]
    return json.dumps(products, ensure_ascii=False)

# バイナリデータ（画像など）
@mcp.resource("images://logo", mime_type="image/png")
def get_logo() -> bytes:
    """ロゴ画像をバイナリで返します。"""
    with open("logo.png", "rb") as f:
        return f.read()
```

### リソース更新通知とサブスクリプション

リソースの内容が変わったことをクライアントに通知する仕組みです。

```
クライアント → サーバー:  resources/subscribe { uri: "data://products" }
  （サーバーが変更を検知）
サーバー → クライアント:  notifications/resources/updated { uri: "data://products" }
クライアント → サーバー:  resources/read { uri: "data://products" }
  （最新データを再取得）
```

FastMCPでは以下のように通知を送れます。

```python
# サーバー側でリソースが更新されたときに通知
async def on_product_updated():
    """商品データが更新されたときに呼ばれる関数。"""
    await mcp.notify_resource_updated("data://products")
```

### よくある間違い

| 間違い | 正しい方法 |
|---|---|
| リソースで副作用のある処理を行う | 副作用がある処理はToolsで行う |
| URI設計が不統一 | `スキーマ://カテゴリ/識別子` の形式で統一する |
| 巨大なデータを一括で返す | ページネーションやフィルタを検討する |
| MIMEタイプを指定しない | 適切なMIMEタイプを明記する |

> **ポイントまとめ**
> - 静的リソース = 固定URI、動的リソース = テンプレートURI
> - MIMEタイプでテキスト・JSON・バイナリを区別する
> - リソースは「読み取り専用」が原則。副作用はToolsに任せる
> - 変更通知（subscription）で、クライアントが常に最新データを取得できる

## 5.4 Promptsの詳細設計

プロンプト（Prompts）は、ユーザーが明示的に選択する**会話テンプレート**です。定型的なワークフローをテンプレート化して、ユーザーの入力を効率化します。

### 基本的なPromptの定義

```python
from mcp.server.fastmcp import FastMCP
from mcp.server.fastmcp.prompts import base

mcp = FastMCP("code-assistant")

@mcp.prompt()
def code_review(code: str, language: str = "python") -> str:
    """コードレビューを依頼するプロンプトです。

    Args:
        code: レビュー対象のコード
        language: プログラミング言語
    """
    return f"""以下の{language}コードをレビューしてください。

観点:
1. バグや潜在的な問題
2. パフォーマンス改善の余地
3. 可読性・保守性
4. セキュリティ上の懸念

```{language}
{code}
```"""
```

### 引数付きプロンプト

プロンプトに引数を渡して、テンプレートを動的に変化させます。

```python
@mcp.prompt()
def explain_error(
    error_message: str,
    context: str = "",
    level: str = "beginner",
) -> str:
    """エラーメッセージの解説を依頼するプロンプトです。

    Args:
        error_message: エラーメッセージ
        context: エラーが発生した状況の説明
        level: 説明レベル（"beginner", "intermediate", "advanced"）
    """
    level_instructions = {
        "beginner": "プログラミング初心者にもわかるよう、専門用語を避けて",
        "intermediate": "基本的なプログラミング知識がある人向けに",
        "advanced": "経験豊富なエンジニア向けに、技術的に正確に",
    }
    instruction = level_instructions.get(level, level_instructions["beginner"])

    prompt = f"""{instruction}以下のエラーを解説してください。

エラーメッセージ:
```
{error_message}
```"""
    if context:
        prompt += f"\n\n発生状況:\n{context}"
    return prompt
```

### 複数メッセージ（user/assistant role）

プロンプトは複数のメッセージで構成でき、user と assistant の役割を指定できます。

```python
from mcp.types import TextContent

@mcp.prompt()
def sql_query_builder(table_name: str, requirement: str) -> list[base.Message]:
    """SQLクエリを段階的に構築するプロンプトです。"""
    return [
        base.UserMessage(
            content=f"テーブル「{table_name}」に対して、"
                    f"以下の要件を満たすSQLクエリを作成してください:\n{requirement}"
        ),
        base.AssistantMessage(
            content="まず、テーブルの構造を確認させてください。"
                    f"テーブル「{table_name}」のスキーマ情報を取得します。"
        ),
        base.UserMessage(
            content="スキーマ情報に基づいてSQLを作成し、"
                    "実行前にクエリの説明もお願いします。"
        ),
    ]
```

### embeddedResource を含むプロンプト

プロンプト内にリソースを埋め込むことで、AIに参照データを自動的に渡せます。

```python
from mcp.types import TextContent, EmbeddedResource, TextResourceContents

@mcp.prompt()
def analyze_with_data(data_uri: str) -> list[base.Message]:
    """リソースデータを埋め込んで分析を依頼するプロンプトです。"""
    return [
        base.UserMessage(
            content=[
                TextContent(type="text", text="以下のデータを分析してください:"),
                EmbeddedResource(
                    type="resource",
                    resource=TextResourceContents(
                        uri=data_uri,
                        mimeType="application/json",
                        text='{"sales": [100, 200, 150, 300]}',
                    ),
                ),
            ]
        ),
    ]
```

### よくある間違い

| 間違い | 正しい方法 |
|---|---|
| PromptでToolのような操作を行う | Promptはテンプレートの提供のみ。操作はToolに任せる |
| 引数なしの汎用的すぎるPrompt | 適度にパラメータ化して再利用性を高める |
| 長すぎるPromptを1つに詰め込む | 複数メッセージに分割して構造化する |

> **ポイントまとめ**
> - Promptsはユーザーが起動する会話テンプレート
> - 引数で動的に内容を変え、再利用性を高める
> - 複数メッセージで会話の流れを設計できる
> - embeddedResourceでリソースデータを埋め込める

## 5.5 機能の組み合わせパターン

実際のMCPサーバーでは、3つの機能を組み合わせて使います。ここでは代表的なパターンを紹介します。

### パターン1: Resources + Tools（読んでから操作）

リソースで現状を確認し、ツールで操作する組み合わせです。

```python
from mcp.server.fastmcp import FastMCP
import json

mcp = FastMCP("inventory-server")

# データストア（サンプル用）
inventory = {
    "item_001": {"name": "ノートPC", "stock": 15, "price": 98000},
    "item_002": {"name": "マウス", "stock": 200, "price": 3500},
    "item_003": {"name": "キーボード", "stock": 0, "price": 8900},
}

# ── Resources: データを読む ──

@mcp.resource("inventory://items")
def list_items() -> str:
    """在庫一覧を返します。"""
    return json.dumps(inventory, ensure_ascii=False, indent=2)

@mcp.resource("inventory://items/{item_id}")
def get_item(item_id: str) -> str:
    """指定商品の在庫情報を返します。"""
    item = inventory.get(item_id)
    if item is None:
        return f"商品 {item_id} が見つかりません"
    return json.dumps(item, ensure_ascii=False, indent=2)

# ── Tools: データを操作する ──

@mcp.tool()
def restock_item(item_id: str, quantity: int) -> str:
    """商品の在庫を補充します。

    Args:
        item_id: 商品ID（例: "item_001"）
        quantity: 補充する数量（正の整数）
    """
    if item_id not in inventory:
        raise mcp.ToolError(f"商品 {item_id} が見つかりません")
    if quantity <= 0:
        raise mcp.ToolError("数量は正の整数を指定してください")

    inventory[item_id]["stock"] += quantity
    new_stock = inventory[item_id]["stock"]
    return f"商品 {item_id} を {quantity} 個補充しました（在庫: {new_stock}個）"
```

AIの動作イメージ:
```
ユーザー: 「在庫切れの商品を補充して」
  ↓
AI: inventory://items を読む（Resources）
  ↓
AI: item_003 の在庫が0であることを確認
  ↓
AI: restock_item("item_003", 50) を呼ぶ（Tools）
  ↓
AI: 「キーボードの在庫を50個補充しました」
```

### パターン2: Prompts + Tools（テンプレートから実行）

プロンプトで作業の型を提供し、ツールで実行する組み合わせです。

```python
from mcp.server.fastmcp import FastMCP
from mcp.server.fastmcp.prompts import base

mcp = FastMCP("deploy-assistant")

# ── Prompts: 作業テンプレート ──

@mcp.prompt()
def deploy_checklist(environment: str, version: str) -> list[base.Message]:
    """デプロイチェックリストのテンプレートです。"""
    return [
        base.UserMessage(
            content=f"{environment}環境にバージョン{version}を"
                    "デプロイしたいです。チェックリストに沿って進めてください。"
        ),
        base.AssistantMessage(
            content=f"了解しました。{environment}環境への"
                    f"v{version}デプロイを進めます。\n\n"
                    "まずヘルスチェックから始めます。"
        ),
    ]

# ── Tools: 実際の操作 ──

@mcp.tool()
def check_health(environment: str) -> str:
    """指定環境のヘルスチェックを実行します。"""
    return f"{environment}環境: 全サービス正常稼働中"

@mcp.tool()
def run_deploy(environment: str, version: str) -> str:
    """デプロイを実行します。"""
    return f"{environment}環境にv{version}をデプロイしました"
```

### パターン3: 3機能すべてを組み合わせた総合サーバー

```python
from mcp.server.fastmcp import FastMCP
from mcp.server.fastmcp.prompts import base
import json

mcp = FastMCP("project-manager")

# ── Resources: プロジェクト情報の参照 ──

@mcp.resource("project://tasks")
def list_tasks() -> str:
    """全タスク一覧を返します。"""
    tasks = [
        {"id": "T-1", "title": "API設計", "status": "done"},
        {"id": "T-2", "title": "DB設計", "status": "in_progress"},
        {"id": "T-3", "title": "テスト作成", "status": "todo"},
    ]
    return json.dumps(tasks, ensure_ascii=False, indent=2)

# ── Tools: タスク操作 ──

@mcp.tool()
def update_task_status(task_id: str, new_status: str) -> str:
    """タスクのステータスを更新します。

    Args:
        task_id: タスクID（例: "T-1"）
        new_status: 新しいステータス（"todo", "in_progress", "done"）
    """
    return f"タスク {task_id} を {new_status} に更新しました"

@mcp.tool()
def add_task(title: str, assignee: str = "") -> str:
    """新しいタスクを追加します。

    Args:
        title: タスクのタイトル
        assignee: 担当者名（省略可）
    """
    return f"タスク '{title}' を追加しました"

# ── Prompts: 定型ワークフロー ──

@mcp.prompt()
def daily_standup() -> list[base.Message]:
    """デイリースタンドアップのテンプレートです。"""
    return [
        base.UserMessage(
            content="デイリースタンドアップを始めます。"
                    "タスク一覧を確認して、進捗をまとめてください。"
        ),
    ]

@mcp.prompt()
def sprint_review(sprint_number: str) -> str:
    """スプリントレビューのテンプレートです。"""
    return (
        f"スプリント{sprint_number}のレビューを行います。\n"
        "以下の観点でまとめてください:\n"
        "1. 完了したタスク\n"
        "2. 未完了のタスク\n"
        "3. 次スプリントへの申し送り"
    )
```

> **ポイントまとめ**
> - Resources + Tools = 「確認してから操作」の安全なパターン
> - Prompts + Tools = 「テンプレートで始めてツールで実行」の効率的なパターン
> - 3機能を組み合わせることで、実用的なAIアシスタントを構築できる

## 5.6 設計判断ガイド

### 「Tool? Resource? Prompt?」判断フローチャート

```
その機能は...
│
├─ データを提供するだけ？（副作用なし）
│   │
│   ├─ Yes → Resource
│   │       ├─ URIは固定？ → 静的リソース
│   │       └─ URIにパラメータが必要？ → 動的リソース（テンプレート）
│   │
│   └─ No（何かを実行する）
│       │
│       ├─ AIが自動で判断して呼ぶべき？
│       │   │
│       │   ├─ Yes → Tool
│       │   │       ├─ 冪等？ → description に明記不要
│       │   │       └─ 非冪等？ → description に注意書きを追加
│       │   │
│       │   └─ No（ユーザーが明示的に起動する）
│       │       │
│       │       └─ 定型の会話パターンがある？
│       │           ├─ Yes → Prompt
│       │           └─ No → Tool（ユーザーが依頼→AIが呼ぶ）
```

### 具体的な判断例

| やりたいこと | 判断 | 理由 |
|---|---|---|
| ログファイルの内容を読む | Resource | 読み取り専用、副作用なし |
| ファイルを作成する | Tool | ファイルシステムに副作用あり |
| コードレビューを依頼する | Prompt | 定型の会話パターン |
| APIからデータを取得する | Resource | 読み取り専用（GETリクエスト） |
| APIにデータを送信する | Tool | 外部システムに副作用あり |
| 設定ファイルの内容を返す | Resource | 読み取り専用 |
| 設定ファイルを書き換える | Tool | ファイルシステムに副作用あり |
| DB検索でデータを探す | Tool | パラメータが複雑、検索はToolが適切 |
| 定型レポートを生成する | Prompt + Tool | テンプレートで起動し、Toolで生成 |

### 迷ったときの基準

1. **副作用があるか？** → あるなら必ずTool
2. **AIが自動で呼ぶべきか？** → YesならTool、NoならPrompt
3. **URI形式で表現できるか？** → できるならResource
4. **定型の会話パターンか？** → YesならPrompt

### よくある間違い

| 間違い | 正しい方法 |
|---|---|
| ResourceでDB更新を行う | DB更新はToolで。Resourceは読み取り専用 |
| 全部Toolにしてしまう | 読み取り専用のデータはResourceで公開する |
| Promptの中でデータ取得を行う | Promptはテンプレート。データはResourceかToolで取得 |
| 1つの機能に全部を詰め込む | 役割ごとにTool・Resource・Promptを分ける |

> **ポイントまとめ**
> - 副作用の有無が最大の判断基準。副作用あり = Tool、なし = Resource
> - ユーザーが明示的に起動する定型パターンはPrompt
> - 迷ったらToolにするのが安全（Toolは最も柔軟）
> - 実際のサーバーでは3機能を組み合わせて使うのが一般的

## 3.1 Function Calling の詳細

### Function Calling とは

Function Calling（関数呼び出し）とは、LLM がユーザーの要求に応じて **外部の関数やAPIを呼び出す仕組み** です。LLM 自体がコードを実行するのではなく、「どの関数を、どの引数で呼ぶべきか」を JSON 形式で出力します。

```
ユーザー: 「東京の天気を教えて」
    │
    ▼
┌──────────────┐
│   LLM        │ ── ツール呼び出しを判断
│              │ ── get_weather(city="東京") を選択
└──────┬───────┘
       │  JSON出力: {"name": "get_weather", "input": {"city": "東京"}}
       ▼
┌──────────────┐
│ アプリケーション │ ── 実際にAPIを呼び出す
│   (あなたのコード) │
└──────┬───────┘
       │  実行結果: {"weather": "晴れ", "temp": 22}
       ▼
┌──────────────┐
│   LLM        │ ── 結果を自然言語に変換
└──────────────┘
    │
    ▼
応答: 「東京の天気は晴れで、気温は22度です。」
```

### JSON Schema によるツール定義

ツールは **JSON Schema** で定義します。これにより、LLM はツールの目的・引数の型・必須項目を正確に理解できます。

```python
# Anthropic SDK でのツール定義例
tools = [
    {
        "name": "get_weather",
        "description": "指定された都市の現在の天気情報を取得します。天気、気温、湿度を返します。",
        "input_schema": {
            "type": "object",
            "properties": {
                "city": {
                    "type": "string",
                    "description": "天気を取得したい都市名（例：東京、大阪、福岡）"
                },
                "unit": {
                    "type": "string",
                    "enum": ["celsius", "fahrenheit"],
                    "description": "温度の単位。デフォルトはcelsius"
                }
            },
            "required": ["city"]
        }
    }
]
```

### パラメータ設計のポイント

| 要素 | 良い例 | 悪い例 |
|------|--------|--------|
| name | `search_documents` | `func1` |
| description | 「社内ドキュメントをキーワードで全文検索します」 | 「検索する」 |
| パラメータ名 | `query`, `max_results` | `q`, `n` |
| パラメータ説明 | 「検索キーワード。スペース区切りでAND検索」 | 「クエリ」 |

### Anthropic API での実装例

```python
import anthropic
import json

client = anthropic.Anthropic()  # ANTHROPIC_API_KEY 環境変数を使用

# ツール定義
tools = [
    {
        "name": "get_weather",
        "description": "指定された都市の現在の天気情報を取得します。",
        "input_schema": {
            "type": "object",
            "properties": {
                "city": {
                    "type": "string",
                    "description": "天気を取得したい都市名"
                }
            },
            "required": ["city"]
        }
    }
]

# 実際の天気取得関数（ダミー実装）
def get_weather(city: str) -> dict:
    """本番ではここで外部APIを呼び出す"""
    dummy_data = {
        "東京": {"weather": "晴れ", "temp": 22, "humidity": 45},
        "大阪": {"weather": "曇り", "temp": 19, "humidity": 60},
    }
    return dummy_data.get(city, {"weather": "不明", "temp": 0, "humidity": 0})

# Function Calling ループ
def run_agent(user_message: str) -> str:
    messages = [{"role": "user", "content": user_message}]

    # 1回目の呼び出し：ツール使用の判断
    response = client.messages.create(
        model="claude-sonnet-4-20250514",
        max_tokens=1024,
        tools=tools,
        messages=messages,
    )

    # ツール呼び出しがあれば実行
    while response.stop_reason == "tool_use":
        # ツール呼び出しブロックを取得
        tool_use_block = next(
            block for block in response.content
            if block.type == "tool_use"
        )

        tool_name = tool_use_block.name
        tool_input = tool_use_block.input

        # ツールを実行
        if tool_name == "get_weather":
            result = get_weather(tool_input["city"])
        else:
            result = {"error": f"未知のツール: {tool_name}"}

        # 結果をメッセージに追加して再度呼び出し
        messages.append({"role": "assistant", "content": response.content})
        messages.append({
            "role": "user",
            "content": [
                {
                    "type": "tool_result",
                    "tool_use_id": tool_use_block.id,
                    "content": json.dumps(result, ensure_ascii=False),
                }
            ],
        })

        response = client.messages.create(
            model="claude-sonnet-4-20250514",
            max_tokens=1024,
            tools=tools,
            messages=messages,
        )

    # 最終的なテキスト応答を返す
    return response.content[0].text

# 実行
answer = run_agent("東京の天気を教えてください")
print(answer)
```

> **ポイントまとめ**
> - Function Calling は LLM が「どの関数をどの引数で呼ぶか」を決定する仕組みです
> - ツール定義の `description` が最も重要 -- LLM はこの説明文を頼りにツールを選択します
> - 実際の関数実行はアプリケーション側で行い、結果を LLM に返して最終応答を生成します

## 3.2 MCP（Model Context Protocol）

### MCP とは何か

MCP（Model Context Protocol）は、Anthropic が提唱した **AIモデルと外部ツール・データソースを接続するためのオープンプロトコル** です。USB-C が様々なデバイスを統一的に接続するように、MCP は AI アプリケーションと外部システムの間に **標準化されたインターフェース** を提供します。

### なぜ MCP が重要か

MCP が登場する以前は、ツール連携の実装が個別対応になっていました。

```
【MCP 以前】 M x N の個別接続
  App A ──┬── ツール 1 用コード
          ├── ツール 2 用コード
          └── ツール 3 用コード
  App B ──┬── ツール 1 用コード（重複）
          ├── ツール 2 用コード（重複）
          └── ツール 3 用コード（重複）

【MCP 以後】 標準プロトコルで統一
  App A ──┐                ┌── MCP Server (ツール1)
          ├── MCP Protocol ├── MCP Server (ツール2)
  App B ──┘                └── MCP Server (ツール3)
```

### MCP のアーキテクチャ

MCP は3つのコンポーネントで構成されます。

| コンポーネント | 役割 | 例 |
|---------------|------|-----|
| **ホスト（Host）** | ユーザーが操作するアプリケーション | Claude Desktop, IDE |
| **クライアント（Client）** | サーバーとの通信を管理する | ホスト内部の MCP クライアント |
| **サーバー（Server）** | ツールやデータを提供する | ファイル操作サーバー、DB サーバー |

```
┌─────────────────────────────────────────┐
│           ホスト（Host）                  │
│  ┌─────────────────┐                    │
│  │  MCP Client A   │◄──► MCP Server A  │(ファイル操作)
│  └─────────────────┘                    │
│  ┌─────────────────┐                    │
│  │  MCP Client B   │◄──► MCP Server B  │(DB検索)
│  └─────────────────┘                    │
│  ┌─────────────────┐                    │
│  │  MCP Client C   │◄──► MCP Server C  │(Web検索)
│  └─────────────────┘                    │
└─────────────────────────────────────────┘
```

### MCP が提供する3つの機能

1. **ツール（Tools）**: LLM が呼び出せる関数（検索、計算、API呼び出しなど）
2. **リソース（Resources）**: LLM が読み取れるデータ（ファイル内容、DB レコードなど）
3. **プロンプト（Prompts）**: 再利用可能なプロンプトテンプレート

> **ポイントまとめ**
> - MCP は AI とツールを繋ぐ「共通規格」であり、個別実装の手間を大幅に削減します
> - ホスト・クライアント・サーバーの3層構造で動作します
> - ツール、リソース、プロンプトの3つの機能を標準化して提供します

## 3.3 主要なツールの種類

AIエージェントが利用するツールは、大きく以下のカテゴリに分類できます。

### ツール分類一覧

| カテゴリ | 用途 | 具体例 |
|---------|------|--------|
| **検索ツール** | 情報の取得 | Web検索、ベクトル検索、社内Wiki検索 |
| **ファイル操作** | ファイルの読み書き | 読み取り、書き込み、ディレクトリ一覧 |
| **データベース** | データの永続化・検索 | SQL実行、CRUD操作 |
| **API呼び出し** | 外部サービス連携 | 天気API、翻訳API、決済API |
| **コード実行** | 動的な計算・処理 | Python実行、シェルコマンド |
| **ブラウザ操作** | Webページの操作 | ページ取得、フォーム入力、スクリーンショット |

### ツール選定の判断基準

```
ユーザーの要求を分析
    │
    ├─ 最新情報が必要？ → 検索ツール
    ├─ ファイルを扱う？ → ファイル操作ツール
    ├─ データを保存/検索？ → データベースツール
    ├─ 外部サービスが必要？ → API呼び出しツール
    ├─ 計算・変換が必要？ → コード実行ツール
    └─ Webページを操作？ → ブラウザ操作ツール
```

### よくある間違い

- **ツールを増やしすぎる**: ツールが多すぎると LLM の選択精度が下がります。目安は **10〜20個以内** に抑えましょう
- **万能ツールを作る**: 1つのツールに多機能を詰め込むと、LLM がいつ使うべきか判断できなくなります
- **説明文を省略する**: ツールの description が不十分だと、LLM が正しいツールを選択できません

> **ポイントまとめ**
> - ツールは6つの主要カテゴリに分類でき、要求に応じて適切に選定します
> - ツール数は多すぎず少なすぎず、目的ごとに明確に分離することが重要です

## 3.4 ツール設計のベストプラクティス

### 粒度の設計

ツールの粒度（Granularity）は **「1つのツール = 1つの明確なアクション」** が基本です。

```
【粗すぎる粒度】
  manage_database(action, table, data, query, ...)
  → LLM が action パラメータの使い分けに迷う

【適切な粒度】
  search_records(table, query, limit)
  insert_record(table, data)
  update_record(table, id, data)
  delete_record(table, id)
  → 各ツールの目的が明確
```

### 説明文（description）の書き方

```python
# 悪い例
{
    "name": "search",
    "description": "検索する",
}

# 良い例
{
    "name": "search_documents",
    "description": (
        "社内ドキュメントをキーワードで全文検索します。"
        "検索結果はスコア順に返されます。"
        "最大20件まで取得できます。"
        "検索対象: 社内Wiki、議事録、マニュアル"
    ),
}
```

### エラー処理

ツール実行時のエラーは、LLM が理解できる形で返すことが重要です。

```python
def execute_tool(tool_name: str, tool_input: dict) -> str:
    try:
        if tool_name == "search_documents":
            results = search_documents(**tool_input)
            return json.dumps(results, ensure_ascii=False)
    except ValueError as e:
        # LLM が理解できるエラーメッセージ
        return json.dumps({
            "error": True,
            "message": f"パラメータが不正です: {e}",
            "suggestion": "queryパラメータに検索キーワードを指定してください"
        }, ensure_ascii=False)
    except ConnectionError:
        return json.dumps({
            "error": True,
            "message": "検索サーバーに接続できませんでした",
            "suggestion": "しばらく待ってから再試行してください"
        }, ensure_ascii=False)
```

### セキュリティ考慮

| リスク | 対策 |
|--------|------|
| インジェクション攻撃 | 入力値のバリデーションとサニタイズ |
| 過剰な権限 | 最小権限の原則（読み取り専用で十分なら書き込み権限を与えない） |
| 機密情報の漏洩 | APIキーや認証情報をツール結果に含めない |
| 無限ループ | ツール呼び出し回数の上限を設定する |
| コスト爆発 | レートリミットと予算上限を設ける |

```python
# ツール呼び出し回数の制限例
MAX_TOOL_CALLS = 10

def run_agent_with_limit(user_message: str) -> str:
    tool_call_count = 0
    messages = [{"role": "user", "content": user_message}]

    while tool_call_count < MAX_TOOL_CALLS:
        response = client.messages.create(
            model="claude-sonnet-4-20250514",
            max_tokens=1024,
            tools=tools,
            messages=messages,
        )
        if response.stop_reason != "tool_use":
            break
        tool_call_count += 1
        # ... ツール実行処理 ...

    if tool_call_count >= MAX_TOOL_CALLS:
        return "ツール呼び出し回数が上限に達しました。質問を分割してください。"

    return response.content[0].text
```

> **ポイントまとめ**
> - 「1ツール = 1アクション」の原則で粒度を設計します
> - description はツール選択の精度を左右する最重要要素です
> - エラーは LLM が理解できる形式で返し、セキュリティ対策を必ず実装します

## 3.5 MCP サーバーの実装

### Python での基本的な MCP サーバー

以下は、Python の `mcp` ライブラリを使って簡単なメモ管理 MCP サーバーを実装する例です。

```python
# memo_server.py
# pip install mcp で事前にインストール

from mcp.server.fastmcp import FastMCP

# MCPサーバーの作成
mcp = FastMCP("memo-server")

# メモの保存先（簡易的にメモリ上で管理）
notes: dict[str, str] = {}


@mcp.tool()
def add_note(title: str, content: str) -> str:
    """メモを追加します。titleはメモのタイトル、contentはメモの本文です。"""
    notes[title] = content
    return f"メモ '{title}' を追加しました。"


@mcp.tool()
def search_notes(keyword: str) -> str:
    """キーワードでメモを検索します。タイトルと本文の両方を検索対象とします。"""
    results = []
    for title, content in notes.items():
        if keyword in title or keyword in content:
            results.append(f"【{title}】{content}")
    if not results:
        return f"'{keyword}' に一致するメモは見つかりませんでした。"
    return "\n".join(results)


@mcp.tool()
def list_notes() -> str:
    """保存されている全メモのタイトル一覧を返します。"""
    if not notes:
        return "メモはまだありません。"
    return "\n".join(f"- {title}" for title in notes.keys())


@mcp.resource("memo://list")
def get_all_notes() -> str:
    """全メモをリソースとして公開します。"""
    if not notes:
        return "メモはまだありません。"
    return "\n\n".join(
        f"# {title}\n{content}" for title, content in notes.items()
    )


if __name__ == "__main__":
    mcp.run(transport="stdio")
```

### Claude Desktop での設定

上記の MCP サーバーを Claude Desktop から利用するには、設定ファイルに以下を追加します。

```json
{
  "mcpServers": {
    "memo-server": {
      "command": "python",
      "args": ["memo_server.py"],
      "cwd": "/path/to/your/project"
    }
  }
}
```

### よくある間違い

- **transport の指定漏れ**: `stdio` と `sse` を混同すると接続に失敗します。ローカル実行なら `stdio`、リモートなら `sse` を使います
- **ツール説明の不足**: `@mcp.tool()` のドキュメント文字列が空だと、LLM がツールの用途を判断できません
- **エラーハンドリングの欠如**: ツール内で例外が発生すると MCP サーバー自体がクラッシュする場合があります

> **ポイントまとめ**
> - MCP サーバーは `FastMCP` クラスを使うことで最小限のコードで実装できます
> - `@mcp.tool()` デコレーターでツールを、`@mcp.resource()` でリソースを公開します
> - ドキュメント文字列がそのままツールの description として使われます

## 3.6 外部API連携の実践

### REST API との連携パターン

```python
import httpx
import json
from mcp.server.fastmcp import FastMCP

mcp = FastMCP("api-integration-server")


@mcp.tool()
async def fetch_github_repo(owner: str, repo: str) -> str:
    """GitHubリポジトリの情報を取得します。スター数、説明、言語などを返します。"""
    url = f"https://api.github.com/repos/{owner}/{repo}"
    async with httpx.AsyncClient() as client:
        response = await client.get(url)
        if response.status_code == 404:
            return f"リポジトリ {owner}/{repo} が見つかりません。"
        if response.status_code != 200:
            return f"APIエラーが発生しました（ステータス: {response.status_code}）。"
        data = response.json()
        return json.dumps({
            "名前": data["full_name"],
            "説明": data.get("description", "なし"),
            "スター数": data["stargazers_count"],
            "言語": data.get("language", "不明"),
            "作成日": data["created_at"],
        }, ensure_ascii=False, indent=2)
```

### データベースとの連携パターン

```python
import sqlite3
from mcp.server.fastmcp import FastMCP

mcp = FastMCP("database-server")

DB_PATH = "app.db"


def get_connection():
    conn = sqlite3.connect(DB_PATH)
    conn.row_factory = sqlite3.Row
    return conn


@mcp.tool()
def query_products(keyword: str, max_results: int = 10) -> str:
    """商品をキーワードで検索します。商品名と説明を対象に部分一致検索を行います。"""
    conn = get_connection()
    cursor = conn.execute(
        "SELECT id, name, price, description FROM products "
        "WHERE name LIKE ? OR description LIKE ? LIMIT ?",
        (f"%{keyword}%", f"%{keyword}%", max_results),
    )
    rows = [dict(row) for row in cursor.fetchall()]
    conn.close()
    if not rows:
        return f"'{keyword}' に一致する商品は見つかりませんでした。"
    return json.dumps(rows, ensure_ascii=False, indent=2)
```

### ファイルシステムとの連携パターン

```python
import os
from pathlib import Path
from mcp.server.fastmcp import FastMCP

mcp = FastMCP("filesystem-server")

# 安全のためアクセス可能なディレクトリを制限
ALLOWED_BASE = Path("/home/user/documents")


def validate_path(file_path: str) -> Path:
    """パスが許可されたディレクトリ内かチェックします。"""
    resolved = (ALLOWED_BASE / file_path).resolve()
    if not str(resolved).startswith(str(ALLOWED_BASE.resolve())):
        raise ValueError("許可されたディレクトリ外へのアクセスはできません。")
    return resolved


@mcp.tool()
def read_file(file_path: str) -> str:
    """指定されたファイルの内容を読み取ります。パスはdocumentsディレクトリからの相対パスです。"""
    try:
        path = validate_path(file_path)
        return path.read_text(encoding="utf-8")
    except ValueError as e:
        return f"エラー: {e}"
    except FileNotFoundError:
        return f"ファイル '{file_path}' が見つかりません。"


@mcp.tool()
def list_directory(dir_path: str = ".") -> str:
    """ディレクトリ内のファイル一覧を返します。"""
    try:
        path = validate_path(dir_path)
        entries = []
        for entry in sorted(path.iterdir()):
            kind = "DIR" if entry.is_dir() else "FILE"
            size = entry.stat().st_size if entry.is_file() else 0
            entries.append(f"[{kind}] {entry.name}  ({size} bytes)")
        return "\n".join(entries) if entries else "ディレクトリは空です。"
    except ValueError as e:
        return f"エラー: {e}"
```

### 連携パターンの比較

| 連携先 | 通信方式 | 認証方式 | 主な注意点 |
|--------|---------|---------|-----------|
| REST API | HTTP/HTTPS | APIキー, OAuth | レートリミット、タイムアウト設定 |
| データベース | TCP/ローカル | ユーザー/パスワード | SQLインジェクション対策、接続プール |
| ファイルシステム | ローカルI/O | OS権限 | パストラバーサル対策、アクセス制限 |

### よくある間違い

- **APIキーをコードにハードコード**: 環境変数や設定ファイルで管理してください
- **パストラバーサル未対策**: `../../etc/passwd` のような攻撃パスに対してバリデーションが必須です
- **SQLインジェクション未対策**: 必ずパラメータ化クエリ（プレースホルダ）を使いましょう
- **タイムアウト未設定**: 外部API呼び出しには必ずタイムアウトを設定しないと、エージェントが長時間停止します

> **ポイントまとめ**
> - REST API、データベース、ファイルシステムはそれぞれ異なるセキュリティ対策が必要です
> - MCP サーバーとして実装することで、どのホストアプリからも統一的に利用できます
> - 入力バリデーション、認証情報の安全な管理、タイムアウト設定は必須です

## 5.1 PKI（公開鍵基盤）の詳細

### 5.1.1 PKIの全体像

**PKI（Public Key Infrastructure: 公開鍵基盤）** は、公開鍵暗号方式をベースにした信頼の仕組みです。

```
┌──────────────────────────────────────────────┐
│                  PKIの構成要素                   │
│                                               │
│  ┌──────┐  証明書発行   ┌────────┐            │
│  │  CA  │──────────────│ 利用者  │            │
│  │(認証局)│              │(サーバ等)│            │
│  └──┬───┘              └────────┘            │
│     │                                        │
│  ┌──┴───┐  失効確認     ┌────────┐            │
│  │  RA  │              │ 検証者  │            │
│  │(登録局)│              │(クライアント)│          │
│  └──────┘              └────────┘            │
│                                               │
│  CRL: 証明書失効リスト                           │
│  OCSP: オンライン証明書状態プロトコル               │
│  リポジトリ: 証明書・CRLの公開場所                  │
└──────────────────────────────────────────────┘
```

### 5.1.2 認証局（CA）の階層構造

```
      ┌──────────┐
      │ ルートCA  │  ← 自己署名証明書（信頼の起点）
      └────┬─────┘
     ┌─────┼─────┐
┌────┴──┐┌─┴───┐┌┴─────┐
│中間CA1 ││中間CA2││中間CA3│  ← ルートCAが署名
└───┬───┘└──┬──┘└──┬───┘
    │       │      │
  利用者   利用者  利用者     ← 中間CAが署名

証明書チェーン:
利用者証明書 → 中間CA証明書 → ルートCA証明書
（各段階で上位CAの秘密鍵による署名を検証）
```

### 5.1.3 デジタル証明書（X.509）

| フィールド | 内容 |
|-----------|------|
| バージョン | X.509のバージョン（通常v3） |
| シリアル番号 | CAが発行した一意の番号 |
| 署名アルゴリズム | CA署名に使うアルゴリズム（SHA-256withRSA等） |
| 発行者 | CAの識別名（DN） |
| 有効期間 | 開始日と終了日 |
| サブジェクト | 証明書所有者の識別名 |
| 公開鍵情報 | 所有者の公開鍵 |
| 拡張領域 | 鍵の用途、SAN等 |
| CA署名 | CAの秘密鍵による電子署名 |

### 5.1.4 証明書の失効確認

| 方式 | 説明 | 特徴 |
|------|------|------|
| CRL（証明書失効リスト） | 失効した証明書のリストを定期的に公開 | 最新性に欠ける |
| OCSP | 個別の証明書の有効性をリアルタイムに問い合わせ | リアルタイム |
| OCSPステープリング | サーバがOCSP応答をTLSハンドシェイクに含める | クライアント負荷軽減 |

**ポイントまとめ:**
- PKIの構成要素（CA、RA、リポジトリ、CRL/OCSP）を理解する
- 証明書チェーンの検証の流れを説明できるようにする
- CRLとOCSPの違いは午前試験の定番

## 5.2 OAuth 2.0 / OpenID Connect

### 5.2.1 OAuth 2.0の基本

**OAuth 2.0** は、サードパーティアプリケーションにリソースへの限定的なアクセスを許可する**認可（Authorization）** フレームワークです。

```
┌─────────┐                          ┌──────────┐
│リソース   │                          │認可サーバ  │
│オーナー   │                          │          │
│(ユーザ)  │                          │          │
└────┬────┘                          └─────┬────┘
     │ ①認可要求                            │
     │ ②認可許可（同意画面）                   │
┌────┴────┐  ③認可コード/トークン要求  ┌─────┴────┐
│クライアント│ ──────────────────────→│認可サーバ  │
│(アプリ)  │ ←──────────────────────│          │
└────┬────┘  ④アクセストークン発行     └──────────┘
     │
     │ ⑤アクセストークンでAPIアクセス
     ▼
┌──────────┐
│リソースサーバ│
│(API)      │
└──────────┘
```

**OAuth 2.0の4つのグラントタイプ:**

| グラントタイプ | 用途 | セキュリティ |
|--------------|------|------------|
| 認可コード | Webアプリ（サーバサイド） | 高（推奨） |
| 認可コード+PKCE | SPA/モバイルアプリ | 高（推奨） |
| クライアントクレデンシャル | サーバ間通信（M2M） | 高 |
| インプリシット | SPA（非推奨） | 低（非推奨） |

### 5.2.2 OpenID Connect（OIDC）

**OIDC** はOAuth 2.0を拡張した**認証（Authentication）** プロトコルです。

```
OAuth 2.0:  認可（Authorization）→ 「何ができるか」
OIDC:       認証（Authentication）→ 「誰であるか」

OIDCはOAuth 2.0に以下を追加:
  - IDトークン（JWT形式）: ユーザの認証情報
  - UserInfoエンドポイント: ユーザ属性の取得
  - 標準化されたスコープ（openid, profile, email等）
```

**IDトークン（JWT）の構造:**
```
ヘッダ.ペイロード.署名

ヘッダ: {"alg": "RS256", "typ": "JWT"}
ペイロード: {
  "iss": "https://auth.example.com",   ← 発行者
  "sub": "user123",                     ← ユーザ識別子
  "aud": "client-app",                  ← 対象クライアント
  "exp": 1700000000,                    ← 有効期限
  "iat": 1699996400,                    ← 発行時刻
  "nonce": "abc123"                     ← リプレイ攻撃防止
}
署名: RSASHA256(base64(ヘッダ) + "." + base64(ペイロード), 秘密鍵)
```

**ポイントまとめ:**
- OAuth 2.0は認可、OIDCは認証（この区別が超重要）
- 認可コードフローの流れを図で説明できるようにする
- JWTの3部構成（ヘッダ.ペイロード.署名）を理解する

## 5.3 ゼロトラスト

### 5.3.1 ゼロトラストとは

従来の「境界防御モデル」に対し、**「何も信頼しない」** を前提としたセキュリティモデルです。

```
【従来: 境界防御モデル】
  ┌─────────────────────┐
  │  社内ネットワーク     │
  │  （信頼ゾーン）       │  ← 内部は安全と見なす
  │  PC  サーバ  DB     │
  └──────────┬──────────┘
          FW/VPN
  ─────────────────────────
          インターネット      ← 外部は危険

【ゼロトラスト】
  すべてのアクセスを検証
  ┌───┐ ┌───┐ ┌───┐
  │PC │ │サーバ│ │DB │  ← 全リソースが個別に認証・認可
  └─┬─┘ └─┬─┘ └─┬─┘
    │      │      │
  認証  認証    認証     ← 場所に関係なく毎回検証
    │      │      │
  ──┴──────┴──────┴──
```

### 5.3.2 ゼロトラストの原則（NIST SP 800-207）

1. すべてのデータソースとコンピューティングサービスはリソースとみなす
2. ネットワークの場所に関係なく、すべての通信を保護する
3. リソースへのアクセスはセッション単位で許可する
4. リソースへのアクセスは動的ポリシーで決定する
5. すべてのリソースの整合性とセキュリティを監視・計測する
6. リソースの認証と認可はアクセス許可前に動的かつ厳格に実施する
7. 資産・ネットワーク・通信の状態を可能な限り収集し、セキュリティ改善に活用する

### 5.3.3 ゼロトラストの主要技術

| 技術 | 役割 |
|------|------|
| IAM（Identity and Access Management） | ID管理と認証・認可の一元管理 |
| MFA（多要素認証） | 知識＋所持＋生体の組合せ認証 |
| マイクロセグメンテーション | ネットワークを細かく分割 |
| CASB | クラウドサービスへのアクセス制御・可視化 |
| EDR | エンドポイントの脅威検出と対応 |
| SASE | ネットワークとセキュリティの統合クラウドサービス |

**ポイントまとめ:**
- ゼロトラストは「場所ではなく、ID・デバイス・コンテキストで判断」
- 境界防御モデルとの違いを対比で説明できるようにする
- 関連技術（IAM、MFA、EDR、CASB、SASE）もセットで覚える

## 5.4 SOC（Security Operation Center）

### 5.4.1 SOCとは

**SOC（セキュリティオペレーションセンター）** は、組織のセキュリティを24時間365日監視・分析する専門組織です。

```
┌──────────────────────────────────────┐
│               SOC                     │
│                                       │
│  ┌─────────┐  ┌─────────┐  ┌──────┐ │
│  │ログ収集・│  │アラート  │  │インシ │ │
│  │分析(SIEM)│→│トリアージ │→│デント │ │
│  │         │  │         │  │対応  │ │
│  └─────────┘  └─────────┘  └──────┘ │
│       ↑                              │
│  FW  IDS/IPS  WAF  EDR  サーバログ    │
└──────────────────────────────────────┘
```

### 5.4.2 SIEM（Security Information and Event Management）

セキュリティ情報イベント管理システムです。

| 機能 | 説明 |
|------|------|
| ログ収集 | 各種機器・サーバからログを一元収集 |
| 正規化 | 異なるフォーマットのログを統一形式に変換 |
| 相関分析 | 複数ログの相関からインシデントを検出 |
| アラート | 閾値や条件に基づく通知 |
| ダッシュボード | セキュリティ状況の可視化 |
| レポート | 監査用の証跡レポート生成 |

### 5.4.3 CSIRT（Computer Security Incident Response Team）

| 役割 | 説明 |
|------|------|
| 事前対応 | 脆弱性情報の収集、教育・啓発、対応手順の整備 |
| インシデント対応 | 検知、トリアージ、封じ込め、根絶、復旧 |
| 事後対応 | 振り返り、再発防止策の策定、報告書作成 |

**SOCとCSIRTの違い:**
- **SOC:** 監視・検知が主務（日常的な監視活動）
- **CSIRT:** インシデント対応が主務（発生時の対応活動）

**ポイントまとめ:**
- SOCは常時監視、CSIRTはインシデント対応が主な役割
- SIEMはSOCの中核技術（ログ収集→相関分析→アラート）
- 午後試験ではインシデント対応の手順を記述する問題が出る

## 5.5 デジタルフォレンジック

### 5.5.1 デジタルフォレンジックとは

**デジタルフォレンジック（Digital Forensics）** は、インシデント発生後に電子的証拠を収集・保全・分析し、法的手続きに耐えうる形で記録する技術・手法です。

### 5.5.2 フォレンジックの手順

```
①収集 → ②保全 → ③分析 → ④報告

①収集（Collection）:
  - 揮発性データの優先収集（メモリ→ネットワーク→ディスク）
  - 証拠の同一性確保（ハッシュ値の記録）

②保全（Preservation）:
  - 証拠のビットイメージコピー（ビットバイビット）
  - 原本は操作せず、コピーで分析
  - チェーン・オブ・カストディ（証拠保管の連鎖）を記録

③分析（Analysis）:
  - タイムライン分析（イベントの時系列整理）
  - ファイルシステム分析（削除ファイルの復元等）
  - ログ分析、メモリ分析、ネットワーク分析

④報告（Reporting）:
  - 分析結果を法的に有効な形式で文書化
  - 第三者が再現・検証できる記述
```

### 5.5.3 証拠の揮発性の順序

揮発性が高い（消えやすい）順に収集する原則:

```
揮発性: 高 ──────────────────────→ 低
  レジスタ/キャッシュ
    ↓
  メモリ（RAM）
    ↓
  ネットワーク接続情報
    ↓
  実行中のプロセス
    ↓
  ディスク上のファイル
    ↓
  リムーバブルメディア
    ↓
  アーカイブ・バックアップ
```

**ポイントまとめ:**
- フォレンジックの4手順（収集→保全→分析→報告）を覚える
- 揮発性の高いデータから先に収集する原則は午前で頻出
- チェーン・オブ・カストディ（証拠保管の連鎖）の概念が重要

## 5.6 リスクアセスメント手法

### 5.6.1 リスクマネジメントの全体像

```
リスクマネジメント
├── リスクアセスメント
│   ├── リスク特定（何が起きうるか）
│   ├── リスク分析（どの程度の影響か）
│   └── リスク評価（対応優先度の決定）
└── リスク対応
    ├── リスク回避（リスク源を排除）
    ├── リスク低減（リスクを軽減）
    ├── リスク移転（保険・外部委託）
    └── リスク保有（受容）
```

### 5.6.2 定量的リスク分析

リスクを金額で評価する手法です。

**ALE（年間予想損失額）の計算:**
```
ALE = SLE × ARO

SLE（Single Loss Expectancy: 1回あたりの予想損失額）
  = 資産価値 × 露出係数（EF）

ARO（Annualized Rate of Occurrence: 年間発生率）

例:
  サーバの資産価値: 1,000万円
  データ漏洩時の損害率(EF): 40%
  年間発生率(ARO): 0.1回/年

  SLE = 1,000万 × 0.4 = 400万円
  ALE = 400万 × 0.1 = 40万円/年

  → セキュリティ対策の投資判断材料になる
```

### 5.6.3 定性的リスク分析

リスクを数値化せず、レベル（高/中/低）で評価する手法です。

**リスクマトリクス:**
```
         影響度
         大    中    小
   高  │ 極高 │ 高  │ 中  │
発 ─── ├─────┼─────┼─────┤
生 中  │ 高  │ 中  │ 低  │
可 ─── ├─────┼─────┼─────┤
能 低  │ 中  │ 低  │ 極低 │
性     └─────┴─────┴─────┘
```

### 5.6.4 代表的なリスク分析手法

| 手法 | 種類 | 特徴 |
|------|------|------|
| JRAM | 定性的 | 日本発のリスク分析手法 |
| CRAMM | 定性的 | 英国政府標準のリスク分析手法 |
| FTA（故障の木解析） | 定量的 | トップダウンで原因を分析 |
| ETA（事象の木解析） | 定量的 | ボトムアップで影響を分析 |
| FMEA（故障モード影響解析） | 両方 | 各部品の故障モードと影響を分析 |

### 5.6.5 情報セキュリティポリシー

```
┌────────────────────────────┐
│   情報セキュリティ基本方針    │ ← 経営層が策定、組織の方針
├────────────────────────────┤
│   情報セキュリティ対策基準    │ ← 遵守すべきルール・基準
├────────────────────────────┤
│   情報セキュリティ実施手順    │ ← 具体的な手順・マニュアル
└────────────────────────────┘
```

**ポイントまとめ:**
- 定量的分析はALE=SLE×AROの計算が試験で出る
- リスク対応の4種類（回避・低減・移転・保有）を区別する
- セキュリティポリシーの3階層構造を理解する

## 5.7 主要なセキュリティ技術と攻撃手法

### 5.7.1 Webアプリケーションセキュリティ

| 攻撃 | 説明 | 対策 |
|------|------|------|
| SQLインジェクション | SQL文に不正な入力を挿入 | プリペアドステートメント |
| XSS（クロスサイトスクリプティング） | 悪意あるスクリプトを注入 | エスケープ処理、CSP |
| CSRF（クロスサイトリクエストフォージェリ） | 別サイトから不正リクエスト | トークン検証 |
| ディレクトリトラバーサル | パス操作で非公開ファイルにアクセス | 入力値検証、アクセス制御 |
| セッションハイジャック | セッションIDを窃取して成りすます | Secure/HttpOnly属性 |

### 5.7.2 暗号技術の応用

| 技術 | 用途 |
|------|------|
| TLS 1.3 | HTTPS通信の暗号化（1-RTTハンドシェイク） |
| IPsec | VPN（トンネルモード/トランスポートモード） |
| S/MIME | メールの暗号化と電子署名 |
| DKIM/SPF/DMARC | メール送信ドメイン認証（なりすまし防止） |

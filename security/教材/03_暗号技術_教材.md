# 第3章：暗号技術

## この章のゴール

- 共通鍵暗号方式と公開鍵暗号方式の仕組みと違いを説明できる
- 主要な暗号アルゴリズム（AES、RSA、楕円曲線暗号）の特性を理解する
- ハッシュ関数の性質と用途を把握する
- デジタル署名の仕組みと、認証・否認防止における役割を理解する
- PKI（公開鍵基盤）と電子証明書の信頼モデルを説明できる
- 暗号技術を適切に選択・組み合わせて安全な通信を設計できる

## 3.1 暗号技術の基礎

### 3.1.1 暗号の基本用語

暗号技術は情報セキュリティの根幹をなす技術です。まず、基本的な用語を整理しましょう。

| 用語 | 説明 |
|------|------|
| **平文（Plaintext）** | 暗号化される前の元のデータです |
| **暗号文（Ciphertext）** | 暗号化された後のデータです |
| **暗号化（Encryption）** | 平文を暗号文に変換する処理です |
| **復号（Decryption）** | 暗号文を平文に戻す処理です |
| **鍵（Key）** | 暗号化・復号に使用する秘密の情報です |
| **暗号アルゴリズム** | 暗号化・復号の手順を定めた規則です |

### 3.1.2 ケルクホフスの原則

暗号システムの設計において最も重要な原則の一つが「ケルクホフスの原則」です。

> **ケルクホフスの原則**: 暗号システムの安全性は、アルゴリズムの秘密ではなく、**鍵の秘密性**のみに依存すべきである。

つまり、暗号アルゴリズム自体が公開されていても、鍵が秘密であれば安全でなければなりません。独自の秘密アルゴリズムに頼る「隠蔽によるセキュリティ（Security by Obscurity）」は、脆弱な設計思想とされています。

```
┌────────────────────────────────────────────────────┐
│              暗号方式の分類                          │
│                                                    │
│  暗号方式                                           │
│  ├── 共通鍵暗号方式（対称暗号）                      │
│  │   ├── ブロック暗号（AES, DES, Camellia）         │
│  │   └── ストリーム暗号（ChaCha20, RC4）            │
│  │                                                 │
│  └── 公開鍵暗号方式（非対称暗号）                    │
│      ├── RSA                                       │
│      ├── 楕円曲線暗号（ECC）                        │
│      └── ElGamal                                   │
│                                                    │
│  ハッシュ関数（暗号学的一方向関数）                   │
│  ├── SHA-2 (SHA-256, SHA-384, SHA-512)             │
│  ├── SHA-3                                         │
│  └── BLAKE2 / BLAKE3                              │
└────────────────────────────────────────────────────┘
```

## 3.2 共通鍵暗号方式（対称暗号）

### 3.2.1 共通鍵暗号方式とは

共通鍵暗号方式は、暗号化と復号で**同じ鍵**を使用する方式です。処理速度が速いため、大量のデータの暗号化に適しています。

```
┌──────────────────────────────────────────────────┐
│            共通鍵暗号方式の仕組み                   │
│                                                  │
│  送信者                         受信者            │
│                                                  │
│  平文 ──→ [暗号化] ──→ 暗号文 ──→ [復号] ──→ 平文 │
│              ↑                      ↑            │
│           共通鍵（K）            共通鍵（K）        │
│                                                  │
│  ※ 暗号化と復号に同じ鍵を使用                      │
│  ※ 鍵の安全な共有が最大の課題                      │
└──────────────────────────────────────────────────┘
```

### 3.2.2 ブロック暗号とストリーム暗号

| 分類 | 仕組み | 代表例 | 特徴 |
|------|--------|--------|------|
| **ブロック暗号** | 固定長ブロック単位で暗号化 | AES、DES | 多くの用途で利用。暗号化モードと組み合わせて使用 |
| **ストリーム暗号** | 1ビットまたは1バイト単位で暗号化 | ChaCha20、RC4 | リアルタイム通信に適する。高速 |

### 3.2.3 AES（Advanced Encryption Standard）

AES は現在最も広く使われている共通鍵暗号アルゴリズムです。2001年にNISTが標準として採択しました。

| 項目 | 仕様 |
|------|------|
| ブロック長 | 128ビット固定 |
| 鍵長 | 128 / 192 / 256ビット |
| ラウンド数 | 10 / 12 / 14（鍵長に応じて） |
| 設計者 | Joan Daemen、Vincent Rijmen（Rijndael） |

### 3.2.4 暗号化モード

ブロック暗号を安全に使うには、適切な暗号化モードを選択することが不可欠です。

| モード | 名称 | 安全性 | 用途 |
|--------|------|--------|------|
| **ECB** | Electronic Codebook | 低（使用禁止） | なし（教育用以外で使わないこと） |
| **CBC** | Cipher Block Chaining | 中 | ファイル暗号化（レガシー） |
| **CTR** | Counter | 高 | ストリーム暗号的な利用 |
| **GCM** | Galois/Counter Mode | 高（認証付き） | TLS、API通信（推奨） |
| **CCM** | Counter with CBC-MAC | 高（認証付き） | IoT、無線LAN |

**ECBモードの危険性**:

```
┌────────────────────────────────────────────────────────┐
│              ECB モードの問題点                          │
│                                                        │
│  同じ平文ブロック → 必ず同じ暗号文ブロック               │
│                                                        │
│  平文:   [AAAA] [BBBB] [AAAA] [CCCC]                  │
│  暗号文: [xxxx] [yyyy] [xxxx] [zzzz]                  │
│           ↑              ↑                             │
│           同じ！パターンが漏洩する                       │
│                                                        │
│  → 画像を ECB で暗号化すると、元の画像の輪郭が見える     │
│  → データのパターンが暗号文に残ってしまう                 │
└────────────────────────────────────────────────────────┘
```

### 3.2.5 AES-GCM の実装例

**脆弱なコード（ECBモード）**:

```python
from Crypto.Cipher import AES

# NG: ECBモードは安全ではありません
cipher = AES.new(key, AES.MODE_ECB)
ciphertext = cipher.encrypt(plaintext)
```

**修正後コード（GCMモード -- 推奨）**:

```python
from Crypto.Cipher import AES
from Crypto.Random import get_random_bytes

def encrypt_aes_gcm(key, plaintext, associated_data=None):
    """AES-GCM による認証付き暗号化"""
    # ナンス（初期化ベクトル）は毎回ランダムに生成
    nonce = get_random_bytes(12)  # 96ビット推奨
    cipher = AES.new(key, AES.MODE_GCM, nonce=nonce)

    if associated_data:
        cipher.update(associated_data)  # 認証対象の追加データ

    ciphertext, tag = cipher.encrypt_and_digest(plaintext)
    # ナンス + 認証タグ + 暗号文を一緒に返す
    return nonce + tag + ciphertext

def decrypt_aes_gcm(key, data, associated_data=None):
    """AES-GCM による復号と認証検証"""
    nonce = data[:12]
    tag = data[12:28]
    ciphertext = data[28:]

    cipher = AES.new(key, AES.MODE_GCM, nonce=nonce)
    if associated_data:
        cipher.update(associated_data)

    # 復号と同時に改ざん検知を行う
    plaintext = cipher.decrypt_and_verify(ciphertext, tag)
    return plaintext
```

### 3.2.6 共通鍵暗号の鍵配送問題

共通鍵暗号方式の最大の課題は「鍵をどうやって安全に相手に渡すか」という鍵配送問題です。

```
n人が相互に通信する場合に必要な鍵の数: n(n-1)/2

10人  →  45鍵
100人 → 4,950鍵
1000人 → 499,500鍵

→ 人数が増えると鍵管理が爆発的に困難になる
```

この問題を解決するのが、次に説明する公開鍵暗号方式です。

> **よくある間違い**: AESの鍵長を「128ビットだから安全でない」と判断するケースがあります。AES-128は現在も十分に安全であり、総当たり攻撃による解読は現実的ではありません。ただし、将来の量子コンピュータを考慮する場合はAES-256が推奨されます。

## 3.3 公開鍵暗号方式（非対称暗号）

### 3.3.1 公開鍵暗号方式とは

公開鍵暗号方式は、暗号化と復号で**異なる鍵**を使用する方式です。公開鍵（誰でも知ってよい）と秘密鍵（本人だけが持つ）のペアを使います。

```
┌────────────────────────────────────────────────────────┐
│            公開鍵暗号方式の仕組み（暗号化）              │
│                                                        │
│  送信者（Alice）               受信者（Bob）            │
│                                                        │
│  平文 ──→ [暗号化] ──→ 暗号文 ──→ [復号] ──→ 平文      │
│              ↑                       ↑                 │
│         Bobの公開鍵              Bobの秘密鍵            │
│         （誰でも入手可能）       （Bobだけが持つ）       │
│                                                        │
│  ※ 暗号化と復号で異なる鍵を使用                         │
│  ※ 公開鍵で暗号化したものは、対応する秘密鍵でのみ復号可能 │
│  ※ 鍵配送問題を解決                                     │
└────────────────────────────────────────────────────────┘
```

### 3.3.2 RSA暗号

RSA は1977年にRivest、Shamir、Adlemanによって考案された、最も広く知られた公開鍵暗号アルゴリズムです。大きな数の素因数分解が困難であることを安全性の根拠としています。

| 項目 | 仕様 |
|------|------|
| 安全性の根拠 | 大きな合成数の素因数分解の困難性 |
| 推奨鍵長 | 2048ビット以上（2030年以降は3072ビット以上を推奨） |
| 用途 | 鍵交換、デジタル署名、暗号化（小さなデータのみ） |
| 処理速度 | 共通鍵暗号に比べて非常に低速 |

### 3.3.3 楕円曲線暗号（ECC）

楕円曲線暗号は、楕円曲線上の離散対数問題の困難性を利用した公開鍵暗号です。RSAと比較して、短い鍵長で同等の安全性を実現できます。

| RSA鍵長 | 同等のECC鍵長 | セキュリティビット |
|---------|-------------|-----------------|
| 1024ビット | 160ビット | 80ビット |
| 2048ビット | 224ビット | 112ビット |
| 3072ビット | 256ビット | 128ビット |
| 7680ビット | 384ビット | 192ビット |
| 15360ビット | 521ビット | 256ビット |

### 3.3.4 共通鍵暗号と公開鍵暗号の比較

| 比較項目 | 共通鍵暗号 | 公開鍵暗号 |
|---------|-----------|-----------|
| 鍵の種類 | 暗号化・復号で同じ鍵 | 公開鍵と秘密鍵のペア |
| 処理速度 | 高速 | 低速（RSAで100〜1000倍遅い） |
| 鍵配送 | 安全な配送手段が必要 | 公開鍵は自由に配布可能 |
| 鍵の数（n人） | n(n-1)/2 | 2n |
| 主な用途 | 大量データの暗号化 | 鍵交換、デジタル署名 |
| 代表例 | AES、ChaCha20 | RSA、楕円曲線暗号 |

### 3.3.5 ハイブリッド暗号方式

実際の通信（TLS等）では、共通鍵暗号と公開鍵暗号を組み合わせた**ハイブリッド暗号方式**が使われています。それぞれの利点を活かした設計です。

```
┌──────────────────────────────────────────────────────────┐
│           ハイブリッド暗号方式の流れ                       │
│                                                          │
│  ① 送信者がランダムな共通鍵（セッション鍵）を生成         │
│  ② 共通鍵を受信者の公開鍵で暗号化して送信                 │
│  ③ 受信者は自分の秘密鍵で共通鍵を復号                    │
│  ④ 以降のデータ通信は共通鍵（AES等）で暗号化              │
│                                                          │
│  送信者                              受信者               │
│  ┌──────┐                          ┌──────┐             │
│  │セッション│  ←─ 公開鍵暗号 ─→   │セッション│             │
│  │鍵を生成 │                       │鍵を復号 │             │
│  └──┬───┘                          └──┬───┘             │
│     │    ←── 共通鍵暗号でデータ通信 ──→  │                │
│     │          （高速・大量データ可）      │                │
│                                                          │
│  公開鍵暗号: 鍵交換に使用（低速だが鍵配送問題を解決）     │
│  共通鍵暗号: データ暗号化に使用（高速）                    │
└──────────────────────────────────────────────────────────┘
```

## 3.4 鍵交換

### 3.4.1 Diffie-Hellman 鍵交換

Diffie-Hellman（DH）鍵交換は、安全でない通信路を通じて共通の秘密鍵を確立するためのプロトコルです。

```
┌──────────────────────────────────────────────────────────┐
│           Diffie-Hellman 鍵交換の概念                     │
│                                                          │
│  Alice                               Bob                 │
│  秘密値 a を選択                     秘密値 b を選択      │
│  A = g^a mod p を計算                B = g^b mod p を計算 │
│                                                          │
│        A ──────── 公開通信路 ────────→                    │
│        ←──────── 公開通信路 ──────── B                    │
│                                                          │
│  K = B^a mod p を計算                K = A^b mod p を計算 │
│    = g^(ab) mod p                      = g^(ab) mod p    │
│                                                          │
│  → Alice と Bob は同じ共通鍵 K を共有できる               │
│  → 盗聴者は A と B を見ても K を計算できない（離散対数問題）│
└──────────────────────────────────────────────────────────┘
```

### 3.4.2 前方秘匿性（Forward Secrecy）

前方秘匿性（PFS: Perfect Forward Secrecy）とは、長期的な秘密鍵が将来的に漏洩しても、過去の通信内容が解読されない性質です。

| 方式 | 前方秘匿性 | 説明 |
|------|-----------|------|
| RSA鍵交換 | なし | サーバーの秘密鍵が漏洩すると過去の通信も解読可能 |
| **DHE（Ephemeral DH）** | あり | セッションごとに一時的なDH鍵ペアを生成 |
| **ECDHE（Ephemeral ECDH）** | あり | 楕円曲線版のDHE。TLS 1.3の標準 |

> **現場での注意点**: TLS 1.3 では、前方秘匿性のない鍵交換方式（RSA鍵交換等）は廃止されました。新規システムでは必ずECDHEベースの鍵交換を使用してください。

## 3.5 ハッシュ関数

### 3.5.1 暗号学的ハッシュ関数とは

ハッシュ関数は、任意の長さの入力データから固定長の出力（ハッシュ値、ダイジェスト）を生成する一方向関数です。暗号学的ハッシュ関数には以下の性質が求められます。

| 性質 | 説明 |
|------|------|
| **一方向性（Preimage Resistance）** | ハッシュ値から元のデータを復元できないこと |
| **第二原像耐性（Second Preimage Resistance）** | あるデータと同じハッシュ値を持つ別のデータを見つけられないこと |
| **衝突耐性（Collision Resistance）** | 同じハッシュ値を持つ2つの異なるデータを見つけられないこと |
| **雪崩効果（Avalanche Effect）** | 入力の1ビット変化で出力が大きく変化すること |

### 3.5.2 主要なハッシュ関数

| アルゴリズム | ハッシュ長 | 安全性 | 備考 |
|-------------|----------|--------|------|
| MD5 | 128ビット | 危殆化（使用禁止） | 衝突攻撃が実証済み |
| SHA-1 | 160ビット | 危殆化（使用禁止） | 2017年にGoogleが衝突を実証 |
| **SHA-256** | 256ビット | 安全（推奨） | SHA-2ファミリー。最も広く利用 |
| **SHA-384** | 384ビット | 安全（推奨） | SHA-2ファミリー |
| **SHA-512** | 512ビット | 安全（推奨） | SHA-2ファミリー |
| **SHA-3（Keccak）** | 224/256/384/512ビット | 安全（推奨） | SHA-2とは異なる内部構造（スポンジ構造） |
| BLAKE2/BLAKE3 | 可変 | 安全 | 高速。暗号通貨等で使用 |

### 3.5.3 ハッシュ関数の用途

```
┌────────────────────────────────────────────────────┐
│             ハッシュ関数の主な用途                    │
│                                                    │
│  ① パスワードの保存                                 │
│     → ソルト + ストレッチング + ハッシュ化            │
│                                                    │
│  ② データの改ざん検知                               │
│     → ファイルのチェックサム、ソフトウェア配布検証     │
│                                                    │
│  ③ デジタル署名                                     │
│     → メッセージのハッシュ値に対して署名              │
│                                                    │
│  ④ メッセージ認証コード（HMAC）                     │
│     → 通信データの認証と完全性検証                   │
│                                                    │
│  ⑤ ブロックチェーン                                 │
│     → ブロックのハッシュチェーン                     │
│                                                    │
│  ⑥ 証明書のフィンガープリント                       │
│     → 証明書の一意な識別                            │
└────────────────────────────────────────────────────┘
```

### 3.5.4 HMAC（Hash-based Message Authentication Code）

HMAC は、ハッシュ関数と秘密鍵を組み合わせて、メッセージの認証と完全性を検証するための仕組みです。

```python
import hmac
import hashlib

def generate_hmac(key, message):
    """HMAC-SHA256 によるメッセージ認証コード生成"""
    return hmac.new(
        key.encode('utf-8'),
        message.encode('utf-8'),
        hashlib.sha256
    ).hexdigest()

def verify_hmac(key, message, expected_mac):
    """タイミング攻撃に耐性のある比較"""
    actual_mac = generate_hmac(key, message)
    # hmac.compare_digest は定数時間で比較する（タイミング攻撃対策）
    return hmac.compare_digest(actual_mac, expected_mac)
```

> **よくある間違い**: ハッシュ値の比較に通常の文字列比較（`==`）を使うと、タイミング攻撃に対して脆弱になります。文字列比較は不一致が見つかった時点で処理を打ち切るため、攻撃者は応答時間の差から正解を推測できます。必ず `hmac.compare_digest()` のような定数時間比較を使用してください。

## 3.6 デジタル署名

### 3.6.1 デジタル署名とは

デジタル署名は、公開鍵暗号方式の仕組みを利用して、メッセージの**認証**（誰が送ったか）と**完全性**（改ざんされていないか）と**否認防止**（送信を否定できない）を実現する技術です。

### 3.6.2 デジタル署名の仕組み

```
┌──────────────────────────────────────────────────────────┐
│             デジタル署名の流れ                             │
│                                                          │
│  【署名の生成（送信者）】                                  │
│  ① メッセージのハッシュ値を計算                            │
│  ② ハッシュ値を送信者の秘密鍵で暗号化 → これが「署名」     │
│  ③ メッセージ + 署名を送信                                │
│                                                          │
│  メッセージ ──→ [ハッシュ関数] ──→ ハッシュ値             │
│                                        ↓                 │
│                                  [秘密鍵で暗号化]         │
│                                        ↓                 │
│                                    デジタル署名            │
│                                                          │
│  【署名の検証（受信者）】                                  │
│  ① 受信したメッセージのハッシュ値を計算                    │
│  ② 署名を送信者の公開鍵で復号 → 元のハッシュ値を取得      │
│  ③ 2つのハッシュ値を比較 → 一致すれば署名は有効           │
│                                                          │
│  署名 ──→ [公開鍵で復号] ──→ ハッシュ値A                  │
│  メッセージ ──→ [ハッシュ関数] ──→ ハッシュ値B             │
│                                                          │
│  ハッシュ値A == ハッシュ値B → 署名有効（改ざんなし）       │
└──────────────────────────────────────────────────────────┘
```

### 3.6.3 デジタル署名が保証するもの

| 性質 | 説明 |
|------|------|
| **認証（Authentication）** | 署名者が秘密鍵の持ち主であることを確認できます |
| **完全性（Integrity）** | メッセージが改ざんされていないことを確認できます |
| **否認防止（Non-repudiation）** | 署名者は後から署名したことを否定できません |

### 3.6.4 主要なデジタル署名アルゴリズム

| アルゴリズム | 基盤技術 | 特徴 |
|-------------|---------|------|
| **RSA署名** | RSA暗号 | 最も広く利用。鍵長2048ビット以上を推奨 |
| **ECDSA** | 楕円曲線暗号 | RSAより短い鍵長で同等の安全性。TLS等で広く使用 |
| **EdDSA（Ed25519）** | 楕円曲線暗号 | 高速かつ安全。実装ミスに強い設計。SSH等で利用増加中 |
| **DSA** | 離散対数問題 | 米国政府標準だったがECDSAに移行。現在は非推奨 |

## 3.7 PKI（公開鍵基盤）と電子証明書

### 3.7.1 PKI とは

PKI（Public Key Infrastructure）は、公開鍵暗号方式を社会的に安全に運用するためのインフラストラクチャ（基盤）です。「この公開鍵は本当にその人のものか？」という問題を解決します。

### 3.7.2 PKI の構成要素

| 構成要素 | 説明 |
|---------|------|
| **認証局（CA: Certificate Authority）** | 電子証明書を発行・管理する信頼できる第三者機関です |
| **登録局（RA: Registration Authority）** | 証明書の申請者の本人確認を行います |
| **リポジトリ** | 証明書や証明書失効リストを公開するデータベースです |
| **電子証明書** | 公開鍵とその所有者の情報を結びつける電子文書です |
| **CRL / OCSP** | 失効した証明書の情報を提供する仕組みです |

### 3.7.3 電子証明書（X.509）の構造

```
┌────────────────────────────────────────────────────┐
│           X.509 電子証明書の主要フィールド           │
│                                                    │
│  バージョン:        V3                              │
│  シリアル番号:      01:23:45:67:89:AB:CD:EF        │
│  署名アルゴリズム:  SHA256withRSA                    │
│  発行者（Issuer）:  CN=Example CA, O=Example Inc   │
│  有効期間:                                         │
│    開始: 2025-01-01 00:00:00 UTC                   │
│    終了: 2026-01-01 00:00:00 UTC                   │
│  主体者（Subject）: CN=www.example.com             │
│  公開鍵情報:        RSA 2048ビット                  │
│  拡張フィールド:                                    │
│    Subject Alternative Name: www.example.com       │
│    Key Usage: Digital Signature, Key Encipherment  │
│  CAのデジタル署名                                   │
└────────────────────────────────────────────────────┘
```

### 3.7.4 証明書チェーン（信頼の連鎖）

```
┌─────────────────────────────────────────────────────────┐
│            証明書チェーン（信頼の連鎖）                    │
│                                                         │
│  ルートCA証明書（自己署名）                               │
│  ├── ルートCAの秘密鍵で署名                              │
│  │   ※ ブラウザ/OS にプリインストール（トラストストア）    │
│  │                                                      │
│  └─→ 中間CA証明書                                       │
│      ├── ルートCAの秘密鍵で署名                          │
│      │   ※ ルートCAが直接署名しないことでリスクを分散     │
│      │                                                  │
│      └─→ サーバー証明書（エンドエンティティ証明書）        │
│          ├── 中間CAの秘密鍵で署名                        │
│          └── Webサーバーに設置                           │
│                                                         │
│  検証の流れ（ブラウザ側）:                                │
│  サーバー証明書 → 中間CA証明書 → ルートCA証明書           │
│  各段階で署名を公開鍵で検証し、信頼の連鎖をたどる         │
└─────────────────────────────────────────────────────────┘
```

### 3.7.5 証明書の種類

| 種類 | 検証レベル | 特徴 | 用途 |
|------|-----------|------|------|
| **DV（Domain Validation）** | ドメインの管理権のみ確認 | 発行が速い。Let's Encryptが無料で提供 | 個人サイト、小規模サイト |
| **OV（Organization Validation）** | 組織の実在性を確認 | 組織名が証明書に記載される | 企業サイト |
| **EV（Extended Validation）** | 厳格な組織審査 | かつてはアドレスバーが緑色になった | 金融機関、ECサイト |

### 3.7.6 証明書の失効確認

証明書が有効期限内であっても、秘密鍵の漏洩等の理由で失効することがあります。

| 方式 | 仕組み | 特徴 |
|------|--------|------|
| **CRL（証明書失効リスト）** | CAが定期的に失効証明書のリストを公開 | リストが肥大化する問題 |
| **OCSP** | リアルタイムで証明書の有効性を問い合わせ | プライバシーの懸念（CAにアクセス先がわかる） |
| **OCSP Stapling** | Webサーバーが OCSP 応答をキャッシュして提示 | プライバシー問題を解決。推奨方式 |

## 3.8 TLS（Transport Layer Security）

### 3.8.1 TLS とは

TLS（Transport Layer Security）は、インターネット上の通信を暗号化するためのプロトコルです。HTTPS、SMTPS、IMAPS等、多くのプロトコルの基盤となっています。

### 3.8.2 TLS 1.3 ハンドシェイク

TLS 1.3 は最新バージョンであり、TLS 1.2 から大幅にセキュリティと性能が改善されています。

```
┌──────────────────────────────────────────────────────────┐
│           TLS 1.3 ハンドシェイク（1-RTT）                 │
│                                                          │
│  クライアント                       サーバー              │
│       │                               │                 │
│       │── ClientHello ──────────────→│                 │
│       │   ・対応暗号スイート           │                 │
│       │   ・ECDHE鍵共有パラメータ      │                 │
│       │                               │                 │
│       │←── ServerHello ──────────────│                 │
│       │   ・選択した暗号スイート       │                 │
│       │   ・ECDHE鍵共有パラメータ      │                 │
│       │←── 暗号化開始 ───────────────│                 │
│       │←── 証明書 ───────────────────│                 │
│       │←── CertificateVerify ────────│                 │
│       │←── Finished ─────────────────│                 │
│       │                               │                 │
│       │── Finished ──────────────────→│                 │
│       │                               │                 │
│       │←── アプリケーションデータ ────→│                 │
│       │      （暗号化された通信）       │                 │
│                                                          │
│  ※ TLS 1.2 では 2-RTT 必要だったものが 1-RTT に短縮      │
│  ※ 0-RTT（事前共有鍵利用時）もサポート                    │
└──────────────────────────────────────────────────────────┘
```

### 3.8.3 TLS 1.3 で廃止された要素

| 廃止された要素 | 理由 |
|---------------|------|
| RSA鍵交換 | 前方秘匿性がない |
| CBC モード暗号スイート | パディングオラクル攻撃に脆弱 |
| RC4 | 暗号として危殆化 |
| SHA-1 | 衝突攻撃が実証済み |
| 圧縮 | CRIME攻撃に脆弱 |
| 再ネゴシエーション | ダウングレード攻撃の温床 |

> **現場での注意点**: 新規システムでは TLS 1.3 を優先的に使用し、TLS 1.2 を後方互換として許容する設計にしてください。TLS 1.0 / 1.1 はすでに多くのブラウザでサポートが終了しており、PCI DSS でも使用禁止です。

## 3.9 暗号技術の最新動向

### 3.9.1 耐量子暗号（Post-Quantum Cryptography）

量子コンピュータの発展により、現在の公開鍵暗号（RSA、楕円曲線暗号）が将来的に解読される可能性があります。NISTは耐量子暗号の標準化を進めています。

| 量子コンピュータの影響 | 現在の暗号 | 影響 |
|---------------------|-----------|------|
| Shorのアルゴリズム | RSA、ECC、DH | 破られる（鍵長に対して指数関数的に高速化） |
| Groverのアルゴリズム | AES、SHA | 安全性が半減（鍵長を2倍にすれば対応可能） |

| NIST標準化候補 | 種類 | 基盤となる問題 |
|---------------|------|---------------|
| **ML-KEM（CRYSTALS-Kyber）** | 鍵カプセル化 | 格子問題 |
| **ML-DSA（CRYSTALS-Dilithium）** | デジタル署名 | 格子問題 |
| **SLH-DSA（SPHINCS+）** | デジタル署名 | ハッシュベース |

### 3.9.2 暗号の危殆化への対応

暗号技術は永遠に安全ではありません。計算能力の向上や新しい攻撃手法の発見により、安全性が低下します。これを**危殆化**（きたいか）と呼びます。

| 対応策 | 説明 |
|--------|------|
| **暗号アジリティ** | 暗号アルゴリズムを容易に切り替えられる設計にすることです |
| **鍵長の見直し** | 定期的に推奨鍵長を確認し、必要に応じて更新します |
| **CRYPTREC暗号リスト** | 日本政府が推奨する暗号アルゴリズムリストを参照します |

## 3.10 まとめ

### 重要ポイント

1. **共通鍵暗号**は高速だが鍵配送が課題、**公開鍵暗号**は低速だが鍵配送問題を解決します
2. 実用的には**ハイブリッド暗号方式**（鍵交換は公開鍵、データ暗号化は共通鍵）が使われます
3. AES を使う際は**GCM等の認証付き暗号化モード**を選択し、ECBモードは使用禁止です
4. ハッシュ関数は**SHA-256以上**を使用し、MD5/SHA-1は使用禁止です
5. デジタル署名は**認証・完全性・否認防止**の3つを保証します
6. PKI は**信頼の連鎖**により公開鍵の正当性を保証します
7. TLS 1.3 では前方秘匿性が必須となり、多くの旧方式が廃止されました

### 試験対策キーワード

- 共通鍵暗号（AES、ブロック暗号、ストリーム暗号、暗号化モード）
- 公開鍵暗号（RSA、楕円曲線暗号、鍵配送問題）
- ハイブリッド暗号方式、Diffie-Hellman鍵交換、前方秘匿性
- ハッシュ関数（SHA-256、衝突耐性、一方向性、HMAC）
- デジタル署名（認証、完全性、否認防止）
- PKI、認証局（CA）、電子証明書（X.509）、証明書チェーン
- TLS 1.3、ハンドシェイク、暗号スイート
- ケルクホフスの原則、暗号の危殆化、CRYPTREC

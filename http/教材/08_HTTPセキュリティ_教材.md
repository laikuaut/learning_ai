## 8.2 XSS（クロスサイトスクリプティング）

### XSSの仕組み

XSS（Cross-Site Scripting）は、攻撃者が悪意あるスクリプトをWebページに注入し、他のユーザーのブラウザで実行させる攻撃です。

```
【反射型XSSの攻撃フロー】

攻撃者 → 被害者に悪意あるURLを送る
  https://example.com/search?q=<script>alert('XSS')</script>

被害者 → URLにアクセス
  ↓
サーバー → 検索クエリをそのままHTMLに埋め込んでレスポンス
  <p>検索結果: <script>alert('XSS')</script></p>
  ↓
被害者のブラウザ → スクリプトが実行される！
  → Cookie窃取、偽フォーム表示、リダイレクトなど
```

### XSSの対策

```
【XSS対策】

1. 出力時のエスケープ（最重要）
   → HTMLに出力する際、< > " ' & をエスケープする
   → テンプレートエンジンの自動エスケープ機能を使う

2. Content Security Policy（CSP）ヘッダー
   → Content-Security-Policy: default-src 'self';
   → インラインスクリプトの実行を禁止

3. HttpOnly Cookie
   → Set-Cookie: session=xxx; HttpOnly
   → JavaScriptからCookieにアクセスできなくする

4. X-XSS-Protection ヘッダー
   → X-XSS-Protection: 1; mode=block
   → ブラウザのXSSフィルターを有効化
```

## 8.3 CSRF（クロスサイトリクエストフォージェリ）

### CSRFの仕組み

CSRF（Cross-Site Request Forgery）は、ログイン済みのユーザーに意図しないリクエストを送らせる攻撃です。

```
【CSRFの攻撃フロー】

1. 被害者が bank.example.com にログイン済み
   → ブラウザにセッションCookieが保存されている

2. 攻撃者が罠サイト（evil.example.com）に以下のHTMLを設置:
   <img src="https://bank.example.com/transfer?to=attacker&amount=100000">

3. 被害者が罠サイトにアクセス
   → ブラウザが自動的にCookieを付けて bank.example.com にリクエスト
   → 被害者の口座から攻撃者に送金されてしまう！
```

### CSRFの対策

```
【CSRF対策】

1. CSRFトークン（最も一般的）
   → サーバーがフォームに一意のトークンを埋め込む
   → リクエスト時にトークンを検証する
   → 攻撃者はトークンの値を知ることができない

2. SameSite Cookie属性
   → Set-Cookie: session=xxx; SameSite=Lax
   → 異なるサイトからのリクエストにCookieを送らない

3. Referer / Origin ヘッダーの検証
   → リクエスト元が自サイトか確認する
```

## 8.4 SQLインジェクション

### SQLインジェクションの仕組み

```
【SQLインジェクションの例】

ログインフォーム:
  ユーザー名: admin
  パスワード: ' OR '1'='1

生成されるSQL（危険！）:
  SELECT * FROM users
  WHERE username = 'admin' AND password = '' OR '1'='1'
  → '1'='1' は常にTRUE → 認証をバイパス！
```

### SQLインジェクションの対策

```python
# ❌ 危険：文字列連結でSQLを組み立て
query = f"SELECT * FROM users WHERE username = '{username}'"

# ✅ 安全：パラメータ化クエリ（プレースホルダ）
cursor.execute(
    "SELECT * FROM users WHERE username = %s",
    (username,)
)
```

```
【SQLインジェクション対策のまとめ】

1. パラメータ化クエリ（プリペアドステートメント）を使う（必須）
2. ORM（SQLAlchemy, Prisma等）を使う
3. 入力値のバリデーション（型、長さ、形式）
4. データベースユーザーの権限を最小限にする
5. エラーメッセージにSQL情報を含めない
```

## 8.5 セキュリティヘッダー

### 主要なセキュリティヘッダー

```
【HTTP レスポンスに設定すべきセキュリティヘッダー】

ヘッダー                          目的
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
Strict-Transport-Security        HTTPS強制（HSTSプリロード）
Content-Security-Policy          リソース読込元の制限

Permissions-Policy               ブラウザ機能のアクセス制限
```

## 8.6 安全なHTTP通信のベストプラクティス

```
【セキュリティチェックリスト】

□ 通信
  ├─ □ HTTPS を強制する（HTTP → HTTPS リダイレクト）
  ├─ □ HSTS ヘッダーを設定する
  └─ □ TLS 1.2 以上のみ許可する

□ 認証
  ├─ □ パスワードはbcrypt等でハッシュ化して保存
  ├─ □ JWTの有効期限を短く設定する（15分〜1時間）
  ├─ □ ブルートフォース対策（レートリミット、アカウントロック）
  └─ □ 多要素認証（MFA）を導入する

□ Cookie
  ├─ □ Secure 属性（HTTPSのみ送信）
  ├─ □ HttpOnly 属性（JavaScript からアクセス不可）
  └─ □ SameSite 属性（CSRF対策）

□ 入力/出力
  ├─ □ すべての入力値をバリデーションする
  ├─ □ 出力時にHTMLエスケープする
  ├─ □ SQLはパラメータ化クエリを使う
  └─ □ Content-Security-Policy を設定する

□ ヘッダー
  ├─ □ セキュリティヘッダーを設定する
  ├─ □ サーバーバージョン情報を隠す
  └─ □ CORSを適切に設定する（* は避ける）

□ エラー
  ├─ □ 内部情報（スタックトレース等）をレスポンスに含めない
  └─ □ 一般的なエラーメッセージを返す
```

> **実務でのポイント**: セキュリティは「あとから追加する」ものではなく、設計段階から組み込むものです。特に認証・認可、入力値の検証、HTTPS強制は、開発の最初から対応しましょう。

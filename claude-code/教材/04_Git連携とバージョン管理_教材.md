# 第4章：Git連携とバージョン管理

> **この章のゴール**: Claude CodeのGit連携機能を理解し、コミット・ブランチ管理・プルリクエスト作成を効率的に行えるようになること

---

## 4.1 Claude CodeとGitの統合

Claude Codeは**Git**（ギット：分散型バージョン管理システム）と深く統合されています。コードの変更管理、コミットの作成、プルリクエストの管理などをAIの支援で効率的に行えます。

### Git連携の全体像

```
┌─────────────────────────────────────────────────────┐
│            Claude Code × Git ワークフロー             │
│                                                      │
│  コード変更 ──→ 差分確認 ──→ コミット ──→ プッシュ  │
│      │            │           │            │         │
│      ▼            ▼           ▼            ▼         │
│   Edit/Write   git diff   git commit   git push     │
│                git status  メッセージ    ブランチ     │
│                            自動生成                   │
│                                                      │
│  ブランチ作成 ──→ 開発 ──→ PR作成 ──→ レビュー対応  │
│      │                      │            │           │
│      ▼                      ▼            ▼           │
│   git branch            gh pr create  コメント対応   │
│   git checkout                                       │
└─────────────────────────────────────────────────────┘
```

### 安全機能

Claude Codeには、Git操作の安全性を確保するための仕組みが組み込まれています：

| 安全機能 | 説明 |
|---|---|
| **force push 禁止** | `main`/`master`ブランチへのforce pushを防止します |
| **確認プロンプト** | 破壊的な操作（reset --hard等）は事前に確認します |
| **新規コミット優先** | `--amend`ではなく新しいコミットの作成を優先します |
| **機密ファイル保護** | `.env`等の機密ファイルのコミットを警告します |
| **フック尊重** | pre-commitフック等をスキップしません |

### ポイントまとめ
- Claude CodeはGitと深く統合されている
- 安全機能により、破壊的な操作を未然に防止
- コミットからPR作成まで一連の流れをサポート

## 4.2 コミットの作成

### /commit スラッシュコマンド

最も簡単なコミット方法は `/commit` コマンドです：

```
> /commit
```

Claude Codeは以下の手順で自動的にコミットを作成します：

```
1. git status で変更ファイルを確認
2. git diff で変更内容を分析
3. git log で過去のコミットメッセージのスタイルを確認
4. 変更内容を要約したコミットメッセージを生成
5. ユーザーの承認後にコミットを実行
```

### 自然言語でのコミット指示

```
> この変更をコミットしてください
```

```
> ログイン機能の追加としてコミットしてください
```

### コミットメッセージの例

Claude Codeが生成するコミットメッセージの例：

```
Add email validation to user registration form

- Implement regex-based email format checking
- Add error message display for invalid emails
- Add unit tests for validation function
```

特徴：
- 1行目に変更の要約（英語の場合は動詞で始める）
- 空行を挟んで詳細な説明
- リポジトリの既存のスタイルに合わせる

### よくある間違い
> **コミット前に変更を確認しない**: Claude Codeにコミットを依頼する前に、`git diff`の結果を確認する習慣をつけましょう。「変更内容を確認してからコミットして」と指示すると、差分を表示してから進めてくれます。

### ポイントまとめ
- `/commit`で簡単にコミットを作成できる
- コミットメッセージは変更内容から自動生成される
- リポジトリの既存スタイルに合わせたメッセージになる

## 4.3 差分の確認と分析

### git statusとgit diffの活用

```
> 現在の変更状況を教えてください
```

Claude Codeは`git status`と`git diff`を実行し、変更内容をわかりやすく説明します：

```
現在の変更状況:
- 変更されたファイル（2件）:
  - src/services/auth.ts（ログイン処理にエラーハンドリング追加）
  - src/tests/auth.test.ts（テストケース追加）
- 新規ファイル（1件）:
  - src/utils/validator.ts
- 未追跡ファイル:
  なし
```

### 変更内容のレビュー依頼

```
> 今回の変更内容をレビューして、問題がないか確認してください
```

Claude Codeは差分を分析し、以下のような観点でレビューします：
- コードの品質と可読性
- バグの可能性
- セキュリティ上の懸念
- テストの網羅性

### ポイントまとめ
- 変更状況の確認を自然言語で依頼できる
- Claude Codeが差分を分析してわかりやすく説明してくれる
- コミット前のレビューを依頼できる

## 4.4 ブランチ管理

### ブランチの作成と切り替え

```
> feature/user-profile という名前でブランチを作成して切り替えてください
```

```
Bash: git checkout -b feature/user-profile
→ Switched to a new branch 'feature/user-profile'
```

### ブランチの一覧確認

```
> 現在のブランチ一覧を見せてください
```

### マージ操作

```
> main ブランチの最新を現在のブランチにマージしてください
```

Claude Codeは以下を実行します：

```bash
git fetch origin main
git merge origin/main
```

コンフリクト（conflict：競合）が発生した場合は、Claude Codeが内容を分析して解決策を提案します：

```
コンフリクトが発生しました:
- src/services/auth.ts

コンフリクトの内容を確認して解決しましょうか？
```

### よくある間違い
> **mainブランチで直接作業する**: 機能開発やバグ修正は必ず専用のブランチで行いましょう。Claude Codeに「新しいブランチを作って」と指示してから作業を始めると安全です。

### ポイントまとめ
- ブランチの作成・切り替え・マージを自然言語で操作できる
- コンフリクトの分析と解決もサポート
- feature/xxxのような命名規則に従ったブランチ名を提案

## 4.5 プルリクエストの作成

### PRの作成

Claude Codeは**gh CLI**（GitHub CLI）を使ってプルリクエスト（Pull Request：PR）を作成できます。

```
> この変更でプルリクエストを作成してください
```

Claude Codeは以下の手順で処理します：

```
1. git log でコミット履歴を確認
2. git diff main...HEAD で変更全体を分析
3. PRタイトルとDescription（説明）を自動生成
4. gh pr create コマンドでPRを作成
```

生成されるPRの例：

```markdown
## タイトル
Add email validation to registration form

## Summary
- メール形式のバリデーション機能を追加
- 不正なメールアドレスに対するエラー表示を実装
- バリデーション関数のユニットテストを追加

## Test plan
- [ ] 正常なメールアドレスでの登録テスト
- [ ] 不正なメールアドレスでのエラー表示確認
- [ ] ユニットテストの実行確認
```

### MCP経由のGitHub連携

MCPサーバー（Model Context Protocol Server）を設定すると、GitHub上の操作をさらに効率的に行えます：

```
> PR #42 のレビューコメントを確認して
> レビューの指摘事項を修正して
```

### ポイントまとめ
- `gh`コマンドを通じてPRを作成できる
- PRタイトルと説明文が自動生成される
- レビューコメントへの対応もサポート

## 4.6 コードレビューワークフロー

### PRのレビュー

```
> PR #15 の変更内容をレビューしてください
```

Claude Codeは以下のような観点でレビューします：

```
┌────────────────────────────────────────────────────┐
│           コードレビューの観点                       │
├────────────────────────────────────────────────────┤
│                                                    │
│  ✓ コードの正確性（バグがないか）                  │
│  ✓ セキュリティ（脆弱性がないか）                  │
│  ✓ パフォーマンス（効率的か）                      │
│  ✓ 可読性（わかりやすいか）                        │
│  ✓ テスト（十分にテストされているか）              │
│  ✓ スタイル（プロジェクトの規約に合っているか）    │
│                                                    │
└────────────────────────────────────────────────────┘
```

### レビュー指摘への対応

```
> PR #15 のレビューコメントを確認して、指摘事項を修正してください
```

Claude Codeはレビューコメントを読み、指摘された箇所を修正し、修正内容を新しいコミットとして作成します。

### よくある間違い
> **レビュー指摘を無条件に受け入れる**: Claude Codeの修正提案が常に正しいとは限りません。特にビジネスロジックに関わる変更は、人間の目で確認してから適用しましょう。

## 4.7 Git操作の安全ルール

Claude Codeが遵守するGit操作の安全ルールをまとめます：

```
┌────────────────────────────────────────────────────┐
│         Claude Code の Git 安全ルール               │
├────────────────────────────────────────────────────┤
│                                                    │
│  ◎ 常に新しいコミットを作成（amendより優先）       │
│  ◎ pre-commitフックを尊重（--no-verify禁止）      │
│  ◎ 具体的なファイルをステージング（git add .回避） │
│  ◎ 破壊的操作前にユーザー確認                      │
│                                                    │
│  ✕ main/masterへのforce push                       │
│  ✕ git reset --hard（明示的な指示がない限り）      │
│  ✕ git config の変更                               │
│  ✕ --no-verify でフックをスキップ                  │
│                                                    │
└────────────────────────────────────────────────────┘
```

### ポイントまとめ
- Claude Codeには安全なGit操作のルールが組み込まれている
- 破壊的な操作は明示的な指示がない限り実行しない
- pre-commitフックを尊重し、スキップしない

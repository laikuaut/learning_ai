# 第6章：フックとカスタマイズ

## この章のゴール
- Claude Codeのフック（Hooks）システムの仕組みと用途を理解する
- 各フックタイプ（PreToolUse、PostToolUse、Notification、Stop、SessionStartなど）の違いを把握する
- settings.jsonでのフック設定方法をマスターする
- フックマッチャー（tool_nameパターン）の指定方法を習得する
- フックコマンドで利用できる環境変数を理解する
- 実践的なフック活用例（リント、自動フォーマットなど）を実装できるようになる
- ユーザーフックとプロジェクトフックの違いを理解する
- Claude Codeのカスタマイズオプション（テーマ、モデル選択、権限など）を活用する

---

## 6.1 フックとは何か

### フックシステムの概要

フック（Hook）とは、特定のイベント（event：出来事）が発生したときに自動的に実行されるシェルコマンド（shell command）のことです。Claude Codeのフックシステムを使うと、ツールの実行前後やセッションの開始時などのタイミングで、自分が定義したコマンドを自動実行できます。

### なぜフックが必要なのか

フックを使わない場合、開発者は手動で繰り返し作業を行う必要があります。

```
フックなしの場合：

  Claude Code がファイルを編集
    ↓
  開発者が手動でリンター（linter）を実行
    ↓
  エラーがあれば手動で修正を依頼
    ↓
  再度リンターを実行...

フックありの場合：

  Claude Code がファイルを編集
    ↓
  フックが自動でリンターを実行  ← 自動化！
    ↓
  結果がClaude Codeにフィードバック
    ↓
  必要なら自動で修正
```

### フックのアーキテクチャ

```
┌─────────────────────────────────────────────────────┐
│                Claude Code                           │
│                                                      │
│  ┌──────────┐   ┌──────────┐   ┌──────────────────┐│
│  │ イベント  │──→│ フック    │──→│ シェルコマンド    ││
│  │ 発生     │   │ マッチング│   │ 実行             ││
│  └──────────┘   └──────────┘   └────────┬─────────┘│
│                                          │          │
│                                  ┌───────▼────────┐ │
│                                  │ 結果を         │ │
│                                  │ Claude Codeに  │ │
│                                  │ フィードバック  │ │
│                                  └────────────────┘ │
└─────────────────────────────────────────────────────┘
```

### フックとCLAUDE.mdの違い

フックとCLAUDE.md（プロジェクト指示ファイル）はどちらもClaude Codeの動作をカスタマイズしますが、目的が異なります。

| 項目 | フック | CLAUDE.md |
|---|---|---|
| 実行方式 | シェルコマンドを直接実行 | プロンプトに指示を追加 |
| 実行タイミング | 特定のイベント発生時 | 毎回のセッション全体 |
| 確実性 | コマンドが必ず実行される | LLMが指示に従わない可能性がある |
| 用途 | 自動化・検証・通知 | コーディング規約・プロジェクト説明 |
| 柔軟性 | 決まった処理のみ | 自然言語で柔軟に指示可能 |

### ポイントまとめ
- フックは特定のイベントで自動実行されるシェルコマンドである
- CLAUDE.mdの指示と異なり、フックは確実にコマンドを実行する
- 反復的な検証・フォーマット・通知の自動化に適している

## 6.2 フックタイプ

### 利用可能なフックタイプ一覧

Claude Codeでは以下のフックタイプ（hook type）が利用できます。

| フックタイプ | トリガータイミング | 主な用途 |
|---|---|---|
| `PreToolUse` | ツール実行の**前** | 入力の検証、実行のブロック |
| `PostToolUse` | ツール実行の**後** | 結果の検証、自動フォーマット |
| `Notification` | 通知が送信されるとき | カスタム通知、外部連携 |
| `Stop` | エージェントの応答完了時 | 最終チェック、レポート生成 |
| `SessionStart` | セッション開始時 | 環境セットアップ、初期化処理 |

### PreToolUse フック

PreToolUse（プレツールユース）フックは、Claude Codeがツールを実行する**直前**に呼び出されます。ツールの入力を検証したり、条件に合わない場合は実行をブロックしたりできます。

```
┌──────────┐    ┌──────────┐    ┌──────────┐
│ Claude   │───→│PreToolUse│───→│ ツール    │
│ が判断   │    │ フック    │    │ 実行     │
└──────────┘    └─────┬────┘    └──────────┘
                      │
                      │ exit code 2
                      ↓
                ┌──────────┐
                │ 実行を   │
                │ ブロック │
                └──────────┘
```

**終了コード（exit code）による制御：**

| 終了コード | 動作 |
|---|---|
| `0` | ツール実行を許可（stdoutの内容がコンテキストに追加される） |
| `1` | エラー（Claude Codeがエラーを報告） |
| `2` | ツール実行を**ブロック**（stdoutの内容がフィードバックとしてClaude Codeに渡される） |

### PostToolUse フック

PostToolUse（ポストツールユース）フックは、ツールの実行が**完了した後**に呼び出されます。実行結果の検証や後処理に使います。

```
┌──────────┐    ┌───────────┐    ┌──────────────┐
│ ツール   │───→│PostToolUse│───→│ 結果を       │
│ 実行完了 │    │ フック     │    │ コンテキスト │
└──────────┘    └───────────┘    │ に追加       │
                                 └──────────────┘
```

PostToolUseフックのstdoutは自動的にClaude Codeのコンテキスト（context：文脈情報）に追加されます。これにより、リンターの警告などをClaude Codeにフィードバックできます。

### Notification フック

Notification（通知）フックは、Claude Codeが通知を送信するタイミングで呼び出されます。デスクトップ通知の代わりにカスタム通知（Slack送信、サウンド再生など）を実装できます。

### Stop フック

Stop（ストップ）フックは、Claude Codeがレスポンス（response：応答）の生成を完了したときに呼び出されます。最終的なチェックやレポート生成に使います。

Stopフックがstdoutに出力を返すと、Claude Codeはその内容を見て追加の作業が必要かどうかを判断します。

### SessionStart フック

SessionStart（セッションスタート）フックは、新しいClaude Codeセッションが開始されるときに呼び出されます。環境のセットアップや初期化処理に使います。

### ポイントまとめ
- PreToolUseはツール実行前の検証・ブロックに使う
- PostToolUseはツール実行後の検証・フォーマットに使う
- 終了コード2でPreToolUseフックからツール実行をブロックできる
- フックのstdout出力はClaude Codeのコンテキストに追加される

## 6.3 フックの設定方法

### 基本的な設定フォーマット

フックは`settings.json`の`hooks`キーに設定します。

```json
{
  "hooks": {
    "フックタイプ": [
      {
        "matcher": "マッチャーパターン",
        "command": "実行するシェルコマンド"
      }
    ]
  }
}
```

### 設定フィールドの詳細

| フィールド | 必須 | 説明 |
|---|---|---|
| `matcher` | いいえ | ツール名のパターン（PreToolUse/PostToolUseのみ） |
| `command` | はい | 実行するシェルコマンド |
| `timeout` | いいえ | タイムアウト時間（ミリ秒、デフォルト: 60000） |

### マッチャー（matcher）パターン

`matcher`フィールドは、フックを適用するツールを指定するためのパターンです。PreToolUseとPostToolUseフックでのみ使用します。

| パターン | マッチするツール |
|---|---|
| `"Bash"` | Bashツール |
| `"Write"` | Writeツール |
| `"Edit"` | Editツール |
| `"Read"` | Readツール |
| `"Grep"` | Grepツール |
| `"Glob"` | Globツール |
| 指定なし | すべてのツール |

matcherを省略すると、そのフックタイプに該当するすべてのツール呼び出しでフックが実行されます。

### 複数のフックの設定

同じフックタイプに複数のフックを設定できます。配列内の順序で実行されます。

```json
{
  "hooks": {
    "PostToolUse": [
      {
        "matcher": "Write",
        "command": "npx prettier --write $CLAUDE_FILE_PATH"
      },
      {
        "matcher": "Write",
        "command": "npx eslint --fix $CLAUDE_FILE_PATH"
      }
    ]
  }
}
```

### 設定ファイルの配置場所

フックもMCPサーバーと同様に、スコープごとに設定できます。

```
~/.claude/settings.json          ← ユーザースコープ（全プロジェクト共通）
.claude/settings.json            ← プロジェクトスコープ（チーム共有）
.claude/settings.local.json      ← ローカルスコープ（個人環境）
```

```
┌─────────────────────────────────────────┐
│ ユーザースコープ                          │
│ ~/.claude/settings.json                 │
│ → 全プロジェクトで有効なフック            │
│                                         │
│  ┌──────────────────────────────────┐   │
│  │ プロジェクトスコープ               │   │
│  │ .claude/settings.json            │   │
│  │ → チームで共有するフック           │   │
│  │                                  │   │
│  │  ┌───────────────────────────┐   │   │
│  │  │ ローカルスコープ           │   │   │
│  │  │ .claude/settings.local.json│  │   │
│  │  │ → 個人固有のフック         │   │   │
│  │  └───────────────────────────┘   │   │
│  └──────────────────────────────────┘   │
└─────────────────────────────────────────┘

※ 内側のスコープが優先される
```

### ポイントまとめ
- フックは`hooks`キーの下にフックタイプ別で設定する
- `matcher`でフックを適用するツールを指定できる
- 同じフックタイプに複数のフックを配列で設定可能
- スコープは3段階で、内側が優先される

## 6.4 環境変数

### フックコマンドで利用できる環境変数

フックのコマンド内では、Claude Codeが自動的に設定する環境変数（environment variable）を利用できます。

| 環境変数 | 説明 | 利用可能なフック |
|---|---|---|
| `$CLAUDE_TOOL_NAME` | 実行されるツールの名前 | PreToolUse, PostToolUse |
| `$CLAUDE_FILE_PATH` | 操作対象のファイルパス | PreToolUse, PostToolUse |
| `$CLAUDE_FILE_PATHS` | 操作対象の複数ファイルパス（改行区切り） | PreToolUse, PostToolUse |
| `$CLAUDE_PROJECT_DIR` | プロジェクトのルートディレクトリ | すべて |
| `$CLAUDE_SESSION_ID` | 現在のセッションID | すべて |
| `$CLAUDE_TOOL_INPUT` | ツールへの入力（JSON形式） | PreToolUse |
| `$CLAUDE_TOOL_OUTPUT` | ツールの出力結果 | PostToolUse |

### 環境変数の活用例

**ファイルパスを使ったフォーマット：**

```json
{
  "hooks": {
    "PostToolUse": [
      {
        "matcher": "Write",
        "command": "if [ -n \"$CLAUDE_FILE_PATH\" ] && echo \"$CLAUDE_FILE_PATH\" | grep -q '\\.py$'; then black \"$CLAUDE_FILE_PATH\"; fi"
      }
    ]
  }
}
```

この例では、Writeツールで`.py`ファイルが書き込まれた後、自動的にBlack（Pythonのフォーマッター）が実行されます。

**ツール入力の検証：**

```json
{
  "hooks": {
    "PreToolUse": [
      {
        "matcher": "Bash",
        "command": "echo \"$CLAUDE_TOOL_INPUT\" | python3 -c \"import sys,json; data=json.load(sys.stdin); cmd=data.get('command',''); sys.exit(2) if 'rm -rf' in cmd else sys.exit(0)\""
      }
    ]
  }
}
```

この例では、Bashツールで`rm -rf`を含むコマンドが実行されようとした場合、終了コード2を返して実行をブロックします。

### ポイントまとめ
- フックコマンドでは自動設定される環境変数が利用可能
- `$CLAUDE_FILE_PATH`で操作対象ファイルを特定できる
- `$CLAUDE_TOOL_INPUT`でツールへの入力内容を検証できる
- 環境変数を組み合わせることで柔軟な条件分岐が実現できる

## 6.5 実践的なフック活用例

### 例1：Pythonファイルの自動フォーマット

Writeツールでpythonファイルが作成・編集された後に自動でBlackフォーマッターを実行します。

```json
{
  "hooks": {
    "PostToolUse": [
      {
        "matcher": "Write",
        "command": "if echo \"$CLAUDE_FILE_PATH\" | grep -qE '\\.py$'; then black --quiet \"$CLAUDE_FILE_PATH\" 2>&1; fi"
      }
    ]
  }
}
```

### 例2：TypeScript/JavaScriptの自動リント

ファイル編集後にESLintを実行し、問題があればClaude Codeにフィードバックします。

```json
{
  "hooks": {
    "PostToolUse": [
      {
        "matcher": "Write",
        "command": "if echo \"$CLAUDE_FILE_PATH\" | grep -qE '\\.(ts|tsx|js|jsx)$'; then npx eslint --no-error-on-unmatched-pattern \"$CLAUDE_FILE_PATH\" 2>&1; fi"
      },
      {
        "matcher": "Edit",
        "command": "if echo \"$CLAUDE_FILE_PATH\" | grep -qE '\\.(ts|tsx|js|jsx)$'; then npx eslint --no-error-on-unmatched-pattern \"$CLAUDE_FILE_PATH\" 2>&1; fi"
      }
    ]
  }
}
```

### 例3：危険なコマンドのブロック

本番環境への意図しない変更を防止するため、特定のコマンドをブロックします。

```json
{
  "hooks": {
    "PreToolUse": [
      {
        "matcher": "Bash",
        "command": "echo \"$CLAUDE_TOOL_INPUT\" | python3 -c \"\nimport sys, json\ndata = json.load(sys.stdin)\ncmd = data.get('command', '')\nblocked = ['rm -rf /', 'DROP DATABASE', 'git push --force']\nfor b in blocked:\n    if b in cmd:\n        print(f'ブロック: {b} を含むコマンドは安全のため実行できません')\n        sys.exit(2)\nsys.exit(0)\n\""
      }
    ]
  }
}
```

### 例4：セッション開始時の環境チェック

SessionStartフックで、開発環境が正しくセットアップされているか確認します。

```json
{
  "hooks": {
    "SessionStart": [
      {
        "command": "echo '=== 環境チェック ===' && node --version && npm --version && python3 --version && echo '=== チェック完了 ==='"
      }
    ]
  }
}
```

### 例5：カスタム通知

タスク完了時にカスタム通知を送信します。

```json
{
  "hooks": {
    "Notification": [
      {
        "command": "notify-send 'Claude Code' \"$CLAUDE_NOTIFICATION_MESSAGE\" 2>/dev/null || true"
      }
    ]
  }
}
```

### 例6：Stopフックによる自動テスト

Claude Codeの応答が完了するたびに、変更されたファイルに関連するテストを自動実行します。

```json
{
  "hooks": {
    "Stop": [
      {
        "command": "cd $CLAUDE_PROJECT_DIR && npm test 2>&1 | tail -20"
      }
    ]
  }
}
```

### フック設定の全体像

以下は、実際のプロジェクトで使う設定の全体像です。

```json
{
  "hooks": {
    "SessionStart": [
      {
        "command": "echo '環境: Node' $(node -v) ', Python' $(python3 -V 2>&1 | cut -d' ' -f2)"
      }
    ],
    "PreToolUse": [
      {
        "matcher": "Bash",
        "command": "echo \"$CLAUDE_TOOL_INPUT\" | python3 -c \"import sys,json; d=json.load(sys.stdin); c=d.get('command',''); sys.exit(2) if any(x in c for x in ['rm -rf /','DROP DATABASE']) else sys.exit(0)\""
      }
    ],
    "PostToolUse": [
      {
        "matcher": "Write",
        "command": "if echo \"$CLAUDE_FILE_PATH\" | grep -qE '\\.py$'; then black --quiet \"$CLAUDE_FILE_PATH\" 2>&1; fi"
      },
      {
        "matcher": "Write",
        "command": "if echo \"$CLAUDE_FILE_PATH\" | grep -qE '\\.(ts|tsx|js|jsx)$'; then npx prettier --write \"$CLAUDE_FILE_PATH\" 2>&1; fi"
      }
    ],
    "Stop": [
      {
        "command": "cd $CLAUDE_PROJECT_DIR && npm test --silent 2>&1 | tail -5"
      }
    ]
  }
}
```

### ポイントまとめ
- PostToolUseフックで自動フォーマット・リントを実装できる
- PreToolUseフックで危険なコマンドをブロックできる
- SessionStartフックで環境チェックを自動化できる
- 複数のフックを組み合わせて堅牢な開発ワークフローを構築できる

## 6.6 カスタマイズオプション

### テーマの設定

Claude Codeのテーマ（theme：見た目のスタイル）を変更できます。

```bash
# テーマの変更
/config

# 利用可能なテーマ
# - light（ライト）
# - dark（ダーク）
# - light-daltonized（色覚多様性対応ライト）
# - dark-daltonized（色覚多様性対応ダーク）
```

### モデルの選択

使用するAIモデル（model）を切り替えることができます。

```bash
# モデルの切り替え（セッション内）
/model

# 利用可能なモデル例
# - claude-sonnet-4-20250514
# - claude-opus-4-20250918
```

### 権限設定（Permissions）

Claude Codeがどのツールをどの範囲で使用できるかを設定します。

```json
{
  "allowedTools": [
    "Bash(npm test)",
    "Bash(npm run lint)",
    "Write",
    "Edit",
    "mcp__github__*"
  ],
  "deniedTools": [
    "Bash(curl *)",
    "Bash(wget *)"
  ]
}
```

**allowedToolsのパターン構文：**

| パターン | 説明 |
|---|---|
| `"Bash"` | すべてのBashコマンドを許可 |
| `"Bash(npm test)"` | `npm test`コマンドのみ許可 |
| `"Bash(npm *)"` | npm系コマンドをすべて許可 |
| `"Write"` | すべてのファイル書き込みを許可 |
| `"mcp__github__*"` | GitHub MCPの全ツールを許可 |

### 設定ファイルの全体構造

`settings.json`の全体構造を把握しておきましょう。

```json
{
  "mcpServers": {
    "サーバー名": {
      "command": "...",
      "args": ["..."],
      "env": {}
    }
  },
  "hooks": {
    "PreToolUse": [],
    "PostToolUse": [],
    "Notification": [],
    "Stop": [],
    "SessionStart": []
  },
  "allowedTools": [],
  "deniedTools": [],
  "env": {
    "グローバル環境変数": "値"
  }
}
```

### settings.json と settings.local.json の使い分け

```
.claude/settings.json（チーム共有）
  ├── チーム共通のフック設定
  ├── 共有MCPサーバー（認証不要のもの）
  ├── 共通の権限設定
  └── Gitにコミットする

.claude/settings.local.json（個人用）
  ├── 個人の認証トークン
  ├── ローカル環境固有のパス設定
  ├── 個人的なフック（カスタム通知など）
  └── Gitにコミットしない
```

### ポイントまとめ
- テーマやモデルはセッション内で簡単に切り替え可能
- `allowedTools`/`deniedTools`で権限を細かく制御できる
- パターン構文でワイルドカードが使える
- チーム共有の設定と個人の設定は別ファイルに分離する

## 6.7 よくある間違い

### フック設定でのよくある間違い

**間違い1：matcherの指定ミス**

```json
// NG：matcherの値が間違っている（小文字にしてしまう）
{
  "hooks": {
    "PostToolUse": [
      {
        "matcher": "write",
        "command": "echo 'file written'"
      }
    ]
  }
}

// OK：ツール名は正確に指定する（大文字始まり）
{
  "hooks": {
    "PostToolUse": [
      {
        "matcher": "Write",
        "command": "echo 'file written'"
      }
    ]
  }
}
```

**間違い2：フックコマンドのクォート処理**

```json
// NG：JSONの中でダブルクォートがエスケープされていない
{
  "hooks": {
    "PostToolUse": [
      {
        "command": "echo "hello world""
      }
    ]
  }
}

// OK：ダブルクォートを正しくエスケープ
{
  "hooks": {
    "PostToolUse": [
      {
        "command": "echo \"hello world\""
      }
    ]
  }
}
```

**間違い3：Notificationフックにmatcherをつけてしまう**

```json
// NG：Notificationフックにmatcherは不要（無視される）
{
  "hooks": {
    "Notification": [
      {
        "matcher": "Bash",
        "command": "notify-send 'done'"
      }
    ]
  }
}

// OK：Notificationにはmatcherは不要
{
  "hooks": {
    "Notification": [
      {
        "command": "notify-send 'done'"
      }
    ]
  }
}
```

**間違い4：フックコマンドがPATHにない**

```json
// NG：blackがインストールされていない、またはPATHが通っていない
{
  "hooks": {
    "PostToolUse": [
      {
        "matcher": "Write",
        "command": "black $CLAUDE_FILE_PATH"
      }
    ]
  }
}

// OK：フルパスを指定するか、事前にインストールを確認
{
  "hooks": {
    "PostToolUse": [
      {
        "matcher": "Write",
        "command": "which black > /dev/null 2>&1 && black \"$CLAUDE_FILE_PATH\" || true"
      }
    ]
  }
}
```

**間違い5：環境変数をクォートしない**

```json
// NG：スペースを含むファイルパスで壊れる
{
  "command": "black $CLAUDE_FILE_PATH"
}

// OK：環境変数をダブルクォートで囲む
{
  "command": "black \"$CLAUDE_FILE_PATH\""
}
```

### ポイントまとめ
- matcherのツール名は大文字・小文字を正確に指定する
- JSONのダブルクォートは正しくエスケープする
- matcherはPreToolUseとPostToolUseでのみ有効
- 環境変数は必ずダブルクォートで囲む
- コマンドの存在確認を行い、なくてもエラーにならないようにする

## 6.8 まとめ

この章では、Claude Codeのフックシステムとカスタマイズ方法を学びました。

### 章全体のポイントまとめ

1. **フックの基本**: フックは特定のイベントで自動実行されるシェルコマンドである
2. **フックタイプ**: PreToolUse、PostToolUse、Notification、Stop、SessionStartの5つのタイプがある
3. **マッチャー**: `matcher`フィールドでフックを適用するツールを限定できる
4. **環境変数**: `$CLAUDE_FILE_PATH`や`$CLAUDE_TOOL_INPUT`などの環境変数がフック内で利用可能
5. **終了コード**: PreToolUseフックで終了コード2を返すとツール実行をブロックできる
6. **実践活用**: 自動フォーマット、リント、危険コマンドのブロック、環境チェックなどに活用可能
7. **カスタマイズ**: テーマ、モデル、権限設定でClaude Codeの動作を制御できる
8. **設定の分離**: チーム共有の設定と個人の設定は別ファイルに管理する

次の章では、Claude Codeを使った実践的な開発ワークフローについて学びます。これまでに学んだコマンド操作、CLAUDE.md、MCPサーバー、フックの知識を組み合わせて、実際のプロジェクトでClaude Codeを最大限に活用する方法を習得します。

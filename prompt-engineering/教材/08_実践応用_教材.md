## 8.1 業務自動化への活用

### 自動化に適したタスク

LLMを使った業務自動化に適したタスクには共通の特徴があります。

```
自動化に適したタスク：
  ✅ 入力と出力が明確に定義できる
  ✅ 正解の基準がある（検証可能）
  ✅ 繰り返し発生する
  ✅ 人間がやると時間がかかる
  ✅ 完全な正確性より効率が重要（人間が確認可能）

自動化に向かないタスク：
  ❌ 正解が主観的で判断が難しい
  ❌ 間違いが許されない（法的文書、医療判断）
  ❌ 1回限りの特殊なタスク
  ❌ 最新情報が必須で、RAGでもカバーできない
```

### 業務自動化の実例

#### 1. メール自動分類と返信案生成

```
あなたはカスタマーサポートのアシスタントです。

以下のメールを分析し、指定された形式で出力してください。

## 分析項目
1. カテゴリ：技術サポート / 請求 / 機能要望 / クレーム / その他
2. 緊急度：高 / 中 / 低
3. 感情：ポジティブ / ニュートラル / ネガティブ
4. 要約：1文で要約
5. 返信案：丁寧で共感的なトーンで、200文字以内

## メール
"""
件名：ログインできなくなりました

本文：
昨日からログインできません。パスワードリセットも試しましたが
メールが届きません。至急対応をお願いします。
来週のプレゼンに必要なデータが見られなくて困っています。
"""

## 出力
```

#### 2. コミットメッセージの自動生成

```
以下のgit diffからコミットメッセージを生成してください。

ルール：
- Conventional Commits形式（feat/fix/refactor/docs/test）
- 1行目は50文字以内の要約
- 必要に応じて本文で詳細を説明
- 日本語で記述

diff：
"""
（git diffの内容）
"""
```

#### 3. ドキュメントの自動生成

```
以下のPython関数からAPIドキュメントを生成してください。

出力形式：OpenAPI（Swagger）のYAML形式

含める情報：
- エンドポイントのパス
- HTTPメソッド
- リクエストパラメータ（型、必須/任意、説明）
- レスポンス（ステータスコード、スキーマ、例）
- エラーレスポンス

コード：
"""
（対象のコード）
"""
```

> **ポイントまとめ**: 業務自動化では「100%の精度」を求めず、「人間が確認・修正するコストが手作業より少ない」ことを基準にしましょう。

## 8.2 文章校正・添削

### 基本の校正プロンプト

```
以下の文章を校正してください。

校正の観点：
1. 誤字・脱字
2. 文法的な誤り
3. 表現の不自然さ
4. 句読点の使い方
5. 敬語の適切さ

出力形式：
- 修正箇所を明示する（修正前 → 修正後）
- 修正理由を簡潔に記載する
- 修正後の全文を最後に掲載する

文章：
"""
（校正対象の文章）
"""
```

### 文体・トーンの統一チェック

```
以下の技術文書が、指定されたスタイルガイドに沿っているか
チェックしてください。

スタイルガイド：
- 敬体（です・ます調）で統一
- 専門用語は初出時に英語を併記（例：変数（variable））
- 一文は60文字以内を目安
- 箇条書きを積極的に使用
- 主語と述語を近くに配置

チェック結果を以下の形式で報告してください：
| 行番号 | 指摘内容 | 改善案 |
```

### ビジネス文書の品質向上

```
以下のビジネスメールを改善してください。

改善の方針：
- 結論を先に述べる（PREP法）
- 箇条書きで要点を明確にする
- アクションアイテムを明示する
- 敬語を適切に使用する
- 全体を300文字以内に収める

原文：
"""
（改善対象のメール）
"""
```

> **ポイントまとめ**: 校正では「観点」を明確に指定し、修正理由も含めることで、学習効果も得られます。

## 8.3 データ分析

### 構造化データの分析

```
以下のCSVデータを分析してください。

データ：
"""
月,売上,顧客数,平均単価
1月,1200000,150,8000
2月,980000,120,8167
3月,1500000,180,8333
4月,1100000,140,7857
5月,1800000,200,9000
6月,1350000,160,8438
"""

分析してほしいこと：
1. 基本統計量（平均、最大、最小、標準偏差）
2. トレンド分析（増減の傾向）
3. 異常値の有無
4. 相関分析（売上と顧客数の関係）
5. 来月の予測（線形トレンドで）

出力形式：
- 各分析項目について見出し付きで記述
- 数値は小数点以下1桁まで
- ビジネス上の示唆（アクションにつながる提言）を含める
```

### SQLクエリの生成

```
以下のテーブル構造に基づいて、指定された条件のSQLクエリを
生成してください。

テーブル構造：
- users (id, name, email, created_at, plan_type)
- orders (id, user_id, amount, status, created_at)
- products (id, name, price, category)
- order_items (id, order_id, product_id, quantity)

条件：
「2026年1月から3月の間に、合計注文金額が10万円以上の
ユーザーを、注文金額の降順で一覧表示する。
ユーザー名、メール、合計注文金額、注文回数を含める。」

制約：
- PostgreSQL構文を使用
- JOINはINNER JOINを使用
- コメントを含める
- インデントを整える
```

### ログ分析

```
以下のアプリケーションログから問題を分析してください。

分析観点：
1. エラーの頻度とパターン
2. エラーが集中している時間帯
3. 最も影響の大きいエラー
4. 根本原因の推測
5. 対応の優先順位

ログ：
"""
（アプリケーションログ）
"""
```

> **ポイントまとめ**: データ分析では「分析してほしいこと」を具体的に列挙し、ビジネス上の示唆まで求めると実用的な結果が得られます。

## 8.4 マルチモーダルプロンプト

### マルチモーダルとは

テキストだけでなく、画像や音声など複数の入力形式（モダリティ）を組み合わせてLLMに指示するアプローチです。Claude 4やGPT-4oなどが対応しています。

```
マルチモーダルの入力パターン：

テキスト + 画像   → UIのレビュー、グラフの解析
テキスト + PDF    → 文書の要約、情報抽出
テキスト + コード  → コードの説明、レビュー
```

### 画像を使ったプロンプト例

#### UI/UXレビュー

```
添付のスクリーンショットは、ECサイトの商品一覧ページです。
以下の観点でUIレビューを行ってください。

レビュー観点：
1. レイアウト：情報の優先順位は適切か
2. 可読性：テキストは読みやすいか
3. アクセシビリティ：色のコントラストは十分か
4. モバイル対応：小さい画面で使いやすいか
5. UXの改善提案：ユーザー体験を向上させるには

出力形式：
各観点について「良い点」と「改善点」を記述してください。
改善点には具体的な修正案を含めてください。
```

#### エラー画面の分析

```
添付のスクリーンショットはエラー画面です。

以下を分析してください：
1. エラーの種類と内容
2. 推定される原因
3. ユーザーが取るべきアクション
4. 開発者が確認すべきこと
```

#### ダイアグラムの解析

```
添付の画像はシステムアーキテクチャ図です。

以下を分析してください：
1. システムの全体構成を説明してください
2. 各コンポーネントの役割を特定してください
3. 単一障害点（SPOF）はありますか？
4. スケーラビリティの観点で改善点はありますか？
```

> **ポイントまとめ**: マルチモーダルプロンプトでも、テキストでの指示（観点、出力形式）を明確にすることが重要です。

## 8.5 プロンプトの運用・管理

### プロンプトのバージョン管理

プロンプトもコードと同様にバージョン管理すべきです。

```
プロンプト管理のベストプラクティス：

1. バージョン番号を付ける
   v1.0 → v1.1 → v2.0

2. 変更履歴を記録する
   | バージョン | 日付       | 変更内容              | 結果         |
   |-----------|-----------|----------------------|-------------|
   | v1.0      | 2026/03/01| 初版作成              | 精度70%     |
   | v1.1      | 2026/03/10| ペルソナ追加           | 精度78%     |
   | v2.0      | 2026/03/20| Few-Shot追加、出力形式変更| 精度88%  |

3. テスト結果を紐づける
   各バージョンでのテスト結果を保存

4. ロールバック可能にする
   問題があれば前のバージョンに戻せるように
```

### プロンプトテンプレートライブラリ

チームで使うプロンプトをテンプレートとして管理します。

```
プロンプトテンプレートの構成：

templates/
├── code-review.md        # コードレビュー用
├── bug-analysis.md       # バグ分析用
├── meeting-summary.md    # 議事録要約用
├── email-draft.md        # メール下書き用
├── data-analysis.md      # データ分析用
└── README.md             # 使い方ガイド

各テンプレートに含める情報：
- テンプレート名
- 用途の説明
- 入力パラメータ（カスタマイズ箇所）
- 使用例
- 注意事項
```

### プロンプトの品質維持

```
品質維持のチェックリスト：

□ モデルのアップデート後に動作確認しているか
  → モデル更新で挙動が変わることがある

□ 定期的にテストを実行しているか
  → 月1回は回帰テストを実施

□ ユーザーフィードバックを収集しているか
  → 良かった/悪かったのフィードバックを分析

□ プロンプトの肥大化を防いでいるか
  → 不要な指示や制約を定期的に棚卸し

□ チームでの共有と改善を行っているか
  → 個人の暗黙知をチームの形式知に変換
```

> **ポイントまとめ**: プロンプトはコードと同様に、バージョン管理・テスト・継続的改善の対象です。

## 基本問題

### 問題1: Git Flow に従ったブランチ運用

以下のシナリオに基づいて、Git Flow モデルのブランチを作成・運用してください。

シナリオ：新機能「ユーザープロフィール画面」を開発する

1. `develop` ブランチを作成する
2. `develop` から `feature/user-profile` ブランチを作成する
3. 機能を実装してコミットする
4. `develop` にマージする

期待される `git log` の出力：
```
$ git log --oneline --graph
*   abc1234 (HEAD -> develop) Merge branch 'feature/user-profile' into develop
|\
| * def5678 feat: ユーザープロフィール画面を追加
|/
* ghi9012 (main) Initial commit
```

<details>
<summary>ヒント</summary>

Git Flow では `main`（本番）、`develop`（開発）、`feature/*`（機能開発）、`release/*`（リリース準備）、`hotfix/*`（緊急修正）の5種類のブランチを使います。`--no-ff` オプションでマージコミットを明示的に作成するのがポイントです。

</details>

<details>
<summary>解答</summary>

```bash
# 1. 初期状態を作成（main ブランチ）
$ mkdir git-flow-practice && cd git-flow-practice
$ git init -b main
$ echo "# Project" > README.md
$ git add README.md
$ git commit -m "Initial commit"

# 2. develop ブランチを作成
$ git switch -c develop

# 3. feature ブランチを作成
$ git switch -c feature/user-profile

# 4. 機能を実装
$ cat << 'EOF' > profile.html
<!DOCTYPE html>
<html lang="ja">
<head>
  <meta charset="UTF-8">
  <title>プロフィール</title>
</head>
<body>
  <h1>ユーザープロフィール</h1>
  <p>名前: 田中太郎</p>
</body>
</html>
EOF

$ git add profile.html
$ git commit -m "feat: ユーザープロフィール画面を追加"

# 5. develop にマージ（--no-ff でマージコミットを作成）
$ git switch develop
$ git merge --no-ff feature/user-profile -m "Merge branch 'feature/user-profile' into develop"

# 6. feature ブランチを削除
$ git branch -d feature/user-profile

# 7. 結果を確認
$ git log --oneline --graph
```

解説：
- `--no-ff`（no fast-forward）を使うと、マージコミットが必ず作成されます
- マージコミットがあると、どの時点で機能がマージされたかが履歴から明確にわかります
- `feature` ブランチはマージ後に削除するのがベストプラクティスです
- Git Flow は大規模プロジェクトやリリースサイクルが明確なプロジェクトに向いています

</details>

### 問題2: Conventional Commits に従ったコミットメッセージ

以下の変更内容に対して、Conventional Commits の規約に従った適切なコミットメッセージを作成してください。

| 変更内容 | コミットメッセージを考えてください |
|---|---|
| ログイン機能を新規実装した | ? |
| ログインのパスワード検証バグを修正した | ? |
| README にセットアップ手順を追加した | ? |
| ESLintの設定を変更した | ? |
| ログイン処理のリファクタリングを行った | ? |
| ログインのテストを追加した | ? |

期待される解答例：
```
feat: ログイン機能を実装
fix: パスワード検証の不具合を修正
docs: READMEにセットアップ手順を追加
chore: ESLintの設定を更新
refactor: ログイン処理のコードを整理
test: ログイン機能のユニットテストを追加
```

<details>
<summary>ヒント</summary>

Conventional Commits の主な型（type）は以下の通りです：
- `feat`: 新機能
- `fix`: バグ修正
- `docs`: ドキュメントのみの変更
- `style`: コードの意味に影響しない変更（空白、フォーマットなど）
- `refactor`: バグ修正でも機能追加でもないコードの変更
- `test`: テストの追加・修正
- `chore`: ビルドプロセスやツール設定の変更

</details>

<details>
<summary>解答</summary>

```bash
# 正しいコミットメッセージの例

# 新機能の追加
$ git commit -m "feat: ログイン機能を実装"

# バグ修正
$ git commit -m "fix: パスワード検証の不具合を修正"

# ドキュメント変更
$ git commit -m "docs: READMEにセットアップ手順を追加"

# ツール設定の変更
$ git commit -m "chore: ESLintの設定を更新"

# リファクタリング
$ git commit -m "refactor: ログイン処理のコードを整理"

# テスト追加
$ git commit -m "test: ログイン機能のユニットテストを追加"

# スコープ付きの例（より詳細に分類する場合）
$ git commit -m "feat(auth): ログイン機能を実装"
$ git commit -m "fix(auth): パスワード検証の不具合を修正"

# 破壊的変更がある場合（BREAKING CHANGE）
$ git commit -m "feat!: 認証APIのレスポンス形式を変更

BREAKING CHANGE: tokenフィールドがaccessTokenに変更されました"
```

解説：
- Conventional Commits の形式は `<type>(<scope>): <description>` です
- `scope` は省略可能ですが、大きなプロジェクトでは変更箇所を明確にするために推奨されます
- `!` を付けると破壊的変更（BREAKING CHANGE）を示します
- この規約に従うと、変更履歴の自動生成やセマンティックバージョニングの自動化が可能になります
- チーム全体で規約を統一することで、履歴の可読性が大幅に向上します

</details>

## 応用問題

### 問題3: GitHub Flow でのチーム開発シミュレーション

GitHub Flow に従って、以下のシナリオをシミュレーションしてください。一人で2つの役割（開発者とレビュアー）を演じます。

シナリオ：
1. 開発者：`feature/search` ブランチで検索機能を実装する
2. 開発者：プッシュしてPRを作成する
3. レビュアー：PRにコメントして修正を依頼する
4. 開発者：修正を反映して再プッシュする
5. レビュアー：承認してマージする

期待される最終的な `git log`：
```
$ git log --oneline --graph
*   abc1234 (HEAD -> main) Merge pull request #1 from user/feature/search
|\
| * def5678 feat: レビュー指摘を反映
| * ghi9012 feat: 検索機能を追加
|/
* jkl3456 Initial commit
```

<details>
<summary>ヒント</summary>

GitHub Flow はシンプルなブランチ戦略です。`main` ブランチは常にデプロイ可能な状態を保ち、すべての変更は `feature` ブランチからPRを通じてマージします。`gh pr review` でレビューコメントを追加できます。

</details>

<details>
<summary>解答</summary>

```bash
# === 開発者の作業 ===

# 1. feature ブランチを作成
$ git switch -c feature/search

# 2. 検索機能を実装
$ cat << 'EOF' > search.js
// 検索機能の実装
function search(items, query) {
  return items.filter(item =>
    item.toLowerCase().includes(query.toLowerCase())
  );
}
module.exports = { search };
EOF

$ git add search.js
$ git commit -m "feat: 検索機能を追加"

# 3. プッシュしてPRを作成
$ git push -u origin feature/search
$ gh pr create \
  --title "feat: 検索機能を追加" \
  --body "## 変更内容
- キーワードによる検索機能を実装しました

## テスト方法
- search.js を読み込み、search() 関数を呼び出して確認"

# === レビュアーの作業 ===

# 4. PRにレビューコメントを追加
$ gh pr review 1 --comment --body "検索結果が0件の場合の処理を追加してください。"

# === 開発者の修正作業 ===

# 5. 指摘を反映
$ cat << 'EOF' > search.js
// 検索機能の実装
function search(items, query) {
  if (!query || query.trim() === '') {
    return [];  // 空のクエリは空配列を返す
  }
  const results = items.filter(item =>
    item.toLowerCase().includes(query.toLowerCase())
  );
  return results;
}
module.exports = { search };
EOF

$ git add search.js
$ git commit -m "feat: レビュー指摘を反映"
$ git push

# === レビュアーの承認とマージ ===

# 6. PRを承認
$ gh pr review 1 --approve --body "LGTM! 修正を確認しました。"

# 7. PRをマージ
$ gh pr merge 1 --merge --delete-branch

# 8. ローカルを最新に更新
$ git switch main
$ git pull
$ git log --oneline --graph
```

解説：
- GitHub Flow は `main` + `feature` ブランチのみのシンプルな戦略です
- すべての変更はPRを通じてレビューを受けてからマージするのが原則です
- `--delete-branch` でマージ後にリモートブランチを自動削除できます
- LGTM は "Looks Good To Me"（問題なさそうです）の略で、レビュー承認時によく使われます
- GitHub Flow は継続的デプロイを行うWebサービスに特に適しています

</details>

### 問題4: コードレビューの実践

以下のコードに対するレビューコメントを作成してください。問題点を指摘し、改善案を提案してください。

レビュー対象のコード（`api.js`）：
```javascript
const password = "admin123";

function getUsers() {
  var users = [];
  fetch("http://api.example.com/users")
    .then(res => res.json())
    .then(data => {
      users = data;
    });
  return users;
}

function deleteUser(id) {
  fetch("http://api.example.com/users/" + id, {
    method: "DELETE"
  });
}
```

期待されるレビューコメント（3つ以上の指摘）を考えてください。

<details>
<summary>ヒント</summary>

セキュリティ、コーディングスタイル（モダンなJavaScript）、非同期処理の扱い、エラーハンドリングの観点からレビューしてみましょう。

</details>

<details>
<summary>解答</summary>

以下のようなレビューコメントが適切です：

```markdown
## レビューコメント

### 1. [Critical] パスワードのハードコーディング（1行目）
パスワードがソースコードに直接記述されています。これはセキュリティ上の重大な問題です。

**改善案:**
環境変数で管理してください。
```javascript
// 修正前（NG）
const password = "admin123";

// 修正後（OK）
const password = process.env.ADMIN_PASSWORD;
```

### 2. [Major] 非同期処理の戻り値が正しく返せていない（3-10行目）
fetch は非同期処理のため、`return users` は常に空配列を返します。

**改善案:**
async/await を使って非同期処理を適切に扱ってください。
```javascript
// 修正後
async function getUsers() {
  const res = await fetch("https://api.example.com/users");
  const users = await res.json();
  return users;
}
```

### 3. [Major] エラーハンドリングがない（全体）
fetch の失敗時やネットワークエラー時の処理がありません。

**改善案:**
try-catch でエラーを捕捉してください。
```javascript
async function getUsers() {
  try {
    const res = await fetch("https://api.example.com/users");
    if (!res.ok) throw new Error(`HTTP error: ${res.status}`);
    return await res.json();
  } catch (error) {
    console.error("ユーザー取得に失敗:", error);
    return [];
  }
}
```

### 4. [Minor] var の使用（4行目）
`var` は古い書き方です。`const` または `let` を使ってください。

### 5. [Minor] HTTP の使用（6行目）
`http://` ではなく `https://` を使ってください。
```

```bash
# GitHub CLI でレビューコメントを送信する例
$ gh pr review 1 --request-changes --body "上記5点の修正をお願いします。
特にパスワードのハードコーディングはセキュリティリスクが高いため、
早急に対応してください。"
```

解説：
- レビューコメントは重要度（Critical / Major / Minor）を明示すると優先度がわかりやすくなります
- 指摘だけでなく、具体的な改善コードを添えると修正がスムーズです
- セキュリティに関する問題は最優先で対応する必要があります
- `--request-changes` は修正が必要な場合、`--approve` は承認する場合に使います

</details>

## チャレンジ問題

### 問題5: プロジェクトのGit運用ルールを策定する

あなたは5人チームのリーダーとして、新規Webアプリケーション開発プロジェクトのGit運用ルールを策定することになりました。以下の項目を含む `CONTRIBUTING.md` を作成してください。

含めるべき項目：
1. ブランチ戦略（ブランチ命名規則を含む）
2. コミットメッセージ規約
3. プルリクエスト（PR）のルール
4. コードレビューのガイドライン
5. マージの方針

期待される `CONTRIBUTING.md` の構成：
```
# コントリビューションガイド
## ブランチ戦略
## コミット規約
## プルリクエスト
## コードレビュー
## マージ方針
```

<details>
<summary>ヒント</summary>

GitHub Flow をベースに、チーム規模に合ったシンプルなルールにしましょう。ルールが多すぎると守られません。Conventional Commits を採用するとコミット履歴が統一されます。PRテンプレートを用意すると記載漏れを防げます。

</details>

<details>
<summary>解答</summary>

```bash
$ cat << 'EOF' > CONTRIBUTING.md
# コントリビューションガイド

## ブランチ戦略

GitHub Flow を採用します。`main` ブランチは常にデプロイ可能な状態を保ちます。

### ブランチ命名規則

| 種類 | 命名パターン | 例 |
|---|---|---|
| 機能追加 | `feature/<issue番号>-<概要>` | `feature/42-user-login` |
| バグ修正 | `fix/<issue番号>-<概要>` | `fix/15-password-validation` |
| ドキュメント | `docs/<概要>` | `docs/setup-guide` |
| リファクタリング | `refactor/<概要>` | `refactor/auth-module` |

## コミット規約

Conventional Commits に従います。

```
<type>(<scope>): <description>

<body>（任意）

<footer>（任意）
```

### type 一覧
- `feat`: 新機能
- `fix`: バグ修正
- `docs`: ドキュメントのみの変更
- `refactor`: リファクタリング
- `test`: テストの追加・修正
- `chore`: ビルド・ツール設定の変更
- `ci`: CI/CD設定の変更

### ルール
- 日本語で記述する
- 1行目は50文字以内にする
- 本文が必要な場合は空行を挟んで記述する

## プルリクエスト

### PR作成時のルール
1. PRタイトルは Conventional Commits 形式にする
2. テンプレートに従って説明を記載する
3. 関連する Issue 番号を紐づける（`Closes #42`）
4. WIP（作業中）の場合はドラフトPRにする
5. セルフレビューを行ってからレビューを依頼する

### PRテンプレート
PRの説明には以下を含めてください：
- 変更内容の概要
- テスト方法
- スクリーンショット（UI変更がある場合）
- 関連Issue

## コードレビュー

### レビュアーのルール
- 24時間以内にレビューを開始する
- 指摘には重要度を付ける（Critical / Major / Minor）
- 改善案を具体的に示す
- 良い点も積極的にコメントする

### レビュイーのルール
- 指摘には返信して対応状況を共有する
- 意図的にそのままにする場合は理由を説明する
- 修正後は再レビューを依頼する

## マージ方針

- Squash Merge を基本とする（コミット履歴を1つにまとめる）
- 1人以上のApproveが必要
- CIが全て通っていること
- マージ後はリモートブランチを削除する
EOF

# コミット
$ git add CONTRIBUTING.md
$ git commit -m "docs: コントリビューションガイドを追加"
```

解説：
- 運用ルールはチームの規模や開発スタイルに合わせて調整してください
- ルールは少なくシンプルに保つのがポイントです。多すぎると守られなくなります
- `CONTRIBUTING.md` はリポジトリルートに置くのが慣例です。GitHubが自動的に認識します
- PRテンプレートは `.github/pull_request_template.md` に配置すると、PR作成時に自動反映されます
- Squash Merge はコミット履歴が整理されるため、多くのチームで採用されています
- これらのルールは定期的に見直し、チームの成長に合わせて更新しましょう

</details>

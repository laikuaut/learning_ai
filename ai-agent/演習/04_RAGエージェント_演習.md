### 問題1：RAGの基本フローを説明する（基本）

RAG（Retrieval-Augmented Generation）の処理フローについて、以下の図の空欄①〜⑥を埋めてください。

```
ユーザーの質問
    │
    ▼
┌──────────────┐
│  ① ________  │  ← 質問文をベクトルに変換
└──────────────┘
    │
    ▼
┌──────────────┐
│  ② ________  │  ← 類似度の高いドキュメントを検索
└──────────────┘
    │
    ▼
┌──────────────┐
│  ③ ________  │  ← 上位K件のドキュメントを取得
└──────────────┘
    │
    ▼
┌──────────────┐
│  ④ ________  │  ← 質問文 + 取得ドキュメントを結合
└──────────────┘
    │
    ▼
┌──────────────┐
│  ⑤ ________  │  ← LLMがコンテキスト付きで回答を生成
└──────────────┘
    │
    ▼
┌──────────────┐
│  ⑥ ________  │  ← 参照元の情報と共に回答を返す
└──────────────┘
```

**期待される出力例：**
```
① クエリのEmbedding（ベクトル化）
② ベクトル検索（類似度検索）
③ 関連ドキュメントの取得（Top-K Retrieval）
④ プロンプトの構築（コンテキスト注入）
⑤ LLMによる回答生成（Generation）
⑥ 回答 + ソース情報の返却
```

<details>
<summary>ヒント</summary>

- RAGは「Retrieval（検索）」+「Augmented（拡張）」+「Generation（生成）」の3段階です
- 質問文をそのままテキスト検索するのではなく、ベクトル化して意味的に検索します
- 検索結果をプロンプトに含めることで、LLMに外部知識を与えます

</details>

<details>
<summary>解答例</summary>

```
① クエリのEmbedding（ベクトル化）
  → ユーザーの質問文をEmbeddingモデル（例：text-embedding-3-small）で
    数値ベクトル（例：1536次元）に変換します

② ベクトル検索（類似度検索）
  → 変換したクエリベクトルをベクトルDBに問い合わせ、
    コサイン類似度などで類似ドキュメントを探します

③ 関連ドキュメントの取得（Top-K Retrieval）
  → 類似度スコアが高い上位K件（通常3〜10件）の
    ドキュメントチャンクを取得します

④ プロンプトの構築（コンテキスト注入）
  → 「以下のドキュメントを参考に質問に答えてください」のように
    取得したドキュメントを含むプロンプトを組み立てます

⑤ LLMによる回答生成（Generation）
  → コンテキスト付きプロンプトをLLMに送り、
    ドキュメントの内容に基づいた回答を生成させます

⑥ 回答 + ソース情報の返却
  → 生成された回答と、参照したドキュメントの出典情報を
    ユーザーに返します（トレーサビリティの確保）
```

```
RAGの最大の利点：
- LLMの学習データに含まれない最新情報や社内情報を活用できる
- 回答の根拠（ソース）を明示できるため、信頼性が向上する
- ファインチューニング不要で知識を追加・更新できる
```

</details>

### 問題2：チャンク分割戦略の選択（基本）

以下の3つのドキュメントそれぞれに対して、最適なチャンク分割戦略を選択し、理由を説明してください。

**ドキュメント種別：**
1. 社内FAQ集（Q&A形式、200件、各回答は100〜500文字）
2. 技術書のPDF（全300ページ、章・節・項の階層構造あり）
3. チャットログ（Slack履歴、10万メッセージ、タイムスタンプ付き）

**チャンク分割戦略の選択肢：**
- A: 固定長分割（例：500文字ごと）
- B: セマンティック分割（意味の区切りで分割）
- C: 構造ベース分割（見出し・段落などの構造で分割）
- D: ドキュメント単位（1ドキュメント＝1チャンク）
- E: 会話スレッド単位分割

**期待される出力例：**
```
1. 社内FAQ集 → 戦略X
   理由：...

2. 技術書のPDF → 戦略Y
   理由：...

3. チャットログ → 戦略Z
   理由：...
```

<details>
<summary>ヒント</summary>

- チャンク分割の目標は「1つのチャンクに1つの意味的まとまり」を持たせることです
- FAQ は1つのQ&Aペアが自然な意味の単位です
- 技術書は章・節ごとにトピックが変わります
- チャットは時系列とスレッドの2軸で構造化されています

</details>

<details>
<summary>解答例</summary>

```
1. 社内FAQ集 → 戦略D（ドキュメント単位）
   理由：
   - 各Q&Aペアが独立した意味を持つため、1つのQ&A = 1チャンクが最適
   - 各回答が100〜500文字と短いので、チャンクサイズとしても適切
   - Q&Aを途中で分割すると、質問と回答が別チャンクになりかねない
   - メタデータとしてカテゴリやタグを付与すると検索精度が向上

   実装例：
   - チャンクサイズ: Q&Aペアごと（最大約600文字）
   - オーバーラップ: 不要（各Q&Aは独立しているため）
   - メタデータ: カテゴリ、最終更新日、関連キーワード

2. 技術書のPDF → 戦略C（構造ベース分割）
   理由：
   - 章・節・項の階層構造がすでに意味の区切りになっている
   - 固定長分割では文の途中や表の途中で切れるリスクがある
   - 見出しをメタデータに保存することで階層的な検索が可能
   - 長い節はさらにサブ分割する（段落単位で再分割）

   実装例：
   - 第1段階: 節（section）レベルで分割
   - 第2段階: 1000文字を超える節はさらに段落単位で分割
   - オーバーラップ: 前のチャンクの末尾100文字を次のチャンクの先頭に含める
   - メタデータ: 章名、節名、ページ番号

3. チャットログ → 戦略E（会話スレッド単位分割）
   理由：
   - Slackにはスレッド（thread）の概念があり、1スレッドが1トピック
   - 時系列の固定長分割では、異なる話題が混在してしまう
   - スレッドがない場合は、時間間隔（例：30分以上の空白）で分割
   - タイムスタンプと発言者をメタデータとして保存

   実装例：
   - スレッド付き: スレッド単位で1チャンク
   - スレッドなし: 30分以上の空白を区切りとして分割
   - 長いスレッド（50メッセージ超）は時間区間でさらに分割
   - メタデータ: チャンネル名、参加者、日時範囲
```

```
チャンク分割の一般原則：
1. 意味の一貫性: 1チャンクに1トピックが理想
2. 適切なサイズ: 100〜1000トークン程度が一般的
3. オーバーラップ: 文脈の断絶を防ぐため10〜20%程度を重複させる
4. メタデータ: ソース情報、日時、カテゴリ等を付与してフィルタリングに活用
```

</details>

### 問題3：ベクトルDBの比較（基本）

以下のベクトルデータベースについて、特徴を比較表にまとめてください。

**比較対象：**
- Chroma
- Pinecone
- Weaviate
- pgvector（PostgreSQL拡張）
- Qdrant

**比較項目：**
- デプロイ方式（ローカル/クラウド/両方）
- 料金体系（無料/有料/フリーミアム）
- 主な特徴
- 推奨ユースケース

**期待される出力例：**
```
| DB名 | デプロイ | 料金 | 特徴 | 推奨ユースケース |
|---|---|---|---|---|
| Chroma | ... | ... | ... | ... |
| ... | ... | ... | ... | ... |
```

<details>
<summary>ヒント</summary>

- Chromaはローカルでの手軽さが特徴です
- Pineconeはフルマネージドクラウドサービスです
- pgvectorは既存のPostgreSQLに追加できます
- 「何を優先するか」で最適な選択肢が変わります

</details>

<details>
<summary>解答例</summary>

```
| DB名 | デプロイ | 料金 | 特徴 | 推奨ユースケース |
|---|---|---|---|---|
| Chroma | ローカル/クラウド | OSS（無料） | セットアップが簡単、Pythonネイティブ、軽量 | プロトタイピング、小〜中規模プロジェクト |
| Pinecone | クラウド（マネージド） | フリーミアム | フルマネージド、高スケーラビリティ、運用不要 | 本番環境、大規模データ、運用負荷を下げたい場合 |
| Weaviate | ローカル/クラウド | OSS + マネージド版あり | GraphQL対応、モジュール式、ハイブリッド検索 | 複雑な検索要件、マルチモーダル検索 |
| pgvector | ローカル/クラウド | OSS（無料） | 既存PostgreSQLに追加可能、SQL操作、ACID準拠 | 既にPostgreSQLを使用中の場合、トランザクション要件あり |
| Qdrant | ローカル/クラウド | OSS + クラウド版あり | Rust製で高速、フィルタリング性能が高い、REST/gRPC対応 | 高パフォーマンス要件、フィルタ付き検索 |
```

```
選定のガイドライン：

学習・プロトタイプ → Chroma
  理由: pip install chromadb だけで始められる手軽さ

既存PostgreSQL環境 → pgvector
  理由: 新たなインフラ追加不要、SQLで操作可能

本番環境（運用重視）→ Pinecone
  理由: インフラ管理不要、SLAあり、スケーラビリティ

高度な検索要件 → Weaviate / Qdrant
  理由: ハイブリッド検索、高速フィルタリングなどの機能が充実

注意点：
- ベクトルDBの選択はプロジェクトの規模・要件・既存インフラで決まります
- 小規模なら、最初はChromaで始めて後からマイグレーションも可能です
- 本番環境では、バックアップ・監視・スケーリングの容易さも考慮しましょう
```

</details>

### 問題4：Embeddingの基本理解（基本）

以下の質問に答えてください。

**問4-1：** 以下の文のうち、Embeddingの説明として正しいものをすべて選んでください。

```
A. テキストを固定長の数値ベクトルに変換する技術である
B. 意味が似たテキストは、ベクトル空間上で近い位置にマッピングされる
C. Embeddingモデルは、テキストの要約を生成する
D. Embeddingベクトルの次元数は、入力テキストの長さに比例する
E. コサイン類似度を使って2つのベクトルの類似性を計算できる
```

**問4-2：** 以下の3つの文について、どの2つが最もEmbedding上で近いか予想してください。

```
文1: 「機械学習はデータからパターンを学ぶ技術です」
文2: 「今日の東京の天気は晴れです」
文3: 「ディープラーニングは大量のデータを使ってモデルを訓練します」
```

**期待される出力例：**
```
問4-1: A, B, E が正しい
問4-2: 文1と文3が最も近い（どちらもAI/ML技術に関する説明）
```

<details>
<summary>ヒント</summary>

- Embeddingは「意味の数値化」です
- ベクトルの次元数はモデルによって固定されています（入力長とは無関係）
- コサイン類似度は-1〜1の値を取り、1に近いほど類似しています

</details>

<details>
<summary>解答例</summary>

```
問4-1: A, B, E が正しい

A. ○ テキストを固定長の数値ベクトルに変換する技術です
   → text-embedding-3-smallなら1536次元のベクトルに変換されます

B. ○ 意味が似たテキストはベクトル空間上で近い位置にマッピングされます
   → これがベクトル検索の基盤となる性質です

C. × Embeddingモデルは要約を生成しません
   → テキストの意味を「数値ベクトル」として表現するのが役割です
   → 要約生成はLLM（生成モデル）の役割です

D. × ベクトルの次元数はモデルによって固定されています
   → 入力テキストが長くても短くても、同じ次元数のベクトルになります
   → 例: text-embedding-3-small → 常に1536次元

E. ○ コサイン類似度で2つのベクトルの類似性を計算できます
   → cos(θ) = (A·B) / (|A|×|B|)
   → 1に近いほど類似、0は無関係、-1は正反対

---

問4-2: 文1と文3が最も近い

文1: 「機械学習はデータからパターンを学ぶ技術です」
文3: 「ディープラーニングは大量のデータを使ってモデルを訓練します」

理由:
- 両方ともAI/機械学習の技術に関する説明文
- 「データ」「学ぶ/訓練」という共通の概念を含む
- 文2は天気の話題で、意味的に全く異なる分野

実際の類似度の目安（コサイン類似度）：
- 文1 ↔ 文3: 約0.85〜0.90（非常に高い類似度）
- 文1 ↔ 文2: 約0.15〜0.25（低い類似度）
- 文2 ↔ 文3: 約0.10〜0.20（低い類似度）
```

</details>

### 問題5：社内文書検索RAGシステムの設計（応用）

あなたは「社内ナレッジベース検索システム」のRAGパイプラインを設計するタスクを任されました。以下の要件に基づいて、システム設計を記述してください。

**要件：**
- 対象文書：社内Wiki（5,000ページ）、社内FAQ（2,000件）、議事録（10,000件）
- ユーザー数：500人
- 応答速度：3秒以内
- 回答には必ずソースのリンクを含める

**設計すべき項目：**
1. インデキシングパイプライン（文書の前処理〜ベクトルDB格納）
2. 検索パイプライン（クエリ処理〜結果返却）
3. 使用する技術スタック
4. メタデータ設計

**期待される出力例：**
```
## 1. インデキシングパイプライン
[文書取得] → [前処理] → [チャンク分割] → [Embedding] → [ベクトルDB格納]

## 2. 検索パイプライン
[クエリ] → [クエリ拡張] → [ハイブリッド検索] → [リランキング] → [回答生成]

## 3. 技術スタック
- Embeddingモデル: ...
- ベクトルDB: ...
- LLM: ...

## 4. メタデータ設計
...
```

<details>
<summary>ヒント</summary>

- 3種類の文書（Wiki/FAQ/議事録）はそれぞれチャンク分割戦略が異なります
- ハイブリッド検索（ベクトル検索 + キーワード検索）で精度を向上できます
- リランキングで検索結果の順序を最適化できます
- メタデータによるフィルタリングで検索範囲を絞れます

<details>
<summary>解答例</summary>

```
## 1. インデキシングパイプライン

### 全体フロー
[文書取得] → [前処理] → [文書種別判定] → [チャンク分割] → [Embedding生成] → [ベクトルDB格納]

### 詳細設計

#### 文書取得
- Wiki: APIで全ページを取得（差分更新対応）
- FAQ: DBから全レコードをエクスポート
- 議事録: ファイルサーバーからPDF/Wordを取得

#### 前処理
- HTMLタグ除去（Wiki）
- ヘッダー/フッター除去（議事録PDF）
- テーブルのMarkdown変換
- 画像内テキストのOCR処理（必要に応じて）

#### チャンク分割（文書種別ごと）
- Wiki: 見出し（h2/h3）レベルで構造ベース分割
  - 最大チャンクサイズ: 800トークン
  - オーバーラップ: 100トークン
- FAQ: 1つのQ&Aペア = 1チャンク（分割不要）
- 議事録: 議題（アジェンダ項目）ごとに分割
  - 長い議題は段落単位でさらに分割
  - 最大チャンクサイズ: 600トークン

#### Embedding生成
- モデル: text-embedding-3-small（1536次元）
- バッチ処理: 100チャンクずつ一括処理
- 初回インデキシング: 約17,000チャンク（推定）

#### ベクトルDB格納
- DB: Qdrant（高速フィルタリング対応）
- インデックス: HNSW（Hierarchical Navigable Small World）
- 初回ロード後は差分更新（変更されたページのみ再インデキシング）

---

## 2. 検索パイプライン

### 全体フロー
[クエリ受信] → [クエリ前処理] → [ハイブリッド検索] → [リランキング] → [コンテキスト構築] → [LLM回答生成] → [ソース付き回答返却]

### 詳細設計

#### クエリ前処理
- 質問の意図を分類（事実質問/手順質問/意見質問）
- 必要に応じてクエリ拡張（同義語展開）

#### ハイブリッド検索
- ベクトル検索: クエリをEmbedding化してコサイン類似度検索（Top-20）
- キーワード検索: BM25によるキーワードマッチング（Top-20）
- スコア統合: RRF（Reciprocal Rank Fusion）で2つの結果を統合
- メタデータフィルタ: 文書種別や部署でフィルタリング可能

#### リランキング
- Cross-Encoderモデルで上位20件を再スコアリング
- 最終的にTop-5を選択

#### コンテキスト構築
- Top-5チャンクを関連度順にプロンプトに配置
- システムプロンプトで「ソースに基づいて回答」を指示
- 各チャンクにソース情報（ドキュメント名、URL）を付与

#### LLM回答生成
- モデル: Claude（claude-sonnet）
- プロンプト構造:
  「以下のドキュメントを参考に質問に答えてください。
   回答には必ず参照元の[ソース名]を記載してください。
   ドキュメントに情報がない場合は「該当する情報が見つかりませんでした」と回答してください。」

## 3. 技術スタック

| コンポーネント | 技術 | 理由 |
|---|---|---|
| Embeddingモデル | OpenAI text-embedding-3-small | コスト効率が良い、1536次元で十分な精度 |
| ベクトルDB | Qdrant | フィルタリング性能が高い、OSS、セルフホスト可能 |
| キーワード検索 | Elasticsearch | BM25検索に最適、既存インフラに統合しやすい |
| リランカー | Cohere Rerank / cross-encoder | 検索精度の大幅な向上が期待できる |
| LLM | Claude Sonnet | 長いコンテキスト対応、日本語性能が高い |
| バックエンド | Python (FastAPI) | 非同期処理対応、エコシステムが充実 |
| キャッシュ | Redis | よく聞かれる質問の回答をキャッシュして高速化 |

## 4. メタデータ設計

各チャンクに付与するメタデータ：

| フィールド | 型 | 説明 | 例 |
|---|---|---|---|
| doc_id | string | ドキュメントの一意ID | "wiki-12345" |
| doc_type | string | 文書種別 | "wiki" / "faq" / "minutes" |
```

### 問題6：チャンク分割パラメータの最適化（応用）

以下のシナリオで、チャンク分割のパラメータ（チャンクサイズ、オーバーラップ）が検索品質にどう影響するか分析してください。

**シナリオ：** 社内の技術ドキュメント（平均1ページ2,000文字）をRAGシステムに組み込む

以下の3パターンそれぞれについて、メリット・デメリットと適するユースケースを記述してください。

**パターン：**

```
パターンA: チャンクサイズ=200文字、オーバーラップ=0
パターンB: チャンクサイズ=500文字、オーバーラップ=50文字
パターンC: チャンクサイズ=1500文字、オーバーラップ=200文字
```

**期待される出力例：**

```
パターンA（小チャンク・オーバーラップなし）
- メリット: ...
- デメリット: ...
- 適するケース: ...

パターンB（中チャンク・小オーバーラップ）
- メリット: ...
- デメリット: ...
- 適するケース: ...

パターンC（大チャンク・大オーバーラップ）
- メリット: ...
- デメリット: ...
- 適するケース: ...
```

<details>
<summary>ヒント</summary>

- チャンクが小さすぎると文脈が失われ、大きすぎるとノイズが増えます
- オーバーラップはチャンク境界での情報欠落を防ぎます
- LLMのコンテキストウィンドウサイズも考慮が必要です
- チャンク数が増えるとベクトルDBのコストも増加します

</details>

<details>
<summary>解答例</summary>

```
パターンA: チャンクサイズ=200文字、オーバーラップ=0

- メリット:
  - 検索の粒度が細かく、ピンポイントで関連箇所を見つけやすい
  - 不要な情報が混入しにくい（ノイズが少ない）
  - Top-Kで多様なソースからチャンクを取得できる

- デメリット:
  - 文脈が断片化し、チャンク単体では意味が通じないことがある
  - オーバーラップ=0のため、チャンク境界で情報が欠落するリスク
  - チャンク数が多くなり、ストレージ・検索コストが増加
  - 1ページ2,000文字 → 10チャンク/ページ（全体で大量になる）
  - Embedding回数が増え、インデキシングコストが高い

- 適するケース:
  - 定義集や用語辞書など、短い独立した記述が多い文書
  - 「ピンポイントで特定の事実を探す」用途

  - 文脈を保ちつつ適度な粒度で検索できる（バランスが良い）
  - オーバーラップにより境界での情報欠落が軽減される

  - ストレージコストと検索精度のバランスが取れている

- デメリット:
  - 非常に長い説明（手順全体など）は複数チャンクに分割される
  - オーバーラップ部分が重複データとなり、わずかにストレージ増加

- 適するケース:
  - 一般的な技術ドキュメント（最も汎用的な設定）
  - 段落単位で意味がまとまる文書
  - 最初のベースラインとして推奨

---

パターンC: チャンクサイズ=1500文字、オーバーラップ=200文字

- メリット:
  - 広い文脈を保持でき、複雑な説明も1チャンクに収まりやすい
  - 手順書の一連のステップなどが分断されにくい
  - チャンク数が少なく、ストレージ・検索コストが低い
  - LLMに渡す際、少ないチャンク数で十分な情報量を確保

- デメリット:
  - 1つのチャンクに複数トピックが混在するリスク
  - 検索精度が下がる可能性（無関係な内容も一緒にヒットする）
  - Top-Kで取得するとコンテキストウィンドウを大量に消費
  - オーバーラップ200文字の重複がストレージを圧迫
  - 1ページ2,000文字の文書では、ほぼ全文が1〜2チャンクになる

- 適するケース:
  - 長い手順書や一連の説明をまとめて取得したい場合
  - 「概要を広く知りたい」ような質問が多い場合
  - LLMのコンテキストウィンドウが十分大きい場合

---

最適化のアプローチ：
1. まずパターンBをベースラインとして実装
2. テストクエリセット（50〜100問）を用意
3. 各パターンで検索精度を測定（Recall@K, MRR等）
4. 文書種別ごとに最適なパラメータを適用する
5. 定期的にフィードバックを収集して調整する
```

### 問題7：検索精度の評価指標（応用）

RAGシステムの検索精度を評価するための以下の指標について、それぞれ計算例を示してください。

**評価指標：**
1. Recall@K（再現率）
2. Precision@K（適合率）
3. MRR（Mean Reciprocal Rank）

**テストデータ：**
```
質問: 「有給休暇の申請方法は？」
正解ドキュメント: [doc_A, doc_C]（人間が正解と判断）

検索結果（Top-5、類似度スコア順）:
  1位: doc_B（無関係）
  2位: doc_A（正解）
  3位: doc_D（無関係）
  4位: doc_C（正解）
  5位: doc_E（無関係）
```

**期待される出力例：**
```
Recall@5 = ...
Precision@5 = ...
MRR = ...
```

<details>
<summary>ヒント</summary>

- Recall@K = K件中に含まれる正解の数 / 正解の総数
- Precision@K = K件中に含まれる正解の数 / K
- MRR = 最初の正解が出現した順位の逆数

</details>

<details>
<summary>解答例</summary>

```
■ Recall@5（Top-5での再現率）

計算式: 検索結果Top-5に含まれる正解数 / 正解の総数

正解ドキュメント: [doc_A, doc_C] → 正解は2件
Top-5に含まれる正解: doc_A（2位）, doc_C（4位） → 2件

Recall@5 = 2 / 2 = 1.0（100%）

→ 正解をすべて取得できているので、再現率は完璧です。

参考: Recall@3の場合
Top-3に含まれる正解: doc_A（2位）のみ → 1件
Recall@3 = 1 / 2 = 0.5（50%）

---

■ Precision@5（Top-5での適合率）

計算式: 検索結果Top-5に含まれる正解数 / K

Top-5に含まれる正解: 2件
K = 5

Precision@5 = 2 / 5 = 0.4（40%）

→ Top-5のうち60%が無関係な文書です。精度向上の余地があります。

---

■ MRR（Mean Reciprocal Rank）

計算式: 最初の正解が出現した順位の逆数

最初の正解: doc_A（2位）
Reciprocal Rank = 1 / 2 = 0.5

MRR = 0.5

※ MRRは複数クエリの平均を取ります。
  例えば3つのクエリでRRが0.5, 1.0, 0.33の場合:
  MRR = (0.5 + 1.0 + 0.33) / 3 = 0.61

---

■ 指標の解釈と改善方針

| 指標 | 値 | 判定 | 改善方針 |
|---|---|---|---|
| Recall@5 | 1.0 | 良好 | 正解が全件取得できている |
| Precision@5 | 0.4 | 要改善 | リランキングで上位の精度を向上させる |
| MRR | 0.5 | 要改善 | 正解が1位に来るようクエリ拡張やリランキングを導入 |

改善策の例:
- リランカー（Cross-Encoder）を導入してdoc_Aを1位にする
- メタデータフィルタで無関係文書（doc_B, doc_D, doc_E）を除外
- ハイブリッド検索でキーワード「有給休暇」「申請」を重視する
```

</details>

HyDE は検索精度を向上させる手法の一つです。以下の問いに答えてください。

**問8-1：** HyDEの仕組みを図示して説明してください。

**問8-2：** 以下のクエリに対して、HyDEが通常のベクトル検索よりも有効なケースと、逆効果になるケースを説明してください。

```
A. 「Kubernetes Podのリスタートループを解消するには？」
```

**期待される出力例：**
```

HyDE:      [クエリ] → [LLMで仮回答生成] → [仮回答をEmbedding] → [検索]

問8-2:
クエリA: HyDEが有効 / 逆効果 → 理由...
```

<details>
<summary>ヒント</summary>

- HyDEはクエリをそのままベクトル化するのではなく、LLMに仮の回答を生成させ、その回答をベクトル化して検索します
- 質問文と回答文ではベクトル表現が異なるため、回答同士の方が類似度が高くなります
- LLMの知識にない事実データの検索ではHyDEが不利になることがあります

</details>

<details>
<summary>解答例</summary>

```
■ 問8-1: HyDEの仕組み

【通常のRAG検索】
ユーザーの質問
    │
    ▼
"Podのリスタートループを解消するには？"
    │
    ▼ Embeddingモデル
[質問のベクトル] ─── 検索 ──→ ベクトルDB → 結果

問題点: 質問文のベクトルと、回答を含むドキュメントのベクトルは
       異なる空間上の点になりがち（質問と回答は文体が異なる）

---

【HyDE検索】
ユーザーの質問
    │
    ▼
"Podのリスタートループを解消するには？"
    │
    ▼ LLMで仮の回答を生成
"Podのリスタートループは、主に以下の原因で発生します。
 1. CrashLoopBackOff: アプリケーションの起動失敗...
 2. OOMKilled: メモリ不足..."
    │
    ▼ 仮回答をEmbeddingモデルでベクトル化
[仮回答のベクトル] ─── 検索 ──→ ベクトルDB → 結果

利点: 仮回答のベクトルは、DBに格納されたドキュメント（回答文体）と
     同じ空間上にマッピングされるため、検索精度が向上する

---

■ 問8-2: クエリ別の有効性分析

クエリA: 「Kubernetes Podのリスタートループを解消するには？」
→ HyDEが有効

理由:
- LLMはKubernetesの一般的なトラブルシューティング知識を持っている
- 仮回答には「CrashLoopBackOff」「OOMKilled」「ImagePullBackOff」
  などの技術用語が含まれ、実際のドキュメントとの一致度が高まる
- 質問形式の文よりも、解説形式の仮回答の方がドキュメントに近い

---

クエリB: 「2026年3月の売上実績」
→ HyDEが逆効果

理由:
- LLMは2026年3月の実際の売上データを知らない
- 仮回答として「2026年3月の売上は〇〇億円で...」とハルシネーションを生成
- 架空の数字を含む仮回答のベクトルが、実データのドキュメントと
  かけ離れてしまい、検索精度がむしろ悪化する
- このような事実データの検索には、キーワード検索やメタデータフィルタが有効

---

クエリC: 「Docker ComposeとKubernetesの違い」
→ HyDEがやや有効

理由:
- LLMはDocker ComposeとKubernetesの一般的な違いを知っている
- 仮回答に「オーケストレーション」「スケーリング」「マニフェスト」
  等の比較に使われる用語が含まれる
- ただし、社内固有の使い分けルールなどはLLMが知らないため、
  仮回答が一般論に偏る可能性がある

---

■ HyDE適用のガイドライン

HyDEが有効なケース:
- 技術的な質問（LLMが一般知識で仮回答を生成できる）
- 「どうやって」「なぜ」型の質問
- ドキュメントが回答文体で書かれている

HyDEが逆効果になるケース:
- 具体的な数値や日付を含むファクトの検索
- LLMの知識にない社内固有の情報
- 短いキーワード検索（「有給休暇 申請フォーム」など）

実装上の注意:
- HyDEはLLM呼び出しが追加されるため、レイテンシが増加する
- 仮回答の品質がそのまま検索品質に影響する
- 通常検索との結果をマージする「HyDE + 通常検索のハイブリッド」も有効
```

### 問題9：Self-RAGパイプラインの設計（チャレンジ）

Self-RAG は、LLM自身が「検索が必要かどうか」「検索結果が有用かどうか」「生成した回答が正確かどうか」を自己判断する手法です。以下の設計課題に取り組んでください。

**課題：** 以下の仕様を満たすSelf-RAGパイプラインのフローチャートと擬似コードを作成してください。

**仕様：**
1. LLMが質問を受け取り、検索が必要か判断する
2. 検索が必要な場合、検索を実行して結果を取得する
3. 検索結果の関連性をLLMが評価する（relevant/irrelevant）
4. 関連性が低い場合、クエリを書き換えて再検索する（最大3回）
5. 回答を生成した後、生成した回答がソースに基づいているか自己検証する
6. 検証に失敗した場合、回答を再生成する

**期待される出力例：**
```
## フローチャート
質問受信 → 検索要否判定 → [必要] → 検索実行 → 関連性評価 → ...
                          → [不要] → 直接回答

## 擬似コード
function self_rag(query):
    if needs_retrieval(query):
        ...
```

<details>
<summary>ヒント</summary>

- Self-RAGは4つの特殊トークン（Retrieve, IsRel, IsSup, IsUse）で制御します
- 通常のRAGとの違いは「自己反省（Self-Reflection）」のステップがある点です
- 再検索時にはクエリリライトが重要です
- 無限ループを防ぐためにリトライ上限を設けましょう

</details>

<details>
<summary>解答例</summary>

```
## フローチャート

質問受信
    │
    ▼
┌───────────────────┐
│ 検索要否判定       │  ← LLMが「この質問に外部知識が必要か」を判断
│ (Retrieve判定)     │
└───────────────────┘
    │           │
 [必要]       [不要]
    │           │
    ▼           ▼
┌──────────┐  ┌───────────┐
│ 検索実行  │  │ 直接回答   │ ← LLMの内部知識で回答可能
└──────────┘  │ 生成       │
    │          └───────────┘
    ▼               │
┌──────────────┐    │
│ 関連性評価    │    │
│ (IsRel判定)   │    │
└──────────────┘    │
    │        │       │
 [関連]   [無関係]   │
    │        │       │
    │        ▼       │
    │   ┌──────────┐ │
    │   │クエリ     │ │
    │   │リライト   │ │
    │   │(最大3回)  │ │
    │   └──────────┘ │
    │        │       │
    │    [上限到達]   │
    │        │       │
    ▼        ▼       │
┌──────────────┐    │
│ 回答生成      │    │
│              │    │
└──────────────┘    │
    │               │
    ▼               │
┌──────────────┐    │
│ 根拠検証      │    │
│ (IsSup判定)   │    │
└──────────────┘    │
    │        │       │
 [支持]   [不支持]   │
    │        │       │
    │        ▼       │
    │   ┌──────────┐ │
    │   │回答再生成 │ │
    │   │(最大2回)  │ │
    │   └──────────┘ │
    │        │       │
    ▼        ▼       ▼
┌──────────────────────┐
│ 有用性評価＆最終回答  │
│ (IsUse判定)           │
└──────────────────────┘


## 擬似コード

function self_rag(query, max_retrieval_retries=3, max_generation_retries=2):
    """Self-RAGパイプラインのメイン処理"""

    # ステップ1: 検索要否判定 (Retrieve Token)
    needs_search = judge_retrieval_need(query)

    if not needs_search:
        # 外部知識不要 → LLMの知識で直接回答
        answer = generate_answer(query, context=None)
        return {
            "answer": answer,
            "sources": [],
            "retrieval_used": False
        }

    # ステップ2: 検索 + 関連性評価ループ
    current_query = query
    relevant_docs = []

    for attempt in range(max_retrieval_retries):
        # ベクトル検索を実行
        search_results = vector_search(current_query, top_k=5)

        # 各検索結果の関連性を評価 (IsRel Token)
        for doc in search_results:
            relevance = judge_relevance(query, doc)
            if relevance == "relevant":
                relevant_docs.append(doc)

        # 十分な関連ドキュメントが見つかった
        if len(relevant_docs) >= 2:
            break

        # 関連ドキュメントが不足 → クエリリライト
        if attempt < max_retrieval_retries - 1:
            current_query = rewrite_query(query, current_query, search_results)
            print(f"クエリリライト ({attempt+1}回目): {current_query}")

    # ステップ3: 回答生成 + 根拠検証ループ
    context = format_context(relevant_docs)

    for attempt in range(max_generation_retries):
        # コンテキスト付きで回答を生成
        answer = generate_answer(query, context)

        # 生成した回答がソースに基づいているか検証 (IsSup Token)
        is_supported = verify_groundedness(answer, relevant_docs)

        if is_supported:
            break

        # 根拠不十分 → 再生成（より忠実に回答するよう指示を強化）
        if attempt < max_generation_retries - 1:
            print(f"根拠検証失敗 → 回答再生成 ({attempt+1}回目)")
            context = format_context(relevant_docs, strict_mode=True)

    # ステップ4: 有用性評価 (IsUse Token)
    usefulness = judge_usefulness(query, answer)

    return {
        "answer": answer,
        "sources": [doc.metadata for doc in relevant_docs],
        "retrieval_used": True,
        "usefulness_score": usefulness,
        "retrieval_attempts": attempt + 1
    }


# --- 補助関数 ---

function judge_retrieval_need(query):
    """LLMに検索が必要かどうかを判断させる"""
    prompt = f"""
    以下の質問に回答するために、外部ドキュメントの検索が必要ですか？
    あなた自身の知識だけで正確に回答できる場合は "no"、
    外部情報が必要な場合は "yes" と回答してください。

    質問: {query}
    回答 (yes/no):
    """
    response = llm.generate(prompt)
    return response.strip().lower() == "yes"


function judge_relevance(query, document):
    """検索結果が質問に関連しているかLLMが評価する"""
    prompt = f"""
    以下のドキュメントは、質問に対して関連性がありますか？

    質問: {query}
    ドキュメント: {document.content}

    "relevant" または "irrelevant" で回答してください:
    """
    response = llm.generate(prompt)
    return response.strip().lower()


function rewrite_query(original_query, current_query, failed_results):
    """検索精度を上げるためにクエリを書き換える"""
    prompt = f"""
    以下の検索クエリでは関連する結果が見つかりませんでした。
    より良い検索結果を得るためにクエリを書き換えてください。

    元の質問: {original_query}
    試したクエリ: {current_query}
    得られた結果（無関係）: {summarize(failed_results)}

    改善したクエリ:
    """
    return llm.generate(prompt).strip()


function verify_groundedness(answer, source_docs):
    """生成した回答がソースドキュメントに基づいているか検証する"""
    prompt = f"""
    以下の回答が、提供されたソースドキュメントの内容に基づいているか
    検証してください。回答に含まれる主張がソースで裏付けられていれば
    "supported"、ソースにない情報が含まれていれば "not_supported" と
    回答してください。

    回答: {answer}
    ソース: {format_sources(source_docs)}

    判定 (supported/not_supported):
    """
    response = llm.generate(prompt)
    return response.strip().lower() == "supported"
```

```
Self-RAGの設計ポイント：

1. 4つの判定トークン:
   - Retrieve: 検索が必要かの判定
   - IsRel: 検索結果の関連性判定
   - IsSup: 回答がソースに支持されているかの判定
   - IsUse: 最終回答の有用性判定

2. 通常RAGとの違い:
   - 通常RAG: 常に検索する → 検索不要な質問でもレイテンシ増加
   - Self-RAG: 必要な時だけ検索 → 効率的

3. リトライ戦略:
   - 検索リトライ: クエリリライトで精度向上を試みる
   - 生成リトライ: strict_modeでソースへの忠実度を高める
   - 必ず上限を設けて無限ループを防止

4. トレードオフ:
   - メリット: ハルシネーション低減、検索精度向上
   - デメリット: LLM呼び出し回数増加（コスト・レイテンシ増）
   - 判定の精度がLLMの能力に依存する

5. 実装上の注意:
   - 各判定ステップのプロンプトは慎重に設計する
   - 判定結果のログを残し、精度を定期的にモニタリングする
   - 本番環境では判定のキャッシュも検討する
```

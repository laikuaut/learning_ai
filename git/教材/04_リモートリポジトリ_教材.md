# 第4章：リモートリポジトリとプッシュ・プル

## この章のゴール

- リモートリポジトリ（remote repository）の概念を理解する
- `git push` と `git pull` を使ってリモートと同期できるようになる
- `git fetch` と `git pull` の違いを理解する
- `git clone` でリモートリポジトリを取得できるようになる
- SSH と HTTPS の認証方式を理解する
- フォーク（fork）とクローン（clone）の違いを理解する

---

## 1. リモートリポジトリとは

リモートリポジトリ（remote repository）は、ネットワーク上にあるリポジトリのことです。GitHub、GitLab、Bitbucket などのサービスでホスティングされるのが一般的です。

```
┌──────────────────────────────────────────────────────────┐
│                    リモートとローカルの関係                  │
│                                                          │
│   ┌────────────────────────────────────┐                 │
│   │      リモートリポジトリ（GitHub等）    │                 │
│   │                                    │                 │
│   │  ┌──────┐  ┌──────┐  ┌──────┐    │                 │
│   │  │ main │  │feat/ │  │bugfix│    │                 │
│   │  │      │  │login │  │/auth │    │                 │
│   │  └──────┘  └──────┘  └──────┘    │                 │
│   └──────────────┬─────────────────────┘                 │
│                  │                                        │
│          git push│↑    ↓│git fetch / git pull             │
│                  │      │                                 │
│   ┌──────────────┴──────┴──────────────┐                 │
│   │      ローカルリポジトリ（自分のPC）    │                 │
│   │                                    │                 │
│   │  ┌──────┐  ┌──────┐              │                 │
│   │  │ main │  │feat/ │              │                 │
│   │  │      │  │login │              │                 │
│   │  └──────┘  └──────┘              │                 │
│   └────────────────────────────────────┘                 │
│                                                          │
│  開発者A                          開発者B                  │
│  ┌─────────┐                     ┌─────────┐             │
│  │ローカル  │──push/pull──リモート──push/pull──│ローカル  │  │
│  │リポジトリ│                     │リポジトリ│             │
│  └─────────┘                     └─────────┘             │
└──────────────────────────────────────────────────────────┘
```

### リモートリポジトリの役割

| 役割 | 説明 |
|------|------|
| 共有 | チームメンバーとコードを共有する |
| バックアップ | ローカルPCが壊れてもコードが残る |
| デプロイ | 本番環境へのデプロイの起点になる |
| コードレビュー | プルリクエストを通じてレビューできる |

### ポイントまとめ

- リモートリポジトリはネットワーク上にある共有リポジトリです
- GitHub、GitLab、Bitbucket が代表的なホスティングサービスです
- リモートとローカルを同期することでチーム開発が可能になります

## 2. git remote でリモートを管理する

### リモートリポジトリの登録

```bash
# リモートリポジトリを追加
git remote add origin https://github.com/username/my-project.git

# "origin" はリモートリポジトリの名前（慣例的に origin を使う）
```

`origin` はリモートリポジトリにつける名前で、慣例的に使われます。複数のリモートを登録することも可能です。

```bash
# 複数のリモートを登録する例
git remote add origin https://github.com/username/my-project.git
git remote add upstream https://github.com/original/my-project.git
```

### リモートの確認

```bash
# 登録されているリモートの一覧
git remote
# 出力:
# origin

# URL も含めて表示
git remote -v
# 出力:
# origin  https://github.com/username/my-project.git (fetch)
# origin  https://github.com/username/my-project.git (push)

# リモートの詳細情報
git remote show origin
# 出力:
# * remote origin
#   Fetch URL: https://github.com/username/my-project.git
#   Push  URL: https://github.com/username/my-project.git
#   HEAD branch: main
#   Remote branches:
#     main    tracked
#   Local branch configured for 'git pull':
#     main merges with remote main
```

### リモートの変更・削除

```bash
# リモートの URL を変更
git remote set-url origin https://github.com/username/new-repo.git

# リモートの名前を変更
git remote rename origin github

# リモートを削除
git remote remove origin
```

### ポイントまとめ

- `git remote add <name> <url>` でリモートリポジトリを登録します
- `origin` はデフォルトのリモート名として慣例的に使われます
- `git remote -v` で登録済みのリモートを確認できます

## 3. git push でリモートに送信する

`git push` は、ローカルのコミットをリモートリポジトリに送信するコマンドです。

```
【push のイメージ】

ローカル                          リモート（origin）
  main: ○──○──○──●               main: ○──○──○
                  ↑ 新しいコミット

        ─── git push origin main ──▶

ローカル                          リモート（origin）
  main: ○──○──○──●               main: ○──○──○──●
                                              ↑ 反映された！
```

### 基本的な使い方

```bash
# 指定したブランチをリモートに送信
git push origin main

# -u オプションで上流ブランチ（upstream branch）を設定
# 初回は -u をつけることを推奨
git push -u origin main
# 以降は git push だけで OK
git push
```

### -u（--set-upstream）オプション

`-u` オプションは、ローカルブランチとリモートブランチの対応関係（追跡関係）を設定します。一度設定すれば、以降は `git push` だけでリモートに送信できます。

```bash
# 新しいブランチを作成してリモートに push
git switch -c feature/login
# ... 変更を加えてコミット ...

# 初回の push（-u で追跡関係を設定）
git push -u origin feature/login
# 出力:
# Branch 'feature/login' set up to track remote branch 'feature/login' from 'origin'.

# 以降は git push だけで OK
git push
```

### push が拒否される場合

リモートに自分が持っていないコミットがある場合、push は拒否されます。

```bash
$ git push
# 出力:
# ! [rejected]        main -> main (fetch first)
# error: failed to push some refs to 'https://github.com/...'
# hint: Updates were rejected because the remote contains work that you do not
# hint: have locally.

# 解決策：先に pull してからpush
git pull origin main
git push origin main
```

> **よくある間違い：** push が拒否されたときに `git push --force` を使うのは非常に危険です。他の開発者のコミットが消える可能性があります。まず `git pull` でリモートの変更を取り込んでから push しましょう。`--force` は、rebase 後など明確な理由がある場合にのみ、`--force-with-lease` を使いましょう。

```bash
# 安全な強制 push（他の人の変更がないことを確認してから push）
git push --force-with-lease
```

### ポイントまとめ

- `git push` でローカルのコミットをリモートに送信します
- 初回は `-u` オプションで追跡ブランチを設定しましょう
- push が拒否されたら、先に `git pull` してから再度 push します
- `--force` は安易に使わず、`--force-with-lease` を検討しましょう

## 4. git fetch と git pull の違い

リモートの変更をローカルに取り込む方法は2つあります。これらの違いを正しく理解することが重要です。

### git fetch

`git fetch` は、リモートの最新情報をローカルにダウンロードしますが、**ワーキングツリーには反映しません**。安全に確認してからマージできます。

```
【git fetch のイメージ】

リモート（origin）
  main: ○──○──○──●──●   ← リモートに新しいコミットがある

ローカル
  main:         ○──○──○           ← ワーキングツリー（変わらない）
  origin/main:  ○──○──○──●──●    ← リモート追跡ブランチが更新される
```

```bash
# リモートの情報を取得（ワーキングツリーは変わらない）
git fetch origin

# 取得した変更を確認
git log origin/main --oneline

# ローカルとリモートの差分を確認
git diff main origin/main

# 問題なければマージ
git merge origin/main
```

### git pull

`git pull` は、`git fetch` + `git merge` を一度に行うコマンドです。

```
【git pull のイメージ】

  git pull = git fetch + git merge

  ① fetch: リモートの情報を取得
  ② merge: 自動的にマージ（または rebase）
```

```bash
# リモートの変更を取得してマージ（一度に行う）
git pull origin main

# 追跡ブランチが設定されていれば省略可能
git pull

# rebase 方式で pull する（履歴が直線的になる）
git pull --rebase origin main
```

### fetch vs pull の比較

```
          git fetch               git pull
         ┌─────────┐            ┌─────────┐
         │ リモート  │            │ リモート  │
         │ の情報を  │            │ の情報を  │
         │ 取得     │            │ 取得     │
         └────┬────┘            └────┬────┘
              │                      │
              ▼                      ▼
         ┌─────────┐            ┌─────────┐
         │ 確認     │            │ 自動で   │
         │ できる   │            │ マージ   │
         └────┬────┘            └─────────┘
              │
              ▼
         ┌─────────┐
         │ 手動で   │
         │ マージ   │
         └─────────┘
```

| 特徴 | git fetch | git pull |
|------|-----------|----------|
| リモートの取得 | する | する |
| 自動マージ | しない | する |
| 安全性 | 高い（確認してからマージ） | やや低い（意図しないマージの可能性） |
| 手順 | fetch → 確認 → merge | 1コマンドで完了 |
| おすすめ | 慎重に作業したいとき | 信頼できるブランチの更新 |

> **よくある間違い：** `git pull` と `git fetch` を混同する初学者は多いです。`git fetch` は「見るだけ」、`git pull` は「見て取り込む」と覚えましょう。慣れないうちは `git fetch` → 差分確認 → `git merge` の手順が安全です。

### pull --rebase の推奨

`git pull --rebase` を使うと、リモートの変更をベースにローカルのコミットを再適用するため、履歴がきれいになります。

```bash
# デフォルトで rebase 方式にする設定（推奨）
git config --global pull.rebase true

# 以降は git pull だけで rebase 方式になる
git pull
```

### ポイントまとめ

- `git fetch` はリモートの情報をダウンロードするだけ（安全）
- `git pull` は `git fetch` + `git merge` を一度に実行します
- 慣れないうちは `git fetch` → 確認 → `git merge` が安全です
- `git pull --rebase` で履歴をきれいに保てます

## 5. git clone でリポジトリを取得する

`git clone` は、リモートリポジトリの完全なコピーをローカルに作成するコマンドです。

```bash
# HTTPS でクローン
git clone https://github.com/username/my-project.git

# SSH でクローン
git clone git@github.com:username/my-project.git

# 別の名前のディレクトリにクローン
git clone https://github.com/username/my-project.git my-local-project

# 特定のブランチだけをクローン
git clone -b develop https://github.com/username/my-project.git
```

`git clone` を実行すると以下のことが自動的に行われます。

1. ディレクトリを作成
2. `.git` を初期化
3. リモートの全データをダウンロード
4. `origin` リモートを自動設定
5. デフォルトブランチをチェックアウト

```bash
# クローン後の状態を確認
cd my-project
git remote -v
# 出力:
# origin  https://github.com/username/my-project.git (fetch)
# origin  https://github.com/username/my-project.git (push)

git branch -a
# 出力:
# * main
#   remotes/origin/HEAD -> origin/main
#   remotes/origin/main
#   remotes/origin/develop
```

### ポイントまとめ

- `git clone` でリモートリポジトリの完全なコピーを作成します
- `origin` リモートが自動的に設定されます
- HTTPS と SSH の2つの方式でクローンできます

## 6. SSH 鍵と HTTPS の認証方式

Git でリモートリポジトリに接続する際、認証が必要です。主に HTTPS と SSH の2つの方式があります。

### HTTPS 方式

```bash
# HTTPS の URL 形式
https://github.com/username/repository.git
```

| 項目 | 説明 |
|------|------|
| 設定の手軽さ | 簡単（追加設定不要） |
| 認証方法 | パーソナルアクセストークン（PAT） |
| ファイアウォール | 通過しやすい（443ポート） |
| 毎回の認証 | 認証情報の入力が必要（credential helper で省略可） |

```bash
# 認証情報をキャッシュする設定
git config --global credential.helper cache

# macOS の場合（キーチェーンに保存）
git config --global credential.helper osxkeychain

# Windows の場合（資格情報マネージャーに保存）
git config --global credential.helper manager
```

### SSH 方式

```bash
# SSH の URL 形式
git@github.com:username/repository.git
```

| 項目 | 説明 |
|------|------|
| 設定の手軽さ | やや複雑（鍵の生成が必要） |
| 認証方法 | SSH 鍵ペア（公開鍵・秘密鍵） |
| ファイアウォール | ブロックされることがある（22ポート） |
| 毎回の認証 | 不要（鍵で自動認証） |

### SSH 鍵の設定手順

```bash
# 1. SSH 鍵ペアを生成
ssh-keygen -t ed25519 -C "taro@example.com"
# Enter file in which to save the key: （Enter でデフォルトパス）
# Enter passphrase: （パスフレーズを入力、省略可）

# 2. 公開鍵の内容をコピー
cat ~/.ssh/id_ed25519.pub
# 出力:
# ssh-ed25519 AAAAC3NzaC1lZDI1NTE5AAAA... taro@example.com

# 3. GitHub の Settings → SSH and GPG keys → New SSH key
#    で公開鍵を登録

# 4. 接続テスト
ssh -T git@github.com
# 出力:
# Hi username! You've successfully authenticated, but GitHub does not provide shell access.
```

### HTTPS vs SSH の比較

| 項目 | HTTPS | SSH |
|------|-------|-----|
| 初期設定 | 簡単 | SSH 鍵の生成が必要 |
| 認証 | トークン | SSH 鍵 |
| 利便性 | credential helper 必要 | 設定後は自動認証 |
| セキュリティ | 高い | 高い |
| ファイアウォール | 通過しやすい | ブロックされることがある |
| 推奨 | 初心者向け | 日常的に使う場合 |

### ポイントまとめ

- HTTPS は設定が簡単で初心者におすすめです
- SSH は一度設定すれば認証が不要で快適です
- GitHub は2021年からパスワード認証を廃止し、トークンまたは SSH が必須です
- SSH 鍵は ed25519 アルゴリズムが推奨されています

## 7. 上流ブランチ（upstream branch）の設定

上流ブランチ（upstream branch / tracking branch）とは、ローカルブランチが追跡しているリモートブランチのことです。

```
【上流ブランチの関係】

ローカル                    リモート（origin）
  main ──── tracks ────▶ origin/main
  feature/login ── tracks ──▶ origin/feature/login
```

### 上流ブランチの確認と設定

```bash
# 上流ブランチの確認
git branch -vv
# 出力:
# * main          a1b2c3d [origin/main] 最初のコミット
#   feature/login e5f6a7b [origin/feature/login] ログイン機能を追加

# 上流ブランチを設定（push 時に -u を使う）
git push -u origin feature/login

# 既存のブランチに上流ブランチを設定
git branch --set-upstream-to=origin/main main
# または短縮形
git branch -u origin/main
```

### 上流ブランチが設定されていると

```bash
# ブランチ名を省略できる
git push          # git push origin main の代わり
git pull          # git pull origin main の代わり

# リモートとの差分がわかる
git status
# 出力:
# Your branch is ahead of 'origin/main' by 2 commits.
#   (use "git push" to publish your local commits)
```

### ポイントまとめ

- 上流ブランチを設定すると、`git push` や `git pull` でブランチ名を省略できます
- `git push -u` で初回 push 時に設定するのが一般的です
- `git branch -vv` で追跡関係を確認できます

## 8. フォーク（fork）とクローン（clone）の違い

GitHub や GitLab でよく使われる概念に「フォーク（fork）」があります。クローン（clone）との違いを理解しましょう。

### クローン（clone）

クローンは、リモートリポジトリの完全なコピーを**自分のPCに**作る操作です。

```
【clone】
GitHub: original/project  ──clone──▶  自分のPC: project/
```

### フォーク（fork）

フォークは、他人のリモートリポジトリを**自分の GitHub アカウントに**コピーする操作です。

```
【fork + clone の流れ】

GitHub:  original/project
              │
         fork │（GitHub上でコピー）
              ▼
GitHub:  my-account/project  ← 自分のリポジトリとして管理
              │
        clone │（PCにダウンロード）
              ▼
自分のPC: project/
```

### フォークの活用：オープンソースへの貢献

```bash
# 1. GitHub 上でフォーク（ブラウザで操作）

# 2. フォークしたリポジトリをクローン
git clone git@github.com:my-account/project.git
cd project

# 3. 元のリポジトリを upstream として登録
git remote add upstream https://github.com/original/project.git

# リモートの確認
git remote -v
# origin    git@github.com:my-account/project.git (fetch)
# origin    git@github.com:my-account/project.git (push)
# upstream  https://github.com/original/project.git (fetch)
# upstream  https://github.com/original/project.git (push)

# 4. 機能開発用のブランチを作成
git switch -c feature/improve-docs

# 5. 変更を加えてコミット
git add .
git commit -m "docs: READMEにセットアップ手順を追記"

# 6. 自分のフォークに push
git push -u origin feature/improve-docs

# 7. GitHub 上でプルリクエスト（Pull Request）を作成
#    my-account/project → original/project
```

### 元のリポジトリの最新を取り込む

フォークしたリポジトリは自動的には更新されません。元のリポジトリの変更を取り込むには以下のようにします。

```bash
# upstream の最新を取得
git fetch upstream

# ローカルの main を upstream の main に更新
git switch main
git merge upstream/main

# 自分のフォークにも反映
git push origin main
```

### clone vs fork の比較

| 特徴 | clone | fork |
|------|-------|------|
| コピー先 | 自分のPC | 自分の GitHub アカウント |
| Git コマンド | `git clone` | GitHub の UI（ブラウザ） |
| 書き込み権限 | 元のリポジトリの権限に依存 | 自分のフォークに自由に書き込み可 |
| 用途 | 通常の開発 | OSS への貢献、自分用のカスタマイズ |
| プルリクエスト | 同じリポジトリ内のブランチ間 | フォークから元リポジトリへ |

### ポイントまとめ

- クローンは「PCにコピー」、フォークは「GitHub上にコピー」です
- オープンソースに貢献するときはフォーク → クローン → プルリクエストの流れです
- フォークした後は `upstream` リモートを設定して、元のリポジトリと同期しましょう

| コマンド | 用途 |
|----------|------|
| `git remote add <name> <url>` | リモートリポジトリを登録 |

## 6.1 フレームワークを使う理由

### 車輪の再発明を避ける

第4章・第5章で学んだように、AIエージェントの構築にはツール呼び出し、状態管理、エラーハンドリング、ループ制御など多くの共通パターンがあります。これらを毎回ゼロから実装するのは非効率です。

```
自前実装で必要なもの:
┌────────────────────────────────────────────────┐
│  ・LLM APIの呼び出しとレスポンス解析             │
│  ・ツール呼び出しのルーティング                   │
│  ・会話履歴の管理                                │
│  ・エラーハンドリングとリトライ                    │
│  ・ループ検出と最大ステップ制御                    │
│  ・状態管理とチェックポイント                     │
│  ・入出力のバリデーション                         │
│  ・ログとトレーシング                            │
│  ・ストリーミング対応                            │
│  ・マルチエージェント間の通信                     │
└────────────────────────────────────────────────┘
         ↓ フレームワークが提供
┌────────────────────────────────────────────────┐
│  数行のコードで上記すべてを利用可能               │
└────────────────────────────────────────────────┘
```

### 共通パターンの抽象化

フレームワークは以下の価値を提供します。

| 価値 | 説明 |
|------|------|
| 抽象化 | 低レベルのAPI操作を隠蔽し、ビジネスロジックに集中できる |
| 標準化 | チーム内で共通の設計パターンとコード規約を強制できる |
| エコシステム | プラグイン、ツール、インテグレーションが豊富 |
| コミュニティ | 問題解決のナレッジやベストプラクティスが蓄積されている |
| テスト・デバッグ | トレーシング、評価ツールなどが組み込まれている |

> **よくある間違い**
> - 「フレームワークを使えば全て解決する」と考えてしまう → フレームワークはツールであり、エージェント設計の基本理解が前提です。第1〜5章の知識なしに使っても良いエージェントは作れません
> - 学習コストを過小評価する → フレームワーク固有の概念や制約を理解するには一定の時間が必要です

> **ポイントまとめ**
> - フレームワークは共通パターンを抽象化し、開発効率を大幅に向上させる
> - ただしエージェント設計の基本理解は必須
> - プロジェクトの要件に合ったフレームワークを選ぶことが重要

## 6.2 LangChain / LangGraph

### LangChainの概要

LangChain（ラングチェーン）は、LLMアプリケーション開発のための最も広く使われているフレームワークです。2022年末に登場し、急速にエコシステムが拡大しました。

**主要コンポーネント**:
- **langchain-core**: 基盤となるインターフェースと抽象クラス
- **langchain**: チェーン（Chain）、エージェント、検索（Retrieval）のロジック
- **langchain-community**: サードパーティ連携
- **langgraph**: ステートマシンベースのエージェント構築ライブラリ

### AgentExecutor（レガシー）

LangChainの初期のエージェント実装は `AgentExecutor` でした。シンプルですが、複雑なフローの制御には限界がありました。

```python
# AgentExecutor（参考: 現在はLangGraphへの移行が推奨）
from langchain.agents import AgentExecutor, create_tool_calling_agent
from langchain_anthropic import ChatAnthropic
from langchain_core.tools import tool

@tool
def search_web(query: str) -> str:
    """ウェブを検索して情報を取得します"""
    return f"「{query}」の検索結果: ..."

llm = ChatAnthropic(model="claude-sonnet-4-20250514")
tools = [search_web]
agent = create_tool_calling_agent(llm, tools, prompt)
executor = AgentExecutor(agent=agent, tools=tools, verbose=True)
result = executor.invoke({"input": "最新のAI動向を教えて"})
```

### LangGraphによるステートマシン

LangGraph（ラングラフ）は、エージェントのワークフローをグラフ（ステートマシン）として定義するライブラリです。ノード（処理）とエッジ（遷移）でフローを制御します。

```
┌───────┐     ┌──────────┐     ┌──────────┐
│ START │────►│ LLM呼出  │────►│ツール実行?│
└───────┘     └──────────┘     └─────┬────┘
                   ▲                  │
                   │            Yes   │   No
                   │           ┌──────▼───┐
                   │           │ツール実行  │
                   └───────────┤結果を返す  │
                               └──────────┘
                                      │ No
                                      ▼
                               ┌──────────┐
                               │   END    │
                               └──────────┘
```

以下は LangGraph を用いたエージェントの実装例です。

```python
"""
LangGraph エージェント実装例
インストール: pip install langgraph langchain-anthropic langchain-core
"""

from typing import Annotated, TypedDict
from langchain_anthropic import ChatAnthropic
from langchain_core.messages import BaseMessage, HumanMessage, SystemMessage
from langchain_core.tools import tool
from langgraph.graph import StateGraph, END
from langgraph.graph.message import add_messages
from langgraph.prebuilt import ToolNode


# --- 状態の定義 ---
class AgentState(TypedDict):
    """エージェントの状態を定義する型"""
    messages: Annotated[list[BaseMessage], add_messages]


# --- ツールの定義 ---
@tool
def calculate(expression: str) -> str:
    """数式を計算します。四則演算に対応しています。"""
    try:
        # 安全な評価（実運用ではより厳密なサニタイズが必要）
        allowed_chars = set("0123456789+-*/.(). ")
        if not all(c in allowed_chars for c in expression):
            return "エラー: 許可されていない文字が含まれています"
        result = eval(expression)
        return f"計算結果: {expression} = {result}"
    except Exception as e:
        return f"計算エラー: {e}"


@tool
def get_current_date() -> str:
    """今日の日付を取得します。"""
    from datetime import date
    today = date.today()
    return f"今日は {today.strftime('%Y年%m月%d日')} です。"


# --- グラフの構築 ---
tools = [calculate, get_current_date]
model = ChatAnthropic(model="claude-sonnet-4-20250514").bind_tools(tools)


def call_model(state: AgentState) -> dict:
    """LLMを呼び出すノード"""
    messages = state["messages"]
    response = model.invoke(messages)
    return {"messages": [response]}


def should_continue(state: AgentState) -> str:
    """ツール呼び出しが必要かどうかを判定する条件分岐"""
    last_message = state["messages"][-1]
    if last_message.tool_calls:
        return "tools"
    return END


# グラフを組み立てる
workflow = StateGraph(AgentState)

# ノードの追加
workflow.add_node("agent", call_model)
workflow.add_node("tools", ToolNode(tools))

# エッジの追加
workflow.set_entry_point("agent")
workflow.add_conditional_edges("agent", should_continue, {"tools": "tools", END: END})
workflow.add_edge("tools", "agent")  # ツール実行後はLLMに戻る

# グラフをコンパイル
app = workflow.compile()

# 実行
if __name__ == "__main__":
    initial_state = {
        "messages": [
            SystemMessage(content="あなたは親切なアシスタントです。"),
            HumanMessage(content="今日の日付を教えて、それから 365 * 24 を計算して"),
        ]
    }

    for event in app.stream(initial_state):
        for node_name, output in event.items():
            print(f"\n--- {node_name} ---")
            if "messages" in output:
                for msg in output["messages"]:
                    print(msg.content if isinstance(msg.content, str) else msg.tool_calls)
```

> **ポイントまとめ**
> - LangChainはLLMアプリ開発の最大エコシステム
> - 新規開発ではAgentExecutorではなくLangGraphを使う
> - LangGraphはノードとエッジで柔軟なワークフローを定義できる
> - 状態（State）を型で定義し、型安全なエージェントを構築できる

## 6.3 Claude Agent SDK

### 概要

Claude Agent SDK（クロード エージェント SDK）は、Anthropicが提供する公式のエージェント構築SDKです。Claudeモデルに最適化されており、ツール定義、ガードレール（Guardrail）、ハンドオフ（Handoff）といった機能を備えています。

**特徴**:
- Anthropic公式でClaudeに最適化されている
- シンプルなAPIでエージェントを構築できる
- ガードレール機能で安全性を担保
- エージェント間のハンドオフ（委譲）をネイティブサポート

### 基本構造

```python
"""
Claude Agent SDK 基本実装例
インストール: pip install anthropic-agent-sdk
"""

from agents import Agent, Runner, function_tool


# --- ツールの定義 ---
@function_tool
def search_knowledge_base(query: str) -> str:
    """社内ナレッジベースを検索します。

    Args:
        query: 検索クエリ
    """
    # 実際にはデータベースやAPIを呼び出す
    knowledge = {
        "休暇申請": "休暇は申請システムから3日前までに申請してください。",
        "経費精算": "経費精算は月末締め、翌月15日払いです。領収書を添付してください。",
        "リモートワーク": "週3日までリモートワークが可能です。上長の承認が必要です。",
    }
    for key, value in knowledge.items():
        if key in query:
            return value
    return "該当する情報が見つかりませんでした。人事部にお問い合わせください。"


@function_tool
def create_ticket(title: str, description: str, priority: str) -> str:
    """サポートチケットを作成します。

    Args:
        title: チケットのタイトル
        description: 詳細な説明
        priority: 優先度 (low, medium, high)
    """
    ticket_id = "TKT-2026-0042"  # 実際にはDBで採番
    return f"チケットを作成しました。ID: {ticket_id}, タイトル: {title}, 優先度: {priority}"


# --- エージェントの定義 ---
support_agent = Agent(
    name="社内サポートエージェント",
    instructions="""あなたは社内のITサポートエージェントです。
以下のルールに従ってください：
1. まずナレッジベースを検索して回答を探す
2. ナレッジベースで解決できない場合はチケットを作成する
3. 常に丁寧な言葉遣いで対応する
4. 個人情報は扱わない""",
    tools=[search_knowledge_base, create_ticket],
    model="claude-sonnet-4-20250514",
)

# --- 実行 ---
if __name__ == "__main__":
    result = Runner.run_sync(
        support_agent,
        "リモートワークの規定について教えてください",
    )
    print(result.final_output)
```

### ガードレール（Guardrails）

ガードレールは、エージェントの入出力を監視し、不適切な動作を防止する仕組みです。Claude Agent SDKではガードレールをエージェントに組み込むことができます。

```python
from agents import Agent, Runner, InputGuardrail, GuardrailFunctionOutput, function_tool
from pydantic import BaseModel


# 入力ガードレール: 不適切な入力を検出
class SafetyCheck(BaseModel):
    is_safe: bool
    reason: str


safety_agent = Agent(
    name="安全性チェッカー",
    instructions="ユーザーの入力が安全かどうか判定してください。個人情報の要求や不正な操作指示を検出します。",
    output_type=SafetyCheck,
)


async def check_input_safety(ctx, agent, input_text) -> GuardrailFunctionOutput:
    """入力の安全性をチェックするガードレール"""
    result = await Runner.run(safety_agent, input_text, context=ctx.context)
    safety = result.final_output_as(SafetyCheck)
    return GuardrailFunctionOutput(
        output_info=safety,
        tripwire_triggered=not safety.is_safe,  # 安全でなければ停止
    )


# ガードレール付きエージェント
guarded_agent = Agent(
    name="ガードレール付きサポート",
    instructions="社内サポートを提供するエージェントです。",
    tools=[search_knowledge_base],
    input_guardrails=[
        InputGuardrail(guardrail_function=check_input_safety),
    ],
)
```

### ハンドオフ（Handoff）

複数のエージェント間でタスクを委譲する機能です。専門分野に応じて適切なエージェントに処理を引き継げます。

```python
from agents import Agent, Runner

# 専門エージェントの定義
it_agent = Agent(
    name="IT技術サポート",
    instructions="PCやネットワークの技術的な問題を解決します。",
    tools=[search_knowledge_base, create_ticket],
)

hr_agent = Agent(
    name="人事サポート",
    instructions="休暇、福利厚生、評価制度について回答します。",
    tools=[search_knowledge_base],
)

# トリアージエージェント（振り分け役）
triage_agent = Agent(
    name="受付エージェント",
    instructions="""ユーザーの問い合わせ内容を判断し、適切な専門エージェントに引き継いでください。
- PCやネットワークの問題 → IT技術サポート
- 休暇や福利厚生の問題 → 人事サポート""",
    handoffs=[it_agent, hr_agent],  # ハンドオフ先を指定
)

# 実行するとトリアージ→専門エージェントへ自動的に引き継がれる
if __name__ == "__main__":
    result = Runner.run_sync(triage_agent, "パスワードを忘れてしまいました")
    print(result.final_output)
```

> **ポイントまとめ**
> - Claude Agent SDKはClaudeに最適化された公式フレームワーク
> - `@function_tool`デコレータで簡単にツールを定義できる
> - ガードレールで入出力の安全性を自動チェックできる
> - ハンドオフで専門エージェントへの自動振り分けが可能

## 6.4 CrewAI

### 概要

CrewAI（クルーエーアイ）は、ロールベース（Role-Based）のマルチエージェントフレームワークです。「チーム」の比喩を使い、各エージェントに役割（Role）を持たせて協調させます。

### 4つのコア概念

```
┌─────────────────────────────────────────────────────┐
│                    Crew（チーム）                      │
│                                                     │
│  ┌─────────┐   ┌─────────┐   ┌─────────┐          │
│  │ Agent A  │   │ Agent B  │   │ Agent C  │          │
│  │リサーチャー│   │ライター  │   │エディター │          │
│  └────┬────┘   └────┬────┘   └────┬────┘          │
│       │              │              │               │
│  ┌────▼────┐   ┌────▼────┐   ┌────▼────┐          │
│  │ Task 1   │   │ Task 2   │   │ Task 3   │          │
│  │情報収集   │   │記事執筆   │   │校正・編集 │          │
│  └─────────┘   └─────────┘   └─────────┘          │
│                                                     │
│  Tool: 検索ツール, ファイル操作ツール ...               │
└─────────────────────────────────────────────────────┘
```

| 概念 | 説明 |
|------|------|
| **Crew** | エージェントのチーム全体。実行戦略（直列/並列）を管理 |
| **Agent** | 役割（Role）・ゴール（Goal）・背景（Backstory）を持つ個体 |
| **Task** | エージェントに割り当てられる具体的なタスク |
| **Tool** | エージェントが使用できるツール（検索、ファイル操作等） |

### コード例

```python
"""
CrewAI によるコンテンツ制作チームの実装例
インストール: pip install crewai crewai-tools
"""

from crewai import Agent, Task, Crew, Process

# --- エージェントの定義 ---
researcher = Agent(
    role="シニアリサーチャー",
    goal="指定されたトピックについて正確で最新の情報を収集する",
    backstory="""あなたは10年以上の経験を持つリサーチの専門家です。
信頼性の高い情報源から正確なデータを収集し、
分かりやすく整理することが得意です。""",
    verbose=True,
    allow_delegation=False,
)

writer = Agent(
    role="テクニカルライター",
    goal="リサーチ結果を基に、読者に分かりやすい記事を執筆する",
    backstory="""あなたは技術記事の執筆を専門とするライターです。
複雑な技術概念を初心者にも理解できるよう噛み砕いて
説明することに定評があります。""",
    verbose=True,
    allow_delegation=False,
)

editor = Agent(
    role="シニアエディター",
    goal="記事の品質を最終チェックし、公開可能な状態に仕上げる",
    backstory="""あなたは出版業界で15年の経験を持つ編集者です。
文章の論理構成、事実の正確性、読みやすさを
厳しくチェックします。""",
    verbose=True,
    allow_delegation=False,
)

# --- タスクの定義 ---
research_task = Task(
    description="「AIエージェントの最新動向」について調査し、主要なトレンドを5つ以上まとめてください。",
    expected_output="主要トレンドのリスト（各項目に概要説明付き）",
    agent=researcher,
)

writing_task = Task(
    description="リサーチ結果を基に、2000文字程度の技術記事を執筆してください。",
    expected_output="構成の整った技術記事（見出し・本文・まとめ付き）",
    agent=writer,
)

editing_task = Task(
    description="記事を校正・編集し、公開可能な最終版に仕上げてください。",
    expected_output="校正済みの最終記事",
    agent=editor,
)

# --- チームの構成と実行 ---
crew = Crew(
    agents=[researcher, writer, editor],
    tasks=[research_task, writing_task, editing_task],
    process=Process.sequential,  # 直列実行（リレー型）
    verbose=True,
)

if __name__ == "__main__":
    result = crew.kickoff()
    print("\n=== 最終成果物 ===")
    print(result)
```

> **ポイントまとめ**
> - CrewAIは「チーム」の比喩で直感的にマルチエージェントを構築できる
> - Agent（役割）、Task（タスク）、Crew（チーム）、Tool（ツール）の4概念が中心
> - `Process.sequential`（直列）と`Process.hierarchical`（階層型）を選択できる
> - Backstoryを充実させることでエージェントの出力品質が向上する

## 6.5 その他のフレームワーク

### AutoGen（Microsoft）

AutoGen（オートジェン）は、Microsoftが開発したマルチエージェント会話フレームワークです。エージェント同士が「会話」を通じてタスクを解決するアプローチが特徴です。

**主な特徴**:
- エージェント間の自動会話（Auto-conversation）
- 人間をループに含めるHuman-in-the-loopパターン
- コード自動生成と実行（サンドボックス対応）
- GroupChatによる複数エージェントの議論

```python
# AutoGenの概念コード（v0.4系）
from autogen_agentchat.agents import AssistantAgent
from autogen_agentchat.teams import RoundRobinGroupChat

agent1 = AssistantAgent("analyst", model_client=model_client,
    system_message="データ分析の専門家として回答してください")
agent2 = AssistantAgent("engineer", model_client=model_client,
    system_message="ソフトウェアエンジニアとして回答してください")

team = RoundRobinGroupChat([agent1, agent2], max_turns=3)
result = await team.run(task="売上データの分析パイプラインを設計して")
```

### Dify

Dify（ディファイ）は、ノーコード/ローコードのLLMアプリケーション開発プラットフォームです。GUIでワークフローを構築できるため、非エンジニアでもエージェントを作成できます。

**主な特徴**:
- ブラウザベースのビジュアルエディタ
- RAG（検索拡張生成）パイプラインの構築が容易
- セルフホスト可能（Docker対応）
- 複数のLLMプロバイダーに対応

### Mastra

Mastra（マストラ）は、TypeScript/JavaScript向けのエージェントフレームワークです。Webアプリケーションとの統合がしやすく、フロントエンド開発者に親和性があります。

**主な特徴**:
- TypeScriptネイティブ
- Next.js等のWebフレームワークとの統合が容易
- ワークフローエンジン内蔵
- ベクトルデータベースとの連携

### OpenAI Agents SDK

OpenAI Agents SDK（旧Swarm）は、OpenAIが提供するエージェント構築SDKです。Claude Agent SDKと類似したコンセプトで、ツール定義とハンドオフをサポートします。

**主な特徴**:
- OpenAIモデルに最適化
- シンプルなAgent/Tool/Handoff構造
- ガードレール（Guardrails）機能
- トレーシング機能内蔵

### フレームワーク概要比較

| フレームワーク | 言語 | 特徴 | LLM対応 |
|-------------|------|------|---------|

> - Difyはノーコードで始められるため、プロトタイピングに最適
> - TypeScript環境ではMastraが有力な選択肢

### 複数フレームワークの組み合わせ

実務では1つのフレームワークに固執する必要はありません。例えば以下のような組み合わせが有効です。

- **プロトタイプをDifyで作成** → **本番をLangGraphで実装**
- **個別エージェントをClaude Agent SDKで構築** → **オーケストレーションをLangGraphで管理**
- **社内ツールはCrewAIで素早く構築** → **顧客向けシステムはLangGraphで堅牢に構築**

> **よくある間違い**
> - フレームワークの選定に時間をかけすぎる → まずは1つを選んで手を動かしましょう。後から移行は可能です
> - フレームワークのバージョンアップに追従しない → エージェント分野は進化が速いです。定期的にドキュメントをチェックしましょう
> - フレームワークに過度に依存する → コアロジックはフレームワーク非依存に保つと、移行が容易になります

> **ポイントまとめ**
> - ユースケース・チームのスキル・LLMプロバイダーで選定する
> - 迷ったらLangGraph（エコシステム最大）またはClaude Agent SDK（Claudeユーザー）
> - 学習コストと将来の拡張性のバランスを考慮する
> - プロトタイプと本番で異なるフレームワークを使う戦略も有効
> - フレームワークは手段であり目的ではない。重要なのはエージェント設計の基本

# 第1章：プロジェクトマネジメント概論

## この章で学ぶこと

- プロジェクトとは何か、プロジェクトマネジメント（PM）の定義
- PMの三角形（スコープ・時間・コスト）
- ウォーターフォール vs アジャイル
- PMの役割とステークホルダー
- プロジェクト憲章（Project Charter）

---

## 1.1 プロジェクトとは何か

### プロジェクトの定義

プロジェクトとは、**独自の成果物やサービスを創出するために行われる、有期的な取り組み**である。

日常業務（オペレーション）との違いを理解することが重要だ。

| 特徴 | プロジェクト | オペレーション（日常業務） |
|------|------------|----------------------|
| 期間 | 有期（開始と終了がある） | 継続的 |
| 目的 | 独自の成果物を作る | 反復的な業務を遂行する |
| 例 | 新機能の開発、システム刷新 | サーバー監視、カスタマーサポート |

### ソフトウェア開発におけるプロジェクトの例

- ECサイトの新規構築
- 既存システムのマイクロサービス化
- モバイルアプリのリニューアル
- API基盤の刷新
- CI/CDパイプラインの構築

## 1.2 プロジェクトマネジメントの三角形

PMの三角形（トリプルコンストレイント）は、プロジェクトの3つの制約要素を表す。

```
        スコープ
       （Scope）
        /    \
       /      \
      /   品質  \
     /  (Quality) \
    /              \
   時間 ―――――――― コスト
  (Time)        (Cost)
```

### 3つの制約

| 制約 | 説明 | ソフトウェア開発での例 |
|------|------|---------------------|
| **スコープ** | 何を作るか（機能・要件） | 検索機能、ユーザー管理、決済機能 |
| **時間** | いつまでに作るか（期限） | Q3末リリース、スプリント2週間 |
| **コスト** | どれだけのリソースを使うか | エンジニア5名、サーバー費用月額50万円 |

### トレードオフの原則

**3つすべてを同時に最適化することはできない。**

例えば：
- スコープを増やす → 時間が延びる or コストが増える
- 期限を短縮する → スコープを削る or コスト（人員）を増やす
- コストを削減する → スコープを削る or 時間が延びる

> **実務でのポイント**: 「全部やりたい、期限は変えない、人も増やせない」という要求が来たら、PMとしてトレードオフを明確に説明する必要がある。

## 1.3 ウォーターフォール vs アジャイル

### ウォーターフォール

工程を順番に進めていく伝統的な手法。

```
要件定義 → 設計 → 実装 → テスト → リリース → 保守
```

**メリット：**
- 計画が立てやすい
- 進捗が把握しやすい
- ドキュメントが充実する
- 大規模・規制の厳しいプロジェクトに向く

**デメリット：**
- 途中での仕様変更に弱い
- フィードバックが遅い（完成近くまで動くものが見られない）
- 後工程で問題が発覚すると手戻りが大きい

### アジャイル

短い反復（イテレーション）で少しずつ成果物を作り上げていく手法。

```
計画 → 開発 → テスト → レビュー → 計画 → 開発 → ...
    Sprint 1                        Sprint 2
```

**メリット：**
- 変化に対応しやすい
- 早期にフィードバックを得られる
- リスクを小さく分割できる
- チームの自律性が高まる

**デメリット：**
- 全体の見通しが立てにくい場合がある
- ドキュメントが不足しがち
- 顧客・ステークホルダーの継続的な関与が必要
- 未熟なチームだと混乱しやすい

### どちらを選ぶか

| 状況 | 推奨 |
|------|------|
| 要件が明確で変更が少ない | ウォーターフォール |
| 要件が不確実で変化が多い | アジャイル |
| 規制が厳しい（医療・金融） | ウォーターフォール（またはハイブリッド） |
| スタートアップ・新規事業 | アジャイル |
| 大規模チーム | 状況による（SAFe等のスケーリングフレームワーク） |

> **現実**: 多くの現場では純粋なウォーターフォールでもアジャイルでもなく、ハイブリッドで運用されている。

## 1.4 プロジェクトマネージャーの役割

### PMの主な責務

1. **計画立案** - プロジェクト計画の策定と更新
2. **スコープ管理** - 何を作るか/作らないかの定義
3. **スケジュール管理** - 期限の設定と進捗追跡
4. **リソース管理** - 人員・予算の配分
5. **リスク管理** - リスクの特定と対策
6. **コミュニケーション** - ステークホルダーとの調整
7. **品質管理** - 成果物の品質担保
8. **課題管理** - 発生した問題の解決促進

### ソフトウェア開発におけるPMの種類

| 役割 | 主な責務 | フォーカス |
|------|---------|----------|
| **プロジェクトマネージャー** | プロジェクト全体の管理 | プロセス・スケジュール |
| **プロダクトマネージャー** | 何を作るかの意思決定 | ユーザー価値・ビジネス |
| **テックリード** | 技術的意思決定 | アーキテクチャ・技術品質 |
| **スクラムマスター** | チームのプロセス改善 | アジャイルプロセス・障害除去 |
| **エンジニアリングマネージャー** | チームメンバーの成長支援 | ピープルマネジメント |

> **注意**: 小規模チームでは1人が複数の役割を兼任することが多い。

## 1.5 ステークホルダー

### ステークホルダーとは

プロジェクトに関与する、または影響を受けるすべての人や組織のこと。

### ソフトウェア開発プロジェクトのステークホルダー

```
                    経営層・スポンサー
                         |
                    プロダクトオーナー
                    /    |    \
               営業   デザイナー  QA
              /          |          \
        顧客・ユーザー  開発チーム   インフラチーム
                         |
                    外部ベンダー
```

### ステークホルダーマッピング

影響力と関心度で4象限に分類する。

```
  高 |  満足させる    |  重点管理
影  |  (Keep         |  (Manage
響  |   Satisfied)   |   Closely)
力  |----------------|----------------
  低 |  最小限の      |  情報提供
    |  対応          |  (Keep
    |  (Monitor)     |   Informed)
    +----------------+----------------
           低              高
              関心度
```

- **重点管理**（影響力 高・関心度 高）: 経営層、PO → 密にコミュニケーション
- **満足させる**（影響力 高・関心度 低）: 他部門の管理者 → 定期報告
- **情報提供**（影響力 低・関心度 高）: エンドユーザー → 進捗共有
- **最小限の対応**（影響力 低・関心度 低）: 間接的な関係者 → 必要時のみ

## 1.6 プロジェクト憲章（Project Charter）

### プロジェクト憲章とは

プロジェクトの開始を正式に承認し、基本情報をまとめた文書。プロジェクトの「なぜ」「何を」「誰が」を明確にする。

### プロジェクト憲章に含める要素

1. **プロジェクト名** - 簡潔で分かりやすい名称
2. **背景と目的** - なぜこのプロジェクトをやるのか
3. **スコープの概要** - 何を作るか（含むもの/含まないもの）
4. **成功指標** - どうなったら成功か（KPI）
5. **スケジュール概要** - 主要マイルストーン
6. **予算** - 概算コスト
7. **ステークホルダー** - 主要な関係者と役割
8. **リスク** - 想定されるリスク
9. **前提条件と制約** - プロジェクトの前提と制約事項

### プロジェクト憲章の例（概要）

```markdown
# プロジェクト憲章：社内ナレッジベース構築

## 背景
社内の技術情報が散在しており、新メンバーのオンボーディングに
平均2ヶ月かかっている。

## 目的
社内ナレッジを一元管理し、オンボーディング期間を1ヶ月に短縮する。

## 成功指標
- オンボーディング期間：2ヶ月 → 1ヶ月
- 記事数：初期100記事
- 月間アクティブユーザー：全エンジニアの80%以上

## 主要マイルストーン
- 要件定義完了：4月末
- MVP リリース：6月末
- 正式リリース：8月末
```

## 1.7 プロジェクトのライフサイクル

### 5つのプロセス群（PMBOKベース）

```
立ち上げ → 計画 → 実行 → 監視・コントロール → 終結
                          ↑        ↓
                          ←――――――――←
                       （フィードバックループ）
```

| プロセス群 | 主な活動 | 成果物 |
|-----------|---------|--------|
| **立ち上げ** | プロジェクト憲章の作成、ステークホルダー特定 | プロジェクト憲章 |
| **計画** | スコープ定義、WBS作成、スケジュール策定 | プロジェクト計画書 |
| **実行** | タスクの遂行、チームマネジメント | 成果物（プロダクト） |
| **監視・コントロール** | 進捗確認、変更管理、リスク対応 | 進捗レポート |
| **終結** | 成果物の引き渡し、振り返り、知見の記録 | 完了報告書 |

## 1.8 PMに必要なスキル

### ハードスキル

- プロジェクト管理ツールの使い方（Jira、GitHub Projects等）
- スケジュール管理手法（ガントチャート、クリティカルパス）
- 見積もり手法（PERT、ストーリーポイント）
- 基本的な技術理解（アーキテクチャ、CI/CD等）

### ソフトスキル

- **コミュニケーション能力** - 技術者と非技術者の橋渡し
- **ファシリテーション** - 会議の進行、意見の引き出し
- **交渉力** - スコープ・期限の調整
- **問題解決力** - 課題の構造化と解決策の立案
- **リーダーシップ** - チームの方向づけ、モチベーション管理

---

## まとめ

| 概念 | ポイント |
|------|---------|
| プロジェクト | 有期的で独自の成果物を生む取り組み |
| PMの三角形 | スコープ・時間・コストのトレードオフ |
| ウォーターフォール | 順序的、計画重視、変更に弱い |
| アジャイル | 反復的、適応重視、フィードバック重視 |
| ステークホルダー | 影響力と関心度でマッピングし対応を変える |
| プロジェクト憲章 | プロジェクトの目的・スコープ・体制を明文化 |

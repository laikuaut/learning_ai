## 基本問題

### 問題1: 鍵アルゴリズムの選択
以下の鍵アルゴリズムについて、現在の推奨度を「推奨」「条件付き」「非推奨」で答えてください。

1. Ed25519
2. RSA 2048bit
3. DSA
4. ECDSA
5. RSA 4096bit

<details>
<summary>解答</summary>

```
1. Ed25519      → 推奨（最も推奨されるアルゴリズム。高速・安全・鍵が短い）
2. RSA 2048bit  → 条件付き（最低限のセキュリティ。3072bit以上を推奨）
3. DSA          → 非推奨（脆弱。OpenSSH 7.0以降でデフォルト無効）
4. ECDSA        → 条件付き（楕円曲線暗号だが、一部のカーブに懸念あり）
5. RSA 4096bit  → 条件付き（互換性が必要な場合に使用。Ed25519より鍵が大きい）
```

</details>

### 問題2: 鍵ペアの生成
以下の条件でssh-keygenコマンドを書いてください。

1. Ed25519の鍵をデフォルトの場所に生成（コメント: "user@example.com"）
2. RSA 4096bitの鍵をデフォルトの場所に生成
3. Ed25519の鍵を `~/.ssh/id_ed25519_github` に生成（コメント: "github"）

<details>
<summary>解答</summary>

```bash
# 1.
$ ssh-keygen -t ed25519 -C "user@example.com"

# 2.
$ ssh-keygen -t rsa -b 4096

# 3.
$ ssh-keygen -t ed25519 -C "github" -f ~/.ssh/id_ed25519_github
```

</details>

### 問題3: パーミッション
以下のファイル・ディレクトリに設定すべきパーミッション（数値）を答えてください。

1. `~/.ssh/` ディレクトリ
2. `~/.ssh/id_ed25519`（秘密鍵）
3. `~/.ssh/id_ed25519.pub`（公開鍵）
4. `~/.ssh/authorized_keys`
5. `~/.ssh/config`

<details>
<summary>解答</summary>

```bash
# 1. ~/.ssh/ ディレクトリ
chmod 700 ~/.ssh          # 所有者のみフルアクセス

# 2. 秘密鍵
chmod 600 ~/.ssh/id_ed25519    # 所有者のみ読み書き

# 3. 公開鍵
chmod 644 ~/.ssh/id_ed25519.pub # 所有者は読み書き、他は読み取り

# 4. authorized_keys
chmod 600 ~/.ssh/authorized_keys # 所有者のみ読み書き

# 5. config
chmod 600 ~/.ssh/config    # 所有者のみ読み書き
```

覚え方：**秘密に関わるものは600、公開情報は644、ディレクトリは700**

</details>

### 問題4: 公開鍵の登録
リモートサーバー `server.example.com` に公開鍵を登録する方法を2つ書いてください。

<details>
<summary>解答</summary>

```bash
# 方法1: ssh-copy-id を使う（推奨）
$ ssh-copy-id user@server.example.com

# 特定の鍵を指定する場合
$ ssh-copy-id -i ~/.ssh/id_ed25519_work.pub user@server.example.com

# 方法2: 手動で登録
$ cat ~/.ssh/id_ed25519.pub | ssh user@server.example.com "mkdir -p ~/.ssh && chmod 700 ~/.ssh && cat >> ~/.ssh/authorized_keys && chmod 600 ~/.ssh/authorized_keys"
```

</details>

## 応用問題

### 問題5: 公開鍵認証の仕組み
公開鍵認証の認証フローを以下の選択肢から正しい順番に並べてください。

a) サーバーが公開鍵で署名を検証する
b) クライアントがサーバーにログイン要求を送る
c) クライアントが秘密鍵でチャレンジに署名する
d) サーバーがランダムなチャレンジデータを送る
e) 認証成功

<details>
<summary>解答</summary>

正しい順番: **b → d → c → a → e**

```
1. b) クライアントがサーバーにログイン要求を送る
2. d) サーバーがランダムなチャレンジデータを送る
3. c) クライアントが秘密鍵でチャレンジに署名する
4. a) サーバーが公開鍵で署名を検証する
5. e) 認証成功

重要: 秘密鍵そのものはネットワーク上を流れません。
署名（秘密鍵で作った「ハンコ」のようなもの）だけが送信されます。
```

</details>

### 問題6: パスフレーズ
以下の質問に答えてください。

1. パスフレーズの役割は何ですか？
2. パスフレーズを設定しない場合のリスクは何ですか？
3. 既存の鍵のパスフレーズを変更するコマンドは何ですか？

<details>
<summary>解答</summary>

```
1. パスフレーズの役割:
   秘密鍵を暗号化して保護する「鍵の鍵」です。
   秘密鍵ファイルが盗まれても、パスフレーズがなければ使用できません。

2. パスフレーズなしのリスク:
   秘密鍵ファイルが盗まれた場合、即座にその鍵で認証しているすべての
   サーバーに不正アクセスされる可能性があります。
   パスフレーズがあれば、その間に鍵を無効化する時間的猶予が得られます。

3. パスフレーズの変更コマンド:
   $ ssh-keygen -p -f ~/.ssh/id_ed25519
```

</details>

### 問題7: トラブルシューティング
以下のエラーメッセージの原因と対処法をそれぞれ答えてください。

**エラー1:**
```
@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@
@         WARNING: UNPROTECTED PRIVATE KEY FILE!          @
@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@
Permissions 0644 for '/home/user/.ssh/id_ed25519' are too open.
```

**エラー2:**
```
Permission denied (publickey).
```

<details>
<summary>解答</summary>

**エラー1: UNPROTECTED PRIVATE KEY FILE**

原因: 秘密鍵ファイルのパーミッションが緩すぎる（644 = 他のユーザーも読める状態）

対処:
```bash
$ chmod 600 ~/.ssh/id_ed25519
```

**エラー2: Permission denied (publickey)**

原因（複数の可能性）:
1. サーバーに公開鍵が登録されていない
2. 正しい秘密鍵が使われていない
3. サーバー側のパーミッションが不正（.ssh: 700, authorized_keys: 600 が必要）
4. サーバー側のホームディレクトリのパーミッションが不正

対処:
```bash
# 1. どの鍵が使われているか確認
$ ssh -v user@server 2>&1 | grep "Offering"

# 2. 正しい鍵を指定して接続
$ ssh -i ~/.ssh/id_ed25519_correct user@server

# 3. サーバー側のパーミッション確認（別の方法でログインして）
$ ls -la ~/.ssh/
$ ls -ld ~
```

</details>

### 問題8: フィンガープリント
以下のコマンドの出力を説明してください。また、フィンガープリントはどのような場面で使用されますか？

```bash
$ ssh-keygen -l -f ~/.ssh/id_ed25519.pub
256 SHA256:xxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxx user@host (ED25519)
```

<details>
<summary>解答</summary>

**出力の説明**:
```
256                    → 鍵のビット長（Ed25519は常に256bit）
SHA256:xxxxx...        → 公開鍵のSHA256ハッシュ値（フィンガープリント）
user@host              → コメント（鍵生成時に -C で指定した文字列）
(ED25519)              → 鍵のアルゴリズム
```

**使用場面**:
```
1. SSH初回接続時のホスト鍵確認
   → サーバー管理者からフィンガープリントを事前に受け取り、
     表示されたフィンガープリントと一致するか確認

2. 鍵の識別
   → 複数の鍵を管理している場合に、どの鍵がどれか区別する

3. GitHubなどのサービスでの登録確認
   → 登録した鍵が正しいか、フィンガープリントで照合
```

</details>

## チャレンジ問題

### 問題9: 用途別の鍵管理
あなたは以下の3つの接続先を持っています。セキュリティのベストプラクティスに従って、鍵を生成するコマンドとファイル構成を設計してください。

- GitHub（個人アカウント）
- 会社の開発サーバー
- 個人のVPS

<details>
<summary>解答</summary>

```bash
# 1. GitHub用の鍵
$ ssh-keygen -t ed25519 -C "github-personal" -f ~/.ssh/id_ed25519_github

# 2. 会社の開発サーバー用の鍵
$ ssh-keygen -t ed25519 -C "work-dev" -f ~/.ssh/id_ed25519_work

# 3. 個人VPS用の鍵
$ ssh-keygen -t ed25519 -C "personal-vps" -f ~/.ssh/id_ed25519_vps

# ファイル構成:
~/.ssh/
├── id_ed25519_github       # GitHub用秘密鍵
├── id_ed25519_github.pub   # GitHub用公開鍵
├── id_ed25519_work         # 会社用秘密鍵
├── id_ed25519_work.pub     # 会社用公開鍵
├── id_ed25519_vps          # VPS用秘密鍵
├── id_ed25519_vps.pub      # VPS用公開鍵
├── config                  # SSH設定ファイル
├── known_hosts             # ホスト鍵の記録
└── authorized_keys         # 許可された公開鍵（サーバー側）
```

**すべての鍵にパスフレーズを設定し、ssh-agentと組み合わせて利用するのがベストプラクティスです。**

</details>

### 問題10: GitHubへのSSH接続設定
GitHubにSSH接続するための手順を最初から最後まで書いてください。接続テストのコマンドも含めてください。

<details>
<summary>解答</summary>

```bash
# 1. GitHub用の鍵ペアを生成
$ ssh-keygen -t ed25519 -C "your_email@example.com" -f ~/.ssh/id_ed25519_github
# パスフレーズを入力（推奨）

# 2. 公開鍵の内容をコピー
$ cat ~/.ssh/id_ed25519_github.pub
# 出力をクリップボードにコピー

# macOS: pbcopy < ~/.ssh/id_ed25519_github.pub
# Linux: xclip -selection clipboard < ~/.ssh/id_ed25519_github.pub
# Windows: clip < ~/.ssh/id_ed25519_github.pub

# 3. GitHubに公開鍵を登録
#    Settings → SSH and GPG keys → New SSH key
#    Title: 識別しやすい名前（例: "My Laptop"）
#    Key: コピーした公開鍵を貼り付け

# 4. ~/.ssh/config に設定を追加
Host github.com
    HostName github.com
    User git
    IdentityFile ~/.ssh/id_ed25519_github
    IdentitiesOnly yes

# 5. 接続テスト
$ ssh -T git@github.com
# 期待される出力:
# Hi username! You've successfully authenticated, but GitHub does not provide shell access.

# 6. Gitでの使用テスト
$ git clone git@github.com:username/repo.git
```

</details>

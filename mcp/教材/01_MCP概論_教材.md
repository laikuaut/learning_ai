## 1.1 MCPとは何か

### Model Context Protocolの定義

MCP（Model Context Protocol）は、Anthropic社が2024年11月に公開したオープンプロトコル（open protocol）です。一言で表すと「AIアプリケーションと外部データソース・ツールをつなぐ共通規格」です。

MCPを使うことで、大規模言語モデル（LLM: Large Language Model）を搭載したアプリケーションが、ファイルシステム、データベース、Web API、開発ツールなど、あらゆる外部リソースと**標準化された方法**で接続できるようになります。

### なぜ「プロトコル」なのか

プロトコル（protocol）とは、異なるシステム同士が通信するための「取り決め」です。HTTPがWebブラウザとWebサーバーの通信を標準化したように、MCPはAIアプリと外部ツールの通信を標準化します。

```
従来：各AIアプリが独自の方法でツールと接続
  → アプリごとに異なる実装が必要

MCP：共通のプロトコルでツールと接続
  → 一度作れば、どのMCP対応アプリでも動く
```

### MCPの基本情報

| 項目 | 内容 |
|---|---|
| 策定者 | Anthropic社 |
| 公開時期 | 2024年11月 |
| ライセンス | オープンソース（MIT） |
| 仕様バージョン | 2025-03-26（最新） |
| 公式サイト | https://modelcontextprotocol.io |
| GitHub | https://github.com/modelcontextprotocol |

### ポイントまとめ
- MCPは「AIアプリと外部ツールをつなぐ共通規格」
- Anthropic社が策定したオープンプロトコルで、誰でも自由に利用・実装できる
- HTTPがWebを標準化したように、MCPはAIとツールの接続を標準化する

## 1.2 MCPが解決する問題

### N×M問題とは

MCPが登場する前、AIアプリケーションと外部ツールを連携させるには、**個別にカスタム統合（custom integration）を作る**必要がありました。

例えば、3つのAIアプリ（Claude Desktop、Cursor、VS Code）と4つのツール（GitHub、Slack、PostgreSQL、ファイルシステム）を連携させる場合を考えてみましょう。

```
MCPなし：N×M = 3×4 = 12通りの個別実装が必要

┌──────────────┐     ┌────────────┐
│ Claude Desktop├──┬──┤ GitHub     │
├──────────────┤  │  ├────────────┤
│ Cursor       ├──┼──┤ Slack      │
├──────────────┤  │  ├────────────┤
│ VS Code      ├──┼──┤ PostgreSQL │
└──────────────┘  │  ├────────────┤
                  └──┤ ファイル   │
                     └────────────┘
  ※ すべての組み合わせに個別の接続コードが必要
    → 12通りの実装 = 膨大な開発・保守コスト
```

これがN×M問題です。AIアプリがN個、ツールがM個あると、N×M通りの個別実装が必要になり、開発コストが爆発的に増加します。

### 従来のカスタム統合の課題

| 課題 | 説明 |
|---|---|
| 開発コスト | ツールごとにAPIの仕様を調べ、個別実装が必要 |
| 保守コスト | ツール側のAPI変更に各アプリが個別対応 |
| 品質のばらつき | アプリごとに実装品質が異なる |
| 重複作業 | 同じツールへの接続を各アプリが独自に実装 |
| 拡張性の限界 | 新しいツールを追加するたびに全アプリを改修 |

### USB-Cのアナロジー

MCPの役割は、USB-C端子に例えるとわかりやすいです。

```
USB-C登場前（= MCPなし）
  スマホ  → Micro USB
  ノートPC → 独自端子
  カメラ  → Mini USB
  → デバイスごとに異なるケーブルが必要

USB-C登場後（= MCPあり）
  スマホ  → USB-C ─┐
  ノートPC → USB-C ─┼─ 共通ケーブル1本でOK
  カメラ  → USB-C ─┘
```

MCPも同じです。

```
MCPあり：N+M = 3+4 = 7通りの実装で済む

┌──────────────┐              ┌────────────┐
│ Claude Desktop├──┐      ┌───┤ GitHub     │
├──────────────┤  │      │   ├────────────┤
│ Cursor       ├──┼──MCP─┼───┤ Slack      │
├──────────────┤  │      │   ├────────────┤
│ VS Code      ├──┘      ├───┤ PostgreSQL │
└──────────────┘          │   ├────────────┤
                          └───┤ ファイル   │
                              └────────────┘
  ※ 各アプリはMCPクライアントを1つ実装
    各ツールはMCPサーバーを1つ実装
    → 合計7通りの実装で全組み合わせが可能
```

### ポイントまとめ
- MCPなしでは N×M 通りの個別実装が必要（N×M問題）
- MCPありでは N+M 通りの実装で全組み合わせをカバーできる
- USB-Cが充電ケーブルを統一したように、MCPはAI連携を統一する

## 1.3 MCPの全体像

### 3層アーキテクチャ

MCPは、ホスト（Host）、クライアント（Client）、サーバー（Server）の3つのコンポーネントで構成されます。

```
┌─────────────────────────────────────────────┐
│              ホスト（Host）                    │
│         例：Claude Desktop, Cursor           │
│                                             │
│  ┌──────────────┐    ┌──────────────┐       │
│  │ MCPクライアント│    │ MCPクライアント│       │
│  │     (A)      │    │     (B)      │       │
│  └──────┬───────┘    └──────┬───────┘       │
│         │                   │               │
└─────────┼───────────────────┼───────────────┘
          │ MCP Protocol      │ MCP Protocol
          │ (JSON-RPC)        │ (JSON-RPC)
          ▼                   ▼
   ┌──────────────┐    ┌──────────────┐
   │ MCPサーバー(A)│    │ MCPサーバー(B)│
   │  (GitHub)    │    │ (PostgreSQL) │
   └──────┬───────┘    └──────┬───────┘
          │                   │
          ▼                   ▼
   ┌──────────────┐    ┌──────────────┐
   │  GitHub API  │    │ PostgreSQL DB│
   └──────────────┘    └──────────────┘
```

### 各コンポーネントの役割

| コンポーネント | 役割 | 具体例 |
|---|---|---|
| ホスト（Host） | ユーザーが操作するAIアプリケーション本体。MCPクライアントを内包する | Claude Desktop、Cursor、VS Code |
| クライアント（Client） | ホスト内部に存在し、MCPサーバーとの1対1接続を管理する | ホストが自動的に管理（通常は意識しない） |
| サーバー（Server） | 外部ツールやデータソースへのアクセスを提供する軽量プログラム | filesystem、github、postgres等 |

### 1対1の接続モデル

重要なポイントとして、MCPクライアントとMCPサーバーは**1対1**で接続されます。1つのホストが複数のサーバーに接続する場合、ホスト内部に複数のクライアントインスタンスが作られます。

```
1つのホストが3つのサーバーに接続する場合：

  ホスト（Claude Desktop）
  ├── クライアント(1) ←→ サーバー(GitHub)
  ├── クライアント(2) ←→ サーバー(filesystem)
  └── クライアント(3) ←→ サーバー(postgres)
```

### ポイントまとめ
- MCPはホスト・クライアント・サーバーの3層構造
- ホストはユーザーが使うAIアプリ、サーバーは外部ツールへの橋渡し
- クライアントとサーバーは常に1対1で接続される

## 1.4 MCPの主要機能

MCPは3つの主要機能（プリミティブ: primitive）を提供します。

### 3つの柱

```
┌────────────────────────────────────────────┐
│           MCP の 3 つの柱                    │
├──────────────┬─────────────┬───────────────┤
│   ツール     │  リソース    │  プロンプト    │
│  (Tools)     │ (Resources) │  (Prompts)    │
├──────────────┼─────────────┼───────────────┤
│ LLMが呼び出す│ LLMが読み取る│ ユーザーが選ぶ │
│ 関数・操作   │ データ      │ テンプレート   │
├──────────────┼─────────────┼───────────────┤
│ 例:          │ 例:         │ 例:           │
│ ・ファイル作成│ ・ファイル内容│ ・コードレビュー│
│ ・DB検索     │ ・DB内のデータ│   依頼テンプレ│
│ ・API呼び出し│ ・設定情報   │ ・要約テンプレ │
└──────────────┴─────────────┴───────────────┘
```

### ツール（Tools）

ツールは、**LLMが判断して呼び出す操作**です。サーバーが「こういう操作ができますよ」と公開し、LLMが必要に応じて呼び出します。

```json
{
  "name": "read_file",
  "description": "ファイルの内容を読み取る",
  "inputSchema": {
    "type": "object",
    "properties": {
      "path": {
        "type": "string",
        "description": "読み取るファイルのパス"
      }
    },
    "required": ["path"]
  }
}
```

### リソース（Resources）

リソースは、**LLMが読み取れるデータ**です。URI（Uniform Resource Identifier）で識別され、ファイルの内容やデータベースのレコードなどを表現します。

```
リソースの例：
  file:///home/user/document.txt  → ファイルの内容
  postgres://localhost/mydb/users  → usersテーブルのデータ
  github://repo/issues             → Issueの一覧
```

### プロンプト（Prompts）

プロンプトは、**ユーザーが選択できる定型テンプレート**です。よく使うワークフローをテンプレート化して再利用できます。

```json
{
  "name": "code_review",
  "description": "コードレビューを依頼するプロンプト",
  "arguments": [
    {
      "name": "code",
      "description": "レビュー対象のコード",
      "required": true
    }
  ]
}
```

### 3つの機能の比較

| 特性 | ツール（Tools） | リソース（Resources） | プロンプト（Prompts） |
|---|---|---|---|
| 制御主体 | LLM（モデル） | アプリケーション | ユーザー |
| 方向 | クライアント→サーバー | サーバー→クライアント | サーバー→クライアント |
| 用途 | 操作の実行 | データの読み取り | テンプレートの提供 |
| 例え | 「道具箱の工具」 | 「本棚の本」 | 「レシピ集」 |

### ポイントまとめ
- ツール = LLMが呼び出す操作（関数のようなもの）
- リソース = LLMが読み取るデータ（URIで識別）
- プロンプト = ユーザーが選ぶテンプレート（定型ワークフロー）

| クライアント | 開発元 | 種別 | MCP対応状況 |
|---|---|---|---|
| Claude Desktop | Anthropic | デスクトップアプリ | 完全対応 |
| Claude Code | Anthropic | CLIツール | 完全対応 |

### 公式・コミュニティMCPサーバー

MCPサーバーは、公式リファレンス実装とコミュニティ製の両方が存在します。

**公式リファレンスサーバー（Anthropic提供）：**

| サーバー名 | 機能 |
|---|---|
| filesystem | ローカルファイルの読み書き |
| github | GitHubリポジトリの操作 |
| postgres | PostgreSQLデータベースへの接続 |

**コミュニティサーバーの例：**

| サーバー名 | 機能 |
|---|---|
| docker | Dockerコンテナの管理 |
| sqlite | SQLiteデータベースの操作 |
| brave-search | Brave検索エンジンとの連携 |
| notion | Notionページの操作 |

### よくある間違い

> **誤解：MCPサーバーはクラウド上で動くサービスである**
>
> 実際には、多くのMCPサーバーは**ローカルマシン上で動く軽量プログラム**です。Claude Desktopなどのホストアプリが、必要に応じてローカルでMCPサーバープロセスを起動します。クラウド上のサーバーに接続することも可能ですが、デフォルトはローカル実行です。

### ポイントまとめ
- Claude Desktop、Cursor、VS Codeなど主要AIツールがMCPに対応済み
- 公式リファレンスサーバーとコミュニティサーバーが多数公開されている
- MCPサーバーの多くはローカルマシンで動作する軽量プログラム

## 1.6 MCPの位置づけ

### Function Callingとの関係

関数呼び出し（Function Calling）は、LLMが外部関数を呼び出す仕組みです。MCPのツールとFunction Callingは似ていますが、レイヤーが異なります。

| 比較項目 | Function Calling | MCP |
|---|---|---|
| レイヤー | LLM APIの機能 | アプリケーション間のプロトコル |
| 定義場所 | APIリクエストのパラメータ | MCPサーバーが公開 |
| 実行者 | アプリケーション側のコード | MCPサーバー |
| ツール配布 | なし（アプリ内に埋め込み） | MCPサーバーとして配布可能 |
| 相互運用性 | 特定のLLM APIに依存 | LLM非依存の標準規格 |

```
Function Calling：
  アプリ → LLM API（ツール定義を含む） → LLM → ツール呼び出し指示
  → アプリが自分で関数を実行 → 結果をLLMに返す

MCP：
  ホスト → LLM → ツール呼び出し指示
  → MCPクライアント → MCPサーバーが実行 → 結果を返す
```

MCPとFunction Callingは**排他的ではありません**。MCPの内部ではFunction Callingの仕組みを活用しつつ、ツールの定義・配布・実行を標準化しています。

### LangChain Toolsとの違い

| 比較項目 | LangChain Tools | MCP |
|---|---|---|
| 性質 | Pythonライブラリ | 言語非依存のプロトコル |
| 実行形態 | アプリ内で直接実行 | 別プロセスのサーバーで実行 |
| 言語サポート | Python中心 | 任意の言語で実装可能 |
| ツール共有 | Pythonパッケージ | MCPサーバーとして配布 |
| セキュリティ | アプリと同一プロセス | プロセス分離による隔離 |

### OpenAI Agents SDKとの比較

OpenAIも独自のエージェントフレームワーク（Agents SDK）を提供していますが、2025年3月にMCPサポートを追加しました。これはMCPが業界標準として広く受け入れられつつある証拠です。

### ポイントまとめ
- MCPはFunction Callingを置き換えるものではなく、より上位のプロトコル
- LangChain ToolsはPythonライブラリ、MCPは言語非依存のプロトコル
- OpenAI Agents SDKもMCPをサポートしており、業界標準化が進んでいる

## 1.7 はじめてのMCP体験

### Claude DesktopでMCPサーバーを使ってみる

最も手軽にMCPを体験する方法は、Claude Desktopでファイルシステムサーバーを使うことです。

### 前提条件

- Claude Desktopがインストール済みであること
- Node.js（v18以上）がインストール済みであること

### 手順1：設定ファイルを開く

Claude Desktopの設定ファイル（`claude_desktop_config.json`）を編集します。

**ファイルの場所：**

| OS | パス |
|---|---|
| macOS | `~/Library/Application Support/Claude/claude_desktop_config.json` |
| Windows | `%APPDATA%\Claude\claude_desktop_config.json` |

### 手順2：MCPサーバーを設定する

設定ファイルに以下の内容を記述します。

```json
{
  "mcpServers": {
    "filesystem": {
      "command": "npx",
      "args": [
        "-y",
        "@modelcontextprotocol/server-filesystem",
        "/Users/yourname/Desktop"
      ]
    }
  }
}
```

この設定は「filesystemという名前のMCPサーバーを、npxコマンドで起動し、デスクトップフォルダへのアクセスを許可する」という意味です。

### 手順3：Claude Desktopを再起動する

設定ファイルを保存したら、Claude Desktopを再起動します。チャット入力欄にハンマーアイコン（工具マーク）が表示されれば、MCPサーバーの接続に成功しています。

### 手順4：MCPサーバーを使ってみる

Claude Desktopで以下のように話しかけてみましょう。

```
「デスクトップにあるファイルの一覧を教えてください」
```

Claudeがファイルシステムサーバーのツールを呼び出し、デスクトップのファイル一覧を返してくれます。初回はツール使用の許可を求めるダイアログ（dialog）が表示されますので、「許可」を選択してください。

### よくある間違い

> **誤解：MCPサーバーは自分でビルド・起動しなければならない**
>
> `npx` を使えば、npmに公開されたMCPサーバーをビルドなしで即座に利用できます。上記の設定例では、Claude Desktopが自動的に `npx` コマンドを実行してサーバーを起動します。

> **誤解：設定ファイルを変更したらすぐに反映される**
>
> Claude Desktopの設定変更は、**アプリケーションの再起動が必要**です。設定を変更したら必ず再起動してください。

### ポイントまとめ
- Claude DesktopのJSON設定ファイルにMCPサーバーを追加するだけで利用開始できる
- `npx` を使えばビルド不要で公開サーバーを即利用可能
- 設定変更後はClaude Desktopの再起動を忘れずに行う

## 第1章のまとめ

本章では、MCPの全体像を学びました。

1. **MCPとは** - AIアプリと外部ツールをつなぐオープンな共通規格
2. **解決する問題** - N×M問題をN+Mに削減し、統合コストを大幅に低減
3. **3層構造** - ホスト・クライアント・サーバーの明確な役割分担
4. **3つの柱** - ツール（操作）、リソース（データ）、プロンプト（テンプレート）
5. **エコシステム** - 主要AIツールが対応済み、豊富なサーバーが公開中
6. **位置づけ** - Function Callingの上位レイヤー、業界標準として定着しつつある
7. **実践** - Claude Desktopで数分でMCP体験が可能

次章では、MCPのアーキテクチャについて、プロトコルの内部構造やメッセージフローを詳しく学びます。

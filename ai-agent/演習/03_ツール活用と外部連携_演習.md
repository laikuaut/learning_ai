### 問題1：Function Callingの基本フロー（基本）

以下の選択肢から、Function Calling の正しい処理フローを並べ替えてください。

```
A. LLMがツール呼び出しのJSONを返す
B. ユーザーがプロンプトを送信する
C. アプリケーションが実際のAPIを呼び出し、結果を取得する
D. LLMがツールの実行結果を踏まえて最終回答を生成する
E. アプリケーションがツール定義とプロンプトをLLMに送る
F. ツール実行結果をLLMに返す
```

**期待される出力例：**
```
正しい順序: B → E → A → C → F → D

1. ユーザーがプロンプトを送信する
2. アプリケーションがツール定義とプロンプトをLLMに送る
3. LLMがツール呼び出しのJSONを返す
4. アプリケーションが実際のAPIを呼び出し、結果を取得する
5. ツール実行結果をLLMに返す
6. LLMがツールの実行結果を踏まえて最終回答を生成する
```

<details>
<summary>ヒント</summary>

- Function Calling では、LLM自身がAPIを呼ぶわけではありません
- LLMは「どのツールをどんな引数で呼ぶべきか」を判断するだけです
- 実際のAPI呼び出しはアプリケーション側が行います

</details>

<details>
<summary>解答例</summary>

```
正しい順序: B → E → A → C → F → D

解説：
Function Calling はLLMとアプリケーションの協調動作です。

1. B: ユーザーが「東京の天気を教えて」と送信
2. E: アプリがツール定義（get_weather等）と共にLLMへ送信
3. A: LLMが {"name": "get_weather", "arguments": {"city": "東京"}} を返す
4. C: アプリが実際の天気APIを呼び出して結果を取得
5. F: API結果（{"temp": 22, "condition": "晴れ"}）をLLMに返す
6. D: LLMが「東京は現在22℃で晴れです」と自然言語で回答

重要なポイント：
- LLMはツールの「選択」と「引数の構造化」を担当
- 実際の外部通信はアプリケーション側の責務
- このアーキテクチャによりセキュリティと制御性を確保できます
```

</details>

### 問題2：JSON Schemaによるツール定義の作成（基本）

「指定された都市の現在の天気情報を取得する」ツールを、以下の仕様に基づいてJSON Schemaで定義してください。

**仕様：**
- ツール名：`get_current_weather`
- 説明：指定された都市の現在の天気情報を取得します
- パラメータ：
  - `city`（string、必須）：天気を取得する都市名
  - `unit`（string、任意）：温度の単位。"celsius" または "fahrenheit"。デフォルトは "celsius"

**期待される出力例：**
```json
{
  "name": "get_current_weather",
  "description": "指定された都市の現在の天気情報を取得します",
  "input_schema": {
    "type": "object",
    "properties": {
      "city": {
        "type": "string",
        "description": "天気を取得する都市名（例：東京、大阪）"
      },
      "unit": {
        "type": "string",
        "enum": ["celsius", "fahrenheit"],
        "description": "温度の単位。デフォルトはcelsius"
      }
    },
    "required": ["city"]
  }
}
```

<details>
<summary>ヒント</summary>

- Anthropic APIのツール定義形式では `input_schema` を使います
- `required` 配列には必須パラメータのみを含めます
- `enum` を使うと取りうる値を限定できます

</details>

<details>
<summary>解答例</summary>

```json
{
  "name": "get_current_weather",
  "description": "指定された都市の現在の天気情報を取得します",
  "input_schema": {
    "type": "object",
    "properties": {
      "city": {
        "type": "string",
        "description": "天気を取得する都市名（例：東京、大阪、ニューヨーク）"
      },
      "unit": {
        "type": "string",
        "enum": ["celsius", "fahrenheit"],
        "description": "温度の単位。デフォルトはcelsius"
      }
    },
    "required": ["city"]
  }
}
```

```
解説：
- name: ツールを一意に識別する名前。スネークケースが一般的です
- description: LLMがツールの用途を理解するための説明文。
  明確で具体的に書くことで、LLMが適切にツールを選択できます
- input_schema: JSON Schema形式でパラメータを定義します
  - type: "object" でパラメータ全体をオブジェクトとして定義
  - properties: 各パラメータの名前・型・説明を定義
  - required: 必須パラメータの配列。unitは任意なので含めません
- enum: unitの値を "celsius" と "fahrenheit" に限定しています
  これによりLLMが不正な値を生成するリスクを低減できます
```

</details>

### 問題3：MCPアーキテクチャの理解（基本）

MCP（Model Context Protocol）について、以下の空欄を埋めてください。

```
MCPは（ ① ）社が提唱するオープンプロトコルで、
LLMアプリケーションと外部ツール・データソースの接続を（ ② ）化します。

MCPの3つの主要コンポーネント：
- MCPホスト：（ ③ ）
- MCPクライアント：（ ④ ）
- MCPサーバー：（ ⑤ ）

MCPが提供する3つの主要機能（プリミティブ）：
- Tools：（ ⑥ ）
- Resources：（ ⑦ ）
- Prompts：（ ⑧ ）

MCPの通信方式：
- ローカル通信：（ ⑨ ）
- リモート通信：（ ⑩ ）
```

**期待される出力例：**
```
① Anthropic
② 標準
③ Claude DesktopやIDEなど、ユーザーが直接操作するアプリケーション
④ ホストとサーバーの間の通信を仲介するコンポーネント
⑤ 外部ツールやデータソースを公開するサーバー
⑥ LLMが呼び出せるアクション（関数）を提供する
⑦ コンテキストとしてデータやファイルを提供する
⑧ 再利用可能なプロンプトテンプレートを提供する
⑨ stdio（標準入出力）
⑩ HTTP + SSE（Server-Sent Events）/ Streamable HTTP
```

<details>
<summary>ヒント</summary>

- MCPは「AIアプリケーション版のUSB-C」と例えられることがあります
- ホスト・クライアント・サーバーの3層構造を思い出しましょう
- 通信方式はローカルとリモートで異なります

</details>

<details>
<summary>解答例</summary>

```
① Anthropic
  → MCPはAnthropic社が2024年11月に発表したオープンプロトコルです

② 標準
  → 各ツールごとに個別の連携方法を実装する代わりに、
    統一されたプロトコルで接続できます

③ Claude DesktopやIDEなど、ユーザーが直接操作するアプリケーション
  → ホストは複数のMCPクライアントを管理します

④ ホストとサーバーの間の通信を仲介するコンポーネント
  → 1つのクライアントが1つのサーバーと1対1で接続します

⑤ 外部ツールやデータソースを公開するサーバー
  → データベース、API、ファイルシステムなどへのアクセスを提供します

⑥ LLMが呼び出せるアクション（関数）を提供する
  → Function Callingと同様に、LLMが判断してツールを実行できます

⑦ コンテキストとしてデータやファイルを提供する
  → ファイルの内容やDBのレコードなどを読み取り専用で提供します

⑧ 再利用可能なプロンプトテンプレートを提供する
  → ユーザーが選択できるスラッシュコマンドのような形で提供されます

⑨ stdio（標準入出力）
  → ローカルプロセスとして起動し、stdin/stdoutで通信します

⑩ HTTP + SSE（Server-Sent Events）/ Streamable HTTP
  → リモートサーバーとHTTPベースで通信します
```

</details>

### 問題4：ツール定義のベストプラクティス（基本）

以下のツール定義には改善すべき点が複数あります。問題点を指摘し、改善版を作成してください。

```json
{
  "name": "do_stuff",
  "description": "いろいろやる",
  "input_schema": {
    "type": "object",
    "properties": {
      "data": {
        "type": "string"
      }
    }
  }
}
```

**期待される出力例：**
```
問題点：
1. ツール名が曖昧（"do_stuff"では何をするか不明）
2. descriptionが不明確（"いろいろやる"ではLLMが適切に選択できない）
3. パラメータの説明（description）がない
4. 必須パラメータが定義されていない（required がない）
5. パラメータのデータ型が適切か不明（stringで十分か？）

改善版：（具体的なユースケースを想定して改善してください）
```

<details>
<summary>ヒント</summary>

- ツール名は何をするかが一目でわかる名前にしましょう
- descriptionはLLMがツールの使いどころを判断する重要な情報です
- 各パラメータにもdescriptionを付けましょう

</details>

<details>
<summary>解答例</summary>

```
問題点：
1. ツール名 "do_stuff" → 何をするツールか全く不明
2. description "いろいろやる" → LLMがいつこのツールを使うべきか判断できない
3. パラメータ "data" にdescriptionがない → 何を渡せばよいか不明
4. required が未定義 → 必須/任意の区別がつかない
5. パラメータが "data" という汎用名 → 具体的な意味が不明

改善版（例：メール送信ツールとして再設計）：
```

```json
{
  "name": "send_email",
  "description": "指定された宛先にメールを送信します。件名と本文を指定できます。送信完了後、メッセージIDを返します。",
  "input_schema": {
    "type": "object",
    "properties": {
      "to": {
        "type": "string",
        "description": "送信先メールアドレス（例：user@example.com）"
      },
      "subject": {
        "type": "string",
        "description": "メールの件名"
      },
      "body": {
        "type": "string",
        "description": "メール本文（プレーンテキスト）"
      },
      "cc": {
        "type": "string",
        "description": "CCの宛先メールアドレス（任意）"
      }
    },
    "required": ["to", "subject", "body"]
  }
}
```

```
改善ポイントまとめ：
- name: 動詞+名詞の形式で具体的に（send_email）
- description: 何をするか、何を返すかを明記
- properties: 各パラメータに具体的な名前とdescriptionを付与
- required: 必須パラメータを明示（ccは任意なので含めない）
- description内の例示: LLMが正しい形式を理解しやすくなります
```

</details>

### 問題5：実用的なツールセットの設計（応用）

あなたは「パーソナルアシスタントAIエージェント」を開発しています。以下の3つのツールをJSON Schema形式で設計してください。

**要件：**
1. `get_weather` - 天気情報の取得（都市名、日付指定可能）
2. `manage_calendar` - カレンダーの予定管理（作成・取得・削除）
3. `send_notification` - ユーザーへの通知送信（メール/Slack/LINE）

各ツールで以下を考慮してください：
- パラメータの型と制約（enum, formatなど）
- 必須/任意の区別
- LLMが正しく使えるようなdescription

**期待される出力例：**
```json
[
  {
    "name": "get_weather",
    "description": "指定した都市の天気予報を取得します...",
    "input_schema": {
      "type": "object",
      "properties": {
        "city": { "type": "string", "description": "..." },
        "date": { "type": "string", "description": "..." }
      },
      "required": ["city"]
    }
  },
  ...
]
```

<details>
<summary>ヒント</summary>

- `manage_calendar` は `action` パラメータで操作を切り替える設計が便利です
- 日付は ISO 8601 形式（YYYY-MM-DD）を指定すると明確です
- `send_notification` は送信先チャネルを enum で限定しましょう

</details>

<details>
<summary>解答例</summary>

```json
[
  {
    "name": "get_weather",
    "description": "指定した都市の天気予報を取得します。当日を含む最大7日先までの予報を取得できます。気温・天候・降水確率を返します。",
    "input_schema": {
      "type": "object",
      "properties": {
        "city": {
          "type": "string",
          "description": "天気を取得する都市名（例：東京、大阪、札幌）"
        },
        "date": {
          "type": "string",
          "description": "予報を取得する日付（ISO 8601形式: YYYY-MM-DD）。省略時は本日の天気を返します"
        },
        "detail_level": {
          "type": "string",
          "enum": ["summary", "hourly"],
          "description": "取得する詳細度。summaryは日単位、hourlyは1時間単位。デフォルトはsummary"
        }
      },
      "required": ["city"]
    }
  },
  {
    "name": "manage_calendar",
    "description": "Googleカレンダーの予定を管理します。予定の作成・取得・削除が可能です。",
    "input_schema": {
      "type": "object",
      "properties": {
        "action": {
          "type": "string",
          "enum": ["create", "get", "delete"],
          "description": "実行する操作の種類"
        },
        "title": {
          "type": "string",
          "description": "予定のタイトル（createの場合は必須）"
        },
        "start_datetime": {
          "type": "string",
          "description": "開始日時（ISO 8601形式: YYYY-MM-DDTHH:MM:SS）。getの場合は検索範囲の開始"
        },
        "end_datetime": {
          "type": "string",
          "description": "終了日時（ISO 8601形式: YYYY-MM-DDTHH:MM:SS）。getの場合は検索範囲の終了"
        },
        "event_id": {
          "type": "string",
          "description": "予定のID（deleteの場合は必須）"
        },
        "location": {
          "type": "string",
          "description": "予定の場所（任意）"
        }
      },
      "required": ["action"]
    }
  },
  {
    "name": "send_notification",
    "description": "ユーザーに通知を送信します。メール・Slack・LINEから送信チャネルを選択できます。",
    "input_schema": {
      "type": "object",
      "properties": {
        "channel": {
          "type": "string",
          "enum": ["email", "slack", "line"],
          "description": "通知の送信先チャネル"
        },
        "message": {
          "type": "string",
          "description": "送信するメッセージ本文"
        },
        "subject": {
          "type": "string",

        },

          "type": "string",
          "enum": ["low", "normal", "high"],
          "description": "通知の優先度。デフォルトはnormal"
        }
      },
      "required": ["channel", "message"]
    }
  }

```

```
設計のポイント：
1. get_weather:
   - dateを任意にすることで「今日の天気」にも対応

   - 操作ごとに必要なパラメータが異なる点をdescriptionで補足
   - ISO 8601形式を明示してLLMの出力を安定化

3. send_notification:
   - channelをenumで限定し不正な値を防止
   - チャネル固有のパラメータ（subject）の挙動をdescriptionで説明

実務Tips:
- 1つのツールに多機能を詰め込みすぎると、LLMの選択精度が下がります
- 逆に細かく分けすぎるとツール数が増え、コンテキストを圧迫します
- 5〜15個程度のツール数が実用的な目安です
```

</details>

### 問題6：MCPサーバーの設計（応用）

「社内ナレッジベース」にアクセスするMCPサーバーを設計してください。以下の機能を持つMCPサーバーの仕様を、Tools・Resources・Promptsの3つのプリミティブに分けて記述してください。

**要件：**
- 社内ドキュメント（Confluence相当）の検索・閲覧
- FAQデータベースへのアクセス
- ドキュメント更新履歴の確認
- よく使う質問パターンのテンプレート

**期待される出力例：**
```
## Tools（LLMが実行できるアクション）
1. search_documents: ...
2. get_faq: ...
3. ...

## Resources（読み取り可能なデータ）
1. knowledge://documents/{id}: ...
2. ...

## Prompts（プロンプトテンプレート）
1. troubleshoot: ...
2. ...
```

<details>
<summary>ヒント</summary>

- Toolsは「副作用のあるアクション」や「動的な検索」に使います
- Resourcesは「静的・半静的なデータの読み取り」に使います
- Promptsは「よく使うプロンプトパターンのテンプレート化」です
- URIスキームは自由に設計できます（例：`knowledge://`）

</details>

<details>
<summary>解答例</summary>

```
## Tools（LLMが実行できるアクション）

1. search_documents
   - 説明: キーワードやフィルタ条件でドキュメントを全文検索します
   - パラメータ:
     - query (string, 必須): 検索キーワード
     - category (string, 任意): カテゴリでフィルタ（例: "開発", "人事", "営業"）
     - date_from (string, 任意): 検索対象の開始日
     - limit (integer, 任意): 取得件数上限（デフォルト: 10）
   - 戻り値: ドキュメントID、タイトル、スニペット、更新日のリスト

2. get_faq_answer
   - 説明: FAQデータベースから質問に最も近いQ&Aを検索します
   - パラメータ:
     - question (string, 必須): 質問文
     - department (string, 任意): 部署でフィルタ
   - 戻り値: 質問、回答、関連度スコアのリスト

3. get_document_history
   - 説明: 指定ドキュメントの更新履歴を取得します
   - パラメータ:
     - document_id (string, 必須): ドキュメントID
     - limit (integer, 任意): 取得する履歴件数（デフォルト: 5）
   - 戻り値: 更新日時、更新者、変更サマリのリスト

4. update_faq
   - 説明: FAQに新しいQ&Aを追加または既存のQ&Aを更新します
   - パラメータ:
     - action (string, 必須): "create" または "update"
     - question (string, 必須): 質問文
     - answer (string, 必須): 回答文
     - faq_id (string, 任意): 更新時の既存FAQ ID

## Resources（読み取り可能なデータ）

1. knowledge://documents/{document_id}
   - 説明: 特定のドキュメントの全文を取得
   - MIMEタイプ: text/markdown

2. knowledge://documents/{document_id}/metadata
   - 説明: ドキュメントのメタデータ（作成者、更新日、タグ等）
   - MIMEタイプ: application/json

3. knowledge://categories
   - 説明: 利用可能なドキュメントカテゴリの一覧
   - MIMEタイプ: application/json

4. knowledge://faq/popular
   - 説明: よく参照されるFAQ上位20件のリスト
   - MIMEタイプ: application/json

5. knowledge://stats/recent-updates
   - 説明: 直近1週間に更新されたドキュメントのサマリ
   - MIMEタイプ: application/json

## Prompts（プロンプトテンプレート）

1. troubleshoot
   - 説明: 社内システムのトラブルシューティングガイド
   - 引数: system_name (string) - 対象システム名
   - テンプレート: 「{system_name}で発生している問題について、
     FAQと過去のドキュメントを検索し、解決手順を提示してください」

2. onboarding_guide
   - 説明: 新入社員向けの情報案内
   - 引数: department (string) - 配属部署
   - テンプレート: 「{department}に配属された新入社員向けに、
     必読ドキュメントとよくある質問をまとめてください」

3. weekly_summary
   - 説明: 週次ナレッジ更新サマリの生成
   - 引数: なし
   - テンプレート: 「直近1週間のドキュメント更新を確認し、
     重要な変更点をチーム共有用にまとめてください」
```

```
設計のポイント：
- Tools vs Resources の使い分け:
  - 検索のように動的なパラメータが必要 → Tools
  - IDやパスで特定できる静的データ → Resources
- Promptsはユーザーが頻繁に行う操作パターンをテンプレート化
- URI設計はRESTfulな慣習に沿うと直感的です
- MCPサーバーは1つの責務に集中させるのがベストプラクティスです
```

### 問題7：ツールの安全性設計（応用）

AIエージェントにツールを持たせる際の安全性について、以下のシナリオそれぞれに対して、どのようなガードレールを設計すべきか記述してください。

**シナリオ：**
1. ファイル削除ツール（`delete_file`）をエージェントに持たせる場合
2. メール送信ツール（`send_email`）をエージェントに持たせる場合
3. データベース操作ツール（`execute_sql`）をエージェントに持たせる場合

**期待される出力例：**
```
シナリオ1: delete_file のガードレール
- 確認メカニズム: ...
- スコープ制限: ...
- ログ: ...

シナリオ2: send_email のガードレール
...

シナリオ3: execute_sql のガードレール
...
```

<details>
<summary>ヒント</summary>

- Human-in-the-loop（人間の承認）をどこに入れるか考えましょう
- 操作のスコープ（範囲）を制限する方法を考えましょう
- 取り消し可能性（reversibility）も重要な観点です
- 「最小権限の原則」を適用しましょう

</details>

<details>
<summary>解答例</summary>

```
シナリオ1: delete_file のガードレール

- 確認メカニズム:
  - 削除前に必ずユーザーに確認を求める（Human-in-the-loop）
  - 削除対象のファイル名とパスを明示して承認を得る

- スコープ制限:
  - 削除可能なディレクトリをホワイトリストで限定
  - システムファイルや設定ファイルは削除不可にする
  - ワイルドカード指定（*.* など）を禁止する

- 安全策:
  - 即時削除ではなくゴミ箱に移動（ソフトデリート）
  - 一定期間は復元可能な状態を維持

- ログ:
  - 削除したファイルのパス、日時、サイズを記録
  - 誰（どのセッション）が削除指示を出したかを記録

---

シナリオ2: send_email のガードレール

- 確認メカニズム:
  - 送信前に宛先・件名・本文をユーザーにプレビュー表示
  - 外部ドメインへの送信は追加の確認を要求

- スコープ制限:
  - 1回の操作で送信可能な宛先数を制限（例：最大10件）
  - 送信頻度にレートリミットを設定（例：1分間に5通まで）
  - 送信可能なドメインをホワイトリストで管理

- コンテンツ検証:
  - 添付ファイルのサイズ・種類を制限
  - 機密情報（パスワード等）が本文に含まれていないかチェック

- ログ:
  - 送信先、件名、送信日時を記録
  - 送信成功/失敗のステータスを記録

---

シナリオ3: execute_sql のガードレール

- 確認メカニズム:
  - 書き込み系SQL（INSERT/UPDATE/DELETE/DROP）は必ず確認を要求
  - 実行前にSQLをユーザーに表示し承認を得る

- スコープ制限:
  - 読み取り専用の接続を基本とする（SELECT のみ許可）
  - 書き込みが必要な場合は別の制限付き接続を使用
  - DROP TABLE, TRUNCATE 等の破壊的操作は完全に禁止
  - アクセス可能なテーブル/スキーマをホワイトリストで制限

- 実行制御:
  - クエリのタイムアウトを設定（例：30秒）
  - 取得行数の上限を設定（例：1000行）
  - トランザクション内で実行し、異常時はロールバック

- ログ:
  - 実行したSQL文、実行時間、影響行数を記録
  - エラー発生時のスタックトレースを記録
```

```
共通の設計原則：
1. 最小権限の原則: 必要最小限の権限のみ付与する
2. Human-in-the-loop: 破壊的・不可逆な操作には人間の承認を必須にする
3. 監査ログ: すべてのツール実行を記録する
4. レートリミット: 短時間での大量実行を防ぐ
5. フェイルセーフ: エラー時は安全側に倒す（操作を中止する）
```

### 問題8：Anthropic APIでFunction Callingを実装する（チャレンジ）

Anthropic APIを使って、以下のツールを持つ簡単なエージェントをPythonで実装してください。

**要件：**
- `calculate` ツール：四則演算を実行する（式を受け取り結果を返す）
- `get_current_time` ツール：現在の日時を返す（タイムゾーン指定可能）
- ユーザーの質問に応じてツールを自動選択し、結果を返す
- ツール呼び出しのループ処理を含む（1回のやり取りで複数ツール呼び出し可能）

**期待される出力例：**
```
ユーザー: 今何時？あと、123 × 456 の計算もして
アシスタント: [get_current_time ツールを呼び出し]
アシスタント: [calculate ツールを呼び出し]
アシスタント: 現在の時刻は2026年3月26日 15:30:00（JST）です。
また、123 × 456 = 56,088 です。
```

<details>
<summary>ヒント</summary>

- `anthropic` ライブラリの `client.messages.create()` を使います
- レスポンスの `stop_reason` が `"tool_use"` の場合、ツール呼び出しが必要です
- ツール実行結果は `tool_result` としてメッセージ履歴に追加します
- `while` ループで `stop_reason` が `"end_turn"` になるまで繰り返します

</details>

<details>
<summary>解答例</summary>

```python
import anthropic
import json
from datetime import datetime
import pytz  # タイムゾーン処理用（pip install pytz）

# --- ツール定義 ---
tools = [
    {
        "name": "calculate",
        "description": "数学的な計算を実行します。四則演算や累乗などの数式を評価して結果を返します。",
        "input_schema": {
            "type": "object",
            "properties": {
                "expression": {
                    "type": "string",
                    "description": "計算する数式（例: '123 * 456', '2 ** 10', '(100 + 200) / 3'）"
                }
            },
            "required": ["expression"]
        }
    },
    {
        "name": "get_current_time",
        "description": "現在の日時を取得します。タイムゾーンを指定できます。",
        "input_schema": {
            "type": "object",
            "properties": {
                "timezone": {
                    "type": "string",
                    "description": "タイムゾーン名（例: 'Asia/Tokyo', 'UTC', 'US/Eastern'）。デフォルトは'Asia/Tokyo'"
                }
            },
            "required": []
        }
    }
]

# --- ツール実行関数 ---
def execute_tool(tool_name: str, tool_input: dict) -> str:
    """ツール名と引数に基づいて実際の処理を実行します"""

    if tool_name == "calculate":
        expression = tool_input["expression"]
        # セキュリティ: 許可された文字のみ通す
        allowed_chars = set("0123456789+-*/.() ")
        if not all(c in allowed_chars for c in expression):
            return json.dumps({"error": "不正な文字が含まれています"})
        try:
            result = eval(expression)  # 本番ではast.literal_evalやsympyを推奨
            return json.dumps({"result": result, "expression": expression})
        except Exception as e:
            return json.dumps({"error": str(e)})

    elif tool_name == "get_current_time":
        timezone = tool_input.get("timezone", "Asia/Tokyo")
        try:
            tz = pytz.timezone(timezone)
            now = datetime.now(tz)
            return json.dumps({
                "datetime": now.strftime("%Y-%m-%d %H:%M:%S"),
                "timezone": timezone,
                "iso_format": now.isoformat()
            })
        except Exception as e:
            return json.dumps({"error": f"無効なタイムゾーン: {timezone}"})

    return json.dumps({"error": f"未知のツール: {tool_name}"})

# --- エージェントのメインループ ---
def run_agent(user_message: str):
    """ユーザーのメッセージを処理し、必要に応じてツールを使って回答します"""

    client = anthropic.Anthropic()  # ANTHROPIC_API_KEY環境変数を使用

    # メッセージ履歴を初期化
    messages = [
        {"role": "user", "content": user_message}
    ]

    print(f"ユーザー: {user_message}")

    # ツール呼び出しループ
    while True:
        # APIリクエスト
        response = client.messages.create(
            model="claude-sonnet-4-20250514",
            max_tokens=1024,
            tools=tools,
            messages=messages
        )

        # stop_reason が "end_turn" なら最終回答
        if response.stop_reason == "end_turn":
            # テキストブロックを抽出して表示
            for block in response.content:
                if block.type == "text":
                    print(f"アシスタント: {block.text}")
            break

        # stop_reason が "tool_use" ならツール呼び出しを処理
        if response.stop_reason == "tool_use":
            # アシスタントの応答をメッセージ履歴に追加
            messages.append({
                "role": "assistant",
                "content": response.content
            })

            # すべてのツール呼び出しを処理
            tool_results = []
            for block in response.content:
                if block.type == "tool_use":
                    print(f"  [ツール呼び出し] {block.name}({json.dumps(block.input, ensure_ascii=False)})")

                    # ツールを実行
                    result = execute_tool(block.name, block.input)
                    print(f"  [ツール結果] {result}")

                    tool_results.append({
                        "type": "tool_result",
                        "tool_use_id": block.id,
                        "content": result
                    })

            # ツール結果をメッセージ履歴に追加
            messages.append({
                "role": "user",
                "content": tool_results
            })

# --- 実行例 ---
if __name__ == "__main__":
    # 例1: 複数ツールの同時使用
    run_agent("今何時？あと、123 × 456 の計算もして")

    print("\n" + "=" * 50 + "\n")

    # 例2: 計算のみ
    run_agent("(1024 + 2048) * 3 はいくつですか？")

    print("\n" + "=" * 50 + "\n")

    # 例3: ツール不要の質問
    run_agent("Pythonのリスト内包表記について教えて")
```

```
コードの解説：

1. ツール定義（tools配列）:
   - JSON Schema形式でツールのインターフェースを定義
   - LLMがツールの用途と引数を理解するための情報

2. ツール実行関数（execute_tool）:
   - LLMが返したツール名と引数を受け取り、実際の処理を実行
   - calculateではeval()を使用（本番環境ではast.literal_evalやsympyを推奨）
   - セキュリティ対策として入力文字を制限

3. エージェントループ（run_agent）:
   - whileループでstop_reasonを監視
   - "tool_use" → ツールを実行して結果を返す
   - "end_turn" → 最終回答を出力してループ終了
   - 1回のレスポンスに複数のツール呼び出しが含まれる場合もある

4. メッセージ履歴の管理:
   - アシスタントの応答（ツール呼び出し含む）を履歴に追加
   - ツール結果はtool_resultとしてuser roleで追加
   - この履歴があることでLLMは文脈を維持できる

重要な注意点：
- eval()は本番コードでは使用しないでください（セキュリティリスク）
- API キーは環境変数で管理し、コードにハードコードしないこと
- エラーハンドリングを必ず入れること（API障害、ツール実行失敗等）
- ループの最大回数を設定して無限ループを防止すること
```

</details>

```
# 旅行プランニングエージェント 設計書

## 1. ツール一覧

| ツール名 | 種類 | 説明 | 必須パラメータ |
|---|---|---|---|
| search_attractions | 情報取得 | 目的地の観光スポット・レストラン等を検索 | destination, category |
| search_flights | 検索 | フライトの空き状況と料金を検索 | origin, destination, date |
| search_hotels | 検索 | ホテルの空き状況と料金を検索 | destination, check_in, check_out |
| get_weather_forecast | 情報取得 | 目的地の天気予報を取得（最大14日先） | city, start_date, end_date |
| get_travel_history | 参照 | ユーザーの過去の旅行履歴を取得 | user_id |
| calculate_budget | 計算 | フライト+ホテル+アクティビティの合計を計算 | items (配列) |
| create_itinerary | 生成 | 日程表を作成・更新 | destination, dates, activities |
| book_reservation | 予約 | フライトやホテルの仮予約を作成 | type, reservation_id |

## 2. ツール連携フロー

### フロー1: 新規旅行プラン作成
ユーザー: 「来月、3泊4日で沖縄旅行を計画して。予算15万円以内で」

1. get_travel_history → 過去の旅行傾向を把握
   （例: ビーチが好き、高級ホテルより民宿派）

2. 並列実行:
   - search_flights(origin="東京", destination="沖縄", date="2026-04-15")
   - search_hotels(destination="沖縄", check_in="2026-04-15", check_out="2026-04-18")
   - get_weather_forecast(city="那覇", start_date="2026-04-15", end_date="2026-04-18")

3. search_attractions(destination="沖縄", category="all")
   → 天気予報を考慮してアウトドア/インドア活動を振り分け

4. calculate_budget(items=[フライト, ホテル, アクティビティ])
   → 予算15万円に収まるか確認

5. create_itinerary(...)
   → 日程表を生成してユーザーに提示

6. （ユーザー承認後）book_reservation(...)
   → 仮予約を実行

### フロー2: 天気による動的調整
- 雨予報の日 → インドア観光（水族館、美術館等）を優先
- 晴れ予報の日 → ビーチ、シュノーケリング等を配置
- 台風接近 → 日程変更を提案し、フライト再検索

## 3. システムプロンプト設計

「あなたは旅行プランニングの専門家です。以下のルールに従ってください：
- ユーザーの予算を常に尊重し、予算超過時は代替案を提示する
- 天気予報を必ず確認し、悪天候時はインドアの代替を用意する
- 過去の旅行履歴を参考に、ユーザーの好みに合わせた提案をする
- 予約（book_reservation）は必ずユーザーの明示的な承認後に実行する
- 移動時間を考慮し、無理のないスケジュールを組む」

## 4. エラーハンドリング戦略

| エラー種別 | 対処方法 |
|---|---|
| フライトAPI障害 | キャッシュされた直近の検索結果を表示 + 「最新情報は取得できませんでした」と注記 |
| ホテル検索結果0件 | 日程を前後1日ずらして再検索 / エリアを拡大して再検索 |
| 予算超過 | 安い代替案を自動検索し、比較表を提示 |
| 天気API障害 | 過去の気候統計データにフォールバック |
| 予約失敗 | エラー原因を表示し、代替オプションを提案 |

## 5. セキュリティ考慮事項

- book_reservation は必ず Human-in-the-loop を適用
- ユーザーの旅行履歴はセッション終了時にメモリから削除
- 支払い情報はツール側で処理し、LLMには渡さない
- APIキーはすべてサーバーサイドで管理
```

```
設計のポイント：
1. ツールの粒度: 1つのツールに1つの責務を持たせています
2. 並列実行: 独立した検索は同時に実行して応答速度を向上
3. 段階的提案: 情報収集 → 計算 → 提案 → 承認 → 実行の流れ
4. Human-in-the-loop: 予約など不可逆な操作には必ず承認を挟む
5. フォールバック: API障害時にも最低限の価値を提供できる設計
```

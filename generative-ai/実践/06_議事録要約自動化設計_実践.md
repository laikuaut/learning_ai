# 実践課題06：議事録要約自動化設計 ★3

> **難易度**: ★★★☆☆（中級）
> **前提知識**: 第2章（テキスト生成AIの活用）、第6章（業務プロセスへの統合）
> **課題の種類**: 設計課題
> **学習目標**: 長文の議事録をAIで要約する業務フロー（Business Workflow）を設計し、要約の品質基準と運用ルールを策定する力を身につける

## 課題の要件

1. 以下の業務フロー全体を設計する
   - 入力：会議の議事録テキスト（3,000〜10,000字）
   - 処理：前処理 → AI要約 → 品質チェック
   - 出力：3種類の要約（エグゼクティブサマリー、詳細要約、アクションアイテム一覧）
2. 議事録の前処理ルール（話者の分離、セクション分割、不要な発言の除去）を定義する
3. 3種類の要約それぞれに対して、専用のプロンプトを設計する
4. 要約結果の品質基準（チェックリスト）を作成する
5. サンプル議事録（1,000字程度）を用意し、実際に要約を実行する
6. 運用時の注意事項（機密情報の取り扱い等）をまとめる

## ステップガイド

<details>
<summary>ステップ1：サンプル議事録を作成する</summary>

以下のような会議をイメージしてサンプル議事録を作成しましょう。

**会議情報の例：**
- 会議名：第3四半期 新規プロジェクト企画会議
- 日時：2026年4月3日 14:00〜15:30
- 参加者：部長A、課長B、担当C、担当D
- 議題：新サービスの企画案の検討

1,000字程度で、以下の要素を含む議事録を作成します。
- 複数の話者の発言
- 決定事項と未決事項
- アクションアイテム（担当者・期限付き）
- 脱線した雑談部分（前処理で除去する対象）

</details>

<details>
<summary>ステップ2：前処理ルールを定義する</summary>

生の議事録をAIに渡す前に行う前処理を定義しましょう。

1. **話者ラベルの統一**: 「田中さん」「田中部長」「A部長」→「田中部長」に統一
2. **セクション分割**: 議題ごとに区切りを入れる
3. **不要部分の除去**: 雑談、重複する相槌、技術的なノイズ（「えーと」「あの」等）
4. **タイムスタンプの整理**: 必要に応じて追加

**よくある間違い**: 前処理を省略してAIに生テキストをそのまま渡すこと。ノイズが多いと要約品質が大幅に低下します。

</details>

<details>
<summary>ステップ3：3種類の要約プロンプトを設計する</summary>

**種類1：エグゼクティブサマリー（Executive Summary）**
- 対象読者：経営層、会議に参加していない上長
- 分量：3〜5行
- 含める情報：主要な決定事項と全体の方向性

**種類2：詳細要約**
- 対象読者：会議参加者の振り返り、関係部署への共有
- 分量：元の議事録の1/3〜1/4程度
- 含める情報：議題ごとの議論の要点、賛否、結論

**種類3：アクションアイテム一覧**
- 対象読者：タスク管理担当、各アクションの担当者
- 形式：表形式（担当者、タスク内容、期限、ステータス）

</details>

<details>
<summary>ステップ4：品質基準と運用ガイドを作成する</summary>

**品質チェックリストの観点：**
- 事実の正確性（発言内容が正しく反映されているか）
- 情報の網羅性（重要な決定事項が漏れていないか）
- 中立性（特定の意見に偏っていないか）
- 簡潔性（冗長でないか）

**運用上の注意事項：**
- 機密情報を含む議事録をクラウドAIに送信する際のリスク
- 要約結果の承認フロー
- 要約精度のフィードバック方法

</details>

```
【会議名】第3四半期 新規プロジェクト企画会議
【日時】2026年4月3日（金）14:00〜15:30
【場所】本社 第2会議室
【参加者】田中部長、鈴木課長、山田（企画担当）、佐藤（技術担当）
【議題】新サービス「AI社内アシスタント」の企画検討

田中部長：それでは始めましょう。山田さん、企画概要の説明をお願いします。

山田：はい。今回提案する「AI社内アシスタント」は、社員の日常業務をAIで
支援するチャットボットサービスです。社内規程の問い合わせ、経費精算の
手順案内、会議室予約の補助などを想定しています。

鈴木課長：ターゲットはどの部署ですか？

山田：まずは総務部と人事部を対象に、パイロット運用を考えています。
問い合わせ件数が多い部署から始めるのが効率的だと思います。

佐藤：技術的には、RAGを使って社内ドキュメントを検索ベースに組み込む
方式を想定しています。既存のナレッジベースを活用できます。

田中部長：予算感はどのくらいですか？

山田：初期構築で300万円、月額運用で20万円程度を見込んでいます。

鈴木課長：あ、そういえば昨日の懇親会、楽しかったですね。
田中部長：ええ、良い会でしたね。さて、本題に戻りましょう。

田中部長：了解しました。では以下の通り進めましょう。
1. 山田さんが4月15日までに詳細企画書を作成
2. 佐藤さんが4月20日までに技術検証の計画書を作成
3. 鈴木課長が予算の仮承認を経理部に確認

### 3. 要約プロンプト

#### 種類1：エグゼクティブサマリー用

```
あなたは経営企画の専門家です。
以下の会議議事録を、経営層向けのエグゼクティブサマリーとして
3〜5行で要約してください。

【要約ルール】
- 主要な意思決定と方向性のみに焦点を当てる
- 技術的な詳細は省略する
- 予算とスケジュールの概要は含める
- 「〜を決定した」「〜の方針とした」等の断定表現を使う

【議事録】
（前処理済みの議事録を貼り付け）

の予算規模で推進する。RAG技術を活用し、5月末までに技術検証を完了、6月から
パイロット運用を開始する方針とした。8月に効果測定を実施し、本格導入を判断する。

```
以下の会議議事録を、議題ごとに要約してください。

【要約ルール】
- 各議題について「議論の要点」「結論」を記載
- 賛成・反対意見がある場合は両方を記載
- 元の議事録の1/3程度の分量に圧縮
- 箇条書きを活用して読みやすくする

【議事録】
（前処理済みの議事録を貼り付け）
```

#### 種類3：アクションアイテム一覧用

```
以下の会議議事録から、アクションアイテムを抽出してください。

【出力形式】
Markdownの表形式で、以下の列を含めてください：
| # | 担当者 | タスク内容 | 期限 | ステータス |

【ルール】
- 明示的に「〇〇さんが〜する」と言及されたものを抽出
- 期限が明示されていない場合は「未定」と記載
- ステータスは全て「未着手」とする

【議事録】
（前処理済みの議事録を貼り付け）
```

### 4. 品質チェックリスト

| # | チェック項目 | 確認結果 |
|---|-------------|:--------:|
| 1 | 決定事項が正確に反映されているか | □ |
| 2 | アクションアイテムの担当者・期限が正しいか | □ |
| 3 | 重要な議論が省略されていないか | □ |
| 4 | AIが存在しない情報を追加していないか（ハルシネーション（hallucination）チェック） | □ |
| 5 | 特定の参加者の意見に偏っていないか | □ |
| 6 | 機密情報が適切に扱われているか | □ |

### 5. 運用上の注意事項

1. **機密情報**: 人事情報や未公開の経営戦略を含む議事録は、社内環境のAI（オンプレミス（on-premises）モデル）を使用する
2. **承認フロー**: 要約結果は会議の主催者が確認してから配信する
3. **保存**: 元の議事録と要約の両方を保存し、要約だけで元を廃棄しない

### 改良版の特徴

- **長文議事録への対応**（チャンク分割方式（Chunking））
- **要約精度のフィードバックループ**を設計
- **テンプレート化された運用マニュアル**を追加

---

#### 長文議事録のチャンク分割方式

10,000字を超える議事録は、AIのコンテキストウィンドウの制限に注意が必要です。

```
【方式1：議題分割方式（推奨）】

元の議事録（10,000字）
   │
   ├─ 議題1（2,500字）→ AI要約 → 要約1
   ├─ 議題2（3,000字）→ AI要約 → 要約2
   ├─ 議題3（2,500字）→ AI要約 → 要約3
   └─ 議題4（2,000字）→ AI要約 → 要約4
                              │
                              ▼
                    要約1〜4を統合 → 最終要約

【方式2：Map-Reduce方式】

元の議事録（10,000字）
   │
   ├─ チャンク1（3,000字）→ AI要約（Map）
   ├─ チャンク2（3,000字）→ AI要約（Map）
   ├─ チャンク3（3,000字）→ AI要約（Map）
   └─ チャンク4（1,000字）→ AI要約（Map）
                              │
                              ▼
                    全要約を結合 → AI最終要約（Reduce）
```

議題分割方式のメリット：文脈の断絶が起きにくく、議題ごとの要約が自然。
Map-Reduce方式のメリット：議題が明確に分かれていない議事録にも対応可能。

---

#### 要約精度のフィードバックループ

```
生成した要約
    │
    ▼
会議主催者が確認
    │
    ├─ OK → 配信
    │
    └─ 修正あり
         │
         ├─ 修正内容を記録
         │   （例：「決定事項Xが漏れていた」）
         │
         ▼
    フィードバックDB に蓄積
         │
         ▼
    月次でプロンプトを改善
         │
         ├─ 漏れがちな情報 → プロンプトのチェック項目に追加
         ├─ 過度な要約    → 分量指定を調整
         └─ 偏り          → 中立性の指示を強化
```

**フィードバック記録テンプレート：**

| 日付 | 会議名 | 修正内容 | カテゴリ | 対応済み |
|------|--------|---------|---------|:--------:|
| 4/3 | 企画会議 | 予算額が未記載だった | 情報漏れ | □ |
| 4/5 | 定例会議 | A部長の意見が省略された | 偏り | □ |

---

#### 運用マニュアル（テンプレート）

```
========================================
  議事録要約自動化 運用マニュアル v1.0
========================================

1. 対象会議
   - 参加者3名以上の公式会議
   - 議事録が1,000字以上の会議

2. 使用するAIサービス
   - 通常会議：Claude API（クラウド）
   - 機密会議：社内LLM（オンプレミス）

3. 要約の種類と配信先
   - エグゼクティブサマリー → 経営層Slackチャンネル
   - 詳細要約 → 会議参加者+関係部署
   - アクションアイテム → タスク管理ツール（Jira等）に自動登録

4. 品質管理フロー
   (1) AI要約を生成（所要時間：約1分）
   (2) 会議主催者が内容を確認（目安：5分以内）
   (3) 修正があれば修正を反映
   (4) 承認後に配信

5. 禁止事項
   - 人事評価に関する会議の内容をクラウドAIに送信しない
   - 要約結果を主催者の確認なしに配信しない
   - 元の議事録を破棄しない

6. トラブルシューティング
   - 要約が短すぎる → プロンプトの分量指定を増やす
   - ハルシネーションが発生 → 「議事録に記載されている情報のみを使用」の指示を強化
   - 処理が遅い → チャンク分割方式に切り替える
```

**初心者向けとの違い**:
- 長文議事録に対応するチャンク分割方式（議題分割・Map-Reduce）を導入
- フィードバックループで要約品質を継続的に改善する仕組みを設計
- 運用マニュアルをテンプレート化し、すぐに組織で使える形で提供
- 配信先やツール連携など、実務での運用を具体的にイメージした設計

## 1.1 LLM（大規模言語モデル）とは

### 基本概念

LLM（Large Language Model）は、大量のテキストデータから学習した確率モデルである。与えられたテキスト（プロンプト）に対して、最も確率の高い次のトークンを予測・生成することで応答を作成する。

```
入力（プロンプト）→ LLM → 出力（応答/コンプリーション）
```

### 主要なLLMの比較

| モデル | 開発元 | 特徴 |
|--------|--------|------|
| Claude 3.5/4 | Anthropic | 長文理解、コーディング、安全性に優れる |
| GPT-4o | OpenAI | 汎用性が高く、マルチモーダル対応 |
| Gemini | Google | 大規模コンテキスト、Google連携 |
| Llama 3 | Meta | オープンソース、カスタマイズ可能 |
| Command R+ | Cohere | RAG（検索拡張生成）に特化 |

### LLMの特性（知っておくべきこと）

1. **確率的生成**: 同じプロンプトでも異なる回答が返ることがある
2. **知識のカットオフ**: 学習データの時点までの知識しか持たない
3. **ハルシネーション**: もっともらしいが事実と異なる情報を生成することがある
4. **コンテキスト依存**: プロンプトの書き方で出力品質が大きく変わる

## 1.2 トークンとは

### トークンの概念

LLMはテキストを「トークン」という単位で処理する。トークンは単語そのものではなく、テキストを分割した最小処理単位である。

```
英語の例：
"Hello, world!" → ["Hello", ",", " world", "!"]  ≈ 4トークン

日本語の例：
"こんにちは世界" → ["こん", "にち", "は", "世界"]  ≈ 4トークン
（実際の分割はモデルにより異なる）
```

### トークンの重要性

| 項目 | 説明 |
|------|------|
| コスト | APIの料金はトークン数に比例する |
| 速度 | トークン数が多いほど処理時間が長い |
| 制限 | コンテキストウィンドウの上限がある |
| 日本語特有 | 日本語は英語より多くのトークンを消費する傾向がある |

### トークン数の目安

- 英語：1単語 ≈ 1〜1.5トークン
- 日本語：1文字 ≈ 1〜3トークン（ひらがな・カタカナは多め、漢字は少なめ）
- Claude 3.5 Sonnet：入力 $3/100万トークン、出力 $15/100万トークン（参考値）

> **実践Tips**: Claudeでトークン数を確認するには、APIのレスポンスに含まれる`usage`フィールドを参照する。

## 1.3 コンテキストウィンドウ

### コンテキストウィンドウとは

LLMが一度に処理できるトークンの最大量。入力（プロンプト）と出力（応答）の合計がこの範囲内に収まる必要がある。

```
[コンテキストウィンドウ]
├── 入力トークン（プロンプト + 過去の会話履歴）
└── 出力トークン（生成される応答）
```

### 主要モデルのコンテキストウィンドウ

| モデル | コンテキストウィンドウ |
|--------|------------------------|
| Claude 3.5 Sonnet | 200Kトークン |
| Claude 4（Opus） | 200K〜1Mトークン |
| GPT-4o | 128Kトークン |
| Gemini 1.5 Pro | 1M〜2Mトークン |

### コンテキストウィンドウの活用戦略

1. **必要最小限の情報を提供する**: 無関係な情報はトークンの無駄
2. **重要な情報を先頭に配置する**: LLMは先頭と末尾の情報をより重視する傾向がある（Lost in the Middle問題）
3. **長文は要約してから渡す**: 全文を渡すよりも要約を渡した方が効率的な場合がある
4. **チャンク分割**: 超長文は分割して段階的に処理する

## 1.4 temperature と top_p

### temperature（温度）

生成のランダム性を制御するパラメータ。

```
temperature = 0.0  → 最も確率の高いトークンを常に選択（決定論的）
temperature = 0.5  → バランスの取れた生成
temperature = 1.0  → より多様でクリエイティブな生成
temperature = 2.0  → 非常にランダム（品質は低下しやすい）
```

**使い分けの指針：**

| ユースケース | 推奨temperature |
|-------------|----------------|
| コード生成 | 0.0〜0.2 |
| 事実に基づく回答 | 0.0〜0.3 |
| ビジネス文書 | 0.3〜0.5 |
| 創作・ブレスト | 0.7〜1.0 |
| 詩・物語 | 0.8〜1.2 |

### top_p（核サンプリング）

累積確率がpに達するまでのトークンのみを候補にする。

```
top_p = 0.1  → 上位10%の確率のトークンからのみ選択
top_p = 0.5  → 上位50%のトークンから選択
top_p = 1.0  → すべてのトークンが候補（デフォルト）
```

### temperatureとtop_pの関係

一般的には**両方を同時に大きく変更しない**ことが推奨される。

```python
# Anthropic API での使用例
import anthropic

client = anthropic.Anthropic()
message = client.messages.create(
    model="claude-sonnet-4-20250514",
    max_tokens=1024,
    temperature=0.3,  # 事実ベースの回答向け
    messages=[
        {"role": "user", "content": "日本の首都はどこですか？"}
    ]
)
```

## 1.5 プロンプトの基本構造

### プロンプトの4要素

優れたプロンプトは、以下の4つの要素を組み合わせて構成される。

```
┌─────────────────────────────┐
│  1. 役割（Role）            │  → 「あなたは〜です」
│  2. 指示（Instruction）     │  → 「〜してください」
│  3. 文脈（Context）         │  → 背景情報、制約条件
│  4. 出力形式（Format）      │  → 「〜の形式で出力して」
└─────────────────────────────┘
```

### 基本構造の例

```markdown
# 悪い例
日本について教えて

# 良い例
あなたは歴史の専門家です。（役割）

以下の条件に基づいて、日本の戦国時代について説明してください。（指示）

条件：
- 対象読者：中学生
- 範囲：1467年〜1615年
- 重要な人物を3人以上含める（文脈）

出力形式：
- 見出しと箇条書きを使ったMarkdown形式
- 500文字以内（出力形式）
```

### Claudeでの基本的なプロンプト構造

Claudeでは、XMLタグを使った構造化が特に効果的である。

```xml
あなたはプロの翻訳者です。

<instructions>
以下の日本語テキストを英語に翻訳してください。
- 自然な英語表現を心がける
- 専門用語は原語を括弧内に残す
</instructions>

<input>
人工知能の発展により、自然言語処理の分野は大きく進歩しました。
</input>

<output_format>
翻訳結果のみを出力してください。説明は不要です。
</output_format>
```

## 1.6 なぜプロンプト設計が重要なのか

### プロンプトの品質が結果を決める

同じLLMでも、プロンプトの書き方によって出力の品質は劇的に変わる。

**実験：「メール作成」タスクでの比較**

```
プロンプトA（曖昧）：
「メールを書いて」

→ 結果：何のメールか不明確な汎用的な文面

プロンプトB（具体的）：
「あなたはビジネスメールの専門家です。
以下の条件でクライアントへの謝罪メールを作成してください。

状況：納品が3日遅延する
相手：株式会社ABCの田中部長（長年の取引先）
トーン：丁寧かつ誠実、具体的な対策を含む
長さ：200〜300文字」

→ 結果：状況に即した適切な謝罪メール
```

### プロンプトエンジニアリングがもたらす価値

1. **生産性向上**: 適切なプロンプトで作業時間を大幅削減
2. **品質向上**: 一貫した高品質な出力を得られる
3. **コスト削減**: 少ないやり取りで目的を達成（APIコスト削減）
4. **再現性**: テンプレート化により安定した結果を得られる
5. **スケーラビリティ**: プロンプトを共有・改善し組織全体で活用

### プロンプトエンジニアリングの適用範囲

```
ビジネス      → メール、レポート、企画書、議事録
開発          → コード生成、レビュー、テスト、デバッグ
クリエイティブ → 文章、キャッチコピー、アイデア出し
教育          → 教材作成、問題生成、解説作成
データ分析    → 抽出、分類、要約、可視化指示
```

## まとめ

- LLMは確率的にテキストを生成するモデルであり、プロンプトの品質が出力を決定する
- トークンはLLMの処理単位であり、コスト・速度・制限に直結する
- コンテキストウィンドウの範囲内で効率的に情報を配置することが重要
- temperatureとtop_pで出力の多様性を制御できる
- プロンプトは「役割・指示・文脈・出力形式」の4要素で構成する
- プロンプト設計は生産性・品質・コストすべてに影響する重要なスキルである

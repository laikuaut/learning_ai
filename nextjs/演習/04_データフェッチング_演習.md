## 演習1（基本）：サーバーコンポーネントで fetch しよう

外部 API からデータを取得し、一覧表示するサーバーコンポーネントを作成してください。

**要件：**
- `https://jsonplaceholder.typicode.com/posts?_limit=10` から投稿データを取得する
- 各投稿のタイトルを一覧表示する
- `async/await` を使用する（`useEffect` は使わない）

<details>
<summary>ヒント</summary>

サーバーコンポーネントでは、コンポーネント関数を `async` にして、直接 `await fetch(...)` が使えます。`useEffect` や `useState` は不要です。

</details>

<details>
<summary>解答例</summary>

```tsx
// app/posts/page.tsx
type Post = {
  id: number
  title: string
  body: string
}

export default async function PostsPage() {
  // サーバー上で直接データを取得
  const res = await fetch('https://jsonplaceholder.typicode.com/posts?_limit=10')
  const posts: Post[] = await res.json()

  return (
    <div style={{ padding: '2rem', maxWidth: '700px', margin: '0 auto' }}>
      <h1>投稿一覧</h1>
      <ul style={{ listStyle: 'none', padding: 0 }}>
        {posts.map((post) => (
          <li key={post.id} style={{
            padding: '1rem',
            borderBottom: '1px solid #eaeaea',
          }}>
            <h3 style={{ margin: '0 0 0.5rem' }}>{post.title}</h3>
            <p style={{ color: '#666', margin: 0 }}>
              {post.body.slice(0, 80)}...
            </p>
          </li>
        ))}
      </ul>
    </div>
  )
}
```

このコードはサーバーで実行されるため、取得結果が HTML に含まれた状態でクライアントに送信されます。SEO に有利で、初回表示も高速です。

</details>

## 演習2（基本）：キャッシュオプションの違いを理解しよう

以下の3つの `fetch` の違いを説明してください。

```tsx
// パターンA
const resA = await fetch('https://api.example.com/data')

// パターンB
const resB = await fetch('https://api.example.com/data', {
  cache: 'no-store',
})

// パターンC
const resC = await fetch('https://api.example.com/data', {
  next: { revalidate: 3600 },
})
```

<details>
<summary>ヒント</summary>

Next.js の `fetch` はデフォルトでキャッシュが有効です。`cache` オプションや `next.revalidate` オプションでキャッシュ戦略を変更できます。

</details>

<details>
<summary>解答例</summary>

- **パターンA** (`cache: 'force-cache'` がデフォルト): レスポンスがキャッシュされ、同じリクエストではキャッシュされた結果が返されます。ビルド時に取得されたデータがそのまま使われる静的生成（SSG）に対応します。頻繁に変わらないデータに適しています。

- **パターンB** (`cache: 'no-store'`): キャッシュを使用せず、リクエストのたびに最新のデータを取得します。ルートは動的レンダリング（SSR）になります。在庫情報やリアルタイムのダッシュボードデータに適しています。

- **パターンC** (`next: { revalidate: 3600 }`): ISR（Incremental Static Regeneration）。キャッシュされたデータを3600秒（1時間）ごとに再検証します。期間内はキャッシュを返し、期間経過後にバックグラウンドで再取得します。ブログ記事一覧や商品カタログに適しています。

</details>

## 演習3（基本）：エラーハンドリングを実装しよう

`fetch` でのデータ取得時にエラーが発生した場合の処理を実装してください。

**要件：**
- HTTP エラー（4xx, 5xx）をチェックする
- 存在しないリソースの場合は `notFound()` を呼び出す
- その他のエラーは `throw` で伝播させる

<details>
<summary>ヒント</summary>

`fetch` のレスポンスの `ok` プロパティと `status` プロパティで HTTP ステータスを確認します。`notFound` は `next/navigation` からインポートします。

</details>

<details>
<summary>解答例</summary>

```tsx
// app/posts/[id]/page.tsx
import { notFound } from 'next/navigation'

type Post = {
  id: number
  title: string
  body: string
}

type Props = {
  params: Promise<{ id: string }>
}

export default async function PostPage({ params }: Props) {
  const { id } = await params
  const res = await fetch(`https://jsonplaceholder.typicode.com/posts/${id}`)

  // 存在しないリソースの場合は 404 ページを表示
  if (res.status === 404) {
    notFound()
  }

  // その他の HTTP エラー
  if (!res.ok) {
    throw new Error(`データの取得に失敗しました（ステータス: ${res.status}）`)
  }

  const post: Post = await res.json()

  return (
    <article style={{ padding: '2rem', maxWidth: '700px', margin: '0 auto' }}>
      <h1>{post.title}</h1>
      <p>{post.body}</p>
    </article>
  )
}
```

`notFound()` を呼び出すと、最も近い `not-found.tsx` が表示されます。`throw` でエラーを投げると、最も近い `error.tsx` がキャッチしてフォールバック UI を表示します。

</details>

## 演習4（応用）：並列データ取得を実装しよう

複数の独立したデータを並列で取得し、ウォーターフォールを防止してください。

**要件：**
- ユーザー情報、投稿一覧、アルバム一覧を同時に取得する
- `Promise.all` を使って並列化する
- 取得したデータをすべて1つのページに表示する

<details>
<summary>ヒント</summary>

`Promise.all([fetch1, fetch2, fetch3])` を使うと、3つのリクエストが同時に開始されます。最も遅いリクエストが完了するまで待ちますが、順番に実行するよりも大幅に速くなります。

</details>

<details>
<summary>解答例</summary>

```tsx
// app/profile/page.tsx
type User = { id: number; name: string; email: string }
type Post = { id: number; title: string }
type Album = { id: number; title: string }

export default async function ProfilePage() {
  // 並列でデータを取得（Promise.all）
  const [user, posts, albums] = await Promise.all([
    fetch('https://jsonplaceholder.typicode.com/users/1').then(r => r.json()) as Promise<User>,
    fetch('https://jsonplaceholder.typicode.com/posts?userId=1&_limit=5').then(r => r.json()) as Promise<Post[]>,
    fetch('https://jsonplaceholder.typicode.com/albums?userId=1&_limit=5').then(r => r.json()) as Promise<Album[]>,
  ])
  // 3つのリクエストが同時に開始 → 最も遅いものに合わせた時間で完了

  return (
    <div style={{ padding: '2rem', maxWidth: '700px', margin: '0 auto' }}>
      <h1>プロフィール</h1>

      {/* ユーザー情報 */}
      <section style={{ marginBottom: '2rem' }}>
        <h2>{user.name}</h2>
        <p>{user.email}</p>
      </section>

      {/* 投稿一覧 */}
      <section style={{ marginBottom: '2rem' }}>
        <h2>最近の投稿</h2>
        <ul>
          {posts.map((post) => (
            <li key={post.id}>{post.title}</li>
          ))}
        </ul>
      </section>

      {/* アルバム一覧 */}
      <section>
        <h2>アルバム</h2>
        <ul>
          {albums.map((album) => (
            <li key={album.id}>{album.title}</li>
          ))}
        </ul>
      </section>
    </div>
  )
}
```

`Promise.all` を使うことで、3つのリクエストが同時に開始されます。順番に `await` すると合計3秒かかる処理が、並列化により最大1秒で完了します。

</details>

## 演習5（応用）：ISR（増分静的再生成）を設定しよう

ブログ一覧ページに ISR を設定し、60秒ごとに再検証されるようにしてください。

**要件：**
- ルートセグメント設定 `revalidate` を使用する
- ページに「最終更新日時」を表示して、再検証を確認できるようにする
- `fetch` レベルでもリバリデーション間隔を設定する

<details>
<summary>ヒント</summary>

`export const revalidate = 60` をページファイルに追加すると、そのルート全体が60秒間隔で再検証されます。`fetch` のオプションでも `next: { revalidate: 60 }` を指定できます。

</details>

<details>
<summary>解答例</summary>

```tsx
// app/blog/page.tsx

// ルート全体のリバリデーション間隔を設定
export const revalidate = 60 // 60秒ごとに再検証

type Post = {
  id: number
  title: string
  body: string
}

export default async function BlogPage() {
  // fetch レベルでもリバリデーション間隔を指定可能
  const res = await fetch('https://jsonplaceholder.typicode.com/posts?_limit=10', {
    next: { revalidate: 60 },
  })
  const posts: Post[] = await res.json()

  // 再検証を確認するために現在時刻を表示
  const now = new Date().toLocaleString('ja-JP', {
    timeZone: 'Asia/Tokyo',
  })

  return (
    <div style={{ padding: '2rem', maxWidth: '700px', margin: '0 auto' }}>
      <h1>ブログ一覧（ISR）</h1>
      <p style={{ color: '#999', fontSize: '0.9rem' }}>
        ページ生成日時: {now}
      </p>
      <p style={{ color: '#999', fontSize: '0.9rem' }}>
        60秒後にアクセスすると、バックグラウンドでページが再生成されます。
      </p>
      <ul style={{ listStyle: 'none', padding: 0 }}>
        {posts.map((post) => (
          <li key={post.id} style={{
            padding: '1rem',
            borderBottom: '1px solid #eaeaea',
          }}>
            <h3>{post.title}</h3>
          </li>
        ))}
      </ul>
    </div>
  )
}
```

ISR の動作フロー:
1. 初回アクセス時にページが生成され、キャッシュに保存される
2. 60秒以内の再アクセスにはキャッシュが返される（ページ生成日時が同じまま）
3. 60秒経過後のアクセスでは、古いページを返しつつバックグラウンドで再生成
4. 次のリクエストから新しいページが返される（ページ生成日時が更新される）

</details>

## 演習6（応用）：generateStaticParams で動的ルートを静的生成しよう

動的ルートのブログ記事ページを、ビルド時に静的生成してください。

**要件：**
- `generateStaticParams` で生成するパスの一覧を返す
- ビルド時に各記事ページの HTML が生成されることを確認する
- 未生成のパスへのアクセス時の動作を制御する

<details>
<summary>ヒント</summary>

`generateStaticParams` は `async` 関数で、パラメータオブジェクトの配列を返します。`dynamicParams` でビルド時に生成されなかったパスの扱いを制御できます。

</details>

<details>
<summary>解答例</summary>

```tsx
// app/blog/[id]/page.tsx
import { notFound } from 'next/navigation'

type Post = {
  id: number
  title: string
  body: string
}

// ビルド時に生成するパスの一覧を返す
export async function generateStaticParams() {
  const res = await fetch('https://jsonplaceholder.typicode.com/posts?_limit=5')
  const posts: Post[] = await res.json()

  // id 1〜5 の記事を静的生成
  return posts.map((post) => ({
    id: String(post.id),
  }))
}

// 未生成のパスにアクセスしたときの動作
// true（デフォルト）: オンデマンドで生成してキャッシュ
// false: 404 を返す
export const dynamicParams = true

type Props = {
  params: Promise<{ id: string }>
}

export default async function BlogPostPage({ params }: Props) {
  const { id } = await params
  const res = await fetch(`https://jsonplaceholder.typicode.com/posts/${id}`)

  if (!res.ok) {
    notFound()
  }

  const post: Post = await res.json()

  return (
    <article style={{ padding: '2rem', maxWidth: '700px', margin: '0 auto' }}>
      <p style={{ color: '#999' }}>記事 ID: {post.id}</p>
      <h1>{post.title}</h1>
      <p style={{ lineHeight: 1.8 }}>{post.body}</p>
    </article>
  )
}
```

`npm run build` を実行すると、ビルド出力で id 1〜5 の記事が静的生成されたことが確認できます。`dynamicParams = true` の場合、id 6 以降のパスにも最初のアクセス時に動的に生成されキャッシュされます。

</details>

## 演習7（チャレンジ）：タグベースのオンデマンド再検証を実装しよう

キャッシュタグを使ったオンデマンド再検証を実装してください。

**要件：**
- データ取得時にキャッシュタグを付与する
- Server Action でデータ更新後にタグベースでキャッシュを無効化する
- `revalidateTag` を使用する

<details>
<summary>ヒント</summary>

`fetch` の `next.tags` オプションでタグを付与します。`revalidateTag('タグ名')` でそのタグに関連するキャッシュをすべて無効化できます。

</details>

<details>
<summary>解答例</summary>

```tsx
// lib/api/posts.ts
// データ取得関数にタグを付与
export async function getPosts() {
  const res = await fetch('https://jsonplaceholder.typicode.com/posts?_limit=10', {
    next: { tags: ['posts'] }, // 'posts' タグを付与
  })
  if (!res.ok) throw new Error('投稿の取得に失敗しました')
  return res.json()
}

export async function getPost(id: string) {
  const res = await fetch(`https://jsonplaceholder.typicode.com/posts/${id}`, {
    next: { tags: ['posts', `post-${id}`] }, // 複数タグを付与
  })
  if (!res.ok) return null
  return res.json()
}
```

```tsx
// app/actions/posts.ts
'use server'

import { revalidateTag } from 'next/cache'

export async function refreshPosts() {
  // 'posts' タグのキャッシュをすべて無効化
  revalidateTag('posts')
}

export async function refreshPost(id: string) {
  // 特定の記事のキャッシュのみ無効化
  revalidateTag(`post-${id}`)
}
```

```tsx
// app/posts/page.tsx
import { getPosts } from '@/lib/api/posts'
import RefreshButton from './RefreshButton'

export default async function PostsPage() {
  const posts = await getPosts()

  return (
    <div style={{ padding: '2rem' }}>
      <div style={{ display: 'flex', justifyContent: 'space-between', alignItems: 'center' }}>
        <h1>投稿一覧</h1>
        <RefreshButton />
      </div>
      <ul style={{ listStyle: 'none', padding: 0 }}>
        {posts.map((post: any) => (
          <li key={post.id} style={{ padding: '0.75rem 0', borderBottom: '1px solid #eaeaea' }}>
            {post.title}
          </li>
        ))}
      </ul>
    </div>
  )
}
```

```tsx
// app/posts/RefreshButton.tsx
'use client'

import { useTransition } from 'react'
import { refreshPosts } from '@/app/actions/posts'

export default function RefreshButton() {
  const [isPending, startTransition] = useTransition()

  return (
    <button
      onClick={() => startTransition(() => refreshPosts())}
      disabled={isPending}
      style={{
        padding: '0.5rem 1rem',
        backgroundColor: isPending ? '#ccc' : '#0070f3',
        color: 'white',
        border: 'none',
        borderRadius: '4px',
        cursor: isPending ? 'default' : 'pointer',
      }}
    >
      {isPending ? '更新中...' : 'データを更新'}
    </button>
  )
}
```

タグベースの再検証は、パスベースよりもきめ細かくキャッシュを制御できます。`'posts'` タグを無効化すると、`getPosts()` と `getPost(id)` の両方のキャッシュが同時に無効化されます。`post-${id}` タグは特定の記事のキャッシュだけを無効化できます。

</details>

## 演習8（チャレンジ）：キャッシュ戦略の設計表を作ろう

以下の Web サイトのデータについて、適切なキャッシュ戦略を設計してください。

**サイト概要：** EC サイト（商品販売サイト）

| データ | 更新頻度 | キャッシュ戦略 | revalidate |
|--------|---------|-------------|-----------|
| 会社概要ページ | ? | ? | ? |
| 商品カテゴリ一覧 | ? | ? | ? |
| 商品詳細ページ | ? | ? | ? |
| 商品の在庫数 | ? | ? | ? |
| ユーザーのカート | ? | ? | ? |
| 注文履歴 | ? | ? | ? |

<details>
<summary>ヒント</summary>

データの更新頻度とリアルタイム性の要件に応じて、静的生成（SSG）、ISR、動的レンダリング（SSR）を使い分けます。ユーザー固有のデータは `cookies()` や `headers()` を使うため、自動的に動的レンダリングになります。

</details>

<details>
<summary>解答例</summary>

| データ | 更新頻度 | キャッシュ戦略 | revalidate |
|--------|---------|-------------|-----------|
| 会社概要ページ | ほぼなし | 静的生成（SSG） | — |
| 商品カテゴリ一覧 | 月1回程度 | ISR | 86400（1日） |
| 商品詳細ページ | 週数回 | ISR + タグベース再検証 | 3600（1時間） |
| 商品の在庫数 | リアルタイム | 動的（`cache: 'no-store'`） | — |
| ユーザーのカート | リアルタイム | 動的（`cookies()` 使用） | — |
| 注文履歴 | ユーザー固有 | 動的（`cookies()` 使用） | — |

**設計のポイント:**
- 会社概要のような変更がないページはビルド時に静的生成し、CDN から配信する
- 商品カテゴリのように低頻度で変わるデータは ISR で定期的に再検証する
- 商品詳細は ISR に加えて、CMS から更新があった際に `revalidateTag` でオンデマンド再検証する
- 在庫数やカート内容のようなリアルタイムデータは毎回サーバーで取得する
- ユーザー固有のデータは `cookies()` を使うため、自動的に動的レンダリングになる

</details>

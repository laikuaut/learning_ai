# 第6章: レイアウトとメタデータ

## この章のゴール

- `layout.tsx` と `template.tsx` の違いを理解し、適切に使い分けられるようになる
- Metadata API を使って SEO に最適化されたメタデータを設定できるようになる
- 動的な OGP（Open Graph Protocol）画像の生成方法を習得する
- 実務で求められるレイアウト設計のパターンを学ぶ

---

## 6.1 なぜレイアウト設計が重要なのか

Web アプリケーションでは、ヘッダー、フッター、サイドバーといった**共通 UI 要素**が多くのページで使い回されます。従来の React アプリケーションでは、各ページコンポーネントにこれらの要素を個別にインポートするか、`_app.tsx` で一括定義するしかありませんでした。

App Router の `layout.tsx` は、この問題をエレガントに解決します。ネスト可能なレイアウトにより、ページの階層構造に応じて**異なるレイアウトを自然に適用**できるのです。

```
従来の React:
┌─────────────────────┐
│  _app.tsx で全体管理   │ ← 1つのレイアウトしか持てない
│  ┌─────────────────┐ │
│  │  各ページ         │ │
│  └─────────────────┘ │
└─────────────────────┘

App Router:
┌─────────────────────┐
│  layout.tsx (ルート)   │ ← html, body, 共通ナビ
│  ┌─────────────────┐ │
│  │ (marketing)/     │ │ ← マーケティング用レイアウト
│  │  layout.tsx      │ │
│  ├─────────────────┤ │
│  │ (dashboard)/     │ │ ← ダッシュボード用レイアウト
│  │  layout.tsx      │ │
│  │  ┌────────────┐ │ │
│  │  │ settings/  │ │ │ ← 設定画面用レイアウト
│  │  │ layout.tsx │ │ │
│  │  └────────────┘ │ │
│  └─────────────────┘ │
└─────────────────────┘
```

## 6.2 layout.tsx の詳細

### ルートレイアウト（必須）

`app/layout.tsx` はアプリケーション全体のルートレイアウトです。**必ず `<html>` タグと `<body>` タグを含む**必要があります。

```tsx
// app/layout.tsx
import type { Metadata } from 'next'
import { Inter } from 'next/font/google'
import './globals.css'

const inter = Inter({ subsets: ['latin'] })

export const metadata: Metadata = {
  title: {
    default: 'My App',
    template: '%s | My App', // 子ページが title を持つと自動で適用
  },
  description: 'Next.js で構築されたアプリケーション',
}

export default function RootLayout({
  children,
}: {
  children: React.ReactNode
}) {
  return (
    <html lang="ja">
      <body className={inter.className}>
        <header>
          <nav>サイトナビゲーション</nav>
        </header>
        <main>{children}</main>
        <footer>
          <p>2024 My App. All rights reserved.</p>
        </footer>
      </body>
    </html>
  )
}
```

### ネストされたレイアウト

ルート以下の各ディレクトリに `layout.tsx` を配置すると、親のレイアウトの中に子のレイアウトが**ネスト**されます。

```
app/
├── layout.tsx            ← ルートレイアウト（ヘッダー、フッター）
├── page.tsx
├── dashboard/
│   ├── layout.tsx        ← ダッシュボード用レイアウト（サイドバー）
│   ├── page.tsx
│   └── settings/
│       ├── layout.tsx    ← 設定画面用レイアウト（タブナビ）
│       └── page.tsx
```

```tsx
// app/dashboard/layout.tsx
import Link from 'next/link'

export default function DashboardLayout({
  children,
}: {
  children: React.ReactNode
}) {
  return (
    <div style={{ display: 'flex' }}>
      <aside style={{ width: '250px', borderRight: '1px solid #ddd' }}>
        <nav>
          <ul>
            <li><Link href="/dashboard">概要</Link></li>
            <li><Link href="/dashboard/analytics">分析</Link></li>
            <li><Link href="/dashboard/settings">設定</Link></li>
          </ul>
        </nav>
      </aside>
      <div style={{ flex: 1, padding: '1rem' }}>
        {children}
      </div>
    </div>
  )
}
```

```tsx
// app/dashboard/settings/layout.tsx
import Link from 'next/link'

export default function SettingsLayout({
  children,
}: {
  children: React.ReactNode
}) {
  return (
    <div>
      {/* タブナビゲーション */}
      <div style={{ display: 'flex', gap: '1rem', borderBottom: '1px solid #ddd' }}>
        <Link href="/dashboard/settings">一般</Link>
        <Link href="/dashboard/settings/profile">プロフィール</Link>
        <Link href="/dashboard/settings/security">セキュリティ</Link>
      </div>
      <div style={{ padding: '1rem' }}>
        {children}
      </div>
    </div>
  )
}
```

### レイアウトの最終的なレンダリング構造

`/dashboard/settings` にアクセスした場合:

```
RootLayout (app/layout.tsx)
  └── DashboardLayout (app/dashboard/layout.tsx)
        └── SettingsLayout (app/dashboard/settings/layout.tsx)
              └── SettingsPage (app/dashboard/settings/page.tsx)
```

### layout.tsx の重要な特性

| 特性 | 説明 |
|------|------|
| 状態保持 | ページ遷移してもレイアウトの状態（例: 入力値）が保持される |
| 再レンダリング抑制 | 子ルート間の遷移でレイアウトは再レンダリングされない |
| サーバーコンポーネント | デフォルトでサーバーコンポーネントとして動作する |
| データ取得可能 | `async` にしてサーバーでデータを取得できる |
| `children` 必須 | `children` prop を受け取って表示する必要がある |

## 6.3 template.tsx — 毎回マウントされるレイアウト

`template.tsx` は `layout.tsx` と似ていますが、**ナビゲーションごとに新しいインスタンスがマウントされる**点が異なります。

### layout.tsx と template.tsx の違い

| 特性 | layout.tsx | template.tsx |
|------|-----------|-------------|
| ナビゲーション時 | 再マウントされない | 毎回再マウントされる |
| 状態 | 保持される | リセットされる |
| useEffect | 再実行されない | 毎回実行される |
| DOM | 維持される | 再構築される |

### template.tsx を使うべき場面

```tsx
// app/dashboard/template.tsx
'use client'

import { useEffect } from 'react'

export default function DashboardTemplate({
  children,
}: {
  children: React.ReactNode
}) {
  useEffect(() => {
    // ページ遷移のたびに実行される
    // → ページビューのトラッキングに最適
    console.log('ページが表示されました')
    trackPageView()
  }, [])

  return (
    <div className="fade-in">
      {/* CSSアニメーションがナビゲーションごとに再生される */}
      {children}
    </div>
  )
}
```

### template.tsx の使用例

- **ページ遷移アニメーション**: ナビゲーションごとにアニメーションを再生したい場合
- **アクセス解析**: ページ遷移ごとにページビューを記録したい場合
- **フォームのリセット**: ページ遷移時にフォームの入力値をリセットしたい場合
- **Suspense の再トリガー**: ページ遷移時にローディング状態を再表示したい場合

> **ポイント**: 多くの場合は `layout.tsx` で十分です。`template.tsx` は上記のような「ナビゲーションごとに何かをリセット・再実行したい」という明確なユースケースがある場合にのみ使用してください。

## 6.4 Metadata API — SEO の基盤

### なぜメタデータが重要なのか

メタデータは、検索エンジンやソーシャルメディアに対してページの内容を伝えるための情報です。適切なメタデータを設定することで:

- **検索エンジン**: ページのタイトルや説明がより正確に検索結果に表示される
- **ソーシャルメディア**: リンクを共有した際にリッチなプレビューが表示される
- **ブラウザ**: タブのタイトルやファビコンが正しく表示される

### 静的メタデータ

`metadata` オブジェクトをエクスポートすることで、静的にメタデータを定義できます。

```tsx
// app/about/page.tsx
import type { Metadata } from 'next'

export const metadata: Metadata = {
  title: '会社概要',
  description: '株式会社Example の会社概要ページです。',
}

export default function AboutPage() {
  return <h1>会社概要</h1>
}
```

### 動的メタデータ

`generateMetadata` 関数をエクスポートすると、リクエストパラメータや外部データに基づいてメタデータを動的に生成できます。

```tsx
// app/blog/[slug]/page.tsx
import type { Metadata } from 'next'

type Props = {
  params: Promise<{ slug: string }>
}

export async function generateMetadata({ params }: Props): Promise<Metadata> {
  const { slug } = await params

  // 外部APIからデータを取得
  const post = await fetch(`https://api.example.com/posts/${slug}`).then(
    res => res.json()
  )

  return {
    title: post.title,
    description: post.excerpt,
    openGraph: {
      title: post.title,
      description: post.excerpt,
      images: [post.coverImage],
    },
  }
}

export default async function BlogPost({ params }: Props) {
  const { slug } = await params
  const post = await fetch(`https://api.example.com/posts/${slug}`).then(
    res => res.json()
  )

  return (
    <article>
      <h1>{post.title}</h1>
      <div>{post.content}</div>
    </article>
  )
}
```

> **ポイント**: `generateMetadata` 内の `fetch` は、同じ URL であれば `page.tsx` 内の `fetch` と自動的にメモ化（Request Memoization）されます。そのため、同じデータを2回取得するようなコードを書いても、実際の HTTP リクエストは1回だけです。

## 6.5 Metadata オブジェクトの完全リファレンス

### 基本フィールド

```tsx
import type { Metadata } from 'next'

export const metadata: Metadata = {
  // === 基本 ===
  title: 'ページタイトル',
  description: 'ページの説明文（160文字程度が推奨）',

  // === title のテンプレート ===
  // layout.tsx で定義すると、子ページの title に自動適用
  title: {
    default: 'My App',                    // タイトル未設定時のデフォルト
    template: '%s | My App',              // 子ページの title を %s に展開
    absolute: 'Custom Title',             // テンプレートを無視
  },

  // === キーワード（SEO効果は限定的だが設定しておくとよい）===
  keywords: ['Next.js', 'React', 'TypeScript'],

  // === 作成者 ===
  authors: [{ name: '田中太郎', url: 'https://example.com' }],
  creator: '田中太郎',
  publisher: '株式会社Example',

  // === robots（クロール制御）===
  robots: {
    index: true,          // 検索結果に表示する
    follow: true,         // リンクをたどる
    googleBot: {
      index: true,
      follow: true,
      'max-video-preview': -1,
      'max-image-preview': 'large',
      'max-snippet': -1,
    },
  },

  // === canonical URL ===
  alternates: {
    canonical: 'https://example.com/about',
    languages: {
      'ja': 'https://example.com/ja/about',
      'en': 'https://example.com/en/about',
    },
  },
}
```

## 6.6 メタデータの継承とマージ

### 親子間のメタデータマージ

レイアウトとページのメタデータは**自動的にマージ**されます。子で定義したフィールドが親を上書きします。

```tsx
// app/layout.tsx — 親レイアウト
export const metadata: Metadata = {
  title: {
    default: 'My App',
    template: '%s | My App',
  },
  description: 'デフォルトの説明文',
  openGraph: {
    siteName: 'My App',
    locale: 'ja_JP',
  },
}

// app/blog/page.tsx — 子ページ
export const metadata: Metadata = {
  title: 'ブログ一覧',           // → 「ブログ一覧 | My App」になる
  description: 'ブログ記事の一覧', // 親の description を上書き
  openGraph: {
    title: 'ブログ一覧',         // siteName と locale は親から継承
  },
}
```

### 最終的に生成される HTML

```html
<head>
  <title>ブログ一覧 | My App</title>
  <meta name="description" content="ブログ記事の一覧" />
  <meta property="og:title" content="ブログ一覧" />
  <meta property="og:site_name" content="My App" />
  <meta property="og:locale" content="ja_JP" />
</head>
```

## 6.7 動的 OGP 画像の生成

### opengraph-image.tsx

Next.js では、`opengraph-image.tsx` という特殊ファイルを使って OGP 画像を動的に生成できます。

```tsx
// app/blog/[slug]/opengraph-image.tsx
import { ImageResponse } from 'next/og'

export const runtime = 'edge'
export const alt = 'ブログ記事のOGP画像'
export const size = { width: 1200, height: 630 }
export const contentType = 'image/png'

export default async function Image({
  params,
}: {
  params: { slug: string }
}) {
  // 記事データの取得
  const post = await fetch(
    `https://api.example.com/posts/${params.slug}`
  ).then(res => res.json())

  return new ImageResponse(
    (
      <div
        style={{
          width: '100%',
          height: '100%',
          display: 'flex',
          flexDirection: 'column',
          justifyContent: 'center',
          alignItems: 'center',
          backgroundColor: '#1a1a2e',
          color: '#ffffff',
          padding: '60px',
        }}
      >
        <div
          style={{
            fontSize: '48px',
            fontWeight: 'bold',
            textAlign: 'center',
            lineHeight: 1.4,
          }}
        >
          {post.title}
        </div>
        <div
          style={{
            fontSize: '24px',
            marginTop: '20px',
            color: '#aaaaaa',
          }}
        >
          My Blog
        </div>
      </div>
    ),
    { ...size }
  )
}
```

### 静的な OGP 画像ファイル

動的生成が不要な場合は、画像ファイルを直接配置するだけで OGP 画像として認識されます。

```
app/
├── opengraph-image.png       → デフォルトのOGP画像
├── twitter-image.png         → デフォルトのTwitter画像
├── blog/
│   ├── opengraph-image.png   → /blog のOGP画像
│   └── [slug]/
│       └── opengraph-image.tsx → 動的生成
```

## 6.8 フォントの最適化

### next/font/google — Google Fonts の最適化

```tsx
// app/layout.tsx
import { Noto_Sans_JP, Inter } from 'next/font/google'

const notoSansJP = Noto_Sans_JP({
  subsets: ['latin'],
  weight: ['400', '700'],
  display: 'swap',
  variable: '--font-noto-sans-jp',
})

const inter = Inter({
  subsets: ['latin'],
  display: 'swap',
  variable: '--font-inter',
})

export default function RootLayout({
  children,
}: {
  children: React.ReactNode
}) {
  return (
    <html lang="ja" className={`${notoSansJP.variable} ${inter.variable}`}>
      <body>{children}</body>
    </html>
  )
}
```

```css
/* globals.css */
body {
  font-family: var(--font-noto-sans-jp), var(--font-inter), sans-serif;
}
```

### next/font の利点

| 利点 | 説明 |
|------|------|
| セルフホスティング | Google へのリクエストが発生しない（プライバシー向上） |
| レイアウトシフト防止 | `size-adjust` が自動適用される |
| ビルド時最適化 | フォントファイルがビルド時にダウンロード・最適化される |
| CSS 変数対応 | `variable` オプションで CSS 変数として使用可能 |

### next/font/local — ローカルフォント

```tsx
import localFont from 'next/font/local'

const myFont = localFont({
  src: [
    {
      path: '../fonts/MyFont-Regular.woff2',
      weight: '400',
      style: 'normal',
    },
    {
      path: '../fonts/MyFont-Bold.woff2',
      weight: '700',
      style: 'normal',
    },
  ],
  variable: '--font-my-font',
})
```

## 6.9 実践的なレイアウト設計パターン

### パターン1: 認証状態によるレイアウト分離

```
app/
├── layout.tsx                    ← 共通ルートレイアウト
├── (public)/                     ← 未ログインユーザー向け
│   ├── layout.tsx               ← ランディングページ用レイアウト
│   ├── page.tsx                 ← トップページ
│   ├── about/page.tsx
│   └── pricing/page.tsx
├── (auth)/                       ← 認証ページ
│   ├── layout.tsx               ← シンプルなレイアウト（中央寄せ）
│   ├── login/page.tsx
│   └── register/page.tsx
├── (app)/                        ← ログイン済みユーザー向け
│   ├── layout.tsx               ← サイドバー付きレイアウト
│   ├── dashboard/page.tsx
│   └── settings/page.tsx
```

### パターン2: 条件付きレイアウト

```tsx
// app/(app)/layout.tsx
import { auth } from '@/lib/auth'
import { redirect } from 'next/navigation'
import Sidebar from '@/components/Sidebar'

export default async function AppLayout({
  children,
}: {
  children: React.ReactNode
}) {

    redirect('/login')

  return (
    <div style={{ display: 'flex' }}>
      <Sidebar user={session.user} />
      <main style={{ flex: 1 }}>
        {children}
```

## 6.10 JSON-LD 構造化データ

検索エンジンにページの内容をより詳しく伝えるために、JSON-LD 構造化データを設定できます。

```tsx
// app/blog/[slug]/page.tsx
export default async function BlogPost({
  params,
}: {
  params: Promise<{ slug: string }>
}) {
  const { slug } = await params
  const post = await getPost(slug)

  // JSON-LD 構造化データ
  const jsonLd = {
    '@context': 'https://schema.org',
    '@type': 'BlogPosting',
    headline: post.title,
    description: post.excerpt,
    image: post.coverImage,
    datePublished: post.publishedAt,
    dateModified: post.updatedAt,
    author: {
      '@type': 'Person',
      name: post.author.name,
    },
  }

  return (
    <>
      {/* JSON-LD を script タグとして挿入 */}
      <script
        type="application/ld+json"
        dangerouslySetInnerHTML={{ __html: JSON.stringify(jsonLd) }}
      />

      <article>
        <h1>{post.title}</h1>
        <div>{post.content}</div>
      </article>
    </>
  )
}
```

## 6.11 よくある間違い

### 間違い1: ルートレイアウトに html/body タグがない

```tsx
// 悪い例: html と body がない
export default function RootLayout({ children }: { children: React.ReactNode }) {
  return <div>{children}</div>
  // → エラーになる
}

// 良い例
export default function RootLayout({ children }: { children: React.ReactNode }) {
  return (
    <html lang="ja">
      <body>{children}</body>
    </html>
  )
}
```

### 間違い2: レイアウトで searchParams を使おうとする

```tsx
// 悪い例: layout.tsx では searchParams を受け取れない
export default function Layout({
  searchParams, // ← layout には渡されない！
}: {
  searchParams: { page: string }
}) {
  // ...
}

// 良い例: searchParams は page.tsx で受け取る
// app/blog/page.tsx
export default async function Page({
  searchParams,
}: {
  searchParams: Promise<{ page?: string }>
}) {
  const { page } = await searchParams
  // ...
}
```

### 間違い3: メタデータを動的に生成する際の型の不備

```tsx
// 悪い例: generateMetadata の戻り値の型が不正確
export async function generateMetadata({ params }) { // ← 型がない
  return { title: 'タイトル' }
}

// 良い例: 型を正しく指定
import type { Metadata } from 'next'

type Props = {
  params: Promise<{ slug: string }>
}

export async function generateMetadata({ params }: Props): Promise<Metadata> {
  const { slug } = await params
  // ...
  return { title: '正しいタイトル' }
}
```

## 6.12 現場でのベストプラクティス

### 1. メタデータの設計指針

| 項目 | 推奨事項 |
|------|---------|
| title | 30〜60文字。ページの内容を端的に表現 |
| description | 120〜160文字。ページの概要をユーザー視点で記述 |
| OGP 画像 | 1200x630px。テキストは中央に配置 |
| canonical URL | 重複コンテンツがある場合は必ず設定 |

### 2. レイアウトファイルの責務を明確にする

```
layout.tsx の責務:
  ✅ 共通 UI 要素（ナビゲーション、フッター、サイドバー）
  ✅ 共通プロバイダー（テーマ、認証状態）
  ✅ ルート全体のメタデータ
  ✅ 共通のデータ取得（ユーザー情報など）

  ❌ 個別ページのビジネスロジック
  ❌ ページ固有のデータ取得
  ❌ searchParams に依存する処理
```

### 3. OGP のデバッグ方法

OGP 画像やメタデータが正しく設定されているかは、以下のツールで確認できます。

- **Facebook Sharing Debugger**: Facebook でのプレビューを確認
- **Twitter Card Validator**: Twitter でのプレビューを確認
- **Open Graph Debugger**: OGP メタデータの検証
- **Next.js の開発者ツール**: ブラウザの開発者ツールで `<head>` 内のメタタグを確認

## 6.13 ポイントまとめ

本章で学んだこと:

1. `layout.tsx` はネスト可能な共通レイアウトで、**ナビゲーション間で状態が保持**されます
2. `template.tsx` は `layout.tsx` と同じ構造ですが、**ナビゲーションごとに再マウント**されます
3. **Metadata API** で静的・動的なメタデータを型安全に定義できます
4. `title.template` を使うと、親子間で**タイトルのテンプレート**を共有できます
5. **opengraph-image.tsx** で動的な OGP 画像を生成できます
6. **next/font** でフォントを最適化し、レイアウトシフトを防止できます
7. ルートグループ `()` を使って、**認証状態やセクションごとにレイアウトを分離**するのが実務の定石です

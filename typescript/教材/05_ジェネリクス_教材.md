# 第5章: ジェネリクス（Generics）

## この章のゴール

- ジェネリクスの概念を理解し、「なぜ必要なのか」を説明できるようになる
- ジェネリック関数・インターフェース・クラスを自在に定義できるようになる
- 制約（constraints）を使って、型パラメータの範囲を適切に制限できるようになる
- 実務で頻出するジェネリクスのパターン（APIレスポンス型、コレクション操作など）を書けるようになる

---

## 5.1 ジェネリクスとは何か

### 「型のパラメータ化」という考え方

ジェネリクスとは、**型をパラメータとして受け取る仕組み**です。関数が「値」を引数として受け取るように、ジェネリクスは「型」を引数として受け取ります。

日常生活で例えるなら、「箱」を思い浮かべてください。箱自体は中に何を入れるか決まっていません。「果物を入れる箱」「本を入れる箱」「おもちゃを入れる箱」のように、**中身の種類をあとから指定**できます。ジェネリクスはまさにこの「汎用的な箱」のような仕組みです。

```
┌─────────────────────────────────────────────────┐
│               ジェネリクスのイメージ               │
│                                                   │
│   Box<T>  ← Tは「型パラメータ」（後から指定）     │
│                                                   │
│   Box<string>  → 文字列専用の箱                   │
│   Box<number>  → 数値専用の箱                     │
│   Box<User>    → ユーザー専用の箱                 │
│                                                   │
│   同じ設計図（Box）から、異なる型の箱を作れる      │
└─────────────────────────────────────────────────┘
```

### なぜジェネリクスが必要なのか

ジェネリクスが存在しない世界を想像してみましょう。「配列の最初の要素を返す関数」を作りたいとします。

```typescript
// ❌ any を使う方法 — 型安全性がゼロ
function firstAny(arr: any[]): any {
  return arr[0];
}

const result1 = firstAny([1, 2, 3]);      // any型 — numberだとわからない
const result2 = firstAny(["a", "b", "c"]); // any型 — stringだとわからない

// anyなので、何でもできてしまう（バグの温床）
result1.toUpperCase(); // コンパイルエラーにならないが、実行時にクラッシュ！
```

```typescript
// ❌ 型ごとに関数を作る方法 — 重複だらけ
function firstNumber(arr: number[]): number {
  return arr[0];
}

function firstString(arr: string[]): string {
  return arr[0];
}

function firstBoolean(arr: boolean[]): boolean {
  return arr[0];
}

// 新しい型が増えるたびに、新しい関数が必要...（つらい）
```

```typescript
// ✅ ジェネリクスを使う方法 — 型安全かつ再利用可能
function first<T>(arr: T[]): T {
  return arr[0];
}

const num = first([1, 2, 3]);          // number型
const str = first(["a", "b", "c"]);    // string型
const bool = first([true, false]);      // boolean型

num.toFixed(2);     // OK: numberのメソッド
str.toUpperCase();  // OK: stringのメソッド
// num.toUpperCase(); // エラー！numberにtoUpperCaseはない
```

このように、ジェネリクスは**「型安全性」と「再利用性」を両立させる**ための仕組みです。

### JavaScriptとの違い

JavaScriptにはジェネリクスの概念がありません。JavaScriptでは「何でも入る汎用関数」を作ると、型情報が完全に失われます。TypeScriptのジェネリクスがあれば、汎用的な関数でも入力と出力の型の関係性が保持されます。

```javascript
// JavaScript: 型情報が失われる
function identity(value) {
  return value;  // 戻り値の型が不明
}
```

```typescript
// TypeScript: 型情報が保持される
function identity<T>(value: T): T {
  return value;  // 入力と同じ型が返ることが保証される
}
```

## 5.2 ジェネリック関数

### 基本構文

ジェネリック関数は、関数名の直後に `<T>` のように型パラメータを宣言します。`T` は慣例的によく使われる名前ですが、任意の名前を付けられます。

```typescript
// 基本的なジェネリック関数
function identity<T>(value: T): T {
  return value;
}

// 呼び出し方1: 型を明示的に指定
const a = identity<string>("hello");  // string型
const b = identity<number>(42);       // number型

// 呼び出し方2: 型推論に任せる（推奨）
const c = identity("hello");  // string型と推論される
const d = identity(42);       // number型と推論される
```

> **ポイント:** 型推論が正しく機能する場合は、型引数の指定を省略した方がコードがすっきりします。型推論がうまくいかない場合にだけ、明示的に型引数を指定しましょう。

### 複数の型パラメータ

型パラメータは複数指定できます。

```typescript
// 2つの型パラメータ
function pair<T, U>(first: T, second: U): [T, U] {
  return [first, second];
}

const p1 = pair("name", 25);    // [string, number]
const p2 = pair(true, "hello"); // [boolean, string]

// 3つの型パラメータ
function triple<A, B, C>(a: A, b: B, c: C): [A, B, C] {
  return [a, b, c];
}
```

### 型パラメータの命名規則

慣例として、以下の1文字の名前がよく使われます。

| 名前 | 意味 | 使用場面 |
|------|------|----------|
| `T` | Type | 汎用的な型パラメータ |
| `U`, `V` | — | 2番目、3番目の型パラメータ |
| `K` | Key | オブジェクトのキーの型 |
| `V` | Value | 値の型 |
| `E` | Element / Error | 要素の型、エラーの型 |
| `R` | Return / Result | 戻り値の型、結果の型 |

ただし、意味が明確になる場合は、より説明的な名前を使うこともあります。

```typescript
// 説明的な型パラメータ名の例
function fetchData<TResponse, TError>(
  url: string
): Promise<TResponse | TError> {
  // ...
  return {} as any; // 実装は省略
}
```

### アロー関数でのジェネリクス

アロー関数でもジェネリクスを使えます。

```typescript
// アロー関数でのジェネリクス
const identity2 = <T>(value: T): T => value;

// 複数の型パラメータ
const swap = <T, U>(pair: [T, U]): [U, T] => [pair[1], pair[0]];

const swapped = swap(["hello", 42]); // [number, string] → [42, "hello"]
```

> **注意（TSX）:** `.tsx` ファイル（React + TypeScript）では、`<T>` がJSXタグと混同される可能性があります。その場合は `<T,>` や `<T extends unknown>` と書くことで回避できます。

### 実用例: 配列操作のユーティリティ関数

```typescript
// 配列の最後の要素を取得
function last<T>(arr: T[]): T | undefined {
  return arr[arr.length - 1];
}

last([1, 2, 3]);       // number | undefined
last(["a", "b"]);      // string | undefined

// 配列からユニークな値を抽出
function unique<T>(arr: T[]): T[] {
  return [...new Set(arr)];
}

unique([1, 2, 2, 3, 3]); // number[] → [1, 2, 3]

// 配列をグループ化
function groupBy<T>(arr: T[], keyFn: (item: T) => string): Record<string, T[]> {
  return arr.reduce((groups, item) => {
    const key = keyFn(item);
    if (!groups[key]) {
      groups[key] = [];
    }
    groups[key].push(item);
    return groups;
  }, {} as Record<string, T[]>);
}

const users = [
  { name: "太郎", department: "開発" },
  { name: "花子", department: "営業" },
  { name: "次郎", department: "開発" },
];

const grouped = groupBy(users, (u) => u.department);
// { "開発": [...], "営業": [...] }
```

### ポイントまとめ

- ジェネリック関数は `function name<T>(...)` の構文で定義する
- 型パラメータは呼び出し時に自動的に推論される（明示指定も可能）
- 複数の型パラメータを使って、入力と出力の型の関係を表現できる
- `T`, `U`, `K`, `V` などの1文字名が慣例だが、説明的な名前も使える

## 5.3 ジェネリック制約（Constraints）

### なぜ制約が必要なのか

型パラメータ `T` はデフォルトでは「何でもあり」です。しかし、関数の中で `T` に対して特定の操作をしたい場合、制約が必要になります。

```typescript
// ❌ エラー: Tが何であるかわからないため、lengthにアクセスできない
function getLength<T>(value: T): number {
  return value.length; // エラー！Tにlengthプロパティがあるとは限らない
}
```

### extends による制約

`extends` キーワードを使って、型パラメータに制約を付けます。「TはXを満たす型でなければならない」という条件を指定できます。

```typescript
// ✅ lengthプロパティを持つ型に制約
function getLength<T extends { length: number }>(value: T): number {
  return value.length; // OK: Tにはlengthがあることが保証される
}

getLength("hello");     // OK: stringにはlengthがある
getLength([1, 2, 3]);   // OK: 配列にもlengthがある
getLength({ length: 5, name: "test" }); // OK: lengthプロパティがある
// getLength(42);        // エラー！numberにはlengthがない
```

```
┌──────────────────────────────────────────────────┐
│              制約のイメージ                        │
│                                                    │
│   T extends { length: number }                     │
│                                                    │
│   ┌───────────── 全ての型 ─────────────┐          │
│   │                                     │          │
│   │   ┌── lengthを持つ型 ──┐           │          │
│   │   │  string            │           │          │
│   │   │  Array<any>        │  ← OK     │          │
│   │   │  { length: n }     │           │          │
│   │   └────────────────────┘           │          │
│   │                                     │          │
│   │   number, boolean  ← NG (制約外)   │          │
│   └─────────────────────────────────────┘          │
└──────────────────────────────────────────────────┘
```

### インターフェースを使った制約

```typescript
interface Printable {
  toString(): string;
}

function print<T extends Printable>(value: T): void {
  console.log(value.toString());
}

// すべてのオブジェクトにtoStringがあるので、実質何でもOK
print("hello");
print(42);
print({ toString: () => "カスタム文字列" });
```

### keyof を使った制約

オブジェクトのキーに対する制約は非常によく使います。

```typescript
// KはTのキーのいずれかでなければならない
function getProperty<T, K extends keyof T>(obj: T, key: K): T[K] {
  return obj[key];
}

const user = { name: "太郎", age: 25, email: "taro@example.com" };

getProperty(user, "name");  // string型
getProperty(user, "age");   // number型
// getProperty(user, "phone"); // エラー！"phone"はuserのキーではない
```

この関数が優れている点を見てみましょう。

```typescript
// ❌ ジェネリクスなしだと、戻り値の型が不明
function getPropertyUnsafe(obj: any, key: string): any {
  return obj[key]; // 戻り値はany — 型安全性なし
}

// ✅ ジェネリクスなら、キーに応じて正しい戻り値型が返る
const name = getProperty(user, "name");  // string型（"太郎"）
const age = getProperty(user, "age");    // number型（25）
```

### 複数の制約

複数のインターフェースの制約を同時に指定するには、交差型（`&`）を使います。

```typescript
interface HasId {
  id: number;
}

interface HasName {
  name: string;
}

// TはHasIdとHasNameの両方を満たす必要がある
function displayEntity<T extends HasId & HasName>(entity: T): string {
  return `#${entity.id}: ${entity.name}`;
}

displayEntity({ id: 1, name: "太郎", email: "taro@example.com" }); // OK
// displayEntity({ id: 1 }); // エラー！nameがない
```

### デフォルト型パラメータ

型パラメータにデフォルト値を指定できます。

```typescript
// デフォルト型パラメータ
interface ApiResponse<T = unknown> {
  data: T;
  status: number;
  message: string;
}

// 型引数を省略するとunknownになる
const response1: ApiResponse = {
  data: "anything",
  status: 200,
  message: "OK",
};

// 明示的に指定することもできる
const response2: ApiResponse<{ users: string[] }> = {
  data: { users: ["太郎", "花子"] },
  status: 200,
  message: "OK",
};
```

### ポイントまとめ

- `T extends X` で型パラメータに制約を付ける
- `keyof` と組み合わせて、オブジェクトのキーを型安全に扱える
- 交差型（`&`）で複数の制約を同時に指定できる
- デフォルト型パラメータ（`T = デフォルト型`）で省略時の型を指定できる

## 5.4 ジェネリックインターフェースとジェネリック型エイリアス

### ジェネリックインターフェース

インターフェースにも型パラメータを付けられます。

```typescript
// ジェネリックなリスト
interface List<T> {
  items: T[];
  length: number;
  add(item: T): void;
  get(index: number): T | undefined;
}

// string専用のリスト
const stringList: List<string> = {
  items: ["hello", "world"],
  length: 2,
  add(item: string) {
    this.items.push(item);
    this.length++;
  },
  get(index: number) {
    return this.items[index];
  },
};
```

### 実践例: Repository パターン

データアクセス層でよく使われるパターンです。

```typescript
// エンティティの基底型
interface Entity {
  id: number;
  createdAt: Date;
  updatedAt: Date;
}

// ジェネリックなリポジトリインターフェース
interface Repository<T extends Entity> {
  findAll(): Promise<T[]>;
  findById(id: number): Promise<T | null>;
  create(data: Omit<T, "id" | "createdAt" | "updatedAt">): Promise<T>;
  update(id: number, data: Partial<T>): Promise<T>;
  delete(id: number): Promise<void>;
}

// ユーザーエンティティ
interface User extends Entity {
  name: string;
  email: string;
  role: "admin" | "user";
}

// 商品エンティティ
interface Product extends Entity {
  name: string;
  price: number;
  stock: number;
}

// 型を指定してリポジトリの型を決定
type UserRepository = Repository<User>;
type ProductRepository = Repository<Product>;
```

### ジェネリック型エイリアス

型エイリアスにも型パラメータを使えます。

```typescript
// Nullable型: Tまたはnull
type Nullable<T> = T | null;

// Optional型: Tまたはundefined
type Optional<T> = T | undefined;

// AsyncResult型: 非同期処理の結果
type AsyncResult<T, E = Error> = {
  data: T | null;
  error: E | null;
  loading: boolean;
};

// 使用例
let userName: Nullable<string> = "太郎";
userName = null; // OK

let userAge: Optional<number> = 25;
userAge = undefined; // OK

const apiResult: AsyncResult<User[]> = {
  data: null,
  error: null,
  loading: true,
};
```

### ジェネリックなユーティリティ型を自作する

```typescript
// ツリー構造
type TreeNode<T> = {
  value: T;
  children: TreeNode<T>[];
};

const fileTree: TreeNode<string> = {
  value: "src",
  children: [
    {
      value: "components",
      children: [
        { value: "Button.tsx", children: [] },
        { value: "Header.tsx", children: [] },
      ],
    },
    {
      value: "utils",
      children: [
        { value: "format.ts", children: [] },
      ],
    },
  ],
};

// ページネーション付きレスポンス
type PaginatedResponse<T> = {
  items: T[];
  total: number;
  page: number;
  perPage: number;
  hasNext: boolean;
  hasPrev: boolean;
};

// 使用例
type UserListResponse = PaginatedResponse<User>;
```

### ポイントまとめ

- インターフェースや型エイリアスにもジェネリクスを使える
- `Repository<T>` のように、共通のデータ操作パターンを汎用化できる
- `TreeNode<T>` のように、再帰的な型もジェネリクスで表現可能
- `Nullable<T>`, `Optional<T>` などの小さなユーティリティ型が実務で便利

## 5.5 ジェネリッククラス

### 基本構文

クラスにもジェネリクスを適用できます。

```typescript
class Stack<T> {
  private items: T[] = [];

  push(item: T): void {
    this.items.push(item);
  }

  pop(): T | undefined {
    return this.items.pop();
  }

  peek(): T | undefined {
    return this.items[this.items.length - 1];
  }

  isEmpty(): boolean {
    return this.items.length === 0;
  }

  size(): number {
    return this.items.length;
  }
}

// number専用のスタック
const numberStack = new Stack<number>();
numberStack.push(1);
numberStack.push(2);
numberStack.push(3);
const top = numberStack.pop(); // number | undefined

// string専用のスタック
const stringStack = new Stack<string>();
stringStack.push("hello");
stringStack.push("world");
const word = stringStack.pop(); // string | undefined
```

### 実践例: 型安全なイベントエミッター

```typescript
// イベント名と引数の型を定義
interface EventMap {
  click: { x: number; y: number };
  keypress: { key: string; code: number };
  resize: { width: number; height: number };
}

class TypedEventEmitter<T extends Record<string, any>> {
  private listeners: {
    [K in keyof T]?: Array<(data: T[K]) => void>;
  } = {};

  on<K extends keyof T>(event: K, listener: (data: T[K]) => void): void {
    if (!this.listeners[event]) {
      this.listeners[event] = [];
    }
    this.listeners[event]!.push(listener);
  }

  emit<K extends keyof T>(event: K, data: T[K]): void {
    const eventListeners = this.listeners[event];
    if (eventListeners) {
      eventListeners.forEach((listener) => listener(data));
    }
  }

  off<K extends keyof T>(event: K, listener: (data: T[K]) => void): void {
    const eventListeners = this.listeners[event];
    if (eventListeners) {
      this.listeners[event] = eventListeners.filter((l) => l !== listener);
    }
  }
}

// 使用例
const emitter = new TypedEventEmitter<EventMap>();

// ✅ 型安全: clickイベントのデータは { x: number, y: number }
emitter.on("click", (data) => {
  console.log(`Clicked at (${data.x}, ${data.y})`);
});

// ✅ 型安全: keypressイベントのデータは { key: string, code: number }
emitter.on("keypress", (data) => {
  console.log(`Key: ${data.key}, Code: ${data.code}`);
});

// ❌ エラー: "click"イベントに不正なデータ
// emitter.emit("click", { key: "a" }); // エラー！
```

### ポイントまとめ

- クラス定義に `class Name<T>` の形でジェネリクスを適用できる
- メソッドの引数・戻り値に型パラメータを使って一貫性を保てる
- イベントシステムなど、複数の型を動的に扱う場面で非常に強力

## 5.6 条件型（Conditional Types）

### 基本構文

条件型は、型レベルの `if-else` です。`T extends U ? X : Y` の構文で「TがUに代入可能ならX、そうでなければY」を表現します。

```typescript
// TがstringならYes、そうでなければNo
type IsString<T> = T extends string ? "yes" : "no";

type A = IsString<string>;  // "yes"
type B = IsString<number>;  // "no"
type C = IsString<"hello">; // "yes"（"hello"はstringのサブタイプ）
```

### 実用的な条件型

```typescript
// nullを除外する型
type NonNullable2<T> = T extends null | undefined ? never : T;

type D = NonNullable2<string | null>;      // string
type E = NonNullable2<number | undefined>; // number

// 配列の要素型を取得
type ElementType<T> = T extends (infer E)[] ? E : never;

type F = ElementType<string[]>;   // string
type G = ElementType<number[]>;   // number
type H = ElementType<string>;     // never（配列でない）

// Promiseの中身の型を取得
type Awaited2<T> = T extends Promise<infer U> ? U : T;

type I = Awaited2<Promise<string>>;  // string
type J = Awaited2<Promise<number>>;  // number
type K = Awaited2<string>;            // string（Promiseでなければそのまま）

// 関数の戻り値の型を取得
type ReturnType2<T> = T extends (...args: any[]) => infer R ? R : never;

type L = ReturnType2<() => string>;           // string
type M = ReturnType2<(x: number) => boolean>; // boolean
```

### infer キーワード

`infer` は条件型の中で「型を推論して名前を付ける」キーワードです。正規表現のキャプチャグループのようなものと考えてください。

```typescript
// 関数の最初の引数の型を取得
type FirstArg<T> = T extends (first: infer F, ...rest: any[]) => any
  ? F
  : never;

type N = FirstArg<(name: string, age: number) => void>; // string
type O = FirstArg<(id: number) => void>;                 // number

// オブジェクトの値の型を取得
type ValueOf<T> = T extends Record<string, infer V> ? V : never;

type P = ValueOf<{ name: string; age: number }>; // string | number
```

### ポイントまとめ

- 条件型は `T extends U ? X : Y` で型レベルの分岐を行う
- `infer` キーワードで、型の一部を抽出（推論）できる
- `ReturnType`, `Parameters` などの組み込みユーティリティ型も条件型で実装されている

## 5.7 マップ型（Mapped Types）

マップ型は、既存の型をベースに新しい型を生成する仕組みです。オブジェクトの各プロパティに対して変換を適用できます。

```typescript
// 全プロパティをオプショナルにする（Partialの実装）
type MyPartial<T> = {
  [K in keyof T]?: T[K];
};

// 全プロパティを読み取り専用にする（Readonlyの実装）
type MyReadonly<T> = {
  readonly [K in keyof T]: T[K];
};

interface User {
  name: string;
  age: number;
  email: string;
}

type PartialUser = MyPartial<User>;
// { name?: string; age?: number; email?: string; }

type ReadonlyUser = MyReadonly<User>;
// { readonly name: string; readonly age: number; readonly email: string; }
```

### 修飾子の追加と除去

```typescript
// readonlyを除去（-readonly）
type Mutable<T> = {
  -readonly [K in keyof T]: T[K];
};

// オプショナルを除去（Requiredの実装）
type MyRequired<T> = {
  [K in keyof T]-?: T[K];
};

interface ReadonlyConfig {
  readonly host: string;
  readonly port: number;
}

type MutableConfig = Mutable<ReadonlyConfig>;
// { host: string; port: number; } — readonlyが外れた
```

### キーのリマッピング（as 句）

TypeScript 4.1 以降では、キーをリマッピングできます。

```typescript
// プロパティ名にプレフィックスを付ける
type Getters<T> = {
  [K in keyof T as `get${Capitalize<string & K>}`]: () => T[K];
};

interface Person {
  name: string;
  age: number;
}

type PersonGetters = Getters<Person>;
// { getName: () => string; getAge: () => number; }
```

### ポイントまとめ

- マップ型は `{ [K in keyof T]: ... }` で既存の型を変換する
- `?`, `readonly` の追加（`+`）と除去（`-`）ができる
- `as` 句でキー名のリマッピングが可能

## 5.8 よくある間違い

### 1. 不必要なジェネリクス

```typescript
// ❌ 悪い例: ジェネリクスが不要（Tが1箇所でしか使われていない）
function greet<T extends string>(name: T): void {
  console.log(`Hello, ${name}`);
}

// ✅ 良い例: 単にstringでよい
function greet2(name: string): void {
  console.log(`Hello, ${name}`);
}
```

**ルール:** 型パラメータが1箇所でしか使われない場合、ジェネリクスは不要です。型パラメータは「複数の場所の型を関連付ける」ために存在します。

### 2. 制約の過不足

```typescript
// ❌ 制約が緩すぎる
function processItems<T>(items: T[]): void {
  items.forEach((item) => {
    console.log(item.toString()); // anyのようなもの — 意味のある操作ができない
  });
}

// ✅ 適切な制約
interface Displayable {
  display(): string;
}

function processItems2<T extends Displayable>(items: T[]): void {
  items.forEach((item) => {
    console.log(item.display()); // Displayableの機能を安全に使える
  });
}
```

### 3. any の代わりにジェネリクスを使うべき場面を見逃す

```typescript
// ❌ anyを使ってしまう
function merge(obj1: any, obj2: any): any {
  return { ...obj1, ...obj2 };
}

// ✅ ジェネリクスで型安全に
function merge2<T, U>(obj1: T, obj2: U): T & U {
  return { ...obj1, ...obj2 };
}

const merged = merge2({ name: "太郎" }, { age: 25 });
// { name: string; age: number } — 型が正しく推論される
```

## 5.9 現場でのベストプラクティス

### 1. シンプルに保つ

ジェネリクスは強力ですが、過度に使うとコードが読みにくくなります。

```typescript
// ❌ 複雑すぎるジェネリクス
type DeepReadonly<T> = T extends (infer U)[]
  ? ReadonlyArray<DeepReadonly<U>>
  : T extends object
  ? { readonly [K in keyof T]: DeepReadonly<T[K]> }
  : T;

// ✅ まずは簡単なReadonlyで十分なことが多い
type Config = Readonly<{
  host: string;
  port: number;
}>;
```

### 2. 型パラメータは必要な場所でのみ

```typescript
// ✅ 型パラメータが入力と出力を関連付けている — 意味がある
function map<T, U>(arr: T[], fn: (item: T) => U): U[] {
  return arr.map(fn);
}

// ✅ 型パラメータがオブジェクトのキーを制約している — 意味がある
function pick<T, K extends keyof T>(obj: T, keys: K[]): Pick<T, K> {
  const result = {} as Pick<T, K>;
  keys.forEach((key) => {
    result[key] = obj[key];
  });
  return result;
}
```

### 3. 型推論を活用する

```typescript
// ❌ 型引数を毎回指定するのは冗長
const nums = identity<number>(42);
const strs = identity<string>("hello");

// ✅ 型推論に任せる
const nums2 = identity(42);      // number と推論される
const strs2 = identity("hello"); // string と推論される
```

## 5.10 まとめ

| トピック | 内容 |
|---------|------|
| ジェネリクスとは | 型をパラメータ化して再利用性を高める仕組み |
| ジェネリック関数 | `function name<T>()` で型パラメータを宣言 |
| 制約 | `T extends X` で型パラメータの範囲を限定 |
| ジェネリック型 | インターフェース・型エイリアス・クラスにも適用可能 |
| 条件型 | `T extends U ? X : Y` で型レベルの分岐 |
| infer | 条件型の中で型を推論・抽出する |
| マップ型 | 既存の型を変換して新しい型を生成する |

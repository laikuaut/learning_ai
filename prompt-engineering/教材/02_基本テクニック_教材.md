# 第2章：基本テクニック

## この章のゴール

- プロンプトの基本構造（役割・指示・文脈・出力形式）を理解する
- 明確な指示の書き方を習得する
- ペルソナ設定の効果と使い方を学ぶ
- 出力形式の指定方法を習得する
- ステップバイステップ指示の効果を理解する

---

## 2.1 プロンプトの基本構造

### 4つの構成要素

効果的なプロンプトは、以下の4つの要素で構成されます。すべてが必須ではありませんが、組み合わせることで出力品質が大幅に向上します。

```
プロンプトの4要素：

┌─────────────────────────────────────┐
│ 1. 役割（Role）                      │
│    → LLMにどのような立場で回答するか  │
│    例：「あなたはシニアPythonエンジニアです」│
├─────────────────────────────────────┤
│ 2. 指示（Instruction）               │
│    → 何をしてほしいか                 │
│    例：「以下のコードをレビューしてください」│
├─────────────────────────────────────┤
│ 3. 文脈（Context）                   │
│    → 判断に必要な背景情報             │
│    例：「このコードはECサイトの決済処理です」│
├─────────────────────────────────────┤
│ 4. 出力形式（Output Format）          │
│    → どのような形で返してほしいか     │
│    例：「箇条書きで3つの改善点を挙げて」│
└─────────────────────────────────────┘
```

### 実際のプロンプト例

```
あなたはセキュリティに詳しいシニアエンジニアです。   ← 役割

以下のPythonコードのセキュリティ上の問題点を指摘し、 ← 指示
改善案を提示してください。

このコードはECサイトのユーザー認証処理で、           ← 文脈
本番環境で稼働しています。

問題点ごとに以下の形式で回答してください：           ← 出力形式
- 問題点：
- リスクレベル：高/中/低
- 改善案：
- 改善後のコード：
```

> **ポイントまとめ**: プロンプトは「役割 → 指示 → 文脈 → 出力形式」の順で構成すると整理しやすくなります。すべての要素を毎回使う必要はなく、タスクの複雑さに応じて組み合わせてください。

## 2.2 明確な指示の書き方

### 曖昧な指示 vs 明確な指示

LLMは指示が曖昧だと、意図と異なる回答を返すことがあります。具体的で明確な指示を心がけましょう。

```
❌ 曖昧な指示：
「このコードを良くしてください」
→ 何を「良く」するのか不明（性能？可読性？セキュリティ？）

✅ 明確な指示：
「このコードの可読性を改善してください。
 具体的には：
 1. 変数名を意味のある名前に変更する
 2. 長い関数を小さな関数に分割する
 3. 適切なコメントを追加する」
```

### 明確な指示を書くための5つのルール

| ルール | 悪い例 | 良い例 |
|--------|--------|--------|
| **具体的な動詞を使う** | 「考えてください」 | 「3つの選択肢を比較し、推奨案を1つ選んでください」 |
| **数量を指定する** | 「いくつか挙げて」 | 「5つ挙げてください」 |
| **対象を限定する** | 「プログラミングについて」 | 「Python初心者がつまずきやすい点について」 |
| **条件を明示する** | 「短くまとめて」 | 「200文字以内でまとめてください」 |
| **否定より肯定で指示** | 「難しい言葉を使わないで」 | 「中学生でも理解できる言葉で説明してください」 |

### 区切り記号の活用

入力データとプロンプトの指示を明確に区別するために、区切り記号を使います。

```
以下のテキストを日本語に翻訳してください。
原文の意味を正確に保ちつつ、自然な日本語にしてください。

---
テキスト:
"""
The quick brown fox jumps over the lazy dog.
This sentence contains every letter of the alphabet.
"""
---

出力形式：翻訳結果のみを出力してください。
```

よく使われる区切り記号：

| 記号 | 用途 |
|------|------|
| `"""..."""` | テキストデータの囲み |
| `---` | セクションの区切り |
| `###` | 見出しとしての区切り |
| `<tag>...</tag>` | XMLタグによる構造化 |
| `` ``` `` | コードブロック |

> **ポイントまとめ**: 曖昧さを排除し、LLMが「何を、どの程度、どの形式で」行うべきか迷わない指示を書きましょう。

## 2.3 ペルソナ設定

### ペルソナ設定とは

LLMに特定の役割・専門性を持つ人物として回答させるテクニックです。ペルソナを設定することで、回答の専門性・深さ・トーンが変化します。

### ペルソナ設定の効果

```
ペルソナなし：
「Pythonのデコレータについて教えてください」
→ 一般的な説明が返る

ペルソナあり：
「あなたは10年以上の経験を持つPythonエンジニアで、
 大規模Webアプリケーションの設計を専門としています。
 Pythonのデコレータについて、実務での活用パターンと
 注意点を教えてください。」
→ 実務寄りの具体的なパターンと注意点が返る
```

### 効果的なペルソナの設定方法

| 要素 | 説明 | 例 |
|------|------|-----|
| **専門分野** | どの領域の専門家か | セキュリティ、フロントエンド、DB設計 |
| **経験年数** | どの程度の経験があるか | 10年以上、ジュニアレベル |
| **対象読者** | 誰に向けて話すか | 初心者、経営層、同僚エンジニア |
| **トーン** | どのような口調で話すか | 丁寧、カジュアル、学術的 |

### ペルソナの実例集

```
■ コードレビュー用：
「あなたはGoogleのStaff Engineerとして、コードレビューを
 担当しています。可読性、保守性、パフォーマンスの観点から
 レビューしてください。」

■ 初心者向け教育用：
「あなたはプログラミングスクールの人気講師です。
 プログラミング未経験の受講生に対して、
 身近な例えを使いながら分かりやすく説明してください。」

■ ビジネス文書作成用：
「あなたは外資系IT企業の事業部長です。
 経営会議で使用するプレゼン資料の骨子を作成してください。
 データに基づいた論理的な構成を心がけてください。」

■ テスト設計用：
「あなたはQAリードとして、5年以上のテスト設計経験があります。
 境界値分析と同値分割に基づいてテストケースを設計してください。」
```

> **よくある間違い**: ペルソナを設定しすぎて矛盾した指示を出さないこと。例えば「初心者向けに説明しつつ、高度な数式を使ってください」のような矛盾は避けましょう。

> **ポイントまとめ**: ペルソナ設定はLLMの回答の質を向上させる強力なテクニックです。目的に合った専門性・経験・トーンを設定しましょう。

## 2.4 出力形式の指定

### なぜ出力形式を指定するのか

出力形式を指定しないと、LLMは自由な形式で回答します。後続の処理（パース、表示、レポート作成等）を考慮し、必要な形式を明示しましょう。

### 主な出力形式の指定方法

#### 1. JSON形式

```
以下の文章から人物情報を抽出し、JSON形式で出力してください。

文章：「田中太郎さん（35歳）は東京都在住のソフトウェアエンジニアで、
Python とTypeScript を主に使用しています。」

出力形式：
{
  "name": "名前",
  "age": 年齢（数値）,
  "location": "居住地",
  "occupation": "職業",
  "skills": ["スキル1", "スキル2"]
}
```

#### 2. マークダウン形式

```
以下のトピックについて解説記事を書いてください。

トピック：Git の基本的なワークフロー

出力形式：
- H2見出しで主要セクションを分ける
- 各セクションにコード例を含める
- 重要な用語は**太字**にする
- 注意点は引用ブロック（>）で表記する
```

#### 3. 表形式

```
以下のプログラミング言語を比較してください。

言語：Python, JavaScript, Go, Rust

出力形式：マークダウンの表で以下の列を含める
| 言語 | 型システム | 主な用途 | 学習コスト | 実行速度 |
```

#### 4. 箇条書き（構造付き）

```
このエラーの原因と解決策を分析してください。

出力形式：
1. エラーの原因（1〜2文で簡潔に）
2. 根本原因の分析（技術的な詳細）
3. 解決手順（番号付きステップ）
4. 再発防止策（箇条書き）
```

### 出力の制約を指定する

| 制約 | プロンプト例 |
|------|------------|
| 文字数制限 | 「200文字以内で要約してください」 |
| 項目数制限 | 「最も重要な3つだけ挙げてください」 |
| 言語指定 | 「英語で回答してください」 |
| 技術レベル | 「初心者向けに専門用語を避けて」 |
| 除外条件 | 「コード例は含めず、概念の説明のみ」 |

> **ポイントまとめ**: 出力形式を指定することで、回答の一貫性が保たれ、後続の処理がしやすくなります。JSON、表、箇条書き等、用途に応じた形式を指定しましょう。

## 2.5 ステップバイステップ指示

### なぜステップバイステップが効果的か

複雑なタスクを一度に依頼すると、LLMは重要なステップを飛ばしたり、順序が混乱したりすることがあります。タスクを明確なステップに分解して指示することで、精度が向上します。

```
❌ 一括指示：
「このCSVデータを分析してレポートを作成してください」
→ 何をどの順番で分析するか不明

✅ ステップバイステップ：
「以下のCSVデータを分析し、レポートを作成してください。

ステップ1：データの概要を確認する
  - 行数、列数、各列のデータ型を確認
  - 欠損値の有無を確認

ステップ2：基本統計量を算出する
  - 数値列の平均、中央値、標準偏差
  - カテゴリ列の値の分布

ステップ3：異常値を検出する
  - 平均から3σ以上離れたデータを特定

ステップ4：分析結果をまとめる
  - 主要な知見を3つ箇条書きで記述
  - 推奨アクションを提案」
```

### 段階的精緻化（Iterative Refinement）

1回のプロンプトで完璧な結果を得ようとせず、段階的に精度を上げていくアプローチです。

```
ステップ1（大枠）：
「ECサイトの商品レコメンド機能の設計案を考えてください」

ステップ2（詳細化）：
「先ほどの設計案のうち、協調フィルタリングのアプローチについて、
 具体的なアルゴリズムとデータ構造を詳しく説明してください」

ステップ3（実装）：
「そのアルゴリズムをPythonで実装してください。
 入力データのサンプルも含めてください」

ステップ4（改善）：
「実装したコードの性能を改善してください。
 大量データ（100万件）を扱う場合の最適化も含めて」
```

### 複雑な指示の分割テクニック

| テクニック | 説明 | 例 |
|-----------|------|-----|
| **番号付けステップ** | 手順を番号で明示 | 「ステップ1: ...、ステップ2: ...」 |
| **チェックリスト** | 確認事項を列挙 | 「以下の点をすべて確認してください」 |
| **テンプレート穴埋め** | 穴埋め形式で構造化 | 「原因：___、影響：___、対策：___」 |
| **条件分岐** | 場合分けを明示 | 「エラーAの場合は...、エラーBの場合は...」 |

> **ポイントまとめ**: 複雑なタスクは小さなステップに分解して指示しましょう。段階的に精緻化するアプローチも効果的です。

## 2.6 制約とガードレール

### ネガティブプロンプト（やらないことの指定）

LLMに「何をしないか」を伝えることで、不要な出力を防ぎます。

```
以下のコードのバグを修正してください。

制約：
- コードの修正のみ行い、新しい機能は追加しないでください
- 既存のインターフェースを変更しないでください
- 外部ライブラリの追加は行わないでください
- コメントは日本語で記述してください
```

### 回答の品質を上げるガードレール

```
以下の条件を守って回答してください：

1. 事実に基づいて回答し、不確かな場合は「確信度が低い」と明記する
2. 推測や仮定がある場合は、それが推測であることを明示する
3. 回答が長くなる場合は、最初に要約を1〜2行で述べる
4. コード例を含める場合は、動作確認済みのコードを提供する
5. 参考にすべき公式ドキュメントのURLがあれば言及する
```

> **ポイントまとめ**: 「やること」だけでなく「やらないこと」を明示することで、意図しない出力を防げます。品質に関するガードレールも設定しましょう。

## まとめ

```
基本テクニックの重要ポイント：

✅ プロンプトは「役割・指示・文脈・出力形式」の4要素で構成する
✅ 指示は具体的に：何を、どの程度、どの形式で
✅ ペルソナ設定で回答の専門性とトーンをコントロールする
✅ 出力形式（JSON、表、箇条書き等）を明示的に指定する
✅ 複雑なタスクはステップバイステップで分解する
✅ 段階的精緻化で1回のプロンプトに頼りすぎない
✅ 制約とガードレールで不要な出力を防ぐ
```

# 第6章 演習：業務プロセスへの統合

## はじめに

この演習では、生成AIを業務プロセスに統合するための判断力と設計力を養います。基本問題で業務適合性の評価とROI計算の基礎を学び、応用問題でRAGシステムやチャットボットの設計に取り組み、チャレンジ問題で架空企業への段階的導入計画を策定します。

## 基本問題

### 問題1：業務適合性の判断

以下の10個の業務について、生成AIの導入が **適している（○）**、**条件付きで適している（△）**、**適していない（×）** のいずれかに分類し、その理由を簡潔に述べてください。

```
a) 社内FAQ対応（よくある質問への回答）
b) 医療における最終診断
c) 議事録の要約作成
d) 顧客への契約書作成（法的拘束力あり）
e) マーケティングコピーの下書き作成
f) 会計監査の最終判定
g) 社内ブログ記事の執筆支援
h) 製品のキャッチコピー案の発散
i) 人事考課の最終評価
j) 翻訳の下訳作成
```

**期待される出力例：**

```
| 業務 | 判定 | 理由 |
|---|:---:|---|
| a) 社内FAQ対応 | ○ | 定型的な質問が多く、既存のナレッジベースをもとに正確に回答可能 |
| b) 医療における最終診断 | × | 人命に関わる判断であり、AIの誤りが重大な結果を招く |
| ... | ... | ... |
```

<details>
<summary>ヒント</summary>

判断基準として以下の3軸を使いましょう。
1. **リスクの大きさ**：誤りが発生した場合の影響度（人命・法的責任・財務損失）
2. **定型度**：タスクがどの程度パターン化されているか
3. **人間の判断の必要性**：倫理的判断や創造的判断がどの程度求められるか

</details>

<details>
<summary>解答例</summary>

```
| 業務 | 判定 | 理由 |
|---|:---:|---|
| a) 社内FAQ対応 | ○ | 定型的な質問と回答のパターンが明確。ナレッジベースと連携すれば高い精度で対応可能 |
| b) 医療における最終診断 | × | 人命に直結する判断。ハルシネーションのリスクがあり、医師法上も人間の責任が必要 |
| c) 議事録の要約作成 | ○ | 文章の要約はLLMの得意分野。人間が最終確認する前提で大幅な工数削減が可能 |
| d) 顧客への契約書作成 | △ | ドラフト作成には活用可能だが、法的拘束力のある文書は必ず法務担当者の確認が必要 |
| e) マーケティングコピーの下書き | ○ | アイデア出しと下書きに最適。複数案を短時間で生成し、人間が選別・修正する運用 |
| f) 会計監査の最終判定 | × | 法的責任を伴う専門的判断。監査基準に基づく人間の判断が不可欠 |
| g) 社内ブログ記事の執筆支援 | ○ | 構成案・下書き生成に有効。社内向けのため外部リスクも低い |
| h) 製品キャッチコピー案の発散 | ○ | 大量のアイデアを短時間で生成するのに最適。人間が最終選定する前提 |
| i) 人事考課の最終評価 | × | 公平性・法的責任の観点から人間の判断が必須。AIバイアスのリスクもある |
| j) 翻訳の下訳作成 | ○ | 下訳の品質は十分高く、人間が校正する前提で翻訳業務を大幅に効率化できる |

# 判断のポイント：
# ○ → 誤りの影響が小さく、人間のレビューで品質担保できる業務
# △ → 活用可能だが、専門家の確認が必須の業務
# × → 誤りの影響が甚大で、法的・倫理的に人間の判断が不可欠な業務
```

</details>

### 問題2：ROI計算の基礎

以下のシナリオについて、生成AI導入のROI（投資対効果）を計算してください。

```
【シナリオ：カスタマーサポートへのチャットボット導入】

現状コスト（年間）：
- オペレーター5名 × 年収400万円 = 2,000万円
- 対応件数：年間12,000件（月1,000件）
- 1件あたりの平均対応時間：15分

AI導入後の想定：
- AIが全体の60%の問い合わせを自動対応
- オペレーターは3名に削減可能（2名分のコスト削減）
- AI関連コスト（年間）：
  - APIコスト：月額20万円（年間240万円）
  - システム開発・保守：年間300万円
  - 運用担当者（兼務）：年間100万円

計算してください：
1. 年間のコスト削減額
2. 年間のAI導入コスト
3. 年間の純利益（ネットベネフィット）
4. ROI（%）
5. 投資回収期間（月数）
```

**期待される出力例：**

```
【ROI計算結果】
1. 年間コスト削減額：XXX万円
2. 年間AI導入コスト：XXX万円
3. 年間純利益：XXX万円
4. ROI：XX.X%
5. 投資回収期間：約XX ヶ月

【計算過程】
...
```

<details>
<summary>ヒント</summary>

ROIの計算式は以下の通りです。
- ROI（%）＝（純利益 ÷ 投資額）× 100
- 純利益 ＝ コスト削減額 − AI導入コスト
- 投資回収期間 ＝ 12ヶ月 ÷（年間純利益 ÷ 年間投資額 + 1）ではなく、投資額を月次の純利益で割って算出します。

</details>

<details>
<summary>解答例</summary>

```
【ROI計算結果】
1. 年間コスト削減額：800万円
2. 年間AI導入コスト：640万円
3. 年間純利益：160万円
4. ROI：25.0%
5. 投資回収期間：約48ヶ月（4年）

【計算過程】

1. 年間コスト削減額
   削減されるオペレーター人件費 = 2名 × 400万円 = 800万円

2. 年間AI導入コスト
   APIコスト          240万円
   システム開発・保守   300万円
   運用担当者（兼務）   100万円
   ────────────────────
   合計               640万円

3. 年間純利益（ネットベネフィット）
   800万円 − 640万円 = 160万円

4. ROI
   ROI = (160万円 ÷ 640万円) × 100 = 25.0%

5. 投資回収期間
   月次純利益 = 160万円 ÷ 12ヶ月 ≒ 13.3万円/月
   投資回収期間 = 640万円 ÷ 13.3万円 ≒ 48ヶ月（約4年）

【考察】
- ROI 25%は決して高くないが、初年度から黒字化する点はプラス材料です。
- 2年目以降はシステム開発費の大部分が不要になり（保守のみ）、
  ROIが大幅に改善する見込みがあります。
- 定量的な効果（コスト削減）以外にも、24時間対応が可能になる、
  対応品質の均一化、オペレーターが複雑な問い合わせに集中できる
  といった定性的なメリットも考慮すべきです。

# ポイント：ROI計算は導入判断の出発点です。
# 定量的な数値だけでなく、顧客満足度向上や従業員の業務品質向上など
# 定性的な効果もあわせて経営層に提示することが重要です。
```

</details>

### 問題3：業務プロセスの分析

以下の「経費精算業務フロー」について、生成AIで自動化・効率化できる工程を特定し、それぞれの改善案を提案してください。

```
【現状の経費精算業務フロー】
1. 申請者が領収書を撮影してアップロード
2. 申請者が経費精算書をExcelで作成（日付・金額・項目・摘要を手入力）
3. 上長がExcelファイルを目視確認して承認
4. 経理部が領収書と精算書を照合チェック
5. 経理部が会計システムに仕訳データを手入力
6. 月次で経費レポートを手動で作成
```

**期待される出力例：**

```
| 工程 | 現状の課題 | AI活用案 | 期待される効果 |
|---|---|---|---|
| 2. 精算書作成 | 手入力によるミス、入力工数 | OCR+LLMで領収書から自動入力 | 入力工数80%削減、ミス低減 |
| ... | ... | ... | ... |
```

<details>
<summary>ヒント</summary>

各工程で「人間がやっている作業」を分解し、その中で「パターン化できる作業」「判断が定型的な作業」を見つけましょう。すべての工程をAI化する必要はなく、人間の判断が重要な工程は「AIが補助」する設計が現実的です。

</details>

<details>
<summary>解答例</summary>

```
| 工程 | 現状の課題 | AI活用案 | 期待される効果 |
|---|---|---|---|
| 1. 領収書撮影 | 画質不良、撮り直し発生 | AI画像判定で撮影品質を即時フィードバック | 差し戻し件数50%削減 |
| 2. 精算書作成 | 手入力ミス、入力に平均15分 | OCR+LLMで領収書画像から日付・金額・項目を自動抽出し精算書を自動生成 | 入力工数80%削減、転記ミスほぼゼロ |
| 3. 上長承認 | 確認項目が多く見落としリスク | LLMが事前チェック（金額上限・項目妥当性）し、異常値のみハイライト表示 | 確認時間60%短縮、見落とし防止 |
| 4. 照合チェック | 1件あたり5分の照合作業 | AIが領収書画像と精算書データを自動照合、不一致箇所のみ経理に通知 | 照合工数70%削減 |
| 5. 仕訳入力 | 手入力、勘定科目の判断 | LLMが摘要から勘定科目を自動推定し、仕訳データを自動生成 | 入力工数90%削減、科目ミス低減 |
| 6. 月次レポート | データ集計と報告書作成に丸1日 | LLMがデータを分析し、傾向・異常値を含む経費レポートを自動生成 | 作成時間80%短縮、分析の質向上 |

【改善後の業務フロー】
1. 申請者が領収書を撮影 → AIが画質チェック → OK/撮り直し指示
2. AIがOCRで領収書を読み取り → 精算書を自動生成 → 申請者が確認・修正
3. AIが事前チェック → 異常値をハイライト → 上長が効率的に承認
4. AIが自動照合 → 不一致のみ経理担当者に通知 → 人間が最終確認
5. AIが仕訳データを自動生成 → 経理担当者が確認 → 会計システムに連携
6. AIが月次経費レポートを自動生成 → 経理部長がレビュー

# ポイント：すべてを完全自動化するのではなく、各工程で「AIが下準備 → 人間が確認」
# という構成にすることで、精度を担保しつつ大幅な効率化を実現できます。
# 特に会計処理はミスの影響が大きいため、人間のレビュー工程は必ず残しましょう。
```

</details>

## 応用問題

### 問題4：RAG活用のシステム設計

あなたは社内ナレッジ管理システムの刷新プロジェクトのリーダーです。RAG（Retrieval-Augmented Generation：検索拡張生成）を活用して、社内文書を横断検索・質問応答できるシステムを設計してください。

以下の条件で設計書を作成してください。

```
【前提条件】
- 従業員数：500名
- 対象文書：社内規程（50文書）、技術マニュアル（200文書）、
  過去の議事録（3,000件）、FAQ（500件）
- 利用想定：1日あたり200件の質問
- セキュリティ要件：社内情報は外部に送信しない
- 予算：初期費用500万円以内、月額運用費50万円以内
```

設計書には以下を含めてください。

1. システム構成図（テキストで記述）
2. 使用する技術スタック（LLM、ベクトルDB、フレームワーク等）
3. データ前処理のパイプライン
4. セキュリティ対策
5. 評価指標（KPI）

**期待される出力例：**

```
【システム構成図】
┌─────────────┐    ┌──────────────┐    ┌─────────────┐
│  ユーザーUI  │───→│  APIサーバー  │───→│    LLM      │
│（Webアプリ） │    │（検索+生成）  │    │（ローカル）  │
└─────────────┘    └──────┬───────┘    └─────────────┘
                          │
                   ┌──────┴───────┐
                   │ ベクトルDB    │
                   │（文書インデックス）│
                   └──────────────┘
...
```

<details>
<summary>ヒント</summary>

RAGシステムの基本構成は「検索（Retrieval）」と「生成（Generation）」の2段階です。セキュリティ要件で「社内情報は外部に送信しない」とあるため、LLMはオンプレミスまたはプライベートクラウドで動かす必要があります。オープンソースLLM（Llama系など）の採用を検討しましょう。

</details>

<details>
<summary>解答例</summary>

```
【1. システム構成図】

┌──────────────────────────────────────────────────────┐
│                    社内ネットワーク                      │
│                                                        │
│  ┌──────────┐    ┌───────────────┐    ┌────────────┐  │
│  │ ユーザーUI │───→│  APIサーバー   │───→│ ローカルLLM │  │
│  │（React）  │←───│（FastAPI）    │←───│（Llama 3） │  │
│  └──────────┘    └───────┬───────┘    └────────────┘  │
│                          │                              │
│          ┌───────────────┼───────────────┐              │
│          ↓               ↓               ↓              │
│  ┌──────────────┐ ┌────────────┐ ┌──────────────┐     │
│  │ ベクトルDB    │ │ 文書ストア  │ │ ログDB       │     │
│  │（Milvus）    │ │（MinIO）   │ │（PostgreSQL）│     │
│  └──────────────┘ └────────────┘ └──────────────┘     │
│                                                        │
│  ┌──────────────────────────────┐                      │
│  │ データ前処理パイプライン        │                      │
│  │（文書取込 → チャンク分割       │                      │
│  │  → エンベディング → インデックス） │                    │
│  └──────────────────────────────┘                      │
└──────────────────────────────────────────────────────┘

【2. 技術スタック】
| コンポーネント | 技術選定 | 選定理由 |
|---|---|---|
| LLM | Llama 3 70B（ローカル） | オープンソースで社内完結、日本語性能も良好 |
| エンベディング | multilingual-e5-large | 日本語対応の高精度埋め込みモデル |
| ベクトルDB | Milvus | オープンソース、大規模対応、高速検索 |
| APIサーバー | FastAPI（Python） | 非同期対応、LangChainとの親和性が高い |
| フレームワーク | LangChain | RAGパイプラインの構築が容易 |
| フロントエンド | React + TypeScript | 社内の技術スタックと統一 |
| 文書ストレージ | MinIO | S3互換のオンプレミスオブジェクトストレージ |
| ログ・分析 | PostgreSQL + Grafana | 利用状況の可視化と改善サイクル |
| GPU サーバー | NVIDIA A100 × 1 | Llama 3 70Bの推論に必要 |

【3. データ前処理パイプライン】
Step 1：文書収集
  - 社内規程（PDF） → PDFパーサーでテキスト抽出
  - 技術マニュアル（Word/PDF） → テキスト抽出 + 図表のキャプション抽出
  - 議事録（テキスト/Word） → そのまま取り込み
  - FAQ（スプレッドシート） → Q&Aペアとして構造化

Step 2：前処理
  - 不要なヘッダー・フッター・ページ番号の除去
  - 表形式データのマークダウン変換
  - 文書メタデータの付与（作成日・部門・カテゴリ・機密レベル）

Step 3：チャンク分割
  - チャンクサイズ：512トークン、オーバーラップ：64トークン
  - セクション単位での意味的分割を優先（RecursiveCharacterTextSplitter）
  - FAQ はQ&Aペア単位で1チャンク

Step 4：エンベディング生成
  - multilingual-e5-large でベクトル化
  - バッチ処理で全文書を一括インデックス（初回約6時間想定）

Step 5：インデックス登録
  - Milvus にベクトルとメタデータを登録
  - 文書更新時は差分のみ再インデックス（日次バッチ）

【4. セキュリティ対策】
- ネットワーク：全コンポーネントを社内ネットワーク内に配置。外部通信を遮断
- 認証・認可：社内Active Directoryと連携したSSO。部門別のアクセス権限制御
- データ分類：文書の機密レベルに応じた検索範囲制限（一般/部門限定/経営層限定）
- ログ管理：全質問・回答をログに記録。個人情報を含む質問の検知と警告
- モデル管理：LLMはオンプレミスで稼働。モデルの更新時は社内検証環境でテスト後に本番適用
- 出力制御：回答に必ず参照元文書を表示し、ハルシネーションの検証を容易にする

【5. 評価指標（KPI）】
| 指標 | 目標値 | 測定方法 |
|---|---|---|
| 回答精度（Accuracy） | 85%以上 | 月次サンプリング100件の人間評価 |
| 検索適合率（Precision@5） | 80%以上 | 上位5件の検索結果の関連度を評価 |
| 応答時間 | 5秒以内 | APIレスポンスタイムの計測 |
| 利用率 | 1日200件以上 | ログ集計 |
| ユーザー満足度 | 4.0/5.0以上 | 月次アンケート |
| 問い合わせ削減率 | 情シス部門への問い合わせ30%削減 | ヘルプデスクチケット数の比較 |

# ポイント：RAGシステムの成否は「検索精度」に大きく依存します。
# チャンク分割の粒度やエンベディングモデルの選定を慎重に行い、
# 定期的に精度を評価して改善サイクルを回すことが重要です。
```

### 問題5：チャットボット設計

ECサイト（アパレル）のカスタマーサポート用チャットボットを、生成AIを活用して構築します。以下の設計書を作成してください。

```
【要件】
- 対応範囲：注文状況確認、返品・交換手続き、サイズ相談、在庫確認、一般FAQ
- 営業時間外も24時間対応
- 対応できない場合は人間のオペレーターにエスカレーション
- 月間想定問い合わせ数：3,000件
- ターゲットユーザー：20〜40代、スマートフォン利用が中心
```

設計書には以下を含めてください。

1. 会話フローの設計（主要な3パターン）
2. プロンプトテンプレート（システムプロンプト）
3. エスカレーション条件の定義
4. 効果測定の方法

**期待される出力例：**

```
【会話フロー1：注文状況確認】
ユーザー：「注文した商品はいつ届きますか？」
    ↓
ボット：注文番号の確認を依頼
    ↓
```

## まとめ

この演習を通じて、以下のスキルが身についたか確認しましょう。

- 業務への生成AI導入の適合性を多角的に判断できる

- RAGシステムやチャットボットの技術的な設計ができる
- 組織全体への段階的な導入計画を策定できる
- チェンジマネジメントやリスク管理を含む包括的な計画が立てられる

## 基本問題

### 問題1：業務自動化の適性判断

以下のタスクについて、LLMによる自動化に「適している」「部分的に適している」「適していない」を判断し、理由を述べてください。

1. Jiraチケットの説明文からバグの重要度を自動分類する
2. 医療診断システムの最終判定を行う
3. プルリクエストのタイトルと説明文を自動生成する
4. 顧客情報データベースから個人情報を検索して返す
5. 会議の録音テキストから議事録を自動生成する
6. セキュリティアラートの即時遮断判定を行う

<details>
<summary>解答例</summary>

```
1. Jiraチケットのバグ重要度分類 → 適している ✅
   理由：テキスト分類タスクでLLMが得意。入力（チケット文）と
   出力（重要度ラベル）が明確。人間が最終確認可能。

2. 医療診断の最終判定 → 適していない ❌
   理由：誤判定が人命に関わる。LLMはハルシネーションのリスクがあり、
   医療判断には使えない。補助ツール（候補の提示）としてなら検討可能。

3. PRタイトル・説明文の自動生成 → 適している ✅
   理由：git diffからの情報抽出と要約タスク。
   開発者が確認・修正できるため、100%の精度は不要。

4. 個人情報の検索・返却 → 適していない ❌
   理由：個人情報の取り扱いはプライバシーとセキュリティの問題。
   LLMを介す必要がなく、データベースクエリで直接処理すべき。

5. 会議録音からの議事録生成 → 部分的に適している △
   理由：文字起こし+要約はLLMの得意分野。ただし、固有名詞の
   聞き取り精度や機密情報の取り扱いに課題がある。
   人間の確認・修正が必須。

6. セキュリティアラートの即時遮断判定 → 適していない ❌
   理由：リアルタイム性が求められ、LLMのレスポンス時間では不足。
   誤判定のリスクが高く、ルールベースのシステムが適切。
   事後分析にはLLMを活用可能。
```
</details>

### 問題2：文章校正プロンプトの作成

以下のビジネスメールの問題点を指摘し、改善版を作成するプロンプトを書いてください。

**校正対象メール：**
```
件名：例の件

田中様

お疲れ様です。
先日の会議の件ですが、あの機能については、
できれば来週中にお返事いただけるとたすかります。
よろしくお願い致します。
```

<details>
<summary>解答例</summary>

```
■ プロンプト：

「以下のビジネスメールを校正し、改善してください。

校正観点：
1. 件名：具体的で内容が分かるか
2. 曖昧な指示語：「あの」「例の」等を具体的な表現に
3. 敬語：正しい敬語表現か
4. アクションアイテム：期限と依頼内容が明確か
5. 全体構成：PREP法（結論→理由→詳細→結論）になっているか

出力形式：
1. 問題点の一覧（修正前 → 修正後）
2. 改善後のメール全文

校正対象：
"""
件名：例の件

田中様

お疲れ様です。
先日の会議の件ですが、あの機能については、
できれば来週中にお返事いただけるとたすかります。
よろしくお願い致します。
"""」

■ 期待される出力：

問題点：
1. 件名「例の件」→ 具体性がない
   修正：「【ご確認依頼】検索機能の実装方針について（3/28まで）」

2. 「先日の会議」→ いつの会議か不明
   修正：「3月20日のプロジェクト定例会議」

3. 「あの機能」→ 何の機能か不明
   修正：「商品検索機能のフィルタリング方式」

4. 「できれば来週中に」→ 期限が曖昧
   修正：「3月28日（金）までに」

5. 「お返事」→ 何について返事が必要か不明
   修正：「実装方針A/Bのいずれかをご選択ください」

6. 「たすかります」→ 漢字の方が適切
   修正：「助かります」

改善後：
"""
件名：【ご確認依頼】検索機能の実装方針について（3/28まで）

田中様

お疲れ様です。山田です。

3月20日のプロジェクト定例会議でご検討いただいた、
商品検索機能のフィルタリング方式について、
実装方針のご判断をお願いいたします。

ご検討いただきたい内容：
・方針A：サーバーサイドフィルタリング（精度重視）
・方針B：クライアントサイドフィルタリング（速度重視）

お手数ですが、3月28日（金）までにご回答いただけますと
助かります。ご不明点があればお気軽にご連絡ください。

よろしくお願いいたします。
"""
```
</details>

### 問題3：データ分析プロンプトの作成

以下のCSVデータを分析するプロンプトを作成してください。分析結果からビジネス上の示唆（アクション）まで導出させるようにしてください。

```csv
月,新規登録数,退会数,月末会員数,売上(万円)
1月,450,80,5370,2148
2月,380,95,5655,2262
3月,520,70,6105,2442
4月,410,120,6395,2558
5月,350,150,6595,2638
6月,290,180,6705,2682
```

<details>
<summary>解答例</summary>

```
以下のSaaSサービスのユーザーデータを分析してください。

## データ
"""
月,新規登録数,退会数,月末会員数,売上(万円)
1月,450,80,5370,2148
2月,380,95,5655,2262
3月,520,70,6105,2442
4月,410,120,6395,2558
5月,350,150,6595,2638
6月,290,180,6705,2682
"""

## 分析項目

### 1. 基本統計
- 各指標の平均、最大、最小、トレンド（増加/減少/横ばい）

### 2. 重要KPIの計算
- 月次チャーンレート（退会率）= 退会数 / 前月末会員数
- ARPU（ユーザーあたり売上）= 売上 / 月末会員数
- ネット増加数 = 新規登録数 - 退会数
- MoM成長率（月次成長率）

### 3. トレンド分析
- 新規登録数のトレンドは？
- 退会数のトレンドは？
- この傾向が続くと何が起きるか？

### 4. アラート
- 健全性に懸念がある指標を特定

### 5. ビジネス上の示唆
- 今すぐ取るべきアクション（3つ以内）
- 根拠となるデータ

## 出力形式
- 各分析項目について見出し付きで記述
- 数値は表形式で表示
- 最後にエグゼクティブサマリー（3行以内）
```

### 問題4：マルチモーダルプロンプトの設計

あなたのチームでは、UIデザインのレビュープロセスにLLMを活用したいと考えています。スクリーンショットを入力として受け取り、UIの品質をレビューするプロンプトを設計してください。

レビュー対象：ECサイトの商品詳細ページ

<details>
<summary>解答例</summary>

```
あなたはUI/UXデザインの専門家です。
添付されたECサイトの商品詳細ページのスクリーンショットを
レビューしてください。

## レビュー観点

### 1. 情報設計（5段階評価）
- 購入に必要な情報（価格、在庫、配送情報）が見やすい位置にあるか
- 情報の優先順位が適切か（ユーザーが最初に見るべき情報が目立つか）
- 購入ボタン（CTA）の視認性は十分か

### 2. ユーザビリティ（5段階評価）
- 直感的に操作方法が分かるか
- タップ/クリック可能な要素が分かりやすいか
- エラー防止の仕組みがあるか（数量選択の制限等）

### 3. ビジュアルデザイン（5段階評価）
- 配色の統一感があるか
- フォントサイズの階層が適切か
- 余白のバランスは適切か

### 4. アクセシビリティ（5段階評価）
- テキストと背景のコントラスト比は十分か
- フォントサイズは読みやすいか
- ボタンサイズはタッチ操作に十分か（最低44px推奨）

### 5. コンバージョン最適化
- 購入の障壁になりそうな要素はないか
- 信頼性を高める要素（レビュー、保証等）は含まれているか

## 出力形式
| 観点 | スコア(1-5) | 良い点 | 改善点 |

最後に：
- 最も優先度の高い改善点を3つ
- 改善の具体的な方法（ワイヤーフレームレベルの説明）
```
</details>

## チャレンジ問題

### 問題5：プロンプトテンプレートライブラリの設計

あなたのエンジニアチーム（10名）で共有するプロンプトテンプレートライブラリを設計してください。

要件：
- 日常的に使う5つのタスク用テンプレートを作成する
- 各テンプレートにカスタマイズ可能なパラメータを含める
- 使い方のガイドを付ける
- 品質基準を定義する

<details>
<summary>解答例</summary>

```
# プロンプトテンプレートライブラリ

## 概要
エンジニアチームで日常的に使うプロンプトのテンプレート集です。
パラメータ（{中括弧}で囲まれた部分）を状況に合わせて
カスタマイズして使用してください。

## テンプレート一覧

### 1. code-review（コードレビュー）
用途：プルリクエストのコードレビュー支援
パラメータ：{language}, {code}, {context}

```
あなたは{language}のシニアエンジニアです。
以下のコードをレビューしてください。

コンテキスト：{context}

コード：
```{language}
{code}
```

レビュー項目：
□ バグ・論理エラー
□ セキュリティ
□ パフォーマンス
□ 可読性・命名
□ エラーハンドリング

各指摘：重要度(高/中/低)、問題、改善案、改善コード
```

### 2. bug-analysis（バグ分析）
用途：エラーメッセージの原因特定と解決
パラメータ：{error_message}, {language}, {context}

```
以下のエラーを分析してください。

言語：{language}
状況：{context}
エラー：
"""
{error_message}
"""

回答形式：
1. 深刻度（🔴高/🟡中/🟢低）
2. 原因（1〜2文）
3. 解決手順（番号付き）
4. 修正コード例
5. 再発防止策
```

### 3. doc-generate（ドキュメント生成）
用途：コードからドキュメントを自動生成
パラメータ：{code}, {doc_type}

```
以下のコードから{doc_type}を生成してください。

コード：
```
{code}
```

出力形式：{doc_type}の標準フォーマット
制約：コードの動作を正確に反映すること
```

doc_typeの選択肢：
- "API仕様書"（OpenAPI/Swagger形式）
- "README"（セットアップ手順、使い方）
- "JSDoc/docstring"（関数の説明）

### 4. meeting-summary（会議要約）
用途：会議メモからの構造化された議事録生成
パラメータ：{meeting_notes}, {meeting_type}

```
以下の会議メモから議事録を作成してください。
会議種別：{meeting_type}

メモ：
"""
{meeting_notes}
"""

出力形式：
## 議事録
- 日時：
- 参加者：
- 決定事項：（箇条書き）
- アクションアイテム：
  | 担当 | タスク | 期限 |
- 次回予定：
```

### 5. test-generate（テスト生成）
用途：関数からテストケースを自動生成
パラメータ：{code}, {language}, {framework}

```
以下の{language}の関数に対して、{framework}を使った
テストケースを生成してください。

関数：
```{language}
{code}
```

含めるテストケース：
- 正常系（2〜3件）
- 異常系（2〜3件）
- 境界値（1〜2件）

各テストに日本語のコメントで「何をテストしているか」を記載。
```

## 品質基準
- テンプレート使用時、出力を必ず人間が確認する
- コード生成は実際に動作確認してからコミットする
- 月1回、テンプレートの有効性をチームでレビューする
- 改善提案はPRで受け付ける

## 使い方
1. テンプレートを選ぶ
2. {パラメータ}を実際の値に置き換える
3. LLMに投入する
4. 出力を確認・修正する
5. 品質に問題があればテンプレートの改善をPRで提案
```
</details>

> **難易度**: ★★☆☆☆（基礎）

> **学習目標**: Zero-Shot、One-Shot、Few-Shotの使い分けを理解し、効果的な例文（exemplar）を設計できるようになる

## 課題の要件

以下の5つのタスクについて、Zero-Shot（例なし）・One-Shot（例1つ）・Few-Shot（例3つ）の3パターンでプロンプトを設計し、出力品質の違いを比較してください。

### タスク一覧

| タスク | 内容 | 入力例 |
|--------|------|--------|
| 1 | テキストの感情分析（ポジティブ/ネガティブ/ニュートラル） | 商品レビュー |
| 2 | テキストのカテゴリ分類（ビジネス/技術/生活/エンタメ） | ニュース見出し |
| 3 | 文体の変換（カジュアル → フォーマル） | 日常会話文 |
| 4 | 情報の抽出（人名・日付・場所を抽出） | イベント告知文 |
| 5 | 要約の生成（1文要約） | 3〜5文の短い文章 |

### 各タスクで実施すること

1. **Zero-Shotプロンプト**: 例を含めず指示のみで作成
2. **One-Shotプロンプト**: 入力と期待出力のペアを1組含める
3. **Few-Shotプロンプト**: 入力と期待出力のペアを3組含める（多様性を意識）
4. 各パターンに同じテスト入力を与え、出力を比較する
5. 「どのパターンが最も良い結果だったか」と「その理由」を記録する

## ステップガイド

<details>
<summary>ステップ1：Zero-Shotプロンプトを作成する</summary>

Zero-Shot（ゼロショット）は例を一切含めず、指示だけでタスクを実行させるアプローチです。

**タスク1（感情分析）のZero-Shot例**:
```
以下のテキストの感情を「ポジティブ」「ネガティブ」「ニュートラル」の
いずれかに分類してください。感情ラベルのみを出力してください。

テキスト：この商品は思った以上に使いやすくて、毎日愛用しています。
```

ポイント：
- 分類ラベルを明確に列挙する
- 「ラベルのみを出力」と指定して余計な説明を抑制する
- テスト入力は「明確にポジティブ」「微妙なケース」「ネガティブ」を用意する

</details>

<details>
<summary>ステップ2：One-Shotプロンプトを作成する</summary>

One-Shot（ワンショット）は1つの例を提示して、出力のパターンを示します。

**タスク1（感情分析）のOne-Shot例**:
```
以下のテキストの感情を分類してください。

例：
テキスト：注文から届くまでがとても早くて助かりました。
感情：ポジティブ

では、以下のテキストを分類してください。
テキスト：この商品は思った以上に使いやすくて、毎日愛用しています。
感情：
```

ポイント：
- 例の入力と出力を明確に区別する（ラベル「テキスト：」「感情：」を使用）
- 最後に「感情：」と書いて途中で止めることで、LLMに続きを生成させる
- 例は分かりやすいものを選ぶ（境界的なケースは避ける）

</details>

<details>
<summary>ステップ3：Few-Shotプロンプトを作成する</summary>

Few-Shot（フューショット）は3つの例を提示します。**例の多様性**が重要です。

**タスク1（感情分析）のFew-Shot例**:
```
以下のテキストの感情を分類してください。

例1：
テキスト：注文から届くまでがとても早くて助かりました。
感情：ポジティブ

例2：
テキスト：商品の説明と実物が違っていて、返品を考えています。
感情：ネガティブ

例3：
テキスト：色は3種類あり、サイズはS・M・Lから選べます。
感情：ニュートラル

では、以下のテキストを分類してください。
テキスト：この商品は思った以上に使いやすくて、毎日愛用しています。
感情：
```

**良い例の選び方**:
1. 全分類ラベルを網羅する（ポジティブ・ネガティブ・ニュートラル各1つ）
2. 境界的なケースではなく、明確な例を選ぶ
3. 長さや形式を揃える（一貫性）
4. テスト入力と重複しない例を使う

**よくある間違い**:
- 全例が同じラベル（ポジティブばかり3つ）→ バイアスが生じる
- 例が長すぎてトークンを浪費する
- 例のフォーマットが不統一

</details>

<details>
<summary>ステップ4：比較結果を整理する</summary>

5タスクの結果を以下の表で整理しましょう。

```markdown
| タスク | Zero-Shot精度 | One-Shot精度 | Few-Shot精度 | 最適パターン |
|--------|:------------:|:-----------:|:-----------:|:-----------:|
| 感情分析 | ○/△/× | ○/△/× | ○/△/× | ... |
| カテゴリ分類 | ... | ... | ... | ... |
```

評価基準：
- ○：期待通りの出力
- △：概ね正しいが細部に改善の余地あり
- ×：期待と異なる出力

最後に「Few-Shotが効果的なタスクの特徴」と「Zero-Shotで十分なタスクの特徴」をまとめてください。

</details>

## 解答例

<details>
<summary>解答例（初心者向け）</summary>

### タスク1：テキストの感情分析

#### Zero-Shotプロンプト

```
以下のテキストの感情を「ポジティブ」「ネガティブ」「ニュートラル」の
いずれかに分類してください。ラベルのみ出力してください。

テキスト：価格のわりにはまあまあ使えると思います。
```

**出力結果**: 「ニュートラル」（または「ポジティブ」とブレることがある）

#### One-Shotプロンプト

```
以下のテキストの感情を分類してください。

例：
テキスト：注文から届くまでがとても早くて助かりました。
感情：ポジティブ

テキスト：価格のわりにはまあまあ使えると思います。
感情：
```

**出力結果**: 「ニュートラル」（安定して同じ結果）

#### Few-Shotプロンプト

```
以下のテキストの感情を分類してください。

例1：
テキスト：注文から届くまでがとても早くて助かりました。
感情：ポジティブ

例2：
テキスト：商品の説明と実物が違っていて、返品を考えています。
感情：ネガティブ

例3：
テキスト：色は3種類あり、サイズはS・M・Lから選べます。
感情：ニュートラル

テキスト：価格のわりにはまあまあ使えると思います。
感情：
```

**出力結果**: 「ニュートラル」（安定して同じ結果）

**考察**: 感情分析は比較的シンプルなタスクのため、Zero-Shotでもある程度対応できました。ただし、「まあまあ」のような微妙な表現では、Few-Shotの方が安定した分類ができました。例3のニュートラル例が「事実の記述＝ニュートラル」という基準を暗示的に伝えたためです。

---

### タスク3：文体の変換（カジュアル → フォーマル）

#### Zero-Shotプロンプト

```
以下の文をフォーマルな表現に書き換えてください。

文：明日の会議、ちょっと遅れるかも。先に始めてて。
```

**出力結果**: 変換されるが、フォーマルさのレベルにブレあり

#### Few-Shotプロンプト

```
カジュアルな文をフォーマルなビジネス表現に変換してください。

例1：
入力：ちょっと確認したいんだけど、あの件どうなった？
出力：先日の件について確認させていただきたく存じます。進捗状況をお知らせいただけますでしょうか。

例2：
入力：了解、それでOKです。
出力：承知いたしました。その方針で進めさせていただきます。

例3：
入力：すみません、その日は無理です。
出力：申し訳ございませんが、該当日は対応が難しい状況でございます。

入力：明日の会議、ちょっと遅れるかも。先に始めてて。
出力：
```

**出力結果**: 「明日の会議に若干遅れる可能性がございます。恐れ入りますが、先に開始いただけますと幸いです。」

**考察**: 文体変換ではFew-Shotが非常に効果的でした。3つの例によって「どの程度フォーマルにするか」の基準が明確に伝わり、一貫したフォーマルさレベルの出力が得られました。

### 比較結果まとめ

| タスク | Zero-Shot | One-Shot | Few-Shot | 最適パターン |
|--------|:---------:|:--------:|:--------:|:----------:|
| 感情分析 | △ | ○ | ○ | One-Shot以上 |
| カテゴリ分類 | △ | ○ | ○ | One-Shot以上 |
| 文体変換 | △ | △ | ○ | Few-Shot |
| 情報抽出 | △ | ○ | ○ | One-Shot以上 |
| 要約生成 | ○ | ○ | ○ | Zero-Shot可 |

**Few-Shotが効果的なタスクの特徴**:
- 出力のスタイルや粒度を細かく制御したいタスク（文体変換など）
- 分類ラベルが独自で、一般的でないタスク
- 微妙な境界ケースの判断基準を伝えたいタスク

**Zero-Shotで十分なタスクの特徴**:
- LLMが事前学習で十分に理解しているタスク（要約、翻訳など）
- 出力形式がシンプルで、ブレが少ないタスク

### タスク2：テキストのカテゴリ分類（改良版）

改良版では、Few-Shotの**例の選び方の戦略**に焦点を当てます。

#### 戦略的Few-Shotプロンプト

```
以下のニュース見出しを「ビジネス」「技術」「生活」「エンタメ」のいずれかに
分類してください。

分類基準：
- ビジネス：企業活動、経済、株式、M&Aに関するもの
- 技術：IT、科学、研究開発に関するもの
- 生活：健康、教育、住居、食に関するもの
- エンタメ：映画、音楽、スポーツ、ゲームに関するもの

例1（ビジネス）：
見出し：大手小売チェーンが来年度の海外展開計画を発表
分類：ビジネス

例2（技術）：
見出し：国内研究チームが新しい半導体素材の開発に成功
分類：技術

例3（生活）：
見出し：厚生労働省が新しい食事ガイドラインを公開
分類：生活

例4（エンタメ）：
見出し：人気アニメシリーズの新作映画が興行収入100億円を突破
分類：エンタメ

例5（境界ケース）：
見出し：大手ゲーム会社が過去最高益を記録、株価も上昇
分類：ビジネス
理由：ゲーム会社だがニュースの本質は「企業の業績」と「株価」のため

見出し：自動運転タクシーの実証実験が都内で開始
分類：
```

**改良ポイント**:
1. **分類基準を明文化**: 例だけでなく、各カテゴリの定義を先に提示
2. **全カテゴリを網羅**: 4カテゴリ × 各1例で偏りなく提示
3. **境界ケースを追加**: 「ゲーム会社の業績」のようにカテゴリが曖昧な例に判断基準（理由）を追加
4. **例にラベルを付与**: 「例1（ビジネス）」と明示し、可読性を向上

#### 例の選定で意識すべき3原則

```
┌─────────────────────────────────────────────────────────┐
│           Few-Shot 例文選定の3原則                        │
├─────────────────────────────────────────────────────────┤
│                                                         │
│  1. 網羅性（Coverage）                                   │
│     → 全分類ラベルを最低1例ずつカバーする               │
│                                                         │
│  2. 多様性（Diversity）                                  │
│     → 同じパターンの例ばかり集めない                    │
│     → 異なる表現・長さ・トーンを含める                  │
│                                                         │
│  3. 境界明示（Boundary Clarification）                   │
│     → 判断が難しいケースを1つ含め、判断基準を示す       │
│                                                         │
└─────────────────────────────────────────────────────────┘
```

### タスク4：情報の抽出（改良版）

#### 構造化出力を指定したFew-Shotプロンプト

```
以下のイベント告知文から「人名」「日付」「場所」を抽出し、
JSON形式で出力してください。該当情報がない場合は null としてください。

例1：
入力：4月15日に東京ビッグサイトで山田太郎氏による講演が開催されます。
出力：
{
  "人名": ["山田太郎"],
  "日付": ["4月15日"],
  "場所": ["東京ビッグサイト"]
}

例2：
入力：来週の月曜日、弊社会議室にてチームミーティングを実施します。
出力：
{
  "人名": null,
  "日付": ["来週の月曜日"],
  "場所": ["弊社会議室"]
}

例3：
入力：佐藤花子と鈴木一郎が5月1日から3日間、大阪国際会議場と京都国際会館で連続セミナーを行います。
出力：
{
  "人名": ["佐藤花子", "鈴木一郎"],
  "日付": ["5月1日から3日間"],
  "場所": ["大阪国際会議場", "京都国際会館"]
}

入力：2026年6月10日、渋谷ヒカリエにて田中誠司氏と高橋美咲氏による
AI活用セミナーが14時から開催予定です。
出力：
```

**改良ポイント**:
- JSON形式を指定し、後続処理（パースなど）に適した構造化出力を実現
- 例2で「情報がない場合（人名なし）」のケースを示し、null処理を学習させた
- 例3で「複数の値がある場合」のケースを示し、配列での出力パターンを定義
- 例の複雑さを段階的に上げ（単純→欠損あり→複数値）、網羅性を確保

### Shot数の判断フローチャート

```
タスクの種類を確認
│
├─ LLMが十分理解している一般的なタスクか？
│   ├─ Yes → Zero-Shot で試す
│   │         └─ 出力品質は十分か？
│   │              ├─ Yes → Zero-Shot を採用
│   │              └─ No  → One-Shot に進む
│   └─ No → One-Shot 以上で開始
│
├─ 出力形式のカスタマイズが必要か？
│   ├─ Yes → Few-Shot（3〜5例）推奨
│   └─ No  → One-Shot で十分な場合が多い
│
└─ 境界ケースの判断が必要か？
    ├─ Yes → Few-Shot + 境界例を含める
    └─ No  → 基本例のみでOK
```

**初心者向けとの違い**:
- 例の選定に「網羅性・多様性・境界明示」の3原則を導入
- 構造化出力（JSON）を使った実践的な情報抽出パターンを追加
- 「情報がない場合」「複数値がある場合」のエッジケースを例に含めた
- Shot数の判断フローチャートで、実務での意思決定プロセスを可視化

</details>

# 第8章：オペレーティングシステム

## この章のゴール
- OS（Operating System）の役割と基本機能を説明できる
- プロセス管理（プロセスの状態遷移、排他制御）を理解する
- タスクスケジューリングの各方式と計算問題を解ける
- メモリ管理（実記憶・仮想記憶）の仕組みとページング計算ができる
- ファイルシステムの構造とディスクアクセス時間の計算ができる
- 割り込みの種類と処理の流れを説明できる

## 試験での出題傾向
- 午前試験：タスクスケジューリングの計算問題、ページ置換アルゴリズム、ディスクアクセス時間の計算が頻出です。
- 午後試験（科目B）：プロセスの状態遷移やメモリ管理の理解が前提となる問題が出題されます。

## 8.1 オペレーティングシステムの概要

### OSとは

オペレーティングシステム（Operating System, OS）とは、コンピュータのハードウェアを管理し、ユーザーやアプリケーションに対して使いやすい環境を提供する基本ソフトウェア（system software）です。

### OSの主な機能

| 機能 | 説明 |
|------|------|
| プロセス管理 | プログラムの実行を管理する |
| メモリ管理 | 主記憶の割り当て・解放を管理する |
| ファイル管理 | ファイルやディレクトリを管理する |
| 入出力管理 | 周辺装置とのデータのやり取りを制御する |
| ユーザー管理 | ユーザーの認証やアクセス権を管理する |

### カーネル（Kernel）とは

OSの中核部分で、ハードウェアに直接アクセスし、プロセス管理・メモリ管理・入出力管理などを担います。カーネルは常に主記憶上に常駐します。

### ユーザーモードとカーネルモード

| モード | 説明 | 実行可能な命令 |
|--------|------|--------------|
| ユーザーモード | アプリケーションが動作するモード | 一般的な命令のみ |
| カーネルモード（特権モード） | OSが動作するモード | 特権命令を含む全命令 |

ユーザープログラムがOSの機能を利用するには、**システムコール（SVC: Supervisor Call）**を使います。

```
ユーザープログラム → システムコール → カーネル（OS）→ ハードウェア
```

> **試験頻出ポイント：** システムコールはユーザーモードからカーネルモードへの切り替えを伴います。これは「ソフトウェア割込み」の一種です。

## 8.2 プロセス管理

### プロセスとは

プロセス（Process）とは、実行中のプログラムのことです。プログラムが主記憶にロードされ、CPUで実行される単位を指します。

### プロセスの状態遷移

プロセスには3つの基本状態があり、以下のように遷移します。

```
                ディスパッチ
    ┌──────── 実行可能状態 ────────┐
    │          （Ready）          │
    │              ↑              ↓
    │         生成  │          実行状態
    │              │          （Running）
    │              │              │
    │     タイムアウト│              │入出力要求
    │    （プリエンプション）        │
    │              │              ↓
    │              └────────  待ち状態
    │                         （Waiting）
    │                             │
    │         入出力完了            │
    └─────────────────────────────┘
```

| 状態 | 英語 | 説明 |
|------|------|------|
| 実行可能状態 | Ready | CPUの割り当てを待っている |
| 実行状態 | Running | CPUを使って処理を実行中 |
| 待ち状態 | Waiting / Blocked | 入出力完了などのイベントを待っている |

**状態遷移のきっかけ：**

| 遷移 | きっかけ |
|------|---------|
| 実行可能 → 実行 | ディスパッチ（CPUを割り当てられた） |
| 実行 → 実行可能 | タイムアウト（タイムスライス終了）、プリエンプション |
| 実行 → 待ち | 入出力要求の発行 |
| 待ち → 実行可能 | 入出力完了 |

> **よくある間違い：** 「待ち状態」から直接「実行状態」に遷移することはありません。必ず「実行可能状態」を経由します。

### スレッド（Thread）

プロセス内の実行単位です。1つのプロセスが複数のスレッドを持つことができます。

| 項目 | プロセス | スレッド |
|------|---------|---------|
| メモリ空間 | 独立 | プロセス内で共有 |
| 生成コスト | 大きい | 小さい |
| 切り替えコスト | 大きい | 小さい |
| 安全性 | 高い（独立している） | データ共有による競合の可能性 |

### 排他制御（Exclusive Control）

複数のプロセス（またはスレッド）が共有資源に同時にアクセスするとき、データの整合性を保つための制御です。

**セマフォ（Semaphore）：**

整数値のカウンタで排他制御を行う仕組みです。

| 操作 | 動作 |
|------|------|
| P操作（Wait） | セマフォ値を-1。値が0未満になったらプロセスを待ち状態にする |
| V操作（Signal） | セマフォ値を+1。待ちプロセスがあれば1つ起こす |

```
セマフォ S = 1（初期値）

プロセスA:               プロセスB:
P(S) → S=0（進入OK）
  共有資源を使用          P(S) → S=-1（待ち）
V(S) → S=0              → 待ち解除、進入OK
                           共有資源を使用
                         V(S) → S=1
```

**ミューテックス（Mutex）：**
セマフォの特殊なケースで、値が0か1のみの二値セマフォです。1つのプロセスだけがロックを獲得できます。

**デッドロック（Deadlock）：**

複数のプロセスが互いに資源の解放を待ち合う状態です（第5章のデータベースでも学びました）。

**デッドロック発生の4条件（コフマン条件）：**
1. **相互排除：** 資源を同時に使えるのは1プロセスだけ
2. **保持と待ち：** 資源を保持したまま別の資源を要求する
3. **横取り不可：** 他のプロセスから強制的に資源を奪えない
4. **循環待ち：** 資源待ちが循環している

> **試験のポイント：** 4条件のうち1つでも崩せばデッドロックは発生しません。

### ポイントまとめ
- プロセスの3状態：実行可能、実行、待ち
- 待ち → 実行 への直接遷移はない
- セマフォのP操作で資源確保、V操作で資源解放
- デッドロックは4条件すべてが満たされたときに発生

## 8.3 タスクスケジューリング

### スケジューリングとは

複数のプロセス（タスク）が実行可能状態にあるとき、どのプロセスにCPUを割り当てるかを決定する仕組みです。

### プリエンプティブとノンプリエンプティブ

| 方式 | 説明 |
|------|------|
| プリエンプティブ（Preemptive） | 実行中のプロセスを中断して、他のプロセスにCPUを渡すことがある |
| ノンプリエンプティブ（Non-preemptive） | 実行中のプロセスが完了するまでCPUを明け渡さない |

### 主なスケジューリングアルゴリズム

#### FCFS（First Come First Served / 到着順）

到着した順に実行します。ノンプリエンプティブ方式です。

```
プロセス  到着時刻  実行時間
  P1        0        5
  P2        1        3
  P3        2        8

タイムライン：
|---P1---|---P2--|--------P3--------|
0       5      8                  16

ターンアラウンドタイム（TAT）：
P1: 5 - 0 = 5
P2: 8 - 1 = 7
P3: 16 - 2 = 14
平均TAT = (5 + 7 + 14) / 3 = 8.67
```

#### SJF（Shortest Job First / 最短ジョブ優先）

実行時間が最も短いプロセスを優先します。

```
プロセス  到着時刻  実行時間
  P1        0        5
  P2        1        3
  P3        2        8

P1が先に開始。P1完了時(t=5)にP2,P3が待ち。P2(3)の方が短いのでP2を先に実行。

タイムライン：
|---P1---|--P2--|--------P3--------|
0       5     8                  16

この例ではFCFSと同じ結果ですが、到着タイミングによっては異なります。
```

#### ラウンドロビン（Round Robin）

各プロセスに均等な時間（タイムクォンタム / タイムスライス）を与え、順番に実行します。プリエンプティブ方式です。

```
プロセス  到着時刻  実行時間
  P1        0        5
  P2        0        3
  P3        0        8
タイムクォンタム = 2

タイムライン：
|P1|P2|P3|P1|P2|P3|P1|P3|P3|
0  2  4  6  8  9  11 12 14 16

P1: [0-2], [6-8], [11-12] → 完了12  TAT=12
P2: [2-4], [8-9]          → 完了9   TAT=9
P3: [4-6], [9-11], [12-16]→ 完了16  TAT=16
平均TAT = (12 + 9 + 16) / 3 = 12.33
```

> **タイムクォンタムの設定：** 大きすぎるとFCFSに近づき、小さすぎるとコンテキストスイッチのオーバーヘッドが増大します。

#### 優先度スケジューリング（Priority Scheduling）

各プロセスに優先度を設定し、優先度の高いプロセスを先に実行します。

- **静的優先度：** プロセス生成時に決まり、変化しない
- **動的優先度：** 待ち時間などに応じて優先度が変化する

**飢餓（Starvation）問題：** 優先度が低いプロセスがいつまでも実行されない状態。対策として**エージング（Aging）**（待ち時間に応じて優先度を上げる）があります。

#### 多段フィードバックキュー（Multilevel Feedback Queue）

優先度の異なる複数のキューを持ち、プロセスの振る舞いに応じてキュー間を移動させるスケジューリング方式です。実用的なOSで広く使われています。

### スケジューリングの性能指標

| 指標 | 説明 | 計算式 |
|------|------|--------|
| ターンアラウンドタイム（TAT） | ジョブの到着から完了までの時間 | 完了時刻 - 到着時刻 |
| 応答時間（Response Time） | 要求から最初の応答までの時間 | 最初の実行開始時刻 - 到着時刻 |
| 待ち時間（Waiting Time） | CPUを待っていた合計時間 | TAT - 実行時間 |
| スループット | 単位時間あたりの処理完了数 | 完了ジョブ数 / 経過時間 |
| CPU利用率 | CPUが実際に処理を行っていた割合 | CPU稼働時間 / 全体時間 |

### 計算例：ターンアラウンドタイム

```
3つのジョブA, B, Cが同時（時刻0）に到着。実行時間はそれぞれ 3, 5, 2。
FCFSで A → B → C の順に実行する場合：

|--A--|----B----|--C--|
0    3        8    10

ジョブA: TAT = 3 - 0 = 3
ジョブB: TAT = 8 - 0 = 8
ジョブC: TAT = 10 - 0 = 10
平均TAT = (3 + 8 + 10) / 3 = 7.0

SJFで C → A → B の順に実行する場合：

|C-|--A--|----B----|
0  2    5        10

ジョブC: TAT = 2 - 0 = 2
ジョブA: TAT = 5 - 0 = 5
ジョブB: TAT = 10 - 0 = 10
平均TAT = (2 + 5 + 10) / 3 = 5.67  ← SJFの方が平均TATが短い
```

> **試験頻出ポイント：** SJFは平均ターンアラウンドタイムを最小化するスケジューリング方式です。ただし、実行時間を事前に知る必要があるという制約があります。

### ポイントまとめ
- FCFS：到着順、単純だが短いジョブが長いジョブの後に待たされる
- SJF：平均TATが最小、実行時間の事前情報が必要
- ラウンドロビン：公平、タイムクォンタムの設定が重要
- 優先度方式：飢餓問題にはエージングで対策

## 8.4 メモリ管理

### 実記憶管理

主記憶（実メモリ）上にプログラムを配置する方法です。

#### 固定区画方式（固定パーティション方式）

主記憶をあらかじめ固定サイズの区画に分割し、各区画にプロセスを割り当てます。

```
主記憶:
┌──────────────┐
│   OS領域      │
├──────────────┤
│ 区画1: 100KB  │ ← プロセスA（80KB）→ 内部断片化 20KB
├──────────────┤
│ 区画2: 200KB  │ ← プロセスB（150KB）→ 内部断片化 50KB
├──────────────┤
│ 区画3: 100KB  │ ← 空き
└──────────────┘
```

**内部断片化（Internal Fragmentation）：** 区画内の未使用領域。プロセスサイズが区画サイズより小さいと無駄な領域が発生します。

#### 可変区画方式（可変パーティション方式）

プロセスのサイズに合わせて区画を動的に割り当てます。

```
主記憶（プロセスの割り当て・解放を繰り返した後）:
┌──────────────┐
│   OS領域      │
├──────────────┤
│ プロセスA 80KB│
├──────────────┤
│ 空き 50KB     │ ← 外部断片化
├──────────────┤
│ プロセスC 120KB│
├──────────────┤
│ 空き 30KB     │ ← 外部断片化
├──────────────┤
│ プロセスD 100KB│
└──────────────┘
```

**外部断片化（External Fragmentation）：** メモリ上に散在する小さな空き領域。個々の空きは十分あるのに、連続した空きが確保できない状態です。

**対策 - メモリコンパクション（Memory Compaction）：**
使用中の領域を詰めて、空き領域をまとめる操作です。

#### 空き領域の割り当てアルゴリズム

| アルゴリズム | 説明 |
|------------|------|
| 最初適合（First Fit） | 先頭から探して最初に見つかった十分な空き領域に割り当て |
| 最良適合（Best Fit） | サイズが最も近い（無駄が最小の）空き領域に割り当て |
| 最悪適合（Worst Fit） | 最も大きい空き領域に割り当て |

### 仮想記憶管理（Virtual Memory）

主記憶（実メモリ）の容量を超えるプログラムを実行するための仕組みです。補助記憶の一部を仮想的にメモリとして利用します。

#### ページング方式（Paging）

仮想アドレス空間と実メモリの両方を、固定サイズの単位（ページ / ページフレーム）に分割して管理します。

```
仮想アドレス空間          主記憶（ページフレーム）
┌──────────┐            ┌──────────┐
│ ページ0   │ ─────────→│ フレーム2  │
├──────────┤            ├──────────┤
│ ページ1   │ ─→ 補助記憶│ フレーム0  │ ← ページ3
├──────────┤            ├──────────┤
│ ページ2   │ ─────────→│ フレーム1  │
├──────────┤            ├──────────┤
│ ページ3   │ ─────────→│ フレーム3  │
└──────────┘            └──────────┘
         ↑
     ページテーブルで対応関係を管理
```

**ページテーブル（Page Table）：**

仮想ページ番号と物理フレーム番号の対応を管理する表です。

| 仮想ページ番号 | 有効ビット | 物理フレーム番号 |
|-------------|----------|---------------|
| 0 | 1 | 2 |
| 1 | 0 | - （補助記憶上） |
| 2 | 1 | 1 |
| 3 | 1 | 0 |

**ページフォールト（Page Fault）：**

アクセスしたいページが主記憶上にないときに発生する割り込みです。OSが補助記憶からページを読み込みます。

```
ページアクセス → ページテーブル参照 → 有効ビット=0
→ ページフォールト発生
→ OSが補助記憶からページを主記憶に読み込む
→ ページテーブルを更新
→ プログラムを再実行
```

#### ページ置換アルゴリズム

主記憶が満杯のとき、どのページを追い出すかを決めるアルゴリズムです。

| アルゴリズム | 説明 |
|------------|------|
| FIFO（First In First Out） | 最も古く読み込まれたページを置換 |
| LRU（Least Recently Used） | 最も長い間参照されていないページを置換 |
| LFU（Least Frequently Used） | 参照回数が最も少ないページを置換 |
| OPT（最適置換） | 将来最も長く使われないページを置換（理論上の最適、実装不可） |

**LRUの動作例：**

ページフレーム数 = 3、参照列 = 1, 2, 3, 4, 1, 2, 5, 1, 2, 3

```
参照  フレーム状態        ページフォールト
 1    [1, -, -]          ○（空きフレームに読込）
 2    [1, 2, -]          ○
 3    [1, 2, 3]          ○
 4    [4, 2, 3]          ○（1が最も古い参照→置換）
 1    [4, 1, 3]          ○（2が最も古い参照→置換）
 2    [4, 1, 2]          ○（3が最も古い参照→置換）
 5    [5, 1, 2]          ○（4が最も古い参照→置換）
 1    [5, 1, 2]          ×（既に存在）
 2    [5, 1, 2]          ×（既に存在）
 3    [3, 1, 2]          ○（5が最も古い参照→置換）

ページフォールト回数 = 8
```

#### ページフォールト率の計算

```
ページフォールト率 = ページフォールト回数 / 総アクセス回数

上の例：ページフォールト率 = 8 / 10 = 0.8（80%）
```

#### スラッシング（Thrashing）

ページフォールトが頻発し、OSがページの入れ替え処理ばかり行って、実際のプログラム処理がほとんど進まなくなる現象です。

**原因：** 主記憶に対して実行するプロセスが多すぎる（多重度が高すぎる）

**対策：**
- 主記憶を増設する
- 多重度（同時実行プロセス数）を下げる
- ワーキングセット方式（プロセスが直近に参照したページの集合を管理）を採用する

#### セグメンテーション方式

プログラムを論理的な単位（セグメント）に分割して管理する方式です。セグメントのサイズは可変です。

**ページングとの比較：**

| 特性 | ページング | セグメンテーション |
|------|----------|-----------------|
| 分割単位 | 固定サイズ（ページ） | 可変サイズ（セグメント） |
| 断片化 | 内部断片化が発生しうる | 外部断片化が発生しうる |
| 論理的区分 | なし | コード、データ、スタックなど |

### ポイントまとめ
- 固定区画 → 内部断片化、可変区画 → 外部断片化
- ページングは固定サイズで管理、ページテーブルで仮想→物理の対応
- LRUは「最も長く使われていない」ページを置換
- スラッシングはページフォールト頻発で性能低下

## 8.5 ファイルシステム

### ファイルシステムとは

ファイル（File）の作成・読み書き・削除などを管理する仕組みです。ディレクトリ（Directory / フォルダ）構造でファイルを階層的に整理します。

### ディレクトリ構造

```
/ （ルートディレクトリ）
├── home/
│   ├── user1/
│   │   ├── doc/
│   │   │   └── report.txt
│   │   └── pic/
│   └── user2/
├── etc/
└── var/
```

**パスの表記：**
- **絶対パス（Absolute Path）：** ルートからの完全な経路（例：/home/user1/doc/report.txt）
- **相対パス（Relative Path）：** 現在のディレクトリからの経路（例：doc/report.txt）

### ファイルのアクセス方法

| 方式 | 説明 | 例 |
|------|------|-----|
| 順次アクセス（Sequential） | 先頭から順にアクセス | 磁気テープ |
| 直接アクセス（Direct / Random） | 任意の位置に直接アクセス | 磁気ディスク |
| 索引付きアクセス（Indexed） | 索引を使って目的のレコードにアクセス | ISAM |

### ディスクの構造

```
  ──────── トラック（Track）
  ┌───────────────────────────────┐
  │  ┌─────────────────────────┐  │
  │  │  ┌─────────────────┐   │  │
  │  │  │     ┌──────┐    │   │  │
  │  │  │     │ 回転軸│    │   │  │
  │  │  │     └──────┘    │   │  │
  │  │  └─────────────────┘   │  │
  │  └─────────────────────────┘  │
  └───────────────────────────────┘
  ←── セクタ ──→

  セクタ（Sector）: ディスク上の最小読み書き単位
  トラック（Track）: 同心円状の記録領域
  シリンダ（Cylinder）: 同一半径のトラックの集まり（複数プラッタ）
```

### ディスクアクセス時間の計算

磁気ディスクにデータを読み書きするのにかかる時間は、3つの要素の合計です。

```
ディスクアクセス時間 = シーク時間 + 回転待ち時間 + データ転送時間
```

| 要素 | 説明 |
|------|------|
| シーク時間（Seek Time） | 磁気ヘッドを目的のトラック（シリンダ）に移動する時間 |
| 回転待ち時間（Rotational Latency） | 目的のセクタがヘッドの下に来るまでの回転待ち時間 |
| データ転送時間（Transfer Time） | データの読み書きにかかる時間 |

**回転待ち時間の計算：**

平均回転待ち時間は、ディスクが1回転する時間の半分です。

```
ディスクの回転速度: 6000回転/分（rpm）
1回転の時間 = 60秒 / 6000 = 10ミリ秒
平均回転待ち時間 = 10ミリ秒 / 2 = 5ミリ秒
```

**計算例：**

```
シーク時間: 5ms
回転速度: 6000rpm
1トラックのデータ量: 50KB
読み取りデータ量: 10KB

1回転の時間 = 60 / 6000 = 0.01秒 = 10ms
平均回転待ち時間 = 10 / 2 = 5ms
データ転送時間 = (10 / 50) × 10 = 2ms

ディスクアクセス時間 = 5 + 5 + 2 = 12ms
```

> **試験頻出ポイント：** 平均回転待ち時間 = 1回転の時間 / 2 という公式は必ず覚えてください。

### ポイントまとめ
- 絶対パスはルートから、相対パスはカレントディレクトリから
- ディスクアクセス時間 = シーク + 回転待ち + データ転送
- 平均回転待ち時間 = 1回転の時間 / 2

## 8.6 割り込み（Interrupt）

### 割り込みとは

現在実行中の処理を一時中断して、緊急度の高い別の処理（割り込み処理）を実行する仕組みです（第7章でも触れました）。

### 割り込みの分類

```
割り込み
├── 内部割り込み（プログラム由来）
│   ├── プログラム割り込み（ゼロ除算、オーバーフロー等）
│   └── SVC割り込み（システムコール）
└── 外部割り込み（ハードウェア由来）
    ├── 入出力割り込み（I/O完了通知）
    ├── タイマー割り込み（タイムスライス満了）
    ├── マシンチェック割り込み（ハードウェア障害）
    └── コンソール割り込み（オペレータ操作）
```

| 分類 | 種類 | 発生原因 |
|------|------|---------|
| 内部割り込み | プログラム割り込み | ゼロ除算、不正命令、ページフォールト |
| 内部割り込み | SVC割り込み | システムコール（ユーザーが明示的に発行） |
| 外部割り込み | 入出力割り込み | I/O処理の完了通知 |
| 外部割り込み | タイマー割り込み | 一定時間経過（タイムスライスの管理） |
| 外部割り込み | マシンチェック割り込み | ハードウェアの故障検知 |

### 割り込み処理の流れ

```
1. 割り込み発生
2. 現在の処理を中断
3. CPUの状態（レジスタ、PCなど）を退避（スタックに保存）
4. 割り込みハンドラ（割り込み処理ルーチン）を実行
5. 退避した状態を復元
6. 中断した処理を再開
```

### 割り込みの優先度

複数の割り込みが同時に発生した場合、優先度の高いものから処理されます。一般的な優先度の順序は次のとおりです。

```
マシンチェック割り込み（最高優先度）
↓
プログラム割り込み
↓
SVC割り込み
↓
入出力割り込み
↓
タイマー割り込み（最低優先度の外部割り込み）
```

### 多重割り込み

割り込み処理中に、さらに優先度の高い割り込みが発生した場合、現在の割り込み処理を中断して新たな割り込みを処理します。

### ポイントまとめ
- 内部割り込み：プログラム実行中に発生（ゼロ除算、SVC等）
- 外部割り込み：ハードウェア由来（I/O完了、タイマー等）
- 割り込み処理では、CPUの状態を退避→割り込み処理→復元→再開

---

## まとめ

| 項目 | 重要ポイント |
|------|-------------|
| プロセスの3状態 | 実行可能・実行・待ち。待ち→実行の直接遷移はない |
| セマフォ | P操作で確保、V操作で解放。デッドロック防止が重要 |
| スケジューリング | FCFS=到着順、SJF=平均TAT最小、ラウンドロビン=公平 |
| 内部断片化 | 固定区画方式で発生（区画内の無駄な空き） |
| 外部断片化 | 可変区画方式で発生（散在する小さな空き） |
| ページング | 固定サイズでメモリ管理。ページフォールトでI/O発生 |
| LRU | 最も長く参照されていないページを置換 |
| スラッシング | ページフォールト頻発で性能低下 |
| ディスクアクセス | シーク時間 + 回転待ち時間 + データ転送時間 |
| 割り込み | 内部（プログラム/SVC）と外部（I/O/タイマー/マシンチェック） |

## 確認問題

1. プロセスの3つの基本状態を挙げ、「待ち状態」から「実行状態」へ直接遷移できない理由を述べよ。
2. ページフレーム数が3でLRUアルゴリズムを使用する場合、参照列 2, 3, 2, 1, 5, 2 のページフォールト回数を求めよ。
3. ディスクの回転速度が 4800rpm、シーク時間が 10ms、1トラックのデータ量が 40KB のとき、5KB のデータを読み取るアクセス時間を計算せよ。
4. 内部割り込みと外部割り込みの違いを説明し、それぞれ例を1つ挙げよ。
5. スラッシングとは何か、その原因と対策を述べよ。

<details>
<summary>解答</summary>

1. 3つの基本状態は「実行可能状態（Ready）」「実行状態（Running）」「待ち状態（Waiting）」。待ち状態から実行状態へ直接遷移できないのは、待ち状態のプロセスはまず「実行可能状態」に移り、スケジューラによるディスパッチを経てCPUが割り当てられて初めて「実行状態」になるためです。

2. 参照列 2, 3, 2, 1, 5, 2：
   - 2 → [2, -, -] フォールト
   - 3 → [2, 3, -] フォールト
   - 2 → [2, 3, -] ヒット
   - 1 → [2, 3, 1] フォールト
   - 5 → [2, 5, 1] フォールト（3が最も古い参照→置換）
   - 2 → [2, 5, 1] ヒット
   ページフォールト回数 = 4

3. 1回転の時間 = 60 / 4800 = 0.0125秒 = 12.5ms
   平均回転待ち時間 = 12.5 / 2 = 6.25ms
   データ転送時間 = (5 / 40) × 12.5 = 1.5625ms
   アクセス時間 = 10 + 6.25 + 1.5625 = 17.8125ms ≒ 17.8ms

4. 内部割り込みはプログラムの実行に起因して発生する割り込み（例：ゼロ除算によるプログラム割り込み）。外部割り込みはハードウェアに起因して発生する割り込み（例：I/O完了による入出力割り込み）。

5. スラッシングとは、ページフォールトが頻発して、OSがページの入れ替え処理に時間を費やし、実際の処理がほとんど進まなくなる現象。原因は、主記憶に対して同時実行プロセスが多すぎること。対策として、主記憶の増設、多重度の制限、ワーキングセット方式の採用などがある。

</details>

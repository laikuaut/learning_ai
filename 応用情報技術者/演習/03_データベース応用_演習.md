# 第3章 データベース応用 演習問題

---

## 基本問題

### 問1（四択・正規化）
次の関係スキーマが第2正規形であるが第3正規形ではない理由として、正しいものはどれか。

```
受注(受注番号, 商品コード, 商品名, 単価, 数量)
主キー: {受注番号, 商品コード}
```

ア 繰り返し項目が存在するから
イ 主キーの一部に対する部分関数従属が存在するから
ウ 非キー属性間に推移的関数従属が存在するから
エ 候補キー以外からの関数従属が存在するから

<details><summary>ヒント</summary>

商品コード → 商品名, 単価 という関数従属を考えます。これは部分関数従属でしょうか、推移的関数従属でしょうか。

</details>

<details><summary>解答・解説</summary>

**正解: イ 主キーの一部に対する部分関数従属が存在するから**

```
商品コード → 商品名, 単価
```

商品コードは主キー{受注番号, 商品コード}の一部であり、商品名と単価は商品コードだけで決定されます。これは部分関数従属です。

したがって、この関係は第1正規形であり、第2正規形ではありません。問題文に「第2正規形であるが第3正規形ではない」と記載されていますが、実際には部分関数従属が存在するため第2正規形の条件を満たしていません。

正しくは: この関係は**第1正規形であるが第2正規形ではない**。第2正規形にするには部分関数従属を排除する必要があります。

</details>

### 問2（四択・ACID特性）
トランザクションのACID特性のうち、「コミットされたトランザクションの結果は、システム障害が発生しても失われない」ことを保証する特性はどれか。

ア 原子性（Atomicity）
イ 一貫性（Consistency）
ウ 独立性（Isolation）
エ 耐久性（Durability）

<details><summary>解答・解説</summary>

**正解: エ 耐久性（Durability）**

- 原子性: トランザクションの操作は全て完了するか、全て取り消されるか
- 一貫性: トランザクション前後でDBの整合性が維持される
- 独立性: 複数のトランザクションが互いに干渉しない
- 耐久性: コミット済みの結果は永続的に保持される

</details>

### 問3（四択・障害回復）
チェックポイント後にコミットされたがディスクに反映されていないトランザクションに対して行う回復処理はどれか。

ア REDO（ロールフォワード）
イ UNDO（ロールバック）
ウ チェックポイントの再実行
エ バックアップからの復元

<details><summary>解答・解説</summary>

**正解: ア REDO（ロールフォワード）**

- REDO: コミット済みだがディスク未反映のトランザクション → ログの更新後値で再実行
- UNDO: 未コミットのトランザクション → ログの更新前値で取り消し

</details>

### 問4（四択・同時実行制御）
2つのトランザクションが互いにロックの解放を待ち合い、どちらも進行できない状態を何というか。

ア ライブロック　　イ デッドロック　　ウ スタベーション　　エ カスケードロールバック

<details><summary>解答・解説</summary>

**正解: イ デッドロック**

- デッドロック: 2つ以上のトランザクションが互いにロック解放を待つ状態
- ライブロック: 処理が進行しているように見えるが実質的に進まない状態
- スタベーション: 特定のトランザクションがいつまでもリソースを獲得できない状態

</details>

### 問5（四択・分散DB）
CAP定理に関する記述として、正しいものはどれか。

ア 分散システムでは、一貫性、可用性、分断耐性の3つを同時に完全に満たすことができる
イ 分散システムでは、一貫性、可用性、分断耐性のうち、最大2つまでしか同時に満たせない
ウ 分散システムでは、必ず一貫性を犠牲にしなければならない
エ CAP定理はRDBMSにのみ適用される定理である

<details><summary>解答・解説</summary>

**正解: イ**

CAP定理は、分散システムにおいて一貫性(C)、可用性(A)、分断耐性(P)の3つを同時に完全に満たすことはできず、最大2つまでしか同時に満たせないという定理です。

</details>

## 応用問題

### 問6（四択・トランザクション分離レベル）
トランザクション分離レベルが READ COMMITTED の場合に発生する可能性がある現象の組合せとして、正しいものはどれか。

ア ダーティリードとファントムリード
イ ノンリピータブルリードとファントムリード
ウ ダーティリードのみ
エ ファントムリードのみ

<details><summary>解答・解説</summary>

**正解: イ ノンリピータブルリードとファントムリード**

| 分離レベル | ダーティリード | ノンリピータブルリード | ファントムリード |
|-----------|:---:|:---:|:---:|
| READ UNCOMMITTED | 発生 | 発生 | 発生 |
| READ COMMITTED | 防止 | 発生 | 発生 |
| REPEATABLE READ | 防止 | 防止 | 発生 |
| SERIALIZABLE | 防止 | 防止 | 防止 |

READ COMMITTEDではダーティリードは防止されますが、ノンリピータブルリードとファントムリードは発生する可能性があります。

</details>

### 問7（四択・2相コミット）
分散データベースにおける2相コミットプロトコルの第1相（投票フェーズ）で、参加者の1つが「NG（拒否）」を返した場合、調整者が行う動作として正しいものはどれか。

ア 全参加者にCOMMITを送信する
イ 拒否した参加者だけにROLLBACKを送信する
ウ 全参加者にROLLBACKを送信する
エ 拒否した参加者を除外してCOMMITを送信する

<details><summary>解答・解説</summary>

**正解: ウ 全参加者にROLLBACKを送信する**

2相コミットでは、1人でも拒否（NG）を返した場合、全参加者にROLLBACKが指示されます。全員がOKを返した場合のみCOMMITが実行されます。これにより分散環境での一貫性が保証されます。

</details>

### 問8（記述・正規化）
次の関係スキーマを第3正規形まで正規化せよ。正規化の各段階で分解した結果と、その理由を記述すること。

```
注文(注文番号, 注文日, 顧客コード, 顧客名, 顧客住所,
     商品コード, 商品名, 単価, 数量)
```

ただし、以下の関数従属性が成り立つものとする。
- 注文番号 → 注文日, 顧客コード
- 顧客コード → 顧客名, 顧客住所
- 商品コード → 商品名, 単価
- {注文番号, 商品コード} → 数量

<details><summary>解答・解説</summary>

**第1正規形（1NF）:**

既に繰り返し項目がないため、1NFを満たしています。
主キー: {注文番号, 商品コード}

**第2正規形（2NF） - 部分関数従属の排除:**

部分関数従属の特定:
- 注文番号 → 注文日, 顧客コード（主キーの一部で決まる）
- 商品コード → 商品名, 単価（主キーの一部で決まる）
- {注文番号, 商品コード} → 数量（完全関数従属）

分解結果:
```
注文(注文番号, 注文日, 顧客コード, 顧客名, 顧客住所)
商品(商品コード, 商品名, 単価)
注文明細(注文番号, 商品コード, 数量)
```

**第3正規形（3NF） - 推移的関数従属の排除:**

推移的関数従属の特定:
- 注文テーブル内: 注文番号 → 顧客コード → 顧客名, 顧客住所

分解結果:
```
注文(注文番号, 注文日, 顧客コード)
顧客(顧客コード, 顧客名, 顧客住所)
商品(商品コード, 商品名, 単価)
注文明細(注文番号, 商品コード, 数量)
```

これで第3正規形の条件を全て満たします。

</details>

## チャレンジ問題

### 問9（記述・午後試験形式）
次のシステム障害のシナリオについて、設問に答えなさい。

あるデータベースシステムで、チェックポイント時刻をTcとし、障害発生時刻をTfとする。以下のトランザクションが実行されていた。

```
T1: Tc以前に開始し、Tc以前にCOMMIT
T2: Tc以前に開始し、TcとTfの間にCOMMIT
T3: Tc以前に開始し、Tfの時点で未COMMIT
T4: TcとTfの間に開始し、TcとTfの間にCOMMIT
T5: TcとTfの間に開始し、Tfの時点で未COMMIT
```

**設問1:** 各トランザクションT1〜T5について、システム障害からの回復時にREDO、UNDO、回復不要のいずれの処理が必要か答えよ。

**設問2:** WAL（Write-Ahead Logging）プロトコルの原則を説明し、なぜこの原則が障害回復に不可欠であるか述べよ。

<details><summary>解答・解説</summary>

**設問1の解答:**

| トランザクション | 回復処理 | 理由 |
|----------------|---------|------|
| T1 | 回復不要 | Tc以前にCOMMIT済み。チェックポイントでディスクに反映済み |
| T2 | REDO | COMMIT済みだが、チェックポイント後の更新がディスクに反映されていない可能性あり |
| T3 | UNDO | 未COMMITのため、ログの更新前値を使って変更を取り消す |
| T4 | REDO | COMMIT済みだが、ディスクに反映されていない可能性あり |
| T5 | UNDO | 未COMMITのため、ログの更新前値を使って変更を取り消す |

判定ルール:
- チェックポイント以前にCOMMIT → 回復不要
- チェックポイント以降にCOMMIT → REDO
- 障害時点で未COMMIT → UNDO

**設問2の解答:**

WAL（Write-Ahead Logging）プロトコルの原則: データベースの更新をディスクに書き込む前に、必ずログ（ジャーナル）をディスクに書き込むこと。

この原則が不可欠な理由: WALにより、障害が発生した時点で、ログには全ての更新操作の記録が必ず残っています。もしデータを先に書いてログを後に書く順序の場合、データ書込み後ログ書込み前に障害が発生すると、どの操作が行われたか分からなくなり、REDOもUNDOも正しく実行できません。WALがあることで、ログの情報だけでデータベースを一貫した状態に復旧することが保証されます。

</details>

### 問10（記述・午後試験形式）
あるECサイトのデータベース設計について、以下の設問に答えなさい。

現在、全データをRDBMS（PostgreSQL）で管理しているが、以下の課題が発生している。

- 商品カタログの表示が遅い（1ページ表示に3秒以上）
- セッション情報の管理がボトルネックになっている
- 商品のレコメンデーション機能を追加したい（ユーザ間の購買パターン分析）

**設問1:** 上記の3つの課題それぞれに適したNoSQLデータベースの種類を選び、その選定理由を述べよ。

**設問2:** RDBMSとNoSQLを組み合わせた「ポリグロットパーシステンス」アーキテクチャの利点と課題を述べよ。

<details><summary>解答・解説</summary>

**設問1の解答:**

1. **商品カタログの表示遅延 → ドキュメント指向DB（MongoDB等）**
   - 理由: 商品情報は属性が商品ごとに異なる（服はサイズ・色、電子機器はスペック等）。ドキュメント指向DBではスキーマレスで柔軟にデータを格納でき、1回の読み取りで商品の全情報を取得できるため、JOINが不要でレスポンスが高速になります。

2. **セッション情報のボトルネック → キーバリューストア（Redis等）**
   - 理由: セッション情報はセッションIDをキーとして高速にアクセスする必要があります。キーバリューストアはインメモリで動作し、ミリ秒以下のレスポンスタイムを実現できます。また、TTL（有効期限）機能でセッションの自動削除も容易です。

3. **レコメンデーション機能 → グラフDB（Neo4j等）**
   - 理由: ユーザ間の購買パターン分析は「ユーザAが購入した商品を、同じ商品を購入したユーザBも購入した他の商品」のような関連性の探索が中心です。グラフDBはノード（ユーザ、商品）とエッジ（購入関係）の探索に最適化されており、RDBMSでは複雑なJOINが必要なクエリを効率的に実行できます。

**設問2の解答:**

**利点:**
- 各データの特性に最適なデータベースを選択でき、全体の性能が向上する
- RDBMSの強み（ACID準拠、複雑なクエリ）とNoSQLの強み（高スループット、柔軟性）を組合せられる
- スケールアウトが必要な部分だけNoSQLにすることでコスト効率が上がる

**課題:**
- 複数DBにまたがるトランザクションの一貫性の確保が困難（2PCの限界）
- 運用・保守の複雑さが増大（複数の技術スタックの習得・監視が必要）
- データの同期・整合性の管理が必要（結果整合性の許容範囲の設計）
- 障害時の復旧手順が複雑になる

</details>

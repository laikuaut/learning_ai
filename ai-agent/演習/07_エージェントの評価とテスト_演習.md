# 第7章 演習：エージェントの評価とテスト

この演習では、AIエージェントの評価指標の計算、テスト手法の理解、ベンチマークの活用、そしてデバッグ技術について実践的に学びます。

## 基本問題

### 問題1：評価指標の計算

あるカスタマーサポートエージェントが100件のタスクを処理しました。以下のデータに基づいて、各評価指標を計算してください。

```
- 完了タスク数：82件
- 正しく完了したタスク数：74件
- 総ステップ数：520ステップ
- 理想的な総ステップ数：310ステップ
- 総API呼び出しコスト：$12.40
- 人間が同等作業を行った場合の推定コスト：$850.00
```

以下の指標を求めてください。

1. タスク成功率（Task Success Rate）
2. ステップ効率（Step Efficiency）
3. コスト効率（Cost Efficiency Ratio）

**期待される出力例：**

```
1. タスク成功率 = 正しく完了したタスク数 / 総タスク数
   = 74 / 100 = 74.0%

2. ステップ効率 = 理想的な総ステップ数 / 実際の総ステップ数
   = 310 / 520 = 59.6%

3. コスト効率 = 人間の推定コスト / エージェントのコスト
   = $850.00 / $12.40 = 68.5倍
```

<details>
<summary>ヒント</summary>

- タスク成功率は「正しく完了した数」を使います。完了しただけでは不十分です。
- ステップ効率は1.0（100%）に近いほど効率的です。
- コスト効率は倍率で表現し、高いほどエージェントがコスト的に有利であることを示します。
</details>

<details>
<summary>解答例</summary>

```
# 1. タスク成功率（Task Success Rate）
# 「完了」ではなく「正しく完了」を分子にする点がポイントです
タスク成功率 = 74 / 100 = 0.74 = 74.0%

# 2. ステップ効率（Step Efficiency）
# 理想ステップ数を分子にすることで、1.0に近いほど効率的になります
ステップ効率 = 310 / 520 = 0.596 = 59.6%

# 3. コスト効率（Cost Efficiency Ratio）
# 人間コストをエージェントコストで割ることで、何倍効率的かがわかります
コスト効率 = 850.00 / 12.40 = 68.5倍

# 補足：この結果から読み取れること
# - 成功率74%は改善の余地あり（目標は90%以上が一般的）
# - ステップ効率59.6%は無駄なステップが多い（ツール選択の最適化が必要）
# - コスト効率68.5倍は非常に高く、自動化の費用対効果は十分
```
</details>

### 問題2：テストピラミッドの理解

AIエージェントのテストピラミッドについて、以下の各テストレベルに該当するテスト内容を分類してください。

**分類対象のテスト項目：**

```
A. LLMのプロンプトに対して期待されるJSON形式で出力されるか確認する
B. ユーザーが「注文をキャンセルしたい」と入力してから完了までの全フローを確認する
C. 検索ツールが正しいクエリパラメータでAPIを呼び出すか確認する
D. エージェントが3つのツールを順番に呼び出して正しい結果を返すか確認する
E. プロンプトテンプレートに変数を埋め込んだ結果が正しいか確認する
F. 本番環境と同等の条件で100件のシナリオを連続実行して成功率を測定する
G. ツールの戻り値をパースして次のアクションを正しく決定するか確認する
H. メモリモジュールが会話履歴を正しく保存・取得するか確認する
```

**テストレベル：**

1. 単体テスト（Unit Test）
2. 統合テスト（Integration Test）
3. エンドツーエンドテスト（E2E Test）

**期待される出力例：**

```
1. 単体テスト：A, C, E, H
2. 統合テスト：D, G
3. エンドツーエンドテスト：B, F
```

<details>
<summary>ヒント</summary>

- 単体テストは「個別のコンポーネントが単独で正しく動作するか」を確認します。
- 統合テストは「複数のコンポーネントが連携して正しく動作するか」を確認します。
- E2Eテストは「ユーザー視点で全体のフローが正しく動作するか」を確認します。
</details>

<details>
<summary>解答例</summary>

```
1. 単体テスト（Unit Test）：A, C, E, H
   - A: プロンプト→出力形式の単一コンポーネントテスト
   - C: ツール単体の入出力テスト
   - E: テンプレートエンジン単体のテスト
   - H: メモリモジュール単体の読み書きテスト

2. 統合テスト（Integration Test）：D, G
   - D: 複数ツールの連携動作テスト（オーケストレーション層+ツール層）
   - G: LLM出力パース+アクション決定の連携テスト（推論層+実行層）

3. エンドツーエンドテスト（E2E Test）：B, F
   - B: ユーザー入力から完了までの全体フローテスト
   - F: 本番相当環境での大規模シナリオテスト

# テストピラミッドの原則：
# - 単体テストが最も多く（高速・低コスト・安定）
# - E2Eテストは最も少なく（低速・高コスト・不安定になりやすい）
# - エージェント開発ではLLMの非決定性があるため、
#   統合テスト以上では「許容範囲」を定義することが重要です
```
</details>

### 問題3：ベンチマークの分類

以下のベンチマークを、評価対象の能力別に分類してください。

**ベンチマーク一覧：**

```
A. SWE-bench（ソフトウェアのバグ修正タスク）
B. HumanEval（コード生成タスク）
C. WebArena（Webブラウザ操作タスク）
D. GAIA（汎用AIアシスタントタスク）
E. ToolBench（ツール使用タスク）
F. AgentBench（マルチターン対話エージェントタスク）
G. MLAgentBench（機械学習実験タスク）
H. OSWorld（OS操作タスク）
```

**分類カテゴリ：**

1. コーディング能力
2. ツール使用・環境操作能力
3. 汎用エージェント能力

**期待される出力例：**

```
1. コーディング能力：A, B, G
2. ツール使用・環境操作能力：C, E, H
3. 汎用エージェント能力：D, F
```

<details>
<summary>ヒント</summary>

- コーディング能力はソースコードの生成・修正・実験に関するものです。
- ツール使用・環境操作能力はAPIやブラウザ、OSなど外部環境との対話に関するものです。
- 汎用エージェント能力は特定ドメインに限定されない総合的な能力を測定するものです。
</details>

<details>
<summary>解答例</summary>

```
1. コーディング能力：A, B, G
   - A（SWE-bench）：実際のGitHubリポジトリのバグ修正能力を評価
   - B（HumanEval）：関数レベルのコード生成能力を評価
   - G（MLAgentBench）：ML実験の設計・実装・改善能力を評価

2. ツール使用・環境操作能力：C, E, H
   - C（WebArena）：Webサイト上でのナビゲーション・操作能力を評価
   - E（ToolBench）：多数のAPIツールの選択・呼び出し能力を評価
   - H（OSWorld）：デスクトップOS上でのファイル操作・アプリ操作能力を評価

3. 汎用エージェント能力：D, F
   - D（GAIA）：検索・計算・推論を組み合わせた汎用タスク能力を評価
   - F（AgentBench）：多様な環境での対話型問題解決能力を評価

# 補足：ベンチマーク選定のポイント
# - 自分のエージェントの用途に合ったベンチマークを選ぶことが重要です
# - 単一のベンチマークで全能力を測ることはできません
# - ベンチマークのスコアだけでなく、失敗パターンの分析が改善に役立ちます
```
</details>

### 問題4：エージェント評価スイートの設計

あなたは「社内ドキュメント検索エージェント」の評価スイートを設計する担当者です。このエージェントはユーザーの質問に対して社内ドキュメントを検索し、回答を生成します。

以下の要件に基づいて評価スイートを設計してください。

**エージェントの機能：**
- 自然言語の質問を受け付ける
- 社内ドキュメントDBを検索する
- 検索結果を要約して回答を生成する
- 回答の根拠となるドキュメントのリンクを提示する

**設計すべき内容：**
1. 評価指標（最低5つ）とその計算方法
2. テストケースのカテゴリ（最低4カテゴリ）と各カテゴリの具体例
3. 自動評価と人間評価の使い分け方針

**期待される出力例：**

```
【評価指標】
1. 回答正確性（Answer Accuracy）
   - 計算方法：正しい回答数 / 総質問数
   - 目標値：85%以上
   ...（以下同様に5つ以上）

【テストケースカテゴリ】
1. 単一ドキュメント参照型
   - 例：「有給休暇の申請方法を教えてください」
   ...（以下同様に4つ以上）

【自動評価と人間評価の使い分け】
- 自動評価：...
- 人間評価：...
```

<details>
<summary>ヒント</summary>

- 評価指標は「正確性」「関連性」「網羅性」「応答速度」「コスト」など多角的に考えましょう。
- テストケースは「簡単→難しい」「単一情報源→複数情報源」のように段階的に設計します。
- 自動評価はスケーラブルですが、回答の質の微妙なニュアンスは人間評価が必要です。
</details>

<details>
<summary>解答例</summary>

```
【評価指標】

1. 回答正確性（Answer Accuracy）
   - 計算方法：正解と判定された回答数 / 総質問数
   - 目標値：85%以上
   - 判定方法：正解データセットとの照合（自動+人間レビュー）

2. 検索適合率（Retrieval Precision）
   - 計算方法：関連ドキュメント取得数 / 総取得ドキュメント数
   - 目標値：80%以上
   - 判定方法：取得されたドキュメントの関連性を人間が評価

3. 検索再現率（Retrieval Recall）
   - 計算方法：取得された関連ドキュメント数 / 全関連ドキュメント数
   - 目標値：75%以上
   - 判定方法：正解ドキュメントセットとの照合

4. 応答時間（Response Latency）
   - 計算方法：質問受信から回答返却までの平均時間
   - 目標値：5秒以内
   - 判定方法：自動計測

5. 根拠提示率（Citation Rate）
   - 計算方法：根拠リンク付き回答数 / 総回答数
   - 目標値：95%以上
   - 判定方法：回答にリンクが含まれるかの自動チェック

6. 回答の幻覚率（Hallucination Rate）
   - 計算方法：根拠のない情報を含む回答数 / 総回答数
   - 目標値：5%以下
   - 判定方法：回答内容と参照ドキュメントの整合性チェック

【テストケースカテゴリ】

1. 単一ドキュメント参照型（難易度：低）
   - 例：「有給休暇の申請方法を教えてください」
   - 特徴：1つのドキュメントで完結する質問

2. 複数ドキュメント統合型（難易度：中）
   - 例：「東京オフィスと大阪オフィスの会議室予約ルールの違いは？」
   - 特徴：複数ドキュメントの情報を統合して回答する質問

3. 最新情報要求型（難易度：中〜高）
   - 例：「今年度のセキュリティポリシーの変更点は？」
   - 特徴：ドキュメントの更新日時を考慮する必要がある質問

4. 回答不可能型（難易度：特殊）
   - 例：「来年の会社の売上予測を教えてください」
   - 特徴：社内ドキュメントに情報がない質問（「わかりません」と正しく回答すべき）

5. あいまい質問型（難易度：高）
   - 例：「あの件、どうなった？」
   - 特徴：文脈が不足しており、確認が必要な質問

【自動評価と人間評価の使い分け】

- 自動評価で測定するもの：
  - 応答時間、根拠提示率、出力フォーマットの正確性
  - キーワードベースの正解照合（固有名詞、数値など）
  - 回帰テスト（既知の質問に対する回答の一貫性）
  - 実行頻度：毎デプロイ時 + 日次バッチ

- 人間評価で測定するもの：
  - 回答の自然さ・読みやすさ
  - 複数ドキュメント統合時の要約品質
  - 幻覚の検出（微妙なケース）
  - 回答不可能型の判定品質
  - 実行頻度：週次サンプリング（50件程度）
```
</details>

<summary>ヒント</summary>

- ステップ2と4を比較してみましょう。最初から適切なクエリを使えたはずです。

- 「理想的な実行」は何ステップで完了するか考えてみましょう。

<details>
<summary>解答例</summary>

```
【非効率なステップ】

- ステップ2（web_search: "天気"）：
  クエリが曖昧すぎて一般的な結果しか得られず、無駄な検索になっている。
  ユーザーが「東京の今日の天気」と明示しているのに、
  その情報をクエリに含めていない。

- ステップ6（web_search: "降水確率 何パーセント 傘 必要"）：
  降水確率60%で傘が必要かどうかは一般常識で判断可能。
  LLMの知識だけで十分回答できる内容であり、
  ツール呼び出しのコストと時間が無駄。

【根本原因】

1. クエリ生成の品質不足：
   ユーザー入力の重要な情報（東京、今日）を初回検索に反映できていない。
   プロンプトにクエリ生成のガイドラインが不足している可能性がある。

2. ツール使用判断の不適切さ：
   LLMの持つ知識で判断できる内容にまで外部ツールを使用している。
   「ツールを使うべきか」の判断基準が明確でない。

3. 計画フェーズの不足：
   実行前にタスクを分解し、必要な情報と手段を計画していない。
   行き当たりばったりで実行している。

【改善案】

1. クエリ最適化の指示をプロンプトに追加：
   「検索クエリを生成する際は、ユーザーの質問に含まれる
   固有名詞・日時・場所を必ず含めること」という指示を追加する。
   → ステップ2の無駄な検索を防止（1ステップ削減、約2.3秒短縮）

2. ツール使用の判断基準を明確化：
   「一般常識や広く知られている情報に基づく判断は、
   ツールを使わずにLLMの知識で回答すること」という指示を追加する。
   → ステップ6の不要な検索を防止（1ステップ削減、約1.9秒短縮）

3. 実行前の計画ステップを導入：
   最初のthinkステップで「必要な情報」「使用ツール」
   「期待されるステップ数」を明示的に計画させる。
   → 全体の実行効率が向上

【改善後の理想的な実行（4ステップ）】
   Step1: think → 計画：東京の今日の天気を検索し、降水確率から判断する
   Step2: web_search("東京 天気 今日") → 結果取得
   Step3: think → 降水確率60%なので傘が必要と判断
   Step4: respond → 回答

改善効果：
  - ステップ数：8 → 4（50%削減）
  - 推定時間：7800ms → 約3500ms（55%短縮）
  - 推定コスト：$0.0142 → 約$0.008（44%削減）
```

### 問題6：テストケースの作成

以下のエージェント仕様に対して、テストケースを作成してください。

**エージェント仕様：「会議スケジュール管理エージェント」**
- カレンダーAPIで予定の検索・作成・変更・削除が可能
- 参加者の空き時間を確認して最適な時間を提案
- 会議室の予約も同時に行う
- 日本語で自然な対話が可能

**作成するテストケース：**
1. 正常系テスト3件
2. 異常系テスト3件
3. エッジケーステスト2件

各テストケースには以下を含めてください。
- テストID
- テスト名
- 入力（ユーザーの発言）
- 期待される動作
- 期待される出力（エージェントの応答の要約）

**期待される出力例：**

```
【正常系テスト】
TC-001: 単純な会議作成
  入力：「明日の14時から1時間、田中さんと会議を設定して」

  期待出力：「明日14:00-15:00で田中さんとの会議を設定しました。会議室Aを予約しました。」
...
```

## チャレンジ問題

### 問題7：継続的改善パイプラインの設計

AIエージェントの品質を継続的に改善するためのパイプラインを設計してください。対象は「ECサイトの商品レコメンドエージェント」です。

**エージェントの概要：**
- ユーザーの購買履歴・閲覧履歴を分析
- 自然言語での質問に対して商品を推薦
- 推薦理由を説明
- 在庫状況・価格情報も提供

**設計すべきパイプライン：**

```
評価（Evaluate）→ 分析（Analyze）→ 改善（Improve）→ 再評価（Re-evaluate）
```

各フェーズについて以下を詳細に設計してください。

1. **評価フェーズ**：KPI定義、データ収集方法、評価の自動化
2. **分析フェーズ**：失敗パターンの分類、根本原因分析の手法
3. **改善フェーズ**：改善アクションの優先順位付け、A/Bテスト計画
4. **再評価フェーズ**：改善効果の測定、回帰テスト、ロールバック基準

さらに、パイプライン全体の運用ルール（実行頻度、担当者、エスカレーション基準）も定義してください。

**期待される出力例：**

```
=== 継続的改善パイプライン設計書 ===

【1. 評価フェーズ（Evaluate）】

■ KPI定義
| KPI名 | 計算方法 | 目標値 | 測定頻度 |
|---|---|---|---|
| レコメンド精度 | クリック数/表示数 | 15%以上 | 日次 |
| ... | ... | ... | ... |

■ データ収集方法
...

【2. 分析フェーズ（Analyze）】
...

【3. 改善フェーズ（Improve）】
...

【4. 再評価フェーズ（Re-evaluate）】
...

【運用ルール】
...
```

<details>
<summary>ヒント</summary>

- 評価フェーズではビジネスKPIとシステムKPIの両方を定義しましょう。
- 分析フェーズでは「なぜ失敗したか」を構造的に分類する枠組みが重要です。
- 改善フェーズではROI（投資対効果）に基づく優先順位付けが必要です。
- 再評価フェーズでは「改善が他の指標を悪化させていないか」の確認が重要です。
- 全体の運用では自動化できる部分と人間の判断が必要な部分を明確に分けましょう。
</details>

<details>
<summary>解答例</summary>

```
=== 継続的改善パイプライン設計書 ===
=== 対象：ECサイト商品レコメンドエージェント ===

【1. 評価フェーズ（Evaluate）】

■ KPI定義

| KPI名 | 計算方法 | 目標値 | 測定頻度 |
|---|---|---|---|
| レコメンドCTR | クリック数/表示数 | 15%以上 | 日次 |
| 購入転換率 | 購入数/クリック数 | 8%以上 | 日次 |
| 推薦理由の満足度 | 5段階評価の平均 | 4.0以上 | 週次 |
| 応答時間 | 平均レスポンスタイム | 2秒以内 | リアルタイム |
| API呼び出しコスト | 1推薦あたりの平均コスト | $0.005以下 | 日次 |
| カバレッジ率 | 推薦可能カテゴリ数/全カテゴリ数 | 95%以上 | 週次 |

■ データ収集方法

- ユーザーインタラクションログ：
  全リクエスト・レスポンスをトレーシングシステムに記録
  （リクエスト内容、推薦結果、クリック有無、購入有無）

- フィードバックデータ：
  推薦結果表示時に「役立った/役立たなかった」ボタンを設置
  週次でNPS（Net Promoter Score）調査を実施

- システムメトリクス：
  応答時間、トークン使用量、エラー率をPrometheus等で自動収集

■ 評価の自動化

- 日次バッチ：KPIダッシュボードの自動更新（Grafana）
- アラート：KPIが目標値を下回った場合にSlack通知
  - CTR 10%以下 → 警告（Yellow）
  - CTR 5%以下 → 緊急（Red）
- 週次レポート：自動生成してステークホルダーにメール配信

【2. 分析フェーズ（Analyze）】

■ 失敗パターンの分類

| 失敗カテゴリ | 具体例 | 検出方法 |
|---|---|---|
| 無関連推薦 | スポーツ用品を探しているのに食品を推薦 | CTR低下 + ユーザーフィードバック |
| 在庫切れ推薦 | 在庫のない商品を推薦 | 推薦→在庫API照合の自動チェック |
| 重複推薦 | 購入済み商品を再推薦 | 購買履歴との自動照合 |
| 価格帯不一致 | 予算を伝えているのに高額商品を推薦 | ユーザー入力と推薦価格の比較 |
| 説明品質低下 | 推薦理由が曖昧・不正確 | 満足度評価の低下 + 人間レビュー |

■ 根本原因分析の手法

1. エラー分類の自動タグ付け：
   失敗ケースを上記カテゴリに自動分類し、頻度を集計

2. 代表的失敗ケースの深掘り分析（週次）：
   各カテゴリの上位5件をトレーシングログで詳細分析
   - どのステップで判断を誤ったか
   - 使用した情報は正しかったか
   - プロンプトのどの部分が不足していたか

3. 傾向分析：
   時系列での失敗率変化を追跡
   特定の商品カテゴリ・ユーザーセグメントでの偏りを確認

【3. 改善フェーズ（Improve）】

■ 改善アクションの優先順位付け

優先度の決定基準（スコアリング方式）：
  影響度（1-5）× 実装容易性（1-5）× 緊急度（1-5）= 優先スコア

| 改善アクション | 影響度 | 容易性 | 緊急度 | スコア |
|---|:---:|:---:|:---:|:---:|
| 在庫チェックの事前フィルタ追加 | 4 | 5 | 5 | 100 |
| プロンプトへの価格帯考慮指示追加 | 4 | 4 | 4 | 64 |
| 購入済み商品の除外ロジック追加 | 3 | 5 | 3 | 45 |
| 推薦理由生成プロンプトの改善 | 3 | 3 | 3 | 27 |

■ A/Bテスト計画

- トラフィック分割：改善版20%、現行版80%（初期）
- 最小実行期間：7日間（曜日による偏りを排除）
- 統計的有意性：p < 0.05 で効果ありと判定
- 主要評価指標：CTRと購入転換率
- ガードレール指標：応答時間とエラー率（悪化していないこと）

【4. 再評価フェーズ（Re-evaluate）】

■ 改善効果の測定

- A/Bテスト結果の統計分析
- 改善前後のKPI比較レポート作成
- 副作用チェック：改善対象外のKPIが悪化していないか確認

■ 回帰テスト

- 既存のテストスイート（200件以上）を改善版で実行
- 合格基準：既存テストの成功率が95%以上を維持
- 新たに発見された失敗パターンはテストスイートに追加

■ ロールバック基準

以下のいずれかに該当する場合は即座にロールバック：
  - エラー率が改善前の2倍以上に増加
  - 応答時間が目標値（2秒）を超過
  - CTRが改善前より5ポイント以上低下
  - ユーザーからの苦情が急増（通常の3倍以上）

【運用ルール】

■ 実行頻度
- 評価フェーズ：日次（自動）+ 週次レポート
- 分析フェーズ：週次（毎週月曜）
- 改善フェーズ：スプリント単位（2週間）
- 再評価フェーズ：改善デプロイの1週間後

■ 担当者
- 評価・分析：MLエンジニア（自動化部分はシステム）
- 改善実装：開発チーム
- 再評価・ロールバック判断：テックリード

■ エスカレーション基準
- 緊急（Red）アラート → テックリードに即時報告
- 2スプリント連続でKPI未達 → マネージャーにエスカレーション
- ロールバック実施 → 振り返り会を開催し再発防止策を策定
```
</details>

### 問題1：アジャイルマニフェストの理解

以下のシナリオが、アジャイルマニフェストの4つの価値観のどれに関連するか答えてください。

1. 仕様書を100ページ作成する前に、まず動くプロトタイプを2週間で作ってユーザーに見せた。
2. 契約で決まったスコープを厳守するのではなく、市場の変化に合わせて優先機能を変更した。
3. メールで仕様を確認するのではなく、POとエンジニアが毎日15分間の打ち合わせで認識を合わせた。
4. Jiraの運用ルールを厳密に定めるより、チームメンバーが自律的に協力できる文化を育てた。

**期待される出力例：**

```
1. 「包括的なドキュメントよりも動くソフトウェア」
```

<details>
<summary>解答例</summary>

```
1. 「包括的なドキュメントよりも動くソフトウェア」
   → 100ページの仕様書よりも、2週間で動くプロトタイプを
     作ってフィードバックを得ることを優先した。

2. 「契約交渉よりも顧客との協調」
   → 契約で固定されたスコープに縛られるのではなく、
     顧客と協力して市場変化に対応した。

3. 「プロセスやツールよりも個人と対話」
   → メール（ツール）ではなく、直接の対話（15分の打ち合わせ）で
     コミュニケーションの質を高めた。

4. 「プロセスやツールよりも個人と対話」
   → Jira（ツール）の運用ルールよりも、
     チームメンバーの自律性と協力関係を重視した。
```

</details>

### 問題2：スクラムロールの判断

以下の行動は、スクラムの3つのロール（PO / SM / 開発チーム）のうち、誰が行うべきですか？

1. プロダクトバックログの優先順位を決める
2. デイリースクラムが15分を超えないようにする
3. スプリントバックログのタスクに対して「どう実装するか」を決める
4. チームの障害（例：テスト環境のアクセス権がない）を解決する
5. スプリントレビューで、次にどの機能を作るべきか判断する
6. レトロスペクティブをファシリテーションする

**期待される出力例：**

```
1. PO（プロダクトオーナー）
```

<details>
<summary>解答例</summary>

```
1. PO（プロダクトオーナー）
   → バックログの優先順位を決めるのはPOの最も重要な責務。
     ビジネス価値に基づいて判断する。

2. SM（スクラムマスター）
   → スクラムのルール（タイムボックス）を守る支援はSMの役割。
     ただし「管理」ではなく「促進」の立場。

3. 開発チーム
   → 「どう作るか」の技術的判断は開発チームが自律的に決める。
     POやSMが実装方法を指示するのはスクラムに反する。

4. SM（スクラムマスター）
   → チームの障害を除去するのはSMの主要な責務。
     環境アクセス権の取得をIT部門と調整する。

5. PO（プロダクトオーナー）
   → 次に何を作るかの判断はPO。スプリントレビューでの
     フィードバックを基に、バックログを調整する。

6. SM（スクラムマスター）
   → レトロスペクティブのファシリテーションはSMの役割。
     チームが安心して振り返りできる場を作る。

【よくある間違い】
× PMがバックログの優先順位を決める → POの責務
× PMが「こう実装しろ」と指示する → 開発チームの自律性を侵害
× SMが進捗を管理する → SMは管理者ではなくサーバントリーダー
```

</details>

### 問題3：ベロシティとスプリント計画

チームの過去5スプリントのベロシティデータが以下の通りです。

| スプリント | 完了SP |
|-----------|-------|
| SP1 | 18 |
| SP2 | 22 |
| SP3 | 25 |
| SP4 | 28 |
| SP5 | 27 |

1. 平均ベロシティを求めてください
2. 次のスプリント（SP6）で取り込むべきSPの目安を答えてください
3. 残りバックログが80SPの場合、完了までのスプリント数を予測してください（楽観・現実的・悲観）

<details>
<summary>解答例</summary>

```
■ 1. 平均ベロシティ
  (18 + 22 + 25 + 28 + 27) / 5 = 120 / 5 = 24 SP

■ 2. 次のスプリントの取り込み目安
  直近3スプリントの平均がより実態に近い：
  (25 + 28 + 27) / 3 ≈ 26.7 SP → 約25〜27 SP

  推奨：25 SPを目安に計画する。
  理由：チームは上昇トレンドにあるが、過度な期待は避ける。

■ 3. 残り80SPの完了予測

  楽観（最高ベロシティ: 28 SP）：
    80 / 28 ≈ 2.9 → 3スプリント

  現実的（直近3SP平均: 27 SP）：
    80 / 27 ≈ 3.0 → 3スプリント

  悲観（全体平均: 24 SP）：
    80 / 24 ≈ 3.3 → 4スプリント

  → 3〜4スプリント（6〜8週間）で完了見込み

■ 補足
  ・SP1(18)は立ち上げ期の数値と思われるため、
    直近3スプリントの平均を使う方が精度が高い
  ・ベロシティが安定してきている（SP3〜SP5: 25〜28）ので
    予測の信頼性は比較的高い
  ・ただし、祝日やメンバーの休暇は別途考慮が必要
```

</details>

### 問題4：カンバンボードの設計

あなたのチームは、Webアプリの運用・保守を担当しています。以下の作業が日々発生します。

- バグ修正（緊急度：高〜低）
- 機能改善リクエスト
- インフラの定期メンテナンス
- セキュリティパッチの適用
- ドキュメント更新

カンバンボードを設計してください。列の名前、各列のWIP制限、緊急チケットの扱いを定義してください。

<details>
<summary>解答例</summary>

```
■ カンバンボードの設計

┌──────────┬─────────┬──────────┬─────────┬──────────┬─────────┐
│ Backlog  │ Ready   │ In Dev   │ Review  │ Testing  │  Done   │
│          │ (WIP:5) │ (WIP:3)  │ (WIP:2) │ (WIP:2)  │         │
├──────────┼─────────┼──────────┼─────────┼──────────┼─────────┤
│          │         │          │         │          │         │
│ 通常レーン│         │          │         │          │         │
│          │         │          │         │          │         │
├──────────┼─────────┼──────────┼─────────┼──────────┼─────────┤
│          │         │          │         │          │         │
│ 緊急レーン│  WIP制限に含めない（即時対応）              │         │
│          │         │          │         │          │         │
└──────────┴─────────┴──────────┴─────────┴──────────┴─────────┘

■ 列の定義

| 列名     | 定義                              | WIP制限 |
|---------|-----------------------------------|---------|
| Backlog | 未着手の全チケット。優先度順に並べる | なし    |
| Ready   | 着手可能な状態。要件が明確         | 5       |
| In Dev  | 開発中                            | 3       |
| Review  | コードレビュー待ち / レビュー中    | 2       |
| Testing | テスト中                          | 2       |
| Done    | 完了（デプロイ済み）               | なし    |

■ WIP制限の根拠
  チーム4名の場合：
  ・In Dev: 3 → 1名はレビューやペアプロに回れるように
  ・Review: 2 → レビュー待ちの滞留を防止
  ・Testing: 2 → テスト作業の集中を確保

■ 緊急チケットの扱い

  分類基準：
  ・P0（緊急）：本番障害、セキュリティ脆弱性 → 緊急レーン
  ・P1（高）：重要機能のバグ → 通常レーンの先頭に挿入
  ・P2（中）：機能改善、軽微なバグ → 通常レーンで優先度付け
  ・P3（低）：ドキュメント更新等 → 通常レーン

  緊急レーンのルール：
  ・WIP制限に含めない（割り込みを許可）
  ・ただし緊急レーンは同時に1件まで
  ・緊急対応中は通常レーンの新規着手を停止
  ・対応完了後、根本原因分析（RCA）を実施

■ 運用ルール
  ・毎朝10分のスタンドアップで滞留チケットを確認
  ・WIP制限を超えたら新規着手禁止（既存チケットの完了を優先）
  ・週次で「リードタイム」「スループット」を計測
  ・月次でWIP制限とプロセスを見直す
```

</details>

### 問題5：レトロスペクティブの実施

以下のスプリントの状況を読んで、KPT形式でレトロスペクティブの結果をまとめてください。

**Sprint 8の状況：**
- ベロシティ：20SP（目標25SP、前回28SP）
- 原因：途中でメンバー1名がインフルエンザで3日間離脱
- デプロイは自動化できており、リリース作業は30分で完了
- コードレビューのSLAを導入した結果、レビュー時間が3日→1日に改善
- しかし、レビューコメントが表面的になったとの声がある
- POとの認識齟齬で1つのストーリーが手戻りになった
- 新メンバーが加入し、ペアプロでオンボーディングを進行中

<details>

## チャレンジ問題

### 問題6：スクラム導入計画

あなたの会社（50名のエンジニア組織）では、これまでウォーターフォール型で開発してきました。経営陣から「アジャイル開発を全社に導入したい」と言われました。

導入計画を作成してください。以下の観点を含めてください。
- フェーズ分け（段階的導入）
- パイロットチームの選定基準
- 研修計画
- 成功指標
- 想定される抵抗と対策

<details>
<summary>解答例</summary>

```
■ アジャイル導入計画（全社50名エンジニア組織）

【基本方針】
一度に全社導入するのではなく、パイロットチームで成功体験を作り、
段階的に展開する。教条的なスクラムの押しつけではなく、
各チームの状況に合わせたカスタマイズを許容する。

───────────────────────────────
Phase 1: 準備（1〜2ヶ月目）
───────────────────────────────
□ アジャイルコーチの採用または外部コンサルの契約
□ 経営層へのアジャイル研修（半日）
  ・アジャイルの価値観と原則
  ・経営層の役割（障害除去、リソース確保）
□ パイロットチームの選定
  選定基準：
  ・チームサイズ：5〜7名
  ・メンバーの柔軟性が高い（変化に前向き）
  ・POが明確で、密にコミュニケーションできる
  ・プロジェクトの規模が中程度（3〜6ヶ月）
  ・失敗しても致命的でないプロジェクト
□ パイロットチームへの研修（2日間）
  ・スクラムの基礎
  ・ロールプレイ（PO、SM、開発チーム）
  ・ツール研修（Jira等）

───────────────────────────────
Phase 2: パイロット（3〜5ヶ月目）
───────────────────────────────
□ パイロットチーム（1チーム）でスクラムを実践
  ・2週間スプリント
  ・全イベントを実施（プランニング、デイリー、レビュー、レトロ）
  ・アジャイルコーチがSMを兼任
□ 週次で振り返りと改善
□ メトリクスの収集
  ・ベロシティの推移
  ・チーム満足度
  ・バグ発生率
  ・リードタイム
□ 3ヶ月後にパイロットの評価レポートを作成

───────────────────────────────
Phase 3: 展開（6〜9ヶ月目）
───────────────────────────────
□ パイロットの成功事例を全社に共有
□ 追加2〜3チームへ展開
□ 各チームからSM候補を育成
□ チーム間の連携方法を確立（スクラム・オブ・スクラムズ）
□ 組織のプロセス・ルールをアジャイルに適合させる
  ・評価制度の見直し（個人→チーム成果の重視）
  ・予算承認プロセスの柔軟化

───────────────────────────────
Phase 4: 定着（10〜12ヶ月目以降）
───────────────────────────────
□ 全チームへの展開完了
□ 社内アジャイルコミュニティの設立
□ 継続的な研修・勉強会の運営
□ スケーリングの検討（SAFe、LeSS等）

【成功指標】
| 指標                    | 目標                    |
|------------------------|------------------------|
| チーム満足度            | 4.0/5.0以上             |
| リリース頻度            | 月1回 → 隔週            |
| バグ発生率              | 20%削減                 |
| リードタイム            | 30%短縮                 |
| ベロシティの安定性       | 変動係数20%以内          |

【想定される抵抗と対策】

| 抵抗                              | 対策                               |
|----------------------------------|-----------------------------------|
| 「ドキュメントがなくなるのでは？」   | 必要なドキュメントは書く。            |
|                                  | アジャイル≠ドキュメント不要と説明     |
| 「見積もりが出せないと契約できない」 | ベロシティベースの予測を説明。         |
|                                  | ハイブリッドアプローチも検討          |
| 「毎日ミーティングは無駄だ」       | 15分のスタンドアップの目的を説明。    |
|                                  | 効果を体験で実感してもらう            |
| 「PMの仕事がなくなるのでは？」     | PMはPOやSMとして活躍できると説明。   |
|                                  | キャリアパスを明示する               |
| 「今のやり方で問題ない」           | 現状の課題（リリース遅延等）を         |
|                                  | データで示し、改善の必要性を共有       |
```

</details>

## 課題の要件

1. **カスタマーサポートエージェント**を評価対象として、評価フレームワークを設計すること
   - エージェントの概要: FAQ検索、チケット作成、エスカレーションを行うサポートエージェント
2. 以下の**4つの評価次元**について、具体的な指標を定義すること
   - タスク達成率（Task Completion Rate）
   - ツール使用の適切性（Tool Usage Accuracy）
   - 回答品質（Response Quality）
   - 安全性・コンプライアンス（Safety & Compliance）
3. 各評価次元について**テストケース**を5件以上設計すること（合計20件以上）
4. テストケースの**自動判定方法**（LLM-as-a-Judge / ルールベース / 人手）を明記すること
5. **スコアリング基準**と**合格ライン**を定義すること
6. **回帰テスト戦略**（モデル更新時にどう評価するか）を設計すること

## ステップガイド

<details>
<summary>ステップ1：評価次元と指標を定義する</summary>

各評価次元について「何を測るか」「どう測るか」を明確にしましょう。

| 評価次元 | 測定対象 | 指標の例 |
|---------|---------|---------|
| タスク達成率 | 正しくタスクを完了できたか | 成功/部分成功/失敗の3段階 |
| ツール使用適切性 | 正しいツールを正しい引数で呼べたか | ツール選択正答率、引数正答率 |
| 回答品質 | 回答内容の正確性・有用性 | 正確性スコア、有用性スコア（1-5） |
| 安全性 | 禁止行動を行わないか | 違反率、プロンプトインジェクション耐性 |

</details>

<details>
<summary>ステップ2：テストケースを設計する</summary>

各テストケースには以下の要素を含めましょう。

```json
{
  "test_id": "TC001",
  "category": "task_completion",
  "input": "ユーザーの質問や依頼",
  "expected_behavior": "期待されるエージェントの行動",
  "expected_output": "期待される出力（正解）",
  "evaluation_method": "auto_rule | llm_judge | human",
  "difficulty": "easy | medium | hard"
}
```

**よくある間違い**: テストケースが「正常系」ばかりになること。境界ケース（曖昧な質問、矛盾する質問）や異常系（存在しない情報を聞く、禁止行動を要求する）も必ず含めましょう。

</details>

<details>
<summary>ステップ3：スコアリングと合格ラインを決める</summary>

各評価次元のスコアリング方法と、リリース判定の閾値を決めます。

考慮すべきポイント:
- すべての次元で均等にスコアを付けるか、重み付けするか
- 1つの次元だけ低い場合、総合スコアが合格ラインでもリリースを止めるか
- 安全性は他の次元より厳しい基準にすべきか

</details>

<details>
<summary>ステップ4：自動化と回帰テストを設計する</summary>

モデルを更新するたびに全テストを手動で実行するのは非現実的です。

自動化のポイント:
- テストケースの入力をAPIに投げる
- 出力を自動判定ルールまたはLLM-as-a-Judge で評価する
- 結果をダッシュボードに集約する
- 前回のスコアとの比較（回帰検知）

</details>

### 1. 評価次元と指標

#### 次元1: タスク達成率

| 指標名 | 定義 | 計算方法 |
|--------|------|---------|
| 完全達成率 | ユーザーの要求を完全に満たした割合 | 完全達成数 / 総テスト数 |
| 部分達成率 | 一部は満たしたが不完全だった割合 | 部分達成数 / 総テスト数 |
| 失敗率 | 全く要求を満たせなかった割合 | 失敗数 / 総テスト数 |

#### 次元2: ツール使用適切性

| 指標名 | 定義 | 計算方法 |
|--------|------|---------|
| ツール選択正答率 | 正しいツールを選べた割合 | 正しい選択数 / ツール呼び出し総数 |
| 引数正答率 | 正しい引数でツールを呼べた割合 | 正しい引数数 / 引数総数 |
| 不要呼び出し率 | 不必要なツール呼び出しの割合 | 不要な呼び出し数 / 呼び出し総数 |

#### 次元3: 回答品質

| 指標名 | 定義 | 評価方法 |
|--------|------|---------|
| 正確性 | 回答内容が事実と一致するか | LLM-as-a-Judge（1-5スケール） |
| 有用性 | ユーザーの問題解決に役立つか | LLM-as-a-Judge（1-5スケール） |
| 明瞭性 | 回答がわかりやすいか | LLM-as-a-Judge（1-5スケール） |

#### 次元4: 安全性

| 指標名 | 定義 | 評価方法 |
|--------|------|---------|
| 禁止行動違反率 | 禁止されたアクションを行った割合 | ルールベース自動判定 |
| 情報漏洩率 | 機密情報を出力した割合 | キーワード検出 + LLM判定 |
| インジェクション耐性 | 攻撃プロンプトを拒否できた割合 | ルールベース自動判定 |

### 3. スコアリング基準

| 評価次元 | 重み | 合格ライン | 計算方法 |
|---------|------|-----------|---------|
| タスク達成率 | 30% | 80%以上 | 完全達成率 |
| ツール使用適切性 | 25% | 85%以上 | ツール選択正答率 |
| 回答品質 | 25% | 平均3.5/5以上 | 3指標の平均 |
| 安全性 | 20% | 95%以上 | (1 - 違反率) |

**総合スコア** = Σ（各次元スコア × 重み）

**リリース判定**:
- 総合スコア 80%以上 AND 安全性 95%以上 → リリースOK
- 安全性が95%未満 → 他がどれだけ高くてもリリースNG

### 4. 回帰テスト戦略

モデル更新時に以下のテストを実行します。

1. **全テストスイートの実行**（20件以上）
2. **前回スコアとの比較**: 各次元のスコアが5%以上低下した場合はアラート
3. **結果レポート**の自動生成

### 2. LLM-as-a-Judge の評価プロンプト

```python
# 擬似コード: LLM-as-a-Judge の実装

JUDGE_PROMPT_TEMPLATE = """
あなたはAIエージェントの品質を評価する審査員です。
以下のやりとりを評価してください。

## 評価対象
ユーザーの質問: {user_input}
エージェントの回答: {agent_response}
期待される行動: {expected_behavior}

## 評価基準
以下の各項目を1〜5で評価してください。

1. 正確性（Accuracy）: 回答内容は事実に基づいているか
   - 5: 完全に正確
   - 4: ほぼ正確（軽微な不正確さあり）
   - 3: 部分的に正確
   - 2: 重要な誤りあり
   - 1: 大部分が不正確

2. 有用性（Helpfulness）: ユーザーの問題解決に役立つか
   - 5: 問題を完全に解決
   - 4: 問題をほぼ解決
   - 3: 部分的に役立つ
   - 2: あまり役立たない
   - 1: 全く役立たない

3. 明瞭性（Clarity）: 回答がわかりやすいか
   - 5: 非常にわかりやすい
   - 4: わかりやすい
   - 3: 普通
   - 2: わかりにくい
   - 1: 非常にわかりにくい

## 出力形式
以下のJSON形式で回答してください。
{{
  "accuracy": <1-5>,
  "helpfulness": <1-5>,
  "clarity": <1-5>,
  "reasoning": "<評価の理由を簡潔に>"
}}
"""

def evaluate_with_llm_judge(user_input, agent_response, expected_behavior):
    """LLM-as-a-Judgeで回答品質を評価する"""

    prompt = JUDGE_PROMPT_TEMPLATE.format(
        user_input=user_input,
        agent_response=agent_response,
        expected_behavior=expected_behavior
    )

    # 評価のブレを減らすため、3回実行して中央値を取る
    scores = []
    for _ in range(3):
        result = llm.generate(prompt, temperature=0.0)
        scores.append(json.loads(result))

    return {
        "accuracy": median([s["accuracy"] for s in scores]),
        "helpfulness": median([s["helpfulness"] for s in scores]),
        "clarity": median([s["clarity"] for s in scores]),
        "reasoning": scores[0]["reasoning"]  # 1回目の理由を使用
    }
```

### 3. 評価パイプラインの擬似コード

```python
# 擬似コード: 評価パイプラインの全体構成

class EvaluationPipeline:
    def __init__(self, agent, test_suite):
        self.agent = agent
        self.test_suite = test_suite
        self.results = []

    def run(self):
        """全テストを実行し、結果を集計する"""

        for test_case in self.test_suite:
            # 1. エージェントにテスト入力を投入
            response = self.agent.run(test_case.input)

            # 2. 評価方法に応じてスコアリング
            if test_case.evaluation_method == "auto_rule":
                score = self.evaluate_by_rule(test_case, response)
            elif test_case.evaluation_method == "llm_judge":
                score = self.evaluate_by_llm_judge(test_case, response)
            elif test_case.evaluation_method == "human":
                score = self.queue_for_human_review(test_case, response)

            self.results.append({
                "test_id": test_case.test_id,
                "category": test_case.category,
                "score": score,
                "response": response,
                "pass": score >= test_case.threshold
            })

        # 3. 集計とレポート生成
        return self.generate_report()

    def evaluate_by_rule(self, test_case, response):
        """ルールベースの自動評価"""
        score = 1.0

        # 期待されるツール呼び出しのチェック
        if test_case.expected_tool_calls:
            actual_tools = [t.name for t in response.tool_calls]
            expected_tools = [t["tool"] for t in test_case.expected_tool_calls]
            if actual_tools != expected_tools:
                score -= 0.5

        # 禁止出力のチェック
        if test_case.forbidden_output_contains:
            for forbidden in test_case.forbidden_output_contains:
                if forbidden in response.text:
                    score = 0.0  # 禁止出力があれば即0点
                    break

        # 必須キーワードのチェック
        if test_case.expected_output_contains:
            for keyword in test_case.expected_output_contains:
                if keyword not in response.text:
                    score -= 0.2

        return max(0.0, score)

    def generate_report(self):
        """評価レポートを生成する"""
        report = {
            "summary": {
                "total_tests": len(self.results),
                "passed": sum(1 for r in self.results if r["pass"]),
                "failed": sum(1 for r in self.results if not r["pass"]),
                "pass_rate": sum(1 for r in self.results if r["pass"]) / len(self.results)
            },
            "by_category": {},
            "failed_tests": [r for r in self.results if not r["pass"]],
            "timestamp": datetime.now().isoformat()
        }

        # カテゴリ別集計
        for category in ["task_completion", "tool_usage", "response_quality", "safety"]:
            cat_results = [r for r in self.results if r["category"] == category]
            report["by_category"][category] = {
                "pass_rate": sum(1 for r in cat_results if r["pass"]) / len(cat_results),
                "avg_score": sum(r["score"] for r in cat_results) / len(cat_results)
            }

        return report
```

### 4. 回帰テスト戦略（改良版）

```python
# 擬似コード: 回帰テスト + 統計的有意性検定

class RegressionTestRunner:
    def __init__(self, baseline_results, current_agent):
        self.baseline = baseline_results  # 前回のテスト結果
        self.current_agent = current_agent

    def run_regression_test(self, test_suite, n_runs=5):
        """回帰テストを複数回実行し、統計的に比較する"""

        # LLMの出力は非決定的なので、複数回実行して統計を取る
        all_results = []
        for i in range(n_runs):
            pipeline = EvaluationPipeline(self.current_agent, test_suite)
            results = pipeline.run()
            all_results.append(results)

        # カテゴリ別に前回との比較
        comparison = {}
        for category in ["task_completion", "tool_usage", "response_quality", "safety"]:
            current_scores = [
                r["by_category"][category]["avg_score"]
                for r in all_results
            ]
            baseline_score = self.baseline["by_category"][category]["avg_score"]

            avg_current = sum(current_scores) / len(current_scores)
            degradation = baseline_score - avg_current

            comparison[category] = {
                "baseline": baseline_score,
                "current_avg": avg_current,
                "current_std": std_dev(current_scores),
                "degradation": degradation,
                "alert": degradation > 0.05  # 5%以上の低下でアラート
            }

        return comparison
```

**プロの知見**: エージェントの評価で最も見落とされがちなのは「評価の評価」（メタ評価）です。LLM-as-a-Judge を使う場合、Judge自体の精度が低ければ評価結果も信頼できません。定期的に人手評価との一致率を計測し、Judgeプロンプトを改善するサイクルを回すことが重要です。また、テストケースは「過去に発生した実際の問題」から優先的に作成すると、実用的なテストスイートになります。

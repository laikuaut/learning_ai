## 課題の要件

1. **ソフトウェア開発チーム**を模した5体のエージェントシステムを設計すること
   - PM（プロジェクトマネージャー）: 要件整理、タスク分解、進捗管理
   - Architect（アーキテクト）: 設計、技術選定、設計レビュー
   - Developer（デベロッパー）: コード実装、デバッグ
   - Tester（テスター）: テスト設計、テスト実行、バグ報告
   - Reviewer（レビュアー）: コードレビュー、品質チェック
2. 以下の設計書一式を作成すること
   - **エージェント定義書**（5体すべて: ロール、プロンプト、ツール、入出力）
   - **ワークフロー設計**（タスクの流れ、分岐、ループ）
   - **通信プロトコル**（メッセージ形式、共有メモリ）
   - **状態管理設計**（プロジェクト全体の状態遷移）
   - **品質ゲート設計**（各フェーズの合格条件）
   - **エラーハンドリング**（各エージェントの失敗時の復旧）
   - **運用設計**（監視、KPI、改善サイクル）
3. 具体的な開発タスク例（「REST APIにユーザー登録エンドポイントを追加」）を用いて、**全エージェントが協調して動作する実行トレース例**を記述するこ��
4. **擬似コード**でシステム全体のオーケストレーションロジックを記述すること
5. このシステムの**限界と将来の拡張**について考察すること

| エージェント | する | しない |
|------------|------|--------|

| Developer | コード実装、単体テスト | 要件定義、設計判断 |

             └────── フィードバック ───┘

**よくある間違い**: 一方向のパイプラインだけを設計してしまうこと。実際の開発では「レビューで問題発覚 → 設計からやり直し」のようなフィードバックループが頻繁に発生します。

| フェーズ | 品質ゲート | 判定者 |
|---------|-----------|--------|
| 要件定義 | 要件が明確で矛盾がない | PM |

| 実装 | コードが設計に従い、単体テスト通過 | Developer → Reviewer |
| テスト | テスト計画の実行完了、バグなし | Tester |
| レビュー | コード品質基準を満たす | Reviewer |

### 6. エラーハンドリング

| エラー | 対応 |
|--------|------|
| Architectの設計が不明確 | PMが追加要件をヒアリングし、Architectに補足情報を提供 |
| Developerの実装が設計と異なる | Reviewerが検出 → 設計書との差分を指摘 → 修正 |
| テストが3回以上失敗 | PMがArchitectに設計の見直しを依頼 |
| レビューが3回以上却下 | PMがDeveloper + Architectの3者でミーティングを設定 |
| エージェントがタイムアウト | PMが検知し、タスクを再割り当て |

### 1. 状態遷移マシン

```python
# 擬似コード: プロジェクト状態管理

class ProjectState:
    """プロジェクト全体の状態"""
    REQUIREMENTS = "requirements"       # 要件定義中
    DESIGN = "design"                   # 設計中
    DESIGN_REVIEW = "design_review"     # 設計レビュー中
    IMPLEMENTATION = "implementation"   # 実装中
    TESTING = "testing"                 # テスト中
    BUG_FIX = "bug_fix"               # バグ修正中
    CODE_REVIEW = "code_review"        # コードレビュー中
    REVISION = "revision"              # 修正中（レビュー指摘対応）
    COMPLETED = "completed"            # 完了
    FAILED = "failed"                  # 失敗（人間にエスカレーション）

class TaskState:
    """個別タスクの状態"""
    BACKLOG = "backlog"
    IN_PROGRESS = "in_progress"
    IN_REVIEW = "in_review"
    DONE = "done"
    BLOCKED = "blocked"

class DevelopmentWorkflow:
    # 状態遷移テーブル
    TRANSITIONS = {
        "requirements": {
            "requirements_done": "design",
            "requirements_unclear": "requirements",  # ユーザーに再確認
        },
        "design": {
            "design_done": "design_review",
        },
        "design_review": {
            "design_approved": "implementation",
            "design_rejected": "design",  # 設計やり直し
        },
        "implementation": {
            "implementation_done": "testing",
        },
        "testing": {
            "all_tests_passed": "code_review",
            "bugs_found": "bug_fix",
        },
        "bug_fix": {
            "fix_done": "testing",  # 再テスト
            "fix_failed_max_retries": "failed",
        },
        "code_review": {
            "review_approved": "completed",
            "review_changes_requested": "revision",
        },
        "revision": {
            "revision_done": "code_review",  # 再レビュー
            "revision_needs_redesign": "design",  # 設計からやり直し
        },
    }

    def __init__(self):
        self.state = ProjectState.REQUIREMENTS
        self.bug_fix_count = 0
        self.revision_count = 0
        self.max_retries = 3

    def transition(self, event):
        """状態遷移を実行する"""
        transitions = self.TRANSITIONS.get(self.state, {})
        if event not in transitions:
            raise InvalidTransition(
                f"状態 '{self.state}' でイベント '{event}' は無効です"
            )

        # リトライ回数のチェック
        if event == "bugs_found":
            self.bug_fix_count += 1
            if self.bug_fix_count > self.max_retries:
                self.state = ProjectState.FAILED
                return

        if event == "review_changes_requested":
            self.revision_count += 1
            if self.revision_count > self.max_retries:
                self.state = ProjectState.FAILED
                return

        self.state = transitions[event]
```

```python
class DevTeamOrchestrator:
    """開発チーム型マルチエージェントシステムのオー��ストレータ"""

    def __init__(self, llm, tools):
        # エージェントの初期化
        self.pm = Agent(
            name="pm",
            system_prompt=PM_PROMPT,
            llm=llm,
            tools=[tools.create_task, tools.assign_task, tools.update_status]
        )
        self.architect = Agent(
            name="architect",
            system_prompt=ARCHITECT_PROMPT,
            llm=llm,
            tools=[tools.read_codebase, tools.create_design_doc]
        )
        self.developer = Agent(
            name="developer",
            system_prompt=DEVELOPER_PROMPT,
            llm=llm,
            tools=[tools.read_file, tools.write_file, tools.run_tests, tools.run_linter]
        )
        self.tester = Agent(
            name="tester",
            system_prompt=TESTER_PROMPT,
            llm=llm,
            tools=[tools.run_tests, tools.create_bug_report, tools.check_coverage]
        )
        self.reviewer = Agent(
            name="reviewer",
            system_prompt=REVIEWER_PROMPT,
            llm=llm,
            tools=[tools.read_file, tools.analyze_code, tools.check_security]
        )

        self.workflow = DevelopmentWorkflow()
        self.shared_memory = SharedMemory()
        self.message_bus = MessageBus()

    def execute(self, user_request):
        """開発リクエストを処理する"""

        # Phase 1: 要件定義
        requirements = self.pm.run(
            f"以下の開発要求を要件定義してください: {user_request}"
        )
        self.shared_memory.store("requirements", requirements)
        self.workflow.transition("requirements_done")

        # Phase 2: 設計
        design = self.architect.run(
            f"以下の要件に基づいて技術設計を行ってください:\n{requirements}"
        )
        self.shared_memory.store("design", design)
        self.workflow.transition("design_done")

        # Phase 2.5: 設計レビュー
        design_review = self.pm.run(
            f"以下の設計書が要件を満たしているかレビューしてください:\n"
            f"要件:\n{requirements}\n設計:\n{design}"
        )

        if not design_review.approved:
            self.workflow.transition("design_rejected")
            # 設計のやり直し（再帰的に処理）
            design = self.architect.run(
                f"設計にフィードバックがあります。修正してください:\n"
                f"{design_review.feedback}"
            )
            self.shared_memory.store("design", design)

        self.workflow.transition("design_approved")

        # Phase 3: 実装
        implementation = self.developer.run(
            f"以下の設計書に従って実装してください:\n{design}"
        )
        self.shared_memory.store("code", implementation)
        self.workflow.transition("implementation_done")

        # Phase 4: テスト（バグ修正ループ付き）
        for test_round in range(self.workflow.max_retries + 1):
            test_result = self.tester.run(
                f"以下のコードをテストしてください:\n"
                f"要件:\n{requirements}\n設計:\n{design}\n"
                f"コード:\n{implementation}"
            )

            if test_result.all_passed:
                self.workflow.transition("all_tests_passed")
                break
            else:
                self.workflow.transition("bugs_found")
                # バグ修正
                implementation = self.developer.run(
                    f"以下のバグを修正してください:\n{test_result.bug_reports}"
                )
                self.shared_memory.store("code", implementation)
                self.workflow.transition("fix_done")

        # Phase 5: コードレビュー（修正ループ付き）
        for review_round in range(self.workflow.max_retries + 1):
            review_result = self.reviewer.run(
                f"以下のコードをレビューしてください:\n"
                f"設計:\n{design}\nコード:\n{implementation}\n"
                f"テスト結果:\n{test_result}"
            )

            if review_result.approved:
                self.workflow.transition("review_approved")
                break
            else:
                self.workflow.transition("review_changes_requested")
                # 修正
                implementation = self.developer.run(
                    f"レビューコメントに対応してください:\n"
                    f"{review_result.comments}"
                )
                self.shared_memory.store("code", implementation)
                self.workflow.transition("revision_done")

        # 完了レポート
        return self._generate_completion_report()

    def _generate_completion_report(self):
        """完了レポートを生成する"""
        return self.pm.run(
            f"以下の情報をもとに完了レポートを生成してください:\n"
            f"要件: {self.shared_memory.get('requirements')}\n"
            f"設計: {self.shared_memory.get('design')}\n"
            f"実装ファイル: {self.shared_memory.get('code')}\n"
            f"テスト結果: {self.shared_memory.get('test_results')}\n"
            f"レビュー結果: {self.shared_memory.get('review_results')}"
        )
```

### 5. 限界と将来の拡張

#### 現在のシステムの限界

| 限界 | 説明 | 影響 |
|------|------|------|
| **シングルタスク前提** | 1つの開発リクエストを逐次処理するため、複数タスクの並列管理ができない | スループットが低い |
| **コンテキストの制限** | 大規模なコードベース全体を把握できない（コンテキストウィンドウの制約） | 既存コードとの整合性が取れない場合がある |
| **創造的な設計判断** | アーキテクトの設計は既知のパターンの組み合わせに限られ、革新的な設計は困難 | 定型的なCRUDには強いが、新しいアーキテクチャの提案は苦手 |
| **暗黙知の不足** | チーム固有の慣習、コーディングスタイル、過去の経緯を知らない | 既存チームの文化に合わないコードを生成する可能性 |
| **実行環境の制約** | 実際のサーバー起動やブラウザテストなどの環境依存テストができない | 統合テストやE2Eテストは人間が行う必要がある |

#### 将来の拡張案

```
Phase 1（短期）: 基本機能の改善
├ タスクの並列実行（設計中に別タスクのテスト進行など）
├ コードベースの長期メモリ（RAGベース）
└ ユーザーフィードバックからの学習

Phase 2（中期）: 高度な機能の追加
├ DevOpsエージェントの追加（CI/CD連携、デプロイ）
├ ドキュメンテーションエージェント（API docs自動生成）
├ セキュリティ専門エージェント（脆弱性スキャン）
└ 複数プロジェクトの同時管理

Phase 3（長期）: 自律的な開発チーム
├ 自己改善メカニズム（過去の失敗から学ぶ）
├ プロアクティブな提案（技術的負債の検出・報告）
├ 人間の開発者との協業（コペイロットではなくチームメイト）
└ プロジェクト横断的な知識の蓄積と活用
```

## 基本問題

### 問題1：4要素の特定

以下のプロンプトから「役割」「指示」「文脈」「出力形式」の4要素を特定してください。

```
あなたはデータベースの専門家です。以下のSQLクエリを最適化してください。
このクエリは月次レポート生成で使用されており、現在30秒かかっています。
目標は5秒以内です。改善点を箇条書きで3つ挙げ、最適化後のSQLも
提供してください。
```

<details>
<summary>解答例</summary>

```
役割：「あなたはデータベースの専門家です」
指示：「以下のSQLクエリを最適化してください」
文脈：「このクエリは月次レポート生成で使用されており、
       現在30秒かかっています。目標は5秒以内です。」
出力形式：「改善点を箇条書きで3つ挙げ、最適化後のSQLも
          提供してください」
```
</details>

### 問題2：曖昧な指示の改善

以下の曖昧なプロンプトを、明確で具体的なプロンプトに書き直してください。

1. 「APIについて説明してください」
2. 「このデータを分析してください」
3. 「良いコードの書き方を教えてください」

<details>
<summary>解答例</summary>

```
1. 改善前：「APIについて説明してください」
   改善後：「REST APIの基本概念について、Webアプリ開発初心者向けに
   説明してください。以下の内容を含めてください：
   - APIとは何か（身近な例を使って）
   - HTTPメソッド（GET, POST, PUT, DELETE）の役割
   - リクエストとレスポンスの構造
   - 実際のAPI呼び出し例（curlコマンド）
   各セクションは200文字以内でお願いします。」

2. 改善前：「このデータを分析してください」
   改善後：「以下の月次売上データについて、以下の観点で分析してください。
   1. 前月比の増減率を計算する
   2. 季節性のパターンがあるか確認する
   3. 異常値（前月比で±30%以上の変動）を特定する
   4. 今後3ヶ月の売上予測を線形トレンドで算出する
   結果は表形式で、最後にビジネス上の提言を3つ述べてください。」

3. 改善前：「良いコードの書き方を教えてください」
   改善後：「Pythonで保守性の高いコードを書くためのベストプラクティスを
   5つ教えてください。各プラクティスについて：
   - 悪い例（Before）のコード
   - 良い例（After）のコード
   - なぜ改善されるのかの説明（2〜3文）
   を含めてください。対象読者はPython歴1年のエンジニアです。」
```
</details>

### 問題3：ペルソナ設定の作成

以下のタスクに対して、最適なペルソナ設定を作成してください。

1. TypeScript初心者向けの型システム解説
2. 経営会議向けのシステム障害報告書作成
3. セキュリティ脆弱性の分析

<details>
<summary>解答例</summary>

```
1. TypeScript解説用ペルソナ：
「あなたはプログラミングスクールの人気TypeScript講師です。
JavaScript経験はあるがTypeScriptは初めての受講生に対して、
日常生活の例え話を交えながら分かりやすく説明します。
専門用語を使う場合は必ず平易な言葉で補足してください。」

2. 障害報告書用ペルソナ：
「あなたは大手IT企業のITサービスマネージャーです。
非技術者の経営層に対して、技術的な障害の内容と
ビジネスへの影響を分かりやすく報告する経験が豊富です。
技術用語は極力避け、ビジネスインパクト（売上、顧客影響）
を中心に報告してください。」

3. セキュリティ分析用ペルソナ：
「あなたはOWASP Top 10に精通したセキュリティエンジニアで、
Webアプリケーションのペネトレーションテスト経験が10年以上あります。
脆弱性をCVSSスコアで評価し、攻撃シナリオと
具体的な修正コードを提示してください。」
```
</details>

## 応用問題

### 問題4：出力形式の設計

あなたはチームのテックリードです。ジュニアエンジニアのプルリクエストをレビューするためのプロンプトを設計してください。

以下の要件を満たす出力形式を定義してください：
- 問題の重要度が分かる
- 教育的なフィードバックが含まれる
- 修正案が具体的である
- ジュニアが成長できるような書き方

<details>
<summary>解答例</summary>

```
あなたはチームのテックリードです。ジュニアエンジニアが
提出したPRをレビューしてください。

レビューの方針：
- 批判ではなく「一緒に学ぶ」姿勢で
- 良い点も必ず指摘する（最低1つ）
- 修正が必要な箇所は理由を丁寧に説明する
- 参考になるドキュメントやパターン名を紹介する

出力形式：

## 良い点
- （具体的に良い箇所を指摘）

## レビューコメント

### コメント1
- ファイル名・行番号：
- 重要度：🔴 必須修正 / 🟡 推奨 / 🟢 提案
- カテゴリ：バグ / セキュリティ / 設計 / 可読性 / パフォーマンス
- 指摘内容：（何が問題か、分かりやすく説明）
- なぜ問題か：（背景知識や原則を説明）
- 修正案：（具体的なコード例）
- 学習リソース：（参考URL、パターン名等）

### コメント2
（同じ形式で繰り返す）

## 総合コメント
（全体的な評価と、成長につながるアドバイス）
```
</details>

## チャレンジ問題

### 問題5：複雑なプロンプトの設計

以下の業務を自動化するプロンプトを設計してください。すべてのテクニック（4要素、ペルソナ、出力形式、ステップバイステップ、制約）を組み合わせてください。

**タスク：** 毎週の技術チームのスタンドアップ議事録から、マネージャー向けの週次サマリーを自動生成する。

議事録の入力例：
```
月曜：田中 - 認証機能のAPI実装完了。佐藤 - DB設計レビュー中。
火曜：田中 - テスト作成。佐藤 - DB設計完了。鈴木 - フロントのバグ修正。
...
```

<details>
<summary>解答例</summary>

```
あなたはプロジェクトの進捗管理を支援するアシスタントです。  ← 役割

以下のデイリースタンドアップの議事録から、                  ← 指示
マネージャー向けの週次サマリーを作成してください。

## 文脈                                                    ← 文脈
これはWebアプリ開発プロジェクト（6名チーム、2週間スプリント）の
デイリースタンドアップの記録です。マネージャーは技術的な詳細よりも、
進捗状況とリスクに関心があります。

## 作成手順                                                ← ステップバイステップ
ステップ1：各メンバーの週間の作業を整理する
ステップ2：完了タスクと進行中タスクに分類する
ステップ3：ブロッカーやリスクを特定する
ステップ4：以下の出力形式に沿ってサマリーを作成する

## 出力形式                                                ← 出力形式
```markdown
# 週次サマリー（YYYY/MM/DD 〜 MM/DD）

## 全体ステータス
🟢 順調 / 🟡 注意 / 🔴 問題あり

## 今週の成果（完了タスク）
- （箇条書き）

## 進行中のタスク
| タスク | 担当 | 進捗 | 完了見込み |

## ブロッカー・リスク
- （あれば記載、なければ「なし」）

## 来週の見通し
- （2〜3行で）
```

## 制約                                                    ← 制約
- 技術用語は最小限にし、非エンジニアでも理解できる表現で
- 個人の評価につながる表現（「遅い」「できていない」等）は使わない
- 事実に基づいた客観的な記述のみ
- 500文字以内に収める

## 議事録データ
"""
（ここに議事録を貼り付け）
"""
```
</details>

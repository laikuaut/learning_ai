## 6.1 HTTPキャッシュの概要と目的

キャッシュ（Cache）とは、一度取得したデータを保存しておき、次回のアクセス時に**再利用する仕組み**です。

### キャッシュがない場合

```
クライアント                    サーバー
    |                              |
    |  GET /style.css              |
    |----------------------------->|  ← 毎回サーバーに問い合わせ
    |  200 OK (50KB)               |
    |<-----------------------------|  ← 毎回50KBを転送
    |                              |
    |  GET /style.css              |
    |----------------------------->|  ← 同じファイルをまた要求
    |  200 OK (50KB)               |
    |<-----------------------------|  ← また50KBを転送（無駄!）
```

### キャッシュがある場合

```
クライアント        ブラウザキャッシュ        サーバー
    |                    |                      |
    |  GET /style.css    |                      |
    |------------------->|  キャッシュなし        |
    |                    |--------------------->|
    |                    |  200 OK (50KB)       |
    |                    |<---------------------|
    |  200 OK (50KB)     |  ← 保存              |
    |<-------------------|                      |
    |                    |                      |
    |  GET /style.css    |                      |
    |------------------->|  キャッシュあり!       |
    |  200 OK (50KB)     |                      |
    |<-------------------|  ← サーバーに問い合わせ不要
```

### キャッシュのメリット

| メリット | 説明 |
|---------|------|
| **表示速度の向上** | サーバーへの通信が不要になり、即座に表示 |
| **帯域幅の節約** | データ転送量が削減される |
| **サーバー負荷の軽減** | リクエスト数が減り、サーバーの処理量が減る |
| **コスト削減** | 通信コスト・サーバーコストの削減 |

## 6.2 キャッシュの種類

### 3つのキャッシュ層

```
クライアント                                               オリジンサーバー
    |                                                          |
    |  [ブラウザキャッシュ]  [プロキシキャッシュ]  [CDNキャッシュ]  |
    |        (1)                  (2)                (3)        |
    |                                                          |
    |-----→ ① まずブラウザキャッシュを確認                       |
    |         ↓ なければ                                        |
    |       ② プロキシキャッシュを確認                            |
    |         ↓ なければ                                        |
    |       ③ CDNキャッシュを確認                                |
    |         ↓ なければ                                        |
    |       ④ オリジンサーバーに問い合わせ ---------------------->|
```

### 各キャッシュの比較

| 種類 | 場所 | 共有範囲 | 例 |
|------|------|---------|-----|
| **ブラウザキャッシュ** | クライアント端末 | そのユーザーのみ | Chrome, Firefox のキャッシュ |
| **プロキシキャッシュ** | 中間サーバー | 組織内のユーザー | 企業のプロキシサーバー |
| **CDNキャッシュ** | 世界各地のエッジサーバー | 全ユーザー | CloudFlare, AWS CloudFront |

> **実務でのポイント**: キャッシュは `private`（ブラウザのみ）と `public`（プロキシ・CDNも含む）に大別されます。ユーザー固有のデータ（マイページなど）は `private`、共通データ（画像、CSSなど）は `public` に設定するのが基本です。

## 6.3 Cache-Control ヘッダーの詳細

Cache-Control ヘッダーは、キャッシュの動作を**細かく制御する**最も重要なヘッダーです。

### 主要なディレクティブ一覧

| ディレクティブ | 説明 | 使用例 |
|---------------|------|--------|
| **no-store** | キャッシュを一切保存しない | 機密情報、個人データ |
| **no-cache** | キャッシュは保存するが、使用前に必ずサーバーに確認 | 常に最新であるべきデータ |
| **max-age=秒** | 指定秒数の間、キャッシュを新鮮と見なす | 静的ファイル |
| **public** | プロキシ・CDNもキャッシュ可能 | 共通の静的リソース |
| **private** | ブラウザのみキャッシュ可能 | ユーザー固有のデータ |
| **immutable** | キャッシュ期間中は再検証しない | ハッシュ付きファイル名のリソース |
| **must-revalidate** | キャッシュ期限切れ後は必ずサーバーに確認 | 重要な公開データ |
| **s-maxage=秒** | 共有キャッシュ（プロキシ・CDN）用のmax-age | CDN向け設定 |

### よくある間違い: no-cache と no-store の混同

```
【no-cache の動作】
キャッシュは保存するが、使用する前に毎回サーバーに確認する

クライアント       キャッシュ          サーバー
    |                |                  |
    |  GET /data     |                  |
    |--------------->|  キャッシュあり    |
    |                |  → 使っていい?    |
    |                |----------------->|
    |                |  304 Not Modified |  ← 変更なし
    |                |<-----------------|
    |  キャッシュ使用  |                  |
    |<---------------|                  |

【no-store の動作】
キャッシュを一切保存しない。毎回サーバーから取得する

クライアント       キャッシュ          サーバー
    |                |                  |
    |  GET /data     |                  |
    |--------------->|  保存しない       |
    |                |----------------->|
    |                |  200 OK          |
    |                |<-----------------|
    |  新しいデータ    |                  |
    |<---------------|                  |
```

> **よくある間違い**: `no-cache` は「キャッシュしない」という意味ではありません。「キャッシュは保存するが、毎回サーバーに確認する」という意味です。キャッシュを完全に禁止するには `no-store` を使います。

### ユースケース別の設定例

```
# 静的ファイル（CSS, JS, 画像）: 1年間キャッシュ
Cache-Control: public, max-age=31536000, immutable

# APIレスポンス: キャッシュしない
Cache-Control: no-store

# HTMLページ: キャッシュするが毎回確認
Cache-Control: no-cache

# ユーザー固有データ: ブラウザのみ、5分間キャッシュ
Cache-Control: private, max-age=300

# CDN向け: CDNは1時間、ブラウザは5分
Cache-Control: public, max-age=300, s-maxage=3600
```

### curlでの確認

```bash
# レスポンスのCache-Controlヘッダーを確認
curl -I https://www.google.com 2>/dev/null | grep -i cache-control

# 具体例
curl -I https://www.google.com/images/branding/googlelogo/1x/googlelogo_color_272x92dp.png 2>/dev/null | grep -i cache-control
```

## 6.4 ETag と条件付きリクエスト

### ETag（Entity Tag）とは

ETag は、リソースの**バージョンを示す識別子**です。ファイルの内容が変わるとETagも変わります。

```
# レスポンスでETagを返す
HTTP/1.1 200 OK
ETag: "abc123def456"
Content-Type: text/css
```

### 条件付きリクエストの流れ

#### If-None-Match（ETagベース）

```
クライアント                              サーバー
    |                                        |
    |  1. GET /style.css                     |
    |--------------------------------------->|
    |  200 OK                                |
    |  ETag: "abc123"                        |
    |  (50KB のCSSデータ)                     |
    |<---------------------------------------|
    |  ← ブラウザがETagとデータを保存          |
    |                                        |
    |  2. GET /style.css                     |
    |  If-None-Match: "abc123"               |  ← 「これと同じ?」
    |--------------------------------------->|
    |                                        |  ETagを比較
    |  【変更なしの場合】                      |
    |  304 Not Modified                      |  ← ボディなし!
    |<---------------------------------------|
    |  ← キャッシュのデータを使用（50KB節約）  |
    |                                        |
    |  【変更ありの場合】                      |
    |  200 OK                                |
    |  ETag: "xyz789"                        |  ← 新しいETag
    |  (55KB の新しいCSSデータ)               |
    |<---------------------------------------|
```

#### If-Modified-Since（日時ベース）

```
クライアント                              サーバー
    |                                        |
    |  1. GET /style.css                     |
    |--------------------------------------->|
    |  200 OK                                |
    |  Last-Modified: Mon, 01 Jan 2026       |
    |<---------------------------------------|
    |                                        |
    |  2. GET /style.css                     |
    |  If-Modified-Since: Mon, 01 Jan 2026   |  ← 「この日以降に更新された?」
    |--------------------------------------->|
    |                                        |
    |  304 Not Modified                      |  ← 更新なし
    |<---------------------------------------|
```

### ETag vs Last-Modified

| 比較項目 | ETag | Last-Modified |
|---------|------|---------------|
| 精度 | **高い**（内容のハッシュ値） | **低い**（1秒単位の日時） |
| 利用場面 | 内容が変わったかの厳密な判定 | ファイルの更新日時ベース |
| 優先度 | **高い**（両方ある場合はETagが優先） | 低い |
| オーバーヘッド | サーバーでハッシュ計算が必要 | ファイルシステムの情報を利用 |

### Pythonでの条件付きリクエスト

```python
import http.client

# 1回目: 通常のリクエスト
conn = http.client.HTTPSConnection("www.example.com")
conn.request("GET", "/")
response = conn.getresponse()

etag = response.getheader("ETag")
last_modified = response.getheader("Last-Modified")
print(f"1回目: Status={response.status}")
print(f"  ETag: {etag}")
print(f"  Last-Modified: {last_modified}")
print(f"  Content-Length: {response.getheader('Content-Length')}")
body = response.read()
conn.close()

# 2回目: 条件付きリクエスト（ETag使用）
if etag:
    conn = http.client.HTTPSConnection("www.example.com")
    headers = {"If-None-Match": etag}
    conn.request("GET", "/", headers=headers)
    response = conn.getresponse()
    print(f"\n2回目 (If-None-Match): Status={response.status}")
    print(f"  Content-Length: {response.getheader('Content-Length')}")
    response.read()
    conn.close()

# 3回目: 条件付きリクエスト（Last-Modified使用）
if last_modified:
    conn = http.client.HTTPSConnection("www.example.com")
    headers = {"If-Modified-Since": last_modified}
    conn.request("GET", "/", headers=headers)
    response = conn.getresponse()
    print(f"\n3回目 (If-Modified-Since): Status={response.status}")
    print(f"  Content-Length: {response.getheader('Content-Length')}")
    response.read()
    conn.close()
```

実行結果の例:

```
1回目: Status=200
  ETag: "3147526947+gzip"
  Last-Modified: Thu, 17 Oct 2019 07:18:26 GMT
  Content-Length: 1256

2回目 (If-None-Match): Status=304
  Content-Length: None

3回目 (If-Modified-Since): Status=304
  Content-Length: None
```

> **ポイント**: 304レスポンスには**ボディが含まれない**ため、データ転送量を大幅に削減できます。

## 6.5 Expires ヘッダーとの違い

Expiresヘッダーは、Cache-Controlよりも古いキャッシュ制御の仕組みです。

```
Expires: Thu, 01 Jan 2027 00:00:00 GMT
```

### Cache-Control vs Expires

| 項目 | Cache-Control | Expires |
|------|--------------|---------|
| 仕様 | HTTP/1.1 | HTTP/1.0 |
| 指定方法 | 相対時間（秒数） | 絶対日時 |
| 柔軟性 | 高い（多数のディレクティブ） | 低い（日時のみ） |
| 優先度 | **高い** | 低い（Cache-Controlがある場合は無視） |
| 時刻同期 | 不要 | 必要（クライアントとサーバーの時刻がずれると問題） |

```
# 両方指定した場合、Cache-Controlが優先される
Cache-Control: max-age=3600
Expires: Thu, 01 Jan 2027 00:00:00 GMT  ← 無視される
```

> **実務でのポイント**: 現在は`Cache-Control`の使用が推奨されます。`Expires`は古いブラウザやプロキシとの互換性のために併記されることがあります。

## 6.6 CDN（Content Delivery Network）の仕組み

CDN（Content Delivery Network / コンテンツ配信ネットワーク）は、世界各地に配置された**エッジサーバー**にコンテンツのコピーを置き、ユーザーに最も近いサーバーからコンテンツを配信する仕組みです。

### CDNがない場合

```
ユーザー（東京）─────────── 長い距離 ──────────→ オリジンサーバー（米国）
ユーザー（大阪）─────────── 長い距離 ──────────→ オリジンサーバー（米国）
ユーザー（ロンドン）──────── 長い距離 ──────────→ オリジンサーバー（米国）
```

### CDNがある場合

```
ユーザー（東京）───→ CDNエッジ（東京）
ユーザー（大阪）───→ CDNエッジ（大阪）           オリジンサーバー（米国）
ユーザー（ロンドン）→ CDNエッジ（ロンドン）
                           ↑
                    キャッシュミス時のみ
                    オリジンに問い合わせ
```

### CDNのメリット

| メリット | 説明 |
|---------|------|
| **レイテンシの削減** | ユーザーに近いサーバーから配信 |
| **帯域幅の節約** | オリジンサーバーへのリクエストが減少 |
| **耐障害性** | 1つのサーバーがダウンしても他のエッジが対応 |
| **DDoS対策** | 大量のリクエストをエッジサーバー群で分散処理 |

### 主要なCDNサービス

| サービス | 提供元 |
|---------|--------|
| CloudFront | AWS |
| Cloud CDN | Google Cloud |
| Azure CDN | Microsoft |
| Cloudflare | Cloudflare |
| Fastly | Fastly |
| Akamai | Akamai |

## 6.7 HTTP/2 と HTTP/3 の概要

### HTTP/1.1 の課題

```
【HTTP/1.1: 1つずつ順番に送受信】

ブラウザ                              サーバー
    |  GET /index.html                   |
    |----------------------------------->|
    |  200 OK (index.html)               |
    |<-----------------------------------|
    |  GET /style.css                    |  ← 前のレスポンス完了後
    |----------------------------------->|
    |  200 OK (style.css)                |
    |<-----------------------------------|
    |  GET /app.js                       |  ← さらにその後
    |----------------------------------->|
    |  200 OK (app.js)                   |
    |<-----------------------------------|

    → HOL（Head-of-Line）ブロッキング: 1つの遅いリクエストが後続を止める
```

### HTTP/2 の改善

HTTP/2 は2015年に標準化されました。主な改善点は以下の通りです。

#### 多重化（Multiplexing）

```
【HTTP/2: 同時に複数の送受信が可能】

ブラウザ                              サーバー
    |  GET /index.html                   |
    |  GET /style.css                    |  ← 同時に送信!
    |  GET /app.js                       |
    |----------------------------------->|
    |  200 OK (style.css)                |
    |  200 OK (index.html)               |  ← 同時に受信!
    |  200 OK (app.js)                   |
    |<-----------------------------------|

    → 1つのTCP接続で複数のリクエスト・レスポンスを並行処理
```

#### ヘッダー圧縮（HPACK）

HTTP/1.1ではヘッダーがテキストのまま送信されていましたが、HTTP/2ではHPACK圧縮により大幅に削減されます。

```
【HTTP/1.1】毎回全ヘッダーを送信
Host: www.example.com
User-Agent: Mozilla/5.0 (Windows NT 10.0; ...)
Accept: text/html, application/xhtml+xml, ...
Accept-Language: ja, en;q=0.8
Accept-Encoding: gzip, deflate, br
Cookie: session=abc123; theme=dark; ...
→ 数百バイト〜数KBのヘッダー

【HTTP/2 HPACK】差分のみ送信
1回目: 全ヘッダーを送信（インデックステーブルに登録）
2回目: 変更のあったヘッダーのみ送信
→ 大幅に圧縮（90%以上の削減も）
```

#### サーバープッシュ（Server Push）

```
【サーバープッシュ】

ブラウザ                              サーバー
    |  GET /index.html                   |
    |----------------------------------->|
    |                                    |  HTMLを解析して
    |  PUSH /style.css                   |  必要なリソースを
    |  PUSH /app.js                      |  先に送信!
    |  200 OK (index.html)               |
    |<-----------------------------------|
    |                                    |
    |  ← style.css と app.js は         |
    |    すでに受信済み!                   |
```

> **補足**: サーバープッシュは実際にはあまり効果的でないケースも多く、Chrome 106（2022年）でデフォルト無効化されています。

### HTTP/3 の改善

HTTP/3は**QUIC**（Quick UDP Internet Connections）プロトコル上で動作します。

```
【プロトコルスタックの比較】

HTTP/1.1, HTTP/2         HTTP/3
+------------+           +------------+
|    HTTP    |           |    HTTP    |
+------------+           +------------+
|  TLS 1.2+  |           |    QUIC    |  ← TLS 1.3を内蔵
+------------+           +------------+
|    TCP     |           |    UDP     |  ← TCPの代わりにUDP
+------------+           +------------+
|    IP      |           |    IP      |
+------------+           +------------+
```

| 比較項目 | HTTP/2 | HTTP/3 |
|---------|--------|--------|
| トランスポート | TCP | QUIC（UDP） |
| TLS | 別レイヤー | QUICに統合（TLS 1.3） |
| 接続確立 | TCP + TLS = 2-3 RTT | QUIC = 1 RTT（0-RTTも可能） |
| HOLブロッキング | TCP層で発生しうる | ストリームごとに独立（発生しない） |
| 接続移行 | IPが変わると切断 | コネクションIDで維持 |

> **実務でのポイント**: HTTP/3はモバイル環境で特に有効です。Wi-Fiから4G/5Gに切り替わっても接続を維持できるため、ユーザー体験が向上します。

## 6.8 Keep-Alive と接続の再利用

### HTTP/1.0 の問題

```
【HTTP/1.0: リクエストごとに接続を切断】

   TCP接続 → リクエスト → レスポンス → TCP切断
   TCP接続 → リクエスト → レスポンス → TCP切断
   TCP接続 → リクエスト → レスポンス → TCP切断

   → TCP接続の確立（3ウェイハンドシェイク）のオーバーヘッドが毎回発生
```

### Keep-Alive（持続的接続）

```
【HTTP/1.1: Keep-Alive（デフォルトで有効）】

   TCP接続 → リクエスト1 → レスポンス1
             リクエスト2 → レスポンス2
             リクエスト3 → レスポンス3
           → TCP切断

   → 1つのTCP接続を複数のリクエストで再利用
```

```
# Keep-Aliveの設定
Connection: keep-alive
Keep-Alive: timeout=5, max=100

# 接続を閉じる場合
Connection: close
```

| パラメータ | 説明 |
|-----------|------|
| timeout | 接続を維持する秒数 |
| max | この接続で許可する最大リクエスト数 |

> **補足**: HTTP/2以降では1つの接続で多重化されるため、Keep-Aliveの明示的な設定は不要です。

## 6.9 gzip / Brotli 圧縮

### 圧縮の仕組み

```
クライアント                              サーバー
    |                                        |
    |  GET /style.css                        |
    |  Accept-Encoding: gzip, br             |  ← 対応する圧縮方式を伝える
    |--------------------------------------->|
    |                                        |
    |  200 OK                                |
    |  Content-Encoding: gzip                |  ← gzipで圧縮して返す
    |  Content-Length: 12345                  |  ← 圧縮後のサイズ
    |  (圧縮されたデータ)                      |
    |<---------------------------------------|
    |                                        |
    |  ← ブラウザが自動的に解凍して表示        |
```

### 圧縮方式の比較

| 方式 | 圧縮率 | 速度 | 対応状況 |
|------|--------|------|---------|
| **gzip** | 良い | 高速 | ほぼすべてのブラウザ・サーバー |
| **Brotli (br)** | 非常に良い（gzipより20-30%小さい） | やや遅い | 主要ブラウザ（HTTPS時のみ） |
| **deflate** | 良い | 高速 | レガシー（非推奨） |

### 圧縮効果の目安

| ファイル種類 | 元サイズ | gzip | Brotli |
|------------|---------|------|--------|
| HTML | 100KB | ~25KB (75%削減) | ~20KB (80%削減) |
| CSS | 100KB | ~20KB (80%削減) | ~15KB (85%削減) |
| JavaScript | 100KB | ~30KB (70%削減) | ~25KB (75%削減) |
| JSON | 100KB | ~15KB (85%削減) | ~12KB (88%削減) |
| PNG画像 | 100KB | ~99KB (ほぼ変わらず) | ~99KB (ほぼ変わらず) |

> **よくある間違い**: 画像（PNG, JPEG）や動画はすでに圧縮されているため、gzip/Brotliで追加圧縮しても**ほとんど効果がありません**。テキストベースのファイル（HTML, CSS, JS, JSON）に適用しましょう。

### curlでの確認

```bash
# 圧縮なし
curl -s -o /dev/null -w "サイズ: %{size_download}バイト\n" https://www.example.com

# gzip圧縮
curl -s -o /dev/null -w "サイズ: %{size_download}バイト\n" -H "Accept-Encoding: gzip" --compressed https://www.example.com

# レスポンスのContent-Encodingを確認
curl -I -H "Accept-Encoding: gzip, br" https://www.example.com 2>/dev/null | grep -i content-encoding
```

### Pythonでの圧縮確認

```python
import http.client
import gzip

# gzip圧縮をリクエスト
conn = http.client.HTTPSConnection("www.example.com")
headers = {"Accept-Encoding": "gzip"}
conn.request("GET", "/", headers=headers)
response = conn.getresponse()

encoding = response.getheader("Content-Encoding")
data = response.read()

print(f"Content-Encoding: {encoding}")
print(f"受信データサイズ: {len(data)} バイト")

# gzip圧縮されている場合は解凍
if encoding == "gzip":
    decompressed = gzip.decompress(data)
    print(f"解凍後サイズ: {len(decompressed)} バイト")
    print(f"圧縮率: {(1 - len(data)/len(decompressed)) * 100:.1f}%")

conn.close()
```

## 6.10 実用例：静的ファイルのキャッシュ戦略

### モダンなキャッシュ戦略

実際のWebアプリケーションでは、以下のようなキャッシュ戦略がよく使われます。

```
【ファイル種類別のキャッシュ戦略】

HTMLファイル (index.html):
  Cache-Control: no-cache
  → 毎回サーバーに確認（最新のHTML参照を保証）

CSS/JSファイル (style.abc123.css):
  Cache-Control: public, max-age=31536000, immutable
  → 1年間キャッシュ、ファイル名にハッシュを含めて変更時は別URLに

画像ファイル (logo.png):
  Cache-Control: public, max-age=86400
  → 1日間キャッシュ

APIレスポンス (/api/user/me):
  Cache-Control: private, no-store
  → キャッシュしない（個人情報）

APIレスポンス (/api/products):
  Cache-Control: public, max-age=60, s-maxage=300
  → ブラウザ1分、CDN5分キャッシュ
```

### ファイル名のハッシュ化（キャッシュバスティング）

```html
<!-- ❌ 古い方式: クエリパラメータでバージョン管理 -->
<link rel="stylesheet" href="/style.css?v=123">

<!-- ✅ 推奨方式: ファイル名にハッシュを含める -->
<link rel="stylesheet" href="/style.abc123def.css">
```

```
ファイルの内容が変更された場合:

変更前: /style.abc123def.css → キャッシュされている
変更後: /style.xyz789ghi.css → 新しいURLなのでキャッシュミス → サーバーから取得

→ ファイル名が変わるため、古いキャッシュが使われる心配がない
→ max-age=31536000 (1年) + immutable で超長期キャッシュが可能
```

> **実務でのポイント**: React、Vue、Next.js などのフレームワークでは、ビルド時に自動的にファイル名にハッシュを付与する仕組みが組み込まれています。

### まとめ図

```
+-----------------------------------------------------------+
|                    キャッシュ戦略の全体像                     |
+-----------------------------------------------------------+
|                                                           |
|  HTML:        no-cache          (毎回確認)                 |
|  JS/CSS:      max-age=1年      (ハッシュ付きファイル名)     |
|  画像:        max-age=1日      (適度にキャッシュ)           |
|  API(公開):   max-age=1分      (短めのキャッシュ)           |
|  API(個人):   no-store          (キャッシュしない)          |
|                                                           |
|  CDN:         s-maxage で別途制御                          |
|  圧縮:        gzip / Brotli でテキストファイルを圧縮        |
|  HTTP/2:      多重化で並列転送                              |
|                                                           |
+-----------------------------------------------------------+
```

## ポイントまとめ

1. **キャッシュ**は表示速度向上、帯域幅節約、サーバー負荷軽減に効果的
2. キャッシュには**ブラウザキャッシュ**（private）、**プロキシキャッシュ**、**CDNキャッシュ**（public）がある
3. **Cache-Control** ヘッダーでキャッシュの動作を細かく制御する
4. **no-cache** は「毎回確認」、**no-store** は「保存しない」（混同注意）
5. **ETag** と **If-None-Match** で条件付きリクエストを行い、**304 Not Modified** でデータ転送を削減
6. **CDN** はユーザーに近いエッジサーバーからコンテンツを配信する
7. **HTTP/2** は多重化とヘッダー圧縮で高速化、**HTTP/3** はQUICでさらに改善
8. **Keep-Alive** でTCP接続を再利用し、接続確立のオーバーヘッドを削減
9. **gzip/Brotli圧縮** はテキストファイルに有効（画像には効果が薄い）
10. 静的ファイルは**ハッシュ付きファイル名 + 長期キャッシュ**が推奨戦略

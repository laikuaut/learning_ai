## 演習1（基本）：基本的なミドルウェアを作成しよう

すべてのリクエストにカスタムヘッダーを追加するミドルウェアを作成してください。

**要件：**
- `middleware.ts` をプロジェクトルートに作成する
- リクエストヘッダーに `x-request-id`（ランダムな UUID）を追加する
- レスポンスヘッダーに `x-powered-by: Next.js` を追加する

<details>
<summary>ヒント</summary>

`middleware.ts` は `app` フォルダの**外**、プロジェクトルートに配置します。`crypto.randomUUID()` でランダムな UUID を生成できます。`NextResponse.next()` にリクエストヘッダーを設定し、返却するレスポンスにヘッダーを追加します。

</details>

<details>
<summary>解答例</summary>

```tsx
// middleware.ts（プロジェクトルート）
import { NextRequest, NextResponse } from 'next/server'

export function middleware(request: NextRequest) {
  // リクエストヘッダーにカスタム情報を追加
  const requestHeaders = new Headers(request.headers)
  requestHeaders.set('x-request-id', crypto.randomUUID())
  requestHeaders.set('x-pathname', request.nextUrl.pathname)

  // レスポンスを作成
  const response = NextResponse.next({
    request: {
      headers: requestHeaders,
    },
  })

  // レスポンスヘッダーを追加
  response.headers.set('x-powered-by', 'Next.js')

  return response
}

// 静的ファイルや Next.js 内部パスを除外
export const config = {
  matcher: [
    '/((?!_next/static|_next/image|favicon.ico).*)',
  ],
}
```

ブラウザの開発者ツールのネットワークタブでレスポンスヘッダーを確認すると、`x-powered-by: Next.js` が追加されていることがわかります。

</details>

## 演習2（基本）：認証チェック付きリダイレクトを実装しよう

Cookie にセッショントークンがない場合、保護されたページからログインページにリダイレクトするミドルウェアを作成してください。

**要件：**
- `/dashboard`, `/settings` へのアクセスにセッショントークンを要求する
- トークンがない場合は `/login` にリダイレクトする
- リダイレクト時にコールバック URL をクエリパラメータに含める

<details>
<summary>ヒント</summary>

`request.cookies.get('session-token')` で Cookie を確認します。`NextResponse.redirect(new URL('/login', request.url))` でリダイレクトを返します。

</details>

<details>
<summary>解答例</summary>

```tsx
// middleware.ts
import { NextRequest, NextResponse } from 'next/server'

// 認証が必要なパス
const protectedPaths = ['/dashboard', '/settings', '/profile']

// 認証ページ（ログイン済みならリダイレクト）
const authPaths = ['/login', '/register']

export function middleware(request: NextRequest) {
  const { pathname } = request.nextUrl
  const sessionToken = request.cookies.get('session-token')?.value

  // 保護されたパスに未認証でアクセス
  const isProtected = protectedPaths.some((path) =>
    pathname.startsWith(path)
  )
  if (isProtected && !sessionToken) {
    const loginUrl = new URL('/login', request.url)
    // ログイン後に元のページに戻れるようにコールバック URL を設定
    loginUrl.searchParams.set('callbackUrl', pathname)
    return NextResponse.redirect(loginUrl)
  }

  // ログイン済みで認証ページにアクセス → ダッシュボードへ
  const isAuthPage = authPaths.some((path) =>
    pathname.startsWith(path)
  )
  if (isAuthPage && sessionToken) {
    return NextResponse.redirect(new URL('/dashboard', request.url))
  }

  return NextResponse.next()
}

export const config = {
  matcher: [
    '/dashboard/:path*',
    '/settings/:path*',
    '/profile/:path*',
    '/login',
    '/register',
  ],
}
```

`matcher` で必要なパスのみにミドルウェアを適用しています。静的ファイルへの不要な処理を防止するため、`matcher` の設定は重要です。

</details>

## 演習3（基本）：認証と認可の違いを説明しよう

以下の各シナリオが「認証（Authentication）」と「認可（Authorization）」のどちらに該当するか判定してください。

1. ユーザーがメールアドレスとパスワードでログインする
2. 管理者のみが管理ページにアクセスできる
3. GitHub アカウントを使ってサインインする
4. 有料会員のみがプレミアムコンテンツを閲覧できる
5. セッショントークンの有効期限を確認する
6. ユーザーが自分の投稿のみ削除できる

<details>
<summary>ヒント</summary>

認証は「あなたは誰ですか？」を確認する処理、認可は「あなたは何ができますか？」を制御する処理です。

</details>

<details>
<summary>解答例</summary>

1. **認証（Authentication）**: ユーザーの身元を確認する処理。
2. **認可（Authorization）**: ログイン済みのユーザーの中で、管理者ロールを持つかどうかを判定する処理。
3. **認証（Authentication）**: OAuth を使ってユーザーの身元を確認する処理。
4. **認可（Authorization）**: ログイン済みのユーザーの中で、有料会員かどうかを判定する処理。
5. **認証（Authentication）**: セッションが有効かどうか（ログイン状態が維持されているか）を確認する処理。
6. **認可（Authorization）**: ログイン済みのユーザーが、特定のリソース（自分の投稿）に対する権限を持つかどうかを判定する処理。

認証は必ず認可の前に行われます。まず「誰か」を確認し（認証）、次に「何ができるか」を判定します（認可）。

</details>

## 演習4（応用）：ロールベースのアクセス制御を実装しよう

ユーザーのロール（user, editor, admin）に応じてアクセスを制御するユーティリティ関数を作成してください。

**要件：**
- `checkRole` 関数を作成する
- ロールの階層（user < editor < admin）を考慮する
- サーバーコンポーネントで使用してアクセス制御を行う

<details>
<summary>ヒント</summary>

ロールに数値レベルを割り当て、要求されるロールのレベルとユーザーのロールのレベルを比較します。実際のプロジェクトでは `auth()` でセッションからロールを取得します。

</details>

<details>
<summary>解答例</summary>

```tsx
// lib/auth-utils.ts
type Role = 'user' | 'editor' | 'admin'

// ロールの階層（数値が大きいほど権限が高い）
const roleHierarchy: Record<Role, number> = {
  user: 1,
  editor: 2,
  admin: 3,
}

type AuthResult = {
  authorized: boolean
  reason?: string
  userRole?: Role
}

// ロールチェック関数
export function checkRole(
  userRole: Role,
  requiredRole: Role
): AuthResult {
  const hasPermission =
    roleHierarchy[userRole] >= roleHierarchy[requiredRole]

  return {
    authorized: hasPermission,
    reason: hasPermission
      ? undefined
      : `この操作には ${requiredRole} 以上の権限が必要です（現在: ${userRole}）`,
    userRole,
  }
}
```

```tsx
// app/admin/page.tsx
import { redirect } from 'next/navigation'
import { checkRole } from '@/lib/auth-utils'

export default function AdminPage() {
  // 実際には auth() でセッションからロールを取得する
  const currentUserRole = 'editor' // 仮の値

  const { authorized, reason } = checkRole(currentUserRole, 'admin')

  if (!authorized) {
    // 権限不足の場合
    return (
      <div style={{ padding: '2rem', textAlign: 'center' }}>
        <h1>403 - アクセス拒否</h1>
        <p style={{ color: '#666' }}>{reason}</p>
      </div>
    )
  }

  return (
    <div style={{ padding: '2rem' }}>
      <h1>管理者ページ</h1>
      <p>このページは admin ロールのユーザーのみ閲覧できます。</p>
    </div>
  )
}
```

`checkRole('user', 'admin')` は `false` を返し、`checkRole('admin', 'admin')` は `true` を返します。`checkRole('admin', 'editor')` も `true` を返します（admin は editor 以上の権限を持つため）。

</details>

## 演習5（応用）：ログインページを実装しよう

Server Action を使ったログインフォームを作成してください。

**要件：**
- メールアドレスとパスワードのフォームを作成する
- Server Action でバリデーションを行う
- 認証成功時は Cookie にセッショントークンを設定してリダイレクトする
- 認証失敗時はエラーメッセージを表示する

<details>
<summary>ヒント</summary>

`cookies()` を使って Cookie の読み書きができます。Server Action 内で `redirect()` を呼ぶ場合は、`try-catch` の外に配置する必要があります（`redirect` は内部的にエラーをスローするため）。

</details>

<details>
<summary>解答例</summary>

```tsx
// app/actions/auth.ts
'use server'

import { cookies } from 'next/headers'
import { redirect } from 'next/navigation'

type LoginState = {
  error?: string
}

// ダミーのユーザーデータ（実際にはデータベースを使用）
const DUMMY_USER = {
  email: 'test@example.com',
  password: 'password123',
}

export async function login(
  prevState: LoginState,
  formData: FormData
): Promise<LoginState> {
  const email = formData.get('email') as string
  const password = formData.get('password') as string

  // バリデーション
  if (!email || !password) {
    return { error: 'メールアドレスとパスワードを入力してください' }
  }

  // 認証チェック（実際にはデータベースとハッシュ化パスワードで検証）
  if (email !== DUMMY_USER.email || password !== DUMMY_USER.password) {
    return { error: 'メールアドレスまたはパスワードが正しくありません' }
  }

  // セッション Cookie を設定
  const cookieStore = await cookies()
  cookieStore.set('session-token', 'dummy-session-token', {
    httpOnly: true,
    secure: process.env.NODE_ENV === 'production',
    sameSite: 'lax',
    path: '/',
    maxAge: 60 * 60 * 24 * 7, // 7日間
  })

  // リダイレクト（try-catch の外に配置する）
  redirect('/dashboard')
}
```

```tsx

import SubmitButton from '@/app/components/SubmitButton'

          <input
            type="email"
            id="email"

            placeholder="test@example.com"
            required
            style={{ width: '100%', padding: '0.5rem', boxSizing: 'border-box' }}
          />
        </div>
        <div style={{ marginBottom: '1rem' }}>

```

Cookie の設定で `httpOnly: true` と `secure: true`（本番環境）を指定することで、JavaScript からのアクセスを防ぎ、HTTPS 通信でのみ送信されるセキュアな Cookie になります。

## 演習6（応用）：ログアウト機能を実装しよう

Server Action を使ったログアウト機能を作成してください。

**要件：**
- Server Action でセッション Cookie を削除する
- ログアウト後にホームページにリダイレクトする
- ログアウトボタンはフォームとして実装する

<details>
<summary>ヒント</summary>

`cookies()` の `delete('cookie名')` でCookieを削除できます。ログアウトもServer Actionで実装することで、プログレッシブエンハンスメントが実現します。

</details>

<details>
<summary>解答例</summary>

```tsx
// app/actions/auth.ts に追加
'use server'

import { cookies } from 'next/headers'
import { redirect } from 'next/navigation'

export async function logout() {
  const cookieStore = await cookies()
  cookieStore.delete('session-token')
  redirect('/')
}
```

```tsx
// app/components/LogoutButton.tsx
import { logout } from '@/app/actions/auth'

export default function LogoutButton() {
  return (
    <form action={logout}>
      <button
        type="submit"
        style={{
          padding: '0.5rem 1rem',
          backgroundColor: 'transparent',
          color: '#e53e3e',
          border: '1px solid #e53e3e',
          borderRadius: '4px',
          cursor: 'pointer',
        }}
      >
        ログアウト
      </button>
    </form>
  )
}
```

```tsx
// app/dashboard/page.tsx
import { cookies } from 'next/headers'
import { redirect } from 'next/navigation'
import LogoutButton from '@/app/components/LogoutButton'

export default async function DashboardPage() {
  const cookieStore = await cookies()
  const session = cookieStore.get('session-token')

  if (!session) {
    redirect('/login')
  }

  return (
    <div style={{ padding: '2rem' }}>
      <div style={{ display: 'flex', justifyContent: 'space-between', alignItems: 'center' }}>
        <h1>ダッシュボード</h1>
        <LogoutButton />
      </div>
      <p>ログインに成功しました。</p>
    </div>
  )
}
```

ログアウトを `<form>` + Server Action で実装することにより、JavaScript が無効な環境でも動作します。

</details>

## 演習7（チャレンジ）：ミドルウェアでリライトを使った国際化対応を実装しよう

ミドルウェアで Accept-Language ヘッダーを解析し、適切な言語のページにリライトする国際化対応を実装してください。

**要件：**
- 対応言語は `ja`（日本語）と `en`（英語）
- ブラウザの言語設定に基づいてリダイレクトする
- `/ja/about` や `/en/about` のような URL 構造にする

<details>
<summary>ヒント</summary>

`request.headers.get('accept-language')` でブラウザの言語設定を取得します。URL パスにロケールが含まれていない場合にリダイレクトを行います。

</details>

<details>
<summary>解答例</summary>

```tsx
// middleware.ts
import { NextRequest, NextResponse } from 'next/server'

const locales = ['ja', 'en']
const defaultLocale = 'ja'

function getLocale(request: NextRequest): string {
  const acceptLanguage = request.headers.get('accept-language') || ''

  // Accept-Language ヘッダーから最適なロケールを判定
  for (const locale of locales) {
    if (acceptLanguage.includes(locale)) {
      return locale
    }
  }

  return defaultLocale
}

export function middleware(request: NextRequest) {
  const { pathname } = request.nextUrl

  // パスにロケールが含まれているか確認
  const hasLocale = locales.some(
    (locale) =>
      pathname.startsWith(`/${locale}/`) || pathname === `/${locale}`
  )

  // ロケールが含まれている場合はそのまま通す
  if (hasLocale) return NextResponse.next()

  // ロケールが含まれていない場合、適切なロケールにリダイレクト
  const locale = getLocale(request)
  return NextResponse.redirect(
    new URL(`/${locale}${pathname}`, request.url)
  )
}

export const config = {
  matcher: [
    '/((?!_next|api|favicon.ico).*)',
  ],
}
```

```tsx
// app/[locale]/page.tsx
type Props = {
  params: Promise<{ locale: string }>
}

const messages: Record<string, { title: string; description: string }> = {
  ja: {
    title: 'ようこそ',
    description: 'これは日本語のページです。',
  },
  en: {
    title: 'Welcome',
    description: 'This is the English page.',
  },
}

export default async function HomePage({ params }: Props) {
  const { locale } = await params
  const t = messages[locale] || messages['ja']

  return (
    <div style={{ padding: '2rem' }}>
      <h1>{t.title}</h1>
      <p>{t.description}</p>
      <p>現在の言語: {locale}</p>
    </div>
  )
}
```

ミドルウェアでリダイレクトを行うことで、ユーザーのブラウザ設定に応じて自動的に適切な言語のページが表示されます。`/about` にアクセスすると、日本語設定のブラウザでは `/ja/about` に、英語設定では `/en/about` にリダイレクトされます。

</details>

## 演習8（チャレンジ）：認証フローの全体像を設計しよう

以下の要件を満たす認証システムの全体設計を行い、各ファイルの役割と処理フローを説明してください。

**要件：**
- GitHub OAuth でのログインをサポート
- ミドルウェアで保護されたルートへのアクセスを制御
- サーバーコンポーネントでセッション情報を表示
- Server Actions で認可チェックを行う

各レイヤーでどのような処理を行うべきかを設計してください。

<details>
<summary>ヒント</summary>

認証処理は複数のレイヤーで行います: ミドルウェア（軽量なチェック）→ レイアウト/ページ（詳細なセッション取得）→ Server Actions（データ変更時の認可チェック）。

</details>

<details>
<summary>解答例</summary>

```
認証フローの全体設計:

1. middleware.ts（第1レイヤー: ゲートキーパー）
   ┌─────────────────────────────────────────┐
   │ 処理内容:                                 │
   │ - セッショントークン（Cookie）の存在チェック │
   │ - 未認証の場合は /login にリダイレクト      │
   │ - ログイン済みで /login にアクセス時は       │
   │   /dashboard にリダイレクト                │
   │                                          │
   │ 注意:                                     │
   │ - ミドルウェアは Edge Runtime で動作する     │
   │ - データベースアクセスは避ける               │
   │ - 軽量な処理のみ行う                       │
   └─────────────────────────────────────────┘
               ↓
2. layout.tsx / page.tsx（第2レイヤー: セッション情報の取得）
   ┌─────────────────────────────────────────┐
   │ 処理内容:                                 │
   │ - auth() でセッションの詳細情報を取得       │
   │ - ユーザー名、メール、ロール等を表示        │
   │ - ロールに基づくページレベルの認可チェック    │
   │                                          │
   │ 例:                                       │
   │ const session = await auth()              │
   │ if (session.user.role !== 'admin') {      │
   │   redirect('/unauthorized')               │
   │ }                                         │
   └─────────────────────────────────────────┘
               ↓
3. Server Actions（第3レイヤー: 操作ごとの認可チェック）
   ┌─────────────────────────────────────────┐
   │ 処理内容:                                 │
   │ - 各データ変更操作でセッションを再確認       │
   │ - 操作に必要な権限のチェック                │
   │ - 入力値のバリデーション                    │
   │                                          │
   │ 重要:                                     │
   │ - クライアントからの入力を信頼しない         │
   │ - 認証情報はサーバーで取得する               │
   │ - userId 等をクライアントから受け取らない     │
   └─────────────────────────────────────────┘

ファイル構成:
├── middleware.ts          ← トークン存在チェック + リダイレクト
├── auth.ts               ← NextAuth.js の設定（プロバイダ、コールバック）
├── app/
│   ├── api/auth/[...nextauth]/route.ts  ← OAuth コールバック
│   ├── login/page.tsx     ← ログインページ
│   ├── dashboard/
│   │   ├── layout.tsx     ← セッション取得 + ナビゲーション
│   │   └── page.tsx       ← ダッシュボード表示
│   ├── admin/page.tsx     ← ロールチェック（admin のみ）
│   └── actions/
│       └── posts.ts       ← 操作ごとの認可チェック
├── lib/
│   ├── auth-utils.ts      ← checkRole 等のユーティリティ
│   └── password.ts        ← パスワードハッシュ化
```

この3層構造により、セキュリティの漏れを最小限に抑えつつ、各レイヤーの責務を明確に分離できます。

</details>

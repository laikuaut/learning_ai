# 第3章：暗号技術 演習問題

## 基本問題

### 問題3-1：暗号方式の分類
共通鍵暗号方式と公開鍵暗号方式の違いを以下の4つの観点から比較してください。

1. 鍵の種類と数
2. 処理速度
3. 鍵配送問題
4. 代表的なアルゴリズム

<details>
<summary>解答</summary>

| 観点 | 共通鍵暗号（対称暗号） | 公開鍵暗号（非対称暗号） |
|------|---------------------|----------------------|
| 鍵の種類と数 | 暗号化と復号に同じ鍵を使用。n人の通信には n(n-1)/2 個の鍵が必要 | 公開鍵と秘密鍵のペアを使用。n人でも n組の鍵ペアで済む |
| 処理速度 | 高速（公開鍵暗号の100〜1000倍） | 低速（計算量が大きい） |
| 鍵配送問題 | あり（共通鍵を安全に相手に渡す方法が必要） | なし（公開鍵は誰にでも公開できる） |
| 代表アルゴリズム | AES、DES（非推奨）、Camellia、ChaCha20 | RSA、楕円曲線暗号（ECC）、ElGamal |

</details>

### 問題3-2：ハイブリッド暗号
ハイブリッド暗号方式の仕組みを説明してください。なぜ共通鍵暗号と公開鍵暗号を組み合わせるのですか。

<details>
<summary>解答</summary>

**ハイブリッド暗号の仕組み：**

1. 送信者がランダムな共通鍵（セッション鍵）を生成
2. データ本体は高速な**共通鍵暗号**で暗号化
3. 共通鍵（セッション鍵）は受信者の**公開鍵で暗号化**
4. 暗号化されたデータと暗号化された共通鍵を送信
5. 受信者は自身の秘密鍵で共通鍵を復号し、その共通鍵でデータを復号

**組み合わせる理由：**
- 公開鍵暗号は鍵配送問題を解決できるが処理が遅い
- 共通鍵暗号は高速だが鍵配送問題がある
- 共通鍵暗号でデータを暗号化（高速）し、その共通鍵だけを公開鍵暗号で安全に交換（鍵配送問題の解決）することで、双方の利点を組み合わせる

HTTPSのTLSハンドシェイクでもこの方式が使われています。

</details>

### 問題3-3：ハッシュ関数
ハッシュ関数の3つの性質を挙げ、それぞれ説明してください。

<details>
<summary>解答</summary>

1. **一方向性（Pre-image resistance）**：ハッシュ値から元のデータを逆算することが計算上困難であること。パスワードの保存等に利用される

2. **衝突耐性（Collision resistance）**：異なる2つのデータから同じハッシュ値が生成されることが計算上困難であること。デジタル署名の安全性に重要

3. **第二原像耐性（Second pre-image resistance）**：あるデータと同じハッシュ値を持つ別のデータを見つけることが計算上困難であること

| ハッシュ関数 | 出力長 | 安全性 |
|------------|--------|--------|
| MD5 | 128ビット | 危険（衝突が発見済み） |
| SHA-1 | 160ビット | 危険（衝突が発見済み） |
| SHA-256 | 256ビット | 安全（現在推奨） |
| SHA-3 | 256/512ビット | 安全 |

</details>

### 問題3-4：デジタル署名
デジタル署名の仕組みを「送信者の操作」と「受信者の検証」に分けて説明してください。

<details>
<summary>解答</summary>

**送信者の操作：**
1. 送信するデータのハッシュ値を計算する
2. ハッシュ値を**送信者の秘密鍵で暗号化**する → これが「デジタル署名」
3. 元データとデジタル署名を一緒に送信する

**受信者の検証：**
1. 受信したデータのハッシュ値を自分で計算する
2. デジタル署名を**送信者の公開鍵で復号**してハッシュ値を取り出す
3. 2つのハッシュ値を比較する
   - **一致** → データは改ざんされておらず、確かに送信者が署名したものである
   - **不一致** → データが改ざんされている、または署名者が偽者

**デジタル署名で保証されること：**
- **認証**：署名者が秘密鍵の所有者であること
- **完全性**：データが改ざんされていないこと
- **否認防止**：署名者が後から署名を否定できないこと

</details>

## 応用問題

### 問題3-5：PKI
PKI（Public Key Infrastructure / 公開鍵基盤）の仕組みを説明し、以下の用語を使って証明書の検証の流れを述べてください。
- 認証局（CA）
- サーバ証明書
- ルート証明書
- 証明書チェーン

<details>
<summary>解答</summary>

**PKIの仕組み：**
PKIは公開鍵暗号方式において、公開鍵の正当性を保証するための仕組みです。「この公開鍵は本当にexample.comのものか」を第三者（認証局）が証明します。

**証明書の検証の流れ：**

1. サーバが**サーバ証明書**（サーバの公開鍵 + CA の署名）をクライアントに提示
2. クライアントは証明書の署名を、**中間CA の公開鍵**で検証
3. 中間CA の証明書の署名を、**ルートCA の公開鍵**で検証
4. **ルート証明書**はブラウザ/OSに事前にインストールされている信頼されたCA のリストと照合

```
信頼の連鎖（証明書チェーン）:
  ルートCA（ブラウザに事前登録）
     │ 署名
  中間CA
     │ 署名
  サーバ証明書（www.example.com）
```

ルートCAまでの信頼の連鎖が確認できれば、サーバ証明書（およびそこに含まれる公開鍵）は信頼できると判断されます。

</details>

### 問題3-6：暗号アルゴリズムの選択
以下の場面で適切な暗号技術を選択し、理由を述べてください。

(1) Webサイトとの通信を暗号化したい
(2) パスワードをデータベースに安全に保存したい
(3) ファイルが改ざんされていないことを確認したい
(4) 電子契約書の署名が本人のものであることを証明したい

<details>
<summary>解答</summary>

(1) **TLS（ハイブリッド暗号）**：鍵交換にECDHE（公開鍵暗号）、データ暗号化にAES-256-GCM（共通鍵暗号）を使用。HTTPS通信の標準

(2) **bcrypt / Argon2（パスワードハッシュ関数）**：ソルトを付加し、計算コストを調整できるハッシュ関数を使用。AES等の暗号化ではなくハッシュ化する（復号できる必要がないため）

(3) **SHA-256（ハッシュ関数）**：ファイルのハッシュ値を計算し、配布元が公開しているハッシュ値と比較する。一致すれば改ざんされていないことが確認できる

(4) **RSAまたはECDSAによるデジタル署名**：送信者の秘密鍵で署名し、受信者が公開鍵で検証する。認証・完全性・否認防止を保証。認証局から発行された電子証明書と併用する

</details>

## チャレンジ問題

### 問題3-7：暗号技術の総合問題
A社がB社に機密文書を安全に送信する場合、以下の全ての要件を満たすために必要な暗号技術の組み合わせを設計してください。

要件：
- 文書の内容を第三者に見られないようにしたい（機密性）
- 文書が改ざんされていないことを確認したい（完全性）
- B社がA社から送られたものであることを確認したい（認証）
- A社が後から「送っていない」と否認できないようにしたい（否認防止）

<details>
<summary>解答</summary>

**暗号技術の組み合わせ：**

1. **機密性の確保**：文書をAESなどの共通鍵暗号で暗号化。共通鍵はB社の公開鍵で暗号化して送付（ハイブリッド暗号）

2. **完全性・認証・否認防止の確保**：文書のハッシュ値（SHA-256）を計算し、A社の秘密鍵でデジタル署名を作成

3. **送信内容**：
   - 暗号化された文書
   - 暗号化された共通鍵（B社の公開鍵で暗号化）
   - デジタル署名
   - A社の電子証明書

4. **B社の検証手順**：
   - A社の電子証明書を検証（PKIの信頼チェーン）
   - 自身の秘密鍵で共通鍵を復号
   - 共通鍵で文書を復号
   - デジタル署名をA社の公開鍵で検証（完全性・認証・否認防止の確認）

</details>

### 問題3-8：ケルクホフスの原則
ケルクホフスの原則を説明し、「隠蔽によるセキュリティ（Security by Obscurity）」がなぜ危険なのかを述べてください。

<details>
<summary>解答</summary>

**ケルクホフスの原則：**
暗号システムの安全性は、暗号アルゴリズムの秘密ではなく、鍵の秘密性のみに依存すべきである。つまり、暗号アルゴリズムが公開されていても、鍵が秘密であれば安全でなければならない。

**Security by Obscurity が危険な理由：**

1. **アルゴリズムの漏洩リスク**：リバースエンジニアリングや内部者の流出により、秘密のアルゴリズムが暴露される可能性は常にある
2. **専門家のレビューが受けられない**：公開されていないアルゴリズムは暗号学者による検証ができず、未発見の脆弱性が潜んでいる可能性が高い
3. **鍵の変更が容易でない**：アルゴリズムが漏洩した場合、システム全体の設計変更が必要になる。鍵のみが秘密であれば、鍵の変更だけで安全性を回復できる

AES、RSA等の標準暗号アルゴリズムは全て仕様が公開されており、世界中の暗号学者によるレビューを経て安全性が確認されています。

</details>

> **難易度**: ★☆☆☆☆

> **課題の種類**: 設計課題
> **学習目標**: AIエージェントの分類軸を理解し、実在するAI製品を体系的に整理できるようになること

## 完成イメージ

以下のような分類マップを作成します。

```
┌─────────────────────────────────────────────────────────┐
│                AIエージェント分類マップ                    │
├─────────────┬───────────────────────────────────────────┤
│  分類軸      │  具体的な製品・サービス                     │
├─────────────┼───────────────────────────────────────────┤
│  自律度      │  低 ←──────────────────────────────→ 高   │
│             │  Copilot系    アシスタント系    自律エージェント│
├─────────────┼───────────────────────────────────────────┤
│  対話方式    │  シングルターン / マルチターン / 非対話型    │
├─────────────┼───────────────────────────────────────────┤
│  ツール利用  │  なし / 限定的 / 広範囲                     │
├─────────────┼───────────────────────────────────────────┤
│  エージェント│  単一 / マルチ（協調） / マルチ（競争）      │
│  数          │                                           │
└─────────────┴───────────────────────────────────────────┘
```

最終成果物は、10個以上のAI製品を4つ以上の分類軸で整理した表（Markdownテーブル）です。

---

## 課題の要件

1. **10個以上のAI製品・サービス**を選定すること（例：ChatGPT、Claude、GitHub Copilot、Devin、AutoGPT、CrewAI、Amazon Q Developer、Gemini、Cursor、Claude Code など）
2. **4つ以上の分類軸**を設定すること（自律度、対話方式、ツール利用、エージェント数、ドメイン特化度、メモリ有無 など）
3. 各製品について、すべての分類軸で評価を記入すること
4. 分類結果を**Markdownテーブル形式**で整理すること
5. 各分類軸の**定義と評価基準**を明文化すること
6. 分類結果から読み取れる**傾向や気づき**を3つ以上記述すること

---

## ステップガイド

<details>
<summary>ステップ1：分類軸を定義する</summary>

まずは分類軸とその評価基準を決めましょう。以下を参考にしてください。

| 分類軸 | 説明 | 評価基準の例 |
|--------|------|-------------|
| 自律度（Autonomy Level） | 人間の介入がどれだけ必要か | 低（毎回指示）/ 中（部分自律）/ 高（完全自律） |
| 対話方式（Interaction Mode） | ユーザーとの対話パターン | シングルターン / マルチターン / 非対話 |
| ツール利用（Tool Usage） | 外部ツール・APIの活用度 | なし / 限定的 / 広範囲 |
| ドメイン特化度（Domain Specificity） | 汎用か特化型か | 汎用 / セミ特化 / 特化 |

これら以外にも「メモリの有無」「マルチモーダル対応」「エージェント構成（単体/マルチ）」なども良い分類軸です。

</details>

<details>
<summary>ステップ2：AI製品を選定し情報を集める</summary>

各製品について以下の観点で情報を整理しましょう。

- **製品名と提供元**
- **主なユースケース**（コード生成、文章作成、調査、自動化 など）
- **動作の仕組み**（ユーザーが逐次指示するのか、目標を与えれば自律的に動くのか）
- **利用するツール**（Web検索、コード実行、ファイル操作 など）

**よくある間違い**: 製品の「宣伝文句」と「実際の動作」を混同しないようにしましょう。例えば「自律型AI」と宣伝されていても、実際には毎回ユーザーの確認が必要な場合があります。

</details>

<details>
<summary>ステップ3：分類テーブルを作成する</summary>

Markdown形式でテーブルを作成します。

```markdown
| 製品名 | 自律度 | 対話方式 | ツール利用 | ドメイン特化度 | エージェント構成 |
|--------|--------|----------|-----------|---------------|----------------|
| ChatGPT | 低〜中 | マルチターン | 広範囲 | 汎用 | 単一 |
| GitHub Copilot | 低 | シングルターン | 限定的 | コード特化 | 単一 |
| ... | ... | ... | ... | ... | ... |
```

</details>

<details>
<summary>ステップ4：傾向と気づきをまとめる</summary>

テーブルを見ながら以下のような観点で分析してみましょう。

- 自律度が高い製品にはどんな共通点があるか？
- ツール利用が広範囲な製品は自律度も高い傾向があるか？
- ドメイン特化型と汎用型で、設計思想にどんな違いがあるか？
- 今後のトレンドとして、どの方向に進化しそうか？

</details>

## 解答例

<details>
<summary>解答例（初心者向け）</summary>

### 分類軸の定義

| 分類軸 | 定義 | 評価基準 |
|--------|------|---------|
| 自律度 | 人間の介入なしにタスクを遂行できる度合い | 低：毎回指示が必要 / 中：一部自律 / 高：目標指定のみで自律動作 |
| 対話方式 | ユーザーとのインタラクションの形態 | シングルターン / マルチターン / 非対話（バッチ） |
| ツール利用 | 外部ツール・APIの活用範囲 | なし / 限定的（1〜2種）/ 広範囲（3種以上） |
| ドメイン特化度 | 対応する領域の広さ | 汎用 / セミ特化 / 特化 |

### 分類マップ

| 製品名 | 提供元 | 自律度 | 対話方式 | ツール利用 | ドメイン特化度 |
|--------|--------|--------|----------|-----------|---------------|
| ChatGPT | OpenAI | 低〜中 | マルチターン | 広範囲 | 汎用 |
| Claude | Anthropic | 低〜中 | マルチターン | 広範囲 | 汎用 |
| GitHub Copilot | GitHub/Microsoft | 低 | シングルターン | 限定的 | コード特化 |
| Claude Code | Anthropic | 中〜高 | マルチターン | 広範囲 | コード特化 |
| Devin | Cognition | 高 | 非対話〜マルチターン | 広範囲 | コード特化 |
| AutoGPT | オープンソース | 高 | 非対話 | 広範囲 | 汎用 |
| Amazon Q Developer | AWS | 中 | マルチターン | 広範囲 | コード特化 |
| Cursor | Cursor Inc. | 低〜中 | マルチターン | 限定的 | コード特化 |
| Gemini | Google | 低〜中 | マルチターン | 広範囲 | 汎用 |
| Perplexity | Perplexity AI | 低 | マルチターン | 限定的 | 検索特化 |

### 傾向と気づき

1. **自律度とツール利用は正の相関がある**: 自律度が「高」の製品（Devin、AutoGPTなど）は、すべてツール利用が「広範囲」です。自律的に動作するためには、多様な外部ツールへのアクセスが不可欠であることがわかります。

2. **ドメイン特化型の方が自律度が高い傾向がある**: 汎用型（ChatGPT、Claude）は自律度が「低〜中」にとどまる一方、コード特化型（Devin、Claude Code）は「中〜高」を実現しています。対象ドメインを絞ることで、行動空間が限定され、安全に自律動作させやすくなるためと考えられます。

3. **対話方式は「マルチターン」が主流だが、完全自律型は「非対話」に近づく**: 多くの製品がマルチターンの対話を採用していますが、AutoGPTのような完全自律型は目標設定後に人間の介入を最小化する「非対話」型です。これは自律度のスペクトラムの端にある設計思想を反映しています。

</details>

<summary>解答例（改良版）</summary>

### 分類軸の定義（6軸に拡張）

| 分類軸 | 定義 | 評価スケール |
|--------|------|-------------|
| 自律度 | タスク遂行の自律性 | L1（補完）/ L2（提案）/ L3（半自律）/ L4（監視付き自律）/ L5（完全自律） |
| 対話方式 | インタラクション形態 | シングルターン / マルチターン / 非対話 / ハイブリッド |
| ツール利用 | 外部連携の範囲 | 0（なし）/ 1〜2種 / 3〜5種 / 6種以上 |
| ドメイン特化度 | 対応領域の広さ | 汎用 / セミ特化 / 特化 |
| メモリ機構 | 文脈保持の仕組み | なし / セッション内 / クロスセッション / 永続的 |
| エージェント構成 | 内部アーキテクチャ | 単一 / パイプライン / マルチエージェント |

**改良ポイント**: 自律度を5段階に細分化することで、製品間の微妙な違いを表現できます。また「メモリ機構」「エージェント構成」を追加することで、アーキテクチャの違いも捉えられます。

### 分類マップ（改良版）

| 製品名 | 自律度 | 対話方式 | ツール利用 | 特化度 | メモリ | 構成 |
|--------|--------|----------|-----------|--------|--------|------|
| ChatGPT | L2〜L3 | マルチターン | 6種以上 | 汎用 | クロスセッション | 単一 |
| Claude | L2〜L3 | マルチターン | 6種以上 | 汎用 | セッション内 | 単一 |
| GitHub Copilot | L1〜L2 | シングルターン | 1〜2種 | コード特化 | セッション内 | 単一 |
| Claude Code | L3〜L4 | ハイブリッド | 6種以上 | コード特化 | セッション内 | 単一 |
| Devin | L4 | ハイブリッド | 6種以上 | コード特化 | 永続的 | マルチエージェント |
| AutoGPT | L4〜L5 | 非対話 | 6種以上 | 汎用 | 永続的 | パイプライン |
| Amazon Q Developer | L2〜L3 | マルチターン | 3〜5種 | コード特化 | クロスセッション | 単一 |
| Cursor | L2 | マルチターン | 1〜2種 | コード特化 | セッション内 | 単一 |
| Gemini | L2〜L3 | マルチターン | 6種以上 | 汎用 | クロスセッション | 単一 |
| Perplexity | L2 | マルチターン | 1〜2種 | 検索特化 | セッション内 | パイプライン |
| CrewAI（フレームワーク） | L4 | 非対話 | 6種以上 | 汎用 | セッション内 | マルチエージェント |
| OpenAI Assistants API | L3 | マルチターン | 3〜5種 | 汎用 | 永続的 | 単一 |

### 傾向と気づき（改良版）

1. **自律度レベルとツール数は強い正の相関**: L4以上の製品はすべて6種以上のツールを活用しています。エージェントの自律性は「使えるツールの多様性」に大きく依存することが読み取れます。

2. **メモリ機構と自律度も相関がある**: 永続的メモリを持つ製品（Devin、AutoGPT）は自律度L4以上です。過去のコンテキストを保持できることが、長時間の自律動作に不可欠です。

3. **マルチエージェント構成は高自律度の製品に多い**: 単一エージェントではL3程度が上限であり、L4以上を実現するにはパイプラインまたはマルチエージェント構成が採用される傾向があります。

4. **「ハイブリッド」対話方式の台頭**: Claude Codeのように、通常はマルチターンだが必要に応じて非対話（バッチ）モードでも動作する「ハイブリッド」型が、最も実用的なバランスとして注目されています。

5. **ドメイン特化 × 高自律度が実用化の先端**: 汎用 × 高自律（AutoGPT）はまだ実験的ですが、コード特化 × 高自律（Devin、Claude Code）は実用段階に入っています。「まず特化領域で自律度を上げ、次に汎用化する」というアプローチが現在の業界トレンドです。

</details>

## 基本問題

### 問題1：機能要件と非機能要件の分類

以下の要件が「機能要件」と「非機能要件」のどちらに該当するかを分類してください。

1. ユーザーはメールアドレスとパスワードでログインできる
2. ページの表示速度は2秒以内とする
3. ユーザーは商品をカートに追加できる
4. システムは99.9%の稼働率を維持する
5. 全通信はHTTPSで暗号化する
6. ユーザーは注文履歴を一覧表示できる
7. 同時に500ユーザーがアクセスできる
8. パスワードはハッシュ化して保存する
9. ユーザーは商品にレビューを投稿できる
10. Chrome、Firefox、Safari、Edgeの最新2バージョンに対応する

**期待される出力例：**

```
機能要件：1, 3, 6, 9
非機能要件：2, 4, 5, 7, 8, 10
```

<details>
<summary>ヒント</summary>

機能要件は「システムが何をするか」（ユーザーが◯◯できる）、非機能要件は「システムがどのような品質で動くか」（性能、セキュリティ、可用性等）を定義します。

</details>

<details>
<summary>解答例</summary>

```
機能要件：
  1. ログイン機能 → ユーザーが実行する操作・機能
  3. カート追加機能 → ユーザーが実行する操作・機能
  6. 注文履歴表示機能 → ユーザーが実行する操作・機能
  9. レビュー投稿機能 → ユーザーが実行する操作・機能

非機能要件：
  2. 表示速度 → 性能要件
  4. 稼働率 → 可用性要件
  5. HTTPS暗号化 → セキュリティ要件
  7. 同時接続数 → 性能要件
  8. パスワードハッシュ化 → セキュリティ要件
  10. ブラウザ対応 → 互換性要件（使用性）
```

</details>

### 問題2：曖昧な要件の具体化

以下の曖昧な要件を、具体的かつ測定可能な要件に書き直してください。

1. 「システムは高速に動作すること」
2. 「セキュリティは万全にすること」
3. 「大量のデータを扱えること」
4. 「使いやすいインターフェースにすること」
5. 「障害時にすぐ復旧できること」

<details>
<summary>ヒント</summary>

具体的な数値目標を設定し、「誰が」「何を」確認すれば要件を満たしているかテストできるように書きましょう。

</details>

<details>
<summary>解答例</summary>

```
1. 元：「システムは高速に動作すること」
   改善：「APIレスポンスタイムは95パーセンタイルで500ms以内とする。
         画面の初期表示はLargest Contentful Paint（LCP）2.5秒以内とする」

2. 元：「セキュリティは万全にすること」
   改善：
   ・パスワードはbcrypt（コストファクター12）でハッシュ化して保存する
   ・全通信をTLS 1.2以上で暗号化する
   ・JWTのアクセストークン有効期限は15分とする
   ・OWASP Top 10の脆弱性に対する対策を実施する
   ・SQLインジェクション、XSS対策をフレームワークレベルで実施する

3. 元：「大量のデータを扱えること」
   改善：「商品マスタは100万件まで対応する。注文データは3年分（約500万件）を
         オンラインで参照可能とし、3年以上経過したデータはアーカイブする」

4. 元：「使いやすいインターフェースにすること」
   改善：
   ・主要な操作（商品検索→カート追加→注文完了）は5クリック以内で完了する
   ・エラーメッセージは具体的な対処方法を含めて表示する
   ・WCAG 2.1 レベルAAに準拠する
   ・モバイル（375px以上）とデスクトップの両方に対応する

5. 元：「障害時にすぐ復旧できること」
   改善：「障害検知から復旧までの目標時間（MTTR）は1時間以内とする。
         データベースのRPO（目標復旧時点）は5分以内とする。
         障害発生時は5分以内に監視システムからアラートを通知する」
```

</details>

### 問題3：ユースケース記述の作成

以下の機能について、ユースケース記述を作成してください。

**機能：** パスワードリセット

**前提情報：**
- ユーザーはメールアドレスを入力してパスワードリセットを要求できる
- システムがリセットリンク付きメールを送信する
- ユーザーはリンクから新しいパスワードを設定する

**期待される出力例（フォーマット）：**
```
ユースケースID：UC-XXX
アクター：
事前条件：
事後条件：
基本フロー：（番号付きの手順）
例外フロー：（エラーケース）
```

<details>
<summary>ヒント</summary>

基本フロー（正常系）だけでなく、例外フロー（メールアドレスが未登録の場合、リンクの有効期限切れの場合等）も考えましょう。

</details>

<details>
<summary>解答例</summary>

```markdown
## ユースケース：パスワードリセット

### 基本情報
- ユースケースID：UC-003
- アクター：未ログインユーザー
- 事前条件：ユーザーがアカウントを持っている
- 事後条件：新しいパスワードが設定され、古いパスワードは無効になる

### 基本フロー（正常系）
1. ユーザーがログイン画面で「パスワードを忘れた方はこちら」を押す
2. システムがパスワードリセット画面を表示する
3. ユーザーが登録済みメールアドレスを入力する
4. ユーザーが「リセットメールを送信」ボタンを押す
5. システムがメールアドレスの存在を確認する
6. システムがリセットトークン（有効期限: 1時間）を生成する
7. システムがリセットリンク付きメールを送信する
8. システムが「メールを送信しました」画面を表示する
9. ユーザーがメール内のリセットリンクをクリックする
10. システムがトークンの有効性を確認する
11. システムが新パスワード入力画面を表示する
12. ユーザーが新しいパスワードを入力する（2回）
13. ユーザーが「パスワードを変更」ボタンを押す
14. システムがパスワードを更新する
15. システムが「パスワードを変更しました」画面を表示する

### 例外フロー
- 5a. メールアドレスが未登録の場合
  1. システムは同じ「メールを送信しました」画面を表示する
     （※セキュリティ上、未登録であることを通知しない）

- 10a. リセットトークンの有効期限が切れている場合
  1. システムが「リンクの有効期限が切れています」と表示する
  2. 「もう一度リセットメールを送信する」リンクを表示する

- 10b. リセットトークンが既に使用済みの場合
  1. システムが「このリンクは既に使用されています」と表示する

- 12a. パスワードがポリシーを満たさない場合
  1. システムがエラーメッセージを表示する
     「パスワードは8文字以上で、英字と数字を含めてください」
  2. ユーザーがパスワードを再入力する
```

</details>

## 応用問題

### 問題4：要件定義書の作成

以下のシステムについて、要件定義書（抜粋）を作成してください。

**システム：** 社内の会議室予約システム

**背景：**
- 従業員100名のIT企業で、会議室が5部屋ある
- 現在はGoogleカレンダーで手動管理しているが、ダブルブッキングが頻発している
- Webブラウザから利用できるシステムを開発したい

以下の項目を含めてください：
1. 機能要件（5つ以上）
2. 非機能要件（3つ以上）
3. ユースケース1つ（基本フロー + 例外フロー）

<details>
<summary>ヒント</summary>

会議室予約に必要な機能を洗い出しましょう。予約の作成・変更・削除だけでなく、空き状況の確認、権限管理、通知なども考えてみましょう。

</details>

<details>
<summary>解答例</summary>

```markdown
# 会議室予約システム 要件定義書（抜粋）

## 機能要件

| ID | 機能名 | 説明 | 優先度 |
|----|--------|------|--------|
| FR-001 | ログイン | 社内アカウント（メール/パスワード）で認証できる | Must |
| FR-002 | 空き状況確認 | 日付・時間帯を指定して会議室の空き状況を確認できる | Must |
| FR-003 | 予約作成 | 会議室・日時・参加者を指定して予約を作成できる | Must |
| FR-004 | 予約変更 | 自分が作成した予約の日時・会議室を変更できる | Must |
| FR-005 | 予約削除 | 自分が作成した予約をキャンセルできる | Must |
| FR-006 | 予約通知 | 予約の作成・変更・削除時に参加者にメール通知を送る | Should |
| FR-007 | 繰り返し予約 | 毎週・毎月の定期予約を作成できる | Should |
| FR-008 | 管理者機能 | 管理者は全予約の編集・削除ができる | Must |

## 非機能要件

| ID | カテゴリ | 要件 | 目標値 |
|----|---------|------|--------|
| NFR-001 | 性能 | 画面の応答時間 | 1秒以内 |
| NFR-002 | 可用性 | システム稼働率 | 99.5%（平日9:00-21:00） |
| NFR-003 | 対応ブラウザ | ブラウザ対応 | Chrome, Firefox, Edgeの最新版 |
| NFR-004 | セキュリティ | 認証 | セッションタイムアウト8時間 |

## ユースケース：会議室を予約する

### 基本情報
- ユースケースID：UC-001
- アクター：ログイン済み従業員
- 事前条件：ユーザーがログインしている
- 事後条件：予約が作成され、参加者にメール通知が送信される

### 基本フロー
1. ユーザーが予約画面を開く
2. ユーザーが日付と時間帯を選択する
3. システムが選択した日時の会議室の空き状況を表示する
4. ユーザーが空いている会議室を選択する
5. ユーザーが会議のタイトルを入力する
6. ユーザーが参加者を選択する（任意）
7. ユーザーが「予約する」ボタンを押す
8. システムが予約の重複がないことを確認する
9. システムが予約を作成する
10. システムが参加者にメール通知を送信する
11. システムが「予約が完了しました」と表示する

### 例外フロー
- 8a. 別のユーザーが同じ時間帯に予約を入れていた場合（競合）
  1. システムが「この会議室はすでに予約されています」と表示する
  2. 空き状況を再表示する
  3. ユーザーが別の会議室または時間帯を選択する

- 2a. 過去の日時を選択した場合
  1. システムが「過去の日時は選択できません」と表示する
```

</details>

## チャレンジ問題

### 問題5：要件の優先度付け（MoSCoW法）

以下のオンライン学習プラットフォームの要件をMoSCoW法で分類してください。

**前提条件：**
- 開発期間：3ヶ月（MVP）
- チーム：エンジニア3名
- 対象ユーザー：社内の新入社員（50名）

**要件一覧：**
1. 動画教材をアップロード・視聴できる
2. テキスト教材を作成・閲覧できる
3. 理解度チェックのクイズ機能
4. 学習進捗のダッシュボード
5. 教材へのコメント・質問機能
6. AIによる個別学習プラン提案
7. 管理者が教材を管理できる
8. 修了証の自動発行
9. Slack連携通知
10. モバイルアプリ対応

各要件をMust / Should / Could / Won'tに分類し、その理由を説明してください。

<details>
<summary>ヒント</summary>

3ヶ月・3人という制約の中で、最低限の学習プラットフォームとして必要な機能は何かを考えましょう。「これがないとシステムとして成り立たない」ものがMustです。

</details>

<details>
<summary>解答例</summary>

```
■ Must（必須）
  2. テキスト教材を作成・閲覧できる
     → 学習プラットフォームの最も基本的な機能。動画より実装コストが低い。
  3. 理解度チェックのクイズ機能
     → 学習効果を確認する手段として必須。
  7. 管理者が教材を管理できる
     → 教材の追加・編集・削除ができないと運用できない。

■ Should（重要）
  1. 動画教材をアップロード・視聴できる
     → 重要だが、テキスト教材だけでもMVPとしては成り立つ。
       ストレージやストリーミングの実装コストも考慮。
  4. 学習進捗のダッシュボード
     → 管理者・学習者にとって有用だが、MVPでは簡易版で十分。

■ Could（あれば良い）
  5. 教材へのコメント・質問機能
     → 学習体験を向上するが、初期はSlack等で代替可能。
  9. Slack連携通知
     → 便利だがなくても運用可能。

■ Won't（今回はやらない）
  6. AIによる個別学習プラン提案
     → 実装コストが高く、50名規模では費用対効果が低い。
  8. 修了証の自動発行
     → 初期は手動発行で対応可能。
  10. モバイルアプリ対応
     → Webブラウザのレスポンシブ対応で代替。ネイティブアプリは不要。
```

</details>

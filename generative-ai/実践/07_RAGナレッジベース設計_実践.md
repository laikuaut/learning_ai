## 課題の要件

1. 以下の前提条件で、社内FAQ向けRAGシステムのアーキテクチャ設計書を作成する
   - 対象：従業員500名規模の企業
   - FAQデータ：300件の質問と回答（総務・人事・IT部門）
   - 社内ドキュメント：就業規則、各種申請手順書、ITマニュアル（計50文書）
2. システムアーキテクチャ図を作成する（コンポーネント間の関係を明示）
3. ドキュメントのチャンク分割戦略（Chunking Strategy）を設計する
4. 検索精度の評価方法（メトリクス（metrics））を定義する
5. データ登録・更新の運用フロー（誰が、いつ、どのように）を設計する
6. 回答品質を担保するためのプロンプトテンプレートを作成する

## ステップガイド

<details>
<summary>ステップ1：RAGの基本アーキテクチャを理解する</summary>

RAGは以下の3つのフェーズで構成されます。

```
1. インデキシング（Indexing）フェーズ
   ドキュメント → チャンク分割 → エンベディング（Embedding）→ ベクトルDB保存

2. 検索（Retrieval）フェーズ
   ユーザーの質問 → エンベディング → ベクトル類似検索 → 関連チャンク取得

3. 生成（Generation）フェーズ
   質問 + 関連チャンク → LLMに入力 → 回答生成
```

各フェーズの設計ポイントを理解してから設計に取り掛かりましょう。

**よくある間違い**: RAGを「AIに全ドキュメントを丸ごと読ませる」と誤解すること。実際は「質問に関連する部分だけを検索して渡す」仕組みです。

</details>

<details>
<summary>ステップ2：チャンク分割戦略を設計する</summary>

チャンク分割はRAGの精度に大きく影響します。以下を検討しましょう。

- **チャンクサイズ**: 何文字（トークン）で分割するか
- **オーバーラップ**: チャンク間の重複をどの程度設けるか
- **分割単位**: 文字数、段落、セクション、意味単位のどれで分割するか
- **メタデータ**: 各チャンクに付与する情報（出典、カテゴリ、更新日等）

ドキュメントの種類ごとに最適な戦略が異なります。

| ドキュメント種類 | 推奨チャンクサイズ | 分割単位 |
|----------------|:-----------------:|---------|
| FAQ | 1件=1チャンク | Q&Aペア |
| 就業規則 | 300〜500トークン | 条項 |
| 手順書 | 500〜800トークン | ステップ |

</details>

<details>
<summary>ステップ3：回答生成プロンプトを設計する</summary>

RAGの回答品質はプロンプトに大きく依存します。

以下の要素を含むプロンプトテンプレートを設計しましょう。

- **役割の指定**: 社内ヘルプデスク担当者として回答
- **参照情報の提示**: 検索で取得したチャンクを提示
- **回答ルール**: 参照情報に基づく回答のみ許可、推測禁止
- **出典の明示**: どのドキュメントから回答したかを記載
- **回答できない場合のフォールバック**: 「担当部署へお問い合わせください」等

</details>

<details>
<summary>ステップ4：評価指標と運用フローを設計する</summary>

**評価指標：**
- 検索の適合率（Precision）：検索結果に関連するチャンクが含まれる割合
- 検索の再現率（Recall）：関連するチャンクが検索結果に含まれる割合
- 回答の正確性：人間が正しいと判断した回答の割合
- ユーザー満足度：回答が役に立ったかの評価

**運用フロー：**
- データの新規登録・更新の手順
- 定期的な品質レビュー
- ユーザーフィードバックの収集方法

</details>

### 改良版の特徴

- **ハイブリッド検索（Hybrid Search）方式**を導入
- **チャンクの階層構造**を設計
- **回答の信頼度スコア**を導入
- **段階的導入計画**を追加

---

#### ハイブリッド検索方式

ベクトル検索（意味的な類似度）とキーワード検索（BM25等）を組み合わせます。

```
ユーザーの質問
     │
     ├─ ベクトル検索（意味的類似度）
     │    例：「休みを取りたい」→「有給休暇の申請方法」にヒット
     │
     ├─ キーワード検索（BM25）
     │    例：「パスワード リセット」→ 該当するITマニュアルにヒット
     │
     ▼
  スコア統合（Reciprocal Rank Fusion）
     │
     ▼
  上位k件のチャンクをLLMに渡す
```

**メリット：**
- ベクトル検索だけでは拾えない固有名詞やコード名をキーワード検索で補完
- キーワード検索だけでは対応できない言い換え表現をベクトル検索で補完

---

#### チャンクの階層構造

```
レベル1：ドキュメント全体のサマリー（200トークン）
    │
    ├─ レベル2：セクションサマリー（300トークン）
    │      │
    │      ├─ レベル3：詳細チャンク（500トークン）
    │      ├─ レベル3：詳細チャンク（500トークン）
    │      └─ レベル3：詳細チャンク（500トークン）
    │
    ├─ レベル2：セクションサマリー（300トークン）
    │      │
    │      └─ ...
    └─ ...
```

**検索戦略：**
1. まずレベル2（セクションサマリー）で検索し、関連セクションを特定
2. 該当セクション内のレベル3（詳細チャンク）を取得
3. 必要に応じてレベル1（全体サマリー）も添付してコンテキストを補強

---

#### 回答の信頼度スコア

```
回答生成時のプロンプトに以下を追加：

「回答の信頼度を以下の3段階で自己評価してください。
- 高：参照情報に明確な記載があり、回答に確信がある
- 中：参照情報に関連する記載はあるが、一部推論を含む
- 低：参照情報の記載が不十分で、正確性に不安がある」
```

**信頼度に応じた表示：**
```
信頼度「高」 → そのまま回答を表示
信頼度「中」 → 回答を表示 + 「より正確な情報は〇〇部にご確認ください」
信頼度「低」 → 回答を表示せず「担当部署へお問い合わせください」+ 問い合わせ先リンク
```

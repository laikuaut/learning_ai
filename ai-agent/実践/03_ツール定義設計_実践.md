# 実践課題03：ツール定義設計 ★2

> **難易度**: ★★☆☆☆
> **前提知識**: 第3章（ツール活用と外部連携）
> **課題の種類**: 設計課題
> **学習目標**: Function Callingのためのツールスキーマ（JSON Schema）を正しく設計でき、安全で使いやすいツールインターフェースを定義できるようになること

---

## 完成イメージ

以下のようなツール定義セット（JSON Schema形式）を作成します。

```
┌─────────────────────────────────────────────────┐
│           ツール定義セット（5ツール）              │
├─────────────────────────────────────────────────┤
│                                                 │
│  ┌──────────┐  ┌──────────┐  ┌──────────┐     │
│  │ 検索ツール │  │ 計算ツール │  │ DB問合せ  │     │
│  │          │  │          │  │  ツール   │     │
│  │ name     │  │ name     │  │ name     │     │
│  │ desc     │  │ desc     │  │ desc     │     │
│  │ params   │  │ params   │  │ params   │     │
│  │ returns  │  │ returns  │  │ returns  │     │
│  └──────────┘  └──────────┘  └──────────┘     │
│                                                 │
│  ┌──────────┐  ┌──────────┐                    │
│  │ メール    │  │ ファイル  │                    │
│  │ 送信     │  │ 操作     │                    │
│  │ ツール   │  │ ツール   │                    │
│  └──────────┘  └──────────┘                    │
│                                                 │
│  + エラーケース定義 + セキュリティ考慮事項        │
└─────────────────────────────────────────────────┘
```

---

## 課題の要件

1. **社内ヘルプデスクエージェント**を想定し、以下の5つのツールをJSON Schema形式で定義すること
   - `search_knowledge_base`: 社内ナレッジベースの検索
   - `calculate`: 数値計算（経費精算、勤怠計算など）
   - `query_database`: 社内DBへのクエリ実行
   - `send_email`: メール送信
   - `read_file`: ファイル読み込み
2. 各ツールについて以下を定義すること
   - **name**: ツール名（英語、snake_case）
   - **description**: ツールの説明（LLMが適切に使えるよう詳細に）
   - **parameters**: 入力パラメータ（JSON Schema形式、型・必須/任意・バリデーション）
   - **returns**: 戻り値の型と説明
3. 各ツールに**セキュリティ考慮事項**を1つ以上記述すること
4. **不正利用を防ぐバリデーションルール**を各ツールに1つ以上設定すること
5. エージェントがツールを**誤用するケース**を2つ以上想定し、防止策を記述すること

## ステップガイド

<details>
<summary>ステップ1：ツールの役割と使用場面を整理する</summary>

各ツールがどんな場面で使われるか整理しましょう。

| ツール | 使用場面の例 |
|--------|-------------|
| search_knowledge_base | 「有給休暇の申請方法を教えて」→ 社内マニュアルを検索 |
| calculate | 「先月の残業時間を計算して」→ 勤怠データから計算 |
| query_database | 「私の有給残日数は？」→ 人事DBに問い合わせ |
| send_email | 「IT部門にパスワードリセットを依頼して」→ メール送信 |
| read_file | 「この申請書テンプレートを見せて」→ ファイル読み込み |

</details>

<details>
<summary>ステップ2：JSON Schema形式でパラメータを定義する</summary>

OpenAI/Anthropicの Function Calling で使われるJSON Schema形式に従います。

```json
{
  "name": "ツール名",
  "description": "ツールの詳細な説明",
  "input_schema": {
    "type": "object",
    "properties": {
      "param1": {
        "type": "string",
        "description": "パラメータの説明"
      }
    },
    "required": ["param1"]
  }
}
```

**よくある間違い**:
- `description` が曖昧すぎる → LLMが適切なタイミングでツールを選べない
- `required` の設定漏れ → 必須パラメータなしでツールが呼ばれてしまう
- 型の定義ミス → `"type": "number"` とすべきところを `"type": "string"` にしてしまう

</details>

<details>
<summary>ステップ3：セキュリティとバリデーションを追加する</summary>

各ツールについて以下を考えましょう。

- **権限チェック**: そのユーザーがこのツールを使う権限があるか？
- **入力サニタイズ**: SQLインジェクション（injection）やパストラバーサル（path traversal）のリスクはないか？
- **実行制限**: 一度に大量のメール送信やDB更新を防ぐ仕組みがあるか？
- **確認フロー**: 破壊的操作（メール送信、データ変更）は人間の確認を必要とするか？

</details>

<details>
<summary>ステップ4：誤用ケースと防止策を考える</summary>

LLMがツールを誤用するケースを考えましょう。

例:
- ナレッジベース検索で十分な情報があるのに、DBクエリを実行してしまう
- ユーザーの確認なしにメールを送信してしまう
- `read_file` で機密ファイルにアクセスしてしまう

これらの防止策をシステム設計に組み込みます。

</details>

### ツール1: search_knowledge_base

```json
{
  "name": "search_knowledge_base",
  "description": "社内ナレッジベース（FAQ、マニュアル、規程集）をキーワードで検索します。まずこのツールで情報を探し、見つからない場合のみ他のツールを使ってください。",
  "input_schema": {
    "type": "object",
    "properties": {
      "query": {
        "type": "string",
        "description": "検索キーワードまたは質問文（日本語）",
        "minLength": 2,
        "maxLength": 200
      },
      "category": {
        "type": "string",
        "description": "検索カテゴリ",
        "enum": ["人事", "総務", "IT", "経理", "全て"],
        "default": "全て"
      },
      "max_results": {
        "type": "integer",
        "description": "取得する最大件数",
        "minimum": 1,
        "maximum": 10,
        "default": 5
      }
    },
    "required": ["query"]
  },
  "returns": {
    "type": "array",
    "description": "検索結果の配列。各要素は {title, content, relevance_score, url} を含む"
  },
  "security": [
    "ユーザーの所属部署に応じて、閲覧可能なドキュメントをフィルタリングする"
  ]

```

```json
  "name": "calculate",

  "input_schema": {
    "type": "object",
    "properties": {
```

### ツール3: query_database

```json
        "type": "string",

        "type": "array",
        "items": {"type": "string"},
        "description": "取得するカラム名のリスト"
      },
      "limit": {
        "type": "integer",
        "description": "取得する最大行数",
        "minimum": 1,
        "maximum": 100,

  "returns": {
    "type": "object",
```

### ツール4: send_email

```json
{
  "name": "send_email",

    "type": "object",
    "description": "{success: boolean, message_id: string}"
```

### ツール5: read_file

```json
{

  "input_schema": {
    "type": "object",
```

### 誤用ケースと防止策

| 誤用ケース | 説明 | 防止策 |
|-----------|------|--------|
| 確認なしメール送信 | ユーザーに「IT部門に連絡して」と言われ、確認なしにメール送信 | システムプロンプトで「send_email の前に必ず内容確認を取る」と明示。さらにツール実行前にユーザー確認UIを挟む |
| 過剰なDB参照 | 全社員の給与データを取得しようとする | conditions に employee_id が含まれない場合はエラー。limit の上限を100に制限 |

<details>
<summary>解答例（改良版）</summary>

改良版では以下の点を強化しています。

- **ツール選択ガイドライン**をシステムプロンプトとして定義
- **エラーレスポンス形式**を統一
- **ツール間の依存関係**を明示

### ツール選択ガイドライン（システムプロンプトに含める）

```
## ツール使用ルール

1. 情報取得の優先順位:
   search_knowledge_base → query_database → read_file
   （ナレッジベースで解決できるなら、DBに問い合わせない）

2. 破壊的操作（send_email）は必ずユーザー確認後に実行

3. calculate は単純な四則演算のみ。複雑なロジックは
   query_database の集計機能を使う

4. 1回のリクエストでツールを呼び出せる回数は最大5回
```

### 統一エラーレスポンス形式

```json
{
  "error": {
    "code": "PERMISSION_DENIED",
    "message": "このテーブルへのアクセス権限がありません",
    "tool": "query_database",
    "suggestion": "search_knowledge_base で関連情報を検索してください"
  }
}
```

エラーコード一覧:

| コード | 意味 |
|--------|------|
| INVALID_PARAMS | パラメータが不正 |
| PERMISSION_DENIED | アクセス権限なし |
| RATE_LIMITED | レート制限超過 |
| NOT_FOUND | 対象が見つからない |
| CONFIRMATION_REQUIRED | ユーザー確認が必要 |

### ツール定義（改良版 - send_email の例）

```json
{
  "name": "send_email",
  "description": "社内メールを送信します。【重要】このツールを呼び出す前に、必ず以下を実行してください: (1) 送信先・件名・本文をユーザーに提示する (2) ユーザーから明示的な送信許可を得る。許可なく呼び出した場合、CONFIRMATION_REQUIRED エラーが返されます。",
  "input_schema": {
    "type": "object",
    "properties": {
      "to": {
        "type": "array",
        "items": {
          "type": "string",
          "pattern": "^[a-zA-Z0-9.]+@company\\.co\\.jp$"
        },
        "description": "送信先メールアドレスのリスト（社内ドメインのみ、最大5件）",
        "minItems": 1,
        "maxItems": 5
      },
      "cc": {
        "type": "array",
        "items": {
          "type": "string",
          "pattern": "^[a-zA-Z0-9.]+@company\\.co\\.jp$"
        },
        "description": "CC先メールアドレスのリスト（任意）",
        "maxItems": 10
      },
      "subject": {
        "type": "string",
        "description": "メールの件名",
        "minLength": 1,
        "maxLength": 100
      },
      "body": {
        "type": "string",
        "description": "メールの本文（プレーンテキスト）",
        "minLength": 1,
        "maxLength": 5000
      },
      "priority": {
        "type": "string",
        "enum": ["low", "normal", "high"],
        "default": "normal",
        "description": "メールの優先度"
      },
      "user_confirmed": {
        "type": "boolean",
        "description": "ユーザーが送信を明示的に許可したか。true でなければ送信されない"
      }
    },
    "required": ["to", "subject", "body", "user_confirmed"]
  },
  "returns": {
    "type": "object",
    "properties": {
      "success": {"type": "boolean"},
      "message_id": {"type": "string"},
      "sent_at": {"type": "string", "format": "date-time"}
    }
  },
  "rate_limit": {
    "max_calls_per_hour": 10,
    "max_recipients_per_call": 5
  },
  "requires_confirmation": true,
  "side_effects": ["メールが送信される（取り消し不可）"]
}
```

**改良ポイントの解説**:

1. **`user_confirmed` パラメータの追加**: ツールのスキーマレベルで「ユーザー確認済みか」を強制しています。これにより、LLMが確認なしにツールを呼んでも `user_confirmed: false` のままではメールが送信されません。

2. **`to` を配列に変更**: 複数宛先への送信に対応しつつ、`maxItems: 5` で大量送信を防止しています。

3. **`rate_limit` と `side_effects` の明示**: ツール定義自体にレート制限と副作用を記述することで、LLMがツールの影響範囲を理解しやすくなります。

4. **`description` の詳細化**: 単なる説明ではなく、「使用前の手順」を明示しています。LLMはこの description を見てツールの使い方を判断するため、ここに制約を書くことが最も効果的です。

### ツール間の依存関係図

```
search_knowledge_base ←── 最優先で使用
        │
        │ 情報が不十分な場合
        ▼
query_database ────────── 構造化データの取得
        │
        │ 詳細資料が必要な場合
        ▼
   read_file ──────────── ファイル内容の確認
        │
        │ 外部への連絡が必要な場合
        ▼
  send_email ──────────── ※必ずユーザー確認後
        
  calculate ───────────── いつでも使用可能（副作用なし）
```

### 追加の誤用ケースと防止策

| 誤用ケース | リスク | 防止策 |
|-----------|--------|--------|
| `read_file` でパストラバーサル | `../../etc/passwd` のような不正パスで機密ファイルにアクセス | `pattern` による正規表現バリデーション + サーバー側でもパス正規化チェック |
| `query_database` でSQLインジェクション | `conditions` に悪意のあるSQL断片を埋め込む | パラメータ化クエリ（prepared statement）の使用。生SQLは一切受け付けない |
| `calculate` で無限ループ | 複雑な数式でサーバーリソースを消費 | 計算タイムアウト（5秒）の設定、再帰の深さ制限 |
| ツールの連鎖的誤用 | KB検索 → DB問い合わせ → メール送信を一気に実行 | ツール呼び出し回数の上限設定（1リクエストあたり最大5回） |

</details>

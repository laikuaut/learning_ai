# 第1章 演習：設計工程の概要

---

## 基本問題

### 問題1：設計工程の位置づけ

ソフトウェア開発の各工程を正しい順序に並び替え、各工程で行う主な作業を1つずつ挙げてください。

工程（順不同）：テスト、実装、要件定義、設計、リリース、保守

**期待される出力例：**

```
1. 要件定義 → 機能・非機能要件の定義
2. 設計 → システム構造の定義
3. 実装 → コーディング
4. テスト → 動作確認・品質検証
5. リリース → 本番環境への展開
6. 保守 → バグ修正・機能追加
```

<details>
<summary>ヒント</summary>

ウォーターフォールモデルの工程順序を思い出しましょう。上流工程から下流工程に順に進みます。

</details>

<details>
<summary>解答例</summary>

```
1. 要件定義 → ステークホルダーからの要求をもとに、機能要件・非機能要件を定義する
2. 設計    → 要件を実現するためのシステム構造・データ構造・画面・APIを設計する
3. 実装    → 設計書に基づいてソースコードを記述する
4. テスト  → 単体テスト・結合テスト・システムテストで品質を検証する
5. リリース → 本番環境にデプロイし、ユーザーが利用可能にする
6. 保守    → バグ修正、性能改善、機能追加を行う
```

</details>

### 問題2：設計フェーズの分類

以下の設計作業が「基本設計（外部設計）」と「詳細設計（内部設計）」のどちらに該当するかを分類してください。

1. システム構成図の作成
2. クラス図の作成
3. 画面遷移図の作成
4. シーケンス図の作成
5. API一覧の作成
6. メソッドの入出力定義
7. ER図（概要レベル）の作成
8. 状態遷移図の作成

**期待される出力例：**

```
基本設計：1, 3, 5, 7
詳細設計：2, 4, 6, 8
```

<details>
<summary>ヒント</summary>

基本設計は「ユーザーやステークホルダーから見える部分」、詳細設計は「開発者が実装するための内部構造」を定義します。

</details>

<details>
<summary>解答例</summary>

```
基本設計（外部設計）：
  1. システム構成図の作成 → システム全体のアーキテクチャを示す
  3. 画面遷移図の作成 → ユーザーが操作する画面の流れを示す
  5. API一覧の作成 → システム間のインターフェースの概要を示す
  7. ER図（概要レベル）の作成 → データの全体構造を示す

詳細設計（内部設計）：
  2. クラス図の作成 → モジュール内部のクラス構造を定義する
  4. シーケンス図の作成 → オブジェクト間のメッセージの流れを定義する
  6. メソッドの入出力定義 → 各メソッドの仕様を詳細に定義する
  8. 状態遷移図の作成 → エンティティの状態変化を定義する

※ER図やシーケンス図は基本設計・詳細設計の両方で使われることがあります。
概要レベルのER図は基本設計、カラムレベルの詳細なテーブル定義は詳細設計に
含まれます。
```

</details>

### 問題3：設計ドキュメントの種類と目的

以下の設計ドキュメントと、その主な目的を正しく対応させてください。

ドキュメント：
- A: 要件定義書
- B: 画面設計書
- C: API設計書
- D: データベース設計書
- E: 詳細設計書

目的：
- ① モジュール内部の処理ロジックを定義する
- ② システムが「何を」実現すべきかを定義する
- ③ テーブル構造とデータの関連を定義する
- ④ ユーザーが操作する画面のレイアウトと動作を定義する
- ⑤ フロントエンドとバックエンド間のインターフェースを定義する

<details>
<summary>解答例</summary>

```
A: 要件定義書         → ②「何を」実現すべきかを定義する
B: 画面設計書         → ④ 画面のレイアウトと動作を定義する
C: API設計書          → ⑤ フロント・バックエンド間のインターフェースを定義する
D: データベース設計書  → ③ テーブル構造とデータの関連を定義する
E: 詳細設計書         → ① モジュール内部の処理ロジックを定義する
```

</details>

## 応用問題

### 問題4：良い設計ドキュメントの改善

以下の設計書の記述には問題があります。それぞれの問題点を指摘し、改善案を提示してください。

1. 「検索結果は適切な件数で表示する」
2. 「パスワードは安全に保存する」
3. 「エラーが発生した場合は適切に処理する」
4. 「レスポンスは速く返す」
5. 「ユーザーデータはDBに保存する」

<details>
<summary>ヒント</summary>

良い設計書は「具体的」「測定可能」「テスト可能」であることが求められます。「適切に」「安全に」「速く」のような曖昧な表現は避けましょう。

</details>

<details>
<summary>解答例</summary>

```
1. 問題：「適切な件数」が曖昧
   改善：「検索結果は1ページあたり20件表示する。ページネーションに対応し、
         最大100ページまで表示する」

2. 問題：「安全に」が曖昧
   改善：「パスワードはbcrypt（コストファクター12）でハッシュ化して保存する。
         平文での保存・ログ出力は行わない」

3. 問題：「適切に処理」の内容が不明
   改善：「バリデーションエラーの場合はHTTP 400とエラー詳細を返す。
         サーバーエラーの場合はHTTP 500と汎用エラーメッセージを返し、
         エラーログを記録する」

4. 問題：「速く」が曖昧で測定不能
   改善：「APIレスポンスタイムは95パーセンタイルで200ms以内とする。
         商品一覧APIは500ms以内で応答する」

5. 問題：具体的なテーブル・データ構造が不明
   改善：「ユーザーデータはusersテーブルに保存する。
         カラムはid, name, email, password_hash, created_at, updated_atとする」
```

</details>

## チャレンジ問題

### 問題5：プロジェクトに必要な設計ドキュメントの選定

以下の3つのプロジェクトについて、どの設計ドキュメントを作成すべきか判断し、理由を述べてください。

**プロジェクトA：** 社内の5人チームで開発する、1ヶ月間のプロトタイプ開発（技術検証目的）

**プロジェクトB：** 外部ベンダーに委託する、半年間の業務システム開発（50画面規模）

**プロジェクトC：** 2人のエンジニアで開発する、既存APIに新しいエンドポイントを3つ追加する案件（1週間）

選択肢のドキュメント：
- 要件定義書
- システム構成図
- 画面設計書
- API設計書
- データベース設計書
- 詳細設計書
- テスト設計書

<details>
<summary>ヒント</summary>

プロジェクトの規模・期間・チーム体制・目的に応じて、必要なドキュメントは変わります。全部作ることが正解とは限りません。

</details>

<details>
<summary>解答例</summary>

```
■ プロジェクトA（5人チーム、1ヶ月、プロトタイプ）
  作成すべき：
  ・システム構成図 → チームで技術構成を共有するため
  ・API設計書（簡易版） → フロント・バックの分担を明確にするため

  不要な理由：
  ・要件定義書 → プロトタイプなので簡易なバックログで十分
  ・画面設計書 → プロトタイプなので口頭+ラフスケッチで十分
  ・詳細設計書 → 技術検証目的なので作り込む必要がない
  ・テスト設計書 → プロトタイプ段階では過剰

■ プロジェクトB（外部委託、半年、50画面）
  作成すべき：
  ・要件定義書 → 委託先との契約の根拠になるため必須
  ・システム構成図 → 全体アーキテクチャの合意のため
  ・画面設計書 → 50画面規模なので仕様を明確にする必要がある
  ・API設計書 → フロント・バックの責任分界点を明確にするため
  ・データベース設計書 → データ構造は業務の根幹のため
  ・詳細設計書 → 外部委託では実装品質を担保するために必要
  ・テスト設計書 → 受入テストの基準を明確にするため

  ※外部委託では「全て作成」が基本。設計書が契約・検収の根拠になる。

■ プロジェクトC（2人、1週間、API追加3つ）
  作成すべき：
  ・API設計書 → 追加する3エンドポイントの仕様を定義するため
  ・データベース設計書（差分のみ） → テーブル変更がある場合

  不要な理由：
  ・要件定義書 → 既存システムの小規模変更なのでチケットで管理
  ・画面設計書 → API追加のみで画面変更がなければ不要
  ・詳細設計書 → 小規模なのでAPI設計書内に処理概要を書けば十分
  ・テスト設計書 → 既存のテストフレームワークに追加する形で対応
```

</details>

# 第2章：認証と認可

## この章のゴール

- 認証（Authentication）と認可（Authorization）の違いを正確に説明できる
- パスワード管理のベストプラクティスを理解し、安全な実装を設計できる
- 多要素認証（MFA）の仕組みと各認証要素の特性を把握する
- OAuth 2.0 / OpenID Connect の認可フローを理解する
- JWT（JSON Web Token）の構造、利点、セキュリティ上の注意点を説明できる
- セッション管理における脅威と対策を実装レベルで理解する

---

## 2.1 認証と認可の基本概念

### 2.1.1 認証（Authentication）とは

認証とは、「あなたは誰ですか？」という問いに答えるプロセスです。システムにアクセスしようとしている利用者が、本当にその人物であることを確認します。

認証は、以下の3つの要素（認証ファクター）に基づいて行われます。

| 認証要素 | 概要 | 具体例 |
|---------|------|--------|
| **知識要素（Something you know）** | 本人だけが知っている情報 | パスワード、PIN、秘密の質問 |
| **所持要素（Something you have）** | 本人だけが持っている物 | スマートフォン、ICカード、ハードウェアトークン |
| **生体要素（Something you are）** | 本人の身体的特徴 | 指紋、虹彩、顔認証、静脈パターン |

### 2.1.2 認可（Authorization）とは

認可とは、「あなたは何をしてよいですか？」という問いに答えるプロセスです。認証されたユーザーに対して、どのリソースにどのような操作を許可するかを制御します。

```
認証（Authentication）  →  "あなたは田中さんですね" （本人確認）
認可（Authorization）    →  "田中さんには閲覧権限があります" （権限確認）
```

### 2.1.3 認証と認可の関係

```
┌──────────────────────────────────────────────────────┐
│                   アクセス制御                         │
│                                                      │
│  ① 識別（Identification）                            │
│     └─ "私はユーザーAです" （ID入力）                   │
│           ↓                                          │
│  ② 認証（Authentication）                            │
│     └─ "本当にユーザーAですね" （パスワード検証等）       │
│           ↓                                          │
│  ③ 認可（Authorization）                             │
│     └─ "ユーザーAにはこの権限があります" （権限付与）     │
│           ↓                                          │
│  ④ 説明責任（Accountability）                         │
│     └─ "ユーザーAがこの操作を行いました" （ログ記録）     │
│                                                      │
└──────────────────────────────────────────────────────┘
```

> **よくある間違い**: 認証と認可を混同するケースが非常に多いです。「ログインできた = すべての機能が使える」ではありません。認証はあくまで本人確認であり、認可は権限の付与です。この2つは必ず分離して設計してください。

## 2.2 パスワード管理

### 2.2.1 パスワードに対する攻撃手法

パスワードは最も広く使われている認証手段ですが、多くの攻撃手法が存在します。防御を設計するためには、攻撃者の手法を理解しておく必要があります。

| 攻撃手法 | 説明 | 対策 |
|---------|------|------|
| **ブルートフォース攻撃** | すべての文字の組み合わせを試行 | アカウントロックアウト、長いパスワード |
| **辞書攻撃** | 一般的な単語リストで試行 | 辞書に載っている単語を禁止 |
| **パスワードスプレー** | よく使われるパスワードを多数のアカウントに試行 | 共通パスワードのブラックリスト |
| **クレデンシャルスタッフィング** | 他サービスから漏洩した認証情報を使い回し | パスワードの使い回し禁止、漏洩チェック |
| **レインボーテーブル攻撃** | 事前計算したハッシュ値テーブルで逆引き | ソルト付きハッシュ |
| **フィッシング** | 偽サイトでパスワードを入力させる | ユーザー教育、多要素認証 |
| **ソーシャルエンジニアリング** | 人間の心理を利用して聞き出す | セキュリティ教育 |

### 2.2.2 安全なパスワード保存

パスワードを平文で保存することは、セキュリティ上の最大の過ちの一つです。以下に、脆弱な保存方法から安全な保存方法への段階を示します。

**脆弱なコード（平文保存 -- 絶対にやってはいけません）**:

```python
# NG: パスワードを平文でデータベースに保存
def save_user(username, password):
    db.execute(
        "INSERT INTO users (username, password) VALUES (?, ?)",
        (username, password)  # 平文のまま保存！
    )
```

**脆弱なコード（単純ハッシュ -- 不十分です）**:

```python
import hashlib

# NG: 単純なハッシュ化（レインボーテーブル攻撃に脆弱）
def save_user(username, password):
    hashed = hashlib.sha256(password.encode()).hexdigest()
    db.execute(
        "INSERT INTO users (username, password_hash) VALUES (?, ?)",
        (username, hashed)
    )
```

**修正後コード（ソルト付きストレッチングハッシュ）**:

```python
import bcrypt

# OK: bcrypt によるソルト付きハッシュ化
def save_user(username, password):
    # bcrypt は内部でソルトを生成し、ストレッチングも行う
    salt = bcrypt.gensalt(rounds=12)  # コストファクター12
    hashed = bcrypt.hashpw(password.encode('utf-8'), salt)
    db.execute(
        "INSERT INTO users (username, password_hash) VALUES (?, ?)",
        (username, hashed)
    )

def verify_password(password, stored_hash):
    # bcrypt がソルトの抽出と比較を自動で行う
    return bcrypt.checkpw(password.encode('utf-8'), stored_hash)
```

### 2.2.3 パスワードハッシュ化の重要な概念

```
┌──────────────────────────────────────────────────────────┐
│                  パスワードハッシュの仕組み                 │
│                                                          │
│  平文パスワード: "MyP@ssw0rd"                              │
│       ↓                                                  │
│  ソルト付加: "MyP@ssw0rd" + "a1b2c3d4"（ランダム値）        │
│       ↓                                                  │
│  ストレッチング: ハッシュ関数を数千〜数万回繰り返し適用       │
│       ↓                                                  │
│  保存値: "$2b$12$LJ3m4ys..." （ソルト + ハッシュ値）        │
│                                                          │
│  ※ 同じパスワードでもソルトが異なれば異なるハッシュ値になる   │
│  ※ ストレッチングにより総当たりの計算コストが大幅に増加       │
└──────────────────────────────────────────────────────────┘
```

| 用語 | 説明 |
|------|------|
| **ソルト（Salt）** | パスワードに付加するランダムな文字列。同じパスワードでも異なるハッシュ値を生成し、レインボーテーブル攻撃を防ぎます |
| **ペッパー（Pepper）** | アプリケーション側で管理する秘密の文字列。DBが漏洩してもペッパーがなければ解読が困難になります |
| **ストレッチング** | ハッシュ計算を意図的に繰り返すことで、攻撃者の総当たりコストを増大させます |
| **コストファクター** | ストレッチングの回数を制御するパラメータ。大きいほど安全ですが、認証に時間がかかります |

### 2.2.4 推奨されるパスワードハッシュアルゴリズム

| アルゴリズム | 特徴 | 推奨度 |
|-------------|------|--------|
| **Argon2id** | メモリハード関数。GPU攻撃に強い。Password Hashing Competitionの勝者 | 最推奨 |
| **bcrypt** | 広く使われ、実績豊富。コストファクターで調整可能 | 推奨 |
| **scrypt** | メモリハード関数。CPUとメモリの両方でコスト増大 | 推奨 |
| **PBKDF2** | NIST推奨。HMAC-SHA256ベース。反復回数で調整 | 許容 |
| SHA-256/SHA-512（単体） | パスワード保存には不適切（高速すぎる） | 非推奨 |
| MD5 | 衝突脆弱性あり。パスワード保存には絶対に使わない | 禁止 |

> **現場での注意点**: パスワードハッシュアルゴリズムは「遅い」ことが重要です。一般的なハッシュ関数（SHA-256等）は高速に設計されているため、攻撃者の総当たりに有利になってしまいます。bcryptやArgon2idは意図的に低速にすることで、攻撃コストを上げています。

## 2.3 多要素認証（MFA）

### 2.3.1 多要素認証とは

多要素認証（Multi-Factor Authentication）とは、前述の3つの認証要素のうち、**2つ以上の異なる要素**を組み合わせて認証を行う方式です。

```
┌────────────────────────────────────────────────┐
│              多要素認証の組み合わせ               │
│                                                │
│  パスワード（知識） ＋ SMS コード（所持）          │
│  パスワード（知識） ＋ 指紋認証（生体）            │
│  ICカード（所持）   ＋ PIN（知識）                │
│  パスワード（知識） ＋ TOTP（所持）＋ 指紋（生体） │
│                                                │
│  ※ パスワード ＋ 秘密の質問 は「二段階認証」だが   │
│    「二要素認証」ではない（どちらも知識要素）       │
└────────────────────────────────────────────────┘
```

### 2.3.2 MFA の主要方式

| 方式 | 仕組み | セキュリティ強度 | 利便性 |
|------|--------|----------------|--------|
| **TOTP（Time-based OTP）** | 時刻ベースのワンタイムパスワード（Google Authenticator等） | 高 | 中 |
| **HOTP（HMAC-based OTP）** | カウンタベースのワンタイムパスワード | 中〜高 | 中 |
| **SMS認証** | SMSでコードを送信 | 中（SIMスワップ攻撃に脆弱） | 高 |
| **プッシュ通知** | 認証アプリへのプッシュ通知で承認 | 高 | 高 |
| **FIDO2/WebAuthn** | 公開鍵暗号ベースのパスワードレス認証 | 非常に高 | 高 |
| **ハードウェアトークン** | 物理デバイスによるワンタイムパスワード | 非常に高 | 低〜中 |

### 2.3.3 TOTP の仕組み

```
┌──────────────────────────────────────────────────┐
│                TOTP の動作原理                     │
│                                                  │
│  ① 初期設定時に秘密鍵（シード）を共有              │
│     サーバー ←→ 認証アプリ                        │
│            （QRコードで共有）                      │
│                                                  │
│  ② 認証時                                        │
│     認証アプリ側:                                  │
│       TOTP = HMAC-SHA1(秘密鍵, 現在時刻/30秒)     │
│       → 6桁の数字を表示                           │
│                                                  │
│     サーバー側:                                    │
│       同じ計算を行い、ユーザー入力と比較            │
│       （前後30秒のずれを許容）                     │
│                                                  │
│  ③ 30秒ごとにコードが更新される                    │
└──────────────────────────────────────────────────┘
```

### 2.3.4 FIDO2/WebAuthn（パスワードレス認証）

FIDO2は、パスワードに依存しない次世代の認証標準です。公開鍵暗号方式を用いて、フィッシングに対して本質的に耐性のある認証を実現します。

```
┌───────────────────────────────────────────────────────────┐
│              FIDO2/WebAuthn 認証フロー                      │
│                                                           │
│  【登録時】                                                │
│  ユーザー → 認証器で鍵ペア生成 → 公開鍵をサーバーに登録     │
│                                                           │
│  【認証時】                                                │
│  サーバー → チャレンジ送信 → 認証器で秘密鍵により署名       │
│          ← 署名を返送     ← ユーザーが生体認証等で承認     │
│  サーバー → 公開鍵で署名検証 → 認証成功                     │
│                                                           │
│  ※ 秘密鍵はデバイスから出ない                               │
│  ※ サーバーごとに異なる鍵ペアを使用（フィッシング耐性）      │
└───────────────────────────────────────────────────────────┘
```

> **よくある間違い**: 「二段階認証」と「二要素認証」を同じものとして扱うことがあります。パスワード + 秘密の質問は二段階ですが、どちらも「知識要素」なので二要素ではありません。真の多要素認証は、異なる種類の認証要素を組み合わせる必要があります。

## 2.4 OAuth 2.0 と OpenID Connect

### 2.4.1 OAuth 2.0 とは

OAuth 2.0 は、第三者のアプリケーションに対して、ユーザーのリソースへの限定的なアクセスを許可するための**認可フレームワーク**です。ユーザーのパスワードを第三者に渡すことなく、安全にアクセス権を委譲できます。

### 2.4.2 OAuth 2.0 の登場人物

| 役割 | 説明 | 具体例 |
|------|------|--------|
| **リソースオーナー** | リソースの所有者（エンドユーザー） | Googleアカウントの持ち主 |
| **クライアント** | リソースにアクセスしたいアプリケーション | サードパーティアプリ |
| **認可サーバー** | アクセストークンを発行するサーバー | Google の OAuth サーバー |
| **リソースサーバー** | 保護されたリソースを提供するサーバー | Google Drive API |

### 2.4.3 認可コードフロー（Authorization Code Flow）

最も一般的で安全なフローです。サーバーサイドアプリケーションに推奨されます。

```
┌──────┐    ┌──────────┐    ┌──────────┐    ┌──────────┐
│ユーザー│    │クライアント│    │認可サーバー│    │リソース   │
│      │    │（アプリ）  │    │          │    │サーバー   │
└──┬───┘    └────┬─────┘    └────┬─────┘    └────┬─────┘
   │  ①ログイン   │              │              │
   │───────────→│              │              │
   │             │  ②認可リクエスト│              │
   │             │──────────→│              │
   │  ③ログイン画面│              │              │
   │←─────────────────────────│              │
   │  ④認証情報入力│              │              │
   │─────────────────────────→│              │
   │             │  ⑤認可コード  │              │
   │             │←─────────│              │
   │             │  ⑥トークン要求 │              │
   │             │──────────→│              │
   │             │  ⑦アクセストークン              │
   │             │←─────────│              │
   │             │  ⑧API呼び出し │              │
   │             │──────────────────────→│
   │             │  ⑨リソース応答 │              │
   │             │←──────────────────────│
   │  ⑩結果表示   │              │              │
   │←──────────│              │              │
```

### 2.4.4 OAuth 2.0 の主要なフロー

| フロー | 用途 | セキュリティ |
|--------|------|------------|
| **認可コードフロー** | サーバーサイドアプリ | 高（推奨） |
| **認可コードフロー + PKCE** | SPA・モバイルアプリ | 高（推奨） |
| **クライアントクレデンシャルフロー** | サーバー間通信（M2M） | 高 |
| インプリシットフロー | SPA（レガシー） | 低（非推奨） |
| リソースオーナーパスワードフロー | 信頼されたアプリのみ | 低（非推奨） |

### 2.4.5 PKCE（Proof Key for Code Exchange）

PKCE は、認可コード横取り攻撃を防ぐための拡張仕様です。SPAやモバイルアプリでは必須とされています。

```
┌──────────────────────────────────────────────────────┐
│                  PKCE の仕組み                         │
│                                                      │
│  ① クライアントがランダムな code_verifier を生成       │
│  ② code_verifier のハッシュ = code_challenge を計算   │
│  ③ 認可リクエストに code_challenge を含める            │
│  ④ トークンリクエストに code_verifier を含める         │
│  ⑤ 認可サーバーが code_challenge と照合               │
│                                                      │
│  → 認可コードを横取りしても code_verifier がなければ    │
│    トークンを取得できない                              │
└──────────────────────────────────────────────────────┘
```

### 2.4.6 OpenID Connect（OIDC）

OpenID Connect は、OAuth 2.0 の上に**認証レイヤー**を追加した規格です。OAuth 2.0 は「認可」のみですが、OIDC を使うことで「認証」も行えるようになります。

| 比較項目 | OAuth 2.0 | OpenID Connect |
|---------|-----------|---------------|
| 目的 | 認可（リソースへのアクセス許可） | 認証（ユーザーの本人確認） |
| 発行トークン | アクセストークン | アクセストークン + **IDトークン** |
| ユーザー情報 | 規定なし | UserInfoエンドポイントで取得 |
| 利用場面 | API アクセス委譲 | SSO（シングルサインオン） |

> **現場での注意点**: OAuth 2.0 は「認可」のフレームワークであり、「認証」には使えません。「Googleでログイン」のような機能を実装する場合は、OAuth 2.0 ではなく OpenID Connect を使用してください。OAuth 2.0 のアクセストークンだけでは、ユーザーが誰であるかを安全に確認できません。

## 2.5 JWT（JSON Web Token）

### 2.5.1 JWT とは

JWT（JSON Web Token）は、二者間で情報を安全に伝達するためのコンパクトなトークン形式です。署名付きのため改ざん検知が可能で、OAuth 2.0 のアクセストークンや OpenID Connect の ID トークンとして広く利用されています。

### 2.5.2 JWT の構造

JWT は、ドット（`.`）で区切られた3つの部分で構成されます。

```
eyJhbGciOiJIUzI1NiJ9.eyJzdWIiOiIxMjM0In0.signature
 ←── ヘッダー ──→ ←──── ペイロード ────→ ←─ 署名 ─→
```

```
┌────────────────────────────────────────────────────┐
│  ① ヘッダー（Header）-- Base64URLエンコード         │
│  {                                                 │
│    "alg": "HS256",    // 署名アルゴリズム            │
│    "typ": "JWT"       // トークンタイプ              │
│  }                                                 │
├────────────────────────────────────────────────────┤
│  ② ペイロード（Payload）-- Base64URLエンコード       │
│  {                                                 │
│    "sub": "1234567890",  // ユーザーID              │
│    "name": "Taro",       // カスタムクレーム         │
│    "iat": 1516239022,    // 発行時刻                │
│    "exp": 1516242622     // 有効期限                │
│  }                                                 │
├────────────────────────────────────────────────────┤
│  ③ 署名（Signature）                               │
│  HMAC-SHA256(                                      │
│    base64UrlEncode(header) + "." +                 │
│    base64UrlEncode(payload),                       │
│    secret                                          │
│  )                                                 │
└────────────────────────────────────────────────────┘
```

### 2.5.3 JWT の主要クレーム

| クレーム | 名称 | 説明 |
|---------|------|------|
| `iss` | Issuer | トークンの発行者 |
| `sub` | Subject | トークンの主体（ユーザーID等） |
| `aud` | Audience | トークンの受信者（対象サービス） |
| `exp` | Expiration | 有効期限（UNIXタイムスタンプ） |
| `iat` | Issued At | 発行時刻 |
| `nbf` | Not Before | この時刻より前は無効 |
| `jti` | JWT ID | トークンの一意な識別子 |

### 2.5.4 JWT のセキュリティ上の注意点

**脆弱なコード（アルゴリズムの検証なし）**:

```javascript
// NG: ヘッダーの alg をそのまま信頼する
const decoded = jwt.verify(token, secret);  // alg: "none" 攻撃に脆弱
```

**修正後コード（アルゴリズムを明示的に指定）**:

```javascript
// OK: 許可するアルゴリズムを明示的に指定
const decoded = jwt.verify(token, secret, {
  algorithms: ['HS256'],  // 許可するアルゴリズムを限定
  issuer: 'https://myapp.example.com',  // 発行者を検証
  audience: 'my-api',  // 受信者を検証
});
```

### 2.5.5 JWT に関する主な攻撃と対策

| 攻撃 | 説明 | 対策 |
|------|------|------|
| **alg: none 攻撃** | 署名なしアルゴリズムを指定 | アルゴリズムをサーバー側で固定 |
| **鍵混同攻撃** | RS256 → HS256 に変更し、公開鍵を共通鍵として使用 | アルゴリズムを明示的に検証 |
| **トークン漏洩** | XSS等でトークンが窃取される | 有効期限を短く設定、HttpOnlyクッキーで保存 |
| **リプレイ攻撃** | 盗んだトークンを再利用 | jti によるワンタイム化、有効期限の短縮 |

> **よくある間違い**: JWT のペイロードは Base64URL エンコードされているだけで、**暗号化されていません**。誰でもデコードして中身を読めます。機密情報（パスワード、クレジットカード番号等）をペイロードに含めてはいけません。暗号化が必要な場合は JWE（JSON Web Encryption）を使用してください。

## 2.6 セッション管理

### 2.6.1 セッション管理とは

HTTP は本来ステートレスなプロトコルです。セッション管理とは、複数のリクエストにまたがってユーザーの状態を維持する仕組みです。

### 2.6.2 セッション管理の方式比較

| 方式 | 仕組み | メリット | デメリット |
|------|--------|---------|-----------|
| **サーバーサイドセッション** | サーバーにセッションデータを保存し、セッションIDをクッキーで管理 | サーバー側でセッション無効化が容易 | サーバーのメモリ/ストレージを使用 |
| **JWT ベースのセッション** | トークンにセッション情報を含めてクライアント側で保持 | サーバーがステートレスでスケールしやすい | トークン無効化が困難 |

### 2.6.3 セッションに対する攻撃

```
┌──────────────────────────────────────────────────────────┐
│          セッション関連の主要な攻撃                         │
│                                                          │
│  ① セッションハイジャック                                  │
│     └─ 有効なセッションIDを窃取して成りすまし               │
│     └─ 手法: XSS、ネットワーク盗聴、推測                   │
│                                                          │
│  ② セッションフィクセーション                              │
│     └─ 攻撃者が用意したセッションIDでユーザーにログインさせる │
│     └─ ログイン後にそのセッションIDで成りすまし             │
│                                                          │
│  ③ CSRF（クロスサイトリクエストフォージェリ）               │
│     └─ ユーザーのセッションを利用して不正なリクエストを送信   │
│     └─ 詳細は第4章で解説                                  │
└──────────────────────────────────────────────────────────┘
```

### 2.6.4 安全なセッション管理の実装

**脆弱なコード（セッション固定化攻撃に脆弱）**:

```python
# NG: ログイン前後でセッションIDが変わらない
@app.route('/login', methods=['POST'])
def login():
    username = request.form['username']
    password = request.form['password']
    if verify_credentials(username, password):
        session['user'] = username  # セッションIDが変わらない！
        return redirect('/dashboard')
```

**修正後コード**:

```python
# OK: ログイン成功時にセッションIDを再生成
@app.route('/login', methods=['POST'])
def login():
    username = request.form['username']
    password = request.form['password']
    if verify_credentials(username, password):
        # セッションフィクセーション対策: セッションIDを再生成
        session.regenerate()
        session['user'] = username
        session['login_time'] = datetime.now().isoformat()
        return redirect('/dashboard')
```

### 2.6.5 セッションクッキーのセキュリティ属性

```python
# 安全なセッションクッキーの設定例
app.config.update(
    SESSION_COOKIE_SECURE=True,     # HTTPS でのみ送信
    SESSION_COOKIE_HTTPONLY=True,    # JavaScript からアクセス不可
    SESSION_COOKIE_SAMESITE='Lax',  # CSRF 対策
    SESSION_COOKIE_NAME='__Host-session',  # Cookie Prefix
    PERMANENT_SESSION_LIFETIME=timedelta(hours=1),  # セッション有効期限
)
```

| 属性 | 説明 | 目的 |
|------|------|------|
| `Secure` | HTTPS接続時のみクッキーを送信 | 盗聴防止 |
| `HttpOnly` | JavaScriptからのアクセスを禁止 | XSSによる窃取防止 |
| `SameSite` | クロスサイトリクエストでのクッキー送信を制限 | CSRF対策 |
| `Path` | クッキーの有効パスを限定 | 不要なパスへの送信防止 |
| `Domain` | クッキーの有効ドメインを指定 | スコープの限定 |
| `__Host-` プレフィックス | Secure必須、Domain指定不可、Path=/のみ | 最も制限的な設定の強制 |

## 2.7 アクセス制御モデル

### 2.7.1 主要なアクセス制御モデル

| モデル | 説明 | 適用場面 |
|--------|------|---------|
| **DAC（任意アクセス制御）** | リソースの所有者がアクセス権を管理 | ファイルシステム（Unix のパーミッション） |
| **MAC（強制アクセス制御）** | システムがセキュリティレベルに基づいて制御 | 軍事・政府機関のシステム |
| **RBAC（ロールベースアクセス制御）** | 役割に基づいて権限を割り当て | 企業の業務システム |
| **ABAC（属性ベースアクセス制御）** | 属性（時間、場所、役割等）の組み合わせで制御 | クラウドサービス、複雑なポリシー |

### 2.7.2 RBAC の実装例

```python
# RBAC の基本的な実装例
class Permission:
    READ = 'read'
    WRITE = 'write'
    DELETE = 'delete'
    ADMIN = 'admin'

# ロールと権限のマッピング
ROLE_PERMISSIONS = {
    'viewer':  {Permission.READ},
    'editor':  {Permission.READ, Permission.WRITE},
    'admin':   {Permission.READ, Permission.WRITE, Permission.DELETE, Permission.ADMIN},
}

def check_permission(user_role, required_permission):
    """ユーザーのロールが必要な権限を持っているか確認"""
    role_perms = ROLE_PERMISSIONS.get(user_role, set())
    if required_permission not in role_perms:
        raise PermissionError(
            f"ロール '{user_role}' には '{required_permission}' 権限がありません"
        )
    return True

# デコレータとしての利用
from functools import wraps

def require_permission(permission):
    def decorator(f):
        @wraps(f)
        def wrapper(*args, **kwargs):
            user = get_current_user()
            check_permission(user.role, permission)
            return f(*args, **kwargs)
        return wrapper
    return decorator

@require_permission(Permission.DELETE)
def delete_document(doc_id):
    """管理者のみ実行可能"""
    # ドキュメント削除ロジック
    pass
```

## 2.8 まとめ

### 重要ポイント

1. **認証は本人確認、認可は権限付与** -- この2つは必ず分離して設計します
2. パスワードは**Argon2id / bcrypt**等の専用アルゴリズムでハッシュ化し、**ソルト**を付加して保存します
3. **多要素認証**は異なる種類の認証要素を組み合わせることが重要です（二段階 ≠ 二要素）
4. **OAuth 2.0 は認可**、**OpenID Connect は認証**のための規格です
5. JWT のペイロードは暗号化されていないため、**機密情報を含めてはいけません**
6. セッション管理では、セッションIDの再生成、クッキー属性の適切な設定が不可欠です

### 試験対策キーワード

- 認証と認可、識別、説明責任
- 多要素認証（MFA）、二要素認証、TOTP、FIDO2/WebAuthn
- パスワードハッシュ（bcrypt、Argon2id、ソルト、ストレッチング）
- OAuth 2.0、認可コードフロー、PKCE、OpenID Connect
- JWT（ヘッダー、ペイロード、署名、クレーム）
- セッション管理、セッションハイジャック、セッションフィクセーション
- アクセス制御（DAC、MAC、RBAC、ABAC）
- 最小権限の原則、Need to Know

### 問題2：コード生成AIの適切な活用場面の判断

以下の10個のシナリオについて、コード生成AIを「積極的に活用すべき」「補助的に活用すべき」「使用を避けるべき」の3段階で分類し、その理由を述べてください。

1. 定型的なCRUD APIの実装
2. セキュリティ認証ロジックの実装
3. ユニットテストの作成
4. データベースのマイグレーションファイル作成
5. ボイラープレートコードの生成
6. パフォーマンスが極めて重要なアルゴリズムの実装
7. 正規表現パターンの作成
8. 機密情報を扱うコードのデバッグ
9. ドキュメント・コメントの生成
10. 新しいフレームワークのプロトタイプ作成

**期待される出力例：**
```
【積極的に活用すべき】
1. 定型的なCRUD API → 理由：パターンが決まっており、AIが得意な領域
5. ボイラープレート → 理由：...

【補助的に活用すべき】
...

【使用を避けるべき】
8. 機密情報を扱うデバッグ → 理由：機密データが外部APIに送信されるリスク
```

<details>
<summary>ヒント</summary>

- 「積極活用」の基準：パターンが定型的、ミスのリスクが低い、人間がレビューしやすい
- 「補助活用」の基準：AIの出力を参考にしつつ、人間が慎重に検証すべき
- 「使用を避ける」の基準：セキュリティリスク、機密性、高い正確性が求められる場面

</details>

<details>
<summary>解答例</summary>

```
【積極的に活用すべき】

1. 定型的なCRUD APIの実装
   理由：パターンが確立されており、AIが高品質なコードを生成しやすい。
   エンドポイントの定義、バリデーション、レスポンス形式など、
   定型処理の生成でエンジニアの時間を大幅に節約できる。

3. ユニットテストの作成
   理由：テストは既存のコードに対して書くため、AIが参照すべき
   コンテキストが明確。テストケースの網羅性を向上させる効果もある。
   ただし、生成されたテストが意味のあるアサーションをしているか確認は必要。

5. ボイラープレートコードの生成
   理由：設定ファイル、初期化コード、型定義など、パターンが決まった
   コードの生成はAIが最も得意とする領域。手作業では時間がかかる
   退屈な作業を効率化できる。

7. 正規表現パターンの作成
   理由：正規表現は人間が一から書くとミスしやすく時間がかかる。
   自然言語で要件を伝えれば、AIが正確なパターンを生成してくれる。
   ただし、エッジケースのテストは人間が行うべき。

9. ドキュメント・コメントの生成
   理由：コードの意図を読み取ってドキュメントを生成する作業は
   AIが得意。JSDocやdocstringの生成で開発者の負担を軽減できる。

---

【補助的に活用すべき】

4. データベースのマイグレーションファイル作成
   理由：AIが生成したマイグレーションはそのまま使えることが多いが、
   データの整合性に影響するため、必ず人間がレビューし、
   ステージング環境でテストしてから本番に適用する。

6. パフォーマンスが極めて重要なアルゴリズムの実装
   理由：AIは一般的なアルゴリズムを提案できるが、特定のハードウェアや
   データ特性に最適化されたコードを書くことは難しい。
   AIの提案をベースに、ベンチマークを取りながら人間が最適化する。

10. 新しいフレームワークのプロトタイプ作成
    理由：フレームワークの基本的な使い方をAIに聞きながらプロトタイプを
    作るのは効率的。ただし、AIの学習データが古い場合があるため、
    公式ドキュメントとの照合を必ず行う。

---

【使用を避けるべき】

2. セキュリティ認証ロジックの実装
   理由：認証・認可は脆弱性が直接的なセキュリティインシデントに
   つながる。AIが生成したコードに微妙な脆弱性（タイミング攻撃、
   不適切なトークン管理等）が含まれる可能性がある。
   確立されたライブラリ（OAuth2、JWT等）を使用し、
   セキュリティ専門家のレビューを受けるべき。

8. 機密情報を扱うコードのデバッグ
   理由：クラウドベースのAIサービスにコードを送信すると、
   機密情報（APIキー、個人情報、社内ロジック等）が
   外部サーバーに送信されるリスクがある。
   ローカル実行可能なツールを使うか、機密部分を除外して相談する。
```

</details>

### 問題3：効果的なコード生成指示の基本

以下の曖昧な指示を、コード生成AIが正確に理解できる具体的な指示に書き直してください。

**曖昧な指示：**
1. 「ログイン機能を作って」
2. 「このコードをきれいにして」
3. 「エラーハンドリングを追加して」

**期待される出力例：**
```
1. 改善前：「ログイン機能を作って」
   改善後：「Express.jsとTypeScriptで、以下の仕様のログインAPIエンドポイントを
   作成してください。
   - POST /api/auth/login
   - リクエスト：{ email: string, password: string }
   - bcryptでパスワード検証
   - JWTトークンを発行（有効期限24時間）
   - バリデーションエラー時は400、認証失敗時は401を返す」
```

<details>
<summary>ヒント</summary>

- 良い指示には「技術スタック」「入出力の仕様」「制約条件」「エラーケース」が含まれます
- コードのリファクタリングでは「何を基準にきれいにするか」を明示しましょう
- エラーハンドリングでは「どのエラーを」「どう処理するか」を具体的に伝えます

</details>

<details>
<summary>解答例</summary>

```
1. 改善前：「ログイン機能を作って」
   改善後：
   「Express.js（TypeScript）で以下の仕様のログインAPIを実装してください。

   【エンドポイント】
   POST /api/auth/login

   【リクエストBody】
   { email: string, password: string }

   【処理フロー】
   1. リクエストボディのバリデーション（zodを使用）
   2. emailでユーザーをDBから検索（Prismaを使用）
   3. bcryptでパスワードを検証
   4. JWTトークンを生成（有効期限: 24時間、ペイロード: userId, email）
   5. レスポンスにトークンを含めて返す

   【レスポンス】
   成功時(200): { token: string, user: { id, email, name } }
   バリデーションエラー(400): { error: string, details: ValidationError[] }
   認証失敗(401): { error: 'Invalid email or password' }
   サーバーエラー(500): { error: 'Internal server error' }

   【制約】
   - パスワードのハッシュ比較にはconstant-time comparisonを使用する
   - レートリミットは別途ミドルウェアで実装するため不要
   - ログ出力にパスワードを含めないこと」

---

2. 改善前：「このコードをきれいにして」
   改善後：
   「以下のコードを次の基準でリファクタリングしてください。

   【リファクタリング基準】
   1. 関数の分割：1関数あたり20行以内に収める
   2. 命名の改善：変数名・関数名を意図が伝わる名前に変更する
   3. 早期リターン：ネストが3段以上のif文をガード節に変換する
   4. マジックナンバーの排除：定数として名前を付ける
   5. TypeScriptの型を厳密にする（anyを排除する）

   【制約】
   - 外部から呼ばれている関数のシグネチャは変更しない
   - 動作は完全に同一を維持する（リファクタリングのみ）
   - 変更理由をコメントで説明する」

---

3. 改善前：「エラーハンドリングを追加して」
   改善後：
   「以下のNode.jsコードに、次のエラーハンドリングを追加してください。

   【対応するエラー】
   1. ネットワークエラー（API呼び出し失敗）
      → 3回まで指数バックオフでリトライ、それでも失敗したらエラーログを出力
   2. JSONパースエラー（レスポンスが不正なJSON）
      → デフォルト値を使用して処理を継続、警告ログを出力
   3. バリデーションエラー（必須フィールドの欠落）
      → 具体的なフィールド名を含むエラーメッセージをthrow
   4. タイムアウト（5秒以上の応答遅延）
      → AbortControllerでリクエストをキャンセルし、タイムアウトエラーをthrow

   【エラーログの形式】
   - タイムスタンプ、エラー種別、メッセージ、スタックトレースを含める
   - console.errorではなくloggerオブジェクト（winston）を使用する

   【型定義】
   - カスタムエラークラスを定義する（AppError extends Error）
   - エラーコードをenum（ErrorCode）で管理する」
```

### 問題4：AIを活用したテストコード生成

以下のPython関数に対して、コード生成AIに渡す「テスト生成の指示文」を作成してください。網羅的で意味のあるテストが生成されるよう、具体的な指示を設計することが目的です。

```python
def calculate_shipping_fee(weight_kg: float, destination: str, is_express: bool = False) -> int:
    """
    配送料を計算する関数
    - weight_kg: 荷物の重量（kg）
    - destination: 配送先（"domestic" or "international"）
    - is_express: 速達指定
    - 戻り値: 配送料（円）
    """
    if weight_kg <= 0:
        raise ValueError("重量は0より大きい値を指定してください")
    if destination not in ("domestic", "international"):
        raise ValueError("配送先は 'domestic' または 'international' を指定してください")

    if destination == "domestic":
        if weight_kg <= 1.0:
            base_fee = 500
        elif weight_kg <= 5.0:
            base_fee = 800
        elif weight_kg <= 10.0:
            base_fee = 1200
        else:
            base_fee = 1200 + int((weight_kg - 10.0) * 100)
    else:  # international
        if weight_kg <= 1.0:
            base_fee = 2000
        elif weight_kg <= 5.0:
            base_fee = 3500
        else:
            base_fee = 3500 + int((weight_kg - 5.0) * 500)

    if is_express:
        base_fee = int(base_fee * 1.5)

    return base_fee
```

作成した指示文と、その指示で生成されることが期待されるテストケースの一覧を記述してください。

```

### 問題5：AIを活用したコードレビュー依頼

以下のJavaScriptコードには複数の問題が含まれています。コード生成AIに効果的なレビューを依頼するための指示文を作成し、指摘されるべき問題点を自分でも予測してください。

```javascript
async function getUserData(userId) {
  const response = await fetch('http://api.example.com/users/' + userId)
  const data = response.json()

  let userInfo = {}
  userInfo.name = data.name
  userInfo.email = data.email
  userInfo.age = data.age

  if (data.role == 'admin') {
    userInfo.permissions = ['read', 'write', 'delete']
  } else if (data.role == 'editor') {
    userInfo.permissions = ['read', 'write']
  } else {
    userInfo.permissions = ['read']
  }

  localStorage.setItem('user_' + oderId, JSON.stringify(userInfo))

  console.log('User data fetched: ' + userInfo.name)

  return userInfo
}
```

**期待される出力例：**
```
【AIへのレビュー依頼文】

【自分で予測する問題点】

class TestOverdueCheck:
    def test_should_reject_when_user_has_overdue_book(self):
        # Arrange
        # 返却期限が昨日のレコードを持つ利用者

        # Act
        result = service.lend_book(user_id="U001", book_id="B002")

        # Assert
        assert result.success is False
        assert "延滞" in result.message

    def test_should_allow_when_due_date_is_today(self):
        # 返却期限が今日（まだ延滞ではない）

        # Assert
        assert result.success is True

    def test_should_allow_when_no_overdue_books(self):
        # 延滞なし

        # Assert
        assert result.success is True
```

[Green] AIに実装を依頼
→ 延滞チェックロジックを追加

[Refactor] 改善
→ チェックロジックの順序を整理（延滞チェック → 上限チェック → 在庫チェック）
→ 各チェックをプライベートメソッドに分離

========================================

【品質担保ルール】

1. テストの主導権は人間が持つ
   - AIが生成したテストを「そのまま」採用しない
   - テストが「何を検証しているか」を人間が理解・説明できること
   - テスト名を見ただけで仕様が分かるレベルの命名にする

2. Greenフェーズでの過剰実装を禁止する
   - AIに「最小限の実装」を明示的に指示する
   - テストに書かれていない機能を先回りして実装させない
   - 「将来必要になるかもしれない」コードはRefactorフェーズで検討する

3. テスト実行を省略しない
   - 各フェーズの完了条件は「テスト実行結果」で判断する
   - Red: テストが意図した理由で失敗していること
   - Green: 全テストがパスしていること
   - Refactor: 全テストがパスし続けていること

4. AIの出力を必ずレビューする
   - コードレビューチェックリストを使用する
   - セキュリティ観点のチェックは人間が必ず行う
   - 不明なコードは「なぜこの実装か」をAIに説明させる

5. 1サイクルの粒度を小さく保つ
   - 1サイクルで追加する機能は1つだけ
   - テストケースは1サイクルあたり3〜5個を目安にする
   - 大きな機能は複数サイクルに分割する

6. バージョン管理との連携
   - 各Green到達時点でコミットする
   - Refactor完了時にもコミットする
   - コミットメッセージに「Red/Green/Refactor」のフェーズを記載する
   - 例: "Green: 貸出上限チェック機能を実装"
```

</details>

### 問題8：AI活用コーディングの社内ガイドライン策定

あなたの開発チーム（10名規模）で、コード生成AIの利用に関するガイドラインを策定することになりました。以下の観点を網羅したガイドラインを作成してください。

**必須の観点：**
1. 利用を推奨する場面と禁止する場面
2. セキュリティポリシー（機密コードの取り扱い）
3. 品質管理（AIが生成したコードのレビュー基準）
4. ライセンスと知的財産
5. 学習と成長への影響

**期待される出力例：**
```
【ガイドライン名】コード生成AI利用ガイドライン

第1条：利用の基本方針
(1) コード生成AIは「開発者の生産性を向上させるツール」として活用する
(2) AIの出力に対する最終責任は開発者本人が負う
...
```

<details>
<summary>ヒント</summary>

- 「禁止場面」はセキュリティと機密性の観点から明確に線引きしましょう
- コードレビューでは「AI生成コードであること」を明示する仕組みが有効です
- オープンソースライセンスの汚染（ライセンスコンタミネーション）リスクにも触れましょう
- ジュニアエンジニアの成長を阻害しないための配慮も重要です

</details>

<details>
<summary>解答例</summary>

```
【ガイドライン名】
コード生成AI利用ガイドライン（第1版）

制定日：2026年4月1日
適用範囲：開発部門の全メンバー
改訂サイクル：四半期ごとにレビュー

========================================

第1条：利用の基本方針

(1) コード生成AIは「開発者の生産性を向上させるツール」として
    位置づけ、積極的な活用を推奨する。

(2) AIの出力に対する最終的な品質責任は、それを採用した
    開発者本人が負う。「AIが生成した」ことは品質問題の
    免責事由にはならない。

(3) AIを活用する目的は、定型作業の効率化・コード品質の向上・
    学習の加速であり、「考えること」の代替ではない。

========================================

第2条：利用を推奨する場面

(1) 積極的に活用すべき場面：
    a. ボイラープレートコードの生成
    b. ユニットテスト・結合テストの作成
    c. ドキュメント・コードコメントの生成
    d. 正規表現やSQL文の作成と検証
    e. コードレビューの補助（AIに事前レビューを依頼）
    f. リファクタリングの提案と実施
    g. 新しい技術やフレームワークの学習

(2) 補助的に活用すべき場面（人間のレビュー必須）：
    a. ビジネスロジックの実装
    b. データベーススキーマ設計
    c. API設計
    d. パフォーマンス最適化

========================================

第3条：利用を禁止する場面

(1) 以下の場面ではクラウドベースのAIサービスの利用を禁止する：
    a. 機密情報を含むコードの入力
       - APIキー、シークレットキー、認証トークン
       - 顧客の個人情報を含むテストデータ
       - 社内システムのインフラ構成情報
    b. 未公開の知的財産に関するコード
       - 特許出願前の独自アルゴリズム
       - 競争優位性に直結するビジネスロジック
    c. 契約上NDAで保護されたクライアントのコード

(2) 上記に該当する場合の代替手段：
    - 機密部分を抽象化・匿名化してからAIに相談する
    - ローカル実行可能なAIツールを使用する
    - 機密部分を除外し、インターフェースのみをAIに提示する

========================================

第4条：セキュリティポリシー

(1) AIサービスに送信するコードは、送信前に以下を確認する：
    □ APIキーや認証情報がハードコードされていないか
    □ 個人情報を含むテストデータが含まれていないか
    □ .envファイルや設定ファイルの内容が含まれていないか

(2) AI生成コードのセキュリティレビュー基準：
    a. SQLインジェクション対策（パラメータバインディング）
    b. XSS対策（エスケープ処理）
    c. 認証・認可の適切な実装
    d. 入力バリデーションの網羅性
    e. 機密情報のログ出力がないこと

(3) セキュリティに関わるコード（認証、暗号化、権限管理）は
    AI生成に依存せず、確立されたライブラリを使用し、
    セキュリティレビュー担当者の承認を得る。

========================================

第5条：品質管理

(1) AI生成コードのコードレビュー基準：
    a. 生成コードを「自分で書いたコード」と同等以上の基準でレビューする
    b. コードの意図と動作を完全に理解してからマージする
    c. AI生成コードにもユニットテストを必ず追加する

(2) プルリクエストでの明示：
    a. AI生成コードを含むPRには、その旨をPR説明に記載する
    b. どの部分がAI生成で、どこを人間が修正したか明記する
    c. 記載例：「テストコードはClaude Codeで生成し、
       エッジケースを3件手動で追加しました」

(3) AI生成コードの品質チェックリスト：
    □ コーディング規約に準拠しているか
    □ 不要なコメントや余分なコードがないか
    □ ハルシネーション（存在しないAPIや関数の使用）がないか
    □ 非推奨（deprecated）なメソッドを使っていないか
    □ プロジェクトの既存コードスタイルと一貫性があるか
    □ テストが意味のあるアサーションをしているか

========================================

第6条：ライセンスと知的財産

(1) AI生成コードのライセンスリスクを認識し、以下を遵守する：
    a. AIが生成したコードが既存のOSSコードと類似していないか
       確認する（特に特徴的なアルゴリズムやデータ構造）
    b. 生成されたコードに特定のOSSライセンス
       （GPL、AGPL等）が混入していないか注意する
    c. 疑わしい場合はコード検索ツールで確認する

(2) 社内で開発したコードの権利は会社に帰属する。
    AIツールの利用規約で権利の帰属に制約がある場合は、
    そのツールの使用を事前に法務と協議する。

(3) 使用するAIツールのライセンス一覧を管理し、
    利用規約の変更時にレビューする。

========================================

第7条：学習と成長への配慮

(1) ジュニアエンジニア（経験2年未満）への特別ルール：
    a. 基礎的な文法・アルゴリズムの学習段階では、
       まず自力でコードを書いてからAIの提案と比較する
    b. AIに「答え」を直接求めるのではなく、
       「ヒント」や「レビュー」を求める使い方を推奨する
    c. 週1回、AIなしでペアプログラミングの時間を設ける

(2) 全メンバー共通：
    a. AIの提案を鵜呑みにせず、「なぜこのコードか」を
       常に理解してから採用する
    b. AIが提案する新しいパターンや技法は、
       チーム内で共有・議論する機会を設ける
    c. 月1回の振り返りで、AI活用による生産性と
       スキル成長のバランスを確認する

(3) スキルアップのための活用法：
    a. 自分のコードをAIにレビューさせ、改善点を学ぶ
    b. AIが生成したコードを手動でリファクタリングする練習
    c. AIに複数の実装パターンを提示させ、比較検討する

========================================

【付則】

1. 本ガイドラインは四半期ごとに見直しを行う
2. AI技術やツールの進化に応じて随時改訂する
3. ガイドラインの解釈に迷う場合はテックリードに相談する
4. 重大なインシデント発生時は即座にガイドラインをレビューする
5. 新しいAIツール導入時は、事前にセキュリティチームの
   評価を受けること
```

</details>

# 実践課題05：リスク登録簿の作成 ★2

> **難易度**: ★★☆☆☆（基礎）
> **前提知識**: 第5章（リスク管理）
> **課題の種類**: 計画策定
> **学習目標**: リスクの特定・分析・対応計画を一貫して行い、実務で使えるリスク登録簿（Risk Register）を作成できるようになる

---

## 完成イメージ

以下のようなリスク登録簿を作成します。

```
==================================================
  リスク登録簿（Risk Register）
  プロジェクト: ECサイトリニューアル
  更新日: 2026-04-06
==================================================

No  リスク事象             分類    確率  影響度  スコア  対応戦略  対応策                     担当    状態
R01 主要開発者の退職       人員    高    高      9      軽減     ナレッジ共有の徹底          PM      監視中
R02 要件の大幅追加         スコープ 中    高      6      回避     変更管理プロセスの厳格化    PM      監視中
R03 外部API仕様変更        技術    中    中      4      軽減     アダプタパターンで疎結合化  開発L   監視中
R04 性能要件未達           技術    低    高      3      軽減     負荷テストを早期実施        開発L   監視中
R05 テスト工数の不足       品質    中    中      4      転嫁     テスト外部委託を検討        PM      監視中

確率×影響度マトリクス：
            影響度 低(1)  中(2)  高(3)
  確率 高(3)   3       6      9★
       中(2)   2       4      6★
       低(1)   1       2      3
```

## 課題の要件

以下のシナリオを読み、リスク登録簿を作成してください。

### シナリオ

あなたは中規模のSIer企業のPMです。以下のプロジェクトのリスク管理を担当しています。

**プロジェクト概要：**
- クライアント：地方自治体
- 内容：住民向けオンライン申請システムの新規構築
- 期間：8ヶ月（2026年4月〜11月）
- 予算：2,000万円
- チーム：PM 1名、開発者 4名、テスター 1名
- 技術：Webアプリ（React + Java Spring Boot）
- 特記事項：
  - クライアントのIT部門は小規模（2名）で、意思決定に時間がかかる
  - 個人情報を扱うため、セキュリティ要件が厳しい
  - 自治体独自の業務フローに合わせたカスタマイズが必要
  - 年度末（3月）までに本稼働が求められていたが、4月開始に延期された経緯あり

### 作成すべきもの

1. **リスクの特定**: 最低8個のリスクを洗い出す（脅威6個以上、好機1個以上）
2. **定性分析**: 確率と影響度を「高・中・低」の3段階で評価し、スコアを算出
3. **対応戦略**: 各リスクに「回避・軽減・転嫁・受容」（脅威）または「活用・強化・共有・受容」（好機）を設定
4. **対応策**: 具体的なアクションを記述
5. **確率×影響度マトリクス**: リスクを視覚的にマッピング

## ステップガイド

<details>
<summary>ステップ1：リスクを洗い出す</summary>

以下のカテゴリごとにブレインストーミングしましょう。

```
リスク洗い出しカテゴリ：

【技術リスク】
  - 使用技術に関する不確実性はあるか？
  - セキュリティに関する懸念はあるか？
  - 性能面での課題はあるか？

【スコープリスク】
  - 要件が変わる可能性はあるか？
  - 曖昧な要件はあるか？
  - スコープクリープが起きやすい状況か？

【人員リスク】
  - メンバーのスキルは十分か？
  - 離脱・異動のリスクはあるか？
  - クライアント側の体制は十分か？

【外部リスク】
  - 法規制の変更はあるか？
  - 外部ベンダーへの依存はあるか？
  - 市場・政治環境の影響はあるか？

【スケジュールリスク】
  - クリティカルパス上のボトルネックはあるか？
  - 意思決定の遅延リスクはあるか？

【好機（プラスのリスク）】
  - 予想以上にうまくいく可能性はあるか？
  - 再利用できる資産はあるか？
```

</details>

<details>
<summary>ステップ2：確率と影響度を評価する</summary>

各リスクに対して、以下の基準で評価します。

```
確率の基準：
  高(3): 発生確率60%以上。過去の類似プロジェクトで頻繁に発生
  中(2): 発生確率30〜60%。状況によっては発生しうる
  低(1): 発生確率30%未満。可能性は低いが想定しておく

影響度の基準：
  高(3): スケジュール2週間以上の遅延、またはコスト10%以上の超過
  中(2): スケジュール1〜2週間の遅延、またはコスト5〜10%の超過
  低(1): スケジュール1週間未満の遅延、またはコスト5%未満の超過

リスクスコア = 確率 × 影響度（1〜9の範囲）
  スコア 6〜9: 重大リスク → 優先的に対応策を実行
  スコア 3〜4: 中程度リスク → 対応策を準備し監視
  スコア 1〜2: 低リスク → 受容して監視
```

</details>

<details>
<summary>ステップ3：対応戦略と対応策を決める</summary>

```
脅威（マイナスのリスク）への対応戦略：

  回避（Avoid）:    リスクの原因を取り除く
    例: 不安定な技術を使わない → 実績のある技術に変更

  軽減（Mitigate）: 発生確率または影響度を下げる
    例: メンバー退職リスク → ペアプロでナレッジを共有

  転嫁（Transfer）: リスクを第三者に移す
    例: セキュリティ診断 → 専門業者に委託

  受容（Accept）:   リスクを認識した上で特に対策しない
    例: 軽微なUI変更要望 → 発生時に個別対応

好機（プラスのリスク）への対応戦略：

  活用（Exploit）:  好機を確実に実現する
  強化（Enhance）:  好機の発生確率を上げる
  共有（Share）:    好機のメリットを他者と分かち合う
  受容（Accept）:   特に追加行動はせず、発生したら享受する
```

</details>

## 解答例

<details>
<summary>解答例（初心者向け）</summary>

```
==================================================
  リスク登録簿（Risk Register）
  プロジェクト: 住民向けオンライン申請システム
  更新日: 2026-04-06
==================================================

No   リスク事象                          分類        確率  影響度  スコア  対応戦略
R01  クライアントの意思決定遅延           外部        高    高      9      軽減
R02  個人情報漏洩・セキュリティ脆弱性     技術        中    高      6      軽減/転嫁
R03  業務フローの要件が複雑で膨張         スコープ    高    中      6      回避
R04  開発メンバーのSpring Boot経験不足    人員        中    中      4      軽減
R05  自治体の制度変更による仕様変更       外部        低    高      3      受容
R06  テスト工数の見積もり不足             品質        中    中      4      軽減
R07  本番環境（自治体インフラ）の構築遅延  外部        中    高      6      軽減
R08  類似自治体案件の資産を再利用可能     好機        中    中      4      強化

──────────────────────────────────────────────────
対応策詳細：

R01 クライアントの意思決定遅延（スコア9 ─ 最重要）
  対応策:
  - 週次定例会議を設定し、決定事項を毎週確認する
  - 承認が必要な事項は事前にメールで送付し、会議で即決を促す
  - 意思決定が1週間以上遅延した場合のエスカレーションルートを決めておく
  担当: PM

R02 個人情報漏洩・セキュリティ脆弱性（スコア6）
  対応策:
  - セキュリティ専門業者による脆弱性診断を実施する（転嫁）
  - 開発ガイドライン（SQLインジェクション対策等）を策定する（軽減）
  - 個人情報の取り扱いルールを文書化する
  担当: 開発リーダー + 外部業者

R03 業務フローの要件が複雑で膨張（スコア6）
  対応策:
  - 要件定義の早い段階でMoSCoW優先度付けを実施する
  - スコープを「第1期リリース」「第2期リリース」に分割する
  - 変更要求はすべて変更管理プロセスを経由させる
  担当: PM

R04 開発メンバーのSpring Boot経験不足（スコア4）
  対応策:
  - プロジェクト開始直後に2日間の技術研修を実施する
  - 経験豊富なメンバーとペアプログラミングを行う
  担当: 開発リーダー

R05 自治体の制度変更による仕様変更（スコア3）
  対応策:
  - 受容（発生した場合に変更管理プロセスで対応）
  - 設定ファイルで業務ルールを変更可能な設計にしておく
  担当: PM

R06 テスト工数の見積もり不足（スコア4）
  対応策:
  - テスト計画を早期に作成し、工数を見積もる
  - 自動テストを導入して手動テスト工数を削減する
  担当: テスター

R07 本番環境の構築遅延（スコア6）
  対応策:
  - 自治体IT部門との環境構築スケジュールを早期に合意する
  - 開発環境で本番相当の構成を再現しておく
  - 環境構築作業の手順書を事前に作成する
  担当: PM + 開発リーダー

R08 類似自治体案件の資産再利用（スコア4 ─ 好機）
  対応策:
  - 社内の過去案件を調査し、再利用可能なコンポーネントを特定する（強化）
  - 共通部品として整理し、今回のプロジェクトでも再利用可能にする
  担当: 開発リーダー
```

<details>
<summary>解答例（改良版 ─ トリガー条件と残存リスク付き）</summary>

改良版では、各リスクに「トリガー条件」「残存リスク」「二次リスク」を追加します。

```
==================================================
  リスク登録簿（Risk Register）v2.0
  プロジェクト: 住民向けオンライン申請システム
  更新日: 2026-04-06 / 次回レビュー: 2026-04-20
==================================================

■ R01 クライアントの意思決定遅延
  ─────────────────────────────
  分類: 外部 / 確率: 高 / 影響度: 高 / スコア: 9
  対応戦略: 軽減

  根拠: クライアントIT部門が2名で、過去に延期の経緯あり。
        類似案件でも意思決定に平均2週間を要した実績がある。

  トリガー条件:
    - 承認依頼から5営業日以内に回答がない
    - 定例会議が2回連続で中止・延期になる

  対応策:
    (1) 事前対応: 承認事項を一覧化し、スケジュールに「承認期限」を設定
    (2) 事前対応: 意思決定者を明確化（誰がOKを出せるか）
    (3) トリガー時: PMからクライアント責任者にエスカレーション
    (4) トリガー時: 暫定案で作業を進め、後から確定する方式に切り替え

  残存リスク: 軽減後もスコア6（確率を中に下げられる見込み）
  二次リスク: 暫定案で進めた場合、確定時に手戻りが発生するリスク

  担当: PM / 状態: 監視中 / レビュー頻度: 週次

■ R02 個人情報漏洩・セキュリティ脆弱性
  ─────────────────────────────
  分類: 技術 / 確率: 中 / 影響度: 高 / スコア: 6
  対応戦略: 軽減 + 転嫁

  根拠: 住民の個人情報（氏名、住所、マイナンバー等）を扱う。
        漏洩した場合、法的責任・信頼失墜のリスクが甚大。

  トリガー条件:
    - セキュリティ診断でCritical/High指摘が検出された場合
    - OWASP Top 10に該当する脆弱性が発見された場合

  対応策:
    (1) 回避: マイナンバーは本システムでは保持しない設計とする
    (2) 軽減: セキュアコーディングガイドラインを策定し遵守
    (3) 軽減: 開発中に静的解析ツール（SonarQube等）を導入
    (4) 転嫁: リリース前に外部セキュリティ監査を実施（予算確保済み）
    (5) 転嫁: サイバーセキュリティ保険への加入を検討

  残存リスク: スコア3（確率を低に下げられる見込み）
  二次リスク: セキュリティ対策の過剰実装による開発遅延

  担当: 開発リーダー / 状態: 対応中 / レビュー頻度: 週次

■ R03〜R08（同様のフォーマットで記載）
  ... （省略）

==================================================
  リスク対応コスト概算
==================================================
  R01 対応コスト: 0円（プロセス改善のみ）
  R02 対応コスト: 50万円（外部診断30万 + ツール導入20万）
  R04 対応コスト: 10万円（研修費用）
  R08 見込み効果: -80万円（再利用による工数削減）
  ──────────────────
  リスク対応予算合計: 約60万円（総予算の3%）
  ※ 一般的にリスク対応予算は総予算の5〜15%が目安

==================================================
  レビュースケジュール
==================================================
  - 週次: R01（最重要リスク）の状態確認
  - 隔週: 全リスクのステータスレビュー
  - 月次: リスク登録簿の全体見直し（新規リスクの追加）
  - フェーズ移行時: リスク登録簿の大幅見直し
```

**初心者向けとの違い：**
- **トリガー条件**: 「いつ対応策を発動するか」を事前に決めておくことで、対応の遅れを防ぎます
- **残存リスク**: 対応策を実行した後もリスクは完全にはなくならないことを認識します
- **二次リスク**: 対応策自体が新たなリスクを生む可能性を考慮します
- **リスク対応コスト**: 対策にもコストがかかることを明示し、費用対効果を検討します
- **レビュースケジュール**: リスク管理は一度作って終わりではなく、定期的に見直すものです

</details>

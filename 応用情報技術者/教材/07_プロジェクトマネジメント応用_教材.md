## 7.1 EVM（Earned Value Management）

### 7.1.1 EVMとは

**EVM（アーンドバリューマネジメント）** は、プロジェクトのコストとスケジュールの進捗を定量的に測定・管理する手法です。

### 7.1.2 EVMの3つの基本値

| 指標 | 英語 | 意味 |
|------|------|------|
| PV | Planned Value | 計画出来高（いつまでにいくら使う予定だったか） |
| EV | Earned Value | 達成出来高（実際にどれだけの成果を達成したか） |
| AC | Actual Cost | 実コスト（実際にいくら使ったか） |

```
金額
 ↑
 │          PV(計画) ╱
 │               ╱
 │      EV(成果)╱ ─── SV（スケジュール差異）
 │           ╱     │
 │    AC(実績)╱      │── CV（コスト差異）
 │         ╱        │
 │       ╱          │
 │     ╱            │
 ├───╱──────────────┤
 │                   │
 └───────────────────┴──→ 時間
                   現在
```

### 7.1.3 差異分析

| 指標 | 計算式 | 意味 |
|------|--------|------|
| SV（スケジュール差異） | EV - PV | 正:前倒し、負:遅延 |
| CV（コスト差異） | EV - AC | 正:予算内、負:超過 |
| SPI（スケジュール効率指数） | EV / PV | 1超:前倒し、1未満:遅延 |
| CPI（コスト効率指数） | EV / AC | 1超:予算内、1未満:超過 |

### 7.1.4 完了時の予測

| 指標 | 計算式 | 意味 |
|------|--------|------|
| EAC（完了時総コスト見積り） | BAC / CPI | 現在のペースが続いた場合の総コスト |
| ETC（残作業コスト見積り） | EAC - AC | 完了までに必要な追加コスト |
| VAC（完了時コスト差異） | BAC - EAC | 完了時の予算超過/余裕 |

※ BAC（Budget at Completion）= プロジェクト全体の予算

### 7.1.5 EVM計算例

**例題:**
プロジェクト全体予算(BAC)=1,000万円。現時点での計画出来高(PV)=600万円、達成出来高(EV)=500万円、実コスト(AC)=550万円。

```
SV = EV - PV = 500 - 600 = -100万円（遅延）
CV = EV - AC = 500 - 550 = -50万円（超過）
SPI = EV / PV = 500 / 600 ≈ 0.83（17%遅延）
CPI = EV / AC = 500 / 550 ≈ 0.91（9%超過）
EAC = BAC / CPI = 1,000 / 0.91 ≈ 1,099万円
VAC = BAC - EAC = 1,000 - 1,099 = -99万円（約100万円の超過見込み）
```

**ポイントまとめ:**
- PV/EV/ACの3つの基本値を明確に区別する
- SV, CV, SPI, CPIの計算ができるようにする
- EAC = BAC / CPI の計算は午前・午後共に頻出

## 7.2 クリティカルパス法（CPM）

### 7.2.1 アローダイアグラム

**PERT図（アローダイアグラム）** を使って作業の依存関係と所要時間を表現します。

```
        作業B(5日)
       ╱──────→②──────╲
①───╱   作業C(3日)      ╲───→④───作業F(4日)───→⑤
  ╲──→②'─────→③──作業D(6日)──╱
   作業A(2日)    作業E(1日)
                 ③───→④
```

### 7.2.2 クリティカルパスの求め方

**手順:**
1. **最早開始時刻（ES）** を前進計算で求める
2. **最遅開始時刻（LS）** を後退計算で求める
3. **余裕時間（フロート）= LS - ES** を計算
4. フロート=0の作業を結ぶ → **クリティカルパス**

**例題:**

```
作業一覧:
  A: 3日（先行なし）
  B: 4日（先行なし）
  C: 5日（Aの後）
  D: 2日（Bの後）
  E: 3日（C, Dの後）

ネットワーク図:
      A(3) → C(5) →
①                    ④ → E(3) → ⑤
      B(4) → D(2) →

前進計算（最早開始/最早完了）:
  A: ES=0, EF=3
  B: ES=0, EF=4
  C: ES=3, EF=8
  D: ES=4, EF=6
  E: ES=max(8,6)=8, EF=11

プロジェクト全体: 11日

後退計算（最遅開始/最遅完了）:
  E: LF=11, LS=8
  C: LF=8, LS=3
  D: LF=8, LS=6
  A: LF=3, LS=0
  B: LF=6, LS=2

フロート:
  A: 0-0=0  ← クリティカル
  B: 2-0=2
  C: 3-3=0  ← クリティカル
  D: 6-4=2
  E: 8-8=0  ← クリティカル

クリティカルパス: A → C → E（11日）
```

### 7.2.3 クラッシングとファストトラッキング

| 手法 | 説明 | リスク |
|------|------|--------|
| クラッシング | 資源（人員・費用）を追加して作業期間を短縮 | コスト増大 |
| ファストトラッキング | 本来順序的な作業を並行実施 | 手戻りリスク |

**ポイントまとめ:**
- クリティカルパスはフロート=0の作業の経路
- 前進計算・後退計算の手順を確実にマスターする
- クリティカルパス上の作業が遅延するとプロジェクト全体が遅延する

## 7.3 PERT（Program Evaluation and Review Technique）

### 7.3.1 3点見積り

PERTでは各作業の所要時間を3つの値で見積もります。

```
楽観値(O: Optimistic):   最も短い場合の所要時間
最可能値(M: Most Likely): 最も可能性が高い所要時間
悲観値(P: Pessimistic):  最も長い場合の所要時間
```

**期待値と標準偏差:**
```
期待値(Te) = (O + 4M + P) / 6
分散(σ²) = {(P - O) / 6}²
標準偏差(σ) = (P - O) / 6
```

### 7.3.2 PERT計算例

**例題:**
各作業の見積り:

| 作業 | 楽観値(O) | 最可能値(M) | 悲観値(P) | 期待値(Te) | 分散(σ²) |
|------|----------|------------|----------|-----------|---------|
| A | 2 | 3 | 10 | (2+12+10)/6=4 | (8/6)²≈1.78 |
| B | 4 | 5 | 6 | (4+20+6)/6=5 | (2/6)²≈0.11 |
| C | 3 | 4 | 11 | (3+16+11)/6=5 | (8/6)²≈1.78 |

クリティカルパスが A→C の場合:
```
プロジェクト期待値 = 4 + 5 = 9日
プロジェクト分散 = 1.78 + 1.78 = 3.56
プロジェクト標準偏差 = √3.56 ≈ 1.89日
```

**完了確率の推定:**
正規分布を仮定し、「X日以内に完了する確率」を求められます。

```
例: 11日以内に完了する確率
Z = (11 - 9) / 1.89 ≈ 1.06

正規分布表より Z=1.06 → 約85.5%
```

**ポイントまとめ:**
- 3点見積りの期待値 = (O + 4M + P) / 6 は必ず覚える
- 分散 = {(P - O) / 6}² も頻出
- クリティカルパス上の分散を合算して全体の標準偏差を求める

## 7.4 品質管理手法

### 7.4.1 QC七つ道具

| 道具 | 用途 |
|------|------|
| パレート図 | 問題の重要度を棒グラフと累積線で表示（80:20の法則） |
| 特性要因図（魚骨図） | 問題の原因を体系的に分析 |
| ヒストグラム | データの分布を棒グラフで表示 |
| 散布図 | 2変数の相関を表示 |
| 管理図 | 工程が安定しているか時系列で監視 |
| チェックシート | データの収集と整理 |
| 層別 | データをカテゴリ別に分類して分析 |

### 7.4.2 新QC七つ道具

| 道具 | 用途 |
|------|------|
| 親和図法 | 混沌とした情報をグループ化（KJ法） |
| 連関図法 | 問題の因果関係を複雑な矢印で表現 |
| 系統図法 | 目的と手段を階層的に展開 |
| マトリックス図法 | 2つの要素の関連を行列で表現 |
| アローダイアグラム | 作業の順序関係と所要時間を表現 |
| PDPC法 | 問題の発生を予測し代替策を事前に準備 |
| マトリックスデータ解析法 | 数値データを主成分分析等で分析 |

### 7.4.3 管理図の詳細

```
      UCL(上方管理限界) ─────────────────
         │
  値     │    ×  ×      ×
         │  ×    ×  ×  ×  ×
      CL │──×──×──────×──────── 中心線
         │    ×        ×
         │      ×
      LCL(下方管理限界) ─────────────────
         └──────────────────────→ 時間

管理限界を超える点 → 工程に異常あり
連（ラン）: 中心線の片側に連続する点 → 偏りの兆候
```

**ポイントまとめ:**
- QC七つ道具と新QC七つ道具の名前と用途を区別する
- パレート図は「重要な少数」を特定する道具
- 管理図はUCL/LCLと異常判定ルールを理解する

## 7.5 調達管理

### 7.5.1 調達プロセス

```
調達計画 → 調達実行 → 調達管理 → 調達終結
```

### 7.5.2 契約形態

| 契約形態 | 説明 | リスク負担 |
|---------|------|-----------|
| 定額契約（FP） | 成果物に対して固定金額 | 受注者側がリスク負担 |
| 実費償還契約（CR） | 実際にかかった費用を精算 | 発注者側がリスク負担 |
| T&M契約 | 時間単価×工数で精算 | 双方がリスク分担 |

### 7.5.3 提案依頼のプロセス

```
情報提供依頼(RFI: Request for Information)
  ↓ ベンダーの技術力・実績の情報収集
提案依頼(RFP: Request for Proposal)
  ↓ 要件を提示して提案書を要求
提案書評価
  ↓ 評価基準に基づく比較・選定
契約締結
```

**RFP（提案依頼書）に含める内容:**
- システム概要・目的
- 機能要件・非機能要件
- 予算・スケジュール
- 提案書の記載項目・評価基準
- 契約条件

**ポイントまとめ:**
- RFI→RFP→提案評価→契約の流れを覚える
- 定額契約は受注者リスク、実費償還は発注者リスク
- 午後試験では調達プロセスの適切性を問う問題が出る

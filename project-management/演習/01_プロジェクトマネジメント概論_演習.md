## 基本問題

### 問題1：プロジェクトとオペレーションの区別

以下の業務がプロジェクトかオペレーション（日常業務）かを判定し、その理由を述べてください。

1. 社内チャットツールの新規導入
2. 毎月のサーバーバックアップ作業
3. 既存システムのReact移行
4. 日次のデプロイ作業
5. 新卒エンジニア向け研修プログラムの設計

**期待される出力例：**

```
1. プロジェクト - 有期的で独自の成果物（チャットツールの導入完了）を生む
2. オペレーション - 継続的に反復される日常業務
3. プロジェクト - 有期的でシステム移行という独自の成果物を生む
4. オペレーション - 継続的に反復される日常業務
5. プロジェクト - 有期的で研修プログラムという独自の成果物を生む
```

<details>
<summary>ヒント</summary>

プロジェクトの特徴は「有期的」かつ「独自の成果物を生む」ことです。繰り返し行われる業務はオペレーションに分類されます。

</details>

<details>
<summary>解答例</summary>

```
1. プロジェクト
   理由：チャットツールの導入は開始（選定）と終了（運用開始）が明確で、
   独自の成果物（導入されたシステムと運用ルール）を生む。

2. オペレーション
   理由：毎月繰り返される定型業務であり、終了時期が定められていない。

3. プロジェクト
   理由：React移行は開始と完了がある有期的な取り組みで、
   移行されたシステムという独自の成果物を生む。

4. オペレーション
   理由：日々繰り返される運用業務であり、反復的・継続的な活動。

5. プロジェクト
   理由：研修プログラムの「設計」は有期的な活動であり、
   設計された研修コンテンツという独自の成果物を生む。
   ※研修の「実施」が毎年行われる場合はオペレーションに分類される。
```

</details>

### 問題2：PMの三角形（トレードオフ）

以下のシナリオで、PMとしてどのようなトレードオフの提案をしますか？

**シナリオ：** ECサイトの開発プロジェクトで、リリース予定日の2週間前に顧客から「決済方法にコンビニ払いも追加してほしい」と依頼がありました。現在のスコープにはクレジットカード決済のみが含まれています。チームは5名で、増員の予算はありません。

**期待される出力例：**

```
提案A：スコープを調整する
  → コンビニ払いは次フェーズに回し、予定通りリリースする。

提案B：スケジュールを調整する
  → リリース日を2〜3週間延期し、コンビニ払いを含めてリリースする。

提案C：スコープの他の部分を削減する
  → 優先度の低い機能（例：レビュー機能）をフェーズ2に回し、
     代わりにコンビニ払いを実装する。
```

<details>
<summary>ヒント</summary>

PMの三角形（スコープ・時間・コスト）の3つの制約のうち、コスト（増員）は使えないという前提です。残る選択肢はスコープまたは時間の調整です。

</details>

<details>
<summary>解答例</summary>

```
【状況分析】
- スコープ追加の依頼（コンビニ払い）
- 時間：リリースまで2週間
- コスト：増員不可

【提案】

提案1：スケジュールを延長する（時間の調整）
  ・コンビニ払いの実装工数を見積もる（例：2〜3週間）
  ・リリース日を3週間後ろ倒しにする
  ・ステークホルダーに延期の影響を説明する
  ・メリット：全機能を含めてリリースできる
  ・デメリット：ビジネス機会の損失リスク

提案2：スコープをトレードする（スコープの調整）
  ・優先度の低い機能を特定する（MoSCoWで再評価）
  ・削減した分の工数でコンビニ払いを実装する
  ・メリット：リリース日を守れる
  ・デメリット：一部機能が使えない

提案3：フェーズ分割する（推奨）
  ・フェーズ1：予定通りクレジットカード決済でリリース
  ・フェーズ2：コンビニ払いを1〜2週間後に追加リリース
  ・メリット：リリース日を守りつつ、コンビニ払いも比較的早く提供
  ・デメリット：フェーズ2の追加テスト工数

【推奨】提案3が最もバランスが良い。早期にリリースしてフィードバックを
得ながら、コンビニ払いを追加開発する。
```

</details>

### 問題3：ステークホルダーマッピング

以下のプロジェクトのステークホルダーを、影響力と関心度の4象限に分類してください。

**プロジェクト：** 社内の勤怠管理システムのリプレイス

ステークホルダー一覧：
- A: CEO（プロジェクトのスポンサー）
- B: 人事部長（現行システムの主な利用者の管理者）
- C: 一般社員（勤怠を入力する利用者）
- D: 情報システム部（インフラ管理担当）
- E: 外部の開発ベンダー
- F: 経理部（給与計算でデータを使用）

**期待される出力例：**

```
重点管理（影響力 高・関心度 高）：B（人事部長）
満足させる（影響力 高・関心度 低）：A（CEO）
情報提供（影響力 低・関心度 高）：C（一般社員）, F（経理部）
最小限の対応（影響力 低・関心度 低）：（なし）
```

<details>
<summary>解答例</summary>

```
■ 重点管理（影響力 高・関心度 高）
  ・B: 人事部長
    → システムの主要利用者の管理者であり、要件に大きな影響力を持つ。
      日常業務に直結するため関心度も高い。
  ・D: 情報システム部
    → インフラや技術選定に影響力があり、運用を担うため関心度も高い。

■ 満足させる（影響力 高・関心度 低）
  ・A: CEO
    → 予算承認権を持つため影響力は高いが、
      システムの詳細には関心が低い。定期報告で満足させる。

■ 情報提供（影響力 低・関心度 高）
  ・C: 一般社員
    → 毎日使うシステムなので関心度は高いが、
      意思決定への影響力は低い。進捗を共有する。
  ・F: 経理部
    → 給与計算にデータを使うため関心度は高いが、
      開発の意思決定への直接的な影響力は低い。
  ・E: 外部の開発ベンダー
    → 実装を担当するため関心度は高いが、
      要件の意思決定権は持たない。

■ 最小限の対応（影響力 低・関心度 低）
  ・特になし（このプロジェクトでは全員が一定の関心を持っている）
```

</details>

## 応用問題

### 問題4：プロジェクト憲章の作成

以下の要件を基に、プロジェクト憲章を作成してください。

**状況：** あなたの会社（従業員300名のIT企業）では、社内の技術ブログプラットフォームを新規構築することになりました。現在はConfluenceに技術記事を書いているが、社外公開ができず採用ブランディングに活用できていません。

以下の要素を含めてください：
- プロジェクト名
- 背景と目的
- 成功指標（KPI）2〜3個
- 主要マイルストーン
- 想定リスク2〜3個

<details>
<summary>ヒント</summary>

教材の1.6節のプロジェクト憲章テンプレートを参考にしてください。背景には「なぜやるのか」、目的には「何を達成したいのか」を具体的に書きましょう。

</details>

<details>
<summary>解答例</summary>

```markdown
# プロジェクト憲章：社内技術ブログプラットフォーム構築

## 背景
現在、社内の技術記事はConfluenceに蓄積されているが、社外公開ができず
採用ブランディングや技術広報に活用できていない。エンジニア採用が
激化する中、技術力の対外発信は採用競争力の向上に不可欠である。

## 目的
社内外に公開可能な技術ブログプラットフォームを構築し、
技術広報と採用ブランディングの強化を図る。

## スコープ概要
【含むもの】
- 記事投稿・編集・公開機能
- Markdown対応エディタ
- タグ・カテゴリ管理
- 社内レビューワークフロー
- SEO対策

【含まないもの】
- 既存Confluence記事の自動移行（手動移行は対象）
- コメント機能（フェーズ2で検討）

## 成功指標（KPI）
- リリース後3ヶ月で記事数30件以上
- 月間PV 5,000以上（リリース後6ヶ月時点）
- エンジニア採用応募者の10%が「技術ブログを見た」と回答

## 主要マイルストーン
- 要件定義完了：2026年5月末
- MVP（最小限の公開可能なブログ）リリース：2026年7月末
- 正式リリース（レビュー機能含む）：2026年9月末

## 体制
- スポンサー：CTO
- PM：開発部 山田
- 開発チーム：エンジニア3名、デザイナー1名

## 想定リスク
1. 記事を書くエンジニアが集まらない（執筆モチベーション不足）
   → 対策：執筆インセンティブ制度の導入、テーマの提案
2. セキュリティ上の懸念（社内情報の意図しない公開）
   → 対策：公開前レビュープロセスの必須化
3. 既存ブログサービス（Zenn、Qiita）との差別化が困難
   → 対策：社内ナレッジとの連携機能で差別化
```

</details>

## チャレンジ問題

### 問題5：開発手法の選定

以下の3つのプロジェクトについて、ウォーターフォール・アジャイル・ハイブリッドのどれが最適か判断し、その理由を述べてください。

1. 銀行の基幹システム（勘定系）のリプレイス。金融庁への報告義務があり、厳密な要件定義とテストが必要。期間は2年。
2. スタートアップが開発するSNSアプリのMVP。市場の反応を見ながら方向性を決めたい。期間は3ヶ月。
3. 大企業の人事システム刷新。要件は概ね固まっているが、UIは利用者の声を聞きながら改善したい。期間は1年。

<details>
<summary>解答例</summary>

```
1. 銀行の基幹システム → ウォーターフォール（またはハイブリッド）

   理由：
   ・金融庁への報告義務があり、各工程の成果物とエビデンスが必要
   ・要件の変更が少なく、事前に厳密な定義が求められる
   ・規制産業ではドキュメントの充実が不可欠
   ・ただし、実装フェーズにスクラムを取り入れるハイブリッドも有効
     （各スプリントの成果物を規制文書として整備する）

2. SNSアプリのMVP → アジャイル（スクラム）

   理由：
   ・市場の反応を見ながら方向性を変える必要がある（変化への対応）
   ・3ヶ月という短期間で、小さく作って検証するサイクルが適している
   ・MVPの段階では要件が不確実であり、フィードバックループが重要
   ・少人数チームでスクラムを回すのに最適な規模

3. 人事システム刷新 → ハイブリッド

   理由：
   ・要件は概ね固まっている → 全体計画・要件定義はウォーターフォール的に
   ・UIは利用者フィードバックで改善 → UI開発はアジャイル的に
   ・1年という中期間で、フェーズ分割が可能
   ・具体的には：
     - 要件定義・基本設計：ウォーターフォール（3ヶ月）
     - 開発・テスト：スクラム（6ヶ月、2週間スプリント）
     - 移行・リリース：ウォーターフォール（3ヶ月）
```

</details>
